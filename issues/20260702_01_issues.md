# 1. 全体所見

- 前回 (2026-06-30) レビューで High に分類された 3 件 (A-01: StateStore リーク、A-02: SIGTERM ビジースピン、A-03: エフェメラルメッセージ DB 保存) がすべて修正済みであり、信頼性の高いリファクタリングが実施されている。
- `session_restore.py` の重複システムメッセージ問題 (旧 A-05) も修正済み。現行実装は `ctx.conv.system_prompt_content` を正規ソースとして使用し、DB 取得メッセージから `system` ロールを除外するパターンで正しく実装されている。
- `DiagnosticStore` の JSONL 二重書き込み (旧 A-06) も削除済み。ドキュメントも SQLite 単一チャンネルに整合するよう更新されている。
- `workflow_require_approval` のドキュメント未記載問題 (旧 A-04) は修正済み。`docs/05_agent_08_configuration.md` に詳細が追加されている。
- `ctx.services: Any` 型問題 (旧 A-07) は未修正のまま残存する。コメントには「暫定措置」と記載されており、明示的な改善タスクが必要。
- `auto_inject_notes` フィールドは `ToolConfig` から削除済みであることが `check_no_compat.py:46` で禁止チェックに追加されていることから確認できる。
- `ToolLoopGuard` の dedup/cycle/retry ヒントは LLM コンテキストへ注入せずに診断チャンネルのみに保存するパターンが維持されており、ドキュメントの記述 (`docs/05_agent_03_turn-processing-flow.md:100`) と依然乖離している。
- `RuntimeStats` に `stat_serialization_events`、`stat_serialization_total_overhead_ms`、`stat_memory_consistency_failures`、`stat_memory_circuit_open`、`stat_memory_fts_fallback_count` の各フィールドが追加されているが、`docs/05_agent_04_state-and-persistence.md` の `RuntimeStats` テーブルに未記載である。
- `stat_partial_completions` が `RuntimeStats` ではなく `LLMClient` インスタンス (`ctx.services.llm`) に存在する設計は一貫性の観点で改善余地があるが、運用上の問題は限定的。
- アーキテクチャ分離度は引き続き高い。`AgentREPL` は UI のみ、`StartupOrchestrator` が起動シーケンスを担い、`Orchestrator` がターン処理を担う構造は維持されている。

## A-07 (旧 A-07)
**Status:** 継続 (未修正)

**Title:** `ctx.services` が `Any` 型であり型安全性が失われている

**Category:** Design flaw / Testability gap

**Severity:** Medium

**Affected area:** `scripts/agent/context.py:186` — `self.services: Any = None`

**Why it is a problem:**
`AgentContext.services` が `Any` 型として宣言されている (`context.py:186`)。コメントには「cascading union-attr errors を避けるための暫定措置」と記載されているが、改善が行われていない。`orchestrator.py:358` (`ctx.services.hist_mgr`)、`orchestrator.py:362` (`ctx.services.audit_logger`)、`orchestrator.py:513` (`_build_turn_end_event()` 内 `ctx.services.llm`) など多数箇所でノーガードアクセスが行われており、型チェッカーがこれらの誤りを検出できない。`services_required` プロパティ (`context.py:189`) が存在するが、使用率は低い。

**Evidence:**
- コードから確認: `context.py:186` — `self.services: Any = None`
- コードから確認: `context.py:185` — コメント「cascading union-attr errors」暫定措置
- コードから確認: `orchestrator.py:358, 362, 513` など多数の `ctx.services.xxx` 直接アクセス
- コードから確認: `repl.py:307` — `svc = self._ctx.services; if svc is not None:` のような手動 None チェック

**Recommended direction:**
`ctx.services` を `AppServices | None` に変更し、`orchestrator.py` と `llm_turn_runner.py` での `ctx.services.xxx` アクセスを `ctx.services_required.xxx` 経由に変更する。

## A-09 (旧 A-09)
**Status:** 継続 (部分確認)

**Title:** `HistoryManager._call_compress_llm()` の HTTP タイムアウト未設定

**Category:** Reliability/recovery weakness

**Severity:** Low

**Affected area:** `scripts/agent/history.py:207–214` — `await self._http.post(self._llm_url, json={...})` — timeout パラメータなし

**Why it is a problem:**
`_call_compress_llm()` が `httpx.AsyncClient.post()` を呼ぶ際に timeout を明示的に指定していない。`factory.py:114` で `httpx.AsyncClient(timeout=ctx.cfg.llm.http_timeout)` として作成されたクライアントを HistoryManager が共有しているため、クライアントデフォルトの timeout (30秒) が適用される。圧縮 LLM が応答しない場合に最大 30 秒ブロックするが、HistoryCompressionError が発生した場合は `_fallback_truncate()` で適切にフォールバックされる。直接的な危険性は低いが、明示的な timeout パラメータがない点は保守性の観点で改善余地がある。

**Evidence:**
- コードから確認: `history.py:207` — timeout 指定なし
- コードから確認: `factory.py:114` — `httpx.AsyncClient(timeout=ctx.cfg.llm.http_timeout)`

**Recommended direction:**
`_call_compress_llm()` の POST 呼び出しに `timeout=httpx.Timeout(self._http.timeout)` または専用設定フィールド `context_compress_timeout` を追加する。

## A-13 (新規)
**Title:** `ctx.services.llm.stat_partial_completions` が `RuntimeStats` でなく `LLMClient` に存在する設計的非一貫性

**Category:** Design flaw

**Severity:** Low

**Affected area:**
- `scripts/agent/repl.py:385–388` — `ctx.services.llm.stat_partial_completions` への直接アクセス
- `scripts/agent/orchestrator.py:616` — 同フィールドへのインクリメント

**Why it is a problem:**
`stat_partial_completions` は `LLMClient` インスタンス上に定義されており (`shared/llm_client.py:78`)、`RuntimeStats` にはない。他の統計フィールド (`stat_turns`, `stat_tool_calls` 等) はすべて `RuntimeStats` (`ctx.stats.*`) に集約されているが、この 1 フィールドのみが `ctx.services.llm.*` に存在する。`repl.py` で `ctx.services.llm` が `None` でないことを確認してからアクセスするパターンが必要であり、`_persist_session_diagnostics()` 内でも `llm is not None` の手動チェックが行われている。

**Evidence:**
- コードから確認: `repl.py:385` — `_prev_partial = llm.stat_partial_completions if llm is not None else 0`
- コードから確認: `orchestrator.py:616` — `ctx.services.llm.stat_partial_completions += 1`
- コードから確認: `context.py:90–107` — `RuntimeStats` に `stat_partial_completions` なし

**Recommended direction:**
`stat_partial_completions` を `RuntimeStats` に移動し、`LLMClient` の該当フィールドは削除するか参照を `ctx.stats.stat_partial_completions` に切り替える。A-07 の `ctx.services` 型強化と合わせて実施することが望ましい。

## A-14 (新規)
**Title:** `_recover_pending_approvals()` が StateStore を close() せずに例外伝播する可能性がある

**Category:** Reliability/recovery weakness

**Severity:** Low

**Affected area:** `scripts/agent/startup.py:173–196` — `_recover_pending_approvals()`

**Why it is a problem:**
`startup.py:178–179` で `store = StateStore(); result = store.find_latest_pending_approval(); store.close()` というパターンを使用しているが、`find_latest_pending_approval()` が例外を投げた場合 (SQLite エラー等) は `store.close()` が呼ばれない。`_handle_workflow_engine()` や `_init_workflow_task()` では `try/finally` で `store.close()` が保証されているが、この箇所には `try/finally` がない。

**Evidence:**
- コードから確認: `startup.py:178–179` — `store.find_latest_pending_approval()` と `store.close()` が try/finally なしに順次実行
- 対照: `orchestrator.py:210,263` — `store = StateStore()` ... `finally: store.close()` パターン

**Recommended direction:**
`startup.py:_recover_pending_approvals()` の StateStore 操作を `try/finally: store.close()` で保護する。

```
        → T-01 完了後に実施; 小規模変更
  T-06  /undo 圧縮警告表示
        → ユーザー体験向上
```

**根拠:**
- High に分類される未修正問題がなく、全体的にコードの安全性は高い状態
- T-01 は型安全性の基盤であり、T-05 がその恩恵を受けるため Phase 1 に配置
- T-02/T-03 はコード変更なしのドキュメント作業であり低リスク
- T-04 は安全側の保護だが実際の影響は限定的なため Phase 3
