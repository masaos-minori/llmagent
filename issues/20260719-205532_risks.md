## Orphaned duplicate MDQ module files from incomplete rename (commit ca6b7bfe) — original finding, now resolved

**Type:** Implementation bug (dead code / maintenance hazard)

**Impact scope:** `scripts/mcp_servers/mdq/models.py`, `scripts/mcp_servers/mdq/service.py`, `scripts/mcp_servers/mdq/server.py`, `scripts/mcp_servers/mdq/tools.py`

**Summary:**

Commit `ca6b7bfe` ("refactor: rename MCP server files to eliminate duplicate filenames") renamed `models.py` → `mdq_models.py`, `service.py` → `mdq_service.py`, `server.py` → `mdq_server.py`, `tools.py` → `mdq_tools.py` across MCP server subpackages (mdq, browser, cicd, git, github, rag_pipeline, shell, web_search) and updated all internal imports and `__init__.py`/`__main__.py` wiring to the new names.

For the `mdq` subpackage specifically, the old-named files (`models.py`, `service.py`, `server.py`, `tools.py`) were never deleted. They remain in the tree, byte-identical to their `mdq_`-prefixed counterparts except for self-referential import paths (verified via `diff scripts/mcp_servers/mdq/service.py scripts/mcp_servers/mdq/mdq_service.py` and the equivalent for `server.py`).

`scripts/mcp_servers/mdq/__init__.py` and `__main__.py` — the actual runtime entry points — both import exclusively from the `mdq_`-prefixed modules. Nearly all `tests/test_mdq_*.py` files also import the `mdq_`-prefixed modules. The four orphaned files are therefore dead code: never imported by production code or by the large majority of the test suite.

**Evidence:**
- `git show ca6b7bfe --stat` does not list `scripts/mcp_servers/mdq/{models,service,server,tools}.py` as touched — only the new `mdq_*.py` files and `__init__.py`/`__main__.py`/a handful of other mdq files were part of that commit.
- `git log --oneline -- scripts/mcp_servers/mdq/service.py` and `.../server.py` show pre-rename history, i.e. these are the original files left behind, not new files.
- `diff` shows the old and new files differ only in import statements referencing themselves (`from mcp_servers.mdq.models import ...` vs. `from mcp_servers.mdq.mdq_models import ...`).

**Risk:** A future contributor searching the codebase for "server.py", "service.py", "models.py", or "tools.py" (e.g. by grepping a generic name, or via editor fuzzy-file-open) may edit the orphaned copy instead of the live `mdq_*.py` file, producing changes that compile, lint, and appear correct in review, but never execute at runtime. This nearly affected the current planning cycle: the source requirement file (`requires/20260717_26_require.md`, authored before the rename) lists the pre-rename filenames as its "Target files", and only a diff-based cross-check against `__init__.py`/`__main__.py` imports caught the discrepancy before any implementation was planned against the dead files.

**Recommended action:**
1. Confirm the four orphaned files (`models.py`, `service.py`, `server.py`, `tools.py` under `scripts/mcp_servers/mdq/`) have zero import references outside of each other (already confirmed for `service.py`/`server.py`; verify `models.py` and `tools.py` similarly).
2. `git rm` the four orphaned files.
3. Check whether the same incomplete-rename pattern affects any of the other seven subpackages listed in commit `ca6b7bfe` (`browser`, `cicd`, `git`, `github`, `rag_pipeline`, `shell`, `web_search`) — the commit message claims 36 files renamed across all of them; if the same "old file never deleted" pattern recurs elsewhere, it should be cleaned up in the same pass.
4. Re-run the full test suite and `check-mcp-docs` after removal to confirm nothing was silently depending on the orphaned copies.

**Why this is unmitigated within the current plan:** The current plan (`plans/20260719-205532_plan.md`) works around this by verifying and targeting only the live `mdq_*.py` files — it does not fix the underlying dead-code duplication, since that cleanup is unrelated to the MDQ authorization/config-safety requirement being planned. This is flagged separately so the duplication itself is not lost as a known issue.

**Notes for AI reference:** If a future task references `scripts/mcp_servers/mdq/server.py`, `service.py`, `models.py`, or `tools.py` (unprefixed) by name, first verify with `rg "mcp_servers.mdq.(server|service|models|tools)\b" scripts/ tests/ -g '!scripts/mcp_servers/mdq/{server,service,models,tools}.py'` whether the reference is to the live `mdq_`-prefixed sibling or genuinely to the orphaned file; do not assume the unprefixed name is current.
