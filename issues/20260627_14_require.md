## [ops][runtime] Define operational visibility for the internal orchestrator and role MCP services

### Summary

中央 Orchestrator は内部コンポーネントであり、role MCP services とは異なる運用可視化が必要である。Phase 1 では、内部 workflow 状態と外部 MCP health を分けて監視する運用モデルを定義する。

### Problem

orchestrator を MCP server と誤認すると、/mcp 表示や watchdog の対象が混線する。実際には、workflow 状態は orchestrator/state_manager から表示し、role MCP の health は mcp_registry.py が監視する必要がある。

### Scope
- internal orchestrator visibility
- role MCP monitoring
- Event Bus monitoring(role MCP watchdog とは別経路であることの明確化)
- REPL / status command integration
- watchdog expectations

### Task
- internal orchestrator status 表示方法を定義する
- role MCP service status 表示方法を定義する
- Event Bus (`scripts/eventbus/`) の健全性監視経路を定義する(role MCP watchdog の対象外であるため)
- watchdog / health check の責務分離を定義する
- workflow count / phase progress の表示項目を定義する

### Operational Model

#### Internal Orchestrator Visibility
- orchestrator は Agent プロセス内部コンポーネントとして扱う
- workflow 状態は state_manager / get_workflow_status API から取得する
- 表示項目: active workflow count, current phase, blocked/running task counts, waiting_approval count

#### Role MCP Visibility
- planner / retriever / patchworker / integrator / validator / publisher の MCP servers は mcp_registry.py で監視する
- health source: GET /health
- capability source: GET /v1/tools
- 表示項目: endpoint status, last health check, capability status

#### Watchdog Expectations
- watchdog は role MCP services の health を定期確認する
- orchestrator 自体はプロセス内部のため workflow heartbeat / event lag で監視する
  - event lag は Event Bus (`scripts/eventbus/`) の `GET /replay?since_seq=N` が返す最新 `seq` と、orchestrator が `processed_events` に最後に記録した event に対応する `seq` との差分から算出する(Event Bus 自体は consumer ごとの lag を計算しないため、orchestrator 側で算出する)
  - **注意**: Event Bus は `config/agent.toml` の `[mcp_servers.*]` に登録された role MCP ではなく、`deploy/setup_services.sh` が直接 `python -m eventbus.app` で起動する独立 HTTP サービス(port 8015)。したがって MCP watchdog(`mcp_registry.py` / `agent/repl_health.py`)の監視対象には含まれない。Event Bus 自体の `GET /health` 監視(DB unavailable, DLQ task stopped 等の `degraded_reasons` を含む)は、role MCP watchdog とは別の監視経路として新設する必要がある — Phase 1 でこの経路をどう用意するかは未決(このセクションの Scope に追加が必要)
- persistent failure 時は restart / alert / degraded mode を定義する

### Policy Alignment Notes

This requirement must follow the Phase 1 integration policy below.

1. **Do not create `orchestrator-mcp`.**
   - The central Orchestrator is an internal Agent component, not an MCP server.
   - Role MCP services may be added and monitored via MCP health/capability checks, but the Orchestrator itself must not be registered as an MCP server.
2. **Add internal Orchestrator APIs.**
   - REPL / command layers should call internal APIs such as `start_workflow`, `resume_workflow`, `cancel_workflow`, `get_workflow_status`, `get_workflow_artifacts`, and `retry_phase`.
3. **Use `state_manager.py` as the source of truth.**
   - Workflow / task / artifact / retry / approval state must be persisted and read through `state_manager.py`.
   - REPL, `AgentContext`, WorkerRuntime, JSONL logs, and UI code must not own canonical workflow state.
4. **Extend `workflow.sqlite`; event messages stay in the existing Event Bus.**
   - Phase 1 must reuse and extend `workflow.sqlite` for workflow metadata, task state, artifact metadata references, retries, and approvals.
   - Do not introduce a separate `orchestrator.sqlite` in Phase 1.
   - Event messages themselves are NOT duplicated into `workflow.sqlite`. They are already persisted by the existing Event Bus (`eventbus.sqlite`, managed by `scripts/eventbus/`), which provides publish/subscribe, replay, ack/nack, and DLQ over HTTP (see `docs/06_eventbus_*.md`). `workflow.sqlite` only tracks which Event Bus `event_id`s a workflow has already consumed, via the `processed_events` table (idempotency — see `scripts/agent/workflow/idempotency_ops.py`).

### Expected Result

orchestrator と role MCP services の運用可視化が混線せず、問題箇所を明確に切り分けられる。

### Acceptance Criteria
- internal orchestrator visibility が定義されている
- role MCP monitoring が定義されている
- Event Bus の健全性監視経路が role MCP watchdog とは別軸として定義されている
- watchdog の責務分離が定義されている
- 表示項目が定義されている
- orchestrator を MCP service として扱わない方針が明記されている
