## [orchestrator][api] Define the minimum internal orchestrator API

### Summary

中央 Orchestrator は MCP server としては追加せず、既存 Agent プロセス内の中核コンポーネントとして実装する。Phase 1 では REPL や command 層から呼び出す最小内部 API を定義する。

### Problem

外部 API / MCP tool 前提の設計は、統合仕様の「orchestrator-mcp は追加しない」という方針に反する。代わりに、REPL から直接呼び出される内部 API 契約を固定しなければならない。

### Scope
- orchestrator internal API design
- input/output contract
- command / REPL integration points

### Task
- start_workflow を定義する
- resume_workflow を定義する
- cancel_workflow を定義する
- get_workflow_status を定義する
- get_workflow_artifacts を定義する
- retry_phase を定義する
- minimum request / response DTO を定義する

### Required Internal API
- start_workflow(session_id, request_text, mode)
- resume_workflow(workflow_id)
- cancel_workflow(workflow_id)
- get_workflow_status(workflow_id)
- get_workflow_artifacts(workflow_id)
- retry_phase(workflow_id, phase, reason)

### Minimum Request Fields
- session_id
- workflow_id (resume / status / artifact / retry / cancel 用)
- request_text (start 用)
- mode
- phase (retry 用)
- reason (retry / cancel 用)

### Minimum Response Fields
- workflow_id
- workflow_state
- current_phase
- started_at / updated_at
- summary
- artifact_refs
- next_action_hint

### Design Rules
- API は Agent プロセス内部でのみ使用する
- 永続 state の読み書きは state_manager 経由とする
- response は DTO として構造化する
- REPL / command はこの API を使って workflow を起動・確認する

### Expected Result

Phase 1 で必要なオーケストレータ操作を、外部 MCP 化せず内部 API として安定提供できる。

### Acceptance Criteria
- start_workflow が定義されている
- resume_workflow が定義されている
- cancel_workflow が定義されている
- get_workflow_status が定義されている
- get_workflow_artifacts が定義されている
- retry_phase が定義されている
- request DTO が定義されている
- response DTO が定義されている
