# [Risk] Sole-authority routing removes the safety net for discovery failure

## Priority

Medium

## Source

Identified during planning for `requires/20260720_01_require.md` ("Unify routing
authority: RuntimeToolRegistry becomes sole routing source"). Plan:
`plans/20260720-134821_plan.md`.

## Summary

Once `ToolRouteResolver.resolve()` no longer falls back to `ToolRegistry`,
`RuntimeToolRegistry` must be populated (via `ToolExecutor.set_runtime_registry()`) before
any tool call can succeed. Today, `scripts/agent/startup.py`'s
`StartupOrchestrator._check_services()` step 4 wraps
`McpToolDiscoveryService(ctx).discover_all()` in a bare `except Exception` that is
**non-fatal**:

```python
except Exception:  # noqa: BLE001
    pipeline.add_skipped(
        "mcp_tool_discovery", f"MCP tool discovery skipped: {exc}"
    )
```

If discovery throws (e.g. every MCP server is unreachable at startup, or a bug in
discovery itself), `discovery.registry` is never built, `set_runtime_registry()` is never
called, and `ctx.services_required.tools._resolver._runtime_registry` stays `None` for the
entire agent session. The REPL loop still starts (this failure is not fatal), but under
sole-authority routing **every single tool call** for that session will immediately raise
`ValueError` from `ToolRouteResolver.resolve()`.

Under the current (pre-change) two-tier behavior, this same failure mode still leaves the
agent partially functional: `ToolRegistry`'s compile-time frozensets keep resolving tool
names to server keys, so at least the tools whose backing servers happen to be reachable
can still be invoked. Sole-authority routing turns a partial/degraded-mode failure into a
full-session tool-call outage, with only a non-fatal "skipped" line in the startup log to
signal it — easy to miss in interactive use.

## Why this was not mitigated in the plan

The requirement (`requires/20260720_01_require.md`) explicitly lists
`scripts/agent/services/mcp_tool_discovery.py` under "What NOT to change" and states for
`scripts/agent/startup.py`: "No functional change needed... The difference is that now
the resolver mutation is clean." Changing the fatality of the discovery exception handler,
or adding a new post-startup health gate, would go beyond the stated scope of "unify
routing authority" and was judged out of scope for this planning cycle.

## Recommended follow-up (separate requirement)

Pick one (or combine):

1. In production security profile, make `mcp_tool_discovery` failures fatal (raise instead
   of `pipeline.add_skipped`) so the agent refuses to start rather than start silently
   crippled — mirrors the existing `SecurityProfile.PRODUCTION` fatal-on-subprocess-failure
   pattern already used elsewhere in `startup.py` (`_start_servers()`).
2. Add an explicit startup/health check (e.g. in `agent/repl_health.py`) that asserts
   `ctx.services_required.runtime_tools` is non-empty after discovery, surfaced as a
   `FATAL` or prominent `WARNING` pipeline outcome rather than a generic "skipped" line.
3. At minimum, upgrade the log level / wording of the "skipped" case so it's unambiguous
   that **all** tool calls will fail for the session, not just discovery-dependent
   diagnostics.

## Acceptance criteria for the follow-up

- A discovery-failure scenario (e.g. all MCP servers unreachable at startup) either:
  (a) fails startup fast with a clear fatal message in production profile, or
  (b) is surfaced as an unmistakable, high-visibility warning distinct from routine
  per-server "unreachable" warnings.
- Existing tests for `StartupOrchestrator._check_services()` gain a case covering total
  discovery failure and assert the chosen behavior.
