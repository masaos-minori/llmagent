## Issue 13: test_history_manager.py — `HistoryCompressionError` raised instead of returning original history

**Labels:** bug
**Affected tests:** 3 tests in `tests/test_history_manager.py`

```
FAILED tests/test_history_manager.py::TestCompressWithLLM::test_returns_original_history_on_empty_llm_response
FAILED tests/test_history_manager.py::TestCompressWithLLM::test_returns_original_history_on_llm_failure
FAILED tests/test_history_manager.py::TestCountTokensAsync::test_delegates_to_get_token_count_when_no_last_input
```

### Description

`compress_with_llm` が空または失敗の LLM レスポンスに対して元の history を返さず
`HistoryCompressionError` を raise する。また `count_tokens_async` 関連テストも1件追加で失敗。

```
agent.history.HistoryCompressionError: Context compression: LLM returned empty summary
```

### Root cause

`HistoryManager.compress_with_llm` のエラーハンドリングが graceful fallback から raise に変更された。
`TestCountTokensAsync` の失敗は `compress_with_llm` の変更に連鎖する可能性がある。

### Fix

- `compress_with_llm` 内で例外をキャッチして元 history を返す (graceful fallback を復元)、または
- テストを `pytest.raises(HistoryCompressionError)` に書き換える (raise が意図的な場合)
- `test_delegates_to_get_token_count_when_no_last_input` は個別に原因調査が必要

## Issue 14: test_embedding_client.py — Circuit breaker state leaks between tests

**Labels:** bug, test-fix
**Affected tests:** 4 tests in `tests/test_embedding_client.py`

```
FAILED tests/test_embedding_client.py::TestErrorHandling::test_generic_exception_returns_error
FAILED tests/test_embedding_client.py::TestRetry::test_retries_on_failure_then_succeeds
FAILED tests/test_embedding_client.py::TestSuccessResetsFailCount::test_success_resets_fail_count
FAILED tests/test_embedding_client.py::TestSuccessfulEmbedding::test_valid_response_returns_embedding
```

### Description

`EmbeddingClient` の circuit breaker が test 間で状態を持ち越す。
先行テストが circuit open 状態にすると後続テストが `error_kind=circuit_open` を受け取る。

```
EmbeddingResult(success=False, error_kind=circuit_open)
```

### Root cause

circuit breaker の状態がインスタンスに保持されており、テスト間でリセットされない。

### Fix

- テストごとに新しい `EmbeddingClient` インスタンスを使う、または
- `client.reset()` メソッドを追加して teardown で呼ぶ

## Issue 19: test_rag_ingester.py — chunk count doubled (6 instead of expected 3)

**Labels:** bug
**Affected tests:** 2 tests in `tests/test_rag_ingester.py`

```
FAILED tests/test_rag_ingester.py::TestChunkMetadataStorage::test_chunk_index_stored_correctly
FAILED tests/test_rag_ingester.py::TestChunkMetadataStorage::test_chunking_strategy_stored_in_documents
```

### Description

3チャンクのドキュメントに対してテストは chunk insert が3回であることを期待するが、
実際には6回になる。

```
AssertionError: assert 6 == 3
```

## Issue 23: test_lifecycle.py — stdio echo server cannot import `shared` module

**Labels:** bug
**Affected tests:** 7 tests in `tests/test_lifecycle.py`

### Description

ライフサイクル統合テストが echo MCP サーバーをサブプロセスとして起動するが、
`echo_server.py` が `ModuleNotFoundError: No module named 'shared'` で失敗する。

```
shared.tool_executor.TransportError: stdio server closed connection (key='echo')
```

### Root cause

`scripts/mcp/echo_server.py` が `from shared.json_utils import ...` をインポートするが、
テストが起動するサブプロセスは `scripts/` を `sys.path` に含まない。

### Fix

- `echo_server.py` の起動時に `sys.path.insert` でパスを追加する、または
- テスト fixture がサブプロセスを `PYTHONPATH=scripts` 付きで起動するよう修正する

## Issue NEW-1: test_installer_templates.py — unknown role raises ValueError instead of falling back

**Labels:** bug
**Affected tests:** 2 tests in `tests/test_installer_templates.py`

```
FAILED tests/test_installer_templates.py::TestGenerateConfigTomlForRole::test_unknown_role_falls_back_to_basic
FAILED tests/test_installer_templates.py::TestGenerateAgentTomlMcpSnippet::test_unknown_role
```

### Description

テストは未知のロール (`"unknown"`) を渡した場合に graceful fallback (basic ロールへのフォールバック)
を期待するが、実装は `ValueError` を raise する。

```
ValueError: Unknown role 'unknown'. Must be one of ['ci', 'git', 'shell', 'sqlite']
```

### Root cause

`_validate_role` が未知ロールを strict エラーとして拒否するようになった。
テストは以前の「不明ロールは basic にフォールバック」という動作を期待している。

### Fix

- `generate_config_toml_for_role` / `generate_agent_toml_mcp_snippet` で未知ロールを
  basic にフォールバックする動作を復元する、または
- テストを `pytest.raises(ValueError)` に書き換える (strict 検証が意図的な場合)
