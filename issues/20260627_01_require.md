## [orchestrator][state] Define the workflow-oriented state model

### Summary

現在の session-level / turn-level state だけでは、中央 Orchestrator による multi-phase workflow 実行を安全に追跡できない。Phase 1 では workflow_id を中心とした run-oriented state model を正式化する。

### Problem

新設計では、workflow / task / artifact / retry / approval を独立した正本として管理し、REPL や WorkerRuntime に workflow state を残してはいけない。状態の二重管理は resume / retry / approval 整合性を破壊する。

### Scope
- workflow-level state design
- task / artifact / retry / approval state design
- SQLite と JSONL の責務分離
- session と workflow の境界定義

### Task
- workflow-level state structure を定義する
- task / artifact / retry / approval の状態項目を定義する
- SQLite に保存する内容と JSONL に保存する内容を分離する
- session state と workflow state の責務境界を明文化する
- 禁止される状態保持箇所を明記する

### State Model Rules

#### Source of truth
- workflow / task / artifact / retry / approval の正本は state_manager.py のみ
- REPL は workflow state の正本を保持しない
- AgentContext は conversation state / 表示状態 / 最小メタ情報のみを保持する
- WorkerRuntime は stateless executor とし、永続 state の正本を保持しない

#### Workflow-level fields
- workflow_id
- session_id
- goal
- current_phase
- workflow_state
- created_at / updated_at
- requested_mode
- final_summary

#### Task-level fields
- task_id
- workflow_id
- role
- phase
- state
- dependency_ids
- target_files
- assigned_worker_id
- started_at / ended_at
- failure_class
- retry_count

#### Artifact-level fields
- artifact_id
- workflow_id
- producing_role
- artifact_type
- content_ref or inline_content
- content_hash
- size_bytes
- created_at

#### Retry-level fields
- retry_id
- workflow_id
- phase
- role
- error_class
- attempt_no
- decision
- backoff_ms
- reason

#### Approval-level fields
- approval_id
- workflow_id
- target_action
- patch_set_hash
- policy_result
- approval_state
- approver
- approved_at

#### Persistence split
- SQLite: workflow/task/artifact/retry/approval の正本、依存関係、検索可能な監査メタ情報
- JSONL: append-only event log、診断用イベント、replay 補助情報
- JSONL は正本ではない

### Prohibited Patterns
- AgentContext に workflow/task/retry/approval state を保持する
- WorkerRuntime ごとの独自 state 正本を持つ
- UI 層に retry / approval state を保持する
- JSONL のみを参照して workflow 正否を判定する

### Policy Alignment Notes

This requirement must follow the Phase 1 integration policy below.

1. **Do not create `orchestrator-mcp`.**
   - The central Orchestrator is an internal Agent component, not an MCP server.
   - Role MCP services may be added and monitored via MCP health/capability checks, but the Orchestrator itself must not be registered as an MCP server.
2. **Add internal Orchestrator APIs.**
   - REPL / command layers should call internal APIs such as `start_workflow`, `resume_workflow`, `cancel_workflow`, `get_workflow_status`, `get_workflow_artifacts`, and `retry_phase`.
3. **Use `state_manager.py` as the source of truth.**
   - Workflow / task / artifact / retry / approval state must be persisted and read through `state_manager.py`.
   - REPL, `AgentContext`, WorkerRuntime, JSONL logs, and UI code must not own canonical workflow state.
4. **Extend `workflow.sqlite`.**
   - Phase 1 must reuse and extend `workflow.sqlite` for workflow metadata, task state, artifact metadata references, retries, approvals, and events.
   - Do not introduce a separate `orchestrator.sqlite` in Phase 1.

### Expected Result

システムは 1 workflow を 1 論理単位として安全に追跡し、resume / retry / approval / audit を一貫して実行できる。

### Acceptance Criteria
- workflow-level state fields が定義されている
- task / artifact / retry / approval state が定義されている
- SQLite / JSONL の責務分離が定義されている
- session と workflow の境界が定義されている
- 禁止される状態保持箇所が明記されている
