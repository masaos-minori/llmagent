# tool_results_turn_max_chars removal request rests on an incorrect premise — needs human decision

## Background

`requires/done/20260709_05_require.md` ("Clean up legacy tool execution and
scheduling compatibility settings") asked to remove
`ToolConfig.tool_results_turn_max_chars`, citing:

> `tool_results_turn_max_chars` is deprecated and no longer enforced because
> the old `tool_results` table has been removed.

This claim was verified against the implementation while planning that
requirement (`plans/done/20260709-130926_plan.md`) and found to be
**incorrect**.

## Evidence

- `ctx.cfg.tool.tool_results_turn_max_chars` (`scripts/agent/config_dataclasses.py:219`,
  default `50000`) is actively read in
  `scripts/agent/tool_runner.py:181` and enforced by
  `_apply_turn_char_limit()` (`scripts/agent/tool_runner.py:222-235`), which
  truncates the *cumulative per-turn* tool-result text sent to the LLM,
  replacing it with a hint (`TURN_LIMIT_HINT`) once the running total
  exceeds the configured limit.
- This has no relationship to the `tool_results` SQLite persistence table
  removed in commit `e63b2628` ("refactor: remove deprecated tool_results
  table and clean up stale references") — that was a database table; this
  is an unrelated in-memory turn-processing limit. The shared name
  (`tool_results`) appears to be the source of the confusion.
- It is also distinct from `tool_result_max_llm_chars`
  (`scripts/agent/config_dataclasses.py:217`, default `8000`), which
  truncates a *single* tool result's text, not the cumulative per-turn
  total. Neither field supersedes the other; both are active,
  non-redundant safety limits.
- `docs/05_agent_08_configuration.md:260` currently states:
  `` `tool_results_turn_max_chars` | `50000` | Deprecated — `tool_results`
  table removed; no longer enforced `` — this stale documentation is the
  likely original source of the requirement's incorrect premise.

## Decision needed

Removing `tool_results_turn_max_chars` as originally requested would
eliminate a working, non-redundant safety limit (protection against
unbounded per-turn LLM context growth from large tool outputs), based on a
documentation error rather than a real deprecation. Options for a human to
choose between:

1. **Keep the field; fix only the stale doc claim** in
   `docs/05_agent_08_configuration.md:260` (remove "Deprecated — `tool_results`
   table removed; no longer enforced," replace with an accurate description
   of the per-turn limit it actually enforces).
2. **Remove the field anyway**, if there is a separate, not-yet-documented
   reason to consolidate it with `tool_result_max_llm_chars` or another
   mechanism — would need explicit rationale before implementation, since
   this is a behavior change (loosens the per-turn cap), not a no-op cleanup.
3. Something else the original requirement author intended that this
   analysis missed.

## Recommendation

Option 1 (fix the doc, keep the field) — the field is functioning as
designed and is not redundant with any other current mechanism.
