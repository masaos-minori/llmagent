## [worker-runtime][extensibility] Define extension points for future roles

### Summary

Phase 1 の設計は初期 6 role に最適化しつつも、将来の role 追加を hardcode せずに拡張できる必要がある。

### Problem

role binding を Python constant や if/else に埋め込むと、Reviewer / Researcher / Memory Manager のような Phase 2 以降の追加が困難になり、policy や tool list の再利用も壊れる。

### Scope
- role registry design
- config-driven extension points
- future role policies

### Task
- role registry design を定義する
- new role の追加手順を定義する
- role config path pattern を定義する
- policy / runtime / schema の extension point を定義する

### Role Registry Design
- role registration は mcp_registry.py と role preset config で管理する
- orchestrator は role 名と preset を見て runtime を起動する
- WorkerRuntime 自体は role-agnostic とする

### How New Roles Are Added
- 新しい runtime / preset を追加する
- 必要なら新しい MCP server を追加する
- role-specific config を TOML/JSON に追加する
- input/output schema を定義する
- allowed_tools / denied_tools / concurrency / timeout を設定する
- policy_engine に必要な role-specific restriction を追加する

### Config Path Pattern
- [<role>]
- runtime = "<runtime_module>"
- transport = "http|internal"
- endpoint = "..."
- allowed_tools = [...]
- denied_tools = [...]
- concurrency_limit = N
- timeout_sec = N

### Future Role Policies Location
- role-specific policies は config + policy_engine.py で管理する
- repository / branch / path / action constraints は共通フレームワークを再利用する
- write capability の既定値は deny とする

### Key Non-hardcoding Rules
- role 名一覧をコードに固定しない
- tool policy をハードコードしない
- input/output schema を preset/config で差し替え可能にする
- WorkerRuntime は role-specific business logic を持たない

### Policy Alignment Notes

This requirement must follow the Phase 1 integration policy below.

1. **Do not create `orchestrator-mcp`.**
   - The central Orchestrator is an internal Agent component, not an MCP server.
   - Role MCP services may be added and monitored via MCP health/capability checks, but the Orchestrator itself must not be registered as an MCP server.
2. **Add internal Orchestrator APIs.**
   - REPL / command layers should call internal APIs such as `start_workflow`, `resume_workflow`, `cancel_workflow`, `get_workflow_status`, `get_workflow_artifacts`, and `retry_phase`.
3. **Use `state_manager.py` as the source of truth.**
   - Workflow / task / artifact / retry / approval state must be persisted and read through `state_manager.py`.
   - REPL, `AgentContext`, WorkerRuntime, JSONL logs, and UI code must not own canonical workflow state.
4. **Extend `workflow.sqlite`.**
   - Phase 1 must reuse and extend `workflow.sqlite` for workflow metadata, task state, artifact metadata references, retries, approvals, and events.
   - Do not introduce a separate `orchestrator.sqlite` in Phase 1.

### Expected Result

Phase 1 設計のまま、将来 role を追加してもアーキテクチャを壊さずに拡張できる。

### Acceptance Criteria
- role registry design が定義されている
- new role 追加手順が定義されている
- role config path pattern が定義されている
- non-hardcoding rules が定義されている
