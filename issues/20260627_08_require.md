## [worktree][design] Define the worktree lifecycle rules

### Summary

PatchWorker は isolated Git worktree を用いて file 単位 patch 生成を安全に分離する。Phase 1 で naming / create / cleanup / artifact collection を固定する。

### Problem

worktree の責務と lifecycle が曖昧だと、並列 task の干渉、orphaned directory、artifact 回収漏れが発生する。

### Scope
- worktree manager design
- creation / cleanup lifecycle
- artifact collection
- same-file conflict handling

### Task
- worktree path naming を定義する
- create / cleanup タイミングを定義する
- artifact collection path を定義する
- conflict handling policy を定義する
- finalize 時 cleanup 責務を定義する

### Worktree Path Naming Rules
- Base path: worktrees/<workflow_id>/<worker_id>/
- worker_id format: <role>-<task_id>
- Example: worktrees/abc123/patchworker-file_001/

### Create and Cleanup Rules
- worktree は対象 file task 実行前に作成する
- cleanup 実行主体は worktree_manager.py
- normal cleanup は worker task 終了後に実行する
- final safety cleanup は orchestrator finalize / abort cleanup で実行する
- orphaned worktrees は active workflow state と worktrees/ の差分で検出する

### Artifact Collection Paths
- Worker metadata: runs/<workflow_id>/workers/<worker_id>/
- Work artifacts: artifacts/<workflow_id>/
- Worker logs: logs/worker_<id>.log

### Conflict Handling Rules
- same-file tasks は scheduler が直列化する
- different-file tasks は別 worktree で並列実行できる
- worktree isolation で cross-worker file conflict を回避する
- conflicting patch が残る場合は Integrator が整合チェックで検出する

### Policy Alignment Notes

This requirement must follow the Phase 1 integration policy below.

1. **Do not create `orchestrator-mcp`.**
   - The central Orchestrator is an internal Agent component, not an MCP server.
   - Role MCP services may be added and monitored via MCP health/capability checks, but the Orchestrator itself must not be registered as an MCP server.
2. **Add internal Orchestrator APIs.**
   - REPL / command layers should call internal APIs such as `start_workflow`, `resume_workflow`, `cancel_workflow`, `get_workflow_status`, `get_workflow_artifacts`, and `retry_phase`.
3. **Use `state_manager.py` as the source of truth.**
   - Workflow / task / artifact / retry / approval state must be persisted and read through `state_manager.py`.
   - REPL, `AgentContext`, WorkerRuntime, JSONL logs, and UI code must not own canonical workflow state.
4. **Extend `workflow.sqlite`.**
   - Phase 1 must reuse and extend `workflow.sqlite` for workflow metadata, task state, artifact metadata references, retries, approvals, and events.
   - Do not introduce a separate `orchestrator.sqlite` in Phase 1.

### Expected Result

worker file systems は分離され、cleanup と artifact 回収が一貫して実行される。

### Acceptance Criteria
- naming rule が定義されている
- cleanup policy が定義されている
- finalize / abort cleanup 責務が定義されている
- artifact collection policy が定義されている
- conflict handling policy が定義されている
