# エージェント層レビュー (2026-06-30)

レビュー対象: `scripts/agent/` および直接依存する `shared/` 層。
コード証拠は `scripts/agent/` 以下の実ファイルから確認したもの。ドキュメント参照先は `docs/05_agent_*.md`。

---

# 1. 全体所見

- アーキテクチャ分離度は高い。`AgentREPL` は UI ループのみを持ち、ターン処理は `Orchestrator` → `LLMTurnRunner` → `ToolRunner` へと明確に委譲されている。
- `AgentContext` による DI パターンは一貫しており、サービス間の直接依存が少ない。ただし `ctx.services` が `Any` 型であるため静的型安全性に穴がある。
- Workflow エンジンが主系 (workflow_mode="required") として稼働している実装と、ドキュメント上の「fail-closed」記述は一致しているが、`_init_workflow_task` 内に未クローズ `StateStore` リークが存在する。
- エフェメラルメッセージ (`_memory_injected`, `_ephemeral`) が DB 圧縮スナップショット (`replace_messages`) でフィルタされないため、セッションリロード後に余分なシステムメッセージが LLM コンテキストに混入するリスクがある。
- SIGTERM ハンドラ内で `time.sleep(0.1)` を使うビジースピンがある。asyncio イベントループ上で呼ばれた場合にループをブロックするリスクがある。
- `workflow_require_approval` フィールドが `AgentConfig` (config_dataclasses.py:528) に存在するが、`docs/05_agent_08_configuration.md` に記載がなく、ドキュメントと実装が乖離している。
- 診断チャンネル (`DiagnosticStore`) の二重書き込みパターン (JSONL + SQLite) がレガシー残存として存在し、`docs/05_agent_04_state-and-persistence.md` でも「将来的に非推奨」と注記されている。実際の削除パスが未定義。
- ヘルスチェック (`repl_health.py`) は外部サービス依存のチェックを確実に実行しているが、MCP サーバー個別の依存関係は HTTP ヘルスエンドポイントの存在を前提としており、`stdio` サーバーへの適用が不均等。
- テスタビリティは中程度。`CLIView` に `Writer/Reader` プロトコルがある一方、`Orchestrator` は `AgentContext` への直接依存が強く、ユニットテストに大型モックが必要。
- `notes` テーブルがスキーマに残っているが書き込み実装は削除されており、`auto_inject_notes` 設定フィールドは `ToolConfig` に残存するがコード上で実際に注入する実装が見当たらない (Needs confirmation)。

# 2. 問題点の指摘

## A-01

**Title:** `_init_workflow_task` 内の匿名 `StateStore` インスタンスが未クローズ

**Category:** Reliability/recovery weakness

**Severity:** High

**Affected area:** `scripts/agent/orchestrator.py` — `Orchestrator._init_workflow_task()` (行 275)

**Why it is a problem:**
`StateStore.__init__()` が `SQLiteHelper("workflow").open(write_mode=True)` を呼び出し、DB 接続を保持する。`_init_workflow_task()` の `existing_task_id is None` 分岐 (行 275) では `StateStore().create_task(...)` を匿名で呼び出しており、返り値を変数に束縛せず、`close()` も呼ばない。これにより workflow.sqlite への接続が GC タイミングに依存して解放されることになる。WAL モードでは接続リークがチェックポイントを阻害し、長時間セッションでファイルディスクリプタ枯渇につながる可能性がある。

**Evidence:**
- コードから確認: `orchestrator.py:275` — `StateStore().create_task(...)` — 戻り値を変数に束縛せず `close()` なし
- 対照: `existing_task_id is not None` 分岐 (行 289–295) では `store = StateStore()` → `store.close()` を明示的に呼び出している
- `_handle_workflow_engine()` 内の `store = StateStore()` (行 210) は `finally: store.close()` (行 263) で適切にクローズしている

**Recommended direction:**
`_init_workflow_task()` の `existing_task_id is None` 分岐にも `store = StateStore()` を変数に束縛し、`try/finally` で `store.close()` を呼ぶよう修正する。あるいは `StateStore` をコンテキストマネージャとして実装する。

## A-02

**Title:** SIGTERM ハンドラ内の `time.sleep()` がイベントループをブロックする

**Category:** Reliability/recovery weakness

**Severity:** High

**Affected area:** `scripts/agent/repl.py` — `AgentREPL.run()._sigterm_handler()` (行 456–461)

**Why it is a problem:**
`loop.add_signal_handler(signal.SIGTERM, _sigterm_handler)` により `_sigterm_handler` が asyncio イベントループのスレッドで同期的に呼ばれる。内部で `while self._ctx.conv.is_processing and time.monotonic() < deadline: time.sleep(0.1)` というビジースピンを行っており、イベントループを最大 10 秒間ブロックする。この間は非同期タスクが一切進行できないため、進行中のターンは完了できず、最大 10 秒後に強制終了される。graceful shutdown の意図と逆効果になる。

**Evidence:**
- コードから確認: `repl.py:454–461` — `_sigterm_handler()` 内で `time.sleep(0.1)` をループ
- `loop.add_signal_handler()` で登録したコールバックは asyncio のイベントループスレッドで同期実行される (Python asyncio 仕様)

**Recommended direction:**
`_sigterm_handler` から `time.sleep` を除去し、`ctx.conv.shutdown_requested = True` のセットと `shutdown_event.set()` のみに留める。ターン完了待機は `_repl_loop` の `asyncio.wait_for()` タイムアウト (行 341–349) で行う。

## A-03

**Title:** `_memory_injected` / `_ephemeral` マーカーが `replace_messages()` でフィルタされず DB に永続化される

**Category:** Implementation bug

**Severity:** High

**Affected area:**
- `scripts/agent/session_message_repo.py` — `SessionMessageRepository.replace_messages()` (行 118–144)
- `scripts/agent/orchestrator.py` — `_handle_memory_injection()` (行 371–388), `_classify_and_inject_mode()` (行 523–542)

**Why it is a problem:**
`_handle_memory_injection()` はメモリ注入メッセージを `{"role": "system", ..., "_memory_injected": True}` としてインメモリの `ctx.conv.history` に追加する。`_classify_and_inject_mode()` は `{"role": "system", ..., "_ephemeral": True}` を追加する。

`_handle_history_compression()` から `replace_messages()` が呼ばれるとき、これらのマーカー付きメッセージがそのまま DB に書き込まれる。`replace_messages()` はマーカーキーをフィルタしない (コードから確認: `session_message_repo.py` に `_memory_injected`/`_ephemeral` への言及なし)。

セッションリロード後、`fetch_messages()` がこれらの行を返し `ctx.conv.history` に復元される。次ターン開始時に `_sync_system_prompt()` (行 507–521) は `_ephemeral` マーカーを除去するが、`_memory_injected` マーカーを除去しない。その結果、以前のセッションで注入されたメモリ断片が LLM コンテキストに永続的に混入する。

**Evidence:**
- コードから確認: `session_message_repo.py:118–144` — `replace_messages()` で `_memory_injected`/`_ephemeral` のフィルタなし
- コードから確認: `orchestrator.py:513` — `_sync_system_prompt()` は `_ephemeral` のみ除去
- コードから確認: `orchestrator.py:386` — `_memory_injected: True` マーカー追加
- 対照: `undo_service.py:41` では `_memory_injected` を認識して除去する実装がある

**Recommended direction:**
`replace_messages()` に渡す前に、または `replace_messages()` 内部で、`_memory_injected=True` または `_ephemeral=True` を持つメッセージをフィルタアウトする。`_sync_system_prompt()` でも `_memory_injected` マーカーを除去するよう対称的な処理を追加する。

## A-05

**Title:** `restore_session()` がセッションリロード後のシステムメッセージ重複を引き起こす

**Category:** Implementation bug

**Severity:** Medium

**Affected area:**
- `scripts/agent/services/session_restore.py` — `restore_session()` (行 33–34)
- `scripts/agent/startup.py` — `_setup_prompt()` (行 222–239)

**Why it is a problem:**
`restore_session()` は `system_msgs = [m for m in ctx.conv.history if m["role"] == "system"]` でインメモリの現在の system メッセージを取り出し、`ctx.conv.history = system_msgs + messages` として復元する。一方、`messages` (DB から取得) にも `role=system` のメッセージが含まれる場合がある (`_setup_prompt()` で最初の system プロンプトを追加して保存しているため)。これにより同一セッションのシステムプロンプトが二重に含まれる。また `_memory_injected=True` を持つインメモリの system メッセージも引き継がれるため、注入済みメモリ断片も混入する。

**Evidence:**
- コードから確認: `session_restore.py:33–34` — フィルタリングなしで system_msgs を先頭に追加
- コードから確認: `session_message_repo.py:162` — `fetch_messages()` は role 無差別に返す (system も含む)
- ドキュメント: `docs/05_agent_04_state-and-persistence.md` に `/session load` の挙動が記載されているが、重複については言及なし

**Recommended direction:**
`restore_session()` で `system_msgs` として保持するのはインメモリの現在のシステムプロンプト (先頭 `history[0]`) のみに限定する。または `fetch_messages()` から返ってくる system メッセージとの重複排除ロジックを追加する。`_memory_injected`/`_ephemeral` マーカーを持つメッセージを取り込まないよう明示的にフィルタする。

## A-09

**Title:** `HistoryManager._call_compress_llm()` が HTTP タイムアウトを `context_char_limit` 評価前に設定しない

**Category:** Reliability/recovery weakness

**Severity:** Medium

**Affected area:**
- `scripts/agent/history.py` — `HistoryManager._call_compress_llm()` (行 198–228)
- `scripts/agent/factory.py` — `_build_history_manager()` (行 189–206)

**Why it is a problem:**
`HistoryManager` に渡された `httpx.AsyncClient` はメイン LLM 呼び出し用に作成されたもの (timeout=`cfg.llm.http_timeout`, デフォルト 30 秒) と同じインスタンスを共有している。圧縮 LLM 呼び出しは `_call_compress_llm()` で `await self._http.post(...)` を直接呼ぶが、timeout を指定していない。圧縮 LLM の応答が遅い場合、メイン LLM 呼び出しに使う HTTP クライアントのデフォルトタイムアウトで制限される。ただしドキュメントには圧縮の LLM パラメータ (温度、トークン数) は記載されているがタイムアウトについての記載がない。圧縮失敗時は fallback truncation で適切に処理されるため影響は限定的だが、圧縮が長時間ハングすることでメインの LLM ループが遅延するリスクがある。

**Evidence:**
- コードから確認: `history.py:207` — `await self._http.post(self._llm_url, json={...})` — timeout 指定なし
- コードから確認: `factory.py:149` — `httpx.AsyncClient(timeout=ctx.cfg.llm.http_timeout)` — 同じインスタンスを HistoryManager と LLMClient が共有

**Recommended direction:**
`_call_compress_llm()` の POST 呼び出しに明示的な timeout パラメータを追加する (例: `timeout=httpx.Timeout(30.0)`)。または圧縮専用の設定フィールド `context_compress_timeout` を追加する。

**Title:** `ToolLoopGuard.check_dedup()` が `tool_dedup_max_repeats` に達した際にループを抜けるが hint メッセージが LLM コンテキストに注入されない

## A-11
