# [worktree][design] Define the worktree lifecycle rules

## Summary
Patch workers will use isolated Git worktrees.

## Problem
We need clear rules for:
- where worktrees are created
- how they are named
- how they are cleaned up
- how artifacts are collected

## Scope
- worktree manager design
- artifact collection design

## Worktree Path Naming Rules

- Base path: `worktrees/<workflow_id>/<worker_id>/`
- worker_id format: `<role>-<task_id>` (e.g., `patchworker-file_001`)
- Example: `worktrees/abc123/patchworker-file_001/`

## Create and Cleanup Rules

- Worktree created before PatchWorker starts processing a file task
- Worktree cleaned up after all tasks for that worker complete
- Cleanup triggered by `worktree_manager.cleanup(workflow_id, worker_id)`
- Orphaned worktrees detected by comparing `worktrees/` against active workflow state

## Artifact Collection Paths

- Worker metadata: `runs/<workflow_id>/workers/<worker_id>/`
- Work artifacts: `artifacts/<workflow_id>/` (diff, logs, summaries)
- Individual worker log: `logs/worker_<id>.log`

## Conflict Handling Rules

- Same file from multiple workers is serialized by the task scheduler
- Git worktree isolation prevents cross-worker file conflicts
- If two tasks target the same file, the second task waits for the first to complete

## PatchWorker Worktree Lifecycle (from spec)

- Basic unit is `file` — each file task gets its own worktree
- Write execution is not included in PatchWorker; only patch generation
- Worktree isolation is a prerequisite for safe parallel execution
- Multiple file tasks can be executed in parallel via separate worktrees

## Expected Result
Worker file systems are isolated and manageable.

## Acceptance Criteria
- [ ] worktree naming rule is defined
- [ ] cleanup policy is defined
- [ ] artifact collection policy is defined
- [ ] conflict handling policy is defined
