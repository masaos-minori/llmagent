## Issue 2: test_agent_cmd_db.py — `_cmd_db` converted to async; tests call synchronously

**Labels:** bug, test-fix
**Affected tests:** 25 tests in `tests/test_agent_cmd_db.py`

### Description

`_cmd_db` was made `async` in `agent/commands/cmd_db.py`, but the tests call it synchronously.
Python creates a coroutine object without executing it, so all assertions on output fail silently.

```
RuntimeWarning: coroutine '_DbMixin._cmd_db' was never awaited
```

### Root cause

`CommandDef("/db", is_async=True, ...)` was set in the registry, but tests were not updated to
`await` the call or use `asyncio.run(...)`.

### Fix

Update all test cases in `tests/test_agent_cmd_db.py` to call `await cmd._cmd_db(...)` (or
wrap with `asyncio.run()`). Fixtures that use `cmd._cmd_db(...)` without `await` must be made
async.

## Issue 3: test_tool_approval.py — MagicMock not awaitable for `tools.execute`

**Labels:** bug, test-fix
**Affected tests:** 12 tests in `tests/test_tool_approval.py`

### Description

`_build_preview_with_dry_run` in `tool_approval.py` awaits `ctx.services.tools.execute(...)`, but
tests mock `ctx.services.tools` with a plain `MagicMock`, making `await` raise a `TypeError`.

```
TypeError: object MagicMock can't be used in 'await' expression
```

### Root cause

`ctx.services.tools` must be an `AsyncMock` (or its `execute` method must be an `AsyncMock`) in
any test that exercises the approval path.

### Fix

In `tests/test_tool_approval.py` fixtures, change:
```python
ctx.services.tools = MagicMock()
```
to:
```python
ctx.services.tools = AsyncMock()
```
or set `ctx.services.tools.execute = AsyncMock(return_value=(...))`.

## Issue 8: SQLite disk I/O error during full test suite run (resource exhaustion)

**Labels:** flaky, infrastructure
**Affected tests:** ~20 tests in `test_create_schema.py`, `test_workflow_state_store.py`, `test_workflow_schema.py`

### Description

Tests fail with `sqlite3.OperationalError: disk I/O error` when the full test suite runs, but
pass when executed individually. This suggests a tmpfs space or file descriptor exhaustion during
parallel database fixture creation.

```
sqlite3.OperationalError: disk I/O error
```

### Root cause

Multiple test files create SQLite databases in `/tmp` concurrently. The tmpfs may fill up or
exceed inode limits when all tests run at once. Individual runs succeed because they use a fresh
tmpfs state.

### Fix

- Add `tmp_path` fixture cleanup between test classes, or
- Increase tmpfs size in the CI/test environment, or
- Serialize database-intensive tests with `pytest-xdist` worker limits or a shared tmpdir cleaner.

## Issue 14: test_embedding_client.py — Circuit breaker state leaks between tests

**Labels:** bug, test-fix
**Affected tests:** 4 tests in `tests/test_embedding_client.py`

### Description

`EmbeddingClient` has a circuit breaker that accumulates failure state across test invocations.
When a prior test triggers the circuit open state, subsequent tests receive
`error_kind=circuit_open` even though they mock a healthy server.

```
EmbeddingResult(success=False, error_kind=circuit_open)
```

### Root cause

The circuit breaker state is held on the `EmbeddingClient` instance. If tests share an instance
or don't reset the circuit, later tests see the open state from earlier failures.

### Fix

Reset the circuit breaker state between tests in `tests/test_embedding_client.py` using a
fresh `EmbeddingClient` per test, or expose a `client.reset()` method and call it in teardown.
