# [Medium] Complete the stubbed/unwired parts of the existing `mdq-mcp` Markdown compression engine

## Labels
priority/medium, area/mcp, area/mdq, tech-debt

## Summary
A feature request ("Markdown Context Compression Engine": AST parse → heading-based
semantic index → BM25/grep/heading-traverse retrieval → LLM input, with a
`search_docs`/`get_chunk`/`outline` API and a `.mdq/index.sqlite`-style local index) was
proposed for investigation. Investigation shows this feature **already exists** as
`mdq-mcp` (port 8013, `status = "production"` in `config/mdq_mcp_server.toml`), with the
proposed API, heading-based chunking, FTS5/BM25 search, and CLI (`/mdq ...`) all
implemented and covered by 12 test files. This requirement therefore does **not** propose
a new feature; it enumerates the concrete stubs, unwired handlers, and always-empty
columns found in the existing implementation, and defines the work needed to close them.

## Background
Confirmed in code (all paths below exist and were read in full):

- `docs/04_mcp_04_04_mdq.md:18-31` documents `mdq-mcp` (port 8013), 9 tools
  (`search_docs`, `get_chunk`, `outline`, `index_paths`, `refresh_index`, `stats`,
  `grep_docs`, `fts_consistency_check`, `fts_rebuild`), DB at `/opt/llm/db/mdq.sqlite`.
- `scripts/mcp_servers/mdq/` (14 files, ~5,890 lines incl. tests) implements: Markdown
  AST-ish parsing with heading path/ordinal/parent tracking
  (`parser.py:47-198`), chunking + stable chunk-id generation + diff-based refresh
  (`indexer.py:41-342`), FTS5 BM25 + hybrid(RRF) search + path/heading/tag filters
  (`search.py:28-215`), schema with `chunks_fts` FTS5 virtual table + sync triggers +
  conditional `chunks_vec` (`db_schema.py:21-163`), regex grep with context lines
  (`db_grep.py:18-107`), fail-closed path authorization (`auth.py:12-38`), and a health
  endpoint (`health_check.py:40-117`).
- `scripts/agent/commands/cmd_mdq.py:1-338` implements the `/mdq
  status|index|refresh|search|outline|get|grep` slash commands.
- `search_docs`, `get_chunk`, `outline` are real, callable functions — not stubs — at
  `service.py:143/175/234` and dispatched via `server.py:132/137/142`.
- `.mdq/index.sqlite` (the path implied by the original proposal) does **not** exist;
  the real index lives at `/opt/llm/db/mdq.sqlite` per `config/mdq_mcp_server.toml`.
  This is a naming difference only, not a gap — no action needed.
- The following gaps were found, each confirmed by reading the implementation directly
  (see Required Changes for line-level detail):
  1. Vector/embedding search is a hard-coded stub (`search.py:151-158`) that always
     returns `[]`, already tracked as a known issue at
     `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md:78` ("MDQ-02").
  2. `tags_json` is always written as the literal empty string
     (`indexer.py:121`) — no frontmatter tag extraction exists, so the `tag_filter`
     parameter in `search.py:209-212` can never match anything.
  3. `token_count` is always written as `None` (`indexer.py:119`) despite a
     dedicated schema column (`db_schema.py:75`) and response field
     (`models.py:38,147`) existing for it.
  4. The summary cache (`service.py:152-173`) does not call an LLM; for
     `summary_model == "default"` it returns `None`, and the actual truncation
     fallback in `indexer.py:337` just slices raw content — not a semantic summary.
  5. `fts_consistency_check` and `fts_rebuild` have full tool schemas
     (`tools.py:162-173`) and implementations (`service.py:428-447`), but are **not**
     registered in `server.py:167-175`'s `_DISPATCH_TABLE`, so calling them today
     returns "Unknown tool" — already noted as a known inconsistency in
     `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md:79-84`.
- No existing requirement under `requires/done/` or `issues/` addresses these five gaps
  directly (verified via `grep -rl mdq requires/done/ issues/` — all hits are prior
  feature-build requirements, not gap-closure requirements).
- There is no repository-wide rule forcing agents to prefer `search_docs` over
  `read_file`; only a per-turn routing hint exists
  (`scripts/agent/mdq_rag_classifier.py`, documented at
  `docs/04_mcp_05_04_mdq-rag-boundary.md:96` as non-binding). This is a documented
  design choice, not a bug — no action proposed here.

## Required Changes
1. **Wire up `fts_consistency_check` / `fts_rebuild`** (smallest, highest-confidence fix):
   - `scripts/mcp_servers/mdq/server.py:167-175`: add both tool names to
     `_DISPATCH_TABLE`, mapping to the existing `service.py:428-447` implementations.
   - Add/extend a test in `tests/test_mdq_service.py` or a new
     `tests/test_mdq_fts_tools_dispatch.py` asserting both tools are callable via
     `/v1/call_tool` and return the expected shape, not "Unknown tool".
2. **Implement `tags_json` extraction**:
   - `scripts/mcp_servers/mdq/parser.py`: parse YAML frontmatter `tags:` (list or
     comma-separated string) during document parsing.
   - `scripts/mcp_servers/mdq/indexer.py:121`: replace the hard-coded `""` with the
     parsed tags serialized as JSON.
   - Confirm `search.py:209-212`'s `tag_filter` then matches real data with a new test.
3. **Implement `token_count` calculation**:
   - `scripts/mcp_servers/mdq/indexer.py:119`: replace `None` with an actual count,
     using the same tokenizer/approach as `scripts/shared/token_counter.py` (reuse,
     do not reimplement) if that module is import-layer-compatible with
     `mcp_servers` (see Unknowns — import layer contract in `AGENTS.md:56-68` allows
     `mcp_servers → db, shared`, so importing from `shared` is permitted).
4. **Decide and document the summary cache behavior** — currently every code path in
   `service.py:152-173` returns `None` regardless of `summary_model`'s value (the
   `if summary_model == "default"` branch and the fallback both return `None`). Either:
   - implement an actual LLM-backed summary call (bigger effort, needs its own
     LLM client wiring analogous to `scripts/agent/memory/embedding_client.py`), or
   - explicitly document that summary caching is presently unimplemented and remove
     the misleading early-return structure that implies model-specific behavior exists.
   Which path to take is an open decision — see Unknowns.
5. **Update `docs/04_mcp_04_04_mdq.md` and
   `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md`** to reflect whichever of the
   above items are completed, and to remove/close the "MDQ-02" and
   "fts_consistency_check/fts_rebuild unwired" known-issue entries once fixed.

## Acceptance Criteria
- Calling `fts_consistency_check` and `fts_rebuild` via `/v1/call_tool` (or `/mdq`
  equivalents, if added to `cmd_mdq.py`) succeeds and returns their documented
  response shape, not "Unknown tool".
- A Markdown file with a YAML frontmatter `tags:` field, once indexed, is retrievable
  via `search_docs(..., tag_filter=<that tag>)`.
- A newly indexed chunk has a non-null `token_count` in both the DB row and the
  `get_chunk`/`search_docs` response payload.
- The summary cache behavior (implemented or explicitly documented as unimplemented)
  matches whatever decision is made in Required Changes item 4 — no code path silently
  implies functionality that does not exist.
- `docs/04_mcp_04_04_mdq.md` and `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md` no
  longer list as open/unimplemented any item that was closed by this requirement.

## Non-Goals
- Do not implement MDQ-02 (embedding/vector hybrid search) as part of this
  requirement — it is a materially larger effort (LLM client integration, embedding
  storage/retrieval logic) than the other four items and should be scoped as its own
  requirement if prioritized.
- Do not add a repository-wide enforced rule requiring agents to call `search_docs`
  before `read_file` — the current per-turn hint (`mdq_rag_classifier.py`) is an
  existing, separate design choice not raised as a gap by this investigation.
- Do not rename or relocate the index DB to `.mdq/index.sqlite` — the current
  `/opt/llm/db/mdq.sqlite` path is intentional production configuration, not a defect.
- Do not add support for Setext headings, HTML blocks, GFM tables, inline HTML, or MDX
  parsing — documented as known non-goals of the parser
  (`scripts/mcp_servers/mdq/parser.py:18-30`, `docs/04_mcp_04_04_mdq.md:60-64`).
