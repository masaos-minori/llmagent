# Orphaned duplicate-module files from rename commit ca6b7bfe cause live/dead module-identity mismatches

**RESOLVED (2026-07-19), with a new critical follow-on finding.** Commit `abcf0820` ("chore: clean up
stale MCP server file references after rename") deleted the orphaned `models.py`/`server.py`/`tools.py`
in `web_search` (and the equivalent trio in all 7 other affected subpackages) — confirmed via
`ls scripts/mcp_servers/web_search/*.py`, only `web_search_models.py`/`web_search_server.py`/
`web_search_tools.py` remain. R1 (the exception-handler class-identity mismatch described below) no
longer applies as described, because the file it was rooted in (`server.py`) no longer exists. R4 (the
"other 7 servers" follow-up) is also resolved — the same cleanup was applied to all 8 subpackages, not
just `web_search`.

**However, this cleanup did not update `config/agent.toml`**, whose `cmd` entry for `web_search` (and
the other 7 servers) still launches the now-deleted bare `server.py`. This is a new, more severe,
currently-live issue — see `issues/20260720-070458_mcp_server_agent_toml_cmd_points_to_deleted_files.md`
for full details and required action. `plans/20260719-193357_plan.md` and
`plans/20260719-195330_plan.md` have been updated to reflect both findings.

## Context (original, 2026-07-19 — describes the now-resolved orphaned-duplicate state)

While planning `requires/20260717_22_require.md` (web_search `search_web` error-classification and
timeout work — see `plans/20260719-193357_plan.md`), inspection of
`scripts/mcp_servers/web_search/` turned up a concrete bug pattern traced to commit `ca6b7bfe`
("refactor: rename MCP server files to eliminate duplicate filenames"). That commit's stated intent
was `models.py -> <server>_models.py`, `server.py -> <server>_server.py`,
`service.py -> <server>_service.py`, `tools.py -> <server>_tools.py` across 8 servers (`browser`,
`cicd`, `git`, `github`, `mdq`, `rag_pipeline`, `shell`, `web_search`) — but it added the new
`<server>_*.py` files without removing the old ones. Both copies remain tracked in git (`git status`
is clean; `git ls-files` lists both `models.py` and `web_search_models.py` side by side).

## Verified in web_search (in scope of the plan above, fixed there)

- `scripts/mcp_servers/web_search/models.py` and `web_search_models.py` are byte-identical
  (`diff` produces no output).
- `scripts/mcp_servers/web_search/tools.py` and `web_search_tools.py` are byte-identical.
- `scripts/mcp_servers/web_search/server.py` and `web_search_server.py` differ only in 3 import-path
  lines and the `app_module` string.
- `config/agent.toml:314` wires the **actually-deployed** command to the **old**-named
  `scripts/mcp_servers/web_search/server.py` (`cmd = [..., "/opt/llm/scripts/mcp_servers/web_search/server.py"]`).
- `server.py` imports `WebSearchUpstreamError`/`WebSearchConfig` from the **old** `models.py`, while
  `search_provider.py` and `formatters.py` (both single-copy, non-duplicated files) import the same
  *named* classes from the **new** `web_search_models.py`. Because these are two distinct Python
  modules with two distinct class objects, `server.py`'s `@app.exception_handler(WebSearchUpstreamError)`
  does not match exceptions raised via `web_search_models.WebSearchUpstreamError` (no shared class
  identity, `issubclass()` is false across the two definitions) — a live latent bug, independent of
  any new feature work.
- Side effect: `WebSearchConfig.load()` fail-fast-loads `config/web_search_mcp_server.toml` twice at
  import time (once per module identity) — harmless today (idempotent load), but wasted work and a
  trap for any future stateful config object.

## Not yet verified (scope of this issue, not the plan above)

- Whether the same live-file-imports-orphaned-module pattern exists in the other 7 renamed packages
  (`browser`, `cicd`, `git`, `github`, `mdq`, `rag_pipeline`, `shell`). Spot checks during this
  planning cycle confirmed `browser/models.py` / `browser_models.py` and `shell/models.py` /
  `shell_models.py` both still exist as duplicate pairs, but which file each server's
  `config/agent.toml` `cmd` entry actually launches was not checked per-server.
- Whether any of those servers register an exception handler keyed on a domain exception class,
  making them susceptible to the same class-identity mismatch as `web_search`'s R1
  (`mdq`'s `MdqServiceError` hierarchy is the most likely candidate, given `mdq/models.py` and
  `mdq_models.py` are also both present).

## Suggested resolution

1. For each of the 8 renamed packages, determine which file (old- or new-named) is the one actually
   referenced by `config/agent.toml`'s `[mcp_servers.<name>]` `cmd` entry.
2. Delete the orphaned, unreferenced duplicate (`git rm`) rather than leaving both — this was
   evidently the original intent of commit `ca6b7bfe`'s commit message ("Renamed files (36 total)"),
   which did not actually complete as a rename (no old files were removed).
3. Before deleting, grep the whole repo (`scripts/`, `tests/`, `config/`) for any remaining import of
   the old module path to confirm it is truly dead, the same check this planning cycle used for
   `web_search`.
4. Re-run each server's test suite and `check-mcp-docs` after cleanup.

## Severity

Medium — confirmed exploitable in `web_search` (exception-handler mismatch, silent-ish 500 instead of
502 for classified errors once those are introduced by the plan above); severity in the other 7
servers is unconfirmed but plausible given the identical duplicate-file evidence. No user-facing
outage reported to date — this is a latent-bug risk, not an active incident.
