## Issue 5: test_config_loader.py — `ConfigLoader` cannot load test files by relative path

**Labels:** bug, test-fix
**Affected tests:** 10 tests in `tests/test_config_loader.py`

### Description

Tests call `ConfigLoader` with relative filenames like `"test.toml"`, but `ConfigLoader` resolves
paths from the project `config/` directory, not from the current working directory.

```
ValueError: Config file not found: test.toml
```

### Root cause

`ConfigLoader.load_all()` or its internal path resolution is tied to the project config directory.
Tests assume it will resolve from CWD or an absolute path provided in the test.

### Fix

Either:
- Update `ConfigLoader` to accept an optional `base_dir` parameter for tests, or
- Update tests to use `tmp_path` (pytest fixture) and pass absolute paths.

## Issue 8: SQLite disk I/O error during full test suite run (resource exhaustion)

**Labels:** flaky, infrastructure
**Affected tests:** ~20 tests in `test_create_schema.py`, `test_workflow_state_store.py`, `test_workflow_schema.py`

**再調査結果 (2026-06-19 15:00): 現在は再現せず → クローズ候補**

### 再調査手順

```
uv run pytest tests/test_create_schema.py tests/test_workflow_state_store.py tests/test_workflow_schema.py -v
# → 41 passed (単独実行)

uv run pytest tests/test_create_schema.py tests/test_workflow_state_store.py tests/test_workflow_schema.py -n 4
# → 41 passed (xdist 4ワーカー並列)

uv run pytest tests/ --ignore=tests/test_lifecycle.py -p no:randomly
# → 3ファイルはすべて PASSED。51 failed は他 issue 由来
```

### 再現しない理由 (推定)

元の 162 件 (+ 70 errors) の失敗が引き起こした fixture setup/teardown の嵐が、
WAL ファイル操作を WSL2 仮想 FS 上で衝突させていた。
16 件の fix で失敗が ~51 件に減り、同時競合が解消された。

`disk I/O error` の直接原因は「tmpfs 枯渇」ではなく、
**多数の失敗テストが WAL ファイルを開いたままにした状態での並列書き込み競合**。

### 残存する潜在リスク (今は壊れていないが要修正)

**`test_create_schema.py` の fixture がコネクションを閉じない**

```python
# 現状 (NG): return で返すため __del__ 頼みになる
@pytest.fixture
def rag_tmp_db(tmp_path: Path) -> sqlite3.Connection:
    ...
    return sqlite3.connect(str(db_file))   # 明示的 close なし
```

正しくは `yield` + teardown にするべき:

```python
@pytest.fixture
def rag_tmp_db(tmp_path: Path) -> Generator[sqlite3.Connection, None, None]:
    ...
    conn = sqlite3.connect(str(db_file))
    yield conn
    conn.close()
```

CPython の参照カウントで通常は GC されるが、フルスイートで多数積み重なると
fd 枯渇 (`Too many open files`) を引き起こす可能性がある。

### 対処方針

1. **Issue 8 はクローズ** — 現在のコードベースでは再現しない
2. **`test_create_schema.py` の `rag_tmp_db` / `session_tmp_db` fixture を `yield` 方式に修正** (別 issue または PR で対処)

## Issue 13: test_history_manager.py — `HistoryCompressionError` raised instead of returning original history

**Labels:** bug
**Affected tests:** 2 tests in `tests/test_history_manager.py`

### Description

Tests expect that an empty or failed LLM response causes `compress_with_llm` to return the
original history unchanged. Instead, `HistoryCompressionError` is raised.

```
agent.history.HistoryCompressionError: Context compression: LLM returned empty summary
```

### Root cause

Error handling in `HistoryManager.compress_with_llm` was changed to raise instead of gracefully
returning the original history on LLM failure.

### Fix

Either:
- Restore graceful fallback: catch the error inside `compress_with_llm` and return the original
  history, or
- Update tests to `pytest.raises(HistoryCompressionError)` if raising is the intended behavior.

## Issue 14: test_embedding_client.py — Circuit breaker state leaks between tests

**Labels:** bug, test-fix
**Affected tests:** 4 tests in `tests/test_embedding_client.py`

### Description

`EmbeddingClient` has a circuit breaker that accumulates failure state across test invocations.
When a prior test triggers the circuit open state, subsequent tests receive
`error_kind=circuit_open` even though they mock a healthy server.

```
EmbeddingResult(success=False, error_kind=circuit_open)
```

### Root cause

The circuit breaker state is held on the `EmbeddingClient` instance. If tests share an instance
or don't reset the circuit, later tests see the open state from earlier failures.

### Fix

Reset the circuit breaker state between tests in `tests/test_embedding_client.py` using a
fresh `EmbeddingClient` per test, or expose a `client.reset()` method and call it in teardown.

## Issue 23: test_lifecycle.py — stdio echo server cannot import `shared` module

**Labels:** bug
**Affected tests:** 7 tests in `tests/test_lifecycle.py`

### Description

Lifecycle integration tests start the echo MCP server as a subprocess, but `echo_server.py`
fails with `ModuleNotFoundError: No module named 'shared'`.

```
shared.tool_executor.TransportError: stdio server closed connection (key='echo')
```
(Underlying cause: `ModuleNotFoundError: No module named 'shared'` in echo_server.py)

### Root cause

`scripts/mcp/echo_server.py` imports `from shared.json_utils import ...`, but the subprocess
launched by the test does not have `scripts/` in its `sys.path`.

### Fix

Fix `echo_server.py` to use an absolute path bootstrap (`sys.path.insert`) at startup, or
update the test fixture to launch the server with `PYTHONPATH=scripts` in the subprocess
environment.
