# [worker-runtime][design] Design a reusable WorkerRuntime

## Summary
The current agent is built for one interactive REPL.
The new system needs reusable worker runtimes.

## Problem
We need one shared runtime that can run different roles:
- Planner
- PatchWorker
- Validator
- Integrator
- Publisher

## Scope
- worker runtime design
- role-specific runtime settings

## Task
- define the WorkerRuntime concept
- define which parts are shared across roles
- define which parts are different per role
- separate UI-specific code from runtime logic

## Expected Result
The system can launch role-specific workers without copying the full agent implementation.

## WorkerRuntime Concept

WorkerRuntime is a reusable runtime that launches role-specific workers without copying the full agent implementation.

### Shared parts across all roles
- MCP HTTP transport (`http://127.0.0.1:8009`)
- Health check via `GET /health` and capability discovery via `GET /v1/tools`
- Tool execution via `POST /v1/call_tool`
- State persistence to `workflow.sqlite`
- Event logging to JSONL
- Retry controller integration
- Barrier synchronization
- Audit logging

### Role-specific parts
- Prompt configuration
- Allowed/denied tool lists
- LLM endpoint configuration
- Input/output schema
- Worktree isolation (PatchWorker only)
- Concurrency limits per role

## Publisher Runtime Settings
- MCP server: `publisher_mcp_server.py`
- Transport: HTTP (`http://127.0.0.1:8009`)
- Input: validation-verified `patch_set`, repository context, approval/policy state
- Output: branch name, commit plan, push result, Draft PR payload, final summary
- Isolated from Integrator at approval boundary; Publishers only handles post-approval publishing
- Write API access via `repository_gateway.py` exclusively (no direct github-mcp/file-mcp)

## Acceptance Criteria
- [ ] shared worker runtime design is defined
- [ ] role-specific settings are defined
- [ ] UI-dependent parts are separated from runtime logic
