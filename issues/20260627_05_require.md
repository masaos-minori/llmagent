# [orchestrator][phase] Define the phase state machine

## Summary
The multi-agent system needs a phase-based execution model.

## Problem
The system must control these phases:
- plan
- retrieve
- implement
- integrate
- validate
- publish
- finalize

## Scope
- orchestration phase logic
- phase transition rules

## Task
- define all phase states
- define allowed transitions
- define start and end conditions for each phase

## Phase States (from spec)

### Workflow states
- `pending` — workflow created but not yet started
- `running` — workflow in progress
- `waiting_approval` — waiting for approval before proceeding
- `completed` — all phases completed successfully
- `failed` — unrecoverable error occurred
- `cancelled` — workflow was cancelled

### Task states
- `queued` — task registered but not yet ready to run
- `ready` — dependencies satisfied, ready to execute
- `running` — currently executing
- `succeeded` — completed successfully
- `failed` — execution failed
- `blocked` — blocked by dependency state
- `skipped` — skipped due to upstream failure

### Recommended phases (from spec section 3.3.2)
1. **plan** — Planner generates task graph and change_sets
2. **retrieve** — Retriever collects evidence and context
3. **implement** — PatchWorker generates file patches in parallel
4. **integrate** — Integrator merges all patches into patch_set
5. **validate** — Validator runs syntax/lint/unit tests on patch_set
6. **publish** — Publisher creates branch, commit, push, Draft PR
7. **finalize** — Orchestrator finalizes workflow state and cleanup

## Expected Result
The orchestrator can move a run through clear phases.

## Acceptance Criteria
- [ ] all Phase 1 states are defined
- [ ] transitions are defined
- [ ] completion conditions are defined
- [ ] invalid transitions are defined
