## [orchestrator][phase] Define the phase state machine

### Summary

multi-role workflow を安全に進行させるため、workflow state / task state / phase transition の正式な state machine を定義する。

### Problem

phase と terminal state の定義が曖昧だと、retry / approval wait / publish gate / finalize 処理に矛盾が生じる。

### Scope
- workflow phase logic
- workflow state transitions
- task state transitions
- invalid transition rules

### Task
- workflow states を定義する
- task states を定義する
- phase 一覧を定義する
- 各 phase の開始条件 / 終了条件を定義する
- invalid transitions を定義する

### Workflow States
- pending
- running
- waiting_approval
- completed
- failed
- cancelled

### Task States
- queued
- ready
- running
- succeeded
- failed
- blocked
- skipped

### Phase Order
- plan
- retrieve
- implement
- integrate
- validate
- publish
- finalize

### Per-phase Entry / Exit Conditions
- **plan**: request を goal/change_sets/tasks/dependencies/acceptance_criteria に展開できたら完了
- **retrieve**: 必要 evidence/context package が揃ったら完了
- **implement**: runnable file tasks がすべて終了したら完了
- **integrate**: patch_set と validation artifact が生成されたら完了
- **validate**: syntax/lint/unit test 結果と retry_hint が確定したら完了
- **publish**: policy pass + approval hash match + publish action 完了で完了
- **finalize**: final summary / artifact index / audit closeout が完了したら終了

### Invalid Transition Rules
- plan 未完了で retrieve へ遷移しない
- retrieve 未完了で implement へ遷移しない
- implement 未完了で integrate へ遷移しない
- validate 未完了で publish へ遷移しない
- approval 未完了で waiting_approval から publish 実行に遷移しない
- completed / failed / cancelled から running に戻さない（resume/retry は明示 API のみ）

### Policy Alignment Notes

This requirement must follow the Phase 1 integration policy below.

1. **Do not create `orchestrator-mcp`.**
   - The central Orchestrator is an internal Agent component, not an MCP server.
   - Role MCP services may be added and monitored via MCP health/capability checks, but the Orchestrator itself must not be registered as an MCP server.
2. **Add internal Orchestrator APIs.**
   - REPL / command layers should call internal APIs such as `start_workflow`, `resume_workflow`, `cancel_workflow`, `get_workflow_status`, `get_workflow_artifacts`, and `retry_phase`.
3. **Use `state_manager.py` as the source of truth.**
   - Workflow / task / artifact / retry / approval state must be persisted and read through `state_manager.py`.
   - REPL, `AgentContext`, WorkerRuntime, JSONL logs, and UI code must not own canonical workflow state.
4. **Extend `workflow.sqlite`.**
   - Phase 1 must reuse and extend `workflow.sqlite` for workflow metadata, task state, artifact metadata references, retries, approvals, and events.
   - Do not introduce a separate `orchestrator.sqlite` in Phase 1.

### Expected Result

workflow は明確な phase と state を持ち、phase 境界を破らずに進行できる。

### Acceptance Criteria
- workflow states が定義されている
- task states が定義されている
- phase order が定義されている
- phase ごとの開始条件 / 終了条件が定義されている
- invalid transitions が定義されている
