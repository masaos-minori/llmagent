# テスト実行によるプロセス終了問題

## 概要

`tests/test_lifecycle.py` のインテグレーションテスト実行が、LLM エージェントプロセスごと終了させる問題を調査。
同時に、同じパターンを持つ他のテストファイルの有無を調査し、対応要否を判断した。

---

## 1. `tests/test_lifecycle.py` — 原因分析

### Status: 未修正

### 要因

#### 1-a. インテグレーションテストが実サブプロセスを起動

`TestHttpLifecycleStderrLog` (L476-553) の 3 テストは `@pytest.mark.integration` だが、
`subprocess.Popen` / `os.killpg` / `httpx.AsyncClient` をモックしておらず、**実プロセスを生成する。**

特に `test_start_timeout_stderr_from_log` (L515-532) は `time.sleep(60)` でブロックする子プロセスを生成し、
1秒後のヘルスチェックタイムアウト → `HttpServerLifecycleManager.start()` → `_terminate_with_timeout()` で
`os.killpg(pgid, SIGTERM)` を実呼び出しする。

#### 1-b. `_terminate_with_timeout` のバックグラウンドスレッドリーク

`scripts/agent/http_lifecycle.py:102`:

```python
await asyncio.wait_for(asyncio.to_thread(proc.wait), timeout=timeout)
```

`asyncio.to_thread(proc.wait)` は `ThreadPoolExecutor` のスレッドで `proc.wait()` を実行する。
`os.killpg` / `proc.terminate` が失敗すると、このスレッドは子プロセスの終了を無限に待ち続ける。

スレッドは非デーモンであるため、Python 終了時の `ThreadPoolExecutor` atexit フックがブロックし、
**テストプロセス全体がハングする。** この状態でシェルタイムアウトなどが force-kill を発行すると、
同一プロセスグループ内の LLM エージェントも影響を受ける。

#### 1-c. PID 再利用の競合条件

`start()` (L217-219):

```python
try:
    self._http_pgids[server_key] = os.getpgid(proc.pid)
except OSError:
    self._http_pgids[server_key] = proc.pid   # PID → PGID として保存
```

`os.getpgid()` が `OSError` (プロセスが既に終了) を上げると `proc.pid` を PGID として記録する。
この PID が他のプロセスに再利用された後に `_terminate_with_timeout` で `os.killpg(reused_pid, SIGTERM)` を呼ぶと、
**無関係なプロセスグループにシグナルが送信され、テストランナーや LLM エージェントが終了する可能性がある。**

#### 1-d. モック漏れ

`_patch_open_to_tmp` で `_open_stderr_log` のみパッチし、以下はモックされていない:

- `subprocess.Popen` → 実プロセス起動
- `os.killpg` / `os.getpgid` → 実際のシグナル送信
- `httpx.AsyncClient` → `http://127.0.0.1:9999/health` への実接続試行

`test_start_populates_http_pgids` (L745) の docstring でも「モックしないとスレッドリークでハングする」と注意喚起されている。

### 対応案

- インテグレーションテストに `subprocess.Popen` / `os.killpg` / `os.getpgid` / `httpx.AsyncClient` のモックを追加
- `_terminate_with_timeout` の SIGTERM 後タイムアウトを 3s → 1s に短縮し SIGKILL へのフォールバックを確実化
- 実サブプロセスを要するテストは `tests/integration/` へ隔離

## 2. `tests/test_mcp_tools_validation.py` — 同種リスクあり

### Status: 未修正

### リスク内容

#### 2-a. `_terminate_process_group` が `os.killpg` を実呼び出し

`_terminate_process_group` (L91-138) は `http_lifecycle._terminate_with_timeout` の独立複製。
`os.getpgid` / `os.killpg` のモックなしに実シグナルを送信する。

#### 2-b. `mcp_server` fixture が実サブプロセスを起動

`mcp_server` fixture (L141-175):

```python
proc = subprocess.Popen(cmd, ..., start_new_session=True)   # L157-164
...
_terminate_process_group(proc, timeout=5)                    # L175 — teardown
```

teardown で `_terminate_process_group` を毎回実行する。`os.killpg` にモックはない。

### 2-a と `test_lifecycle` の差異

| 項目 | `test_lifecycle` | `test_mcp_tools_validation` |
|---|---|---|
| スレッドリーク | `asyncio.to_thread(proc.wait)` — あり | `proc.wait(timeout=timeout)` — 同期待ち、**なし** |
| リスク度 | **HIGH** | **MEDIUM** (スレッドリークなし) |
| PID 再利用 race | あり | あり (同程度) |

### 対応案

- `_process_group_gone` (L80-88) の `os.killpg(pgid, 0)` プローブを `proc.poll()` ベースに変更し、
  プロセスグループプローブの副作用を排除
- `_terminate_process_group` 内で `os.killpg` 呼び出しを `try/except` でガード (既に実装済みだが、
  `_process_group_gone` アサーションが `os.killpg` を呼ぶ点が未対処)

## 3. 問題なしと判断したファイル

| ファイル | 理由 |
|---|---|
| `test_shell_mcp_service.py` | `os.killpg` / `asyncio.create_subprocess_exec` ともに `patch()` で完全モック |
| `test_agent_factory.py` | `_http_mgr` 全体が `AsyncMock()`、実プロセス操作なし |
| `test_startup.py` | lifecycle メソッドはすべて `AsyncMock()` |

---

## 4. 推奨アクション

### Priority: High

1. **`test_lifecycle.py`**: `TestHttpLifecycleStderrLog` の 3 テストに `subprocess.Popen` / `os.killpg` /
   `os.getpgid` / `httpx.AsyncClient` のモックを追加する
2. **`test_lifecycle.py`**: `test_start_timeout_stderr_from_log` の `time.sleep(60)` をモックプロセスに置き換え、
   実プロセス生成をやめる

### Priority: Medium

3. **`test_mcp_tools_validation.py`**: `_process_group_gone` の `os.killpg(pgid, 0)` を `proc.poll()` に変更する
4. **`test_mcp_tools_validation.py`**: `mcp_server` fixture の teardown で `os.killpg` が
   実シグナルを送信しないようガードを強化する

### Priority: Low

5. **`scripts/agent/http_lifecycle.py`**: `_terminate_with_timeout` の SIGTERM 後タイムアウトを
   3s → 1s に短縮し、SIGKILL フォールバックまでの待機時間を削減する
