# テスト実行・エージェント終了時のプロセス残留問題

## 概要

`tests/test_lifecycle.py` のインテグレーションテスト実行が LLM エージェントプロセスごと終了させる問題として調査を開始。
同種リスクを持つ他のテストファイルの洗い出しと修正を経て(C節、解決済み)、修正後もなお「エージェント終了時にサブプロセスで起動した
MCP サーバーが終了しない」という事象が報告されたため、コードレベルで再調査した(A節、**現在オープン**)。
また調査の過程で、回帰テスト実行時に Claude Code CLI 自身のプロセスが終了する別の事象が発生しており、これも記録する
(B節、**原因特定済み・未修正** — `tests/test_lifecycle.py` 内のテストが `os.killpg(1, ...)` を実呼び出しし、POSIX の
特殊なブロードキャスト形式として全プロセスにシグナルが配送されることが確定した)。

本ファイルは、並行して作成されていた `issues/20260709_01_issue.md`(別セッションによる独立した再監査)の内容を統合したものである。
統合にあたり、`01` 側の記述と実コードを突き合わせて検証し、既に修正済みの内容が誤って「未修正」と記載されていた箇所を訂正した
(詳細は「統合に関する注記」参照)。テストケース一覧(D節)は `01` 側の網羅表を検証・訂正の上で取り込んだ。

## A. オープンな問題: シャットダウン処理自体への二重割り込み

### Status: 未修正(対策方針は決定済み — 下記「対策」参照)

### 症状

エージェント終了時、サブプロセスで起動した MCP サーバーが終了せず残り続ける。
メモリ不足・ネットワーク不調などの外部要因は、ネットワークが安定していた過去や別 PC の Linux 環境でも同事象が再現することから除外し、
コードレベルの原因を特定した。

### 原因

`HttpServerLifecycleManager.shutdown_all()` (`scripts/agent/http_lifecycle.py:290-315`) は、設定された複数の MCP サーバーを
1 台ずつ順番に `await self._terminate_with_timeout(...)` で終了させる(1 台あたり最大 10 秒: SIGTERM 待ち 5s + SIGKILL 待ち 5s)。
`AgentREPL.run()` の `finally` 節(`scripts/agent/repl.py:452-456`)からこれが呼ばれる際:

- 1 回目の Ctrl-C (SIGINT) は正しくこの `finally` 節に到達し、`shutdown_all()` を開始させる
  (Python 3.13 の `asyncio.run()` では単発の SIGINT は task の finally 節まで正しく伝播することを確認済み)。
- しかし `shutdown_all()` の逐次ループが複数秒〜数十秒かかりうるため、待たされたユーザーが焦って**2 回目の Ctrl-C を押す**、
  あるいは外部監視プロセスが 2 回目のシグナルを送るといった状況が起きやすい。
- **2 回目の割り込みがこのループの実行中(`await self._terminate_with_timeout(...)` の最中)に届くと、ループがその場で中断される。**
  まだ処理していない残りの MCP サーバーには終了シグナルが一切送られない。

安全な最小再現スクリプトで実証済み(実際の MCP サーバー・リポジトリコードには非接触):
1 回目の SIGINT → クリーンアップ開始・正常進行 / 2 回目の SIGINT(0.8 秒後、クリーンアップ実行中)→ `KeyboardInterrupt` が
`asyncio.run()` まで伝播し、クリーンアップが 10 ステップ中 2 ステップで中断。

これは Python/asyncio のシグナル配送・タスクキャンセルという言語仕様レベルの挙動であり、
**メモリ量・ネットワーク状態・マシン環境に非依存で決定的に再現する**。

### 寄与要因

MCP サブプロセスは `start_new_session=True` (H-8 設計) でエージェント自身のプロセスグループ/セッションから意図的に切り離されている。
そのため、このクリーンアップコードが最後まで実行される以外に、OS レベルの安全網(`PR_SET_PDEATHSIG` 等による親プロセス死亡時の
自動終了)が一切存在しない。一度この経路が中断されれば、当該 MCP サーバーは完全に孤立し、永続的に起動し続ける。

### 既存テストのカバレッジ

`tests/test_lifecycle.py`・`tests/test_agent_factory.py`・`tests/test_startup.py` はいずれも `shutdown_all()`/`_terminate_with_timeout`
を個別にモック化した単体テストのみで、実シグナル割り込み(まして 2 回連続の割り込み)を伴うエンドツーエンドのシャットダウン
シナリオを検証するテストは存在しない(テストケース一覧は D 節参照)。

テストスイート全体を対象に、他に実プロセスを kill するテストがないか全数監査した。結果、`tests/integration/conftest.py` の
`stdio_echo_server` フィクスチャ(実 `asyncio.create_subprocess_exec` + `proc.terminate()`、`killpg` なし・`start_new_session` なし)
が唯一の該当箇所だったが、**現状どのテストからも呼び出されていない未使用フィクスチャ**であり対応不要と判断。それ以外の
`os.killpg`/`subprocess.Popen` はすべてモック済みか実プロセス操作なしを確認済み。

### 対策(選定済み・未実装)

**採用: 対策1** — `shutdown_all()` のクリーンアップ処理実行中は、追加のシグナル割り込みを一時的に無視/吸収するようガードする
(多くの CLI ツールが採る「終了処理中に再度 Ctrl-C を押しても処理を中断させない」パターン)。

不採用/保留とした対策案:
- `shutdown_all()` 内の複数サーバー終了処理を `asyncio.gather()` で並列化し、ユーザーが痺れを切らす待ち時間の窓自体を短縮する
- Linux 固有の `prctl(PR_SET_PDEATHSIG)` を MCP サブプロセス起動時に設定し、OS レベルの安全網を追加する
- クリーンアップ処理中の割り込みを再現する統合テストの追加

## B. 確定した問題: `os.killpg(1, sig)` が POSIX の特殊なブロードキャスト形式として解釈され、
   Claude Code CLI 自身を含む同一ユーザーの全プロセスにシグナルが配送される

### Status: 修正済み (2026-07-11)

### 症状

本リポジトリの改修作業中、Claude Code CLI 上から `pytest` による回帰テストを実行した際に、テスト対象のプロセスではなく
**Claude Code CLI 自身のプロセスが終了し、セッションの再起動が必要になる** 事象が本調査中に複数回発生した。フルテスト
スイートを1ファイルずつ実行しながら開始・終了をログファイルに記録する方式で追跡した結果、
**`tests/test_lifecycle.py::TestStartHttpSubprocess::test_zero_timeout_skips_health_polls`** の実行中にクラッシュが
発生することを **2 回連続で完全に同一箇所にて再現**し、原因を確定した。

### 根本原因

1. `test_zero_timeout_skips_health_polls` は `mock_proc = _make_mock_proc()` で `proc.pid` を明示的に設定していない
   (未設定の `MagicMock`)。かつ `os.getpgid`/`os.killpg` もモックしていない。
2. `HttpServerLifecycleManager.start()` は `os.getpgid(proc.pid)` を実呼び出しする。`MagicMock().pid` は `__index__()`
   経由で **`1`** に変換される(`int(MagicMock()) == 1` は `unittest.mock` のデフォルト仕様。安全な分離実験で確認済み)。
3. `os.getpgid(1)` は実在する PID 1(`systemd`)の pgid `1` を正しく返す → `self._http_pgids["s"] = 1` として保存される。
4. `cfg.startup_timeout_sec=0` のため、ヘルスチェックループが即座にスキップされ `_terminate_with_timeout()` に到達する。
5. `os.killpg(1, signal.SIGTERM)` が実呼び出しされる。**ここが本質的な問題**: `os.killpg(pgrp, sig)` は内部的に
   `os.kill(-pgrp, sig)` として実装されている。`pgrp=1` の場合 `-pgrp = -1` となり、これは POSIX で
   「プロセスグループ 1 へのシグナル」ではなく **`kill(-1, sig)` = 呼び出し元ユーザーがシグナル可能な全プロセスへの
   一斉送信(ブロードキャスト)** という完全に別の特殊な意味を持つ。
6. 実際に検証(シグナル 0 = 権限確認のみで安全、実シグナルは送信されない)したところ、`os.killpg(1, 0)` は権限エラー
   なく成功した。単一 PID 宛の `os.kill(1, 0)` は正しく `PermissionError` になることと対比すると、`killpg(1, ...)` が
   `kill(-1, ...)` のブロードキャスト経路を通っていることの直接証拠になる。
7. `mock_proc.poll()` は `exit_code=None` のまま定数のため、最初の `_wait_exited` は必ずタイムアウトし、
   `os.killpg(1, signal.SIGKILL)` へのエスカレーションも発生する。SIGKILL は無視・捕捉不可能なため、
   **同一ユーザー(masaos)が所有する全プロセス — Claude Code CLI 自身を含む — に配送され得る**。

この説明は、これまで観測してきた全ての事実と整合する:

- CLI と(別途起動していた)バックグラウンドの検証スクリプトが同時に終了したこと
- カーネルレベルの OOM killer ログが一切ないこと(OOM ではなく意図的な SIGKILL 配送のため)
- WSL2 VM 自体は再起動していないこと(root 所有の `systemd` 等は UID 不一致で対象外になるため)
- メモリ・ネットワーク状態に依存せず、別 PC・別環境でも決定的に再現すること(Python/OS のシグナル配送仕様そのものが
  原因のため)
- 本調査自身のセッション PID が `17009` → `41109` → `55435` → `63035` と複数回変化していたこと(いずれも一次証拠として
  システム記録から確認済み)

### 除外した仮説(誤りだったと判明したが、調査プロセスの記録として残す)

以下は本当の原因が判明する前に検討・除外した仮説。結果的にはいずれも的外れだったが、除外に至った根拠自体は正しい。

1. **`tests/test_mcp_tools_validation.py` の `os.killpg`**: 隔離実験で呼び出し元セッションへの影響なしを確認し、
   PID 単体シグナルに変更済み(C-2 節)。変更後も同事象が再発したため主要因ではないと判断していたが、これは正しい
   判断だった(真の原因は `test_lifecycle.py` 側にあった)。
2. **メモリ逼迫・ネットワーク不調**: 状況証拠はあったが、ユーザーからの「ネットワーク安定時・別 PC でも再現する」との
   フィードバックにより除外。これも結果的に正しかった(真の原因は環境非依存の POSIX シグナル仕様バグ)。

### 同じ脆弱パターンを持つ他のテスト

`tests/test_lifecycle.py::TestStartHttpSubprocess` 内で、同一パターン(`_make_mock_proc()` で `pid` 未設定 +
`os.getpgid` 未モック + `startup_timeout_sec=0` により即座に終了処理へ到達)を持つテストがもう 1 件存在する:

- `test_raises_on_timeout` (L187)

同クラスの他のテストは、ヘルスチェックが成功する・早期終了で完結する・`_terminate_with_timeout` 自体をモックしている、
のいずれかにより安全であることを確認済み。

### 実施した対応

この修正は、並行して作業していた別セッションによっても独立に同一の根本原因が特定され、コミット `f2cec00b`
("fix: mock os.getpgid and os.killpg in lifecycle tests to prevent parent process kill") と `12a27fcf`
("fix: add _terminate_with_timeout patch for zero timeout test") として先に実装・コミットされていた
(本セッション側でも独立に `_make_mock_proc` へのデフォルト安全 pid 付与という同種の修正を行ったが、
先にコミットされていた上記2件に一本化された)。実際に採用された内容:

- `tests/test_lifecycle.py` の `_make_mock_proc()` に `pid: int = 999999`(実在しない PID)をデフォルト設定。
  `os.getpgid(999999)` は `OSError` を送出し、呼び出し元が明示的に上書きしない限り安全側に倒れる
- 加えて、`subprocess.Popen` をモックする全テストに `os.getpgid`/`os.killpg` の明示的モックを追加(多重防御)
- `test_zero_timeout_skips_health_polls` に不足していた `_terminate_with_timeout` のパッチを追加
- `test_raises_on_timeout` を `timeout=0` から `timeout=1` に変更し、実際にヘルスポーリングが走るよう修正
- `tests/conftest.py` に、各テストの開始・終了をログファイル(`/tmp/test_lifecycle_crash.log`)へ記録する
  `pytest_runtest_setup`/`pytest_runtest_teardown`/`pytest_sessionfinish` フックを追加。これは本調査で
  本ファイル(`issues/20260709_00_issue.md`)側が用いた「1ファイルずつ手動でログを取る」手法と全く同じ狙いを、
  恒久的な pytest フックとして実装したものであり、両セッションが独立に同じ診断手法へ収束したことになる

**検証**: 修正前は `test_zero_timeout_skips_health_polls` 実行中に確実にセッション再起動が発生していたが、
修正後は `tests/test_lifecycle.py` 全 42 件を実行し、全てパスかつセッション PID の変化なし(CLI 生存)を確認した。

**検討したが不採用とした対策**:

- 本番コード側(`HttpServerLifecycleManager`)に、`os.killpg`/`os.kill` 呼び出し直前で pgid が `0`/`1`/負数など
  POSIX の特殊値でないことを検証する防御的ガードを追加する案。`start_new_session=True` により本番の
  `proc.pid` は常に新規セッションの実PID(1ではあり得ない)となるため、本番コードでこの値になることは
  構造的に発生し得ない。テストのモック不備が原因であり、本番コードの防御的ガードは対象外と判断した。

### 再発時に追加調査すべき項目(今後、類似バグの早期発見のために)

- `~/.claude/sessions/*.json` の `startedAt`/PID 変化で CLI プロセスの再起動を事後検出できることを確認済み。
  今後も同様の手法で会話ログのタイムスタンプと突き合わせる
- フルテストスイートを 1 ファイルずつ実行し開始・終了をログファイルに記録する方式(本調査で使用)は、
  クラッシュ発生ファイルの特定に有効であることを実証済み

## C. 解決済み: テスト実行によるプロセスリーク・スレッドリーク

### Status: 修正済み (2026-07-10, commit `3b272c1a` "fix: eliminate thread-leak and process-leak risks in MCP subprocess lifecycle")

### C-1. `tests/test_lifecycle.py` — スレッドリーク

`TestHttpLifecycleStderrLog` (旧 L476-553) の 3 テストは `@pytest.mark.integration` だが、`subprocess.Popen` / `os.killpg` /
`httpx.AsyncClient` をモックしておらず実プロセスを生成していた。特に `test_start_timeout_stderr_from_log` は
`time.sleep(60)` でブロックする子プロセスを生成し、ヘルスチェックタイムアウト経由で `_terminate_with_timeout()` の
`os.killpg(pgid, SIGTERM)` を実呼び出ししていた。

根本原因は `scripts/agent/http_lifecycle.py` の `_terminate_with_timeout` が使う
`await asyncio.wait_for(asyncio.to_thread(proc.wait), timeout=timeout)`:
`asyncio.to_thread(proc.wait)` は `ThreadPoolExecutor` のスレッドで `proc.wait()` を実行し、`os.killpg`/`proc.terminate` が
失敗するとこのスレッドは子プロセスの終了を無限に待ち続ける。スレッドは非デーモンのため、Python 終了時の
`ThreadPoolExecutor` atexit フックがブロックし、プロセス全体がハングしうる。

**実施した対応**:
- `scripts/agent/http_lifecycle.py`: `_terminate_with_timeout` の待機方式を、スレッドを一切生成しないポーリング方式
  (`_wait_exited`、`proc.poll()` + `asyncio.sleep`)に置き換え、非デーモンスレッドの atexit ハングリスクを根絶。
- `tests/test_lifecycle.py`: `TestProcessGroupShutdown` を新実装に合わせて書き換え、`asyncio.to_thread` が呼ばれないことを
  直接検証する回帰テスト(`test_terminate_never_uses_thread_even_when_process_never_exits`)を追加。

副次的に見つかっていた PID 再利用の競合条件(`os.getpgid()` が `OSError` を上げた際に `proc.pid` を PGID として誤って保存する
フォールバック)も同時に修正済み。現在の `start()` (`scripts/agent/http_lifecycle.py:235-238`) は `except OSError: pass` のみで、
PID を PGID として誤って保存する経路は存在しない。

### C-2. `tests/test_mcp_tools_validation.py` — 同種リスク

`mcp_server` フィクスチャが `subprocess.Popen(..., start_new_session=True)` で実サブプロセスを起動し、teardown で
`_terminate_process_group`(`http_lifecycle._terminate_with_timeout` の独立複製)を呼ぶが、`os.killpg`/`os.getpgid` に
モックがなく実シグナルを送信していた。

**実施した対応**:
- `_find_free_port()` による動的ポート割当を、固定ポート + 事前バインドチェックに変更(サーバー側が `--host`/`--port`
  CLI 引数を一切パースしないため、動的ポート割当が機能していなかったことも判明・修正)。
- `_terminate_process_group` を `_terminate_process` に置き換え。

**設計変更の経緯**: 当初案は `os.killpg` を維持しつつモックのみ撤去する方針だったが、検証中に Claude Code CLI プロセス終了が
複数回報告されたため不採用とし、`os.killpg`/`os.getpgid` を完全に排除して PID 単体への `proc.terminate()`/`proc.kill()` のみに
変更した。これらのテストは `/v1/tools` のみを叩き `call_tool` を実行しないため、プロセスグループ単位で子孫を回収する必要がない。
なお、この変更後も B 節の事象(Claude Code CLI プロセス終了)は再度報告されており、本対応だけでは B 節を解消できていない。

### C-3. 問題なしと判断したファイル

| ファイル | 理由 |

### C-4. 未対応のまま保留とした項目(Priority: Low)

- `scripts/agent/http_lifecycle.py`: `_terminate_with_timeout` の SIGTERM 後タイムアウトを 3s → 1s に短縮する案は、
  本番相当環境での `shutdown_all()` 実測データが無いため保留(A 節の対策と合わせて再検討)。
