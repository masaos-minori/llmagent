## Original problem (still valid, but not yet triggerable)

`plans/20260717-180307_plan.md` documents the `config_dependent` / `enabled` / `disabled_reason`
metadata contract, to be written into a new file, `docs/04_mcp_03_06_tool-runtime-availability-metadata.md`,
based on field names/types/semantics only *planned* in requirement 15, which had not landed in real
source at authorship time. The original risk: if requirement 15 lands with a different shape than
assumed, the new doc would describe an inaccurate contract until manually corrected — and because this
is a documentation-only plan, it cannot mitigate the risk by testing against real implementation.

## Suggested resolution

1. No action is needed on the documentation itself right now — there is nothing to correct because
   nothing has been written yet.
2. When `implementations/20260718-101441_04_mcp_03_06_tool-runtime-availability-metadata.md` is
   executed and the file is created, verify the callout survived exactly as specified (its own
   Validation plan already checks this) — do not treat that as optional.
3. Once requirements 14/15/16/17 actually land in `scripts/`, re-read
   `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` against the real code and remove/update
   the "Implementation status" callout as part of that landing work — same follow-up trigger as
   originally suggested, unchanged.
4. This note and `issues/20260717-175327_risks.md` share the same root cause (requirement 15 not yet
   landed) and now also share the same evidence trail (the 2026-07-18 `implementations/` batch). They
   remain two separate files by design — one tracks the runtime-registry wiring gap, this one tracks
   the documentation-authoring gap — but a reader investigating either should read both.
