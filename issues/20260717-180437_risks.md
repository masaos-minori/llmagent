# Unmitigated risk: documentation may drift from requirement 15's actual field semantics

## Context

Plan: `plans/20260717-180307_plan.md` (from `requires/done/20260717_19_require.md`).

## Problem

`plans/20260717-180307_plan.md` documents the `config_dependent` / `enabled` /
`disabled_reason` metadata contract based on the field names, types, and semantics
currently planned in `plans/20260717-174024_plan.md` (requirement 15). As of this
writing, requirement 15 has not been implemented — verified via
`grep -rln "config_dependent\|disabled_reason\|RuntimeToolRegistry" scripts/` (zero
matches in production code).

This is the same underlying dependency already flagged as an open, unmitigated risk
for requirement 17 in `issues/20260717-175327_risks.md` ("RuntimeToolRegistry depends
on unimplemented sibling requirement 15"). This documentation plan inherits the same
exposure one layer further: if requirement 15 is ultimately implemented with a
different field name, type, or semantics than assumed (e.g. `disabled_reason: str |
None` instead of `str`, or a nested `availability` object instead of flat
`enabled`/`disabled_reason` fields), then the new
`docs/04_mcp_03_06_tool-runtime-availability-metadata.md` file this plan creates would
describe an inaccurate contract until manually corrected.

Because this is a documentation-only plan, it cannot mitigate this risk by writing
code or tests against the real implementation — there is no implementation yet to
verify against, and re-verifying doc accuracy against code is inherently a step that
must happen after requirement 15 (and 14/16/17) actually land.

## Suggested resolution

- When requirement 15 (and 14/16/17) are implemented, add a follow-up documentation
  task (or fold into those implementations' own validation plans) to re-read
  `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` against the landed code
  and correct any drift before removing the "Implementation status: target design,
  not yet implemented" callout this plan adds to that file.
- Treat the callout box (Design item 7 / Implementation step 3 in
  `plans/20260717-180307_plan.md`) as the trigger: whoever implements requirement 15
  should search for that callout and update/remove it as part of their own
  Implementation steps, rather than leaving stale "not yet implemented" text after
  the feature ships.
- Consider consolidating this risk with the already-open
  `issues/20260717-175327_risks.md` under a single cross-cutting tracking note, since
  both stem from the same root cause (requirement 15 not yet landed) — left as a
  human coordination decision, not resolved automatically here.

## Severity

Low — does not block writing or merging the documentation now (the content is
correct relative to the current plans, which are the best available source of
truth), but carries a follow-up-required flag once requirement 15's real
implementation lands, to keep docs and code from silently diverging.
