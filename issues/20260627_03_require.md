## [worker-runtime][roles] Define role presets for Planner / Retriever / PatchWorker / Integrator / Validator / Publisher

### Summary

各 worker role には、入力、出力、tool policy、禁止事項、runtime 制約を含む明確な preset が必要である。

### Problem

role ごとの目的と制約が曖昧だと、PatchWorker が write を行う、Integrator が publish を担う、Validator が approval を跨いで副作用を起こす、といった重大な責務逸脱が起きる。

### Scope
- role preset definitions
- prompt / input / output / tool policies
- role-specific constraints

### Task
- Planner preset を定義する
- Retriever preset を定義する
- PatchWorker preset を定義する
- Integrator preset を定義する
- Validator preset を定義する
- Publisher preset を定義する

### Planner Preset
- **Input**: user request, session context, repository summary
- **Output**: goal, change_sets, tasks, dependencies, acceptance_criteria
- **Tool policy**: read-only tools
- **Prohibited**: implementation, repository write, publish
- **Responsibility**: task graph 設計と acceptance criteria 定義のみ

### Retriever Preset
- **Input**: request, change_set, repository / document context
- **Output**: evidence list, context package, source metadata
- **Tool policy**: retrieval/search/read tools のみ
- **Prohibited**: implementation, repository write, publish
- **Responsibility**: RAG/evidence/context の正式生成主体

### PatchWorker Preset
- **Input**: file task, evidence/context, repository target, constraints
- **Output**: file patch, diff, rationale, changed file summary
- **Tool policy**: read tools + worktree 内の change proposal 系のみ
- **Prohibited**: repository write, GitHub write, publish
- **Responsibility**: file 単位 patch 生成のみ
- **Isolation**: task ごとの worktree が必要

### Integrator Preset
- **Input**: file patch group, repository context, validation prerequisites
- **Output**: patch_set, integrated diff, validation artifact, integration summary
- **Tool policy**: read / merge / consistency-check tools のみ
- **Prohibited**: repository write, Git operations, publish
- **Responsibility**: patch_set 統合と整合確認のみ

### Validator Preset
- **Input**: patch_set, repository state, validation scope
- **Output**: syntax result, lint result, unit test result, failed cases, logs, summary, retry_hint
- **Tool policy**: shell-mcp + read tools
- **Prohibited**: repository write, publish
- **Responsibility**: patch_set 全体検証と machine-readable 結果生成

### Publisher Preset
- **Input**: validation-verified patch_set, repository context, approval state, policy state
- **Output**: branch name, commit plan, push result, Draft PR payload, final summary
- **Tool policy**: repository_gateway.py 経由の write API のみ
- **Prohibited**: patch integration, approval bypass, direct raw github/file write
- **Responsibility**: approval 後の公開責務のみ

### Common Preset Rules
- 各 preset は stateless WorkerRuntime 上で動作する
- state の正本は role ごとに保持しない
- prompt policy / tool policy / input-output schema を preset に含める
- role 追加時は config-driven に拡張する

### Policy Alignment Notes

This requirement must follow the Phase 1 integration policy below.

1. **Do not create `orchestrator-mcp`.**
   - The central Orchestrator is an internal Agent component, not an MCP server.
   - Role MCP services may be added and monitored via MCP health/capability checks, but the Orchestrator itself must not be registered as an MCP server.
2. **Add internal Orchestrator APIs.**
   - REPL / command layers should call internal APIs such as `start_workflow`, `resume_workflow`, `cancel_workflow`, `get_workflow_status`, `get_workflow_artifacts`, and `retry_phase`.
3. **Use `state_manager.py` as the source of truth.**
   - Workflow / task / artifact / retry / approval state must be persisted and read through `state_manager.py`.
   - REPL, `AgentContext`, WorkerRuntime, JSONL logs, and UI code must not own canonical workflow state.
4. **Extend `workflow.sqlite`; event messages stay in the existing Event Bus.**
   - Phase 1 must reuse and extend `workflow.sqlite` for workflow metadata, task state, artifact metadata references, retries, and approvals.
   - Do not introduce a separate `orchestrator.sqlite` in Phase 1.
   - Event messages themselves are NOT duplicated into `workflow.sqlite`. They are already persisted by the existing Event Bus (`eventbus.sqlite`, managed by `scripts/eventbus/`), which provides publish/subscribe, replay, ack/nack, and DLQ over HTTP (see `docs/06_eventbus_*.md`). `workflow.sqlite` only tracks which Event Bus `event_id`s a workflow has already consumed, via the `processed_events` table (idempotency — see `scripts/agent/workflow/idempotency_ops.py`).

### Expected Result

orchestrator は全 role を安全かつ一貫して起動・管理できる。

### Acceptance Criteria
- Planner preset が定義されている
- Retriever preset が定義されている
- PatchWorker preset が定義されている
- Integrator preset が定義されている
- Validator preset が定義されている
- Publisher preset が定義されている
- 各 preset に tool policy と prohibited responsibility が含まれている
