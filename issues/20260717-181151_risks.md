# Requirement 20 (deferred include_disabled / disabled_code evaluation): both risks still open, one has an independent workaround (updated 2026-07-18)

## Context

Plan: `plans/done/20260717-181151_plan.md` (requirement 20, from `requires/done/20260717_20_require.md`).
Depends on: requirement 19 (`plans/done/20260717-180307_plan.md`), requirement 15
(`plans/done/20260717-174024_plan.md`), requirement 14 (`plans/done/20260717-173602_plan.md`).

All four plans are filed under `plans/done/` — that reflects planning-phase completion only. This note
was originally filed 2026-07-17 and is fully rewritten here after a follow-up investigation (2026-07-18).

## Risk 1: Future-options content has no landed home yet (depends on unimplemented requirement 19)

### Original problem (still valid)

`plans/20260717-181151_plan.md` designs a "Future / deferred design options" section intended to live
inside `docs/04_mcp_03_06_tool-runtime-availability-metadata.md`. That file was planned by requirement
19, which had not landed at authorship time — so the content would sit unattached to the canonical doc,
undiscoverable via the normal `04_mcp_00` navigation path, until someone appends it later.

### Current status (verified 2026-07-18)

- `find docs -iname "04_mcp_03_06*"` still returns **zero matches** — requirement 19 has not landed.
  `grep -rl "config_dependent\|disabled_reason" scripts/` also still returns zero matches (requirements
  14/15 unlanded too).
- `implementations/20260718-102011_04_mcp_03_06_future_options_appendix.md` (already noted in
  `issues/20260717-180437_risks.md`) holds this plan's "Future / deferred design options" content in
  reserve and re-checks `find docs -iname "04_mcp_03_06*"` before attempting to append — correctly
  designed as a conditional, not yet executed.
- **New finding**: a second, independent implementation procedure now exists —
  `implementations/20260718-102052_04_mcp_90_deferred_include_disabled_disabled_code.md`. Unlike the
  appendix above, its target file (`docs/04_mcp_90_inconsistencies_and_known_issues.md`) **already
  exists** today (confirmed via `find docs -iname "04_mcp_90*"`), so this entry can be written *now*,
  without waiting on requirement 19. It adds a "known gap / deferred decision" entry stating
  `include_disabled` and `disabled_code` were evaluated and intentionally deferred, cross-referencing
  `plans/20260717-181151_plan.md` and the (not-yet-existing) `04_mcp_03_06` appendix. This substantially
  reduces the original "undiscoverable" concern: even before `04_mcp_03_06` exists, a reader browsing
  `04_mcp_90`'s known-issues doc would find this feature's status.
- Neither procedure has been executed yet: `grep -c "include_disabled\|disabled_code"
  docs/04_mcp_90_inconsistencies_and_known_issues.md` returns `0`, and no matching file exists in
  `implementations/done/`.

### Suggested resolution (updated)

- Execute `implementations/20260718-102052_04_mcp_90_deferred_include_disabled_disabled_code.md` first
  — it has no dependency on requirement 19 and directly mitigates the discoverability gap this risk
  raised, sooner than waiting for the `04_mcp_03_06` appendix path.
- When requirement 19 lands and `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` is created,
  still execute `implementations/20260718-102011_04_mcp_03_06_future_options_appendix.md` as originally
  planned — the two docs serve different readers (known-issues browsers vs. the canonical spec) and are
  not a substitute for one another.

### Severity

Low — unchanged. Does not block requirement 20's own evaluation work, and now has a concrete,
independently-executable path to partially close the discoverability gap without waiting on
requirement 19.

## Risk 2: `disabled_code` proposal may drift from requirement 15's actual (unimplemented) `disabled_reason` design

### Original problem (still valid)

The `disabled_code` enum values in `plans/20260717-181151_plan.md`'s Design section are mapped against
today's `requires_config` per-server semantics, not against requirement 15's actual field shape, since
requirement 15 has not been implemented. If requirement 15 lands with a different mechanism, the
proposed `disabled_code` mapping table would need revision before anyone treats it as authoritative.

### Current status (verified 2026-07-18)

- `grep -rn "disabled_reason" scripts/` and `grep -rn "disabled_code" scripts/` both still return
  **zero matches** — requirement 15 has not landed, and `disabled_code` remains an unimplemented,
  purely evaluated proposal. Nothing has changed that would let this mapping be re-verified yet.
- `grep -rn "requires_config" scripts/` returns 51 occurrences across 10 files (unchanged file count
  from the plan's own baseline; the occurrence count differs slightly from the plan's cited "47" — per
  `implementations/20260718-101407_requires_config_sequencing_check_for_availability_metadata_docs.md`,
  this is attributed to grep-invocation/counting differences, not an actual code change — requirement
  14 has also not landed).
- `implementations/20260718-101407_requires_config_sequencing_check_for_availability_metadata_docs.md`
  (new since the original note) defines a re-grep-and-branch procedure for requirement 19's own doc
  edits to the 4 MCP catalog docs, specifically to avoid a clobbering edit against requirement 14's
  eventual `requires_config` → `config_dependent` rename. This improves cross-plan sequencing hygiene
  in the surrounding area but does not itself touch or resolve this risk's `disabled_code` mapping
  question — that remains fully gated on requirement 15 landing.

### Suggested resolution (unchanged)

- Re-review the `disabled_code` mapping table in `plans/20260717-181151_plan.md` once requirement 15
  actually lands, before promoting it from "evaluated proposal" to any implementation plan. Follow the
  same landing-check pattern already established for requirement 17
  (`implementations/20260718-094534_requirement_17_disabled_reason_landing_check.md`): grep for
  `disabled_reason` in `scripts/mcp_servers/`, and only then compare field shapes.

### Severity

Low — unchanged. This proposal is explicitly labeled deferred/unimplemented; the risk only
materializes if/when someone decides to implement `disabled_code`, which itself cannot happen before
requirement 15 lands.
