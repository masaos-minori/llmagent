# Unmitigated risks: requirement 20 (deferred include_disabled / disabled_code evaluation)

## Context

Plan: `plans/20260717-181151_plan.md` (from `requires/done/20260717_20_require.md`).

## Risk 1: Future-options content has no landed home yet (depends on unimplemented requirement 19)

`plans/20260717-181151_plan.md` designs a "Future / deferred design options" section
intended to live inside `docs/04_mcp_03_06_tool-runtime-availability-metadata.md`.
That file does not exist yet — it is itself only planned, by a different,
also-unimplemented plan: `plans/20260717-180307_plan.md` (requirement 19). Verified:
`find docs -iname "04_mcp_03_06*"` returns nothing, and
`grep -rl "config_dependent" implementations/ implementations/done/` returns zero
matches, confirming requirement 19 has not been implemented.

This is the third plan in the same chain to depend on requirement 19/15 landing
(the other two are already tracked in `issues/20260717-175327_risks.md` and
`issues/20260717-180437_risks.md`). Requirement 20's plan mitigates authorship risk by
keeping its Future-options content fully specified inside the plan document itself
(so the evaluation work is not blocked), but the content will sit unattached to the
canonical doc — and thus undiscoverable via the normal `04_mcp_00` navigation path —
until someone appends it after requirement 19 lands.

### Suggested resolution

- When requirement 19 (`plans/20260717-180307_plan.md`) is implemented and
  `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` is created, append this
  plan's "Future / deferred design options" subsection (verbatim from the Design
  section of `plans/20260717-181151_plan.md`) as part of that same implementation
  pass or a prompt follow-up.
- Consider consolidating this note with `issues/20260717-175327_risks.md` and
  `issues/20260717-180437_risks.md` into a single cross-cutting tracking issue for
  "requirements 14-20 chain ordering," since all three stem from the same root cause
  (requirement 15/19 not yet landed) — left as a human coordination decision.

### Severity

Low — does not block this plan's own documentation-evaluation work (the content is
fully captured regardless), but the deliverable is not yet discoverable in the
canonical doc tree until a follow-up append happens.

## Risk 2: `disabled_code` proposal may drift from requirement 15's actual (unimplemented) `disabled_reason` design

The `disabled_code` enum values in `plans/20260717-181151_plan.md`'s Design section
are mapped against today's `requires_config` per-server semantics (verified via
`grep -rl "requires_config" scripts/` — 10 files, 51 occurrences), not against
requirement 15's actual field shape, since requirement 15
(`plans/20260717-174024_plan.md`) has not been implemented — verified via
`grep -rln "disabled_reason" scripts/` (zero matches). If requirement 15 lands with a
different mechanism (e.g. a list of reasons, or a nested object instead of a flat
`disabled_reason` string), the proposed `disabled_code` mapping table would need
revision before anyone treats it as authoritative.

### Suggested resolution

- Re-review the `disabled_code` mapping table in `plans/20260717-181151_plan.md`
  once requirement 15 actually lands, before promoting it from "evaluated proposal"
  to any implementation plan.

### Severity

Low — this proposal is explicitly labeled deferred/unimplemented in the plan; the
risk only materializes if/when someone decides to implement `disabled_code` later.
