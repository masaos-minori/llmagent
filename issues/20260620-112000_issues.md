## ISSUE-A: build_execution_groups 戻り値型変更 (22 件)

### 対象テストファイル

- `tests/test_tool_scheduler.py` — 16 件
- `tests/test_tool_scheduler_comprehensive.py` — 6 件

### 失敗症状

```
AssertionError: assert 2 == 1
  where 2 = len(([[...]], _GroupMetadata(...)))
```

`len(groups)` が期待値 (グループ数) ではなくタプルの要素数 2 を返す。
インデックスアクセス時は `groups[0]` がグループのリスト全体を返し、
イテレーション時は `TypeError: list indices must be integers or slices, not str` が発生する。

### 根本原因

`scripts/agent/tool_scheduler.py` の `build_execution_groups()` の戻り値型が
`list[list[ToolCall]]` から `tuple[list[list[ToolCall]], _GroupMetadata]` に変更された。
テストはいずれも戻り値をそのままリストとして扱っており、タプルのアンパックを行っていない。

### 改修方針

`build_execution_groups()` を呼び出している全テスト行を以下のように修正する。

```python
# 修正前
groups = build_execution_groups(tcs, meta)

# 修正後
groups, _ = build_execution_groups(tcs, meta)
```

`_GroupMetadata` の内容を検証するテストについては `_` の代わりに変数名を付ける。
変更箇所はテストコードのみ。プロダクションコード変更不要。

## ISSUE-B: _slow_embed モックのシグネチャ不一致 (1 件)

### 対象テスト

- `tests/test_memory_layer.py::TestEmbeddingClientTimeout::test_timeout_returns_failure`

### 失敗症状

```
TypeError: _slow_embed() takes 4 positional arguments but 5 were given
```

### 根本原因

`scripts/agent/memory/embedding_client.py:146` の `_fetch_embedding()` 呼び出し引数が
`(text, http, url, query_prefix)` の 4 引数から `(text, http, url, query_prefix, embed_dim)` の
5 引数に変更された。テスト内の `_slow_embed` モック関数が旧シグネチャ (4 引数) のままである。

実際の呼び出し:
```python
# embedding_client.py:146
result = await asyncio.wait_for(
    _fetch_embedding(
        text,
        self._http,
        self._config.embed_url,
        self._config.query_prefix,
        self._config.embed_dim,   # <-- 追加された第5引数
    ),
    timeout=self._config.timeout,
)
```

テストのモック:
```python
# test_memory_layer.py (修正前)
async def _slow_embed(
    text: str, http: object, url: str, query_prefix: str
) -> EmbeddingResult:
    ...
```

### 改修方針

テスト内の `_slow_embed` に `embed_dim: int` パラメータを追加する。

```python
# 修正後
async def _slow_embed(
    text: str, http: object, url: str, query_prefix: str, embed_dim: int
) -> EmbeddingResult:
    await asyncio.sleep(10)
    return EmbeddingResult(success=True, embedding=[0.1])
```

変更箇所はテストコードのみ。プロダクションコード変更不要。

## ISSUE-C: security_lockdown_enabled 参照パスの不整合 (2 件)

### 対象テスト

- `tests/test_repl_health.py::TestAuditSecurityDefaults::test_sqlite_config_empty_db_allowlist_warns`
- `tests/test_repl_health.py::TestAuditSecurityDefaults::test_git_config_empty_allowed_repo_paths_warns`

### 失敗症状

`sqlite.db_allowlist` / `git.allowed_repo_paths` が空であるにもかかわらず、
`warnings` リストに対応する警告文字列が追加されない。

```
assert any("sqlite.db_allowlist" in w for w in warnings)
AssertionError: assert False
```

### 根本原因

`scripts/agent/repl_health.py:443` では `security_lockdown_enabled` を
`ctx.cfg.mcp` から正しく取得している。

```python
# line 443
lockdown = getattr(ctx.cfg.mcp, "security_lockdown_enabled", False)
```

しかし lines 457, 472, 487, 506 の各条件式では `lockdown` 変数を使わず
`ctx.cfg.security_lockdown_enabled` を直接参照している。

```python
# line 472 (誤り)
if not sqlite_cfg.db_allowlist and not ctx.cfg.security_lockdown_enabled:
```

テストの `_make_ctx` は `ctx.cfg.mcp.security_lockdown_enabled = False` と設定しているが
`ctx.cfg.security_lockdown_enabled` は設定していない。`ctx` が `MagicMock` のため
`ctx.cfg.security_lockdown_enabled` は `MagicMock()` (truthy) を返し、
`not ctx.cfg.security_lockdown_enabled` が `False` になって警告が発行されない。

### 改修方針

プロダクションコード (`repl_health.py`) の各条件式を `lockdown` 変数に統一する。

```python
# 修正前 (lines 457, 472, 487, 506)
if not xxx and not ctx.cfg.security_lockdown_enabled:

# 修正後
if not xxx and not lockdown:
```

変更対象: `scripts/agent/repl_health.py` lines 457, 472, 487, 506 の計 4 箇所。
テストコード変更不要。
