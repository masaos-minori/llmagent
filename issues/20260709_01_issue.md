# テスト実行によるプロセス終了問題

## 概要

`tests/test_lifecycle.py` のインテグレーションテスト実行が、LLM エージェントプロセスごと終了させる問題を調査。
同時に、同じパターンを持つ他のテストファイルの有無を調査し、対応要否を判断した。

---

## 1. `tests/test_lifecycle.py` — 原因分析

### Status: 部分的に修正済み（1-bのみ修正済み）

### 要因

#### 1-a. インテグレーションテストが実サブプロセスを起動 — 部分的に修正済み

`TestHttpLifecycleStderrLog` (L476-553) の 3 テストは `@pytest.mark.integration` だが、
現行では `subprocess.Popen` がモック済み (`monkeypatch.setattr(mod.os, "killpg", MagicMock())`)。

ただし `test_start_timeout_stderr_from_log` (L533-565) は `exit_code=None` でプロセスを生き続けさせており、
`_wait_exited` のタイムアウト待ちは依然として存在。`os.getpgid` はモックされていない。

#### 1-b. `_terminate_with_timeout` のバックグラウンドスレッドリーク — **修正済み**

コミット `3b272c1a` (fix: eliminate thread-leak and process-leak risks in MCP subprocess lifecycle) で対応済み。

`scripts/agent/http_lifecycle.py:L87-101` の `_wait_exited` が `asyncio.to_thread(proc.wait)` から
`proc.poll()` ポーリング方式に変更された:

```python
async def _wait_exited(self, proc: subprocess.Popen[bytes], timeout: float) -> bool:
    """Poll proc.poll() (non-blocking) until it exits or timeout elapses."""
    deadline = time.monotonic() + timeout
    while proc.poll() is None:
        if time.monotonic() >= deadline:
            return False
        await asyncio.sleep(self._TERMINATE_POLL_INTERVAL_SEC)
    return True
```

L746-764の `test_terminate_never_uses_thread_even_when_process_never_exits` がこれを検証している。

#### 1-c. PID 再利用の競合条件 — **未修正**

`start()` (L217-219):

```python
try:
    self._http_pgids[server_key] = os.getpgid(proc.pid)
except OSError:
    self._http_pgids[server_key] = proc.pid   # PID → PGID として保存
```

`os.getpgid()` が `OSError` (プロセスが既に終了) を上げると `proc.pid` を PGID として記録する。
この PID が他のプロセスに再利用された後に `_terminate_with_timeout` で `os.killpg(reused_pid, SIGTERM)` を呼ぶと、
**無関係なプロセスグループにシグナルが送信され、テストランナーや LLM エージェントが終了する可能性がある。**

#### 1-d. モック漏れ — 部分的に修正済み

`_patch_open_to_tmp` で `_open_stderr_log` のみパッチし、以下はモック済み/未対応:

- `subprocess.Popen` → **モック済み**
- `os.killpg` → **モック済み**
- `os.getpgid` → **未モック** (1-cのリスクに関連)
- `httpx.AsyncClient` → **モック済み**

### 対応案

- `os.getpgid` のモック追加 (1-cに関連)
- 実サブプロセスを要するテストは `tests/integration/` へ隔離

## 2. `tests/test_mcp_tools_validation.py` — 同種リスクあり

### Status: 修正済み

### リスク内容

#### 2-a. `_terminate_process_group` が `os.killpg` を実呼び出し — **修正済み**

`L84-111` の `_terminate_process` が `os.killpg` から `proc.terminate()/proc.kill()` に変更された。
ドキュメントコメントにもその経緯が記載されている:

```python
def _terminate_process(proc: subprocess.Popen, timeout: float) -> None:
    """Send SIGTERM to proc; escalate to SIGKILL if it doesn't exit in time.

    Signals the specific PID only — never os.killpg()/a process group. These
    tests only ever hit /health and /v1/tools, never /v1/call_tool, so the
    spawned MCP server never forks a child of its own that would need
    group-wide reaping. Using os.killpg() here previously risked signaling a
    process group broader than intended (observed to disrupt the surrounding
    shell/CLI session in some environments); a single Popen.terminate()/kill()
    call is scoped strictly to this one PID and cannot have that effect.
    """
```

#### 2-b. `mcp_server` fixture が実サブプロセスを起動 — **修正済み**

teardown で `_terminate_process` を呼んでいるが、これは単一PIDのみを対象とする。

### 2-a と `test_lifecycle` の差異

| 項目 | `test_lifecycle` | `test_mcp_tools_validation` |
|---|---|---|
| スレッドリーク | **修正済み** | **修正済み** |
| リスク度 | MEDIUM (1-cのみ) | LOW |
| PID 再利用 race | あり | なし |

## 3. 問題なしと判断したファイル

| ファイル | 理由 |
|---|---|
| `test_shell_mcp_service.py` | `os.killpg` / `asyncio.create_subprocess_exec` ともに `patch()` で完全モック |
| `test_agent_factory.py` | `_http_mgr` 全体が `AsyncMock()`、実プロセス操作なし |
| `test_startup.py` | lifecycle メソッドはすべて `AsyncMock()` |

---

## 4. 推奨アクション

### Priority: Medium

1. **`scripts/agent/http_lifecycle.py`**: `start()` の `os.getpgid()` OSError フォールバック処理を改善し、
   PID を PGID として保存しないようにする

### Priority: Low

2. **`scripts/agent/http_lifecycle.py`**: `_terminate_with_timeout` の SIGTERM 後タイムアウトを
   3s → 1s に短縮し、SIGKILL フォールバックまでの待機時間を削減する
