## [orchestrator][barrier] Define barrier rules between phases

### Summary

phase 間進行を安全に制御するため、orchestrator が判定する barrier ルールを正式化する。

### Problem

scheduler が runnable task を dispatch できても、phase gate が明確でなければ integrate / validate / publish が不正なタイミングで始まり得る。

### Scope
- phase barrier logic
- scheduler と barrier の責務分離
- barrier state persistence

### Task
- plan barrier を定義する
- retrieve barrier を定義する
- implement barrier を定義する
- integrate barrier を定義する
- publish barrier を定義する
- barrier state を run state にどう保存するか定義する

### Responsibility Split
- **task_scheduler.py**: runnable task 抽出、dependency 解決、並列 dispatch、同一 file 直列化
- **orchestrator_barrier.py / orchestrator**: phase 遷移ゲート判定、phase 開放条件確認、waiting/release 制御

### Barrier Definitions

#### Plan Barrier
- Planner 完了まで retrieve を開始しない
- Open 条件: goal + change_sets + tasks + dependencies + acceptance_criteria が確定

#### Retrieve Barrier
- Retriever 完了まで implement を開始しない
- Open 条件: change_set ごとの evidence list / context package / source metadata が揃う

#### Implement Barrier
- 全 independent file task 完了まで integrate を開始しない
- Open 条件: implement phase の対象 task が succeeded または terminal state に整理済み
- same-file tasks は scheduler が直列化する

#### Integrate Barrier
- Integrator が patch_set と validation artifact を確定するまで validate を開始しない
- Open 条件: integrated patch_set 完成、整合確認済み

#### Publish Barrier
- validate success + policy pass + approval granted + patch_set_hash match まで publish を開始しない
- Open 条件: approval record と artifact hash 一致、publish 前条件すべて満たす

### Barrier State Fields
- barrier_name
- workflow_id
- status (closed / open / released / blocked)
- required_inputs
- satisfied_at
- blocker_reason

### Policy Alignment Notes

This requirement must follow the Phase 1 integration policy below.

1. **Do not create `orchestrator-mcp`.**
   - The central Orchestrator is an internal Agent component, not an MCP server.
   - Role MCP services may be added and monitored via MCP health/capability checks, but the Orchestrator itself must not be registered as an MCP server.
2. **Add internal Orchestrator APIs.**
   - REPL / command layers should call internal APIs such as `start_workflow`, `resume_workflow`, `cancel_workflow`, `get_workflow_status`, `get_workflow_artifacts`, and `retry_phase`.
3. **Use `state_manager.py` as the source of truth.**
   - Workflow / task / artifact / retry / approval state must be persisted and read through `state_manager.py`.
   - REPL, `AgentContext`, WorkerRuntime, JSONL logs, and UI code must not own canonical workflow state.
4. **Extend `workflow.sqlite`.**
   - Phase 1 must reuse and extend `workflow.sqlite` for workflow metadata, task state, artifact metadata references, retries, approvals, and events.
   - Do not introduce a separate `orchestrator.sqlite` in Phase 1.

### Expected Result

phase 遷移は scheduler のみで決まらず、barrier で安全に統制される。

### Acceptance Criteria
- scheduler と barrier の責務分離が定義されている
- 各 barrier の open 条件が定義されている
- same-file serialization の関与点が明記されている
- barrier state が run state に含まれている
