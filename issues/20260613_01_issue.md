## BUG-1: `create_schema.py` モジュールレベルで `common.toml` を要求 → テスト collection 失敗

**症状**

```
ERROR tests/test_create_schema.py
ValueError: Config file not found: common.toml
```

**根本原因**

`scripts/db/create_schema.py:34` でモジュールトップレベルに

```python
logger = Logger(__name__, _get_schema_log_path())
```

があり、`_get_schema_log_path()` が `ConfigLoader().load("common.toml")` を実行する。
テスト実行時にはワーキングディレクトリに `common.toml` が存在しないため、
import 段階で `ValueError` が発生し、`test_create_schema.py` 全件が実行不可。

**影響テスト数**: 11 件 (test_create_schema.py 全件)

**修正方針**

`_get_schema_log_path()` 呼び出しをモジュールレベルから遅延初期化に変更するか、
Logger を関数スコープで初期化する。

## BUG-2: テストが `LLMMessage` を `rag.types` から import → `ImportError`

**症状**

```
ImportError: cannot import name 'LLMMessage' from 'rag.types'
```

**根本原因**

`tests/test_history_selection_policy.py:9` および `tests/test_history_manager.py:17` に

```python
from rag.types import LLMMessage  # 誤り
```

があるが、`LLMMessage` は `rag/types.py` には存在せず `shared/types.py` に定義されている。
プロダクションコード (`agent/history_selection_policy.py` 等) は正しく
`from shared.types import LLMMessage` を使っている。

**影響テスト数**: 2 ファイル全件 (collection 失敗)

**修正方針**

両テストファイルの import を `from shared.types import LLMMessage` に修正。

## BUG-3: テストヘルパーが `build_agent_config()` に `mcp_servers` なし dict を渡す → `ValueError` (最多)

**症状**

```
ValueError: mcp_servers config section is missing or empty.
Add [mcp_servers] to config/mcp_servers.toml.
```

**根本原因**

以下のテストファイルがローカルの `_cfg()` ヘルパーで最小 dict を構築して
`build_agent_config(defaults)` を呼んでいるが、`"mcp_servers"` キーが含まれていない。
`shared/mcp_config.py:108-116` の `_build_mcp_servers()` は `mcp_servers` セクションを
必須とし、存在しない場合に `ValueError` を raise する。

**影響ファイル** (代表):

| ファイル | 推定失敗件数 |
|---|---|
| `tests/test_tool_policy.py` | 12 件 |
| `tests/test_tool_audit.py` | 5 件 |
| `tests/test_tool_loop_guard.py` | 多数 |
| `tests/test_tool_runner.py` | 多数 |
| その他多数 | — |

**合計推定**: 200+ 件

**修正方針** (2案)

- A: 各テストヘルパーの `defaults` dict に `"mcp_servers": {"_dummy": {"transport": "http", "url": "http://127.0.0.1:9999"}}` を追加
- B: `_build_mcp_servers()` が空の場合は空 dict を返す (デフォルト変更) → 本番動作への影響に注意

## BUG-4: `test_tool_executor_routing.py` が `ToolCallResult` を tuple として unpack → `TypeError`

**症状**

```
TypeError: cannot unpack non-iterable ToolCallResult object
```

**根本原因**

`scripts/shared/tool_executor.py:48-54` の `ToolCallResult` は frozen dataclass (4 フィールド)。
テストは以下のように 3 要素 tuple として unpack している:

```python
result, is_err, x_req_id = await transport.call("my_tool", {})
```

`ToolCallResult` は `__iter__` を持たないため `TypeError` になる。
`transport.call()` / `_raw_execute()` / `execute()` の戻り値型が
tuple から dataclass に変わった際にテストが未更新のまま残されたと推定。

**影響テスト数**: `test_tool_executor_routing.py` の 15+ 件

**修正方針**

テスト内の unpack を属性アクセスに変更:

```python
res = await transport.call("my_tool", {})
result, is_err, x_req_id = res.output, res.is_error, res.request_id
```

## BUG-5: `TestFormatTransportError` が `TransportErrorInfo` を dict 扱い → `TypeError`

**症状**

```
TypeError: argument of type 'TransportErrorInfo' is not iterable
```

**根本原因**

`tests/test_llm_client.py:620-636` の `_call()` ヘルパーは
`cast(dict[str, str], format_transport_error(...))` で戻り値を dict とキャストしているが、
`cast` は型ヒントのみで実行時変換をしない。
実際の戻り値は `TransportErrorInfo` dataclass であるため、
`assert "summary" in result` が `TypeError` になる。

**影響テスト数**: `TestFormatTransportError` の 5 件

**修正方針**

テストの `_call()` 戻り値型と使用箇所を修正:

```python
def _call(self, ...) -> TransportErrorInfo:
    return format_transport_error(...)

# アクセス箇所
assert result.summary  # "summary" in result → result.summary
assert result.detail   # result["detail"]   → result.detail
```

## BUG-6: `test_mdq_service.py::test_logs_query` — INFO ログが存在しない → テスト失敗

**症状**

```
AssertionError: assert 'foo bar' in 'WARNING  mcp.mdq.search:search.py:52 MDQ FTS5 search failed: ...'
```

**根本原因**

`scripts/mcp/mdq/search.py` はクエリ文字列を INFO レベルでログに出力しない。
`_search_docs_structured()` が DB アクセスに失敗すると WARNING が出るのみ。
テストの `service` fixture は `_init_db` をパッチして DB を初期化しないため、
`sections_fts` テーブルが存在せず `OperationalError` が発生、
WARNING ログのみが記録される。

`test_logs_query` は `"foo bar"` が `caplog.text` に現れることを期待しているが、
クエリログの実装がないため常に失敗する。

**影響テスト数**: 1 件

**修正方針** (2案)

- A: `search.py` に `logger.info("MDQ search query: %s", req.query)` を追加
- B: テストが誤りなら削除または意図に合わせたログを検証するよう修正
