## [orchestrator] Freeze the Phase 1 scope

### Summary

Phase 1 は「REPL を UI に縮退し、中央 Orchestrator を内部コンポーネントとして導入する最初の動作可能リリース」の範囲に固定する。

### Problem

scope を明確に凍結しないと、orchestrator-mcp のような非採用案や、Phase 2 以降の role/機能が流入して実装が肥大化する。

### Scope
- Phase 1 roadmap
- MUST / SHOULD / LATER decision
- completion criteria

### Task
- Phase 1 MUST を定義する
- Phase 1 SHOULD を定義する
- Phase 2 以降へ送る項目を定義する
- completion criteria を明記する

### Phase 1 Scope

#### MUST
- orchestrator.py — workflow orchestration core (internal component)
- workflow_engine.py — phase state machine
- task_scheduler.py — DAG-based task scheduling with parallel execution
- retry_controller.py — retry policy per role/error class
- state_manager.py — workflow/task/artifact/retry/approval persistence
- approval_gate.py — write approval gating
- policy_engine.py — repository/branch/path/action constraints
- repository_gateway.py — unified repository I/O abstraction
- mcp_registry.py — role MCP endpoint registration and health/capability monitoring
- audit_logger.py — approval/retry/write audit logging
- orchestrator_barrier.py — phase barrier synchronization
- planner_runtime.py
- retriever_runtime.py
- patch_worker_runtime.py
- integrator_runtime.py
- validator_runtime.py
- publisher_runtime.py
- worker_runtime.py — reusable worker runtime
- worker_launcher.py — worker process management
- worktree_manager.py — worktree lifecycle management
- orchestrator_models.py — DTO definitions
- workflow.sqlite schema reuse/extension
- internal APIs: start_workflow, resume_workflow, cancel_workflow, get_workflow_status, get_workflow_artifacts, retry_phase

#### SHOULD
- status command / REPL progress integration
- role MCP watchdog integration
- worker log file output
- artifacts/<workflow_id>/ directory output
- orphaned worktree cleanup job

#### LATER (Phase 2+)
- Reviewer role
- Researcher role
- Memory Manager role
- external artifact store separation
- advanced retry backoff strategies
- dynamic concurrency limits per role
- richer approval UI / dashboard

### Completion Criteria
- workflow 作成・状態更新・task 登録・event 記録が単体で動作する
- REPL が internal orchestrator API の呼び出しに限定される
- RAG が Retriever に完全分離される
- repository write が Publisher 専用となる
- approval / policy / hash check を経ない publish が存在しない

### Expected Result

チームは最初の working release に本当に必要な要素だけを実装対象として共有できる。

### Acceptance Criteria
- MUST list が存在する
- SHOULD list が存在する
- LATER list が存在する
- Phase 1 completion criteria が明確である
- orchestrator-mcp が scope から除外されている
