# [worker-runtime][extensibility] Define extension points for future roles

## Summary
Future roles are planned for Phase 2 and later.

## Problem
If role handling is hardcoded for only 4 roles, later extension will be harder.

## Scope
- role registry design
- future role extension design

## Role Extension Rule (from spec section 3.2)

### Role registry design
- Roles are registered via `mcp_registry.py`
- New roles must implement the MCP tool contract:
  - `GET /health` — health check
  - `GET /v1/tools` — capability discovery
  - `POST /v1/call_tool` — task execution

### How new roles are added
1. Create a new MCP server file (e.g., `reviewer_mcp_server.py`)
2. Register the server in `mcp_registry.py`
3. Add role-specific configuration to config file
4. Define role-specific tool lists (allowed/denied)

### Role configuration extension
- Role configs are stored in the MCP server config TOML
- Each role has its own `[<role>-mcp]` section
- Config includes: transport, url, startup_mode, allowed_tools, denied_tools

## New Role Config Path (from spec section 3.2)

### Config path pattern
```toml
[<role>-mcp]
transport = "http"
url = "http://127.0.0.1:8009"
startup_mode = "persistent"
allowed_tools = [...]
denied_tools = [...]
```

## Future Role Policies Location

### Policy storage
- Role-specific policies live in `policy_engine.py`
- Policies are loaded from config files, not hardcoded
- New roles inherit the same policy framework:
  - Repository constraints
  - Branch constraints
  - Path constraints
  - Action constraints

## Current Design Does Not Hardcode Only 4 Roles

### Key design decisions that prevent hardcoding
- `mcp_registry.py` is the single source of truth for MCP endpoints — new roles are registered dynamically
- Role configs are TOML-based, not Python constants
- Policy engine uses config-driven rules, not role-specific branches
- Worker runtime is role-agnostic — it only knows about the task graph and barrier state

## Expected Result
The Phase 1 design stays extensible.
