# [validator][execution] Define the Validator execution contract

## Summary
The Validator role needs a clear execution and result contract.

## Problem
Validation must produce machine-readable results for the orchestrator.

## Scope
- validator runtime design
- command execution path
- validation result format

## Task
- define how Validator runs tests and checks
- define the result format:
  - pass/fail
  - summary
  - logs
  - retry hint
- define whether shell-mcp is enough or a dedicated execution path is needed

## Validator Execution Path (from spec section 3.3.5)

### Execution approach
- Validator uses shell-mcp for test execution
- No dedicated execution path needed — shell-mcp provides sufficient isolation
- Tests run in the context of the integrated `patch_set`

### Result format (from spec section 3.3.5)
```json
{
  "syntax": { "status": "pass" | "fail", "details": [...] },
  "lint": { "status": "pass" | "fail", "details": [...] },
  "unit_test": { "status": "pass" | "fail", "details": [...] },
  "failed_cases": [
    {
      "test_name": "...",
      "error": "...",
      "traceback": "..."
    }
  ],
  "logs": [...],
  "summary": "..."
}
```

### Retry hint format
- `retry_hint` field includes:
  - Error class category (syntax/lint/unit_test)
  - Suggested action (fix syntax / update lint rules / review test)
  - Whether retry is allowed based on retry policy

## Expected Result
The orchestrator can understand validator results and react automatically.

## Acceptance Criteria
- [ ] validator execution path is defined
- [ ] validator result format is defined
- [ ] retry hint format is defined
