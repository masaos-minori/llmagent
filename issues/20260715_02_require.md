# [Medium] Add the missing Browser MCP server; document the naming/architecture gaps in the "MCP integration plugin system" proposal

## Labels
priority/medium, area/mcp, feature, docs

## Summary
A feature request ("MCP統合プラグインシステム": a unified MCP Gateway with
`list_files`/`read_file`/`search_file`/`write_file`/`query_sqlite`/`invoke_script`,
covering Git/Browser/FileSystem/CI-CD/SQLite/Script-Runner MCP servers, path/command
allowlists, size limits, readonly/readwrite separation, and tunnel/auth-based
publishing) was proposed for investigation. Investigation shows this architecture is
**already implemented**, as 10 specialized MCP servers (not one "Gateway" component)
unified by `ToolExecutor` + `.importlinter` layer enforcement, with allowlists, command
allowlists, and size limits already present per-server. The only genuinely missing
MCP server is **Browser MCP**, which was already identified and explicitly deferred in
two prior planning documents. One conflict was also found: the proposal implicitly
assumes a "Local DB (SQLite) MCP" that was **deliberately removed** on 2026-07-08
(commit `479db1f6`) — re-adding it requires an explicit decision, not a default action.

## Background
Confirmed in code (see full investigation for exhaustive line-level citations):

- **10 existing MCP servers** in `scripts/mcp_servers/`: `git`, `github`, `cicd`,
  `file` (split into `file_read`/`file_write`/`file_delete`), `shell`, `rag_pipeline`,
  `web_search`, `mdq`. Each has its own `config/<name>_mcp_server.toml`.
- **`AGENTS.md:82` is stale**: it still lists `sqlite/service` in its test-coverage
  inventory, but `scripts/mcp_servers/sqlite/` and `config/sqlite_mcp_server.toml`
  do not exist. `tests/test_route_resolver.py:81-84` explicitly asserts
  `query_sqlite` must never resolve: `"""sqlite-mcp was removed; query_sqlite must
  not resolve to any server."""`. `docs/04_mcp_05_03_fail-open-fail-closed-and-risk-tiers.md:111`
  also still references a `db_allowlist` config key that exists nowhere in current
  `config/*.toml` — a second stale reference to the removed sqlite-mcp.
- **Gateway function names in the proposal do not match actual tool names** — this is
  a naming difference, not a functional gap:
  - `list_files` → actual: `list_directory` (`scripts/mcp_servers/file/read_tools.py:49`)
  - `read_file` → actual: `read_text_file` (`read_tools.py:70`)
  - `search_file` → actual: `search_files` (plural, `read_tools.py:91`)
  - `write_file` → matches exactly (`scripts/mcp_servers/file/write_tools.py:13`,
    `write_service.py:121`)
  - `query_sqlite` → removed; the only DB-adjacent server today is `mdq` (Markdown
    FTS5 index), not general SQLite query access
  - `invoke_script` → actual: `shell_run` (`scripts/mcp_servers/shell/tools.py:13`)
- **Execution guards already exist, per-server rather than centrally**:
  - path allowlist: `scripts/mcp_servers/file/common.py:27-41` (`resolve_safe`),
    `scripts/mcp_servers/mdq/auth.py:12-38` (`authorize_path`), both fail-closed on
    empty `allowed_dirs`; `config/git_mcp_server.toml`'s `allowed_repo_paths = []`
    (fail-closed, per its own comment)
  - command allowlist: `scripts/mcp_servers/shell/models.py:37`
    (`command_allowlist: list[str]`), `config/shell_mcp_server.toml`'s
    `command_allowlist = ["pytest", "python", "python3", "git", "ruff", "mypy"]`
  - size limits: `file/common.py`'s `check_size_limit`, plus per-server config keys
    (`max_read_bytes`, `max_write_bytes`, `max_output_kb`, `max_file_size_kb`,
    `max_log_size_kb`) — unit and field-name conventions are **not** standardized
    across servers (bytes vs KB, different key names)
  - readonly/readwrite separation: achieved either by splitting into separate
    processes (`file_read` vs `file_write` vs `file_delete`, each its own port and
    `allowed_dirs`) or by an in-server flag (`config/git_mcp_server.toml`'s
    `read_only = true`) — two different patterns coexist, not one
- **A unified "MCP Gateway" component, as a single named class/module, does not
  exist.** `grep -rni "mcp gateway|mcp_gateway|MCPGateway"` across `scripts/` returns
  zero hits; all hits are in planning documents. What exists instead is
  `scripts/shared/tool_executor.py:41` (`class ToolExecutor(ToolTransportInvoker)`),
  which routes every tool call through HTTP to the appropriate MCP server, combined
  with `.importlinter`'s `agent → mcp_servers` allowed / `mcp_servers → agent`
  forbidden layering (`AGENTS.md:56-68`). This already satisfies the proposal's
  stated goal ("Worker から直接 OS へ触らせず、MCP Gateway 経由を原則とする") —
  no Worker/Agent code can reach the OS except through an MCP server call.
- **Browser MCP does not exist.** `grep -rn "browser|playwright|Playwright"
  --include="*.py" scripts/` returns zero hits. The only web-adjacent server,
  `web_search`, wraps `duckduckgo_search.DDGS().text()`
  (`scripts/mcp_servers/web_search/search_provider.py:14,21-26`) — text search only,
  no page rendering, clicking, or screenshots. This gap was already identified twice
  before and explicitly deferred:
  - `plans/done/20260606-190413_plan.md:73`: "Phase 3 | Browser MCP | ✗ 未実装"
  - `plans/done/20260606-200114_plan.md:18,25`: "Browser MCP: 未実装、別instructionで扱う"
- **Tunnel/port-forward publishing does not exist, and was already deliberately
  deferred** — `plans/done/20260606-190413_plan.md:74-75`: "Phase 3 | Tunnel /
  port-forward 運用ルール | ✗ 未実装（ローカルファースト維持）"; the same plan's
  Scope Out section lists this explicitly as low priority for the local-first model.
- **Auth token middleware exists but is disabled by default** —
  `scripts/mcp_servers/server.py:78,90` implements Bearer-token verification; all
  current `config/*.toml` set `auth_token = ""` (disabled), documented as
  intentional for the loopback-only local dev default
  (`config/cicd_mcp_server.toml`'s comment: "Empty string = auth disabled
  (loopback-only default)"). `docs/04_mcp_06_17_local-to-production-auth-migration.md`
  already documents the production migration path.
- **Orchestrator/Worker integration (proposal section 4.3.6) is out of scope for this
  requirement** — `scripts/agent/orchestrator.py` (531 lines) implements single-agent
  turn-level control (compress → LLM call → tool loop), not the multi-Worker
  (Planner/Retriever/PatchWorker/Validator/Integrator) architecture described in
  `issues/multi-agent-orchestration-proposal.md`, which remains design-only
  (`grep -rln "PatchWorker|class Planner|class Retriever|class Validator|class
  Integrator"` returns zero hits). That is tracked separately and should not be
  conflated with this MCP-server-inventory requirement.

## Required Changes
1. **Add a Browser MCP server**, following the existing addition pattern documented
   in `docs/04_mcp_06_15_new-mcp-server-addition-checklist.md`:
   - New directory `scripts/mcp_servers/browser/` (server.py, service.py, tools.py,
     models.py, following the shape of `scripts/mcp_servers/shell/` as the closest
     existing analog for a sandboxed-execution-style server).
   - New `config/browser_mcp_server.toml` with `allowed_dirs`/`allowed_domains`-style
     fail-closed defaults (empty = deny all), a `read_only` toggle if write actions
     (form submission, downloads) are in scope, and explicit size/timeout limits.
   - Register the server in `config/agent.toml`'s `[mcp_servers.browser]` section and
     in whatever startup/lifecycle manifest the other 10 servers use.
   - Update `docs/04_mcp_04_01_web-search-file-read-github.md` (or add a new catalog
     doc following the `04_mcp_04_0N` numbering) to document the new server.
   - **This item requires an explicit scoping decision before implementation begins**
     (see Unknowns) — what browser actions are in scope (read-only page fetch/render
     vs. interactive clicking/form-fill/navigation) materially changes the security
     surface and effort.
2. **Fix the two stale references to the removed sqlite-mcp**:
   - `AGENTS.md:82`: remove `sqlite/service` from the test-coverage inventory line.
   - `docs/04_mcp_05_03_fail-open-fail-closed-and-risk-tiers.md:111`: remove or
     rewrite the `db_allowlist` reference, since no current server config uses that
     key.
3. **Document the naming difference** between the proposal's Gateway function names
   and actual tool names (`list_files`/`read_file`/`search_file`/`invoke_script` vs.
   `list_directory`/`read_text_file`/`search_files`/`shell_run`) in whichever doc
   currently serves as the MCP tool catalog index
   (`docs/04_mcp_00_document-guide.md`), so future proposals reference real names.
   This is a documentation clarification, not a rename of working tools — renaming
   live tool names is out of scope (see Non-Goals).
4. **Do not silently re-add SQLite MCP** — if a general-purpose SQLite query tool is
   still wanted, that must be a separate, explicit requirement, since the prior
   removal (`479db1f6`) was a deliberate decision whose rationale is not recorded in
   this repository (see Unknowns, blocking).

## Acceptance Criteria
- A Browser MCP server exists, is registered in `config/agent.toml`, has a
  fail-closed default config, and is documented in the MCP server catalog docs.
- `AGENTS.md` and `docs/04_mcp_05_03_fail-open-fail-closed-and-risk-tiers.md` no
  longer reference the removed sqlite-mcp (`sqlite/service`, `db_allowlist`).
- The MCP tool catalog documentation clarifies the actual names of the Gateway-style
  functions (list/read/search/write/run), removing ambiguity for future proposals.
- No SQLite MCP server is re-added as part of this requirement unless a separate,
  explicit decision authorizes it.

## Non-Goals
- Do not re-add a SQLite MCP server (`query_sqlite`) as part of this requirement —
  its removal was deliberate; reintroducing it needs its own requirement with an
  explicit rationale, not a default "restore what the proposal assumed."
- Do not implement tunnel/port-forward publishing — already explicitly deferred in
  `plans/done/20260606-190413_plan.md:74-75` for the local-first model; this
  requirement does not revisit that decision.
- Do not build a new unified "MCP Gateway" class/module to replace the existing
  per-server + `ToolExecutor` architecture — the proposal's stated goal (no direct
  OS access, centralized audit) is already met architecturally; consolidating into
  one component would be a large refactor with no described benefit over the current
  design.
- Do not implement Worker-role-based automatic `allowed_tools` assignment
  (orchestrator section 4.3.6) — that belongs to the separate, still design-only
  multi-agent orchestration track (`issues/multi-agent-orchestration-*`), not to this
  MCP-server-inventory requirement.
- Do not standardize the size-limit unit/field-naming conventions across existing
  servers (bytes vs. KB, differing key names) — flagged as an inconsistency, but
  changing it would touch every existing server's config schema, which is a larger
  refactor than "add a browser server."

## Unknowns
- **BLOCKING**: What Browser MCP's action scope should be — read-only page
  fetch/render (lower risk, closer to `web_search`) vs. interactive automation
  (clicking, form-fill, navigation, downloads — materially higher risk, needs its
  own sandboxing/allowlist design). This must be decided before Required Changes
  item 1 can be implemented.
- Why sqlite-mcp was removed (commit `479db1f6`'s message is the only record; no
  design doc explains the tradeoff) — needed before deciding whether any future
  SQLite-access requirement should re-create the old design or start fresh.
- Whether the current `file_read`/`file_write`/`file_delete` three-way split should
  eventually be presented as a single logical "FS MCP" in documentation/tooling, or
  whether the physical split is permanent — an open question already recorded in
  `requires/done/20260528_02_require.md:53`, not resolved by this requirement.
