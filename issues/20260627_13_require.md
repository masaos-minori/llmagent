# [orchestrator-mcp][response] Define structured MCP response handling for orchestration data

## Summary
The orchestration server needs more structured output than normal MCP tools.

## Problem
The current MCP pattern is mostly:
- `result`
- `is_error`

But orchestration needs:
- text summary
- structured run data
- metadata

## Scope
- orchestrator response format
- client-side result handling

## Structured Response Fields (from spec section 3.3.7)

### `result_text` — Human-friendly summary
- Contains a readable summary of the orchestration result
- Used for display in CLI / UI
- Example: "Workflow completed successfully. 3 tasks ran, 0 failed."

### `result_data` — Machine-readable structured data
- Contains the full structured data for programmatic consumption
- Includes workflow state, task results, artifact references
- Used by the orchestrator to make decisions (retry, abort, continue)

### `meta` — Metadata fields
- `workflow_id` — current workflow identifier
- `worker_id` — worker that produced this result
- `phase` — current phase name
- `timestamp` — when the result was produced

## Compatibility with Current MCP Result Model

### Current model
```json
{
  "result": "...",
  "is_error": false
}
```

### Mapping strategy
- `result_text` maps to `result` field (backward compatible)
- `is_error` maps based on validation status
- `result_data` and `meta` are added as additional fields when present
- Clients that only understand the current model will still work with `result_text`

## Client-Side Handling Strategy

### Two-tier handling
1. **Simple clients**: Read `result` + `is_error` only (backward compatible)
2. **Full clients**: Read `result_text`, `result_data`, `meta` for structured processing

## Expected Result
The agent can read human-friendly text and machine-friendly data from orchestration tools.
