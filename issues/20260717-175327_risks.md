# RuntimeToolRegistry / requirement-15 dependency: still unresolved, now tracked by explicit landing-check procedures (updated 2026-07-18)

## Context

Plan: `plans/done/20260717-175327_plan.md` (requirement 17, from `requires/done/20260717_17_require.md`).
Dependency plan: `plans/done/20260717-174024_plan.md` (requirement 15, "add `enabled`/`disabled_reason`
to `/v1/tools`").

Both plans are filed under `plans/done/` — that only means the **planning phase** is complete, not
that the described code changes have landed. This note was originally filed when requirement 17 was
authored (2026-07-17) and is fully rewritten here after a follow-up investigation (2026-07-18) into
what actually exists in `scripts/` today.

## Original problem (still valid)

`RuntimeToolRegistry` / `mcp_tool_discovery.py` was designed to consume `enabled` / `disabled_reason`
fields on each MCP server's `/v1/tools` response. Those fields were, at authorship time, only
*planned* (requirement 15), not implemented anywhere in the codebase. The original note flagged three
risks: (1) the feature has no real effect until requirement 15 lands, (2) requirement 15 might land
with a different field shape than assumed, (3) there was no cross-plan tracking mechanism to know when
requirement 15 actually lands.

## Current status (verified 2026-07-18)

**Requirement 17 (RuntimeToolRegistry) — partially implemented, not wired in**

- `scripts/shared/runtime_tool.py`, `scripts/shared/runtime_tool_registry.py`, and
  `scripts/agent/services/mcp_tool_discovery.py` exist as real source and implement the base shape
  (`RuntimeTool` dataclass, `build_runtime_tool()`, `RuntimeToolRegistry`, `McpToolDiscoveryService`).
- However, `grep -rln "McpToolDiscoveryService\|RuntimeToolRegistry" scripts/` returns matches only
  inside those three files themselves — nothing in `agent/context.py`, `agent/startup.py`,
  `agent/factory.py`, `route_resolver.py`, `tool_policy.py`, or `tool_runner.py` references them.
  `AppServices` has no `runtime_tools` field. The registry is built by nothing and consulted by
  nothing at runtime.
- `runtime_tool_registry.py`'s own module docstring already states this explicitly: "additive and
  unused until a later implementation step... wires existing call sites... to actually consult it."
  `docs/04_mcp_03_02_tool-registry.md:134` matches this — it documents the feature under a heading
  literally titled "将来の追加: RuntimeToolRegistry とライブ検出（未配線）" (future addition, not wired
  in). Code and docs agree; there is no drift here.
- The implemented `mcp_tool_discovery.py` does **not** read `enabled`/`disabled_reason` at all — it
  validates/consumes `status`, `is_write`, `requires_serial`, `resource_scope` only. This is exactly
  the "different field shape" scenario the original risk anticipated, except the actual outcome is
  simpler: requirement 15 never landed, so requirement 17 shipped independent of it rather than
  shipping against a mismatched shape.

**Requirement 15 (`enabled`/`disabled_reason`) — not implemented in source**

- `grep -rn "disabled_reason" scripts/` returns **zero matches**. The four target handlers
  (`scripts/mcp_servers/file/read_server.py`, `write_server.py`, `delete_server.py`,
  `scripts/mcp_servers/git/server.py`) are unchanged from their pre-plan `list_tools()` shape — no
  `enabled`/`disabled_reason` merge logic exists.
- So the original risk's central concern ("every live server will report fallback defaults, i.e.
  every tool will appear enabled regardless of true state") is currently moot in a stronger sense:
  the discovery service that would even read those fields isn't invoked at all, since it is unwired
  per the point above.

**New since the original note: explicit landing-check procedures now exist (2026-07-18)**

A large batch of `implementations/*.md` procedure docs was authored today addressing exactly the gap
this note raised — but these are documents-only, not source changes yet
(`implementations/20260718-095246_requirements_14_15_landing_check_for_availability_metadata_schema_tests.md`
states outright: "no real source changes have been made by ANY plan in this batch yet — this entire
batch produces `implementations/*.md` design/procedure docs only, not code"). Relevant docs:

- `implementations/20260718-111454_context_py_appservices_runtime_tools.md` — designs the missing
  `AppServices.runtime_tools` wiring (adds the field, populates it from `McpToolDiscoveryService` in
  `StartupOrchestrator._check_services()`). This is the fix for the "not wired in" gap above.
- `implementations/20260718-094534_requirement_17_disabled_reason_landing_check.md` — a mechanical
  pre-merge check: `grep -rn "disabled_reason" scripts/mcp_servers/` before merging requirement 17's
  work; branches into "add integration test against real shape" (if landed) vs. "keep defaulting path,
  document the dependency in code comments" (if not landed). This directly answers the original risk's
  concern #3 (no cross-plan tracking) with a concrete, repeatable procedure.
- `implementations/20260718-095246_requirements_14_15_landing_check_for_availability_metadata_schema_tests.md`
  — an analogous gate for requirement 18's schema tests, using `xfail(strict=True)` markers on
  assertions that depend on requirement 14/15 landing, so an unexpected pass (drift undetected) itself
  becomes a hard test failure.
- A parallel, already-designed (not yet real-source) lineage for requirement 15 itself exists:
  `implementations/20260718-085909_read_tools.py.md` through
  `implementations/20260718-091922_full_validation_pass_call_tool_disabled_gate.md`, plus
  `implementations/20260718-090617_read_server.py.md`, `..._write_server.py.md`,
  `..._delete_server.py.md`, `..._git_server.py.md`, `..._common.py.md`.

## Suggested resolution

1. Turn the design docs into real source, in this order (each unblocks the next):
   - `implementations/20260718-111454_context_py_appservices_runtime_tools.md` (wires
     `RuntimeToolRegistry` into `AppServices`/startup — makes requirement 17 have any runtime effect
     at all).
   - Requirement 15's own doc lineage (`..._read_server.py.md`, `..._write_server.py.md`,
     `..._delete_server.py.md`, `..._git_server.py.md`, `..._common.py.md`) — lands
     `enabled`/`disabled_reason` server-side.
   - Run the landing-check procedure in `implementations/20260718-094534_requirement_17_disabled_reason_landing_check.md`
     once both are real, and update `mcp_tool_discovery.py`'s parsing/tests accordingly.
2. Until step 1 lands, do not treat requirement 17 as "done" in any user-facing sense — no LLM
   tool-calling behavior is affected by `RuntimeToolRegistry` today; it is inert.
3. When these implementation docs are turned into code and moved to `implementations/done/`, this
   issue file should be re-verified once more and closed if the dependency chain is confirmed landed
   end-to-end (registry wired in AND consuming real `enabled`/`disabled_reason` values AND an
   integration test proves it).

## Severity

Medium, unchanged from the original assessment — still does not block other work, and a concrete plan
now exists to resolve it (the landing-check docs above), but the feature remains fully inert in
production until the wiring + requirement-15 landing both happen.
