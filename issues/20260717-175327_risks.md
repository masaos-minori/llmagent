# Unmitigated risk: RuntimeToolRegistry depends on unimplemented sibling requirement 15

## Context

Plan: `plans/20260717-175327_plan.md` (from `requires/done/20260717_17_require.md`).

## Problem

`RuntimeToolRegistry` / `mcp_tool_discovery.py` (this plan) is designed to consume `enabled` /
`disabled_reason` fields on each MCP server's `/v1/tools` response. As of this plan's authorship,
those fields do not exist anywhere in the codebase — verified via:

```
grep -rn "disabled_reason" .   # (excluding caches) — zero matches
```

The fields are only *planned*, not implemented, in sibling requirement 15's plan
(`plans/20260717-174024_plan.md`, targeting `scripts/mcp_servers/file/read_server.py` and
`scripts/mcp_servers/git/server.py`).

This plan's `mcp_tool_discovery.py` includes a defaulting fallback (`enabled=True,
disabled_reason=""` when the fields are absent from a server's response) so that the code can be
written and unit-tested today against mocked payloads. That keeps *authorship* unblocked, but it
does **not** mitigate the deeper risk:

- Until requirement 15's plan is actually implemented and deployed, this requirement's entire
  feature (disabled tools hidden from the LLM, shown in diagnostics) has nothing real to operate
  on — every live server will report the fallback defaults, i.e. every tool will appear
  "enabled" regardless of its true state.
- If requirement 15's plan is implemented with a different field name, type, or semantics than
  assumed here (e.g. `disabled_reason: str | None` instead of `str`, or a nested `availability`
  object instead of flat fields), `mcp_tool_discovery.py`'s parsing logic will need rework that
  this plan cannot anticipate in advance.
- There is no cross-plan tracking mechanism in this repo (no shared ticket/issue linking
  `plans/20260717-174024_plan.md` and this plan) beyond this note and the "SHARED WITH" callouts
  in each plan's own text — a human coordinating implementation order needs to know requirement 15
  should land (or at minimum have its exact `/v1/tools` response shape finalized) before this
  requirement's discovery/registry work is considered "done" in the full, non-fallback sense.

## Suggested resolution

- Sequence implementation: land `plans/20260717-174024_plan.md` (requirement 15) first, or at
  minimum finalize and freeze its exact per-tool `/v1/tools` field names/types before writing
  `mcp_tool_discovery.py`'s parsing code for this plan.
- When implementing this plan, add an explicit integration test (not just mocked-payload unit
  tests) against the real, updated `/v1/tools` handler in `scripts/mcp_servers/file/read_server.py`
  / `scripts/mcp_servers/git/server.py` once requirement 15 lands, to catch shape drift early.
- If requirement 15 is deprioritized or changes shape substantially, this plan's Design section
  (RuntimeTool fields, discovery defaulting logic) must be revisited before implementation
  proceeds past the unit-test stage.

## Severity

Medium — does not block writing/reviewing this plan or its code structure, but blocks the feature
from having any real effect until the dependency is resolved, and carries integration-shape risk
if the two plans diverge.
