# [orchestrator-mcp][ops] Define how orchestrator-mcp is added to /mcp and watchdog

## Summary
The new orchestrator server must fit into the current MCP operations model.

## Problem
We need a consistent way to:
- start it
- monitor it
- show it in `/mcp`
- restart it with watchdog

## Scope
- MCP server config
- operational visibility

## Service Naming (from spec section 3.6)

### Service name
- `orchestrator-mcp` — follows existing convention (`integrator-mcp`, `patchworker-mcp`)

## MCP Server Config Entry (from spec section 3.6)

### Config format
```toml
[orchestrator-mcp]
transport = "http"
url = "http://127.0.0.1:8009"
startup_mode = "persistent"
```

## `/mcp` Status Behavior

### Status endpoint response
- Shows orchestrator-mcp as a registered MCP server
- Displays health status from `mcp_health_status` table
- Shows active workflow count

## Watchdog Monitoring (from spec section 3.6)

### Watchdog monitoring
- Health check via `GET /health` endpoint
- Watchdog monitors health status every N seconds
- On failure: restart service, log event, alert if persistent

## Expected Result
The new server works like other MCP services in operations.

- [ ] service naming is defined
- [ ] MCP config entry shape is defined
- [ ] `/mcp` integration behavior is defined
- [ ] watchdog expectations are defined
