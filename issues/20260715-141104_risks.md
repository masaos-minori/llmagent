## Issue: Consider widening cache-bypass to other mutating RAG/MDQ tools not named in the requirement

### Priority

Medium

### Type

Follow-up / Safety

### Background

The requirement `requires/20260715_20_require.md` names an explicit list of tools that must bypass `ToolExecutor` caching: `write_file`, `delete_file`, `shell_run`, `git_push`, `github_merge_pull_request`, `trigger_workflow`, `rag_delete_document`, `fts_rebuild`. The plan implements exactly this list by adding `RAG_WRITE_TOOLS = {rag_delete_document}`, `CICD_WRITE_TOOLS = {trigger_workflow}`, `MDQ_WRITE_TOOLS = {fts_rebuild}` to `scripts/shared/tool_constants.py` and unioning them into `_SIDE_EFFECT_TOOLS`.

However, the following tools also mutate state but are **not** in the requirement's list and therefore remain cacheable after this plan is implemented:

- `rag_run_pipeline` — triggers an ingestion pipeline run
- `rag_debug_pipeline` — likely also triggers pipeline execution (debug mode)
- `index_paths` — triggers an MDQ index build (already classified as `_WRITE_TOOLS` locally inside `scripts/mcp_servers/mdq/tools.py`, separate from `shared/tool_constants.py`)
- `refresh_index` — triggers an MDQ index refresh (same local classification as above)

A cached response for any of these could create the same false impression the requirement is trying to eliminate for `rag_delete_document`/`fts_rebuild`: that a pipeline run or index rebuild happened, when in fact a stale cached text was returned instead.

### Why this was not included in the plan

- Non-goal per the requirement: "do not include speculative steps — only steps required by the stated goal."
- No reviewer/product-owner confirmation was available during planning (no interactive question tool in this session) to confirm whether these tools were deliberately left out or simply not considered.
- The codebase already has multiple, inconsistent notions of "write" tool across `shared/tool_constants.py`, `agent/tool_policy.py._API_WRITE_TOOLS`, `mcp_servers/mdq/tools.py._WRITE_TOOLS`, and `config/agent.toml [tool_safety_tiers]` (server-level tier, not per-tool). Reconciling all of these is a larger refactor than this requirement asks for.

### Suggested next step

Confirm with the requirement owner whether `rag_run_pipeline`, `rag_debug_pipeline`, `index_paths`, `refresh_index` should also bypass cache. If yes, extend `RAG_WRITE_TOOLS`/`MDQ_WRITE_TOOLS` accordingly and add matching bypass tests, following the same pattern established by this plan.

### Acceptance Criteria (if taken up)

- Decision recorded on whether each of the four tools above should bypass cache.
- If yes: tools added to the appropriate `tool_constants.py` write-set; `is_side_effect()` covers them; new `test_<tool>_bypasses_cache` tests added.
- If no: rationale documented (e.g. "these are idempotent / safe to serve stale for N seconds") so the decision is not re-litigated later.
