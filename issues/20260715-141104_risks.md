# Follow-up issue from plan `plans/20260715-140914_plan.md` — RESOLVED

Source requirement: `requires/done/20260715_20_require.md` ("Make ToolExecutor cache safe
for side-effecting tools and preserve cache attribution")

This issue originally flagged four tools (`rag_run_pipeline`, `rag_debug_pipeline`,
`index_paths`, `refresh_index`) as candidates for `ToolExecutor` cache-bypass, on the
theory that all four mutate state the same way `rag_delete_document`/`fts_rebuild` do.
Follow-up investigation found this was only half right, and one factual claim below was
wrong. The resolution has been applied to the not-yet-implemented design documents
(`implementations/20260715-152212_tool_constants.py.md`,
`implementations/20260715-152212_tool_executor_helpers.py.md`,
`implementations/20260715-152212_test_tool_constants.py.md`) — no source code has
changed yet, since implementation had not started.

## Resolution

### `index_paths` / `refresh_index` (MDQ) — added to cache-bypass scope

**Confirmed as a real gap, now fixed in the design docs.** Both tools write to
`mdq.sqlite` (`scripts/mcp_servers/mdq/indexer.py`: `INSERT`/`DELETE`/`commit`) and are
already self-classified as write tools locally
(`scripts/mcp_servers/mdq/tools.py:177`'s `_WRITE_TOOLS`, plus `is_write=True`/
`requires_serial=True` in their tool schemas) — but were left out of
`shared/tool_constants.py`'s `MDQ_WRITE_TOOLS`, which is what `is_side_effect()` (and,
once implemented, `ToolExecutor.execute()`'s cache-bypass) actually consults. That
inconsistency was real: a cached response could return a stale `"Indexing complete"`
message or stale `indexed_count` without the index actually being rebuilt.

`MDQ_WRITE_TOOLS` now reads `frozenset({"fts_rebuild", "index_paths", "refresh_index"})`
in the updated design docs; the corresponding test doc's read-only assertion list has
had `index_paths`/`refresh_index` removed and replaced with positive
`is_side_effect() is True` assertions for all three.

### `rag_run_pipeline` / `rag_debug_pipeline` — NOT added; original issue text was wrong

**The original claim that these "trigger an ingestion pipeline run" is factually
incorrect.** Both are read-only retrieval/search pipelines
(`scripts/mcp_servers/rag_pipeline/service.py:146-183`: MQE → search → RRF → rerank →
augment), confirmed to contain zero database write statements
(`scripts/rag/pipeline.py` has no `INSERT`/`UPDATE`/cursor-write calls). The in-process
`SemanticCache` they use is a memory-only cache, not persisted state. Their responses
(`augmented_text`, `selected_hits`, etc.) are search results, not "operation completed"
confirmations — so a stale cached response does not carry the same false-completion
risk that `rag_delete_document`/`fts_rebuild`/`index_paths`/`refresh_index` do. These
two tools are intentionally left out of `RAG_WRITE_TOOLS` and remain cacheable, same as
`search_docs` or other read tools. `tests/test_tool_constants.py`'s
`test_rag_read_tools_not_side_effect` already asserts this and needs no change.

## Why this was worth a follow-up pass rather than accepting the original issue as-is

- The original issue treated all four tools as one risk class. They are not: two are
  genuine write operations missing from the classification, two are read operations
  incorrectly described as writes.
- No design/product decision was actually required here — the RAG-vs-MDQ distinction is
  resolvable from source code alone (DB write statements present or absent), which is
  why this was corrected directly rather than re-escalated as a new open question.

## Status

Closed. No remaining action beyond applying the (already updated) design documents
during implementation of `plans/done/20260715-140914_plan.md`.
