# [orchestrator][storage] Decide whether to reuse workflow.sqlite or add orchestrator.sqlite

## Summary
The current platform already has `workflow.sqlite`.
The orchestration plan also suggests a new `orchestrator.sqlite`.

## Problem
We need one clear storage strategy for multi-agent run state.

## Scope
- workflow DB reuse analysis
- orchestrator DB design choice

## Task
- compare `workflow.sqlite` reuse vs new `orchestrator.sqlite`
- list benefits and risks of both options
- choose one approach for Phase 1

## Storage Decision (from spec section 3.5)

### Recommended: Reuse `workflow.sqlite`
- The existing `workflow.sqlite` already has the tables needed:
  - `workflows`, `workflow_tasks`, `workflow_dependencies`, `workflow_artifacts`
  - `workflow_retries`, `workflow_approvals`, `workflow_events`
  - `mcp_endpoints`, `mcp_health_status`
- Adding orchestrator-specific columns to existing tables is simpler than maintaining two databases

### Artifact storage strategy (Phase 1)
- Use DB inline for artifacts in Phase 1:
  - diff
  - `patch_set`
  - validator logs
  - evidence bundle
  - Draft PR payload
- Future: separate external artifact store when size becomes a concern

### Risk mitigation
- Set artifact size limits per column
- Define truncation policy for large fields
- Leave room for future external artifact store separation

## Expected Result
The project has one clear orchestration metadata storage plan.

## Acceptance Criteria
- [ ] reuse vs new DB decision is made
- [ ] decision reasons are documented
- [ ] schema direction is defined
