# Plan: Strict Two-Element Tuple Validation for Plugin Tool Return Values

## Goal

Enforce that plugin tool return values are exactly `tuple[str, bool]` (two elements) at runtime, making runtime validation consistent with the registration-time annotation check.

## Scope

### In-Scope

- Update runtime validation in `ToolExecutor.execute()` to reject tuples with length != 2
- Update the module-level constant `_PLUGIN_RESULT_TUPLE_LENGTH` to reflect strict equality
- Add test cases for all six acceptance criteria from the requirement
- Update plugin documentation to state the strict two-element contract

### Out-of-Scope

- Changing plugin tool dispatch priority
- Changing MCP tool result format
- Changing plugin command behavior
- Changing the registration-time annotation check (already strict)

## Assumptions

1. The current `< _PLUGIN_RESULT_TUPLE_LENGTH` (i.e., `< 2`) guard was an oversight, not an intentional design decision; no existing plugin relies on returning a 3+-element tuple.
2. Plugin code that already declares `-> tuple[str, bool]` and returns exactly two elements will not be broken by this change.
3. The strict check should raise `ValueError` (same exception type as the current path), not `TypeError`, to maintain consistent error handling in `execute()`.
4. The doc file `docs/05_agent_11_extension-points.md` is the canonical plugin documentation; no other doc file needs updating for this change.

## Implementation Steps

### Step 1: Update `_PLUGIN_RESULT_TUPLE_LENGTH` usage in `tool_executor.py`

**File:** `scripts/shared/tool_executor.py`

Change line 570 from:
```python
or len(result_raw) < _PLUGIN_RESULT_TUPLE_LENGTH
```
to:
```python
or len(result_raw) != _PLUGIN_RESULT_TUPLE_LENGTH
```

Update the error message on lines 572-575 to say "must return exactly `tuple[str, bool]` (2 elements)" instead of "must return `tuple[str, bool]`".

Also update the module-level comment on line 37 from:
```python
# Plugin tool return value: (output: str, is_error: bool)
_PLUGIN_RESULT_TUPLE_LENGTH = 2
```
to clearly indicate this is a strict equality check.

**Completion criterion:** The condition `len(result_raw) != 2` raises `ValueError` for both 1-element and 3-element tuples.

### Step 2: Add plugin return value validation tests to `test_tool_executor.py`

**File:** `tests/test_tool_executor.py`

Add a new test class `TestPluginReturnValidation` with the following cases:

1. `test_non_tuple_return_raises`: plugin returns a `str` → `ValueError` raised, `ToolCallResult` not returned
2. `test_one_element_tuple_raises`: plugin returns `("ok",)` → `ValueError` raised
3. `test_valid_two_element_tuple`: plugin returns `("ok", False)` → success, `output="ok"`, `is_error=False`
4. `test_three_element_tuple_raises`: plugin returns `("ok", False, "extra")` → `ValueError` raised (new behavior; previously this was silently accepted)
5. `test_wrong_output_type_raises`: plugin returns `(123, False)` → `TypeError` raised
6. `test_wrong_is_error_type_raises`: plugin returns `("ok", "no")` → `TypeError` raised

Each test should register a plugin function via `plugin_registry._tools` directly (bypassing `@register_tool` annotation check) to allow injecting bad return values, and call `executor.execute()` with the tool name.

**Completion criterion:** All 6 test cases pass; `uv run pytest tests/test_tool_executor.py::TestPluginReturnValidation -v` exits 0.

### Step 3: Update plugin documentation

**File:** `docs/05_agent_11_extension-points.md`

In the `@register_tool` section, add a note under "Return-type validation (fail-fast):" that the runtime validation enforces **exactly** two elements:

> **Runtime return value validation:** `ToolExecutor.execute()` validates the actual return value at call time. It checks that the return is a `tuple` with exactly 2 elements (`len == 2`), that `result[0]` is `str`, and that `result[1]` is `bool`. A tuple with more or fewer than 2 elements raises `ValueError`.

**Completion criterion:** The doc explicitly states that tuples with != 2 elements are rejected at runtime.
