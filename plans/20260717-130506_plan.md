# Plan: Consolidate tool startup validation into MCP discovery service

Source requirement: `requires/20260717_09_require.md`

## Goal

All tool-validation logic currently split across `check_tool_definitions_startup()`, `check_routing_drift_vs_live()`, duplicate-ownership detection, and malformed-`/v1/tools`-response checks (all in `scripts/agent/repl_health.py`) is consolidated into `scripts/agent/services/mcp_tool_discovery.py` (requirement 03) — discovery and validation happen in one service, one fetch per server, not several independent passes producing overlapping findings.

## Scope

**In scope**
- Migrate `check_tool_definitions_startup()` (`repl_health.py:373-376`, thin wrapper over `_check_tool_definitions(ctx, strict=...)`) into the discovery service.
- Migrate `check_routing_drift_vs_live()` (`repl_health.py:379-459`) **in full** — this function already implements duplicate-live-tool-ownership detection (lines 426-440), 4-way drift detection (tool-in-live-not-registry, tool-in-registry-not-live, wrong-owner, duplicate-ownership), and strict/non-strict severity branching — this is the most substantial migration in this requirement.
- Migrate the malformed-`/v1/tools`-response validation (`_validate_tools_response()`, `repl_health.py:158-186`) — already partially planned as part of requirement 03's own discovery-validation logic; this requirement confirms it belongs in the same consolidated service, not left in `repl_health.py`.
- `scripts/agent/startup.py`: update its `_check_services()` pipeline to call the consolidated discovery-and-validation service instead of the current 3 separate imports (`check_tool_definitions_startup`, `check_routing_drift`, `check_routing_drift_vs_live`).
- `scripts/shared/tool_routing_validation.py`: `validate_routing_against_live()` (currently consumed only by `check_routing_drift_vs_live()`) becomes an internal helper called from within the discovery service, not a separately-imported public function for `repl_health.py`.
- **Supersedes a recommendation in requirement 03's own plan** (`plans/20260717-124907_plan.md`'s Unknowns section, which recommended keeping the two `/v1/tools` fetches independent "for this requirement's scope" while flagging unification as "a natural follow-up") — this requirement *is* that follow-up, made explicit and immediate rather than deferred.

**Out of scope**
- `check_routing_drift()` (`repl_health.py:531`, config-vs-registry drift, not live-vs-registry) — this checks static config `tool_names` lists against the registry, a different concern from live `/v1/tools` validation; not named in this requirement's scope and not touched.
- `check_routing_safety_tiers()` (`repl_health.py:562`) — not named in scope; a distinct config-validation concern.
- Any change to the actual duplicate-tool-name *policy* (production=fatal, local=warning) — that's requirement 03's design; this requirement only relocates *where* the check runs, not its severity rules.

## Assumptions

1. `check_tool_definitions_startup()` (`repl_health.py:373-376`, confirmed by direct read) is a 2-line wrapper: `strict = getattr(ctx.cfg.tool, "tool_definitions_strict", False); return await _check_tool_definitions(ctx, strict=strict)` — the actual logic lives in `_check_tool_definitions()` (not read in this pass, but referenced consistently across both `check_tool_definitions_runtime()` and `check_tool_definitions_startup()`, lines 368-376), which this requirement's migration must also relocate (or leave as a shared helper the discovery service calls, if `check_tool_definitions_runtime()` — a distinct, non-startup runtime check — needs to keep using it independently).
2. `check_routing_drift_vs_live()` (`repl_health.py:379-459`, confirmed by full direct read) **already implements everything this requirement asks the discovery service to own**: it fetches per-server tool lists (via `_collect_server_tool_names_per_server()`), builds a discovery map with duplicate detection (`build_discovery_map()` from `route_resolver.py`), validates against the registry (`validate_routing_against_live()`), and branches strict-mode-raises vs. non-strict-mode-warns for both drift and duplicates. This is not a small function to reference-and-wrap — its full body is the substance of what requirement 03's discovery service needs to do anyway (per that plan's own Design section on validation rules), just currently pointed at the static `ToolRegistry` (`get_registry()`, line 397/427) instead of building/validating against a fresh `RuntimeToolRegistry`.
3. `check_tool_definitions_runtime()` (`repl_health.py:368-370`) is a **separate, non-startup** runtime check (used, per its docstring, for warnings-only re-validation, likely from a `/mcp status` or periodic health-check path) — this requirement's scope is startup consolidation; `check_tool_definitions_runtime()`'s own call site(s) must be checked before deciding whether `_check_tool_definitions()` can be fully absorbed into the discovery service or must remain a shared, independently-callable helper.
4. `scripts/shared/tool_routing_validation.py`'s `validate_routing_against_live()` (confirmed via the requirement 03 investigation: takes pre-fetched `live_tool_lists`, delegates to `registry.validate_live_tools_match()`, returns mismatch messages) has no other caller besides `check_routing_drift_vs_live()` — confirm this via grep before assuming it can become a private implementation detail of the discovery service rather than a still-public shared function.
5. This requirement and requirement 03 are tightly coupled and should likely be implemented together (or requirement 03 first, then this one immediately after) — requirement 03's plan already anticipated this exact consolidation as a named follow-up in its own Risks section ("if unification... becomes desirable later, that removes the redundant round-trip").

## Design

### Migration is substitution, not re-invention

Per Assumption 2, `check_routing_drift_vs_live()`'s logic (fetch → build discovery map with duplicate detection → validate against registry → strict/warn branching) is structurally identical to what requirement 03's discovery service already needs to do to build `RuntimeToolRegistry` safely. Rather than writing new duplicate/drift-detection logic from scratch (as requirement 03's plan originally scoped, deferring unification), this requirement's implementation should **directly port `check_routing_drift_vs_live()`'s body** into the discovery service, adapting it to validate against the *newly-fetched* data it's simultaneously using to build `RuntimeToolRegistry`, rather than validating a separate fetch against the old static `ToolRegistry`.

### One fetch, one service

The consolidated flow: discovery service fetches each server's `/v1/tools` once → validates response shape (`_validate_tools_response()`'s logic) → builds `RuntimeTool` entries → detects duplicate names across servers in the same pass (no second `build_discovery_map()` call against a redundant fetch) → applies severity (production=fatal, local=warning, per requirement 03's own design) → returns both the built `RuntimeToolRegistry` and a validation report for the startup pipeline. This directly satisfies "duplicate MCP tool names are detected before registry construction completes" (this requirement's own acceptance criterion) since detection now happens *during* the same pass that builds the registry, not as a separate, later validation step.

### Preserving strict/production behavior

Per this requirement's "strict and production behavior are preserved" acceptance criterion: the exact same severity semantics from `check_routing_drift_vs_live()` (strict mode raises `RuntimeError`, non-strict logs warnings) must carry over — this maps directly onto requirement 03's own `security_profile`-gated fatal/warning distinction (production=fatal, local=warning), so both requirements' severity models should be reconciled into one consistent scheme during implementation, not left as two parallel severity mechanisms (one from "strict mode," one from "security profile") that could disagree.

## Risks

- **Risk**: This requirement directly supersedes a "keep independent for now" recommendation in requirement 03's own plan — if requirement 03 is implemented first without awareness of this requirement's consolidation ask, the discovery service could be built in a way that's harder to retrofit for consolidation (e.g. tightly coupled to a single-fetch assumption that doesn't leave room for the drift/duplicate logic to plug in cleanly).
  **Mitigation**: flag this dependency explicitly — ideally requirements 03 and 09 are implemented as one combined effort, or requirement 03's implementer is made aware of this requirement's follow-on scope before finalizing that service's internal structure.
- **Risk**: `check_tool_definitions_runtime()` (non-startup) could depend on the shared `_check_tool_definitions()` helper in a way that's disrupted if that helper is fully absorbed into the discovery-service's private implementation.
  **Mitigation**: Implementation Step 1's grep explicitly checks this before any function is moved or privatized.
- **Risk**: Reconciling two previously-independent severity mechanisms ("strict mode" flag vs. "security profile" enum) into one scheme could silently change behavior for an edge case where the two previously disagreed (e.g. `strict=True` in a `local` security profile, or vice versa) — a combination that was possible under the old, separate mechanisms but may not map cleanly onto a single unified scheme.
  **Mitigation**: Implementation Step 5 should explicitly enumerate the 4 combinations (strict×production, strict×local, non-strict×production, non-strict×local) and confirm each one's resulting behavior is deliberate, not accidental, before finalizing the unified scheme.
