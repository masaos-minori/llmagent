# Plan: Restrict legacy use_tool_dag=false in production

Source requirement: `requires/20260709_05_require.md`

## Scope note (read first)

This requirement bundles two sub-tasks. Research during planning found that
the first sub-task's premise is factually wrong (see Unknowns/UNK-01 below).
Per user decision (asked during planning), **this plan covers only the
second sub-task** (`use_tool_dag=false` production restriction). The first
sub-task (`tool_results_turn_max_chars` removal) is deferred to a GitHub
issue for human review — see `issues/` output from this plan's Step 5-equivalent
below — and is explicitly **not** implemented by this plan.

## Goal

Make `security_profile="production"` startup fail fast (`RuntimeError`,
via `ProductionConfigValidator`) when `use_tool_dag=false`, while local/
development keeps allowing it with a warning that it is legacy
compatibility mode.

## Scope

In scope:
- `shared/production_config_validator.py`: add `"use_tool_dag"` to the
  required-true keys already checked by `ProductionConfigValidator.validate()`
  (from `plans/done/20260709-130206_plan.md`, requirement 03).
- `scripts/agent/repl_health.py`: pass `use_tool_dag` into the existing
  `ProductionConfigValidator().validate()` call in `audit_security_defaults()`.
- `docs/05_agent_08_configuration.md`: describe `use_tool_dag=false` as
  legacy compatibility mode; document the production requirement.
- `docs/04_mcp_06_configuration_and_operations.md`: document resource-scoped
  DAG scheduling as the standard, and legacy round-wide serialization
  (`use_tool_dag=false`) as non-production.
- `tests/test_production_config_validator.py`: add `use_tool_dag` cases to
  the `validate()` tests.
- `tests/test_repl_health.py`: add production/local integration tests.

Out of scope (deferred — see Scope note above):
- Removing `tool_results_turn_max_chars` from `ToolConfig` / config
  builder / `config/agent.toml` / docs. **This field is not deprecated** —
  see Unknowns/UNK-01. No files related to this sub-task are touched by
  this plan.
- `tests/test_tool_scheduler.py` — the requirement asks to "use existing
  tests... for scheduler behavior tests," but this plan makes no scheduler
  changes (DAG scheduling logic itself is untouched, per the Non-Goal "do
  not rewrite the tool scheduler"); no edits needed, only a regression run
  in Validation plan.
- Removing `use_tool_dag` entirely — explicit Non-Goal ("do not remove
  use_tool_dag immediately unless explicitly approved").
- `resource_scope` / `requires_serial` semantics — explicit Non-Goal,
  untouched.

## Assumptions

1. `use_tool_dag` defaults to `True` (`config_dataclasses.py:221`,
   `config_builders.py:188`) and gates DAG-vs-legacy scheduling in
   `tool_runner.py:450` (`if ctx.cfg.tool.use_tool_dag and not
   ctx.cfg.tool.serial_tool_calls`) — verified by reading all three files.
   No existing check anywhere enforces `use_tool_dag=True` based on
   `security_profile`.
2. `ProductionConfigValidator` (from `plans/done/20260709-130206_plan.md`)
   is designed as a flat "these keys must all be true" checker
   (`_REQUIRED_STRICT_KEYS` tuple + `.get(key, False)` loop) — `use_tool_dag`
   fits this exact shape, so no new method is needed on the validator class
   (unlike requirement 04's unknown-tool-tiers check, which needed a
   different method because it isn't a simple boolean). Just add
   `"use_tool_dag"` to the tuple and pass its value into the same `cfg` dict
   the caller already builds.
3. Current documentation (`docs/05_agent_08_configuration.md:261,265-275`,
   `docs/04_mcp_06_configuration_and_operations.md:730-776`) already
   describes DAG scheduling mechanics (resource_scope, requires_serial
   defaults, round-wide serialization triggers) in detail, but does not
   currently state that `use_tool_dag=false` is legacy/non-production —
   this is a net-new addition, not a correction.

## Unknowns

| ID | Unknown | Resolution |
|---|---|---|
| UNK-01 | The requirement's Background claims `tool_results_turn_max_chars` "is deprecated and no longer enforced because the old `tool_results` table has been removed." Verified against implementation: `ctx.cfg.tool.tool_results_turn_max_chars` (`config_dataclasses.py:219`, default `50000`) is actively read in `tool_runner.py:181` and enforced by `_apply_turn_char_limit()` (`tool_runner.py:222-235`), which truncates cumulative per-turn tool-result text sent to the LLM. This has no relationship to the `tool_results` SQLite table removed in commit `e63b2628` ("refactor: remove deprecated tool_results table and clean up stale references") — that was a DB persistence table, unrelated to this in-memory turn-processing limit. It is also distinct from `tool_result_max_llm_chars` (`config_dataclasses.py:217`, default `8000`), which truncates a *single* tool result's text, not the cumulative per-turn total. `docs/05_agent_08_configuration.md:260`'s claim `"Deprecated — tool_results table removed; no longer enforced"` is itself the likely source of the requirement's incorrect premise. | **Resolved by user decision during planning**: defer this sub-task entirely; do not remove the field. Filed as a GitHub issue (see below) for a human to decide whether to (a) fix the stale doc claim only, keeping the field, or (b) something else — removing live LLM-context-protection functionality should not happen based on a doc error alone. |

No other blocking unknowns. UNK-01 is resolved (deferred, not blocking
this plan's reduced scope).

## Design

**`shared/production_config_validator.py`** — extend the existing tuple
(from requirement 03's plan):

```python
_REQUIRED_STRICT_KEYS: tuple[str, ...] = (
    "plugin_strict",
    "tool_definitions_strict",
    "routing_drift_strict",
    "use_tool_dag",
)
```

No other change to the class — `validate()`'s existing loop
(`if not cfg.get(key, False): errors.append(f"{key} must be true in
production")`) already handles the new key generically.

**`repl_health.py`** — extend the dict already being built for the
`ProductionConfigValidator().validate(...)` call (from requirement 03's
plan) with one more key:

```python
    prod_result = ProductionConfigValidator().validate(
        {
            "plugin_strict": ctx.cfg.tool.plugin_strict,
            "tool_definitions_strict": ctx.cfg.tool.tool_definitions_strict,
            "routing_drift_strict": ctx.cfg.tool.routing_drift_strict,
            "use_tool_dag": ctx.cfg.tool.use_tool_dag,
        }
    )
```

The existing raise-in-production / warn-in-local branching immediately
after this call (already designed in requirement 03's plan) requires no
further change — it already generically raises/warns based on
`prod_result.errors`, which will now include `"use_tool_dag must be true
in production"` when applicable.

**Docs**:

1. `docs/05_agent_08_configuration.md:261` — expand the `use_tool_dag` row:
   ```markdown
   | `use_tool_dag` | `True` | Dependency-aware scheduling: independent reads run concurrently; writes serialized per resource scope. `false` = legacy round-wide serialization compatibility mode — not permitted in production (enforced by `ProductionConfigValidator`; startup raises `RuntimeError`). Local/development allows it with a warning. |
   ```
2. `docs/04_mcp_06_configuration_and_operations.md` — add near the existing
   resource-scoped scheduling explanation (~line 730): a sentence stating
   resource-scoped DAG scheduling (`use_tool_dag=true`) is the standard/
   required behavior in production, and legacy round-wide serialization
   (`use_tool_dag=false`) is local/development-only.

## Risks

- **Risk**: Folding `use_tool_dag` into the same generic
  `_REQUIRED_STRICT_KEYS` tuple as `plugin_strict`/`tool_definitions_strict`/
  `routing_drift_strict` produces a generic error message
  (`"use_tool_dag must be true in production"`) rather than a message
  explaining *why* (legacy round-wide serialization risk) — less
  actionable than a bespoke message.
  → **Mitigation**: Acceptable trade-off for consistency and simplicity;
  the docs changes (Design, item 1) carry the "why," and the error message
  still names the exact key to fix.
- **Risk**: Deferring `tool_results_turn_max_chars` removal means this
  requirement is only partially closed; if the requirement's issue tracker
  auto-closes on "implementation complete," it would incorrectly mark the
  first sub-task done too.
  → **Mitigation**: The filed GitHub issue (`issues/20260709-130926_risks.md`)
  exists specifically so the first sub-task remains separately tracked and
  visible, not silently dropped.
