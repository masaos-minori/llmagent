## 1. Goal

Fix the `/health` endpoint in `scripts/mcp/mdq/server.py` to reflect the current production schema (`chunks`/`chunks_fts`/`chunks_a*` triggers), add `row_factory`, and align all health checks with acceptance criteria.

## 2. Scope

- **In-Scope**:
  - `scripts/mcp/mdq/server.py` — rewrite `health()` and `_check_stale_documents()` for new schema
  - `tests/test_mdq_health_stale.py` — update fixtures and queries from `sections` to `chunks`/`documents`
  - New test file `tests/test_mdq_health_endpoint.py` — full acceptance-criteria coverage
- **Out-of-Scope**:
  - DB schema changes (schema is already `chunks`/`chunks_fts`)
  - `MdqService` internals other than `_get_db_connection`
  - Other MCP servers

## 3. Assumptions

- The production DB schema is `chunks` + `chunks_fts` + `documents` + `index_state` (verified in `service.py:_init_db`).
- Triggers are `chunks_ai`, `chunks_ad`, `chunks_au` (verified in `service.py`).
- `_check_stale_documents` should query `documents` (mtime_ns / indexed_at) instead of `sections`.
- `sqlite3.connect()` in `health()` must set `row_factory = sqlite3.Row` before any `.fetchone()["col"]` access.
- HTTP 200 when ready=true; HTTP 503 when ready=false (MCP-08 guidance).
- `stub: true` must not appear in the response.

## 6. Implementation Steps

1. **Phase 1: Fix `health()` in `server.py`**
   - [ ] Add `conn.row_factory = sqlite3.Row` immediately after `sqlite3.connect(db_path)` in `health()`.
   - [ ] Replace table checks: `sections` → `chunks`, `sections_fts` → `chunks_fts`.
   - [ ] Replace trigger checks: `sections_ai/ad/au` → `chunks_ai/ad/au`.
   - [ ] Replace FTS5 probe query: `SELECT COUNT(*) FROM sections_fts WHERE sections_fts = 'delete' LIMIT 1` → `SELECT COUNT(*) FROM chunks_fts WHERE chunks_fts = 'delete' LIMIT 1`.
   - [ ] Replace stats queries: `sections` → `chunks` / `DISTINCT file_path` → `DISTINCT source_path`; add `doc_count` from `documents` table.
   - [ ] Replace `last_indexed` query: `MAX(file_mtime) FROM sections` → `MAX(indexed_at) FROM documents`.

2. **Phase 2: Fix `_check_stale_documents()`**

## 8. Risks & Mitigations

- **Risk**: `_check_stale_documents` schema mismatch silently swallows exceptions → `stale_document_count: None` → operators miss stale data. **Mitigation**: Log the exception before returning `None`; add a test that validates None is returned (not raised) on corrupt DB.
- **Risk**: HTTP 503 on `ready: false` is a behavior change — watchdog will now restart the server on health failure. **Mitigation**: This is correct per MCP-08; document the change in the commit message. Verify watchdog behavior in staging.
- **Risk**: `test_mdq_health_stale.py` legacy fixtures become dead code if not updated. **Mitigation**: Update fixtures in Phase 3; mark old schema tests as superseded with a comment.
