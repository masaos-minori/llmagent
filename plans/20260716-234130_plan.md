## Assumptions

1. `scripts/agent/workflow/models.py` defines `StageDefinition` and `RetryPolicy` as plain dataclasses (not Pydantic models) — to be confirmed by direct read in Step 3; the removal procedure (delete a field, delete its keyword argument at each construction site) is the same either way, but exact syntax depends on which it is.
2. `_search_docs_structured()` in `search.py` is a synchronous function called directly (not `await`ed) from the async `search_docs()`. Wrapping it in `asyncio.wait_for(asyncio.to_thread(...), timeout=...)` bounds the *caller's* wait but cannot forcibly abort the SQLite call running in the background thread if it truly hangs — Python has no safe preemptive cancellation for a blocking C-extension call. This is accepted as a known limitation of `search_timeout_sec` (documented in its Validation plan and Risks) rather than a blocker, since it still solves the stated goal (bound how long a caller waits) even if it doesn't kill a wedged query.
3. MDQ's `max_chunk_chars`/`max_file_chars` have no existing enforcement point to hook into (confirmed: `indexer.py`'s `_index_single_file` writes `section["content"]` verbatim with no length check, and there is no file-size gate before `parse_markdown` is called). The chosen enforcement design (skip oversized files with a warning log for `max_file_chars`; truncate oversized chunk content for `max_chunk_chars`) is a new, additive behavior — not a fix to an existing bug — and needs review during implementation for whether "skip" vs. "truncate-and-index-anyway" is the right choice for `max_file_chars` (flagged as Unknown #1 below since the requirement didn't specify this).
4. Removing `RAGConfig.web_search_max_results` does not remove `WebSearchConfig.default_max_results`/`max_results_limit` (the web-search server's own, separate config) — these survive and become the sole source of truth per the requirement's Option A wording ("consolidate on the web-search server's own config, remove the rest").
5. `SearchRequest`'s Pydantic `Field(DEFAULT_MAX_RESULTS, ge=1, le=MAX_RESULTS_LIMIT, ...)` bounds are evaluated once at class-definition (module-import) time. Sourcing them from `WebSearchConfig.load()` at import time (rather than per-request) is consistent with how `_cfg: WebSearchConfig = WebSearchConfig.load()` already works in `server.py` — config is fail-fast-loaded once at process startup, not hot-reloaded per request. If `web_search_mcp_server.toml` changes, the process must restart to pick it up (already true for `_cfg` today; not a new limitation this plan introduces).
6. `workflow_engine.py`'s `_run_execute_with_retry` retrying only `"execute"` today is a hardcoded stage-name check, not a structural constraint — `_run_stage` already takes an arbitrary `stage_id`, so generalizing the retry wrapper to wrap `plan`/`verify` too (gated on `stage_def.retryable`) is a mechanical refactor, not a redesign. Confirmed via the `run()` method's structure: it calls `_run_stage(task, "plan", plan_fn)` and `_run_stage(task, "verify", verify_fn)` directly today, un-wrapped by any retry logic — those two call sites are what change.
7. No config file other than `config/agent.toml` and `config/workflows/default.json` needs a schema-version bump or migration note for these removals, since none of the removed keys are referenced by any other `config/*.toml` file (confirmed during the originating audit — no cross-file key reuse of `healthcheck_mode`, `web_search_max_results`, `status`, `description`, or `backoff` was found outside their home files).

### 2. MDQ indexing knobs (Group 2 — wire up)

`indexer.py` currently hardcodes `path.rglob("*.md")` at 4 call sites (`_index_directory` line 138; `refresh_paths`'s force-mode directory branch line 218; incremental-mode directory branch line 256; deletion-detection scan line 282). Introduce one shared helper to avoid duplicating glob-matching logic 4 times:

```python
def _iter_indexable_files(service: MdqService, directory: Path) -> Iterator[Path]:
    """Yield files under *directory* matching service.include_globs, excluding service.exclude_globs."""
    seen: set[Path] = set()
    for pattern in service.include_globs:
        for candidate in sorted(directory.rglob(pattern)):
            if candidate in seen or not candidate.is_file():
                continue
            rel = candidate.relative_to(directory) if candidate.is_relative_to(directory) else candidate
            if any(rel.match(excl) or candidate.match(excl) for excl in service.exclude_globs):
                continue
            seen.add(candidate)
            yield candidate
```

Replace each of the 4 `rglob("*.md")` call sites with `_iter_indexable_files(service, path)` (or `dir_path` at the deletion-scan site — same signature). This changes the deletion-detection logic's `current_md_files` from `.md`-only to whatever `include_globs` says — correct, since a file that's no longer in-scope per `include_globs`/`exclude_globs` should be treated as "gone" the same way a deleted file is.

**`max_file_chars`** (skip-and-warn per UNK-01's resolution): in `_index_single_file`, after `path.stat().st_size` is available (line 85) but before writing chunks, check the file's character length (read `path.read_text()`'s length, or approximate via `st_size` if avoiding a double-read is preferred — use `st_size` as a fast pre-check since it's already available, and only fall back to counting actual characters if UTF-8 multi-byte chars make the byte-count check ambiguous near the boundary). If it exceeds `service.max_file_chars`, log a warning (`logger.warning("Skipping oversized file: %s (%d chars > max_file_chars=%d)", path, ..., service.max_file_chars)`) and `return` before parsing — mirrors the existing `if not sections: ... return` early-exit pattern already in this function.

**`max_chunk_chars`** (truncate): in the per-section loop (`for section in sections:`, line 91), when computing `char_count = len(section["content"])` (line 94), if it exceeds `service.max_chunk_chars`, truncate `section["content"]` to that length before it's inserted into the `chunks` table and before `content_hash` is computed from it (so the hash reflects what's actually stored) — insert the truncation *before* line 92's `content_hash = hashlib.sha256(section["content"].encode())...` so hash and stored content stay consistent.

**`max_snippet_chars`**: `search.py:114`'s `snippet=row["content"][:150]` → `snippet=row["content"][: service.max_snippet_chars]`.

**`search_timeout_sec`**: wrap `_search_docs_structured`'s invocation in `search_docs()` (the `async def` caller) with:
```python
try:
    result = await asyncio.wait_for(
        asyncio.to_thread(_search_docs_structured, service, req),
        timeout=service.search_timeout_sec,
    )
except asyncio.TimeoutError as exc:
    raise MdqConsistencyError(f"Search timed out after {service.search_timeout_sec}s") from exc
```
Per Assumption 2, this bounds the caller's wait but does not forcibly kill the background SQLite call if it's truly hung — document this caveat in the doc's Validation plan and in this plan's Risks.

### 3. Web-search config consolidation

In `scripts/mcp_servers/web_search/models.py`: keep `DEFAULT_MAX_RESULTS`/`MAX_RESULTS_LIMIT` as fallback literals (rename semantics stay the same), but have `SearchRequest` be constructed with `Field` bounds sourced from a module-level `_cfg = WebSearchConfig.load()` called once at import time (matching `server.py`'s existing `_cfg: WebSearchConfig = WebSearchConfig.load()` pattern) — i.e. `max_results: int = Field(_cfg.default_max_results, ge=1, le=_cfg.max_results_limit, ...)`. ### 4. Workflow `retryable`

Generalize `_run_execute_with_retry` into a stage-agnostic method:

```python
async def _run_stage_with_retry(
    self, task: TaskRecord, stage_id: str, fn: StageCallback
) -> None:
    stage_def = self._wdef.get_stage(stage_id)
    if stage_def is None or not stage_def.retryable:
        await self._run_stage(task, stage_id, fn)
        return
    policy = self._wdef.retry_policy
    attempt = 0
    while True:
        attempt += 1
        try:
            await self._run_stage(task, stage_id, fn, attempt=attempt)
            return
        except WorkflowTimeoutError:
            raise
        except Exception as exc:  # noqa: BLE001
            if attempt >= policy.max_attempts:
                raise WorkflowHaltError(f"{stage_id} stage halted after {attempt} attempts") from exc
            wait = policy.backoff_sec
            ... # same logging/span/sleep as today's _run_execute_with_retry
            await asyncio.sleep(wait)
```

In `run()`, replace all 3 direct/legacy calls (`_run_stage(task, "plan", plan_fn)`, `_run_execute_with_retry(task, execute_fn)`, `_run_stage(task, "verify", verify_fn)`) with `_run_stage_with_retry(task, "plan", plan_fn)`, `_run_stage_with_retry(task, "execute", execute_fn)`, `_run_stage_with_retry(task, "verify", verify_fn)`. Since `config/workflows/default.json` sets `retryable: true` only for `execute` today (verify at implementation time — Step 3), this preserves current behavior exactly (plan/verify still don't retry, since their `retryable` is presumably `false`) while making that behavior config-driven instead of hardcoded by stage name. Delete `_run_execute_with_retry` once its logic is folded into `_run_stage_with_retry`.

## Risks

### R-1: `search_timeout_sec`'s bound only limits the *caller's wait*, not the underlying SQLite call
**影響:** A truly wedged query (e.g. lock contention) keeps running in a background thread after the caller sees a timeout error — under repeated timeouts this could accumulate background threads/connections.
**緩和:** Document this explicitly in the MDQ doc and in the new test's docstring. `_search_docs_structured` already opens and closes its own connection in a `finally` block, so the thread will eventually exit on its own once the underlying SQLite call returns (it is not a permanent leak, just a delayed one) — acceptable given MDQ's existing `_index_lock`-based serialization already bounds concurrent write contention, and reads under WAL mode do not block on writes. If timeout-driven thread accumulation becomes an observed problem in production, a future requirement should add an explicit thread-pool cap; not needed for this plan's scope.

### R-2: Generalizing `_run_execute_with_retry` to `_run_stage_with_retry` changes `plan`/`verify` stage behavior if their `retryable` flag in `config/workflows/default.json` is not already `false`
**影響:** If `plan` or `verify` currently has `retryable: true` in the JSON (not yet confirmed — Implementation Step 5's second checklist item), this refactor would newly cause those stages to retry on failure, a behavior change beyond "wire up an unused field."
**緩和:** Implementation Step 5 explicitly gates on verifying the current JSON values before considering the refactor complete; if `plan`/`verify` are `true`, treat that as a separate decision point requiring explicit confirmation (not a silent behavior change) rather than assuming the removal-plan's Group-2 framing ("wire up, preserve current behavior") automatically holds.

### R-3: `mdq_mcp_server.toml`'s `include_globs`/`exclude_globs` semantics for `Path.match()` (used in the proposed `_iter_indexable_files` helper) differ subtly from shell-style globbing for patterns like `.git/**`
**影響:** `pathlib.Path.match()` treats `**` differently from `Path.rglob()`'s recursive-wildcard semantics; an incorrect exclude-check could let `.git/**`-matching files through, or exclude too much/too little.
**緩和:** Implementation Step 3 must include a concrete test with the exact default `exclude_globs` value (`[".git/**", "__pycache__/**"]`) against a realistic directory tree to empirically confirm exclusion works as intended before considering the feature done — do not trust the pseudocode in Design §2 as final; adjust the matching logic (e.g. switch to `fnmatch` against the relative POSIX path string if `Path.match()` proves unreliable for `**`) based on that test's outcome.

### R-4: `scripts/shared/mcp_config.py` is a widely-imported, high-centrality module (per its role building every server's transport config at startup) — removing `HealthcheckMode` risks a missed reference elsewhere
**影響:** A missed import or isinstance-check elsewhere referencing `HealthcheckMode` would cause an `ImportError` or `AttributeError` at startup, not caught until the agent process actually boots.
**緩和:** Implementation Step 2's `rg` sweep explicitly includes `HealthcheckMode` (not just `healthcheck_mode`) as a search term; Implementation Step 7's full `pytest -v` + `mypy scripts/` pass is the empirical backstop (a missed reference would surface as an import error or type error, not silently pass).
