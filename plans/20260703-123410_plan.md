# Plan: Add MCP documentation consistency checks for routing, audit, health, and transport invariants

## Goal

Extend `scripts/check_mcp_docs_consistency.py` with nine new stale-pattern detectors that prevent stale MCP documentation from being reintroduced across routing authority, audit log format, transport error semantics, startup mode, and watchdog behavior invariants.

---

## Scope

### In-Scope

- Add new check functions to `scripts/check_mcp_docs_consistency.py`
- Wire new checks into `main()` with `--skip` flag support
- Add unit tests to `tests/test_check_mcp_docs_consistency.py`
- Remove `"ondemand"` from `VALID_STARTUP_MODES` in `scripts/check_mcp_docs_consistency.py` (it was already removed as an active transport; the constant incorrectly still lists it)
- Narrow allowlist exceptions only where explicitly justified

### Out-of-Scope

- Changes to `scripts/checks/check_mcp_docs_consistency.py` (older, pre-`DocFile` version)
- Changes to documentation files themselves
- Changes to implementation code (`shared/`, `mcp/`, `agent/`)
- Changes to `scripts/checks/check_docs_consistency.py` (RAG-focused)

---

## Assumptions

1. `scripts/check_mcp_docs_consistency.py` (root-level `scripts/`) is the canonical target, not `scripts/checks/check_mcp_docs_consistency.py`. The test file `tests/test_check_mcp_docs_consistency.py` imports from the root-level file via `conftest.py` sys.path injection, and all 21 current tests pass against it.
2. The nine stale patterns in the requirement represent invariants that are no longer true (e.g., `ondemand` removed, `stdio` removed, `AUDIT session=` is now only one of two audit formats) — not patterns that are currently valid.
3. `"ondemand"` in `VALID_STARTUP_MODES` (line 32 of `scripts/check_mcp_docs_consistency.py`) is incorrect and should be removed, since `startup_mode = "ondemand"` is a stale value the requirement says should be flagged.
4. The check for `HttpTransport returns is_error=True for transport failures` should flag if docs still assert this behavior — confirming with existing docs that the behavior may have changed to a different error shape.
5. The `--skip` argument pattern in `main()` already exists with choices `["startup", "failopen", "routing", "active", "toolcount"]`; new checks should follow the same pattern with new skip tokens.
6. Pattern matching should be line-by-line (consistent with existing check functions), not multi-line regex.
7. Historical section markers (`HISTORICAL_MARKERS` in the old checker: `"legacy"`, `"historical"`, `"archive only"`, `"resolved"`, `"was:"`, `"removed"`) should exempt matching lines from new checks.
8. Allowlist exceptions (files or line contexts where a pattern is legitimately present) must be explicitly justified per the acceptance criteria.

## Implementation Steps

### Step 1: Remove `"ondemand"` from `VALID_STARTUP_MODES`

**File:** `scripts/check_mcp_docs_consistency.py` line 32

Change `VALID_STARTUP_MODES: set[str] = {"persistent", "ondemand", "subprocess"}` to `{"persistent", "subprocess"}`. This makes `check_startup_modes` flag any doc that still says `startup_mode = "ondemand"`.

**Completion criteria:** `uv run pytest tests/test_check_mcp_docs_consistency.py` still passes (the test at line 83 uses `"ondemand"` — that test must be updated to use `"subprocess"` or removed, since `ondemand` is now invalid).

---

### Step 2: Add `check_live_discovery_routing` — flag "live discovery overrides registry routing" stale wording

**File:** `scripts/check_mcp_docs_consistency.py`

Add a new check function that detects lines asserting that live discovery (Priority 1) overrides the registry for routing decisions when the implementation has changed to make the registry the primary source. Specifically, flag lines containing patterns like `"discovery.*overrides.*registry"` or `"discovery map.*wins"` outside of explicitly historical sections. Add corresponding `--skip discoveryrouting` support in `main()`.

**Allowlist exception:** Lines in `04_mcp_90_inconsistencies_and_known_issues.md` (the known-issues file) are exempt.

**Completion criteria:** New check function is callable, registered in `check_all`, and covered by at least two tests (pass case: no stale language; fail case: stale language triggers ERROR).

---

### Step 3: Add `check_routing_authority_v1tools` — flag `/v1/tools is routing authority`

**File:** `scripts/check_mcp_docs_consistency.py`

Add a check that flags lines asserting `/v1/tools` is the routing authority (not merely a drift validation source). Pattern: lines matching `"/v1/tools.*routing authority"` or `"/v1/tools.*single source"` where the claim is positive (not a negation). Complement the existing `check_routing_authority` which focuses on `ToolRegistry` wording.

**Allowlist exception:** Lines containing negation phrases (`"not.*routing authority"`, `"not.*source of truth"`) should be skipped.

**Completion criteria:** Function added, skip token `"v1toolsrouting"` added to `main()`, tests written.

---

### Step 4: Add `check_tool_names_routing_input` — flag `config tool_names is a routing input`

**File:** `scripts/check_mcp_docs_consistency.py`

Port the `check_tool_names_as_routing_input` logic from `scripts/checks/check_mcp_docs_consistency.py` to the newer checker using the `DocFile`/`Issue` dataclass pattern. Pattern: `TOOL_NAMES_ROUTING_PATTERN = r"(?:tool_names.*(routing\s+input|routing\s+drives?|routing\s+determines?)|(?:(?:routing\s+drives?|routing\s+determines?).*tool_names))"`. Exclude lines that contain `"not a routing input"` or `"not routing inputs"`.

**Allowlist exception:** `04_mcp_90_` and `04_mcp_00_` files are exempt.

**Completion criteria:** Function added, skip token `"toolnamesrouting"` added, tests written.

---

### Step 5: Add `check_audit_log_single_format` — flag "shared audit log uses only AUDIT key=value format"

**File:** `scripts/check_mcp_docs_consistency.py`

Add a check that flags lines asserting a single audit log format for `audit.log` without acknowledging the two-format reality (key-value for MCP servers, JSON-lines for agent-side). Pattern: lines matching `"audit.log.*key.value"` or `"AUDIT session=.*format"` used as the sole format description outside of code blocks and historical sections. Also flag `"AUDIT session="` appearing in prose (not in a fenced code block) as a format example without a JSON-lines caveat nearby.

**Note:** This is subtle — the pattern must not flag legitimate code block examples. Use a `in_fenced_block` tracker consistent with existing checks.

**Completion criteria:** Function added, skip token `"auditformat"` added, tests cover code-block exemption.

---

### Step 6: Add `check_transport_error_is_error` — flag stale `is_error=True` for transport failures

**File:** `scripts/check_mcp_docs_consistency.py`

Add a check that flags lines asserting `HttpTransport` returns `is_error=True` for transport failures if this behavior has been changed. Pattern: `r"HttpTransport.*is_error\s*=\s*True"` or `r"is_error=True.*transport"` in prose (not code blocks). Apply only outside known-issues and allowlisted files.

**Note:** If `is_error=True` is still the correct behavior after reviewing `shared/tool_executor.py`, this check should be scoped narrowly to future-guard against incorrect removal and should use WARNING severity instead of ERROR.

**Completion criteria:** Function added with appropriate severity, skip token `"transportiserror"`, tests written.

---

### Step 7: Add `check_stdio_active_transport` — flag stdio as active transport

**File:** `scripts/check_mcp_docs_consistency.py`

This is already partially covered by `check_active_stdio_refs` in `scripts/checks/check_mcp_docs_consistency.py` (old checker), but is not in the newer checker. Port and adapt: flag lines asserting `stdio` is an active transport for any MCP server, in files outside the stdio allowlist. Pattern: `r"(?:^|[^a-zA-Z])stdio(?:[^a-zA-Z]|$)"` and `r"StdioTransport"`. The allowlist from the old checker (`04_mcp_02`, `04_mcp_06`, `04_mcp_05`, `04_mcp_00`) applies.

**Completion criteria:** Function added, skip token `"stdiotransport"`, tests written (allowlist verified).

---

### Step 8: Add `check_watchdog_restarts_on_dependency_failure` — flag stale watchdog behavior

**File:** `scripts/check_mcp_docs_consistency.py`

Add a check that flags lines asserting the watchdog restarts servers on every dependency failure. The correct behavior is that the watchdog only checks HTTP status code — not body fields — so dependency failures (which may return HTTP 503) do trigger restart, but dependency issues that return HTTP 200 with degraded body fields do NOT trigger restart. Pattern: flag `"watchdog.*restart.*dependency"` or `"dependency.*failure.*watchdog.*restart"` without a qualifying caveat about HTTP status code being the sole trigger.

**Completion criteria:** Function added, skip token `"watchdogrestart"`, tests written.

---

### Step 9: Add `check_strict_validation_skips_unreachable` — flag stale strict validation behavior

**File:** `scripts/check_mcp_docs_consistency.py`

Add a check that flags lines asserting that `tool_definitions_strict` (or strict validation) skips all-unreachable servers. The correct behavior is fail-fast (`RuntimeError`). Pattern: `r"strict.*skip.*unreachable"` or `r"skip.*unreachable.*strict"`. Historical sections exempt.

**Completion criteria:** Function added, skip token `"strictskip"`, tests written.
