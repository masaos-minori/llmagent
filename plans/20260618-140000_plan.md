# Plan: Define strict tool-definition verification behavior clearly

## Goal

Document and clarify startup validation behavior for `tool_definitions_strict` mode when servers are unreachable, distinguishing between unreachable servers and tool mismatches.

## Scope

**In scope:**
- Startup verification logic in `scripts/agent/repl_health.py` (`_check_tool_definitions`, `_collect_server_tool_names`)
- `/v1/tools` comparison behavior
- Startup documentation
- `/mcp` diagnostics display

**Out of scope:**
- Changing the actual tool definition comparison logic (behavior is correct; needs documentation)
- Modifying the strict_mode enforcement (raises RuntimeError on mismatch — this is intentional)
- Changes to MCP server implementations
- Changes to `/v1/tools` endpoint behavior

## Assumptions

1. The current code already handles unreachable servers correctly by logging warnings and including them in the `unreachable` list, but the behavior is not documented clearly for operators.
2. When all servers are unreachable, validation is skipped with an info log — this is correct behavior but should be explicitly documented.
3. The distinction between "tool mismatch" (tools exist in config but not on servers, or vice versa) and "server unreachable" (cannot query /v1/tools) needs to be clearer in startup output.

## Implementation Steps

1. **Clarify error messages in `_check_tool_definitions`**
   - When strict_mode raises RuntimeError, format the message to clearly separate:
     - Tool mismatches (missing_in_server / extra_on_servers)
     - Unreachable servers
   - Example: "Strict mode: tool definition mismatch detected. Mismatches: missing_in_server=[foo]. Unreachable servers: [bar]."

2. **Add docstrings to `_collect_server_tool_names` and `_check_tool_definitions`**
   - Document the behavior for: one server unreachable, multiple servers unreachable, all servers unreachable, tool name mismatch
   - Include examples of expected log output for each case
   - Completion criterion: Docstrings describe all four scenarios with examples

3. **Document strict-mode behavior in ops docs**
   - Add a section in `04_mcp_06_configuration_and_operations.md` or create a dedicated startup validation doc
   - Document the three cases:
     - Partial unreachable: validation proceeds with reachable servers; unreachable servers logged as warnings
     - All unreachable: validation skipped; info log emitted
     - Mismatch: warnings (non-strict) or error (strict mode)

4. **Improve `/mcp` diagnostics display**
   - If `/mcp` endpoint shows tool definition status, include unreachable server count
   - Completion criterion: `/mcp` output includes unreachable server information
