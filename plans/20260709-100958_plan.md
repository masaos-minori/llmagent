# Plan: Keep production MCP auth validation startup-only

Source requirement: `requires/20260708_36_require.md` (M-4)

## Goal

Confirm production-mode `auth_token` validation stays startup-only and
unreachable from `/reload`, and document the startup/reload boundary
explicitly.

## Scope

In scope:
- `docs/04_mcp_05_security_and_safety_model.md` — add one clarifying
  sentence that `/reload` never re-runs this validation and never applies
  `auth_token` changes at runtime.
- Confirmation only (no code change) for
  `scripts/agent/repl_health.py::audit_security_defaults()` and
  `scripts/agent/startup.py`.

Out of scope:
- `scripts/agent/repl_health.py::audit_security_defaults()` itself — verified
  below it is called only from `startup.py`, never from any `/reload` path;
  no change needed to preserve "startup-only."
- `scripts/agent/services/config_reload.py` — the "does not apply auth_token
  changes" / "reports restart-required" behavior is already fully specified
  by the H-4 plan (`plans/20260709-095233_plan.md`); this requirement adds
  no new production behavior on top of that.
- New test coverage for `audit_security_defaults()` startup failure —
  already extensively covered by `tests/test_audit_security_failures.py`
  and `tests/test_repl_health.py` (11+ existing tests); adding more would be
  redundant.

## Assumptions

1. `audit_security_defaults(ctx, production_mode)`
   (`scripts/agent/repl_health.py:655-690`) is called from exactly one
   place, `scripts/agent/startup.py:190`, inside the startup sequence —
   verified by `grep -rn "audit_security_defaults" scripts/`. No `/reload`
   code path calls it, so it cannot be re-triggered or bypassed at runtime;
   this is already true today and unaffected by the H-1..H-9 batch.
2. `tests/test_audit_security_failures.py` (8 tests) and
   `tests/test_repl_health.py` (12+ tests in
   `TestAuditSecurityDefaultsProductionMode`) already cover: missing
   `auth_token` raising in production mode, warning-only in local mode, and
   several related fail-closed scenarios — verified by reading both files'
   test names. This satisfies the requirement's first test-plan item
   ("start with missing token and verify startup validation fails")
   without new tests.
3. The requirement's remaining test-plan items ("reload with changed token…
   verify restart-required… verify running token unchanged") are the exact
   scenario already specified in the H-4 plan
   (`plans/20260709-095233_plan.md` § Design,
   `test_auth_token_change_reports_restart_not_deferred`). No duplicate test
   is added here.
4. `docs/04_mcp_05_security_and_safety_model.md:179-186` already documents
   the production/local auth requirement and its enforcement point
   (`audit_security_defaults()`), but does not mention `/reload`'s
   relationship to this check — verified by reading that section. This is
   the one genuine documentation gap this requirement asks to fill.

## Design

Add to `docs/04_mcp_05_security_and_safety_model.md`, immediately after the
existing "Enforcement point" sentence (line 186):

```markdown
**Reload boundary:** `/reload` never re-runs this check and never applies
`auth_token` changes to a running MCP server — token changes are always
reported as restart-required (see
[Configuration: Hot-reload eligibility](05_agent_08_configuration.md#config-file-ownership-and-hot-reload-eligibility)).
Production auth validation only ever runs at startup; there is no runtime
path that can weaken or bypass it.
```

No production code changes — this requirement is fully satisfied by
existing code (Assumption 1) plus the H-4 plan's behavior (Assumption 3),
with one documentation addition closing the "explicitly startup-only" gap.

## Implementation steps

1. Confirm `audit_security_defaults()` is still called only from
   `startup.py` after the H-1..H-9 batch lands (no code touches
   `repl_health.py`'s security audit function in that batch).
2. Confirm the H-4 plan's `auth_token` behavior
   (`plans/20260709-095233_plan.md`) is implemented — restart-required
   reporting, no mutation.
3. Add the clarifying sentence to
   `docs/04_mcp_05_security_and_safety_model.md` per Design above.
4. Run the full validation sequence (`rules/toolchain.md`), including
   `uv run check-mcp-docs`.

## Risks

- **Risk**: This plan intentionally adds no new tests, relying entirely on
  existing coverage plus the H-4 plan's tests. If either of those plans is
  implemented incorrectly or skipped, this requirement's acceptance criteria
  would silently fail to be verified.
  → **Mitigation**: Implementation steps 1-2 explicitly require confirming
  both dependencies before considering this requirement done; Validation
  plan runs both the existing security tests and the H-4 reload tests
  together.
- **Risk**: The new documentation sentence references a doc anchor
  (`#config-file-ownership-and-hot-reload-eligibility`) that could drift if
  that section is renamed.
  → **Mitigation**: Low risk — same anchor is already used elsewhere in the
  docs set (`docs/05_agent_07_cli-and-commands.md:275`), so it is an
  established, presumably stable cross-reference.
