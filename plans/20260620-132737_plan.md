# Plan: Remove cache-key ambiguity between documentation and implementation (requirement 20260619_22)

## Goal

Eliminate ambiguity around cache key generation by renaming the unused MD5-based `tool_call_key` function to clarify its actual purpose (failed-call tracking, not caching), updating its docstring, and adding a regression test for the actual cache key behavior.

## Scope

**In scope:**
- Rename `tool_call_key()` → `tool_hash_key()` in `scripts/shared/tool_executor.py`
- Update docstring to clarify it is for failed-call tracking, NOT for caching
- Update all import sites: `scripts/agent/tool_loop_guard.py`, `scripts/agent/tool_runner.py`
- Update test imports: `tests/test_tool_loop_guard.py`, `tests/test_tool_executor_helpers.py`
- Add regression test for cache key format in `test_tool_executor_routing.py`

**Out of scope:**
- Changing cache eviction policy
- Redesigning the full caching subsystem
- Changing the MD5 hash strategy used by tool_loop_guard/tool_runner

## Assumptions

1. The cache key (`f"{tool_name}:{_json_dumps(args)}"`) is the canonical approach; documented correctly in both `docs/04_mcp_03_routing_lifecycle_and_execution.md:131` and `docs/05_agent_06_tool-execution-and-approval.md:136`
2. `tool_call_key()` is NOT used for caching — it's used exclusively for failed-call tracking in `tool_loop_guard.py:157` and `tool_runner.py:141`
3. The MD5 hash strategy for failed-call tracking is intentional (stable identity across LLM arg reordering); no behavioral change needed
