# Plan: Document the responsibility boundary between `SessionMessageRepository` and `SQLiteSessionStore`

## Goal

The responsibility boundary between `SessionMessageRepository` (business-logic layer) and `SQLiteSessionStore` (DB adapter layer) is explicitly documented in all relevant docs so future developers know where validation and encoding logic belongs.

## Scope

**In-Scope**:
- Add a `SessionMessageRepository vs SQLiteSessionStore` responsibility boundary section to `docs/90_shared_05_db_api_and_operations.md` §4
- Expand/standardize the existing boundary section in `docs/05_agent_09_data-layer.md` (currently Japanese-only, minimal)
- Optionally add a cross-reference entry to `docs/90_shared_90_inconsistencies_and_known_issues.md` if a DESIGN issue entry is warranted

**Out-of-Scope**:
- Code changes to `scripts/agent/session_message_repo.py` or `scripts/db/store_impl.py`
- Schema changes
- New tests

## Assumptions

- `SessionMessageRepository` (`scripts/agent/session_message_repo.py`) is the sole entry point for saving/fetching conversation messages in the agent layer; `SQLiteSessionStore.message_save` is not called directly outside of tests
- `SQLiteSessionStore` does NOT perform role validation, `content=None` normalization, or `tool_calls` JSON encoding; these are confirmed to live exclusively in `SessionMessageRepository`
- The existing Japanese section in `05_agent_09_data-layer.md` (lines 60-78) is accurate but incomplete; no contradictory information exists in other docs
