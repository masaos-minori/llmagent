# Plan: Add server-level monitoring for tool errors

## Goal

Distinguish transport-level errors from tool-level errors in monitoring, add per-server tool error counters, and improve operational visibility for repeated tool failures.

## Scope

**In scope:**
- MCP metrics collection (transport vs tool errors)
- Per-server tool error counters
- Audit aggregation for error reporting
- `scripts/shared/tool_executor.py` — transport layer
- `scripts/mcp/server.py` — MCP server base class

**Out of scope:**
- Adding new external monitoring systems (Prometheus, Datadog, etc.)
- Changing the HTTP response format (still 200 with is_error=true for tool errors)
- Modifying individual MCP server implementations
- Real-time alerting rules

## Assumptions

1. Tool-level errors currently return HTTP 200 with `is_error=true`, which makes them indistinguishable from transport failures in simple metrics.
2. The audit log already records per-server information, but error counters are not aggregated at the server level.
3. Current metrics collection does not separate transport errors (connection failures, timeouts) from tool errors (tool execution returned is_error=true).
4. Operators need to distinguish between "MCP server is down" and "this specific tool keeps failing."

## Implementation Steps

1. **Distinguish transport vs tool errors in metrics**
   - Add error type attribute to existing metrics/tracing:
     - `error_type=transport` for connection failures, timeouts, server unreachable
     - `error_type=tool` for tool execution returning is_error=true
   - Completion criterion: Error metrics include error_type discriminator

2. **Add per-server tool error counters**
   - Track `server_tool_errors_total` counter per MCP server name
   - Increment on each tool execution with is_error=true
   - Completion criterion: Per-server tool error count is queryable

3. **Improve audit aggregation for repeated failures**
   - Add detection of repeated tool failures (e.g., same tool failing 3+ times in N minutes)
   - Log warning when threshold exceeded
   - Completion criterion: Repeated failures trigger operator-visible warnings

4. **Document new monitoring fields**
   - Update ops docs to describe transport vs tool error distinction
   - Document per-server counter format
   - Completion criterion: Docs explain how to monitor and interpret new metrics
