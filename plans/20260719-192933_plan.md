# Plan: Validate WebSearchConfig and align search_web input schema

Source requirement: `requires/20260717_21_require.md` (moved to `requires/done/` in Step 7 of this cycle)

## Goal

Make `WebSearchConfig` fail fast on invalid configuration values, and make the `search_web` MCP tool's LLM-facing `inputSchema` in `tools.py` match the runtime `SearchRequest` constraints (query length, max_results bounds), with a normalized query policy and drift tests, so the LLM cannot construct tool calls that pass schema checks but fail runtime validation.

## Scope

**In-Scope**
- `WebSearchConfig.from_dict()` / `.load()` validation: `default_max_results >= 1`, `max_results_limit >= 1`, `default_max_results <= max_results_limit`, `max_results_limit <= HARD_MAX_RESULTS_LIMIT` (new constant, recommended value 100).
- `tools.py::TOOL_LIST` `search_web.inputSchema`: add `query.minLength=1`, `query.maxLength=500`, `max_results.minimum=1`, `max_results.maximum=<max_results_limit>`, sourced from shared constants/config rather than duplicated literals.
- Query normalization policy in `SearchRequest` (or a validator on it): trim leading/trailing whitespace, reject empty-after-trim, reject NUL, reject/normalize other control characters; ensure the normalized value is what flows to provider call, response, audit log, and metrics.
- New tests: `tests/test_web_search_models.py` additions for invalid-config cases; new `tests/test_web_search_tool_schema.py` for schema/model drift and normalization.
- Doc update: `docs/04_mcp_04_01_web-search-file-read-github.md` (the actual current file covering `search_web`; note the 2026-07-17 entry at line 48 already documents the config→Field wiring — extend/adjust it to reflect the new validation and normalization behavior once implemented).

**Out-of-Scope**
- Changes to the search provider itself (`search_provider.py` / DuckDuckGo integration logic).
- Changes to `server.py` audit/dispatch mechanics beyond passing through the already-normalized query.
- Any other MCP server's config or schema.
- Actually writing code — this cycle produces a plan only.

## Assumptions

1. `HARD_MAX_RESULTS_LIMIT = 100` is an acceptable safety ceiling (falsifiable: if product/ops wants a different cap, change one constant).
2. The single source of truth for the schema bounds should be constants exported from `models.py` and imported into `tools.py` (per requirement's "Notes for implementer" first option) rather than generating the schema from `SearchRequest.model_json_schema()`, because `tools.py` currently has zero imports from `models.py` and the MCP schema format (`inputSchema`) is not a literal Pydantic JSON schema dump — hand-written schema with shared constants is the lower-risk increment.
3. Control-character rejection will use a conservative rule: reject if the trimmed query contains any character with Unicode category `Cc` (control) — this covers NUL and other control chars uniformly rather than special-casing NUL separately.
4. `SearchRequest` field-level validation (min_length/max_length/ge/le) stays on the Pydantic model; normalization (trim, reject-on-empty-after-trim, reject-on-control-char) is added as a Pydantic `field_validator` on `query`, since Pydantic already runs before any handler code executes.

## Design

1. **`models.py`** (currently: `DEFAULT_MAX_RESULTS=5`, `MAX_RESULTS_LIMIT=20` at L32-33; `WebSearchConfig` dataclass L37-42; `from_dict()` L44-49 does unchecked `int(...)` conversion; `load()` L52-54 calls `ConfigLoader().load(...)` then `from_dict`; module-level singleton `_cfg = WebSearchConfig.load()` at L65; `SearchRequest(BaseModel)` L68-82 with `query: Field(..., min_length=1, max_length=500)` and `max_results: Field(_cfg.default_max_results, ge=1, le=_cfg.max_results_limit)`):
   - Add `HARD_MAX_RESULTS_LIMIT: int = 100` near the existing constants (L32-33).
   - In `from_dict()`, after building the int values, validate the four invariants from the requirement and raise `ValueError` with an actionable message (e.g. `f"max_results_limit ({v}) must be >= 1"`) instead of silently constructing an invalid instance. Keep `from_dict` raising synchronously so `load()` propagates fail-fast at process/import time (consistent with the existing "fail-fast" docstring on `load()`).
   - Add a Pydantic `field_validator("query", mode="before")` (or `mode="after"`) on `SearchRequest` implementing: strip, reject if empty after strip, reject if any `unicodedata.category(ch) == "Cc"` character remains. Because `_cfg` and `SearchRequest` are both defined in this same module, no new inter-module import is needed for this part.

2. **`tools.py`** (currently: `TOOL_LIST` L11-30; `search_web.inputSchema.properties.query` L20 has only `{"type": "string", "description": ...}`; `max_results` L21-24 has only `{"type": "integer", "description": ...}`; no imports from `models.py` today):
   - Import `DEFAULT_MAX_RESULTS`, `MAX_RESULTS_LIMIT` (or a getter reflecting the live `_cfg`, see Design note below) from `models.py`.
   - Design note / open tradeoff to record explicitly per Step 9 rule ("if two approaches are equally valid, state the tradeoff"): `tools.py` importing the *module-level* `_cfg` singleton value at import time captures whatever `max_results_limit` was in the TOML file at process startup — same lifecycle as `SearchRequest`'s own Field bound, so behavior stays consistent between schema and runtime model within a single process. Import `models._cfg.max_results_limit` (or expose a small `get_max_results_limit()` accessor in `models.py` for a cleaner public surface) rather than the static `MAX_RESULTS_LIMIT` constant, since the static constant does not reflect a TOML override.
   - Update `inputSchema.properties.query` to add `"minLength": 1, "maxLength": 500` and `max_results` to add `"minimum": 1, "maximum": <imported value>`.

3. **Tests**:
   - Extend `tests/test_web_search_models.py` with the invalid-config cases listed in the requirement's Test Plan section, plus normalization cases (whitespace-only, leading/trailing whitespace, NUL, newline/control chars).
   - Create `tests/test_web_search_tool_schema.py` that imports both `TOOL_LIST` and `SearchRequest`, and asserts numeric/string equality between the hand-written schema values and the live Pydantic field constraints (e.g. `SearchRequest.model_fields["query"].metadata` min/max, `SearchRequest.model_fields["max_results"]` ge/le) — this is the "schema drift test" required by the requirement's Acceptance Criteria.

4. **Docs**: extend `docs/04_mcp_04_01_web-search-file-read-github.md` line 39 schema table and the line-48 note to describe the new bounds and normalization rule, once implemented (per `rules/coding.md`'s "Accepted current specification" classification — this will be a plain-prose update, not an "Implementation fix required" issue, since it describes the new intended behavior, not a gap).

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Adding strict `from_dict()` validation breaks server startup if `config/web_search_mcp_server.toml` ever drifts from valid values (e.g. during a future manual edit) | Low | Current TOML values (5, 20) already satisfy all four invariants (verified above); add the invalid-config tests so any future regression is caught before deploy; `load()` already documents itself as fail-fast, so this is consistent with existing intent |
| Query normalization (trim/reject control chars) changes behavior for any existing caller relying on today's pass-through semantics | Low-Medium | No current normalization exists, so there is no "existing correct behavior" being broken for valid queries (trimmed non-empty printable strings are unaffected); add explicit before/after test cases for whitespace-only, leading/trailing whitespace, NUL, and control-character queries per the requirement's test plan to lock in the new contract |
| Sourcing `tools.py`'s schema `maximum` from the live `_cfg` singleton (Design note in step 2) ties the LLM-facing schema to process-startup-time config, so a config change requires a server restart to be reflected in `/v1/tools` — could confuse an operator who edits the TOML and expects immediate effect | Low | This matches the existing lifecycle of `SearchRequest`'s own Field bound (already startup-time-bound, per the models.py L61-64 comment), so no new inconsistency is introduced; document this explicitly in the doc update (Design section, item 4) so it is not mistaken for a bug |
| New `tests/test_web_search_tool_schema.py` could become stale if someone changes `SearchRequest` bounds without re-running tests locally | Low | Standard CI/pre-commit gate (`uv run pytest`) catches this on every change; no additional mitigation needed beyond existing CI discipline |

All identified risks have a stated mitigation and are assessed Low or Low-Medium likelihood; none are being deferred to an issue file.
