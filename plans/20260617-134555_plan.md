# Plan: [RAG][Search][Tests] Add Japanese FTS regression tests

## Goal

Create integration tests that verify Japanese FTS search quality — ensuring `normalized_content` is correctly indexed by the `chunks_ai` trigger and morphological variation queries return relevant results.

## Scope

**In scope:**
- New file: `tests/test_fts_japanese.py` (or add to existing test file)
- Read-only reference: `scripts/rag/repository.py` — `_build_fts_query()`, `_build_fts_tokens_ja()`, `fts_search()`
- Read-only reference: `scripts/db/schema_sql.py` — FTS5 schema + triggers (`chunks_ai`, `chunks_ad`, `chunks_au`)
- Read-only reference: `tests/test_document_repo.py` — testing patterns (in-memory SQLite, `_FakeSQLiteHelper`)

**Out of scope:**
- Fixing BUG-2 (`normalized_content=None` in ingester) — that is a separate task
- Changes to Sudachi tokenizer or FTS5 trigger definitions
- Tests for English FTS (covered by existing tests or out of scope)

## Assumptions

1. The `chunks_ai` trigger uses `COALESCE(new.normalized_content, new.content)` — when `normalized_content` is populated (Japanese chunks), FTS5 indexes the Sudachi-normalized form; when NULL (English/code), it falls back to raw content.
2. `_build_fts_tokens_ja()` extracts `normalized_form()` of nouns/verbs/adjectives (名詞/動詞/形容詞) from Japanese query text — this must match the tokenization used during indexing.
3. `_build_fts_query()` detects Japanese via regex `[぀-ヿ一-鿿]` and applies Sudachi tokenization; otherwise uses ASCII word extraction.
4. Tests will use in-memory SQLite with the RAG schema (pattern from test_document_repo.py).
5. `chunks_vec` is a stub table (vec0 not needed for FTS tests).

## Implementation Steps

1. **Create test schema fixture** (adapt from test_document_repo.py):
   - In-memory SQLite with full RAG schema (documents, chunks, chunks_fts, triggers)
   - `_FakeSQLiteHelper` for DB abstraction

2. **Test `chunks_ai` trigger — normalized_content indexing**:
   - `test_ja_normalized_content_indexed_in_fts` — insert chunk with `normalized_content`, verify FTS finds it via normalized tokens
   - `test_ja_raw_content_fallback_when_normalized_null` — insert English chunk (normalized=NULL), verify FTS uses raw content
   - `test_trigger_uses_coalesce_order` — verify COALESCE(normalized, content) behavior: normalized takes precedence

3. **Test `_build_fts_query()` Japanese tokenization**:
   - `test_build_fts_query_ja_extracts_nouns_verbs_adjectives` — verify POS filtering (particles excluded)
   - `test_build_fts_query_en_uses_ascii_extraction` — verify ASCII regex for English text
   - `test_build_fts_query_escapes_fts_metacharacters` — verify quote stripping

4. **Test FTS search quality with morphological variation**:
   - `test_ja_morphological_variant_returns_same_results` — query with inflected form finds document indexed with dictionary form
   - `test_ja_search_does_not_fall_back_to_raw_only` — verify normalized_content is actually used (not content) by testing specific token matches
   - `test_empty_japanese_query_returns_no_hits` — edge case: no valid tokens → empty FTS query → no results

5. **Test trigger lifecycle**:
   - `test_update_trigger_reindexes_normalized_content_change` — UPDATE normalized_content, verify FTS reflects new value
   - `test_delete_trigger_removes_from_fts` — DELETE chunk, verify it is no longer searchable

6. **Validation:** run `uv run pytest tests/test_fts_japanese.py -v` and confirm all tests pass.
