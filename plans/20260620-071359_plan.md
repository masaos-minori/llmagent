# Plan: Clarify Ingestion Artifact File Format (.txt = JSON) (req_38)

## 1. Goal

Eliminate operator confusion about `.txt` files containing JSON payloads in `rag-src/`
and `rag-src/chunk/` by making the format contract explicit in documentation and
module docstrings.

## 2. Scope

**In-Scope:**
- Add a clear format disclaimer to `docs/03_rag_02_ingestion_pipeline.md`:
  explain why `.txt` is used, what it actually contains, and how to read/validate the files
- Update module docstrings in `scripts/rag/ingestion/chunk_splitter.py`,
  `ingester.py`, and `crawler.py` to explicitly state `.txt = JSON payload`
- Add an operator warning: "Do not process `.txt` as plain text; always parse as JSON"
- Add a compatibility-preserving proposal for future extension rename
  (marked as: Needs confirmation)

**Out-of-Scope:**
- Changing file extensions now (compatibility risk; requires migration planning)
- Redesigning the ingestion artifact pipeline
- Breaking file-path compatibility without migration

## 3. Confirmed Facts

**File format (confirmed):**
- `rag-src/yyyymmddhhmmss-{slug}.txt` — crawler output: JSON payload
  (fields: `url`, `title`, `lang`, `fetched_at`, `content`, `code_blocks`, `etag`, `last_modified`)
- `rag-src/chunk/{stem}-{idx:04d}.txt` — chunk output: JSON payload
  (fields: `url`, `title`, `lang`, `source_file`, `chunk_index`, `chunk_type`,
  `content`, `normalized_content`, `etag`, `last_modified`, `fetched_at`)
- `rag-src/registered/{stem}-{idx:04d}.txt` — moved chunk files: same JSON format

**Hardcoded `.txt` references (confirmed — changing would break):**
- `ingester.py:105`: `self._chunk_dir.glob("*.txt")` — discovers all chunk files
- `chunk_splitter.py:217`: `f"{src_path.stem}-{idx:04d}.txt"` — writes chunk files
- `chunk_splitter.py:85`: `f"{src_path.stem}-0000.txt"` — sentinel file check
- `crawler_utils.py:X`: crawler output filename
- All `glob("*.txt")` and sentinel checks must match extension

**Documentation status (partial):**
- `docs/03_rag_02_ingestion_pipeline.md:64-66`: table shows "JSON" in rightmost column
- No explicit disclaimer that operators must not treat `.txt` as plain text
- No explanation of why `.txt` extension is used for JSON payloads
- Module docstrings say "rag-src/*.txt (JSON files)" in passing but not prominently

## 4. Assumptions

1. The `.txt` extension was chosen historically for simplicity (no special handling
   by OS file type associations) and to avoid per-platform issues with `.json` being
   associated with editors that might auto-format. This assumption is unconfirmed —
   mark as "historical reason unknown" in docs.
2. Any external tooling that processes `.txt` as plain text is not in the current
   scope — the operator guidance is the correct mitigation.
3. Renaming to `.json` is a valid future option but requires:
   - Updating all `glob("*.txt")`, sentinel checks, and filename generation
   - A migration script for existing `registered/` and `chunk/` files
   - Backwards-compatibility window or version bump
   This requires explicit confirmation before implementing.
