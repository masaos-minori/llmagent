# Plan: Define MCP tool capability naming convention

Source requirement: `requires/20260717_13_require.md`

## Goal

A standard, optional `domain.action`-style capability naming convention (e.g. `filesystem.read`, `git.write`, `github.issue.write`) is documented for future MCP tool schema adoption; the discovery service tolerates tools with no capabilities declared, since this requirement only defines the convention — it does not mandate applying it to any existing tool.

## Scope

**In scope**
- A new documentation file defining the naming convention: domain prefix, action suffix, read/write/delete/admin distinction, and support for multiple capabilities per tool — placed following this repo's existing doc-naming convention (see Assumption 1), not literally at the requirement's stated `docs/tool_capabilities.md` path.
- `scripts/shared/runtime_tool.py`: add an optional `capabilities: tuple[str, ...] = ()` field (or similar) to `RuntimeTool` (requirement 02's dataclass) so a tool *may* declare capabilities once the convention exists — this requirement defines the field's existence and naming rules; it does not populate real capability values on any actual MCP tool.
- `scripts/agent/services/mcp_tool_discovery.py`: parsing tolerates a missing `capabilities` field on any `/v1/tools` entry (default to empty), consistent with the "discovery service tolerates missing capabilities" acceptance criterion.
- MCP server `tools.py` files: **no mandatory change** — the requirement's own acceptance criteria say schemas "may adopt capabilities consistently," not "must adopt them now."

**Out of scope**
- Actually annotating any real MCP tool's `TOOL_LIST` entry with a capability string — this requirement defines the convention only; per-tool adoption is explicitly deferred (the requirement's own Reason section: "before being added broadly").
- Building any policy/grouping/discoverability feature that *consumes* capabilities (e.g. a `/mcp tools --capability filesystem.write` filter) — the requirement's own Reason section frames this as supporting *future* policy/grouping work, not building it now.
- Enforcing a fixed, closed vocabulary of domains/actions — the convention should be extensible (new domains/actions addable later), not a hardcoded enum.

## Design

### Convention structure

`{domain}.{action}` or `{domain}.{subdomain}.{action}` (per the GitHub example's 3-part form), all lowercase, dot-separated, no spaces/underscores within segments. Domains correspond to logical resource areas (`filesystem`, `git`, `github`, `process`, `search`), not necessarily 1:1 with MCP server names (e.g. `filesystem.*` could span `file_read`/`file_write`/`file_delete` servers). Actions follow a small, extensible vocabulary anchored on the read/write/delete/admin distinction the requirement explicitly asks for: `read`, `write`, `delete`, `execute` (for process/shell-like actions), and domain-specific verbs where needed (e.g. `github.issue.write` rather than a generic `github.write`, per the requirement's own example showing sub-domain granularity for GitHub specifically).

### Multiple capabilities per tool

A single `RuntimeTool` may declare more than one capability (e.g. a tool that both reads and triggers a side effect) — hence `capabilities: tuple[str, ...]`, not a single string field.

### Discovery tolerance

Per the acceptance criteria, a missing `capabilities` key on a `/v1/tools` entry defaults to an empty tuple — no validation error, no warning; this is a purely additive, optional metadata field with no enforcement behavior in this requirement's scope.

## Implementation steps

1. **Determine the doc's exact filename/number** (resolves the Unknown) by listing the current `04_mcp_*` sequence and picking a consistent slot.
2. **Write the convention document**: domain prefix rules, action suffix rules (including the read/write/delete/admin distinction), multi-capability-per-tool support, and the worked examples from the requirement (`filesystem.read`, `filesystem.write`, `filesystem.delete`, `git.read`, `git.write`, `github.issue.write`, `process.execute`, `search.web`).
3. **Add the `capabilities` field to `RuntimeTool`** (`shared/runtime_tool.py`, requirement 02's dataclass) as `capabilities: tuple[str, ...] = ()`.
4. **Update `mcp_tool_discovery.py`**'s response parsing to read an optional `capabilities` list per tool entry, defaulting to empty when absent.
5. **Verification**: no functional test suite changes needed beyond confirming the new optional field doesn't break existing `RuntimeTool`/discovery construction (a single test asserting a tool with no `capabilities` key parses with `capabilities=()`).

## Risks

- **Risk**: Defining a naming convention without any real tool adopting it risks the convention drifting from actual practice by the time real adoption happens (a common "designed in isolation" risk for conventions).
  **Mitigation**: keep the convention document explicitly marked as a proposal/standard to be validated against real adoption when requirement 13's follow-on work (actually annotating tools) happens — not claimed as already-proven-in-practice.
- **Risk**: This requirement's `capabilities` field addition to `RuntimeTool` is the second such addition in this batch (after requirement 07's `operation_type`) — if both are implemented independently without coordination, `RuntimeTool`'s final shape could end up inconsistent or reviewed twice for the same dataclass.
  **Mitigation**: flag both additions together when either requirement 07 or 13 is implemented, so `RuntimeTool`'s shape is finalized once with both fields considered together, not patched twice in uncoordinated passes.
