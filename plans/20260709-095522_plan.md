# Plan: Classify MCP server add, remove, and rename as restart-required

Source requirement: `requires/20260708_29_require.md` (H-7)

## Goal

Detect MCP server keys added or removed between the running config and a
reloaded config, report each as a single `needs_restart` entry
(`mcp/<key> (new server)` / `mcp/<key> (removed server)`), never mutate
`ctx.cfg.mcp.mcp_servers`, and never duplicate the same change under
`skipped`.

## Scope

In scope:
- `scripts/agent/services/config_reload.py` — extends the H-1 design
  (`plans/20260709-094640_plan.md`) with a removed-server detection loop;
  this was missing from the original H-1 design (which only iterated the new
  config's keys) and has been added there in place while planning this
  requirement.
- `tests/test_config_reload.py` — add/confirm add/remove/rename assertions.

Out of scope: no additional files.

## Assumptions

1. No code currently appends to `ConfigReloadOutcome.skipped` for MCP servers
   except the line being removed
   (`scripts/agent/services/config_reload.py:289`,
   `result.skipped.append(f"mcp/{key} (new server — restart required)")`) —
   verified by `grep -rn "\.skipped\b" scripts/ tests/`; no test asserts that
   specific string, so removing it is safe.
2. "Rename" has no dedicated representation in `McpServerConfig` — a renamed
   server is indistinguishable, from a dict-diff perspective, from one key
   being removed and a different key being added. The requirement confirms
   this is the intended treatment ("Treat server key rename as remove +
   add."), so no rename-specific detection code is needed.

None.

## Design

Extends the H-1 design (`plans/20260709-094640_plan.md` § Design) with a
second loop over keys present only in the old config:

```python
    for key in old_mcp:
        if key not in new_mcp:
            result.needs_restart.append(f"mcp/{key} (removed server)")
```

(Already incorporated into the canonical `_classify_mcp_server_changes()`
listing in the H-1 plan — this plan does not repeat the full function body.)

Rename handling: no special-cased logic. A rename from `old_key` to `new_key`
naturally produces `mcp/old_key (removed server)` (old_key absent from
`new_mcp`) and `mcp/new_key (new server)` (new_key absent from `old_mcp`) —
exactly the "remove + add" treatment the requirement specifies.

The removed `result.skipped.append(...)` line (old line 289) is what the
requirement's Background calls out as ambiguous ("reported as both skipped
and restart-required"); dropping it (already reflected in the H-1 plan)
resolves this requirement's "no duplicate skip/restart semantics" criterion
directly.

## Implementation steps

1. Confirm `_classify_mcp_server_changes()` in
   `scripts/agent/services/config_reload.py` includes both loops (new/changed
   keys, then removed keys) as specified in the H-1 plan's revised Design.
2. Add to `tests/test_config_reload.py`:
   - a test where `new_mcp` contains a key absent from `old_mcp` →
     `mcp/<key> (new server)` in `needs_restart`, nothing in `skipped`.
   - a test where `old_mcp` contains a key absent from `new_mcp` →
     `mcp/<key> (removed server)` in `needs_restart`.
   - a test simulating rename (old key removed, new key added in the same
     call) → both entries present, no cross-contamination (old server's
     other fields don't leak into the new key's report).
3. Run the full validation sequence (`rules/toolchain.md`) and fix any findings.

## Risks

- **Risk**: Iterating `old_mcp` for removed-key detection assumes
  `ctx.cfg.mcp.mcp_servers` is a plain `dict`; if it becomes some other
  mapping type without a stable `in`/iteration contract, this breaks.
  → **Mitigation**: Low likelihood — `McpConfig.mcp_servers` is typed as
  `dict[str, McpServerConfig]` throughout the codebase (see
  `config_dataclasses.py`, `config_builders.py`).
- **Risk**: A rename that also changes other fields (e.g. renamed AND URL
  changed) reports two independent entries (`removed` + `new server`) rather
  than a single "renamed and changed" entry, which could look like two
  unrelated events in `/reload` output.
  → **Mitigation**: Matches the requirement's explicit instruction to treat
  rename as remove + add; a combined single-entry representation was not
  requested and would require correlating unrelated dict keys by heuristic
  (e.g. same URL), which risks false-positive correlation.
