## Goal
MDQ サマリーキャッシュの書き込みパスを設計・実装し、`chunk_summaries` テーブルへのサマリー登録フロー（LLM 呼び出しまたは外部 API 経由）を定義する。

## Scope
**In-Scope**:
- `MdqService` にサマリー生成・キャッシュ書き込みメソッド（`_generate_and_cache_summary`）を追加
- `get_chunk` でサマリーが存在しない場合に生成を試みるロジックの追加
- 生成失敗時のフォールバック（生の truncated コンテンツを返す）の保証
- `summary_model = "default"` の場合のスタブ実装（外部 LLM 未統合）

**Out-of-Scope**:
- 実際の LLM API 呼び出し統合（`summary_model` が "default" のうちはスタブ）
- `chunk_summaries` テーブルスキーマの変更
- 既存のキャッシュ読み込みパスの変更

## Assumptions
1. `summary_cache_enabled = false` がデフォルトであり、有効化しない限り生成は行わない
2. `summary_model = "default"` の場合は生成をスキップしてフォールバック（外部統合の準備インタフェースのみ定義）
3. サマリー生成は非同期で行い、`get_chunk` のレスポンスをブロックしない設計
4. `chunk_summaries` テーブルへの INSERT は `content_hash` を伴う（既存スキーマそのまま）

## Unknowns
| Unknown | Evidence missing | Resolution | Blocking |
|---|---|---|---|
| `summary_model = "default"` 時のサマリー生成先 | 実際の LLM クライアント統合が未定 | スタブ（即時 return None）として実装し、将来の統合ポイントをコメントで明示 | False |
| サマリー生成の非同期化方針 | `get_chunk` が既存の同期 DB 呼び出しを含む | `asyncio.create_task` でバックグラウンド生成 or 同期生成後キャッシュ保存どちらか | False |

**分析結果**: `get_chunk` は `async def` 。生成は試みるが失敗してもフォールバックする形にする。

## Implementation Steps
1. **Phase 1: 書き込みメソッドの追加**
   - [ ] `MdqService._generate_and_cache_summary(conn, chunk_id, content, content_hash) -> str | None` を追加
   - [ ] `summary_model == "default"` の場合は `None` を返すスタブ
   - [ ] 生成成功時は `INSERT OR REPLACE INTO chunk_summaries` で保存
   - [ ] 例外は except して `None` を返す（フォールバック保証）

2. **Phase 2: get_chunk への統合**

   - [ ] 戻り値が `None` の場合は生の truncated コンテンツを返す（既存フォールバック）

3. **Phase 3: テスト追加**
