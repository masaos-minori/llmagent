# Plan: Strengthen MDQ read-time authorization, deny-all visibility, and config validation

Source requirement: `requires/20260717_26_require.md` (High priority)

## Goal

MDQ's runtime behavior matches its current `allowed_dirs` configuration at all times: `search_docs`, `get_chunk`, and `grep_docs` re-check authorization against the live config before returning any indexed content, operators can see deny-all state (`allowed_dirs = []`) in `/health`, and `MdqService` construction fails fast on invalid config values — with no database migration and no change to the existing fail-closed `authorize_path()` semantics.

## Scope

**In scope:**
- Read-time authorization enforcement in `search_docs`, `get_chunk`, `grep_docs` (filter/deny by current `allowed_dirs`, do not leak unauthorized metadata)
- `/health` detail fields `allowed_dirs_count` and `deny_all`, plus a startup warning log when `allowed_dirs` is empty
- A new `MdqConfig` pydantic model (or equivalent validator) enforcing types/ranges on all listed config fields, wired into `MdqService.__init__`
- New/extended tests for all of the above
- Doc updates to `docs/04_mcp_04_04_mdq.md` and `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md` reflecting the new enforcement (both currently state read-time tools do NOT authorize — this plan makes that statement stale and it must be corrected, not left as a silent inconsistency)

**Out of scope:**
- Database migration / deleting indexed rows for now-unauthorized paths (explicitly forbidden by the requirement)
- Wiring mdq-mcp into the global `audit_security_defaults()` startup audit (a separate, already-documented gap in `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md`; not requested here)
- Cleaning up the orphaned duplicate module set `scripts/mcp_servers/mdq/{models,service,server,tools}.py` (see Unknowns/Risks — real but unrelated pre-existing issue)
- Any change to `stats` (not listed in acceptance criteria; it returns aggregate counts only, no per-path content)
- Any change to `outline`/`index_paths`/`refresh_index` authorization (already enforced pre-change, no gap there)

## Assumptions

1. The requirement's target file list (`auth.py`, `search.py`, `service.py`, `db_grep.py`, `health_check.py`, `models.py`, `server.py`) uses pre-rename module names; the actually-live modules today are `mdq_service.py`, `mdq_models.py`, `mdq_server.py` (verified in Unknowns below) — this plan targets the live files.
2. `search_docs`'s existing size/char truncation logic in `search.py` runs on the already-filtered result set — filtering unauthorized rows happens before truncation math, otherwise reported "total" counts would include unauthorized (invisible) rows and read as inconsistent to the caller.
3. `get_chunk` must add `source_path` to its existing `SELECT heading, content, content_hash FROM chunks WHERE chunk_id = ?` query since it does not currently fetch it.
4. `grep_docs`'s existing `req_paths` path list (when supplied) should be authorized before being handed to `db_grep.grep_docs()`; when no `paths` filter is supplied, the query must add a `source_path IN (<authorized set>)` filter rather than post-filtering matches, per the requirement's "limit scanning to currently authorized source_path values" wording — however enumerating all indexed distinct `source_path` values and authorizing each is the only structural option available (no stored per-row authorization flag exists), so this is a scan-then-authorize-then-filter implementation even though the requirement frames it as "limit scanning".
5. `enable_grep` config gate check (already implemented, unrelated to this requirement) is preserved unchanged.
6. Health check's new `allowed_dirs_count`/`deny_all` fields are computed by reading `allowed_dirs` directly from `ConfigLoader` in `health_check.py` (matching its current pattern of reading `db_path` the same way), not by instantiating `MdqService` — avoids adding a service dependency to the standalone health function.
7. "Keep current valid defaults unchanged" means the new `MdqConfig` model's field defaults must match the literal defaults currently hardcoded in `MdqService.__init__` (e.g. `max_snippet_chars=500`, `search_timeout_sec=30`, etc.), verified against `config/mdq_mcp_server.toml`.

## Design

### 1. Read-time authorization (`search_docs`, `get_chunk`, `grep_docs`)

- `search.py::_search_docs_structured`: after building `fts_results`, filter out any `SearchResultItem` whose `source_path` fails `authorize_path(Path(row["source_path"]), service.allowed_dirs)`, before the `results = fts_results` assignment. Unauthorized rows are silently dropped (not counted in `total`), so search's existing truncation/"N found" messaging stays internally consistent and does not leak the existence/count of unauthorized matches.
- `mdq_service.py::get_chunk`: extend the `SELECT` to also fetch `source_path`; after `row is None` check, call `authorize_path(Path(row["source_path"]), self.allowed_dirs)`; raise `MdqAuthorizationError` (not `MdqNotFoundError`, to match the requirement's "raises `MdqAuthorizationError`" acceptance criterion) when unauthorized, without including the chunk's source path in the exception message (avoid leaking).
- `mdq_service.py::grep_docs`: two cases per requirement —
  - `req.paths` supplied: authorize each path via `authorize_path(Path(p), self.allowed_dirs)` up front; raise `MdqAuthorizationError` if any requested path is unauthorized (fail the whole call, do not silently drop — matches "rejects unauthorized paths" acceptance criterion, distinct from search's silent-drop because here the caller explicitly named the path).
  - `req.paths` empty/`None`: query `SELECT DISTINCT source_path FROM chunks`, filter to only `authorize_path()`-passing paths, and pass that resolved list into `db_grep.grep_docs()` as the `source_path IN (...)` filter (reusing the existing `req_paths` parameter, which already supports this via the `WHERE source_path IN (...)` clause) instead of leaving it empty (unfiltered).
- `stats` is unchanged — it returns aggregate counts, not path-identified content, and the requirement's acceptance criteria do not mention it.

### 2. Deny-all visibility

- `health_check.py::check_health()`: after loading `mdq_cfg`, read `allowed_dirs = mdq_cfg.get("allowed_dirs") or []`; add `details["allowed_dirs_count"] = len(allowed_dirs)` and `details["deny_all"] = len(allowed_dirs) == 0`. Do not add the raw `allowed_dirs` paths themselves to `details` (existing security policy per `04_mcp_05_05` — do not expose full allowed paths in health).
- Startup warning: `MdqService.__init__`, after validating `self._allowed_dirs`, add `if not self._allowed_dirs: logger.warning("MDQ allowed_dirs is empty — all path-based operations will be denied. Configure allowed_dirs in mdq_mcp_server.toml to enable indexing and path operations.")`. This is scoped to `MdqService` construction (module-local logger), not the global `audit_security_defaults()` (that remains out of scope, per Scope section).

### 3. Config validation (`MdqConfig`)

- New pydantic `BaseModel` `MdqConfig` in `mdq_models.py`:
  ```python
  class MdqConfig(BaseModel):
      allowed_dirs: list[str] = []
      include_globs: list[str] = ["*.md"]
      exclude_globs: list[str] = [".git/**", "__pycache__/**"]
      max_snippet_chars: int = Field(default=500, gt=0)
      max_chunk_chars: int = Field(default=10000, gt=0)
      max_file_chars: int = Field(default=100000, gt=0)
      search_timeout_sec: int = Field(default=30, gt=0)
      max_results_limit: int = Field(default=100, gt=0)
      max_chars_per_chunk: int = Field(default=10000, gt=0)
      max_total_result_chars: int = Field(default=100000, gt=0)
      max_outline_items: int = Field(default=500, gt=0)
      max_grep_matches: int = Field(default=200, gt=0)
      max_chars_per_match: int = Field(default=500, gt=0)
      context_before: int = Field(default=2, ge=0)
      context_after: int = Field(default=2, ge=0)
      enable_grep: bool = True
      max_outline_depth: int = Field(default=6, gt=0)
      sqlite_busy_timeout: int = Field(default=5000, gt=0)
  ```
  Pydantic's default strict-enough type coercion (with `model_config = ConfigDict(strict=False)` default) already rejects wrong container types for `list[str]` fields (e.g. a string or dict) and enforces `bool` for `enable_grep` — matches the acceptance criteria for list/bool validation without a custom validator, but add an explicit `field_validator` if plain pydantic v2 coercion (e.g. `"true"` string → bool) proves too permissive during test-writing (verify with a quick unit test before relying on default coercion).
- `MdqService.__init__`: replace the sequence of individual `mdq_cfg.get(...)` calls with `cfg = MdqConfig(**mdq_cfg)` (wrapped so a `pydantic.ValidationError` propagates as a clear startup failure — do not catch and swallow it, per "fail fast" requirement), then assign `self.db_path = db_path or mdq_cfg.get("db_path", ...)` (unchanged, `db_path` is not in the validated list) and `self._allowed_dirs = cfg.allowed_dirs`, etc., reading every other attribute off `cfg` instead of `mdq_cfg`.
- Remove the now-redundant manual `isinstance(self._allowed_dirs, list)` check (superseded by `MdqConfig`'s type validation, which raises rather than warning-and-defaulting — a deliberate behavior tightening the requirement asks for: "invalid config values fail during service construction").

## Risks

- **Risk:** Silently dropping unauthorized rows in `search_docs`/`grep_docs` (no-filter case) could make the tool's "total"/"found" counts appear to under-report when `allowed_dirs` is narrowed after indexing, confusing users who don't understand why previously-visible results vanished. — **Likelihood:** Medium — **Mitigation:** Keep this as intentional fail-closed behavior (matches requirement's "do not leak unauthorized metadata"); no count of hidden results is surfaced, by design. Covered by acceptance criteria; no further action needed.
- **Risk:** `grep_docs`'s no-filter path requires an extra `SELECT DISTINCT source_path FROM chunks` plus per-path `authorize_path()` calls before the main query — on a large index this adds O(distinct paths) authorization overhead per call. — **Likelihood:** Low (MDQ workloads are documented as "low-to-medium volume, thousands to tens of thousands of documents" per `docs/04_mcp_05_04_mdq-rag-boundary.md`) — **Mitigation:** Accept the cost; if it becomes measurable, cache the authorized-path set per request (not across requests, to avoid staleness) — no code change needed now, just noted for future profiling.
- **Risk:** Tightening `isinstance` warn-and-default to a hard pydantic `ValidationError` on `MdqService.__init__` is a behavior change: a previously-tolerated malformed `allowed_dirs` (e.g., a string instead of a list) now crashes server startup instead of degrading to deny-all. — **Likelihood:** Low — **Mitigation:** This is the explicit intent of the requirement ("invalid config should fail early"); document the behavior change in the doc updates (Phase 4) so operators upgrading are not surprised by a startup crash instead of a warning.
- **Risk:** Orphaned duplicate module set (`scripts/mcp_servers/mdq/{models,service,server,tools}.py`, byte-identical to the live `mdq_*.py` files except import paths, left behind by commit `ca6b7bfe`) is dead code that could be mistakenly edited by a future contributor searching for "server.py" instead of "mdq_server.py", silently producing changes that never run. — **Likelihood:** Medium (this exact confusion nearly affected this plan's own target-file list, which was written against the pre-rename names) — **Mitigation:** This plan targets the correct live files (documented in UNK-01); however, fixing the root cause (deleting or otherwise resolving the duplicate files) is out of scope for this security-focused requirement. **Filed as an unmitigated risk** — see `issues/20260719-205532_risks.md`.
