# Plan: Fix `config/agent.toml` `cmd` paths for 8 renamed MCP servers

Source issue: `issues/20260720-070458_mcp_server_agent_toml_cmd_points_to_deleted_files.md` (Critical —
discovered during plan review of `requires/20260717_21-31_require.md`, not from its own requirement
file).

## Goal

Every `[mcp_servers.<name>]` entry in `config/agent.toml` launches a `cmd` script path that actually
exists on disk, so all 8 MCP servers affected by the `ca6b7bfe`/`abcf0820` rename (`shell`, `git`,
`web_search`, `github`, `cicd`, `rag_pipeline`, `mdq`, `browser`) can start; a regression test locks
this invariant so a future rename/cleanup cannot silently reintroduce the same class of bug.

## Scope

**In scope:**
- `config/agent.toml` — update the `cmd` entry for 8 `[mcp_servers.<name>]` blocks (lines 293, 305,
  314, 358, 367, 376, 388, 397) from `.../<name>/server.py` to `.../<name>/<name>_server.py`.
- New test asserting, for every `[mcp_servers.*]` entry with `startup_mode = "subprocess"`, that the
  last element of `cmd` resolves to an existing file relative to the repo root — a regression guard
  for this exact bug class (a config `cmd` pointing at a script that does not exist).
- Verify no other `agent.toml` key references a stale bare filename (`models.py`/`tools.py`/
  `service.py`) — already checked in this planning cycle (see Assumptions), re-verify at
  implementation time as a final gate.
- Update `issues/20260720-070458_mcp_server_agent_toml_cmd_points_to_deleted_files.md`'s status to
  resolved once the fix lands and tests pass.

**Out of scope:**
- Any change to `scripts/mcp_servers/**/*.py` — the `<name>_server.py` files and their internal
  `app_module` self-references are already correct (verified for all 8 in this cycle); only the config
  file is wrong.
- `file_delete`/`file_write`/`file_read` MCP servers — different naming convention
  (`delete_server.py`/`write_server.py`/`read_server.py`), never touched by the `ca6b7bfe` rename, not
  affected.
- Extending `tools/check_mcp_docs_consistency.py` (`check-mcp-docs`) to also validate `cmd` path
  existence — a reasonable future enhancement (it already validates other doc/config consistency
  classes) but broader than this fix requires; a plain pytest test is the minimal correct fix (see
  Unknowns UNK-01).
- Restarting/redeploying any currently-running MCP server process — this plan only changes the repo's
  config file; actual redeployment requires the `deploy` skill and explicit user approval, per this
  session's operating mode (plan only, no implementation/deployment yet).
- Any change to `startup_mode`, `url`/port, `auth_token`, `role`, or `tool_names` for any of the 8
  entries — only the `cmd` script path is wrong.

## Assumptions

1. The correct replacement path for each server is `scripts/mcp_servers/<name>/<name>_server.py` —
   verified by `ls scripts/mcp_servers/<name>/*_server.py` returning exactly one file per subpackage
   for all 8 (`shell_server.py`, `git_server.py`, `web_search_server.py`, `github_server.py`,
   `cicd_server.py`, `rag_pipeline_server.py`, `mdq_server.py`, `browser_server.py`), and each file's
   own `app_module` class attribute already self-references the same `<name>_server` module path
   (e.g. `mcp_servers.web_search.web_search_server:app`), confirming these are the files each server
   class expects to be run as `__main__`.
2. No `agent.toml` key other than the 8 `cmd` entries references a stale bare filename — verified via
   `grep -n 'models\.py\|tools\.py\|service\.py' config/agent.toml` returning no matches.
3. All 8 affected entries use `startup_mode = "subprocess"` (confirmed by reading each block), so all
   8 are actually launched via `subprocess.Popen(cfg.cmd, ...)` in
   `scripts/agent/http_lifecycle.py::HttpLifecycleManager.start()` — this is not a latent/unused config
   path, it is the exact code path that fails today.
4. `McpServerConfig`'s existing validation (`scripts/shared/mcp_config.py`, `__post_init__`) only
   checks that `cmd` is non-empty for `startup_mode="subprocess"` — it does not check that the script
   file exists. This is why the bug was not caught by existing config-loading tests
   (`tests/test_mcp_config_validation.py`, `tests/test_config_loader.py`) and is the gap the new
   regression test closes.
5. `deploy/deploy.sh` copies `config/agent.toml` via a plain `cp` (not a per-key transform), so no
   `deploy.sh` change is needed — only the TOML content changes.
6. Changing a `cmd` string is a config-content edit, not a schema change — no `McpServerConfig`
   dataclass field is added/removed/renamed, so no migration or additional config-loader code change
   is needed beyond the new test.

## Design

1. **`config/agent.toml` fix** — for each of the 8 blocks, change only the last path segment of `cmd`:
   ```toml
   # before
   cmd = ["uv", "run", "--directory", "/opt/llm", "python", "/opt/llm/scripts/mcp_servers/web_search/server.py"]
   # after
   cmd = ["uv", "run", "--directory", "/opt/llm", "python", "/opt/llm/scripts/mcp_servers/web_search/web_search_server.py"]
   ```
   Applied identically (with the server-specific name) to `shell`, `git`, `web_search`, `github`,
   `cicd`, `rag_pipeline`, `mdq`, `browser`. No other field in any of these 8 blocks changes.
2. **Regression test** — parse the real `config/agent.toml` (via the existing `ConfigLoader`/
   `McpServerConfig` construction path used elsewhere in the test suite, not a hand-built dict), filter
   to entries with `startup_mode == StartupMode.SUBPROCESS`, and for each assert
   `Path("/opt/llm/scripts/...")`'s repo-relative equivalent (map the `/opt/llm/scripts/` prefix back
   to this repo's `scripts/` directory, matching how other config tests already resolve repo-relative
   paths — verify the exact helper/pattern at implementation time per UNK-01) exists on disk. This
   test would have failed against the pre-fix config and passes after, proving it is not vacuous.
3. **No code change** — `scripts/shared/mcp_config.py`, `scripts/agent/http_lifecycle.py`, and every
   `<name>_server.py` are already correct; this plan touches only the config file and adds test
   coverage.
