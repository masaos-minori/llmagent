# Plan: Add workflow-level observability

## Goal

Add workflow and task identifiers to all observability data (audit logs, OTel spans, diagnostics.jsonl) so that execution can be traced at workflow/task level, not only at turn level.

## Scope

**In scope:**
- Add `workflow_id` field to all audit event writers (`tool_audit.py`)
- Add `workflow_id`, `session_id` to event DTOs (`agent/shared/models.py`)
- Add `workflow_id` to `WorkflowState` context model (`agent/context.py`)
- Add `workflow_id` to `TaskRecord` and persist it via `StateStore` (`agent/workflow/models.py`, `agent/workflow/state_store.py`)
- Add `workflow_id` column to `tasks` table in workflow DB schema (`db/workflow_schema.py`)
- Add `workflow_id`, `task_id` attributes to OTel spans emitted by `WorkflowEngine` and `LLMTurnRunner`
- Add `workflow_id`, `session_id` to `_ContextFilter` context keys (`shared/logger.py`)
- Add `workflow_id`, `task_id` to session diagnostics output (`agent/repl.py`, `agent/diagnostic_store.py`)
- Add `RetryEvent` TypedDict and `artifact_uri` field to `ToolExecEvent`

**Out of scope:**
- New DB tables (retry_events table is deferred)
- MCP server-side audit log changes (`mcp/audit.py`) — deferred to separate task
- OTel exporter configuration changes
- UI/REPL commands for workflow inspection

## Assumptions

1. `workflow_id` is a session-level identifier that groups multiple tasks across turns (e.g., a user session may spawn one or more workflows). It is distinct from `task_id` which is per-task.
2. `session_id` is already available in all audit call sites via the existing context; it just needs to be passed through to event DTOs.
3. The workflow engine (`WorkflowEngine`) already has access to `task_id` and `workflow_version`; adding `workflow_id` requires passing it from the orchestrator during task creation.
4. OTel span attribute limits are not a constraint for adding 2-3 extra string attributes per span.
5. Diagnostics.jsonl schema changes are additive only (new fields with default `null`).

| Audit event DTO changes break backward compatibility for log consumers | med | Add new fields as required (workflow_id, session_id); consumers that ignore unknown fields are unaffected |

| _ContextFilter context key changes affect all structured logs (including non-observability) | med | workflow_id/task_id will be null when not set; null fields are safe in JSON output |
