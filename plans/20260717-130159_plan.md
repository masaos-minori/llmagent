# Plan: Use RuntimeToolRegistry for policy and approval classification

Source requirement: `requires/20260717_07_require.md`

## Goal

`classify_operation_type()`/`classify_risk()` (consumed by `RepositoryGateway` and `tool_approval.py`) derive operation type, safety tier, and approval requirement from `RuntimeToolRegistry` metadata instead of static `tool_constants.py` frozensets — `RepositoryGateway` remains the enforcement boundary unchanged, but its classification *inputs* switch source; unknown tools are classified fail-safe (high risk, approval required, LLM exposure disabled).

## Scope

**In scope**
- `scripts/agent/tool_policy.py`: `classify_operation_type()` (currently frozenset-membership checks against `WRITE_TOOLS`/`DELETE_TOOLS`/`_EXEC_TOOLS`/`_GITHUB_MUTATION_TOOLS`) and `classify_risk()` (currently `cfg.approval.tool_safety_tiers` → `tool_constants.py` fallback) both read from `RuntimeToolRegistry` as their primary source.
- `scripts/agent/tool_approval.py`: remove the direct `GITHUB_WRITE_TOOLS | GITHUB_DANGEROUS_TOOLS` import/`_GITHUB_MUTATION_TOOLS` construction (line 41) — this classification now flows through `RuntimeToolRegistry`-backed `classify_risk()`/`classify_operation_type()` instead of a second, independent static-set check.
- `scripts/agent/repository_gateway.py`: **no structural change** to `RepositoryGateway.execute()`'s enforcement flow (`classify_operation_type()` at line 74, `classify_risk()` at line 93) — it continues calling the same two functions; only their internal implementation changes.
- `scripts/shared/runtime_tool_registry.py`: **extend requirement 02's planned interface** with whatever additional field/method is needed to support the `UNKNOWN` classification and the WRITE/DELETE/EXECUTE/API_WRITE distinction this requirement asks for (see Design — this is a gap in requirement 02's original 13-field/9-method plan, not something already covered).
- `scripts/agent/tool_enums.py`: add `UNKNOWN` to `OperationType` (currently `WRITE`/`DELETE`/`EXECUTE`/`API_WRITE`/`READ` only, per direct read).
- `tests/test_tool_policy.py`, `tests/test_tool_approval.py`, `tests/test_repository_gateway.py` (real paths — requirement's stated `tests/agent/...` paths don't exist, per the pattern confirmed in requirements 04/05/06).

**Out of scope**
- Changing `RepositoryGateway`'s enforcement *logic* itself (the write/delete/API-write gating structure) — only its classification inputs change source, per the requirement's own "keep RepositoryGateway as the enforcement boundary" framing.
- Routing/scheduler/LLM-schema consumers — requirements 04-06 (separately tracked).
- Deleting `tool_constants.py` outright — requirement 10's scope; this requirement only stops it being the *primary* source for policy/approval, mirroring requirement 06's identical "primary vs. fallback" framing for side-effect detection.

## Design

### Resolving the `UNKNOWN`/DELETE/API_WRITE gap (Unknown #1)

Recommend option (a) from the Unknown's resolution path: add an `operation_type: OperationType` field directly to `RuntimeTool` (populated at discovery/annotation time, alongside `is_write`), rather than option (b) (keeping `tool_constants.py` sets as classification *input data*, which would perpetuate hand-maintained set membership as the true source of truth, just relocated). This keeps `RuntimeToolRegistry.classify_operation_type(tool_name)` as a simple field read (`self.get(tool_name).operation_type`), consistent with how `is_side_effect()` became a simple field read in requirement 06. Safe default for an unannotated tool: `OperationType.UNKNOWN` (not `READ`, since defaulting an unclassified tool to the *safest-looking* category would contradict this requirement's own "unknown tools classified conservatively" instruction) — `classify_risk()` must then map `UNKNOWN` to a high/dangerous tier explicitly (see below), not silently fall through a switch statement with no matching case.

### `classify_risk()`'s new fallback source

Priority order preserved (per Assumption 2): (1) `approval_risk_rules` explicit override unchanged; (2) `tool_safety_tiers` config unchanged; (3) fallback becomes `RuntimeTool.agent_safety_tier` (requirement 02's field) instead of `tool_constants.py` set membership. `OperationType.UNKNOWN` and an unannotated `agent_safety_tier` (defaulting to `"WRITE_DANGEROUS"` per requirement 02) both map to `RiskLevel.HIGH` via `_TIER_TO_RISK`, satisfying "high risk or dangerous safety tier" for unknown tools without new mapping logic — `_TIER_TO_RISK`'s existing `"WRITE_DANGEROUS"` → `MEDIUM` mapping (per the prior investigation) may need reconsidering here: **note the prior session's investigation found `WRITE_DANGEROUS` maps to `MEDIUM`, not `HIGH`**, in the existing `_TIER_TO_RISK` dict — this requirement's "high risk" ask for unknown tools may require either using `ADMIN`→`HIGH` as the unknown-tool default instead of `WRITE_DANGEROUS`→`MEDIUM`, or updating `_TIER_TO_RISK`'s mapping; flag this explicitly during implementation rather than assuming the existing dict already produces "high risk" for the unknown case.

## Risks

- **Risk**: Extending `RuntimeTool` with a new `operation_type` field (Unknown #1's recommended resolution) after requirement 02's plan already fixed its 13-field shape could require touching requirement 02's implementation if it has already landed by the time this requirement is implemented — a cross-plan dependency that must be tracked explicitly.
  **Mitigation**: flag this cross-dependency clearly when implementing; if requirement 02 is already implemented, this becomes a small additive field change to an existing dataclass, not a redesign — low risk in practice, but must not be silently skipped because "requirement 02 is done."
- **Risk**: `_TIER_TO_RISK`'s existing `WRITE_DANGEROUS` → `MEDIUM` mapping (not `HIGH`) could mean the "unknown tools = high risk" acceptance criterion silently fails even after this plan's other changes land, if the mapping itself isn't also corrected.
  **Mitigation**: Implementation Step 5 explicitly calls this out as a check-and-possibly-fix item, not an assumption that existing code already satisfies it.
- **Risk**: `RepositoryGateway`'s enforcement boundary (the actual acceptance criterion this requirement cares most about preserving) could regress silently if a `RuntimeToolRegistry`-sourced classification differs from the old static-set classification for even one tool during the transition.
  **Mitigation**: Implementation Step 8's full-suite run, plus a targeted comparison (for every currently-known tool, assert old-classification == new-classification) would catch this — consider adding such a comparison test explicitly if the full suite doesn't already provide this coverage.
