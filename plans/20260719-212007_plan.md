## Design

### 1. Serialization tests (`tests/test_mdq_index_serialization.py`)

Add two new test functions alongside the existing
`test_concurrent_refresh_calls_are_serialized`, reusing the same `service`/`md_file` fixtures and
the same `monkeypatch`-the-protected-section technique:

- `test_concurrent_index_paths_calls_are_serialized` — same shape as the existing test, but
  wraps `mcp_servers.mdq.mdq_service._index_paths` (imported into `mdq_service` as `_index_paths`
  from `indexer.py`, confirmed at `mdq_service.py` import block) instead of `_refresh_paths`, and
  calls `service.index_paths(IndexPathsRequest(paths=[...]))` twice via `asyncio.gather`.
- `test_index_paths_and_refresh_index_are_serialized` — one call to `index_paths`, one call to
  `refresh_index`, both racing through `asyncio.gather`; assert the recorded `order` list shows
  no interleaving (`["start", "end", "start", "end"]` regardless of which op runs first — assert
  on `order.count("start") == 2` and that "end" always immediately follows "start" at the same
  index, since the requirement is "operations do not interleave," not a fixed op order). Both the
  `index_paths`-side and `refresh_index`-side protected functions need to be monkeypatched
  (`_index_paths` and `_refresh_paths` respectively) so either ordering is observable.
- `test_is_indexing_warning_visible_during_active_index` (resolves UNK-02) — start a slow
  `index_paths` call as a background `asyncio.create_task`, yield with a short `asyncio.sleep(0)`
  or the existing monkeypatch hook to guarantee the lock is held, call `service.search_docs(...)`
  and assert the returned string contains the known warning substring
  (`"[WARNING: Index is being updated"` — exact text confirmed at `mdq_service.py`
  `search_docs()`), then `await` the background task to completion before the test ends.

## Risks

- **Risk:** `asyncio.gather(index_paths(...), index_paths(...))` ordering may be
  nondeterministic across Python/event-loop versions, making the new serialization tests flaky.
  **Likelihood:** low-medium. **Mitigation:** follow the existing test's proven pattern exactly
  (monkeypatch the protected section + `asyncio.sleep(0.05)` to force a scheduling yield inside
  the lock), which already produces a deterministic `["start","end","start","end"]` sequence in
  the current `test_concurrent_refresh_calls_are_serialized`; assert on non-interleaving
  structurally (paired start/end) rather than on which operation "wins" the race, so the test is
  order-agnostic and only fails on genuine interleaving.

- **Risk:** The new `_is_indexing` visibility test could be inherently timing-sensitive (needs
  `search_docs` to run *while* the lock is held). **Likelihood:** medium. **Mitigation:** use
  `asyncio.create_task` for the indexing call plus an explicit `await asyncio.sleep(0)` (single
  event-loop tick) before invoking `search_docs`, relying on the monkeypatched protected section
  still sleeping 0.05s (comfortably longer than one tick) to guarantee `_is_indexing` is still
  `True` when `search_docs` runs; always `await` the background task at the end of the test to
  avoid "Task was destroyed but it is pending" warnings turning into CI noise.

- **Risk:** The new agent/shared boundary scans could produce false negatives if a future
  contributor accesses MDQ's DB through a wrapper function that doesn't contain the literal
  string `"mdq.sqlite"` (e.g., builds the path via `os.path.join` from parts, or reads
  `mdq_mcp_server.toml`'s `db_path` key indirectly). **Likelihood:** low today (Assumption 4
  confirms no such access exists yet), but the detection method is inherently substring-based
  and not exhaustive. **Mitigation:** documented explicitly in the new tests' docstrings as
  "string/import-based static scan, not a runtime guarantee," matching the same limitation already
  accepted by the existing `rag.sqlite`/`mdq.sqlite` scans in this file; this is a known,
  pre-existing category of limitation for this whole test module, not a new gap introduced by
  this plan, so it does not need a separate issue filed — flagged here for visibility only.

- **Risk:** Requirement's "Target files" list names `tests/test_tool_server_layer_consistency.py`
  but Design/Scope decided not to add new tests there (UNK-01). If a reviewer disagrees, the
  acceptance criteria could be read as under-delivered. **Likelihood:** low. **Mitigation:**
  Scope section states the reasoning explicitly and Acceptance Criteria are otherwise fully
  covered (both consistency files already assert the write/serial flags this issue is protecting;
  the issue's own "Required changes" §1-2 only ask for the two files this plan targets). No
  further mitigation needed beyond this explicit call-out; not escalated to an issue file since
  it is a scope-interpretation judgment call, not an unmitigated technical risk.

All risks above have mitigations recorded in this plan; none require escalation to a separate
`issues/*_risks.md` file for this cycle.
