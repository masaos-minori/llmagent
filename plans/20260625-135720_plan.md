# Plan: [Medium][eventbus][health] /health エンドポイントの劣化状態レポート拡張

Source: `requires/20260625_19_require.md`

## Goal

`/health` エンドポイントを拡張し、DB クエリレイテンシ・SSE バックログ・DLQ ループ遅延を含む劣化状態インジケータを返すことで、オペレーターがサービスの完全停止と深刻なランタイム劣化を区別できるようにする。

## Scope

**In-Scope**
- `scripts/eventbus/app.py` の `health()` 関数の拡張
- DB クエリレイテンシ測定 (SELECT 1 の応答時間)
- DLQ ループの最終実行時刻追跡と遅延検出
- SSE バックログ (未配信イベント件数) の近似値
- 劣化状態の HTTP レスポンスセマンティクスの定義

**Out-of-Scope**
- Prometheus / OpenTelemetry メトリクスエクスポート
- DB スキーマ変更
- 他のエンドポイントの変更

## Assumptions

1. `health()` は同期 `SELECT 1` を実行しているが、拡張後もシンプルな実装を維持する
2. DLQ ループ遅延は `_dlq_task` の最終完了時刻をグローバル変数で追跡する
3. SSE バックログは直接追跡できないため、`events` テーブルの最新 seq と DB 全体のイベント数から近似する
4. 劣化状態でも HTTP 200 を返す (500 は完全停止用、200 は通常/劣化の区別は status フィールドで行う)

## Design

拡張後の `/health` レスポンス例:

```json
{
  "status": "degraded",
  "db": "ok",
  "db_latency_ms": 45.2,
  "dlq_task": "running",
  "dlq_last_run_age_s": 125.3,
  "event_count": 1523,
  "degraded_reasons": ["dlq_loop_delayed"]
}
```

劣化判定ロジック:
- `db_latency_ms > 100` → `db_slow`
- `dlq_last_run_age_s > 120` (DLQ_INTERVAL × 2) → `dlq_loop_delayed`
- `db` が "unavailable" または `dlq_task` が "stopped" → `status: "unavailable"`

`_dlq_last_run` グローバル変数を追加し、`_dlq_loop()` の各反復後に更新する。

```python
_dlq_last_run: float = 0.0  # time.monotonic()

async def _dlq_loop() -> None:
    global _dlq_last_run
    while True:
        ...
        _dlq_last_run = time.monotonic()
        await asyncio.sleep(_DLQ_INTERVAL)
```

## Implementation Steps

1. `_dlq_last_run` グローバル変数を追加し、`_dlq_loop()` で更新
2. `health()` に DB レイテンシ測定を追加 (`time.monotonic()` で計測)
3. `health()` に DLQ ループ遅延チェックを追加
4. `health()` に `event_count` (SELECT COUNT(*) FROM events) を追加
5. `degraded_reasons` リストのロジックを追加
6. `status` フィールドの3値 (ok / degraded / unavailable) を定義
7. 既存の health テスト (`tests/test_eventbus_health.py`) を更新
8. 劣化状態を再現するテストを追加

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| `health()` 内の `SELECT COUNT(*)` がパフォーマンスに影響する | Low | INDEX 済みの seq カラムで高速。問題があれば `SELECT MAX(seq)` に変更 |
| DLQ ループが一度も実行されていない起動直後に誤った degraded を返す | Medium | `_dlq_last_run == 0.0` の場合は "starting" として degraded 扱いしない |
| 後方互換: 既存クライアントが新フィールドで壊れる | Low | 新フィールドの追加のみで既存フィールドは維持 |
