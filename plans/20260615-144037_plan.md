## Goal

llmagent の agent ターン実行（plan / execute / verify / retry）をフォーマルなワークフロー定義に昇格し、タスク状態・試行履歴・処理済みイベント・アーティファクト参照を独立した `workflow.sqlite` で管理することで、冪等な実行と障害後の再開を保証する。

---

**In**:
- `config/workflows/default.json` — ワークフロー定義（ステージ・タイムアウト・リトライポリシー）

- `scripts/agent/orchestrator.py` — WorkflowEngine への最小接続（最小変更）

**Out**:
- 既存 `01_plan.md` / `02_design.md` / `03_implementation.md`（作業管理ツール）との統合
- UI / CLI 表示レイヤーの変更
- MCP サーバー群の変更
- `session.sqlite` / `rag.sqlite` のスキーマ変更

## Assumptions

1. `orchestrator/state/` は `scripts/agent/workflow/` として agent パッケージ内に配置する（import-linter の `root_packages = agent, db, mcp, rag, shared` に準拠）。
2. Metadata DB は `workflow.sqlite`（独立 DB）。`session.sqlite` / `rag.sqlite` とは別接続とし WAL モードを強制する。
3. 既存の `01_plan.md` / `02_design.md` / `03_implementation.md` は llmagent の作業管理ツールであり、要件の「plan/execute/verify/retry ステージ」とは無関係。
4. plan = RAG + MQE コンテキスト準備 / execute = LLM 呼び出し + ツール実行 / verify = ツール結果検証 + ループガード / retry = リトライポリシー適用、と `agent/orchestrator.py` の `handle_turn()` に対応づける。
5. `idempotency_key` = `session_id + turn_number` の組み合わせから生成する。
6. `workflows/default.json` は `config/workflows/default.json` に配置し、`ConfigLoader` 経由で読み込む。

## Design

### パッケージ配置

```
config/
  workflows/
    default.json          # ワークフロー定義

scripts/
  agent/
    workflow/             # 新規パッケージ
      __init__.py
      models.py           # TaskRecord / AttemptRecord / ArtifactRef / WorkflowDef dataclass
      workflow_loader.py  # config/workflows/*.json ロードと JSON Schema 検証
      state_store.py      # workflow.sqlite CRUD + 冪等性チェック
      workflow_engine.py  # ステージ遷移 / タイムアウト / リトライ制御
  db/
    workflow_schema.py    # Metadata DB DDL + 初期化
```

### ステージと `handle_turn()` の対応

| ステージ | `orchestrator.py` 対応箇所 | 内容 |
|---|---|---|
| plan | `_handle_turn_start()` + `_handle_memory_injection()` + `_handle_note_injection()` | ユーザー入力受付・RAG コンテキスト準備 |
| execute | `_handle_llm_turn()` → `LLMTurnRunner.run()` | LLM 呼び出し + ツール実行ループ |
| verify | `_handle_history_compression()` + `_handle_turn_end()` | 結果検証・ループガード確認 |
| retry | `_handle_llm_transport_error()` | トランスポートエラー時のリトライポリシー適用 |

### Metadata DB スキーマ（workflow.sqlite）

4 テーブル: `tasks`, `attempts`, `processed_events`, `artifacts`（要件定義書 Design セクション参照）。

加えて `tasks.workflow_version TEXT NOT NULL` フィールドを追加し、定義変更時の不整合を追跡する。

### 冪等性フロー

```
ステージ実行前:
  SELECT event_id FROM processed_events WHERE event_id = ?
    → 存在: 前回 artifact を返してスキップ
    → 不在: ステージ実行

ステージ完了後（同一トランザクション）:
  INSERT INTO processed_events (event_id, task_id, stage_id)
  INSERT INTO artifacts (artifact_id, task_id, stage_id, uri)
  → 失敗: ROLLBACK → attempts.status = 'failed' → リトライ判定
```

## Implementation Steps

推奨実施順（依存関係順）:

1. **`scripts/db/config.py` + `scripts/db/helper.py` + `scripts/db/workflow_schema.py` + `deploy/init_db.sh`**
   - `db/config.py`: `DbConfig` dataclass に `workflow_db_path: str = "/opt/llm/db/workflow.sqlite"` を追加
   - `db/helper.py`: `SQLiteHelper.__init__` の `target` 許容値に `"workflow"` を追加し、`workflow_db_path` を参照するよう `_resolve_path()` を更新
   - `db/workflow_schema.py`: tasks / attempts / processed_events / artifacts テーブルの DDL を実装（`tasks` テーブルには `workflow_version TEXT NOT NULL` フィールドを含める）
   - `deploy/init_db.sh`: `workflow.sqlite` 初期化ステップを追加
   - `deploy/deploy.sh`: `db/workflow_schema.py` と `agent/workflow/` の rsync を追加
   - 完了条件: `PYTHONPATH=scripts python -m db.workflow_schema` で DB 生成、`uv run mypy scripts/db/` が 0 errors

2. **`config/workflows/default.json`**
   - ステージ定義・タイムアウト・リトライポリシーを記述
   - 完了条件: `jsonschema` でバリデーション通過

3. **`scripts/agent/workflow/models.py`**
   - `TaskRecord`, `AttemptRecord`, `ArtifactRef`, `WorkflowDef` dataclass を定義
   - 完了条件: `uv run mypy scripts/agent/workflow/models.py` が 0 errors

4. **`scripts/agent/workflow/workflow_loader.py`**
   - `config/workflows/*.json` のロードと検証を実装
   - 完了条件: 不正 JSON でエラー、正常 JSON で `WorkflowDef` を返す

5. **`scripts/agent/workflow/state_store.py`**
   - `workflow.sqlite` への CRUD と冪等性チェックを実装
   - 完了条件: 単体テスト（重複 event_id でスキップ、新規 event_id で実行）pass

6. **`scripts/agent/workflow/workflow_engine.py`**
   - plan → execute → verify → (retry) ステージ遷移・タイムアウト・リトライを実装
   - 完了条件: 統合テストで失敗→リトライ→halt シナリオが pass

7. **`scripts/agent/orchestrator.py` への統合**
   - `WorkflowEngine` の参照を `handle_turn()` に追加（最小変更）
   - 完了条件: `uv run pytest` でリグレッションなし

8. **`PYTHONPATH=scripts uv run lint-imports` による境界確認**
   - `agent.workflow → db` の依存は既存コントラクト（`agent → rag, db, shared, mcp`）内で許容済みのため `.importlinter` 変更不要
   - 完了条件: `lint-imports` 0 violations（変更なし）

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| `agent/orchestrator.py` への統合で既存ターン処理が壊れる | 中 | WorkflowEngine は `handle_turn()` の外側でラップする形とし、既存ロジックを最小限に変更する |
