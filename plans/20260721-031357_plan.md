# Plan: Make MCP tool-discovery failure visible/fatal (sole-authority routing follow-up)

Source requirement: `requires/20260721_02_require.md` (moved to `requires/done/` in Step 7 of this
cycle). Formalizes the "Recommended follow-up" section of `issues/20260720-135023_risks.md`, which
was filed during the planning cycle for `requires/done/20260720_01_require.md` (routing
sole-authority change). Do not re-litigate that investigation here — this plan only designs the
fix.

## Goal

When `StartupOrchestrator._check_services()`'s MCP tool-discovery step (`scripts/agent/startup.py`,
step 4) hits its outer `except Exception` branch (i.e. `McpToolDiscoveryService(ctx).discover_all()`
itself raises, rather than completing with per-server findings), the failure must be surfaced with
severity that matches its actual impact — a full-session tool-call outage — instead of today's
silent `add_skipped(...)` outcome, which is both non-fatal and invisible in the CLI (the pipeline
display method never renders `SKIPPED` outcomes at all).

## Scope

**In scope:**
- The `except Exception as exc:` block wrapping step 4 (`mcp_tool_discovery`) in
  `StartupOrchestrator._check_services()` (`scripts/agent/startup.py`, currently lines 245-248).
- `StartupOrchestrator._display_pipeline_results()` (lines 289-297): add rendering for `SKIPPED`
  outcomes so a non-production skip is actually visible to the interactive user, not just to
  `logger.error` (which today only fires if some *other* step is also fatal).
- A new unit test in `tests/test_startup.py` covering this exact branch with
  `production_mode=True`, plus (if needed) confirming the existing non-production case's message
  wording/visibility.

**Out of scope (per requirement's explicit exclusions, confirmed against source):**
- `McpToolDiscoveryService` internals (`scripts/agent/services/mcp_tool_discovery.py`) — its
  per-server / per-finding fatal/warning classification is already correct and untouched.
- `ToolRouteResolver` / `ToolExecutor` sole-authority routing behavior (`scripts/shared/route_resolver.py`,
  `scripts/shared/tool_executor.py`).
- The per-outcome handling already inside the `try` block of step 4 (lines 229-244): the
  `for outcome in discovery.findings` / `for key in discovery.unreachable` / `add_ok` logic is
  unchanged.
- Steps 1, 2, 3, 5, 5b, 6 of `_check_services()` — not touched, though step 3's `except Exception`
  pattern (fatal unconditionally) and step 5/5b's pattern (warning unconditionally) are read as
  precedent, not modified.
- `scripts/agent/repl_health.py` — the requirement's option 2 (a distinct post-startup health-gate
  module) is not selected (see Design below); no new file/module is added there.

## Assumptions

1. `production_mode` (computed once at `_check_services()` line 192 from
   `ctx.cfg.mcp.security_profile == SecurityProfile.PRODUCTION`) is the correct signal to key the
   fatal/non-fatal branch on, since it is already the exact signal `_start_servers()` uses for its
   analogous fatal-on-subprocess-failure decision, and it is already in scope at the point in
   `_check_services()` where step 4 lives.
2. `pipeline.add_fatal(...)` is safe to call from step 4's except block without any other change,
   because `_check_services()` already aggregates all `add_fatal` calls across all 6 steps and
   raises once at the end (lines 281-287) — adding one more source of `add_fatal` requires no
   change to that aggregate-raise logic.
3. No caller of `_check_services()` distinguishes *which* step produced a fatal `RuntimeError` —
   confirmed by reading the aggregate raise: `raise RuntimeError(f"Startup validation failed:
   {fatal_str}")` joins all fatal messages generically. Adding a new fatal source does not require
   changing any caller.
4. `OutputTag.SKIPPED = "[skipped]"` (already defined in `scripts/agent/output_tags.py:27`) is
   available and unused elsewhere for this exact purpose — using it for the new `SKIPPED` rendering
   branch in `_display_pipeline_results()` introduces no new dependency.
5. Existing tests that assert `SKIPPED` status for this branch (e.g.
   `test_routing_drift_live_skipped_on_exception` in `tests/test_startup.py`, lines 802-816, which
   runs with implicit `production_mode=False`) are expected to keep asserting `SKIPPED` after this
   change, since that test does not set `production_mode=True` — only new production-profile test(s)
   are added, not a change to that existing test's expected status.

## Design

### Change 1 — step 4 except block (`scripts/agent/startup.py`, lines 245-248)

Current:
```python
        except Exception as exc:  # noqa: BLE001
            pipeline.add_skipped(
                "mcp_tool_discovery", f"MCP tool discovery skipped: {exc}"
            )
```

Proposed:
```python
        except Exception as exc:  # noqa: BLE001
            msg = f"MCP tool discovery failed - ALL tool calls will fail this session: {exc}"
            if production_mode:
                pipeline.add_fatal(
                    "mcp_tool_discovery",
                    msg,
                    remediation="Investigate MCP server connectivity/discovery failure before restarting.",
                )
            else:
                pipeline.add_skipped("mcp_tool_discovery", msg)
```
This reuses `production_mode`, already computed at line 192 and already in scope for the whole
method (it is passed into steps 1 and 3 today). No new imports are needed: `SecurityProfile` is
already imported for the `production_mode` computation itself; `pipeline.add_fatal` /
`pipeline.add_skipped` are already used elsewhere in this exact method.

Because `_check_services()`'s existing bottom-of-method aggregate check
(`if pipeline.has_fatal: ... raise RuntimeError(...)`, lines 281-287) already fires generically
across all sources, no change is needed there — the new `add_fatal` call is picked up automatically.

### Change 2 — `_display_pipeline_results()` (lines 289-297)

Current:
```python
    def _display_pipeline_results(self, pipeline: StartupValidationResult) -> None:
        """Display startup validation warnings and fatal errors via the CLI view."""
        for outcome in pipeline.outcomes:
            if outcome.status == StartupCheckStatus.WARNING:
                self._view.write_warning(f"{OutputTag.NON_FATAL} {outcome.message}")
            elif outcome.status == StartupCheckStatus.FATAL:
                self._view.write_warning(f"{OutputTag.FATAL} {outcome.message}")
                if outcome.remediation:
                    self._view.write_warning(f"  Remediation: {outcome.remediation}")
```

Proposed addition (new `elif` branch, using the already-defined but previously-unused
`OutputTag.SKIPPED`):
```python
            elif outcome.status == StartupCheckStatus.SKIPPED:
                self._view.write_warning(f"{OutputTag.SKIPPED} {outcome.message}")
```
This makes the non-production "all tool calls will fail" message actually reach the interactive
CLI user, closing the gap the Step-3 research flagged (today `SKIPPED` outcomes are silently
dropped by this method and only reach `logger.error` if some unrelated step is also fatal).

Note: this branch renders **all** `SKIPPED` outcomes project-wide (there is one other producer:
`rag_consistency`'s except block, line 277), not just `mcp_tool_discovery`'s. This is judged
acceptable and arguably a secondary improvement in the same spirit (a `RagMaintenanceService()`
crash being silently invisible has the same visibility gap) but should be flagged in the PR
description as a slightly broader blast radius than the requirement's literal ask, in case the
project prefers a `mcp_tool_discovery`-only conditional instead of a blanket `SKIPPED` branch. Both
are one-line-different implementations; recommend the blanket branch as the simpler, more general
fix consistent with why `OutputTag.SKIPPED` exists as a tag in the first place.

### Why not option 2 (a `repl_health.py` post-startup health gate)

Rejected as unnecessary additional surface area: the existing `pipeline`/`StartupValidationResult`
mechanism in `startup.py` already provides exactly the fatal/warning/skipped vocabulary and the
single aggregate raise point this requirement needs. Introducing a second, parallel health-check
module (`agent/repl_health.py`) that separately asserts `ctx.services_required.runtime_tools` is
non-empty would duplicate logic already available in-line at the discovery step, and would run
after `_check_services()` already returned — i.e., strictly later than necessary, when the fatal
condition is already known at the point of the exception. Combination note: if a future requirement
wants defense-in-depth (in case some other future code path bypasses `_check_services()` entirely),
a `repl_health.py` assertion could be added later without conflicting with this change.
