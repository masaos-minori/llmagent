# Plan: Freshness Detection for Local `file://` Ingestion

## 1. Goal

Detect `file://` file changes automatically using mtime + SHA-256 so local file
updates are ingested without requiring `--force`, and unchanged files are skipped safely.

## 2. Scope

**In-Scope:**
- `crawler.py` `crawl_local_file()`: compute `mtime` (ISO string) and `sha256` hash before crawling;
  store both in the crawl payload JSON under `file_mtime` and `file_sha256` fields
- `ingester.py` `_register_url()`: for `file://` URLs, compare stored `last_modified` (mtime)
  and `etag` (sha256) to incoming values before deciding to skip or re-ingest
  — If same mtime AND same sha256: skip (unchanged)
  — If different mtime OR different sha256: auto-force (re-ingest)
- Log the decision: "skipped (unchanged)", "re-ingesting (mtime changed)", "re-ingesting (content changed)"
- Preserve existing `--force` behavior (always re-ingest regardless of freshness)

**Out-of-Scope:**
- Remote HTTP freshness changes
- New schema columns — reuse `last_modified` for mtime and `etag` for SHA-256

## 3. Assumptions

1. `etag` column in `documents` table is safe to use as SHA-256 storage for `file://` URLs
   because HTTP ETags are not meaningful for local files.
2. `last_modified` column holds the mtime as ISO string (confirmed: ingester uses it for HTTP too).
3. Reading file content for SHA-256 adds latency but is necessary for reliable freshness;
   mtime-only is used as a fast-path check (if mtime unchanged → assume unchanged, skip SHA-256).
4. `_register_url()` has access to the ingested `etag` and `last_modified` values from the
   chunk payload (confirmed: line 152 reads `etag` from `first_data.get("etag")`).

## 6. Implementation Steps

1. **Add mtime + SHA-256 to crawl payload in `crawler.py`**
   ```python
   import hashlib, os
   stat = path.stat()
   mtime_iso = datetime.fromtimestamp(stat.st_mtime).isoformat(timespec="seconds")
   sha256 = hashlib.sha256(content.encode() if isinstance(content, str) else content).hexdigest()
   payload.update({
       "etag": sha256,          # reuse etag field for content hash
       "last_modified": mtime_iso,
   })
   ```

2. **Add `_is_file_unchanged()` to `ingester.py`**
   ```python
   def _is_file_unchanged(
       self, existing_etag: str | None, existing_last_modified: str | None,
       new_etag: str | None, new_last_modified: str | None
   ) -> bool:
       """Return True when both mtime and SHA-256 are identical."""
       if existing_etag is None or new_etag is None:
           return False  # no hash stored — can't determine
       if existing_etag != new_etag:
           return False  # content hash differs — changed
       return True  # sha256 match → unchanged (mtime is informational)
   ```

3. **Update `_register_url()` freshness check for `file://` URLs**
   ```python
   if not force and url.startswith("file://") and existing_doc_id is not None:
       row = db.fetchone("SELECT etag, last_modified FROM documents WHERE doc_id=?", (existing_doc_id,))
       if self._is_file_unchanged(row["etag"], row["last_modified"], etag, last_modified):
           logger.info("file:// unchanged (sha256 match): %s", url)
           return existing_doc_id  # skip
       logger.info("file:// changed — auto re-ingesting: %s", url)
       force = True  # fall through to re-ingest
   ```

4. **Tests**
   - Test: identical file → `_is_file_unchanged()` returns True → skipped
   - Test: changed content (different SHA-256) → returns False → re-ingested
   - Test: mtime changed but SHA-256 same → returns True → skipped
   - Test: `--force` still re-ingests regardless of freshness

5. **Update docs**
   - Add "Local file freshness detection" section to RAG ingestion doc

## 8. Risks

- **Risk:** Computing SHA-256 of large files (e.g. 50MB PDFs) adds significant ingestion latency
  → **Mitigation:** use mtime as fast-path: if mtime unchanged → skip SHA-256 entirely → return unchanged
- **Risk:** Using `etag` column for SHA-256 overloads its HTTP ETag semantics for file:// URLs
  → **Mitigation:** this is safe — HTTP ETags are never set for file:// URLs; document explicitly
- **Risk:** `_register_url()` currently reads `etag` from the chunk payload, not from the DB.
  Freshness check requires reading stored `etag` from the DB — add a `fetchone` query
  → **Mitigation:** 1 extra query per URL; acceptable for ingestion batch context
