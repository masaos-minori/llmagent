# Plan: Remove backward-compatible DB migration documentation and replace with DB recreation policy

**Source requirement:** `requires/20260702_42_require.md`

---

## 1. Goal

- Remove all documentation that implies automatic schema migration, `ALTER TABLE` support, or backward-compatible column additions.
- Replace with an explicit DB recreation policy: when the schema changes, recreate DB files from the latest DDL in `schema_sql.py`.
- Align `create_schema.py` docstring and `schema_sql.py` DDL with the no-migration policy.
- Document `schema_sql.py` as the canonical schema source.

---

## 2. Scope

- **In-Scope**:
  - Remove or rewrite references to incremental migration, `ALTER TABLE`, duplicate column suppression, and backward-compatible schema additions in:
    - `docs/90_shared_04_db_architecture_and_schema.md`
    - `docs/90_shared_05_db_api_and_operations.md`
    - `docs/90_shared_00_document-guide.md`
    - `scripts/db/create_schema.py` (module docstring and `_migrate_*` private functions)
  - Add DB recreation policy text and operational guidance (archive + recreate steps).
  - Clarify that `schema_sql.py` is the canonical latest schema definition.
  - Sync `schema_sql.py` DDL to include columns currently added only via `_migrate_*` (`chunk_type`, `source_file`, `undone`) so that a freshly recreated DB has the full schema without migration helpers.

- **Out-of-Scope**:
  - Implementing migration tools.
  - Preserving old DB compatibility with pre-migration-applied DBs.
  - Changing runtime behavior of `SQLiteHelper`, store protocols, or maintenance functions.
  - Event Bus runtime (publisher/subscriber/dispatcher/DLQ worker).

---

## 3. Assumptions

- All `_migrate_*` functions in `create_schema.py` have already been applied to the production DB:
  - `chunks.chunk_type TEXT` — present (confirmed via `PRAGMA table_info`)
  - `chunks.source_file TEXT` — present (confirmed)
  - `tool_results.undone INTEGER NOT NULL DEFAULT 0` — present (confirmed)
  - `tasks.workflow_id TEXT` — present (confirmed)
  - `tool_results` FK to `sessions(session_id) ON DELETE CASCADE` — present (confirmed)
  - `documents.chunking_strategy TEXT NOT NULL DEFAULT 'text'` — present in both DDL and production DB
- The production DB recreation procedure will be: stop service, archive existing DB files, run `create_schema()` against new empty paths.
- No other code in `scripts/` calls `_migrate_*` functions directly (they are private).
- `schema_sql.py` DDL is currently missing `chunk_type`, `source_file`, and `undone` columns (they are only added via migration helpers). These must be added to the DDL templates to make `schema_sql.py` the true canonical source.

## 8. Risks & Mitigations

- **Risk**: Removing `_migrate_*` functions from `create_schema.py` breaks a production DB that has NOT had migrations applied → **Mitigation**: Confirmed via `PRAGMA table_info` that all migrate-added columns exist in production DB. Safe to remove.
- **Risk**: `schema_sql.py` DDL does not include `chunk_type`/`source_file` → fresh DB creation would omit these columns → **Mitigation**: Phase 2 adds these columns to `_RAG_SCHEMA_TEMPLATE` before any code is touched in `create_schema.py`.
- **Risk**: Tests in `test_create_schema.py` assert that `_migrate_*` functions exist or are callable → **Mitigation**: Update tests in Phase 7; confirm with `grep -n "_migrate_" tests/test_create_schema.py` first.
- **Risk**: Documentation consumers (AI agents reading these docs) may cache the old migration-enabled description → **Mitigation**: Docs are updated in Phases 4–6 with explicit "Schema migration is not supported" statements.
