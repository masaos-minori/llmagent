# Plan: Restrict DLQ requeue to events currently in the DLQ

## Goal
`POST /dlq/{event_id}/requeue` returns 409 Conflict for non-DLQ events and 404 for unknown events, ensuring `dlq_requeue_count` increments only when `dlq_at IS NOT NULL`.

## Scope
**In-Scope**:
- `scripts/eventbus/db.py` — `requeue_event()` must enforce `dlq_at IS NOT NULL`
- `scripts/eventbus/app.py` — `dlq_requeue` handler must return 409 vs 404 correctly
- `tests/test_eventbus_dlq.py` — add `test_requeue_non_dlq_event_fails`, `test_requeue_unknown_event_returns_404`
- `tests/test_eventbus_concurrent.py` — concurrent requeue test must assert exactly 1 success + 4 conflicts (409)
- `docs/06_eventbus_02_http_api_and_runtime.md` — document 409 response for non-DLQ requeue

**Out-of-Scope**:
- DB schema changes (no migration needed)
- Other endpoints
- `delivery_failure_count` reset behavior (already preserved as-is)

## Assumptions
- SQLite-level `rowcount` on `UPDATE ... WHERE dlq_at IS NOT NULL` provides sufficient atomicity under the existing `_db_lock` to prevent concurrent double-requeue.
- HTTP 409 Conflict is the preferred response (over 404) for "event exists but not in DLQ", per the requirement.
