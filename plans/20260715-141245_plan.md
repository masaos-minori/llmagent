# Work Plan: Unify GitHub policy and approval behavior

Source requirement: `requires/20260715_21_require.md`
Generated: 2026-07-15 14:12:45

## Goal

Unify GitHub mutation-tool classification across `agent/tool_policy.py` and
`agent/tool_approval.py` on the shared `shared/tool_constants.py` sets, make
`gitops_push_blocked` scope explicit and tested, and make dry-run failure
handling explicit/tested for high-risk GitHub mutation tools — while
confirming the repo-allowlist comparison (found already exact-match) stays
correct and regression-covered.

## Scope

**In scope**
- `scripts/agent/tool_policy.py`: replace local `_API_WRITE_TOOLS` with the
  shared union `GITHUB_WRITE_TOOLS | GITHUB_DANGEROUS_TOOLS`.
- `scripts/agent/tool_approval.py`: replace local `_GITHUB_WRITE_TOOLS` with a
  shared-set-derived, explicitly documented constant for the
  `gitops_push_blocked` guard; add high-risk dry-run-failure blocking.
- `scripts/agent/tool_exceptions.py`: add a distinct exception (or attribute)
  so "dry-run returned an error" can be handled differently from "dry-run
  unsupported / connection failure".
- `tests/test_tool_policy.py`, `tests/test_tool_policy_comprehensive.py`:
  verify classification still matches after switching to shared constants.
- `tests/test_tool_approval_repos.py`: add `gitops_push_blocked` coverage for
  the shared mutation set and for `github_create_issue` /
  `github_add_issue_comment` parity.
- `tests/test_tool_approval_preflight.py`: add high-risk dry-run-failure
  blocking tests.
- `docs/05_agent_06_01_tool-execution-and-approval-execution.md`: document
  the unified classification, `gitops_push_blocked` scope, and dry-run
  failure semantics.

**Out of scope**
- GitHub server-side dry-run implementation (already implemented in
  `scripts/mcp_servers/github/service_dispatch.py`; not part of this task).
- Relaxing repository allowlist semantics.
- Changing GitHub MCP tool APIs.
- Any code change to `check_allowed_repo()`'s comparison logic — verified
  already correct (see Assumptions/Unknowns).

## Design

1. **`scripts/agent/tool_policy.py`** — import
   `GITHUB_DANGEROUS_TOOLS, GITHUB_WRITE_TOOLS` from `shared.tool_constants`
   alongside the existing `DELETE_TOOLS, SHELL_TOOLS, WRITE_TOOLS` import;
   replace the module-level `_API_WRITE_TOOLS` frozenset with
   `_GITHUB_MUTATION_TOOLS: frozenset[str] = GITHUB_WRITE_TOOLS | GITHUB_DANGEROUS_TOOLS`
   and update all three call sites (`classify_operation_type`,
   `check_allowed_repo`) to reference it. No other logic changes — the
   union is numerically identical to today's local set.

2. **`scripts/agent/tool_approval.py`** — import the same shared constants
   and define:
   ```python
   from shared.tool_constants import GITHUB_DANGEROUS_TOOLS, GITHUB_WRITE_TOOLS

   _GITHUB_MUTATION_TOOLS: frozenset[str] = GITHUB_WRITE_TOOLS | GITHUB_DANGEROUS_TOOLS
   # gitops_push_blocked guards repository-content and PR mutations.
   # Issue-tracker mutations are intentionally excluded — see
   # docs/05_agent_06_01_tool-execution-and-approval-execution.md.
   _GITOPS_BLOCKABLE_TOOLS: frozenset[str] = _GITHUB_MUTATION_TOOLS - {
       "github_create_issue",
       "github_add_issue_comment",
   }
   ```
   Replace `tool_name in _GITHUB_WRITE_TOOLS` in the `gitops_push_blocked`
   guard (currently line 117) with `tool_name in _GITOPS_BLOCKABLE_TOOLS`.

3. **High-risk dry-run failure blocking** — in `_build_preview_with_dry_run`,
   when `result.is_error` is True, raise `ApprovalPreviewBlockingError`
   instead of the generic `ApprovalPreviewError` (keep the RuntimeError/OSError
   branch raising the existing `ApprovalPreviewError` — that path represents
   "dry-run unsupported/unreachable", not an explicit error result). In
   `check_approval`, compute `risk` before the preview call (already the
   case), then:
   - if `risk == RiskLevel.HIGH` and `ApprovalPreviewBlockingError` is
     raised → deny immediately (`audit_approval(..., "denied_dry_run_error")`,
     `emit_denied(...)`), do not prompt.
   - otherwise (MEDIUM risk, or `ApprovalPreviewError` for
     unsupported/connection failure) → keep current fallback-to-text-preview
     behavior unchanged.

4. **Tests** — add:
   - `classify_operation_type("github_create_issue")` and
     `("github_add_issue_comment")` return `OperationType.API_WRITE` in both
     `tool_policy` classification and are covered by `check_allowed_repo`.
   - `gitops_push_blocked=True` blocks `github_push_files`,
     `github_create_or_update_file`, `github_delete_file`,
     `github_merge_pull_request`, `github_create_branch`,
     `github_create_pull_request`, `github_update_pull_request`; does
     **not** block `github_create_issue` / `github_add_issue_comment`
     (documents the intentional split from Assumption 4/UNK-03).
   - dry-run returns `is_error=True` for a HIGH-risk tool (e.g.
     `github_delete_file`) → `check_approval` returns `False` without
     prompting, audited as `denied_dry_run_error`.
   - dry-run raises `RuntimeError` (unsupported/connection failure) for a
     HIGH-risk tool → `check_approval` still falls back to text preview and
     prompts (unchanged behavior — regression guard for
     `test_dry_run_exception_does_not_abort_approval`).

5. **Docs** — add a short subsection to
   `docs/05_agent_06_01_tool-execution-and-approval-execution.md` describing:
   the single shared GitHub mutation set now used by both policy and
   approval; the narrower `gitops_push_blocked` scope and why; and the new
   dry-run-error-blocks-high-risk-tools behavior vs. the unchanged
   dry-run-unsupported fallback.

## Risks

- **Risk**: Importing shared constants removes today's local duplication,
  but nothing stops a *future* edit to `shared/tool_constants.py` from
  silently changing both policy and approval classification at once without
  a test catching an unintended change.
  → **Mitigation**: add an explicit assertion test (e.g. in
  `tests/test_tool_policy.py`) that `classify_operation_type()` for every
  tool in `GITHUB_WRITE_TOOLS | GITHUB_DANGEROUS_TOOLS` returns
  `OperationType.API_WRITE`, so any future set change is caught by CI.
  Likelihood: low; Impact: medium.

- **Risk**: Scoping `gitops_push_blocked` to exclude
  `github_create_issue`/`github_add_issue_comment` (Assumption 4 / UNK-03)
  may not match the security team's actual intent — if "block everything"
  was intended, issue creation/comments would be under-blocked during a
  GitOps freeze.
  → **Mitigation**: make the split explicit in code comments and docs;
  call it out prominently in the PR description for reviewer sign-off
  before merge; the change is a single frozenset difference and easily
  reversible if reviewers want the stricter scope. Likelihood: medium;
  Impact: medium — flagged for review, not blocking implementation.

- **Risk**: The new `ApprovalPreviewBlockingError` path changes approval
  behavior for HIGH-risk GitHub tools when dry-run is enabled and returns an
  explicit error — an operator relying on "always downgrade to preview"
  could see previously-succeeding approvals now denied.
  → **Mitigation**: scope blocking strictly to `result.is_error` (explicit
  server-reported failure), not to unsupported/connection-failure cases;
  add a regression test proving unsupported/connection-failure still falls
  back; document the behavior change in the docs update and PR/commit
  message as a deliberate tightening. Likelihood: low today — no GitHub
  tool is in `approval_dry_run_tools` by default
  (`_DEFAULT_DRY_RUN_TOOLS` in `config_builders.py` only lists filesystem
  tools), so this only activates once an operator explicitly opts a GitHub
  tool into dry-run; Impact: low-medium.

- **Risk**: `tests/test_tool_approval.py` named in the requirement does not
  exist; adding tests to the wrong split file could reduce discoverability.
  → **Mitigation**: follow the existing topic-based split convention
  (`_repos`, `_preflight`, `_risk`, `_paths`) as resolved in UNK-01; do not
  create a new test file unless review feedback requests consolidation.
  Likelihood: low; Impact: low.

- **Risk**: `classify_risk()`'s fallback default in `tool_policy.py`
  (`else: base = RiskLevel.MEDIUM`) means any *new* GitHub tool added to the
  MCP server in the future that is not registered in
  `GITHUB_WRITE_TOOLS`/`GITHUB_DANGEROUS_TOOLS`/`_DEFAULT_APPROVAL_RISK_RULES`/
  `tool_safety_tiers` silently defaults to MEDIUM risk, and would not be
  covered by the new HIGH-risk dry-run-blocking logic.
  → **Mitigation**: out of scope for this fix; documented as a known gap in
  the docs update. Optionally (stretch goal, not required for acceptance
  criteria) add a `tests/test_tool_constants.py` assertion that every tool
  in `GITHUB_TOOLS` appears in `_DEFAULT_APPROVAL_RISK_RULES` or
  `tool_safety_tiers`, to catch future omissions early. Likelihood: low;
  Impact: medium (defense-in-depth gap, not a regression from this change).

All risks above carry documented mitigations within this plan; none require
a separate blocking issue file. No `issues/*_risks.md` file is produced for
this plan.
