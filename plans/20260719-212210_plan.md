## Goal

Make MDQ's documented behavior for fenced-code-block indexing, audit-log policy, and Markdown feature coverage explicitly match the implementation, and relocate the removed-config-key history out of the active config file into a dedicated doc — without changing any runtime behavior.

## Scope

**In scope:**
- Documentation precision fix in `docs/04_mcp_04_04_mdq.md` for the fenced-code-block compatibility table row (currently conflates "heading suppression inside fences" with "content indexing", only the former is stated).

**Out of scope:**
- Any change to `scripts/mcp_servers/mdq/parser.py` parsing logic — Option B (exclude fenced code content from indexed section text) is already the implemented and tested behavior; this plan documents/locks it, it does not implement Option A.
- Any new dedicated `mdq_audit.log` mechanism or `audit_log_path` config key — evidence gathered (see Assumptions) shows this is not needed.
- The orphaned pre-rename duplicate MDQ files (`service.py`, `models.py`, `server.py`, `tools.py`) — already tracked in `issues/20260719-205532_risks.md`; not re-investigated here.
- Any change to `scripts/mcp_servers/mdq/mdq_service.py`, `mdq_server.py`, `mdq_tools.py`, `mdq_models.py`, or any other runtime module.

## Design

1. **Fenced code block content behavior (Option B, no code change).**
   `parser.py:83-99` already implements: on entering/inside a fence (`in_fence=True`), lines are skipped via `continue` before ever being appended to a section's content list. This means fence content is excluded from both heading-detection (already documented/tested) *and* from the indexed/searchable section text (currently implemented but not fully documented or tested). Fix: rewrite the fallback-behavior cell for "フェンス付きコードブロック" in `docs/04_mcp_04_04_mdq.md:71` from "フェンス内の `#` は見出しとして扱われない" to explicitly state both facts — headings inside fences are not recognized, **and** fence content itself is excluded from the indexed section text (not searchable via `search_docs`/`grep_docs` against reconstructed section content). Classify per `rules/coding.md`: Documentation fix required (the current cell under-describes real behavior).

2. **Audit-log policy (Accepted current specification, no code change).**
   Add one explicit dated note stating the dedicated-audit-log question (raised by this requirement) was evaluated on 2026-07-19 and resolved as: shared audit logging via `_audit_log()`/`server_key="mdq"` into `/opt/llm/logs/audit.log` is authoritative and sufficient; no dedicated `mdq_audit.log` or `audit_log_path` config key will be reintroduced without a concrete implementation need. Place near `docs/04_mcp_04_04_mdq.md:40`.

3. **Config-key history relocation.**
   Create `docs/mdq_config_history.md` with a short intro sentence plus the six `NOTE` blocks currently at `config/mdq_mcp_server.toml:50-95` (`audit_log_path`, `concurrency_limit`, `max_search_results`, embedding keys, summary-cache keys, `enable_refresh`, `status`), copied verbatim (not paraphrased) to preserve exact rationale and removal dates. Replace those 46 lines in the config file with a short 2-3 line pointer comment: e.g. "# Historical notes on removed config keys have moved to docs/mdq_config_history.md" plus a one-line list of the six key names for grep-ability.

4. **Markdown feature coverage documentation.**
   Already complete: `docs/04_mcp_04_04_mdq.md:66-79` table already lists Setext headings, HTML blocks, MDX, GFM tables, and inline HTML tags as "No" with fallback behavior stated. No table changes required for these five items. Optionally add a one-line cross-reference near the table pointing to the existing frontmatter-tags note (lines 43-48) so a reader scanning the compatibility table isn't left wondering whether frontmatter tags/categories are searchable (they are, as of 2026-07-19).

5. **Test lock for content-exclusion.**
   Add a new test (in `tests/test_mdq_service.py` per UNK-03 default) that: indexes a markdown file containing a fenced code block with a unique token (e.g. `UNIQUE_FENCE_TOKEN_XYZ`) not present anywhere else in the file, calls `search_docs` for that token, and asserts zero results — locking that fence content is excluded from the search index end-to-end, not merely from heading detection. This satisfies the requirement's own Test Plan item ("code fence containing a unique searchable term; indexing and searching the unique term").

## Risks

- **Risk:** The new fence-content-exclusion test could unexpectedly fail (i.e. content is *not* actually excluded from the search index by some other code path such as `search.py` snippet generation reading raw file content instead of parsed sections). Likelihood: low — `parser.py:83-99` skips fence lines via `continue` before any section-content append, so downstream indexing/search operates on the already-fence-stripped section text. Mitigation: run the new test against current code *before* finalizing any doc wording; if it fails, stop and file a "Documentation fix required" → actually "Implementation fix required" issue under `issues/` per `rules/coding.md`, rather than writing docs to match a bug.
- **Risk:** Trimming `config/mdq_mcp_server.toml:50-95` could lose institutional knowledge if the new `docs/mdq_config_history.md` doesn't capture something precisely. Likelihood: low. Mitigation: copy the six `NOTE` blocks verbatim (not paraphrased) and diff-review before deleting the originals; the original comment text also remains recoverable via `git log`/`git show` on the toml file regardless.
- **Risk:** Doc wording changes in `docs/04_mcp_04_04_mdq.md` could trip `check-mcp-docs`'s pattern-based consistency checks (fail-open wording, routing-authority language, tool-count checks) if new phrasing accidentally matches a flagged pattern. Likelihood: low-medium. Mitigation: run `uv run check-mcp-docs` as part of Phase 4 validation before considering the doc edits complete; the tool is already listed in `rules/toolchain.md` for exactly this purpose.
- **Risk:** This requirement is the last of a 6-file batch (26-30 already processed) touching overlapping doc/config files (`docs/04_mcp_04_04_mdq.md`, `config/mdq_mcp_server.toml`); if an earlier cycle's plan is implemented concurrently with or after this one, line numbers cited here (e.g. `:71`, `:40`, `:50-95`) may have shifted. Likelihood: medium. Mitigation: at implementation time, re-run `rg -n "フェンス付きコードブロック|Audit:|NOTE: audit_log_path"` against the current file state to relocate the exact lines rather than trusting this plan's line numbers verbatim.

All risks above have a concrete mitigation folded into the plan; none are being carried forward as unmitigated. No `issues/*_risks.md` file is written for this cycle.
