## Goal

Make `deploy/init_db.sh` actively verify that all 5 required workflow tables (`tasks`, `attempts`, `processed_events`, `artifacts`, `approvals`) exist in `workflow.sqlite` after schema initialization, and fail with a non-zero exit code and remediation message if any are missing.

## Scope

**In-Scope:**
- `deploy/init_db.sh`: replace the existing informational `.tables` echo for `workflow.sqlite` with an actual per-table existence check that aborts the script on any missing table

**Out-of-Scope:**
- `scripts/db/create_schema.py` / `scripts/db/schema_sql.py`: no changes — direct code read confirms `create_schema.py` already calls `create_workflow_schema()` (line 90, within the wrapper `create_schema()` that `init_db.sh` invokes), and the DDL already uses `IF NOT EXISTS` for idempotent re-runs. The requirement's premise ("the deployment flow must ensure it is executed") is already satisfied by existing code; only the *verification* step is missing.
- Adding equivalent post-init verification for `rag.sqlite`/`session.sqlite`/`eventbus.sqlite`'s tables — the requirement's Scope and Tasks name only the workflow tables

## Assumptions

1. `deploy/init_db.sh` already invokes `db/create_schema.py`, which already calls `create_workflow_schema()` — confirmed by direct read of both files at planning time. Tasks #1-#2 of the requirement ("verify whether init_db.sh calls create_workflow_schema(); if missing, update") resolve to "already done, no action needed."
2. `create_workflow_schema()`'s DDL creates exactly these 5 tables in `workflow.sqlite`: `tasks`, `attempts`, `processed_events`, `artifacts`, `approvals` (per the module's own docstring, line 12) — matching the requirement's Background exactly.
3. The existing `sqlite3 "${DEPLOY_DB}/workflow.sqlite" ".tables"` line in `init_db.sh` is purely informational (its output is never captured or compared against anything) — this is the exact gap the requirement targets. It should be replaced, not merely supplemented, to avoid duplicate/confusing output.
4. Table-existence should be checked via `SELECT name FROM sqlite_master WHERE type='table' AND name='<table>'` per table, rather than parsing `.tables`'s column-formatted text output — the `sqlite_master` query is exact and avoids any risk of substring/word-boundary false positives from `.tables`'s multi-column layout.

## Unknowns

No unknowns — the existing code fully determines the required behavior; both files needing evidence (`init_db.sh`, `create_schema.py`) were read directly at planning time.

## Design

Replace the existing block:
```bash
sqlite3 "${DEPLOY_DB}/workflow.sqlite" ".tables"
# expected: artifacts  attempts  approvals  processed_events  tasks
```
with:
```bash
echo "--- workflow.sqlite テーブル確認 ---"
REQUIRED_WORKFLOW_TABLES="tasks attempts processed_events artifacts approvals"
MISSING_TABLES=""
for t in ${REQUIRED_WORKFLOW_TABLES}; do
  FOUND=$(sqlite3 "${DEPLOY_DB}/workflow.sqlite" \
    "SELECT name FROM sqlite_master WHERE type='table' AND name='${t}';")
  if [ -z "${FOUND}" ]; then
    MISSING_TABLES="${MISSING_TABLES} ${t}"
  fi
done
if [ -n "${MISSING_TABLES}" ]; then
  echo "[FATAL] Workflow database schema is missing or incomplete." >&2
  echo "Missing table(s):${MISSING_TABLES}" >&2
  echo "Run the workflow schema initialization step before starting the agent." >&2
  exit 1
fi
echo "OK: all required workflow.sqlite tables present (${REQUIRED_WORKFLOW_TABLES})"
```

This keeps the other three `.tables` informational checks (`rag.sqlite`, `session.sqlite`, `eventbus.sqlite`) unchanged, per the Out-of-Scope decision — only the workflow table check gains active verification.
