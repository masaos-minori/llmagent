## Design

### 1. Integration Testing Architecture

**Process boundary model.** The existing suite already establishes the
right pattern for two of the three subsystems and this plan extends it
rather than replacing it:

- **HTTP-based MCP (existing, TC-A/3B/3D):** `respx.mock()` intercepts at
  the `httpx.AsyncClient` transport level, driving `ToolExecutor` directly
  with hand-built `McpServerConfig` objects and a monkeypatched
  `_resolver.resolve`. This is correct for testing `ToolExecutor`'s own
  retry/health-check/error-classification logic, but it never launches a
  real subprocess, so it cannot observe pipe-level failure (SIGPIPE,
  partial-write-then-close, non-JSON garbage on stdout).
- **stdio-based MCP (new, gap):** must use a real
  `asyncio.create_subprocess_exec` (the existing `stdio_echo_server`
  fixture's own technique) so that killing the process or closing its
  stdin/stdout actually exercises the OS pipe layer, not a mock of it.
  Since no agent-layer `StdioLifecycleManager` exists (Assumption 3), tests
  target the subprocess/pipe boundary directly and assert on what a caller
  reading from the pipe observes (`BrokenPipeError`, EOF, or a JSON decode
  failure on a truncated line) — this is what any future stdio transport
  implementation would need to survive.
- **SQLite (existing, TC-B):** real `sqlite3` connections against a
  `tmp_path` file — correct pattern, reused as-is for the new corruption
  scenario. The gap is scenario coverage (corruption + recovery), not
  architecture.
- **RAG↔LLM (existing, TC-C, but shallow):** the existing tests mock at
  the wrong boundary — they mock `HybridRetriever`/`LLMClient` internals
  directly rather than driving the actual `LLMTurnRunner.run()` loop. The
  new tests fix this by constructing a real `LLMTurnRunner` with a
  `MagicMock`-based `ctx.services_required` (mirroring the existing,
  working pattern already used in `tests/test_llm_turn_runner.py`) and a
  `respx`-mocked RAG tool HTTP response, so the full round-trip
  (tool-call → tool-result → `ctx.conv.history` → next LLM stream call) is
  exercised end-to-end within one test process — no new "architecture," just
  correct application of the pattern the unit tests already use, at one
  layer higher.

**Mock/real boundary summary:**

| Layer | Existing approach | This plan's approach |
|---|---|---|
| HTTP MCP transport | `respx` (keep) | unchanged |
| stdio MCP transport | none (fixture unused) | real subprocess via `asyncio.create_subprocess_exec` |
| SQLite | real file (keep) | real file, add corruption via byte-level truncation |
| RAG↔LLM round-trip | mocked internals, no round-trip | real `LLMTurnRunner` + `respx`-mocked RAG tool response, driven through `execute_all_tool_calls` |

**Concurrency model:** all new async tests use `pytest.mark.asyncio`
(consistent with `asyncio_mode = "auto"` already set in `pyproject.toml`,
though the codebase's convention is to mark explicitly — followed here for
consistency with 100% of existing async tests read during this analysis).
Subprocess-based tests use `asyncio.create_subprocess_exec` +
`asyncio.wait_for(..., timeout=N)` for deterministic timeout assertions,
matching the pattern already demonstrated in `tests/test_repl.py:311-323`
(delayed-task + `wait_for` timeout).

### 3. Robustness & Chaos Injection Strategy

This plan does **not** add a new "chaos" abstraction — `ToolLoopGuard` and
`ErrorInjectionService` are already exercised by `test_robustness_chaos.py`
(3A, 3B) at the correct level (unit-adjacent, direct class instantiation).
Instead:

- **New chaos surface (stdio):** `test_mcp_transport_crash.py`'s D01–D04
  extend the *existing* chaos style (real fault injection, deterministic
  assertions, no bare-`sleep` races) to a subsystem the existing 3D suite
  never reached (raw process/pipe, as opposed to HTTP transport).
- **New chaos surface (DB corruption):** `test_session_recovery.py`'s
  E01–E04 extend 3C's SQLite chaos style (real file manipulation, not
  mocked `sqlite3.connect`) to actual on-disk corruption + the recovery
  path, which 3C does not touch (3C covers busy-lock and disk-full-*
  simulation only, not corruption-and-restore).
- **Injection technique for D-series (process-level):** use
  `proc.kill()` / `proc.terminate()` / closing `proc.stdin` mid-write —
  these are the three concrete "chaos" primitives available for a real
  subprocess, chosen because they map directly to what a crashed or
  hung MCP server subprocess looks like from the agent's side, and because
  `HttpServerLifecycleManager` already has documented handling
  (`_terminate_with_timeout`, `_wait_exited`) that D05 cross-checks against.
- **Injection technique for E-series (corruption):** byte-level truncation
  of a WAL-mode file (`f.seek(n); f.truncate()`), which is a standard,
  reproducible SQLite corruption technique (per UNK-02's resolution path)
  — avoids the flakier alternative of trying to kill a process mid-`COMMIT`
  to induce corruption naturally, which is non-deterministic.
- **Injection technique for F-series (RAG/LLM):** `respx` side-effects
  already used by 100% of existing HTTP-boundary tests in this repo —
  reused, not reinvented. The only "new" injection here is placing the
  fault at the *second* `_stream_llm()` call (F03), which none of C01–C10
  do (they each test a single isolated call in isolation).
- **No new fault-injection library is introduced.** `respx` (HTTP),
  `asyncio.create_subprocess_exec` + signal methods (process), and raw
  `open()`/`sqlite3` file manipulation (DB) are the three primitives
  already present in the codebase's test dependencies
  (`pyproject.toml` dev deps) — sufficient for every scenario above.
- **Flakiness guard:** per the existing convention documented in
  `test_robustness_chaos.py`'s own module docstring ("Run 5x for flakiness
  check"), all three new files must pass 5 consecutive runs
  (`for i in {1..5}; do uv run pytest tests/integration/test_mcp_transport_crash.py tests/integration/test_session_recovery.py tests/integration/test_rag_turn_integration.py -v --timeout=30; done`)
  before being considered complete — timing-sensitive subprocess tests
  (D03, D04) are the primary flakiness risk.

### 4. Example Test Skeleton

Copy-pasteable boilerplate for one complex integration scenario (D03: stdio
subprocess hang, bounded by `asyncio.wait_for`), following the existing
repo's async test conventions (`pytest.mark.asyncio`, no custom event-loop
fixtures needed since `asyncio_mode = "auto"` is already set):

```python
"""tests/integration/test_mcp_transport_crash.py

Integration tests: Agent Loop <-> MCP Servers, stdio transport crash modes.

Companion to test_agent_mcp_integration.py (TC-A, HTTP-only). These tests
exercise the raw subprocess/pipe boundary directly since no agent-layer
StdioLifecycleManager exists in this codebase today (see plan Assumption 3,
plans/20260709-000000.md) -- assertions target what any future stdio
transport implementation would need to survive.

Run 5x for flakiness check (subprocess timing is the main risk):
    for i in {1..5}; do uv run pytest tests/integration/test_mcp_transport_crash.py -v --timeout=30; done
"""

from __future__ import annotations

import asyncio
import json

import pytest


async def _hanging_stdio_server() -> asyncio.subprocess.Process:
    """Start a subprocess that reads one line from stdin, then sleeps forever."""
    script = (
        "import sys, time\n"
        "sys.stdin.readline()\n"
        "time.sleep(3600)\n"
    )
    proc = await asyncio.create_subprocess_exec(
        "python",
        "-c",
        script,
        stdin=asyncio.subprocess.PIPE,
        stdout=asyncio.subprocess.PIPE,
    )
    return proc


# ── D03: stdio subprocess never responds — bounded read must time out ───────


@pytest.mark.asyncio
async def test_d03_stdio_hang_times_out_and_process_is_reaped() -> None:
    proc = await _hanging_stdio_server()
    try:
        assert proc.stdin is not None
        assert proc.stdout is not None

        request = {"id": "1", "name": "some_tool", "arguments": {}}
        proc.stdin.write((json.dumps(request) + "\n").encode())
        await proc.stdin.drain()

        # Bounded read: a real caller must not hang indefinitely waiting
        # for a response from a wedged MCP server subprocess.
        with pytest.raises(asyncio.TimeoutError):
            await asyncio.wait_for(proc.stdout.readline(), timeout=1.0)
    finally:
        # Cleanup must succeed even though the process is unresponsive to
        # stdin -- this is the actual "chaos" assertion: termination must
        # not itself hang.
        proc.kill()
        await asyncio.wait_for(proc.wait(), timeout=5.0)
        assert proc.returncode is not None  # reaped, not a zombie


# ── D01: subprocess killed mid-response — reader sees EOF, not a hang ───────


@pytest.mark.asyncio
async def test_d01_stdio_killed_mid_response_yields_eof() -> None:
    script = (
        "import sys, time\n"
        "sys.stdin.readline()\n"
        "time.sleep(5)\n"  # simulate slow processing before the kill lands
        "sys.stdout.write('should never be seen\\n')\n"
        "sys.stdout.flush()\n"
    )
    proc = await asyncio.create_subprocess_exec(
        "python",
        "-c",
        script,
        stdin=asyncio.subprocess.PIPE,
        stdout=asyncio.subprocess.PIPE,
    )
    assert proc.stdin is not None
    assert proc.stdout is not None

    proc.stdin.write(b'{"id": "1"}\n')
    await proc.stdin.drain()
    await asyncio.sleep(0.1)  # let the subprocess start its sleep

    proc.kill()
    await proc.wait()

    # After kill, a read on stdout must return EOF (b""), not hang or raise.
    line = await asyncio.wait_for(proc.stdout.readline(), timeout=2.0)
    assert line == b""
```
