# Plan: Make memory retrieval branch-aware and use stored branch context explicitly

Source: `requires/20260626_04_require.md`

---

## Goal

Pass `branch` through all retrieval and injection call sites so that branch context is used as an active filter/boost boundary, not just scoring metadata, and add tests that verify cross-branch memories are not injected into the current task.

---

## Scope

**In-Scope**
- Pass `branch` parameter in `MemoryInjectionService.on_user_prompt()` calls to `HybridRetriever.search()`
- Distinguish "global memories" (empty branch field) from "branch-specific memories" in retrieval: boost branch-match, do not hard-exclude global memories
- Confirm `MemoryInjectionService` constructor already receives `branch` (confirmed in `__init__`); fix the call sites that drop it
- Add tests: unrelated branch memories not injected; branch-specific memories score higher than global

**Out-of-Scope**
- Full git integration redesign
- Replacing memory metadata model
- Changing the branch field storage schema

---

## Assumptions

1. `MemoryInjectionService.__init__` already stores `self._branch` (confirmed in injection.py:42).
2. `FtsRetriever.search()` already accepts `branch` param and calls `_context_boost(entry, project, repo, branch)` (confirmed in retriever.py:154).
3. The fix is to pass `branch=self._branch` in `on_user_prompt()` search calls (currently absent, lines 84–104).
4. Policy: branch-match adds `_CONTEXT_MATCH_BOOST + 0.05` (0.15); empty branch entries (global) still pass through — filtering is boost-based, not exclusion-based.
5. If strict exclusion is needed, that requires a schema change and is out of scope.
