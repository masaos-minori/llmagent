# Plan: Clarify use_rrf=False Behavior and Quality Implications (req_40)

## 1. Goal

optimization switch), and expose the active fusion mode in runtime diagnostics.

## 3. Confirmed Facts

**`use_rrf=False` path (confirmed from `fusion.py` and `repository.py`):**
- Calls `_dedup_hits(ctx.search_results)` â€” deduplicates by `chunk_id`, keeps first occurrence
- All merged results get `rrf_score=0.0` â€” no rank-weighted scoring
- Preserves original search-result order within each list; cross-list priority by insertion

**`use_rrf=True` path (RRF):**
- Assigns each result a score of `ÎŁ 1/(rrf_k + rank)` across all result lists
- Results ranked higher in multiple lists get amplified scores
- `rrf_k=60` (default) â€” provides stability against outlier ranks

**Quality tradeoff:**
- `use_rrf=True`: robust cross-list ranking; benefits from MQE (multi-query expansion)
  because results appearing in multiple query result sets get promoted
- `use_rrf=False`: no rank signal; if the same chunk appears in multiple lists, only
  the first occurrence matters; MQE-generated queries provide no additional re-ranking benefit

**Stage tracking (confirmed):**
- `pipeline.py:140-141`: `if not self._cfg.use_rrf: return "fallback", "use_rrf=False"` in
  `_get_stage_status()` â†’ `StageResult(status="fallback", fallback_reason="use_rrf=False")`
- Visible in `--debug`: `~ FusionStage: fallback â€” use_rrf=False`
- Not in `get_diagnostics()` â€” no dedicated `fusion_mode` field
