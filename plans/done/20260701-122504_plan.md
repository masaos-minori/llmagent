## Goal
`Orchestrator._init_workflow_task()` の `existing_task_id is None` ブランチで匿名生成されている `StateStore()` インスタンスを変数に束縛し、`try/finally` で `close()` を確実に呼び出すことで、DB 接続リークを修正する。

## Scope
**In-Scope**:
- `scripts/agent/orchestrator.py` — `_init_workflow_task()` の `existing_task_id is None` ブランチのみ

**Out-of-Scope**:
- `_handle_workflow_engine()` の `store` — 既に `try/finally` で管理済み
- `existing_task_id is not None` ブランチ — 既に `store.close()` を呼び出し済み
- `StateStore` クラスへの `__enter__`/`__exit__` 追加（要確認事項）
- 他の `StateStore` 使用箇所

## Assumptions
1. `StateStore.__init__()` は呼び出し時点で `workflow.sqlite` への接続を開く
2. `close()` を呼ばなければ GC 依存となり、asyncio 長期実行環境でリークする
3. `__enter__`/`__exit__` の追加はスコープ外（確認済み：本 Issue では不要）

## Design
`existing_task_id is None` ブランチを以下のパターンに変換する（同メソッド L289–295 と同形式）：

```python
# Before
task = StateStore().create_task(
    session_id=session_id,
    turn_number=ctx.stats.stat_turns,
    workflow_version=self._workflow_def.version,
    workflow_id=workflow_id,
)

# After
store = StateStore()
try:
    task = store.create_task(
        session_id=session_id,
        turn_number=ctx.stats.stat_turns,
        workflow_version=self._workflow_def.version,
        workflow_id=workflow_id,
    )
finally:
    store.close()
```
