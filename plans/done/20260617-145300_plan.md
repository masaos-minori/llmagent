# Work Plan: Isolate delete-order logic for chunks_vec → chunks → documents

## Goal
- consolidate force-delete logic into one dedicated, reusable function
- make the delete order (chunks_vec → chunks → documents) explicit and documented
- add tests that verify correct delete order and no orphaned vectors

## Scope
- `scripts/rag/ingestion/ingester.py` — `_delete_existing_document()` method
- `scripts/agent/document_repo.py` — `delete_document()` method (duplicate logic)
- `tests/test_document_repo.py` — existing test patterns for delete order verification
- `tests/test_rag_ingester.py` — new tests for ingester delete helper

## Assumptions
- Both `RagIngester._delete_existing_document()` and `DocumentRepository.delete_document()` share the same delete order requirement
- No other code paths perform this three-table delete pattern
- `chunks_vec` has no FK constraint to `chunks`, requiring explicit deletion first
- Existing `_FakeSQLiteHelper` test pattern in `test_document_repo.py` can be reused for ingester tests

## Unknowns
- [ ] Are there any other callers that delete chunks_vec/chunks/documents that need consolidation?
  - Analyzed: only `ingester.py` and `document_repo.py` have this pattern. No others found.
- [ ] Should the helper accept `doc_id` or `url` as parameter?
  - `ingester.py` already has `doc_id`; `document_repo.py` resolves url→doc_id. Helper should take `doc_id`.
- [ ] Does `chunks_fts` need explicit deletion or does ON DELETE CASCADE handle it?
  - Schema shows FTS5 delete triggers fire automatically when chunks are deleted. No explicit chunks_fts delete needed.
- [ ] Should the helper be in a shared module (e.g. `db/store_impl.py`) or stay module-local?
  - Recommendation: keep in `ingester.py` (primary owner) and extract to shared only if a third caller appears.

## Design
### Proposed helper signature
```python
def delete_document_chain(
    db: SQLiteHelper,
    doc_id: int,
) -> None:
    """Delete chunks_vec → chunks → documents for doc_id.

    chunks_vec has no FK to chunks, so it must be deleted first.
    chunks deletion triggers FTS5 triggers automatically.
    documents deletion cascades to remaining chunks rows.
    """
```

### Integration points
1. `RagIngester._delete_existing_document()` → becomes thin wrapper calling `delete_document_chain(db, doc_id)`
2. `DocumentRepository.delete_document()` → replace inline SQL with `delete_document_chain(db, doc_id)` after resolving url→doc_id
3. If shared module needed: place in `db/store_impl.py` or `db/helper.py`; otherwise keep in `ingester.py` and import where needed

## Implementation steps
1. Create `delete_document_chain(db, doc_id)` function in `scripts/rag/ingestion/ingester.py`
   - Include docstring documenting delete order rationale
   - Execute three DELETE statements in order: chunks_vec → chunks → documents
2. Refactor `RagIngester._delete_existing_document()` to call the new function
3. Update `DocumentRepository.delete_document()` to use the same logic
   - Option A: import from ingester.py (creates cross-module dependency)
   - Option B: duplicate in document_repo.py (DRY violation but preserves module boundaries)
   - Recommendation: Option B for now; extract to shared db module if a third caller appears
4. Add test in `tests/test_document_repo.py`:
   - Insert document + chunks + chunks_vec rows
   - Call delete_document()
   - Assert zero rows remain in all three tables
5. Add tests in `tests/test_rag_ingester.py`:
   - Verify delete order via SQL query inspection after each DELETE
   - Verify no orphaned chunks_vec rows after force reinsert
   - Verify idempotent delete (second call on deleted doc is no-op)

## Validation plan
- Unit test: `test_delete_no_orphaned_vectors` — insert doc+chunks+vec, delete, assert 0 vec rows
- Unit test: `test_delete_order_explicit` — mock execute to capture SQL order, verify chunks_vec before chunks before documents
- Unit test: `test_force_reinsert_clean` — insert, force-delete, re-insert; verify no leftover rows
- Existing `test_document_repo.py` tests continue to pass (no behavior change)

## Risks
| Risk | Mitigation |
|---|---|
| Cross-module import cycle (ingester ↔ document_repo) | Keep helper in ingester.py; duplicate in document_repo.py if needed; avoid circular imports |
| FTS5 triggers not firing after refactor | Schema shows triggers use `ON DELETE` on chunks table; delete_chunks still fires them |
| Test coverage gap for parallel ingest paths | Delete logic is synchronous; no parallel concerns |
| Breaking existing force reinsert behavior | Behavior-lock tests in test_document_repo.py; add new tests before refactoring |
