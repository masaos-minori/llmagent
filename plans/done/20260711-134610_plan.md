## Goal

Make watchdog `/health` probe handling explicit and diagnosable: distinguish unreachable, non-200, malformed-JSON, and degraded-body (with/without `restart_recommended`) cases in logs and in `HealthRegistry`, without changing which of the existing restart/no-restart decisions fires.

## Scope

**In-Scope:**
- `scripts/agent/shared/health_models.py`: add `parse_failed`/`parse_error` fields to `McpHealthProbeResult`
- `scripts/agent/repl_health.py`: `_probe_mcp_health_detail()` sets the new fields on JSON parse failure; `_watchdog_check_http()` emits 5 explicitly distinct log messages (one per failure class) instead of today's 2 generic ones, and records `"malformed_health_response"` as the degraded reason for parse failures
- `tests/test_mcp_health_degraded.py` (existing watchdog-integration section): add tests for malformed JSON and for the `restart_recommended=True` degraded case

**Out-of-Scope:**
- `McpServerHealthRegistry`'s own state-transition logic (`record_degraded()`'s `UNAVAILABLE`/`HALF_OPEN` guard) — covered by the sibling, already-planned requirement (`plans/20260711-134244_plan.md`); this plan only changes *what reason string* is passed into `record_degraded()`, not its internal behavior
- Changing which of the 3 existing decision branches (healthy / degraded-no-restart / restart-or-give-up) fires for any given probe result — Acceptance Criteria explicitly requires "Existing restart behavior remains unchanged unless explicitly modified," and no explicit change is proposed here
- Auto-restarting solely because of malformed JSON — explicitly excluded per the requirement's Notes

## Assumptions

1. `_probe_mcp_health_detail()` (line 40) currently swallows any JSON-parse exception into `body = {}` with a bare `except Exception` (already `# noqa: BLE001`-justified) — confirmed by direct read; this is the "malformed JSON collapsed into an empty body" the requirement describes.
2. Because `restart_recommended`/`operator_action_required` are both derived from `body.get(...)` and `body` is `{}` on parse failure, a parse-failure probe **already** defaults to `restart_recommended=False` today — meaning it already falls into the "reachable, not restart-recommended" (degraded, no-restart) branch of `_watchdog_check_http()`, not the restart branch. This confirms the Notes' requirement ("do not restart solely because of malformed JSON") is **already satisfied by accident** today; this plan only needs to make that existing, correct routing *explicit and diagnosable* — not change it.
3. `McpHealthProbeResult` is `@dataclass(frozen=True)` (confirmed by direct read) — adding `parse_failed: bool = False` and `parse_error: str | None = None` with defaults is purely additive; the only two production construction sites (`repl_health.py:51,64`) and the three test construction sites (`tests/test_watchdog.py:90,116,145`) all use keyword arguments, so none require updating.
4. Resolving the requirement's Task "Decide whether malformed health responses should be treated as operator-action-required": **yes, for logging/visibility purposes only** — a parse failure is logged at `WARNING` with the raw diagnostic detail every time (equivalent to today's `operator_action_required=True` logging path), but this is done by the watchdog's own logging decision, not by setting `probe.operator_action_required=True` on the model (there is no body to read that flag from during a parse failure, and forcing it to `True` would be a fabricated value, not an observed one). This satisfies the Acceptance Criteria ("operators can identify the exact health failure class") without inventing data the probe never actually received, and does not change `restart_recommended`'s already-correct `False` default (Assumption 2), honoring the Notes.
5. `_watchdog_check_http()` already has cyclomatic complexity C (18) per `radon cc` — high for this codebase's stated threshold (CC ≥ 15 flags "additional test coverage before changes" per `skills/python-issue-to-plan/workflow.md` Step 6). Adding 5 differentiated log branches directly inline would push this further; the Design below extracts a small, separately-testable classification/logging helper instead of adding more inline branches to `_watchdog_check_http()` itself.
6. `repl_health.py` has high churn (62 commits) — consistent with it being this codebase's central, actively-evolving health/watchdog module (confirmed by this session's own recent, unrelated edits to it for workflow-schema-version checks). No bus-factor concern (multiple unrelated changes across many commits, not single-author-dominated).

## Unknowns

No unknowns remain. The one Task requiring a design decision ("operator-action-required for malformed responses") is resolved in Assumption 4 using the requirement's own Acceptance Criteria and Notes as deciding evidence.

## Design

### `McpHealthProbeResult` (health_models.py)

```python
@dataclass(frozen=True)
class McpHealthProbeResult:
    reachable: bool
    status_code: int | None
    restart_recommended: bool
    operator_action_required: bool
    body: dict[str, object]
    parse_failed: bool = False
    parse_error: str | None = None
```

### `_probe_mcp_health_detail()` (repl_health.py)

```python
try:
    body: dict[str, object] = resp.json()
except Exception as exc:  # noqa: BLE001 — health check must not fail on body parse errors
    return McpHealthProbeResult(
        reachable=True,
        status_code=resp.status_code,
        restart_recommended=False,
        operator_action_required=False,
        body={},
        parse_failed=True,
        parse_error=f"{exc} (raw={resp.text[:200]!r})",
    )
```

(Unreachable-path and successful-parse-path returns are unchanged.)

### New classification helper (repl_health.py, new small function)

```python
def _classify_health_failure(probe: McpHealthProbeResult) -> str:
    """Return a short, log-friendly label for a non-fully-healthy probe result.

    Does not affect any restart/degrade decision — for diagnostic logging only.
    """
    if not probe.reachable:
        return "unreachable"
    if probe.parse_failed:
        return f"malformed JSON ({probe.parse_error})"
    if probe.status_code != HTTPStatus.OK:
        return f"non-200 (status={probe.status_code})"
    if probe.restart_recommended:
        return "degraded (restart_recommended=true)"
    return "degraded (restart_recommended=false)"
```

### `_watchdog_check_http()` changes (repl_health.py)

- In the "reachable and not restart_recommended" branch: log
  `logger.warning("Watchdog: %r (%s) — %s", key, srv_cfg.url, _classify_health_failure(probe))`
  before the existing `operator_action_required` check; when `probe.parse_failed`, call
  `health_registry.record_degraded(key, reason="malformed_health_response")` instead of the
  existing `body.get("reason")/body.get("message")` extraction (which is meaningless on an
  empty body).
- In the "unreachable or restart_recommended" branch: replace the single generic
  `"health check failed, restarting"` log with one that includes
  `_classify_health_failure(probe)`, so "unreachable" and "degraded
  (restart_recommended=true)" are visibly distinct in the log line, while the restart
  decision itself (`count >= max_restarts` / `lifecycle.restart()`) is untouched.
- No change to the "fully healthy" branch (`reachable and status_code == 200 and not
  parse_failed` — note: add `and not probe.parse_failed` to this branch's condition too,
  since today a 200-status response with genuinely malformed JSON is silently treated as
  "fully healthy," which is itself a small, real gap the requirement's "malformed JSON
  collapsed into an empty body" concern implies should not happen).

### Tests (tests/test_mcp_health_degraded.py, watchdog-integration section)

- `test_watchdog_malformed_json_calls_record_degraded_with_reason` — probe with `parse_failed=True`; assert `record_degraded` reason is `"malformed_health_response"` and a log call occurred (via `caplog`).
- `test_watchdog_degraded_with_restart_recommended_true_logs_distinctly` — extends the existing `test_watchdog_reachable_restart_does_not_call_record_degraded` case by also asserting (via `caplog`) that the log message differs from the unreachable case's message.
- `test_watchdog_unreachable_logs_distinctly_from_degraded` — asserts the unreachable case's log text differs from the degraded case's log text (regression guard for "operators can identify the exact health failure class from logs").
- `test_classify_health_failure_all_five_cases` — direct unit test of the new `_classify_health_failure()` helper against all 5 constructed `McpHealthProbeResult` variants (unreachable, non-200, malformed, degraded-no-restart, degraded-restart-true).

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Adding the `and not probe.parse_failed` condition to the "fully healthy" branch is a genuine (if narrow) behavior change: a 200-status response with malformed JSON now falls into the "degraded, no restart" branch instead of "fully healthy" | Low | This is a deliberate, in-scope fix (not accidental scope creep) directly serving the requirement's core complaint ("malformed JSON collapsed into an empty body"); the new branch it falls into still does not restart (Assumption 2), so "existing restart behavior... unless explicitly modified" is satisfied — this is the one explicit modification, called out here for reviewer visibility |
| `_watchdog_check_http()`'s already-high complexity (CC 18) could still creep up slightly even with a helper extracted, if log-statement changes add new conditionals inline | Medium | Keep the two modified branches to a single added log line + a conditional `record_degraded` call each — no new `if`/`elif` structure beyond what already exists; verify with `radon cc` in the validation plan before considering the step done |
| `caplog`-based assertions on log message text are more brittle than state-based assertions (wording changes silently break tests) | Low | Assert on distinguishing substrings (e.g., `"unreachable"` vs `"malformed"` vs `"non-200"`) rather than full message equality, consistent with how log-based tests are typically written; the messages themselves are also cross-checked via `_classify_health_failure()`'s direct unit test, so log-text drift is caught at the source function, not just at the integration-test log-capture layer |
