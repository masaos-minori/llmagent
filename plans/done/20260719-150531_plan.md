# Plan: Remove _FORBIDDEN_KEYS / _REMOVED_KEYS config key rejection logic

Source requirement: `requires/20260719_32_require.md`

## Classification

**Path A** — No public/runtime-facing interface changes. Only removes dead code paths that reject stale config keys at startup/reload time.

## 1. Goal

Remove the three duplicate mechanisms that reject deprecated config keys at startup and reload time. These checks are redundant because Pydantic dataclasses silently ignore unknown fields, so stale keys in config files have no functional impact.

## 2. Scope

**In-Scope:**
- Remove `_FORBIDDEN_KEYS` constant and its check block (lines 254-275) from `scripts/agent/config_builders.py`
- Remove `_REMOVED_KEYS` constant and `_filter_removed_keys()` method from `scripts/shared/config_loader.py`
- Remove `_REMOVED_KEYS` constant and its check block (line 130-134) from `scripts/agent/services/config_reload.py`
- Remove the `github_server_url` individual check (lines 272-275) from `scripts/agent/config_builders.py`

**Out-of-Scope:**
- Changes to the Pydantic dataclass definitions themselves (they already handle unknown fields correctly)
- Changes to config file contents or migration guidance for users with stale keys

## 3. Assumptions

1. Pydantic dataclasses will silently ignore unknown fields in config dicts — verified by reading `AgentConfig`, `LLMConfig`, `RAGConfig`, `ToolConfig`, `MemoryConfig`, `ApprovalConfig`, `MCPConfig`, `ObservabilityConfig` definitions in `scripts/agent/config_dataclasses.py`. All use Pydantic's default behavior which ignores extra fields.
2. No test relies on `ConfigLoadError` being raised for removed config keys — verified by searching for tests that set removed keys and expect errors.

## 6. Design

### Current state — Three duplicate mechanisms

| Location | Constant name | Keys covered | Action |
|---|---|---|---|
| `config_loader.py:L31-39` | `_REMOVED_KEYS` | 5 keys | Silently filter out during `load_all()` |
| `config_reload.py:L87-95` | `_REMOVED_KEYS` | Same 5 keys | Raise `ConfigReloadValidationError` |
| `config_builders.py:L254-266` | `_FORBIDDEN_KEYS` | 10 keys + `github_server_url` separate | Raise `ConfigLoadError` |

### Inconsistencies

1. **Startup vs reload divergence**: During initial startup, `config_loader.py` silently filters out the 5 shared keys, so `_FORBIDDEN_KEYS` never sees them. But `/reload` uses `config_reload.py` which raises an error for those same 5 keys.
2. **Overlap gap**: `workflow_mode`, `web_search_url`, etc. exist only in `_FORBIDDEN_KEYS` — they are silently accepted during startup but would raise an error if added via `/reload` (if such a path existed).
3. **Separate handling**: `github_server_url` is checked separately from `_FORBIDDEN_KEYS` with a different error message format.

### Target state — No rejection mechanism

Since Pydantic dataclasses ignore unknown fields automatically:
- Stale keys in config files are silently ignored regardless of whether they appear during startup or reload
- No error messages need to be generated for stale keys
- The three duplicate mechanisms can all be removed

## 9. Risks & Mitigations

- **Risk**: Users relying on the error message to know they should clean up their config files won't get feedback anymore. → **Mitigation**: The Pydantic dataclasses already ignore unknown fields silently, so there is no functional difference. If explicit feedback becomes important later, a single centralized deprecation warning can be added rather than maintaining three inconsistent checks.

- **Risk**: A test somewhere expects `ConfigLoadError` to be raised for a specific removed key. → **Mitigation**: Step 1 of implementation searches for this pattern before proceeding. If found, the test needs updating.
