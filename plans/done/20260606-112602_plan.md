# Markdown Context Compression Engine MCP Server — Implementation Plan

## Goal
Implement a dedicated MCP server that compresses large Markdown while preserving meaning and retrieves only the necessary parts for injection.

## Scope
- AST parsing of Markdown files, or heading-aware parsing
- section-level chunk generation
- index construction equivalent to `.mdq/index.sqlite`
- `search_docs(query)`
- `get_chunk(chunk_id)`
- `outline(path)`
- path / tag / heading filters
- BM25 / grep / heading traverse
- incremental indexing
- MCP server implementation
- Agent-side policy update so that `search_docs` is preferred over `read_file`

## Assumptions
- Current ingestion pipeline includes `web_crawler.py`, `chunk_splitter.py`, `rag_ingester.py`
- `chunk_splitter.py` already supports Markdown snippet splitting by heading boundary via `md_index_enable`
- The current Agent assumes HTTP MCP servers using `/v1/call_tool` and `/v1/tools`
- The current `file-mcp` provides `read_text_file`, `search_files`, `grep_files` but not Markdown heading structure or heading-path based partial retrieval
- Introduce a new MCP server: `mdq-mcp` that is separate from `file-mcp` and provides a Markdown-specific index with partial retrieval APIs optimized for large Markdown documents

## Design
- Use heading-aware parsing based on existing `chunk_splitter.py` logic
- Implement SQLite index with dedicated tables for Markdown documents and chunks
- Use FTS5 for search engine
- Implement MCP server with HTTP protocol
- Add new tools: `search_docs`, `get_chunk`, `outline`
- Update agent configuration to prefer `search_docs` over `read_file`

## Validation plan
- Unit tests for heading extraction, outline correctness, incremental indexing, grep/BM25/hybrid search
- Integration tests for index_paths -> search_docs -> get_chunk flow
- Performance tests with large SKILL.md and ADR files
- Verify connectivity through /mcp

## Risks
- Incomplete Markdown parser coverage - addressed by using heading-aware parse in Phase 1
- SQLite index growth - addressed by incremental indexing, content hash comparison, snippet length limits
- Agent keeps using read_file - addressed by adding explicit rule to tool_definitions and system prompt
- Variability in Japanese search precision - addressed by using normalized FTS fields as in current RAG system
