# Plan: M-1 — Remove Tool Result Summarization Config

**Requirement:** `requires/20260708_11_require.md`
**Timestamp:** 20260708-101731

---

## Goal

Remove all traces of the Tool Result Summarization feature from the agent configuration
layer. The two fields `use_tool_summarize` and `tool_summarize_threshold` must be deleted
from the dataclass, builder, reload service, and `/config` display. Existing config files
that still contain those keys must be handled consistently.

---

## Scope

- Remove `ToolConfig.use_tool_summarize` and `ToolConfig.tool_summarize_threshold` from
  `agent/config_dataclasses.py`.
- Remove both fields from `_build_tool_config()` in `agent/config_builders.py`.
- Remove `use_tool_summarize` and `tool_summarize_threshold` handling from
  `ConfigReloadService._apply_tool_params()` in `agent/services/config_reload.py`.
- Remove both lines from `_print_execution_settings()` in
  `agent/commands/cmd_config_display.py`.
- Remove the summarization call-site in `agent/tool_runner.py` (lines 152–166).
- Remove the summarization guard in `agent/tool_result_formatter.py` (`is_summarized()`,
  lines 31–45) — or replace it with a stub that always returns `False` depending on
  downstream callers.
- Update all test files that pass `use_tool_summarize` / `tool_summarize_threshold` as
  fixture fields.
- Decide and implement the handling strategy for old config files (see Unknowns).

Out of scope:
- Removing `summarize_tool_result()` and related HTTP helpers (depends on whether they
  are used elsewhere — see Unknowns).
- Database schema changes.
- Any UI / documentation changes outside the listed files.

---

## Assumptions

- The summarization feature has been confirmed non-functional or intentionally removed at
  the execution layer; removing the config fields completes the cleanup.
- `ToolConfig.__post_init__` does not validate `use_tool_summarize` or
  `tool_summarize_threshold`, so no validator removal is needed.
- The import-layer contract (agent → all layers) is unaffected; no cross-layer imports
  are introduced.

## Unknowns

### U-1: Handling strategy for old config keys

The requirement lists three options:
1. Ignore with warning (log at WARNING, continue).
2. Fail validation (raise `ConfigLoadError` at startup if keys present).
3. One compatibility release with deprecation warnings, then hard-fail.

**Resolution (from source analysis):** A precedent already exists in `build_agent_config()`:

```python
_FORBIDDEN_KEYS = {"workflow_mode", "workflow_require_approval"}
found = _FORBIDDEN_KEYS & set(cfg.keys())
if found:
    raise ConfigLoadError(
        f"Config contains removed keys: {sorted(found)}. "
        "Remove them from your config file."
    )
```

This pattern (option 2 — fail validation with a clear error) is already established.
Applying the same mechanism to `use_tool_summarize` / `tool_summarize_threshold` is
consistent and safe. **Resolution: add both keys to `_FORBIDDEN_KEYS`.**

### U-2: Is `summarize_tool_result()` used anywhere other than `tool_runner.py`?

**Resolution (from source analysis):** A grep of `scripts/` shows the call appears only
in `agent/tool_runner.py`. The helper function itself likely lives in a separate module
(e.g. `agent/tool_summarizer.py` or similar). That module can be deleted or left dormant;
if it is imported only by `tool_runner.py`, it becomes dead code after this change.
Removing the import and the function is recommended but is a low-risk follow-up.

Confirmed callers:
- `agent/tool_runner.py` lines 152–166 (the only active call-site).
- `agent/tool_result_formatter.py` `is_summarized()` — reads the two cfg fields only to
  determine display annotation; with fields removed this function becomes a no-op stub or
  can be deleted.

## Design

### Field removal strategy

1. Delete `use_tool_summarize` and `tool_summarize_threshold` from `ToolConfig` in
   `config_dataclasses.py`.
2. In `_build_tool_config()`, remove the two `cfg.get(...)` lines.
3. In `build_agent_config()`, extend `_FORBIDDEN_KEYS`:
   ```python
   _FORBIDDEN_KEYS = {
       "workflow_mode",
       "workflow_require_approval",
       "use_tool_summarize",        # added
       "tool_summarize_threshold",  # added
   }
   ```
4. In `config_reload.py` `_apply_tool_params()`, delete the two `_apply_bool` /
   `_apply_int` blocks for those keys.
5. In `cmd_config_display.py` `_print_execution_settings()`, remove the two
   `self._out.write(...)` lines that reference `use_tool_summarize` and
   `tool_summarize_thr`.
6. In `tool_runner.py`, replace the summarization branch (lines 152–166) with the
   existing else-branch body only (truncate or pass through), removing the `if`/`else`
   structure around it.
7. In `tool_result_formatter.py`, delete `is_summarized()` and its callers, or replace
   with a stub `return False` if call-sites exist outside the direct change scope.
8. Update all affected tests to remove the two cfg keys; remove test cases that exercise
   summarization behavior directly.

### Backward compatibility

Old config files with `use_tool_summarize = true` or `tool_summarize_threshold = 3000`
will fail at startup with a clear `ConfigLoadError` message — identical behavior to the
already-removed `workflow_mode`/`workflow_require_approval` keys. This is the chosen
policy (fail-closed, explicit error > silent ignore).

## Risks

### R-1: Tests referencing removed fields will fail immediately after field deletion

**Impact:** Medium — CI breaks until tests are updated.
**Mitigation:** Steps 1–6 (source changes) and step 7 (test updates) must be committed
together in a single changeset. Do not commit field deletion without simultaneously
updating tests.

### R-2: `is_summarized()` called from `_collect_tool_result_msgs()` in `tool_runner.py`

**Impact:** Low — `_collect_tool_result_msgs` calls `is_summarized(ctx.cfg, text,
llm_text, is_error)`. After removing cfg fields, the function will crash unless updated.
**Mitigation:** Delete `is_summarized()` and remove the call at line 195 of
`tool_runner.py` simultaneously with the cfg field removal. The `summarized` variable is
only used in `tool_result_store.store(summary=...)`. Replace with `summarized = False`
constant or remove the `summary` argument if the store API allows it.

### R-3: `/reload` path does not go through `_FORBIDDEN_KEYS` check

**Impact:** Low — if a user runs `/reload` while their config file still contains the old
keys, `apply_config_dict` will silently ignore them (keys not present in
`_apply_tool_params`). This is safe but may be confusing.
**Mitigation:** Acceptable behavior. Document in release notes that `/reload` silently
ignores unknown keys; the hard-fail only applies at startup via `build_agent_config`.

### R-4: `summarize_tool_result` import in `tool_runner.py` becomes a dead import

**Impact:** Low — `ruff check` will flag unused import.
**Mitigation:** Remove the import of `summarize_tool_result` when removing the call-site.
