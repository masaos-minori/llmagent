## 1. Goal

`tests/test_mcp_tools_validation.py::mcp_server` フィクスチャが、実サブプロセスの起動・終了を安全かつ確実に行えるようにする（プロセスリークとteardownの`assert`失敗を解消し、統合テストが実際に意図した検証を行えるようにする）。

## 3. Assumptions

1. `mcp_server` フィクスチャが検証している対象（`mcp.shell.server` / `mcp.cicd.server` / `mcp.mdq.server`）は、通常の運用では各自のハードコードされたポート（8009/8012/8013）で起動され、CLIの `--host`/`--port` を受け取る設計にはなっていない。
2. 開発機上でこれら3ポートが常時使用中（本番相当のインスタンスが起動中）であることは稀であり、ポート使用中を検知した場合はテストをスキップする方針で十分安全側に倒せる。
3. `_terminate_process_group` 内の `proc.poll() is not None` による早期return（`tests/test_mcp_tools_validation.py:98-99`）により、フィクスチャが自ら起動した直後のプロセスに対しては、PID再利用競合が発生する余地は実質的にない（起動〜終了までの間に他プロセスがそのPIDを再利用する時間的余地がない）。

## 6. 再現手順（検証済み）

以下を実際に実行し、リグレッションを直接確認した。

```bash
timeout 90 uv run pytest tests/test_mcp_tools_validation.py -k "shell" -v --timeout=80
```

結果: `test_v1_tools_each_has_name_and_description[shell]` が **setup時にエラー**:
```
AssertionError: process group <pid> still has live members after terminate+kill —
MCP server subprocess tree was not fully reaped
```
実行後、`ps` で該当PIDの `mcp.shell.server` プロセスが実際に生き残っていることを確認し、手動で `SIGTERM`→`SIGKILL` して後片付けした（本来はフィクスチャのteardownがこれを行うべき処理）。

原因は二重になっている:
1. `mcp_server` fixture (`tests/test_mcp_tools_validation.py:139`) が `os.killpg` を丸ごと `MagicMock()` に差し替えているため、`_terminate_process_group` 内の `os.killpg(pgid, SIGTERM/SIGKILL)` 呼び出しが**常に「成功」したことになり例外を投げない**。結果として `except (ProcessLookupError, OSError): proc.terminate()` のフォールバック（実際にプロセスを止められる唯一の経路）が一度も実行されない。
2. さらに、`_MCP_SERVERS` の各サーバー (`scripts/mcp/shell/server.py:157-159`、`scripts/mcp/cicd/server.py:134-136`、`scripts/mcp/mdq/server.py:302-304`) の `if __name__ == "__main__":` ブロックは `sys.argv` を一切参照しない。フィクスチャが渡す `--host`/`--port`（`_find_free_port()` で動的に採番したポート）は無視され、サーバーは常にハードコードされた `http_port`（8009/8012/8013）でbindする。そのためヘルスチェック（動的ポート宛）は必ずタイムアウトし、`pytest.skip` に落ちる分岐（`tests/test_mcp_tools_validation.py:163-167`）を経由するが、そこでも同じ `_terminate_process_group` を呼ぶため上記1のバグを踏む。

つまり現状、`mcp_server` フィクスチャを使うテストは **必ず** teardown（またはskip直前のcleanup）で失敗し、かつ実サブプロセスをリークする。`pyproject.toml` の `addopts = "-v --tb=short"` には `-m "not integration"` が含まれていないため、`uv run pytest`（フルスイート）実行時は常にこの経路を踏む。

## 9. Risks & Mitigations

- **Risk**: Phase 2でハードコードポートに固定した結果、CI/開発機で当該ポートが真に使用中の場合にテストがskipされ続け、実質検証されなくなる → **Mitigation**: skip時に理由（ポート番号・"already in use"）を明示するメッセージを出し、CI実行ログで可視化する。恒常的にskipされる場合は別途調査する。
- **Risk**: Phase 1でモックを外した結果、`_terminate_process_group` が実際に `os.killpg` を呼ぶようになるため、テスト実行環境によっては予期せぬ既存プロセス（同名PGIDを共有する無関係プロセス）に影響する可能性が理論上ゼロではない → **Mitigation**: `proc.poll()` による直前の生存確認、`os.getpgid(proc.pid)` の都度取得（キャッシュしない）という既存ガードで実質的にカバーされている。念のためテスト実行前後で `ps` により対象プロセス以外への影響がないことを1回手動確認する。
- **Risk**: issue 1-bで指摘された `asyncio.to_thread(proc.wait)` の非デーモンスレッドリーク（SIGKILLが真に効かないD-state等の稀なケース）は、本計画では対応しない残留リスクとして残る → **Mitigation**: 発生確率は低く（SIGKILLはユーザー空間から捕捉不能）、issue自身も具体策を提示していないため対応保留とし、次回issue化する。
- **Risk**: UNK-01（CLI引数対応 vs ハードコードポート案）の判断を誤ると、将来的にテストの並列実行時にポート競合が起きる可能性がある → **Mitigation**: 本計画はシングル実行を前提としたB案を採用。並列実行が要件化された時点でA案（`MCPServer.main()`共通化）を再検討する。
