# Plan: セッションID生成の同時実行テストの追加 — test_agent_session.py

## Goal

`tests/test_agent_session.py` に `TestSessionIdConcurrency` クラスを新設し、
`ThreadPoolExecutor` で複数スレッドから並行して `AgentSession.start()` を呼び出した際に、
各セッションが一意の `session_id` を取得することを検証する。

## Scope

**In-Scope:**
- `tests/test_agent_session.py` — `TestSessionIdConcurrency` クラスの新設

**Out-of-Scope:**
- `scripts/agent/session.py` の実装変更
- 他のテストクラスの変更

## Assumptions

1. `start()` は `INSERT INTO sessions (title) VALUES (NULL)` + `cur.lastrowid` で session_id を取得する。
   SQLite AUTOINCREMENT はシリアライズされた書き込みを保証するため、一意性は SQLite 側で担保される。
2. 並行テストでは `check_same_thread=False, timeout=5` で単一の in-memory 接続を共有する。
3. 各スレッドに独立した `AgentSession` インスタンスを割り当てる (session_id はインスタンスフィールド)。
4. `agent.session.SQLiteHelper`・`agent.session_message_repo.SQLiteHelper`・
   `agent.note_repo.SQLiteHelper` の 3 つをパッチする (既存 `session` fixture と同様)。
5. `DiagnosticStore` は `session_id` を取るが、テスト用 DB にそのスキーマが必要なければパッチする。

## Design

### 共有接続ヘルパー

```python
def _make_concurrent_session_factory(conn):
    """Returns a factory function that patches SQLiteHelper with conn."""
    def _make(target="rag"):
        return _FakeSQLiteHelper(conn)
    return _make
```

### TestSessionIdConcurrency

**`test_concurrent_starts_produce_unique_ids`**:
1. `conn = sqlite3.connect(":memory:", check_same_thread=False, timeout=5)` で接続
2. スキーマを適用
3. `patch.start()` で SQLiteHelper の 3 箇所をパッチ
4. `ThreadPoolExecutor(max_workers=4)` で 8 スレッドそれぞれに独立した `AgentSession` インスタンスを作成し `start()` を呼ぶ
5. 各インスタンスの `session_id` を収集
6. 期待:
   - 全 8 件が `int` 型
   - `len(set(ids)) == 8` (全て一意)
7. `patch.stop()` で後処理

**`test_concurrent_starts_all_persisted`**:
1. 上記と同様のセットアップ
2. 全スレッド完了後: `SELECT count(*) FROM sessions` == 8
3. 全件が実際に DB に存在することを確認
