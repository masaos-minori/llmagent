# Work Plan: MCP サーバ群 fail-fast 化・型強化・責務分離

## Goal

MCP service 層から `_get_cfg()` キャッシュ、`HTTPException` 使用、`[ERROR]` 文字列返却、`except Exception` を全面排除し、型付き config dataclass + domain 例外 + constructor injection に統一する。

## Scope

**In scope:**
- `scripts/mcp/` 配下の全 MCP サーバモジュール（8 servers × models/service/server）
- 共通モデル `scripts/mcp/file/common.py`
- Config dataclass 定義と domain 例外クラスの追加
- `_get_cfg()` の廃止と constructor injection への変更
- `service.py` からの `HTTPException` 除去と domain 例外への置換
- `git/service.py` からの `[ERROR]` 文字列返却廃止
- `except Exception` の個別例外型への置換
- 対応する server.py への exception handler 追加

**Out of scope:**
- `scripts/shared/` 配下の共通モジュール（前回コミット済み）
- HTTP サーバの起動ロジック (`mcp/server.py` の stdio ハンドラ)
- config ファイル自体の変更（TOML/JSON）
- デプロイスクリプトの変更
- テストファイルの新規作成（既存テストは regression 確認のみ）

## Assumptions

1. `mcp/github/models.py` の `GitHubConfig.load()` パターンが標準パターンとして採用されている
2. domain 例外 → HTTP 変換は各 server.py の `@app.exception_handler` で行う（FastAPI の標準機能）
3. 既存テストは変更後の型シグネチャに合わせて修正可能（例: `cfg: dict[str, Any]` → `XxxConfig`）
4. `_LazyXxxService` lazy singleton パターンは各 server に存在し、config injection で初期化される
5. `mcp/file/common.py` は file サーバ間で共有されるため、domain 例外はここで定義するか共通モジュールとして配置する

## Design

### Domain 例外クラス設計

各 MCP サーバごとに domain 例外を定義する。GitHub パターンに倣う：

```python
class CicdAuthorizationError(RuntimeError): ...    # HTTP 403
class CicdNotFoundError(RuntimeError): ...         # HTTP 404
class CicdValidationError(ValueError): ...         # HTTP 400/422
class CicdUpstreamError(RuntimeError): ...         # HTTP 5xx

# File servers (shared in common.py)
class FileAuthorizationError(RuntimeError): ...
class FileValidationError(ValueError): ...

# Git
class GitServiceError(RuntimeError): ...

# Web search
class SearchUpstreamError(RuntimeError): ...
```

### Config dataclass 設計

GitHubConfig に倣ったパターン：

```python
@dataclasses.dataclass
class CicdConfig:
    auth_token: str = ""
    api_url: str = ""
    # ... other fields from cicd.toml

    @classmethod
    def from_dict(cls, d: dict[str, Any]) -> "CicdConfig":
        return cls(
            auth_token=str(d.get("auth_token", "")),
            api_url=str(d.get("api_url", "")),
            # ... explicit conversion for each field
        )

    @classmethod
    def load(cls) -> "CicdConfig":
        """Load from cicd.toml; raises on failure (fail-fast)."""
        return cls.from_dict(ConfigLoader().load("cicd.toml"))
```

### Constructor injection 設計

```python
# Before:
class CicdService:
    def __init__(self, cfg: dict[str, Any]) -> None:
        self._cfg = cfg

# After:
class CicdService:
    def __init__(self, cfg: CicdConfig) -> None:
        self._cfg = cfg
```

### Exception handler 設計

各 server.py に追加：

```python
@app.exception_handler(CicdAuthorizationError)
async def _cicd_auth_error(request: Request, exc: CicdAuthorizationError) -> JSONResponse:
    return JSONResponse({"error": str(exc)}, status_code=403)

@app.exception_handler(CicdValidationError)
async def _cicd_validation_error(request: Request, exc: CicdValidationError) -> JSONResponse:
    return JSONResponse({"error": str(exc)}, status_code=400)
```

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| 既存テストが新しい型シグネチャで失敗する | high | Step 8 で全テストを実行し、失敗した場合は test fixture を更新 |
| config TOML に新しい key が不足している | medium | `from_dict()` で default 値を設定し、missing key は None ではなく安全なデフォルト値を使用 |
| lazy singleton の初期化タイミングで循環 import が発生する | low | `TYPE_CHECKING` ガードと遅延 import を使用 |
| exception handler の順序が誤って意図しない例外変換を行う | medium | specific exception → base exception の順に handler を登録 |
| `mypy` が dataclass のフィールド型でエラーを出す | medium | `from_dict()` で明示的な型変換を全てのフィールドに追加 |
