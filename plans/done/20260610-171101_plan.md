# RAG パイプライン fail-fast 化・型強化・責務分離 作業計画

## Goal

`rag/` モジュール全体を fail-fast・厳密型付け・責務分離の原則で再構成し、`except Exception` による暗黙の失敗吸収と `Any`/loosely-typed インタフェースを廃止する。

---

## Scope

**In:**
- `scripts/rag/types.py` — 型追加・RagHit 分割
- `scripts/rag/stage.py` — stage result model 追加
- `scripts/rag/utils.py` — `validate_url` の `except Exception` 廃止
- `scripts/rag/repository.py` — Sudachi fallback 廃止・row model 導入・DB 失敗 fail-fast 化
- `scripts/rag/llm.py` — `_get_cfg` fallback 廃止・broad exception 除去・応答 strict parse
- `scripts/rag/pipeline.py` — safe wrapper 廃止・SQLiteHelper 直接利用排除・段階分離
- `scripts/rag/stages/mqe.py`, `search.py`, `fusion.py`, `rerank.py`, `augment.py` — `except Exception` 除去・typed stage result 対応
- `tests/test_rag_pipeline.py`, `test_rag_pipeline_stage.py`, `test_rag_stages.py`, `test_rag_utils.py`, `test_rag_get_cfg.py` — 異常系テスト追加

**Out:**
- DB スキーマ変更なし
- `scripts/agent/` の内部実装変更なし（ただし `RagHit` 型変更の影響がある場合は最小限の修正のみ）
- `scripts/mcp/rag_pipeline/service.py`: 例外が伝播する場合のみ最小限の `sqlite3.OperationalError` / `RagPipelineError` catch を追加（それ以外は変更なし）
- 新規 MCP サーバー追加なし
- `docs/` 更新は実装完了後の別タスク扱い

---

## Assumptions

1. `RagHit` は現在 `TypedDict, total=False` — 後方互換性維持なしで分割・変更可能。
2. `SQLiteHelper().open()` の失敗は `sqlite3.OperationalError` / `sqlite3.DatabaseError` として捕捉できる。
3. Sudachi (`sudachipy`) は optional dependency のまま — ただし未インストール時は fallback ではなく明示的な `ImportError` として扱う。
4. `_get_cfg()` の module-level cache パターンは廃止し、config を constructor 注入または呼び出し元が解決する形にする。
5. `PipelineContext.observers` は現在未使用であり、typed callback への変更は後方互換影響なし。
6. `scripts/mcp/rag_pipeline/service.py` は `RagHit` と `RagPipeline` を import しているため、型変更に伴う最小限の修正が必要になる場合がある。

## Implementation Steps

### Step 1: 型と失敗契約の固定 (`types.py`, `stage.py`)

**変更内容:**
- `RagHit` を 4 段階型に分割:
  - `RawHit` — vector/FTS 検索直後 (chunk_id, content, url, title + distance or bm25_score)
  - `MergedHit` — RRF 後 (+ rrf_score)
  - `RankedHit` — rerank 後 (+ rerank_score)
  - `RagHit` を上記 3 型の Union alias として保持 (既存 import 互換)
- `RagQuery`, `PipelineStageResult` dataclass 追加
- `PipelineContext` の `list[Any]` → `list[RawHit]` / `list[MergedHit]` / `list[RankedHit]`
- `observers` を `list[Callable[[str, PipelineStageResult], None]]` に

**完了条件:** `uv run mypy scripts/rag/types.py scripts/rag/stage.py` no new errors

---

### Step 2: `utils.py` の fail-fast 化

**変更内容:**
- `validate_url`: `except Exception` → `except ValueError` (urlparse は ValueError を送出しない — try 不要; 削除して直接 return)
- `_validate_float_list`: 全要素を `isinstance(v, int | float)` で検証; NaN/inf を `math.isfinite` で拒否
- `floats_to_blob`: 入力 NaN/inf を明示的に拒否

**完了条件:** `uv run pytest tests/test_rag_utils.py` pass; NaN/inf/non-numeric テスト追加

---

### Step 3: `repository.py` の fail-fast 化

**変更内容:**
- `_build_fts_tokens_ja`: `except Exception` を `except (sudachipy.MorphemeList, RuntimeError)` に限定; non-ASCII fallback を削除して `ImportError`/`RuntimeError` をそのまま送出
- `_get_sudachi_tokenizer`: `ImportError` はそのまま送出 (optional dep)
- `RagRepository.vector_search` / `fts_search`: rows を `RawHit` 型の dataclass/TypedDict に変換; 空 rows と DB exception を区別
- `fts_search`: `sqlite3.OperationalError` を捕捉して `[]` に変換するパターンを廃止し、呼び出し元に送出
- `SemanticCache.lookup`/`put`: 次元不一致を `ValueError` で拒否
- `_dedup_hits`/`deduplicate_chunks`: 型を `list[RawHit]` / `list[MergedHit]` に合わせる

**完了条件:** `uv run pytest tests/test_rag_pipeline_stage.py tests/test_rag_stages.py` pass

---

### Step 4: `llm.py` の再構成

**変更内容:**
- `_get_cfg()`: module-level cache + `except Exception` + `{}` fallback を廃止; `_get_cfg()` 自体を削除し、`RagLLM.__init__` / module-level 関数の呼び出し元が `llm_url`/`embed_url` を渡す形に
- `_parse_mqe_response`: JSON 不正時は `[original_query]` 返却をやめ `MqeParseError` を送出
- `get_embedding`: `{}` fallback なし; `ValueError` を送出
- `RagLLM.expand_queries`: `except Exception` を廃止; `httpx.HTTPStatusError`, `httpx.RequestError`, `orjson.JSONDecodeError`, `MqeParseError` のみを捕捉して `RagExpansionError` に wrap して送出
- `RagLLM.cross_encoder_rerank`: 同様に broad exception を廃止; fallback を廃止して失敗時は `RagRerankError` を送出
- `RagLLM.summarize_tool_result`: `except Exception` を廃止; 失敗時は送出 (呼び出し元で判断)
- `RagLLM.refine_context`: 変更なし (既に raise する設計)

**完了条件:** `uv run pytest tests/test_rag_get_cfg.py` pass; LLM 応答不正テスト追加

---

### Step 5: `pipeline.py` と stages の段階分離

**変更内容:**
- `expand_queries_safe()` 廃止; `MqeStage` が失敗時は `RagExpansionError` を `PipelineStageResult` に記録して上位に返す
- `search_queries()` の `except Exception` 廃止; embedding 失敗は `PipelineStageResult.failure_reason` に
- `rerank_candidates()` の `except Exception` 廃止; `RagRerankError` を記録して上位へ
- `augment()` の `SQLiteHelper().open()` 直接利用を廃止; repository factory または注入された db を使用
- `_augment_http()`, `_augment_refiner()` の `except Exception` を具体的な例外型に置換
- `_get_cfg()` 廃止; `_llm_url` / `_embed_url` は constructor 注入または `cfg` 経由
- `stages/mqe.py` の `except Exception` 廃止
- `stages/search.py` の `db._http` 非標準アクセスを `http` inject に変更; `except Exception` 廃止
- `stages/rerank.py` の `except Exception` 廃止
- `stages/augment.py` の `rag.pipeline` 循環 import 解消 (`sanitize_document` を `utils.py` または `repository.py` へ移動)

**完了条件:** `uv run pytest tests/test_rag_pipeline.py tests/test_rag_stages.py` pass

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| `RagHit` 分割が `agent/` や `mcp/rag_pipeline/service.py` の呼び出し元を破壊する | medium | Step 1 で `RagHit = Union[RawHit, MergedHit, RankedHit]` として互換 alias を維持; agent/ への影響は最小限 |
| `fts_search` の `sqlite3.OperationalError` を上位に送出すると既存の `mcp/rag_pipeline/service.py` が未処理例外でクラッシュ | high | `pipeline.augment()` または `service.py` の `run_pipeline()` で `sqlite3.OperationalError` / `RagPipelineError` を明示的に捕捉してエラーレスポンスを返す (Step 5 と同時) |
| `_get_cfg` 廃止で `llm_url`/`embed_url` が未設定のまま実行されると `httpx.RequestError` が多発 | medium | constructor 注入で `llm_url` が空の場合は `ValueError` を即時送出 |
| stages/ の `except Exception` 廃止後、未テストの例外パスが本番で顕在化 | medium | Step 7 の異常系テストで事前カバレッジ確保 |
| Sudachi の `ImportError` 伝播が日本語クエリ全体を失敗させる | low | `ImportError` 発生時のみ FTS を skip し、vector search のみで継続する設計を pipeline layer で持つ |
