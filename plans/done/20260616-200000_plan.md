# Plan: Fix McpServerHealthRegistry state transitions

## Goal

Make `McpServerHealthRegistry` actually track transport failures and successes so that repeated failures transition a server from `HEALTHY` to `DEGRADED` to `UNAVAILABLE`, and successful calls reset it to `HEALTHY`.

## Scope

**In scope:**
- `shared/tool_executor.py` — call `record_failure()` / `record_success()` in `_raw_execute()`
- `shared/mcp_config.py` — (no changes expected; registry already has working methods)

**Out of scope:**
- Health check endpoint logic
- Transport-level retry logic
- UI/dashboard for health states
- Tool-level error handling (server responds OK with is_error=true in response body)

## Assumptions

1. Only transport-layer failures (network errors, timeouts, process crashes) should increment the failure counter. Tool-level errors (server responds with `is_error=true` in JSON response) must NOT affect health state.
2. Any successful transport response (even if the tool itself returned an error) resets the failure counter to 0 (HEALTHY).
3. `is_unavailable()` blocking dispatch is already implemented and working; only the failure/success recording is missing.

## Implementation Steps

1. **Add `TransportError` exception class** in `shared/tool_executor.py` — a lightweight marker exception raised by transports for transport-level failures only.
2. **Refactor `HttpTransport.call()`** — raise `TransportError` instead of returning `ToolCallResult(is_error=True)` for httpx exceptions. Keep existing error message formatting.
3. **Refactor `StdioTransport.call()`** — raise `TransportError` instead of returning `ToolCallResult(is_error=True)` for timeout/OS errors. Keep existing error message formatting.
4. **Update `_raw_execute()`** — wrap `transport.call()` in try/except:
   - On `TransportError`: call `record_failure(server_key)`, build error result, return it.
   - On success (any response from transport): call `record_success(server_key)`, return the result as-is.
5. **Run all validations** — ruff, mypy, lint-imports, tests.
