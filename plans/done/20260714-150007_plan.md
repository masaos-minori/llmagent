## Design

### Error Classification Model

Based on implementation inspection, classify each error type:

```
HTTP 429 (Too Many Requests):     Retryable — backoff
HTTP 502 (Bad Gateway):           Retryable — backoff
HTTP 503 (Service Unavailable):   Conditional — depends on response headers
HTTP 504 (Gateway Timeout):       Retryable — backoff
Timeout:                          ??? — needs implementation inspection
Connection Refused:               Non-retryable — server down
Malformed Response:               Non-retryable — data corruption
```

### Watchdog Decision Order (confirmed)

```
1. Server unreachable:
   - subprocess mode: restart candidate
   - persistent or external mode: warning only

2. Reachable and operator_action_required=true:
   - no restart
   - warning only

3. Reachable and restart_recommended=true:
   - subprocess mode: restart candidate
   - persistent or external mode: warning only

4. Reachable and degraded but restart_recommended=false:
   - no restart
   - warning only

5. Healthy:
   - no action
```

### Health Status Edge Cases

```
HTTP 503 + operator_action_required=true:   Warning only, no restart
HTTP 503 + restart_recommended=true:        Subprocess: restart candidate; Persistent: warning only
HTTP 503 + both flags false:                Warning only, no restart
Unreachable subprocess server:              Restart candidate
Unreachable persistent server:              Warning only
```

### /mcp status Display Fields vs Lifecycle State vs Watchdog Action

```
/mcp status display fields:    Current observable state (healthy/degraded/unreachable)
Lifecycle state:               Internal state machine transition (running/stopped/restarting)
Watchdog restart action:       Whether watchdog will attempt restart based on decision rules
```

These are independent concepts that must not be conflated in documentation.

### Change Strategy

1. **Inspect implementation** to determine actual retry behavior for each error type.
2. **Fix contradictions** in error classification tables.
3. **Document watchdog decision order** consistently across all affected docs.
4. **Add explicit edge case documentation** for HTTP 503 scenarios.
5. **Clarify** the distinction between display fields, lifecycle state, and watchdog action.
6. **Add tests** for timeout retry behavior and watchdog decision branches.

### Consistency Rules

- After changes, every mention of retry behavior must match implementation exactly.
- Every mention of watchdog restart must follow the strict priority order.
- No document should conflate `/mcp status` display with lifecycle state or watchdog action.
- `operator_action_required=true` must be documented for degraded responses too, not just HTTP 200 cases.
