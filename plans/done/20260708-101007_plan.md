## Unknowns

### U-1: `_collect_tool_result_msgs` の `result_id` 用途

**状況:** `tool_runner.py` L.196-210 で `result_id = ctx.tool_result_store.store(...)` の戻り値を `_apply_turn_char_limit` に渡している。`tool_result_store` を削除すると `result_id` が得られなくなる。

**分析結果:**
- `_apply_turn_char_limit` は `result_id` を hint 文字列に埋め込むだけ (`id=<result_id>`)。
- `tool_result_store` を削除した場合、`result_id` は定数 (`0` や `-1`) またはカウンタで代替可能。
- **解決策**: `result_id = 0` (または削除してヒント文字列から `id=` 部分を除去) で代替。

**=> 解決: result_id を 0 固定とし、_apply_turn_char_limit のシグネチャは維持する。**

### U-2: `_ToolingMixin._tool_list` / `_tool_show` の代替

**状況:** `cmd_tooling.py` で `/tool list` / `/tool show` が `ctx.tool_result_store` に依存。削除後は `/tool` コマンドが機能しなくなる。

**分析結果:**
- `ToolResultStore` 自体は残るため、`cmd_tooling.py` が直接 `ToolResultStore()` をインスタンス化して使用する方法がある。
- ただしそれは「AgentContext から取り除く」だけであって「ToolResultStore を使わない」ではない。
- **解決策**: `_ToolingMixin` 内で `ToolResultStore()` を直接使うか、コマンド全体を削除するかは仕様次第。

**=> 要確認 (U-2 は未解決)**: require に「cmd_tooling.py から ctx.tool_result_store 参照を削除する」と明記されているが、`/tool` コマンドの代替動作については記載なし。

### U-3: `undo_service.py` の `n_artifacts_marked` 戻り値

**状況:** `undo_last_turn()` は `UndoResult(n_removed, n_artifacts_marked)` を返す。`tool_result_store` 削除後、`n_artifacts_marked` は常に 0 になる。

**分析結果:**
- `UndoResult.n_artifacts_marked` はテストで検証されている。
- 常に 0 を返すか、フィールドを削除するかの判断が必要。
- **解決策**: `n_artifacts_marked=0` 固定で問題なし。`UndoResult` モデルの変更は不要。

**=> 解決: n_artifacts_marked=0 固定。**

### U-4: `repl.py` の `tool_result_summary` ブロック

**状況:** `_persist_session_diagnostics` (L.161-189) が `ctx.tool_result_store` を参照するが、実際には直接 `SQLiteHelper("session")` でクエリしている (L.176)。`tool_results` 変数は `None` チェックのガードにしか使われていない (L.174)。

**分析結果:**
- L.161: `tool_results = ctx.tool_result_store` — 取得
- L.174: `if session_id is not None and tool_results is not None:` — ガード
- ガード条件を `if session_id is not None:` に変更するだけで削除できる。

**=> 解決: L.161 の代入と L.174 の `and tool_results is not None` を削除。**

## Design

### 方針

`AgentContext` から `tool_result_store` フィールドを削除し、各呼び出し箇所の動作を以下の方針で処理する:

| 呼び出し箇所 | 処置 |
|---|---|
| `tool_runner._collect_tool_result_msgs` | `store()` 呼び出し削除。`result_id = 0` で代替 |
| `llm_transport_errors.handle_partial_completion` | `tool_result_store.store()` ブロック全体削除 (try/except 含む)。`diagnostic_store` への書き込みは維持 |
| `error_injection_service.inject_mid_turn_error` | `ctx.tool_result_store.store(...)` 行削除。`diagnostics.save(...)` は維持 |
| `undo_service.undo_last_turn` | `if turn_to_mark > 0 and ctx.tool_result_store is not None:` ブロック削除。`n_marked = 0` 固定 |
| `cmd_tooling._tool_list` / `_tool_show` | `ctx.tool_result_store` → `ToolResultStore()` に置き換え (直接インスタンス化) |
| `repl._persist_session_diagnostics` | `tool_results = ctx.tool_result_store` 削除、ガード条件から `and tool_results is not None` を削除 |

### U-2 の解決方針

`cmd_tooling.py` は引き続き `/tool list` / `/tool show` を提供するが、`ctx.tool_result_store` の代わりに `ToolResultStore()` をローカルにインスタンス化して使用する。これにより:
- `AgentContext` からの依存は断ち切れる
- `/tool` コマンドの機能は維持される
- importlinter の層契約: `agent` → `db` は許可されているため問題なし

## Risks

### R-1: `test_undo_artifact_consistency.py` の大幅改修

**リスク:** このテストファイルは `undo_last_turn` と `ToolResultStore` の整合性を検証するテストだが、`ctx.tool_result_store` 削除後は `n_artifacts_marked` が常に 0 になり、ほぼ全テストケースの期待値が変わる。

**緩和策:** `undo_last_turn` がアーティファクトを mark しなくなる変更を明示するコメントをテストに追加し、`n_artifacts_marked == 0` を期待値とするように書き換える。テストを削除するのではなく書き換えて `undo` の動作は引き続き確認する。

### R-2: `/tool list` / `/tool show` で直接 `ToolResultStore()` を使う場合の DB パス依存

**リスク:** `cmd_tooling.py` が `ToolResultStore()` を直接インスタンス化すると、DB 設定が `SQLiteHelper("session")` ハードコードに依存する。テスト時のパッチが必要になる場合がある。

**緩和策:** 既存の `test_agent_cmd_tooling.py` でのテスト手法を確認し、`ToolResultStore()` を直接インスタンス化する場合も同じパッチ戦略で対応できることを確認してから実装する。

### R-3: repl.py の `tool_result_summary` ブロックは `tool_results` ではなく直接 SQLiteHelper を使う

**リスク:** U-4 の分析通り、実際には `tool_results` 変数は `None` チェックのガード目的にしか使われておらず、実際の DB クエリは `SQLiteHelper` 経由。意図せず guard を緩めてしまう可能性がある。

**緩和策:** ガード条件を `if session_id is not None:` のみとする。`session_id is None` のケースは `AgentSession.session_id` が None であることを示し、DB クエリを安全にスキップできる。

### R-4: Path B 分類 (変更ファイル数 > 3)

**リスク:** 変更ファイルが 7 production + 7 test + 3 doc = 17 ファイルに及ぶ。各ファイルの変更は小さいが、広範囲のリファクタリングとなる。

**緩和策:** ファイルごとに `grep -n tool_result_store` で事前確認してから変更する。`pytest -x` で途中失敗を早期検出する。
