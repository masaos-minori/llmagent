# 改善計画書 — scripts/mcp/ 配下の MCP サーバーを個別にコマンドラインから起動可能にする

## 0. 結論(先出し)

各 `scripts/mcp/<name>/server.py` には `if __name__ == "__main__": server.run_http()` が既にあり単体起動口自体は存在するが、実測の結果、dev venv (`.venv`) では `ModuleNotFoundError: No module named 'mcp.audit'` で必ず失敗することを確認した。原因は自作パッケージ `mcp` が PyPI 公式 Model Context Protocol SDK (`mcp`, dev 依存 `semgrep` 経由で推移的にインストールされる) と名前衝突しているため。恒久対策として `scripts/mcp` を `scripts/mcp_servers` にリネームし、その上でパッケージ外の統一ランチャーを新設する方針を確定した。

## 1. Goal

`scripts/mcp/` 配下に実装された各 MCP サーバーを、dev / prod いずれの環境でも個別にコマンドラインから確実に起動できるようにする。

## 2. Scope

**In-Scope:**
- `scripts/mcp` → `scripts/mcp_servers` へのパッケージリネーム(名前衝突の根治)
- `scripts/mcp_launcher.py` の新設(パッケージ外の統一起動ランチャー、自動探索レジストリ)
- ポート衝突ガード(起動前の軽量ヘルスチェック)の `mcp_launcher.py` への実装
- `skills/deploy/workflow.md` の Agent restart decision criteria 更新(既存 `[mcp_servers.*]` エントリの `cmd` 等変更時のフルリスタート要件の明記)
- 関連ドキュメント更新(`docs/04_mcp_06_configuration_and_operations.md`、`skills/mcp-server-add/SKILL.md`、`.importlinter`、`AGENTS.md`)

**Out-of-Scope:**
- プロセスロック・PID ファイル等の重い排他制御機構(軽量ヘルスチェックのみ採用。TOCTOU は許容)
- `semgrep` を dev 依存から外す/隔離するなど、名前衝突の別解決策(パッケージリネームを採用するため不要)
- `cicd-mcp` / `rag-pipeline-mcp`(`startup_mode="none"`)固有の依存関係(GITHUB_TOKEN、embed_url)不足時の健全性改善 — 個別起動自体は他サーバーと同様に可能だが、依存未設定時に `/health` が 503 を返すのは既存仕様のままとし変更しない

## 3. Assumptions

1. `shared/config_loader.py::ConfigLoader.restrict_to` の config 探索は `Path(__file__).resolve().parent.parent.parent / "config"` という物理パス基準の絶対パス解決であり、cwd に依存しない。リネーム後もこの挙動は保たれ、個別起動時の config 解決に追加対応は不要。
2. 本番 venv (`/opt/llm/venv`) は `uv sync --system-certs`(dev グループ非インストール)のため `semgrep`/公式 `mcp` SDK が存在せず、現状は名前衝突が表面化していない。今回のリネームは主に開発者のローカル `.venv` での個別起動を成立させることが目的であり、本番動作を変えるものではない。
3. `pyproject.toml` の `[tool.setuptools.packages.find] where = ["scripts"]` はディレクトリ自動検出のため、`scripts/mcp_servers/` へのリネーム後も追加設定は不要。
4. エージェントは MCP サブプロセスをプロセスグループで管理しており(`scripts/agent/http_lifecycle.py` の `start_new_session=True` + `os.killpg()`)、エージェント自身のシャットダウン時に `shutdown_all()` (`http_lifecycle.py:290`) が全サブプロセスを明示的に terminate する。したがって `cmd` パス変更(リネーム)の反映は「エージェント本体の再起動」1回で完結し、10 サーバーを個別に手動再起動する作業は不要。

**呼び出し元・運用面の影響:**
- `config/agent.toml` の `cmd` フィールド変更は `/reload` で反映されない(`docs/04_mcp_06_configuration_and_operations.md` に既存記載)。リネーム反映にはエージェントのフルリスタートが必須。

## 6. Design

### 6.1 パッケージリネーム: `scripts/mcp` → `scripts/mcp_servers`

起動時ラッパーで `sys.path` を場当たり的に矯正する案も検討したが、根本原因が「PyPI の `mcp` と自作パッケージの `mcp` が同名」である以上、リネームが恒久対策として妥当と判断した。

### 6.2 統一起動ランチャー: `scripts/mcp_launcher.py`

```
uv run python scripts/mcp_launcher.py <server_key>   # 個別サーバーを単体起動
uv run python scripts/mcp_launcher.py --list          # 起動可能なサーバー一覧を表示
uv run python scripts/mcp_launcher.py <server_key> --force  # ポート衝突ガードを無視して起動
```

`mcp_servers` パッケージの外(`scripts/` 直下)に置く独立スクリプトとする。理由は上記 5 節「新設ファイル」の通り。

### 6.3 サーバーレジストリ: 自動探索方式

手書き `SERVER_REGISTRY` dict は持たない。`mcp_servers` パッケージ配下を `pkgutil.walk_packages` + `inspect` で走査し、`MCPServer` サブクラスを自動収集する(キーは `server_key` 属性、なければ `server_name` の `-mcp` サフィックスを除去して導出)。走査中に個別モジュールの import が失敗しても他のサーバー探索を止めない(try/except で個別隔離)。

これによりサーバー一覧の管理箇所は実質「各 `server.py` 内のクラス定義そのもの」のみとなり、`config/agent.toml [mcp_servers.*]`(実行時 SSOT)とは目的の異なる独立した情報源として共存する(二重管理にはあたらない)。新規サーバー追加時にランチャー側の更新は不要になる。

### 6.4 ポート衝突ガード

`mcp_launcher.py` はサーバー起動前に、対象ポートへ短タイムアウト(暫定 0.5 秒、UNK-01)で `GET /health` を1回試行する。

- 応答あり → `"port <port> is already responding — <server_key> may be running under the agent. Use --force to start anyway."` を表示して中断(デフォルト拒否)
- 応答なし(接続不可/タイムアウト) → そのまま起動続行
- `--force` で事前チェックをスキップ可能

TOCTOU(チェック直後に他プロセスが起動する競合)は理論上残るが、単発の開発ツールとして許容する。

### 6.5 deploy skill への反映

`skills/deploy/workflow.md` の Agent restart decision criteria(現状 L51-53)に以下を追記する:

```diff
 Restart `llama-agent` ONLY if changes are in:
 - `agent/repl.py`, `agent/context.py`, `agent/config.py`, or any file under `agent/commands/`
 - `config/agent.toml` with a new `mcp_servers` entry (requires full restart)
+- `config/agent.toml` with an existing `[mcp_servers.*]` entry's `cmd`/`url`/`transport`/
+  `startup_mode`/`env` value changed (e.g. package rename affecting launch paths) —
+  `/reload` does not apply these fields; see 04_mcp_06 §Reload vs. restart
```

Phase 3 の Service restart commands 直後に検証手順を追記する:

```diff
 Check status after restart:
 curl -s http://127.0.0.1:<PORT>/health
+
+# cmd パス変更を伴うデプロイ(パッケージリネーム等)の場合は追加で:
+# /mcp status で全 MCP サーバーの PID が再起動後の値に更新されていることを確認
```

## 9. Risks & Mitigations

- **Risk**: 94 ファイルの import 書き換え + config 10 箇所 + `.importlinter` + `AGENTS.md` + docs 多数という大規模変更になり、レビュー負荷が高い → **Mitigation**: `libcst`/`bowler` による機械的一括置換とし、手動編集を最小限にする。`python-refactoring` skill の 6 フェーズツールチェーン(フォーマット→リント→型→アーキ→セキュリティ→テスト)を経由してから 1 PR にまとめる。
- **Risk**: 大規模 `git mv` + 一括置換により、当該範囲の `git blame` 追跡性が下がる → **Mitigation**: リネームコミットと機能追加コミット(ランチャー新設等)を分離し、リネームコミットは「pure rename + mechanical replace」のみに限定する(レビュー時に diff の意味を分離しやすくする)。
- **Risk**: 本番デプロイ時、`config/agent.toml` の `cmd` 変更はエージェントのフルリスタートが必須であり、通常の「影響を受けたサービスのみ再起動」という deploy skill の原則から外れる → **Mitigation**: 7節 Phase 3 で `skills/deploy/workflow.md` の restart 判定基準に明示的な条件を追加済み。エージェントの `shutdown_all()` により全 MCP サブプロセスは自動的に再 spawn されるため、追加の手動操作は発生しない(3節 Assumption 4)。
- **Risk**: `mcp_launcher.py` の自動探索が、依存未設定の `cicd`/`rag_pipeline` モジュールの import 時に何らかの副作用で例外を出す可能性(UNK-02) → **Mitigation**: 走査ループ内で個別モジュールの import を try/except し、失敗しても他サーバーの探索・一覧表示を継続する設計を Phase 2 に織り込み済み。
- **Risk**: ポート衝突ガードは TOCTOU により完全な排他制御にはならない → **Mitigation**: 個別起動は開発者による単発のデバッグ用途であり、悪意ある同時実行や高頻度自動実行は想定しない。実運用上のリスクは無視できるレベルと判断する。
