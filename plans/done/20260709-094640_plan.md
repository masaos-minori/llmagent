# Plan: Replace MCP runtime reload with restart-required classification

Source requirement: `requires/20260708_23_require.md` (H-1)

## Goal

Replace `ConfigReloadService._apply_mcp_url_reload()` with a comparison-only
`_classify_mcp_server_changes()` that reports every MCP server definition
change through `ConfigReloadOutcome.needs_restart` and never mutates
`ctx.cfg.mcp.mcp_servers`.

## Scope

In scope:
- `scripts/agent/services/config_reload.py`: replace `_apply_mcp_url_reload()`
  with `_classify_mcp_server_changes()`; update the call site in
  `apply_config_dict()`.
- `tests/test_config_reload.py`: update `TestDeferredReload` to assert
  no-mutation, restart-only behavior instead of deferred/applied mutation.
- `tests/test_config_reload_classification.py`: update the two
  `patch.object(svc, "_apply_mcp_url_reload", ...)` call sites to the new
  method name.

Out of scope:
- LLM, RAG, tool cache, SSE, memory, and approval reload logic (unaffected by
  this change; acceptance criteria requires them to remain unchanged).
- `scripts/shared/mcp_config.py` / `_build_mcp_servers()` (comparison source,
  not modified).
- `scripts/agent/commands/cmd_config.py` (already renders `needs_restart`,
  `applied`, `deferred` generically; no change needed).
- `tests/test_cmd_config_refactor.py` (constructs `ConfigReloadOutcome`
  directly to test rendering, does not call the classification function).

## Assumptions

1. `cmd_config.py`'s rendering of `applied` / `needs_restart` / `deferred` is
   generic (iterates whatever the lists contain) and requires no change —
   verified by reading `scripts/agent/commands/cmd_config.py:54-101`.
2. `_apply_mcp_url_reload` has no callers outside `config_reload.py` and the
   two test files listed above — verified by
   `grep -rn "_apply_mcp_url_reload" scripts/ tests/`.
3. `McpServerConfig` is a plain `@dataclass` with a generated `__eq__` that
   compares all fields except `key` (`compare=False`), so `old_srv != new_srv`
   is a valid, safe way to detect any definition change — verified by reading
   `scripts/shared/mcp_config.py:50-71`.

## Design

> Revised after planning `requires/20260708_25_require.md` (H-3): that
> requirement's acceptance criteria requires the exact rendered form
> `[RESTART] - mcp/<server_key>.url` (verified against
> `scripts/agent/commands/cmd_config.py:69-76`, which renders
> `needs_restart` items verbatim as `f"  [RESTART] - {item}"`). A bare
> `key` entry cannot produce that string, so entries must be field-qualified.

Replace the current mutate-and-classify function with a pure, field-level
comparison:

```python
_MCP_SERVER_FIELDS = (
    "transport",
    "url",
    "startup_mode",
    "healthcheck_mode",
    "call_timeout_sec",
    "startup_timeout_sec",
    "tool_names",
    "auth_token",
    "role",
    "cmd",
    "env",
)


def _diff_mcp_server_config(old: McpServerConfig, new: McpServerConfig) -> list[str]:
    """Return names of McpServerConfig fields that differ between old and new.

    Pure comparison — never mutates either argument. Field order follows
    _MCP_SERVER_FIELDS, so output is deterministic for a given pair of inputs.
    """
    return [
        field_name
        for field_name in _MCP_SERVER_FIELDS
        if getattr(old, field_name) != getattr(new, field_name)
    ]


def _classify_mcp_server_changes(
    self,
    ctx: AgentContext,
    new_cfg: dict[str, Any],
) -> ConfigReloadOutcome:
    """Classify MCP server definition changes as restart-required, field by field.

    MCP server definitions are restart-time snapshots: ToolExecutor and
    HttpTransport are built from them at startup, so mutating
    `ctx.cfg.mcp.mcp_servers` here would desync already-running instances
    from the reported config. This method only compares; it never writes.
    """
    from agent.config_builders import (
        _build_mcp_servers,  # noqa: PLC0415 — lazy: avoids circular import at module level
    )

    result = ConfigReloadOutcome()
    new_mcp = _build_mcp_servers(new_cfg)
    old_mcp = ctx.cfg.mcp.mcp_servers
    for key, new_srv in new_mcp.items():
        old_srv = old_mcp.get(key)
        if old_srv is None:
            result.needs_restart.append(f"mcp/{key} (new server)")
            continue
        for field_name in _diff_mcp_server_config(old_srv, new_srv):
            result.needs_restart.append(f"mcp/{key}.{field_name}")
    for key in old_mcp:
        if key not in new_mcp:
            result.needs_restart.append(f"mcp/{key} (removed server)")
    return result
```

Call site in `apply_config_dict()`:
```python
result = self._classify_mcp_server_changes(ctx, new_cfg)
```

Notes:
- Entries are field-qualified (`mcp/<key>.<field>`), matching the rendered
  `[RESTART] - mcp/<server_key>.url` form required by H-3 and the naming
  convention the old code already used for `deferred` entries
  (`mcp/{key}.auth_token`).
- Iterating the fixed field tuple covers all 11 fields named in H-2
  (`requires/20260708_24_require.md`), a superset of the three fields the old
  code checked individually.
- No entries are ever added to `applied`, `deferred`, or `skipped` for MCP
  server changes — satisfies "reported only as restart-required."
- The second loop (`for key in old_mcp`) reports removed servers — added
  while planning `requires/20260708_29_require.md` (H-7), which also covers
  renames as remove-old + add-new (no special-cased rename detection needed;
  a rename naturally produces one `(removed server)` entry for the old key
  and one `(new server)` entry for the new key).
- The per-field comparison is extracted into the module-level
  `_diff_mcp_server_config(old, new) -> list[str]` helper (added while
  planning `requires/20260708_33_require.md`, M-1) so it can be unit-tested
  directly, independent of `ConfigReloadService`/`ctx`. It returns bare
  field names (not pre-formatted `mcp/<key>.<field>` strings) because it has
  no `key` parameter; `_classify_mcp_server_changes()` does the
  `mcp/{key}.{field_name}` formatting at the call site. `key`/`removed
  server`/`new server` handling stays in `_classify_mcp_server_changes()`,
  outside the helper, per M-1's explicit instruction.

   - rename the class to reflect restart-only classification (e.g.

   - assert `needs_restart` contains `mcp/<key>.auth_token`,
     `mcp/<key>.startup_mode`, and `mcp/<key>.url` for the corresponding
     changes,
   - assert a server present only in the old config produces
     `mcp/<key> (removed server)` in `needs_restart` (see H-7,
     `requires/20260708_29_require.md`),
   - assert `applied`, `deferred`, and `skipped` contain no MCP entries,
   - assert `old_srv` field values are unchanged after the call (no mutation).
4. In `tests/test_config_reload_classification.py`, rename the two
   `patch.object(svc, "_apply_mcp_url_reload", ...)` targets (lines 90, 99) to
   `_classify_mcp_server_changes`.

## Risks

- **Risk**: Checking all 11 fields individually (instead of the three
  previously-checked fields) could flag restarts for fields that were not
  restart-sensitive before (e.g. `role`, `tool_names`).
  → **Mitigation**: Acceptable per requirement intent ("restart-time
  snapshots") and explicitly required by H-2
  (`requires/20260708_24_require.md`), which lists all 11 fields as
  must-detect. Call out in the PR description so reviewers can confirm this
  is intended.
- **Risk**: `tests/test_cmd_config_refactor.py` still uses an MCP-flavored
  example string (`"mcp/svc.auth_token"`) in a `deferred=[...]` fixture that no
  longer reflects real MCP behavior post-change.
  → **Mitigation**: Out of scope for this change (it tests generic rendering,
  not classification); low risk since the string is just an example key, not a
  behavioral assertion. Optionally relabel in a follow-up for clarity.
- **Risk**: Other reload call sites depending on MCP entries appearing in
  `applied`/`deferred` (e.g. log messages, telemetry) could silently lose that
  signal.
  → **Mitigation**: `grep` confirmed `cmd_config.py` is the only consumer and
  it already branches generically on whichever lists are non-empty.
