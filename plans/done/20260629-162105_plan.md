## 1. Goal
Harden MDQ SQLite connection and transaction management to make concurrent reads during indexing safe, reduce "database locked" errors, and provide FTS5 consistency check/rebuild operations.

## 2. Scope
- **In-Scope**:
  - `_init_db()` in `service.py`: apply WAL mode and busy_timeout at DB-init time
  - `_get_db_connection()` in `service.py`: already sets WAL/busy_timeout — verify isolation level
  - `_index_single_file()` in `indexer.py`: wrap DELETE + INSERT sequence in an explicit write transaction; make it atomic-replace per document
  - `refresh_paths()` in `indexer.py`: ensure `conn.commit()` is inside proper transaction scope; avoid partial commits mid-loop
  - `_delete_file_from_index()` in `indexer.py`: remove trigger drop/recreate pattern; use proper FTS delete inside transaction
  - `fts_consistency_check()` in `service.py`: already implemented; add integrity_check PRAGMA call
  - `fts_rebuild()` in `service.py`: already implemented; verify it uses explicit BEGIN/COMMIT
  - Tests: add/extend unit tests for concurrent read safety, transaction atomicity, FTS consistency, and rebuild path
- **Out-of-Scope**:
  - DB schema changes (no column/table additions)
  - Embedding/vector table handling
  - `_migrate_from_legacy()` (legacy path not production-critical)
  - Summary cache logic

## 3. Assumptions
- SQLite WAL mode is already set in `_get_db_connection()` but NOT in `_init_db()` — the `_init_db()` call uses a bare `sqlite3.connect()` without WAL/busy_timeout.
- The `isolation_level=None` (autocommit) mode is not currently set; Python sqlite3 defaults to deferred transaction mode; explicit `BEGIN IMMEDIATE` is needed for write transactions.
- `_delete_file_from_index()` drops and recreates the `chunks_ad` trigger to work around the FTS ON DELETE trigger — this is fragile under concurrent access and should be replaced with a direct FTS delete.
- `_index_single_file()` commits inside the section loop (line 93), which means a partial index can be written if the loop fails midway — atomicity requires moving `conn.commit()` outside the loop.
- `fts_rebuild()` uses `INSERT INTO chunks_fts(chunks_fts) VALUES('rebuild')` which is the correct FTS5 rebuild command.
- No connection pooling is used; each operation opens and closes its own connection.

## 8. Risks & Mitigations
- **Risk**: Changing `isolation_level=None` (autocommit mode) breaks existing callers that rely on implicit transactions → **Mitigation**: Audit all `_get_db_connection()` callers before changing isolation level; if risk is high, keep autocommit and use `conn.execute("BEGIN IMMEDIATE")` explicitly only in write paths without changing `isolation_level`
- **Risk**: `BEGIN IMMEDIATE` in `_index_single_file` blocks read connections during indexing → **Mitigation**: WAL mode separates reader and writer lock; WAL readers are never blocked by a writer, so this is safe in WAL mode
- **Risk**: Removing trigger drop/recreate in `_delete_file_from_index` requires fetching rowids before deletion → **Mitigation**: Fetch rowids with `SELECT id FROM chunks WHERE doc_id = ?` before DELETE; use them in manual FTS delete; test with multi-chunk documents
- **Risk**: `fts_rebuild` locks the FTS table exclusively → **Mitigation**: WAL mode reduces impact; document in code that this is a maintenance operation and should not be called during active search traffic
