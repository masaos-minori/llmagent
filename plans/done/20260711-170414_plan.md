## Design

### `ToolCallResult` field documentation (scripts/shared/transport_dto.py)

```python
@dataclass(frozen=True)
class ToolCallResult:
    """Typed result from a single tool call execution."""

    output: str  # tool result text, or error message when is_error=True
    is_error: bool
    request_id: str  # x-request-id from HTTP transport; "" for plugin/cache
    server_key: str  # server key that handled the call; "" for plugin tools
    source: str = (
        ""  # "mcp" for MCP tools, "plugin" for plugin tools, "" for cache/error paths
    )
    error_type: str = (
        ""  # "transport" | "tool" | "plugin_contract" | "" (empty on success)
    )
```

### Doc fixes (docs/90_shared_03_02_runtime_and_execution-plugin-and-tool-runtime.md)

Replace:
```
    command_shadows: int
```
with:
```
    command_shadows_rejected: int  # commands rejected due to strict-mode conflict with a builtin
```

Expand line 57's bypass sentence to:
"`@register_tool` handlers are checked by `ToolExecutor.execute()` **before** `ToolRegistry` route resolution, the tool-result cache, the health gate, lifecycle `ensure_ready()`, and `HttpTransport` — a plugin tool never reaches any of these MCP-routing mechanisms."

Add one new sentence near the `PluginLoadError` description: "In `strict_mode=True`, **all** plugins are attempted first; a single aggregated `PluginLoadError` (naming every load failure, tool-conflict rejection, and command-conflict rejection together) is raised only after every plugin has had a chance to load — not on the first failure."
