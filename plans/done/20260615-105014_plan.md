## Design

### Phase 1: 後方互換レイヤ確認・削除

**調査済みで対象なし。** `repl_debug.py`, `context_detection.py`, `_audit_logger_instance`,
config.py/context.py の flat compat コードは全て既に削除済み。Phase 1 は実装不要。

### Phase 2: hidden global state 排除

**調査結果**: `SQLiteHelper` は 117 箇所・20+ ファイルで使われており、全面 DI 化は
工数に見合わない。現 Phase の対象を新規コードに絞る。

主な対象:
- `db/tool_results.py`: `ToolResultStore` が `SQLiteHelper("session")` を直接呼ぶ3箇所を
  コンストラクタ DI（`db: SQLiteHelper | None = None`）に変更し、テスト差し替えを可能にする

### Phase 3: 責務分離

**`tool_executor.py` (597行, 10 クラス/関数) の分割案**:

`ToolRouteResolver` は既に `shared/route_resolver.py` に分離済み。残りの分割:
- `shared/tool_cache.py` (新規) — `_CacheEntry`, `_execute_with_cache` を移動
- `shared/tool_executor.py` (残存) — `ToolCallResult`, `TransportErrorInfo`, `HttpTransport`,
  `StdioTransport`, `LifecycleProtocol`, `is_side_effect`, `format_transport_error`,
  `tool_call_key`, `ToolExecutor` を残す

**`llm_client.py` (622行, 4 クラス)**:

`LLMTransportError`, `RobustSSEParser`, `_anext_or_done`, `LLMClient` の 4 クラス。
クラス数が少なく分割よりも内部整理（メソッド抽出・可視性整理）が適切。
分割は行わない。

### Phase 4: エラー処理統一

- `orchestrator.py` の partial completion / mid-turn transport failure を共通ポリシー化
- `format_transport_error()` は既に `source`, `phase`, `kind`, `url`, `status_code`, `retryable`,
  `partial` の 7 引数を持つ充実実装（`tool_executor.py:369`）。拡充は不要。
  対象: `orchestrator.py` の LLM 失敗ハンドリング統一のみ。
- `git_helper.py` は存在しない（要望書の記載は廃止済みファイルへの言及）。対象外。

### Phase 5: 型・スキーマ強化

**`LLMMessage.role` の Literal 化**:
```python
# 変更前
class LLMMessage(TypedDict):
    role: str  # 任意文字列

# 変更後
from typing import Literal
class LLMMessage(TypedDict):
    role: Literal["system", "user", "assistant", "tool"]
```

**`mcp_config.py` の enum 化** (影響: `.transport ==` 17箇所, `startup_mode ==` 7箇所, 9ファイル):
```python
# 変更前
transport: str      # "http" | "stdio"
startup_mode: str   # "persistent" | "ondemand" | "subprocess"

# 変更後
class TransportType(str, Enum):
    HTTP = "http"
    STDIO = "stdio"

class StartupMode(str, Enum):
    PERSISTENT = "persistent"
    ONDEMAND = "ondemand"
    SUBPROCESS = "subprocess"
```

`str(Enum)` を使うことで既存の `== "http"` 比較も動作するが、
mypy が未対応の比較を警告するため、全 17+7+9 箇所を enum 比較に書き換える必要がある。

影響内訳:
- `transport ==` 17 箇所、`startup_mode ==` 7 箇所、`healthcheck_mode` 9 箇所 (合計 33 箇所)
- `healthcheck_mode` 影響ファイル (3 ファイル): `shared/mcp_config.py` (5箇所),
  `agent/repl_health.py` (1箇所), `mcp/installer_templates.py` (1箇所)
- `transport`/`startup_mode` 影響ファイル (9ファイル): `mcp_config.py`, `lifecycle.py`,
  `cmd_mcp.py`, `factory.py`, `mcp/server.py`, `installer_templates.py`, テスト 3 ファイル

### Phase 6: ルーティング単一ソース化

- `WRITE_TOOLS | DELETE_TOOLS | {"shell_run"}` の固定集合を `ToolSpec.capability` に統合
- `route_resolver.py` の static fallback に strict mode を追加

### Phase 7: ToolSpec 正式化・DAG 実行

- `resource_scope`, `requires_serial` は `tool_runner.py` に既に実装済み
- `ToolSpec` dataclass を `shared/tool_spec.py` に作成して正式化
- 同一 resource への write は自動直列化する実行計画生成

## Implementation Steps

各 Phase は独立してコミット可能。Phase 1 → 2 → 5 → 4 → 6 → 3 → 7 の順を推奨
（影響範囲が小さい順）。

1. ~~**Phase 1**~~ — 調査済みで対象なし。スキップ。

2. **Phase 2** — `db/tool_results.py` の DI 化
   - `ToolResultStore` の `SQLiteHelper("session")` 直接呼び出し 3 箇所をコンストラクタ DI に変更
   - 完了条件: `uv run mypy scripts/` + `uv run pytest tests/test_tool_results*.py` pass

3. **Phase 5** — 型・スキーマ強化
   - `LLMMessage.role` を Literal 化
   - `McpServerConfig` の `transport`, `startup_mode`, `healthcheck_mode` を enum 化（合計 33 箇所）
   - 完了条件: `uv run mypy scripts/` でエラー減少、`ruff check scripts/` 0 errors

4. **Phase 4** — エラー処理統一
   - `orchestrator.py` の LLM 失敗処理を一元化（`format_transport_error()` は拡充不要）
   - `git_helper.py` は存在しないため対象外
   - 完了条件: `uv run pytest tests/test_orchestrator.py` pass

5. **Phase 6** — ルーティング単一ソース化
   - `ToolSpec` メタモデルに side_effect / capability を統合
   - `route_resolver.py` strict mode 追加
   - 完了条件: `uv run pytest tests/test_tool_*.py` pass

6. **Phase 3** — 責務分離
   - `tool_executor.py` から `_CacheEntry`, `_execute_with_cache` を `shared/tool_cache.py` に移動
     （`ToolRouteResolver` は既に `shared/route_resolver.py` に分離済み）
   - `llm_client.py` は分割せず内部整理のみ（4 クラス、分割効果なし）
   - 完了条件: `lint-imports` 0 violations + 全テスト pass

7. **Phase 7** — ToolSpec 正式化・DAG 実行
   - `shared/tool_spec.py` 作成、`ToolSpec` dataclass 定義
   - 完了条件: `uv run pytest tests/test_tool_runner.py` pass
