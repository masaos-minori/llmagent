# Plan: Fix stale `/db rag consistency` hint in `/stats` by adding `/session rag-consistency`

Source requirement: `requires/20260719_01_require.md`
Source issue: `issues/20260717-133541_stale_db_rag_consistency_hint.md`

## Classification

**Path B (Large Task)** — fails the Path A "no public/runtime-facing interface
changes" criterion, since this plan adds a new slash-command action
(`/session rag-consistency`) in addition to the string-only bug fix. All 10
steps run.

## 1. Goal

`/stats` no longer recommends a slash command that the REPL rejects as
unrecognized: its RAG hint points at a working, on-demand
`/session rag-consistency` command that reuses the existing (already tested)
`RagMaintenanceService.consistency()` check, instead of the removed
`/db rag consistency`.

## 2. Scope

**In-Scope**:
- Add `rag-consistency` as a new subcommand key in `_cmd_session`'s
  `db_dispatch` table (`scripts/agent/commands/cmd_session.py`), backed by a
  new `_rag_consistency` method that calls `RagMaintenanceService().consistency()`
  and prints `is_consistent` + `issues` (mirroring the existing inline
  `_db_session_stats` pattern in the same file, which calls
  `DbMaintenanceService().stats()` directly rather than delegating to
  `DbSessionOps`).
- Update the `/session` `CommandDef` help string in
  `scripts/agent/commands/command_defs_list.py` to include
  `rag-consistency` in the subcommand list.
- Update the stale hint string in
  `scripts/agent/commands/cmd_config_stats.py:146` from
  `"Run /db rag consistency for index integrity status"` to
  `"Run /session rag-consistency for index integrity status"`.
- Add regression tests:
  - `tests/test_agent_cmd_session.py` — new `TestCmdSessionRagConsistency`
    class (consistent case, inconsistent case with issues printed).
  - A general regression test (new or added to
    `tests/test_agent_cmd_config.py` / `tests/test_cmd_config_char.py`,
    whichever already covers `_cmd_stats`) asserting the `/stats` hint, when
    printed, references a command name present in `_COMMANDS` — so this
    exact class of drift cannot silently recur.
- Update documentation:
  - `docs/05_agent_07_08_cli-and-commands-slash-commands-session-mcp.md`
    (the `/session stats|health|checkpoint|...` table, line ~51-56) — add a
    `/session rag-consistency` row.
  - `docs/05_agent_07_07_cli-and-commands-migration-notes.md` (line 57) —
    change the `/db consistency` → `後継コマンドなし` row to point at
    `/session rag-consistency`, since a successor now exists.
  - `docs/05_agent_10_04_operations-and-observability-validation-and-troubleshooting-part2.md`
    (line 96) — update the `/stats` hint description to match the new
    string, per `rules/coding.md`'s "Documentation fix required" category
    (the doc was accurate before; it must track the corrected code).

**Out-of-Scope**:
- Any change to `RagMaintenanceService.consistency()`, `check_rag_consistency()`,
  or `RagConsistencyReport`/`RagConsistencyResult` — reused as-is, not modified.
- Any change to `_get_rag_db_configured()` semantics in `cmd_config_stats.py`
  (it only checks that a RAG DB path is configured, not that the index is
  consistent — a distinct, correctly-named concern per the requirement's
  "Notes for implementer").
- Any change to `agent/startup.py`'s automatic startup consistency check
  (step 7) — left untouched; this plan only adds an on-demand path alongside it.
- Adding a `/db` command back in any form, or a new top-level `/rag` command
  namespace (the requirement's alternative option) — `/session rag-consistency`
  is the chosen approach (see Assumptions).
- Any change to `DbSessionOps` (`db_session_ops.py`) — that class is scoped to
  `session.sqlite` via `DbMaintenanceService`; RAG consistency targets
  `rag.sqlite` via a different service, so it does not belong there (matches
  how `_db_session_stats` already bypasses `DbSessionOps` for the same reason).

## 3. Assumptions

1. `/session rag-consistency` (verb-scoped under the existing `/session`
   command) is the correct home for this action, not a new top-level `/rag`
   command — falsifiable by: precedent in
   `docs/05_agent_07_07_cli-and-commands-migration-notes.md`, where the old
   `/db rag <subcmd>` functions that had successors all moved under
   `/session <subcmd>` (e.g. `/db session stats` → `/session stats`), and the
   RAG-scoped `/db rag rebuild-fts`/`/db rag urls` functions that had no
   successor were left with none rather than gaining a new namespace. Since
   this plan does add a successor, following the same target namespace
   (`/session`) is the smaller, more consistent change. The requirement
   document flags a dedicated `/rag consistency` command as an acceptable
   alternative if a different naming philosophy is preferred (see Risks).
2. `RagConsistencyResult.issues: list[str]` is safe to print directly as
   human-readable lines — falsifiable by reading
   `scripts/db/rag_consistency.py`'s `summarize_issues()` (not yet read in
   full; must confirm during Implementation step 1 that it returns
   presentation-ready strings, not raw dataclass reprs).
3. No `SubcommandSpec` entries need to be added to the `/session` `CommandDef`
   in `command_defs_list.py` — falsifiable by: `grep -n "SubcommandSpec"
   scripts/agent/commands/command_defs_list.py` returns only the module
   docstring reference, confirming `subcommands=[]` (default, unused) on every
   existing `CommandDef` today, so only the free-text `help` string needs
   updating, consistent with how `stats|health|checkpoint|vacuum|purge|recover`
   are already listed as plain text rather than `SubcommandSpec` entries.
4. Adding one dict key to `db_dispatch` in `_cmd_session` does not require
   restructuring that method despite its existing `C`-grade cyclomatic
   complexity — falsifiable by: `radon cc` (Step 6 evidence below) shows
   `_cmd_session` at CC 13 today, entirely from the `if sub == ...` chain
   before `db_dispatch`; the dict itself contributes O(1) per entry, not
   incremental branching, so this change does not raise the method's CC grade.

## 8. Risks & Mitigations

- **Risk**: Choosing `/session rag-consistency` over a dedicated `/rag
  consistency` namespace may not match the repo owner's actual naming
  preference (Assumption 1 is a judgment call, not a confirmed decision). →
  **Mitigation**: The requirement doc already states either naming is
  acceptable and explicitly recommends the `/session`-scoped form as most
  consistent with existing precedent; if reviewed and rejected, only the
  dispatch key, help strings, and doc table rows need renaming — the
  `_rag_consistency` method body and its tests are unaffected either way.
- **Risk**: `summarize_issues()`'s actual return format (UNK-01) may not be
  simple flat strings, requiring extra formatting logic not yet designed. →
  **Mitigation**: Implementation step 1 reads that function before any
  handler code is written, so the output-formatting code is written against
  verified behavior, not assumed behavior.
- **Risk**: `_cmd_session`'s existing CC-13 complexity could compound with
  future subcommand additions, eventually crossing the CC-15
  extra-test-coverage threshold from `workflow.md`. → **Mitigation**: this
  plan's addition is a single dict entry (no new `if` branch), verified via
  `radon cc` to not raise the method's grade; if a future subcommand pushes
  it over CC 15, that is a separate refactoring concern, not introduced here.
- **Risk**: Updating `docs/05_agent_07_07_cli-and-commands-migration-notes.md`'s
  `/db consistency` row changes a migration-history document that other docs
  may cross-reference verbatim (e.g. the issue file itself quotes "後継コマンドなし"). →
  **Mitigation**: `tools/check_agent_docs_consistency.py`'s `commanddrift`
  check (run in Validation Plan) is designed to catch exactly this class of
  doc/code drift; also, once `/db rag consistency` is superseded, that issue
  file and requirement become historical records (moved to `done/`) rather
  than live documentation, so a stale quote inside them is expected and not
  a defect.
- **Risk (discovered during this plan's authorship, out of scope but
  adjacent)**: `RagConsistencyResult.issues` (i.e.
  `scripts/db/rag_consistency.py`'s `summarize_issues()`) itself embeds two
  more references to a removed command — `Run '/db rag rebuild-fts' to
  repair` (WARNING, fts_gap) and the same text at CRITICAL severity for
  fts_orphan_count (data-loss-risk case). Since `_rag_consistency` (this
  plan) prints `.issues` verbatim, the new on-demand command will surface
  this same stale guidance whenever an inconsistency is found — the same
  agent-startup path already does today. → **Mitigation**: filed as
  `issues/20260719-091341_rag_consistency_repair_guidance_points_to_removed_db_command.md`
  rather than folded into this plan's scope (keeps the "expose
  `consistency()`" decision and the "expose/rewrite `rebuild_fts()` guidance"
  decision independently reviewable, per this repo's established issue-triage
  discipline). Residual risk: if that issue is not resolved before this plan
  ships, `/session rag-consistency` will faithfully reproduce a pre-existing
  stale-command reference rather than introduce a new one — no regression,
  but the fix is incomplete from the operator's point of view until both
  land. Recommend resolving both together in practice, even though tracked
  separately.
