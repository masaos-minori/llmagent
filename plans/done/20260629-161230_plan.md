## 1. Goal

Align `search_docs` request schema and implementation so every exposed field has real effect and results include sufficient metadata for direct follow-up `get_chunk` calls.

## 2. Scope

- **In-Scope**:
  - Narrow `mode` to `bm25` only; remove or stub `hybrid` (embeddings not implemented)
  - Remove `tag_filter` from `SearchDocsRequest` and tool schema until tag indexing is implemented
  - Verify `heading_prefix` filtering works end-to-end
  - Preserve `path_prefix` filtering (already implemented)
  - Ensure `SearchResultItem` exposes `chunk_id`, `source_path`, `heading_path`, `score`, `start_line`, `end_line`, `token_count`, `snippet` (already present)
  - Fix `chunk_id` generation to be stable (content-addressed, not position-dependent)
  - Add tests: `path_prefix` filter, `heading_prefix` filter, empty results, `chunk_id` stability

- **Out-of-Scope**:
  - Implementing tag extraction in the indexer
  - Implementing hybrid/vector search (depends on separate "stable chunk IDs for MDQ chunks" ticket)
  - MDQ SQLite schema redesign (listed as dependency; proceed without it — schema already has production tables)
  - Changes to `get_chunk`, `outline`, `grep_docs`, or other MDQ tools

## 3. Assumptions

- The production MDQ SQLite schema (`documents`, `chunks`, `chunks_fts`) is already in place; no DDL migration needed for this task.
- `tag_filter` removal is safe because `tags_json` is always stored as `""` (empty string) in the indexer — the filter currently silently returns nothing useful.
- `mode` field can be narrowed to `bm25` only; `hybrid` is advertised in the tool schema description already as "not yet supported" but still present in `SearchDocsRequest`.
- Stable `chunk_id` generation means content-addressed (SHA-256 of `doc_id + heading_path + ordinal`) rather than `doc_id + heading + start_line`, which is fragile when line numbers shift.
- All filter logic lives in `_build_search_where` in `scripts/mcp/mdq/search.py`.

## 8. Risks & Mitigations

- **Risk**: Removing `tag_filter` from Pydantic model is a breaking change if any deployed agent sends `tag_filter` in requests.
  **Mitigation**: Grep `scripts/agent/` and all TOML configs for `tag_filter` before removal. If found, add a deprecated field that is silently ignored rather than hard-removing.

- **Risk**: Changing `chunk_id` generation invalidates all existing `chunk_id` references in agent conversation history or external stores.
  **Mitigation**: Deploy during maintenance window; run `refresh_index --force` immediately after deploy. Document that stored `chunk_id` values must be refreshed.

- **Risk**: `heading_prefix` filter uses `LIKE` on `heading_path` which requires FTS5 join. Incorrect column reference could silently return unfiltered results.
  **Mitigation**: Add `test_heading_prefix_filter` that asserts both positive (matches returned) and negative (non-matching heading not returned) outcomes.

- **Risk**: `_search_vector` function removal (if done) breaks `hybrid` mode gracefully — current code already returns `[]` as placeholder.
  **Mitigation**: Remove only the `if mode == "hybrid"` branch; leave `_search_vector` and `_merge_hybrid` in place as dead code or remove in a follow-up refactor ticket.
