## Goal

Remove `workflow_mode` and `WorkflowExecutionPolicy` entirely so that workflow execution is an unconditional runtime invariant with no configuration path to disable or degrade it.

## Scope

**In**:
- Remove `AgentConfig.workflow_mode` field and `_validate_workflow_mode()` from `scripts/agent/config_dataclasses.py`
- Remove `AgentConfig.workflow_require_approval` field is handled by plan 20260707-095942 (req 05); keep out of this plan
- Remove `scripts/agent/workflow_execution_policy.py` (delete file)
- Remove `workflow_mode` parameter from `Orchestrator.__init__()` and all policy branches in `scripts/agent/orchestrator.py`
- Remove `workflow_mode` read from `scripts/agent/config_builders.py` (line 290)
- Remove `WorkflowExecutionPolicy` usage from `scripts/agent/startup.py` (lines 94, 130)
- Remove `workflow_mode` display from `scripts/agent/commands/cmd_context.py` (line 83)
- Remove `workflow_mode` from `config/common.toml` (line 18)
- Reject config dicts that still contain `workflow_mode` key (raise `ConfigLoadError`)
- Remove and rewrite all tests referencing `workflow_mode`
- Update `deploy/deploy.sh` if `workflow_execution_policy.py` copy line exists

**Out**:
- `workflow_require_approval` removal (handled by req 05 / plan 20260707-095942)
- Approval gate logic changes (req 05)
- Documentation-only updates
- Migration shims or backward compatibility

## Assumptions

1. `workflow_mode="required"` is the sole previously-valid production mode; removing the field makes that the only behavior.
2. `config/common.toml` line 18 (`workflow_mode = "auto"`) must be removed; no other config files in `config/` reference `workflow_mode`.
3. `WorkflowCreationError` docstring reference to "direct-execution fallback is disabled" can be updated as part of this cleanup.
4. `deploy/deploy.sh` copies `workflow_execution_policy.py`; that cp line must be removed.
5. Deleting `workflow_execution_policy.py` is safe once all importers are cleaned.
