**Requirement:** `requires/20260622_74_require.md`

## Assumptions

1. `plugin_strict=False` is the dataclass default; CI auto-detects via `os.getenv("CI")` in `config_builders.py:197`. Production deployments that don't set `plugin_strict=true` in `agent.toml` are intentionally fail-open.
2. `override_policy="reject"` (default, `plugin_tool_override=False`) already removes shadowing tools — the gap is that this removal is silent even in strict mode.
3. `docs/05_agent_11_extension-points.md` line 95 says "By default, plugin tools **shadow** MCP tools" — this is incorrect; the default is rejection. The correct sentence is already given in the Conflict Detection subsection that follows.
4. `_validate_command_conflicts()` correctly logs command shadows but does not (and should not) raise — command shadows are informational, not blocking.
5. `/plugin` command exists in `cmd_plugins.py` but is absent from the CLI commands doc.

## Design

### Change 1: `plugin_registry.py` — `_validate_tool_conflicts()` strict mode

**Before signature:**
```python
def _validate_tool_conflicts(
    known_tools: frozenset[str],
    override_policy: str,
) -> tuple[int, int]:
```

**After signature:**
```python
def _validate_tool_conflicts(
    known_tools: frozenset[str],
    override_policy: str,
    strict_mode: bool = False,
) -> tuple[int, int]:
```

In the `else:` branch (override_policy != "allow"):
```python
else:
    del _tools[tool_name]
    shadowed_count += 1
    logger.info(
        "[plugin] conflict: tool '%s' in '%s' shadows MCP tool — rejected",
        tool_name, module_name,
    )
```

After the loop, add:
```python
if strict_mode and shadowed_count:
    raise PluginLoadError(
        f"Plugin strict mode: {shadowed_count} tool(s) rejected for MCP conflict"
        f" ({', '.join(t for t in known_tools if t not in _tools)})"
    )
```

**Note:** The exact rejected tool names must be captured before the loop since `del _tools[tool_name]` removes them. Refactor to collect `rejected_names: list[str]` in the loop.

**`load_plugins()` change:** Pass `strict_mode` to `_validate_tool_conflicts()`:
```python
shadowed, allowed = _validate_tool_conflicts(known_tools, override_policy, strict_mode)
```

**Ordering concern:** The current code raises `PluginLoadError` for import failures after all plugins are loaded. The tool conflict check runs after import. If BOTH import failures and tool conflicts exist in strict mode, the import failure is raised first (existing behavior). The tool conflict raise in `_validate_tool_conflicts()` would be reached only if there are no import failures (or if we restructure). To keep the aggregated behavior: collect the tool conflict error and add it to `failures` before the final strict-mode raise.

**Revised approach:** Instead of raising directly in `_validate_tool_conflicts()`, return the rejected tool names and let `load_plugins()` aggregate:

```python
def _validate_tool_conflicts(
    known_tools: frozenset[str],
    override_policy: str,
    strict_mode: bool = False,
) -> tuple[int, int, list[str]]:
    """Returns (shadowed_count, allowed_count, strict_rejected_names)."""
    ...
    strict_rejected: list[str] = []
    ...
    if strict_mode:
        strict_rejected.append(tool_name)
    ...
    return (shadowed_count, allowed_count, strict_rejected)
```

In `load_plugins()`:
```python
shadowed, allowed, strict_rejected = _validate_tool_conflicts(
    known_tools, override_policy, strict_mode
)
...
if strict_mode and (failures or strict_rejected):
    parts = [f.error for f in failures]
    if strict_rejected:
        parts.append(
            f"Tool MCP conflicts rejected: {', '.join(strict_rejected)}"
        )
    raise PluginLoadError(
        f"Plugin load failed ({len(parts)} error(s)): {'; '.join(parts)}"
    )
```

This preserves the "load all, then raise once" contract.

### Change 3: `docs/05_agent_11_extension-points.md` — fix sentence

**Before (line 95):**
```
Plugin tools are registered via the `@register_tool()` decorator in plugin files
located in the `plugins/` directory. By default, plugin tools **shadow** MCP tools
when they share the same name — this is because plugin tool lookup in
`ToolExecutor.execute()` runs before MCP routing.
```

**After:**
```
Plugin tools are registered via the `@register_tool()` decorator in plugin files
located in the `plugins/` directory. By default (`plugin_tool_override = false`),
plugin tools that share a name with a known MCP tool are **rejected at startup** —
the conflicting tool is removed from the registry before the session starts.
See Conflict Detection below.
```
