# Plan: Req 30 — Improve token budgeting accuracy beyond chars // 4

## Goal
Replace the inaccurate `chars // 4` fallback token estimator with a category-aware estimator that applies different character-to-token ratios based on message content type (natural language text vs. structured JSON tool calls). Add per-category token breakdown to `/context` output. Improve inline documentation for exact vs estimated token counting.

## Design

### Category-aware token estimation

```
_estimate_tokens(history) → (total_tokens, token_breakdown)

For each message:
  if role == "assistant" and tool_calls:
    → tool_call_json chars / 2.5
  elif role == "system":
    → content chars / 3.5
  else (user, assistant text-only, tool):
    → content chars / 4.0

token_breakdown = {"text": N, "tool_calls": N, "system": N}
```

Rationale for ratios:
- **Text (4.0)**: English natural language averages ~4 chars/token; covers user messages, assistant text responses, and tool result payloads
- **Tool call JSON (2.5)**: Serialized JSON with braces, quotes, field names ("id", "function", "arguments") is more verbose per token
- **System (3.5)**: System prompts often contain code/instructions mixing natural language and structured formats

### API changes

**`shared/token_counter.py`:**
```python
# Before:
def _estimate_chars(history) -> int: ...  # returns char count

# After:
def _estimate_tokens(history) -> tuple[int, dict[str, int]]: ...  # returns (total, breakdown)
# _estimate_chars() retained for backward compat if needed elsewhere
```

**`agent/services/models.py`:**
```python
@dataclass(frozen=True)
class ContextBudget:
    system: int
    history: int
    tool_results: int
    # New optional fields:
    token_system: int | None = None
    token_history: int | None = None
    token_tool_results: int | None = None
```

**`agent/commands/cmd_context.py`:**
- Add token breakdown section below character breakdown when estimate is available
- Update source label from `"chars/4"` → `"category-aware estimate"`

### Backward compatibility
- `_estimate_chars()` function retained (may be used by other callers); `_estimate_tokens()` is new
- `ContextBudget` fields are optional with `None` defaults — existing code that doesn't expect them continues to work
- `HistoryManager.count_tokens()` signature unchanged; only internal fallback logic changes

## Risks

| Risk | Mitigation |
|------|-----------|
| New ratios produce different token counts that shift compression timing | Compression is based on char limits primarily; token limit is secondary. Changes will make it *more* accurate, not less. Monitor via logs. |
| `_estimate_chars()` may have other callers not yet discovered | Search codebase for `_estimate_chars` references before removing; retain function if found |
| Token breakdown display adds noise to `/context` output | Use compact format matching existing budget breakdown style; only show when estimate available |
| Test assertions for exact token counts will break | Tests for fallback path use small inputs where `chars//4` and new estimator may differ — update expected values |
| `orjson.dumps()` length vs actual serialized size | `orjson.dumps()` is deterministic; length is reliable proxy for tokenization input size |
