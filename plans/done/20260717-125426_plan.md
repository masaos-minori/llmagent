# Plan: Use RuntimeToolRegistry for LLM tool schema and help output

Source requirement: `requires/20260717_05_require.md`

## Goal

The LLM receives its tool-calling schema from `RuntimeToolRegistry.llm_tool_definitions()` (requirement 02) instead of static `ctx.cfg.tool.tool_definitions`, so policy-disabled tools are no longer exposed to the LLM; `/help` reports RuntimeToolRegistry-derived tool counts (total discovered, LLM-enabled) instead of `len(ctx.cfg.tool.tool_definitions)`.

## Scope

**In scope**
- `scripts/agent/llm_turn_runner.py`: `_stream_llm()`'s call to `ctx.services_required.llm.stream(llm_url, ctx.conv.history, ctx.cfg.tool.tool_definitions)` (line 213) replaced with `ctx.services_required.runtime_tools.llm_tool_definitions()`.
- `scripts/shared/llm_client.py`: confirm `stream()`'s `tools`/`tool_definitions` parameter accepts the same JSON-schema list shape `llm_tool_definitions()` produces (per requirement 02's Design, already re-keyed to `parameters` for LLM compatibility) — likely no change needed if the shape matches, per Assumption 2.
- `scripts/agent/commands/registry.py`: `_cmd_help()`'s `n_tools = len(ctx.cfg.tool.tool_definitions)` (line 105) replaced with RuntimeToolRegistry-derived counts (total discovered + LLM-enabled), and the help output line (121) updated to show both.
- `tests/test_llm_turn_runner.py` (the requirement's stated path `tests/agent/test_llm_turn_runner.py` doesn't exist — see Assumption 3): update tests asserting `stream()`'s tool-definitions argument.
- Help-related tests: the requirement's stated `tests/agent/commands/test_help.py` doesn't exist — actual `/help`-adjacent assertions live in `tests/test_command_def_sync.py` (confirmed via grep) — update whichever tests assert on `_cmd_help()`'s tool-count output.

**Out of scope**
- Building `RuntimeToolRegistry`/`llm_tool_definitions()` itself — requirement 02.
- Populating the registry at startup — requirement 03.
- Wiring routing/scheduler/policy consumers — requirements 04, 06, 07 (separately tracked; this plan only touches the LLM-schema and help-output consumers).
- Removing `ctx.cfg.tool.tool_definitions` itself from `AgentConfig`/`config/agent.toml` — that's requirement 10's scope ("Redefine static tool_definitions... as compatibility-only"), not this one; this plan only stops *reading* it for these two specific consumers.

## Assumptions

1. `scripts/agent/llm_turn_runner.py:213`'s `ctx.cfg.tool.tool_definitions` is the sole call site inside this file feeding the LLM's tool schema — confirmed via direct grep (single match) — `_stream_llm()` (lines 202-215) passes it as the third positional argument to `ctx.services_required.llm.stream(llm_url, ctx.conv.history, ctx.cfg.tool.tool_definitions)`.
2. `shared/llm_client.py`'s `stream()` method's tool-schema parameter expects the same JSON-schema-array shape currently produced by the static `[[tool_definitions]]` TOML blocks (`name`/`description`/`parameters` per requirement 02's plan) — `llm_tool_definitions()` (requirement 02's method) is designed to produce this exact shape (re-keying MCP's `inputSchema` to `parameters`), so no change to `llm_client.py` itself should be needed; this plan verifies this compatibility during implementation rather than assuming it silently holds.
3. **The requirement's stated test-file paths don't match the current repo layout**: `tests/agent/` and `tests/agent/commands/` do not exist (confirmed via `ls` errors) — the real test file for `llm_turn_runner.py` is `tests/test_llm_turn_runner.py` (flat `tests/` dir). No dedicated `test_help.py` exists; `/help`-related assertions are found in `tests/test_command_def_sync.py` (confirmed via grep: `test_help_output_mentions_all_command_names`, `test_dispatch_help_returns_true`) — this plan targets the real files.
4. `_cmd_help()`'s current tool-count line (`registry.py:105`, `n_tools = len(ctx.cfg.tool.tool_definitions)`, displayed at line 121: `f"Tools: {n_tools} | LLM: {ctx.cfg.llm.llm_url} | {sid}"`) is the only tool-count display in the help output — confirmed via direct grep of `registry.py` for `tool_definitions`/`n_tools`.
5. `ctx.services_required.runtime_tools` (the field added by requirement 02, populated by requirement 03) is available by the time both `_stream_llm()` and `_cmd_help()` run — both are invoked well after startup completes, so this dependency ordering is safe to assume without further verification.

## Design

`_stream_llm()`'s change is a one-line substitution once `RuntimeToolRegistry` is populated (requirements 02-03 land first) — `llm_tool_definitions()` already filters to `enabled_for_llm=True` tools per requirement 02's plan, directly satisfying "policy-disabled tools are not exposed to the LLM" without any additional filtering logic needed in `llm_turn_runner.py` itself.

`_cmd_help()`'s change needs two counts: `len(ctx.services_required.runtime_tools.all_tools())` (total discovered) and `len(ctx.services_required.runtime_tools.llm_tool_definitions())` (LLM-enabled) — both methods already exist per requirement 02's planned `RuntimeToolRegistry` interface, so no new registry method is needed here.

## Implementation steps

1. **Verify `llm_client.py`'s `stream()` shape compatibility** (resolves Unknown #1): read its tools-parameter handling in full; confirm `llm_tool_definitions()`'s output (per requirement 02) matches what it expects.
2. **Update `llm_turn_runner.py`**: replace line 213's `ctx.cfg.tool.tool_definitions` with `ctx.services_required.runtime_tools.llm_tool_definitions()`.
3. **Update `registry.py`'s `_cmd_help()`**: replace the tool-count line with two RuntimeToolRegistry-derived counts (total discovered, LLM-enabled); choose and apply a display format (resolves Unknown #2).
4. **Update `tests/test_llm_turn_runner.py`**: adjust any test asserting on the exact tools-argument passed to `stream()` to use a `RuntimeToolRegistry` fixture instead of static `tool_definitions`.
5. **Update `tests/test_command_def_sync.py`** (or wherever the real `/help`-output assertions live): adjust for the new two-count display format.
6. **Verification**: run the standard validation sequence scoped to changed files, then the full suite.

## Risks

- **Risk**: If `llm_tool_definitions()`'s output shape (built by requirement 02, not yet implemented at planning time) doesn't exactly match what `llm_client.py`'s `stream()` expects, this plan's one-line substitution could silently send a malformed schema to the LLM.
  **Mitigation**: Implementation Step 1 explicitly verifies shape compatibility before the substitution, not after; if a mismatch is found, fix it in `llm_tool_definitions()` itself (requirement 02) rather than adding shape-adaptation code in `llm_turn_runner.py`.
- **Risk**: Changing `/help`'s tool-count display format could break any test asserting on the exact old string (`f"Tools: {n_tools} | ..."`).
  **Mitigation**: Implementation Step 5 explicitly re-checks `tests/test_command_def_sync.py` for such assertions before finalizing the new format.
- **Risk**: This plan assumes `ctx.services_required.runtime_tools` is always populated by the time `_stream_llm()`/`_cmd_help()` run (Assumption 5) — if requirements 02-04 aren't fully landed first, this substitution would crash on a `None`/missing attribute.
  **Mitigation**: this plan must not be implemented before requirements 02-04 land; sequence dependency should be explicit in any implementation tracking (this is an ordering risk inherent to the whole 20-item batch, not unique to this plan).
