MCP ツール呼び出しキャッシュキーを「plain string」に一本化し、MD5ベースの代替実装と混同されないようにコード・ドキュメント・テストを整合させる。

## Background: 調査結果

### キャッシュキーの実装 (verified from source)

**`scripts/shared/tool_executor.py` line 608:**
```python
cache_key = f"{tool_name}:{_json_dumps(args)}"
```
`_json_dumps` = `from shared.json_utils import dumps as _json_dumps` (orjson ベースのシリアライザ)

**`tool_hash_key` (line 384-389):**
```python
def tool_hash_key(name: str, args: dict[str, Any]) -> str:
    """Return a stable MD5 hash for a (tool name, args) pair; used for failed-call tracking
    (NOT for cache keys). Cache keys use plain string concatenation: f'{name}:{json_dumps(args)}'."""
    return hashlib.md5(...).hexdigest()
```
- 用途: `tool_runner.py` (failed_keys セット管理) + `tool_loop_guard.py` (failed call チェック)
- キャッシュには**使用されていない**

**`tool_loop_guard.py` 直接 `hashlib.md5` 呼び出し (lines 70, 113):**
- ラウンドレベルのサイクル検知 (`round_key`) とターンレベルの dedup (`key`)
- どちらもキャッシュとは無関係

## Assumptions

1. `_json_dumps` は `from shared.json_utils import dumps as _json_dumps` — orjson ベース、OPT_SORT_KEYS 付き。`orjson_dumps` は旧名称または古いドキュメント由来の表記。
2. `tool_hash_key` は dead code ではない — `tool_runner.py:141` と `tool_loop_guard.py:157` で `failed_calls` セットのキー生成に使用。削除は適切でない。
3. `tool_loop_guard.py` の直接 `hashlib.md5` 呼び出しは意図的: `tool_hash_key` が使う `_json_dumps(parsed_args)` とは異なり、LLM メッセージの raw `arguments` 文字列をそのまま使うため別ロジックが必要。
