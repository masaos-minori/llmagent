# Plan: M-5 — Align /mcp status health parsing with watchdog health parsing

## Goal

Update `McpStatusService._get_http_status()` to parse the same `/health` body fields as `_probe_mcp_health_detail()` in `repl_health.py` (`restart_recommended`, `operator_action_required`, `reason`, `message`) and populate the corresponding `McpProbeResult` fields, so that `/mcp status` shows a consistent health interpretation with the watchdog.

## Scope

**In**: `scripts/agent/services/mcp_status.py` — use `_probe_mcp_health_detail()` from `repl_health.py` (or extract it to a shared helper), populate `McpProbeResult.restart_recommended`, `operator_action_required`, `health_reason` fields (added by M-4).

**Out**: No changes to `mcp_health.py`, `cmd_mcp.py` renderer structure (display is handled by M-4), `repl_health.py` watchdog logic, `context.py`, or DB schema. No duplication of body parsing — reuse existing function.

## Assumptions

1. `_probe_mcp_health_detail()` in `repl_health.py` is the canonical health body parser; it already parses `restart_recommended`, `operator_action_required`, and captures `body` dict.
2. `reason` and `message` are available via `probe.body.get("reason", "")` and `probe.body.get("message", "")`.
3. M-4 has added `restart_recommended`, `operator_action_required`, and `health_reason` fields to `McpProbeResult`; this plan depends on M-4.
4. `_probe_mcp_health_detail()` handles malformed JSON gracefully (confirmed: falls back to `restart_recommended=False`).
5. The import of `_probe_mcp_health_detail` from `repl_health` into `mcp_status` is acceptable per the import-linter architecture (both are in `agent` layer).

## Implementation Steps

1. **Import `_probe_mcp_health_detail` in `mcp_status.py`**
   - Add `from agent.repl_health import _probe_mcp_health_detail` (or move function to `agent.shared` if import-linter blocks it).
   - Criterion: import resolves; `lint-imports` passes.

2. **Replace `_get_http_status()` with probe detail call in `_resolve_endpoint()`**
   - Instead of calling `_get_http_status()`, call `await _probe_mcp_health_detail(probe, cfg.url)`.
   - Derive `McpAvailability` from probe result:
     - `probe.reachable and probe.status_code == 200` → `OK`
     - `probe.reachable and probe.status_code != 200` → `HTTP_ERROR`
     - `not probe.reachable` → `FAIL`
   - Extract `sandbox_backend` from `probe.body.get("details", {}).get("sandbox_backend", "")`.
   - Return probe result alongside availability for use in `_probe_single_server()`.
   - Criterion: existing availability semantics unchanged.

3. **Populate new `McpProbeResult` fields in `_probe_single_server()`**
   - `restart_recommended = probe.restart_recommended`
   - `operator_action_required = probe.operator_action_required`
   - `health_reason = str(probe.body.get("reason", "") or probe.body.get("message", ""))`
   - Criterion: fields populated when `/health` returns those fields.

4. **Add tests**
   - Test: `/health` body with `restart_recommended=true` → `McpProbeResult.restart_recommended=True`.
   - Test: `/health` body with `operator_action_required=true` → `McpProbeResult.operator_action_required=True`.
   - Test: `/health` body with `reason="OOM"` → `McpProbeResult.health_reason="OOM"`.
   - Test: malformed JSON → safe fallback, no exception.
   - Criterion: `uv run pytest tests/ -k "mcp_status" -v` passes.
