# Plan: Wire enabled_for_llm from /v1/tools response into build_runtime_tool()

## Goal

Fix the critical gap where `enabled_for_llm` defaults to `False` because the `/v1/tools` response's `enabled` field is never read during tool discovery, causing ALL tools to be hidden from the LLM.

## Scope

- `scripts/agent/services/mcp_tool_discovery.py` — add `enabled_for_llm` parameter to `build_runtime_tool()` call in `_dedupe_and_build()`
- `scripts/shared/runtime_tool.py` — no changes needed (`enabled_for_llm` parameter already exists)
- `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` — update outdated statement about `enabled`/`disabled_reason` not being implemented
- `tests/test_mcp_tool_discovery*.py` — add test for `enabled_for_llm` wiring

## Assumptions

- `enabled` field in `/v1/tools` response is a boolean (or absent)
- When `enabled` is absent, tools should default to `enabled_for_llm=True` (backward compatible)
- `disabled_reason` is preserved for diagnostics but not exposed to LLM

## Unknowns

Resolved — all unknowns answered by code inspection:

1. `enabled_for_llm` parameter already exists in `build_runtime_tool()` with default `None` (line 75 of `runtime_tool.py`). No need to add it.
2. Current default is `False` when `enabled_for_llm=None` (line 102: `resolved_enabled_for_llm = enabled_for_llm if enabled_for_llm is not None else False`). This is the bug — every tool gets `False`.
3. `_dedupe_and_build()` currently passes 10 parameters to `build_runtime_tool()`, none mapping to `enabled`.
4. `_validate_and_normalize_entry()` does not validate `enabled` field — it would need to be added if we want strict validation.

## Affected areas

- Tool discovery pipeline — `enabled` field now consumed during discovery
- LLM tool-calling surface — previously hidden tools will become visible again
- RuntimeToolRegistry — core filtering logic restored to correct behavior

## Design

### Code change in `_dedupe_and_build()`

Add one line to the `build_runtime_tool()` call:

```python
built[name] = build_runtime_tool(
    name=name,
    server_key=server_key,
    server_url=server_url,
    description=str(entry.get("description", "")),
    input_schema=entry.get("inputSchema", entry.get("input_schema")),
    raw_definition=entry,
    status=str(entry.get("status", "active")),
    is_write=entry.get("is_write"),
    requires_serial=entry.get("requires_serial"),
    enabled_for_llm=bool(entry.get("enabled", True)),  # ← NEW
    capabilities=tuple(entry.get("capabilities", []) or []),
)
```

Rationale:
- `entry.get("enabled", True)` — default `True` for backward compatibility with servers not sending `enabled` yet
- `bool(...)` — cast to bool for type safety even though `entry.get()` may already return bool

### Documentation update

Update `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` line 22 to reflect that `enabled`/`disabled_reason` ARE now implemented in MCP server responses AND wired into RuntimeToolRegistry.

## Risks

### Risk 1: Tools reappear unexpectedly

Tools previously hidden due to the bug will now appear in the LLM's tool-calling surface. If operators relied on the broken behavior (e.g., disabled tools were hidden because they always were), this is a behavioral change. Mitigation: verify with operator before deploying; document the change clearly.

### Risk 2: Missing `enabled` field in some MCP servers

Servers that don't send `enabled` will default to `True` (visible to LLM). This is intentional for backward compatibility but means those tools won't benefit from the disabled-tool filtering until the server adds support. Mitigation: monitor tool availability after deployment.

### Risk 3: No validation of `enabled` field type

If an MCP server sends `enabled: "true"` (string instead of boolean), `bool("true")` returns `True` regardless. Consider adding type validation in `_validate_and_normalize_entry()` if strict enforcement is needed.
