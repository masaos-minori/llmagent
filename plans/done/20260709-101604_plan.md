# Plan: Update tests for MCP restart-required reload behavior

Source requirement: `requires/20260708_38_require.md` (M-6)

## Goal

Verify every test-update task this requirement lists is already assigned to
a specific plan from this batch (H-1 through M-5), correct the requirement's
target-file list against the actual test layout, and add one consolidated
validation pass across the full affected surface.

## Scope

This requirement is a completeness audit over the H-1..M-5 batch, not a
source of new production behavior. No new implementation tasks are
introduced beyond what earlier plans already specify; this plan's
contribution is the cross-reference table below plus a combined test run.

In scope:
- Cross-referencing each M-6 implementation task to the plan file that
  already covers it (table below).
- Correcting 3 target file names that do not exist in the repository.
- Adding one aggregate validation step spanning every test file touched
  across the batch (no single earlier plan runs all of them together).

Out of scope: no file requires a change beyond what H-1..M-5 already specify.

1. Of the 7 target files listed in the requirement, 3 do not exist in the
   repository — verified with `ls tests/test_mcp_reload.py
   tests/test_http_transport.py tests/test_cmd_reload.py` (all missing).
   Their intended content already lives elsewhere:
   - `test_mcp_reload.py` → `tests/test_config_reload.py` +
     `tests/test_config_reload_classification.py`.
   - `test_http_transport.py` → `tests/test_tool_executor.py::TestHttpTransportRetry`
     (HTTP transport tests are not split into their own file).
   - `test_cmd_reload.py` → `tests/test_agent_cmd_config.py` +
     `tests/test_cmd_config_refactor.py`.
2. `tests/test_tool_executor.py` exists but its only relevant class
   (`TestHttpTransportRetry`) covers retry logic, not config reload — no
   plan in this batch touches it, and none needs to: H-3's plan
   (`plans/20260709-095138_plan.md`) already confirmed `HttpTransport`
   snapshots its URL once at construction and is never touched by
   `config_reload.py`.
3. `tests/test_lifecycle.py` and `tests/test_repl_health.py` exist and were
   inspected while planning H-5, H-7, H-8, and M-4; none required changes
   (verified in each of those plans' Assumptions/Out-of-scope sections).

## Risks

- **Risk**: Treating this requirement as "fully covered elsewhere" could
  cause it to be marked done without anyone actually running the aggregate
  test command, leaving cross-plan interactions (e.g. two plans editing the
  same test class) undetected until CI.
  → **Mitigation**: The aggregate command in the Validation plan is the
  actual deliverable; it must be run as a distinct step after all of
  H-1..M-5 are implemented, not skipped because "each plan already tested
  itself."
- **Risk**: If a future requirement renames or restructures
  `test_config_reload.py`, the aggregate command above would need updating.
  → **Mitigation**: Low likelihood within this batch's timeframe; flagged
  for visibility only.
