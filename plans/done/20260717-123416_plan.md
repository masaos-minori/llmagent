# Plan: Remove plugin subsystem completely

Source requirement: `requires/20260717_01_require.md`

## Goal

The Agent no longer loads plugins, dispatches plugin commands, executes plugin tools, exposes `/plugin`, or validates plugin-specific configuration — every plugin runtime, config, command, tool-execution, diagnostics, test, and documentation path is removed, since all tools will instead be provided by MCP servers and indexed through the forthcoming `RuntimeToolRegistry` (requirement 02).

## Scope

**In scope**
- Delete `scripts/shared/plugin_registry.py`, `plugin_auto_discover.py`, `plugin_registries.py`, `plugin_result.py`, `plugin_tool_invoker.py`, and `plugin_conflicts.py` (a dependency of `plugin_auto_discover.py` not listed in the requirement's own target-file list, but part of the same subsystem — see Assumption 6).
- Delete `scripts/agent/commands/cmd_plugins.py` (`_PluginsMixin`).
- `scripts/agent/commands/registry.py`: remove `_PluginsMixin` from `CommandRegistry`'s base classes; remove `_dispatch_plugin()`; remove the `return await self._dispatch_plugin(line)` fallback in `dispatch()` so it returns `False` when no built-in command matches.
- `scripts/agent/commands/command_defs_list.py`: remove the `/plugin` `CommandDef` entry.
- `scripts/shared/tool_executor.py`: remove the `PluginToolInvoker` import, `self._plugin_invoker` instantiation, and the `try_execute()` call site — `ToolExecutor` becomes MCP-only.
- `scripts/agent/factory.py`: remove `_init_plugin_registry()` and its call site (plugin loading at startup).
- `scripts/agent/config_dataclasses.py`: remove `plugin_tool_override`, `plugin_strict` fields and their validator imports (`_v_tool_ps`, `_v_tool_pto`).
- `scripts/agent/config_builders.py`: remove the two plugin config-key reads.
- `scripts/agent/services/config_validators.py`: remove `validate_tool_plugin_tool_override()`, `validate_tool_plugin_strict()`.
- `scripts/agent/services/config_reload.py`: remove the `plugin_strict` reload-diff branch.
- `scripts/shared/production_config_validator.py`: remove the `"plugin_strict"` list entry.
- `scripts/shared/transport_dto.py`: update `ToolCallResult`'s field-comment documentation to drop the `"plugin"`/`"plugin_contract"` semantics (the dataclass shape itself is otherwise unchanged — `source`/`error_type` remain generic strings).
- `config/agent.toml`: remove `plugin_tool_override`, `plugin_strict` keys.
- `plugins/`: delete the directory (only `example.py`, a template with no production use, plus `__pycache__/`).
- Remove/migrate the 21 plugin-referencing test files (enumerated in Affected areas).
- Remove/update the plugin-mentioning documentation (36 files found; dedicated plugin docs are deleted outright, incidental mentions are trimmed).

**Out of scope**
- Building `RuntimeTool`/`RuntimeToolRegistry` itself — that is requirement 02's scope; this plan only removes the system being replaced.
- Any change to MCP server/tool dispatch mechanics beyond deleting the plugin short-circuit in `ToolExecutor`.
- Redesigning `ToolCallResult`'s dataclass shape — only its documentation comments change.

## Assumptions

1. No real plugin implementation exists in this codebase today — `plugins/` contains only `example.py` (a template, explicitly documented as inactive unless manually placed there) and `__pycache__/`. Removing the directory has no runtime behavior loss beyond the example itself.
2. `PluginLoadError`/`PluginLoadResult`/`PluginFailure` (in `plugin_result.py`) have no consumers outside the plugin subsystem itself and `cmd_plugins.py`'s `/plugin status` display — confirmed via the import survey; safe to delete alongside their sole consumers.
3. `plugin_strict` is the only plugin config key with reload support (`config_reload.py:452-454`); `plugin_tool_override` is not hot-reloadable today — this asymmetry is pre-existing and not something this removal needs to preserve or fix, since both keys are being deleted entirely.
4. `ToolExecutor`'s plugin short-circuit (`tool_executor.py:211-213`) runs *before* MCP dispatch when a tool name matches a registered plugin tool — since no real plugin tools are registered in production (Assumption 1), removing this path changes no observable production behavior; it only removes a code path that today is always a no-op (`plugin_result is always None` in practice).
5. `production_config_validator.py`'s single `"plugin_strict"` reference is a required/known-keys list entry (not a semantic check) — removing the entry is a pure subtraction, not a logic change.
6. **`scripts/shared/plugin_conflicts.py` (95 lines, `validate_command_conflicts`/`validate_tool_conflicts`) is part of this subsystem** even though the requirement's own target-file list omits it — confirmed as a direct dependency of `plugin_auto_discover.py` (imported at line 10, called at lines 78-81). This file has no other purpose and must be deleted alongside the rest of the subsystem, or `plugin_auto_discover.py`'s deletion leaves a dangling import if only the requirement's literal list is followed.
7. `CommandRegistry.dispatch()`'s fallback to `_dispatch_plugin()` (registry.py:153) is the sole plugin-command dispatch path — once removed, `dispatch()` naturally returns `False` for any unmatched line (the existing `for cmd in _COMMANDS` loop's fallthrough), matching the requirement's own "Implementation notes" instruction exactly with no additional code needed beyond deleting the fallback call.

## Design

This is a pure subtractive change with one structural nuance: `CommandRegistry.dispatch()`'s existing control flow already naturally degrades to "return `False` if nothing in `_COMMANDS` matches" once the plugin fallback line is deleted — no new logic is needed, only deletion (per Assumption 7).

The plugin subsystem has three extension points (per the requirement's own framing): plugin commands (`_dispatch_plugin`), plugin tools (`PluginToolInvoker`), and plugin pipeline hooks (`get_pipeline_post_stages`/`run_pipeline_stages`, consumed by `rag/pipeline.py` — the one file not in the requirement's own list, flagged as UNK-03/blocking above). All three must be traced to zero remaining call sites, not just the two explicitly named in the requirement.

Deletion order matters for avoiding transient import errors during a multi-commit rollout: delete consumers first (registry.py's fallback, tool_executor.py's invoker call, factory.py's init call, rag/pipeline.py's hook call if confirmed plugin-only), then delete the now-unreferenced subsystem files themselves (plugin_registry.py, plugin_auto_discover.py, plugin_registries.py, plugin_result.py, plugin_tool_invoker.py, plugin_conflicts.py, cmd_plugins.py), so that at every commit boundary the tree still imports cleanly.

## Risks

- **Risk**: `scripts/rag/pipeline.py`'s dependency on `plugin_registry` was outside the requirement's own target-file list, so it could have been missed entirely if not found during planning.
  **Mitigation**: already resolved during planning (confirmed plugin-only by direct read, see Assumptions/Unknowns); included explicitly in Affected areas and Implementation Step 1 so it isn't silently skipped at implementation time.
- **Risk**: `scripts/agent/commands/registry.py` (25 commits/30d) and `scripts/shared/tool_executor.py` (32 commits/30d) are high-churn files — a concurrent in-flight change could conflict with this removal.
  **Mitigation**: land the consumer-removal step (Implementation Step 1) as its own small, fast commit; rebase immediately before merging; re-run the full test suite after rebase.
- **Risk**: Some of the 21 plugin-referencing test files may test adjacent (non-plugin) behavior that happens to share a fixture or module scope with plugin tests — an overly broad deletion could remove real coverage.
  **Mitigation**: Implementation Step 6 requires reading each file in full before deciding delete-vs-trim, not deleting based on filename/grep-hit alone.
- **Risk**: Removing `ToolExecutor`'s plugin short-circuit changes its method signature/control flow in a high-centrality file — any test asserting on `ToolExecutor`'s exact call sequence (e.g. via mocks) could break even if plugin behavior itself was never exercised.
  **Mitigation**: Implementation Step 9's full-suite run is the empirical backstop; a mock-related failure would surface immediately as a test failure, not silently pass.
- **Risk**: Deleting 5 dedicated plugin docs (rather than converting them to removal notes) could break cross-references from other docs that still link to them.
  **Mitigation**: Implementation Step 7 requires checking for inbound links/`related:` front-matter references before outright deletion; convert to a short dated removal note instead of hard-deleting if any live cross-reference is found, matching this repo's established convention for fully-removed features.
