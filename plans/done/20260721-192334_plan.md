## Goal
Restrict approval dry-run execution to tools known to support `dry_run`, preventing HIGH-risk false denials when unsupported tools are listed in `approval_dry_run_tools`.

## Scope
**In-Scope**:
- Define canonical list of dry-run-capable tools (currently only file-oriented tools from `_DEFAULT_DRY_RUN_TOOLS`)
- Validate `approval_dry_run_tools` at startup — warn and ignore unsupported entries
- Modify `_build_preview_with_dry_run()` to distinguish dry-run failure categories
- Update `scripts/agent/tool_approval.py` — skip dry-run for unsupported tools, use static preview
- Update `scripts/agent/config_dataclasses.py` — add capability flag or validation logic
- Update `scripts/agent/config_builders.py` — add startup validation for `approval_dry_run_tools`
- Update `scripts/agent/services/config_validators.py` — add dry-run tool validation
- Update tests: `tests/test_tool_approval_preflight.py`, `tests/test_tool_approval_repos.py`, `tests/test_tool_approval_risk.py`

**Out-of-Scope**:
- Adding new dry-run-capable tools beyond the existing file-oriented defaults
- Implementing dry-run for GitHub/Git/CICD/shell tools
- Changes to MCP server configurations unrelated to dry-run validation

## Assumptions
1. Only file-oriented tools (`write_file`, `edit_file`, `create_directory`, `delete_file`, `delete_directory`, `move_file`) currently support `dry_run`
2. Warning + ignoring unsupported entries is preferred over strict rejection unless strict config mode is enabled
3. Dry-run failures from transport/server errors should be treated as unavailable (downgrade to text-only preview), not as blocking errors

## Risks & Mitigations
- **Risk**: Existing operators rely on dry-run for non-file tools and expect it to work | Likelihood: low → **Mitigation**: Startup warning clearly identifies unsupported tools; operator can remove them from config or implement dry-run support
- **Risk**: Distinguishing dry-run failure categories proves unreliable via exception types | Likelihood: medium → **Mitigation**: Conservative approach — treat all dry-run exceptions as unavailable; only deny HIGH-risk for confirmed valid dry-run results with `is_error=True`
- **Risk**: Adding dry-run capability metadata creates maintenance burden | Likelihood: low → **Mitigation**: Use simple frozenset constant rather than per-tool capability flags; add new tools by updating the constant
- **Risk**: Startup validation warnings flood logs during normal operation | Likelihood: low → **Mitigation**: Log once per session, not per request; use WARNING level (not ERROR)
