# Plan: Remove healthcheck_mode="" auto-inference; derive from transport (Option B)

Source requirement: `requires/20260709_07_require.md`

## Decision record

The requirement presents 3 implementation options. Asked the user during
planning; **Option B** was chosen ("Derive healthcheck mode from transport.
Remove the empty-string auto-inference branch. Keep explicit `"http"`
only."), based on evidence gathered during research:
- `config/agent.toml` already sets `healthcheck_mode = "http"` explicitly
  for every one of its 10+ MCP server entries — no shipped config relies on
  the empty-string inference path.
- `tests/test_mcp_config.py::test_ping_tool_healthcheck_rejected` proves a
  second mode (`"ping_tool"`) was previously implemented and has already
  been removed — non-HTTP healthcheck modes are not a live future
  direction, they are a tried-and-reverted one.
- `grep -rn 'healthcheck_mode\s*=\s*""'` across `scripts/`, `config/`,
  `tests/` found zero explicit empty-string usages anywhere in the repo.

## Goal

Replace the empty-string sentinel-based auto-inference in
`_build_single_server()` with an explicit `_derive_healthcheck_mode(transport)`
helper, called only when the `healthcheck_mode` key is *absent* from
config. An explicitly-present-but-empty `healthcheck_mode = ""` becomes an
invalid value (parsed the same as any other unrecognized string) rather
than a second way to request auto-inference.

## Scope

In scope:
- `shared/mcp_config.py`: add `_derive_healthcheck_mode()`; restructure
  `_build_single_server()`'s `healthcheck_mode` resolution to distinguish
  "key absent" (derive) from "key present" (strict enum parse, no more
  empty-string special case).
- `tests/test_mcp_config.py`: add a test that explicit
  `healthcheck_mode: ""` now raises `ValueError`; confirm the
  omitted-key/explicit-`"http"`/invalid-value cases the requirement lists
  are all covered (most already are).
- `docs/04_mcp_06_configuration_and_operations.md`: update the
  `healthcheck_mode` field-table row and add a short note describing the
  historical empty-string compatibility path and its removal (the "selected
  deprecation path" per the requirement's Documentation section).

Out of scope:
- `config/agent.toml` — already fully compliant (every entry already says
  `healthcheck_mode = "http"` explicitly); no change needed. Listed in the
  requirement's Target Files as a verification target, not an edit target.
- `startup_mode`, health endpoint semantics, non-HTTP transports — all
  explicit Non-Goals, untouched.
- `HealthcheckMode` enum itself — stays a single-member enum (`HTTP`); this
  plan does not add, remove, or rename enum members.

## Assumptions

1. `McpServerConfig.healthcheck_mode`'s dataclass-level default
   (`shared/mcp_config.py`, `healthcheck_mode: HealthcheckMode =
   HealthcheckMode.HTTP`) already defaults directly to `HealthcheckMode.HTTP`
   with no empty-string sentinel involved — verified by reading the
   dataclass field declaration and confirmed by
   `tests/test_mcp_config.py::test_valid_http_config` (constructs
   `McpServerConfig` directly, omitting `healthcheck_mode`, asserts
   `HealthcheckMode.HTTP`). **The empty-string auto-inference logic exists
   only in `_build_single_server()`'s TOML-dict parsing path** — this plan
   only touches that function, not the dataclass default.
2. `dict.get("healthcheck_mode", "")` (current code) cannot distinguish
   "key absent" from "key present with value `''`" — both currently hit the
   same `if not raw_hc:` branch identically, so no existing test currently
   exercises "explicit empty string" as a case distinct from "omitted" —
   verified by reading `_build_single_server()` and
   `tests/test_mcp_config.py::test_toml_string_values_default_when_absent`
   (the one existing "absent" test uses a dict that omits the key entirely,
   not one that sets it to `""`). This plan's restructure is what makes the
   two cases distinguishable and therefore separately testable, per the
   requirement's explicit ask to test "omitted" and "explicit empty"
   separately.
3. `tests/test_mcp_config.py` already covers "explicit `\"http\"`" (via
   `config/agent.toml`'s real usage pattern and
   `test_toml_string_values_default_when_absent`'s assertion target value)
   and "invalid healthcheck mode" (3 existing tests:
   `test_invalid_healthcheck_mode_raises`,
   `test_direct_runtime_invalid_healthcheck_mode_raises`,
   `test_ping_tool_healthcheck_rejected`) — only the "explicit empty string"
   case is new.

## Design

**`shared/mcp_config.py`** — add near `_build_single_server()`:

```python
def _derive_healthcheck_mode(transport: TransportType) -> HealthcheckMode:
    """Derive the healthcheck mode from transport type.

    HTTP is currently the only supported transport, so this always
    returns HealthcheckMode.HTTP. Kept as its own function (rather than a
    literal) so a future second transport has one place to add a mapping.
    """
    return HealthcheckMode.HTTP
```

Replace the current block in `_build_single_server()`:

```python
    # Resolve healthcheck_mode with auto-inference from transport.
    raw_hc = v.get("healthcheck_mode", "")
    if not isinstance(raw_hc, str):
        raise ValueError(
            f"mcp_servers[{key!r}].healthcheck_mode must be str, got {type(raw_hc).__name__}"
        )
    if not raw_hc:
        healthcheck_mode = HealthcheckMode.HTTP
    else:
        healthcheck_mode = HealthcheckMode(raw_hc)
```

with:

```python
    # healthcheck_mode: derive from transport when the key is absent.
    # An explicitly-present value (including "") is parsed strictly — the
    # historical empty-string auto-inference sentinel is removed.
    if "healthcheck_mode" not in v:
        healthcheck_mode = _derive_healthcheck_mode(TransportType(transport))
    else:
        raw_hc = v["healthcheck_mode"]
        if not isinstance(raw_hc, str):
            raise ValueError(
                f"mcp_servers[{key!r}].healthcheck_mode must be str, got {type(raw_hc).__name__}"
            )
        healthcheck_mode = HealthcheckMode(raw_hc)
```

Note: `HealthcheckMode("")` already raises `ValueError` naturally (empty
string is not a valid enum value) — no new error-handling code is needed
for the "explicit empty string" case, only the removal of the special-case
branch that used to intercept it.

**`docs/04_mcp_06_configuration_and_operations.md`** — replace the field
row (line 77):

```markdown
| `healthcheck_mode` | `str` | *(derived)* | `"http"` — the only current transport/healthcheck mode; derived automatically when the key is omitted. If present, the value must be exactly `"http"`. |
```

Add a note below the field table (or near it):

```markdown
**Deprecation note:** Earlier versions accepted `healthcheck_mode=""` as an
explicit request for auto-inference (compatibility with configs predating
this field). That empty-string sentinel has been removed — omit the key
entirely to get the derived value, or set it to the exact string `"http"`.
An explicit empty string is now rejected as an invalid value, the same as
any other unrecognized string.
```

## Risks

- **Risk**: If any config file outside this repo (e.g. a deployment
  environment's local override not tracked in version control) explicitly
  sets `healthcheck_mode = ""`, this change would newly break its startup
  with a `ValueError` where it previously started successfully.
  → **Mitigation**: `grep` across the entire tracked repo found zero such
  usages (Decision record); this is a real but low-probability risk for
  untracked/external configs. The new error message
  (`"'' is not a valid HealthcheckMode"`) is actionable — it tells the
  operator exactly which key to fix or remove.
- **Risk**: `_derive_healthcheck_mode()` is a trivial single-line function
  today (always returns `HealthcheckMode.HTTP`); a reviewer might flag it
  as over-engineered for a single-transport codebase.
  → **Mitigation**: Explicitly justified in its own docstring — it exists
  so a future second transport has one obvious place to add a mapping,
  rather than another empty-string-style special case being invented later.
