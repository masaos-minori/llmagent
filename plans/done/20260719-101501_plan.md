# Plan: Add a read-only Browser MCP server; fix stale sqlite-mcp doc references; document Gateway-proposal naming gaps

Source requirement: `requires/20260715_02_require.md`

## Goal

A new, read-only Browser MCP server (page fetch + render-to-text, no
interactive automation) exists alongside the 10 current MCP servers,
registered and documented following the project's existing add-a-server
checklist; the two stale "sqlite-mcp" doc references are corrected; and the
MCP tool catalog documents the real tool names corresponding to the
original "MCP統合プラグインシステム" proposal's Gateway-style function
names, so future proposals reference reality.

## Assumptions

1. The canonical add-a-server checklist
   (`docs/04_mcp_06_15_new-mcp-server-addition-checklist.md:20-28`,
   verified) is: create `server.py` (subclass `MCPServer`, override
   `dispatch()`), declare `own_config_file`, create the per-server TOML
   (self-contained — does not read `agent.toml`), add
   `[[tool_definitions]]` to `agent.toml`, register in
   `shared/tool_constants.py`'s frozenset, add to `deploy/deploy.sh`'s
   copy list, add startup steps to `deploy/setup_services.sh`, add
   `tool_safety_tiers` entries, update `routing.md` if needed. Browser MCP
   follows this exactly — no deviation.
2. `scripts/mcp_servers/shell/` (verified: `models.py` 144L, `server.py`
   170L, `service.py` 342L, `tools.py` 57L) is the closest existing
   template for a sandboxed-access server with allowlisting + size limits,
   per the source requirement's own suggestion — confirmed suitable:
   `_check_command()`-style allowlist-and-reject pattern (raises a typed
   `*AuthorizationError`, mapped to HTTP 403) and
   `truncate_output()`-style silent-truncation-with-flag pattern (for size
   limits) both translate directly to a domain-allowlist-and-reject /
   content-length-truncation-with-flag design for Browser MCP.
3. No shared `create_app()` FastAPI factory exists across servers — each
   server wires its own `app = FastAPI(...)` and, inconsistently, only 3
   of the ~8 existing servers (`cicd`, `git`, `mdq`) call
   `attach_auth_middleware()` (verified: `shell/server.py` does not).
   Given Browser MCP performs outbound network fetches (a materially
   different risk profile than `shell`'s local-command execution), this
   plan chooses to **wire `attach_auth_middleware()`** (following the
   `cicd`/`git` pattern) as a deliberate, conservative default — not
   because it's universally required by convention, but because it costs
   nothing and matches the higher-risk-surface precedent.
4. `.importlinter` has no per-server contract entries (verified: `grep -in
   "git\|shell" .importlinter` → zero matches) — a new
   `scripts/mcp_servers/browser/` package automatically inherits the
   single root-level `mcp_servers-no-agent` contract; no new `.importlinter`
   entry is needed.
5. **No `mcp_servers → rag` contract exists either** (verified: only
   5 contracts total, none naming `rag` as forbidden for `mcp_servers`) —
   but the documented intended hierarchy (`.importlinter`'s own header
   comment: "mcp_servers → db, shared") and every existing server's
   self-contained-config convention (Assumption 1) both indicate
   `mcp_servers` packages are not meant to depend on `rag`. This plan
   therefore does **not** import `scripts/rag/ingestion/crawler.py`'s
   `WebCrawler` (which already does httpx+BeautifulSoup page fetching for
   RAG ingestion) even though it is technically lint-imports-legal —
   Browser MCP gets its own small, self-contained fetch+parse function
   instead, consistent with every other server's independence.
6. `httpx` and `beautifulsoup4` are already project dependencies
   (`pyproject.toml:6`, confirmed) and already used for exactly this kind
   of "fetch a URL, parse HTML" task in `scripts/rag/ingestion/crawler.py`
   (verified) — no new dependency is needed for the **read-only, no-JS**
   scope chosen. `playwright` is confirmed **not** installed/declared
   (`import playwright` fails; no `pyproject.toml` entry) — deliberately
   not added, since the user's scope decision (read-only fetch/render, no
   interactive automation) does not require executing page JavaScript.
   This means Browser MCP will not fully render JavaScript-heavy
   (client-side-rendered) pages — see Risks.
7. `config/agent.toml`'s `[mcp_servers.*]` entries (verified for `shell`,
   `git`, `web_search`) share a common field set: `transport`, `url`,
   `startup_mode`, `auth_token`, `role`, `tool_names`, `cmd`. A minor
   inconsistency exists (`cicd`/`rag_pipeline` also carry
   `healthcheck_mode="http"` and use `startup_mode="persistent"` instead
   of `"subprocess"`, per a documented fix for a startup-stall issue) —
   not standardized by this plan; `[mcp_servers.browser]` follows the more
   common `shell`/`git`/`web_search` shape (`startup_mode="subprocess"`)
   unless implementation discovers a similar startup-stall risk.

## Design

1. **`BrowserService.fetch(url, **params)`**: (a) parse the URL's
   hostname, check it against `allowed_domains` (fail-closed: empty list
   denies everything, mirroring `shell`'s empty-`command_allowlist`
   warning-and-deny and `file/common.py`'s `resolve_safe` fail-closed
   pattern) — on mismatch, raise a typed `BrowserAuthorizationError`
   (mapped to HTTP 403 in `server.py`, same shape as
   `ShellAuthorizationError`→403); (b) `httpx.AsyncClient` GET the URL
   with a configured `timeout_sec`; (c) parse the response body with
   `BeautifulSoup(html, "lxml")` (same parser already used in
   `crawler.py`) and extract visible text (strip `<script>`/`<style>`,
   join block-level text) — no JS execution, so client-side-rendered
   content is not visible (documented limitation, Risks); (d) truncate
   the extracted text to `max_response_kb` with a `truncated: bool` flag
   on the response, mirroring `shell`'s `truncate_output` pattern exactly
   rather than inventing new truncation semantics.
2. **Tool schema** (`tools.py`): one tool, `browser_fetch`, params `url`
   (required, string), optionally `max_response_kb` (caller override,
   clamped to server max, same pattern as `shell_run`'s `max_output_kb`).
3. **`server.py`**: `BrowserMCPServer(MCPServer)` with
   `server_name="browser-mcp"`, a new port (next available — confirm the
   current highest-used port among the 10 servers during implementation
   and take the next one), `own_config_file="browser_mcp_server.toml"`;
   FastAPI app wiring `attach_auth_middleware()` per Assumption 3;
   exception handler mapping `BrowserAuthorizationError` → 403.
4. **Docs**:
   - `04_mcp_04_06_browser.md`: new catalog page following the existing
     `04_mcp_04_0N` pages' structure/frontmatter convention (confirm exact
     template by reading one sibling page, e.g. `04_mcp_04_05_git.md`,
     immediately before writing — not done in this investigation pass to
     avoid redundant reads, per token-efficiency guidance).
   - `04_mcp_00_document-guide.md`: add the File Index row; add the
     Gateway-naming-clarification bullet (Migration Notes section,
     matching the existing terse note style, e.g. the `POST /v1/search`
     removal note already there).
   - `AGENTS.md:83`: `... shell/service, rag_pipeline/service, sqlite/service, cicd/service, server (base class)`
     → `... shell/service, rag_pipeline/service, cicd/service, server (base class)`.
   - `docs/04_mcp_05_03_...md:111`: the current line

## Risks

- **Risk**: Read-only, no-JS-execution scope means client-side-rendered
  (SPA/React-style) pages will return little or no meaningful text —
  users may expect "browser" to mean "renders anything a real browser
  would." → **Mitigation**: explicitly documented in
  `04_mcp_04_06_browser.md` as a stated limitation of the read-only scope
  decision (Assumption 6); if this proves insufficient in practice, adding
  Playwright-based rendering is a natural, separately-scoped follow-up
  (interactive-automation scope was already deferred for the same
  dependency-size reason).
- **Risk**: Outbound URL fetching from an MCP server is an SSRF-adjacent
  surface (a caller could point `browser_fetch` at internal/loopback
  addresses if `allowed_domains` is misconfigured too permissively). →
  **Mitigation**: fail-closed empty-by-default `allowed_domains` (same
  convention as every other server's allowlist), plus recommend the
  implementation additionally reject literal IP-literal targets and
  loopback/link-local ranges by default regardless of the domain
  allowlist (defense in depth) — call this out explicitly as an
  implementation-step detail, not left as a residual gap.
- **Risk**: `[mcp_servers.cicd]`/`[mcp_servers.rag_pipeline]`'s
  `startup_mode="persistent"` fix (for a health-check-stall issue,
  Assumption 7) suggests Browser MCP could hit a similar issue if its
  health check depends on any slow-to-initialize resource. → **Mitigation**:
  Browser MCP's health check should be a trivial liveness check with no
  external dependency (no network call at healthcheck time), avoiding the
  same failure mode by design; confirmed feasible since Design step 1's
  fetch logic is per-request, not a persistent connection/resource the
  health check would need to probe.
- **Risk**: Adding a new server touches many small, easy-to-miss files
  (checklist has 8 steps across `agent.toml`, `tool_constants.py`,
  `deploy/deploy.sh`, `deploy/setup_services.sh`, `routing.md`) — a
  partially-completed addition (e.g. forgetting the `deploy/deploy.sh`
  copy-list entry) would work in dev but fail in production deploys. →
  **Mitigation**: Implementation steps 1-7 follow the checklist in its
  documented order exactly; the Validation Plan's `check-mcp-docs` and
  `test_route_resolver.py`/`test_tool_constants.py` runs are specifically
  the project's existing safety net for exactly this class of
  partial-registration mistake.
