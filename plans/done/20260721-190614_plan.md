# Plan: Unify startup config validation with reload validation and fix missing defaults

## Goal

Make startup config loading use the same strict typed validators as runtime reload, eliminating silent coercion bugs and ensuring deterministic configuration behavior.

## Scope

- `scripts/agent/config_builders.py` — replace all implicit `bool()`, `list()`, `dict()` conversions with typed validators
- `scripts/agent/services/typed_validators.py` — may need minor additions for startup-specific needs
- `scripts/agent/config_dataclasses.py` — verify defaults for `tool_concurrency_limits` and `tool_safety_tiers`
- `tests/test_agent_config*.py` — add/update tests for strict startup validation

## Assumptions

- `ConfigReloadValidationError` can be reused for startup errors (or a new `ConfigLoadError` should be introduced)
- All typed validators already handle the edge cases correctly (bool excluded from int, etc.)
- Existing test fixtures will need updates to reflect stricter validation

## Unknowns

Resolved — all unknowns answered by code inspection:

1. Typed validators (`_get_bool`, `_get_int`, `_get_list`, `_get_dict`, etc.) already exist in `typed_validators.py` but are never imported or used in `config_builders.py`.
2. `tool_concurrency_limits` and `tool_safety_tiers` both use `dict(cfg.get(...), {})` — defaults are correct but validation is missing.
3. `_apply_*` helpers exist in `typed_validators.py` and could simplify the migration.
4. Exception class is `ConfigReloadValidationError` — may need a separate `ConfigLoadError` for startup context.

## Affected areas

- Configuration loading pipeline — startup becomes as strict as reload
- Security settings — tool safety tiers, approval rules validated at startup
- Memory settings — numeric constraints enforced at startup
- Test suite — many existing tests will fail due to stricter validation

## Design

### Migration approach

Replace all implicit conversions in `config_builders.py` with typed validators:

```python
# Before:
serial_tool_calls = bool(cfg.get("serial_tool_calls", False))

# After:
serial_tool_calls = _get_bool(cfg, "serial_tool_calls") or False
```

For dict/list fields with non-trivial defaults:

```python
# Before:
tool_concurrency_limits = dict(cfg.get("tool_concurrency_limits", {}))

# After:
tool_concurrency_limits = _get_dict(cfg, "tool_concurrency_limits") or {}
```

Consider using `_apply_*` helpers where setter patterns match.

### Additional validations per requirements

After typed validator migration, add specific business rule validations:

- `memory_retention_days >= 1`
- `memory_embed_dim >= 1`
- `memory_embed_timeout_sec > 0`
- `memory_max_inject_semantic >= 0`
- `memory_max_inject_episodic >= 0`
- `tool_safety_tiers` values must be one of: `READ_ONLY`, `WRITE_SAFE`, `WRITE_DANGEROUS`, `ADMIN`

### Exception handling strategy

Option A: Reuse `ConfigReloadValidationError` for both startup and reload contexts. Simpler but less precise error messaging.

Option B: Create `ConfigLoadError` as a distinct exception for startup failures. Better error semantics but more code changes.

Recommend Option A initially — consistent error type simplifies caller logic. Can split later if needed.

## Implementation steps

### Step 1: Import typed validators into config_builders.py

Add imports for `_get_bool`, `_get_int`, `_get_float`, `_get_str`, `_get_list`, `_get_dict` from `services.typed_validators`.

### Step 2: Replace boolean conversions

Replace all `bool(cfg.get("key", default))` patterns with `_get_bool(cfg, "key") or default`.

### Step 3: Replace list conversions

Replace all `list(cfg.get("key", default))` patterns with `_get_list(cfg, "key") or default`.

### Step 4: Replace dict conversions

Replace all `dict(cfg.get("key", default))` patterns with `_get_dict(cfg, "key") or default`.

### Step 5: Add business rule validations

Add validation for memory numeric fields and safety tier values after typed validator checks pass.

### Step 6: Update tests

Update existing test fixtures that rely on implicit coercion. Add new tests for:
- Invalid bool type rejection
- Invalid list type rejection
- Invalid dict type rejection
- Bool-as-int rejection
- Missing concurrency limits / safety tiers
- Invalid safety tier values
- Invalid memory numeric values
- Minimal valid config acceptance

### Step 7: Run validation

- `uv run pytest` — fix failing tests
- `uv run mypy scripts/` — type check passes
- `uv run ruff check scripts/` — lint clean

## Risks

### Risk 1: Widespread test breakage

Stricter validation will cause many existing tests to fail. Mitigation: update tests incrementally, starting with the most critical ones first.

### Risk 2: Backward compatibility

Configs that previously worked via implicit coercion will now fail. Mitigation: document breaking changes clearly; consider adding a deprecation warning path before full enforcement.

### Risk 3: Incomplete validator coverage

If any implicit conversion is missed during migration, the inconsistency remains. Mitigation: grep for remaining `bool(`, `list(`, `dict(` patterns in `config_builders.py` after migration.
