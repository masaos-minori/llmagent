# Plan: Remove hardcoded MCP server/tool total-count assertions from docs and tests

Source requirement: `requires/20260716_19_require.md`

## Goal

No documentation file or test in the repository asserts a literal total count
(global or per-server) of MCP servers or MCP tools, or compares an MCP tool
collection to a hardcoded expected-name-set literal; name-level drift checks
between independently-maintained implementation sources and structural/
classification tests remain intact and green.

## Assumptions

1. The disambiguating rule from the requirement — "keep checks that compare
   two independent implementation sources; remove checks that compare an
   implementation source to a number or name-list literal written into the
   test itself" — was confirmed interactively with the user and applies
   uniformly across all 4 test files. No file needs a different rule.
2. `docs/04_mcp_04_*.md`'s `**ツール（N個）:**` heading is followed immediately
   by the tool-name listing (verified by direct read of all 5 catalog files:
   `docs/04_mcp_04_01_web-search-file-read-github.md:62,103`,
   `docs/04_mcp_04_02_file-write-file-delete-shell.md:26,58,90`,
   `docs/04_mcp_04_03_rag-pipeline-and-cicd.md:25,75`,
   `docs/04_mcp_04_04_mdq.md:24`, `docs/04_mcp_04_05_git.md:25`) — dropping
   the count leaves the heading grammatically and structurally intact
   (`**ツール:**` followed by the same comma-separated tool list).
3. `tools/check_mcp_docs_consistency.py`'s Check 5 is the only doc-count
   validator in the CI script — verified by reading the full file: Checks
   1–4 and 6–13 validate unrelated wording/terminology drift (startup modes,
   fail-open language, routing authority, stdio references, etc.) and do not
   reference `_TOOL_COUNT_RE` or `_SERVER_TOOLS_MAP`.
4. `.github/workflows/mcp-docs-consistency.yml` references
   `scripts/checks/check_mcp_docs_consistency.py`, which does not exist in
   the current tree (`scripts/checks/` is not a directory; confirmed via
   `ls scripts/checks/` — no such path). The real script lives at
   `tools/check_mcp_docs_consistency.py` and is registered as the
   `check-mcp-docs` console script in `pyproject.toml` via module name
   `check_mcp_docs_consistency`. This means the GitHub Actions workflow
   already cannot run this script today, independent of this plan — Check 5's
   removal introduces no new CI breakage since the workflow that would
   exercise it cannot locate the script at all. Fixing the workflow's path is
   out of scope for this plan (unrelated pre-existing defect).
5. Removing `TestRegistryTotalCounts`/`TestMdqToolsCount`/`TestCheckToolCounts`/
   the 7 `TestToolConstants` methods does not reduce coverage of any
   *behavior* — each removed method's only assertion is a literal count or a literal
   name-set equal to what `shared/tool_constants.py`/`mcp_servers/mdq/tools.py`
   already define verbatim in production code; no branch or edge case is
   uniquely exercised by these methods (confirmed by reading each test body
   in full — none call a function under test with varying inputs, they only
   inspect a static collection's cardinality/membership).
6. `docs/04_mcp_01_system_overview.md`'s replacement wording does not need to
   name a specific number of servers or tools to remain useful — pointing to
   `config/agent.toml`'s `[mcp_servers.*]` sections and
   `shared/tool_constants.py` as the live sources of truth satisfies the
   same reader need (verified: `config/agent.toml` has exactly 10
   `[mcp_servers.*]` table headers today, confirming this doc line was
   already stale — "65個" vs. the registry's actual 63 total from
   `tests/test_tool_registry_counts.py`).

## Design

**Doc heading edit pattern** (5 files, 8 occurrences) — mechanical
find/replace, same shape in every case:

```diff
-**ツール（9個）:** `search_docs`, `get_chunk`, ...
+**ツール:** `search_docs`, `get_chunk`, ...
```

**System overview total-count sentence** — replace the stale, drift-prone
sentence with a pointer to the two canonical sources instead of a number:

```diff
-- 10個のMCPサーバ、合計65個のtool（すべて `tool_constants.py` のfrozensetで管理され、`ToolRegistry` 経由で登録される）
+- MCPサーバは `config/agent.toml` の `[mcp_servers.*]` に定義され、各サーバが提供するtoolは `tool_constants.py` のfrozensetで管理され、`ToolRegistry` 経由で登録される
```

**Test removal pattern** — delete each named test method/class in full
(signature through closing assertion), plus any docstring line that states a
now-untracked count. Do not replace removed count assertions with `>= 1` or
similar weakened checks — the requirement's acceptance criteria state no
count-shaped assertion should remain, and a weakened count check would still
require touching the test on every tool-inventory change (the exact
maintenance burden being removed).

**`test_no_overlapping_tools` and `test_v1_tools_returns_all_tools`** — these
two are partial edits, not full removals, since each contains one count
assertion alongside a still-valuable non-count assertion (overlap detection;
`"tools"` key presence):

```diff
 def test_no_overlapping_tools(self) -> None:
     ...
             all_tools.update(tools_set)
-
-        # Total should be 42 tools (all frozensets; github-mcp uses prefix routing separately)
-        assert len(all_tools) == 42
```

```diff
         assert "tools" in body
-        assert len(body["tools"]) == 7
```

**CI script Check 5 removal** — delete the five module-level constants/
functions specific to Check 5, its call site inside `main()`'s `if
"toolcount" not in skip:` block, and its `"toolcount"` entry in
`skip_choices`; leave Checks 1–4 and 6–13 (and their own constants/functions)
untouched since none of them reference the Check-5-only symbols.

## Risks

### R-1: A removed count assertion was silently also the only check catching a real drift case not covered by the retained cross-source tests
**影響:** A future tool-inventory change could pass CI/tests while actually broken (e.g. a tool present in `TOOL_LIST` but missing from `MDQ_TOOLS`) if no retained test happens to cover that specific pairing.
**緩和:** Verified during Assumption 5 that every removed method's assertion is a pure literal-count/literal-name-set check with no unique behavioral coverage; the retained files (`test_mdq_tool_layer_consistency.py`, `TestMdqNoUnmappedTools`, `test_mdq_tools_match_server_definition`, `test_v1_tools_names_match_mdq_tools`) already cover every cross-source pairing (`TOOL_LIST`↔`_DISPATCH_TABLE`↔`MDQ_TOOLS`↔`ToolRegistry`↔live `/v1/tools`) independently of this plan. Implementation Step 8's full regression pass is the empirical confirmation.

### R-2: Doc heading edits are mechanical across 5 files; a missed occurrence would leave a stale count next to tool names that later change
**影響:** Low — a leftover `（N個）` is a cosmetic staleness risk, not a functional break, but defeats the purpose of this plan for that one heading.
**緩和:** Step 1's `rg` grep check after editing is the closing verification; re-run it, not just visually reread the diff.

### R-3: `.github/workflows/mcp-docs-consistency.yml` references a path (`scripts/checks/check_mcp_docs_consistency.py`) that does not exist in the current tree
**影響:** Confirmed via `grep -rn "check_mcp_docs_consistency" --include="*.yml" --include="*.yaml" --include="*.toml" .` and `ls scripts/checks/` (no such directory) — this workflow already cannot run the script today, on `master`, independent of this plan. It passes no `--skip` flags (just a bare invocation), so there was never a `--skip toolcount` CI dependency to break. This is a pre-existing defect, not introduced or worsened by this plan.
**緩和:** Out of scope to fix here (not part of the requirement). No mitigation needed for *this* plan's risk profile — Step 8's `check-mcp-docs` console-script smoke test (`pyproject.toml`'s `[project.scripts]` entry, module `check_mcp_docs_consistency`) is the actually-reachable invocation path and is exercised directly in Implementation Step 8 and the Validation Plan. Flagging the workflow's broken path as a candidate follow-up requirement, not implemented speculatively here.
