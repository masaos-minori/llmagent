# Plan: Replace requires_config with config_dependent in MCP tool definitions

Source requirement: `requires/20260717_14_require.md` (moved to `requires/done/` on completion of this workflow)

## Goal

Remove the ambiguous `requires_config` field from every MCP tool definition and
replace it with a clearly-named static field, `config_dependent: bool`, that
only asserts "this tool's behavior depends on server configuration" — without
conflating that with runtime availability (which is a separate, currently
unimplemented, dynamic concern).

## Scope

**In scope** — rename `requires_config` -> `config_dependent` (same boolean
value, no behavior change) in the 10 files that currently define it:

- `scripts/mcp_servers/file/read_tools.py` (9 occurrences, all `False`)
- `scripts/mcp_servers/file/write_tools.py` (4 occurrences, all `False`)
- `scripts/mcp_servers/file/delete_tools.py` (2 occurrences, all `False`)
- `scripts/mcp_servers/git/tools.py` (10 occurrences, all `True`)
- `scripts/mcp_servers/shell/tools.py` (1 occurrence, `True`)
- `scripts/mcp_servers/cicd/tools.py` (4 occurrences, all `True`)
- `scripts/mcp_servers/github/tools_file.py` (4 occurrences, all `True`)
- `scripts/mcp_servers/github/tools_issues.py` (5 occurrences, all `True`)
- `scripts/mcp_servers/github/tools_pull_requests.py` (6 occurrences, all `True`)
- `scripts/mcp_servers/github/tools_repository.py` (6 occurrences, all `True`)

Also in scope:

- Add a schema-validation test asserting no `requires_config` key exists in any
  server's `TOOL_LIST` and that `config_dependent` is present and is a `bool`
  on every tool. Add this to `tests/test_mcp_tools_validation.py` (the file
  named in the requirement, `tests/mcp_servers/test_tool_schema.py`, does not

- Update the 4 doc files that mention `requires_config`:
  `docs/04_mcp_04_05_git.md`, `docs/04_mcp_04_02_file-write-file-delete-shell.md`,
  `docs/04_mcp_04_03_rag-pipeline-and-cicd.md`,
  `docs/04_mcp_04_01_web-search-file-read-github.md`.

**Out of scope** (per requirement's own Scope section, which only lists the
static-field rename — confirmed no code implements this today):

- Implementing the dynamic `enabled` / `disabled_reason` runtime fields
  described in the requirement's "Intent" section. No code path currently
  computes or returns these; adding them is a materially larger feature
  (needs access to each server's live config state at request time) and is
  not listed in the requirement's Scope or Acceptance criteria.
- `scripts/mcp_servers/rag_pipeline/tools.py`, `web_search/tools.py`,
  `mdq/tools.py` — verified these have no `requires_config` field today (only

- A `sqlite` MCP server — verified it no longer exists in
  `scripts/mcp_servers/` (removed since an earlier plan referenced it), so no
  action needed.

## Assumptions

1. `config_dependent` carries the exact same boolean value `requires_config`
   had for every tool (pure rename, zero semantic change) — this matches the
   requirement's Scope ("all file tools ... true" / "all git tools ... true",
   which is what they already are as `requires_config`).
2. The requirement's target file `tests/mcp_servers/test_tool_schema.py` is
   stale/aspirational — verified no `tests/mcp_servers/` directory exists.
   The actual relevant tests are `tests/test_mcp_tools_validation.py` (has
   `test_read_tools_schema_matches_hand_written`, a natural home for the new
   assertion) and `tests/test_mcp_tool_schema_exports.py`. This plan adds the
   validation to `tests/test_mcp_tools_validation.py`.
3. No runtime code reads `requires_config` today (verified via
   `grep -rn "requires_config" scripts/agent scripts/shared scripts/rag scripts/db`
   — zero matches). The rename is therefore safe with no call-site fallout in
   `agent/`, `shared/route_resolver.py`, or `shared/tool_registry.py`
   (`tool_registry.py` only tracks `name`/`server_key`, never the metadata
   dict — confirmed by reading `scripts/shared/tool_registry.py` in full).
4. `list_tools_with_server_key()` in `scripts/mcp_servers/server.py` and each
   server's inline `/v1/tools` handler (`{**t, "server_key": ...}`) both just
   spread the tool dict verbatim, so `config_dependent` passes through
   automatically with no endpoint code changes needed.

## Design

- Pure mechanical rename at the dict-literal level: `"requires_config"` key
  becomes `"config_dependent"`, value unchanged. No schema/model class exists
  today that needs a field renamed (tool dicts are plain
  `list[dict[str, Any]]` — verified in `read_tools.py`), so this is a
  find-and-replace per file, not a type change.
- Add one new assertion-style test to `tests/test_mcp_tools_validation.py`
  that iterates every server's `TOOL_LIST` (import each of the 10+ modules)
  and asserts: (a) no dict contains the key `"requires_config"`, (b) every
  dict contains `"config_dependent"` and its value is `isinstance(..., bool)`.
  This directly satisfies the acceptance criteria "No MCP tool definition
  contains requires_config" and "All MCP tool definitions include boolean
  config_dependent" at test-collection time (fails fast, no server process
  needed — unlike the `@pytest.mark.integration` HTTP tests already in that
  file).
- Do not touch `enabled`/`disabled_reason` — no existing runtime hook to
  attach them to; introducing them would require new design work (per-request
  config-state inspection) not authorized by this requirement's Scope.
- Doc updates are text replacements of the `requires_config: true/false`
  inline-code mentions to `config_dependent: true/false`, preserving the
  surrounding Japanese doc prose.

## Implementation steps

1. In each of the 10 Python files listed in Affected areas, replace every
   `"requires_config"` key with `"config_dependent"` (value unchanged). Do
   this as a literal key rename, not a re-derivation, to guarantee no value
   drifts during the change.
2. In `tests/test_mcp_tools_validation.py`, add a new test function (near
   `test_read_tools_schema_matches_hand_written`) that imports `TOOL_LIST`
   from all 10 affected modules (plus the 3 unaffected ones, to guard against
   future regressions) and asserts the no-`requires_config` /
   has-`config_dependent`-bool invariant across all of them.
3. Update the 4 doc files' `requires_config` mentions to `config_dependent`.
4. Run the Validation plan below.
5. Confirm via `grep -rn "requires_config" scripts/ tests/ docs/` that zero
   matches remain anywhere in the repo.
