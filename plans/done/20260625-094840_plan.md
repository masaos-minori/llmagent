## Design

### New `┬¦Routing Source of Truth` section (at top of `04_mcp_03 ┬¦Tool Registry`)

```markdown
## Routing Source of Truth

The `ToolRouteResolver` resolves tool calls using exactly one authoritative source and two optional validation layers:

| Input | Role | Requirement |
|---|---|---|
| `shared/tool_constants.py` frozensets | **Authoritative static source** | Required ŌĆö all tool names must be in a frozenset |
| `shared/tool_registry.py` | **Routing authority** ŌĆö populated from frozensets at import time | Read-only at runtime; changes require code edit |
| Config `tool_names` (in `mcp_servers.toml`) | **Validation-only** ŌĆö checked for drift at startup | Optional; routing works without it |
| Live `/v1/tools` discovery | **Validation-only** ŌĆö startup drift check | Optional; skipped if servers unreachable |

**Summary of ownership rules:**
- To add a tool: only `tool_constants.py` is required. All other inputs are optional or automatic.
- `tool_names` in config: optional drift-check input. Not consulted for routing decisions.
- Static fallback: the frozensets in `tool_constants.py` ARE the static source. There is no runtime fallback ŌĆö if a tool name is missing from the registry, routing raises a `KeyError`.
```
