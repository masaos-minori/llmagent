# Plan: Normalize MDQ search production behavior and remove unsupported hybrid/embedding surface

Source requirement: `requires/20260716_09_require.md`

## Goal

Confirm (with fresh evidence) that MDQ's search surface no longer advertises
or executes an unsupported hybrid/embedding path, that `max_search_results`
vs `max_results_limit` ambiguity is resolved, and that RAG/session vector
schema remains untouched — then close this requirement as already resolved
rather than re-implementing work already completed.

## Scope

**In scope**

- Re-verify, with direct evidence gathered today, that every acceptance
  criterion in this requirement is already satisfied by the MDQ
  compatibility-cleanup batch implemented earlier on 2026-07-16
  (`plans/done/20260716-131500_plan.md`, "Simplify MDQ search configuration
  and remove unsupported hybrid search placeholders").
- Confirm the RAG/session vector schema (`chunks_vec`, `memories_vec` in
  `scripts/db/schema_sql.py`) is a distinct module from MDQ's own
  (now fully removed) vector-table code, so this requirement's "Scope
  protection" section was never actually at risk.
- Document the evidence trail in this plan.

**Out of scope**

- Any code change — none is needed.
- Implementing real hybrid/embedding search — explicitly a non-goal in both
  this requirement and the earlier plan that already resolved it.
- Any change to `scripts/db/schema_sql.py`'s `chunks_vec`/`memories_vec` —
  confirmed unrelated and untouched.

## Assumptions

1. This requirement was filed independently of (or before) the 2026-07-16
   MDQ compatibility-cleanup batch and describes the same root cause already
   fixed by `plans/done/20260716-131500_plan.md` — confirmed by comparing
   this requirement's "Background"/"Problem" sections (hybrid stub,
   `use_embedding`/`vector_table`/`embedding_model`/`embedding_dims`,

2. `scripts/db/schema_sql.py`'s `chunks_vec`/`memories_vec` virtual tables
   (RAG/session embedding storage) are a completely separate module from
   MDQ's `scripts/mcp_servers/mdq/db_schema.py` — verified by direct read;
   MDQ's own vector-table creation code was removed entirely (not merely
   disabled) by the earlier plan, so there was never any risk of the two
   being conflated in the implementation, only in this requirement's
   cautionary wording.

## Risks

- **Risk**: A future contributor unfamiliar with the 2026-07-16 cleanup
  batch could re-open this requirement and re-add `use_embedding`/hybrid
  scaffolding to MDQ, reintroducing the exact ambiguity this batch closed.
  **Mitigation**: `docs/04_mcp_04_04_mdq.md` and
  `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md` both document the
  removal with rationale and date; `tests/test_mdq_search_modes.py` would
  fail immediately if `mode` were widened beyond `Literal["bm25"]` without
  updating the tests deliberately.
- **Risk**: A future contributor could confuse MDQ's (removed) vector table
  with the RAG/session `chunks_vec`/`memories_vec` tables in
  `scripts/db/schema_sql.py` and mistakenly touch the wrong one.
  **Mitigation**: the two now live in entirely separate modules
  (`scripts/mcp_servers/mdq/db_schema.py` no longer creates any vector table
  at all) with no shared code path, minimizing confusion risk going forward.
