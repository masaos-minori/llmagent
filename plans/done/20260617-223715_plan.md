# Plan: Change SQLiteHelper.target to a safer typed value (req 68)

## Goal

Replace `SQLiteHelper(target: str)` with a typed target parameter so static type checkers can enforce only `"rag"`, `"session"`, or `"workflow"` values, and update docs to show the final accepted values.

## Scope

**In scope:**
- `db/helper.py` — add `DbTarget` StrEnum; change `SQLiteHelper.__init__` parameter type from `str` → `DbTarget | str`; keep string acceptance for backward compatibility with callers using string literals
- `docs/06_shared_04_db_architecture_and_schema.md` — update §4 to show new type annotation and valid values
- `docs/06_shared_05_db_api_and_operations.md` — update §2 to show new type annotation

**Out of scope:**
- No changes to ~28 call sites across `scripts/` (they all use string literals which are accepted via `DbTarget(str)` conversion)
- No changes to tests (they pass string literals; the constructor accepts both `str` and `DbTarget`)
- No changes to config files or schema files

## Assumptions

1. The safer type should be an Enum (not just `Literal["rag", "session", "workflow"]`) because:
   - The codebase already uses StrEnum for similar purposes (`TransportType`, `StartupMode`, `HealthcheckMode` in `mcp_config.py`)
   - StrEnum accepts string literals at call sites while providing enum-typed access for callers that want type safety
   - Example: `SQLiteHelper(DbTarget.RAG)` works, and `SQLiteHelper("rag")` also works (backward compatible)

2. The constructor should accept both `DbTarget` and `str` to avoid breaking ~28 existing call sites that use string literals.

3. Runtime validation stays the same — the `target not in ("rag", "session", "workflow")` check in `__init__` line 33 remains as a safety net for unexpected string values.

## Design

### New enum in `db/helper.py`

```python
from enum import StrEnum


class DbTarget(StrEnum):
    """SQLite database target type."""
    RAG = "rag"
    SESSION = "session"
    WORKFLOW = "workflow"
```

### Updated constructor signature

```python
def __init__(self, target: DbTarget | str = "rag") -> None:
    # Convert string to enum if needed
    if isinstance(target, str):
        try:
            self._target = DbTarget(target)
        except ValueError:
            raise ValueError(
                f"target must be 'rag', 'session', or 'workflow', got: {target!r}"
            )
    else:
        self._target = target  # already a DbTarget
    
    # Keep existing validation as safety net
    if self._target not in (DbTarget.RAG, DbTarget.SESSION, DbTarget.WORKFLOW):
        raise ValueError(
            f"target must be 'rag', 'session', or 'workflow', got: {self._target!r}"
        )
    
    self._default_load_vec = self._target == DbTarget.RAG
```

### Updated target comparison logic

All `if self._target == "rag"` comparisons in the existing code need to be updated to use enum members:
- Line 46: `if target == "rag":` → `if self._target in (DbTarget.RAG, "rag"):` or keep string comparison since StrEnum values are strings
- Lines 48-51: same pattern

Actually, since `DbTarget` is a **StrEnum**, `self._target == "rag"` still works — `DbTarget.RAG == "rag"` returns `True`. So no comparison logic changes are needed if we store the enum value. But to keep it simple and avoid breaking any edge cases, we can store `self._target` as a string (the `.value` of the enum) for all existing comparisons.

Simpler approach — store as string internally but accept both types at the constructor:

```python
def __init__(self, target: DbTarget | str = "rag") -> None:
    if isinstance(target, DbTarget):
        resolved = target.value
    else:
        # Validate string value
        if target not in ("rag", "session", "workflow"):
            raise ValueError(
                f"target must be 'rag', 'session', or 'workflow', got: {target!r}"
            )
        resolved = target
    
    self._target = resolved  # stored as str for backward compatibility
```

This approach:
- Accepts `DbTarget.RAG` (enum) or `"rag"` (string) at the constructor
- Stores internally as `str` so all existing comparisons (`self._target == "rag"`) work unchanged
- No changes needed to any of the 28+ call sites
- Type checkers can still see that `DbTarget` is a valid input type
