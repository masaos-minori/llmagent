# Plan: Clean up MDQ indexing control configuration and serialization behavior

Source requirement: `requires/20260716_06_require.md`

## Goal

MDQ indexing/refresh concurrency is provably serialized by
`MdqService._index_lock` (with a test proving it), `enable_refresh` is
removed from config and service (confirmed dead — read but never enforced),
`enable_grep` remains as a documented, enforced feature gate, and no
reference to the already-removed `concurrency_limit` remains in docs or
examples.

## Scope

**In scope**

- **`concurrency_limit`**: confirm zero runtime footprint (already true) and
  audit docs/examples for stray mentions beyond the existing removal note.
- **Serialization consistency**: confirm (already true, add proof via test)
  that `index_paths` and `refresh_index` both carry `requires_serial: True`
  in `tools.py` and both acquire `MdqService._index_lock` before running.
  Add `tests/test_mdq_index_serialization.py` (new — the requirement's
  `tests/test_mdq_index*.py` target does not exist yet) with a concurrency
  test that starts two overlapping `index_paths`/`refresh_index` calls via
  `asyncio.gather` and asserts they do not interleave (e.g. via a
  monkeypatched sleep inside the critical section, or by asserting
  `_is_indexing` toggles correctly and no double-execution occurs).
- **`enable_refresh` gate cleanup**: remove `self.enable_refresh` from
  `service.py` and `enable_refresh` from `config/mdq_mcp_server.toml` —
  selecting removal over enforcement (see Design for why).
- **`enable_grep` gate classification**: no code change — confirm (already
  true) it is a supported, enforced production feature gate; keep as-is.
  Add one clarifying doc line distinguishing it from the removed
  `enable_refresh` (both looked similar in config, but only one was ever
  wired to real behavior).
- Update `docs/04_mcp_04_04_mdq.md` (drop `enable_refresh` from the field
  list) and `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md` (document the
  internal-lock serialization model explicitly as the production mechanism,
  and note `enable_refresh` removal alongside the existing
  `concurrency_limit` removal note).

**Out of scope**

- `audit_log_path` (requirement 01), `max_search_results`/hybrid search
  (requirement 04), summary cache (requirement 05) — disjoint config keys in
  the same files.

## Assumptions

1. `concurrency_limit` has zero runtime footprint today — already verified
   and documented in `config/mdq_mcp_server.toml:62-66`
   ("removed (2026-07-13)... never read anywhere in scripts/mcp_servers/mdq/")
   and `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md:83-87`. This plan
   only needs to confirm no regression reintroduced it — re-verified via
   `rg -n "concurrency_limit" scripts/` (no matches outside the removal-note
   comments).
2. `self.enable_refresh` is read in `service.py:65`
   (`mdq_cfg.get("enable_refresh", True)`) but never referenced anywhere else
   — verified via `rg -n "enable_refresh" scripts/mcp_servers/mdq/*.py`
   (single hit, the assignment itself). `refresh_index()` has no gate check
   on it. This is a genuinely dead flag, not a documented future feature —
   unlike `enable_grep`.
3. `self.enable_grep` **is** enforced: `grep_docs()` raises
   `MdqValidationError("grep_docs is disabled by configuration")` when
   `not self.enable_grep` (`service.py:393-394`), and has direct test
   coverage (`tests/test_mdq_service.py:632-644`,
   `TestGrepDocsConfigGate` — verified by direct read). This is the key
   evidence distinguishing it from `enable_refresh`: one gate is wired to
   real behavior, the other is not.
4. `requires_serial: True` is already set on both `index_paths` and
   `refresh_index` in `tools.py:106-107,124-125` (verified), and both
   `MdqService.index_paths()` and `MdqService.refresh_index()` acquire
   `self._index_lock` (an `asyncio.Lock`, lazily created) before running
   (verified in `service.py:319-342`). No code change is needed here — only
   a test proving the lock actually prevents interleaved execution, since no
   such test exists today (verified — no `test_mdq_index*.py` file exists,
   and `tests/test_mdq_incremental_refresh.py` tests functional correctness
   only, not concurrency).
5. `requires_serial` at the `tools.py`/`MCPToolSchema` level is a *different*
   serialization mechanism from `MdqService._index_lock`: the former is
   consumed by `scripts/agent/tool_scheduler.py` as a global barrier across
   concurrent tool calls *within one agent turn* (verified —
   `tool_scheduler.py:75,131,133`); the latter is an in-process lock inside
   `MdqService` that protects against any overlapping `index_paths`/
   `refresh_index` execution regardless of caller. Both are legitimate and
   complementary — this plan documents the distinction rather than merging
   or removing either.

## Design

`enable_refresh` is **removed**, not enforced, because:

- It has no documented intent as a future feature (unlike the hybrid-search
  scaffolding in requirement 04) — it reads as an oversight, not a planned
  gate.
- The established pattern in this same config file for read-but-unenforced
  keys is removal: `audit_log_path` (requirement 01) and `concurrency_limit`
  (already removed 2026-07-13) both followed this precedent. Adding
  enforcement instead would be new behavior (a way for operators to disable
  `refresh_index`), which was never requested and is exactly the kind of
  speculative feature addition the project's `AGENTS.md` policy warns
  against ("Do not generate code... speculatively").
- The requirement's own task list offers both options ("If unused, remove...
  If retained, enforce...") and its "Intent" section frames the overall goal
  as simplification, which favors removal absent an active use case.

The serialization test uses the internal lock directly rather than mocking,
so it proves the real production mechanism:

```python
import asyncio

async def test_concurrent_refresh_calls_are_serialized(service, md_file):
    order: list[str] = []

    original_refresh = service._is_indexing  # baseline reference, illustrative

    async def tracked_refresh(paths):
        order.append("start")
        result = await service.refresh_index(RefreshIndexRequest(paths=paths))
        order.append("end")
        return result

    results = await asyncio.gather(
        tracked_refresh([str(md_file)]),
        tracked_refresh([str(md_file)]),
    )
    # With serialization, order is start,end,start,end — never start,start,end,end
    assert order in (["start", "end", "start", "end"],)
```

(Exact assertion mechanics to be finalized during implementation — the key
property is that `_index_lock` prevents the second call's critical section
from beginning before the first's ends.)

## Risks

- **Risk**: The concurrency test could be flaky if it relies on real
  `asyncio` scheduling timing rather than a deterministic hook.
  **Mitigation**: use a monkeypatched delay or an explicit event/barrier
  inside the locked section (not a bare `asyncio.sleep` race) so the test is
  deterministic; finalize the exact mechanism during implementation review
  rather than committing to real-time timing here.
- **Risk**: Overlap with `requires/20260716_01_require.md`, which originally
  listed `enable_refresh` in its own task list.
  **Mitigation**: plan 01 explicitly defers this key to this requirement
  (06); implementing in filename order avoids duplicate edits to
  `service.py` and `config/mdq_mcp_server.toml`.
