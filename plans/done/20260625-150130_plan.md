# Plan: [High][eventbus][architecture] subscribe() をリプレイ (SQLite) とライブ (in-memory) のハイブリッドに再設計

Source: `requires/20260625_37_require.md`

## Goal

`/subscribe` の SSE ジェネレーターを「SQLite からのリプレイ → broker queue からのライブ配信」の2フェーズ構成に再設計し、ポーリングを廃止しながら `since_seq`・`consumer_id` のリジューム語義を維持する。

## Scope

**In-Scope**
- `scripts/eventbus/app.py` の `subscribe()` SSE ジェネレーター全面改定
- リプレイフェーズ: SQLite から `seq > since_seq` を全件取得後に broker queue に切替
- ライブフェーズ: `asyncio.Queue.get()` (await) で待機、新イベントを即座に yield
- リプレイ→ライブ遷移時の重複除去 (broker に登録後にリプレイ開始し、seq でフィルタ)
- 再接続リカバリー: 同じ `since_seq` + SQLite リプレイで自動復旧
- `poll_interval_ms`, `offset_checkpoint_interval` 設定の廃止 (deprecated)

**Out-of-Scope**
- broker 実装 (req #35 対象)
- publish() の broker 通知 (req #36 対象)
- topic fan-out の詳細 (req #39 対象)
- 新テスト (req #38 対象)

## Assumptions

1. req #35 (broker) と req #36 (publish notify) が実装済みであること
2. broker への登録は SSE ジェネレーター開始時に行い、リプレイ前に完了する
3. リプレイフェーズで取得した最後の `seq` (= `replay_ceil`) を記録し、queue から取り出したイベントの seq が replay_ceil 以下なら重複として破棄する
4. `since_seq == 0 AND consumer_id` の場合は既存の `read_offset()` でオフセットを復元 (現行ロジック維持)
5. ライブフェーズで `asyncio.CancelledError` を受けたら broker から unsubscribe して終了
6. `poll_interval_ms` 設定は config に残すが subscribe() からは参照しない (deprecated コメント追加)

## Design

**新 `subscribe()` SSE ジェネレーターの疑似コード:**

```
subscribe():
  start_seq = オフセット復元 (consumer_id あり && since_seq == 0)
  assert _broker is not None

  # Step 1: broker に登録 (リプレイ前に行うことで遷移期イベントを queue に積む)
  sub = _broker.subscribe(topic)
  try:
    # Step 2: SQLite リプレイ (since_seq より新しいイベントを全件取得)
    replay_rows = DB.execute("SELECT ... WHERE seq > start_seq ORDER BY seq").fetchall()
    replay_ceil = start_seq  # リプレイで届けた最後の seq
    for row in replay_rows:
      yield SSE(row)
      replay_ceil = row["seq"]

    # Step 3: ライブ配信 (broker queue から)
    while True:
      event = await sub.queue.get()
      if event is None:  # shutdown sentinel
        break
      if event["seq"] <= replay_ceil:
        continue  # リプレイ済み → 重複破棄
      yield SSE(event)

  except asyncio.CancelledError:
    logger.info("subscribe disconnected consumer=%s", consumer_id)
  finally:
    _broker.unsubscribe(sub)
```

**リプレイ→ライブ遷移の競合安全性:**

```
時刻  イベント
T0   broker.subscribe() → sub (queue 空)
T1   publish(A) → DB commit → broker.publish(A) → sub.queue.put(A)  [seq=5]
T2   リプレイ: DB query → rows=[A(seq=5), ...]  replay_ceil=5
T3   queue から A を取り出し → seq=5 <= replay_ceil=5 → 重複破棄 ✓
T4   publish(B) → broker.publish(B) → sub.queue.put(B)  [seq=6]
T5   queue から B を取り出し → seq=6 > replay_ceil=5 → yield ✓
```

**再接続リカバリー:**
- SSE 切断後に再接続 → `consumer_id` + offset / `since_seq` でリプレイから再開
- broker queue は再接続前の publish を受け取っていない → 全件 SQLite リプレイが補完する
- miss なし、gap なし

## Implementation Steps

1. `subscribe()` の SSE ジェネレーターを2フェーズ構成に書き換え
2. リプレイフェーズ: SQLite クエリ (asyncio.to_thread 経由) → yield
3. `replay_ceil` を記録し broker queue 読み取りへ切替
4. ライブフェーズ: `await sub.queue.get()` → seq フィルタ → yield
5. finally ブロックで `_broker.unsubscribe(sub)`
6. `poll_interval_ms` 参照を削除、設定項目には deprecated コメント追加
7. `events_since_checkpoint` と mid-stream write_offset を削除 (req #29 に既に設計済みの場合は統合)

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| リプレイフェーズが長い場合 queue が溢れる | Medium | `maxsize=1000` と drop+WARNING で対処。リプレイが完了するまでに溢れたイベントは replay 自体が補完する |
| broker.unsubscribe() が finally で呼ばれない場合 (例外パス) | Low | `try/finally` で確実に実行 |
| `asyncio.to_thread` 内で SQLite をクエリする場合、broker queue への put と ordering が逆転しないか | Low | リプレイクエリは to_thread で実行。broker.publish() はイベントループ上で実行。リプレイ後に queue を読むため ordering は常に正しい |
