# Plan: Add fallback title when session title generation fails

## Goal

Ensure sessions always have useful titles even when LLM-based title generation fails. Fix race conditions where empty titles appear, and extend fallback to all session creation paths.

## Scope

**In scope:**
- Verify existing fallback in `_generate_session_title()` works correctly (already uses truncated first input)
- Fix race condition: show "(generating...)" or similar when title generation is in progress
- Extend title generation trigger to `/context new` sessions (currently only fires on REPL startup first turn)
- Add explicit "pending" state for title generation lifecycle tracking
- Handle edge case: empty first input → use deterministic fallback like "(New Session)"

**Out of scope:**
- Changing the LLM prompt or title generation algorithm
- Adding manual title editing command (user can use `/rename` today)
- Title generation retry logic with exponential backoff
- Persistent "pending" state across session reloads (pending is session-scoped)

## Assumptions

1. Fallback already exists in `cmd_session.py:40-46`: truncated first input (max 32 chars) when title generation fails.
2. `SessionTitleService.generate()` runs as detached `asyncio.create_task()` — race condition with session list is expected.
3. `/context new` creates sessions mid-session via `conversation_service.py:30-36`; currently does NOT trigger title generation because `stat_turns == 1` check prevents it.
4. The `sessions.title` column is nullable — NULL means no title set yet or generation failed.

## Unknowns

| Unknown | Evidence missing | Resolution | Blocking |
|---|---|---|---|
| Whether `/context new` should trigger title generation | It creates a new session with potentially different context | If yes, need to pass first user message of the new session to title generator; if no, use generic fallback like "(New Session)" | no |
| How to track "pending" state without DB schema change | No `title_status` column exists | Use in-memory tracking in `AgentSession` or `Context`; persist fallback title to DB immediately when generation fails (already done) | no |

## Implementation Steps

1. **Strengthen existing fallback in `_generate_session_title()`**
   - Current: truncated first input (max 32 chars)
   - Add: if first_input is empty or whitespace-only, use `"(New Session)"` as fallback
   - Add: log level info (not just warning) when fallback is used, so users can see it in diagnostics
   - Completion criterion: all failure paths produce a non-empty title

2. **Add pending state tracking**
   - Add `_title_pending: bool = False` flag to `AgentSession` in `session.py`
   - Set `_title_pending = True` when title generation task is created
   - Set `_title_pending = False` when generation completes (success or failure)
   - Expose `is_title_pending()` method for CLI display

3. **Fix race condition in session list display**
   - In `_session_list()`: check if current session has pending title
   - If pending, show `"(generating...)"` instead of empty string
   - Completion criterion: session list never shows blank title for current session

4. **Extend title generation to `/context new` sessions**
   - In `conversation_service.py`: after creating new session, trigger title generation with a prompt derived from the last few messages before the switch (or use generic "(New Session)" if no context)
   - In `orchestrator.py`: decouple title generation from `stat_turns == 1` check; allow explicit trigger via `on_new_session` callback
   - Completion criterion: all new sessions get title generation attempt

5. **Update session list UX**
   - Show `(generating...)` for pending titles
   - Show `(no title)` for NULL titles that are not pending (generation already failed and no fallback was applied — should not happen after step 1)
   - Completion criterion: session list always shows meaningful title text

6. **Update command docs**
   - Document fallback behavior in `/db` or session-related help
   - Note that title generation is async and may take a few seconds
