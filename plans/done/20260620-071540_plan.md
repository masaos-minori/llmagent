# Plan: Elevate RAG Consistency Checking to Normal Operational Diagnostics (req_39)

## 1. Goal

Make RAG index consistency (chunks/FTS/vec sync) visible in normal operations —
on startup warning and in `/db consistency` output — without requiring a separate
maintenance-time manual check.

## 2. Scope

**In-Scope:**
- Add lightweight RAG consistency check to `StartupOrchestrator._check_services()`:
  if rag.sqlite exists, run `check_rag_consistency()` and emit `write_warning()` if
  inconsistent
- Enhance `/db consistency` output: show numeric details (`fts_gap`, `orphan_vec_count`,
  `vec`, `chunks`) in addition to the OK/FAIL summary
- Document the operational role of consistency checks and startup warning in
  `docs/03_rag_02_ingestion_pipeline.md` or relevant doc

**Out-of-Scope:**
- Automatic repair of consistency issues
- Replacing SQLite triggers
- Threshold-based partial-OK policy (marked as: Needs confirmation)

## 3. Confirmed Facts

**`check_rag_consistency()` is read-only and fast (3 COUNT queries):**
```python
chunks = db.execute("SELECT COUNT(*) FROM chunks").fetchone()[0]
fts    = db.execute("SELECT COUNT(*) FROM chunks_fts_docsize").fetchone()[0]
vec    = db.execute("SELECT COUNT(*) FROM chunks_vec").fetchone()[0]
orphan_vec_count = db.execute("SELECT COUNT(*) FROM chunks_vec ...").fetchone()[0]
```
No write operations; safe to run on every startup.

**`is_consistent()` check is strict zero:** `fts_gap==0 and fts_orphan_count==0 and orphan_vec_count==0 and vec==chunks`

**Current `/db consistency` output:**
- "RAG consistency: OK (chunks/FTS/vec in sync) [RAG]" on success
- `write_error(f"Consistency issue: {issue}")` per issue on failure
- **Missing:** numeric details (fts_gap=N, orphan_vec_count=N, etc.)

**Startup (`startup.py:_check_services()`):**
- Already calls `check_readiness()` and `check_tool_definitions_runtime()` with warnings
- Does NOT run `check_rag_consistency()`
- Pattern exists for `write_warning()` in startup

**Ingestion (`ingester.py:123`):**
- `check_rag_consistency()` is called post-ingest — but only when `ingest_all()` runs,
  not on normal agent startup

## 4. Assumptions

1. rag.sqlite may not exist (fresh deployment, RAG disabled) — guard with `try/except` or
   file-existence check before running startup consistency check
2. Startup consistency check must be non-blocking and fast — acceptable since it's 3 COUNTs
3. Threshold policy (e.g. fts_gap <= 5 is acceptable) is NOT defined — keep strict-zero
   check for now; mark as "Needs confirmation" for future tuning
4. The consistency check should NOT appear in the startup banner (too verbose for every
   session) — use `logger.info` on success; `write_warning()` only on failure

## 6. Affected Areas

**Affected Files:**
- `scripts/agent/startup.py`:
  - In `_check_services()`: add RAG consistency check after existing readiness checks
  ```python
  try:
      consistent, issues = RagMaintenanceService().consistency()
      if consistent:
          logger.info("RAG consistency: OK")
      else:
          for issue in issues:
              self._view.write_warning(f"[RAG] Consistency issue: {issue}")
  except Exception as e:
      logger.debug("RAG consistency check skipped: %s", e)
  ```

- `scripts/agent/commands/cmd_db.py`:
  - In `_db_consistency()`: show numeric details from `RagConsistencyReport`
  ```python
  def _db_consistency(self) -> None:
      consistent, issues = RagMaintenanceService().consistency()
      report = check_rag_consistency(db)  # get raw report for numbers
      self._out.write(
          f"  chunks: {report.chunks}  fts: {report.fts}  vec: {report.vec}"
          f"  fts_gap: {report.fts_gap}  orphan_vec: {report.orphan_vec_count}"
      )
      if consistent:
          self._out.write_success("RAG consistency: OK [RAG]")
      else:
          for issue in issues:
              self._out.write_error(f"Consistency issue: {issue} [RAG]")
  ```
  Note: need to expose `check_rag_consistency` in `RagMaintenanceService` or call it directly

  - Document startup warning and `/db consistency` command output format
  - Note: threshold policy is strict-zero (Needs confirmation for tuning)

## 9. Risks

- **Risk:** Startup consistency check adds latency if rag.sqlite is large
  → **Mitigation:** 3 COUNT queries are O(1) on SQLite; safe for startup path
- **Risk:** rag.sqlite does not exist on fresh deployment — check raises
  → **Mitigation:** wrap in `try/except Exception` in startup, log and skip
- **Risk:** `_db_consistency()` now requires two calls to get raw report and summarized issues
  → **Mitigation:** refactor to call `check_rag_consistency(db)` once, derive both
    `is_consistent()` and `summarize_issues()` from the same report
