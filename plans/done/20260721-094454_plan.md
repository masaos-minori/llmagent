# Plan: Restore `[[tool_definitions]]` coverage for git/mdq/file_write tools; fix inert `routing_drift_strict`

Source requirement: `requires/20260721_05_require.md`

## Goal

`config/agent.toml` exposes complete, accurate `[[tool_definitions]]` (LLM function-calling schema) entries
for all 19 currently-registered-but-undefined tools (file_write: `edit_file`, `move_file`; git: 10 tools;
mdq: 7 tools), and `routing_drift_strict = true` actually causes `StartupOrchestrator` to abort startup with
a `RuntimeError` when routing drift is detected — not just log a warning — while `routing_drift_strict = false`
(the default) continues to only warn.

## Scope

**In scope:**
- Add 19 `[[tool_definitions]]` TOML blocks to `config/agent.toml`, following the existing style used for the
  45 current entries (see Design → A. TOML pattern).
- Pass `strict=ctx.cfg.tool.routing_drift_strict` into the `check_routing_drift(ctx)` call at
  `scripts/agent/startup.py:252`.
- Narrow or restructure the surrounding `except Exception` at `scripts/agent/startup.py:254` so that a
  `RuntimeError` raised by `check_routing_drift(..., strict=True)` actually propagates as a fatal startup
  outcome instead of being swallowed and downgraded to a warning (see Design → C, and Unknowns → resolved item).
  This is a necessary companion change discovered during planning; it is required to satisfy the requirement
  doc's own Acceptance Criteria ("`routing_drift_strict=true` aborts startup on injected drift via the real
  `startup.py` code path"), even though the requirement doc's "Required Changes" section only mentions adding
  `strict=`.
- Add/extend a regression test exercising the real `StartupOrchestrator._check_services()` code path (not a
  direct `check_routing_drift(strict=True)` unit call — that already exists and already passes) to prove both
  `routing_drift_strict=true` (aborts) and `routing_drift_strict=false` (warns only) end-to-end.

**Out of scope (explicitly, per requirement doc "Non-Goals"):**
- `ProductionConfigValidator` / production-hardening logic from `requires/done/20260709_03_require.md` /
  `requires/done/20260709_08_require.md`.
- The already-filed `[mcp_servers.*].cmd` stale-path bug (`requires/done/20260721_01_require.md`).
- Any change to `RuntimeToolRegistry` dynamic discovery behavior — this task only touches the static
  `config/agent.toml` fallback list.
- Adding `fts_consistency_check` / `fts_rebuild` to any config — confirmed removed from the codebase (see
  Unknowns).

4. Fixing the `except Exception` swallow at `startup.py:254` for the routing-drift check only does not change

   this is a narrowly-scoped, low-blast-radius change.
5. No other call site invokes `check_routing_drift()` in a way that would be affected by widening its
   exception handling at the `startup.py:252` site (falsifiable: `rg "check_routing_drift"` across `scripts/`

## Design

### A. `[[tool_definitions]]` TOML pattern to replicate (verbatim reference from file_read fix)

Confirmed existing style (from `config/agent.toml`, e.g. `list_directory` lines 410-424, `read_text_file`
lines 425-446, `read_multiple_files` lines 1209-1223):

```toml
[[tool_definitions]]
type = "function"

[tool_definitions.function]
name        = "<tool_name>"
description = "<Japanese description>"

[tool_definitions.function.parameters]
type     = "object"
required = ["<required_param_1>", ...]

[tool_definitions.function.parameters.properties.<param>]
type        = "string" | "integer" | "boolean" | "array"
description = "<Japanese description>"

# for array-typed params, add a nested .items sub-table:
[tool_definitions.function.parameters.properties.<array_param>.items]
type = "string"
```

All 19 new blocks must follow this exact shape. `config/agent.toml` currently has zero `[[tool_definitions]]`
entries for any of the 19 target tools (confirmed via `grep -n '^name        = "'`), and `[mcp_servers.*].tool_names`
/ `[tool_safety_tiers]` already list/tier all 19 correctly — only the `[[tool_definitions]]` layer is missing,
unlike the file_read fix which also had to extend `tool_names`.

### B. file_write server — 2 tools (source: `scripts/mcp_servers/file/write_tools.py`)

| Tool | Required | Optional | Description basis |
|---|---|---|---|
| `edit_file` | `path` (string), `edits` (array of objects, each with required `old_text` (string) + `new_text` (string)) | `dry_run` (boolean) | "Apply string replacements to a file. When dry_run=true, return only the diff without writing" |
| `move_file` | `source` (string), `destination` (string) | `dry_run` (boolean) | "Move or rename a file or directory. When dry_run=true, return feasibility info without moving" |

`edit_file`'s `edits` array needs a nested object schema: `properties.edits.items` = `{type: "object",
properties: {old_text: {type: string}, new_text: {type: string}}, required: [old_text, new_text]}` — this is
a new pattern not present in any existing `[[tool_definitions]]` entry (existing array params, e.g.
`read_multiple_files.paths`, are flat string arrays). TOML nested-array-of-objects syntax must be verified to
render correctly (e.g. via `python -c "import tomllib; ..."` round-trip) during implementation, since this is
structurally more complex than any existing entry.

### E. `routing_drift_strict` wiring fix (source: `scripts/agent/startup.py`, `scripts/agent/repl_health.py`)

Current code (`startup.py:250-255`):
```python
# 5. Routing drift (static)
try:
    for msg in check_routing_drift(ctx):
        pipeline.add_warning("routing_drift", msg)
except Exception as exc:  # noqa: BLE001
    pipeline.add_warning("routing_drift", f"Routing drift check failed: {exc}")
```

`check_routing_drift(ctx: AgentContext, *, strict: bool = False) -> list[str]` (`repl_health.py:365`) always
computes and returns/logs drift warnings; when `strict=True` and drift is found, it additionally
`logger.error(...)` and `raise RuntimeError(msg)` (`repl_health.py:383-387`).

`ctx.cfg.tool.routing_drift_strict: bool = False` is defined in `ToolConfig`
(`scripts/agent/config_dataclasses.py:164`, comment: "Treat config/registry routing drift as fatal at startup;
False = warn only"). It is currently **absent from `config/agent.toml`** entirely (confirmed via grep — no
`routing_drift_strict` key anywhere in the file), so the dataclass default `False` is always in effect
regardless of intent — this is a second, independent reason the feature is currently inert beyond the missing
`strict=` kwarg (operators cannot even set it to `true` today without adding the key to the TOML themselves,
which is undocumented as a gap but not itself a bug — `false` is a safe default).

Required fix — two coordinated changes at `startup.py:250-255`:
```python
# 5. Routing drift (static)
try:
    for msg in check_routing_drift(ctx, strict=ctx.cfg.tool.routing_drift_strict):
        pipeline.add_warning("routing_drift", msg)
except RuntimeError as exc:
    pipeline.add_fatal("routing_drift", str(exc))
except Exception as exc:  # noqa: BLE001
    pipeline.add_warning("routing_drift", f"Routing drift check failed: {exc}")
```
(Exact exception variable naming / message format to be finalized during implementation; the key structural
requirement is that the `strict=True` `RuntimeError` path reaches `pipeline.add_fatal`, not
`pipeline.add_warning`, so that `pipeline.has_fatal` at `startup.py:281` triggers the final
`raise RuntimeError(f"Startup validation failed: {fatal_str}")` at line 287.)

This changes the requirement doc's literal "Required Changes §2" (which only says to add `strict=`) — the
plan expands this into two changes because the one-line fix alone would not satisfy the doc's own Acceptance
Criteria (see Unknowns → UNK-02).

### Relationship to other requirement docs

- `requires/done/20260711_04_require.md`: established the exact fix pattern this plan reuses for the
  `[[tool_definitions]]` gap (file_read → now file_write/git/mdq). That doc's own coverage snapshot (git 10/10,
  mdq 9/9) has regressed to 0/10 and 0/7 respectively since 2026-07-11; this plan is not a re-application of an
  unfixed bug but a fresh regression in previously-fixed servers, plus first-time coverage for file_write.
- `requires/done/20260709_03_require.md` / `requires/done/20260709_08_require.md`: propose a
  `ProductionConfigValidator` that would make `routing_drift_strict=false` fatal specifically *in production*
  security profiles. Those designs assume `check_routing_drift()`'s `strict` parameter is already correctly
  wired to the config value at the `startup.py` call site. **This plan's Gap 2 fix is a prerequisite** for
  that future production-hardening logic to have any real effect — without it, `ProductionConfigValidator`
  would validate a config flag (`routing_drift_strict`) whose value never reaches the actual drift-detection
  code path. This plan does not implement `ProductionConfigValidator` itself (explicit non-goal), only removes
  the blocking prerequisite bug.

## Risks

- **Risk**: The nested array-of-objects schema for `edit_file.edits` (Design → B) has no existing precedent
  in `config/agent.toml`'s TOML dotted-table syntax, so the correct TOML encoding is non-obvious and could be
  written incorrectly (e.g., producing a schema that doesn't actually validate `old_text`/`new_text` as
  required within each array item). Likelihood: medium.
  **Mitigation**: During implementation, round-trip the new TOML through `tomllib.load()` and compare the
  resulting Python dict structure against the equivalent `inputSchema` dict literal in
  `scripts/mcp_servers/file/write_tools.py` (lines 36-73) to confirm structural equivalence before considering
  the block correct.
- **Risk**: Restructuring the `except` block at `startup.py:254` (Design → E) is a change beyond what the
  requirement doc's "Required Changes" section literally lists, and could be seen as scope creep or could
  introduce an unintended behavior change to the 7 other checks in `_check_services()` if the edit is not
  narrowly scoped to only the routing_drift try/except block. Likelihood: low (each check has its own
  independent try/except already).
  **Mitigation**: Confirm via code read that each of the 8 checks in `_check_services()` has its own separate
  try/except (already confirmed in Step 3 research — yes), so narrowing exception handling for check #5 only
  cannot affect checks #1-4, #5b, #6. Document this expansion explicitly in the plan (done above) so a reviewer
  can evaluate it against the requirement doc's Acceptance Criteria rather than being surprised by it.
- **Risk**: `check-mcp-docs --skip toolcount` note in `rules/toolchain.md` implies there is a `toolcount`
  consistency check that could be sensitive to the tool_definitions count changing from ~45 to ~64. Likelihood:
  medium (an automated check specifically about tool counts is a plausible place for this change to surface
  friction).
  **Mitigation**: Run `uv run check-mcp-docs` (without `--skip toolcount`) as part of Phase 4 validation to
  confirm no doc/count mismatch is introduced; if it fails, investigate whether `docs/*.md` tool-count
  references (e.g., in `docs/04_mcp_04_server_catalog.md` or similar) need a corresponding update — flagged
  here as a follow-up validation step rather than a certainty, since the actual check's sensitivity was not
  directly inspected during this planning cycle.
- **Risk**: `mode` parameter on `search_docs` is documented as enum `["bm25"]` (single-value enum) — if the
  TOML schema for enums is written incorrectly (e.g., as a plain string type without an `enum` constraint), the
  LLM-facing schema would be less precise than the actual server-side `inputSchema`, though not functionally
  broken (the server still validates). Likelihood: low.
  **Mitigation**: Match the source `inputSchema` exactly, including the `enum` array, if the existing
  `[[tool_definitions]]` TOML style in this file supports representing enums (verify by checking whether any
  of the 45 existing entries already uses an enum-constrained parameter; if not, this would be a new pattern
  requiring the same round-trip verification as the array-of-objects case above).

All risks above have documented mitigations within this plan; none require escalation to
`issues/{timestamp}_risks.md` at this time (Step 6 issue-filing is skipped — all risks are addressable during
implementation without external input).
