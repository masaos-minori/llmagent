## 1. Goal

Enforce safety, serialization, and cache-invalidation controls for `index_paths` and `refresh_index` so they are treated as side-effect tools throughout the agent and MCP layers.

## 2. Scope

- **In-Scope**:
  - Add `index_paths` and `refresh_index` to `_SIDE_EFFECT_TOOLS` in `scripts/shared/tool_executor.py`
  - Add a dedicated `MDQ_WRITE_TOOLS` frozenset in `scripts/shared/tool_constants.py` and reference it from `tool_executor.py`
  - Fix `config/agent.toml` `[tool_safety_tiers]`: replace the incorrect `mdq = "WRITE_DANGEROUS"` entry with per-tool entries (`index_paths`, `refresh_index`, `fts_rebuild` → `WRITE_DANGEROUS`; `fts_consistency_check` → `ADMIN`; remaining MDQ read tools → `READ_ONLY`)
  - Add cache invalidation: after a successful `index_paths` or `refresh_index` call, invalidate MDQ-keyed entries in `ToolExecutor._cache` (prefix eviction or selective clear)
  - Verify `MdqService._index_lock` (asyncio.Lock) already serializes concurrent index ops within a single process (already implemented — confirm test coverage)
  - Confirm `_is_indexing` flag in `MdqService.search_docs` gives a degraded-search warning during indexing (already implemented — confirm test coverage)
- **Out-of-Scope**:
  - Cross-process / multi-worker locking (SQLite WAL + busy_timeout is the existing inter-process guard)
  - Adding `resource_scope` to MCP tool schema (already present as `requires_serial: True` in `tools.py`)
  - Hybrid search implementation (separate MDQ-02 issue)
  - Changing the MDQ SQLite schema

## 3. Assumptions

- The agent runs a single `ToolExecutor` instance per session; asyncio.Lock in `MdqService` is sufficient for intra-process serialization.
- SQLite WAL mode + `busy_timeout=5000ms` is the production guard against inter-process races (multiple uvicorn workers).
- `_SIDE_EFFECT_TOOLS` in `tool_executor.py` drives the parallel→serial downgrade in `tool_runner._execute_standard`; adding MDQ write tools there is the correct integration point.
- `clear_cache()` in `ToolExecutor` drops all cached entries indiscriminately; a selective MDQ-prefix eviction is preferable but requires a small refactor of `_execute_with_cache`.
- `fts_rebuild` (admin tool) also modifies the index and should be treated as `WRITE_DANGEROUS` / `ADMIN`.

## 8. Risks & Mitigations

- **Risk**: Adding MDQ write tools to `_SIDE_EFFECT_TOOLS` downgrades all batches containing those tools to serial execution, increasing latency when an LLM batches `search_docs` with `index_paths`. → **Mitigation**: This is the intended behavior per the requirement. The LLM should not batch read and write MDQ tools together. If needed, add an explicit warning in `_execute_standard` log when this downgrade occurs for MDQ tools.
- **Risk**: `evict_server("mdq")` iterates the full `_cache`; on large caches this may cause a momentary pause. → **Mitigation**: Cache max size is bounded by `tool_cache_max_size` (config default). Iteration is O(n) over a small set (default ≤ 128 entries); acceptable.
- **Risk**: `fts_consistency_check` classified as `ADMIN` requires "yes" confirmation, which may surprise users who treat it as a read operation. → **Mitigation**: `fts_consistency_check` is a read-only consistency probe; reclassifying as `READ_ONLY` is also defensible. Defer to operator preference. Document in config comment.
- **Risk**: The existing `mdq = "WRITE_DANGEROUS"` entry in `tool_safety_tiers` currently has no effect (no tool is named `mdq`) — removing it is safe but should be noted in the commit message to avoid confusion.
