# Markdown Context Compression Engine MCP Server — Implementation Plan

## Goal
Implement a dedicated MCP server that compresses large Markdown while preserving meaning and retrieves only the necessary parts for injection.

## Scope
- AST parsing of Markdown files, or heading-aware parsing
- section-level chunk generation
- index construction equivalent to `.mdq/index.sqlite`
- `search_docs(query)`
- `get_chunk(chunk_id)`
- `outline(path)`
- path / tag / heading filters
- BM25 / grep / heading traverse
- incremental indexing
- MCP server implementation
- Agent-side policy update so that `search_docs` is preferred over `read_file`

## Assumptions
- Current ingestion pipeline includes `web_crawler.py`, `chunk_splitter.py`, `rag_ingester.py`
- `chunk_splitter.py` already supports Markdown snippet splitting by heading boundary via `md_index_enable`
- The current Agent assumes HTTP MCP servers using `/v1/call_tool` and `/v1/tools`
- The current `file-mcp` provides `read_text_file`, `search_files`, `grep_files` but not Markdown heading structure or heading-path based partial retrieval
- Introduce a new MCP server: `mdq-mcp` that is separate from `file-mcp` and provides Markdown-specific index and partial retrieval APIs

## Unknowns
- What specific Markdown files need to be indexed and how to determine the indexing scope
- How to handle the integration of the new `mdq-mcp` server with the existing agent system
- What the exact structure of the new database schema should be for the Markdown index
- How to implement the heading-aware parsing and chunking logic

## Affected areas
- New files: `scripts/mdq_mcp_server.py`, `scripts/mdq_indexer.py`, `scripts/mdq_parser.py`, `scripts/mdq_search.py`, `scripts/mdq_models.py`, `config/mdq_mcp_server.json`, `init.d/mdq-mcp`
- Modified files: `tool_executor.py`, `agent_repl.py`, `agent_commands.py`, `deploy/deploy.sh`, `deploy/setup_services.sh`
- Reused existing code: Markdown heading-splitting design in `chunk_splitter.py`, `SQLiteHelper`, `logger`, `config_loader`, and shared MCP server base

## Design
- Use heading-aware parsing in Phase 1, leveraging existing logic from `chunk_splitter.py`
- Implement a dedicated SQLite database at `.mdq/index.sqlite` for Markdown indexing
- Create MCP server with HTTP transport that implements `search_docs`, `get_chunk`, and `outline` tools
- Follow existing MCP server patterns for HTTP interface, tool definitions, and watchdog integration
- Implement incremental indexing by path with content hash/mtime comparison

## Implementation steps
1. Create work plan file
2. Read and analyze existing codebase files
3. Implement the new MCP server (`mdq-mcp`)
4. Implement the indexer (`mdq_indexer.py`)
5. Implement the parser (`mdq_parser.py`)
6. Implement the search logic (`mdq_search.py`)
7. Implement the models (`mdq_models.py`)
8. Configure the server (`config/mdq_mcp_server.json`)
9. Create init.d script (`init.d/mdq-mcp`)
10. Update tool_executor.py to route to mdq
11. Update agent_repl.py to enforce preferred flow
12. Update agent_commands.py to include mdq-mcp in /mcp display
13. Update deployment scripts

## Validation plan
- Unit tests for heading extraction, outline correctness, incremental indexing, grep/BM25/hybrid search
- Integration tests for `index_paths -> search_docs -> get_chunk` and `outline -> get_chunk`
- Tool calling from the Agent
- `/v1/tools` definition diff check
- Watchdog/OpenRC startup verification
- Performance tests with large SKILL.md and ADR files

## Risks
- Incomplete Markdown parser coverage: Use heading-aware parse in Phase 1, defer complex syntax handling to AST parser in Phase 2
- SQLite index growth: Use incremental indexing by path, content hash/mtime comparison, snippet length limits, pruning of old summary cache entries
- Agent keeps using `read_file`: Explicitly add rule to `tool_definitions` and system prompt, optionally issue warning on large `read_text_file` calls
- Variability in Japanese search precision: Use normalized FTS fields as in current RAG system
