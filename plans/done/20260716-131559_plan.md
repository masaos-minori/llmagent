# Plan: Remove MDQ summary cache stub and `use_summary` request compatibility path

Source requirement: `requires/20260716_05_require.md`

## Goal

`get_chunk()` always returns raw (optionally truncated) chunk content with no
summary branch, `GetChunkRequest` no longer accepts `use_summary`, and no
config key or model type advertises non-functional summary generation —
because `_generate_and_cache_summary()` has always returned `None` for the
only supported `summary_model` value (`"default"`) and the indexer's
`_generate_summaries()` writes a truncated copy of the raw content into
`chunk_summaries`, not an actual summary, so no code path in the repository
today ever produces genuine summarized text.

  - Remove the summary-cache-check block inside `get_chunk()` (the
    `if req.use_summary and self.summary_cache_enabled and ...` branch and its

    chunk, truncate to `max_chars_per_chunk` if needed, return.

  - Remove `ChunkSummary` (`TypedDict`) and `GetChunkSummaryResponse`

  - Remove the `if service.summary_cache_enabled: _generate_summaries(...)`
    call in `_index_single_file()` and the `_generate_summaries()` function

- `scripts/mcp_servers/mdq/db_schema.py`:
  - Decision: **drop** the `chunk_summaries` table creation (see Design for

  `max_chars_per_chunk`, chunk-not-found error, and that `use_summary` is no

## Assumptions

   `self.summary_model == "default"` (the only value ever set, since
   `summary_model` has no other production caller that changes it), and its
   only non-stub branch is dead commented-out code
   (`# Future: replace with actual LLM call`). Verified by direct read of
   `service.py:152-173`.

   inserts `section["content"][:service.summary_threshold]` (a truncated
   verbatim copy of the raw chunk) into `chunk_summaries.summary`, labeled by
   a `summary_model` value that itself never triggers real generation.
   Verified by direct read of `indexer.py:315-341`. This confirms the
   requirement's "Problem" statement precisely — the feature is not partially
   working, it never wrote a real summary.
3. `ChunkSummary` and `GetChunkSummaryResponse` have zero importers anywhere
   in `scripts/` or `tests/` — verified via
   `rg -n "ChunkSummary|GetChunkSummaryResponse" scripts/ tests/` (only the
   two definition sites in `models.py` match).
4. No documentation beyond the `04_mcp_04_04_mdq.md` config field list
   mentions summary-cache behavior — verified via
   `rg -n -i "summary" docs/04_mcp_90_inconsistencies_and_known_issues.md
   docs/99_documentation_sync_report.md` (no matches), so unlike the hybrid
   search item (requirement 04), there is no existing "known issue" bullet

## Design

`chunk_summaries` table creation is **dropped** from `db_schema.py` (not left
in place) because:

- No code will read or write it after this change — an unused table is
  clutter, not a compatibility surface worth preserving.
- Unlike the `chunks_vec` vector table (requirement 04), which is explicitly
  documented as future "Phase 2" scaffolding, `chunk_summaries` was never
  documented as an intentional placeholder for later work — it silently
  produced misleading data (truncated content mislabeled as `summary`), which
  is precisely the "misleading compatibility surface" this entire batch of
  requirements is about removing.
- This matches Non-goal alignment: dropping the `CREATE TABLE` statement is
  not "implementing LLM summarization" or "adding background generation" (the
  requirement's stated non-goals) — it is pure subtraction.

`get_chunk()` after this change becomes:

```python
async def get_chunk(self, req: GetChunkRequest) -> str:
    """Retrieve a Markdown chunk by its ID."""
    request_limit = req.max_chars_per_chunk
    config_cap = self.max_chars_per_chunk
    max_chars = (
        min(request_limit, config_cap) if request_limit is not None else config_cap
    )
    conn = self._get_db_connection()
    try:
        row = conn.execute(
            "SELECT heading, content, content_hash FROM chunks WHERE chunk_id = ?",
            (req.chunk_id,),
        ).fetchone()
        if row is None:
            raise MdqNotFoundError(f"Chunk {req.chunk_id} not found")
        content = row["content"]
        truncated = False
        if len(content) > max_chars:
            content = content[:max_chars]
            truncated = True
        result = f"## {row['heading']}\n\n{content}"
        if truncated:
            result += (
                f"\n\n[Truncated — {len(row['content'])}/{max_chars} chars. "
                f"Use a narrower chunk_id or reduce max_chars_per_chunk.]"
            )
        return result
    finally:
        conn.close()
```

## Implementation steps

1. **Remove service-layer summary logic** — delete the 3 config fields,
   `_generate_and_cache_summary()`, and simplify `get_chunk()` to the Design
   snippet above.
2. **Remove model-layer summary types** — `use_summary` field,
   `ChunkSummary`, `GetChunkSummaryResponse` from `models.py`.
3. **Remove indexer-layer summary generation** — `_generate_summaries()` and
   its call site in `indexer.py`.
4. **Remove schema-layer summary table** — `chunk_summaries` table creation
   in `db_schema.py`.
5. **Remove config surface** — 3 keys from
   `config/mdq_mcp_server.toml` + removal note.
6. **Replace tests** — delete `tests/test_mdq_summary_cache.py`; add
   `tests/test_mdq_get_chunk_behavior.py` per Scope; resolve the
   `extra="forbid"` question from Unknowns while writing the
   removed-field test.
7. **Update documentation** — `docs/04_mcp_04_04_mdq.md` field list.
8. **Deployment/verification** — restart `mdq-mcp` in dev, call `get_chunk`
   on an existing indexed chunk and confirm normal (non-summary) output;
   confirm indexing a large file no longer touches `chunk_summaries`.

## Risks

- **Risk**: A live `mdq.sqlite` database may already contain populated
  `chunk_summaries` rows (truncated-content "summaries") from past indexing
  runs where `summary_cache_enabled=true` was set; removing the read path
  means any operator-facing tool relying on that data (none found) would
  silently stop working.
  **Mitigation**: Assumption 3 confirms no code reads `chunk_summaries`
  outside the removed paths; Implementation step 8 manually verifies
  end-to-end behavior in a dev environment before this is considered done.
- **Risk**: Overlap with `requires/20260716_01_require.md`, which originally
  listed `summary_cache_enabled`/`summary_threshold`/`summary_model` in its
  own task list.
  **Mitigation**: plan 01 explicitly defers these keys to this requirement
  (05); implementing in filename order avoids duplicate/conflicting edits to
  `service.py` and `config/mdq_mcp_server.toml`.
