# Plan: Update operations documentation for MCP server definition restart requirement

Source requirement: `requires/20260708_41_require.md` (L-3)

## Goal

Make it explicit in the operations docs that `/reload` detects but never
applies MCP server definition changes, that `/mcp` status reflects only the
running config, and that watchdog restart is not a substitute for a full
agent restart when the server *definition* changes — and fix one stale
troubleshooting row that currently suggests `/reload` as a remedy for an
unavailable MCP server.

## Design

**`docs/05_agent_10_operations-and-observability.md`** — fix the
troubleshooting row (line 476):

```markdown
| `/mcp` shows UNAVAILABLE server | Health registry marked server unavailable | Check watchdog logs for auto-restart attempts; if the server *definition* (URL, auth, transport, etc.) changed, a full agent restart is required — `/reload` does not apply MCP config changes |
```

Add an operations note after the `workflow_mode` note (after line 405),
following the same style:

```markdown
**Note:** MCP server definitions (`transport`, `url`, `startup_mode`,
`healthcheck_mode`, `call_timeout_sec`, `startup_timeout_sec`, `tool_names`,
`auth_token`, `role`, `cmd`, `env`) are restart-time snapshots. `/reload`
detects changes to `[mcp_servers.*]` and reports them as restart-required
(`[RESTART] - mcp/<server>.<field>`), but never applies them to the running
process. `/mcp` / `/mcp status` always reflects the running (pre-restart)
server config, not pending `/reload` changes. Watchdog-triggered restarts
(`watchdog_loop()`) restart a failed subprocess using its *current* startup
config — this is health-driven recovery, not config reload, and does not
apply pending MCP server definition changes either. Only a full agent
restart applies a changed MCP server definition.
```

**`docs/04_mcp_06_configuration_and_operations.md`** — add a note near the
watchdog configuration section (after line 32, `mcp_watchdog_max_restarts`):

```markdown
**Reload vs. restart:** `/reload` never modifies `[mcp_servers.*]` at
runtime — MCP server definition changes (URL, auth token, startup mode,
transport, command, environment) are always reported as restart-required
and require a full agent restart to take effect. The watchdog
(`mcp_watchdog_interval`, `mcp_watchdog_max_restarts` above) restarts a
*failed* subprocess using its existing startup config; it does not read or
apply any pending `/reload` config change. See
[Agent Operations: MCP restart requirement](05_agent_10_operations-and-observability.md)
for the full explanation.
```

## Risks

- **Risk**: The new cross-reference from
  `docs/04_mcp_06_configuration_and_operations.md` to
  `docs/05_agent_10_operations-and-observability.md` assumes the target
  file/anchor remains stable; if the operations doc's heading structure
  changes later, the link could break.
  → **Mitigation**: Link to the file, not a specific anchor, minimizing
  breakage risk; low likelihood within this batch's timeframe.
- **Risk**: The troubleshooting-row fix removes `/reload` as a suggested
  action entirely; if a future field actually becomes hot-reloadable for
  MCP servers (contradicting this whole batch's intent), the row would need
  revisiting again.
  → **Mitigation**: Acceptable — this batch's explicit intent (H-1..H-9) is
  that MCP server definitions are permanently restart-time snapshots, not a
  temporary state.
