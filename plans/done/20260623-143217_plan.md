# Plan: Expose Workflow Execution Mode and Approval-Pending State

**Requirement:** `requires/20260622_81_require.md`

---

## Goal

Make the workflow execution mode and approval-pending state visible in normal runtime views. Two gaps exist: (1) `/stats` doesn't show `approval_pending` when True, (2) when a workflow turn suspends for approval, no visible terminal message is emitted ‚Äî only a `logger.warning()` (Python log only, not user-visible).

**Note:** The startup banner, `/context`, and `/stats` already show workflow mode. The gaps are approval-pending visibility during the session.

---

## Scope

**In:**
- `scripts/agent/orchestrator.py` ‚Äî emit a visible terminal notice (`emit_*` call) when `WorkflowPendingApprovalError` is caught and `approval_pending` is set
- `scripts/agent/commands/models.py` ‚Äî add `approval_pending: bool = False` to `StatsViewModel`
- `scripts/agent/commands/cmd_config_stats.py` ‚Äî add approval_pending display to `/stats` (shown only when True)

**Out:**
- Changing REPL prompt format to include APPROVAL PENDING (requires prompt re-engineering)
- Redesigning workflow execution
- Changing when approval is required

---

## Assumptions

1. Startup banner already shows `Workflow: {status}` ‚úì
2. `/context` already shows "Approval pending: Yes -> use /approve or /reject" ‚úì
3. `/stats` shows "Workflow mode : {mode}" via `StatsViewModel.workflow_mode` ‚úì
4. The gap is in approval-pending: no terminal message when suspension occurs, no `/stats` indicator.
5. `orchestrator.py` already calls `logger.warning("[workflow] Approval required. Use /approve [reason] or /reject [reason].")` ‚Äî but `logger.warning` is invisible to the operator's terminal.
6. `emit_*` functions from `tool_output.py` write to `_DEFAULT_OUT` (the terminal). Adding an `emit_approval_notice()` call in the orchestrator would make the message visible.
7. `ctx.workflow.approval_pending` is set in `orchestrator.py:200` and checked in `cmd_context.py:95`.
8. `StatsViewModel.approval_pending` doesn't exist ‚Äî `cmd_config_stats.py:57` reads `workflow_mode` but not `approval_pending`.

## Design

### Change 1: `tool_output.py` ‚Äî `emit_approval_pending_notice()`

Add a new emit function:
```python
def emit_approval_pending_notice(
    approval_id: str,
    task_id: str,
    output: OutputPort | None = None,
) -> None:
    """Write a visible terminal notice when a workflow turn is suspended for approval."""
    out = output if output is not None else _DEFAULT_OUT
    out.write(
        f"\n[APPROVAL PENDING] Workflow task '{task_id}' is waiting for approval."
        f" Use /approve [reason] or /reject [reason]. (id: {approval_id})"
    )
```

### Change 2: `orchestrator.py` ‚Äî call `emit_approval_pending_notice()`

In the `except WorkflowPendingApprovalError as exc:` block (line ~186), after setting `ctx.workflow.approval_pending = True`:
```python
from agent.tool_output import emit_approval_pending_notice  # noqa: PLC0415
emit_approval_pending_notice(
    approval_id=exc.approval_id,
    task_id=exc.task_id or "unknown",
)
```

### Change 3: `models.py` ‚Äî `StatsViewModel` field

Add:
```python
approval_pending: bool = False
```

### Change 4: `cmd_config_stats.py` ‚Äî `/stats` display

In `_collect_stats()`, add:
```python
approval_pending=ctx.workflow.approval_pending if ctx.workflow is not None else False,
```

In `_cmd_stats()`, add after workflow_mode:
```python

    self._out.write("  Approval       : PENDING ‚Äî use /approve or /reject")
```

### Change 5: `cmd_workflow.py` ‚Äî reset approval_pending on resolution

In `_cmd_approve()` (line ~43):
```python

self._ctx.workflow.approval_pending = False  # add this
```

In `_cmd_reject()` (same pattern):
```python

self._ctx.workflow.approval_pending = False  # add this
```
