## Goal
`tests/test_mdq_routing.py` を拡張し、MDQ ツール名が tool_constants、tool_registry、`/v1/tools` HTTP レスポンス、LLM 向け tool definitions、安全ティアにわたって一貫していることを保証するテストを網羅する。

## Scope
**In-Scope**:
- `MDQ_TOOLS` のカウントを 7 (production) / 9 (全体) の区別を含むテストに整理
- `/v1/tools` HTTP エンドポイント経由でのツール名一貫性テストを追加
- 安全ティア分類のテスト（read-only vs write/admin のツール別確認）
- `_MCP_TOOLS` status フィールドによる production ツール絞り込み確認

**Out-of-Scope**:
- `MDQ_TOOLS` の定義変更（9 → 7 への縮小は行わない）
- `tools.py` の tool 定義変更
- `agent.toml` の変更

## Assumptions
1. `MDQ_TOOLS` は 9 ツール（production 7 + admin 2: `fts_consistency_check`, `fts_rebuild`）を保持する
2. 要件の「7 expected tools」は production status のツールを指す
3. `/v1/tools` は `test_mdq_routing.py` で TestClient または requests を使ってテスト可能
4. `_MCP_TOOLS` の各エントリが `status` フィールドを持つことは確認済み（tools.py:L20-L173）

## Unknowns
| Unknown | Evidence missing | Resolution | Blocking |
|---|---|---|---|
| `/v1/tools` テストで TestClient vs 実サーバーどちらを使うか | 他の routing テストで HTTP テストの実績なし | FastAPI の `TestClient` を使用（pytest fixture） | False |
| `agent.toml` の `tool_safety_tiers` で mdq 配下の per-tool 設定があるか | 現テストは mdq 全体の tier しか確認していない | agent.toml を参照して判断 | False |

**分析結果**:
- `test_mdq_routing.py` は 9 ツールとしてテストしているが、要件は 7 (production) を強調
- `/v1/tools` の HTTP テストは存在しない
- per-tool 安全ティアは `_WRITE_TOOLS` frozenset として tools.py で定義済み (`index_paths`, `refresh_index`)
