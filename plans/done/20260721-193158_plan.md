## Goal
Fix critical wiring gap where `enabled_for_llm` is never passed from MCP tool discovery into `build_runtime_tool()`, causing ALL discovered tools to have `enabled_for_llm=False` and be filtered out of the LLM-facing tool list.

## Scope
**In-Scope**:
- Add `enabled_for_llm=bool(entry.get("enabled", True))` to `build_runtime_tool()` call in `scripts/agent/services/mcp_tool_discovery.py:_dedupe_and_build()`
- Update documentation in `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` to reflect current implementation state
- Add test coverage for `enabled_for_llm` wiring path

**Out-of-Scope**:
- Changes to MCP server implementations (they already return `enabled`/`disabled_reason`)
- Adding new query params (`include_disabled`) or structured enums (`disabled_code`) â€” these were evaluated but not selected
- Changes to any other MCP server configurations beyond the discovery wiring fix

## Assumptions
1. `enabled` field defaults to `True` if absent from `/v1/tools` response (backward compatibility with servers that don't send it yet)
2. The `disabled_reason` field is preserved for diagnostics but does not affect `enabled_for_llm` computation
3. No existing tests directly verify `enabled_for_llm` values on discovered tools
