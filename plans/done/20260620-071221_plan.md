# Plan: HTTP RAG Mode Result Classification and Observability (req_37)

## 1. Goal

Distinguish between "remote success with context", "remote success with empty context",
and "remote failure with in-process fallback" in HTTP RAG mode — visible in `--debug`
output, `get_diagnostics()`, and stage results.

## 2. Scope

**In-Scope:**
- Add `"http_remote_empty"` as `fallback_reason` on `StageResult` for `HttpAugment`
  when `call_rag_service()` returns `""` (valid empty result, not a failure)
- Add `http_result_kind: str | None` to `get_diagnostics()` output:
  `"remote_nonempty"`, `"remote_empty"`, `"in_process_fallback"`, or `None` (non-HTTP mode)
- Document the 3-way classification in `docs/03_rag_03_query_pipeline.md`

**Out-of-Scope:**
- Changing the `""` vs `None` return semantics in `call_rag_service()`
- Redesigning the HTTP protocol
- Changing fallback behavior (in-process fallback remains on `None`)

## 3. Confirmed Facts

**Current behavior (confirmed):**
- `call_rag_service()` returns: `str` (non-empty) | `""` (valid empty) | `None` (error)
- `augment()`: `http_status = "success" if result is not None else "fallback"`
  - `result == ""` → `status="success"`, `fallback_reason=None` (empty not visible)
  - `result != ""` → `status="success"`, `fallback_reason=None` (same as above)
  - `result is None` → `status="fallback"`, `fallback_reason="http_client_error: ..."` etc.
- `_print_rag_debug()` shows `~ Refiner: fallback — reason` for fallback entries
  but empty result is `✓ HttpAugment: success` with NO distinguishing marker

**StageResult type** (from `stage.py:15-22`):
```python
class StageResult(TypedDict):
    stage_name: str
    status: str  # "success" | "fallback" | "failure"
    elapsed_seconds: float
    fallback_reason: str | None
```
- `fallback_reason` is `str | None` and already used for failure path
- Using `fallback_reason="http_remote_empty"` on `status="success"` is technically valid
  (TypedDict does not enforce semantic pairing)

**`get_diagnostics()` return value (from `pipeline.py:499-526`):**
- Contains `stage_results`, `timings`, `fetch_result`, `fallback_count`, `fallback_reasons`, `hit_counts`
- Does NOT contain a dedicated HTTP result classification field

## 4. Assumptions

1. Overloading `fallback_reason` for the empty case (`status="success"`) is pragmatic
   and avoids adding a new TypedDict field. The meaning is "this was an observed non-failure
   anomaly, not a fallback". `_print_rag_debug()` will display it as suffix.
2. `http_result_kind` in `get_diagnostics()` is the cleaner, structured path for
   programmatic inspection; `fallback_reason` is for the `--debug` display.
3. `http_result_kind` is `None` when `rag_service_url` is empty (in-process mode only).

## 7. Implementation Steps

1. **Update `augment()` in `pipeline.py`**
   ```python
   # After: result = await self._augment_http(...)
   http_empty = result is not None and not result  # valid but empty remote result
   http_status = "success" if result is not None else "fallback"
   http_fallback_reason = (
       "http_remote_empty"
       if http_empty
       else (http_fallback_reasons[0] if http_fallback_reasons else "in-process fallback")
       if result is None
       else None
   )
   self.last_stage_results.append(
       StageResult(
           stage_name="HttpAugment",
           status=http_status,
           elapsed_seconds=elapsed,
           fallback_reason=http_fallback_reason,
       )
   )
   ```
   Note: `fallback_reason="http_remote_empty"` on `status="success"` — intentional labeling

2. **Update `get_diagnostics()` in `pipeline.py`**
   ```python
   def _http_result_kind(self) -> str | None:
       """Classify the HTTP RAG result for the last augment() call."""
       http_stage = next(
           (r for r in self.last_stage_results if r["stage_name"] == "HttpAugment"),
           None,
       )
       if http_stage is None:
           return None
       if http_stage["status"] == "fallback":
           return "in_process_fallback"
       if http_stage.get("fallback_reason") == "http_remote_empty":
           return "remote_empty"
       return "remote_nonempty"
   ```
   Add to `get_diagnostics()` return dict:
   ```python
   "http_result_kind": self._http_result_kind(),
   ```

3. **Update `docs/03_rag_03_query_pipeline.md`**
   - Under HTTP mode section, add:
     ```
     HTTP mode result classification (visible in --debug, get_diagnostics()):
     | http_result_kind     | Condition |
     |----------------------|-----------|
     | remote_nonempty      | HTTP 200 with non-empty context field |
     | remote_empty         | HTTP 200 with null/absent/empty context — valid empty result |
     | in_process_fallback  | HTTP error; in-process pipeline ran |
     | None                 | rag_service_url not set; HTTP mode not used |
     ```

4. **Tests**
   - `test_pipeline_http_result_kind.py`:
     - Mock `call_rag_service` returning non-empty str → `http_result_kind="remote_nonempty"`
     - Mock returning `""` → `http_result_kind="remote_empty"`, `fallback_reason="http_remote_empty"`
     - Mock returning `None` → `http_result_kind="in_process_fallback"`
     - No `rag_service_url` → `http_result_kind=None`

## 9. Risks

- **Risk:** `fallback_reason` on `status="success"` is semantically inconsistent with
  the field's original intent (failure reason)
  → **Mitigation:** rename meaning in docs to "classification reason"; keep implementation minimal.
  Alternative: add `result_kind: str | None` to `StageResult` TypedDict — cleaner but wider change.
  If this becomes a pattern, promote to a proper `result_kind` field in a follow-up.
- **Risk:** Tests that assert `StageResult` for `HttpAugment` with no `fallback_reason`
  on empty result will now fail
  → **Mitigation:** search for such assertions before implementation; update them
