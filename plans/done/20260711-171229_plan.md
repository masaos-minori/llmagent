2. `docs/04_mcp_03_03_transport-and-health.md:28-32` (confirmed by direct read, previously read in full during this session's requirement-11 research) already states: retryable statuses `429/502/503/504`; retry max 3 with the exact `4, 2, 1` second delay sequence and formula `2^(RETRY_MAX - attempt - 1)`; explicitly notes this is not exponential backoff; explicitly notes timeout (`httpx.TimeoutException`) and non-retryable status codes propagate immediately without retry. This is a complete, accurate description of `scripts/shared/http_transport.py`'s actual implementation (confirmed by direct read of `_call()`'s retry loop, lines 88-144) — including the fact that the final (3rd) retryable-status attempt still sleeps before the loop's `else:` clause raises `TransportError`. Resolving the requirement's Task ("decide whether final retry attempt should sleep before retry exhaustion"): **preserve current behavior (it does sleep)** — the Notes' explicit caution against changing test-locked timing, combined with this being a minor (1-second) inefficiency on an already-failing call path rather than a correctness bug, outweighs the low value of eliminating one wasted second on an error path that is about to fail regardless.
3. `scripts/shared/tool_executor.py::_record_success()` (lines 169-176) and `_record_transport_error()` (lines 178-186), confirmed by direct read: tool-level errors (`result.is_error and result.error_type == "tool"`) still call `self._health_registry.record_success(server_key)` (the transport succeeded; only the tool itself reported an error) and increment `self.stat_tool_errors[server_key]`; transport-level errors call `self._health_registry.record_failure(server_key)` and increment `self.stat_transport_errors[server_key]`. Both counters already exist and are already correctly separated in code — this matches the requirement's Tasks 40-41 exactly. No code change needed; the one gap is that the target documentation file doesn't yet state this split as explicitly as the code implements it.
4. `repeated_tool_error_threshold` (`tool_executor.py:66,77`, confirmed by direct read) is accepted as a constructor parameter and stored as `self._tool_error_threshold`, but `grep -n "repeated_tool_error_threshold\|_tool_error_threshold"` across the entire file confirms it is **never read again anywhere** — genuinely inert. `grep -rn "repeated_tool_error_threshold" scripts/ tests/` (excluding its own definition) confirms **zero callers** pass a non-default value anywhere in the codebase. Resolving the requirement's Task ("decide whether... should be implemented or removed"): **remove it** — it is 100% dead, no caller depends on a non-default value (removal is behavior-preserving for every existing caller), and "implement" would require inventing new, unspecified threshold-triggered behavior with no design guidance in the requirement itself, unlike the clear, low-risk removal path already established as this session's pattern for confirmed-dead settings.

Replace:

class LifecycleProtocol(Protocol):
    """Protocol for MCP server lifecycle managers injected into ToolExecutor."""

    async def ensure_ready(self, server_key: str) -> None:
        """Ensure the MCP server identified by server_key is ready to accept calls."""
        ...

from shared.tool_lifecycle import LifecycleProtocol  # noqa: F401 — re-exported for backward compatibility
