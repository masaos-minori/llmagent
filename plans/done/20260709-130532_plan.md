# Plan: Detect unknown tool_safety_tiers keys

Source requirement: `requires/20260709_04_require.md`

## Goal

Detect `tool_safety_tiers` keys that are not real registered tool names
(the reverse of the existing missing-tier check), warn in local/development,
and fail startup (`RuntimeError`, via `ProductionConfigValidator`) in
production. Clean up the one known stale key (`mdq`) in
`config/agent.toml`.

  `check_tool_safety_tiers()`'s shape but computing the reverse set
  difference.
- `shared/production_config_validator.py`: add a second method,

- `scripts/agent/repl_health.py`: call the new helper inside

- `config/agent.toml`: replace the stale `mdq = "WRITE_DANGEROUS"` line

- `tests/test_production_config_validator.py`: add tests for the new

Out of scope:

- Changing risk tier enum values, or default behavior for tools *absent*

- `startup.py`'s `check_routing_safety_tiers` call site (step 5b, always

     index entries, matching `write_file`/`edit_file`'s tier) → `WRITE_SAFE`.
   - `fts_rebuild` — description in `tools.py:169` is `"Rebuild the FTS5
     index. Admin operation — may take time on large datasets."` — a heavy,
     disruptive rebuild (matches `delete_file`/`delete_directory`'s tier,
     both irreversible/high-impact operations) → `WRITE_DANGEROUS`.
   `docs/04_mcp_04_server_catalog.md:298` corroborates the production/admin

## Design

**`shared/tool_routing_validation.py`** — add after `check_tool_safety_tiers()`:

```python
def check_unknown_tool_safety_tiers(
    registry: "ToolRegistry | None" = None,
    tool_safety_tiers: dict[str, str] | None = None,
) -> list[str]:
    """Return the tool_safety_tiers keys that are not registered tool names.

    Detects stale or misconfigured entries (e.g. a server key like "mdq"
    used instead of its individual tool names). Returns empty list when
    tool_safety_tiers is not configured.
    """
    if not tool_safety_tiers:
        return []
    if registry is None:
        from shared.tool_registry import get_registry

        registry = get_registry()
    known = registry.get_all_tool_names()
    return sorted(set(tool_safety_tiers) - known)
```

**`shared/production_config_validator.py`** — add a method:

```python
    def validate_unknown_tool_safety_tiers(
        self, unknown_keys: list[str]
    ) -> ConfigValidationResult:
        errors = [
            f"tool_safety_tiers contains unknown key {k!r} (not a registered tool name)"
            for k in unknown_keys
        ]
        return ConfigValidationResult(errors=errors, warnings=[])
```

**`repl_health.py`** — inside `audit_security_defaults()`, add (near the
new production-flags block from requirement 03's plan, or anywhere before
the function's final summary):

```python
    from shared.tool_routing_validation import check_unknown_tool_safety_tiers

    unknown_tiers = check_unknown_tool_safety_tiers(
        tool_safety_tiers=getattr(ctx.cfg.approval, "tool_safety_tiers", {})
    )
    if unknown_tiers:
        unknown_result = ProductionConfigValidator().validate_unknown_tool_safety_tiers(
            unknown_tiers
        )
        if production_mode:
            raise RuntimeError(
                "Production config validation failed: "
                + "; ".join(unknown_result.errors)
            )
        for err in unknown_result.errors:
            logger.warning("Security: %s", err)
            warnings.append(f"Security: {err}")
```

## Risks

- **Risk**: `fts_consistency_check`'s proposed `READ_ONLY` tier (UNK-01) may
  not match the actual intent behind its `"admin"` status label.
  → **Mitigation**: Explicitly flagged as Needs confirmation in both the
  Unknowns table and the `config/agent.toml` inline comment; the detection
  mechanism itself does not depend on which tier value is ultimately
  chosen.
- **Risk**: This plan depends on `shared/production_config_validator.py`
  from requirement 03's plan, which may not be implemented yet. If someone
  implements this plan first without creating that file, step 2 must create
  it (with at least the new method), and requirement 03's implementation
  must later add its own method to the same class without overwriting this
  one.
  → **Mitigation**: Called out explicitly in Assumption 3 and Implementation
  step 2.
- **Risk**: Replacing `mdq` with 9 individual entries increases
  `config/agent.toml`'s `[tool_safety_tiers]` section length; if any other
  code or doc iterates `tool_safety_tiers` and assumed one entry per
  *server* (not per tool), it could behave unexpectedly.
  → **Mitigation**: `grep -rn "tool_safety_tiers" scripts/` at
  implementation time to confirm all consumers (`mcp_status.py`,
  `check_tool_safety_tiers()`, approval risk logic) already operate
  per-tool-name, not per-server — consistent with every other entry in the
  existing `[tool_safety_tiers]` table already being an individual tool
  name (e.g. `read_text_file`, `write_file`), not a server key.
