# Work Plan: Make ToolExecutor cache safe for side-effecting tools and preserve cache attribution

Source requirement: `requires/20260715_20_require.md`

## Goal

`ToolExecutor.execute()` never returns a cached result for a side-effecting MCP tool call (write/delete/shell/git-write/github-write-or-dangerous/rag-delete/trigger-workflow/fts-rebuild); read-only MCP tools keep TTL caching, LRU eviction, and stampede protection; and cache-hit results retain the `server_key` of the server that originally produced them.

## Scope

**In scope**
- `scripts/shared/tool_executor.py` — route side-effecting tools to `_raw_execute()` directly in `execute()`.
- `scripts/shared/tool_cache.py` — add `server_key` field to `CacheEntry`; thread it through `ToolResultCache` for consistency (same dataclass, same file listed as a target).
- `scripts/shared/transport_dto.py` — no field changes required; `ToolCallResult` already has `server_key` and `source`. Confirm cache-hit paths populate them correctly.
- `scripts/shared/tool_executor_helpers.py` — no change to `is_side_effect()` signature; extend the underlying classification data it reads from `tool_constants.py` (see Design).
- `scripts/shared/tool_constants.py` — add three new minimal frozensets so `is_side_effect()` covers the three additional unsafe tools named in the requirement: `rag_delete_document`, `trigger_workflow`, `fts_rebuild`.
- `tests/test_tool_executor.py` — change existing cache tests off `write_file` onto `read_text_file`; add new tests proving side-effect bypass and cache-hit attribution.
- `tests/test_tool_constants.py` — add classification tests for the three new frozensets, following the existing `TestGitToolClassification` / `TestGithubToolClassification` pattern.

**Out of scope (explicit non-goals, per requirement)**
- Removing caching entirely.
- Changing plugin execution semantics (`PluginToolInvoker` bypass stays as-is).
- Changing transport error classification, except the minimal `source` addition needed for cache attribution.
- Reconciling the wider, pre-existing inconsistency across the codebase between `shared/tool_constants.py`, `agent/tool_policy.py._API_WRITE_TOOLS`, `mcp_servers/mdq/tools.py._WRITE_TOOLS`, and `agent.toml [tool_safety_tiers]` (each uses a different, non-identical notion of "write"). This plan only closes the specific gap needed to satisfy the requirement's named examples.
- Extending cache-bypass to `rag_run_pipeline`, `rag_debug_pipeline`, `index_paths`, `refresh_index` — these also mutate state but are not named in the requirement's "Caching is unsafe for" list or its Suggested Tests. Flagged as a Risk below with a follow-up-issue mitigation.

## Assumptions

1. `is_side_effect()` (in `scripts/shared/tool_executor_helpers.py`) is the correct, already-designated shared classification function to call from `ToolExecutor.execute()` — the requirement's Implementation Notes explicitly say side-effect detection must come from shared classification logic, not a local list in `ToolExecutor`. Verified: it exists and is already used by `agent/tool_runner.py` for serial-execution downgrade.
2. Reusing `is_side_effect()` in `execute()` means `rag_delete_document`, `trigger_workflow`, and `fts_rebuild` must be added to its underlying `_SIDE_EFFECT_TOOLS` set for the requirement's acceptance criteria (and its named Suggested Tests: `test_trigger_workflow_bypasses_cache`, `test_fts_rebuild_bypasses_cache`) to hold. Verified by reading `_SIDE_EFFECT_TOOLS` — it currently only unions `WRITE_TOOLS | DELETE_TOOLS | {shell_run} | GIT_WRITE_TOOLS | GITHUB_WRITE_TOOLS | GITHUB_DANGEROUS_TOOLS`; none of `RAG_TOOLS`, `CICD_TOOLS`, `MDQ_TOOLS` is included today.
3. `RAG_TOOLS`, `CICD_TOOLS`, `MDQ_TOOLS` in `tool_constants.py` are flat (read+write mixed) sets whose exact membership and total count (44) are pinned by existing tests (`test_tool_constants.py::test_no_overlapping_tools`, `test_mdq_routing.py`, `test_rag_tools_consistency.py`, `test_tool_registry_counts.py`). Adding new, disjoint, purely-additive frozensets (`RAG_WRITE_TOOLS`, `CICD_WRITE_TOOLS`, `MDQ_WRITE_TOOLS`) that are subsets of the existing flat sets — without redefining `RAG_TOOLS`/`CICD_TOOLS`/`MDQ_TOOLS` themselves — will not break those pinned tests, mirroring how `GIT_READ_TOOLS`/`GIT_WRITE_TOOLS` coexist with `GIT_TOOLS`.
4. Because `agent/tool_runner.py._execute_standard()` already calls `is_side_effect()` to decide serial-vs-parallel execution, widening `_SIDE_EFFECT_TOOLS` will also cause `rag_delete_document`, `trigger_workflow`, and `fts_rebuild` calls to force serial execution in that code path. This is judged to be a desirable, not merely incidental, side effect (these ARE mutations), and is within the spirit of the requirement, but it is a behavior change beyond `ToolExecutor` itself — called out explicitly since it was not asked for directly.
5. Introducing `source="cache"` on cache-hit `ToolCallResult`s (the requirement's "Optional improvement") is adopted, because it directly serves the requirement's stated observability goal and the blast radius is small (one existing assertion in `tests/test_tool_executor.py:762` expects `source == ""` on a cache hit and must be updated; `agent/tool_audit.py::audit_tool_exec` already treats non-empty `source` as pass-through, so audit logs will show `source="cache"` instead of falling back to `"agent"` — this is strictly more informative, not a break).
6. `CacheEntry` is a shared dataclass used by both `ToolExecutor`'s internal cache dict and the currently-unused `ToolResultCache` utility class (both live in `scripts/shared/tool_cache.py`, both target files per the requirement). Adding `server_key: str = ""` to `CacheEntry` and threading it through `ToolResultCache.store_if_success` / `get_result` too (not just `ToolExecutor`) keeps the file internally consistent, at negligible cost since `ToolResultCache` has no other production caller today (confirmed via `rg` — the only importers are `tool_cache.py`, `tool_executor.py`, and `tests/shared/test_tool_result_cache.py`).
7. `cache_ttl=300`, `cache_max_size=200` (from `config/agent.toml` / `AgentConfig` defaults) are unaffected by this change; no config surface changes are required.
8. No DB schema change, no new module, no new MCP server — `deploy/deploy.sh` requires no update (all changed files already exist and are already copied).

## Design

### 1. `tool_constants.py` additions (additive only)

```python
RAG_WRITE_TOOLS: frozenset[str] = frozenset({"rag_delete_document"})
CICD_WRITE_TOOLS: frozenset[str] = frozenset({"trigger_workflow"})
MDQ_WRITE_TOOLS: frozenset[str] = frozenset({"fts_rebuild"})
```

Do not redefine `RAG_TOOLS`, `CICD_TOOLS`, `MDQ_TOOLS` in terms of these (keeps `test_no_overlapping_tools`'s count of 44 and the exact-membership tests in `test_mdq_routing.py` / `test_rag_tools_consistency.py` untouched — those tests do not iterate `RAG_WRITE_TOOLS` etc., only the flat top-level sets).

### 2. `tool_executor_helpers.py` — extend the union only

```python
_SIDE_EFFECT_TOOLS: frozenset[str] = (
    WRITE_TOOLS
    | DELETE_TOOLS
    | frozenset({"shell_run"})
    | GIT_WRITE_TOOLS
    | GITHUB_WRITE_TOOLS
    | GITHUB_DANGEROUS_TOOLS
    | RAG_WRITE_TOOLS
    | CICD_WRITE_TOOLS
    | MDQ_WRITE_TOOLS
)
```
`is_side_effect()` itself is unchanged (no signature or docstring behavior change beyond the wider set it checks against).

### 3. `tool_cache.py` — attribution field

```python
@dataclass(frozen=True)
class CacheEntry:
    output: str
    is_error: bool
    cached_at: float
    server_key: str = ""
```

`ToolResultCache.store_if_success` stores `server_key=result.server_key`; `get_result` returns `ToolCallResult(..., server_key=cached.server_key, source="cache" if not cached.is_error else "")`. Error entries are never stored today (`store_if_success` early-returns on `result.is_error`), so `source="cache"` only ever applies to successful cache hits — consistent with the requirement's framing ("only successful results are cached").

### 4. `tool_executor.py` — bypass routing + attribution

```python
async def execute(self, tool_name, args) -> ToolCallResult:
    plugin_result = await self._plugin_invoker.try_execute(tool_name, args)
    if plugin_result is not None:
        return plugin_result
    if is_side_effect(tool_name):
        return await self._raw_execute(tool_name, args)
    return await self._execute_with_cache(tool_name, args)
```

`_store_and_evict` stores `server_key=result.server_key` on the `CacheEntry`. `_execute_with_cache`'s cache-hit branch returns `ToolCallResult(output=cached.output, is_error=cached.is_error, request_id="", server_key=cached.server_key, source="cache" if not cached.is_error else "", error_type=...)`.

Because `_execute_with_cache()` and `_execute_with_stampede_protection()` are no longer reachable from `execute()` for side-effecting tools, they are called directly only by tests going forward (and, incidentally, by any other future caller) — no signature changes needed there, so `agent/tool_runner.py`'s existing call path (`ctx.services_required.tools.execute(name, args)`) is unaffected apart from side-effecting tools now genuinely re-executing instead of hitting cache.

### 5. Interaction with `agent/tool_runner.py` (no code change, behavior change)

`_execute_standard()` already calls `is_side_effect()` to decide serial vs. parallel scheduling. Widening `_SIDE_EFFECT_TOOLS` means `rag_delete_document`, `trigger_workflow`, `fts_rebuild` calls will now also be downgraded to serial execution when mixed with other tool calls in one turn. This is judged safe/desirable (matches their already-mutating nature) and requires no code change in `tool_runner.py`, but the existing test suite for that module (`tests/test_tool_runner.py`) must be re-run to confirm no test hardcodes the old set of side-effect tools in a way that would now fail.

## Risks

- **Risk**: Widening `_SIDE_EFFECT_TOOLS` also changes `agent/tool_runner.py._execute_standard()`'s serial-vs-parallel scheduling for `rag_delete_document`/`trigger_workflow`/`fts_rebuild`, a behavior change not explicitly requested. Likelihood: high (certain, given shared classification reuse). Impact: low-medium (forces safer serial execution for genuinely mutating calls; could change turn latency when these tools are mixed with others). → **Mitigation**: call this out explicitly in the PR description; run `tests/test_tool_runner.py` and `tests/test_agent_repl_tool_exec.py` as part of Phase 5 to catch any test that assumed the old (narrower) side-effect set.
- **Risk**: `rag_run_pipeline`, `rag_debug_pipeline`, `index_paths`, `refresh_index` remain cacheable despite being mutating operations, because they are not named in the requirement. Likelihood: medium (a cached ingestion/index-trigger response could mislead a caller into believing an index refresh ran when it did not, mirroring exactly the bug class this requirement fixes for other tools). Impact: medium. → **Mitigation**: do not silently expand scope in this plan (Non-Goals forbid speculative steps); instead file a follow-up issue after this plan is approved, proposing `RAG_WRITE_TOOLS`/`MDQ_WRITE_TOOLS` be widened to include these tools with product-owner sign-off, since it changes caching behavior for tools not currently flagged unsafe by the requirement author.
- **Risk**: `tool_executor.py` is high-churn (55 all-time commits, 37 in the last 30 days) and imported by `agent/factory.py`, `agent/context.py`, `agent/repository_gateway.py` — a merge conflict or an in-flight unrelated change could land concurrently. Likelihood: medium. Impact: low (conflict, not correctness). → **Mitigation**: keep the `execute()` diff minimal (3-line branch, Design §4) and land Phase 1-2 (constants/cache attribution) as an independently revertable commit before Phase 3 (bypass routing), per the skill's "independently committable steps" rule.
- **Risk**: Introducing `source="cache"` could be read by some future/undiscovered consumer as meaning "not a real MCP call" and treated specially in a way that suppresses legitimate audit trail. Likelihood: low (searched all `.source` consumers; only `agent/tool_audit.py` and test assertions found). Impact: low. → **Mitigation**: `audit_tool_exec()`'s existing fallback (`source if source else "agent"`) already handles any non-empty value gracefully; no code change needed there, but note this explicitly in the PR description so a reviewer double-checks no downstream dashboard/alert filters on `source == ""` specifically for "was this a live call".
- **Risk**: Diff-cover ≥ 90% on changed lines may be hard to hit if the new `RAG_WRITE_TOOLS`/`CICD_WRITE_TOOLS`/`MDQ_WRITE_TOOLS` frozensets or the `ToolResultCache` propagation are not directly exercised by a test. Likelihood: low. Impact: low (would fail Phase 6 gate, not a silent regression). → **Mitigation**: Phase 1/2's added tests (classification tests, `ToolResultCache` round-trip test) directly cover every new line before Phase 6 runs `diff-cover`.
