# Plan: Consolidate tool-level and workflow-level approval into one canonical model

Source: `requires/20260625_01_require.md`

---

## Goal

tool-level approval (tool_approval.py) と workflow-level approval (workflow_engine.py) の境界を明確に定義し、両者が同一操作に対して重複起動しないよう制御を追加したうえで、ドキュメントを一元化する。

---

## Scope

**In-Scope**
- 全 approval エントリポイントの監査（完了 → 下記 Affected Areas に記載）
- canonical approval モデルの定義と設計記録（ADR）
- `orchestrator.py` での `require_approval` 引数の扱いを明示化
- tool approval が workflow コンテキスト内で有効な場合の抑制フラグ追加（または設計上の排他保証）
- `cmd_workflow.py` の `/approve` / `/reject` が workflow-level approval のみを解決することを明示化
- approval 関連ドキュメントの一本化

**Out-of-Scope**
- `tool_executor.py` への approval ロジック追加（現行: approval 責務なし。変更しない）
- DB スキーマ変更（`workflow_approvals` テーブルはそのまま利用）
- approval UI の変更（CLI プロンプト形式は維持）
- workflow approval の `require_approval` をデフォルト有効化（設計決定待ち）

---

## Assumptions

1. `WorkflowEngine(require_approval=False)` は現行コードで固定されており、workflow-level approval は本番では起動していない。
2. tool-level approval (`tool_approval.py`) が per-tool の書き込みリスク判定の主担当である。
3. workflow-level approval はターン全体の human-in-the-loop ゲートとして設計されており、per-tool approval とは粒度が異なる。
4. `tool_executor.py` は approval 責務を持たない（ルーティング + 実行のみ）。これは変更しない。
5. Python 3.13、SQLite、uv 環境での実装。

## Design

### Approval モデルの分類

| 軸 | Tool-level Approval | Workflow-level Approval |
|---|---|---|
| 実装 | `agent/tool_approval.py` | `agent/workflow/workflow_engine.py` |
| 粒度 | tool call 単位 | ターン（execute → verify 間）|
| 状態 | ephemeral（メモリ内） | DB 永続（workflow_approvals テーブル）|
| 解決手段 | stdin インタラクティブ入力 | `/approve` / `/reject` コマンド |
| 現行有効 | 常時有効 | 無効（`require_approval=False` 固定）|

### Canonical モデル設計（UNK-01〜03 解決済み）

両モデルを共存させる。「canonical」の意味は「排他」ではなく「境界と責任の明文化」とする。

| レイヤー | 責任 | 粒度 | 状態 | 解決手段 |
|---|---|---|---|---|
| tool-level approval (`tool_approval.py`) | per-tool の書き込みリスク判定と interactive 確認 | tool call 単位 | ephemeral | stdin インタラクティブ |
| workflow-level approval (`workflow_engine.py`) | execute → verify 間の human-in-the-loop ゲート | ターン単位 | DB 永続 | `/approve` / `/reject` |

**共存時の動作（require_approval=True の場合）**
1. execute ステージ内: tool approval が per-tool に起動（MEDIUM/HIGH リスクツールのみ）
2. execute ステージ完了後: workflow approval ゲートで `/approve` または `/reject` 待機
3. 二重の確認になるが、粒度が異なるため矛盾しない

**主要変更方針**
- `orchestrator.py` の `require_approval=False` 固定を config 値から取得するよう変更（将来の有効化に備える）
- `/approve` / `/reject` は workflow-level approval のみ解決することをコード・ドキュメントで明示
- `tool_approval.py` は変更しない（workflow コンテキスト依存を持ち込まない）
- docs に境界図を追加
