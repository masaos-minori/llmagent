# Plan: Simplify web-search-mcp to DuckDuckGo Only

**Source requirement:** `requires/20260623_00_require.md`
**Task type:** Refactor (provider simplification)
**Path classification:** B (Large Task — affects 5 files)

---

## Goal

Remove Brave and Bing provider code from `web-search-mcp` so that DuckDuckGo is
the only search provider, eliminating API key handling, fallback logic, and
multi-provider config complexity.

---

## Scope

**In-scope:**
- Remove `_search_brave()` and `_search_bing()` from `server.py`
- Remove `BRAVE_API_KEY` / `BING_API_KEY` env var reads and startup warnings from `server.py`
- Remove `_PROVIDER_FUNCS` dict and `_try_provider()` fallback loop from `server.py`
- Inline DuckDuckGo call directly in the `search()` endpoint
- Simplify `/health` to remove `brave_api_key` / `bing_api_key` dependency checks
- Remove `brave_search_url`, `bing_search_url`, `search_providers` fields from `WebSearchConfig`
- Remove `http_timeout` from `WebSearchConfig` and config TOML (no httpx calls remain)
- Remove `httpx` and `os` imports from `server.py`
- Remove `brave_search_url`, `bing_search_url`, `search_providers`, `http_timeout` keys from `config/web_search_mcp_server.toml`
- Update `docs/04_mcp_04_server_catalog.md` — Purpose, Provider fallback section, Config table, Health example
- Update `tests/test_web_search_models.py` — remove assertions referencing removed fields

**Out-of-scope:**
- Renaming the `search_web` MCP tool
- Replacing DuckDuckGo with another provider
- Changing `SearchResult.provider` or `SearchResponse.provider` schema fields (keep for informational value)
- Refactoring any other MCP server
- Modifying `deploy/deploy.sh` (web_search_mcp_server.toml is already handled by existing cp line)

---

## Assumptions

1. `duckduckgo_search.DDGS` is already installed and available in the venv (confirmed: present in `server.py`)
2. `http_timeout` is unused after removing Brave/Bing httpx calls; DDGS uses its own internal timeout via `proxies` — config field is safe to remove
3. `search_providers` config field exists solely to support multi-provider selection; removing it simplifies the config without breaking the `search_web` MCP tool contract
4. No external service (agent, config, startup script) reads `BRAVE_API_KEY` or `BING_API_KEY` for web-search-mcp specifically
5. The `/etc/conf.d/web-search-mcp` environment file may still define `BRAVE_API_KEY`/`BING_API_KEY` — their presence is harmless since the server will no longer read them
6. `deploy/deploy.sh` already copies `config/web_search_mcp_server.toml`; no new cp line is needed

## Design Decisions

### D1: Remove `search_providers` entirely (not just set to `["duckduckgo"]`)

Keeping `search_providers` as a list field in config/model implies future multi-provider
support. The requirement states "update config/model handling so it no longer assumes
multiple providers." Removing the field is the correct interpretation.

The `search()` endpoint will call `_search_duckduckgo()` directly, eliminating the
provider-selection loop.

### D2: Remove `http_timeout` from config/model

`http_timeout` was used exclusively for httpx calls in `_search_brave` and `_search_bing`.
After their removal, no httpx calls remain. `asyncio.to_thread` wrapping DDGS has no
configurable timeout at the `WebSearchConfig` level. Removing the field reduces config
surface to only what the code actually uses.

| Risk | Likelihood | Mitigation |
|---|---|---|
| `WebSearchConfig.load()` fails if TOML still contains removed keys | Low — `from_dict` uses `.get()` with defaults, so extra keys in TOML are ignored; the reverse (missing keys) also safe | Verify by running the server after config update |
