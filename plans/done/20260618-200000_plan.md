# Plan: Record and analyze side-effect round serialization

## Goal

Record when and why round-wide serialization happens for side-effect tools, add observability before attempting any optimization to parallelism behavior.

## Scope

**In scope:**
- Tool execution flow (`scripts/agent/tool_scheduler.py`)
- Serialization event recording (logging/metrics)
- Side-effect scheduling behavior
- Execution observability

**Out of scope:**
- Changing the serialization logic itself (safe by design, no optimization yet)
- Modifying tool definition or execution order
- Adding new parallelism strategies
- Changes to `requires_serial`, `resource_scope`, `is_write` rule evaluation

## Assumptions

1. When one side-effect tool appears in a round, the entire round becomes serialized — this is intentional for safety but may reduce parallelism more than necessary.
2. The serialization rules (`requires_serial`, `resource_scope`, `is_write`) determine grouping, but there is no logging of why specific grouping decisions are made.
3. Operators and developers need data on how often serialization happens and which tools trigger it before considering optimizations.

## Implementation Steps

1. **Record serialization events**
   - Log when round-wide serialization is triggered:
     ```
     ROUND_SERIALIZATION: triggered by {tool_name} ({reason}) — {N} tools serialized in this round
     ```
   - Reason values: `requires_serial`, `resource_scope_conflict`, `is_write_overlap`
   - Completion criterion: Every serialization event is logged with trigger tool and reason

2. **Record serialization impact metrics**
   - Track: number of tools affected, total serialization count per session
   - Example: "Serialization impact: 3 tools grouped serially (normally would run in parallel)"
   - Completion criterion: Metrics capture serialization frequency and scope

3. **Add observability endpoint for serialization stats**
   - If `/mcp` or status endpoint exists, include serialization statistics:
     ```json
     {"serialization": {"total_events": 5, "tools_affected_last_round": 3}}
     ```
   - Completion criterion: Serialization stats accessible via status endpoint

4. **Document serialization behavior**
   - Explain why serialization is necessary (safety for side-effect tools)
   - Document how to interpret serialization logs
   - Note that optimization attempts require data from this observability layer first
   - Completion criterion: Docs explain serialization rationale and log interpretation
