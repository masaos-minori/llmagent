## Goal

Fix the one genuinely stale documentation section that still describes `workflow_mode` as a live, 3-valued config option (contradicting the already-mandatory, no-disable reality), add explicit source/runtime path pairing where both matter, and cross-link the workflow validation / DB-initialization / startup-validation documentation sections — without a mechanical, context-blind find-replace of "workflow definition file" across the whole doc set.

## Scope

**In-Scope:**
- `docs/01_overview-arch-pipelines.md`: replace the stale "`workflow_mode` の3種" table (still describing `auto`/`required`/`disabled` modes) with accurate text stating workflow is unconditionally mandatory, no modes exist
- `docs/02_deployment.md`: confirm (or add, if not already present after `plans/done/20260710-153811_plan.md` lands) that both the source path (`config/workflows/default.json`) and runtime/deployed path (`/opt/llm/config/workflows/default.json`) are stated together, not just one or the other, in the workflow-responsibilities additions
- Add cross-links between: `docs/02_deployment.md` (deploy-time validation), `docs/05_agent_03_turn-processing-flow-workflow-engine.md` (loader validation rules — already documents the exact rule set), and `docs/05_agent_10_operations-and-observability-validation-and-troubleshooting.md` (startup validation + runbook, from `plans/done/20260710-153954_plan.md`)

- A mechanical, blanket replacement of every "workflow definition file" occurrence with "required workflow deployment artifact" — direct investigation (see Assumptions) found that every other occurrence of this phrase already appears in a context that is unambiguously mandatory (paired with "unconditionally requires/validates", "no config toggle to skip"), so a global terminology swap would not measurably improve clarity and risks awkward, robotic-sounding prose. This plan instead introduces "required workflow deployment artifact" as a defined term in one authoritative location (Design section) and leaves the naturally-already-unambiguous surrounding prose as-is.
- Re-documenting `workflow_mode`/`workflow_require_approval`'s removal — `docs/05_agent_08_configuration-loading-agent-config.md` (lines 77, 132-136) and `docs/05_agent_10_operations-and-observability-validation-and-troubleshooting.md` (lines 29, 36-38) already correctly and clearly document this as a forbidden/rejected config key, confirmed by direct read; no changes needed there

## Design

### `docs/01_overview-arch-pipelines.md` replacement

Replace:
```markdown
**workflow_mode の3種**

| workflow_mode | 動作 | 失敗時挙動 |
|---|---|---|
| `auto` (デフォルト) | ワークフロー定義が存在すれば有効化 | ロード失敗は警告ログで継続 |
| `required` | ワークフロー定義が必須 | ロード失敗は `RuntimeError` で起動中断 |
| `disabled` | 常にダイレクト実行 | ワークフローを完全バイパス |
```
with:
```markdown
**ワークフローは常時必須(モード設定なし)**

`workflow_mode` は設定キーとして存在しない(`build_agent_config()` の `_FORBIDDEN_KEYS` に含まれ、設定すると `ConfigLoadError` で起動不可)。ワークフロー定義 (`config/workflows/default.json` としてデプロイされる **required workflow deployment artifact**) は常に必須であり、存在しない・不正な場合は起動前に `RuntimeError` で中断する。ダイレクト実行へのフォールバックや、ワークフローを無効化する経路は一切存在しない。

詳細: [02_deployment.md §Workflow deployment checklist](02_deployment.md) / [Workflow Deployment Runbook](05_agent_10_operations-and-observability-validation-and-troubleshooting.md#workflow-deployment-runbook)
```

### `docs/02_deployment.md` path-pairing confirmation

When implementing `plans/done/20260710-153811_plan.md`'s additions, ensure the workflow-responsibilities bullets state both paths together at least once, e.g.:
```markdown
The workflow definition is a **required workflow deployment artifact**:
source `config/workflows/default.json` → deployed to `/opt/llm/config/workflows/default.json`.
```
(rather than mentioning only one path in isolation in any single bullet)

### Cross-links

See also: [02_deployment.md](02_deployment.md) for deploy-time validation of these same rules,
