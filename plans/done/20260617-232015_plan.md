# Plan: Add strict and best-effort modes for maintenance operations (req 76)

## Goal

Define clear `MaintenanceMode` enum (`STRICT`, `BEST_EFFORT`) for maintenance operations. In `STRICT` mode, exceptions propagate directly. In `BEST_EFFORT` mode, exceptions are caught, logged, and included in a structured result. Improve result reporting across all maintenance functions.

## Scope

**In scope:**
- `db/maintenance.py` — add `MaintenanceMode` StrEnum; add `MaintenanceResult` dataclass; add `mode` parameter to key functions; improve return types
- `tests/test_db_maintenance.py` — add tests for best-effort mode behavior (exception catching, result reporting)
- `docs/06_shared_05_db_api_and_operations.md` — document maintenance modes in API reference

**Out of scope:**
- No changes to `agent/services/db_maintenance_service.py` — it wraps maintenance functions but doesn't need mode awareness itself
- No changes to CLI commands — `/db` commands use STRICT by default (existing behavior preserved)

## Assumptions

1. The existing functions that raise exceptions directly: `vacuum_db()`, `checkpoint_wal()` (via `db.checkpoint()`), `purge_old_sessions()` (via `db.execute()`), `prune_old_memories()` (via store's `delete_memories_before()`).
2. Functions that already return structured results: `recover_corruption()` returns `RecoveryResult`; `rotate_rag_db()`, `rotate_session_db()` return `Path`.
3. Existing tests expect exceptions to propagate in the current code — STRICT mode must be the default to preserve backward compatibility.

## Design

### New types

```python
from enum import StrEnum

class MaintenanceMode(StrEnum):
    STRICT = "strict"       # Exceptions propagate (default, preserves current behavior)
    BEST_EFFORT = "best_effort"  # Exceptions caught, logged, result returned with failure detail

@dataclass(frozen=True)
class MaintenanceResult:
    """Structured result of a best-effort maintenance operation."""
    success: bool
    action: str  # e.g. "vacuum", "purge", "prune"
    mode: MaintenanceMode
    detail: str | None = None
    data: dict | None = None  # Optional payload (e.g. PurgeCounts, row counts)
```

### Changes per function

**`vacuum_db()`:**

```python
def vacuum_db(db: SQLiteHelper, mode: MaintenanceMode = MaintenanceMode.STRICT) -> MaintenanceResult:
    """Run VACUUM to rebuild the DB file and reclaim freed pages.
    
    In STRICT mode (default), raises on failure (current behavior).
    In BEST_EFFORT mode, returns MaintenanceResult(success=False, detail=str(exc)).
    """
    try:
        db.vacuum()
        return MaintenanceResult(
            success=True, action="vacuum", mode=mode,
            data={"message": "VACUUM completed successfully"},
        )
    except (sqlite3.OperationalError, RuntimeError) as e:
        logger.error("VACUUM failed: %s", e)
        if mode == MaintenanceMode.STRICT:
            raise
        return MaintenanceResult(
            success=False, action="vacuum_failed", mode=mode, detail=str(e),
        )
```

### Backward compatibility

- Default `mode=STRICT` preserves existing exception-propagation behavior.
- Existing callers that don't pass `mode` get the same behavior as before.
- Return type changes: `vacuum_db()` now returns `MaintenanceResult` instead of `None`. Callers that ignore the return value are unaffected. Callers that check for exceptions are unaffected (STRICT mode raises).
