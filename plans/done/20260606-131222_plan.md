# Markdown Context Compression Engine MCP Server — Implementation Plan

## Goal
Implement a dedicated MCP server that compresses large Markdown while preserving meaning and retrieves only the necessary parts for injection.

## Scope
- AST parsing of Markdown files, or heading-aware parsing
- section-level chunk generation
- index construction equivalent to `.mdq/index.sqlite`
- `search_docs(query)`
- `get_chunk(chunk_id)`
- `outline(path)`
- path / tag / heading filters
- BM25 / grep / heading traverse
- incremental indexing
- MCP server implementation
- Agent-side policy update so that `search_docs` is preferred over `read_file`

## Assumptions
- Current ingestion pipeline includes `web_crawler.py`, `chunk_splitter.py`, `rag_ingester.py`
- `chunk_splitter.py` already supports Markdown snippet splitting by heading boundary via `md_index_enable`
- The current Agent assumes HTTP MCP servers using `/v1/call_tool` and `/v1/tools`
- The current `file-mcp` provides `read_text_file`, `search_files`, `grep_files` but not Markdown heading structure or heading-path based partial retrieval
- Introduce new MCP server: `mdq-mcp` separate from `file-mcp` that provides Markdown-specific index and partial retrieval APIs

## Unknowns
- What specific Markdown files need to be indexed
- What the exact structure of the `mdq-mcp` server implementation will be
- How the integration with existing tools will work
- What the exact schema for the new database tables will be

## Design
- Use FastAPI + Uvicorn for HTTP MCP server implementation
- Use SQLite with FTS5 for indexing
- Reuse existing Markdown heading-splitting logic from `chunk_splitter.py`
- Implement `search_docs`, `get_chunk`, `outline` tools
- Add incremental indexing capability
- Follow existing MCP server patterns for integration with watchdog and `/mcp`

## Implementation steps
1. Create work plan file
2. Read relevant existing files
3. Analyze unknowns and risks
4. Create implementation plan
5. Implement MCP server components
6. Integrate with existing system
7. Test implementation

## Validation plan
- Unit tests for heading extraction, outline correctness, incremental indexing
- Integration tests for `index_paths -> search_docs -> get_chunk` flow
- Performance tests with large Markdown files
- Verify `/v1/tools` definition diff check works
- Verify watchdog / OpenRC startup verification

## Risks
- Incomplete Markdown parser coverage - addressed by using heading-aware parse in Phase 1
- SQLite index growth - addressed by incremental indexing and content hash comparison
- Agent keeps using `read_file` - addressed by explicit rule in tool_definitions and system prompt
- Variability in Japanese search precision - addressed by using normalized FTS fields like current RAG system
