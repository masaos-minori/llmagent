# Plan: Make Workflow/Run State a First-Class Execution Model

Derived from `01_plan.md` using requirement `requires/20260618_22_require.md`.

---

## Goal

Workflow/run state (not session/turn history) is the primary execution model; conversation history is a subordinate concern.

---

## Scope

**In scope:**
- `scripts/agent/workflow/models.py` — add approval state; decouple from session_id/turn_number
- `scripts/db/workflow_schema.py` — add approval table; optionally decouple tasks from session
- `scripts/agent/workflow/state_store.py` — CRUD for new approval state
- `scripts/agent/workflow/workflow_engine.py` — integrate approval gates into stage transitions
- `scripts/agent/orchestrator.py` — shift primary model from turn to workflow
- `config/workflows/*.json` — may need schema update for approval stages
- `docs/05_agent_03_turn-processing-flow.md` — update to reflect workflow-first model
- `tests/` — update existing tests; add approval state tests

**Out of scope:**
- RAG pipeline changes
- MCP server changes
- AgentREPL UI layer changes (input loop, command dispatch)
- LLM client / streaming changes
- Memory layer changes

---

## Assumptions

1. The existing `TaskRecord` → `AttemptRecord` → `ArtifactRef` chain is sound and only needs extension, not replacement.
2. Approval decisions can be modeled as a `pending → approved | rejected` state attached to either a task or a stage, depending on design choice.
3. Separating workflow state from session/turn state does not require migration of existing session data (new sessions use the new model).
4. The `Orchestrator` is the correct integration point for workflow-first execution.

## Implementation Steps

1. **Define approval state model**
   - Design: per-task (single approval gate) vs per-stage (approval per step)
   - Add `ApprovalRecord` dataclass to `models.py`
   - Add `approvals` table to `workflow_schema.py`
   - Criterion: models + schema compile cleanly

2. **Decouple TaskRecord from session/turn**
   - Make `session_id` and `turn_number` optional (nullable) or remove
   - Update `state_store.py` queries that reference these fields
   - Criterion: existing workflow engine tests still pass

3. **Integrate approval into WorkflowEngine**
   - Add `request_approval()` and `resolve_approval()` methods
   - Engine pauses stage execution when approval is pending
   - Criterion: new approval gate tests pass

4. **Update Orchestrator**
   - Shift `handle_turn()` to create workflow context first, then attach turn
   - Criterion: turn processing still works end-to-end

5. **Update CLI/commands** (if needed)
   - Add `/approve`, `/reject` slash commands in `CommandRegistry`
   - Criterion: approval flow works from REPL

6. **Update documentation**
   - Update `docs/05_agent_03_turn-processing-flow.md` to describe workflow-first model
   - Document approval state lifecycle

7. **Lint, type, arch, security validation**
   - `ruff check scripts/` — 0 errors
   - `mypy scripts/` — no new errors
   - `lint-imports` — 0 violations
