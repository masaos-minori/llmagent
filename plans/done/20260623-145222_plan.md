# Plan: Surface RAG Quality Degradation to Operators

**Requirement:** `requires/20260622_85_require.md`

---

## Goal

Make RAG retrieval and ingestion degradation visible in terminal output and diagnostic artifacts. `SearchDiagnostics` (embed_failed, fts_errors) is already tracked internally but never reaches operator-visible surfaces ‚Äî only `logger.warning()`.

---

## Current State

| Signal | Tracked? | Logged? | Terminal-visible? | Diagnostic artifact? |
|---|---|---|---|---|
| Search embed_failed | Yes (`ctx.search_diagnostics`) | `logger.warning()` | No | No |
| Search fts_errors | Yes (`ctx.search_diagnostics`) | `logger.warning()` | No | No |
| Zero results (normal) | Yes (empty context) | No | "No results found." | No |
| Zero results (degraded) | No distinction | No | "No results found." (same) | No |
| Refiner fallback | Yes (`stage_results`) | Yes | `[warn] refiner fallback: ...` | stage_results |
| Ingestion embed_failed | Yes (`n_embed_failed`) | `logger.info()` final summary | No | No |
| Post-ingest consistency | Yes | `logger.warning()` | No | No |

**Refiner fallback warning already exists** in `cmd_ingest.py:159-163` as precedent for `[warn]` surface.

**`pipeline.get_diagnostics()`** at `pipeline.py:458-489` includes: stage_results, timings, fetch_result, fusion_mode, fallback_count, fallback_reasons, hit_counts ‚Äî but NOT `search_diagnostics`.

## Scope

**In:**
- `scripts/rag/pipeline.py` ‚Äî expose `last_search_diagnostics` attribute; include in `get_diagnostics()`; add session-level cumulative `stat_search_embed_failed` and `stat_search_fts_errors` counters
- `scripts/agent/commands/cmd_ingest.py` ‚Äî show `[warn]` terminal line when `last_search_diagnostics` is degraded; enhance "No results found." message when degraded
- `scripts/agent/services/ingest_workflow.py` ‚Äî emit terminal warning when `n_embed_failed > 0` in ingestion
- `scripts/agent/commands/cmd_config_stats.py` ‚Äî add RAG session degradation line to `/stats` when non-zero
- `scripts/agent/commands/models.py` ‚Äî add `rag_search_embed_failed: int = 0` and `rag_search_fts_errors: int = 0` to `StatsViewModel`

**Out:**
- Changing graceful degradation policy
- Adding per-turn warnings to the user's message stream
- Changing `EmbeddingClient` behavior
- Adding degradation signals to post-MCP-tool results (different scope)

---

## Assumptions

1. `RagPipeline.run()` sets `self.last_stage_results`; similarly adding `self.last_search_diagnostics` is consistent with this pattern.
2. `pipeline.augment()` calls `run()` internally and returns the context block ‚Äî after it returns, `last_stage_results` is current. Same pattern applies for `last_search_diagnostics`.
3. `cmd_ingest.py._cmd_rag()` creates a new `RagPipeline` per invocation ‚Äî session-level counters accumulate across multiple `/rag search` calls within the same pipeline instance. Since a new instance is created each time, per-call stats are the right granularity for `/stats`. The agent turn's `rag_stage_outcomes` list in `repl.py:276` already aggregates across tools calls per turn; same approach applies here.
4. The precedent of `[warn] refiner fallback: ...` in `cmd_ingest.py:160-163` confirms that post-pipeline warnings are appropriate in the `/rag search` output.
5. `IngestWorkflow` already emits per-step status messages ‚Äî adding embed_failed warning fits the existing pattern.

## Design

### Change 1: `pipeline.py` ‚Äî `last_search_diagnostics` + `get_diagnostics()` + cumulative counters

In `RagPipeline.__init__()`, add:
```python
self.last_search_diagnostics: SearchDiagnostics = SearchDiagnostics()
self.stat_search_embed_failed: int = 0
self.stat_search_fts_errors: int = 0
```

In `run()`, after `ctx.search_diagnostics = diagnostics` (currently in `SearchStage.run()`), save to pipeline after `run()` completes:
```python
# After self.last_stage_results = list(ctx.stage_results)
self.last_search_diagnostics = ctx.search_diagnostics
self.stat_search_embed_failed += ctx.search_diagnostics.embed_failed
self.stat_search_fts_errors += ctx.search_diagnostics.fts_errors
```

In `get_diagnostics()`, add to the returned dict:
```python
"search_diagnostics": {
    "embed_ok": self.last_search_diagnostics.embed_ok,
    "embed_failed": self.last_search_diagnostics.embed_failed,
    "fts_errors": self.last_search_diagnostics.fts_errors,
    "degraded": (
        self.last_search_diagnostics.embed_failed > 0
        or self.last_search_diagnostics.fts_errors > 0
    ),
},
```

### Change 2: `cmd_ingest.py` ‚Äî `/rag search` degradation terminal output

After `context = await pipeline.augment(query, debug_fn=debug_fn)`:

```python
diag = pipeline.last_search_diagnostics
search_degraded = diag.embed_failed > 0 or diag.fts_errors > 0

if not context:
    if search_degraded:
        parts = []
        if diag.embed_failed > 0:
            parts.append(f"embed_failed={diag.embed_failed}")
        if diag.fts_errors > 0:
            parts.append(f"fts_errors={diag.fts_errors}")
        self._out.write(f"No results found [warn: search degraded ‚Äî {', '.join(parts)}]")
    else:
        self._out.write("No results found.")
else:
    self._out.write(context)

# After the refiner fallback check, add:
if search_degraded:
    parts = []
    if diag.embed_failed > 0:
        parts.append(f"embed_failed={diag.embed_failed}")
    if diag.fts_errors > 0:
        parts.append(f"fts_errors={diag.fts_errors}")
    self._out.write(
        f"[warn] search degraded: {', '.join(parts)}"
        " ‚Äî results may be incomplete (FTS-only fallback)"
    )
```

### Change 3: `ingest_workflow.py` ‚Äî emit terminal warning for embed_failed

After ingestion completes and `result.n_embed_failed > 0`:
```python
if result.n_embed_failed > 0:
    messages.append(
        self._emit(
            f"[warn] {result.n_embed_failed} chunk(s) failed embedding ‚Äî "
            "stored without vec index; FTS search still works",
            on_status,
        )
    )
```

### Change 4: `models.py` + `cmd_config_stats.py` ‚Äî `/stats` RAG degradation row

In `StatsViewModel`, add:
```python
rag_search_embed_failed: int = 0
rag_search_fts_errors: int = 0
```

In `_collect_stats()`, these would need a reference to `RagPipeline` instance. However, `RagPipeline` is instantiated per-command call ‚Äî there is no long-lived pipeline instance in the agent context.

**Resolution for `/stats`:** The session-level degradation counters should live on the `AgentContext` or `SessionServices`, not on `RagPipeline`. Instead, after each `/rag search` call that completes with degradation, update a session-level counter on `ctx`.

Update design:
- Add `rag_search_embed_failed: int = 0` and `rag_search_fts_errors: int = 0` fields to `AgentContext.services` (or a new `RagStats` dataclass on services)
- After `/rag search` completes, if degraded: `ctx.services.rag_stats.embed_failed += diag.embed_failed`
- `/stats` reads from `ctx.services.rag_stats`

**Simplified alternative:** Since `RagPipeline` is per-call and agent services don't yet have a `rag_stats` slot, start with the simpler approach: show degradation in `/rag search` terminal output (Change 2) and in `get_diagnostics()` (Change 1). Skip `/stats` integration until `rag_stats` accumulator is added to agent services ‚Äî mark as a follow-up in the plan.

**In-scope for this plan:** Changes 1, 2, 3. Change 4 (`/stats` RAG row) tracked as follow-up ‚Äî requires adding a new `rag_stats` accumulator to `AgentContext.services`.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| `[warn] search degraded` shown even on transient single-query embed failure in multi-query search | Low | Display is per-run (one `/rag search` call); the `embed_failed` count accurately reflects the degradation scope |
| Ingestion embed_failed warning is noisy during large batch ingests where a few chunks legitimately fail | Low | Warning includes count and notes "FTS still works" ‚Äî operators can assess severity; no action required unless count is high |
| `last_search_diagnostics` is stale between searches (instance-per-call pattern) | Low | Each `/rag search` creates a new `RagPipeline`; `last_search_diagnostics` is per-instance and read immediately after `augment()` ‚Äî no staleness possible |
