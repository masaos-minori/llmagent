## Goal

Remove the disabled-mode direct LLM/tool execution path from `Orchestrator.handle_turn()` so that no user turn can execute without a workflow task, stage, and audit context.

## Scope

**In**:
- Remove the `if not self._policy.is_workflow_enabled()` branch and its `logger.info("Workflow mode=disabled — direct execution")` log in `scripts/agent/orchestrator.py` (line 195–208)
- Remove the fallback `await self._process_turn(...)` / `await self._handle_turn_end(...)` call that follows the direct-execution branch (lines 203–208)
- Ensure the `else: await self._handle_workflow_engine(...)` path is the only turn path
- Remove `is_workflow_enabled()` from `WorkflowExecutionPolicy` (covered by plan 20260707-095938 which deletes the class; this plan documents the specific code site)
- Remove any tests that assert tool or LLM execution is possible without a workflow task

**Out**:
- Removal of `WorkflowExecutionPolicy` class (covered by plan 20260707-095938)
- Startup fallback removal (covered by plan 20260707-095941)
- Approval gate changes (covered by plan 20260707-095942)

**Note**: This requirement (req 02) overlaps significantly with req 03 (plan 20260707-095940). Both target the disabled-mode direct execution branch in `orchestrator.py:195–208`. These two plans should be executed together or req 02 should be treated as a subset of req 03. Req 03 is the authoritative, cleaner statement of the same change.

## Assumptions

1. After this change, `handle_turn()` must always go through `_handle_workflow_engine()`.
2. The `elif self._workflow_def is None: self._log_fallback(...)` branch (line 197) becomes the only failure mode when workflow is unavailable.
3. `_process_turn()` remains callable from within `_handle_workflow_engine()` stage callbacks; only the top-level direct call is removed.
4. No test legitimately tests "tool called without workflow" as a success path.

         await self._handle_workflow_engine(line, ctx, turn_started_at)

     await self._handle_turn_end(...)
