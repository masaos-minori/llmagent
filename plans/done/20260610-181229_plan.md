# GitHub MCP fail-fast 化・責務分離・型強化 作業計画

## Goal

`mcp/github/` 全体を fail-fast・厳密型付け・責務分離の原則で再構成し、`_get_cfg()` module-cache / `HTTPException` service 混在 / `except Exception` による暗黙失敗吸収を廃止する。

---

## Scope

**In:**
- `scripts/mcp/github/models.py` — `_get_cfg()` + `{}` fallback 廃止、型付き設定モデル導入
- `scripts/mcp/github/service.py` — `HTTPException` 除去・domain 例外導入・`_get_cfg()` 参照廃止・formatter/mapper/audit 分離
- `scripts/mcp/github/tools.py` — `_MCP_TOOLS` 手書きリスト廃止・models から schema 生成
- `scripts/mcp/github/server.py` — 統一例外ハンドラ・thin transport layer 化・route declarative 登録
- `tests/test_github_mcp_service.py` — 異常系テスト追加

**Out:**
- `docs/04_mcp-github.md` 更新は実装完了後の別タスク
- `config/github_mcp_server.toml` スキーマ変更なし
- `shared/` 層の変更なし
- DB スキーマ変更なし

---

## Assumptions

1. `models.py` の `_get_cfg()` 廃止後は FastAPI lifespan または `_LazyGitHubService.start()` で設定を一度ロードし、型付き config object を service に注入する。
2. `service.py` が現在投げている `HTTPException` は `server.py` の統一例外ハンドラで HTTP response に変換する。
3. `tools.py` の `inputSchema` を Pydantic `model_json_schema()` から生成しても、既存の MCP client が受け取る JSON は変わらない。
4. `get_dispatch_table()` の廃止後は `server.py` 側で tool 名 → service メソッドの静的 mapping を持つ。
5. テスト (`test_github_mcp_service.py`) は `_get_cfg()` を monkeypatch して使っているため、config injection への切り替え後はコンストラクタ/fixture を更新する必要がある。

## Implementation Steps

### Step 1: 型付き設定モデルと domain 例外の定義 (`models.py`)

**変更内容:**
- `_get_cfg()` / `_cfg` module 変数を削除
- `@dataclasses.dataclass` `GitHubConfig` を追加 (allowed_repos, allowed_repos_mode, path_denylist, protected_branches, max_file_size_kb, audit_log_path, allow_force_push, require_pr_review, default_per_page, max_per_page, llm_url)
- `GitHubConfig.from_dict(d: dict[str, Any]) -> GitHubConfig` class method を追加
- `DEFAULT_PER_PAGE = 10` をモジュール定数に変更 (設定依存を排除)
- domain 例外クラスを `models.py` または専用 `exceptions.py` に追加:
  - `GitHubAuthorizationError(RuntimeError)` — 403 相当
  - `GitHubNotFoundError(RuntimeError)` — 404 相当
  - `GitHubValidationError(ValueError)` — 400/422 相当
  - `GitHubConflictError(RuntimeError)` — 409 相当
  - `GitHubUpstreamError(RuntimeError)` — 5xx 相当
  - `GitHubAuditError(RuntimeError)` — 監査失敗

**完了条件:** `uv run mypy scripts/mcp/github/models.py` no new errors

---

### Step 2: `service.py` の fail-fast 化・config injection

**変更内容:**
- `GitHubService.__init__` に `cfg: GitHubConfig` 引数を追加; `_get_cfg()` 呼び出しを削除
- `_assert_allowed_repo()`, `_assert_allowed_path()`, `_assert_allowed_branch()` が `_get_cfg()` ではなく `self._cfg` を使うよう変更
- `HTTPException` を全削除; 代わりに Step 1 で定義した domain 例外を使用
- `_handle_github_error()` を GithubException の status code ごとに domain 例外へ変換するよう再設計
- `_write_github_audit_log()` が失敗したとき warning のみにしない; `GitHubAuditError` を送出 or 上位に伝播

**完了条件:** `uv run ruff check scripts/mcp/github/service.py` 0 errors; HTTPException import が消える

---

### Step 3: `service.py` formatter / mapper 分離

**変更内容:**
- `fmt_*` 系メソッド群を `mcp/github/formatter.py` へ移動
- `_issue_to_info()`, `_pr_to_info()` を `mcp/github/mapper.py` へ移動
- `_dry_run_preview()` を `formatter.py` へ移動
- `service.py` は business logic のみを持ち、文字列整形を持たない形に

**完了条件:** `uv run pytest tests/test_github_mcp_service.py -q` pass

---

### Step 4: `tools.py` の schema 単一化

**変更内容:**
- `_MCP_TOOLS` 手書きリストを廃止
- 各 request モデル (`SearchRepositoriesRequest` 等) から `model.model_json_schema()` でスキーマを生成する `build_tool_list()` 関数を追加
- `name`, `description` はツール名→説明の dict として別管理
- tool schema と request model fields の一致を検証するテストを追加

**完了条件:** `_MCP_TOOLS` と `build_tool_list()` 生成結果の JSON diff が 0; テストがモデルとスキーマの一致を assert する

---

### Step 5: `server.py` の thin transport 化

**変更内容:**
- 統一例外ハンドラ (`@app.exception_handler`) を追加: `GitHubAuthorizationError` → 403, `GitHubNotFoundError` → 404, `GitHubValidationError` → 400, `GitHubConflictError` → 409, `GitHubUpstreamError` → 502
- FastAPI lifespan で `GitHubConfig.from_dict(ConfigLoader().load(...))` を実行し `app.state.cfg` に格納
- `Depends` で service instance を注入
- route 関数を request model → `service.method()` → response model の 3 段階に統一
- `list_tools()` を `tools.build_tool_list()` から動的生成に変更

**完了条件:** `uv run mypy scripts/mcp/github/server.py` no new errors; 全 route が 10 行以内

---

### Step 6: 異常系テスト追加

追加テストケース:
- `test_config_load_failure_raises` — ConfigLoader 失敗時に例外が伝播すること
- `test_unauthorized_repo_raises_authorization_error` — 無許可 repo で `GitHubAuthorizationError`
- `test_unauthorized_path_raises_authorization_error` — path denylist にマッチで `GitHubAuthorizationError`
- `test_protected_branch_raises_authorization_error` — 保護ブランチ操作で `GitHubAuthorizationError`
- `test_github_api_404_raises_not_found_error` — GithubException(404) → `GitHubNotFoundError`
- `test_github_api_403_raises_authorization_error` — GithubException(403) → `GitHubAuthorizationError`
- `test_audit_failure_raises_audit_error` — 監査ログ書き込み失敗で `GitHubAuditError`
- `test_tool_schema_matches_request_model` — tool schema と Pydantic model fields の一致

**完了条件:** `uv run pytest tests/test_github_mcp_service.py -v` 全 pass; diff-cover ≥ 90%

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| `server.py` の lifespan DI 移行で `_service` singleton と `_GITHUB_TOKEN` の既存 import が壊れる | high | Step 5 で `_service` と `_GITHUB_TOKEN` の module-level 公開を維持するか、`app.state` から取得する薄い wrapper を残す |
| `_MCP_TOOLS` と `model_json_schema()` 生成結果が細部で差異を持ち MCP client が誤動作する | medium | Step 4 で JSON diff テストを先に書き、差異を事前に確認してから切り替える |
| `test_github_mcp_service.py` (868 行) が `_get_cfg()` monkeypatch に依存しており、config injection 移行で多数のテストが壊れる | high | Step 2 と同時に fixture を更新し、`GitHubConfig` インスタンスを直接渡すパターンに移行する |
| `GitHubAuditError` を必須送出にすると、本番環境で audit_log_path 未設定時に書き込み操作が全失敗する | medium | `audit_log_path` が空の場合は audit をスキップする policy を `GitHubConfig` に持たせ、明示的に設定する |
| formatter.py / mapper.py の分離で `test_github_mcp_service.py` の import パスが変わる | low | テスト内の import を一括更新; formatter/mapper は `from mcp.github.service import fmt_*` の backward compat re-export を一時的に残す |
