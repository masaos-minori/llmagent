# Plan: Add DB Recreation Procedure to Operations Documentation

**Source requirement:** `requires/20260702_46_require.md`

---

## 1. Goal

Add a clear DB recreation procedure to the operations documentation so that operators have a documented, official workflow for applying schema changes by recreating DB files from the latest DDL.

---

## 2. Scope

**In-Scope:**
- Add a "DB Recreation" section to `docs/90_shared_05_db_api_and_operations.md`
- Add a reference/cross-link to the DB recreation procedure in `docs/90_shared_04_db_architecture_and_schema.md`
- Document the archive step using `rotate_all_dbs()` before deletion
- Document removal of existing DB files (`rag.sqlite`, `session.sqlite`, `workflow.sqlite`)
- Document schema recreation using `create_schema()`
- Explicitly state that automatic migration is unsupported
- Explicitly state that recreated DBs are empty and use the latest DDL
- Explicitly state that existing records are not migrated automatically

**Out-of-Scope:**
- Automated migration tooling
- Data export/import tooling
- Changes to `maintenance.py`, `create_schema.py`, or any Python source files
- `eventbus.sqlite` recreation (not mentioned in requirement; see Unknowns)

---

## 3. Assumptions

1. DB paths are configured in `common.toml` (not hardcoded in docs) — docs should reference `common.toml` keys (`rag_db_path`, `session_db_path`, `workflow_db_path`) rather than hardcode `/opt/llm/db/*.sqlite`.
2. `rotate_all_dbs()` in `scripts/db/maintenance.py` covers all three DB files (`rag.sqlite`, `session.sqlite`, `workflow.sqlite`) via `_archive_db_file()`, which uses the SQLite online backup API for WAL-consistent snapshots — confirmed by code inspection.
3. `create_schema()` in `scripts/db/create_schema.py` creates schemas for all four DB files (`rag.sqlite`, `session.sqlite`, `workflow.sqlite`, `eventbus.sqlite`) — confirmed by code inspection.
4. The archive directory defaults to `sqlite_archive_dir` in `common.toml` (default: `/opt/llm/db/archive`) — confirmed by `_archive_db_file()` in `maintenance.py`.
5. The documentation change is docs-only; no Python source changes are needed.
6. The target docs use Markdown with section headers; new sections should follow the existing heading style.
