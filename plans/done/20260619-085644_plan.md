# Plan: Fix field loss in RagIngester._read_chunk_json()

## Goal

Preserve all chunk metadata fields from chunk JSON files through to the database. Fix the downstream consumers that currently drop fields after `_read_chunk_json()` returns them.

## Scope

**In scope:**
- Schema migration: add `chunk_type TEXT` and `source_file TEXT` columns to `chunks` table (idempotent ALTER TABLE)
- Update `_insert_chunk()` to accept and store `chunk_type` and `source_file`
- Update `_embed_and_store()` to extract and pass all fields from chunk JSON dict
- Add schema migration helper for idempotent column addition
- Ensure FTS5 and chunks_vec tables are not affected (chunk_type/source_file are not indexed)

**Out of scope:**
- Adding `chunking_strategy` as a per-chunk column (already stored at document level from first chunk; per-chunk strategy would require schema redesign)
- Migrating existing data (new columns default to NULL for existing chunks)
- Adding retrieval queries that filter by chunk_type or source_file (future enhancement)
- Changing the chunk JSON file format

## Assumptions

1. `_read_chunk_json()` already returns the full dict with all 10 fields — the bug is in downstream consumers, not in this method.
2. The requirement's mention of `normalized_content` and `chunk_index` being dropped is incorrect — both are already preserved and stored. Only `source_file` and `chunk_type` are lost.
3. `IF NOT EXISTS` on CREATE TABLE means existing tables are protected; ALTER TABLE for new columns needs idempotent handling.
4. New columns will be NULL for existing chunks — acceptable since this is additive metadata.

## Implementation Steps

   - In `schema_sql.py`: add `_ALTER_CHUNKS_ADD_COLUMNS` SQL that checks for column existence before ALTER
   - Use Python-side check: query `PRAGMA table_info(chunks)` and only ALTER if column missing
   - Call migration from `create_rag_schema()` after CREATE TABLE statements (idempotent)
   - Completion criterion: running schema creation on existing DB adds new columns without errors

2. **Update `_insert_chunk()` in `ingester.py`**
   - Add `chunk_type: str = ""` and `source_file: str = ""` parameters with defaults
   - Include these fields in the INSERT statement
   - Completion criterion: _insert_chunk accepts and stores chunk_type and source_file

3. **Update `_embed_and_store()` in `ingester.py`**
   - Extract `chunk_type` and `source_file` from the parsed JSON dict (already available via `_read_chunk_json()`)
   - Pass them to `_insert_chunk()`
   - Completion criterion: chunk_type and source_file flow from chunk file → DB

4. **Verify field preservation**
   - Trace: chunk JSON file → `_read_chunk_json()` → `_embed_and_store()` → `_insert_chunk()` → DB
   - Confirm all 10 fields are either stored (chunk_type, source_file) or already preserved (content, normalized_content, chunk_index, url, title, lang, etag, last_modified, chunking_strategy)
   - Completion criterion: all chunk fields accounted for

5. **Update tests**
   - Add test verifying new columns exist in chunks table after schema creation
   - Add test verifying chunk_type and source_file are stored correctly
   - Update existing tests if INSERT assertions need updating
