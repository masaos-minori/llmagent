# Plan: Add `from __future__ import annotations` to `scripts/shared/tool_executor.py`

Source requirement: `requires/20260721_14_require.md`
Source issue: `issues/20260721-120414_tool_executor_missing_future_annotations_nameerror.md`

## Goal

Make `scripts/shared/tool_executor.py` importable again by adding the missing
`from __future__ import annotations` import, eliminating the startup-breaking
`NameError: name 'RuntimeToolRegistry' is not defined` that currently crashes every
import of `agent.factory`, `agent.context`, and `agent.repository_gateway`.

## Scope

**In scope:**
- Add `from __future__ import annotations` as the first import line in
  `scripts/shared/tool_executor.py`, placed immediately after the module docstring
  and before all other imports (matching the convention in
  `scripts/agent/tool_policy.py` and `scripts/agent/repository_gateway.py`).
- Add or extend a regression test asserting that `shared.tool_executor` and
  `agent.factory` import successfully (smoke-import test), per the requirement's
  Acceptance Criteria.

**Out of scope (per requirement's Non-Goals):**
- No change to the `TYPE_CHECKING` guard, the `RuntimeToolRegistry` import, or the
  `set_runtime_registry` method signature.
- No `from __future__ import annotations` audit/rollout to other files.
- No fix for unrelated pre-existing lint/architecture/bandit findings noted in the
  issue (`rag_pipeline_server.py` formatting, `eventbus` layering violations, etc.).
- No refactor of `ToolExecutor` / `ToolRouteResolver` / `RuntimeToolRegistry` routing
  logic.

## Assumptions

1. The current `master` state matches the issue's reproduction: `tool_executor.py`
   has no `from __future__ import annotations` and raises `NameError` on import.
   (Verified directly in Step 3 — confirmed below.)
2. Adding `from __future__ import annotations` only defers annotation evaluation; it
   does not change runtime behavior for any annotation in this file, since none of
   the file's annotations are otherwise runtime-inspected (verified in Step 3 via
   grep — no `get_type_hints`/`inspect.signature` calls target this module).
3. mypy already treats annotations as deferred by default (per `pyproject.toml`
   config confirmed in `rules/coding.md`), so no new mypy errors are expected from
   this change.
4. A smoke-import regression test does not yet exist for `shared.tool_executor` /
   `agent.factory` — one must be added or an existing test extended to satisfy the
   requirement's Acceptance Criteria. This will be determined precisely at
   implementation time by searching `tests/` for an existing "import smoke test"
   pattern; if one already exists (e.g. covering other modules), it can be extended
   rather than duplicated.

### Step 4 resolution of UNK-01

Directly verified (not assumed) via:
```
rg -l "def test_.*import" tests/
rg -n "^import shared\.tool_executor|^from agent import factory|^import agent\.factory" tests/
ls tests/ | grep -i -E "smoke|import|startup"
```
Findings:
- No file in `tests/` contains a dedicated, explicit "does this module import
  successfully" smoke test targeting `shared.tool_executor` or `agent.factory`.
- `tests/test_agent_factory.py` already imports `agent.factory` at module level
  (line 16: `from agent.factory import _ServerLifecycleRouter, build_agent_context,
  init_tracer`), so a collection failure there would *implicitly* surface this
  exact `NameError` today — but there is no explicit assertion naming this failure
  mode, so a future regression could resurface as a confusing collection error
  rather than a clearly labeled regression test.
- `tests/test_startup.py` exists but is a `StartupOrchestrator`-focused behavior
  test suite (heavy mocking via `unittest.mock`), not a plain import-smoke test —
  not a good fit to extend for this narrow purpose.
- **Resolution**: create a new, small, dedicated file `tests/test_import_smoke.py`
  containing explicit test functions that import `shared.tool_executor`,
  `agent.factory`, `agent.context`, and `agent.repository_gateway` and assert no
  exception is raised. This directly satisfies the requirement's Acceptance
  Criteria wording ("a regression test ... asserts that `shared.tool_executor` and
  `agent.factory` import successfully ... so this exact failure mode cannot
  silently regress again undetected") with a clearly labeled, single-purpose test,
  rather than relying on an implicit side effect of an unrelated test file's
  collection succeeding.

## Design

Confirmed by direct inspection (Explore sub-agent read of full file contents, plus
grep run directly by the planner — not assumed):

- `scripts/shared/tool_executor.py` current top-of-file structure (as read):
  1. Shebang line `#!/usr/bin/env python3`
  2. Module docstring (lines ~2-11, describing `ToolExecutor` as the MCP tool
     execution layer)
  3. `import asyncio` / `import logging` / `import time` / `from collections import
     OrderedDict` / `from typing import TYPE_CHECKING, Any`
  4. `if TYPE_CHECKING:` guard (lines ~19-20) importing `RuntimeToolRegistry` from
     `shared.runtime_tool_registry`
  5. `import httpx` and further `shared.*` imports
  6. Class `ToolExecutor(ToolTransportInvoker)` whose method `set_runtime_registry`
     (line ~76) is annotated `registry: RuntimeToolRegistry` — this bare name is
     evaluated eagerly at class-body execution time without `from __future__ import
     annotations`, causing the `NameError` (matches the issue's reproduction
     traceback exactly).
- The fix: insert `from __future__ import annotations` as the first import,
  immediately after the module docstring and before `import asyncio`, exactly
  mirroring the placement in `scripts/agent/tool_policy.py` (line 9) and
  `scripts/agent/repository_gateway.py` (line 13) — both place it directly after
  the module docstring, before any other import.
- **Required-Changes confirmation check (run directly, not delegated, not
  assumed):**
  ```
  rg -n "get_type_hints|inspect\.signature" scripts/ tests/
  ```
  Result: the only matches are in `tests/test_command_registry_consistency.py`
  (lines 59 and 81), which concern `CommandRegistry` function signatures and have
  no relationship to `ToolExecutor` or `scripts/shared/tool_executor.py`. **No
  caller anywhere in `scripts/` or `tests/` calls `get_type_hints()` or
  `inspect.signature()` against `ToolExecutor`, `tool_executor.py`, or
  `set_runtime_registry`.** This confirms the fix is side-effect-free: no code path
  depends on eager annotation evaluation for this module, so deferring evaluation
  via `from __future__ import annotations` cannot change any runtime behavior in
  this file or any caller of it.
- No other annotation in the file is runtime-fragile: the Explore sub-agent
  confirmed no `dataclass` fields, no `Protocol`/`@runtime_checkable` usage, and no
  other bare `TYPE_CHECKING`-only names used as annotations elsewhere in the file —
  `set_runtime_registry`'s parameter is the sole occurrence of this pattern.

## Risks

- **Risk**: The smoke-import test location is ambiguous until Phase 1's search is
  run, which could lead to creating a duplicate test file if an equivalent already
  exists.
  → **Mitigation**: Phase 1 explicitly requires searching `tests/` before writing
  the test; implementer must extend an existing import-smoke test if one is found,
  and only create a new file if none exists.
- **Risk**: A change this small (1 import line) could be mistaken for
  "obviously safe" and skip the full validation sequence, missing a regression in
  an unrelated part of the test suite that happens to also be affected by the same
  root cause.
  → **Mitigation**: Implementation steps explicitly mandate running the full
  `rules/toolchain.md` validation sequence, not just the three Acceptance-Criteria
  import checks, before considering the fix complete.
- **Risk**: The pre-existing 25 failed / 17 errors in the full suite might not all
  clear after this fix, if some failures share superficial symptoms but have a
  different root cause, leading to an incorrectly "expanded" scope for this fix.
  → **Mitigation**: Requirement's Acceptance Criteria and this plan's Validation
  Plan both explicitly state that residual failures unrelated to this root cause
  must be triaged separately, not silently absorbed into this fix's scope — keeps
  the fix minimal and auditable.
- **Risk (considered, deemed negligible)**: Deferred evaluation via
  `from __future__ import annotations` could theoretically change behavior for code
  that inspects annotations at runtime.
  → **Mitigation**: Already resolved by direct evidence in the Design section — a
  repo-wide grep for `get_type_hints`/`inspect.signature` found no caller targeting
  `ToolExecutor` or `tool_executor.py`. No further mitigation needed; this risk is
  fully closed by evidence, not merely assumed.
