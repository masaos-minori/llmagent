# Plan: [RAG][Remote] Define rag_service_url auth, retry, and fallback behavior

## Goal

Define and document (and optionally implement) authentication headers, retry policy, timeout behavior, and exact fallback conditions for the external RAG service delegation path (`rag_service_url`).

## Scope

**In scope:**
- `scripts/rag/pipeline_service.py` — `call_rag_service()` (current implementation)
- `scripts/rag/pipeline.py` — `_augment_http()`, `augment()` (delegation entry point)
- Configuration: `rag_service_url`, new config fields for auth/retry
- Documentation update in `03_rag_03_query_pipeline.md` and `03_rag_90_inconsistencies_and_known_issues.md` (OQ-1)

**Out of scope:**
- Changes to the external RAG service itself (rag-pipeline-mcp server)
- Changes to in-process pipeline behavior
- MCP transport layer modifications
- Semantic cache behavior changes

## Assumptions

1. `call_rag_service()` currently sends no authentication headers — this is a gap that needs definition.
2. No retry policy exists — any single HTTP failure triggers immediate fallback to in-process mode.
3. The fallback condition is: `result is None` (any exception) → in-process pipeline; `result is not None` (including empty string) → use result.
4. `rag_service_url=""` is hardcoded in MCP adapter to prevent HTTP loops (confirmed in models.py:93).
5. The external RAG service (`rag-pipeline-mcp`) runs on the same host or a trusted network — this affects auth strategy.
6. Existing behavior: any failure (4xx, 5xx, connection error, timeout, JSON parse error) returns `None` → fallback.

## Implementation Steps

1. **Define auth strategy** — document recommended approach (e.g., `X-RAG-Token` header from config); add optional `rag_auth_token` to RagConfigProtocol and MCP models.
2. **Add explicit timeout** to `call_rag_service()` — set `timeout=10.0` on the httpx.post() call (or use a client-level timeout).
3. **Add retry policy** — implement simple retry with exponential backoff (e.g., 2 retries, `min(2**i, 5)` sec delay) for transient errors (5xx, connection errors); do NOT retry on 4xx client errors.
4. **Clarify fallback conditions**:
   - `None` returned → fallback to in-process (connection error, timeout, JSON parse error, 5xx after retries exhausted)
   - `""` returned → accept as valid empty result (no fallback)
   - `"context"` returned → accept as valid result
5. **Update OQ-1** in `03_rag_90_inconsistencies_and_known_issues.md` — resolve the open question with documented decisions.
6. **Update docs** in `03_rag_03_query_pipeline.md` §1 — add HTTP mode details (auth, retry, fallback).
7. **Validation:** run `uv run pytest tests/ -x -q` to confirm no regressions.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Adding auth headers to existing code may break deployments that rely on no-auth setup | low | Make auth header optional (empty string = no auth); default behavior unchanged. |
| Retry policy may cause longer perceived latency before fallback | medium | Limit retries to 2 max; use short backoff (1s, 2s); document expected latency impact. |
| Empty results accepted as valid may mask remote service issues | low | Log a WARNING when empty results received from remote service (only once per query, not per chunk). |
