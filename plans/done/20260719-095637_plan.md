# Plan: Enable the persistent memory layer by default and add the missing chunking step

Source requirement: `requires/20260714_15_require.md`

## Goal

A fresh checkout has `use_memory_layer = true` and `memory_embed_enabled =
true` in effect by default, starts without a `ValueError` from the
`memory_local_only`/`embed_url` check, and no longer silently truncates
long extracted memory content — long content is split into multiple
stored/embedded chunks instead of being cut and discarded.

## Scope

**In scope**:
- Add a chunking stage to `scripts/agent/memory/extract.py` (replacing the
  hard truncation) so `_try_extract_from_assistant`/`_try_extract_from_user`
  produce multiple `MemoryEntry` rows for over-long content instead of one
  truncated entry.
- Flip `use_memory_layer` and `memory_embed_enabled` defaults to `true` in
  both `scripts/agent/config_dataclasses.py` and `config/agent.toml`.
- Resolve the `memory_local_only`/`embed_url` conflict via **Option B**
  (user-confirmed): set `config/agent.toml`'s `memory_local_only = false`;
  `embedding_client.py`'s `_LOCAL_PREFIXES` is left unchanged.
- Update the three target docs
  (`docs/05_agent_12_03_memory-module-ref-core-and-store.md`,
  `docs/05_agent_12_04_memory-module-ref-retrieval-and-injection.md`,
  `docs/05_agent_08_01_configuration-loading-agent-config-part1.md`) with
  the new defaults and the new chunking stage, including the accepted
  retrieval-fragmentation limitation (see Design).
- Confirm and state explicitly (no code change) that the SQLite
  `memories`/`memories_fts` schema remains the permanent "Vector DB" stage —
  satisfies Required Change 5 as a documentation/confirmation item.

**Out of scope** (per the requirement's Non-Goals, and per the two
user-confirmed decisions):
- No DB schema change (`scripts/db/schema_sql.py` untouched) — chunking is
  **one row per chunk with no parent/child linkage**, per the user's
  "simple" mapping-strategy choice.
- No change to `scripts/agent/memory/retriever.py` or `rrf.py` — retrieval
  continues to treat every row independently; a single long source event
  split into N chunks may surface as N independent hits in results. This is
  an accepted, documented limitation for this plan, not solved here.
- No change to `embedding_client.py`'s `_LOCAL_PREFIXES` (Option B chosen,
  not Option A).
- No graph-based memory representation, no new embedding-provider
  mechanism, no dedicated vector DB product, no semantic/episodic table
  split, no ADR/code-fix-intent/API-dependency/git-history ingestion — all
  explicitly deferred per the requirement's own Non-Goals, reconfirmed here.
- Reuse of `scripts/rag/ingestion/chunk_splitter.py` — investigated and
  rejected (see Design); memory gets its own minimal splitter instead.

## Design

1. **Chunking helper** (new, in `extract.py` or a small new
   `scripts/agent/memory/chunk.py` module — implementer's choice, not fixed
   by this plan): `def _split_content(content: str, max_chars: int) ->
   list[str]`. Simple, dependency-free strategy: split on paragraph
   boundaries (`\n\n`) first, greedily packing paragraphs into chunks up to
   `max_chars`; if a single paragraph itself exceeds `max_chars`, hard-cut
   it at the character boundary (same fallback the current code already
   uses, just applied per-chunk instead of once-and-discard). This directly
   reuses the existing truncation's fallback semantics rather than
   inventing new ones, and avoids depending on `chunk_splitter.py`'s
   Sudachi-tokenizer/RAG-coupled machinery (Assumption 3).
2. **`_try_extract_from_assistant`/`_try_extract_from_user`**: instead of
   truncating `content` once and building one `MemoryEntry`, call
   `_split_content(content, max_content_chars)` and build one `MemoryEntry`
   per returned chunk (reusing the existing `_make_entry`/
   `_make_entry_with_importance` builders at `extract.py:126,160` for each
   chunk) — all other fields (`session_id`, `turn_id`, `project`, `repo`,
   `branch`, `memory_type`, `importance`, timestamps) are identical across
   chunks from the same source message, only `content` (and derived
   `summary`, if any) differs per chunk. Return `list[MemoryEntry]` (or
   `list[MemoryEntry] | None` if the source message yields nothing) instead
   of `MemoryEntry | None`.
3. **`extract_memories`**: change the call sites that currently do
   `entry = _try_extract_from_assistant(...); if entry: entries.append(entry)`
   to `chunks = _try_extract_from_assistant(...); if chunks:
   entries.extend(chunks)` (and the analogous change for the user-message
   path).
4. **No other pipeline changes** (per Assumption 2 / the "simple" mapping
   decision): `ingestion.py`'s existing per-entry loop, `embedding_client.py`,
   `retriever.py`, and `rrf.py` are all reused unchanged. Each chunk becomes
   an ordinary, independently embedded/stored/retrieved `memories` row.
5. **Config**: flip the three flags/values listed in Affected areas. No new
   config keys are introduced; `memory_max_content_chars` is reinterpreted
   (documentation-only change) as "max chars per chunk" rather than "max
   chars total".
6. **Docs**: `12_03` gets a new subsection describing the chunking stage
   (what triggers it, that `memory_max_content_chars` is now a per-chunk
   limit, that no data is discarded); `12_04` gets a short note under
   retrieval/injection behavior stating that multiple chunks from the same
   long source may appear as separate hits in results — explicitly labeled
   as an accepted limitation, not a bug, per `rules/coding.md`'s
   "Current behavior" classification table (this is "Accepted current
   specification" once implemented, so plain prose, not a flagged-gap
   framing); `08_01` gets its default-value text updated for the three
   flipped config keys.

## Risks

- **Risk**: Splitting on `\n\n` paragraph boundaries may not suit all
  content shapes extracted from conversation turns (e.g. a single long
  code block or a long unbroken sentence) — could still frequently hit the
  hard-cut fallback. → **Mitigation**: the hard-cut fallback is exactly
  today's existing truncation behavior, just applied per-chunk instead of
  once — no regression versus current behavior, only strictly better
  (nothing is discarded). Fallback quality can be improved in a future
  iteration if needed; not a blocker for this plan's acceptance criteria.
- **Risk (accepted, per user decision)**: retrieval may surface multiple
  fragment-hits from the same long source event, since chunks have no
  parent/grouping metadata (retriever.py/rrf.py unchanged). →
  **Mitigation**: explicitly documented in `12_04` as a known, accepted
  limitation rather than silently shipped; the user has confirmed this
  tradeoff is acceptable for this scope. A future requirement could add
  `chunk_index`/`parent_id` schema support and retrieval-side grouping if
  this proves problematic in practice.
- **Risk**: Flipping `use_memory_layer`/`memory_embed_enabled` to `true` by
  default changes real runtime behavior for every existing deployment on
  upgrade (new HTTP calls to `embed_url`, new DB writes to `memories`,
  additional startup work in `_build_memory_services`) — not just a config
  file default, but a behavior change for anyone who upgrades without
  reading release notes. → **Mitigation**: this is exactly what the
  requirement mandates (explicit product decision, "given directly by the
  user" per the requirement's own Summary) — flagged here for visibility,
  not as an open question. The documentation updates (Design step 6) are
  the primary mitigation for operator surprise.
- **Risk**: `config_dataclasses.py` line numbers cited in the requirement
  (`:243`, `:249`) did not exactly match this investigation's read-out
  (`:208`, `:214`) — likely due to intervening comment lines or file drift
  since the requirement was authored. → **Mitigation**: Implementation
  step 3 must re-locate the exact current line numbers via `grep -n
  "use_memory_layer\|memory_embed_enabled" scripts/agent/config_dataclasses.py`
  immediately before editing, not rely on either cited line number.
