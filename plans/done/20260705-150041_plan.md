## Goal

Fix the concurrent-execution bug for same-resource-scope write tool calls in the DAG scheduler, and add audit metadata fields so scheduling decisions are fully traceable.

## Scope

**In**: Fix `tool_runner.py` so same-resource-scope write calls run sequentially (not gathered). Add `resource_scope`, `is_write`, `requires_serial`, `scheduling_decision` fields to `_SerializationEvent`. Add tests covering all serialization cases.

**Out**: Changing the public ToolSpec schema, rewriting the whole tool execution layer, documentation-only changes.

## Assumptions

1. `asyncio.gather()` over a group executes all tools in the group concurrently.
2. Same-resource-scope write tools that run concurrently can corrupt shared state.
3. `build_execution_groups()` currently returns same-scope write tools in one group; the inner `asyncio.gather` in tool_runner makes them run concurrently — this is the bug.
4. `requires_serial=True` tools already work correctly (each in its own single-tool batch).
5. `write_first` tools (no scope) running concurrently with each other is acceptable per spec — the only constraint is they run before reads.
6. `shell_run` is already handled by `SHELL_TOOLS` injection at tool_runner.py:69 (`requires_serial=True`).

## Design

**Root cause**: In `tool_runner.py` lines ~297-310:
```python
for concurrent_batch in metadata.concurrent_groups:
    batch_results = await asyncio.gather(
        asyncio.gather(*(execute_one_tool_call(ctx, tc, turn) for tc in group))
        for group in concurrent_batch
    )
```
Each group is executed with inner `asyncio.gather` — all tools in the group run concurrently. For resource-scope groups like `[write_file_a, write_file_b]`, this is a bug.

**Fix**: When a group contains more than one tool, run them sequentially if they are all write calls sharing the same resource_scope. The scheduler already knows this — `_SerializationEvent` with `reason="resource_scope_conflict"` identifies these groups.

Simplest fix: In `_GroupMetadata`, add a flag to mark resource-scope groups as "serialize_within". Then in `tool_runner.py`, for groups marked as `serialize_within`, replace inner `asyncio.gather` with a sequential for-loop.

Alternative: Add a `ScheduledGroup` dataclass with `calls: list[dict]` and `serialize: bool`.
