# Plan: [RAG][Cache] Make SemanticCache safe for concurrent use

## Goal

Add concurrency protection to `SemanticCache` so that `put()`, `lookup()`, and `prune()` are safe when called from multiple async tasks sharing a single cache instance.

## Scope

**In scope:**
- `scripts/rag/cache.py` — add asyncio.Lock for thread safety
- New file: `tests/test_semantic_cache_concurrency.py` (concurrent access tests)
- Read-only reference: `scripts/rag/pipeline.py` — where cache is shared (`self.semantic_cache`)

**Out of scope:**
- Changes to cache algorithm (cosine similarity, pruning strategy)
- Changes to CacheEntry dataclass or CacheService protocol
- Changes to external RAG service caching behavior

## Assumptions

1. The cache is created once per `RagPipeline` instance and shared across all `augment()` calls (pipeline.py:107-109).
2. Multiple async tasks may call `augment()` concurrently on the same pipeline instance (orchestrator supports concurrent tool calls).
3. Current implementation has no locking — `lookup()`, `put()`, and `prune()` all mutate or read `_entries` list without synchronization.
4. Python's asyncio provides `asyncio.Lock` for coroutine-level mutual exclusion within a single event loop.
5. The `CacheService` protocol must remain unchanged (backward compatibility with any custom implementations).

## Implementation Steps

1. **Add asyncio.Lock** to `SemanticCache.__init__()`:
   - `self._lock = asyncio.Lock()`
2. **Protect `lookup()`** — wrap entire method body in `async with self._lock:`.
3. **Protect `put()`** — wrap entire method body (including `prune()`) in `async with self._lock:`.
4. **Protect `prune()`** — no longer needed as standalone async method since `put()` holds the lock; add docstring noting it is internal-only when called from `put()`. If called directly, still protect.
5. **Update `CacheService` protocol** — mark `lookup()` and `put()` as `async def` (breaking change? check callers).
   - **Alternative (non-breaking):** keep methods sync but use `threading.Lock`; callers await nothing. This avoids protocol breakage but is less idiomatic for asyncio code.
   - **Recommended:** Keep methods synchronous; use `threading.RLock()` instead of `asyncio.Lock`. This is safer since `_entries` is a plain list and operations are fast (no I/O). No protocol changes needed.
6. **Create concurrency tests**:
   - `test_concurrent_lookups_no_crash` — 100 concurrent lookups, verify no exceptions
   - `test_concurrent_puts_no_data_loss` — 100 concurrent puts, verify all entries present after
   - `test_concurrent_put_and_lookup_consistent` — put and lookup interleaved, verify no corrupted state
   - `test_prune_under_concurrent_access` — concurrent prune calls during heavy put/lookup
7. **Validation:** run `uv run pytest tests/ -x -q` to confirm no regressions.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Adding a lock may introduce contention under high concurrency (many parallel augment calls) | low | Cache operations are fast (CPU-bound, no I/O); lock hold time is microseconds. Benchmark if needed. |
| Using threading.Lock in async code could block the event loop briefly | very low | Lock acquisition is O(1) for uncontended case; critical section is fast list iteration. Acceptable. |
| Protocol breaking change if methods made async | medium | Recommended approach: keep methods sync, use threading.RLock. No API breakage. |
