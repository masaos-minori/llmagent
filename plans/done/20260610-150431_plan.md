# Agent Runtime 改修計画

## Goal

`scripts/agent/` の 19 ファイルを fail-fast・型厳密・責務単一に再設計し、`except Exception` による曖昧な継続・`None`/`[]`/`False`/`"?"` によるエラー吸収・shared mutable state の過剰な集約を除去する。

---

## Scope

**In scope**

- `scripts/agent/` 以下の対象 19 ファイル（instruction 記載のファイルすべて）
- 各ファイルの fail-fast 化（broad exception 削除・fallback 戻り値削除）
- repository 層への `SQLiteHelper` 注入化
- `AgentContext` の状態分離
- `config.py` の validation パイプライン一方向化
- lifecycle 層（`http_lifecycle.py`, `stdio_lifecycle.py`, `lifecycle.py`, `lifecycle_protocol.py`）の状態遷移厳密化
- 異常系テスト追加（instruction の作業ステップ 7 に列挙された失敗シナリオ）

**Out of scope**

- `scripts/agent/commands/` 配下（直近 `d7be0d6` で改修済み）
- `scripts/agent/memory/` 配下（直近 `d23dc2a` で改修済み）
- `scripts/agent/services/` 配下（直近 `a41912e` で改修済み）
- `scripts/mcp/`, `scripts/rag/`, `scripts/shared/`, `scripts/db/` への変更
- 新機能追加・インターフェースの外部公開変更
- `deploy/deploy.sh` の変更（production ファイルへの追加なし）

---

## Assumptions

1. Python 3.13 + uv 環境は動作している（`uv run pytest` が実行可能）。
2. `SQLiteHelper("session")` は `db/helper.py` の既存 API であり、注入可能なシグネチャへの変更は `db/helper.py` に影響する。ただし `db/helper.py` のテスト（`test_db_helper.py`）は既存のため、契約変更は最小限にとどめる。
3. `AgentContext` を分割した場合でも、`build_agent_context()` の呼び出し側（`repl.py`）インターフェースは維持する。
4. `lifecycle_protocol.py` の Protocol は `http_lifecycle.py` と `stdio_lifecycle.py` が実装している唯一の契約であり、第三の実装は存在しない。
5. 異常系テストは `tests/test_agent_*.py` パターンで追加する。

## Implementation Steps

各ステップは独立してコミット可能。instruction の作業ステップ順に従う。

### Step 1: 型と失敗契約の固定

1. **`lifecycle.py`** — `LifecycleState` の到達可能状態と遷移規則を `dataclass` + `Enum` で明文化する。
2. **`lifecycle_protocol.py`** — Protocol メソッドの戻り型を `None` / `LifecycleState` に統一し、sync/async 混在を解消する。
3. **`config.py`** — `Any` 型フィールドを除去し、`__post_init__` を「ロード→正規化→field validation→cross-field validation」の一方向パイプラインに整理する。mutable runtime config と immutable startup config を分離する。
4. **`context.py`** — `AgentContext` を `ConversationState` / `TurnState` / `RuntimeStats` / `AppServices` の責務別 state container に再構成し、不変にできる部分を frozen dataclass にする。
5. **`orchestrator.py`** — `TurnResult` を厳密な結果型にし、`except Exception → TurnResult(success=False)` の変換をなくす。turn lifecycle を phase ごとに明示化する。

### Step 2: Repository 層の fail-fast 化

6. **`session_message_repo.py`** — broad exception を `sqlite3.Error` など個別例外に置き換え、`None`/`[]` fallback を削除する。`SQLiteHelper` を constructor 注入に変更する。
7. **`document_repo.py`** — 同上。`[]`/`False` fallback を削除する。
8. **`note_repo.py`** — 同上。`None`/`[]`/`False` fallback を削除する。
9. **`session.py`** — façade 化。broad exception を削除し、repository へ例外を委譲する。

### Step 3: Lifecycle 層の再設計

10. **`http_lifecycle.py`** — 起動・停止・restart・health poll を fail-fast 化。`StartupFailure` / `HttpStartupError` 例外体系を一本化する。
11. **`stdio_lifecycle.py`** — `TransportState` を遷移可能な状態機械として明示。broad exception を個別例外に置き換える。idle shutdown と on-demand 起動の責務を分離する。
12. **`lifecycle_protocol.py`** への適合確認（Step 2 の Protocol と http/stdio 実装の整合）。

### Step 4: History / Turn 実行の整理

13. **`history_selection_policy.py`** — 純粋関数化。typed model を返すように変更する。
14. **`history.py`** — 圧縮失敗を `None` で吸収しない。`CompressResult` を厳密な結果型にし、失敗を Orchestrator へ伝播する。
15. **`llm_turn_runner.py`** — error mapping を fail-fast にし、typed request/response model を導入する。

### Step 5: Factory / REPL の thin coordinator 化

16. **`factory.py`** — builder を fail-fast 化。`_ServerLifecycleRouter` を protocol/adapter ベースに置き換える。plugin registry・tracer 初期化の副作用を明示する。
17. **`repl_health.py`** — `except Exception` を除去。構造化診断結果モデルへ変更。restart 判定と実行を分離する。
18. **`repl.py`** — 初期化詳細を factory へ移譲。`print()` を view 経由へ統一。startup/shutdown フェーズを分解する。

### Step 6: View と Context の整理

19. **`cli_view.py`** — `Writer`/`Reader`/`CLIView` を Protocol 主導にし、出力抽象化を完成させる。
20. **`error_injection_service.py`** — policy を明示化し、テスト専用範囲を限定する。

### Step 7: 異常系テスト追加

各失敗シナリオの単体テストを追加する（`tests/test_agent_*.py`）:
- Config load error（ファイル不存在・型不正・値不正・cross-field 不整合）
- SQLite failure（接続失敗・書き込み失敗）
- HTTP subprocess start failure
- stdio transport restart failure
- health-check timeout
- history compression failure
- turn runner failure
- note / document / session repository failure

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| `session.py` façade 化により既存 command/service の呼び出し側が broken | high | Step 2 前に `rg "AgentSession\."` で全呼び出しを洗い出し、interface を固定してから変更する |
| `AgentContext` 再構成により factory.py / repl.py の wiring が広く壊れる | high | Step 1 (context.py) を先行し、Step 5 (factory/repl) で追従。各ステップで `pytest` を通してから次ステップへ進む |
| `config.py` の `Any` 削除が `mypy` エラーを大量に誘発する | medium | config.py は単独ブランチで作業し、mypy strict ではなく incremental に型を付ける |
| `SQLiteHelper` 注入化が `db/helper.py` の API 変更を要求し波及範囲が広がる | medium | `db/helper.py` を読んで provider pattern の追加コストを事前確認し、ラッパーで吸収できる場合はそちらを選択する |
| broad exception 削除により既存テストが通らなくなる | low | 各 step で `uv run pytest` を実行し、失敗したテストを即座に修正してから次へ進む |
| `lifecycle_protocol.py` Protocol の変更が HTTP/stdio 両実装に同時影響 | low | Step 3 で Protocol → HTTP → stdio の順に変更し、各変更後に pytest を実行する |
