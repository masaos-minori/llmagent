# Plan: Add `/diff` slash command to review files the agent wrote/edited this session

Source requirement: `requires/20260714_14_require.md`

## Goal

A user can run `/diff` in the agent REPL and see, for every file the agent
wrote or edited during the current session (via `write_file`/`edit_file`
tool calls still present in `ctx.conv.history`), that file's current
working-tree diff — grouped by file, with paths outside any git repository
(or denied by the git MCP server's allowlist) reported with a clear per-file
notice instead of aborting the whole command.

## Scope

**In scope**:
- Add `CommandDef("/diff", ...)` to `_COMMANDS` in
  `scripts/agent/commands/command_defs_list.py`.
- Implement `_cmd_diff` as an async handler in `scripts/agent/commands/cmd_context.py`'s
  `_ContextMixin` (alongside `_cmd_history`, per the requirement's own
  suggested placement — confirmed as the correct home: `_ContextMixin` is
  already part of `CommandRegistry`'s MRO, so no new mixin/inheritance wiring
  is needed).
- Scan `ctx.conv.history` for `write_file`/`edit_file` tool calls, extract
  distinct touched file paths, group them by containing git repository, call
  `git_diff` once per distinct repository (not once per file — see Design),
  and print each touched file's diff hunk (or a "no diff" / "not in a git
  repo" / "repo access denied" notice) grouped under that file's path.
- Update `docs/05_agent_07_10_cli-and-commands-slash-commands-workflow-debug.md`
  with a new `/diff` row/section matching its existing 3-column table format.
- Add tests mirroring `tests/test_agent_cmd_context.py`'s conventions.

**Out of scope** (per the requirement's own Non-Goals, confirmed still
correct after investigation):
- No new DB-backed change-tracking table (history-scan approach only; the
  `/compact`/`/clear` limitation is documented, not solved — see Design/Risks
  and the resolved Unknown below).
- No surfacing of workflow-approval diffs (`/approve`/`/reject` flow).
- No git write operations.
- No modification to the `git_diff` MCP tool's schema or
  `scripts/mcp_servers/git/*` — confirmed via investigation that `git_diff`
  has no path/pathspec filter parameter (only `repo_path`, `staged`,
  `commit` — verified in `scripts/mcp_servers/git/tools.py:58-83` and
  `format_output.py:64-72`, which always calls `repo.git.diff(...)` with no
  path argument). Extending that tool's schema to accept a path filter would
  be a cleaner long-term design but is a bigger, separately-scoped change
  affecting a "production"-status MCP tool; this plan works around the gap
  client-side instead (see Design).
- No change to `config/git_mcp_server.toml`'s `allowed_repo_paths` (currently
  `[]`, i.e. empty/fail-closed — see Risks). Populating it is an operator
  configuration action, not a code change.

## Assumptions

1. `write_file`/`edit_file` tool-call arguments use the JSON key `"path"`
   (verified: `scripts/mcp_servers/file/write_tools.py:18,42,80` — both
   schemas declare `"path"` as a required property).
2. The MCP tool name to invoke is exactly `"git_diff"` (verified:
   `scripts/mcp_servers/git/tools.py:59`).
3. `ctx.services_required.tools.execute(name, args)` is `async`, returns a
   `ToolCallResult` with `.output: str` / `.is_error: bool`, and does not
   raise for tool-level failures (verified via `cmd_mdq.py:80-87` usage
   pattern) — `/diff`'s handler must be registered with `is_async=True` in
   its `CommandDef`.
4. `ConversationState.history` entries are plain `LLMMessage` dicts;
   assistant messages carry `tool_calls: list[dict]` with
   `tool_calls[i]["function"]["name"]` / `["arguments"]` (a JSON string) —
   verified in `scripts/shared/types.py:12-38`. No dedicated file-path index
   exists; `/diff` must parse this structure directly.
5. `git_diff` returns the diff of the **entire working tree** for a given
   `repo_path` (no per-file filter) — verified by reading
   `scripts/mcp_servers/git/format_output.py:64-72`. This falsifies the
   requirement's literal description ("invoke `git_diff`... to obtain a
   working-tree diff" once "for each path") as written; Design section below
   adapts this to call `git_diff` once per distinct repository and split the
   result client-side.
6. `config/git_mcp_server.toml`'s `allowed_repo_paths = []` today (verified)
   means every `git_diff` call currently returns a `"[DENIED] repo_path ...
   is not in allowed_repo_paths"` string (from
   `scripts/mcp_servers/git/git_security.py`), not a raised exception —
   `/diff` will need to recognize and cleanly report this case too, distinct
   from "not a git repo". This is expected, working-as-designed fail-closed
   behavior, not a defect to fix in this plan (see Risks).

### Resolution: `/compact`/`/clear` retention question

Asked the user (this plan's authoring session). **Decision: the simpler
history-scan approach is sufficient — `/diff` is NOT required to show
changes from before a `/compact` or `/clear` in the same session.** No
durable, DB-backed tracking mechanism is required. This confirms the
requirement's Non-Goals stance and keeps `_cmd_diff`'s implementation
scoped to a pure `ctx.conv.history` scan. The known limitation must still be
documented (in both the command's own behavior — e.g. a one-line note when
history looks compacted — and in
`docs/05_agent_07_10_cli-and-commands-slash-commands-workflow-debug.md`).

## Design

1. **`CommandDef`**: `CommandDef("/diff", False, True, "_cmd_diff", "Show diffs for files written/edited this session")`
   — exact-match (no sub-args needed for v1), async.
2. **`_cmd_diff(self) -> None` (async)** in `cmd_context.py`:
   a. Walk `ctx.conv.history`; for each message with `role == "assistant"`
      and a non-empty `tool_calls` list, for each tool call where
      `function.name in ("write_file", "edit_file")`, `json.loads` the
      `function.arguments` string and collect `args["path"]`. Deduplicate
      into an ordered list of distinct touched paths, preserving
      first-seen order.
   b. If no paths were collected: `self._out.write("No files written or edited this session.")` and return (satisfies the "nothing to show" acceptance criterion).
   c. For each distinct touched path, resolve its containing git repository
      root (e.g. walk parent directories looking for a `.git` entry, using
      `pathlib.Path`, stdlib only — no new dependency). Group paths by
      resolved repo root; paths with no resolvable repo root go into an
      "outside any git repository" bucket.
   d. For each distinct repo root (not each file — see Assumption 5): call
      `await ctx.services_required.tools.execute("git_diff", {"repo_path": str(repo_root), "commit": ""})`
      once. If `result.is_error` or the output starts with `"[DENIED]"`
      (matching `git_security.py`'s literal denial-string prefix, verified),
      report once per repo: `f"[{repo_root}] git diff unavailable: {result.output}"`
      and move on — do not abort remaining repos/files.
   e. Otherwise, split the returned unified diff text on
      `^diff --git a/... b/...` headers (stdlib `re`, no new dependency) and
      keep only the hunks whose header path matches one of this repo's
      touched paths (relative-path match against the repo root). Print each
      touched file's matching hunk under its own path header; if a touched
      path has no matching hunk in the diff (e.g. already reverted, or
      untracked/gitignored), print `f"{path}: no working-tree diff"` instead
      of silently omitting it — this keeps every originally-touched path
      accounted for in the output, which the Acceptance Criteria imply
      ("lists each touched file path").
   f. For paths in the "outside any git repository" bucket: print
      `f"{path}: not inside a git repository (skipped)"` per path — satisfies
      the explicit per-file-notice acceptance criterion.
3. **Docs**: new row (or small subsection) in
   `docs/05_agent_07_10_cli-and-commands-slash-commands-workflow-debug.md`,
   explicitly noting: (a) only tool calls still present in
   `ctx.conv.history` are considered — a `/compact` or `/clear` earlier in
   the session hides prior activity from `/diff` (per the resolved Unknown
   above, this is accepted, not fixed); (b) `git_diff` requires the
   repository's path to be present in `config/git_mcp_server.toml`'s
   `allowed_repo_paths`, which is empty by default — operators must
   populate it for `/diff` to return real diffs rather than denial
   messages (per Assumption 6).

## Implementation steps

1. Add the `CommandDef("/diff", ...)` entry to `_COMMANDS` in
   `command_defs_list.py`.
2. Implement `_cmd_diff` in `cmd_context.py` per Design steps 2a-2f
   (history scan → path grouping by repo root → one `git_diff` call per
   repo → client-side diff-splitting/matching → per-path notices for
   denied/non-repo/no-diff cases).
3. Add tests in `tests/test_agent_cmd_context.py`: (a) no tool calls in
   history → "nothing to show" message; (b) one touched path inside a repo
   with a real diff returned → path header + matching hunk printed; (c) one
   touched path with no matching hunk in the returned diff → "no
   working-tree diff" message; (d) one touched path outside any git
   repository → "not inside a git repository (skipped)" message, and the
   command still completes (does not raise) when combined with case (b) in
   the same run; (e) `git_diff` returns a `[DENIED]`-prefixed string (or
   `result.is_error=True`) → per-repo denial message printed, command still
   completes.
4. Update `docs/05_agent_07_10_cli-and-commands-slash-commands-workflow-debug.md`
   per Design step 3.
5. Run the Validation Plan below.

## Risks

- **Risk**: `git_diff` has no path filter, so the client-side diff-splitting
  (Design step 2e) must correctly parse unified-diff `diff --git a/<path>
  b/<path>` headers, including edge cases (renamed files, files with spaces
  in paths, binary-file diff markers). — **Mitigation**: use a conservative
  regex anchored on the literal `diff --git a/` prefix per hunk boundary,
  and fall back to printing "no working-tree diff" (not a parse error) for
  any touched path whose header isn't found verbatim; add a dedicated test
  for a path containing a space, since `unified_diff`/`git diff` path
  quoting is a known sharp edge.
- **Risk**: `config/git_mcp_server.toml`'s `allowed_repo_paths = []` today
  means `/diff` will report "access denied" for every real repo out of the
  box until an operator configures it — this could read as `/diff` being
  broken rather than working-as-designed fail-closed security.
  — **Mitigation**: the denial message (Design step 2d) must be
  distinguishable from a genuine error, and the new docs entry (Design step
  3b) explicitly calls out the `allowed_repo_paths` prerequisite so
  operators aren't left guessing.
- **Risk**: Determining a file's containing git repo root by walking parent
  directories for `.git` (Design step 2c) duplicates logic already present
  server-side in `scripts/mcp_servers/git/service.py`'s `_open_repo`
  (`git.InvalidGitRepositoryError` detection) — a client-side
  reimplementation could disagree with the server's own detection in edge
  cases (e.g. `.git` as a file for worktrees/submodules, which GitPython
  handles but a naive walk-up might not). — **Mitigation**: use GitPython's
  own `git.Repo(path, search_parent_directories=True)` (already a project
  dependency per `scripts/mcp_servers/git/service.py`'s use of the `git`
  module) inside `_cmd_diff` for repo-root detection, catching
  `git.InvalidGitRepositoryError` for the "outside any git repository"
  bucket, rather than a hand-rolled directory walk — this reuses the exact
  same detection semantics the server already relies on.
- **Risk (accepted, not mitigated further — see resolved Unknown)**: `/diff`
  will not show changes made before a `/compact` or `/clear` earlier in the
  same session. The user has explicitly confirmed this is acceptable for
  v1; documented in the new docs entry so it's not a silent gap.
