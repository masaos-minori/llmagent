## Assumptions

1. `scripts/rag/cache.py::SemanticCache.prune()` (lines 92-96) does `self._entries[-self._max_size:]` â€” confirmed by direct read that when `max_size=0`, Python evaluates `-0 == 0`, and `list[0:]` returns the **entire** list, not an empty one. The cache silently grows unbounded when `max_size=0`, exactly as the requirement describes.
2. Resolving "Decide whether `semantic_cache_max_size=0` means disabled or invalid": **`0` is valid and means "capacity zero" (the cache holds no entries; `put()` remains a normal call but every entry is immediately pruned away)** â€” not rejected as invalid. Reasoning: `RagPipeline`/`RagConfig` already has a dedicated, separate `use_semantic_cache: bool` flag (confirmed by direct read of `scripts/rag/pipeline.py:410` and `scripts/mcp_servers/rag_pipeline/models.py:59`) that is the actual "cache on/off" switch; treating `max_size=0` as *also* meaning "disabled" (rather than *rejected*) gives a second, independent, defense-in-depth way to guarantee zero cache growth even if `use_semantic_cache` is mistakenly left `True` â€” without inventing a second orthogonal validation error for a value (`0`) that has a perfectly sensible, boundary-case meaning as a capacity.
3. Negative `semantic_cache_max_size` has no sensible interpretation (Python's negative-index slicing would keep everything-except-the-first-N entries, growing unbounded just like the `0` bug) â€” resolved: **reject negative values in `RagConfigValidator`** (an `errors` entry, not just a warning, since the value is unusable, not just suboptimal), and defensively clamp in `prune()` itself too (`max(self._max_size, 0)`), matching this codebase's established pattern (seen repeatedly this session) of guarding at both the validator and the low-level implementation.
4. `scripts/shared/config_validator.py::RagConfigValidator.validate()` (lines 19-43) does `rag = cfg.get("rag", {})` unconditionally â€” confirmed by direct read. `scripts/rag/pipeline.py:120-122` confirms the **actual runtime call**: `result = validator.validate(_resolved_cfg)` where `_resolved_cfg` is either `module_cfg` (a flat dict, when the MCP service passes one) or `_ModuleConfig.get()`'s result (nested, since it comes from `ConfigLoader().load_all()` â€” a full agent.toml load with top-level `[rag]`/`[mcp]`/etc. sections, confirmed by direct read). **When called with the MCP path's flat `module_cfg`, `cfg.get("rag", {})` always returns `{}`, so every single validation check silently no-ops today** â€” this is the exact, verified bug behind "MCP module config path receives meaningful validation."
5. `scripts/mcp_servers/rag_pipeline/service.py::start()`'s `module_cfg` dict (lines 75-85, confirmed by direct read) currently includes `llm_url`/`embed_url`/`rag_db_path`/`sqlite_vec_so`/`sqlite_timeout`/`sqlite_busy_timeout_ms`/`mqe_*`/`rerank_prompt_template` â€” but **not** `use_rrf`, `semantic_cache_max_size`, or `semantic_cache_threshold`, even though `RagPipelineConfig` (the `cfg` object these values are pulled from) has all three as real fields with real values (confirmed by direct read of `scripts/mcp_servers/rag_pipeline/models.py:48,57-58`). Fixing `RagConfigValidator`'s shape-normalization alone is **necessary but not sufficient** â€” `module_cfg` itself must also carry these 3 keys for validation of the MCP path to be meaningful, not just correctly-shaped-but-empty. This file is added to this plan's Affected Areas even though the requirement's Scope list doesn't explicitly name it, because the gap was directly verified during planning and leaving it out would make the plan's own stated Acceptance Criterion factually false once implemented.
6. `rag/models_data.py::CacheEntry.generation` (line 64) is set at creation (`generation=self._generation` in `put()`) but confirmed by `grep` to never be read/compared anywhere else in the codebase. `SemanticCache.invalidate()` (`cache.py:102-106`) achieves staleness-freedom by physically `.clear()`-ing `_entries`, not by filtering on generation at lookup time. Resolving the requirement's Task: **`CacheEntry.generation` is observability-only today** (useful for logging/debugging which cache epoch an entry belongs to) â€” it plays no role in stale-entry filtering, since eager physical clearing already makes that unnecessary. No code change follows from this; it is a documentation clarification.
7. `grep -rln "test_rag_cache" docs/` at planning time found **no** existing reference to `tests/test_rag_cache*.py` anywhere in the current doc set â€” the requirement's Task ("remove references to non-existent `tests/test_rag_cache*.py`") is not currently applicable; either it was already cleaned up by a prior/concurrent change, or the stale reference never existed in the docs actually shipped. This plan creates the real file (`tests/test_rag_cache.py`), which makes the concern moot regardless. Re-run the same grep during implementation (Phase 5) in case a reference reappears from other in-flight work.
8. `SemanticCache.size`/`generation` (properties at `cache.py:98-100`, `108-110`) currently read `self._entries`/`self._generation` **without** acquiring `self._lock` â€” confirmed by direct read â€” while every mutating method (`lookup`, `put`, `prune`, `invalidate`) already does acquire it. This is a genuine, if narrow, thread-safety gap (a concurrent read during a `put()`/`invalidate()` could observe a torn state) that the requirement's Tasks explicitly call out ("Make `SemanticCache.size` read under lock," "Make `SemanticCache.generation` read under lock").

## Design

### `SemanticCache.prune()` fix (scripts/rag/cache.py)

```python
def prune(self) -> None:
    """Remove oldest entries (FIFO) until len(self._entries) <= max_size.

    max_size <= 0 means the cache holds zero entries (effectively disabled
    via capacity, independent of the separate use_semantic_cache toggle).
    """
    with self._lock:
        if self._max_size <= 0:
            self._entries = []
        elif len(self._entries) > self._max_size:
            self._entries = self._entries[-self._max_size :]
```

### `size`/`generation` under lock

```python
@property
def size(self) -> int:
    with self._lock:
        return len(self._entries)

@property
def generation(self) -> int:
    with self._lock:
        return self._generation
```

### `RagConfigValidator` shape normalization + max_size check (scripts/shared/config_validator.py)

```python
class RagConfigValidator:
    def validate(self, cfg: dict[str, Any]) -> ConfigValidationResult:
        errors: list[str] = []
        warnings: list[str] = []

        # Normalize: nested {"rag": {...}} and flat {...} both supported.
        # A flat dict has no "rag" key by construction (MCP module_cfg never
        # nests); a nested dict is agent.toml's ConfigLoader().load_all() shape.
        rag = cfg["rag"] if "rag" in cfg else cfg

        embed_dim = rag.get("embedding_dim")
        vec_dim = rag.get("vec_dim")
        if embed_dim and vec_dim and embed_dim != vec_dim:
            errors.append(f"embedding_dim={embed_dim} != vec_dim={vec_dim}")

        if not rag.get("use_rrf", True):
            warnings.append(
                "use_rrf=false degrades retrieval quality; use only for diagnostics"
            )

        threshold = rag.get("semantic_cache_threshold", 0.92)
        if threshold < 0.5:
            warnings.append(f"semantic_cache_threshold={threshold} is unusually low")

        max_size = rag.get("semantic_cache_max_size", 100)
        if max_size < 0:
            errors.append(
                f"semantic_cache_max_size={max_size} is negative; must be >= 0"
            )

        return ConfigValidationResult(errors=errors, warnings=warnings)
```

(`max_size == 0` deliberately produces neither an error nor a warning, per Assumption 2 â€” it is a valid, if degenerate, configuration.)

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Any existing deployment currently relying on the buggy `max_size=0` behavior (unbounded cache growth) would see its cache start actually pruning to empty after this fix, changing observed query latency/behavior | Low | This is the entire point of the fix â€” the current behavior is an unintended bug, not a documented feature; the new, explicit `max_size=0` semantics (Assumption 2) is announced in documentation (Phase 5) so this is a visible, deliberate behavior change, not a silent one |
| Adding `semantic_cache_max_size`/`use_rrf`/`semantic_cache_threshold` to `module_cfg` could shadow or conflict with how `_resolved_cfg.get(...)` is used elsewhere in `pipeline.py` if any of those 3 keys are read there with different expected semantics | Low | Confirmed by `grep -n "_resolved_cfg.get"` in `pipeline.py` that none of the 3 new keys are currently read from `_resolved_cfg` anywhere (only `llm_url`/`embed_url`/`rag_db_path`/`sqlite_*` are) â€” these 3 keys are added purely for `RagConfigValidator` to see, with no other consumer to conflict with |
