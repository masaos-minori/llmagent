## Design decisions

- Handle `sqlite3.DatabaseError` as NON_FATAL warning in non-production mode, FATAL in production mode — database corruption should halt startup in production but allow graceful degradation in development.
- Handle `ConnectionRefusedError` / `OSError` (network errors) as NON_FATAL warning — RAG is optional and should not block startup.
- Handle `ValueError` (invalid data returned from DB) as NON_FATAL warning — indicates corrupted data but not necessarily a fatal condition.
- Handle `RuntimeError` (schema missing/incompatible) as FATAL — schema incompatibility means RAG will produce incorrect results.
- Keep a catch-all `Exception` handler at WARNING level for truly unexpected errors — prevents startup failure from obscure edge cases.

## Alternatives considered

- Make ALL RAG errors FATAL: too aggressive — RAG is optional and operators should be able to start the agent even if RAG fails.
- Only log the error without changing the pipeline status: loses visibility — operators need to know RAG is unavailable.
- Add a configuration option to control RAG error severity: adds complexity for a relatively simple problem.

  except ValueError as exc:
      pipeline.add_warning("rag_consistency", f"[RAG] Invalid data: {exc}")
  
  except RuntimeError as exc:
      pipeline.add_fatal("rag_consistency", f"[RAG] Schema incompatibility: {exc}")
  
  except Exception as exc:
      pipeline.add_warning("rag_consistency", f"[RAG] Unexpected error: {exc}")
