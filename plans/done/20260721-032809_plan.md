# Plan: Fix `enabled_for_llm` never being set True for discovered tools

Source requirement: `requires/20260721_03_require.md` (moved to `requires/done/` at Step 7)

## Goal

After a successful MCP tool discovery cycle, every discovered tool that is not
explicitly reported as unavailable by its owning server (`enabled: false` in
its `/v1/tools` entry) is visible to the LLM's tool-calling payload; a tool a
server explicitly disables stays hidden; and an end-to-end test exercises the
real `build_runtime_tool()` call site inside `McpToolDiscoveryService` so this
class of bug cannot regress silently.

## Scope

**In scope:**
- `scripts/agent/services/mcp_tool_discovery.py` — `_dedupe_and_build()`'s
  `build_runtime_tool(...)` call: add `enabled_for_llm=entry.get("enabled", True)`.
- `tests/agent/services/test_mcp_tool_discovery.py` — add coverage asserting
  `enabled_for_llm` is derived correctly from discovery entries.
- A new or extended integration test exercising
  discovery → `RuntimeToolRegistry` → `_filter_disabled_tool_definitions()`
  end-to-end (real `build_runtime_tool()` call site, not a hand-built
  `RuntimeTool` fixture). Likely home: `tests/test_runtime_tool_routing_integration.py`
  (already exercises this chain with hand-built tools) or a new test module.
- Optional: `scripts/agent/commands/cmd_mcp.py` diagnostics — decide whether to
  surface `disabled_reason` alongside the existing `enabled_for_llm` column
  (see Design §2).

**Out of scope:**
- `ToolRouteResolver` / `RuntimeToolRegistry.resolve()` — routing is independent
  of `enabled_for_llm` and must not change.
- `scripts/shared/runtime_tool.py::build_runtime_tool()`'s default-to-`False`
  behavior for *omitted* `enabled_for_llm` — that default is intentional and
  correct for hand-constructed/test callers that don't pass it; the bug is that
  the discovery call site never passes it at all, not that the default itself
  is wrong.
- `RuntimeToolRegistry.apply_policy()` — confirmed (Step 3) it only ANDs
  `enabled_for_llm` down (`enabled and tool.enabled_for_llm`), never sets it
  True; it is a config-reload-time narrowing mechanism, not a source of truth,
  and needs no change here.
- The other two sibling requirements (`requires/20260721_01_require.md`,
  `requires/20260721_02_require.md`) — already processed in separate cycles.

## Assumptions

1. `entry.get("enabled", True)` — defaulting to `True` when a server's
   discovery entry omits the `enabled` key entirely — is the correct semantic,
   because a server that hasn't adopted requirement 15's `enabled`/
   `disabled_reason` schema has no way to signal unavailability and should not
   have all of its tools silently hidden as a side effect of not implementing
   an unrelated schema. (Verified in Step 4 below — no counter-evidence found.)
2. `entry["enabled"]`, when present, is always a JSON boolean (`true`/`false`),
   consistent with the 4 servers that emit it today
   (`file/read`, `file/write`, `file/delete`, `git`); `_validate_and_normalize_entry()`
   does not currently type-check `enabled`/`disabled_reason`, so a malformed
   value (e.g. a string) would pass through as truthy/falsy via Python's own
   truthiness — this is treated as a pre-existing minor schema-validation gap,
   not part of this fix's scope, but is worth a one-line defensive read
   (`bool(entry.get("enabled", True))`) to avoid a non-bool value poisoning
   `RuntimeTool.enabled_for_llm`'s declared `bool` type.
3. The fix belongs on the producer side (`mcp_tool_discovery.py`), not the
   consumer side (`llm_turn_runner.py::_filter_disabled_tool_definitions()`) —
   the requirement's own "Target files" section lists the consumer only as a
   "possibly" alternative, and Step 3 confirms `_filter_disabled_tool_definitions()`'s
   logic is otherwise correct: it already does the right thing (hide names in
   `known_names - visible_names`) — the input data (`visible_names`, sourced
   from `llm_tool_definitions()`) is what is wrong, not the filter logic.
4. No other code path sets `enabled_for_llm=True` for a discovered tool between
   `build_runtime_tool()` construction and the LLM call (`apply_policy()` only
   narrows, per Assumption/Out-of-scope above) — confirmed by grep: the only
   `apply_policy()` call site is `scripts/agent/services/config_reload.py:187`,
   invoked on `/reload`, not at startup discovery time.

Resolved during Step 4 (static analysis, no unknowns left unresolved from the
requirement's own flagged item):

- **Requirement's own flagged unknown** ("is `entry.get('enabled', True)` the
  right default?"): **Resolved — yes.** Verified all 6 servers that do not
  implement the `enabled`/`disabled_reason` schema
  (`shell`, `cicd`, `rag_pipeline`, `mdq`, `github`, `web_search`) omit the
  `enabled` key entirely from their `/v1/tools` tool dicts — none include a

## Design

### 1. Producer-side fix (the core fix)

In `scripts/agent/services/mcp_tool_discovery.py::_dedupe_and_build()`, change:

```python
built[name] = build_runtime_tool(
    name=name,
    server_key=server_key,
    server_url=server_url,
    description=str(entry.get("description", "")),
    input_schema=entry.get("inputSchema", entry.get("input_schema")),  # type: ignore[arg-type]
    raw_definition=entry,
    status=str(entry.get("status", "active")),
    is_write=entry.get("is_write"),  # type: ignore[arg-type]
    requires_serial=entry.get("requires_serial"),  # type: ignore[arg-type]
    capabilities=tuple(entry.get("capabilities", []) or []),  # type: ignore[arg-type]
)
```

to add one keyword argument:

```python
    enabled_for_llm=bool(entry.get("enabled", True)),
```

Rationale for `bool(...)` wrapper: defends against a non-bool `enabled` value
passing through unchanged (Assumption 2 / UNK-02); costs nothing since all
current servers emit real booleans or omit the key.

This directly implements requirement's "Required changes #1" verbatim, plus
the defensive `bool()` cast as a small improvement.

### 2. `disabled_reason` diagnostics (UNK-01, recommended)

`RuntimeToolRegistry.diagnostics()` (line 173-193) already returns a
`disabled_reason` field per row, but it is hardcoded from `tool.status`
(config-dependent status string), not the discovery-time `disabled_reason`
from the server's `/v1/tools` entry. Since `RuntimeTool` has no dedicated
`disabled_reason` field (only `raw_definition`, which holds the original
entry dict), the cleanest low-risk approach is:

- Read `disabled_reason` display value from `tool.raw_definition.get("disabled_reason", "")`
  inside `diagnostics()`, falling back to the existing `status`-derived value
  when absent (preserves current behavior for tools that never had a
  `disabled_reason` key, e.g. all non-4-server tools).
- Add a `disabled_reason` column to `cmd_mcp.py`'s `/mcp status` table output
  (it already prints `row.get("disabled_reason", "-")` internally per Step 3's
  read at line 93 — confirm the table's header row includes this column; if
  not, add it).

This is presented as a design option, not mandatory for the core fix's
correctness — call out to implementer to confirm scope with reviewer before
implementing, since it touches a second file's display logic beyond the
minimal 1-line fix.

### 3. Test design

**Unit-level** (`tests/agent/services/test_mcp_tool_discovery.py`):
- Extend/add a fixture returning a fake `/v1/tools` response with two tools:
  one with no `enabled` key, one with `"enabled": False, "disabled_reason": "..."`.
  Run `McpToolDiscoveryService.discover_all()` against a mocked HTTP client
  (existing test infra in this file already mocks `ctx.services_required.http`,
  per file naming — confirm exact mock pattern when implementing).
- Assert: tool #1's `RuntimeTool.enabled_for_llm is True`; tool #2's
  `RuntimeTool.enabled_for_llm is False`.

**Integration-level** (new or extended in `tests/test_runtime_tool_routing_integration.py`):
- Run the same `discover_all()` → registry flow, then call
  `RuntimeToolRegistry.llm_tool_definitions()` and confirm tool #1's name is
  present, tool #2's is absent.
- Optionally construct an `LlmTurnRunner`-equivalent minimal harness (or import
  `_filter_disabled_tool_definitions` logic directly if the test file already
  has a pattern for this — check at implementation time) and assert a
  config-defined tool definition matching tool #1's name survives filtering
  while one matching tool #2's name does not.
- This test must call the real `build_runtime_tool()` via
  `McpToolDiscoveryService`, not construct `RuntimeTool` by hand — this is the
  requirement's explicit acceptance criterion and the reason the bug went
  undetected by all ~15 existing `enabled_for_llm=True` call sites.

## Risks

- **Risk**: The fix flips a large number of tools from invisible to visible in
  one change; some tool the LLM now sees might not actually be safe to expose
  in every config (e.g. a tool intended to stay hidden pending a future
  feature flag, if any such case exists beyond the `enabled`/`disabled_reason`
  schema). Likelihood: low. **Mitigation**: `enabled_for_llm` only ever gates
  LLM visibility, not routing/execution (confirmed Out-of-scope note) — a tool
  becoming visible does not grant it new execution permissions;
  `requires_approval`/`agent_safety_tier`/`allowed_tools` (via
  `apply_policy()` on `/reload`) remain the actual authorization gates. Any
  tool needing to stay hidden from the LLM specifically (not just
  execution-gated) should have its owning server report `enabled: false` via
  the existing schema — this is the intended mechanism per requirement 15.

- **Risk**: The 6 servers without the `enabled`/`disabled_reason` schema could
  add it later with a bug that emits a bad `enabled` value (non-bool, or
  inverted polarity), silently re-hiding tools with no diagnostic trail —
  echoing the original failure mode. Likelihood: low-medium (future work, not
  this change). **Mitigation**: Step 4 (UNK-02) defensive type-check in
  `_validate_and_normalize_entry()` catches non-bool values as a WARNING
  finding at discovery time, giving the exact diagnostic trail this
  requirement's "Reason for change" section says was originally missing.
  Recommend including this in the implementation (Implementation step 4).

- **Risk**: The new end-to-end integration test could pass "by accident" (e.g.
  mocking too much of the chain, not actually exercising
  `build_runtime_tool()`), reintroducing the exact blind spot this requirement
  is trying to close. Likelihood: medium (test-design risk, not code risk).
  **Mitigation**: explicitly verify the new test fails against the pre-fix
  code (temporarily revert the 1-line change, confirm the new test fails, then
  reapply) before considering the test authoritative — call this out as a
  mandatory sub-step of Implementation step 3/6, not optional.

- **Risk**: Diff-scoped coverage (≥90%) could be hard to hit if the 1-line
  producer fix is bundled with the optional UNK-01/UNK-02 changes without
  matching test additions for those paths too. Likelihood: low.
  **Mitigation**: if UNK-01/UNK-02 are implemented, add matching test
  assertions for each (a non-bool `enabled` value case for UNK-02; a
  `disabled_reason` display assertion for UNK-01) in the same commit/PR as
  those changes, not deferred.

No risk in this plan was identified as unmitigated after the above — no
`issues/{timestamp}_risks.md` file is required for this plan.
