# Work Plan: Fix side-effect scheduling and ToolSpec metadata generation

Source requirement: `requires/20260715_18_require.md`

## Goal

Make the tool scheduler safe by default: (1) `write_first` groups execute their
tools serially instead of concurrently, and (2) `_build_tool_meta()` classifies
Git/GitHub/CI/CD/RAG/MDQ side-effecting tools as write-capable via shared
classification, so they are never silently placed in the default parallel
read-only group even when `tool_definitions` lacks explicit metadata.

## Scope

**In scope:**
- `scripts/agent/tool_scheduler.py` — `build_execution_groups()`: change the
  `write_first` `ScheduledBatch` from `serialize_flags=[False]` to
  `serialize_flags=[True]`.
- `scripts/agent/tool_runner.py` — `_build_tool_meta()`: broaden `is_write` /
  `requires_serial` classification beyond `WRITE_TOOLS` / `DELETE_TOOLS` /
  `SHELL_TOOLS` to cover Git write, GitHub write/dangerous, CI/CD
  `trigger_workflow`, RAG `rag_delete_document`, and MDQ
  `index_paths` / `refresh_index` / `fts_rebuild`, while still respecting
  explicit `resource_scope` / `requires_serial` / `is_write` keys when present
  in a tool definition's `function` dict.
- `scripts/shared/tool_constants.py` — add granular write/admin subsets for
  CI/CD and MDQ (existing `CICD_TOOLS` / `MDQ_TOOLS` mix read and write tools
  and cannot be used directly for side-effect classification). `RAG_TOOLS`
  needs a similar split, or a single `rag_delete_document` reference is used
  directly since it is the only RAG write tool today.
- `scripts/shared/tool_executor_helpers.py` — extend `_SIDE_EFFECT_TOOLS` (the
  set behind `is_side_effect()`) with the same new write/admin subsets, so
  `_build_tool_meta()` can consume `is_side_effect()` as specified in
  "Implementation Notes" of the requirement, instead of duplicating tool-name
  sets in `tool_runner.py`.
- `tests/test_tool_scheduler.py` — update the one existing assertion that
  encodes the old `write_first` `serialize_flags=[False]` behavior
  (`TestConcurrentGroups.test_write_first_gets_own_sequential_batch`, which
  currently only checks batch count/grouping, not the flag value — needs a new
  assertion added) and add new regression tests.
- `tests/test_tool_runner.py` and/or `tests/test_tool_executor_helpers.py` —
  add regression tests for the new classification.

**Out of scope:**
- No change to `resource_scope`-based serialization (already correct).
- No change to the `requires_serial` barrier mechanism itself.
- No new MCP servers, no config schema changes.
- Not disabling parallel execution for genuinely read-only tools.
- Not requiring every `tool_definitions` entry to declare metadata manually.

## Assumptions

1. "Serialize `write_first` groups" means setting
   `ScheduledBatch(groups=[write_first], serialize_flags=[True])` in
   `build_execution_groups()` (the primary acceptable design named in the
   requirement), not the alternative "split into single-tool batches" design —
   this is the smaller, more localized change and keeps `write_first` as one
   scheduled batch. *(Falsifiable: verified against `_run_group_calls()`,
   which executes a group sequentially when `serialize=True`.)*
2. `_build_tool_meta()` is the sole production caller that must reflect the
   new classification; no other code path builds `ToolSpec` from
   `tool_definitions`. *(Verified: `ToolSpec(` is only constructed in
   `tool_runner.py` and in `tests/test_tool_scheduler.py` / `test_tool_runner.py`
   test helpers.)*
3. `trigger_workflow`, `rag_delete_document`, `index_paths`, `refresh_index`,
   and `fts_rebuild` do not currently appear in `WRITE_TOOLS`, `DELETE_TOOLS`,
   `GIT_WRITE_TOOLS`, `GITHUB_WRITE_TOOLS`, or `GITHUB_DANGEROUS_TOOLS` in
   `scripts/shared/tool_constants.py`, so they fall through to `parallel` under
   current `_build_tool_meta()` logic. *(Verified by reading
   `tool_constants.py` in full — confirmed absent from all write-classified
   sets.)*
4. Adding tools to `_SIDE_EFFECT_TOOLS` in `tool_executor_helpers.py` is safe:
   its only consumer is `_execute_standard()` in `tool_runner.py`, which uses
   it purely to decide whether a *non-DAG* round should fall back to serial
   execution — expanding it only makes that fallback path more conservative,
   never less. *(Verified: single call site at `tool_runner.py:360`.)*
5. `fts_rebuild` should be classified as `requires_serial=True` (serial
   barrier) rather than merely `is_write=True`, because it operates on a
   shared FTS index and the requirement's acceptance criteria phrase it as
   "write or serial" — treating it as a serial barrier is the safer choice and
   is explicitly allowed ("MDQ write/admin tools ... allow it to become a
   serial barrier").
6. `index_paths` and `refresh_index` are classified as `is_write=True` (write
   group), not `requires_serial=True`, since the requirement lists them
   alongside `fts_rebuild` under "MDQ write/admin tools" without singling them
   out for the serial-barrier treatment reserved for "admin-like or
   operationally dangerous" tools.

## Design

### 1. `scripts/agent/tool_scheduler.py`

In `build_execution_groups()`, change:

```python
if write_first:
    cgr.append(ScheduledBatch(groups=[write_first], serialize_flags=[False]))
```

to:

```python
if write_first:
    cgr.append(ScheduledBatch(groups=[write_first], serialize_flags=[True]))
```

This is the only line that changes in this file. It makes `_run_group_calls()`
execute all `write_first` tools sequentially (per its `if serialize:` branch)
instead of via `asyncio.gather`.

### 2. `scripts/shared/tool_constants.py`

Add three new frozensets, following the existing `GIT_READ_TOOLS` /
`GIT_WRITE_TOOLS` / `GIT_TOOLS` pattern, so side-effect classification never
needs to duplicate raw tool-name literals in `tool_executor_helpers.py` or
`tool_runner.py`:

```python
CICD_WRITE_TOOLS: frozenset[str] = frozenset({"trigger_workflow"})
CICD_READ_TOOLS: frozenset[str] = CICD_TOOLS - CICD_WRITE_TOOLS

RAG_WRITE_TOOLS: frozenset[str] = frozenset({"rag_delete_document"})
RAG_READ_TOOLS: frozenset[str] = RAG_TOOLS - RAG_WRITE_TOOLS

MDQ_WRITE_TOOLS: frozenset[str] = frozenset({"index_paths", "refresh_index"})
MDQ_SERIAL_TOOLS: frozenset[str] = frozenset({"fts_rebuild"})
MDQ_READ_TOOLS: frozenset[str] = MDQ_TOOLS - MDQ_WRITE_TOOLS - MDQ_SERIAL_TOOLS
```

Keep `CICD_TOOLS`, `RAG_TOOLS`, `MDQ_TOOLS` unchanged (still the union, still
used by `get_all_mcp_tool_names()`).

### 3. `scripts/shared/tool_executor_helpers.py`

Extend `_SIDE_EFFECT_TOOLS`:

```python
from shared.tool_constants import (
    CICD_WRITE_TOOLS,
    DELETE_TOOLS,
    GIT_WRITE_TOOLS,
    GITHUB_DANGEROUS_TOOLS,
    GITHUB_WRITE_TOOLS,
    MDQ_SERIAL_TOOLS,
    MDQ_WRITE_TOOLS,
    RAG_WRITE_TOOLS,
    WRITE_TOOLS,
)

_SIDE_EFFECT_TOOLS: frozenset[str] = (
    WRITE_TOOLS
    | DELETE_TOOLS
    | frozenset({"shell_run"})
    | GIT_WRITE_TOOLS
    | GITHUB_WRITE_TOOLS
    | GITHUB_DANGEROUS_TOOLS
    | CICD_WRITE_TOOLS
    | RAG_WRITE_TOOLS
    | MDQ_WRITE_TOOLS
    | MDQ_SERIAL_TOOLS
)
```

Update the module docstring/comment above `_SIDE_EFFECT_TOOLS` to mention the
newly covered families (CI/CD, RAG, MDQ).

### 4. `scripts/agent/tool_runner.py` — `_build_tool_meta()`

Replace the narrow `_is_write` computation with one based on the extended
`is_side_effect()`, while adding a small explicit serial set for MDQ, and
preserving explicit per-definition overrides (`fn.get(...)` calls already do
this — no change needed there, just to the defaults feeding them):

```python
from shared.tool_constants import MDQ_SERIAL_TOOLS, SHELL_TOOLS
from shared.tool_executor_helpers import is_side_effect, tool_hash_key

def _build_tool_meta(
    tool_definitions: list[dict],
) -> dict[str, ToolSpec]:
    """Build tool metadata map from tool_definitions for DAG execution."""
    tool_meta: dict[str, ToolSpec] = {}
    for td in tool_definitions:
        fn = td.get("function", {})
        name = fn.get("name", "")
        if name:
            _is_write = is_side_effect(name)
            _requires_serial = (
                fn.get("requires_serial", False)
                or name in SHELL_TOOLS
                or name in MDQ_SERIAL_TOOLS
            )
            _default_scope = name if _is_write else ""
            tool_meta[name] = ToolSpec(
                call_id="",
                name=name,
                resource_scope=fn.get("resource_scope", _default_scope),
                requires_serial=_requires_serial,
                is_write=fn.get("is_write", _is_write),
            )
    return tool_meta
```

Notes on this design:
- `is_side_effect()` now already includes `WRITE_TOOLS | DELETE_TOOLS`, so the
  old `name in WRITE_TOOLS or name in DELETE_TOOLS` check is fully subsumed —
  no double logic.
- `fn.get("is_write", _is_write)` preserves explicit `is_write` from
  `tool_definitions` per requirement item 3 ("Preserve explicit metadata where
  safe"), while still defaulting unclassified-but-known-side-effecting tools
  to `True`.
- `fn.get("resource_scope", _default_scope)` is unchanged in shape but now
  `_default_scope` is populated for the newly-classified tool families too
  (e.g. `trigger_workflow` → default scope `"trigger_workflow"`), so
  `trigger_workflow` calls land in a resource-scope group (serialized among
  themselves) rather than the generic `write_first` group — this still
  satisfies "does not fall into the default read-only parallel group".
- `MDQ_SERIAL_TOOLS` (`fts_rebuild`) is folded into `requires_serial` directly
  in `_build_tool_meta()` rather than through `is_side_effect()`, matching how
  `SHELL_TOOLS` is already handled (a name-set check for a barrier condition
  distinct from write-classification).
- `SHELL_TOOLS` import is currently already used in this file for the
  `requires_serial` check — no new import needed for that one; only
  `MDQ_SERIAL_TOOLS` and `is_side_effect` (already imported) are additions.
  `WRITE_TOOLS` / `DELETE_TOOLS` imports in `tool_runner.py` can be dropped
  from the `_build_tool_meta` path once `is_side_effect()` fully replaces
  their use here — but check for other in-file usages before removing the
  import (see Implementation Steps, Step 2).

## Risks

- **Risk**: Broadening `is_write`/`requires_serial` classification could
  cause previously-parallel tool calls (e.g. `trigger_workflow`,
  `rag_delete_document`, `index_paths`/`refresh_index`, `fts_rebuild`) to now
  serialize in real agent turns, increasing wall-clock latency for tool-heavy
  rounds. → **Mitigation**: This is the explicit intent of the fix (safety
  over parallelism for these tools); acceptable per requirement's Non-Goals
  ("Do not disable all parallel tool execution" — only the newly-classified
  write/admin tools are affected, read-only tools are untouched). Document the
  latency trade-off in the PR description.
- **Risk**: `fts_rebuild` becoming `requires_serial=True` makes it a global
  serial barrier — if it is ever called alongside many unrelated read tools
  in the same round, it forces those into serial ordering too? → **Mitigation**:
  Verified false — `requires_serial` tools each get their own one-element
  group and their own `ScheduledBatch`; other tools' concurrency is
  unaffected (confirmed by reading `build_execution_groups()`: serial-barrier
  tools do not merge with the parallel batch).
  Add a regression test asserting this (`test_fts_rebuild_does_not_serialize_unrelated_reads`).
- **Risk**: Existing external callers/tests may hard-code expectations that
  `trigger_workflow`/`rag_delete_document`/MDQ admin tools are `is_write=False`
  (e.g. relying on old default `parallel` bucket). → **Mitigation**: Searched
  `tests/` and `scripts/` for these tool names outside `tool_constants.py`
  and `tool_executor_helpers.py`; no test currently asserts a
  `parallel`/`read`/`is_write=False` expectation for these five tool names
  (confirmed via `rg "trigger_workflow|rag_delete_document|index_paths|refresh_index|fts_rebuild" tests/`
  during Step 3 read-through of related tests). Re-verify with the same
  `rg` command immediately before implementation to catch any file added
  since this plan was written.
- **Risk**: `build_execution_groups` (CC=23, grade D) and `_execute_with_dag`
  (CC=25, grade D) are both high-complexity, high-churn (9 and 46 commits
  respectively) functions; even a small edit inside them carries above-average
  regression risk per the `python-issue-to-plan` skill's guidance ("Modules
  with CC ≥ 15 require additional test coverage before changes"). →
  **Mitigation**: Keep the scheduler edit to the single `serialize_flags`
  line (no other logic touched); rely on the full existing
  `tests/test_tool_scheduler.py` suite (33 tests) plus new regression tests
  as the behavior-lock; do not refactor either function as part of this fix
  (refactoring, if desired, is out of scope and should be a separate
  `python-refactoring` task with behavior-lock tests acquired first, per
  `AGENTS.md`'s test-coverage section listing `tool_runner` as a
  behavior-locked module).
- **Risk**: Removing now-redundant `WRITE_TOOLS`/`DELETE_TOOLS` imports from
  `tool_runner.py` (Design §4 note) could break if those names are used
  elsewhere in the file beyond `_build_tool_meta()`. → **Mitigation**: Grep
  the file for other usages before deleting the import (Implementation Steps,
  Phase 3, first bullet); if still used elsewhere, keep the import and only
  change the `_build_tool_meta()` body.

All risks above have mitigations already specified; none require opening an
`issues/*_risks.md` file (Step 6 of the workflow: only *unmitigated* risks
are escalated to an issue file).
