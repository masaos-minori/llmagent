# Plan: Add guardrails to keep MDQ `TOOL_LIST`, dispatch table, and ToolRegistry aligned

Source requirement: `requires/20260716_03_require.md`

## Goal

A new test module fails immediately if any future change adds an MDQ tool to
`TOOL_LIST`, `_DISPATCH_TABLE`, `MDQ_TOOLS`, or `config/agent.toml`'s
`tool_names` without adding it to all four — the same class of drift that
`requires/20260716_02_require.md` fixes for the current `fts_consistency_check`
/ `fts_rebuild` case.

## Scope

**In scope**

- Add a new test module (`tests/test_mdq_tool_layer_consistency.py`) with the
  guardrail assertions listed in Design, covering:
  - `TOOL_LIST` (schema) vs `_DISPATCH_TABLE` (runtime dispatch), both
    directions.
  - `MDQ_TOOLS` (registry population) vs `TOOL_LIST` names, both directions.
  - `MDQ_TOOLS` vs `ToolRegistry` live state (no missing, no stale entries).
  - `MDQ_WRITE_TOOLS` vs each tool's `is_write` flag in `TOOL_LIST`.
  - Tools with `requires_serial: True` in `TOOL_LIST` are exactly
    `{"index_paths", "refresh_index"}` (the two write tools that mutate the
    shared SQLite index) — see Unknowns for how this set is pinned down.
- Add one doc-clarification note to `docs/04_mcp_03_01_dispatch-and-routing.md`
  stating the single responsibility of each layer (`tools.py` = LLM-visible
  schema, `server.py::_DISPATCH_TABLE` = runtime dispatch,
  `tool_constants.py::MDQ_TOOLS` = registry population source,
  `config/agent.toml::tool_names` = deployed/active tool allowlist).

**Out of scope**

- Live `/v1/tools` drift validation against `ToolRegistry` — this already
  exists generically in `shared/tool_routing_validation.py`
  (`validate_routing_against_live`) and is not MDQ-specific; no changes
  needed there per the requirement's own "Implementation notes" (do not merge
  tool-definition layers).
- Generalizing this guardrail to every other MCP server (github, shell, git,
  cicd, rag_pipeline, sqlite, file) — none of them currently have this class
  of test either, but the requirement scopes this issue to MDQ only. Worth a
  follow-up requirement, not implemented here (see Risks).
- Any functional change to MDQ tools — this plan is test-only, plus one doc
  note.

## Assumptions

1. This guardrail plan is implemented **after**
   `requires/20260716_02_require.md`'s plan lands (i.e., MDQ is down to 7
   tools with `fts_consistency_check`/`fts_rebuild` fully removed from every
   layer). Writing the guardrail against the pre-removal 9-tool state would
   immediately need rework once 02 lands. The calling workflow processes
   requirement files sequentially in filename order (01→02→03→...), so 02's
   implementation is expected to complete before 03's is implemented.
2. `MDQ_WRITE_TOOLS` (post-02-removal) will be `{"index_paths",
   "refresh_index"}` — verified today it is
   `{"fts_rebuild", "index_paths", "refresh_index"}`
   (`scripts/shared/tool_constants.py:87-89`); once `fts_rebuild` is removed
   by requirement 02, only the two indexing tools remain.
3. `requires_serial: True` is currently set in `TOOL_LIST` only on
   `index_paths` and `refresh_index` (`scripts/mcp_servers/mdq/tools.py:106-107,
   124-125`) — verified by direct read. No canonical "serial tools" constant
   exists elsewhere in `shared/tool_constants.py`; `requires_serial` is
   declared per-tool directly in each server's `tools.py`. The guardrail test
   therefore compares `TOOL_LIST`'s own `requires_serial` flags against
   `MDQ_WRITE_TOOLS` (both write tools should require serialization since they
   mutate the SQLite index under `MdqService._index_lock`), not against a
   separate registry-level source of truth.
4. No existing test in the codebase currently checks `TOOL_LIST` vs
   `_DISPATCH_TABLE` consistency for any MCP server — verified via
   `rg -n "_DISPATCH_TABLE" tests/` (no matches). This confirms the guardrail
   gap is real and not already covered elsewhere under a different name.

## Design

Four independent guardrail checks, each importing the real modules directly
(no mocking) so a future edit to any one file trips the corresponding test:

```python
from mcp_servers.mdq.tools import TOOL_LIST
from mcp_servers.mdq.server import _DISPATCH_TABLE
from shared.tool_constants import MDQ_TOOLS, MDQ_WRITE_TOOLS
from shared.tool_registry import get_registry

def test_tool_list_subset_of_dispatch_table():
    schema_names = {t["name"] for t in TOOL_LIST}
    assert schema_names <= set(_DISPATCH_TABLE)

def test_dispatch_table_subset_of_tool_list():
    schema_names = {t["name"] for t in TOOL_LIST}
    assert set(_DISPATCH_TABLE) <= schema_names

def test_mdq_tools_matches_tool_list():
    assert MDQ_TOOLS == {t["name"] for t in TOOL_LIST}

def test_mdq_tools_registered_in_registry():
    registry = get_registry()
    assert set(registry.get_tool_names("mdq")) == MDQ_TOOLS

def test_write_tools_flagged_is_write():
    for t in TOOL_LIST:
        if t["name"] in MDQ_WRITE_TOOLS:
            assert t.get("is_write") is True

def test_serial_tools_flagged_requires_serial():
    for t in TOOL_LIST:
        if t["name"] in MDQ_WRITE_TOOLS:
            assert t.get("requires_serial") is True
```

This directly satisfies the requirement's task list ("every `TOOL_LIST` entry
exists in `_DISPATCH_TABLE`", "every `_DISPATCH_TABLE` entry exists in
`TOOL_LIST`", "every supported MDQ tool is registered in `ToolRegistry`", "no
removed MDQ tools remain in `ToolRegistry`", "`MDQ_TOOLS` matches the
supported MDQ tool names", "write tools have `is_write=True`", "serial tools
have `requires_serial=True`").

The "drift validation expectations for MDQ live discovery" task item is
intentionally not duplicated here — `shared/tool_routing_validation.py`
already implements generic live-discovery drift checking
(`validate_routing_against_live`) that applies to MDQ automatically once
`config/agent.toml`'s `mcp_servers.mdq` config is passed through it; no
MDQ-specific version is needed.

## Implementation steps

1. **Confirm requirement 02 has landed** — re-run
   `uv run pytest tests/test_mdq_routing.py -v` and confirm it reflects the
   7-tool state before writing guardrail assertions against it.
2. **Write the new test module** — implement the six test functions from
   Design in `tests/test_mdq_tool_layer_consistency.py`.
3. **Verify the guardrail actually catches drift** — temporarily remove one
   `_DISPATCH_TABLE` entry locally and confirm the new test fails, then
   restore it (manual verification step, not committed).
4. **Add the doc note** — layer-responsibility paragraph in
   `docs/04_mcp_03_01_dispatch-and-routing.md`.
5. **Deployment/verification** — test-only change; no service restart
   required, but run the full MDQ test suite once more to confirm no
   collateral failures.

## Risks

- **Risk**: Implementing this before requirement 02 lands would bake
  assertions against the current 9-tool state, requiring immediate rework.
  **Mitigation**: Implementation step 1 explicitly gates on 02's completion;
  the calling workflow's sequential per-file processing order already
  enforces this.
- **Risk**: This guardrail is MDQ-only; the same drift class could recur in
  any other MCP server (github, shell, git, cicd, rag_pipeline, sqlite,
  file) since none of them have an equivalent test today.
  **Mitigation**: out of scope per the requirement's own "Implementation
  notes" section; flagged here as a candidate follow-up requirement rather
  than implemented speculatively.
