# Plan: Enforce production fail-fast settings with a dedicated ProductionConfigValidator

Source requirement: `requires/20260709_03_require.md`

## Goal

Add a standalone `ProductionConfigValidator` (mirroring the existing
`RagConfigValidator`'s shape, but topically unrelated) that flags
`plugin_strict`, `tool_definitions_strict`, and `routing_drift_strict` as
required-`True` settings; wire it into `audit_security_defaults()` so
production startup raises `RuntimeError` if any is `False`, while local
startup only warns.

## Design

**`shared/production_config_validator.py`** (new file):

```python
"""shared/production_config_validator.py — Startup validator for production
fail-fast settings. Separate from RagConfigValidator (RAG cross-file
consistency); this validator is topic-agnostic and only checks the flags
that must be True when security_profile == "production"."""

from __future__ import annotations

from typing import Any

from shared.config_validator import ConfigValidationResult

_REQUIRED_STRICT_KEYS: tuple[str, ...] = (
    "plugin_strict",
    "tool_definitions_strict",
    "routing_drift_strict",
)


class ProductionConfigValidator:
    """Flags production fail-fast settings that are not enabled.

    Does not itself know about security_profile — always reports which of
    the required keys are falsy. The caller (audit_security_defaults())
    decides whether to raise (production) or warn (local) based on the
    result.
    """

    def validate(self, cfg: dict[str, Any]) -> ConfigValidationResult:
        errors: list[str] = []
        for key in _REQUIRED_STRICT_KEYS:
            if not cfg.get(key, False):
                errors.append(f"{key} must be true in production")
        return ConfigValidationResult(errors=errors, warnings=[])
```

**`repl_health.py`** — inside `audit_security_defaults()`, add a new block
(placement: after the existing `shell_sandbox_backend` check and before the
`git allowed_repo_paths` check is a reasonable spot, since both are
early "hard settings" checks; exact placement is not load-bearing):

```python
    # Check production fail-fast settings via the dedicated validator
    from shared.production_config_validator import ProductionConfigValidator

    prod_result = ProductionConfigValidator().validate(
        {
            "plugin_strict": ctx.cfg.tool.plugin_strict,
            "tool_definitions_strict": ctx.cfg.tool.tool_definitions_strict,
            "routing_drift_strict": ctx.cfg.tool.routing_drift_strict,
        }
    )
    if not prod_result.ok:
        if production_mode:
            raise RuntimeError(
                "Production config validation failed: "
                + "; ".join(prod_result.errors)
            )
        for err in prod_result.errors:
            logger.warning("Security: %s", err)
            warnings.append(f"Security: {err}")
```

Design note: the validator itself is *not* `security_profile`-aware — it
always reports which required keys are falsy, regardless of profile. The
caller (`audit_security_defaults`) is what decides severity based on
`production_mode`, exactly matching the existing pattern used for
`shell_sandbox_backend == "none"` and HTTP `auth_token` earlier in the same
function. This keeps the validator a pure, trivially-testable function of
its input dict (satisfies the requirement's test case "local +
non-strict settings => warning or no production error" as a caller-level
behavior, not a validator-level branch).

## Risks

- **Risk**: `ProductionConfigValidator` lives in `shared/`, which per the
  architecture contract (`shared must not import from agent, mcp, rag, or
  db`) must not import anything from `agent/`. The design above only
  imports `shared.config_validator`, so this should hold — verify with
  `lint-imports` in the Validation plan regardless, since `repl_health.py`
  (in `agent/`) is the one importing `shared.production_config_validator`,
  not the reverse.
- **Risk**: Placing the new check between `shell_sandbox_backend` and `git
  allowed_repo_paths` in `audit_security_defaults()` is a judgment call;
  if a future reviewer expects strict ordering by topic, the check might
  look misplaced.
  → **Mitigation**: Low impact — the function already interleaves
  unrelated checks (shell, git, github, tools, cicd) without a strict
  topical grouping; order does not affect behavior, only readability.
- **Risk**: The three doc edits in `04_mcp_05_security_and_safety_model.md`'s
  table use a placeholder column value ("Keep `true` in production
  (enforced by `ProductionConfigValidator`)") for all 3 new/fixed rows —
  if the table's existing style varies per-row, this could look
  inconsistent.
  → **Mitigation**: The existing `shell_sandbox_backend` row already uses a
  similar imperative phrasing ("Set `\"firejail\"` in production"), so this
  matches the table's established voice.
