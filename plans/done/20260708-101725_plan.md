# Plan: H-10 — Stop creating tool_results table in new session schema

## Goal

Remove `CREATE TABLE IF NOT EXISTS tool_results` and its index
`CREATE INDEX IF NOT EXISTS idx_tool_results_session` from the session schema
DDL template so that new `session.sqlite` databases no longer contain this
backward-compatibility table. Existing databases are not altered.

---

## Scope

- `scripts/db/schema_sql.py` — remove DDL block for `tool_results` and
  `idx_tool_results_session` from `_SESSION_SCHEMA_TEMPLATE`
- `scripts/db/create_schema.py` — update `create_session_schema()` docstring
  (remove "tool_results" from the list of created objects)
- `tests/test_create_schema.py` — update the stub `_SESSION_SCHEMA_NO_VEC0`
  (remove `tool_results` DDL) and replace the positive assertion
  `test_creates_tool_results_table` with a negative assertion
  `test_does_not_create_tool_results_table`

Docs that reference session.sqlite tables (e.g. `docs/05_agent_09_data-layer.md`,
`docs/90_shared_04_db_architecture_and_schema.md`) will also need updating to
remove `tool_results` from the session.sqlite table list.

**NOT in scope:** DROP migration, changes to `db/tool_results.py` (runtime
`ToolResultStore`), or any schema change to workflow/rag/eventbus.

---

## Assumptions

- `ToolResultStore` in `scripts/db/tool_results.py` continues to work against
  existing databases that still have the `tool_results` table. This is safe
  because schema creation is separate from runtime code.
- The requirement explicitly says "do not add destructive DROP migration."
  This plan makes no such change.
- Path A routing applies (2 source files + 1 test file, no interface/schema
  change for existing callers).

## Unknowns

### U-1: Actual file paths — scripts/db/ vs db/
Require file references `db/schema_sql.py` and `db/create_schema.py`, but the
actual files are at `scripts/db/schema_sql.py` and `scripts/db/create_schema.py`.

**Resolution (confirmed by file search):** The canonical source files are
under `scripts/db/`. The `db/` prefix used in docstrings and module paths
(e.g. `from db.schema_sql import ...`) is the Python package import path,
mapping to `scripts/db/` via the project's sys.path configuration.

### U-2: Test coverage for negative assertion
The existing test `test_creates_tool_results_table` asserts that the table IS
created. After this change, the test must be inverted to assert it is NOT
created.

**Resolution:** Replace the test method body. The `_SESSION_SCHEMA_NO_VEC0`
stub used by the fixture must also drop the `tool_results` DDL block; otherwise
the fixture would create the table even after the production template is
updated.

### U-3: Documentation sync
`docs/05_agent_09_data-layer.md` section "session.sqlite Tables" still lists
`tool_results`. `docs/90_shared_04_db_architecture_and_schema.md` section 6
still documents the `tool_results` table and its columns.

**Resolution:** Both docs must be updated to remove `tool_results` from the
"created by new schema" table list, while noting that existing databases may
still contain the table (backward-compatibility note if appropriate).

## Design

### DDL change (scripts/db/schema_sql.py)

Remove from `_SESSION_SCHEMA_TEMPLATE`:

```sql
    CREATE TABLE IF NOT EXISTS tool_results (
        id         INTEGER PRIMARY KEY AUTOINCREMENT,
        session_id INTEGER REFERENCES sessions(session_id) ON DELETE CASCADE,
        turn       INTEGER NOT NULL,
        tool_name  TEXT    NOT NULL,
        args_masked  TEXT,
        full_text  TEXT    NOT NULL,
        summary    TEXT,
        is_error   INTEGER NOT NULL DEFAULT 0,
        undone     INTEGER NOT NULL DEFAULT 0,
        created_at TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))

    CREATE INDEX IF NOT EXISTS idx_tool_results_session
        ON tool_results(session_id);
```

## Risks

### R-1: Test stub out of sync with production template
If `_SESSION_SCHEMA_NO_VEC0` in the test file still contains `tool_results` DDL
after the production template is updated, the negative assertion would fail
because the fixture itself creates the table.

**Mitigation:** Step 3 explicitly updates `_SESSION_SCHEMA_NO_VEC0` before
changing the assertion.

### R-2: ToolResultStore runtime failure on new databases
`ToolResultStore` issues `INSERT INTO tool_results ...` at runtime. If a new
session.sqlite is created without the table, any call to `ToolResultStore.save()`
would raise `sqlite3.OperationalError: no such table: tool_results`.

**Mitigation:** This is a known pre-existing design conflict. The requirement
states "runtime code should not use it" for new databases. The actual
deactivation of `ToolResultStore` for new databases is outside this task scope.
This plan should note the residual risk explicitly: until `ToolResultStore`
usage is disabled or guarded by a table-existence check, running the agent
against a fresh new-schema database will raise on the first tool call.
Implementer must confirm whether `ToolResultStore` is already conditionally
disabled in the runtime path before closing this task.

### R-3: Documentation divergence with db layer source-of-truth
`docs/90_shared_04_db_architecture_and_schema.md` §10 names
`scripts/db/schema_sql.py` as DDL source of truth. After this change, the
document must reflect the removal — otherwise future readers will expect
`tool_results` in new databases.

**Mitigation:** Step 5 covers doc updates. Both doc files are included in scope.
