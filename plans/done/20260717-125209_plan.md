# Plan: Use RuntimeToolRegistry for routing and MCP execution

Source requirement: `requires/20260717_04_require.md`

## Goal

`ToolExecutor` routes exclusively through `RuntimeToolRegistry` (from requirements 02-03) instead of the static `ToolRegistry` singleton, and executes MCP tools only — the static `ToolRegistry`-backed `ToolRouteResolver` and the plugin-first execution check (removed separately by requirement 01) are both retired from the execution path.

## Scope

**In scope**
- `scripts/shared/route_resolver.py`: replace `ToolRouteResolver`'s internal `self._registry = get_registry()` (the static `ToolRegistry` singleton) with a `RuntimeToolRegistry` instance passed in at construction; `resolve()`'s logic (try lookup, raise/warn on miss) stays structurally the same, just backed by different data.
- `scripts/shared/tool_executor.py`: `ToolExecutor.__init__` now takes/builds its `ToolRouteResolver` (or resolves directly) against a `RuntimeToolRegistry` instance instead of implicitly pulling the global `ToolRegistry`; the plugin-check-first line at `execute()` (L211-213) is removed here too (shared boundary with requirement 01 — see Assumption 5).
- `scripts/shared/tool_transport_invoker.py`: no routing-logic change (it never consulted `ToolRegistry` directly — confirmed), but verify `server_key`-to-transport dict construction still receives correct data once `RuntimeToolRegistry` is the resolver's backing data source.
- `scripts/shared/transport_dto.py`: no dataclass shape change required (per Assumption 4) — `error_type="tool"` remains the shared bucket for both unknown-tool-name and missing-transport cases, satisfying "deterministic tool error" for both per the requirement's own wording.
- `tests/test_route_resolver.py` (the requirement's stated path `tests/shared/test_route_resolver.py` doesn't match reality — see Assumption 6): substantial rewrite, since it's tightly coupled to `ToolRouteResolver`'s exact constructor and to literal `"ToolRegistry"` error-message text.
- `tests/test_tool_executor.py` (same path correction) and `tests/test_tool_executor_routing.py` (**not in the requirement's target-file list at all** — flagged as a gap, see Unknowns): both assert resolver behavior via `ex._resolver` and must be checked/updated even though only one is named by the requirement.

**Out of scope**
- Changing `scripts/shared/http_transport.py` — confirmed to have zero routing/registry dependency; POSTs to `/v1/call_tool` unchanged, per the requirement's own "existing HttpTransport behavior remains compatible" acceptance criterion.
- Removing the plugin-check line itself — that is requirement 01's scope; this plan only needs `execute()`'s control flow to end up MCP-only, which requirement 01's removal already accomplishes independently; this plan focuses on the *routing* half (registry swap), not the plugin-removal half.
- Deleting `shared/tool_registry.py`/`ToolRegistry` outright — per requirement 02's plan, that decision is deferred; this plan only stops *routing* from using it, it may still be referenced elsewhere (e.g. drift validation in `repl_health.py`) until a later cleanup requirement addresses that explicitly.
- Building `RuntimeToolRegistry` itself (requirement 02) or populating it at startup (requirement 03) — this plan only wires `ToolExecutor`/`ToolRouteResolver` to consume it once it exists.

## Assumptions

1. `ToolExecutor.execute()`'s current flow (`tool_executor.py:204-216`, confirmed by direct read) is: plugin-check-first (line 211-213, unconditional) → side-effect check bypasses cache (214) → cache-or-raw-execute (216). Routing only happens inside `_raw_execute()` (86-128), specifically at line 92: `server_key = self._resolver.resolve(tool_name)` — the sole call into the routing layer.
2. `ToolRouteResolver` (`route_resolver.py:62-102`) currently imports the global `ToolRegistry` singleton via `get_registry()` and stores it as `self._registry`; its `server_configs` constructor parameter is accepted but explicitly documented as unused ("backward compatibility" only, line 74-75) — confirming that today's constructor signature already has a natural slot to repurpose for a `RuntimeToolRegistry` reference without adding a new parameter, only changing what the existing slot means.
3. `resolve()`'s control flow (`route_resolver.py:104-118`: try registry lookup → strict-mode raise or warn-then-raise on miss) needs no structural change — only the object it delegates to (`self._registry.get_server_for_tool(tool_name)` → `runtime_tool_registry.resolve(tool_name)`, per requirement 02's method name) changes.
4. `ToolCallResult`'s `error_type` field (`transport_dto.py`, confirmed a bare `str`, no enum) already buckets both "unknown tool" (though today this actually raises `ValueError` uncaught, not a `ToolCallResult` at all — see Unknown #1) and "missing transport for resolved server key" (`error_type="tool"`, via `_transport_missing_msg`/`_error_result` in `tool_transport_invoker.py:99-115`) under the same generic value. Per the requirement's own wording ("deterministic tool error" for both cases, not "distinguishable error"), no new error-type constant is needed — both cases can continue sharing `error_type="tool"`.
5. Requirement 01 (plugin subsystem removal) already removes `execute()`'s plugin-check line and `PluginToolInvoker` entirely; this plan does not need to re-remove it, only ensure the *routing* half (registry swap) is independent of that removal and doesn't conflict with it — both plans touch `tool_executor.py` but at different, non-overlapping lines (plugin-check line vs. resolver construction/usage).
6. **The requirement's stated test-file paths (`tests/shared/test_route_resolver.py`, `tests/shared/test_tool_executor.py`) don't match the actual repo layout** — confirmed via direct investigation: the real files are `tests/test_route_resolver.py` (398 lines, 34 tests, flat `tests/` dir, not under `tests/shared/`) and `tests/test_tool_executor.py` (973 lines, 42 tests, same). This plan targets the real, existing files.
7. `tests/test_tool_executor_routing.py` (934 lines, 64 tests) — **not listed in the requirement's target files at all** — directly asserts resolver behavior (`ex._resolver.resolve(...)`, resolver identity surviving `apply_config`) and will be affected by this change; this is a gap in the requirement's own scope statement that must be flagged, not silently worked around.
8. `tests/test_route_resolver.py` is tightly coupled to `ToolRouteResolver`'s exact constructor and to literal `"ToolRegistry"` text in strict-mode error messages (line 168) — confirmed by direct read; this file needs substantial rewrite, not incremental patching, since its assertions are about the *old* registry's identity, not just its behavior contract.
9. `tests/test_tool_executor.py` stubs the resolver as an injectable `MagicMock()` attribute (`ex._resolver = MagicMock()`) rather than constructing a real `ToolRouteResolver` — confirmed lower rewrite risk, since it's already resolver-implementation-agnostic.

## Design

### Registry injection point

Per Assumption 2, `ToolRouteResolver`'s existing `server_configs` constructor parameter is unused for routing today — rather than adding a new parameter, the cleanest change is to add an explicit `runtime_registry: RuntimeToolRegistry` parameter (required, not optional) and remove the implicit `get_registry()` pull, making the dependency explicit at construction time instead of a hidden global-singleton reach-in. `ToolExecutor.__init__` then must itself receive or build a `RuntimeToolRegistry` reference (sourced from `AppServices`/`AgentContext`, per requirements 02-03) and pass it down when constructing its internal `ToolRouteResolver`.

### Preserving `resolve()`'s error-handling contract

`resolve()`'s existing strict/warn-mode behavior (raise `ValueError` in strict mode on unknown tool; warn-then-raise otherwise) is preserved verbatim — only the data source changes from `ToolRegistry.get_server_for_tool()` to `RuntimeToolRegistry.resolve()` (per requirement 02's method, which per that plan's Design section mirrors the same `None`-on-unknown contract). This keeps `_raw_execute()`'s uncaught-`ValueError`-propagates-on-unknown-tool behavior unchanged, satisfying "unknown tool name should return deterministic tool error" only if `_raw_execute()`'s caller already catches this — confirm during implementation whether this exception is caught upstream into a `ToolCallResult` or genuinely propagates as an unhandled exception (this may already be existing, unrelated behavior to preserve, not something this plan needs to newly implement).

### Missing-transport case is unaffected

`ToolTransportInvoker`'s `self._transports.get(server_key)` dict lookup and `_transport_missing_msg`/`_error_result(error_type="tool")` path (per Assumption 1's survey) is entirely downstream of routing and untouched by this registry swap — a resolved-but-transportless `server_key` continues to produce the same `error_type="tool"` result regardless of which registry resolved it.

## Implementation steps

1. **Resolve the design Unknown**: confirm (or decide, since either is acceptable) whether `ToolRouteResolver` stays as a thin wrapper — recommend yes, per Design section, to minimize blast radius on `tool_executor_routing.py`'s 64 existing tests.
2. **Modify `route_resolver.py`**: change `ToolRouteResolver.__init__` to take a required `runtime_registry: RuntimeToolRegistry` parameter; remove the `get_registry()` import/call; update `_lookup_registry()` to call `self._runtime_registry.resolve(tool_name)` (or `.get(tool_name).server_key` depending on requirement 02's final method signatures — confirm during implementation).
3. **Modify `tool_executor.py`**: `ToolExecutor.__init__` accepts/receives a `RuntimeToolRegistry` reference (from `AppServices`, per requirements 02-03) and passes it into its internally-constructed `ToolRouteResolver`.
4. **Modify `factory.py`'s `_build_tool_executor()`**: pass the `RuntimeToolRegistry` instance (built by requirement 03's discovery step) into `ToolExecutor`'s constructor.
5. **Rewrite `tests/test_route_resolver.py`**: replace `ToolRegistry`/`get_registry()`-based fixtures with hand-built `RuntimeToolRegistry` fixtures; update the strict-mode error-message assertion (currently checks for literal `"ToolRegistry"` text) to match whatever the new implementation's error message says.
6. **Update `tests/test_tool_executor.py`**: minimal changes expected given its existing `MagicMock()`-based resolver stubbing; verify no test constructs a real `ToolRouteResolver`/`ToolRegistry` pairing that would break.
7. **Update `tests/test_tool_executor_routing.py`** (the flagged gap — not in the requirement's own list): verify/update its 64 tests' assumptions about resolver construction and identity-across-`apply_config` behavior against the new `RuntimeToolRegistry`-backed resolver.
8. **Verification**: run the standard validation sequence scoped to changed files, then the full suite — pay particular attention to the 3 affected test files (34 + 42 + 64 = 140 tests) for regressions.

## Risks

- **Risk**: `tests/test_tool_executor_routing.py` (64 tests) is not in the requirement's own target-file list — if implementation follows the requirement's literal file list without cross-checking, this file's assertions could silently break without being caught until CI.
  **Mitigation**: explicitly called out in this plan's Scope/Affected areas/Implementation steps; do not rely on the requirement's own list as exhaustive.
- **Risk**: `tests/test_route_resolver.py`'s literal `"ToolRegistry"` string assertion (line 168) is exactly the kind of brittle test-coupling that could cause a confusing failure if not caught during the rewrite (test expects old text, new code says something else, failure looks like a logic bug rather than a stale assertion).
  **Mitigation**: Implementation Step 5 explicitly calls out updating this assertion, not just the fixtures.
- **Risk**: If `_raw_execute()`'s uncaught `ValueError` on unknown-tool-name is genuinely never caught upstream (i.e., it's a latent bug, not intentional "deterministic tool error" behavior), this plan's Design section's assumption that it's pre-existing, acceptable behavior could be wrong.
  **Mitigation**: Implementation Step 2's confirmation step should trace this exception's actual handling before assuming no change is needed here; if it turns out to be an unhandled crash path, flag it as a separate finding rather than silently preserving a bug under this plan's "no change needed" assumption.
- **Risk**: Making `ToolRouteResolver`'s `runtime_registry` parameter required (removing the `get_registry()` fallback) means any test or code path that constructs `ToolRouteResolver` without one will now fail at construction time rather than silently using the wrong (global) registry.
  **Mitigation**: this is treated as a feature, not a risk to avoid — an explicit dependency is safer than an implicit global; the full-suite run (Validation plan) will surface any missed construction site immediately as a clear `TypeError`, not a silent behavioral drift.
