## Goal

Add `workflow_id` directly to the `approvals` table schema and require it on approval creation, so that every approval record carries its own workflow context without requiring a task join.

## Scope

**In**:
- Add `workflow_id TEXT NOT NULL` column to the `approvals` table in the workflow SQLite schema definition
- Change `request_approval(db, task_id, ...)` in `scripts/agent/workflow/approval_ops.py` to accept and persist `workflow_id: str` (required)
- Update `WorkflowEngine._gate_approval()` in `scripts/agent/workflow/workflow_engine.py` to pass `workflow_id` (from `task.workflow_id`) to `request_approval()`
- Update `ApprovalRecord` dataclass in `scripts/agent/workflow/models.py` to include `workflow_id: str`
- Update `get_pending_approval()`, `find_approval_by_id()`, `find_latest_pending_approval()`, `find_pending_approval_by_session()` in `approval_ops.py` to populate `workflow_id` from the row
- Update `_emit_approval_audit()` in `cmd_workflow.py` to use `approval.workflow_id` directly instead of joining through `store.get_task_by_id(task_id)` — remove the task join
- Ensure audit event never has `workflow_id=null` or empty string
- Update all affected tests

**Out**:
- Supporting legacy approval rows without `workflow_id` (migration compatibility excluded)
- Backfill of existing `workflow_id=NULL` rows
- Schema migration scripts
- Changing the strategy to "task join only" (option 2 rejected — direct storage is safer for audit)

**Decision**: Strategy 1 (store `workflow_id` directly on approvals) is selected because:
- Avoids task join in audit paths — single-table lookup
- Consistent with req07 (all events carry full workflow context)
- Simpler for startup recovery display and `/approve` output
- After req09, `tasks.workflow_id` is always non-null, so denormalization is safe

## Assumptions

1. The `approvals` table schema is defined in a SQL schema file or `create_workflow_schema()` function — migration is out of scope; schema is recreated from scratch.
2. `request_approval()` is called only from `WorkflowEngine._gate_approval(task)` (line 148); `task.workflow_id` is available there after req09 makes it required.
3. `ApprovalRecord` is a dataclass; adding `workflow_id: str` field does not break existing construction sites once `approval_ops.py` is updated.
4. `_emit_approval_audit()` currently calls `store.get_task_by_id(task_id)` to get `workflow_id`; after this change it reads `approval.workflow_id` directly.
