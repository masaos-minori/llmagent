# Plan: Remove obsolete and non-functional MDQ configuration compatibility paths

Source requirement: `requires/20260716_01_require.md`

## Goal

`MdqService` in `scripts/mcp_servers/mdq/service.py` no longer reads the dead
`audit_log_path` config key, and the MDQ configuration surface documents which
follow-up requirement owns each remaining "review" item so no duplicate or
conflicting cleanup work happens across the batch of MDQ compatibility
requirements filed on 2026-07-16.

## Scope

**In scope**

- Remove `self.audit_log_path` and its config read from `MdqService.__init__`
  (`scripts/mcp_servers/mdq/service.py:67-69`).
- Confirm no other runtime code reads `audit_log_path` (verified — none found).
- Confirm audit events continue to flow through `_audit_log()` into
  `mdq-mcp.log` (verified — implemented in `scripts/mcp_servers/mdq/server.py`,
  unaffected by this change).
- Add/adjust a regression test asserting `MdqService` has no `audit_log_path`
  attribute and that construction succeeds without the key in config.
- Update `docs/04_mcp_04_04_mdq.md` config field list to drop `audit_log_path`
  from the "already removed" parenthetical note once the code no longer reads
  it (the key was already removed from `config/mdq_mcp_server.toml` on
  2026-07-13; only the code-level read lagged behind).

**Out of scope (owned by other requirements in this batch — do not duplicate here)**

| Item | Deferred to |
|---|---|
| `max_search_results` vs `max_results_limit` consolidation | `requires/20260716_04_require.md` |
| `use_embedding`, `embedding_dims`, `vector_table`, `embedding_model` | `requires/20260716_04_require.md` |
| `summary_cache_enabled`, `summary_threshold`, `summary_model` | `requires/20260716_05_require.md` |
| `enable_refresh` enforcement/removal | `requires/20260716_06_require.md` |

This plan only executes the `audit_log_path` removal task list items from the
source requirement. The requirement's remaining checklist items (summary
cache, embedding, search limits, `enable_refresh`) are explicitly re-scoped
into their own dedicated requirement files, confirmed by reading all 7
`requires/20260716_*.md` files before planning began — each of those items has
a requirement file with a narrower, more detailed task list than this one.

## Assumptions

1. `audit_log_path` in `scripts/mcp_servers/mdq/service.py:67-69` has zero
   runtime readers besides the assignment itself — verified via
   `rg -n "audit_log_path" scripts/ tests/` (only the definition line and the
   already-removed-key comment in `config/mdq_mcp_server.toml` reference it).
2. `config/mdq_mcp_server.toml` no longer defines `audit_log_path` (removed
   2026-07-13 per the NOTE comment at line 56) — so today `mdq_cfg.get(...)`
   always falls through to the hardcoded default `"/opt/llm/logs/mdq_audit.log"`,
   which is never used for anything.
3. Removing the attribute does not change `mdq-mcp` audit behavior, because
   audit events are written by `_audit_log()` in `server.py`, which does not
   reference `MdqService.audit_log_path` at all.
4. No external caller (agent CLI, other MCP servers) reads
   `MdqService.audit_log_path` — verified via
   `rg -n "\.audit_log_path" scripts/` returning only the `service.py`
   definition site.

## Unknowns

None blocking. One non-blocking note: no test currently exercises
`MdqService.__init__` config parsing directly for `audit_log_path`, so this
plan adds a small regression test rather than modifying an existing one.

## Design

`MdqService.__init__` currently does:

```python
self.audit_log_path: str | None = mdq_cfg.get(
    "audit_log_path", "/opt/llm/logs/mdq_audit.log"
)
```

This line is deleted outright. No replacement field is introduced — the
requirement's intent is removal, not renaming, since the value has no
consumer. `MdqDatabaseError`/other imports are unaffected.

## Implementation steps

1. **Remove the dead field** — delete lines 67-69 in
   `scripts/mcp_servers/mdq/service.py` (the `self.audit_log_path = ...`
   assignment). Independently committable and revertable.
2. **Add regression coverage** — add a test in `tests/test_mdq_service.py`
   asserting `not hasattr(service, "audit_log_path")` and that `MdqService`
   still constructs successfully from a config dict without the key.
3. **Update documentation** — adjust the parenthetical note in
   `docs/04_mcp_04_04_mdq.md:27` to state the code-level read was removed on
   the current date, matching the pattern already used for
   `concurrency_limit`.
4. **Deployment/verification** — no deploy.sh change needed (no new/removed
   module file). Run the validation sequence below and manually restart
   `mdq-mcp` in a dev environment to confirm the service still starts and
   `stats`/`search_docs` tools respond normally.

## Risks

- **Risk**: A future contributor might reintroduce `audit_log_path` expecting
  it to control audit output, repeating the original confusion.
  **Mitigation**: the doc note (Implementation step 3) explicitly states the
  path is not read and directs to `_audit_log()`/`mdq-mcp.log` instead, same
  pattern already used for `concurrency_limit`.
- **Risk**: Overlap with the other 3 requirement files in this batch
  (`04_require.md`, `05_require.md`, `06_require.md`) if implemented
  out of order, causing merge conflicts in `service.py` and
  `docs/04_mcp_04_04_mdq.md`.
  **Mitigation**: this plan touches only the `audit_log_path` lines; the
  other plans touch disjoint config keys in the same file/doc, so sequential
  implementation (one requirement fully merged before the next starts, per
  the calling workflow's own per-file cycle rule) avoids conflicts.
