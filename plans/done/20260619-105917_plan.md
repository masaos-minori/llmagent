# Plan: Define a better incremental update strategy for file:// ingestion

## Goal

Define and document an incremental update strategy for local file (`file://`) ingestion that is more efficient than the current all-or-nothing approach. Document how it differs from web ingestion (Conditional GET / ETag).

## Scope

**In scope:**
- Define mtime-based or hash-based no-op strategy for local files
- Document the incremental update flow for `file://` URLs
- Document differences between local file and web ingestion strategies
- Add design documentation in `docs/` for the incremental update strategy

**Out of scope:**
- Implementing the incremental update logic (implementation is a separate task)
- Modifying the DB schema (would require a migration plan)
- Adding mtime/hash columns to the `documents` table
- Changing web ingestion behavior (already has Conditional GET)

## Assumptions

1. The current sentinel-based skip (`chunk_dir/stem-0000.txt`) is too coarse — it skips entire files regardless of content changes.
2. An mtime-based approach would be simpler to implement but less robust than hash-based.
3. A hash-based approach (SHA-256) would be more robust but adds computation cost during ingestion.
4. The requirement asks to "define" the strategy, not implement it — documentation and design are the primary deliverables.

## Implementation Steps

1. **Define mtime-based incremental update strategy**
   - Store file `mtime` in `documents` table (new column: `file_mtime TEXT`)
   - Before re-ingesting a `file://` URL, compare stored `mtime` with current file `mtime`
   - If `mtime` unchanged → skip ingestion (no-op)
   - If `mtime` changed → re-ingest (same as current `--force` behavior)
   - This is the default strategy

2. **Define hash-based incremental update strategy (opt-in)**
   - Store file content hash (SHA-256) in `documents` table (new column: `content_hash TEXT`)
   - Before re-ingesting, compute current file hash and compare
   - Hash comparison is more robust than mtime (catches content changes even if mtime is unchanged)
   - Adds computation cost (hash every ingestion run)
   - Opt-in via config flag (`incremental_strategy: "mtime"` or `"hash"`)

3. **Document the incremental update flow**
   - Document the decision tree for `file://` ingestion:
     1. Check if URL exists in DB
     2. If not → ingest normally
     3. If yes → compare mtime (or hash) with current file
     4. If unchanged → log "skip (unchanged)" and continue
     5. If changed → re-ingest (delete old, insert new)
   - Document how this differs from web ingestion:
     - Web: server-driven (Conditional GET, 304 Not Modified)
     - Local: client-driven (mtime/hash comparison)

4. **Document differences from web ingestion**
   - Create a comparison table in documentation:
     | Aspect | Web (http/https) | Local (file://) |
     |---|---|---|
     | Change detection | Conditional GET (ETag, Last-Modified) | mtime or content hash |
     | Server cooperation | Yes (server returns 304) | No (client checks locally) |
     | Network overhead | HTTP request + headers | None (local file access) |
     | Computation cost | None (server-driven) | mtime: negligible, hash: moderate |
     | Reliability | Depends on server support | Fully deterministic |

5. **Document operational implications**
   - When to use `--force` vs relying on incremental check
   - How to handle edge cases:
     - File moved/renamed (URL changes → new document)
     - File permissions changed (mtime changes but content same)
     - Large files (hash computation may be slow)

| Risk | Likelihood | Mitigation |
|---|---|---|
