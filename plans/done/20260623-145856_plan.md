# Plan: Elevate RAG Consistency Check to Normal Operations Diagnostics

**Requirement:** `requires/20260622_90_require.md`

---

## Goal

Surface RAG consistency check results during normal operation, not only as an on-demand maintenance command.

---

## Current State

| Path | Consistency check run? | Terminal-visible? |
|---|---|---|
| Agent startup | Yes (`startup.py:170-176`) | Yes — `write_warning("[RAG] Consistency issue: ...")` |
| Post-ingest (per URL) | Yes (`ingester.py:131-135`) | **No** — `logger.warning()` only |
| On-demand (`/db rag consistency`) | Yes | Yes |
| `/stats` | No | No |

**Gap:** Post-ingest consistency warnings go to `logger.warning()` only. Operators running `/ingest run` will not see index issues unless they actively inspect logs or run `/db rag consistency` manually.

**Already correct:** Startup check is already operator-visible. `summarize_issues()` already labels issues as `[WARNING]` / `[CRITICAL]` with repair guidance.

---

## Scope

**In:**
- `scripts/rag/ingestion/ingester.py` — add `consistency_issues: list[str]` to `IngestUrlResult`; populate from `check_rag_consistency()` result
- `scripts/agent/services/ingest_workflow.py` — emit terminal warnings for each consistency issue after ingest
- `scripts/agent/commands/cmd_config_stats.py` — add hint line in `/stats` when RAG DB is configured: "Run /db rag consistency for index integrity status"

**Out:**
- Auto-repair of detected issues (out of scope per requirement)
- Running consistency check on every `/stats` call (too heavy for a frequent command)
- Changing startup consistency check (already correct)
- Changing on-demand `/db rag consistency` (already correct)

---

## Assumptions

1. `IngestUrlResult` is a frozen dataclass — adding `consistency_issues: list[str] = dataclasses.field(default_factory=list)` with a default is safe
2. `ingest_workflow.py` has access to `on_status` callback for emitting messages — same pattern as `n_embed_failed` warning (req 85)
3. `check_rag_consistency()` in `ingester.py:131` already runs after ingestion — the issues need only be propagated up, not generated anew
4. The hint in `/stats` is one line and adds no DB read overhead

## Design

### Change 1: `ingester.py` — `IngestUrlResult.consistency_issues`

In `IngestUrlResult`:
```python
@dataclasses.dataclass(frozen=True)
class IngestUrlResult:
    url: str
    n_success: int
    n_failed: int
    skipped: bool
    n_embed_failed: int = 0
    consistency_issues: tuple[str, ...] = ()  # use tuple for frozen dataclass
```

In `_ingest_url()` (around line 131-135):
```python
consistency_issues: tuple[str, ...] = ()
try:
    report = check_rag_consistency(db)
    issues = summarize_issues(report)
    consistency_issues = tuple(issues)
    for issue in issues:
        logger.warning("Post-ingest consistency: %s", issue)
except Exception:
    logger.exception("Post-ingest consistency check failed")

return IngestUrlResult(
    url=url,
    n_success=inserted,
    n_failed=failed,
    skipped=skipped,
    n_embed_failed=embed_failed,
    consistency_issues=consistency_issues,
)
```

### Change 2: `ingest_workflow.py` — emit terminal warnings

After `results = await loop.run_in_executor(...)` (ingest completes), collect and emit:
```python
all_issues: list[str] = []
for r in results:
    all_issues.extend(r.consistency_issues)
for issue in all_issues:
    messages.append(
        self._emit(f"[warn] RAG consistency: {issue}", on_status)
    )
```

### Change 3: `cmd_config_stats.py` — `/stats` hint line

In `_cmd_stats()`, after the DB stats section:
```python
if ctx.cfg is not None and ctx.cfg.tool is not None:
    # Hint: run consistency check manually
    self._out.write("  RAG index    : run /db rag consistency to verify FTS/vec sync")
```

Or more targeted — only when a `rag.sqlite` is configured. Since the RAG DB path is always configured when the agent is used with RAG, this hint is always relevant:
```python
self._out.write("  RAG index    : /db rag consistency (FTS/vec sync check)")
```

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Post-ingest consistency warnings fire for every URL in a large batch, creating noise | Low | Issues are per-batch not per-URL — `check_rag_consistency()` runs once after all URLs for a source dir are processed; duplicates rare |
| Adding `consistency_issues` field to frozen `IngestUrlResult` breaks existing construction sites | Low | Use `tuple[str, ...] = ()` with default — existing call sites that don't pass the field still work |
| `/stats` hint line clutters the output | Low | Single short line; operators can correlate with `/db rag consistency` result |
