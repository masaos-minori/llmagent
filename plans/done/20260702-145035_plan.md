## Goal

Resolve ambiguity around `artifact_type` by deciding it is ingestion-pipeline metadata only (Option B) and updating documentation and code comments to reflect that decision explicitly.

## Scope

**In**: 
- Document `artifact_type` as ingestion-pipeline-only metadata in `docs/03_rag_04_data_model_and_interfaces.md` and `docs/03_rag_02_ingestion_pipeline.md`
- Add an inline comment to `ChunkJsonRaw.artifact_type` in `scripts/rag/ingestion/pipeline_utils.py` clarifying it is not persisted
- No schema changes

**Out**:
- Option A (adding `artifact_type TEXT NOT NULL DEFAULT ''` to the `chunks` table) — explicitly rejected; see Rationale below
- Any changes to `ChunkDocument`, `ChunkRecord`, `scripts/db/schema_sql.py`
- Broad artifact-management redesign

## Rationale for Option B

Evidence gathered from static analysis:

1. **`read_json_file()` drops the field** (`scripts/rag/ingestion/pipeline_utils.py:67–95`): the function reads `url`, `content`, `title`, `lang`, `code_blocks`, `etag`, `last_modified`, `chunking_strategy`, `normalized_content`, `chunk_index`, `source_file`, `chunk_type` — but explicitly omits `artifact_type`. `ChunkDocument` is constructed without it.
2. **`ChunkDocument` has no `artifact_type` field** (`scripts/rag/models_data.py:27–39`): the dataclass does not carry the value through the ingestion pipeline.
3. **`chunks` table has no `artifact_type` column** (`scripts/db/schema_sql.py:27–34`): `chunks` stores only `chunk_id`, `doc_id`, `chunk_index`, `content`, `normalized_content`.
4. **`_validate_artifact()` is a guard, not a persistence path** (`scripts/rag/ingestion/ingester.py:296–305`): the method checks that the JSON file's `artifact_type` matches the expected value (`"crawl"` or `"chunk"`) and returns a boolean. It does not write to the DB.
5. **No downstream query or filter references `artifact_type`**: `grep -rn artifact_type scripts/ tests/ docs/` returns no hits in query-side code (`rag/retriever.py`, `rag/query_pipeline.py`, or any MCP tool). The field is used only in `crawler.py`, `chunk_splitter.py`, and `ingester.py` (validation).
6. **Adding the column would require a migration** with no current consumer — zero ROI.

Conclusion: `artifact_type` is a pipeline-level artifact provenance tag used only for input validation. Persisting it would add schema complexity with no query-side benefit at this time.

## Assumptions

1. There is no undocumented code path that queries `artifact_type` from the DB.
2. Future search-filtering by artifact type, if needed, can be introduced as a separate issue.
3. The `artifact_type` values `"crawl"` and `"chunk"` are stable enough to document as the canonical value set.
