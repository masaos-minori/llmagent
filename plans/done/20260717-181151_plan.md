# Plan: Evaluate future options for disabled-tool filtering and structured disabled codes

Source requirement: `requires/20260717_20_require.md` (moved to `requires/done/` on
completion of this workflow)

## Goal

Evaluate — and document, but not implement — two forward-looking design options for
the still-unimplemented MCP tool runtime-availability contract (`config_dependent` /
`enabled` / `disabled_reason`, requirements 14-19): (1) an `include_disabled=false`
query parameter on `/v1/tools` for filtered tool discovery, and (2) a structured
`disabled_code` enum alongside the human-readable `disabled_reason` string. This is a
documentation/evaluation-only requirement (Priority: Low); no source code, tests, or
runtime behavior changes are made, and the plan must explicitly preserve today's
default `/v1/tools` behavior (always return every tool, disabled ones included) as
the assumption for any future implementation.

## Scope

**In scope**:

- Add a "Future / deferred design options" section documenting the two evaluated
  options to the canonical MCP tool-availability doc that requirement 19's plan
  (`plans/20260717-180307_plan.md`) is designing:
  `docs/04_mcp_03_06_tool-runtime-availability-metadata.md`. (See Assumptions/Risks —
  this file does not exist yet at the time of this plan's authorship; requirement
  19's plan has not been implemented.)
- Document the `include_disabled=false` query-param option: proposed semantics,
  explicit statement that the default must remain `include_disabled=true` (i.e.
  today's always-return-all-tools behavior), and that disabled tools are omitted only
  when a caller explicitly opts in.
- Document the `disabled_code` structured-enum option: candidate values
  (`EMPTY_ALLOWED_DIRS`, `EMPTY_ALLOWED_REPO_PATHS`, `READ_ONLY`,
  `EMPTY_COMMAND_ALLOWLIST`, `EMPTY_WORKFLOW_ALLOWLIST`), how it would coexist with
  (not replace) the free-text `disabled_reason` string, and which servers each code
  would map from (file read/write/delete -> `EMPTY_ALLOWED_DIRS`; git ->
  `EMPTY_ALLOWED_REPO_PATHS`/`READ_ONLY`; shell -> `EMPTY_COMMAND_ALLOWLIST`; cicd ->
  `EMPTY_WORKFLOW_ALLOWLIST`).
- State explicitly, in the new section, that neither option is required for or
  blocks the initial migration (requirements 14-19), per this requirement's own
  Acceptance criteria.
- Add a short "known gap / deferred decision" cross-reference entry to
  `docs/04_mcp_90_inconsistencies_and_known_issues.md` (mirroring the pattern
  requirement 19's plan already uses for that file) noting these two options are
  evaluated-but-deferred, with pointers back to this plan and to
  `requires/done/20260717_20_require.md`.

**Out of scope**:

  any test file — this requirement's Acceptance criteria explicitly states "No

- Implementing the `config_dependent`/`enabled`/`disabled_reason` contract itself —

- Adding an actual `/v1/tools?include_disabled=` FastAPI query parameter or a

## Design

- New subsection **"Future / deferred design options"** to be appended to
  `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` (after its "Implementation
  status" callout, per that file's planned Design item 7 in
  `plans/20260717-180307_plan.md`), containing two clearly-labeled proposals:
  1. **`include_disabled` query parameter** — `GET /v1/tools?include_disabled=false`
     as an opt-in filter. Default (no query param, or `include_disabled=true`)
     preserves today's behavior: every tool is returned, disabled tools included
     with `enabled=false`/`disabled_reason` set. Only when a caller explicitly
     passes `include_disabled=false` are disabled tools omitted from the `tools`
     array. State plainly: this is unimplemented and requires no immediate action.
  2. **`disabled_code` structured field** — a machine-readable enum companion to the
     free-text `disabled_reason`, proposed values mapped 1:1 to today's
     `requires_config`-gated servers: `EMPTY_ALLOWED_DIRS` (file read/write/delete),
     `EMPTY_ALLOWED_REPO_PATHS` (git, precedence over `READ_ONLY`), `READ_ONLY` (git
     write tools), `EMPTY_COMMAND_ALLOWLIST` (shell — reserved, per requirement 15's
     plan's own scoping-out of shell), `EMPTY_WORKFLOW_ALLOWLIST` (cicd — reserved,
     same reason). Both fields would be present together on a disabled tool
     (`disabled_reason` for humans/logs, `disabled_code` for programmatic dispatch),
     never `disabled_code` alone.
  3. Explicit closing statement: "Neither option is implemented. Both are deferred
     design decisions tracked for future evaluation; the initial RuntimeToolRegistry
     migration (requirements 14-19) does not depend on either."
- New entry in `docs/04_mcp_90_inconsistencies_and_known_issues.md` (format to match
  whatever convention that file already uses — read in full before writing, matching
  requirement 19's plan's own Implementation step 1 discipline): a short row/bullet
  noting "`include_disabled` filtering and `disabled_code` structured codes were
  evaluated per requirement 20 and intentionally deferred (not implemented); see
  `plans/20260717-181151_plan.md` and the Future-options section of
  `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` once that file exists."
- No FastAPI route signatures, Pydantic models, or dispatch-table changes are
  designed here — this plan produces prose/documentation content only.

- **Base file dependency on requirement 19 (unresolved ordering)**: this plan's
  primary content (the "Future options" subsection) is designed to live inside
  `docs/04_mcp_03_06_tool-runtime-availability-metadata.md`, a file that does not
  exist yet and is owned by a different, also-unimplemented plan
  (`plans/20260717-180307_plan.md`). If requirement 19 is never implemented, or is
  implemented with a different file name/structure than currently planned, this
  plan's Future-options content has nowhere to land except as a standalone
  artifact. Mitigation applied: Implementation step 1's fallback branch keeps the
  content fully specified inside this plan document itself (Design section), so
  the evaluation work is not lost or blocked regardless of requirement 19's status;
  the `04_mcp_90` known-gap entry (step 3) is independent of `04_mcp_03_06`'s
  existence and lands regardless. Residual risk: the content may sit unattached to
  the canonical doc for an indeterminate time. This mirrors the exact same
  unmitigated-dependency shape already tracked in `issues/20260717-180437_risks.md`
  and `issues/20260717-175327_risks.md` — filed as a further issue below rather than
  silently left implicit, since a third plan now depends on the same unlanded
  chain.
- **`disabled_code` value/shape drift**: the proposed enum values are mapped against
  `requires_config`'s current per-server semantics, not against requirement 15's
  actual (unimplemented) `disabled_reason` design. If requirement 15 lands with
  different reason strings or a different mechanism (e.g. a list of reasons instead
  of one), the `disabled_code` mapping table in this plan's Design section would
  need revision before anyone builds on it. Mitigation: the Design section already
  labels both proposals as explicitly deferred/unimplemented and the closing
  statement disclaims any dependency, so no reader can mistake this for a committed
  contract. Residual risk (shape drift after requirement 15 lands) is not fully
  mitigable now and is filed below.
- **Doc-consistency test gap**: if requirement 19's file is created between this
  plan's authorship and someone appending the Future-options section, step 2's
  `test_check_mcp_docs_consistency.py` run could be skipped by mistake. Mitigation:
  Implementation step 1 makes the branch explicit and step 2 ties the test run
  directly to that branch; no further action needed — this is fully mitigated by
  the plan's own steps.
