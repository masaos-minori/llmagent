# Plan: Redefine static tool_definitions and tool_constants as compatibility-only

Source requirement: `requires/20260717_10_require.md`

## Goal

Static `config/agent.toml` `[[tool_definitions]]` and `shared/tool_constants.py`'s frozensets are no longer consulted as the *primary* runtime source anywhere in the main execution path ‚Äî `RuntimeToolRegistry` (requirements 02-03) is the sole runtime authority ‚Äî and every remaining reference to the static sources is explicitly marked as compatibility/fallback/test-fixture-only, in code comments and documentation, so no future contributor mistakes a fallback path for the real source of truth.

## Scope

**In scope**
- Confirm (post-implementation of requirements 04-09) that every runtime consumer of static `tool_definitions`/`tool_constants.py` identified in this batch has actually been migrated to `RuntimeToolRegistry` as its *primary* source ‚Äî this requirement is the closing verification pass, not new migration work (requirements 04-09 already did the migration; this requirement audits completeness and adds explicit compatibility labeling).
- `scripts/agent/config_dataclasses.py`: add/confirm a doc-comment on `ToolConfig.tool_definitions` stating it is fallback/test-fixture-only, not the runtime source.
- `scripts/agent/config_builders.py`: confirm no code path here treats `tool_definitions` as authoritative beyond building the fallback config value.
- `scripts/agent/services/config_reload.py`: confirm `/reload` doesn't re-derive anything runtime-authoritative from static `tool_definitions` (should already be true per requirement 08's plan, which only touches `apply_policy()`).
- `scripts/shared/tool_constants.py`: add an explicit module-docstring statement that these frozensets are compatibility/fallback data feeding `RuntimeTool` construction or serving as a secondary check, not the runtime source of truth (mirrors this requirement's own wording).
- `scripts/shared/tool_registry.py`: add an explicit module-docstring statement (updating its existing "primary source of truth for MCP tool ownership and routing" claim, confirmed still present at line 4 ‚Äî see Assumption 1) to state it is non-authoritative for *runtime execution* specifically, since `RuntimeToolRegistry` has superseded it there (per requirement 04's routing migration) ‚Äî but it may still serve other purposes (e.g. drift-detection input data) that this requirement should not silently break.
- `scripts/shared/tool_executor_helpers.py`, `scripts/agent/tool_policy.py`: confirm (not re-implement) that requirements 06-07's fallback demotions are marked with an explicit code comment stating the static-set path is secondary/fallback.
- Documentation: update all docs describing tool definitions/routing/side-effect classification/policy as static-config-sourced, to state MCP `/v1/tools` (via `RuntimeToolRegistry`) is the normal runtime source, with static definitions positioned as fallback/compatibility.
- Tests using static `tool_definitions`/`tool_constants` as fixtures: confirm these remain valid as *fixture* data (this requirement doesn't forbid using static-shaped data in tests ‚Äî it forbids the *runtime* depending on it as primary), and are not mistakenly asserting runtime authority that no longer holds.

**Out of scope**
- Deleting `tool_constants.py`, `tool_definitions`, or `ToolRegistry` outright ‚Äî this requirement explicitly keeps them as compatibility/fallback, not removes them.
- Re-doing requirements 04-09's actual migration work ‚Äî this requirement assumes those land first and audits/labels the result.
- Changing any test's fixture *data* ‚Äî only confirming fixtures don't assert stale claims about runtime authority.

## Assumptions

1. `scripts/shared/tool_registry.py`'s module docstring (confirmed by direct read in requirement 02's investigation) currently states: "Primary source of truth for MCP tool ownership and routing... ToolRegistry is the sole routing authority" (lines 4, 18-19) ‚Äî this claim becomes **false** once requirement 04 lands (`RuntimeToolRegistry` becomes the routing authority instead) and must be corrected as part of this requirement's documentation-accuracy scope, not left stating an outdated claim.
2. This requirement is sequenced *after* requirements 02-09 in this batch (all of which do the actual migration work) ‚Äî it is fundamentally a verification-and-labeling pass, not new migration logic. Its own Scope section's wording ("Redefine... as fallback or test fixture only") is about *documentation and comments*, not code behavior changes, since requirements 04-09 already specify the actual behavior changes.
3. `tool_constants.py`'s frozensets (per requirements 06-07's plans) are retained as a *fallback* input for `is_side_effect()`/`classify_operation_type()` when a tool is absent from `RuntimeToolRegistry` ‚Äî this requirement's "compatibility-only" framing is consistent with (not contradicting) those plans' "demote to fallback, don't delete" designs.
4. `config/agent.toml`'s `[[tool_definitions]]` block itself is not deleted by this requirement ‚Äî it may still serve as: (a) a fallback source if `RuntimeToolRegistry` is empty/unavailable (an edge case worth confirming exists or explicitly deciding doesn't need to exist), and (b) the shape template for test fixtures. This requirement's job is to make sure code/docs *say* this clearly, not to change whether the TOML block itself remains in the config file.
5. No test file needs its fixture *data* changed by this requirement ‚Äî tests that construct `RAGConfig`/`ToolConfig`/etc. with static `tool_definitions`-shaped dicts remain valid as long as they're testing fallback/compatibility behavior explicitly, not asserting that this is the "normal" runtime path.

## Design

This requirement is best modeled as a **verification-and-labeling checklist** run after requirements 02-09 land, not independent new logic:

1. For each of the 4 "primary source" roles this requirement's Reason section names (tool schema, routing seed, side-effect classification, policy classification), confirm the corresponding requirement (05 for schema, 04 for routing, 06 for side-effect, 07 for policy) has actually landed and `RuntimeToolRegistry` is the primary source in each case.
2. For each of the 3 static-source files (`tool_constants.py`, `tool_registry.py`'s `ToolRegistry`, `config/agent.toml`'s `tool_definitions`), add/correct a docstring or comment explicitly stating its compatibility/fallback status and pointing at `RuntimeToolRegistry` as the authoritative replacement.
3. Sweep `docs/` for any doc still describing static config as the normal/authoritative tool source (this is a real content audit, not a mechanical find-replace, since some docs may describe the compatibility path intentionally and correctly once this migration is complete).

## Implementation steps

1. **Confirm requirements 04-09 have landed** (or note explicitly if this requirement is being implemented ahead of its dependencies ‚Äî in which case, stop and flag rather than proceeding against unmigrated code).
2. **Correct `tool_registry.py`'s module docstring**: update the "sole routing authority" claim (lines 4, 18-19) to state it is no longer the runtime routing authority (superseded by `RuntimeToolRegistry` per requirement 04), while clarifying what it *does* still do (e.g., if requirement 09's consolidation still uses its data as one input to drift detection ‚Äî confirm against requirement 09's actual implementation).
3. **Add compatibility framing to `tool_constants.py`'s module docstring**: note its 3 named consumers (`tool_registry.py` seed data, `tool_executor_helpers.py`/`tool_policy.py` fallback classification) are now fallback paths, not primary sources.
4. **Add a doc-comment to `ToolConfig.tool_definitions`** stating its fallback/test-fixture status.
5. **Verify requirements 06-07's fallback comments exist** in `tool_executor_helpers.py`/`tool_policy.py`; add if those plans' implementations didn't already include them.
6. **Resolve the Unknown**: check requirement 03's startup-failure severity model to confirm whether an empty/failed `RuntimeToolRegistry` is already handled as fatal (no static fallback needed) or whether a genuine fallback path exists and needs the same compatibility-only labeling.
7. **Documentation sweep**: `rg -rl "tool_definitions\|tool_constants\|ToolRegistry" docs/` and read each hit to determine whether it needs updating to reflect the new primary-source reality.
8. **Verification**: run the standard validation sequence; since this requirement is primarily documentation/comments, the main check is that requirements 04-09's actual behavior (already validated by their own test suites) is accurately *described*, not that new behavior is introduced.

## Risks

- **Risk**: This requirement's own Scope conflates "redefine as compatibility-only" (a documentation/labeling task) with the actual migration work (a code-behavior task already assigned to requirements 04-09) ‚Äî if implemented out of order (this requirement before 04-09 land), it would produce documentation describing a state the code doesn't yet match.
  **Mitigation**: Implementation Step 1 explicitly gates on requirements 04-09 having landed first; this plan should not be executed prematurely.
- **Risk**: The documentation sweep (Implementation Step 7) has an unbounded exact scope until the grep is actually run ‚Äî could surface more files needing updates than anticipated.
  **Mitigation**: treat any additional finding as in-scope, same-pattern documentation correction, not a reason to abandon the sweep partway through.
- **Risk**: Correcting `tool_registry.py`'s docstring to say it's "no longer the routing authority" could be misleading if requirement 09's consolidation still uses `ToolRegistry` data as an input to some validation check (e.g. drift detection) ‚Äî an overly broad "not authoritative for anything" claim would itself be inaccurate.
  **Mitigation**: Implementation Step 2 explicitly checks requirement 09's actual implementation before finalizing this docstring's wording, to avoid overcorrecting into a new inaccuracy.
