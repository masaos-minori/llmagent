# Plan: Align embedding dimension defaults with production configuration

## Goal

Ensure embedding dimension consistency across config, runtime, and validation layers. Add startup checks to detect mismatches and fix gaps where dimensions are not validated.

## Scope

**In scope:**
- Verify all config defaults are 384 (confirmed: already aligned)
- Add startup check for embedding dimension consistency
- Fix `MemoryStore` instantiation in `factory.py` to pass `embed_dim`
- Add `EmbeddingClient` output dimension validation
- Update diagnostics to show authoritative embedding dimension

**Out of scope:**
- Changing the actual embedding dimension value (384 is correct)
- Adding runtime dimension switching (not supported)
- Modifying embedding model selection logic

## Assumptions

1. All config files and code defaults are already 384 — no mismatch exists in static defaults.
2. The requirement's mention of "768 dataclass default" may refer to a past state that was already fixed.
3. The real issue is lack of validation: embedding server could return wrong dimensions without detection.
4. `MemoryStore()` is instantiated without `embed_dim` in `factory.py:236`, making its dimension validation ineffective.

## Implementation Steps

1. **Fix MemoryStore embed_dim initialization**
   - In `factory.py`: pass `embed_dim=cfg.memory.memory_embed_dim` to `MemoryStore()` constructor
   - Verify `MemoryStore.__init__` accepts `embed_dim` parameter
   - Completion criterion: MemoryStore has non-None _embed_dim, dimension validation active

2. **Add EmbeddingClient output dimension validation**
   - In `embedding_client.py`: after receiving embedding from server, validate `len(embedding) == expected_dims`
   - Raise ValueError with clear message if dimension mismatch
   - Log warning with actual vs expected dimensions
   - Completion criterion: wrong-dimension embeddings are caught at runtime

3. **Add startup dimension consistency check**
   - In `startup.py`: after config loading, verify:
     - `cfg.embedding_dims == cfg.memory.memory_embed_dim` (config consistency)
     - If embedding server has `/health` endpoint, probe it and compare reported dimensions
   - Log error and exit if mismatch detected
   - Completion criterion: startup fails fast on dimension mismatch

4. **Update diagnostics to show authoritative dimension**
   - Add embedding dimension to health/status output
   - Show: `embedding_dims=384` in relevant diagnostic commands
   - Completion criterion: diagnostics reflect real configured dimension

5. **Add tests**
   - Test: MemoryStore receives correct embed_dim from factory
   - Test: EmbeddingClient rejects wrong-dimension embeddings
   - Test: Startup check detects config mismatch
   - Completion criterion: all new validation paths tested
