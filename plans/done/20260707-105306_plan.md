## Goal

Change `/approve` and `/reject` commands to always require an explicit `<approval_id>` argument, removing the latest-pending fallback so that both commands are safe when multiple approvals exist.

## Scope

**In**:
- Change `/approve` handler in `scripts/agent/commands/cmd_workflow.py` to reject the call when `explicit_id is None`, regardless of pending approval count
- Change `/reject` handler to reject the call when `explicit_id is None`, regardless of pending approval count
- Remove the `count == 1` branch that calls `find_latest_pending_approval()` in both handlers
- Remove `find_latest_pending_approval` from `cmd_workflow.py` imports (keep function in `approval_ops.py` — it may be used by startup recovery)
- Update help text docstrings: `[approval_id]` → `<approval_id>` (required)
- Update validation error messages to show the required syntax
- Ensure startup recovery notice (if emitted by `startup._recover_pending_approvals()`) includes the approval ID so operators can copy it
- Remove tests that test ID-less approval with count == 1
- Add tests for: `/approve` without ID → validation error; `/reject` without ID → validation error; multiple pending approvals → correct explicit-ID resolution

**Out**:
- Removing `find_latest_pending_approval()` from `approval_ops.py` (may still be used by startup recovery)
- Changing the `_parse_approval_arg()` return type
- Legacy ID-less approval syntax compatibility
- Migration support

## Assumptions

1. `_parse_approval_arg(arg)` returns `(None, reason)` when no UUID is detected; this remains unchanged.
2. The only change is: when `explicit_id is None`, always return a validation error (not only when `count > 1`).
3. `find_latest_pending_approval()` remains in `approval_ops.py` for startup recovery; only the import from `cmd_workflow.py` is removed.
4. Startup recovery in `startup._recover_pending_approvals()` already displays the approval ID in the notice; if not, it must be added.
