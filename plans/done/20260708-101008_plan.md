## Goal

Remove `/tool list` and `/tool show <id>` built-in commands entirely.
Preserve `/plan` (toggle plan mode). `/help` and tab completion must no longer mention `/tool`.

---

## Scope

### In scope
- Remove `CommandDef("/tool", ...)` from `command_defs_list.py`
- Remove `/tool`-specific code from `cmd_tooling.py`: `_cmd_tool`, `_tool_list`, `_tool_show`, `DecodedArgs`, `_decode_args`, `_to_tool_result_view`
- Remove `/tool`-specific imports from `cmd_tooling.py`: `ToolResultRow`, `ToolResultView`, `MaskedArgs`, `orjson`
- Move or keep `_cmd_plan` — two options (see Design)
- Update `registry.py` comment if mixin is renamed or new mixin introduced
- Update `test_agent_cmd_tooling.py` — remove `/tool` tests; add `/plan` test if not present

### Out of scope
- `ToolResultStore` itself (retained; used by other parts of the system)
- Tab-completion code (driven by `_COMMANDS`; removing the `CommandDef` is sufficient)
- `/help` output (driven by `_COMMANDS`; removing the `CommandDef` is sufficient)

## Unknowns

### U-1: Is `_cmd_plan` test coverage present?
- **Analysis**: `test_agent_cmd_tooling.py` covers `_tool_list`, `_tool_show`, `_cmd_tool` dispatch, and undone display. There is NO test for `_cmd_plan` in this file.
- **Resolution**: After removing `/tool` tests, add a `_cmd_plan` behavior-lock test in the same file (or rename it `test_agent_cmd_plan.py`).

### U-2: Option A vs Option B for `_cmd_plan` placement
- **Option A**: Keep `_cmd_plan` in a slimmed-down `_ToolingMixin` (filename unchanged: `cmd_tooling.py`). Simpler change; fewer files touched.
- **Option B**: Move `_cmd_plan` to a new `cmd_plan.py` (`_PlanMixin`). Cleaner naming; one more file introduced.
- **Resolution (chosen)**: **Option A** — keep `_cmd_plan` in `cmd_tooling.py` (renamed mixin docstring only). Rationale: fewer files, no naming ambiguity, registry.py comment update is trivial.

### U-3: Does `_cmd_tool` appear in any plugin or test fixture outside the test file?
- **Analysis**: `registry.py` references `_ToolingMixin` which exposes `_cmd_tool` via inheritance. Removing `_cmd_tool` and removing `CommandDef("/tool")` are both necessary; the fail-fast validator in `__init__` checks handler names against `_COMMANDS`, so a dangling handler is harmless (fail-fast only checks that listed handlers exist, not the reverse).
- **Resolution**: Removing the `CommandDef` entry is sufficient to remove it from dispatch and help. Removing the implementation cleans dead code.
