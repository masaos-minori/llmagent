# 改善計画書 — 本番コードのスレッドリーク由来「巻き添えkill」不具合 (issue 1-b フォローアップ)

前提: この計画は `plans/20260710-104315_plan.md`（テスト側 `mcp_server` フィクスチャの修正）の続きとして、`issues/20260709_00_issue.md` の 1-b で指摘され、かつ前回の回答で「本計画のスコープ外」と保留した**本番コード側**の問題を対象にする。

## 0. 結論（先出し）

`scripts/agent/http_lifecycle.py:86-124` の `_terminate_with_timeout` が使う `await asyncio.wait_for(asyncio.to_thread(proc.wait), timeout=timeout)` は、**エージェントの通常終了パスで必ず通過する経路**であり、かつ**それを保護する上位のタイムアウト・watchdog・`os._exit()` が一切存在しない**ことを確認した。原因は特定でき、修正方針も決まっている（7節）。

## 1. Goal

MCPサブプロセスの終了処理 (`_terminate_with_timeout`) がバックグラウンドスレッドを一切生成しないようにし、対象プロセスがどれだけ終了に時間がかかっても（あるいは終了しなくても）、エージェント本体のプロセス終了が道連れでハングしない状態にする。

## 2. Scope

**In-Scope:**
- `scripts/agent/http_lifecycle.py` — `_terminate_with_timeout` の実装をスレッド不使用の非同期ポーリング方式に置き換える
- `tests/test_lifecycle.py` — `TestProcessGroupShutdown` クラス（3テスト）を新実装に合わせて書き換え、スレッド不使用であることを直接検証する新規テストを追加する

**Out-of-Scope:**
- `_terminate_with_timeout` のデフォルトタイムアウト値（3.0s/5.0s）自体の変更 — 前回計画（`20260710-104315_plan.md`）のUNK-02で保留した論点であり、本計画でも変更しない
- `subprocess.Popen` ベースの実装を `asyncio.create_subprocess_exec` に全面移行すること — 検討したが不採用（9節参照）
- D-state（割り込み不可能なスリープ）に陥ったプロセス自体をkillする手段の追加 — カーネルレベルの制約でありPythonコードでは解決不能。本計画は「そのプロセスを待つスレッドが道連れでエージェント全体をハングさせない」ことのみを目的とする

## 3. Assumptions

1. `proc.poll()` は非ブロッキングであり（内部で `os.waitpid(pid, os.WNOHANG)` 相当を使用）、対象プロセスがD-stateであっても即座に「まだ終了していない」を返す。ノンブロッキングポーリングに置き換えても、ブロッキングwaitと比べて検出精度が落ちることはない。
2. `time.monotonic()` ベースのデッドリープ判定は `scripts/agent/http_lifecycle.py` の `start()` (L228) で既に同じパターンが使われており、コーディングスタイルとして一貫性がある。

## 6. 不具合メカニズムの詳細（技術的根拠）

1. `_terminate_with_timeout` (`scripts/agent/http_lifecycle.py:104,119`) は `asyncio.wait_for(asyncio.to_thread(proc.wait), timeout=timeout)` を2回（SIGTERM後・SIGKILL後）呼ぶ。`asyncio.to_thread` は同期関数 `proc.wait()`（引数なし＝無期限ブロッキングの `os.waitpid(pid, 0)`）を、イベントループの**デフォルト `ThreadPoolExecutor`** のワーカースレッドで実行する。
2. `asyncio.wait_for` はタイムアウト時にラップしたコルーチン（Task）をキャンセルするが、**`run_in_executor` で既に投入済みの同期呼び出し自体は中断できない** — ワーカースレッドは `proc.wait()` の中でブロックされたまま、独立してバックグラウンドで動き続ける。このため `_terminate_with_timeout` 自体は `except TimeoutError:` で捕捉して正常にreturnする（＝この関数はハングしない）。
3. 問題はその**後**に起きる。CPythonの `concurrent.futures.thread` モジュールは `threading._register_atexit(_python_exit)` によって、インタプリタ終了時に **`_threads_queues` に記録された全ワーカースレッドを無条件に `t.join()` する**（Python 3.13のソースで確認済み。ワーカースレッドはdaemonではない — 3.9以降変更のない既知の標準ライブラリ仕様）。手順2で生き残ったスレッドがまだ `proc.wait()` でブロックされていれば、`t.join()` はそのプロセスが実際に終了するまで（D-stateなどで永久に終了しなければ**無期限に**）ブロックする。
4. `run()` (`repl.py:458`) の `finally` 節が `_close_resources()` → `shutdown_all()` → `_terminate_with_timeout` を経て正常にreturnし、`asyncio.run()` も正常にreturnし、Pythonスクリプト本体の実行が終わった**その後**に、上記3のatexitフックが発火する。ユーザーからは「REPLは終了メッセージを出したのに、プロセスがいつまで経っても終わらない」という形で観測される。
5. この状態で外部の監視機構（シェルの `timeout` コマンド、CI、supervisor等）が「反応がないプロセス」を強制終了しようとして**プロセスグループ単位**でシグナルを送った場合、同じグループに属する他プロセス（issueが懸念する「LLMエージェント」自身、あるいは親シェル）が巻き添えになり得る。これは issue 1-b が指摘していた経路そのものであり、テスト実行時に限らず、**本番の通常終了フローでも理論上発生し得る**ことを確認した。

## 7. Implementation Steps

### Phase 1: `_terminate_with_timeout` をスレッド不使用のポーリング方式に置き換え

- [ ] `HttpServerLifecycleManager` に非公開ヘルパー `_wait_exited(proc, timeout) -> bool` を追加する。`time.monotonic()` によるデッドライン管理と `proc.poll()`（非ブロッキング）＋ `await asyncio.sleep(interval)` のループのみで構成し、`asyncio.to_thread`/`asyncio.wait_for` を一切使わない。
- [ ] `_terminate_with_timeout` (`http_lifecycle.py:86-124`) の2箇所の `await asyncio.wait_for(asyncio.to_thread(proc.wait), timeout=timeout)` 呼び出しを `await self._wait_exited(proc, timeout)` に置き換え、`except TimeoutError:` 分岐を `if not await self._wait_exited(...):` の条件分岐に書き換える。
- [ ] ロジック（SIGTERM→タイムアウト→SIGKILL→タイムアウト→警告ログ）自体は変更しない。挙動の外部インターフェースは維持する。

## 9. Risks & Mitigations

- **Risk**: ポーリング方式への変更により、`proc.poll()` を呼ぶ頻度が上がる（`interval`ごとに1回）。長時間起動したままの多数のMCPサーバーを一斉にshutdownする場合、CPU負荷がわずかに増える可能性がある → **Mitigation**: `interval` を0.05〜0.1s程度に設定し、シャットダウン処理はタイムアウト（3s/5s）以内で完結する短時間の処理であるため、実運用上のCPU負荷増は無視できるレベルと判断する。懸念があれば実装後に `radon`/実測で確認する。
- **Risk**: `asyncio.create_subprocess_exec` への全面移行（本計画では不採用）の方が「本来あるべき非同期ネイティブな設計」だが、`start()` のpgid管理・`start_new_session=True`によるプロセスグループ隔離ロジック全体を書き直す必要があり、変更範囲・リスクが大幅に増える → **Mitigation**: 今回問題になっているのは「スレッドが生き残ること」であり、`proc.poll()`ベースのポーリングだけで根本原因（非デーモンスレッドの残留）を完全に排除できるため、より小さい変更で同等の効果を得られるポーリング方式を採用する。
- **Risk**: D-stateで真に終了しないプロセスが存在する場合、`_wait_exited` はタイムアウトで諦めて警告ログを出すのみで、当該プロセス自体は残り続ける（本計画で解決する範囲外） → **Mitigation**: これはOS/カーネルの制約でありPythonコードでは解決不能。今回の修正は「その残留プロセスを待つスレッドがエージェント全体を巻き添えにしない」ことを保証するものであり、残留プロセス自体の解消は別問題として切り分ける（運用監視・OSレベルの対応が必要）。
- **Risk**: Phase 2のテスト書き換えで、`mod.asyncio.sleep` のno-op化が不十分だと、テストが実際に `interval`×タイムアウト回数分の実待機をしてしまい、テストスイートの実行時間が微増する → **Mitigation**: 実装時に `monkeypatch.setattr(mod.asyncio, "sleep", AsyncMock())` で確実にno-op化し、`uv run pytest tests/test_lifecycle.py -v --durations=10` で実行時間を確認する。
