## Goal

前回セッションで対象外(descope)とした残り2件の後方互換機能 — (A) `github.allowed_repos_mode="fail_open"`、(B) eventbus の `poll_interval_ms` / `offset_checkpoint_interval` no-op フィールド — を、ユーザーが承認した方針(A: `fail_open` モード完全削除、B: 削除しTOMLにキーが残っていたら起動時エラー)に従って除去する。

## Scope

- **In-Scope**
  - (A) `GitHubConfig.allowed_repos_mode` フィールド、`_assert_allowed_repo()` のモード分岐、`ProductionConfigValidator` の関連チェック・`validate_github_fail_open()` メソッド、`GitHubAuditConfig.allowed_repos_mode`、`config/github_mcp_server.toml` の該当キー、関連テスト・ドキュメントの全面書き換え
  - (B) `EventBusConfig.poll_interval_ms` / `offset_checkpoint_interval` フィールドと `DeprecationWarning` ロジックの削除、`load_config()` への起動時バリデーション(該当キーが TOML に残っていた場合 `ValueError`)追加、関連テスト・ドキュメントの更新
- **Out-of-Scope**
  - `allowed_repos`(リポジトリ許可リスト自体)の挙動変更 — fail_closed(空リスト=全拒否)に一本化するのみで、許可リストの判定ロジックそのものは変更しない
  - eventbus の他の設定フィールド(`max_retry`, `host`, `allow_public_bind` 等)
  - `_MCP_TOOLS` 回帰テスト(compat shim ではなく防御的テストのため対象外、既存方針を維持)

## Assumptions

1. `config/github_mcp_server.toml` は現在 `allowed_repos_mode = "fail_closed"` を明示設定済みであり、本リポジトリの実運用設定は `fail_open` に依存していない(確認済み)。
2. `config/eventbus.toml` は現在 `poll_interval_ms` / `offset_checkpoint_interval` を設定しておらず、デフォルト値(500, 10)が使用されている(確認済み)。そのため今回の変更は本リポジトリ自身の設定ファイルには影響しない。
3. `_assert_allowed_repo()` の `mode` 分岐は「`allowed_repos` が空リストの場合」にのみ意味を持つ。`allowed_repos` が非空のテストケースにおける `allowed_repos_mode` キーは実質的に無効(vestigial)であり、削除しても挙動に影響しない。
4. `tests/test_github_mcp_service.py` 内で `{"allowed_repos": [], "allowed_repos_mode": "fail_open"}` を使用している約26箇所のうち、`TestGitHubDryRun` 等の実サービスメソッド呼び出し(例: `svc.fmt_create_branch(...)`)を伴うテストは `_assert_allowed_repo()` を実際に経由するため、`allowed_repos` に当該テストが使用する owner/repo の組を明示的に追加する必要がある。
5. eventbus 側の `poll_interval_ms` / `offset_checkpoint_interval` は `load_config()` 経由でのみ TOML から読み込まれる(`EventBusConfig` を直接構築する呼び出し元はテスト以外に存在しない、confirmed by grep)。
