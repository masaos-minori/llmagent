# Plan: Reduce startup risk around workflow_mode="required" in production defaults

Source: `requires/20260626_06_require.md`

---

## Goal

Make `workflow_mode="required"` failures produce actionable error messages that identify the missing definition, the active mode, and why startup is blocked, and document the local/prod behavior gap clearly.

---

## Scope

**In-Scope**
- Improve the error message in `orchestrator.py` when `WorkflowLoader().load()` fails under `workflow_mode="required"`: include filename expected, mode value, and remediation hint
- Add preflight validation: log the workflow file path and mode at startup before attempting load (INFO level)
- Add/update docs (`05_agent_08_configuration.md`, `05_agent_10_operations-and-observability.md`) to document the local/prod difference and required preflight check
- Add test: `workflow_mode="required"` + missing definition → raises RuntimeError with actionable message

**Out-of-Scope**
- Changing `workflow_mode="required"` production default
- Redesigning workflow mode semantics
- Adding a soft fallback for `workflow_mode="required"` without explicit design approval

---

## Assumptions

1. `Orchestrator.__init__()` is where the failure occurs (confirmed: orchestrator.py:130–136).
2. `WorkflowLoader().load()` raises `WorkflowLoadError` when the definition file is missing (need to confirm the file path it looks for).
3. The error already includes some guidance (`"Check workflow definition file or set workflow_mode=auto"`), but does not identify the expected file path.
4. Adding a pre-load INFO log line (`logger.info("workflow: mode=%s loading from ...", mode, path)`) is sufficient as preflight validation.
