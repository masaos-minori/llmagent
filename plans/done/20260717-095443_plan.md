## Risks

- **Risk**: Reverting `serialize_flags` to `False` changes runtime execution behavior (write-first tool calls with no resource scope now run concurrently instead of sequentially) — if any code added since `12f8e8c9` (2026-07-15) came to depend on the sequential (buggy) behavior, this fix could introduce a new regression.
  **Mitigation**: Implementation Step 4's `rg -n "write_first"` sweep must be run and its output reviewed before applying Step 5's fix, not skipped as a formality — the requirement's own P-1 instructions already anticipated this exact check.
- **Risk**: Updating `test_command_docs_sync.py`'s file list to scan all 11 real files could surface a larger set of stale command references than expected, expanding P-3's scope beyond a single-row fix.
  **Mitigation**: Implementation Step 2 explicitly budgets for this (UNK-02) — treat any additional finding as an in-scope, same-pattern fix (remove/update a stale command reference), not a reason to abandon the file-list correction.
- **Confirmed process risk (no longer hypothetical)**: commit `84b78dfb` resolved two test failures in one commit by changing test expectations rather than verifying the underlying code — for P-2 this was correct (the code being tested, `/db` non-registration, was genuinely right), but for P-1 it was wrong (it re-codified a real regression from `12f8e8c9` as the new expected behavior, and the same day). This is exactly the "bundle multiple changes, one gets it wrong" pattern this requirement's own "Reason for change" section warns about — now confirmed to have happened twice in the same two-day window across two different commits (`12f8e8c9`'s original unrelated 1-line slip, and `84b78dfb`'s uncritical acceptance of it).
  **Mitigation**: report this pattern explicitly to the user alongside the P-1 fix, as a process observation independent of code changes — worth considering whether test-expectation changes should require a comment justifying *why* the new expectation is correct (not just that it matches current code), to prevent a third recurrence.
- **Risk**: Because P-2 is already fixed, marking it "verification only" could cause it to be under-tested if the full-suite run in Step 3 isn't actually inspected carefully (e.g., a shared fixture regression from `84b78dfb` that doesn't manifest as an obvious `test_removed_commands.py` failure).
  **Mitigation**: Step 6's full-suite pass is mandatory regardless of Steps 1-5's individual outcomes, specifically to catch any such collateral effect.
