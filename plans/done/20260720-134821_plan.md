# Plan: Unify routing authority — RuntimeToolRegistry becomes sole routing source

Source requirement: `requires/20260720_01_require.md` (moved to `requires/done/` at Step 7).

## Goal

Remove `ToolRegistry` from `ToolRouteResolver.resolve()`'s resolution path entirely so
`RuntimeToolRegistry` is the sole routing authority, while `ToolRegistry` continues to
exist purely as drift-detection/seed data — and make `ToolExecutor.set_runtime_registry()`
mutate the existing resolver instance instead of replacing it.

## Assumptions

1. `RuntimeToolRegistry` is populated via `set_runtime_registry()` before any real
   user-facing tool call, because `McpToolDiscoveryService.discover_all()` runs in
   `StartupOrchestrator._check_services()` (step 4), which completes before
   `_setup_prompt()` and the REPL input loop begin. Verified by reading
   `scripts/agent/startup.py:227-248`.
2. No production caller passes `known_tools` to `ToolRouteResolver.__init__()` today —
   confirmed by the existing docstring in `route_resolver.py` and by the fact that only
   `scripts/shared/tool_executor.py` constructs `ToolRouteResolver` in production code
   (grep confirmed the only non-test import site).
3. `ToolExecutor` is the sole production consumer of `ToolRouteResolver`; no other module
   constructs it directly.
4. Removing the legacy-fallback branch does not change `resolve()`'s public signature or
   its `ValueError`-based error contract — callers already handle `ValueError` from
   unresolved tools.
5. The 52 tests currently passing in `tests/test_route_resolver.py` and
   `tests/test_runtime_tool_routing_integration.py` (verified via `uv run pytest`) form
   the baseline; a meaningful subset explicitly encodes legacy-fallback behavior and must
   be rewritten (not merely left to fail).

## Design

`ToolRouteResolver.resolve()` becomes:

```python
def resolve(self, tool_name: str) -> str:
    if (key := self._lookup_runtime_registry(tool_name)) is not None:
        return key
    if self._strict_mode:
        self._raise_strict_error(tool_name)
    if self._warn_on_missing:
        logger.warning(
            "ToolRouteResolver: tool %r not found in RuntimeToolRegistry; "
            "ensure MCP servers are healthy and discovery completed.",
            tool_name,
        )
    raise ValueError(f"Unknown tool: {tool_name!r}")
```

`__init__` no longer imports/constructs a `ToolRegistry` instance at all — the
`_legacy_registry` field and its `get_registry()` bootstrap (lines 99-106 today) are
deleted outright, since `ToolRegistry` is no longer consulted anywhere in this class.

`_log_routing_coverage()` only checks `_lookup_runtime_registry()`; the docstring's
"mapped via either registry" language is corrected to "mapped via RuntimeToolRegistry".

`set_runtime_registry()` is added to `ToolRouteResolver` (mutates `self._runtime_registry`
in place) and `ToolExecutor.set_runtime_registry()` is simplified to a single delegating
call — no more resolver reconstruction, no more `discovery_map={}` reset side effect.

`ToolRegistry` (`tool_registry.py`) is untouched structurally; only its docstring is
corrected to remove the now-false "routing fallback" role description. It keeps serving
`shared.tool_routing_validation`'s drift-detection functions.

## Risks

- **Risk: Requirement's own Acceptance Criteria / Test Plan sections contradict its
  Title/Intent/Required-Changes sections** (see UNK-01). → **Mitigation:** This plan
  explicitly documents which sections are authoritative and why (Required-Changes gives
  concrete code; Intent states the goal in plain language; both agree with the document
  title). Implementer should flag this contradiction to the requirement's author when
  the implementation PR is opened, so the requirement doc itself gets corrected upstream.

- **Risk: Removing the `ToolRegistry` fallback eliminates the safety net for the window
  where `RuntimeToolRegistry` is empty/uninitialized** — most importantly, if
  `McpToolDiscoveryService.discover_all()` throws and is caught by `startup.py`'s
  `except Exception: pipeline.add_skipped(...)` (non-fatal), `set_runtime_registry()` is
  never called, `_runtime_registry` stays `None` for the entire session, and **every**
  tool call raises `ValueError` immediately — a full-session outage where today's
  fallback would keep the agent partially functional via `ToolRegistry`. → **Mitigation
  attempted:** none available within this requirement's stated scope, since it explicitly
  forbids modifying `McpToolDiscoveryService` and states "no functional change needed" for
  `startup.py`. → **Residual risk: unmitigated within scope.** Written up in
  `issues/20260720-134821_risks.md` as a follow-up (e.g., consider making
  `mcp_tool_discovery` failures fatal in production security profile, or adding a
  post-startup health check that surfaces "zero routable tools" prominently instead of
  only a non-fatal skip).

- **Risk: Test suite has significant fallback-behavior coverage** (~15 tests in
  `test_route_resolver.py`, ~7 in `test_runtime_tool_routing_integration.py`) that will
  fail or become meaningless once the fallback branch is removed. → **Mitigation:**
  Phase 1 of Implementation Steps requires a full read and explicit rewrite/delete
  decision for each test *before* touching production code, preventing silent coverage
  loss. Diff-cover gate (≥90% on changed lines) in the Validation Plan catches
  under-tested rewrites.

- **Risk: Documentation drift across 8 files describing the old two-tier model** — if
  not all updated consistently, `check-mcp-docs`'s "routing authority language
  consistency" check may fail, or docs may contradict each other. → **Mitigation:**
  Phase 3 explicitly lists all 8 files with line-number anchors found via grep, and the
  validation plan runs `uv run check-mcp-docs` as a gate.

- **Risk: `_log_routing_coverage()` currently has no production caller** (per its own
  docstring — `known_tools` is never passed by `tool_executor.py`), so rewriting it
  carries low direct production risk but the rewrite could still silently diverge from
  `resolve()`'s real behavior if not kept in lockstep. → **Mitigation:** Implementation
  step explicitly rewrites `_log_routing_coverage()` in the same edit as `resolve()`
  (both live in `route_resolver.py`), and existing/rewritten unit tests exercise it
  directly (`test_registry_mapped_tool_is_mapped`-style tests, renamed appropriately).
