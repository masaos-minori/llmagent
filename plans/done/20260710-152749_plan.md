## Goal

Add a pre-flight check at the top of `deploy/setup_services.sh` that verifies the deployed workflow definition (`/opt/llm/config/workflows/default.json`) exists and is valid, and that `workflow.sqlite`'s 5 required tables exist — aborting before any service (Event Bus, LLM, MCP) is started if any check fails.

## Scope

**In-Scope:**
- `deploy/setup_services.sh`: add a pre-flight block, placed before the "LLM サービスのサブプロセス起動" section, that:
  1. Checks `/opt/llm/config/workflows/default.json` exists
  2. Runs the workflow-validation CLI (from `plans/done/20260710-152424_plan.md`'s `scripts/agent/workflow/validate.py`, once implemented) against that deployed path
  3. Checks `/opt/llm/db/workflow.sqlite` exists and its 5 required tables (`tasks`, `attempts`, `processed_events`, `artifacts`, `approvals`) are present
  4. On any failure: print remediation instructions and `exit 1` before the Event Bus (or any other service) is started

**Out-of-Scope:**
- Modifying any workflow definition file — this is a read-only pre-flight check per the requirement's Notes
- Re-implementing the validation CLI or the table-check logic — this plan reuses the exact tools/patterns already designed in `plans/done/20260710-152424_plan.md` (CLI validator) and `plans/done/20260710-152554_plan.md` (table-existence check pattern), applied to the deployed (`/opt/llm/...`) paths instead of the repo-source paths those two plans targeted
- Any change to `deploy/deploy.sh` or `deploy/init_db.sh` themselves — this is a third, independent layer of defense at the service-startup stage, not a modification of the earlier deploy stages

## Assumptions

1. This plan depends on two prior plans in this session having been implemented first: `plans/done/20260710-152424_plan.md` (creates `scripts/agent/workflow/validate.py`) and, implicitly, the existence checks from `plans/done/20260710-152235_plan.md` / `plans/done/20260710-152554_plan.md` (deploy.sh/init_db.sh checks) — those checks should already have caught most failure modes *before* `setup_services.sh` runs. This plan's checks are a deliberate second/third line of defense for the case where `setup_services.sh` is run independently, out of order, or after a manual change to `/opt/llm` post-deploy.
2. The workflow database path is `/opt/llm/db/workflow.sqlite`, matching `scripts/db/config.py`'s `workflow_db_path` default (`"/opt/llm/db/workflow.sqlite"`) and the same path already hardcoded in `deploy/init_db.sh` via `${DEPLOY_DB}/workflow.sqlite` where `DEPLOY_DB="/opt/llm/db"` — `setup_services.sh` should use the same hardcoded convention (not read `common.toml` from bash) for consistency with the existing sibling deploy script, rather than introduce TOML parsing into a shell script for the first time in this deploy pipeline.
3. `scripts/agent/workflow/validate.py` (from the dependency in Assumption 1) is invoked the same way in this script as designed for `deploy.sh`: `PYTHONPATH=/opt/llm/scripts uv run python -m agent.workflow.validate <path>` — using the absolute `/opt/llm/scripts` path (not `${REPO_ROOT}/scripts`) since `setup_services.sh` operates entirely against the deployed tree, unlike `deploy.sh` which operates against the repo checkout.
4. `setup_services.sh` currently has no pre-flight section at all (confirmed by direct read) — the entire block described in Design is new, not a modification of existing logic.

## Design

Insert the following block immediately after the `REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"` line and before the `echo "=== setup_services.sh: サービス設定開始 ==="` banner:

```bash
echo "--- Pre-flight: workflow definition and schema check ---"

WORKFLOW_JSON="/opt/llm/config/workflows/default.json"
WORKFLOW_DB="/opt/llm/db/workflow.sqlite"

if [ ! -f "${WORKFLOW_JSON}" ]; then
  echo "[FATAL] Missing required workflow definition: ${WORKFLOW_JSON}" >&2
  echo "Run deploy/deploy.sh before deploy/setup_services.sh." >&2
  exit 1
fi

if ! PYTHONPATH=/opt/llm/scripts uv run python -m agent.workflow.validate "${WORKFLOW_JSON}"; then
  echo "[FATAL] Deployed workflow definition failed validation: ${WORKFLOW_JSON}" >&2
  echo "Run the workflow schema initialization step before starting the agent." >&2
  exit 1
fi

if [ ! -f "${WORKFLOW_DB}" ]; then
  echo "[FATAL] Workflow database schema is missing or incomplete." >&2
  echo "Run the workflow schema initialization step before starting the agent." >&2
  exit 1
fi

REQUIRED_WORKFLOW_TABLES="tasks attempts processed_events artifacts approvals"
MISSING_TABLES=""
for t in ${REQUIRED_WORKFLOW_TABLES}; do
  FOUND=$(sqlite3 "${WORKFLOW_DB}" \
    "SELECT name FROM sqlite_master WHERE type='table' AND name='${t}';")
  if [ -z "${FOUND}" ]; then
    MISSING_TABLES="${MISSING_TABLES} ${t}"
  fi
done
if [ -n "${MISSING_TABLES}" ]; then
  echo "[FATAL] Workflow database schema is missing or incomplete." >&2
  echo "Missing table(s):${MISSING_TABLES}" >&2
  echo "Run the workflow schema initialization step before starting the agent." >&2
  exit 1
fi

echo "OK: workflow definition and schema pre-flight checks passed"
echo ""
```

This block runs entirely before the existing "LLM サービスのサブプロセス起動" section, so a failure here occurs strictly before the Event Bus (or anything else) is spawned — satisfying "Stop setup before starting Event Bus or other services if validation fails."

   - [ ] Restore the DB file; if possible, test against a scratch DB missing one required table and confirm the per-table check names the missing table and exits 1.

## Validation plan

| Check | Tool | Target |
|---|---|---|
| Shell syntax | `bash -n deploy/setup_services.sh` | 0 errors |
| Positive test | `bash deploy/setup_services.sh` with valid deployed artifacts | Pre-flight prints "OK: ...", Event Bus and other services start as before |
| Negative test: missing definition | Rename `/opt/llm/config/workflows/default.json` away, run the script | Exits 1 before any service starts; no new `eventbus.app` process |
| Negative test: invalid definition | Corrupt the deployed JSON's required fields, run the script | Exits 1 via the validation-CLI check, before any service starts |
| Negative test: missing DB | Rename `/opt/llm/db/workflow.sqlite` away, run the script | Exits 1 via the DB-existence check |
| Negative test: incomplete schema | Scratch DB missing a required table, run the script (adjust `WORKFLOW_DB` path for the test) | Exits 1, names the missing table |
| No partial service startup | `ps aux` before/after each negative test | No Event Bus, LLM, or MCP process appears after a failed pre-flight check |

| Risk | Likelihood | Mitigation |
|---|---|---|
| Negative tests performed directly against the real `/opt/llm` deployment could leave it in a broken state if not restored correctly | Medium | Always restore the renamed/corrupted file immediately after each negative test observation, and re-run the positive-path test at the end to confirm the environment is back to a working state |
