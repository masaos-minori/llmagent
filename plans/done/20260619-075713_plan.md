# Plan: Separate diagnostic data more clearly from normal conversation history

## Goal

Move synthetic error messages and guard hints out of conversation history into the diagnostic channel, keeping user-facing history clean while preserving debugging and replay capability.

## Scope

**In scope:**
- `ErrorInjectionService.inject_mid_turn_error()`: stop injecting synthetic tool messages into `ctx.conv.history`; instead store in `DiagnosticStore` with kind `"mid_turn_error"`
- `ToolLoopGuard`: stop injecting CYCLE_HINT/DEDUP_HINT user messages into history; instead store in `DiagnosticStore` with kind `"guard_hint"`
- `LLMTurnRunner._handle_llm_error()`: update to use diagnostic-only error reporting
- `Orchestrator._handle_llm_transport_error()`: keep existing partial completion behavior (already diagnostic-only); add mid-turn error consolidation
- Ensure replay works: diagnostic store entries are available for session restoration if needed
- Update history export (`export_formatter.py`) to not include diagnostic hints (already filtered by role check)

**Out of scope:**
- Changing the DiagnosticStore schema (use existing `kind` field for categorization)
- Modifying compression behavior for diagnostic entries (they won't be in history anymore)
- Adding new CLI commands for viewing diagnostics (covered by `/tool show` and `/audit`)
- Memory/notes injection separation (system-level injections are intentional context, not diagnostics)

## Assumptions

1. `DiagnosticStore` already supports arbitrary `kind` strings — we will use `"mid_turn_error"` and `"guard_hint"` as new kind values.
2. The synthetic error message in history currently serves two purposes: (a) inform the LLM of the error for recovery, (b) preserve the error for debugging. Purpose (a) can be handled by returning the error in `TurnResult.answer` without injecting into history.
3. Guard hints in history currently influence LLM behavior on subsequent tool calls. Without them in history, the guard logic itself (cycle detection, dedup) still prevents the problematic behavior — so hints are informational, not functional.
4. Session restoration from DB only loads `messages` table entries — diagnostic entries are not restored into history on reconnect. This is acceptable because diagnostics are session-scoped, not persistent across sessions.

## Unknowns

| Unknown | Evidence missing | Resolution | Blocking |
|---|---|---|---|
| Whether removing guard hints from history breaks cycle/dedup recovery behavior | Guard hints are informational; the guard itself prevents the bad behavior | Test: remove hints, verify cycle detection and dedup still fire correctly via unit tests | no |
| Whether mid-turn errors need to be visible in history for LLM recovery | TurnResult.answer already carries error info to orchestrator; LLM gets a fresh turn | If LLM needs context about the error, orchestrator can pass it as a system message on the next turn (not injected into persistent history) | no |

## Implementation Steps

1. **Modify `ErrorInjectionService.inject_mid_turn_error()`**
   - Remove: `ctx.conv.history.append(...)` line that injects synthetic tool message
   - Add: `ctx.diagnostics.save(kind="mid_turn_error", content={...})` with error details (error_type, detail, turn_id)
   - Return error info in a structured format that `LLMTurnRunner` can use
   - Completion criterion: mid-turn errors are stored in diagnostic store only; history is not modified

2. **Modify `ToolLoopGuard` hint injection**
   - Remove: `ctx.conv.history.append(...)` calls for CYCLE_HINT, DEDUP_HINT in `check_cycle()`, `check_dedup()`, `check_retry()`
   - Add: `ctx.diagnostics.save(kind="guard_hint", content={...})` with guard type and context
   - Completion criterion: guard events are stored in diagnostic store; history is not modified

3. **Update `LLMTurnRunner._handle_llm_error()`**
   - Instead of calling `ErrorInjectionService.inject_mid_turn_error()`, call a new `report_error()` method that stores in diagnostics and returns error info
   - Set `TurnResult.answer` to include error summary for orchestrator visibility
   - Completion criterion: LLM errors are reported via diagnostics; turn result carries error context

4. **Update `Orchestrator` error handling**
   - When receiving `TurnResult(action="fail")` with error info, optionally inject a non-persistent system message (not appended to history, not persisted) for LLM recovery on the next turn
   - Consolidate mid-turn errors: if multiple errors occur in one turn, store as single diagnostic entry with error chain
   - Completion criterion: orchestrator handles diagnostic-only errors; recovery context is available without polluting history

5. **Update tests**
   - `test_error_injection.py`: verify no history modification on error; verify diagnostic store entry created
   - `test_tool_loop_guard.py`: verify no history modification on guard fire; verify diagnostic store entry created
   - Add integration test: full turn with mid-turn error → verify history is clean, diagnostics contain error

6. **Verify history export and display**
   - `export_formatter.py`: already filters by role, so synthetic messages won't appear in Markdown export — confirm no changes needed
   - `/history` command: already filters to user/assistant roles only — guard hints (if they were user-role) would have been hidden anyway

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| LLM loses context about mid-turn errors and cannot recover | low | Orchestrator passes error summary in TurnResult.answer; next turn can include non-persistent system context |
| Guard hints removal changes LLM behavior on subsequent tool calls | med | Unit tests verify cycle detection and dedup still work; if LLM behavior degrades, add back as non-persistent context |
| DiagnosticStore entries grow with error/hint data | low | Diagnostics are per-session; session end cleanup already handles retention |
| Existing scripts/grep pipelines that parse history for errors break | low | Error messages were synthetic and hard to distinguish; moving to structured diagnostic store improves parseability |
