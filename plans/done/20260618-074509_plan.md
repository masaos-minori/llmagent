## Goal

Workflow/run becomes a first-class execution model where workflow, task, retry, approval, and artifact states are explicitly managed and persisted, separate from conversation history.

## Scope

**In:**
- Workflow state model definition (session-scoped workflow context)
- Task state model (task CRUD + status transitions)
- Retry state (attempt tracking + policy enforcement)
- Approval state (approval gate lifecycle)
- Artifact references as part of workflow state
- `workflow.sqlite` schema (tables already exist: `tasks`, `attempts`, `processed_events`, `artifacts`, `approvals`)
- `ctx.workflow` sub-structure in `AgentContext` for runtime workflow state
- Orchestrator integration to use workflow engine as primary execution path

**Out:**
- No new DB files (uses existing `workflow.sqlite`)
- No new MCP servers
- No schema migration beyond adding `workflow_id` if needed to `tasks`
- No workflow CLI/REPL commands (existing `/approve` and `/reject` suffice)
- No changes to `ConversationState` history management
- No new `WorkflowDef` JSON workflow definitions (design only)

## Assumptions

1. `workflow.sqlite` tables (`tasks`, `attempts`, `processed_events`, `artifacts`, `approvals`) are already created and match `db/workflow_schema.py`.
2. `StateStore`, `WorkflowEngine`, `WorkflowLoader` in `agent/workflow/` are functional and test-covered.
3. The existing `ctx.turn.pending_approval_id` field is sufficient to bridge workflow suspension into the turn lifecycle.
4. `workflow_mode` parameter on `Orchestrator.__init__` (values: `"auto"`, `"disabled"`, `"required"`) is the correct extension point.
5. No other concurrent agents will modify the same `workflow.sqlite` rows (single-process assumption).

## Design

### WorkflowState model (new dataclass in `context.py`)

```python
@dataclass
class WorkflowState:
    active: bool = False
    current_task_id: str | None = None
    current_workflow_version: str | None = None
    approval_pending: bool = False
    # session_id last used to create a task; None = no task created this session
    last_session_id: str | None = None
```

Added to `AgentContext.__init__()`:

```python
self.workflow = WorkflowState()
```

### Canonical state model

| State | Scope | Location | Persisted |
|---|---|---|---|
| WorkflowState.active | session | `ctx.workflow.active` | No (transient) |
| WorkflowState.current_task_id | session | `ctx.workflow.current_task_id` | No (transient; task_id lives in workflow.sqlite) |
| TaskRecord | task | `workflow.sqlite.tasks` | Yes |
| AttemptRecord | attempt | `workflow.sqlite.attempts` | Yes |
| ApprovalRecord | approval | `workflow.sqlite.approvals` | Yes |
| ArtifactRef | artifact | `workflow.sqlite.artifacts` | Yes |
| RetryPolicy | workflow def | `config/workflows/*.json` (via `WorkflowDef.retry_policy`) | Yes (file) |

### Orchestrator integration

The fallback paths in `Orchestrator.handle_turn()` (lines 108–127) would be removed. Workflow engine runs every turn:

```
handle_turn(line)
  → if workflow_mode == "disabled": direct execution (preserved as opt-out)
  → else:
       store = StateStore()
       task = store.create_task(session_id, turn_number, workflow_version)
       ctx.workflow.current_task_id = task.task_id
       ctx.workflow.active = True
       engine = WorkflowEngine(wdef, store, tracer=tracer)
       await engine.run(task, plan_fn, execute_fn, verify_fn)
       ctx.workflow.active = False
       ctx.workflow.current_task_id = None
```

### Retry state integration

`AttemptRecord` already captures per-attempt status (`running`, `completed`, `failed`, `error_msg`). No new state needed. The retry policy from `WorkflowDef.retry_policy` drives `max_attempts`.

## Implementation Steps

1. **Add `WorkflowState` dataclass** to `scripts/agent/context.py` — new nested dataclass with `active`, `current_task_id`, `current_workflow_version`, `approval_pending`, `last_session_id` fields; wire into `AgentContext.__init__()`.
2. **Add query methods to `StateStore`** — `get_task_by_session(session_id)`, `get_latest_task(session_id)`, `list_tasks(limit)` in `scripts/agent/workflow/state_store.py`.

5. **Update docs** — `05_agent_02_runtime-architecture.md` (add WorkflowState to component diagram), `05_agent_04_state-and-persistence.md` (add WorkflowState table, update user-facing description).
6. **Run validation** — lint, typecheck, arch check, test.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Removing fallback paths in `Orchestrator.handle_turn()` breaks workflows when `workflow.sqlite` is unavailable | Low (already gated by `workflow_mode`) | Keep `workflow_mode="disabled"` path; add explicit try/except around `StateStore()` construction |
| `StateStore()` opens connection at construction (not lazy) — resource leak if not closed | Medium | Use `contextlib.closing()` pattern in `Orchestrator.handle_turn()` |
| `session_id` type mismatch (TEXT in workflow.sqlite vs INTEGER in session.sqlite) causes incorrect correlations | Low | `StateStore.create_task()` stores `str(session_id)` — verify cast is sufficient |
| Adding `WorkflowState` to `AgentContext` requires updating all call sites that iterate `ctx` fields | Low | `WorkflowState` is a single field; no iteration over ctx fields exists in the codebase |
