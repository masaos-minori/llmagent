# Work Plan for Memory Layer Refactoring

## Goal
Refactor the memory layer to remove backward-compatibility features, standardize public APIs, centralize SQLite persistence, and unify retrieval scoring configuration.

## Scope
- Remove all backward-compatibility features from memory layer components
- Standardize public API boundaries across memory layer components
- Centralize SQLite persistence responsibilities in the store layer
- Unify all retrieval scoring configuration under `ScoringPolicy`
- Refactor specific files: `layer.py`, `types.py`, `retriever.py`, `scoring.py`, `ingestion.py`, `store.py`, `embedding_client.py`, `injection.py`, `extract.py`

## Assumptions
- The system has Python 3.13 installed with required dependencies
- The memory layer components are currently functional
- All existing tests and integrations are aware of the upcoming changes
- The refactoring can be done in a specific priority order

## Unknowns
- Specific details of how each component currently works and what backward-compatibility features exist
- Exact behavior changes that will occur with each refactoring step
- Impact on existing tests and integrations
- Whether all backward-compatibility features are properly identified

## Affected areas
- Memory layer components: `agent/memory/layer.py`, `agent/memory/types.py`, `agent/memory/retriever.py`, `agent/memory/scoring.py`, `agent/memory/ingestion.py`, `agent/memory/store.py`, `agent/memory/embedding_client.py`, `agent/memory/injection.py`, `agent/memory/extract.py`
- Tests and integrations that may depend on backward-compatibility features

## Design
- Follow the specified priority order for refactoring
- Remove backward-compatibility features systematically
- Standardize APIs to prevent internal attribute access
- Centralize SQLite operations in the store layer
- Unify scoring configuration under `ScoringPolicy`
- Ensure all changes are properly documented
