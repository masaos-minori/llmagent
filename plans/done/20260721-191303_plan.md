# Plan: Restrict approval dry-run tools to dry-run-capable tools

## Goal

Prevent dry-run execution for tools that don't support the `dry_run` parameter, avoiding false denials for HIGH-risk tools during approval preview.

## Scope

- `scripts/agent/tool_approval.py` — add dry-run capability check before injecting `dry_run=True`
- `scripts/agent/config_builders.py` — validate `approval_dry_run_tools` at startup, warn on unsupported entries
- `scripts/agent/config_dataclass.py` — may need validation addition
- `tests/test_tool_approval_preflight.py` — add/update tests for dry-run capability filtering

## Assumptions

- Tool capabilities can be determined by checking against a known set of dry-run-capable tools
- The default dry-run tools list (`write_file`, `edit_file`, `create_directory`, `delete_file`, `delete_directory`, `move_file`) is authoritative
- Custom dry-run tools added via config should be validated against known capabilities

## Unknowns

Resolved — all unknowns answered by code inspection:

1. Dry-run is currently controlled by `approval_dry_run_tools` list membership (line 64 of `tool_approval.py`). Tools not in the list skip dry-run entirely.
2. `dry_run=True` is injected at line 72: `{**args, "dry_run": True}` before calling `ctx.services_required.tools.execute()`.
3. Dry-run failures raise `ApprovalPreviewBlockingError` for HIGH risk (denied outright) or `ApprovalPreviewError` for MEDIUM risk (fallback to text-only preview).
4. `build_preview()` exists in `tool_result_formatter` and is called as fallback.
5. Default dry-run tools are defined in both `config_builders.py` (line 107-114) and `config_dataclasses.py` (line 331-340) — consistent but no validation.

## Affected areas

- Approval preview mechanism — dry-run restricted to capable tools only
- HIGH-risk tool denial logic — no longer triggered by missing dry-run support
- Configuration validation — invalid dry-run tool names reported at startup

## Design

### Approach: Whitelist-based dry-run capability

Instead of relying solely on `approval_dry_run_tools` list membership, add a second guard: check whether the tool actually supports `dry_run` before executing it.

```python
# In _build_preview_with_dry_run():
if tool_name not in ctx.cfg.approval.approval_dry_run_tools:
    return build_preview(tool_name, args)

if ctx.services_required.tools is None:
    return build_preview(tool_name, args)

# NEW: Check dry-run capability
if not _is_dry_run_capable(tool_name):
    logger.warning("Tool %s does not support dry_run; using static preview", tool_name)
    return build_preview(tool_name, args)

# Existing dry-run injection
try:
    result = await ctx.services_required.tools.execute(..., dry_run=True)
    ...
except ApprovalPreviewBlockingError:
    ...
```

### Dry-run capability determination

Option A: Hardcoded whitelist matching defaults + known additions. Simplest, most reliable.

Option B: Dynamic capability discovery via introspection. Complex, fragile.

Recommend Option A — explicit is better than implicit.

### Startup validation

When `approval_dry_run_tools` contains entries not in the dry-run-capable whitelist, log a warning and exclude them. This avoids breaking startup while alerting operators.

```python
def _validate_dry_run_tools(tools: list[str]) -> list[str]:
    """Validate and filter dry-run tools, returning only supported ones."""
    supported = {t for t in _DEFAULT_DRY_RUN_TOOLS} | _KNOWN_DRY_RUN_CAPABLE_TOOLS
    filtered = []
    for tool in tools:
        if tool not in supported:
            logger.warning("Dry-run tool '%s' is not supported; ignoring", tool)
        else:
            filtered.append(tool)
    return filtered
```

## Implementation steps

### Step 1: Define dry-run capable tool set

Add `_DRY_RUN_CAPABLE_TOOLS` constant in `tool_approval.py` containing all known dry-run-capable tool names. Include current defaults plus any other tools confirmed to support `dry_run`.

### Step 2: Add capability check in `_build_preview_with_dry_run()`

Before injecting `dry_run=True`, check if the tool is in the dry-run-capable set. If not, fall through to `build_preview()`.

### Step 3: Add startup validation for `approval_dry_run_tools`

In `_build_approval_config()`, after loading `approval_dry_run_tools`, validate against the capable set and warn on unsupported entries.

### Step 4: Update HIGH-risk denial logic

Ensure HIGH-risk denial only triggers for meaningful dry-run blocking results, not for unsupported dry-run arguments.

### Step 5: Add/update tests

In `tests/test_tool_approval_preflight.py`:
- Supported dry-run tool appends dry-run output to preview
- Unsupported dry-run tool uses static preview (no `tools.execute()` call with `dry_run=True`)
- HIGH-risk unsupported tool is NOT denied due to unsupported dry-run
- HIGH-risk supported tool with real dry-run blocking error IS denied

### Step 6: Run validation

- `uv run pytest` — ensure no regressions
- `uv run mypy scripts/` — type check passes
- `uv run ruff check scripts/` — lint clean

## Risks

### Risk 1: Missing dry-run-capable tool entries

If new tools gain `dry_run` support but aren't added to the whitelist, they'll use static preview instead. Mitigation: document the whitelist clearly; review when adding new tools.

### Risk 2: False positives in startup validation

If a custom tool supports `dry_run` but isn't in the whitelist, operators will see warnings. Mitigation: allow operators to override via a strict mode flag if needed.

### Risk 3: Change in HIGH-risk denial behavior

Previously, HIGH-risk tools without dry-run support could be denied if dry-run returned an error. Now they won't be denied solely due to missing dry-run support. This is intentional per requirements but represents a behavioral change.
