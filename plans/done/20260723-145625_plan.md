## Goal

Wrap the memory injection call in `StartupOrchestrator._setup_prompt()` in a try/except block so that memory failures do not prevent the agent from starting.

## Scope

**In:**
- `scripts/agent/startup.py`: Add try/except around `ctx.services_required.memory.on_session_start()` in `_setup_prompt()`

**Out:**
- Modifying `session.start()` behavior
- Changing the memory layer itself
- Any other file modifications

## Assumptions

1. The memory layer is explicitly optional — disabled via `use_memory_layer=False` means no memory functionality at all.
2. On failure, the agent should continue with the base system prompt (without memory snippets) rather than failing hard.
3. The warning should use `OutputTag.NON_FATAL` to distinguish from fatal errors.
4. The existing `logger.warning()` call for truncated snippets (line 355) provides an acceptable logging pattern to follow.

## Design decisions

- Catch all exceptions (not specific types) since the memory layer can raise various errors (DB connection, network timeout, etc.) and we want to handle them uniformly.
- Use `logger.warning()` for structured logging and `self._view.write_warning(OutputTag.NON_FATAL)` for user-facing output to ensure visibility.

## Alternatives considered

- Catch specific exception types (e.g., `sqlite3.Error`, `ConnectionRefusedError`): would require maintaining the list as the memory layer evolves; catching `Exception` is simpler and safer.
- Return a boolean flag from `_setup_prompt()` to indicate success/failure: would complicate the caller's control flow unnecessarily.

## Risks & Mitigations

- **Risk**: Operator does not notice the warning because it appears among normal startup output → **Mitigation**: Use `self._view.write_warning()` which produces a distinct visual format compared to regular info logs. Document the expected warning message in operations docs.
- **Risk**: Memory state inconsistency after partial startup → **Mitigation**: The memory injection happens after session creation but before tool execution. If injection fails, the session is valid but has no injected memories — this is equivalent to having memory disabled.
