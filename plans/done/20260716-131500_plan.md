# Plan: Simplify MDQ search configuration and remove unsupported hybrid search placeholders

Source requirement: `requires/20260716_04_require.md`

## Goal

`search_docs` only advertises and accepts search modes that actually run
different logic (`bm25`), MDQ config exposes exactly one canonical
result-limit key (`max_results_limit`), and no embedding/hybrid-search
scaffolding remains in config, service, schema, or search code — since
`_search_vector()` has always returned an empty list (a permanent stub, not
work in progress) and `mode="hybrid"`/`mode="grep"` were never functionally
different from the default FTS5 path.

## Scope

  `Literal["bm25"]` (see Design for why `"bm25"` is the only supported value,

- `scripts/mcp_servers/mdq/tools.py`: change `search_docs.mode` description

  supported" wording (the requirement's own removal target).

  `_search_vector()`, `_merge_hybrid()`, and the `_RRF_K` constant — all code

- `scripts/mcp_servers/mdq/service.py`: remove `self.use_embedding`,
  `self.vector_table`, `self.embedding_model`, `self.embedding_dims` and the

- Add `tests/test_mdq_search_modes.py` (new — the requirement's
  `tests/test_mdq_search*.py` target does not exist yet) covering: default
  mode behavior, `mode="bm25"` explicit, unsupported mode rejected with a

- Update docs: `docs/04_mcp_04_04_mdq.md` (drop

## Assumptions

1. `_search_vector()` in `search.py` is a permanent, explicitly-labeled
   placeholder ("Placeholder: actual embedding generation and search would
   require an LLM API call ... For now, return empty results") — not
   in-progress work — so removing the `mode="hybrid"` path loses no working
   functionality. Confirmed by direct read of `search.py:150-157`.
2. `self.max_search_results` in `service.py` has no reader anywhere in the
   codebase besides its own assignment — verified via
   `rg -n "max_search_results" scripts/ tests/`. `self.max_results_limit` is
   the value actually enforced in `search.py::search_docs`
   (`getattr(req, "max_results_limit", None)` / `service.max_results_limit`).
3. `mcp_servers.mdq.models.EmbeddingResult` (a `TypedDict`, unrelated to the
   same-named `agent.memory.types.EmbeddingResult` dataclass used elsewhere)
   has zero importers anywhere in `scripts/` or `tests/` — verified via
   `rg -n "from mcp_servers.mdq.models import"` across the repo; no result
   imports `EmbeddingResult` from this module.
4. **Resolved deviation from the requirement's "Suggested supported modes"
   list** (`bm25`, `grep`): `search.py` never implements a `mode="grep"`
   branch — any `mode` value other than `"hybrid"` (including `"grep"` or
   garbage strings) silently falls through to the default FTS5 path today.
   Regex-based search already exists as the fully separate `grep_docs` tool
   with its own dedicated implementation (`db_grep.grep_docs`). Introducing a
   `mode="grep"` alias inside `search_docs` that internally called
   `grep_docs` logic would be new functionality, not cleanup, and is not
   requested elsewhere in this batch. This plan therefore restricts
   `SearchDocsRequest.mode` to `Literal["bm25"]` only, and the tools.py
   description is updated to match — not to `bm25`/`grep`. This is called
   out explicitly since it differs from the requirement text's suggestion;
   flagged as a non-blocking judgment call, not a blocking unknown (see
   Unknowns).

## Design

Search mode is restricted to a `Literal["bm25"]` type on
`SearchDocsRequest.mode` (pydantic `BaseModel`), so an unsupported value
raises a `pydantic.ValidationError` automatically at request-parsing time —
satisfying "Reject unsupported modes with a validation error" without new
custom validation code. `search.py`'s `mode = getattr(req, "mode", None) or
"bm25"` line becomes unnecessary (the field will always be `"bm25"` or
absent, defaulting via the model) but the fallback is kept for defensive
clarity since `mode` remains optional.

`max_results_limit` request-override-capped-by-config behavior is unchanged
— it is already implemented correctly in `search.py:36-41`; this plan only
removes the unused `max_search_results` duplicate, so no behavioral change
to the enforced path, only removal of dead config surface.

## Implementation steps

1. **Restrict search mode** — change `SearchDocsRequest.mode` to
   `Literal["bm25"] | None = "bm25"` in `models.py`; update `tools.py`
   description text.
2. **Remove hybrid code path** — delete the `mode == "hybrid"` branch,
   `_search_vector()`, `_merge_hybrid()`, `_RRF_K` from `search.py`.
3. **Remove embedding config surface** — `service.py` field removals;
   `db_schema.py` parameter and vec0-block removal; `config/mdq_mcp_server.toml`
   key removals + note.
4. **Remove duplicate result-limit config** — delete
   `self.max_search_results` from `service.py`; delete `max_search_results`
   from `config/mdq_mcp_server.toml`.
5. **Check for a pre-existing `chunks_vec` table** — run
   `sqlite3 /opt/llm/db/mdq.sqlite ".tables"` in the deploy environment (or
   confirm via docs/ops) to record whether any live database actually has
   the vector table, and note the finding in the commit message.
6. **Remove obsolete tests, add new ones** — delete
   `tests/test_mdq_hybrid_search.py`; add `tests/test_mdq_search_modes.py`.
7. **Update documentation** — the two doc files listed in Affected areas.
8. **Deployment/verification** — no deploy.sh change; restart `mdq-mcp` in
   dev and run `search_docs` with `mode="bm25"` (should work) and an invalid
   mode (should be rejected) to confirm end-to-end behavior.

## Risks

- **Risk**: A production `mdq.sqlite` database might already contain a
  `chunks_vec` virtual table from when `use_embedding=true` was previously
  enabled somewhere; removing the creation code doesn't remove the table,
  potentially leaving orphaned schema.
  **Mitigation**: Implementation step 5 explicitly checks for this before
  the change is deployed; `config/mdq_mcp_server.toml`'s current default
  (`use_embedding = false`) makes this unlikely but not impossible.
- **Risk**: Restricting `mode` to `Literal["bm25"]` instead of the
  requirement's suggested `bm25`/`grep` could be seen as not fully
  implementing the requirement's task list.
  **Mitigation**: documented explicitly in Assumption 4 and the Unknowns
  table with the evidence-based rationale; a reviewer can request a `grep`
  alias be added as a follow-up if actually wanted.
- **Risk**: Overlap with `requires/20260716_01_require.md` (also touches
  `max_search_results`/`max_results_limit` and embedding fields in its
  original task list) if both are implemented without respecting the
  deferral this plan and plan 01 both document.
  **Mitigation**: plan 01 explicitly defers these exact items to this
  requirement (04); implementing in filename order avoids conflicts.
