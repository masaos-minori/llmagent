### Additional Finding: Double-truncation in `_process_stdio_line`

`_handle_stdio_request` (line 350) already calls `_truncate_with_meta(dispatch_result.output)` and stores `tr.text` as `result`.
Then `_process_stdio_line` (line 279) calls `_truncate_with_meta(result)` **again** on the already-truncated text.
The second call captures `truncated`, `total_bytes`, `actual_visible_bytes` but discards the new `tr.text`.
For the response payload, `result` (which already has the truncation suffix) is used, but `truncated`, `total_bytes`,
`actual_visible_bytes` come from the second call — which measures the suffix-appended text, not the original.
This is a minor metadata inaccuracy: `total_bytes` in the second call includes the suffix length.
However, since the second call's `truncated` will be `False` for a normally truncated result (the suffix makes the text > max_bytes),
the metadata fields sent in `_write_stdio_response` are `truncated=False, total_bytes=<suffix-included>, actual_visible_bytes=<suffix-included>`.
This is a real inconsistency that should be fixed by removing the double-truncation.

## 8. Risks & Mitigations
- **Risk**: Refactoring `_StdioRequestResult` to carry truncation metadata may miss the error path (is_error=True) → **Mitigation**: Error responses should have `truncated=False`, `total_bytes=0`, `actual_visible_bytes=0`; keep existing defaults and only populate from `TruncationResult` on non-error dispatch
- **Risk**: Double-truncation fix may alter the `result` text if `_process_stdio_line` was accidentally relying on the second `_truncate_with_meta` call → **Mitigation**: The second call's `tr.text` is already discarded in current code; removing the call changes only metadata, not `result`
- **Risk**: Existing tests for `TestRunStdio` may not assert `truncated`/`actual_visible_bytes` fields → **Mitigation**: Add explicit assertions in new tests; existing tests remain valid as they test non-truncation path
