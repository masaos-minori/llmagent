# Plan: Make fail-open and fail-closed security settings visible at startup

## Goal

Show fail-open vs fail-closed behavior clearly at startup, expose important empty-list behaviors in `/mcp`, and improve operator-facing documentation for security defaults.

## Scope

**In scope:**
- Startup security summary output
- `/mcp` status display
- Security documentation
- `scripts/agent/repl_health.py` — `audit_security_defaults()` function

**Out of scope:**
- Changing the actual security defaults (fail-open vs fail-closed behavior is intentional per-server)
- Modifying individual MCP server configs
- Changes to allowlist/denylist enforcement logic

## Assumptions

1. The `audit_security_defaults()` function in `repl_health.py` already checks several security settings (auth_token, shell sandbox, GitHub workflow allowlist, allowed_tools) but does not explicitly categorize them as fail-open or fail-closed.
2. Operators assume all allowlists behave the same way (deny when empty), but some are fail-open (allow when empty) — this is the core risk.
3. The current startup logs show individual warnings but do not provide a summary of security posture.

## Implementation Steps

1. **Add fail-open/fail-closed summary to startup output**
   - In `audit_security_defaults()`, after collecting all warnings, print a summary:
     ```
     Security posture:
       Fail-closed settings (deny when empty): db_allowlist=empty, command_allowlist=empty
       Fail-open settings (allow when empty): workflow_allowlist=empty, allowed_workflows=empty
     ```
   - Completion criterion: Startup logs include security posture summary

2. **Expand `audit_security_defaults()` checks**
   - Add checks for: db_allowlist (sqlite-mcp), command_allowlist (shell-mcp), allowed_repo_paths (git-mcp)
   - Classify each as fail-closed or fail-open in the warning message
   - Completion criterion: All critical allowlist settings are checked at startup

3. **Expose security defaults in `/mcp` status**
   - If `/mcp` endpoint aggregates server health, include a `security_defaults` section
   - Show fail-open/fail-closed status for each server's key settings
   - Completion criterion: /mcp output includes security defaults summary

4. **Document security defaults in ops docs**
   - Add a section explaining fail-open vs fail-closed behavior
   - List each server's default behavior clearly
   - Highlight which settings are dangerous when left empty
   - Completion criterion: Docs clearly distinguish fail-open from fail-closed
