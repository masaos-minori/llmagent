## 3. Assumptions

1. The state transition flow specified in the require file is correct: HEALTHY → DEGRADED → UNAVAILABLE → HALF_OPEN → HEALTHY. This is confirmed by `scripts/shared/mcp_config.py` implementation (`record_failure()` lines 111–125).
2. The default `failure_threshold=3` means 1–2 failures land in DEGRADED, 3+ failures trigger UNAVAILABLE. The diagram can use threshold-based wording ("failure × 1..N-1" for DEGRADED, "failure × threshold" for UNAVAILABLE) rather than exact counts.
3. Both watchdog success and per-call success flow through `record_success()`, which transitions any state → HEALTHY. The diagram can show a single `record_success()` arc from HALF_OPEN → HEALTHY without distinguishing the caller.
4. `docs/04_mcp_90_inconsistencies_and_known_issues.md` does not currently have a "Resolved" section; one will be added.
