# Plan: Expose RuntimeToolRegistry status in MCP diagnostics

Source requirement: `requires/20260717_11_require.md`

## Goal

Operators can inspect `RuntimeToolRegistry`'s contents (discovered tools, per-tool policy annotations) via CLI diagnostics ŌĆö a new `/mcp tools` subcommand family shows `server_key`, tool name, `enabled_for_llm`, `agent_safety_tier`, `requires_approval`, `is_write`, `requires_serial` per tool, including disabled/policy-filtered tools, with output concise and safe for logs.

## Scope

**In scope**
- `scripts/agent/commands/cmd_mcp.py`: `_cmd_mcp()` (currently only accepts `"status"` as a subcommand, per direct read of line 165-171) gains a new `"tools"` subcommand (and optionally `tools dump`/`tools diff` per the requirement's "optionally add" wording); a new `_cmd_mcp_tools()` method renders a table of `RuntimeToolRegistry.all_tools()` with the 7 requested fields.
- `scripts/agent/commands/registry.py`: **no change expected** ŌĆö `/mcp` is already a registered prefix command (`command_defs_list.py:71`); its existing dispatch already routes all `/mcp <anything>` input to `_cmd_mcp()`, which internally subcommand-dispatches ŌĆö no new top-level `CommandDef` entry is needed, only `_cmd_mcp()`'s internal subcommand handling grows.
- `shared/runtime_tool_registry.py`: **no new method needed** ŌĆö `all_tools()` (requirement 02's planned method) already returns everything this diagnostic needs to iterate and display.
- `agent/context.py`: **no change** ŌĆö `ctx.services_required.runtime_tools` is already available per requirements 02-03.
- `tests/test_cmd_mcp.py` (real path ŌĆö requirement's stated `tests/agent/commands/test_cmd_mcp.py` doesn't exist, consistent with the now-repeated path-mismatch pattern across this batch): add tests for the new `/mcp tools` subcommand.
- Help-output tests (real path per requirement 05's finding: `tests/test_command_def_sync.py`, not `tests/agent/commands/test_help.py`): confirm `/mcp`'s help text mentions the new subcommand if the usage line changes.

**Out of scope**
- Building `RuntimeToolRegistry.all_tools()` itself ŌĆö requirement 02.
- Any change to `/mcp status`'s existing health-probe output ŌĆö this requirement adds a sibling subcommand, not a replacement.
- `/mcp tools diff` (comparing two points in time) ŌĆö explicitly "optional" per the requirement; scoped out unless the base `/mcp tools`/`/mcp tools dump` prove insufficient (flag as a possible follow-up, not built speculatively here).

## Assumptions

2. `/mcp` is already registered as a prefix `CommandDef` in `command_defs_list.py:71` ŌĆö confirmed via grep; no new top-level command registration is needed, only `_cmd_mcp()`'s internal subcommand set grows (mirroring how `/session`'s subcommands are dispatched internally per a prior investigation this session).
3. `RuntimeToolRegistry.all_tools()` (requirement 02's planned method) returns all registered `RuntimeTool` instances, each already carrying the 7 fields this requirement wants displayed (`server_key`, `name`, `enabled_for_llm`, `agent_safety_tier`, `requires_approval`, `is_write`, `requires_serial`) ŌĆö no new registry method is needed, only a rendering/formatting function in `cmd_mcp.py`.

5. "Output concise and safe for logs" (this requirement's own wording) implies the table format should avoid dumping full `input_schema`/`raw_definition` (potentially large JSON blobs) ŌĆö only the 7 named scalar fields, following the existing `_format_mcp_table()` helper's row-based style (`cmd_mcp.py:26`, already used for `/mcp status`'s health-probe table).

## Design

`_cmd_mcp()`'s dispatch gains a `"tools"` branch: `elif sub == "tools": await self._cmd_mcp_tools(rest_args)`. `_cmd_mcp_tools()` iterates `ctx.services_required.runtime_tools.all_tools()`, sorts by `server_key` then `name` (for stable, scannable output), and renders one row per tool with the 7 requested fields via a table helper similar to (or a new sibling of) the existing `_format_mcp_table()`. Per Assumption 5, `input_schema`/`raw_definition`/`description` are deliberately excluded from this table to keep output concise ŌĆö a future `/mcp tools dump <name>` (the "optionally add" item) could show a single tool's full detail including schema, but the default table view stays scalar-fields-only.

`"dump"` (if implemented per the optional scope) is a natural sub-subcommand: `/mcp tools dump <tool_name>` shows one tool's full `RuntimeTool` including `raw_definition`, for deep debugging ŌĆö distinct from the default concise table.

## Implementation steps

1. **Add `_cmd_mcp_tools()`** to `cmd_mcp.py`: iterate `all_tools()`, render a table with `server_key`, `name`, `enabled_for_llm`, `agent_safety_tier`, `requires_approval`, `is_write`, `requires_serial` columns.
2. **Wire the `"tools"` subcommand** into `_cmd_mcp()`'s dispatch, updating the `UnknownSubcommandError`'s allowed-subcommand tuple to `("status", "tools")`.
3. **(Optional) Add `/mcp tools dump <name>`**: a variant showing one tool's full detail; only implement if the base table view is confirmed insufficient during review ŌĆö do not build speculatively beyond what's needed.
4. **Update help/usage text**: `_cmd_mcp()`'s usage line and any help-output test asserting on it.
5. **Tests**: add `/mcp tools` output tests to `tests/test_cmd_mcp.py` (including a case with a disabled/policy-filtered tool present, to directly verify the "disabled tools are visible" acceptance criterion).
6. **Verification**: run the standard validation sequence scoped to changed files, then the full suite.

## Risks

- **Risk**: If `RuntimeToolRegistry` contains a large number of tools, `/mcp tools`'s default table could become unwieldy in a terminal (long output, hard to scan).
  **Mitigation**: sort by `server_key` then name for scannability (per Design); consider a `--server <key>` filter as a future refinement if this becomes a real usability problem, not built speculatively now.
- **Risk**: Exposing `agent_safety_tier`/`requires_approval` in a CLI diagnostic output could be misread by an operator as a live security boundary they can toggle from this view, when it's actually read-only diagnostic information.
  **Mitigation**: the command's help text/docstring should state clearly this is read-only inspection, not a configuration interface ŌĆö no toggle/set functionality should be added under this requirement's scope.
