## Assumptions

1. `format_table()` は `cmd_mcp.py` の `_cmd_mcp_status()` からのみ呼ばれている (grep 確認済み)。
2. `print_next_steps()` は `cmd_mcp.py` の `_cmd_mcp_install()` からのみ呼ばれている (grep 確認済み)。
3. `sync_services()` は `apply_config_dict()` 内からのみ呼ばれており、外部からの direct call は現状存在しない (grep 確認済み: callers は cmd_config.py の `apply_config_dict` 経由のみ)。
4. `McpServerStatus` の `status` フィールドを `availability: str` + `health: str` に分割した場合、`format_table()` 側で結合して表示すれば既存の表示形式は維持できる。

6. `ingest_workflow.py` の `logger` は未使用。`except Exception as e` で `logger.exception` を追加するのは低リスクな改善。

## Design

### D-1: `mcp_status.py` — `McpServerStatus` status フィールド分割

```python
@dataclass
class McpServerStatus:
    key: str
    transport: str
    startup_mode: str
    auth: str
    write: str
    role: str
    availability: str   # "OK" | "HTTP NNN" | "FAIL (...)" | "STOPPED" | ...
    health: str         # "HEALTHY" | "DEGRADED" | "UNAVAILABLE"
    endpoint: str
```

`probe_all()` 内の `status = f"{status}/{health_label}"` を削除し、
`availability` と `health` を個別にセットする。
`format_table()` は `cmd_mcp.py` に移動し、そこで `f"{r.availability}/{r.health}"` として表示を組み立てる。

### D-2: `mcp_install.py` — `print_next_steps()` をコマンド層へ移管

`McpInstallService.print_next_steps()` を削除。
`cmd_mcp.py._cmd_mcp_install()` に、同等の出力ロジックをインライン展開する。

### D-3: `config_reload.py` — 差分適用化

`_apply_rag_tool_params()` / `_apply_llm_prompt_params()` / `_apply_sse_reload_params()` の各フィールド反映を、`if "key" in new_cfg:` ガードで保護する。
`sync_services()` の docstring から "and directly by legacy code" の前提を削除し、private に変更する (名称: `_sync_services`)。
`apply_config_dict()` の呼出も `self._sync_services(new_cfg)` に更新する。

### D-4: `ingest_workflow.py` — ロギング改善

各 `except Exception as e:` に `logger.exception(...)` を追加し、スタックトレースが記録されるようにする。
`logger` は既に定義済みなので追加の import は不要。
