   try:
       result = await self._raw_execute(tool_name, args)
       if not inflight.done():

## Design

### Stampede-protection exception fix (scripts/shared/tool_executor.py)

```python
async def _execute_with_stampede_protection(
    self,
    cache_key: str,
    tool_name: str,
    args: dict[str, Any],
) -> ToolCallResult:
    """Share inflight future among concurrent callers to prevent stampede.

    If _raw_execute() raises, the exception is propagated to every concurrent
    waiter via inflight.set_exception() — not just the caller that triggered
    execution — so no waiter hangs indefinitely on a failed shared future.
    """
    inflight = self._inflight.get(cache_key)
    if inflight is not None and not inflight.done():
        return await inflight
    if inflight is not None and inflight.done():
        return inflight.result()
    loop = asyncio.get_running_loop()
    inflight = loop.create_future()
    self._inflight[cache_key] = inflight
    try:
        result = await self._raw_execute(tool_name, args)
    except Exception as exc:  # noqa: BLE001 — must release all inflight waiters regardless of exception type
        if not inflight.done():
            inflight.set_exception(exc)
        raise
    else:
        if not inflight.done():
            inflight.set_result(result)
        return result
    finally:
        self._inflight.pop(cache_key, None)
```

Note: a waiter that does `return await inflight` on a Future whose exception was set will have that same exception re-raised at the `await` point (standard `asyncio.Future` behavior) — this is the correct, expected propagation, not a new failure mode.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Testing genuine concurrent-hang scenarios (two coroutines truly racing on the same `inflight` Future) can itself be flaky if not carefully sequenced | Medium | Use deterministic sequencing (e.g., an `asyncio.Event` the mocked `_raw_execute` waits on before raising, released only after the second caller has already retrieved the shared `inflight` reference) rather than relying on real scheduler timing; wrap the whole test in a short `asyncio.wait_for` timeout so a regression fails fast and visibly instead of hanging CI |
| `except Exception` in the stampede-protection path is broad; the project generally prefers narrower exception types | Low | Justified with an inline `# noqa: BLE001` comment (matching this codebase's own established pattern, e.g. `_probe_mcp_health_detail()`'s identical justification style) — releasing every concurrent waiter regardless of exception type is the entire point of this fix; narrowing the catch would silently reintroduce the hang for any exception type not explicitly listed |
| Choosing to keep `ToolExecutor`'s internal cache as canonical (rather than migrating to `ToolResultCache`) could be revisited later if `ToolResultCache` gains stampede-protection support and becomes the better long-term abstraction | Low | Documented explicitly in the new docstring so a future contributor has full context for reconsidering, rather than the current orphaned/undocumented state where neither cache's status was clear |
