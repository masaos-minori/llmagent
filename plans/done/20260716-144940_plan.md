# Plan: Remove the MCP watchdog feature (health polling + auto-restart loop)

Source requirement: `requires/20260716_20_require.md`

## Goal

Remove the MCP watchdog (background health-poll + auto-restart loop) and every
config key, display line, test, and doc that exists solely to support it,
while leaving the independent `/health` endpoint / `HealthRegistry` /
`/mcp status` probing path fully intact.

## Scope

**Out of scope**

- `/health` endpoint (`scripts/mcp_servers/server.py`), `McpServerHealthRegistry`
  state machine (HEALTHY/DEGRADED/UNAVAILABLE/HALF_OPEN), and
  `McpStatusService.probe_all()` used by `/mcp status` — independent of the
  watchdog and must keep working unchanged.
- Tool-execution-layer `record_success()` / `record_failure()` calls made from
  the dispatch/transport-error path — unrelated caller, unaffected by this
  removal.
- `record_degraded()` and `get_degraded_reason()` on `McpServerHealthRegistry` —
  used by the transport-error path too, not watchdog-only; only
  `record_restart_exhausted()` is watchdog-only and removed.
- Subprocess lifecycle management itself (`start_http_subprocess()`,
  `ensure_ready()`, manual `/mcp restart`) — only the watchdog's automatic
  trigger of `lifecycle.restart()` is removed, not the lifecycle manager.
- Any change to production process-supervision (systemd, container healthcheck,
  etc.) — out of scope for a code-removal plan; see Risks.

## Assumptions

1. No other module in `scripts/` imports `watchdog_loop`, `_watchdog_check_http`,
   or `_classify_health_failure` besides `scripts/agent/repl.py` and their own
   tests — confirmed via `grep -rn "_classify_health_failure\|watchdog_loop\|_watchdog_check_http" scripts/`.
2. `HealthRegistry.record_restart_exhausted()` has exactly one production
   caller (`repl_health.py:650`, inside `_watchdog_check_http`) — confirmed via
   `grep -rn "record_restart_exhausted" scripts/`. Once the watchdog is
   removed, this method has no caller and is dead code; per this repo's
   coding conventions (delete confirmed-unused code rather than leave it as a
   dormant capability), it is removed along with its dedicated tests.
3. `tests/test_config_reload.py` and `tests/test_config_reload_classification.py`
   do not reference `mcp_watchdog_*` — confirmed via grep — so no change is
   needed there beyond what's already listed.
4. `deploy/deploy.sh` and the rest of `deploy/` have no watchdog-specific
   logic (confirmed via grep) — this is a same-file-set removal, not an
   add/remove of deployed files, so no `deploy.sh` copy-list change is needed.
5. This repo's convention for documenting a removed feature (observed in
   `implementations/done/20260629-185000_mdq_stub_language_removal.md`) is to
   keep a short, dated resolution note rather than silently deleting all
   trace — applied to the watchdog doc sweep below.

## Design

No new architecture — this is a subtractive change to existing modules. Two
decisions worth recording explicitly:

1. **What stays.** `McpServerHealthRegistry` (HEALTHY/DEGRADED/UNAVAILABLE/HALF_OPEN),
   its `record_success()`/`record_failure()`/`record_degraded()`/`get_degraded_reason()`
   methods, the `/health` endpoint on each MCP server, and `McpStatusService.probe_all()`
   (used by `/mcp status`) all have callers independent of the watchdog
   (tool-dispatch error handling, manual `/mcp status`). They are unchanged.
2. **What goes with the watchdog.** `record_restart_exhausted()` exists purely
   to let `/mcp status` distinguish "still cycling" from "watchdog gave up"
   (per its own docstring). With the watchdog removed, nothing ever calls it,
   so it is deleted rather than left as unreachable code — consistent with
   this repo's "delete confirmed-unused code" convention and with what
   `vulture` would flag once the sole caller disappears.

Doc rewrite convention: only `04_mcp_06_12` was confirmed (by direct read) to
be *entirely* about the watchdog. It gets its page body replaced with a short
dated note ("MCP watchdog was removed on 2026-07-16; see
`requires/done/20260716_20_require.md`") rather than deleting the file
outright, matching the resolution-note convention observed in
`implementations/done/20260629-185000_mdq_stub_language_removal.md`.
`04_mcp_06_13` part1/part2 turned out, on direct read, to carry substantial
non-watchdog content (general `health_reason`/`HealthRegistry` behavior in
part1; tool-error monitoring and scheduling/serialization in part2) — these
get a targeted partial edit (remove only the watchdog-specific
subsection/sentences, retitle away from "Watchdog Behavior") rather than a
removal note. `04_mcp_03_04` similarly keeps its transport-error-tracing
content and only loses the watchdog section, with a title reconsideration if
"watchdog" was implied by its current filename/title. The remaining 21 docs
get incidental-mention cleanup only, no removal note (they were never about
the watchdog).
