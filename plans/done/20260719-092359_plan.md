# Plan: Fix stale `/db rag rebuild-fts` repair guidance in RAG consistency issue messages

Source issue: `issues/20260719-091341_rag_consistency_repair_guidance_points_to_removed_db_command.md`
Related: `requires/20260719_01_require.md`, `plans/20260719-091140_plan.md` (adds
`/session rag-consistency`, which surfaces this same guidance text verbatim)

## 1. Goal

RAG consistency issue messages (`summarize_issues()` in
`scripts/db/rag_consistency.py`) no longer tell operators to run
`/db rag rebuild-fts` — a command removed with no successor — including in
the CRITICAL, data-loss-risk case (orphan FTS entries). The guidance instead
points at a working on-demand command, `/session rag-rebuild-fts`, backed by
the existing, already-tested `RagMaintenanceService.rebuild_fts()`.

## 2. Scope

**In-Scope**:
- Add `rag-rebuild-fts` as a new subcommand key in `_cmd_session`'s
  `db_dispatch` table (`scripts/agent/commands/cmd_session.py`), backed by a
  new `_rag_rebuild_fts` method that calls
  `RagMaintenanceService().rebuild_fts()` and prints a success message
  (mirroring the existing `checkpoint`/`vacuum` pattern, which call a
  service method with no return value and print a fixed success string) —
  and the new `_rag_consistency` method/dispatch entry from
  `plans/20260719-091140_plan.md`, since both land in the same file and
  should be implemented together (see Assumptions).
- Update the two stale guidance strings in
  `scripts/db/rag_consistency.py`'s `summarize_issues()` (lines 103, 120)
  from `'/db rag rebuild-fts'` to `'/session rag-rebuild-fts'`.
- Update the `/session` `CommandDef` help string in
  `scripts/agent/commands/command_defs_list.py` to also include
  `rag-rebuild-fts` (in addition to `rag-consistency` from the sibling plan).
- Update existing behavior-lock test

  (currently asserts `"/db rag rebuild-fts" in i`) to assert the new command
  string instead — this test currently locks in the bug and must change with it.
- Add new regression tests:
  - `tests/test_agent_cmd_session.py` — new `TestCmdSessionRagRebuildFts`
    class (success case; confirm `RagMaintenanceService.rebuild_fts()` is
    invoked).
  - A drift-guard test (new, alongside the `/stats`-hint drift test added by
    the sibling plan) asserting no string returned by `summarize_issues()`

- Update documentation:
  - `docs/05_agent_07_08_cli-and-commands-slash-commands-session-mcp.md` —
    add a `/session rag-rebuild-fts` row (alongside the `rag-consistency`
    row from the sibling plan).
  - `docs/05_agent_07_07_cli-and-commands-migration-notes.md` (line 58,
    `/db rebuild-fts` row) — change from `後継コマンドなし` to point at
    `/session rag-rebuild-fts`.

**Out-of-Scope**:

- The duplicate, more elaborate `RagDbMaintenanceService.rebuild_fts()` in
  `scripts/rag/maintenance.py` (drops/recreates the FTS table rather than
  delete-all + reinsert) — this is the RAG ingestion pipeline's own

- The third and fourth `summarize_issues()` messages (`orphan_vec_count`,
  `vec != chunks` — both say `Re-run ingestion with 'ingester.py --force'`)
  — verified `ingester.py --force` is a real, current CLI flag
  (`scripts/rag/ingestion/ingester.py:621-622`), so these two messages are

- `RagMaintenanceService.rebuild_vec()` and `.reconcile_url()` — also

- Any change to `agent/startup.py`'s automatic consistency check — it will
  simply print the corrected guidance text automatically once this lands,
  with no code change needed there.

## 3. Assumptions

1. This plan's `/session rag-rebuild-fts` addition lands in the same
   implementation pass as `plans/20260719-091140_plan.md`'s
   `/session rag-consistency` — both touch `cmd_session.py`'s `db_dispatch`
   dict and `command_defs_list.py`'s same `CommandDef.help` string, so
   implementing them separately risks a trivial merge conflict on adjacent
   lines. Falsifiable: if the sibling plan has already landed by the time
   this one is implemented, this plan's Implementation steps reduce to
   adding one more dict entry next to the existing `rag-consistency` one,
   with no conflict (see Risks for the ordering dependency this creates).
2. `RagMaintenanceService().rebuild_fts()` is safe to expose without
   confirmation/dry-run gating — falsifiable by: reading its body (already
   done; it wraps `INSERT ... VALUES('delete-all')` + re-insert in a single
   `SQLiteHelper("rag").open(write_mode=True)` transaction, the same
   operation `/db rebuild-fts` used to perform before removal, and the same
   one `agent/startup.py`'s automatic consistency check already recommends
   indirectly). No destructive/irreversible risk beyond what the removed
   command already carried.
3. No new `SubcommandSpec` entries are needed, matching Assumption 3 from
   `plans/20260719-091140_plan.md` (verified: `/session`'s `CommandDef` uses
   a free-text `help` string, not `SubcommandSpec` entries, for its existing
   subcommands).
4. Fixing the guidance text at both call sites (`fts_gap`, WARNING; and
   `fts_orphan_count`, CRITICAL) to the same replacement command is correct
   — both messages describe the identical repair action (rebuild the FTS
   index), differing only in severity/urgency framing, confirmed by reading
   `summarize_issues()` in full (lines 88-148).

## 8. Risks & Mitigations

- **Risk**: This plan and `plans/20260719-091140_plan.md` both edit the same
  three files (`cmd_session.py`, `command_defs_list.py`, and indirectly the
  same `/session` documentation table) — implementing them out of order or
  in parallel could produce a merge conflict or a duplicated dict-key
  addition. → **Mitigation**: Implementation step 1 explicitly re-checks
  current file state before editing and branches on whether the sibling
  plan already landed; Assumption 1 documents the expected joint-landing
  order. Residual risk is a trivial textual merge conflict, not a logic bug.
- **Risk**: `summarize_issues()`'s CC 25 (grade D) is high enough that a
  reviewer might expect this plan to refactor it. → **Mitigation**:
  explicitly out of scope — this plan changes 2 string literals inside
  existing branches, not the branching structure; refactoring
  `summarize_issues()` for maintainability is an unrelated, separately
  filable concern (not raised by the source issue).
- **Risk**: Two different `rebuild_fts()` implementations exist
  (`agent/services/rag_maintenance_service.py` vs `rag/maintenance.py`) with
  different table-rebuild strategies (delete-all+reinsert vs drop+recreate);
  exposing only the agent-services one via `/session rag-rebuild-fts` could
  read as endorsing one over the other without explanation. → **Mitigation**:
  documented explicitly in Out-of-Scope with the reasoning (agent-layer
  boundary convention); no code unification attempted here. Filed as a
  residual open question rather than silently ignored, in case a future
  reader wants to reconcile the duplication.
