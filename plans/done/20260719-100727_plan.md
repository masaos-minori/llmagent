# Plan: Implement `tags_json`/`token_count` in mdq-mcp; reconcile stale MDQ known-issue docs

Source requirement: `requires/20260715_01_require.md`

## Goal

Real YAML-frontmatter tags and a real per-chunk token count are stored and
returned by `mdq-mcp` (currently hardcoded to `""` and `None` respectively),
and every MDQ-related "known issue" doc entry accurately reflects the
current codebase — including two entries this requirement's premise no
longer matches (see Assumptions).

   small local heuristic instead (resolves the source requirement's own
   Unknown #1).
6. `db_schema.py`'s `chunks.token_count` column (line 72:
   `token_count INTEGER,` — nullable, no default) and `models.py`'s
   `ChunkRecord.token_count` / `SearchResultItem.token_count` (both

   change needed — confirmed via direct read.
7. No test currently exists to update for tags_json/token_count (they were
   never implemented, only hardcoded), so all new tests are pure additions,
   not behavior-lock updates — confirmed via the source requirement's own
   description and this plan's investigation of `indexer.py`'s hardcoded

## Design

1. **Frontmatter tag extraction** (`parser.py`): within the existing
   `---`...`---` detection block (`:92-100`), instead of only skipping,
   capture `lines[i+1:j]`, join and `yaml.safe_load` it (guarding
   non-dict/parse-failure results by treating them as "no frontmatter
   data" rather than raising — a malformed frontmatter block must not
   break indexing of an otherwise-valid document). Extract `tags`: accept
   either a YAML list (`tags: [a, b]`) or a comma-separated string
   (`tags: "a, b"`), normalizing to `list[str]` either way, per the source
   requirement's own framing. Return this (possibly empty) list alongside
   the existing `list[ParsedSection]`.
2. **Threading into `indexer.py`**: `_index_single_file` calls the updated
   `parse_markdown`, receives the document-level tags list once per file
   (not per section — frontmatter is file-level, not section-level),
   `json.dumps(tags)` it, and binds that instead of `""` for every
   section's `tags_json` column in that file (all sections/chunks in one
   document share the same document-level tags, since there is no
   per-heading tag concept in the source Markdown format).
3. **`token_count` heuristic**: a small local function, e.g.
   `_estimate_token_count(text: str) -> int: return max(1, len(text) // 4)`
   (a common chars-per-token-~4 approximation, consistent in spirit with
   `shared/token_estimation.py`'s own ratio-based approach but without its
   chat-category/network dependencies) — computed per section/chunk next
   to the existing `char_count`. Document in a one-line comment that this
   is an approximation, not an exact tokenizer count, since no live
   tokenizer endpoint is available inside `mcp_servers/mdq` without adding
   a new network dependency (out of scope per this plan's size).
4. **Docs**:
   - Delete the stale `docs/04_mcp_90_inconsistencies_and_known_issues.md`
     entry (Assumption 3) — since the discrepancy it described (hybrid
     search stub + live `use_embedding` config) no longer exists in any
     form, this is "Obsolete and removable" per `rules/coding.md`, not a
     reword.
   - Add a short "tags_json/token_count now implemented" closure note to
     `docs/04_mcp_04_04_mdq.md` (near its existing tool/field
     documentation) and `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md`
     (near wherever it currently lists these as open gaps, if it does —
     confirm exact wording during implementation since this plan's
     investigation focused on the fts/summary-cache entries specifically).

## Risks

- **Risk**: A malformed or unusual frontmatter block (e.g. invalid YAML,
  or a `tags:` value that's neither a list nor a string) could raise
  during indexing and break the whole file's indexing pass, regressing
  today's behavior (which simply skips frontmatter without ever parsing
  it, so it never fails). → **Mitigation**: Design step 1 explicitly
  requires catching parse failures and unexpected shapes, falling back to
  an empty tags list rather than propagating an exception — the "malformed
  frontmatter" test case in Implementation step 1 locks this in.
- **Risk**: The `len(text) // 4` token-count heuristic is an approximation,
  not an exact count — could be seen as inconsistent with more precise
  tokenizer-based counts used elsewhere (e.g. `shared/token_counter.py`'s
  real `/tokenize` endpoint calls for chat history). → **Mitigation**:
  explicitly documented as an approximation in code and in the doc closure
  note (Design step 4); the Acceptance Criteria only require "non-null,"
  not "exact," so this satisfies the stated requirement without a larger
  network-dependent implementation. A future requirement could upgrade
  this to a real tokenizer call if precision becomes a concern.
- **Risk**: Deleting the stale `docs/04_mcp_90_...` entry (Design step 4)
  removes a cross-reference some other doc might point to. → **Mitigation**:
  `tools/check_agent_docs_consistency.py`'s broken-internal-link check
  (`--skip links` is the opt-out, so it runs by default) is included in
  the Validation Plan specifically to catch this before it lands.
- **Risk**: Treating Required Changes 1 & 4 as "already resolved by
  removal" (per user decision) means this plan does not restore
  `fts_consistency_check`/`fts_rebuild` or summary caching even though the
  original requirement asked for them — if a future reader only skims the
  original requirement without this plan's Scope/Assumptions sections,
  they might reopen work already deliberately closed. → **Mitigation**:
  this plan states the removal commit and rationale explicitly (Scope,
  Assumption 1) and `requires/done/20260715_01_require.md` (once moved)
  will sit alongside this plan for full traceability; no further mitigation
  needed beyond clear documentation.
