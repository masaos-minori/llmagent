# Plan: Add startup checks for high-impact deny-all configurations

## Goal

Detect high-impact deny-all configurations at startup, distinguish intentional lockdown from likely misconfiguration, and improve warning messages for these cases.

## Scope

**In scope:**
- Startup validation (`scripts/agent/repl_health.py` — `audit_security_defaults()`)
- MCP config checks for empty allowlists/denylists
- Operator-facing warnings
- Documentation of deny-all behavior

**Out of scope:**
- Changing the actual allowlist/denylist enforcement logic
- Modifying individual MCP server configurations
- Adding runtime configuration hot-reload
- Changes to tool execution behavior

## Assumptions

1. Empty allowlists effectively disable all tools for that category (deny-all), which may be intentional (security lockdown) or accidental (missing config).
2. Current startup checks exist but do not specifically identify deny-all states or distinguish them from normal security defaults.
3. Operators need early visibility into deny-all states to avoid accidentally disabling entire MCP servers.

## Implementation Steps

1. **Add deny-all detection to startup checks**
   - In `audit_security_defaults()`, check for empty allowlists that cause deny-all:
     - db_allowlist empty → sqlite-mcp denies all queries
     - command_allowlist empty → shell-mcp denies all commands
     - allowed_repo_paths empty → git-mcp denies all operations
     - allowed_repos empty → github-mcp denies all repo access
   - Log each as: "WARNING: {setting} is empty — {server} will deny ALL operations"
   - Completion criterion: Startup logs explicitly flag deny-all states

2. **Distinguish intentional lockdown from misconfiguration**
   - Add a new config option `security_lockdown_enabled` (boolean, default false)
   - When true, suppress deny-all warnings (operator has acknowledged they are intentional)
   - When false (default), show all deny-all warnings
   - Completion criterion: Config option allows operators to suppress warnings

3. **Improve warning messages**
   - Format: "DENY-ALL detected: {setting} is empty. {server} will reject ALL requests from this category. Verify this is intentional or add allowed values to config."
   - Include suggested config keys in the message
   - Completion criterion: Warnings are actionable and clear

4. **Document deny-all behavior**
   - Add section in security docs about intentional lockdown procedure
   - Document how to verify deny-all state at runtime
   - Completion criterion: Docs describe deny-all behavior and lockdown workflow
