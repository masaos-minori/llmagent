# Plan: Use RuntimeToolRegistry for scheduler metadata and side-effect detection

Source requirement: `requires/20260717_06_require.md`

## Goal

DAG scheduler metadata (`ToolSpec` instances keyed by tool name) and side-effect detection are sourced from `RuntimeToolRegistry` (requirements 02-03) instead of static `tool_definitions`/`tool_constants.py` frozensets ŌĆö `ToolRunner` no longer constructs `ToolSpec` from static config, and `is_side_effect()`'s primary source becomes `runtime_tools.is_side_effect(tool_name)`.

## Scope

**In scope**
- `scripts/agent/tool_runner.py`: replace `_build_tool_meta(tool_definitions: list[dict])` (lines 54-73) with a call to `RuntimeToolRegistry.tool_spec_map()` (per requirement 02's planned method); update `_execute_with_dag()`'s call site (previously read `ctx.cfg.tool.tool_definitions` at line 254, passed into `_build_tool_meta()` at lines 255/259) to instead pull from `ctx.services_required.runtime_tools`.
- `scripts/agent/tool_scheduler.py`: **no structural change expected** ŌĆö `build_execution_groups()` already takes `tool_meta: dict[str, ToolSpec]` as a parameter; it's agnostic to how that dict was built, so only its caller (`tool_runner.py`) changes, not the scheduler itself.
- `scripts/shared/tool_spec.py`: **no change** ŌĆö `ToolSpec`'s existing 6-field shape (`call_id`, `name`, `args`, `resource_scope`, `requires_serial`, `is_write`) already matches the requirement's "ToolSpec requirements" list exactly (confirmed in requirement 02's plan, Assumption 3).
- `scripts/shared/tool_executor_helpers.py`: `is_side_effect()`'s primary implementation becomes a call to `runtime_tools.is_side_effect(tool_name)`; the existing `_SIDE_EFFECT_TOOLS` frozenset-based logic becomes a fallback only (see Design), not deleted outright per this requirement's own wording ("static constants are not the *primary* source," not "must be removed").
- `tests/test_tool_runner.py` and `tests/test_tool_scheduler.py` (real paths ŌĆö requirement's stated `tests/agent/...` paths don't exist, per Assumption 4): update fixtures from static `tool_definitions` lists to `RuntimeToolRegistry` fixtures.
- `tests/shared/test_runtime_tool_registry.py`: add/confirm `tool_spec_map()`/`is_side_effect()` test coverage (this file is newly created by requirement 02 ŌĆö this requirement extends its test coverage, doesn't create it from scratch).

**Out of scope**
- Routing (`route_resolver.py`/`tool_executor.py`) ŌĆö requirement 04.
- LLM schema/help output ŌĆö requirement 05.
- Policy/approval classification (`classify_operation_type`/`classify_risk`) ŌĆö requirement 07.
- Deleting `tool_constants.py` or its frozensets outright ŌĆö this requirement demotes them to fallback/secondary status for side-effect detection only; requirement 10 ("redefine static tool_definitions and tool_constants as compatibility-only") handles their final disposition.

## Assumptions

1. `_build_tool_meta()` (`tool_runner.py:54-73`, confirmed by direct read, already investigated in requirement 02's planning pass) builds `dict[str, ToolSpec]` from `tool_definitions: list[dict]`, deriving `is_write` via `is_side_effect(name)` (line 63), `requires_serial` via `fn.get("requires_serial", False) or name in SHELL_TOOLS` (line 64), `resource_scope` defaulting to `name` when write else `""` (lines 65, 69) ŌĆö this entire function is replaced by a call into `RuntimeToolRegistry.tool_spec_map()`, which per requirement 02's plan is designed to produce the same `dict[str, ToolSpec]` shape from `RuntimeTool` data instead.
2. `ToolSpec`'s dataclass shape (`shared/tool_spec.py`, 6 fields, confirmed frozen/unchanged per requirement 02's Assumption 3) already satisfies this requirement's "ToolSpec requirements" list verbatim (`call_id`, `name`, `args`, `resource_scope`, `requires_serial`, `is_write`) ŌĆö no new field is needed.
3. `is_side_effect()` (`tool_executor_helpers.py:34-38`, confirmed by direct read in requirement 02's planning pass) is currently a plain frozenset-membership check against `_SIDE_EFFECT_TOOLS` (a union of `WRITE_TOOLS | DELETE_TOOLS | {"shell_run"} | GIT_WRITE_TOOLS | GITHUB_WRITE_TOOLS | GITHUB_DANGEROUS_TOOLS | CICD_WRITE_TOOLS | RAG_WRITE_TOOLS | MDQ_WRITE_TOOLS`). This requirement's "static constants are not the primary source" (not "must never be consulted") allows keeping this as a fallback for tools not yet in `RuntimeToolRegistry` (e.g. during a partial-rollout window), rather than requiring instant, total removal.
4. **The requirement's stated test-file paths don't match the current repo layout**: `tests/agent/` does not exist (confirmed in requirement 05's investigation) ŌĆö real files are `tests/test_tool_runner.py` and `tests/test_tool_scheduler.py` (flat `tests/` dir). `tests/shared/test_runtime_tool_registry.py` is a genuinely new file created by requirement 02, extended (not created) by this requirement.
5. `_execute_with_dag()` (`tool_runner.py:244-259`, confirmed in requirement 02's investigation) reads `ctx.cfg.tool.tool_definitions` at line 254 and passes it into `build_execution_groups()` at lines 255/259 via `_build_tool_meta()` ŌĆö this is the sole call site needing to switch its data source to `ctx.services_required.runtime_tools`.
6. Per requirement 02's Design, `RuntimeTool`'s safe defaults for unannotated tools already set `agent_safety_tier="WRITE_DANGEROUS"`, `requires_approval=True`, `enabled_for_llm=False`, `requires_serial=True` (when `is_write` is also unannotated) ŌĆö this requirement's own "Safe defaults" section (side-effecting=true, requires_serial=true, requires_approval=true) is consistent with and already covered by requirement 02's planned defaults; this plan does not need to re-implement default logic, only ensure `tool_spec_map()`'s `is_write` derivation defaults to `True` (side-effecting) for unknown tools, matching this requirement's explicit ask (slightly stronger than requirement 02's own wording, which defaults `is_write` implicitly via the conservative `requires_serial` rule ŌĆö confirm during implementation that an unknown tool's `is_write` itself, not just `requires_serial`, defaults to `True`).

## Design

`_build_tool_meta()`'s replacement is a direct call: `tool_meta = ctx.services_required.runtime_tools.tool_spec_map()` inside `_execute_with_dag()`, replacing the static-list-based construction entirely. Per Assumption 1, `RuntimeToolRegistry.tool_spec_map()` (requirement 02's planned method) is designed to produce the same `dict[str, ToolSpec]` shape from `RuntimeTool.is_write`/`requires_serial`/`resource_scope` fields ŌĆö this is a like-for-like data-source swap with no change to the scheduler's consumption contract.

`is_side_effect()`'s demotion to fallback status: the function's new body checks `runtime_tools.is_side_effect(tool_name)` first (if the registry has an entry for that tool name); only falls back to the static frozenset check for tools genuinely absent from the registry (e.g. a transitional window before requirement 03's discovery fully replaces all tool sourcing, or a tool that legitimately isn't MCP-routed). This satisfies "static constants are not the primary source" without requiring their immediate, total deletion ŌĆö that full removal is requirement 10's explicit scope.

## Risks

- **Risk**: If requirement 02's `RuntimeTool` safe-default logic doesn't set `is_write=True` for unannotated tools (the blocking Unknown), an unclassified write tool could be scheduled as if it were read-only, running concurrently with other calls when it should be serialized/cache-bypassed ŌĆö a real safety regression, not just a cosmetic mismatch.
  **Mitigation**: Implementation Step 1 explicitly verifies and, if needed, corrects this before any other step proceeds; do not implement this requirement against an unverified assumption about requirement 02's exact default behavior.
- **Risk**: Demoting (not deleting) the static frozenset fallback in `is_side_effect()` could mask a real gap in `RuntimeToolRegistry`'s coverage ŌĆö if a tool is silently absent from the registry (e.g. a discovery bug), it falls back to old behavior without any visible warning, delaying detection of the underlying registry-population problem.
  **Mitigation**: log a warning (not silent) whenever the fallback path is taken, so registry-coverage gaps are visible in logs even though execution continues correctly.
- **Risk**: `_build_tool_meta()`'s deletion could break an untracked caller if `rg -n "_build_tool_meta"` isn't checked first.
  **Mitigation**: Implementation Step 3 explicitly requires this grep before deletion.
