## Design

### `_log_routing_coverage()` fix (scripts/shared/route_resolver.py)

```python
def _log_routing_coverage(self, known_tools: frozenset[str]) -> None:
    """Log routing coverage for all known tools at startup.

    "Mapped" means resolvable via ToolRegistry (the sole routing authority,
    same check resolve() uses) ﻗ NOT merely present in discovery_map, which
    is validation-only metadata from live /v1/tools responses and carries no
    routing authority. A tool present only in discovery_map but absent from
    the registry is UNMAPPED for this purpose, since resolve() would raise
    ValueError for it.

    Note: as of this writing, no production caller passes known_tools to
    ToolRouteResolver.__init__(), so this method does not currently run in
    production (see shared/tool_executor.py's construction call). It remains
    available for a future caller that wants startup coverage visibility.
    """
    mapped: list[str] = []
    unmapped: list[str] = []
    for tool_name in sorted(known_tools):
        if self._lookup_registry(tool_name) is not None:
            mapped.append(tool_name)
        else:
            unmapped.append(tool_name)
    total = len(known_tools)
    if unmapped:
        logger.warning(
            "Routing (registry-based): %d/%d tools mapped; %d unmapped: %s",
            len(mapped), total, len(unmapped), unmapped,
        )
    else:
        logger.info("Routing (registry-based): %d/%d tools mapped", total, total)
```

(`discovery_map` is no longer consulted for the mapped/unmapped classification at all ﻗ it remains stored on `self` for potential future diagnostic use, per its own already-correct docstring comment "Validation data from live /v1/tools (not used for routing).")
