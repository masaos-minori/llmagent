# Plan: Consolidate MCP tool contract, routing ownership, side-effect safety, and approval policy

Source requirement: `requires/20260716_13_require.md`

## Goal

Every MCP tool across all 8 servers (mdq, github, shell, git, cicd,
rag_pipeline, file[read/write/delete], web_search) has consistent
classification across schema, dispatch, `ToolRegistry`, side-effect
detection, risk/approval policy, and audit logging — closing the one
concrete drift already found (`classify_operation_type()` misclassifies
every non-generic-`WRITE_TOOLS` write tool as `read`) and generalizing the
MDQ-only drift guardrail (`tests/test_mdq_tool_layer_consistency.py`,
`plans/done/20260716-123746_plan.md`) to all servers.

## Scope

**In scope**

- **Fix the confirmed classification gap**: `agent/tool_policy.py::classify_operation_type()`
  only recognizes the generic `WRITE_TOOLS`, `DELETE_TOOLS`, `_EXEC_TOOLS`
  (`shell_run`), and `_GITHUB_MUTATION_TOOLS` sets — it does not check
  `MDQ_WRITE_TOOLS`, `RAG_WRITE_TOOLS`, `CICD_WRITE_TOOLS`, or
  `GIT_WRITE_TOOLS`. Every write tool in those four sets is currently
  classified as `OperationType.READ`, and since this function feeds
  `agent/tool_audit.py`'s `operation_type` audit field directly, every MDQ/
  RAG/CICD/Git write-tool call is **audited as a read operation today**.
  Extend `classify_operation_type()` to check all `*_WRITE_TOOLS`/
  `*_DELETE_TOOLS` sets from `shared/tool_constants.py`, not a hand-picked
  subset.
- **Generalize the per-server dispatch/schema/registry consistency
  guardrail**: extend the pattern already proven for MDQ
  (`tests/test_mdq_tool_layer_consistency.py`) to all 8 servers, accounting
  for each server's actual dispatch mechanism — see Design section for why
  this cannot be a literal copy-paste of the MDQ test.
- **Document (not merge) the two existing serialization mechanisms**:
  `shared/tool_executor.py`'s `is_side_effect()`-triggered automatic
  batch-to-serial downgrade (project-wide, driven by the union of all
  `*_WRITE_TOOLS`/`DELETE_TOOLS` in `tool_executor_helpers.py`'s
  `_SIDE_EFFECT_TOOLS`) and `agent/tool_scheduler.py`'s per-tool
  `requires_serial` flag (sourced from each server's `tools.py` metadata) —
  clarify in code comments/docs that these are two distinct, complementary
  mechanisms at different granularities, not a duplicate/drifting pair (see
  Unknowns for the one open question this raises).
- **Confirm and, if needed, extend live-discovery drift validation**:
  `shared/tool_routing_validation.py`'s `validate_routing_against_config`/
  `validate_routing_against_live`/`validate_all_routing` already exist and
  are already wired into `agent/repl_health.py` (startup health check) and
  tested in `tests/test_tool_registry.py` — confirm this covers all 8
  servers today, not just a subset, and that a drift result is treated as
  fatal/actionable at startup per this requirement's "Decide which drift
  cases must be fatal at startup" instruction.
- Update the requirement's imprecise target-file list: `agent/approval/*`
  does not exist as a directory — the actual approval-related files are
  `agent/tool_approval.py` and `agent/workflow/approval_ops.py`.

**Out of scope**

- Adding new MCP tools — explicit non-goal.
- Redesigning the approval UI — explicit non-goal.
- Changing individual tool business behavior beyond what's needed to fix
  classification/dispatch drift — explicit non-goal.
- Changing non-MCP command execution behavior — explicit non-goal.
- Merging `is_side_effect()`-based batch downgrade and `requires_serial`
  into one mechanism — not requested, and Unknowns below flags this as a
  question for the user/reviewer rather than a decision this plan makes
  unilaterally.

## Assumptions

1. `classify_operation_type()`'s under-classification is real and
   reproducible today — confirmed by direct read of
   `scripts/agent/tool_policy.py:40-49`, cross-referenced against
   `scripts/shared/tool_constants.py`'s full set of `*_WRITE_TOOLS`
   definitions (`MDQ_WRITE_TOOLS`, `RAG_WRITE_TOOLS`, `CICD_WRITE_TOOLS`,
   `GIT_WRITE_TOOLS` are all defined there but none are imported into
   `tool_policy.py`).
2. This gap is currently low-impact for **approval risk gating**
   specifically, because `classify_risk()` (the function that actually
   gates approval) uses `cfg.approval.tool_safety_tiers` (from
   `config/agent.toml`) as priority-2, ahead of the generic
   `tool_constants.py` fallback that shares the same blind spot — and
   `config/agent.toml`'s `[tool_safety_tiers]` section already has explicit
   entries for every MDQ/RAG/CICD/Git tool (verified by direct read). The
   gap's real, currently-active impact is confined to
   `operation_type` in audit records (`agent/tool_audit.py:51,185`) and
   `agent/repository_gateway.py`'s use of the same classification — both
   consumers are verified via `rg -n "classify_operation_type"`.
3. No existing test exercises `classify_operation_type()` against any
   `MDQ_WRITE_TOOLS`/`RAG_WRITE_TOOLS`/`CICD_WRITE_TOOLS`/`GIT_WRITE_TOOLS`
   member — verified via direct read of
   `tests/test_tool_policy_comprehensive.py:171-188` and
   `tests/test_tool_approval_risk.py:291-334`, which only cover the generic
   `WRITE_TOOLS`/`DELETE_TOOLS`/`_EXEC_TOOLS`/`_GITHUB_MUTATION_TOOLS`
   cases. This is why the gap has never been caught.
4. Every non-MDQ server (github, shell, git, cicd, rag_pipeline, file's
   three sub-servers) dispatches via `_service.get_dispatch_table()` — a
   method on the service instance — not a module-level `_DISPATCH_TABLE`
   dict like MDQ's `server.py`. Verified via direct read of each server's
   `server.py`: `return await dispatch_tool(_service.get_dispatch_table(),
   name, args)`. MDQ is the outlier in this codebase, not the norm — the
   generalized guardrail test must call `get_dispatch_table()` on an
   instantiated service for those 7 servers and only use MDQ's
   `_DISPATCH_TABLE` import for MDQ itself.
5. `web_search` has a single tool (`search_web`) dispatched via a direct
   `if`/`return` rather than any dispatch table — verified by direct read
   of `scripts/mcp_servers/web_search/server.py`. The generalized guardrail
   test must handle single-tool servers without an enumerable dispatch
   table structurally (e.g., assert the one tool name matches
   `WEB_SEARCH_TOOLS` and is reachable, rather than comparing dispatch-table
   key sets).
6. `validate_routing_against_config`/`validate_routing_against_live` are
   already invoked at startup via `agent/repl_health.py` and covered by
   parametrized-by-server-config tests in `tests/test_tool_registry.py` —
   verified via `rg -n "validate_routing_against_config\|validate_all_routing"
   scripts/ tests/`. Whether this coverage is exhaustive across all 8
   servers or only exercised with synthetic/partial server configs in tests
   needs confirmation during implementation (see Unknowns).

## Design

This is intentionally scoped as **incremental, not a rewrite**: the
underlying architecture (schema in `tools.py`, dispatch via
`get_dispatch_table()`/`_DISPATCH_TABLE`, `ToolRegistry` populated from
`tool_constants.py` frozensets, side-effect detection via `is_side_effect()`,
risk/approval via `classify_risk()`) is already sound and mostly consistent
— the concrete problem found is one function
(`classify_operation_type()`) with an incomplete set of checks, plus the
fact that the drift-guardrail pattern proven for MDQ was never generalized.

The generalized guardrail test cannot be a literal parametrization over one
shared dispatch-table shape, because two structurally different dispatch
patterns exist (module-level dict for MDQ vs. `get_dispatch_table()` method
for the other 7 servers, per Assumption 4) plus one single-tool server with
no table at all (Assumption 5). The test design should therefore define a
small per-server adapter (a mapping from server key → a callable that
returns that server's live dispatch key set), rather than assuming a single
uniform access pattern.

## Implementation steps

1. **Fix `classify_operation_type()`** — import and check
   `MDQ_WRITE_TOOLS`, `RAG_WRITE_TOOLS`, `CICD_WRITE_TOOLS`, `GIT_WRITE_TOOLS`
   (and any `*_DELETE_TOOLS` beyond the generic `DELETE_TOOLS`, if any exist
   — confirm during implementation) so no server's write/delete tools fall
   through to `OperationType.READ`.
2. **Add regression tests** for the fixed function, one assertion per
   previously-unclassified write-tool set, in the existing test files that
   already cover `classify_operation_type()`.
3. **Resolve the serialization-mechanism Unknown** with the user before
   touching `tool_executor_helpers.py`/`tool_scheduler.py` beyond
   documentation — this determines whether step 3's scope is "add
   clarifying comments" or "redesign."
4. **Build the per-server dispatch adapter and generalized guardrail test**
   — one adapter function per server key, then parametrized tests mirroring
   `tests/test_mdq_tool_layer_consistency.py`'s six assertions (schema ⊆/⊇
   dispatch, tool-constants set == schema names, registry membership,
   write/serial flag consistency) applied to all 8 servers.
5. **Confirm live-discovery drift validation's actual server coverage** —
   read `tests/test_tool_registry.py` in full, resolve the second Unknown,
   and extend coverage if any server is currently untested.
6. **Update documentation** describing the unified tool lifecycle, per this
   requirement's Design intent, in whichever existing MCP architecture doc
   is the best fit (likely `docs/04_mcp_03_01_dispatch-and-routing.md`,
   already the home of the routing-authority documentation touched by the
   earlier MDQ-only guardrail work).
7. **Deployment/verification** — run the full test suite; no service
   restart needed since no runtime config or deploy-relevant file changes.

## Risks

- **Risk**: Changing `classify_operation_type()`'s output for MDQ/RAG/CICD/
  Git write tools (READ → WRITE/DELETE) could change downstream behavior in
  `agent/repository_gateway.py`, which also consumes this function, beyond
  just audit logging.
  **Mitigation**: Implementation step 1 must read `repository_gateway.py`'s
  use of `classify_operation_type()`'s result in full before changing the
  function, and step 7's validation plan includes running
  `tests/test_repository_gateway.py` (which already mocks this function,
  per the grep evidence gathered) to confirm no behavioral regression.
- **Risk**: The generalized guardrail test could be large and slow if it
  instantiates every service class (some may have real DB/network
  dependencies at construction time, as MDQ's does via `_init_db()`).
  **Mitigation**: Implementation step 4 should reuse each server's existing
  test fixture patterns (e.g., temp DB paths) rather than inventing new
  service-construction logic, and should be scoped to import-time/static
  checks (`TOOL_LIST`, `_DISPATCH_TABLE`/`get_dispatch_table()`,
  `tool_constants.py` sets) wherever possible instead of live HTTP calls.
- **Risk**: This is a large, cross-cutting requirement (8 servers, 2
  approval-adjacent modules, multiple test files) — attempting it as one
  monolithic change risks a large, hard-to-review diff.
  **Mitigation**: the Implementation steps above are already ordered as
  independently committable increments (fix classification → add its tests
  → resolve the serialization Unknown → build the generalized guardrail →
  confirm live-discovery coverage → docs), matching this project's
  incrementalism convention.
