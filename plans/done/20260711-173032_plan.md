## Assumptions

1. **`common.toml` no longer exists.** Confirmed via `ls config/*.toml`: no `common.toml` file is present; `config/agent.toml`'s own header comment (line 2-3) states "Previously split across: common, llm, http, rag, context, tools, memory, otel, security, system_prompts, tools_definitions, and *_mcp_server transport sections," confirming `common.toml` was consolidated into `agent.toml` and is now historical. `scripts/db/helper.py::_connect()` (line 111) raises `ValueError(f"{self._target}_db_path is not configured in common.toml")` — a genuinely wrong, currently-shipping error message.
2. **`workflow_schema_version` is checked, but not through the same clean-error path as the other required tables.** Confirmed by direct read of `scripts/agent/repl_health.py::check_workflow_schema()` (lines 446-494): `REQUIRED_WORKFLOW_TABLES` (446-452) lists `tasks`, `attempts`, `processed_events`, `artifacts`, `approvals` — each checked via a loop that raises a clean `RuntimeError(f"Workflow schema missing table {table!r}. Run create_workflow_schema() to initialize.")` if absent. `workflow_schema_version` is instead queried directly at line 485-488 (`SELECT version FROM workflow_schema_version ORDER BY applied_at DESC LIMIT 1`) with **no prior existence check** — if the table itself is missing (e.g. a workflow DB created by an older schema version, or manually corrupted), this raises a bare `sqlite3.OperationalError: no such table: workflow_schema_version`, not the clean, remediation-bearing `RuntimeError` every other missing table gets.
3. **Migrations are un-named and swallow all `sqlite3.OperationalError`s.** Confirmed by direct read of `scripts/db/schema_sql.py::_WORKFLOW_MIGRATIONS` (lines 241-247, a bare `list[str]` of ALTER TABLE statements, no IDs) and `apply_workflow_migrations()` (lines 255-262): every statement's `sqlite3.OperationalError` is caught and silently passed (`except sqlite3.OperationalError: pass  # column already exists`) — this masks any other `OperationalError` a broken/malformed migration statement might raise (e.g. a syntax error, a reference to a nonexistent table), not just "column already exists." SQLite's actual duplicate-column error message is deterministic and greppable: `"duplicate column name: <col>"` — confirmed via Python's sqlite3 behavior (standard library, not repo-specific) — so the catch can be narrowed to that exact message pattern.
4. **`create_schema.py::create_workflow_schema()` duplicates the migration-apply loop instead of calling `apply_workflow_migrations()`.** Confirmed by direct read: `create_schema.py:80-84` re-implements the identical `for stmt in _WORKFLOW_MIGRATIONS: try: db.execute(stmt) except sqlite3.OperationalError: pass` loop inline, rather than calling `schema_sql.apply_workflow_migrations(db._conn)` (or equivalent). `apply_workflow_migrations()` itself (confirmed via `grep -rn "apply_workflow_migrations"`) is currently called only from `tests/test_workflow_stage_persistence.py`, never from production code. Fixing the swallow-all-errors bug in only one of these two copies would leave the other silently vulnerable, so both must be updated — this plan additionally de-duplicates by having `create_workflow_schema()` call the shared `apply_workflow_migrations()` function instead of maintaining a second copy of the same loop.
5. **`workflow_schema_version` table already has the `version`/`applied_at` columns the requirement asks for.** Confirmed by direct read of `schema_sql.py:204-207`: `CREATE TABLE IF NOT EXISTS workflow_schema_version (version TEXT NOT NULL, applied_at TEXT NOT NULL DEFAULT (...))` — both required columns already exist. This task item is a negative finding: no DDL change is needed, only the *validation-path* gap (Assumption 2) needs fixing so this table's absence produces a clean error like every other table already gets, and so its columns are checked the same way.
6. **`db/maintenance.py:66` and `db/config.py:59` are stale docstrings, not stale runtime behavior.** Confirmed both actually call `ConfigLoader().load("agent.toml")` (maintenance.py:67, config.py:60) — the code is already correct; only the docstring text above each is wrong.

## Design

### 1. `scripts/agent/repl_health.py::check_workflow_schema()`

Fold `workflow_schema_version` into the same explicit required-table/column loop, with its own required columns, before the version-comparison logic runs:

```python
REQUIRED_WORKFLOW_TABLES: dict[str, list[str]] = {
    "tasks": ["task_id", "session_id", "workflow_id", "status", "created_at"],
    "attempts": ["attempt_id", "task_id", "stage_id", "status"],
    "processed_events": ["event_id", "task_id"],
    "artifacts": ["artifact_id", "task_id"],
    "approvals": ["approval_id", "task_id", "status"],
    "workflow_schema_version": ["version", "applied_at"],
}
```

No other change to the table/column-checking loop is needed — it already iterates `REQUIRED_WORKFLOW_TABLES.items()` generically and will now raise the clean `RuntimeError(f"Workflow schema missing table {table!r}. Run create_workflow_schema() to initialize.")` for a missing `workflow_schema_version` table too, before the raw `SELECT version FROM workflow_schema_version ...` query at line 485 ever runs (since that query is unreachable once the table-missing RuntimeError has already fired). Update the version-mismatch error message (line 491-494) to keep its current wording (already clear/actionable — confirmed no change needed there, a negative finding).

### 2. `scripts/db/schema_sql.py` — named migrations + narrowed error handling

```python
_WORKFLOW_MIGRATIONS: list[tuple[str, str]] = [
    ("2026_add_attempts_error_kind", "ALTER TABLE attempts ADD COLUMN error_kind TEXT"),
    ("2026_add_attempts_error_detail", "ALTER TABLE attempts ADD COLUMN error_detail TEXT"),
    ("2026_add_artifacts_workflow_id", "ALTER TABLE artifacts ADD COLUMN workflow_id TEXT"),
    ("2026_add_artifacts_attempt_number", "ALTER TABLE artifacts ADD COLUMN attempt_number INTEGER"),
    ("2026_add_processed_events_workflow_id", "ALTER TABLE processed_events ADD COLUMN workflow_id TEXT"),
]


def apply_workflow_migrations(conn: sqlite3.Connection) -> None:
    """Apply incremental migrations to an existing workflow.sqlite; idempotent.

    Only 'duplicate column name' errors are treated as already-applied and
    skipped (logged at debug level); any other OperationalError is unexpected
    and re-raised.
    """
    for migration_id, stmt in _WORKFLOW_MIGRATIONS:
        try:
            conn.execute(stmt)
        except sqlite3.OperationalError as exc:
            if "duplicate column name" in str(exc):
                logger.debug("Migration %s already applied: %s", migration_id, exc)
                continue
            raise
    conn.commit()
```

Requires adding a module-level `logger = logging.getLogger(__name__)` to `schema_sql.py` (confirmed absent today via full-file read) and `import logging`.

Exact naming convention for `migration_id` values (e.g. date-prefixed vs. sequential integers) to be finalized at implementation time to match any existing convention elsewhere in the repo — no existing precedent was found via `grep -rn "migration_id\|MIGRATION_ID"`, so this plan proposes a simple descriptive-slug format; open to adjustment during implementation without changing the surrounding logic.

### 3. `scripts/db/create_schema.py::create_workflow_schema()`

Replace the inline duplicate migration loop with a call to the shared, now-narrowed function:

```python
def create_workflow_schema() -> None:
    """Create workflow.sqlite tables (tasks, attempts, processed_events, artifacts, approvals, workflow_schema_version)."""
    with SQLiteHelper("workflow").open(write_mode=True) as db:
        try:
            db.executescript(build_workflow_schema_sql())
        except (sqlite3.OperationalError, sqlite3.DatabaseError) as e:
            logger.error("Failed to execute workflow schema DDL: %s", e)
            raise
        apply_workflow_migrations(db._conn)
        _record_workflow_schema_version(db)
        db.commit()
    logger.info("Workflow schema created successfully.")
```

(Exact attribute name for the underlying `sqlite3.Connection` on `SQLiteHelper` — `db._conn` used above matching `apply_workflow_migrations(conn: sqlite3.Connection)`'s signature — to be confirmed against `db/helper.py`'s actual `SQLiteHelper` attribute name at implementation time.) Update the module-level docstring's `create_workflow_schema()` line (create_schema.py:12) to mention `workflow_schema_version` explicitly, per the requirement's "Schema creation documentation" task.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Narrowing the `except sqlite3.OperationalError` clause to only `"duplicate column name"` could break on an existing production workflow.sqlite whose migration history left the DB in a state where the same statement fails with a *different* wording (e.g. SQLite version differences) | Low | `"duplicate column name: <col>"` is SQLite's standard, stable error text for `ALTER TABLE ... ADD COLUMN` on an existing column across all SQLite versions used by this repo (bundled via the standard library `sqlite3` module); the narrowing is on the error re-raise path only — any unexpected error becomes loud and visible instead of silently masked, which is the explicit goal of this fix, and any newly-surfaced failure indicates a real, previously-hidden problem that should be seen, not a regression to hide from |
| Migration ID slugs are introduced with no existing repo convention to follow, risking inconsistency with any future migration-tracking work | Low | Explicitly flagged in Design §2 as open to adjustment; IDs are purely descriptive/logging metadata with no schema or behavioral significance, so a later rename is a no-op change |
| Routing `create_workflow_schema()` through the shared `apply_workflow_migrations()` (instead of its own inline loop) changes which function tests need to mock/patch if any existing test patches the inline loop directly | Low | Confirmed via `grep -rn "_WORKFLOW_MIGRATIONS\|apply_workflow_migrations"` that only `tests/test_workflow_stage_persistence.py` references these symbols directly, and it already imports/calls `apply_workflow_migrations()` — no test currently depends on `create_schema.py`'s inline loop specifically |
