# Plan: [High][eventbus][performance] 高負荷統合テストの追加

## Goal

並列パブリッシュ・サブスクライブ・DLQ 処理の同時実行下で SSE 継続性とパブリッシュレイテンシを検証する統合テストを追加し、イベントループブロッキングのリグレッションを CI で検出できるようにする。

## Scope

**In-Scope**
- `tests/test_eventbus_*.py` への並列負荷テスト追加
- 複数サブスクライバー + 複数パブリッシャーの同時実行シナリオ
- DLQ ループとの干渉検証
- レイテンシしきい値アサーション

**Out-of-Scope**
- 本番環境でのベンチマーク
- 既存テストの変更
- アプリケーションコードの変更

## Assumptions

1. テストは `httpx.AsyncClient` + `pytest-anyio` または標準の `asyncio` で並列実行できる
2. `TestClient` はブロッキングのため高負荷テストには使用しない
3. レイテンシしきい値: publish P99 < 200ms、SSE 配信遅延 < 500ms
4. テストはローカルの ASGI アプリ (インプロセス) に対して実行するため外部プロセス不要

## Design

テストシナリオ:

1. **並列パブリッシュテスト**: 10 スレッドが同時に 5 イベントずつ publish → 重複なく全 50 件が DB に存在することを検証
2. **SSE 継続性テスト**: 1 サブスクライバーが subscribe 中に 20 件 publish → サブスクライバーが全件受信することを検証 (タイムアウトあり)
3. **DLQ 干渉テスト**: DLQ 対象イベントを混在させた状態で publish/subscribe → API レスポンスタイムが劣化しないことを検証
4. **レイテンシ回帰テスト**: publish 10 件の P99 レイテンシが 200ms 以内であることをアサート

```python
import asyncio
import time
from httpx import AsyncClient, ASGITransport

async def test_concurrent_publish(client: AsyncClient):
    tasks = [client.post("/publish", json=make_event(i)) for i in range(50)]
    results = await asyncio.gather(*tasks)
    assert all(r.status_code == 200 for r in results)
```

## Implementation Steps

1. `tests/test_eventbus_load.py` を新規作成し、並列パブリッシュテストを追加

5. `uv run pytest tests/test_eventbus_load.py -v` で通過確認

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| SSE テストがハングする | Medium | タイムアウトを明示的に設定 (`asyncio.wait_for`) |
| テスト間でDBステートが干渉する | Medium | 各テストで一時ファイルDBを使用 (`tmp_path` fixture) |
| レイテンシしきい値がCI環境で不安定 | Medium | P99 ではなく最大値を使うか、CI では緩い閾値を設定 |
