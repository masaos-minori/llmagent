Confirm and document the end-to-end semantic cache invalidation contract: verify that the CLI
ingestion path actually invokes `on_ingest_complete`, confirm that the MCP ingestion path has no
equivalent call, decide whether `CacheService` Protocol should expose `invalidate()` and
`generation`, and update the relevant documentation to reflect the confirmed state.

## Confirmed Findings (from code analysis)

### CLI path (`ingester.py main()`)
- `main()` at line 661 calls `ingester.ingest_all(args.force)` ŌĆö no `on_ingest_complete` passed.
- The callback parameter is accepted but never wired in CLI usage.
- Result: CLI ingestion does NOT invalidate any semantic cache instance.

### MCP path (`mcp/rag_pipeline/service.py`)
- `RagPipelineMCPService` has no ingestion endpoint ŌĆö confirmed from `tools.py` (only `rag_run_pipeline`, `rag_debug_pipeline`, `rag_list_documents`, `rag_delete_document`).
- `delete_document()` removes DB rows but does NOT call `pipeline.semantic_cache.invalidate()`.
- Result: MCP path has no cache invalidation path.

### `CacheService` Protocol
- Currently exposes only `lookup()` and `put()` ŌĆö the query-path interface.
- `invalidate()` and `generation` are only on `SemanticCache` (concrete class).
- `docs/03_rag_04_data_model_and_interfaces.md ┬¦5.6` already documents this deliberately.
- Decision: keep Protocol unchanged; document the rationale explicitly.

### Multiple RagPipeline instances
- Agent REPL: no `RagPipeline` is instantiated directly in the agent ŌĆö confirmed from agent scripts.
- MCP server: `RagPipelineMCPService.start()` creates one `RagPipeline` instance.
- Each instance owns its own `SemanticCache` (initialized in `RagPipeline.__init__`).
- No shared cache exists; invalidation of one instance does not affect others.
