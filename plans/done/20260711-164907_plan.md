## Goal

Make MCP `rag_delete_document` invalidate the semantic cache through a public `RagPipeline.invalidate_cache()` method (only on successful deletion), and give CLI ingestion an explicit, low-risk cache-freshness story (documented operational restart, not an invented cross-process invalidation mechanism) so operators know exactly what to do after either operation.

2. `scripts/mcp_servers/rag_pipeline/service.py::fmt_delete_document()` (lines 197-205) currently calls `self._doc_mgr.delete_document(url)` and returns a formatted string based on its `bool` result, but performs **no cache invalidation at all** today — confirmed by direct read. These 5 facts map exactly onto the requirement's 5 documentation Tasks (38-42); no code investigation gap remains.
6. Resolving "Design a CLI ingestion cache freshness strategy — choose one": **operational requirement to restart the rag-pipeline-mcp service (or the whole agent process) after CLI ingestion, if immediate query freshness is required.** Chosen over the other 3 options because: (a) the Notes explicitly forbid assuming CLI can reach the MCP service's in-memory state, ruling out any same-process shortcut; (b) an admin/invalidation HTTP endpoint or a DB-persisted corpus-version check are both new, non-trivial cross-process mechanisms with their own correctness surface (version-check overhead on every query, or a new unauthenticated-by-default admin route) that the requirement does not mandate building ("if immediate freshness is required" is conditional, not mandatory); (c) MCP-service-managed ingestion would be an architectural move of the ingestion pipeline into the MCP server process, far larger than this requirement's stated scope; (d) the existing agent/MCP lifecycle already supports restarting a single subprocess-mode MCP server (confirmed in this session's earlier work on `agent/lifecycle.py`/watchdog), so "restart it" is an already-available, zero-new-code operator action that fully satisfies the Acceptance Criteria ("Operators know what to do after CLI ingestion if immediate freshness is required," "Cross-process cache invalidation is not treated as if it were a local callback problem").

## Design

### `RagPipeline.invalidate_cache()` (scripts/rag/pipeline.py)

```python
def invalidate_cache(self) -> None:
    """Clear all cached semantic-search entries.

    Call after any corpus-changing operation this pipeline instance is aware
    of (e.g. MCP rag_delete_document) so subsequent queries don't return
    context for a document that no longer exists. Delegates to
    SemanticCache.invalidate(), which is thread-safe.
    """
    self.semantic_cache.invalidate()
```

### `RagPipelineLike` + `fmt_delete_document()` (scripts/mcp_servers/rag_pipeline/service.py)

```python
class RagPipelineLike(Protocol):
    async def augment(self, ...) -> str: ...
    last_fetch_result: Any
    last_timings: dict[str, float]
    def invalidate_cache(self) -> None: ...
```

```python
async def fmt_delete_document(self, args: ToolArgs) -> str:
    raw_url = args.get("url")
    if not isinstance(raw_url, str):
        return "Error: url must be a string."
    url = raw_url.strip()
    if not url:
        return "Error: url is required."
    ok = self._doc_mgr.delete_document(url)
    if ok:
        self._pipeline_or_raise().invalidate_cache()
        logger.info("Semantic cache invalidated after deleting %r", url)
    return f"Deleted: {url}" if ok else f"Not found: {url}"
```

class TestFmtDeleteDocument:
    async def test_found_returns_deleted(self, monkeypatch: Any) -> None:
        service = RagPipelineMCPService()
        service._pipeline = MagicMock()

### Documentation: `on_ingest_complete`/`check_consistency()` (ingester + document-manager docs)

Five explicit statements, each citing the exact code path confirmed in Assumption 5:
1. `RagIngester.ingest_all(force, on_ingest_complete=None)` accepts an optional callback, forwarded to `DocumentManager.check_consistency()`.
2. `check_consistency()` invokes the callback after a successful consistency-check run — including when the report contains issues (issues alone do not suppress the callback).
3. If `chunk_dir` has no `*.json` files, `ingest_all()` returns `None` immediately; the callback is never invoked.
4. If the consistency check itself raises, `check_consistency()` returns `None`; the callback is never invoked.
5. CLI `main()` (`scripts/rag/ingestion/ingester.py`) calls `ingest_all(args.force)` with no callback — the CLI path never invokes `on_ingest_complete` today; it exists for callers (e.g. a future MCP-driven ingestion path) that need a post-ingestion hook.

### Documentation: cache freshness strategy (cache/pipeline doc)

One paragraph: "MCP `rag_delete_document` invalidates the calling MCP-service process's own `RagPipeline.semantic_cache` via `invalidate_cache()` — this only clears the cache **within that one process**. CLI ingestion (`uv run python -m rag.ingestion.ingester`) runs in a **separate process** with no access to the MCP service's in-memory cache; **if immediate query freshness after CLI ingestion is required, restart the rag-pipeline-mcp service (or the agent process, which restarts all subprocess-mode MCP servers) — this is an operational step, not something CLI ingestion does automatically.** Absent a restart, semantic-cache entries populated before the ingestion may return stale context for a bounded window (until the cache's own eviction/TTL — see [cache configuration] — naturally expires them)."

| Risk | Likelihood | Mitigation |
|---|---|---|
