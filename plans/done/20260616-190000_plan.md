# Plan: Add readiness/status metadata to /v1/tools (req 14)

## Goal

Add optional `status`, `experimental`, `requires_config`, and `stub` fields to `/v1/tools` output so clients can distinguish production-ready tools from placeholders or config-dependent ones.

## Scope

**In scope**:
- `mcp/file/read_tools.py` — add optional metadata fields to `_MCP_TOOLS`
- `mcp/file/write_tools.py` — same
- `mcp/file/delete_tools.py` — same
- `mcp/shell/tools.py` — same
- `mcp/sqlite/tools.py` — same
- `mcp/git/tools.py` — same
- `mcp/cicd/tools.py` — same
- `mcp/web_search/tools.py` — same
- `mcp/rag_pipeline/tools.py` — same
- `mcp/mdq/tools.py` — same
- `mcp/github/tools.py` — same
- `mcp/installer_templates.py` — include metadata in generated tool definitions
- All `/v1/tools` endpoints pass through the new fields unchanged (backward compatible)

**Out of scope**:
- Changing the routing logic (`route_resolver.py`)
- Adding per-tool status filtering or management APIs
- Modifying `MCPServer.list_tools_with_server_key()` to add new fields (it spreads existing tool dicts)

## Assumptions

1. All current tools are production-ready (`status: "production"`) — no stubs or experimental tools exist in the codebase today.
2. The metadata fields are all optional; omitting them preserves backward compatibility for existing clients.
3. Tools that require non-trivial server config (sqlite db paths, shell command allowlists, git repo paths, cicd tokens) should be marked `requires_config: true` so clients know they may not work until configured.
4. File read/write/delete and web_search tools do not require per-server config (they use allowed_dirs or system tools), so `requires_config: false`.

## Implementation Steps

1. **Add metadata to file read tools** (`mcp/file/read_tools.py`)
   - Add `"status": "production"` and `"requires_config": false` to each of the 9 tool dicts
   - Keep fields optional (no `experimental` or `stub` since none exist)

2. **Add metadata to file write tools** (`mcp/file/write_tools.py`)
   - Same: `"status": "production"`, `"requires_config": false`

3. **Add metadata to file delete tools** (`mcp/file/delete_tools.py`)
   - Same: `"status": "production"`, `"requires_config": false`

4. **Add metadata to shell tools** (`mcp/shell/tools.py`)
   - `"status": "production"`, `"requires_config": true` (needs command_allowlist, env_allowlist)

5. **Add metadata to sqlite tools** (`mcp/sqlite/tools.py`)
   - `"status": "production"`, `"requires_config": true` (needs db_allowlist, max_rows)

6. **Add metadata to git tools** (`mcp/git/tools.py`)
   - `"status": "production"`, `"requires_config": true` (needs allowed_repo_paths, read_only)

7. **Add metadata to cicd tools** (`mcp/cicd/tools.py`)
   - `"status": "production"`, `"requires_config": true` (needs github_token, repo_allowlist)

8. **Add metadata to web_search tools** (`mcp/web_search/tools.py`)
   - `"status": "production"` only (no special config beyond API key which is auth-level)

9. **Add metadata to rag_pipeline tools** (`mcp/rag_pipeline/tools.py`)
   - `"status": "production"` only

10. **Add metadata to mdq tools** (`mcp/mdq/tools.py`)
    - `"status": "production"` only

11. **Add metadata to github tools** (`mcp/github/tools.py`)
    - `"status": "production"`, `"requires_config": true` (needs auth_token, repo config)

12. **Update installer template** (`mcp/installer_templates.py`)
    - In the generated `_MCP_TOOLS` template, include `"status": "production"` for new tools
