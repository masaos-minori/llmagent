# Plan: Unify slash command registry, completion behavior, and CLI documentation

## Goal

Eliminate the confirmed drift between the slash command registry
(`_COMMANDS`), tab-completion, and the CLI documentation set, and add
guard-rail tests so the drift cannot silently reappear.

## Scope

**In scope:**
- `docs/05_agent_01_system-overview.md` — stale "Current behavior" section
  (completion-exclusion claim no longer true) and stale `/db` command
  reference
- `docs/05_agent_02_runtime-architecture-part1.md` — stale mixin count
- `docs/05_agent_07_03_cli-and-commands-command-registry.md` — mixin count/list
  wording (direct vs. transitive mixins)
- `tests/docs/test_command_docs_sync.py` — stale doc path reference that
  causes a silent no-op check
- New REPL prompt documentation (currently absent) in the CLI reference doc set
- `scripts/agent/commands/command_defs_list.py`, `scripts/agent/repl.py`,
  `scripts/agent/commands/registry.py` — read-only verification; no code
  change expected (see Assumptions)
- New/extended tests guarding `_COMMANDS` ↔ `SLASH_COMMANDS` derivation and
  mixin count consistency

**Out of scope (per requirement Non-goals):**
- Redesigning the slash command framework
- Removing valid slash commands
- Changing command business behavior
- Redesigning the full CLI (including the prompt style itself — document the
  current fixed `"> "` prompt rather than changing it, per Assumption #3)

## Assumptions

1. The completion-drift problem described in the requirement's Background
   ("`AgentREPL.SLASH_COMMANDS`... may omit valid commands") was already fixed
   in commit `16ba870d` ("feat: unify command metadata as single source of
   truth", 2026-07-15): `SLASH_COMMANDS` is now a `cached_property` derived
   from `builtin_command_names() | reserved_repl_command_names()`, i.e. every
   entry in `_COMMANDS` plus `/exit`. (Evidence: Explicit in code,
   `scripts/agent/repl.py:52-89`.) The remaining problem is that
   documentation was not updated to reflect this fix.
2. No command currently needs `completion_excluded` metadata — all 18
   `_COMMANDS` entries are includable in tab completion today, and
   `CommandDef` has no such field. Adding an unused field for a hypothetical
   future exclusion would be premature. (Evidence: Explicit in code —
   `scripts/agent/commands/command_defs.py` field list; zero exclusions
   observed in `scripts/agent/repl.py`.) Document this as a deliberate
   decision instead of adding speculative metadata.
3. The REPL prompt (`_prompt = "> "` in `scripts/agent/repl.py:96-98`) is a
   simple fixed prompt with no session id displayed. Given the Non-goal "do
   not redesign the full CLI," the correct action is to document this
   as-is, not change it. (Evidence: Explicit in code.)
4. "CommandRegistry mixin count" is ambiguous between direct base classes
   (13, per `scripts/agent/commands/registry.py:60-74`) and the full
   transitive mixin tree including `_ConfigMixin`'s own two mixins,
   `_ConfigDisplayMixin` and `_ConfigStatsMixin` (13 + 2 = 15). Both existing
   docs state "15" but one location (`05_agent_02`) presents it as a flat
   component count, which reads as if `CommandRegistry` directly inherits 15
   mixins — it inherits 13 directly. (Evidence: Explicit in code, cross-
   referenced against both doc locations.)

## Design

### Completion/registry: already unified, docs are the actual gap

`AgentREPL.SLASH_COMMANDS` (a `cached_property`) returns
`completion_command_names()`, which is `builtin_command_names() |
reserved_repl_command_names()` — i.e., every name in `_COMMANDS` (18 commands)
plus the REPL-reserved `/exit`. This was implemented in commit `16ba870d`
(2026-07-15) and is confirmed correct by direct code reading. `_COMMANDS`
remains the single declared source (per its own module docstring). No source
change is needed for the registry/completion mechanism itself; the acceptance
criteria that reference "generated from or validated against `_COMMANDS`" are
already satisfied by the existing implementation — this plan's job is to (a)
correct the documentation that still describes the pre-fix behavior, and (b)
add a regression test so a future change cannot silently reintroduce drift
without a test failure.

### `completion_excluded` metadata: document the decision, don't add the field

The design intent asks to "add explicit metadata such as `completion_excluded`"
only "if not all commands should be completed." Since all 18 current commands
are already completable and no exclusion need exists, adding an unused field
to `CommandDef` would be speculative. Document this as a resolved design
decision (Assumption #2) rather than modify `command_defs.py`.

### REPL prompt: document current behavior, do not change it

`AgentREPL._prompt` returns a fixed `"> "` with no session id and no
`agent>`-style prefix. There is currently no documentation of this at all
(confirmed absent from the CLI reference doc set). Per the Non-goal "do not
redesign the full CLI," the correct action is to document the current
behavior as-is in the CLI reference doc set, including that multiline
continuation input (`CLIView.read_multiline`) does not change the prompt
string itself.

### CommandRegistry mixin documentation: qualify instead of asserting a bare count

Two docs currently claim a bare "15 mixins" figure. The number is defensible
as a *total* (13 direct bases of `CommandRegistry` + `_ConfigDisplayMixin` +
`_ConfigStatsMixin` nested one level inside `_ConfigMixin`), but stating it
without qualification invites exactly the kind of staleness the requirement
flags — a future contributor adding or removing a mixin has no test to catch
a now-wrong number. Fix: reword both doc locations to state the split
explicitly (13 direct + 2 nested = 15 total) and add a test that computes
the total from the actual class hierarchy via `inspect.getmro()` so any
future mismatch fails CI instead of silently rotting in docs.

### Doc-sync test: fix the stale path, not just the assertion logic

`tests/docs/test_command_docs_sync.py` still points at
`docs/05_agent_07_cli-and-commands.md`, which no longer exists after the doc
split (now `05_agent_07_01` through `05_agent_07_11`). Because the check is
written as `if not filepath.exists(): continue`, the test currently provides
zero coverage of the split files — it passes not because the split docs are
consistent, but because the check never runs against them. Fix: point the
test at the current split files most likely to reference commands (at least
`05_agent_07_08` through `05_agent_07_11`, whose filenames indicate
slash-command content), keeping `05_agent_01_system-overview.md` as-is since
it already correctly triggers a real (currently failing) assertion.

## Implementation Steps

1. **Fix the currently-failing test first**: remove the stale `/db` row from
   `docs/05_agent_01_system-overview.md`'s command table (line 129) so
   `tests/docs/test_command_docs_sync.py::test_active_docs_match_registered_commands`
   passes. This is an independently committable, immediate fix.
2. **Rewrite the stale "Current behavior" section** in
   `docs/05_agent_01_system-overview.md` (lines 140-146) to state that
   completion is derived from `_COMMANDS` with `/exit` added, and that no
   commands are currently excluded from completion.
3. **Reword mixin count documentation** in both
   `docs/05_agent_02_runtime-architecture-part1.md` and
   `docs/05_agent_07_03_cli-and-commands-command-registry.md` to state the
   13-direct + 2-nested breakdown explicitly instead of a bare "15".
4. **Add REPL prompt documentation**: a short subsection in the CLI reference
   doc set (`docs/05_agent_07_05_cli-and-commands-repl-io.md` is the most
   likely fit given its filename) describing the fixed `"> "` prompt and
   absence of session-id display.
5. **Fix the doc-sync test's stale path**: update
   `tests/docs/test_command_docs_sync.py` to check the current split CLI doc
   files instead of the removed `05_agent_07_cli-and-commands.md`.
6. **Add the completion-drift guard test**: assert
   `AgentREPL.SLASH_COMMANDS == frozenset(cmd.name for cmd in _COMMANDS) |
   reserved_repl_command_names()` in `tests/test_command_registry_consistency.py`
   (or a new focused test module if that file's fixture shape doesn't fit).
7. **Add the mixin-count guard test**: compute the actual total mixin count
   via `inspect.getmro(CommandRegistry)` filtered to class names ending in
   `Mixin`, and assert it matches the documented total (15), so a future
   addition/removal fails CI instead of only rotting in docs.
8. **Verification**: run the full validation sequence scoped to the changed
   test/doc files, then the full suite.

## Risks

- **Risk**: `docs/05_agent_07_*` split-file set has 11 files; guessing which
  ones need the prompt documentation or the doc-sync test's file list could
  miss the intended target → **Mitigation**: re-grep each candidate file's
  content for existing "prompt" or slash-command listings immediately before
  editing, rather than relying solely on filename inference from this plan.
- **Risk**: Widening `test_command_docs_sync.py`'s checked file list could
  surface additional currently-hidden stale command references in the split
  docs (beyond the known `/db` case), expanding scope beyond this plan's
  estimate → **Mitigation**: treat any newly-surfaced failures as expected
  discoveries to fix in the same pass (they are the same class of bug this
  plan targets), not as a reason to abandon the fix; if the count is large,
  split into a follow-up requirement rather than silently reducing the
  doc-sync test's coverage back down.
- **Risk**: The mixin-count guard test (Step 7) depends on class naming
  convention (`*Mixin` suffix) to enumerate the tree; if a future mixin is
  named inconsistently it would be silently excluded from the count →
  **Mitigation**: document the naming-convention dependency directly in the
  test's docstring/comment so it's an explicit, known limitation rather than
  a silent gap.
- **Risk**: `registry.py` and `repl.py` have moderate-to-high churn (25 and 39
  commits in 30 days respectively) → **Mitigation**: this plan's code-adjacent
  changes are test-only (no production code change), which limits merge-
  conflict risk to the test files themselves.
