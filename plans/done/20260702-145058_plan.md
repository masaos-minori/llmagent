## Goal

Add a DB recreation section to the operations and architecture documentation that describes the official procedure for applying schema changes: archive → delete → recreate via `create_schema()`.

---

## Scope

**In:**
- Add "DB Recreation" section to `docs/90_shared_05_db_api_and_operations.md`
- Add cross-reference note to `docs/90_shared_04_db_architecture_and_schema.md` §8 (Schema Generation and Migration Approach)
- Document `rotate_all_dbs()` in the maintenance functions table (currently missing from §7)
- Clarify that automatic migration is unsupported; recreation is the official method
- Clarify that recreated DBs are empty and hold no prior data

**Out:**
- No code changes
- No automated migration tooling
- No data export/import tooling
- `eventbus.sqlite` recreation (out of scope per require — note only, see UNK-01)

## Assumptions

1. `rotate_all_dbs()` covers the three production DBs (`rag.sqlite`, `session.sqlite`, `workflow.sqlite`) — **confirmed**: `scripts/db/maintenance.py` L291-297 archives all three via `build_db_config()` paths.
2. `create_schema()` is the authoritative entry point for schema initialization — **confirmed**: `scripts/db/create_schema.py` L141-147.
3. DB paths are resolved via `common.toml` (`build_db_config()`), not hardcoded — **confirmed**: `db/config.py` via `ConfigLoader`.
4. Deletion of DB files must be done manually by the operator (no helper script exists for deletion).

## Implementation Steps

### Phase 1: Update `90_shared_05_db_api_and_operations.md`

- [ ] 1a. Add `rotate_workflow_db()` and `rotate_all_dbs()` rows to the §7 maintenance functions table (currently missing):
  - `rotate_workflow_db(archive_dir=None)` → Archive `workflow.sqlite`; returns `Path`
  - `rotate_all_dbs(archive_dir=None)` → Archive all three DBs; returns `(rag_dest, session_dest, wf_dest)`
- [ ] 1b. Add a new section **§11 DB Recreation Procedure** (insert before the current §11 "AI Reference Guide", renumber as needed) with:
  - Explain that schema changes require DB recreation (no migration support)
  - Step 1: Archive — run `rotate_all_dbs()`
  - Step 2: Delete — manually remove `rag.sqlite`, `session.sqlite`, `workflow.sqlite`
  - Step 3: Recreate — run `create_schema()`
  - Note: recreated DBs are empty; existing records are NOT migrated automatically
  - Note: `create_schema()` also initializes `eventbus.sqlite` if not yet present
  - Note: paths resolved from `common.toml` keys `rag_db_path`, `session_db_path`, `workflow_db_path`
  - Show the three commands as a bash block matching the require's suggested flow

### Phase 2: Update `90_shared_04_db_architecture_and_schema.md`

- [ ] 2a. In §8 "Schema Generation and Migration Approach", add:
  - A note that compatible migration is unsupported; schema changes require DB recreation
  - A forward reference: "See [90_shared_05 §11](90_shared_05_db_api_and_operations.md#11-db-recreation-procedure) for the full recreation procedure"
