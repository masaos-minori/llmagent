# Plan: Strengthen plugin strictness and startup diagnostics

## Goal

Make plugin failures, tool conflicts, and shadowing detectable at startup rather than at call time, and improve strict mode usability for CI and production.

## Scope

**In scope:**
- Improve `load_plugins()` reporting: log each loaded plugin, each skipped plugin (with reason), and a summary count
- Strengthen `_validate_tool_conflicts()`: add explicit "shadowed" vs "allowed" categorization in logs; log tool name + plugin module for each conflict
- Strengthen `_validate_command_conflicts()`: always log shadowed commands with module info, not just in strict mode
- Add return-value validation at registration time where possible (check `register_tool` decorated function has correct signature)
- Improve `strict_mode` behavior: when True, collect all import errors first, then raise a single aggregated error
- Update `_init_plugin_registry()` in factory.py to surface plugin diagnostics clearly in startup output
- Add `plugin_strict` and `plugin_tool_override` config defaults to documentation comments

**Out of scope:**
- Changing the tool execution path in `tool_executor.py` (validation at call time remains as a safety net)
- Adding new config keys
- Plugin hot-reload or dynamic unloading

## Assumptions

1. `strict_mode=True` means "fail fast on first error" — but for better diagnostics, we will change it to "collect all errors then raise one aggregated message."
2. Tool shadowing detection already exists via `_validate_tool_conflicts()` using `get_all_mcp_tool_names()` — we are improving the reporting, not the detection logic.
3. Return-value validation at call time exists in `tool_executor.py` (line 597-600) — we will add a lighter-weight signature check at registration time only.
4. Plugin modules are discovered by globbing `plugins/*.py` — this behavior does not change.

## Implementation Steps

1. **Improve `load_plugins()` reporting**
   - Log each successfully loaded plugin module: `[plugin] loaded: <module_name>`
   - Log each skipped plugin with reason: `[plugin] skipped: <module_name> (<reason>)`
   - Log summary at end: `[plugin] loaded=N, skipped=M`
   - In strict mode: collect all import errors, then raise `PluginLoadError("Failed to load: mod1 (ImportError); mod2 (SyntaxError)")`
   - Completion criterion: all plugin load outcomes are logged; strict mode raises aggregated error

2. **Strengthen `_validate_tool_conflicts()` logging**
   - For each shadowed tool, log: `[plugin] conflict: tool '<name>' in '<module>' shadows MCP tool — <reject/allow>`
   - Categorize conflicts into `shadowed` (rejected) and `allowed` (override policy) lists
   - Log summary: `[plugin] tool conflicts: shadowed=N, allowed=M`
   - Completion criterion: each conflict is logged with module name and resolution

3. **Strengthen `_validate_command_conflicts()` logging**
   - Always log shadowed commands (not just in strict mode): `[plugin] command shadow: '<name>' in '<module>' shadows built-in`
   - In strict mode, also add error-level log
   - Completion criterion: all command shadows are visible at startup

4. **Add signature validation at registration time**
   - In `register_tool()`: check that the decorated function has a return type annotation; if missing, log warning `[plugin] warn: tool '<name>' missing return type annotation`
   - Do NOT enforce exact `tuple[str, bool]` at registration (too strict for Python typing); rely on call-time validation in `tool_executor.py` as safety net
   - Completion criterion: tools without return annotations produce a startup warning

5. **Improve `_init_plugin_registry()` startup reporting**
   - After `load_plugins()`, log plugin summary to both audit logger and stdout
   - Include: total plugins discovered, loaded, skipped, conflicts shadowed, conflicts allowed
   - Completion criterion: startup banner includes plugin status summary

6. **Add docstrings to plugin config fields**
   - Add clear docstrings to `ToolConfig.plugin_tool_override` and `ToolConfig.plugin_strict` explaining their effect and recommended values for CI/production
   - Completion criterion: docstrings mention "recommended for production: True"

7. **Update tests**
   - Add test for aggregated error message in strict mode
   - Add test for conflict logging output (capture log records)
   - Add test for signature warning on missing return annotation
   - Completion criterion: all new behaviors covered by tests; existing tests pass

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| New log messages change existing log parsing (grep/awk pipelines) | med | Use `[plugin]` prefix consistently; document new format in ops docs |
| Signature warning at registration may be too noisy for existing plugins | low | Warning-level only; existing plugins without annotations are common in Python |
| Aggregated error in strict mode changes exception type/message | med | Define new `PluginLoadError` exception class; tests assert on class, not message text |
| Command shadowing always-logging may produce noise for users with many plugins | low | Log at INFO level (not WARNING); users can filter if needed |
