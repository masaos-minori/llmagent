# Plan: Clean up the /memory CLI surface and align search/status behavior with the new Memory model

## Goal

Remove legacy CLI commands (`/memory rebuild`, `/memory import-jsonl`), update search to hybrid-only, simplify status modes to `hybrid`/`degraded`/`unavailable`, and remove JSONL references from all CLI output.

## Scope

### Files to modify
- `agent/commands/cmd_memory.py` — remove legacy command dispatch, update help text, remove disabled-memory branch
- `agent/commands/memory_data_ops.py` — update search wording, convert to async if needed
- `agent/commands/memory_rebuild_ops.py` — remove rebuild/import-jsonl handlers, update check-consistency to SQLite-only
- `agent/commands/memory_status.py` — remove disabled/fts-only status branches, update display logic

### Files NOT to modify (per requirement spec)
- `agent/factory.py`
- `agent/memory/ingestion.py`
- `agent/memory/services.py`
- `agent/memory/retriever.py`
- `agent/memory/embedding_client.py`
- `agent/config_*`
- Documentation files
- Test files (except command-specific tests if required)

## Assumptions

1. Runtime Memory model changes (always-on MemoryServices, no JSONL, embedding-required) are handled separately in the Memory runtime issue (#12).
2. Config key removal (`use_memory_layer`, `memory_jsonl_dir`, `memory_embed_enabled`) is handled separately in the config issue (#11).
3. After this refactor, MemoryServices is always available (no `None` check needed in cmd_memory.py).
4. `/memory search` should always use hybrid retrieval; when embedding fails at runtime, show degraded/unavailable message instead of silently falling back to FTS-only.
5. `memory_local_only` validation already exists in EmbeddingClientConfig and doesn't need CLI changes.

## Unknowns

1. **What should happen when `/memory search` is called but embedding generation fails?** Options:
   - Raise an error and stop (fail-closed)
   - Show degraded message and continue with FTS-only (but this contradicts the requirement to remove FTS-only fallback)
   - Show unavailable message and stop
2. **After `memory_embed_enabled` config key is removed, how does `rebuild_vec` determine whether to proceed?** Need to check if there's still a way to detect embedding availability at runtime via `EmbeddingClient.get_status()`.
3. **Does `memory_search` need to become async?** Currently calls `mem.retriever.search()` which may or may not be async depending on whether embedding is involved.

## Affected areas

- CLI help text: Remove JSONL rebuild/help, update search description
- Command dispatch: Remove `rebuild` and `import-jsonl` subcommands
- Search behavior: Always hybrid, no silent FTS-only fallback
- Status display: Only `hybrid`/`degraded`/`unavailable` modes
- Consistency check: SQLite-only, no JSONL row counts

## Implementation steps

### Step 1: Remove legacy commands from cmd_memory.py
1. Remove `"rebuild"` and `"import-jsonl"` from dispatch map
2. Update `_MEMORY_HELP` docstring:
   - Remove `/memory rebuild` line
   - Remove `/memory import-jsonl` line
   - Update `/memory search` description to "Hybrid memory search"
   - Update `/memory check-consistency` description to "SQLite consistency check"
   - Update `/memory status` description to "Memory layer activation mode"
3. Remove `if mem is None:` block (lines 113-117)
4. Simplify `_memory_status()`: remove `if mem is None:` branch and assert

### Step 2: Remove rebuild/check-consistency from memory_rebuild_ops.py
1. Remove `rebuild()` method
2. Remove `check_consistency()` method
3. Keep `rebuild_fts()` and `rebuild_vec()` methods
4. Update `rebuild_vec()` to not depend on `memory_embed_enabled` config — use runtime embedding status instead

### Step 3: Simplify check-consistency (new implementation)
1. Add back a simplified `check_consistency()` method:
   - Report SQLite, FTS5, and Vec row counts
   - No JSONL row count
   - Vec check based on runtime embedding availability (not config flag)
   - Repair suggestions remain same

### Step 4: Update search to hybrid-only
1. Check if `mem.retriever.search()` is sync or async
2. If async: convert `memory_search()` to async, update dispatch lambda
3. Wrap search call in try/except to catch embedding failures
4. On embedding failure: show degraded/unavailable message instead of empty results
5. Update docstring: "FTS5 search" → "Hybrid memory search"

### Step 5: Simplify status display
1. Remove deprecated fields from `MemoryStatus` dataclass:
   - `fts_fallback_count`
   - `last_retrieval_mode`
   - `memory_layer_enabled`
   - `embedding_enabled`
2. Rewrite `build_status_table()`:
   - Mode label: "Hybrid mode" / "Degraded mode (circuit open)" / "Unavailable (vector index)"
   - Remove rows for: FTS fallback count, Last retrieval mode, Embedding enabled, Memory layer
3. Rewrite `build_memory_status()`:
   - Remove `if mem is None:` branch
   - Use `mem.get_activation_mode()` directly
   - Remove deprecated field assignments

### Step 6: Clean up remaining references
1. In `cmd_memory.py`: Remove `assert mem is not None` after removing the None check
2. In `memory_data_ops.py`: Update any remaining FTS5 references in comments/docstrings
3. Verify no other code references removed fields/methods

## Validation plan

1. **Command rejection**: Verify `/memory rebuild` and `/memory import-jsonl` are rejected as unknown subcommands
2. **Help text**: Verify help output no longer mentions JSONL rebuild or FTS5 search
3. **Search behavior**: Verify `/memory search` uses hybrid retrieval and shows degraded/unavailable on embedding failure
4. **Status display**: Verify `/memory status` never prints `disabled` or `fts-only`
5. **Consistency check**: Verify `/memory check-consistency` never mentions JSONL
6. **Rebuild commands**: Verify `/memory rebuild-fts` and `/memory rebuild-vec` remain functional
7. **Type checking**: Run type checker to verify no type errors from removed fields/methods

## Risks

1. **Async conversion for search**: If `mem.retriever.search()` needs to become async, this requires careful handling in the dispatch mechanism. Mitigation: check current signature first before deciding on async conversion.
2. **`rebuild_vec` config dependency**: Currently depends on `memory_embed_enabled` config key which is being removed. Mitigation: replace with runtime `EmbeddingClient.get_status()` check.
3. **Status display simplification**: Removing `fts_fallback_count` and `last_retrieval_mode` removes diagnostic information. Mitigation: these metrics can be added back later if needed via telemetry rather than CLI display.
4. **Test breakage**: Existing tests for removed commands will fail. Mitigation: tests are handled separately per the requirement spec.
