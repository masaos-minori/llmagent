## Goal

Add an explicit `workflow_schema_version` table to `workflow.sqlite`, record the current schema version on creation, and validate the recorded version against the expected version at both agent startup and deploy-time setup — failing with a remediation message on mismatch.

## Scope

**In-Scope:**
- `scripts/db/schema_sql.py`: add `workflow_schema_version` table DDL and a `WORKFLOW_SCHEMA_VERSION` constant
- `scripts/db/create_schema.py`: after creating/migrating `workflow.sqlite`, idempotently record the current version
- `scripts/agent/repl_health.py`: extend the existing `check_workflow_schema()` function (already validates required tables/columns) to also compare the recorded schema version against the expected constant
- `deploy/setup_services.sh`: extend the pre-flight block (from `plans/done/20260710-152749_plan.md`) to also print/compare the workflow schema version

**Out-of-Scope:**
- Any change to the existing table/column structure of `tasks`, `attempts`, `processed_events`, `artifacts`, `approvals` — the Notes explicitly forbid breaking existing tables/migrations
- Rewriting `_WORKFLOW_MIGRATIONS` into a versioned-migration framework (e.g. one version bump per migration with rollback support) — out of scope; this plan adds version *tracking*, not a new migration *engine*. Future migrations can bump the version constant when added, per Design.
- `deploy/init_db.sh` — not named in this requirement's Scope (though it is the script that triggers `create_schema.py`, which is where the version gets recorded; no additional changes are needed there beyond what already runs)

## Assumptions

1. `scripts/agent/repl_health.py::check_workflow_schema()` (lines 455-484) already exists and is the actual "startup validation" referenced in the requirement's Task "Include schema version in startup validation output" — confirmed by direct read; it already validates required tables and required columns per table (`REQUIRED_WORKFLOW_TABLES`, lines 446-452), going beyond what `plans/done/20260710-152554_plan.md`'s deploy-time shell check does. This plan extends this existing function rather than creating a new one.
2. `deploy/setup_services.sh`'s pre-flight block (from `plans/done/20260710-152749_plan.md`) is the actual "setup validation output" referenced in the requirement's other Task — this plan extends that same block rather than adding a separate check.
3. `_WORKFLOW_MIGRATIONS` (5 existing `ALTER TABLE` statements in `schema_sql.py`) represents the schema state as it exists today, before this plan — since no version was ever tracked before, there is no prior recorded version to preserve or migrate from. This plan defines `WORKFLOW_SCHEMA_VERSION = "1.0.0"` as the version label for "the current full schema shape, including all 5 pre-existing migrations" — a starting point, not a retroactive reconstruction of version history. Any *future* addition to `_WORKFLOW_MIGRATIONS` should be paired with incrementing this constant (documented in a code comment at the constant's definition).
4. `workflow_schema_version` is designed as an append-only log table (one row per version ever applied, with `applied_at`), not a single-row config table — this matches the Suggested Table Shape's `applied_at TEXT NOT NULL DEFAULT current UTC timestamp` column, which only makes sense if multiple rows can exist over the DB's lifetime. "Current version" is defined as the row with the maximum `applied_at` (equivalently, the most recently inserted row, since versions are only ever appended forward).
5. Recording a new version row happens **only** when the latest existing row's version differs from `WORKFLOW_SCHEMA_VERSION` (or no row exists yet) — this satisfies "Existing DBs can be migrated idempotently" (Acceptance Criteria) by ensuring `create_schema.py` re-runs do not insert duplicate rows for a version already recorded.
6. `deploy/setup_services.sh` (a bash script) needs the expected-version constant without duplicating it as a second hardcoded string in shell — resolved by having the script query it from Python (`uv run python -c "from db.schema_sql import WORKFLOW_SCHEMA_VERSION; print(WORKFLOW_SCHEMA_VERSION)"`) rather than hardcoding `"1.0.0"` a second time, so the two locations cannot drift out of sync.

## Unknowns

No unknowns — all design decisions above are resolved by analysis of the existing, already-read codebase (schema DDL, migration list, existing validation functions).

## Design

### `schema_sql.py`

```python
# Bump this constant whenever a new entry is added to _WORKFLOW_MIGRATIONS.
WORKFLOW_SCHEMA_VERSION = "1.0.0"
```

Add to `_WORKFLOW_SCHEMA` (after the existing `approvals` table DDL):
```sql
CREATE TABLE IF NOT EXISTS workflow_schema_version (
    version    TEXT NOT NULL,
    applied_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))
);
```

### `create_schema.py`

```python
def _record_workflow_schema_version(db) -> None:
    row = db.execute(
        "SELECT version FROM workflow_schema_version ORDER BY applied_at DESC LIMIT 1"
    ).fetchone()
    current = row[0] if row else None
    if current != WORKFLOW_SCHEMA_VERSION:
        db.execute(
            "INSERT INTO workflow_schema_version (version) VALUES (?)",
            (WORKFLOW_SCHEMA_VERSION,),
        )
```
Called at the end of `create_workflow_schema()`, after the existing `_WORKFLOW_MIGRATIONS` loop and before `db.commit()`.

### `repl_health.py`

Extend `check_workflow_schema()` (after its existing table/column loop):
```python
from db.schema_sql import WORKFLOW_SCHEMA_VERSION  # noqa: PLC0415

row = db.fetchone(
    "SELECT version FROM workflow_schema_version ORDER BY applied_at DESC LIMIT 1", ()
)
actual_version = row[0] if row else None
if actual_version != WORKFLOW_SCHEMA_VERSION:
    raise RuntimeError(
        f"Workflow schema version mismatch: expected {WORKFLOW_SCHEMA_VERSION!r}, "
        f"found {actual_version!r}. Run create_workflow_schema() to migrate."
    )
```

### `deploy/setup_services.sh`

Extend the pre-flight block from `plans/done/20260710-152749_plan.md` (after its existing table-existence loop):
```bash
EXPECTED_SCHEMA_VERSION=$(PYTHONPATH=/opt/llm/scripts uv run python -c \
  "from db.schema_sql import WORKFLOW_SCHEMA_VERSION; print(WORKFLOW_SCHEMA_VERSION)")
ACTUAL_SCHEMA_VERSION=$(sqlite3 "${WORKFLOW_DB}" \
  "SELECT version FROM workflow_schema_version ORDER BY applied_at DESC LIMIT 1;")
echo "Workflow schema version: ${ACTUAL_SCHEMA_VERSION:-<none>} (expected: ${EXPECTED_SCHEMA_VERSION})"
if [ "${ACTUAL_SCHEMA_VERSION}" != "${EXPECTED_SCHEMA_VERSION}" ]; then
  echo "[FATAL] Workflow schema version mismatch: expected ${EXPECTED_SCHEMA_VERSION}, found ${ACTUAL_SCHEMA_VERSION:-<none>}." >&2
  echo "Run deploy/init_db.sh to migrate the workflow schema before starting services." >&2
  exit 1
fi
```

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| A future contributor adds a new entry to `_WORKFLOW_MIGRATIONS` without bumping `WORKFLOW_SCHEMA_VERSION`, silently desynchronizing the version label from the actual schema shape | Medium | Mitigated by the code comment placed directly above `WORKFLOW_SCHEMA_VERSION`'s definition ("Bump this constant whenever a new entry is added to `_WORKFLOW_MIGRATIONS`"); a follow-up could add a lightweight test asserting `len(_WORKFLOW_MIGRATIONS)` matches an expected count tied to the version, but that is not required by this requirement's Acceptance Criteria and is left as a future improvement |
| Existing production `workflow.sqlite` files (created before this change) will show `actual_version = None` on first startup/setup check after this change deploys, triggering a hard failure until `create_schema.py` is re-run | Medium | This is the intended, correct behavior per the Acceptance Criteria ("Schema version mismatch is detected") — the remediation message in both `check_workflow_schema()` and `setup_services.sh` explicitly instructs re-running the schema initialization step, and `create_workflow_schema()`'s DDL is additive/idempotent so re-running it against an existing DB is safe and required regardless |
| `deploy/setup_services.sh`'s Python-subprocess call to read `WORKFLOW_SCHEMA_VERSION` adds a second `uv run python` invocation to an already `uv run`-dependent pre-flight block | Low | Accepted — this project's deploy scripts already depend on `uv run python` for the workflow-validation CLI (`plans/done/20260710-152424_plan.md`); one additional lightweight invocation is consistent with that established pattern, not a new class of dependency |
