# Plan: Event Bus の導入 — HTTP publish / SSE subscribe / Replay / DLQ

## 1. Goal

`scripts/eventbus/` パッケージを新規追加し、HTTP publish + SSE subscribe + SQLite 永続化 + since_seq Replay + Dead Letter Queue を備えた at-least-once 配信基盤を既存コードへの変更ゼロで構築する。

---

## 2. Scope

**In-Scope:**
- `scripts/eventbus/__init__.py` — パッケージ初期化
- `scripts/eventbus/app.py` — FastAPI アプリ (`/publish`, `/subscribe`, `/replay`, `/replay`, `/dlq`, `/health`)
- `scripts/eventbus/schema.sql` — `events` テーブル DDL (WAL + UNIQUE constraint)
- `schemas/event_envelope.json` — イベントエンベロープ JSON Schema
- `config/eventbus.toml` — 設定値 (port, DB path, MAX_RETRY, storage dirs)
- `.importlinter` — `eventbus` を root_packages に追加 + isolated contract
- `deploy/deploy.sh` — `config/eventbus.toml` cp 行 + `mkdir -p /opt/llm/{storage,offsets,deadletter}`
- `deploy/setup_services.sh` — Event Bus uvicorn 起動コマンド追加
- `deploy/init_db.sh` — eventbus DB 初期化ステップ追加
- `routing.md` — eventbus doc entry 追加
- `tests/test_eventbus_*.py` — フェーズ別テスト

**Out-of-Scope:**
- 既存 `scripts/agent/`, `scripts/mcp/`, `scripts/rag/`, `scripts/db/`, `scripts/shared/` への変更
- ノード間 replication (Phase 2)
- 認証・認可機構
- メッセージブローカー (Kafka, RabbitMQ 等) の導入

---

## 3. Assumptions

1. `scripts/eventbus/` を `scripts/` 配下に置くことで `deploy.sh` の rsync (`scripts/ → /opt/llm/scripts/`) に自動含まれる。
2. FastAPI (`>=0.115`) と uvicorn (`>=0.30`) は `pyproject.toml` 既存依存として利用可能 — 確認済 (`pyproject.toml:1`)。
3. SSE は Starlette 組み込みの `StreamingResponse(content_type="text/event-stream")` で実装可能 — `sse-starlette` 追加不要。
4. Event Bus の SQLite WAL モードは既存 `SQLiteHelper` と同等パターンで実装できる。
5. Port 8015 は未使用 — 確認済 (config/mcp_servers.toml では 8004–8012 使用; rag_pipeline_mcp_server.toml では 8010; mdq_mcp_server.json では 8013; rag_pipeline_mcp_server.json では 8014)。
6. `eventbus` は他の既存パッケージ (`agent`, `mcp`, `rag`, `db`, `shared`) を import しない — import-linter forbidden contract で強制する。
7. ISO 8601 datetime 文字列は辞書順 = 時系列順として比較可能 (タイムゾーン指定なし形式の場合)。

## 6. Implementation Steps

### Phase 1: コア基盤

**Step 1-A: deploy / config 基盤整備**

- `deploy/deploy.sh` に追加:
  ```bash
  mkdir -p /opt/llm/storage
  mkdir -p /opt/llm/offsets
  mkdir -p /opt/llm/deadletter
  cp "${REPO_ROOT}/config/eventbus.toml" "${DEPLOY_CONFIG}/"
  cp -r "${REPO_ROOT}/schemas/" /opt/llm/schemas/
  ```
- `config/eventbus.toml` を作成:
  ```toml
  port = 8015
  db_path = "/opt/llm/db/eventbus.sqlite"
  storage_dir = "/opt/llm/storage"
  offsets_dir = "/opt/llm/offsets"
  deadletter_dir = "/opt/llm/deadletter"
  max_retry = 3
  ```
- 完了基準: `deploy.sh` が dry-run で mkdir + cp をエラーなし実行できる。

**Step 1-B: `schemas/event_envelope.json` 作成**

- `schemas/` ディレクトリ作成
- `schemas/event_envelope.json` — JSON Schema draft-07:
  - `required`: `event_id`, `topic`, `payload`, `producer`, `published_at`
  - `optional`: `schema_version` (default `"1.0"`)
  - `event_id`: UUID v4 形式 (`pattern: "^[0-9a-f]{8}-..."`)
  - `published_at`: ISO 8601 (`format: "date-time"`)
- 完了基準: `jsonschema` で validate できる。

**Step 1-C: `.importlinter` 更新**

```ini
root_packages =
    agent
    db
    mcp
    rag
    shared
    eventbus        # 追加

[importlinter:contract:eventbus-is-isolated]
name = eventbus must not import from agent, mcp, rag, db, or shared
type = forbidden
source_modules =
    eventbus
forbidden_modules =
    agent
    mcp
    rag
    db
    shared
```

- 完了基準: `lint-imports` が 0 violation で通過する。

**Step 1-D: `scripts/eventbus/schema.sql` 作成**

```sql
PRAGMA journal_mode=WAL;

CREATE TABLE IF NOT EXISTS events (
    seq          INTEGER PRIMARY KEY AUTOINCREMENT,
    event_id     TEXT    NOT NULL UNIQUE,
    topic        TEXT    NOT NULL,
    payload      TEXT    NOT NULL,
    producer     TEXT    NOT NULL,
    published_at TEXT    NOT NULL,
    acked_at     TEXT,
    retry_count  INTEGER NOT NULL DEFAULT 0,
    dlq_at       TEXT
);

CREATE INDEX IF NOT EXISTS idx_events_topic ON events(topic);
CREATE INDEX IF NOT EXISTS idx_events_seq   ON events(seq);
```

- 完了基準: SQLite で `.read schema.sql` がエラーなし。

**Step 1-E: `scripts/eventbus/app.py` — Phase 1 エンドポイント**

- `/health` (GET): `{"status": "ok"}` 返却
- `/publish` (POST):
  1. リクエストボディを `event_envelope.json` スキーマで validate
  2. `INSERT OR IGNORE INTO events(event_id, topic, payload, producer, published_at) VALUES(?, ?, ?, ?, ?)`
  3. `INSERT OR IGNORE` で重複は無視 (冪等性)
  4. `storage/events.jsonl` へ atomic append (write + flush)
  5. 成功: `{"seq": <int>, "event_id": "<uuid>"}` / 重複: HTTP 200 (同一 seq 不定 — IGNORE のため)
- 設定値は `config/eventbus.toml` から読む (tomllib / tomli)

- 完了基準: `tests/test_eventbus_phase1.py` 全テスト pass (publish 正常/冪等/バリデーションエラー/health)。

**Step 1-F: `tests/test_eventbus_phase1.py` 作成**

- `TestClient(app)` を使用
- テスト: `test_health_returns_ok`, `test_publish_inserts_event`, `test_publish_idempotent`, `test_publish_invalid_schema`

### Phase 2: Replay と SSE

**Step 2-A: `/replay` エンドポイント**

- `GET /replay?since_seq=<n>&format=json|sse`
- SQLite: `SELECT * FROM events WHERE seq > ? ORDER BY seq`
- `format=json`: JSON array 返却
- `format=sse` (default): SSE ストリームで各イベントを送信
- 完了基準: `since_seq=0` で全件取得できる。

**Step 2-B: `/subscribe` SSE エンドポイント**

- `GET /subscribe?topic=<t1>&topic=<t2>&since_seq=<n>`
- 実装: async generator で新規 INSERT を polling (500ms 間隔) または SQLite WAL 監視
- `topic` フィルタ: 複数トピック OR 条件
- `since_seq`: 指定時は未送信イベントをまず replay してから live stream
- 切断検知: `asyncio.CancelledError` で finally クリーンアップ
- 完了基準: SSE ストリームで publish したイベントをリアルタイム受信できる。

**Step 2-C: offset ユーティリティ**

- `offsets/<consumer_id>` ファイルへの read/write ユーティリティ関数
- `write_offset(consumer_id: str, seq: int) -> None`
- `read_offset(consumer_id: str) -> int`  (ファイルなし → 0)
- 完了基準: `tests/test_eventbus_phase2.py` 全テスト pass。

**Step 2-D: `tests/test_eventbus_phase2.py` 作成**

- テスト: `test_replay_returns_events_since_seq`, `test_subscribe_receives_published_event`, `test_offset_read_write`

### Phase 3: DLQ とリトライ

**Step 3-A: DLQ 移送ロジック**

- バックグラウンドタスク (FastAPI `lifespan` または `BackgroundTasks`): 定期実行 (60s 間隔)
- `SELECT * FROM events WHERE retry_count >= MAX_RETRY AND dlq_at IS NULL`
- 各行:
  1. `deadletter/<event_id>.json` へ atomic write (temp file + `os.rename`)
  2. `UPDATE events SET dlq_at = ? WHERE event_id = ?`
- 完了基準: `retry_count >= MAX_RETRY` のイベントが `deadletter/` に移送される。

**Step 3-B: `/dlq` エンドポイント**

- `GET /dlq`: DLQ 内イベント一覧 (`dlq_at IS NOT NULL`) を JSON 返却
- `POST /dlq/<event_id>/requeue`: `retry_count=0, dlq_at=NULL` にリセットして再投入
- 完了基準: DLQ に移送されたイベントを `/dlq` で一覧取得し、requeue できる。

**Step 3-C: `tests/test_eventbus_phase3.py` 作成**

- テスト: `test_dlq_promotion_when_retry_exhausted`, `test_dlq_requeue`, `test_dlq_list`

## 8. Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| SSE 接続 fd リーク — 切断クライアントのジェネレータが GC されない | Med | `try/finally` + `asyncio.CancelledError` ハンドリング; heartbeat_timeout を config 値で設定 |
| JSONL 書き込みの partial write — クラッシュ時に行が truncate される | Low | 1 write() call で行全体 + newline を書く; `flush()` 後に return; SQLite が信頼できる唯一ストア |
| `.importlinter` 更新漏れ — `eventbus` が他パッケージを import しても CI が検知しない | Med | Step 1-C を最初に実装し、CI が forbidden import を即時検知できる状態を作る |
| deploy.sh の runtime ディレクトリ未作成 — 起動クラッシュ | High (現状のギャップ) | Step 1-A で `mkdir -p` 行を追加; deploy 後に `ls /opt/llm/{storage,offsets,deadletter}` で確認 |
| SSE polling 間隔 (500ms) による遅延 — リアルタイム性の要件が変化 | Low | polling 間隔を `eventbus.toml` の設定値にし、将来 NOTIFY/WAL hook に差し替え可能にする |
| Port 8015 衝突 — 将来 MCP サーバーが 8015 を使用 | Low | `config/mcp_servers.toml` と `setup_services.sh` を参照し、8015 は Event Bus 専用として明記 |
