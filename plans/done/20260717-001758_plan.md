# Plan: Harden ephemeral message lifecycle, history compression, and memory boundaries

## Goal

Enforce a strict, verified lifecycle boundary for ephemeral system messages
(`_ephemeral`, `_skill_ephemeral`, `_memory_injected`) so they never leak into
session persistence, memory extraction, or compressed history, and so they
reliably reach the LLM for the turn they were created for — and fix the
confirmed defect where they currently do not.

## Scope

**In scope:**
- `scripts/agent/orchestrator.py` — ephemeral/memory-injected cleanup ordering,
  `_sync_system_prompt()`, `_process_turn()`, `_handle_memory_injection()`
- `scripts/agent/mode_classification.py` — hint injection (read-only verification;
  change only if lifecycle enforcement requires it)
- `scripts/agent/history.py` — fallback truncation retention priority, ephemeral
  exclusion, stats
- `scripts/agent/history_selection_policy.py` — compression candidate selection,
  category/priority representation (this is the actual file; the requirement's
  `agent/history_selection.py` does not exist — see Assumptions #1)
- `scripts/agent/session.py` — confirm no ephemeral/memory-injected persistence path
- `scripts/agent/services/undo_service.py` — `/undo` ephemeral handling
- `scripts/agent/commands/cmd_skill.py` — `_skill_ephemeral` interaction with the
  fix (read-only verification unless the fix requires alignment)
- New/extended tests in `tests/test_orchestrator.py`, a new
  `tests/test_mode_classification.py`, `tests/test_history_manager.py`,
  `tests/test_history_selection_policy.py`, `tests/test_regression_undo_artifact.py`

**Out of scope (per requirement Non-goals):**
- Redesigning the memory system (`scripts/agent/memory/*` internals) — only
  verifying the extraction/injection boundary already excludes ephemeral content
- Removing persistent system prompts
- Changing MDQ/RAG classification logic (`scripts/agent/mdq_rag_classifier.py`)
- Changing the session DB schema

## Assumptions

1. `agent/history_selection.py` in the requirement refers to
   `scripts/agent/history_selection_policy.py` — confirmed by
   `tests/test_history_selection_policy.py` importing
   `from agent.history_selection_policy import HistorySelectionPolicy`.
   (Evidence: Explicit in code.)
2. `/undo` should strip preceding `_ephemeral` messages using the same rule it
   already applies to `_memory_injected` messages, since both are turn-scoped
   artifacts that should not outlive the turn they were created in. This is a
   reasonable default reading of "Define `/undo` behavior for ephemeral
   messages," not a literal statement in the requirement. (Evidence: Strongly
   implied by code symmetry with existing `_memory_injected` handling in
   `undo_service.py:42`.)
3. `_skill_ephemeral` messages (from `/skill`) are a subset of `_ephemeral`
   messages (every `_skill_ephemeral: True` message also has `_ephemeral: True`,
   per `cmd_skill.py:51-53`) and should be covered by the same lifecycle fix
   without special-casing. (Evidence: Explicit in code.)
4. The fix must not change the documented behavior that `_ephemeral` messages
   are "re-evaluated every turn" (doc: `docs/05_agent_03_01_turn-processing-flow-overview.md:112-114`)
   — i.e., the hint must still be recomputed each turn, not cached/reused.

## Design

### Confirmed defect (Explicit in code, cross-referenced against docs)

`_process_turn()` (orchestrator.py:431-462) calls, in this order:
1. `_handle_memory_injection(line)` — appends a `_memory_injected` system message
2. `classify_and_inject_mode(line, ctx)` — appends an `_ephemeral` system message
3. `_append_user_message(line)` — which calls `_sync_system_prompt()` **first**

`_sync_system_prompt()` (orchestrator.py:503-522) unconditionally strips all
`_ephemeral` and `_memory_injected` messages from `ctx.conv.history` before
syncing the system prompt and appending the user message.

Because step 3 runs immediately after steps 1-2 **within the same turn**, and
no LLM call happens until step 5 (`_handle_llm_turn`), the messages injected in
steps 1-2 are removed before the LLM ever sees them. This contradicts the
documented design (`docs/05_agent_03_01_turn-processing-flow-overview.md:112-114`):
"`_ephemeral` messages are removed by `Orchestrator._sync_system_prompt()` at
the start of the next turn." In the current code, the removal happens at the
end of the *same* turn's setup phase, not the start of the next one.

This is confirmed by:
- Full manual trace of `_process_turn` → `_append_user_message` → `_sync_system_prompt`.
- No existing test asserts the ephemeral/memory-injected content is present in
  the outgoing LLM payload (152 tests in the affected test files pass today,
  none catch this).
- `git log`/doc cross-reference: the doc describes intended behavior that the
  code does not implement.

### Fix strategy

Split `_sync_system_prompt()`'s two responsibilities:
1. **Leftover cleanup** (remove `_ephemeral`/`_memory_injected` from the
   *previous* turn) — move this to run at the **start** of `_process_turn()`,
   before `_handle_memory_injection()` and `classify_and_inject_mode()`.
2. **System prompt content sync** (refresh `history[0]` or insert it) — keep
   this where it currently runs, immediately before the user message is
   appended, since it must reflect any `ctx.conv.system_prompt_content` changes
   made during the turn (e.g. via `/system` command) before the LLM call.

This preserves the "re-evaluated every turn" property (Assumption #4): the
hint is still recomputed each turn from a clean slate, but the injected
content now survives long enough to reach the LLM call for its own turn.

### Ephemeral message category/priority representation

`HistorySelectionPolicy.classify()` and `classify_importance()` already encode
category and priority, but `select_turns_to_compress()` removes all
system-role messages (`system_msgs`) from the compression candidate pool
*before* `classify()` is ever applied to them — making the "system" branch of
`classify()` unreachable in that call path. Since this already achieves the
requirement's goal (ephemeral messages, being system-role, are never
summarized), no behavior change is needed here. Add a short comment
documenting why system-role messages bypass `classify()` entirely, so a future
reader does not "fix" the apparent dead branch by wiring it in and
accidentally making ephemeral messages compressible.

### `/undo` ephemeral handling

Extend `undo_service.py`'s backward-scan loop (currently
`while cut_idx > 0 and ctx.conv.history[cut_idx - 1].get("_memory_injected")`)
to also match `_ephemeral`, so `/undo` removes both memory-injected and
ephemeral mode-hint messages that preceded the undone user turn — consistent
handling per Assumption #2.

### Diagnostics for fallback truncation

The requirement asks for `/context` or diagnostics to expose fallback
truncation state. `HistoryManager.stat_fallback_truncate_count` already exists
(history.py:89, incremented at line 311). Confirm whether `/context` (or
equivalent status command) currently surfaces `stat_compress_count` but omits
`stat_fallback_truncate_count`, and if so, add it to the existing status
output rather than creating a new command.

## Implementation Steps

1. **Preparation**: reproduce the same-turn ephemeral wipe with a minimal
   failing test (assert the LLM call payload contains the injected hint for
   its own turn) before changing any production code, to lock in the
   regression baseline.
2. **Core fix**: split `_sync_system_prompt()` in `orchestrator.py`; move the
   ephemeral/memory-injected cleanup to the start of `_process_turn()`; keep
   system-prompt-content sync where the user message is appended.
3. **`/undo` fix**: extend `undo_service.py`'s backward-scan predicate to
   include `_ephemeral`.
4. **Diagnostics**: expose `stat_fallback_truncate_count` in the existing
   status/`/context` output if not already present.
5. **Documentation comment**: add the "why system-role messages bypass
   `classify()`" note in `history_selection_policy.py`.
6. **Test coverage**:
   - New `tests/test_mode_classification.py`: hint injection, no duplication
     across repeated tool-loop iterations, hint present in the same turn.
   - Extend `tests/test_orchestrator.py`: ephemeral/memory-injected survive to
     the LLM payload for their own turn, and are gone by the *next* turn's
     payload; `/skill` content survives to the following turn's LLM call.
   - Extend `tests/test_regression_undo_artifact.py`: `/undo` strips a
     preceding `_ephemeral` message.
   - Extend `tests/test_history_manager.py` /
     `tests/test_history_selection_policy.py`: compression success, failure,
     protected turns, fallback order, ephemeral exclusion (already largely
     covered — add any missing fallback-order case).
7. **Verification**: run the full validation sequence (`rules/toolchain.md`)
   scoped to the changed files, then the full suite.
8. **Deploy impact check**: confirm no `deploy/deploy.sh` change is needed
   (no new/removed files under `scripts/`).

## Risks

- **Risk**: `orchestrator.py` has very high churn (50 commits in the last 30
  days, including concurrent edits observed during this same planning session)
  → **Mitigation**: keep the fix to the smallest possible diff (reordering two
  existing calls, not rewriting `_process_turn`), rebase immediately before
  implementation, and re-run the full orchestrator test file after rebase.
- **Risk**: Moving the cleanup call earlier in `_process_turn()` could
  interact with `_handle_memory_injection()`'s or `classify_and_inject_mode()`'s
  own assumptions about what is already in `ctx.conv.history` (e.g., if either
  function reads `ctx.conv.history` expecting last turn's ephemeral hint to
  already be gone) → **Mitigation**: re-read both functions immediately before
  implementation to confirm neither depends on stale ephemeral content being
  present; both currently only *append*, so this risk is low but must be
  re-verified at implementation time.
- **Risk**: `HistorySelectionPolicy.classify()`'s unreachable "system" branch
  could tempt a future contributor (or an automated refactor) to "fix" it by
  routing system messages back into the compressible pool, silently
  reintroducing ephemeral-message summarization → **Mitigation**: the
  documentation comment in Implementation Step 5 exists specifically to
  prevent this.
- **Risk**: No dedicated test file exists for `mode_classification.py` today,
  so any regression there is currently invisible → **Mitigation**: Implementation
  Step 6 adds one before the fix lands, not after.
- **Risk**: This planning session observed a concurrent process committing
  directly to `master` on files including `orchestrator.py` during the same
  session — the affected-area line counts and churn figures above may already
  be stale by the time implementation starts → **Mitigation**: re-run Step 3
  (read related source files) immediately before implementation begins, not
  from this plan's cached findings.
