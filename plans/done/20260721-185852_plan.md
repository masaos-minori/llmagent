# Plan: Make memory retrieval degrade to FTS-only when vector KNN fails

## Goal

When vector KNN search fails in `HybridRetriever.search()`, fall back to FTS-only results instead of propagating the error. This ensures memory injection continues working when vector tables are unavailable.

## Scope

- `scripts/agent/memory/retriever.py` — add exception handling around vector KNN call in `HybridRetriever.search()`
- `scripts/agent/commands/memory_status.py` — verify `/memory status` exposes `last_retrieval_mode` and `fts_fallback_count`
- `tests/test_memory_retriever.py` — add tests for vector failure fallback
- `tests/test_memory_status.py` — verify fallback metrics exposure

## Assumptions

- `HybridRetriever` already has `last_retrieval_mode` and `fts_fallback_count` attributes defined
- `fts_hits` variable exists in `search()` method body before the vector KNN call
- `sqlite3.OperationalError` is the primary exception to catch; other sqlite-vec exceptions may also occur
- Direct `HybridRetriever.knn_search()` should remain fail-fast (no changes needed there)

## Unknowns

Resolved — all four unknowns answered by code inspection:

1. `last_retrieval_mode` and `fts_fallback_count` already exist and are initialized in `HybridRetriever.__init__` (lines 209-210).
2. FTS results variable is `fts_hits` in `search()` (line 226).
3. Only `sqlite3.OperationalError` is known to be raised by `VectorRetriever.knn_search()` (documented at line 157). No evidence of other exception types.
4. `/memory status` already exposes both `last_retrieval_mode` and `fts_fallback_count` via `MemoryStatus` dataclass fields.

## Affected areas

- Memory retrieval pipeline — hybrid search degrades gracefully
- Memory status reporting — fallback metrics become visible
- Test coverage — new regression tests for graceful degradation

## Design

### Code change in `HybridRetriever.search()`

```python
def search(self, query: str, embedding: Optional[list[float]] = None) -> list[dict]:
    fts_hits = self._fts_search(query)
    
    if embedding is not None:
        try:
            vec_hits = self.knn_search(embedding)
            # merge logic...
        except sqlite3.OperationalError as e:
            logger.warning("Vector KNN failed, falling back to FTS-only: %s", e)
            self.last_retrieval_mode = "fts_only"
            self.fts_fallback_count += 1
            return fts_hits
    
    return merged_results
```

Key decisions:
- Only catch `sqlite3.OperationalError` initially (narrow scope per requirements)
- Log warning with exception class and message
- Update both `last_retrieval_mode` and `fts_fallback_count`
- Return raw `fts_hits` without merging

### Alternative consideration

If `fts_hits` is not yet computed before the vector call, we would need to restructure to compute FTS first regardless. Based on the requirement ("keep FTS search fail-fast"), FTS should be computed first anyway.

## Implementation steps

### Step 1: Add exception handling in `HybridRetriever.search()`

Wrap the vector KNN call (`self._vec.knn_search(...)`) with `try/except sqlite3.OperationalError`:
- Log warning with exception details
- Set `last_retrieval_mode = "fts_only"`
- Increment `fts_fallback_count`
- Return `fts_hits`

Note: `last_retrieval_mode` and `fts_fallback_count` already exist and are initialized. `/memory status` already exposes both metrics.

### Step 2: Add/update tests

In `tests/test_memory_retriever.py`:
- Mock `knn_search` to raise `sqlite3.OperationalError`
- Call `search()` with embedding
- Assert FTS results returned
- Assert `last_retrieval_mode == "fts_only"`
- Assert `fts_fallback_count` incremented
- Assert warning logged

In `tests/test_memory_status.py`:
- Verify fallback metrics appear in status output

### Step 3: Run validation

- `uv run pytest` — ensure no regressions
- `uv run mypy scripts/` — type check passes
- `uv run ruff check scripts/` — lint clean

## Risks

### Risk 1: Missing attributes initialization

If `last_retrieval_mode` or `fts_fallback_count` are not initialized on the instance, the fallback will raise `AttributeError`. Mitigation: verify during Step 1; add initialization if missing.

### Risk 2: Narrow exception filtering

If only `sqlite3.OperationalError` is caught and other exceptions (e.g., `sqlite3.DatabaseError`) are raised by sqlite-vec, they will still propagate. Mitigation: consider catching `sqlite3.DatabaseError` as parent class, or add specific sqlite-vec exception types once identified.

### Risk 3: Silent data loss

Users won't know vector search failed unless they check `/memory status`. Mitigation: warning log + status metrics provide visibility. Consider adding a subtle indicator in normal search results.
