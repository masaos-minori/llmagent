# Plan: Record and analyze round-wide serialization caused by side-effect tools

## Goal

Persist detailed serialization events so the team can measure the real cost of safe serialization before changing execution rules. Capture which tool triggered serialization, how many tools were affected, and timing impact.

## Scope

**In scope:**
- Extend `_serialization_stats` from in-memory counters to persistent diagnostic entries
- Record per-round serialization events with: trigger_tool, affected_tool_count, mode (serial/parallel), elapsed_ms, serialization_reason
- Update `write_round_exec()` audit event to include detailed serialization metadata
- Add timing comparison: record what tools could have run in parallel vs actual serial execution
- Expose serialization data in `/mcp stats` or a new `/db` subcommand

**Out of scope:**
- Changing serialization rules (this requirement is purely about measurement)
- Adding optimization based on collected data (future phase)
- Modifying DAG scheduling algorithm
- Real-time alerts or dashboards (data collection only)

## Assumptions

1. `_serialization_stats` in `tool_runner.py:35-44` already tracks aggregate counts — we will extend this with per-event persistence.
2. `DiagnosticStore` can store serialization events as structured entries with kind `"serialization_event"`.
3. `write_round_exec()` in `tool_audit.py:153-176` is the right place for round-level serialization metadata.
4. The `trigger_tool` field already exists in `write_round_exec()` — we will enhance it with full details.
5. Timing impact can be measured by comparing actual serial elapsed_ms vs estimated parallel time (sum of fastest individual tool execution).

## Implementation Steps

1. **Extend RuntimeStats in `context.py`**
   - Add fields: `stat_serialization_events: list[dict]` (per-round events) or persistent counter approach
   - Alternative: keep in-memory for session lifetime, flush to DiagnosticStore on turn/round end
   - Completion criterion: RuntimeStats can track serialization events with trigger_tool, affected_count, elapsed_ms

2. **Enhance serialization tracking in `tool_runner.py`**
   - In `_execute_standard()`: when `has_side_effect=True`, capture:
     - `trigger_tool`: the tool name that triggered serialization (first side-effect tool found)
     - `affected_tools`: list of all tool names in the round
     - `serial_reason`: "side_effect" or "config_serial_tool_calls"
   - In `_execute_with_dag()`: capture DAG-level serialization events from `build_execution_groups()` results
   - Store per-round event in `ctx.stats.stat_serialization_events` (in-memory for session)

3. **Persist serialization events to DiagnosticStore**
   - Add `DiagnosticStore.save_serialization_event()` method
   - Called at end of each round with: round_id, trigger_tool, affected_count, mode, elapsed_ms, reason
   - Kind: `"serialization_event"`, content: structured JSON with all metadata
   - Completion criterion: serialization events persist beyond session lifetime

4. **Enhance `write_round_exec()` in `tool_audit.py`**
   - Add parameters: `trigger_tool`, `affected_tools`, `serial_reason`, `estimated_parallel_ms`
   - Include these fields in the audit event payload
   - Completion criterion: round exec audit events contain full serialization context

5. **Update `/mcp stats` command**
   - Show serialization summary: total events, avg affected tools, top trigger tools
   - Format: key-value table with breakdown by reason (side_effect vs config)
   - Completion criterion: `/mcp stats` displays serialization metrics

6. **Add timing comparison logic**
   - For each round with serialization: record actual elapsed_ms
   - Estimate parallel time: sum of min(elapsed_ms) for each unique tool in round (conservative estimate)
   - Store ratio: `serial_overhead = actual_elapsed / estimated_parallel`
   - Completion criterion: serialization events include timing impact ratio
