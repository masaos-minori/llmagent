# Remove silent fallback from workflow execution in production

**Requirement:** `requires/20260624_61_require.md`

## Goal

Make `workflow_mode` config-driven and fail fast in `required` mode so that production never silently falls back from workflow execution to direct execution when workflow prerequisites are missing.

## Scope

**In:**
- `scripts/agent/config_builders.py` — read `workflow_mode` from config dict in `build_agent_config()`
- `scripts/agent/config_dataclasses.py` — update docstring: workflow_mode default is overridden by config file
- `scripts/agent/orchestrator.py` — raise `RuntimeError` at construction time when `workflow_mode=required` and `WorkflowLoader` fails
- `config/common.toml` — add `workflow_mode = "required"` (deployed to production via `deploy.sh`)
- `docs/05_agent_08_configuration.md` — update `workflow_mode` documentation with production/dev guidance
- `tests/test_orchestrator.py` — add test for required mode failing fast when WorkflowLoader raises
- `tests/test_config_dataclasses.py` — update `test_workflow_mode_default` assertion (stays `"auto"`)

**Out:**
- Approval model redesign
- Workflow phase expansion
- Removing `_log_fallback()` or the `disabled` mode path
- Changes to `WorkflowLoader` or workflow definition format

## Assumptions

1. `workflow_mode` is not currently read from any config file — only the dataclass default `"auto"` is used. Confirmed: `build_agent_config()` does not call `cfg.get("workflow_mode", ...)`.
2. `config/common.toml` is the canonical source for production config; `deploy.sh:58` copies it to `/opt/llm/config/common.toml`.
3. The dataclass default `"auto"` is intentionally kept (safety net for local/dev environments that don't set `workflow_mode` in config).
4. `Orchestrator.__init__()` runs during `startup.py._init_orchestrator()` which is in the critical startup path — raising there is equivalent to startup failure (user sees error immediately).
5. `WorkflowLoader().load()` raises `WorkflowLoadError` (or subclass) when prerequisites are missing.
6. Tests use `workflow_mode="disabled"` to bypass WorkflowLoader in fast unit tests — no change to test patterns.

## Design

### Root Cause

`build_agent_config()` does not read `workflow_mode` from the config dict. The value is always the dataclass default `"auto"`. There is no way to set `workflow_mode=required` via config files.

### Fix

#### 1. `config_builders.py` — read `workflow_mode` from config

Add to `build_agent_config()` return statement:
```python
return AgentConfig(
    ...
    workflow_mode=cfg.get("workflow_mode", "auto"),  # add this line
    ...
)
```

#### 2. `config/common.toml` — set production default

Add after existing runtime config section:
```toml
# workflow_mode: "required" = fail fast when workflow prerequisites missing (production)
#                "auto"     = warn and fall back to direct execution (dev/local)
#                "disabled" = always use direct execution (testing only)
workflow_mode = "required"
```

#### 3. `orchestrator.py` — fail fast in required mode

Change `__init__` catch block:
```python
if self._workflow_mode != "disabled":
    try:
        self._workflow_def = WorkflowLoader().load()
    except (WorkflowLoadError, Exception) as exc:
        if self._workflow_mode == "required":
            raise RuntimeError(
                f"[workflow] mode=required but WorkflowLoader failed: {exc}. "
                "Check workflow definition file or set workflow_mode=auto in config."
            ) from exc
        logger.warning(
            "WorkflowLoader failed — workflow tracking disabled (%s)", exc
        )
```

Also update `_log_fallback()` error message for clarity:
```python
def _log_fallback(self, reason: str) -> None:
    if self._workflow_mode == "required":
        raise RuntimeError(
            f"[workflow] mode=required but workflow unavailable: {reason}. "
            "Use /reload to retry or set workflow_mode=auto to downgrade."
        )
    logger.warning(
        "Workflow tracking disabled (%s), falling back to direct execution", reason
    )
```

#### 4. `config_dataclasses.py` — update docstring

Update the field comment:
```python
# workflow_mode: "auto" (fallback with warning), "required" (hard error), "disabled" (always direct).
# Production default is set via config/common.toml (workflow_mode = "required").
# Dataclass default "auto" is used only when no config file sets workflow_mode.
```
