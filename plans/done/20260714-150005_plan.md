# Work Plan: Consolidate MCP Configuration Ownership and Secret-Based Authentication Model

## Goal

Consolidate all overlapping MCP configuration documentation around the confirmed ownership model and replace raw `auth_token` configuration with secret references. Eliminate ambiguity about which file owns which settings.

## Scope

**In-Scope:**
- Document that `agent.toml [mcp_servers.<key>]` contains only five agent-side fields
- Remove `auth_token`, `tool_names`, `env`, `role`, timeouts from agent.toml field list
- Document `config/*_mcp_server.toml` as owner of server-local application settings
- Replace raw `auth_token` documentation with secret-based references (`auth_token_env`, `auth_token_file`)
- Document `tool_definitions` as canonical location for LLM-visible tool schemas
- Document `tool_names` only as optional startup drift validation metadata
- Document `security_profile` as agent-wide setting in `agent.toml`
- Replace stale split-config file names with current `agent.toml` paths
- Replace stale `routing.md` references with `04_mcp_03_routing_lifecycle_and_execution.md`

**Out-of-Scope:**
- Storing raw tokens in `agent.toml` or `*_mcp_server.toml` for production
- Allowing MCP servers to read `agent.toml`
- Making `tool_names` a routing source
- Reintroducing old split agent config files

## Assumptions

1. The confirmed specification is: `agent.toml [mcp_servers.<key>]` may contain only `startup_mode`, `transport`, `url`, `healthcheck_mode`, `cmd`.
2. Server-local application configuration belongs in `config/*_mcp_server.toml`.
3. Raw bearer tokens must not be stored in either `agent.toml` or `*_mcp_server.toml` for production.
4. Secret references (`auth_token_env`, `auth_token_file`) are the preferred pattern.
5. Production startup fails when an enabled HTTP MCP server has no resolved token.
6. Local/development may allow missing token with warning.
7. The existing plan format (see `plans/20260714-143629_plan.md`) is the canonical template.

## Design

### Two-Layer Model (reiterated for completeness)

```
Layer 1 - Agent Process Configuration (config/agent.toml)
├── mcp_servers.<key>.startup_mode    # subprocess / persistent / none
├── mcp_servers.<key>.transport       # stdio / http
├── mcp_servers.<key>.url             # HTTP endpoint (for http transport)
├── mcp_servers.<key>.healthcheck_mode # probe / manual
└── mcp_servers.<key>.cmd             # Command to start subprocess (for stdio)

Layer 2 - MCP Server-Local Application Configuration (config/*_mcp_server.toml)
├── allowlists / denylists
├── resource limits
├── audit paths
├── allowed_repos / allowed_repos_mode (GitHub-specific)
├── command_allowlist (shell-specific)
├── allowed_dirs (file server-specific)
├── auth_token_env / auth_token_file (secret references)
└── other server-specific settings
```

### Secret-Based Authentication Decision

**Decision: Implement full secret-based auth pattern.**

Production behavior:
- Server side resolves expected Bearer token from `auth_token_env` or `auth_token_file`
- Agent side resolves same token through authorized config/secret resolution path
- Agent injects `Authorization: Bearer <resolved-token>` for MCP HTTP requests
- Production startup fails when enabled HTTP MCP server has no resolved token
- Unresolved env var is fatal in production
- Unreadable secret file is fatal in production

Development behavior:
- Missing token may be allowed with warning (local/dev environments)

Raw `auth_token` handling:
- Reject as forbidden key in production
- Migration warning in development (if legacy configs exist)

### tool_names Decision

**Decision: Document `tool_names` only as optional startup drift validation metadata.**

It must NOT be documented as:
- A routing input
- A required new-tool step for routing
- A valid confirmed `agent.toml [mcp_servers.*]` field

Routing remains controlled only by ToolRegistry.

1. **Restrict** `agent.toml [mcp_servers.*]` field list to five confirmed fields.

3. **Replace** all raw `auth_token` references with secret-based patterns.
4. **Document** `tool_definitions` as canonical location for LLM-visible tool schemas.
5. **Document** `tool_names` only as optional startup drift validation metadata.
6. **Document** `security_profile` as agent-wide setting in `agent.toml`.
7. **Replace** stale split-config file names with current `agent.toml` paths.
8. **Replace** stale `routing.md` references with `04_mcp_03_routing_lifecycle_and_execution.md`.

### Consistency Rules

- After changes, every mention of MCP config must specify which layer it refers to.
- No document should imply that MCP servers can read `agent.toml`.
- No document should imply that `agent.toml` owns MCP server-local policy settings.
- Raw `auth_token` must never be recommended in any config file.
