## Goal

Document the shared `McpServerHealthRegistry` wiring between `ToolExecutor` and the watchdog, and give watchdog restart-limit exhaustion a deterministic, operator-visible outcome (a recorded reason `/mcp` status can display) instead of a silent log-only give-up.

## Scope

**In-Scope:**
- `scripts/shared/mcp_health.py`: add a new `record_restart_exhausted(server_key)` method (does not call `record_failure()`, does not touch `_states`)
- `scripts/agent/repl_health.py`: `_watchdog_check_http()` calls the new method when `count >= max_restarts`
- `scripts/agent/commands/cmd_mcp.py`: extend `/mcp` status output to show `UNAVAILABLE` servers with their recorded reason (today only `DEGRADED` servers get a reason line)
- `docs/04_mcp_03_03_transport-and-health.md`: document that `ToolExecutor` and the watchdog share the same `McpServerHealthRegistry` instance (created once in `agent/factory.py::_build_tool_executor()`), and why replacing/rebuilding it would break dispatch-gating consistency
- `docs/04_mcp_06_13_watchdog-health-reasons-scheduling-part1.md`: document the restart-exhaustion outcome and reason string
- `tests/test_watchdog.py`: add a test exercising `count >= max_restarts` (not currently covered)

**Out-of-Scope:**
- `record_degraded()`'s `UNAVAILABLE`/`HALF_OPEN` guard — covered by the sibling plan `plans/20260711-134244_plan.md`; this plan's new `record_restart_exhausted()` method is deliberately separate from `record_degraded()` so it composes cleanly with that guard (see Assumption 3)
- Watchdog probe classification/logging (unreachable vs. malformed vs. non-200) — covered by the sibling plan `plans/20260711-134610_plan.md`
- Any change to `max_restarts`'s default value or `mcp_watchdog_max_restarts` config semantics

## Assumptions

1. `agent/factory.py::_build_tool_executor()` (lines 209-228) creates exactly one `McpServerHealthRegistry()` instance (line 221), calls `tools.set_health_registry(registry)` on the `ToolExecutor`, and returns the same `registry` object, which becomes `AppServices.health_registry` — confirmed by direct read. `_watchdog_check_http()` (in `repl_health.py`) reads/writes `ctx.services_required.health_registry`, the identical object. This is the "shared wiring" the requirement wants documented; no code change is needed for this part, only documentation.
2. Today, when `restart_counts[key] >= max_restarts`, `_watchdog_check_http()` logs a warning and returns **before** reaching the `record_failure(key)` call at the bottom of the function — confirmed by direct read (the early `return` at the top of the "unreachable or restart_recommended" branch). This means restart exhaustion currently leaves **no trace** in `HealthRegistry` beyond whatever state the prior (successful) restart attempts already produced; there is no distinct "the watchdog gave up" signal.
3. Resolving the requirement's two decision Tasks:
   - **"Decide whether restart limit reached should call `record_failure()`"** → **No.** By the time the restart limit is reached, `failure_threshold` (default 3) has almost always already been crossed by the `record_failure()` calls made after each of the preceding restart attempts (assuming `max_restarts >= failure_threshold`, the deploy default), so the server is already `UNAVAILABLE` and dispatch is already gated — calling `record_failure()` again adds no new information and would grow an already-saturated failure counter unboundedly.
   - **"Decide whether restart limit reached should record a reason such as `restart_limit_reached`"** → **Yes**, via a new, dedicated method — **not** via `record_degraded()`, because (per the sibling plan `plans/20260711-134244_plan.md`) `record_degraded()` will refuse to change state or reason while `UNAVAILABLE`/`HALF_OPEN`, and here the server is expected to already be `UNAVAILABLE`; a separate method that only writes to the reason field (not gated on current state) is the correct, composable design. Both decisions are grounded in the actual code paths read above, not guesses.
4. `scripts/agent/commands/cmd_mcp.py`'s `/mcp` status handler (lines 90-103) only surfaces `registry.get_degraded_reason(key)` for servers whose `get_state(key) == DEGRADED` — confirmed by direct read. `UNAVAILABLE` servers (including ones with restart exhausted) get no reason line at all today, only appearing in the generic "N unreachable" count. This is the concrete gap behind the Acceptance Criteria "Status output... make restart exhaustion actionable."
5. `_degraded_reasons` (the dict `get_degraded_reason()` reads) is a plain, state-independent `dict[str, str]` in `McpServerHealthRegistry` — confirmed by direct read of `scripts/shared/mcp_health.py`. Writing to it from a new method that does not touch `_states` is safe and requires no change to `get_degraded_reason()`'s own signature or behavior.
6. This plan depends on the sibling plan `plans/20260711-134244_plan.md` (the `record_degraded()` guard) having landed first, since both touch `scripts/shared/mcp_health.py`; implement `plans/20260711-134244_plan.md` first, or coordinate as a single combined change to that file to avoid two independent, conflicting edits.

No unknowns remain. Both decision Tasks are resolved in Assumption 3 using the actual control-flow evidence read from `repl_health.py` and `mcp_health.py`, not guesses.

## Design

### `McpServerHealthRegistry.record_restart_exhausted()` (mcp_health.py)

```python
def record_restart_exhausted(self, server_key: str) -> None:
    """Record that the watchdog exhausted its restart attempts for server_key.

    Does not change state (the server is expected to already be UNAVAILABLE from
    the record_failure() calls made during the preceding restart attempts) — only
    tags the reason so operators can distinguish "still cycling" from "watchdog
    gave up; needs manual intervention" in /mcp status.
    """
    self._degraded_reasons[server_key] = "restart_limit_reached"
    logger.warning(
        "Health: %r restart limit reached — manual intervention required", server_key
    )
```

### `_watchdog_check_http()` change (repl_health.py)

In the existing `count >= max_restarts` branch, immediately after the existing warning log:

```python
if count >= max_restarts:
    logger.warning(
        "Watchdog: %r unreachable; restart limit reached (%s)", key, max_restarts,
    )
    if ctx.services_required.health_registry:
        ctx.services_required.health_registry.record_restart_exhausted(key)
    return
```

### `/mcp` status change (cmd_mcp.py)

After the existing `degraded_keys` block, add a parallel `unavailable_keys` block:

```python
unavailable_keys = [
    key
    for key in ctx.cfg.mcp.mcp_servers
    if registry is not None
    and registry.get_state(key) == McpServerHealthState.UNAVAILABLE
]
if unavailable_keys:
    self._out.write("")
    self._out.write("  Unavailable servers:")
    for key in unavailable_keys:
        reason = registry.get_degraded_reason(key) if registry else None
        reason_str = f": {reason}" if reason else ""
        self._out.write(f"    [UNAVAILABLE] {key}{reason_str}")
```

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| A server could reach `UNAVAILABLE` from `record_failure()` calls and then, independently, `record_restart_exhausted()` overwrites a *different*, more specific degraded reason that was set before it went `UNAVAILABLE` (e.g., a real `record_degraded()` reason from an earlier watchdog probe) | Low | Accepted: once restart attempts are exhausted, "restart_limit_reached" is the most operationally relevant reason (it tells the operator what to do next — manually restart or investigate), superseding an earlier, now-stale degraded reason; documented explicitly in the new docstring |
| Adding an `UNAVAILABLE` block to `/mcp` status output changes the command's visible output shape, which could break a script or test that greps `/mcp` output expecting the old format | Low | The new block is purely additive (new lines only, appended after the existing `DEGRADED` block) and only appears when at least one server is `UNAVAILABLE`; existing output when no server is unavailable is byte-identical. Verify via `tests/test_cmd_mcp.py`/`tests/test_cmd_registry_note_removal.py` (both exercise `_cmd_mcp_status()`) in the validation pass. |
| `record_restart_exhausted()`'s reason could be immediately cleared by a subsequent `record_success()` if the server recovers via a `HALF_OPEN` trial dispatch shortly after — potentially making the "restart limit reached" signal short-lived and easy to miss operationally | Low | This is correct, intended behavior: once the server actually recovers, the stale "needs manual intervention" reason should disappear (matches `record_success()`'s existing, already-tested reset of `_degraded_reasons`); not a new risk introduced by this change |
