# Plan: Add production hardening and backward-compatibility validation

## Goal

Add a dedicated production validation layer (`ProductionConfigValidator`) that detects unsafe or compatibility-oriented settings and makes them fatal in production where appropriate, while preserving local/development compatibility modes.

## Scope

### In scope

- Create `shared/production_config_validator.py` with validation logic
- Create `tests/test_production_config_validator.py` with comprehensive test coverage
- Integrate validator into startup path (before REPL becomes available)
- Validate the following settings when `security_profile == "production"`:
  - `plugin_strict=false` → fatal
  - `tool_definitions_strict=false` → fatal
  - `routing_drift_strict=false` → fatal
  - `allowed_repos_mode="fail_open"` → fatal
  - Missing `tool_safety_tiers` entries → fatal
  - Unknown `tool_safety_tiers` keys → fatal
  - `use_tool_dag=false` → fatal or startup audit error
- Emit explicit audit messages for:
  - `allowed_tools=[]` allow-all visibility
  - `allowed_repos_mode="fail_open"` in local/development
  - `use_tool_dag=false` in local/development
- Update documentation to reflect local/development compatibility modes and production hardening requirements

### Out of scope

- Removing local/development compatibility modes
- Removing `allowed_repos_mode="fail_open"`
- Removing `use_tool_dag`
- Changing runtime approval semantics
- Routing plugin tools through MCP

## Assumptions

1. The `security_profile` configuration field already exists and can be `"production"` or `"local"`/other values.
2. The existing `RagConfigValidator` validates different concerns and should not be merged with production hardening checks.
3. Startup path integration point is accessible from `agent/config_builders.py` or similar startup configuration code.
4. Existing test infrastructure supports adding new test files under `tests/`.

## Unknowns

1. **Exact startup integration point**: RESOLVED — `build_agent_config()` in `agent/config_builders.py` is the optimal integration point. This function is called after all config construction and before REPL becomes available.

2. **`allowed_tools=[]` severity in production**: RESOLVED — Warning only. The whitelist layer being in allow-all mode is an informational message, not a policy violation. Other approval and safety controls still apply separately.

3. **Error type for `use_tool_dag=false`**: RESOLVED — Warning only. Legacy scheduling mode notification, not a fatal error. Local/development behavior remains unchanged.

#### 2. ValidationError hierarchy

```python
class ValidationError(Exception):
    """Base validation error."""
    def __init__(self, message: str, level: Literal["warning", "error"]):
        self.message = message
        self.level = level

class FatalError(ValidationError):
    """Fatal validation error that should stop startup in production."""
    def __init__(self, message: str):
        super().__init__(message, "error")
```

## Risks

### Risk 1: Breaking existing local/development workflows

**Mitigation:** Preserve local/development compatibility modes exactly as they are. Only add warnings, do not change behavior. Ensure thorough testing of local/development paths.

### Risk 2: Missing edge cases in tool_safety_tiers validation

**Mitigation:** Analyze existing `tool_safety_tiers` usage patterns before implementation. Consult with team members who understand the current design.

### Risk 3: Startup path integration complexity

**Mitigation:** Carefully analyze existing startup flow before making changes. Add comprehensive logging to debug any issues. May need to coordinate with other team members working on startup code.
