## Design

**追加するセクション構成:**

```markdown
## Delivery Operations

### Retry policy

Events are retried by consumers nacking (POST /events/{id}/nack).
Each nack increments `delivery_failure_count`.
When `delivery_failure_count >= max_retry`, the event is promoted to the DLQ inline.

| Config field | Effect |
|---|---|
| `max_retry` | Number of nacks before DLQ promotion (default: no default; must be set explicitly) |

### DLQ promotion policy

**Primary path**: Inline promotion via nack endpoint (immediate; no delay).
**Safety-net sweep**: Background loop (every 60 s) promotes any events that reached the threshold
  but were not caught inline. Under normal operation, the sweep promotes 0 events.
  Non-zero sweep count indicates a bug — check logs for "swept N orphan(s)" messages.

### Consumer responsibility

Consumers MUST:
1. Call POST /events/{id}/nack when processing fails
2. Call POST /events/{id}/ack?consumer_id=X on success (also advances offset)

Consumers MUST NOT:
- Silently discard events without calling ack or nack
- Assume that offset advancement happens on delivery (it now requires explicit ack)

### Investigating retry backlog and DLQ lag

Use GET /health to inspect:

| Field | Meaning | Action if high |
|---|---|---|
| `retry_backlog` | Events with ≥1 failure but not yet at threshold | Investigate consumer errors |
| `dlq_orphan_count` | Events at threshold but not promoted | Report as bug; check inline promotion |
| `total_delivery_failures` | Cumulative failure count across all events | Monitor for trends |

### Requeue policy

POST /dlq/{event_id}/requeue:
- Clears `dlq_at` (returns event to live state)
- Increments `dlq_requeue_count` (does NOT reset `delivery_failure_count`)
- If `delivery_failure_count` is still >= `max_retry`, the next nack will immediately re-promote to DLQ

To allow re-delivery after requeue:
- Either the consumer must succeed (ack) before nacking again
- Or `max_retry` must be increased and the config reloaded

### Timeout handling

There is currently no automatic timeout-based nack. Consumers are responsible for
calling nack when processing times out. Implement consumer-side timeouts with:
- A deadline on the processing coroutine/thread
- A nack call in the timeout handler
```
