## Design

**`06_eventbus_02_http_api_and_runtime.md` の `/subscribe` セクション改定後:**

```markdown
### GET /subscribe

SSE subscription. Delivers events in two phases:

**Phase 1 — Replay (SQLite)**: On connect, all events with `seq > since_seq` are read from
SQLite and streamed immediately. If `consumer_id` is set and `since_seq == 0`, the saved
offset is used as the start sequence.

**Phase 2 — Live (in-memory broker)**: After replay completes, the subscriber registers with
the in-memory broker. New events published after the replay are pushed directly — no polling.

**Query parameters:**
- `topic` (list[str], default all): filter by topic (broker-level fan-out)
- `since_seq` (int, default 0): starting sequence; overridden by saved offset if consumer_id is set
- `consumer_id` (str, optional): consumer identifier for offset persistence

**Offset advance**: Only when `POST /events/{id}/ack?consumer_id=X` is called.
No mid-stream checkpoint. No offset write on disconnect.

**Reconnect**: disconnect → reconnect with same consumer_id → Phase 1 replay from last acked seq.

Note: `poll_interval_ms` is deprecated and no longer affects live delivery (req #40).
```

**`06_eventbus_04` のオフセットセクション追記内容:**
- ハイブリッドモデルでのオフセット更新パスは ack のみ (req #29 と同じ)
- 再接続は SQLite リプレイが補完するため broker queue の切断による損失はなし
