# Plan: Make discovery failure fatal under PRODUCTION security profile

## Goal

Prevent the agent from starting silently crippled when MCP tool discovery fails, by making the failure fatal under `PRODUCTION` security profile.

## Scope

- `scripts/agent/startup.py` — change discovery exception handler to use `add_fatal` under PRODUCTION
- `tests/test_startup.py` — add test for discovery failure under both profiles

## Assumptions

- `SecurityProfile` enum has `PRODUCTION` and `LOCAL` values
- `pipeline.add_fatal()` and `pipeline.add_warning()` methods exist and work correctly
- The existing `production_mode` variable computed at line 192 can be reused

## Unknowns

Resolved — all unknowns answered by code inspection:

1. `production_mode` is already computed at line 192: `production_mode = ctx.cfg.mcp.security_profile == SecurityProfile.PRODUCTION`. Can be reused directly.
2. `add_skipped` is used in exactly two places in `_check_services()`: MCP tool discovery (line 246) and RAG consistency (line 281). Only the MCP discovery path needs changing.
3. No post-discovery validation of `ctx.services_required.runtime_tools` exists — Option B would require adding new code in `repl_health.py`, which doesn't currently access `runtime_tools`.
4. Option C (upgrade log level) is insufficient because `add_skipped` doesn't contribute to `pipeline.has_fatal`, so the agent still starts despite the error.

## Affected areas

- Startup pipeline — discovery failure becomes fatal under PRODUCTION
- Agent lifecycle — prevents silent catastrophic failure mode
- Test coverage — new regression tests for discovery failure scenarios

## Design

### Recommended approach: Option A (profile-aware exception handling)

Change the discovery exception handler to branch based on `production_mode`:

```python
except Exception as exc:  # noqa: BLE001
    if production_mode:
        pipeline.add_fatal(
            "mcp_tool_discovery", f"MCP tool discovery failed: {exc}"
        )
    else:
        pipeline.add_warning(
            "mcp_tool_discovery", f"MCP tool discovery skipped: {exc}"
        )
```

Rationale:
- Mirrors the existing `SecurityProfile.PRODUCTION` fatal-on-subprocess-failure pattern in `_start_servers()`
- Reuses the already-computed `production_mode` variable
- Minimal code change — one conditional added to the existing handler
- Clear semantic distinction between PRODUCTION (fail fast) and LOCAL (warn and continue)

### Why not Option B

Option B requires adding new validation logic in `repl_health.py` that accesses `ctx.services_required.runtime_tools`. However, `repl_health.py` currently doesn't access `runtime_tools` at all — it only probes server connectivity and compares tool definitions. Adding this dependency would introduce a new coupling between startup validation and runtime state.

### Why not Option C

Option C only upgrades the log level and wording but keeps `add_skipped`. Since `add_skipped` doesn't contribute to `pipeline.has_fatal`, the agent still starts despite the error — the core problem remains unsolved.

## Implementation steps

### Step 1: Update discovery exception handler in _check_services()

In `scripts/agent/startup.py`, modify the exception handler at lines 245-248 to check `production_mode` and use `add_fatal` for PRODUCTION, `add_warning` for LOCAL.

### Step 2: Add test for discovery failure under PRODUCTION

Add a test case in `tests/test_startup.py`:
- Mock `McpToolDiscoveryService.discover_all()` to raise an exception
- Set `security_profile = SecurityProfile.PRODUCTION`
- Assert `pipeline.has_fatal` is True
- Assert startup raises RuntimeError

### Step 3: Add test for discovery failure under LOCAL

Add a test case in `tests/test_startup.py`:
- Same mock setup
- Set `security_profile = SecurityProfile.LOCAL`
- Assert `pipeline.has_fatal` is False
- Assert startup proceeds normally
- Assert warning message contains discovery failure details

### Step 4: Run validation

- `uv run pytest` — ensure no regressions
- `uv run mypy scripts/` — type check passes
- `uv run ruff check scripts/` — lint clean

## Risks

### Risk 1: Production deployments may fail during rolling updates

If some MCP servers are temporarily unreachable during deployment, discovery will fail and the entire agent will refuse to start. Mitigation: ensure all MCP servers are healthy before deploying; consider adding a grace period or retry mechanism.

### Risk 2: LOCAL profile behavior changes slightly

Under LOCAL, the outcome changes from `SKIPPED` to `WARNING`. While semantically more accurate, this may affect tests or monitoring that rely on the exact outcome classification. Mitigation: update affected tests and alerting rules.

### Risk 3: Partial discovery failures masked

If discovery partially succeeds (some servers reachable, others not), the current code reports per-server issues. Making the overall discovery fatal means even partial failures prevent startup. Mitigation: verify that partial success scenarios are rare and that the per-server reporting is sufficient for operators to diagnose.
