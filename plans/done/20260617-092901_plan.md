# Plan: Isolate plugin pipeline hook failures from RAG pipeline (Req 24)

## Goal

Wrap plugin post-rerank pipeline stage execution with error isolation so that a broken plugin hook logs diagnostic context and does not break the RAG pipeline.

## Scope

**In scope:**
- `scripts/shared/plugin_registry.py` — add `run_pipeline_stages()` helper with error isolation
- `scripts/rag/pipeline.py` — integrate post-rerank hooks into `RagPipeline.run()` after `RerankStage`
- `tests/test_plugin_registry.py` — tests for hook error isolation and strict mode
- `docs/05_agent_11_extension-points.md` — update failure behavior documentation

**Out of scope:**
- Adding pre-search or pre-rerank hook support (documented limitation remains)
- Changing the plugin loading mechanism (`load_plugins`)
- Modifying `RagPipeline.augment()` or other pipeline entry points

## Assumptions

1. Post-rerank hooks are synchronous functions `(hits, query) -> hits` (not async), matching the current decorator signature in docs (`register_pipeline_stage`).
2. Hooks are already registered before `RagPipeline.run()` is called (loaded at startup via `load_plugins()`).
3. "Strict mode" for hook execution is a new parameter on `RagPipeline.run()`, separate from `plugin_strict` config which controls plugin loading.
4. Failed hooks should be skipped silently to the caller — the pipeline continues with whatever hits were produced by prior stages.

## Implementation Steps

### Step 1: Add `run_pipeline_stages()` to `plugin_registry.py`

Create a new public function that iterates over registered post-rerank hooks, calls each one, and handles errors:

```python
def run_pipeline_stages(
    hits: list[Any],
    query: str,
    *,
    strict: bool = False,
) -> list[Any]:
    """Execute all registered post-rerank hooks with error isolation.

    Returns the (possibly modified) hits list. Failed hooks are logged
    and skipped; in strict mode the first failure raises.
    """
```

- Use `inspect.iscoroutinefunction()` / `asyncio.iscoroutinefunction()` to detect async hooks
- Support both sync and async hook signatures
- Log: `Plugin hook "%s" failed: %s` with hook `__name__`, error type, and query context
- In strict mode, re-raise the original exception

### Step 2: Integrate hooks into `RagPipeline.run()`

After the `RerankStage` execution (before `AugmentStage`):

```python
# After RerankStage runs, before AugmentStage:
if plugin_stages := get_pipeline_post_stages():
    hits = ctx.reranked
    for stage_fn in plugin_stages:
        # Call the hook; modify hits in place or replace
        result = _call_hook(stage_fn, hits, query)
        if result is not None:
            hits = result
    ctx.reranked = hits
```

Or more cleanly, delegate to `run_pipeline_stages()`:

```python
# After RerankStage completes:
if plugin_stages := get_pipeline_post_stages():
    ctx.reranked = run_pipeline_stages(
        ctx.reranked, query, strict=hook_strict,
    )
```

Add `hook_strict: bool = False` parameter to `RagPipeline.run()`.

### Step 3: Add tests for hook error isolation

In `tests/test_plugin_registry.py`:

- `test_hook_isolation_skips_failed_sync`: sync hook raises → logged, pipeline continues with unmodified hits
- `test_hook_isolation_skips_failed_async`: async hook raises → logged, pipeline continues
- `test_hook_strict_mode_re_raises`: strict=True → exception propagates
- `test_hook_modifies_hits`: successful hook can modify hits list
- `test_multiple_hooks_first_fails`: first hook fails, second runs on original hits
- `test_no_hooks_returns_original`: empty hook list returns hits unchanged
