# Plan: Define and enforce plugin command shadowing policy (Option A)

## Goal

Enforce Option A â€” reject plugin commands that shadow built-in command names by removing
them from the registry at load time, and update all related code, tests, and documentation
to match the new canonical policy.

---

## Scope

### In-Scope

- Modify `_validate_command_conflicts()` to remove shadowed plugin commands from `_commands`
- Add `command_shadows_rejected` counter to `PluginLoadResult` (replacing ambiguous `command_shadows`)
- Update `load_plugins()` strict-mode to raise `PluginLoadError` when command shadows are rejected
- Update `/plugin status` output to reflect rejection semantics
- Update `docs/05_agent_11_extension-points.md` to document the chosen policy
- Fix and extend existing tests to cover rejection behavior

### Out-of-Scope

- Plugin tool shadowing policy (already implemented via `_validate_tool_conflicts`)
- MCP tool conflict behavior
- Built-in command list changes
- Hot-reload or per-plugin enable/disable features

---

## Assumptions

1. Option A (reject) is the selected policy, as recommended in the requirement and consistent
   with the existing tool conflict policy default (`override_policy="reject"`).
2. `command_shadows` field in `PluginLoadResult` currently counts retained shadow commands;
   renaming it to `command_shadows_rejected` is safe because no external caller other than
   `cmd_plugins.py` and `test_cmd_plugins.py` reads this field by name.
3. The strict-mode path in `load_plugins()` should include command shadow rejections alongside
   tool conflict rejections in the `PluginLoadError` message, consistent with the existing
   tool conflict strict-mode pattern.
4. Removing a plugin command from `_commands` during `_validate_command_conflicts()` is safe
   because the function is called after all plugin files are loaded (post-load validation
   phase), matching the same pattern used for tools in `_validate_tool_conflicts()`.
5. The test file `test_plugin_registry.py::TestCommandShadowLogging` tests logging-only
   behavior; it must be updated to verify rejection instead.
6. `deploy/deploy.sh` does not reference any of the affected files directly (confirmed by
   grep â€” no output).

## Implementation Steps

### Step 1: Update `_validate_command_conflicts()` to remove shadowed commands

**File:** `scripts/shared/plugin_registry.py`

- In `_validate_command_conflicts()`, add `del _commands[name]` after counting the shadow, mirroring `_validate_tool_conflicts()`.
- Return value remains `int` (rejection count).
- Add a `strict_rejected: list[str]` accumulator (same pattern as tool conflicts) so strict-mode can raise.
- Update the function docstring to say "reject" not "warn".

**Completion criteria:**
- `_commands` no longer contains the shadowed command name after `_validate_command_conflicts()` returns.
- Function still returns the count.

---

### Step 2: Rename `PluginLoadResult.command_shadows` to `command_shadows_rejected`

**File:** `scripts/shared/plugin_registry.py`

- Change the dataclass field name from `command_shadows` to `command_shadows_rejected` (default 0).
- Update the assignment in `load_plugins()`: `command_shadows=cmd_shadows` â†’ `command_shadows_rejected=cmd_shadows`.
- Update the module docstring if it references `command_shadows`.

**Completion criteria:**
- `PluginLoadResult(loaded_count=1, failed=(), command_shadows_rejected=1)` constructs without error.
- Old field name `command_shadows` raises `AttributeError`.

---

### Step 3: Add command shadow rejection to strict-mode error aggregation

**File:** `scripts/shared/plugin_registry.py`

- In `_validate_command_conflicts()`, populate `strict_rejected: list[str]` with rejected names when `strict_mode=True`.
- Return signature changes to `tuple[int, list[str]]` (count, strict_rejected_names) â€” same shape as `_validate_tool_conflicts` return signature.
- In `load_plugins()`, capture the new return value and add a third part to the `PluginLoadError` message if `strict_rejected` is non-empty:
  `"Command builtin conflicts rejected: /help, /debug"`.

**Completion criteria:**
- `load_plugins(tmp_path, strict_mode=True)` with a plugin registering `/help` raises `PluginLoadError` containing `"Command builtin conflicts rejected"`.
- Non-strict mode with shadow does not raise.
