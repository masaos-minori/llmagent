# Plan: Implement real mdq-mcp search and indexing

## Goal

Replace all placeholder stubs in `mcp/mdq/service.py` with real SQLite FTS5-backed implementations for search, chunk retrieval, indexing, stats, and grep.

## Scope

**In scope:**
- `scripts/mcp/mdq/service.py` — MdqService._init_db(), get_chunk(), refresh_index(), stats(), grep_docs()
- `scripts/mcp/mdq/indexer.py` — _index_single_file(), _index_directory() with real SQLite write logic
- `scripts/mcp/mdq/search.py` — search_docs() with proper FTS5 query and result formatting
- `scripts/mcp/mdq/parser.py` — parse_markdown() to extract heading-based sections/chunks
- `tests/test_mdq_service.py` — update tests to verify real behavior

**Out of scope:**
- MCP server setup (server.py, tools.py) — no changes needed
- Database migration scripts
- Multi-user concurrency handling
- External search engines (Elasticsearch, etc.)

## Assumptions

1. SQLite FTS5 is available (standard in most Linux distributions with sqlite3).
2. The DB path `/opt/llm/db/mdq.db` can be created and written by the service process.
3. Markdown files use standard heading-based section structure (`#`, `##`, `###`, etc.).
4. One chunk = one section (heading + its content until next heading at same or higher level).
5. Existing tests for models, indexer path traversal, and parser file reading should continue to pass.

## Implementation Steps

1. **Add DB schema to service.py** — `_init_db()` creates `sections` table (id, path, heading, content, file_mtime) and FTS5 virtual table `sections_fts`.
2. **Implement parser.py** — Extract heading-based sections from Markdown: each section = heading + content until next heading at same/higher level. Return list of dicts with path, heading, content.
3. **Implement indexer.py** — `_index_single_file()` calls parser to extract sections, then INSERTs into SQLite with file_mtime for change detection.
4. **Implement refresh_index()** — Compare stored file_mtime with current, delete stale sections, re-index changed files.
5. **Implement search.py** — Ensure FTS5 table exists, run MATCH query, return formatted results with path/heading context.
6. **Implement get_chunk()** — Query `sections` table by chunk_id, return content + heading.
7. **Implement stats()** — SELECT COUNT from sections table, return document_count + chunk_count.
8. **Implement grep_docs()** — SQLite REGEXP or LIKE-based pattern search on content column.
9. **Update tests** — Replace placeholder assertions with real DB-backed verification.
