# Plan: Clarify MDQ authorization behavior and exclude empty `auth_token` from compatibility cleanup

Source requirement: `requires/20260716_07_require.md`

## Goal

MDQ's HTTP-layer no-auth-token design and `allowed_dirs`-based path
authorization are documented clearly enough that no contributor working
through the other 6 requirements in this 2026-07-16 MDQ compatibility-cleanup
batch mistakes the empty `auth_token` for dead/removable compatibility code —
because unlike `audit_log_path` or `enable_refresh`, this is a deliberate,
functioning security boundary, not an unused config key.

## Scope

**In scope (documentation only — no code change)**

- Add a clarification section to
  `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md` (the existing doc that
  already describes `authorize_path()`/`allowed_dirs` enforcement in detail)
  stating explicitly:
  - `mdq-mcp` runs with `attach_auth_middleware(app, "")` — an intentionally
    empty Bearer token, meaning the HTTP layer performs no token check
    (verified: `scripts/mcp_servers/server.py:75-95`, `token` falsy branch
    skips the check entirely).
  - This is not an oversight: `server.py:291`'s existing comment states MDQ
    relies on `allowed_dirs`-based path authorization instead of HTTP
    token auth.
  - `allowed_dirs = []` (the config default) is fail-closed / deny-all —
    already stated in both `config/mdq_mcp_server.toml:5-8`'s comment and
    `scripts/mcp_servers/mdq/auth.py:16-18`'s docstring ("An empty allowlist
    means fail-closed (deny all)").
  - An explicit statement that empty `auth_token` is **current, intentional
    MDQ behavior** and must not be grouped with the config keys removed
    elsewhere in this batch (`audit_log_path`, `concurrency_limit`,
    `enable_refresh`, embedding/hybrid keys, summary-cache keys) — those were
    removed because they were read-but-unenforced; `auth_token=""` is the
    opposite case: it is read and its effect (skip HTTP auth) is fully
    enforced and intentional.
  - A closing note: any future change to MDQ's HTTP authentication model
    (e.g., adding a real Bearer token) requires a separate security design
    issue, not a compatibility-cleanup change.
- No new `docs/security*.md` file is created — see Design for why the
  requirement's listed target path is redirected to the existing
  enforcement doc instead.

**Out of scope**

- Any change to `scripts/mcp_servers/mdq/server.py`'s
  `attach_auth_middleware(cast(_FastAPIApp, app), "")` call, `auth.py`'s
  `authorize_path()` logic, or `config/mdq_mcp_server.toml`'s `allowed_dirs`
  default — explicit non-goals in the source requirement.
- Any change to the 6 other requirements' config-removal work — this plan's
  entire purpose is to draw a boundary around what those other plans must
  *not* touch.

## Assumptions

1. `docs/security*.md` does not exist anywhere in the repository today —
   verified via `find docs -iname "security*"` (no output). Creating a new,
   generically-named security doc file would be a speculative addition with
   no established convention to follow (all existing docs use the numbered
   `NN_topic_...md` scheme). The most relevant existing home for this
   content is `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md`, which
   already documents `authorize_path()`/`allowed_dirs` enforcement in depth
   (verified by direct read — it covers the exact two enforcement
   checkpoints: `outline()` and `_validate_paths()`).
2. `attach_auth_middleware(app, "")`'s behavior with an empty token is fully
   defined and intentional, not accidental: `scripts/mcp_servers/server.py:78-81`'s
   own docstring states "When token is empty, auth is skipped and the
   middleware only injects the X-Request-Id response header" — this is a
   documented, shared mechanism used by other MCP servers too (not MDQ-only
   special-casing), so it is core platform behavior, not leftover
   compatibility code.
3. `allowed_dirs = []` deny-all behavior is already implemented correctly and
   documented in two places (`auth.py` docstring, `mdq_mcp_server.toml`
   comment) — verified by direct read. This plan adds a cross-reference, not
   new documentation of the mechanism itself.
4. None of the other 6 requirement plans in this batch (01, 02, 03, 04, 05,
   06 — all already written) propose any change to `auth_token`,
   `allowed_dirs`, or `attach_auth_middleware` — verified by reviewing each
   plan's Scope section; no conflict exists to guard against beyond the
   documentation clarification itself.

## Design

The source requirement lists `docs/security*.md` as a target file, but no
such file exists and the repository has no generic security-doc convention
(docs are topic-numbered, e.g. `04_mcp_05_05_...`). Rather than create a new
file speculatively (against `AGENTS.md`'s "do not generate documentation
speculatively" policy), this plan places the clarification inside the
existing, directly-relevant `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md`,
which already has an "認可チェック" (authorization check) section describing
`authorize_path()` and `allowed_dirs` enforcement. This is evidence-based
redirection, not scope reduction — every task item in the source requirement
is still satisfied, just in a file that exists and fits the codebase's
existing documentation structure.

## Implementation steps

1. **Add the clarification section** to
   `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md`, covering: empty
   `auth_token` is intentional (with `server.py` line references), the
   security model relies on `allowed_dirs`, `allowed_dirs=[]` is deny-all,
   and this behavior is explicitly excluded from compatibility-cleanup scope

## Risks

- **Risk**: A future contributor implementing one of the other 6
  requirements' plans could still accidentally touch `auth_token` while
  editing `config/mdq_mcp_server.toml` for an unrelated key removal in the
  same file.
  **Mitigation**: this plan's documentation explicitly calls out the
  distinction (enforced-and-intentional vs read-but-unenforced) right next
  to the enforcement doc that a reviewer would naturally consult; each of
  the other 5 plans' Scope/Out-of-scope sections also lists only their own
  specific keys, not `auth_token`/`allowed_dirs`.
- **Risk**: Placing this content in
  `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md` instead of a dedicated
  `docs/security*.md` file (as the requirement literally lists) could be
  seen as not fully following the requirement's target file list.
  **Mitigation**: documented explicitly in Design/Assumption 1 with the
  evidence-based rationale (no such file exists, no established convention
  to create one); a reviewer can request a dedicated file if one is
  actually wanted project-wide.
