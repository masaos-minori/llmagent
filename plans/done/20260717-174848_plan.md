# Plan: Reject disabled tools before dispatch in /v1/call_tool and clarify validation policy

Source requirement: `requires/20260717_16_require.md` (moved to `requires/done/` on
completion of this workflow)

## Goal

Make `/v1/call_tool` on the file-read, file-write, file-delete, and git MCP servers
reject disabled tools *before* reaching the service dispatch table, returning
`CallToolResponse(result="Tool disabled: <disabled_reason>", is_error=True)` instead of
executing the tool. Also make argument validation deterministic: every `/v1/call_tool`
handler in these four servers must call `req.validate_args()` before dispatch, with a
consistent error response when it raises. This closes the gap where a direct HTTP
caller could invoke a tool that `/v1/tools` metadata will (per a sibling, not-yet-built
requirement) report as disabled.

## Design

- **Disabled-tool gate**: each of the four servers' `call_tool()` handler computes
  `(enabled, reason)` from its own `_cfg` *before* touching the dispatch table:
  ```python
  @app.post("/v1/call_tool", response_model=CallToolResponse)
  async def call_tool(req: CallToolRequest) -> CallToolResponse:
      enabled, reason = availability_flags(_cfg.allowed_dirs)
      if not enabled:
          return CallToolResponse(result=f"Tool disabled: {reason}", is_error=True)
      try:
          req.validate_args()
      except ValueError as e:
          return CallToolResponse(result=f"Validation error: {e}", is_error=True)
      r = await _dispatch_read_tool(req.name, req.args)
      return _to_call_tool_response(r)
  ```
  (File servers: same tool set is uniformly enabled/disabled together, so no per-tool-name
  branch is needed — this matches the requirement's own per-server rule, not a per-tool
  rule, for file servers.)
  Git server additionally branches per tool name (read vs. write) using
  `_git_tool_availability(_cfg, req.name)`:
  ```python
  def _git_tool_availability(cfg: GitConfig, tool_name: str) -> tuple[bool, str]:
      if not cfg.allowed_repo_paths:
          return False, "allowed_repo_paths is empty"
      if cfg.read_only and tool_name in GIT_WRITE_TOOLS:
          return False, "read_only=true"
      return True, ""
  ```
  This is the identical precedence rule (`allowed_repo_paths` empty beats `read_only`)
  that requirement 15's plan already specifies for `/v1/tools` — defining it once here
  (or reusing it if requirement 15 lands first) is the point of the "SHARED WITH" note.
- **Validation policy**: adopt the requirement's "Recommended policy" — call
  `req.validate_args()` unconditionally (for every tool, not just ones with a
  registered validator; `validate_tool_args()` is already a no-op when no validator is
  registered, per `tool_validators.py:38-42`) right after the disabled-tool gate and
  before dispatch. Wrap in `try/except ValueError` and convert to the same
  `"Validation error: {e}"` message format `dispatch_tool()` already uses internally,
  so callers see one consistent error shape whether validation fails before or during
  dispatch.
- **Ordering of checks**: disabled-tool check first, then argument validation, then
  dispatch. This matches the requirement's own Scope ordering ("Before calling
  `dispatch_tool()`, check whether the requested tool is enabled... Do not call the
  service dispatch table for disabled tools") and avoids running argument validation
  logic for a tool that's disabled anyway.
- **No new Pydantic model/response field** — `CallToolResponse`'s existing two fields
  (`result`, `is_error`) are reused for both the disabled case and the validation-error
  case; this keeps the response contract identical for every caller.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Duplicate/diverging `availability_flags()` / git-precedence logic if this plan and the sibling requirement-15 plan are both implemented without checking for each other's work first | Medium (explicitly the reason for the "SHARED WITH" note) | Implementation step 1 makes checking for the sibling plan's helper an explicit first action, before writing any new code; whichever lands first is the single source of truth, and a repo-wide `grep -n "def availability_flags\|def _git_tool_availability"` after both plans are implemented should show exactly one definition of each |
| Git precedence rule (`allowed_repo_paths` empty beats `read_only=true`) implemented backwards, silently letting write tools through when `allowed_repo_paths` is empty | Medium (easy to swap `if`/`elif` order) | New tests in step 5 explicitly assert all git tools (including reads) are disabled when `allowed_repo_paths=[]` regardless of `read_only`, and that only the 5 named write tools are disabled when `read_only=True` with non-empty `allowed_repo_paths` — pins the precedence order as a regression gate |
| `validate_args()` now runs for every call, including tools with no registered validator — if `validate_tool_args()`'s no-op path has any hidden side effect or is accidentally removed later, every tool call would start raising | Low | `validate_tool_args()` is already unit-tested as a no-op for unregistered tools (`tests/test_mcp_tool_validators.py::TestValidateToolArgs::test_no_registered_validator_is_noop`); the new tests in step 5 add an end-to-end `TestClient` case confirming a tool with no validator still dispatches successfully after the change |
| Disabled-tool response text (`"Tool disabled: <reason>"`) doesn't exactly match what a future `RuntimeToolRegistry` client (agent-side, not yet built) expects to parse | Low | The requirement's Scope specifies this exact string format verbatim; tests assert exact string equality, not substring matching, so any future consumer contract mismatch is caught at the point that consumer is built, not silently |
| Adding `validate_args()` to the hot path changes behavior for tools that previously succeeded with technically-invalid args (e.g. a blank `git_commit` message that server-side git itself might have rejected differently) | Low-medium | `git_commit` and `git_push` validators (the only ones affecting git write tools) are already exercised and passing in `tests/test_mcp_tool_validators.py`; the new pre-dispatch call doesn't change what's validated, only *when* — an existing caller sending genuinely valid args sees no behavior change, only ones sending already-invalid args (previously either silently accepted or failing deeper in the git call) now fail fast and uniformly |
