# Plan: Implement MCP tool discovery and build RuntimeToolRegistry at startup

Source requirement: `requires/20260717_03_require.md`

## Goal

`RuntimeToolRegistry` (from requirement 02) is built once at startup from real MCP `/v1/tools` responses — replacing today's validation-only, throwaway `/v1/tools` fetch — and exists in `AppServices` before the REPL loop starts; `/reload` continues to not rediscover tools (already true today, must remain true).

## Scope

**In scope**
- New `scripts/agent/services/mcp_tool_discovery.py`: an async service that iterates `ctx.cfg.mcp.mcp_servers` (HTTP-transport, non-empty URL), fetches each server's `/v1/tools`, validates response shape, normalizes entries into `RuntimeTool` instances (per requirement 02's shape), and detects duplicate tool names across servers.
- `scripts/agent/startup.py`: insert a discovery step after `_start_servers()` and before/within `_check_services()`, feeding a new startup-check block that reports malformed schemas via the existing `StartupValidationResult` severity model.
- `scripts/agent/factory.py`: since `build_agent_context()` is synchronous but discovery requires `await`, restructure so discovery runs from `startup.py` (async context) after `build_agent_context()` returns, storing the resulting `RuntimeToolRegistry` into the already-constructed `AppServices` instance (or adding it as a post-construction attribute).
- `scripts/agent/context.py`: wire the `RuntimeToolRegistry` field added in requirement 02 to actually receive the discovery result.
- `scripts/agent/repl_health.py`: **do not duplicate** the existing `_collect_server_tool_names_per_server()`/`check_routing_drift_vs_live()` fetch — either have the new discovery module supersede it (fetch once, reuse for both registry-building and drift-validation) or keep them as two independent fetches if unifying them is riskier than this requirement's stated scope (see Unknowns).
- `scripts/shared/mcp_config.py`: read-only — confirm `McpServerConfig`'s fields are sufficient for discovery iteration (already used identically by `repl_health.py`).
- New `tests/agent/services/test_mcp_tool_discovery.py` (new directory `tests/agent/services/` must be created — confirmed not to exist) and extensions to `tests/test_startup.py` (the actual existing file — the requirement's stated path `tests/agent/test_startup.py` does not match reality, see Assumption 6).

**Out of scope**
- Wiring `RuntimeToolRegistry` into actual routing/execution/schema/scheduler/policy call sites — requirements 04-07.
- Changing `/reload`'s behavior — it already correctly does not touch MCP discovery (confirmed); this requirement only needs this property preserved, not implemented.
- Automatic duplicate-tool priority selection — explicitly disallowed by the requirement even in local/warning mode.

## Assumptions

1. **The existing `/v1/tools` fetch is validation-only and discards tool data**: confirmed via direct read of `scripts/agent/repl_health.py:243-287` (`_collect_server_tool_names_per_server`) and `scripts/shared/tool_routing_validation.py:38-60` (`validate_routing_against_live`) — only tool *names* are retained (`{server_key: [tool_name,...]}`), full schemas/descriptions are fetched over HTTP but never kept. This requirement's new discovery step must capture that data instead of discarding it.
2. **`build_agent_context()` (`scripts/agent/factory.py:477`) is synchronous** (`def build_agent_context(...) -> None`), but MCP tool discovery requires async HTTP calls (`await http.get(...)`, mirroring `repl_health.py`'s pattern). Discovery cannot be embedded inside `build_agent_context()` without making it async (a larger, riskier change touching every caller) — it must instead run as a separate async step from `startup.py`, after `build_agent_context()` returns but before `_check_services()` completes, consistent with the required startup-flow ordering (steps 1-4 synchronous, discovery async, then continuing checks).
3. **`AppServices` is constructed once, synchronously, inside `build_agent_context()`** (`factory.py:491-501`) — since discovery must run asynchronously afterward, the `RuntimeToolRegistry` field on `AppServices`/`AgentContext` (added by requirement 02) will necessarily be `None`/empty immediately after `build_agent_context()` returns and only populated once discovery completes in `startup.py`. This means the field must be mutable-after-construction (not required at `__init__` time), unlike `ToolExecutor` which *is* available at construction.
4. **`_check_services()` (`startup.py:187`) already threads a `StartupValidationResult` pipeline through 7 named checks** (`security_audit`, `_check_embedding_dimensions`, `check_readiness`, `check_tool_definitions_startup`, `check_routing_drift`, `check_routing_safety_tiers`, `check_routing_drift_vs_live`, plus RAG consistency) — a new discovery-and-build step is naturally an 8th check in this same pipeline, using `pipeline.add_fatal(...)`/`add_warning(...)` exactly like the existing `security_audit` (lines 194-204) and `tool_definitions` (226-235) checks.
5. **Duplicate-tool severity is governed by `ctx.cfg.mcp.security_profile`** (`SecurityProfile.PRODUCTION`/`LOCAL`, `scripts/shared/mcp_config.py:44-49`, consumed as `ctx.cfg.mcp.security_profile == SecurityProfile.PRODUCTION` at `startup.py:137,190`) — this is the existing production/local switch to reuse for the "production: fatal, local: warning" duplicate-tool rule, not `plugin_strict` (a distinct, unrelated flag being removed by requirement 01 anyway).
6. **The requirement's stated test-file paths don't match the current repo layout**: `tests/agent/services/` does not exist at all (confirmed via `ls` error) — must be created as a new directory alongside a new `tests/agent/services/__init__.py` if the test-collection convention requires one (check sibling `tests/` subdirectories for the pattern). `tests/agent/test_startup.py` (the requirement's stated path) does not exist; the real, existing startup test file is `tests/test_startup.py` (428 lines, 23 test functions) at repo-root `tests/`, alongside `test_startup_validation_pipeline.py`, `test_startup_routing_drift.py`, `test_startup_consistency.py`. This plan extends the real existing file, not a nonexistent nested path.
7. `/reload`'s `ConfigReloadService` (`scripts/agent/services/config_reload.py:96`, entry `apply_config_dict()`→`_sync_services()`) has zero references to `/v1/tools`, `route_resolver`, `tool_registry`, or "discover" — confirmed via grep. This requirement's "`/reload` must not rediscover tools" constraint describes existing correct behavior to explicitly preserve (add a regression test asserting this), not a bug to fix.
8. `scripts/agent/services/` already holds a similarly-scoped module, `mcp_status.py` — the new `mcp_tool_discovery.py` should follow that file's structural conventions (async service class or function set) for consistency, confirmed via directory listing.

## Design

### Startup sequence integration

Per Assumption 2, discovery must run as an async step distinct from the synchronous `build_agent_context()`. The concrete flow in `startup.py`'s `run()`:
1. `_initialize()` (builds context, synchronous) — unchanged.
2. `_start_servers()` (spawns/waits for MCP subprocess servers) — unchanged.
3. **New**: `await self._discover_mcp_tools()` — calls into `mcp_tool_discovery.py`'s async fetch-and-build function, populating `ctx.services.runtime_tools` (or wherever requirement 02 placed the field) with a built `RuntimeToolRegistry`. Malformed-schema findings and duplicate-name findings are collected into the `StartupValidationResult` pipeline that `_check_services()` already threads through — this new step should either run inside `_check_services()` as its 8th check, or immediately before it feeding the same pipeline object, so severity (fatal in production, warning in local) is reported consistently with the other 7 checks.
4. `_check_services()`'s remaining 7 checks proceed as today.

### Discovery validation rules (from the requirement, mapped to concrete checks)

Each server's `/v1/tools` response is validated for: (a) is a JSON object, (b) `tools` is a list, (c) each entry has a non-empty `name`, (d) each entry has a `description`, (e) each entry has an object-shaped `inputSchema` (or equivalent field name), (f) optional `status`/`is_write`/`requires_serial`/`resource_scope` fields, if present, are type-checked (not required, but if present and wrong-typed, flagged). This closely parallels `repl_health.py:158-186`'s existing `_validate_tools_response()` shape-check — the new module should follow the same validation style but capture the full entry (not just extract the name) to build `RuntimeTool` instances.

### Duplicate-tool-name rule

Per Assumption 5, gate severity on `ctx.cfg.mcp.security_profile == SecurityProfile.PRODUCTION`: if the same tool name is discovered from two different servers, `pipeline.add_fatal(...)` in production, `pipeline.add_warning(...)` in local — and explicitly do NOT auto-select a "winning" server (per the requirement's own constraint); in local/warning mode, the registry should either exclude the duplicate tool entirely (safer) or keep the first-seen entry with a loud warning — this exact behavior (exclude vs. first-wins-with-warning) should be decided during implementation and documented in the discovery module's docstring, since the requirement says "avoid automatic priority selection" without specifying the exact non-selection behavior.

### `/reload` regression guard

Add an explicit test in `tests/test_startup.py` (or a new `test_config_reload.py` case) asserting that invoking `ConfigReloadService.apply_config_dict()` does not call the discovery module's fetch function (e.g. via a mock/spy asserting zero calls) — this converts today's *implicit* correct behavior (confirmed by Assumption 7's grep) into an explicit, enforced regression guard per the requirement's acceptance criteria.

## Risks

- **Risk**: Inserting a new async discovery step into `startup.py`'s sequence could change overall startup latency (one more full round of `/v1/tools` fetches across all MCP servers) — acceptable per the requirement's own framing (discovery replaces validation-only usage of the same data), but worth measuring.
  **Mitigation**: if unification with `repl_health.py`'s existing fetch (per the Unknowns discussion) becomes desirable later, that removes the redundant round-trip; for this pass, accept the extra latency as the cost of a correctly-populated registry.
- **Risk**: `AppServices`'s exact mutability model is not yet confirmed (Unknown #2) — if it turns out to be a frozen dataclass, populating `RuntimeToolRegistry` post-construction requires either `object.__setattr__` (fragile) or restructuring `AppServices` itself (bigger diff than expected).
  **Mitigation**: Implementation Step 1 resolves this before any other code is written; if `AppServices` must be restructured, treat that as a small, isolated sub-step verified against requirement 02's own implementation, not a surprise discovered mid-implementation.
- **Risk**: The exact non-selection behavior for local-mode duplicates (exclude vs. first-wins-with-warning, per Design) is underspecified by the requirement — an inconsistent choice here could cause confusing runtime behavior (a tool silently missing vs. silently routed to an arbitrary server).
  **Mitigation**: document whichever choice is made explicitly in the discovery module's docstring and in `docs/` (per requirement 19, "Document MCP runtime availability metadata," later in this batch) rather than leaving it implicit.
- **Risk**: Not unifying the two `/v1/tools` fetch paths (per the Unknowns recommendation) means a future contributor might assume `repl_health.py`'s drift-check data and the new registry's data are the same fetch, when they are actually two independent HTTP calls that could theoretically disagree (e.g. a tool appears/disappears between the two calls in a race).
  **Mitigation**: document this explicitly as a known limitation in both modules' docstrings; treat true unification as a follow-up requirement once both are stable in production, not a silent risk to leave undocumented.
