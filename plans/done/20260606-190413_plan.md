## Implementation Steps

### Step 1: 実装状況確認（完了）

コード調査の結果、Phase 1・Phase 2 は完了。Phase 3 に残存ギャップあり。

### Step 2: /mcp install テンプレートの精査（要確認）
- `scripts/mcp/installer_templates.py` を読み、FS/SQLite/Script/CI 専用テンプレートが存在するかを確認する
- 不足テンプレートがあれば追加する（`python-implementation` スキルを使用）

### Step 3: Worker ロール別 allowed-tools の設計
- `scripts/agent/orchestrator.py` を読み、Worker 生成時に `allowed_tools` を渡せるかを確認する
- 渡せない場合は `AgentConfig.tool.allowed_tools` をワーカーごとに上書きする仕組みを設計し、別プランとして立案する

### Step 4: バリデーション（変更した場合のみ）
- `uv run ruff check scripts/`
- `uv run mypy scripts/`
- `lint-imports`
- `uv run pytest`

### Step 5: instructions/done/ へ移動
- `instructions/20260528_02_instruction.md` を `instructions/done/` に移動する

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Worker ロール別 allowed-tools の実装が orchestrator の大規模リファクタリングを要する | medium | 現状の `allowed_tools` フィールドを Worker 初期化時に注入する小さな変更で済む可能性が高い。スコープを最小化する。 |
| Browser MCP の追加が外部依存（Playwright 等）を増やしてセキュリティリスクになる | medium | Browser MCP は別 instruction で扱い、sandbox 設計を先行させる。 |
| /mcp install テンプレートが不完全で不正なサーバーコードを生成する | low | 生成後に ruff + mypy + pytest で検証するステップを必須にする。 |
