## Goal

Add a retry cooldown mechanism to `_ServerLifecycleRouter` that prevents rapid subprocess start retries for failed MCP servers in non-production mode.

## Scope

**In:**
- `scripts/agent/factory.py`: Add `_failed_starts` dict and cooldown logic to `_ServerLifecycleRouter`

**Out:**
- Modifying production mode behavior (already treats startup failures as fatal)
- Changing the exception propagation path in `ensure_ready()`
- Any other file modifications

## Assumptions

1. The cooldown should apply regardless of whether the failure occurred during initial startup or a later tool call — any failed subprocess start triggers the cooldown.
2. The cooldown duration is configurable via a module-level constant (`COOLDOWN_SECONDS`) rather than a runtime configuration option.
3. The cooldown applies only to HTTP subprocess-mode servers (the same subset affected by the current `ensure_ready()` logic).
4. Successful subprocess start clears the cooldown entry, allowing the next failure to trigger a new cooldown period.

## Unknowns

Resolved: No unresolved unknowns remain.

- **UNK-01** (resolved): Whether the cooldown should also apply to `start_http_subprocess()` and `restart()` methods. Since these methods are called from `StartupOrchestrator._start_servers()` (initial startup) and `/restart` command respectively, they may also benefit from cooldown protection. However, the requirement specifically mentions `ensure_ready()` as the primary concern. For safety, the cooldown should be applied at the `_http_mgr.start()` level so all callers benefit. This requires moving the cooldown check into `HttpServerLifecycleManager.start()` rather than `_ServerLifecycleRouter.ensure_ready()`.

## Implementation Steps

### Phase 1: Core Logic Implementation

- [ ] Step 1: Add `import time` at module level if not already imported.
- [ ] Step 2: Define `COOLDOWN_SECONDS: float = 30` at module level (after `_logger` definition).
- [ ] Step 3: In `_ServerLifecycleRouter.__init__()`, add `self._failed_starts: dict[str, float] = {}` alongside existing instance variables.
- [ ] Step 4: Add a private method `_in_cooldown(self, server_key: str) -> bool` that checks if the server is within the cooldown window using `time.monotonic()`.
- [ ] Step 5: In `ensure_ready()`, before calling `self._http_mgr.verify_running()`, check if the server is in cooldown via `_in_cooldown()`. If yes, log a warning and return early without attempting to start the subprocess.
- [ ] Step 6: After a successful subprocess start in `ensure_ready()` (line 111), remove the server from cooldown: `self._failed_starts.pop(server_key, None)`.
- [ ] Step 7: After a failed subprocess start in `ensure_ready()` (line 112-114), record the failure time: `self._failed_starts[server_key] = time.monotonic()`.
- [ ] Step 8: Apply the same cooldown check to `start_http_subprocess()` and `restart()` methods to prevent cooldown bypass via direct calls.
- [ ] Step 9: Consider moving the cooldown check into `HttpServerLifecycleManager.start()` instead of `_ServerLifecycleRouter.ensure_ready()` so all callers (including `start_http_subprocess()` and `restart()`) share the same cooldown logic. This avoids duplication but requires modifying `lifecycle_protocol.py` interface contract.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Cooldown prevents recovery even after underlying issue resolves | Low | Cooldown duration (30s) is short enough for most transient issues; after cooldown expires, the next tool call will retry automatically. |
| Cooldown state persists across restarts if the process survives but the server config changes | Low | The cooldown is in-memory only; on process restart, it resets naturally. If the server config changes while the process is running, the cooldown would need to be cleared manually — document this limitation. |
| Cooldown applies to production mode despite being intended for non-production only | Medium | The cooldown applies to all modes equally. Since production mode already treats startup failures as fatal (fail-fast), the cooldown has minimal practical impact there. Document this behavior explicitly. |
