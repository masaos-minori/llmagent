# Plan: [RAG][Retrieval] Verify fetch_full_document() ordering after chunk_index fix

## Goal

Add tests that verify `fetch_full_document()` returns chunks in correct document order (by `chunk_index` ascending) for both full-document and windowed retrieval modes.

## Scope

**In scope:**
- New file: `tests/test_fetch_order.py` (or add to existing test file)
- Read-only reference: `scripts/rag/repository.py` — `fetch_full_document()`
- Read-only reference: `tests/test_document_repo.py` — testing patterns (in-memory SQLite, `_FakeSQLiteHelper`)

**Out of scope:**
- Fixing BUG-3 (`chunk_index=0` in ingester) — that is a separate task
- Changes to ingestion pipeline or chunk splitting logic
- Changes to FTS or vector search behavior

## Assumptions

1. `fetch_full_document()` queries chunks ordered by `ORDER BY c.chunk_index ASC` — the SQL is correct; the problem is data-level (all chunk_index=0 currently due to BUG-3).
2. The windowed query uses `BETWEEN max(0, chunk_index - window) AND chunk_index + window` where `chunk_index` comes from the fetched row — this means the center point is dynamic based on the requested chunk's index.
3. Tests will use in-memory SQLite with the RAG schema (pattern from test_document_repo.py).
4. `chunks_vec` can be a stub table (no FK constraint needed for ordering tests).

## Unknowns

| Unknown | Evidence missing | Resolution | Blocking |
|---|---|---|---|
| Does the windowed query use `chunk_index` of the target chunk or always center at 0? | Source code analysis confirms it uses the fetched row's chunk_index (line 253, 268). | Confirmed: window is centered on the requested chunk's index. If all indexes are 0, window covers only index 0. | No |
| What is the expected behavior when two chunks have the same `chunk_index`? | SQLite ORDER BY is stable but does not guarantee secondary sort order when values are equal. | Accept: when chunk_index ties occur (current BUG-3 state), order is implementation-defined (rowid order). Tests should verify that correct indexes produce deterministic order. | No |

## Implementation Steps

1. **Create test schema fixture** (adapt from test_document_repo.py):
   - In-memory SQLite with full RAG schema (documents, chunks, chunks_vec stub)
   - `_FakeSQLiteHelper` for DB abstraction
   - Helper to insert document + chunks with known chunk_index values

2. **Test full-document fetch ordering**:
   - `test_fetch_full_document_returns_chunks_in_index_order` — insert 5 chunks with indices 0-4, verify returned order matches index order
   - `test_fetch_from_middle_chunk_returns_all_chunks` — request from middle chunk_id, verify all document chunks returned in order
   - `test_fetch_nonexistent_chunk_id_returns_empty` — verify not-found handling

3. **Test windowed fetch ordering**:
   - `test_window_fetch_returns_correct_subset` — insert 10 chunks (indices 0-9), request window=2 around index 5, verify indices 3-7 returned
   - `test_window_fetch_ordered_within_subset` — verify returned subset is still in ascending chunk_index order
   - `test_window_fetch_at_boundary` — request window at start (index 0) and end (last index), verify no negative or out-of-range indices
   - `test_window_fetch_no_match_returns_empty` — request window around non-existent chunk_id

4. **Test ordering with duplicate chunk_index** (BUG-3 simulation):
   - `test_duplicate_chunk_index_returns_deterministic_order` — insert chunks with same index, verify rowid-based order (SQLite stability)
   - This test documents the current limitation and will pass once BUG-3 is fixed

5. **Validation:** run `uv run pytest tests/test_fetch_order.py -v` and confirm all tests pass.
