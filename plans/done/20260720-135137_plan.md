# Plan: Merge browser-mcp into web-search-mcp

Source requirement: `requires/20260720_02_require.md` (moved to `requires/done/` as part of this planning cycle)

## Goal

Consolidate the `browser` MCP server (`browser_fetch` tool, port 8016) into the `web_search` MCP server (`search_web` tool, port 8004), so a single FastAPI process on port 8004 serves both tools, `browser_fetch` retains its existing security model (domain allowlist, IP-literal rejection, size limits) and behavioral parity, and all references to the standalone `browser` server are removed from code, config, tests, and docs.

## Scope

**In scope:**
- `scripts/mcp_servers/web_search/{web_search_models,web_search_tools,search_provider,formatters,service,web_search_server}.py` — add browser_fetch models/provider/formatter/tool-schema/routes.
- Deletion of `scripts/mcp_servers/browser/` (entire directory) and `config/browser_mcp_server.toml`.
- `config/web_search_mcp_server.toml` — add `browser_*` keys.
- `config/agent.toml` — remove `[mcp_servers.browser]` block, keep/fix `[mcp_servers.web_search]`'s `tool_names` (add `"browser_fetch"`) and `cmd`, update `tool_safety_tiers`/`tool_definitions` entries as needed (no functional change to their values, just consolidation).
- `scripts/shared/tool_constants.py` — remove `BROWSER_TOOLS`, fold `"browser_fetch"` into `WEB_SEARCH_TOOLS`.
- `deploy/deploy.sh` — remove the `browser_mcp_server.toml` cp line.
- `tests/test_runtime_tool_routing_integration.py` — update `browser_fetch` fixtures/imports to point at the merged server/module; route resolution should now map `browser_fetch` → `web_search` server key.
- Documentation files listed in the requirement's "Files that reference browser-mcp" section.

**Out of scope (per requirement's "What NOT to change"):**
- `search_web` tool behavior, DuckDuckGo provider logic, audit log format/structure.
- Adding new tools beyond `browser_fetch`.
- The separate, already-planned-but-unimplemented fix for `config/agent.toml`'s stale `cmd` paths across 8 renamed MCP servers (`issues/20260720-070458_mcp_server_agent_toml_cmd_points_to_deleted_files.md`, `plans/done/20260720-073534_plan.md`) — this merge plan touches the same `agent.toml` lines for its own reasons (deleting the browser block, editing the web_search block) but does not re-plan that separate fix; see Assumptions.
- No implementation/code changes are performed by this planning cycle itself.

## Assumptions

1. `config/agent.toml`'s `[mcp_servers.web_search].cmd` currently still points at the deleted `scripts/mcp_servers/web_search/server.py` (confirmed live in the file, line ~313) — the separate, unimplemented fix in `plans/done/20260720-073534_plan.md` targets exactly this line (and the analogous `browser` line, which this merge deletes wholesale). Since this merge plan's own agent.toml edit necessarily rewrites the `web_search` `cmd` entry (and removes the `browser` block entirely), the merge implementation should set the correct `.../web_search_server.py` path directly as part of its own edit, which incidentally also resolves that row of the separate stale-cmd-path bug. The `browser` row of that bug becomes moot once its whole config block is deleted. No coordination beyond "get the path right" is required; this plan does not implement or duplicate the separate fix for the other 7 unrelated servers.
2. The merged server keeps `web_search_server.py`'s existing exception-hierarchy pattern (502 for `WebSearchUpstreamError` and subclasses) for search failures, and layers `browser`'s existing 403/422 exception handlers (`BrowserAuthorizationError`, `BrowserValidationError`) alongside it as separate, additional `@app.exception_handler` registrations — i.e., the two error models are NOT collapsed into one hierarchy; they coexist since they represent genuinely different failure semantics (upstream/network failure vs. client policy/validation failure). This matches the requirement's own sample code.
3. `browser_fetch`'s tool schema keeps the `"config_dependent": True` field (distinct from `search_web`'s `"status": "production"`-only style) because `tests/test_runtime_tool_routing_integration.py::TestBrowserToolsConfigDependentMigration` locks this field in; the merge relocates that test class's assertions to import from `web_search_tools.py` instead of deleting the guarantee.
4. Port 8016 and the standalone `browser-mcp` process are retired entirely; nothing outside `config/agent.toml`'s `[mcp_servers.browser]` block references port 8016 (confirmed — no docker-compose/.env.example files exist in this repo, so no container/env cleanup is needed there).
5. `docker-compose.yml`, `docker-compose.dev.yml`, and `.env.example` do not exist in this repository (confirmed via search) — the requirement's file list mentioning them is stale/inapplicable; this is a documentation-fix-required item on the requirement itself, not something to action. No implementation step is planned for them.
6. The existing `web_search_mcp_server.toml`'s empty `auth_token` semantics (browser's `auth_token=""` today is a no-op since `attach_auth_middleware(app, "")` disables auth for empty tokens) mean dropping the middleware call carries no immediate behavioral regression, but see Risks below for the recommended mitigation.

## Design

- **Layering preserved**: the merge follows the web_search package's existing 6-layer structure (models → provider → formatters → service → health/metrics → server) rather than grafting `browser`'s flatter `BrowserService`-class style on top. `search_provider.py` gains a second async function (`fetch_browser`) alongside `search_duckduckgo`; `service.py` gains a second orchestration function (`fetch_browser`) alongside `search_web`; both share the same `formatters.py` dispatch table and the same FastAPI app/router in `web_search_server.py`.
- **Exception handling stays split by semantics, not merged**: `WebSearchUpstreamError` (and subclasses) → 502 for search failures; `BrowserAuthorizationError` → 403 and `BrowserValidationError` → 422 for fetch policy/validation failures. Both sets of `@app.exception_handler` registrations coexist on the same `app` object.
- **Health/metrics become per-tool** (UNK-03) rather than collapsing browser_fetch into the DuckDuckGo-specific singleton, so `/health` degradation reporting stays accurate per tool.
- **Config consolidation**: `browser_*`-prefixed keys land inside `WebSearchConfig`/`web_search_mcp_server.toml`, exactly as the requirement specifies, with `BrowserConfig` constructed from those fields at server startup (as a thin sub-view, not a separately-loaded TOML file).
- **Tool routing**: no hardcoded routing table exists outside live `/v1/tools` discovery + `RuntimeToolRegistry`/`ToolRouteResolver` (confirmed in Step 3 — `scripts/agent/startup.py`, `mcp_tool_discovery.py`, `cmd_mcp.py`, `runtime_tool_registry.py`, `route_resolver.py` have zero "browser" references); only `scripts/shared/tool_constants.py`'s `BROWSER_TOOLS`/`WEB_SEARCH_TOOLS` frozensets need a code edit outside the `mcp_servers/web_search` and `mcp_servers/browser` packages and `config/`.
- **Deploy**: since `scripts/` is rsync'd wholesale, the deleted `scripts/mcp_servers/browser/` directory needs no deploy.sh change beyond removing the standalone `browser_mcp_server.toml` cp line; `rsync --delete` will remove the stale directory from `/opt/llm/scripts/` automatically on next deploy.

8. **Config**: add `browser_*` keys to `config/web_search_mcp_server.toml`; remove `config/browser_mcp_server.toml`; remove `[mcp_servers.browser]` from `config/agent.toml`; update `[mcp_servers.web_search]`'s `tool_names`/`cmd`.

11. **Docs**: update the 8 confirmed-existing doc files (per Affected areas) to remove browser-mcp/port-8016 references.
12. **Deploy/verification (mandatory)**: run full `rules/toolchain.md` validation sequence (ruff, mypy, lint-imports, bandit, pytest full suite + `test_mdq_rag_boundary.py`, diff-cover ≥90%, pre-commit); then run `skills/deploy/SKILL.md` deployment flow to `/opt/llm` and confirm `/v1/tools` on port 8004 lists both `search_web` and `browser_fetch`, `/health` is OK, and port 8016 is no longer bound by any process.

## Risks

- **Risk**: Health/metrics singleton collision (browser_fetch failures skewing search_web's degradation status) if UNK-03 is implemented naively. **Mitigation**: per-tool keying in `health.py`/`metrics.py` as specified in Implementation step 4; add a unit test asserting a `browser_fetch` failure does not flip `search_web`'s `is_degraded()`.
- **Risk**: Auth regression if `attach_auth_middleware` wiring for `browser_auth_token` is dropped silently (UNK-02), since the requirement's literal sample code omits it. **Mitigation**: explicit implementation step 7 item; add a test that sets a non-empty `browser_auth_token` and confirms unauthenticated `browser_fetch` calls are rejected.
- **Risk**: `config/agent.toml` edit collision with the separate, still-unimplemented stale-`cmd`-path fix (`plans/done/20260720-073534_plan.md`) if both land independently and out of order, causing a merge conflict or one edit clobbering the other on the same lines. **Mitigation**: this merge's own `agent.toml` edit sets the correct `web_search` `cmd` path directly (Assumption 1) and deletes the `browser` block outright, so implement this merge's config edit in one atomic step and, before starting, check whether the other fix has landed in the interim (`git log` / `grep` the current `cmd` line) to avoid a stale re-edit.
- **Risk**: Hidden importers of `mcp_servers.browser.*` beyond the ones found in this planning pass (Step 3 research covered the 6 named agent/shared files plus 2 test files, but not an exhaustive repo-wide grep). **Mitigation**: Implementation step 1 explicitly re-greps the whole repo (`rg -l "mcp_servers\.browser|mcp_servers/browser"`) immediately before deletion, as a safety net.
- **Risk**: `browser_fetch`'s existing security guarantees (domain allowlist, IP-literal rejection, byte-safe truncation) are subtly altered during the code move from `browser_service.py` to `search_provider.py` (e.g., a refactor accidentally changes clamp order or truncation byte-boundary handling). **Mitigation**: port the `_check_domain`/`_truncate` logic verbatim (no rewrite) in Implementation step 3; the existing unit tests (domain allowlist, IP rejection, truncation) from `browser`'s test suite (if any exist under `tests/`) should be ported/re-run against the merged code, not just re-derived from the requirement's sample.
- **Risk**: `uv run check-mcp-docs` (validates tool counts against canonical frozensets, per `rules/toolchain.md`) may fail after `BROWSER_TOOLS` removal if any doc still hardcodes the old two-server tool count. **Mitigation**: run `check-mcp-docs` as part of the validation sequence (already in Validation plan) before considering the doc-update step complete; treat any failure as a doc-fix-required, not a blocker requiring a new issue.

No risk from this analysis was left unmitigated after adding the mitigations above — no `issues/*_risks.md` file is needed for this cycle.
