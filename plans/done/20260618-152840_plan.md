# Plan: Strengthen Plugin Strictness and Startup Diagnostics

Derived from `01_plan.md` using requirement `requires/20260618_26_require.md`.

---

## Goal

Plugin failures, conflicts, and shadowing are clearly reported at startup; strict mode is easy to enable and its behavior is predictable.

---

## Scope

**In scope:**
- `scripts/shared/plugin_registry.py` — improve error reporting, add command shadowing detection, add return-type validation at registration time
- `scripts/agent/factory.py` — log `result.failed` breakdown at startup
- `scripts/agent/config_dataclasses.py` — optional: env-var override for `plugin_strict`
- `docs/05_agent_11_extension-points.md` — document diagnostic output format and strict mode behavior
- `docs/05_agent_08_configuration.md` — document env-var override if added

**Out of scope:**
- Runtime plugin management CLI (`/plugin list/reload`)
- Plugin directory configurability
- Plugin tool return-value convention change (design decision deferred)

---

## Assumptions

1. `PluginLoadResult.failed` contains `(name, exception)` tuples that can be logged individually.
2. Command names registered via `@register_command` are accessible for collision checking at import time.
3. Return-type validation at `@register_tool` time does not break existing plugins that happen to match the convention.

## Implementation Steps

1. **Read existing plugins**
   - Inspect `plugins/example.py` and any other plugin files
   - Confirm registration-time validation will not break them
   - Criterion: no false-positive validation failures

2. **Add command-shadowing detection to `load_plugins()`**
   - Compare newly registered command names against known built-in command names
   - Log warning or error based on strict mode
   - Criterion: shadowing visible at startup

3. **Add return-type validation to `@register_tool`**
   - Check that the registered function has `tuple[str, bool]` annotation
   - Log warning on mismatch (non-blocking by default)
   - Criterion: UNDOC-02 addressed

4. **Log individual failures in `factory.py`**
   - Iterate `result.failed` and log each `(name, reason)` pair
   - Criterion: startup log has per-failure visibility

5. **Document new diagnostics**
   - Update `docs/05_agent_11_extension-points.md`
   - Criterion: doc matches implementation

6. **Lint, type, arch validation**
   - `ruff check scripts/` — 0 errors
   - `mypy scripts/` — no new errors
   - `lint-imports` — 0 violations
