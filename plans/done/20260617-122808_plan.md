# Plan: Req 32 — Persist runtime diagnostics and tool-result summaries for later analysis

## Goal

Persist a lightweight session-end diagnostic summary to `diagnostics.jsonl` so that important runtime diagnostics survive beyond the REPL session for post-session analysis.

## Scope

### In scope
- `agent/repl.py`: Add `_persist_session_diagnostics()` method, call from `_run_repl_loop` finally block
- `tests/test_repl.py`: Tests for `_persist_session_diagnostics`
- `docs/05_agent_10_operations-and-observability.md`: Document diagnostics file location and reading examples

### Out of scope
- Full raw payload persistence (tool result full_text already persisted in tool_results table)
- Real-time streaming of diagnostics (batch at session end only)
- Configurable diagnostics output path or format
- Diagnostics cleanup/rotation

## Assumptions
- Diagnostics file stored as `diagnostics.jsonl` next to `session.sqlite` (same parent directory from `DbConfig`)
- One JSON-lines record per session end, appended (not overwritten)
- Fields include: session_id, timestamp, turns, tool_calls, tool_errors, partial_completions, parse_errors, heartbeat_timeouts, reconnects, semantic_cache_hits, input_tokens, output_tokens, compress_count, latency_summary, tool_result_summary
- Persistence failures are non-fatal (logged at DEBUG, do not affect shutdown)

## Design

### Data model (JSON-lines record)
```json
{
  "session_id": 42,
  "timestamp": "2026-06-17T12:00:00Z",
  "turns": 5,
  "tool_calls": 12,
  "tool_errors": 2,
  "partial_completions": 1,
  "parse_errors": 0,
  "heartbeat_timeouts": 0,
  "reconnects": 2,
  "semantic_cache_hits": 3,
  "input_tokens": 1000,
  "output_tokens": 500,
  "compress_count": 2,
  "latency_summary": {
    "llm": {"count": 5, "mean_ms": 1200.5, "max_ms": 2100.3},
    "compress": {"count": 2, "mean_ms": 50.0, "max_ms": 80.0}
  },
  "tool_result_summary": {"total": 8, "errors": 3}
}
```

### Implementation steps
1. **Add imports** to `repl.py`: `json`, `time`, `Path` from stdlib; `build_db_config`, `SQLiteHelper` from `db/`
2. **Add `_persist_session_diagnostics(self, ctx)` method** that:
   - Collects stats from `ctx.stats`, `ctx.services.llm`, `ctx.services.hist_mgr`
   - Queries `tool_results` table for COUNT/SUM per session_id
   - Builds summary dict
   - Writes to `<db_dir>/diagnostics.jsonl` as JSON-lines
   - Catches `OSError, sqlite3.Error` and logs at DEBUG
3. **Hook into `_run_repl_loop` finally block**: call `_persist_session_diagnostics(ctx)` before `_persist_session_memories`
4. **Add tests** in `tests/test_repl.py`:
   - `test_writes_jsonl_file`: verifies correct JSON output with all fields
   - `test_handles_none_session_id`: no crash when session_id is None
   - `test_handles_none_services`: no crash when services is None
5. **Update docs**: Add "Runtime Diagnostics" section to ops doc

## Risks

| Risk | Mitigation |
|------|-----------|
| Diagnostics file grows unbounded over many sessions | File is append-only; operators can clean periodically. Not in scope for this ticket. |
| DB query for tool_result_summary adds latency at session end | Query is a simple COUNT/SUM; negligible overhead |
| Persistence failure could block shutdown | Caught with try/except, logged at DEBUG, non-fatal |
| Sensitive data in diagnostics? | Summary only — no conversation content, no tool result full_text. Only counts and timestamps. |
