# Work Plan for RAG Pipeline MCP Server

## Goal
Implement a dedicated MCP server that consolidates the RAG pipeline into a single integrated service, replacing the current centralized RAG processing in `agent_rag.py`.

## Scope
- Implement `rag-pipeline-mcp` server with integrated RAG pipeline functionality
- Provide tools for running MQE, Search, RRF, Rerank, Dedup, and Augment as a single pipeline
- Support both standard and debug modes for the RAG pipeline
- Integrate with existing MCP operating model (HTTP, OpenRC, `/mcp`, `/v1/tools`, watchdog)
- Update Agent configuration to include the new MCP server and tools

## Assumptions
- Current RAG pipeline is implemented in `agent_rag.py` with six steps: MQE, Search, RRF, Rerank, Dedup, and Augment
- Current Agent assumes HTTP MCP servers using `/v1/call_tool` and `/v1/tools` endpoints
- Current Agent performs `/v1/tools` diff checks and watchdog-based health checks
- The system has Python 3.13 installed with required dependencies
- The database directory `/opt/llm/db/` exists
- The config directory `/opt/llm/config/` exists
- The scripts directory `/opt/llm/scripts/` exists

## Unknowns
- Specific implementation details of each RAG pipeline step (MQE, Search, RRF, Rerank, Dedup, Augment)
- How to properly integrate with existing RAG infrastructure in `agent_rag.py`
- Exact schema and parameter definitions for the RAG pipeline tools
- Whether the database directory `/opt/llm/db/` needs to be created (it already exists)
- Whether the config directory `/opt/llm/config/` needs to be created (it already exists)
- Whether the scripts directory `/opt/llm/scripts/` needs to be created (it already exists)

## Affected areas
- New files: `scripts/rag_pipeline_mcp_server.py`, `config/rag_pipeline_mcp_server.json`, `init.d/rag-pipeline-mcp`
- Modified files: `agent/config.py`, `agent/factory.py`, `agent/repl.py`, `agent/commands/cmd_mcp.py`
- Existing code reuse: `agent_rag.py`, `rag/types.py`, `rag/pipeline.py`, `shared/tool_constants.py`

## Design
- Use FastAPI + Uvicorn for MCP server implementation to align with existing HTTP MCP pattern
- Implement the complete RAG pipeline as a single integrated service
- Create two tools: `rag_run_pipeline` for standard operation and `rag_debug_pipeline` for debugging
- Use existing RAG infrastructure components where possible
- Follow existing MCP server base structure for consistency
- Implement proper error handling and logging

## Validation plan
- Unit tests for each RAG pipeline step
- Integration tests for the complete workflow
- Verify `/v1/tools` definition matches expected format
- Test that the server can be started and stopped properly
- Validate that the MCP server integrates with the agent system
- Confirm that the database is created and used correctly
- Check that tool definitions are properly registered

## Risks
- Incomplete RAG pipeline implementation - addressed by using existing components where possible
- Integration issues with existing RAG infrastructure - addressed by careful code reuse
- Tool definition mismatch - addressed by careful schema definition
- Database access issues - addressed by using existing database patterns
