## Goal

Wrap `ctx.session.start()` in a dedicated try/except that catches both `RuntimeError` and `sqlite3.Error`, emitting user-friendly messages for each case before re-raising so the outer handler always sees consistent error formatting.

## Scope

**In:**
- `scripts/agent/repl.py`: Add nested try/except around `ctx.session.start()` in `_run_repl_loop()`

**Out:**
- Modifying `session.py` — the existing "no such table" error message there is already correct
- Changing the outer `except RuntimeError` handler behavior
- Any other file modifications

## Assumptions

1. The inner handler should catch `RuntimeError` and `sqlite3.Error`, write a user-friendly message via `self._view.write_fatal()`, then re-raise so the outer handler handles cleanup.
2. The outer `except RuntimeError` handler will still receive the re-raised `RuntimeError` but will no longer receive raw `sqlite3.Error` exceptions (since they are caught and handled internally).
3. The `sqlite3.Error` message should suggest checking DB connectivity or running `bash deploy/init_db.sh`.

## Unknowns

Resolved: No unresolved unknowns remain.

- **UNK-01** (resolved): Whether the outer `except RuntimeError` handler should also catch `sqlite3.Error`. Since the inner handler re-raises after writing the fatal message, the outer handler would still receive it for `RuntimeError`. For `sqlite3.Error`, the inner handler catches and writes the message, then re-raises — which means the outer handler would NOT catch it (only catches `RuntimeError`). To prevent double-handling, use a sentinel exception class that the outer handler ignores, or simply do not re-raise and let the finally block handle cleanup (the process will exit naturally after the finally block completes).

### Phase 1: Core Logic Implementation

- [ ] Step 1: In `AgentREPL._run_repl_loop()`, wrap `ctx.session.start()` (line 418) in a nested try/except:
  ```python
  try:
      ctx.session.start()
  except RuntimeError as e:
      msg = str(e)
      if "no such table" in msg.lower():
          self._view.write_fatal("Session schema missing. Run: bash deploy/init_db.sh to initialize the database.")
      else:
          self._view.write_fatal(f"Session start failed: {msg}")
      # Re-raise as RuntimeError with a clean message so the outer handler
      # produces consistent formatting regardless of the original exception type.
      raise RuntimeError(msg) from None
  except sqlite3.Error as e:
      self._view.write_fatal(f"Database unavailable during session start: {e}. Check DB connectivity or run: bash deploy/init_db.sh")
      # Wrap in RuntimeError so the outer handler treats it uniformly.
      raise RuntimeError(f"Database unavailable: {e}") from None
  ```
- [ ] Step 2: The outer `except RuntimeError` handler (line 425) will now always receive a `RuntimeError` with a clean message, producing consistent output in both paths.
