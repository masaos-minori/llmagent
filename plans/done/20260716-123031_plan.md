# Plan: Remove partially wired MDQ FTS admin tools (`fts_consistency_check`, `fts_rebuild`)

Source requirement: `requires/20260716_02_require.md`

## Goal

`fts_consistency_check` and `fts_rebuild` are no longer advertised in the MDQ
tool schema, registry, or config anywhere in the codebase, so every tool MDQ
exposes is actually dispatchable — closing the schema/dispatch drift that
`docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md` already documents as a
known issue.

## Scope

**In scope** (removal path — see Design for why removal, not restoration, is
selected)

- Remove `fts_consistency_check` / `fts_rebuild` from `TOOL_LIST`
  (`scripts/mcp_servers/mdq/tools.py`).
- Remove `FtsConsistencyCheckRequest`, `FtsConsistencyCheckResponse`,
  `FtsRebuildRequest` from `scripts/mcp_servers/mdq/models.py`.
- Remove `MdqService.fts_consistency_check()` / `MdqService.fts_rebuild()`
  from `scripts/mcp_servers/mdq/service.py`, and the now-unused import of
  `fts_consistency_check, fts_rebuild` from `db_fts`.
- Delete `scripts/mcp_servers/mdq/db_fts.py` entirely — both functions it
  defines have no callers outside the two removed service methods (verified).
- Remove `fts_consistency_check`, `fts_rebuild` from `MDQ_TOOLS` and
  `MDQ_WRITE_TOOLS` in `scripts/shared/tool_constants.py`.
- Remove the two tool safety tier lines from `config/agent.toml`
  (`fts_consistency_check = "READ_ONLY"` / `fts_rebuild = "WRITE_DANGEROUS"`,
  lines 240 and 243) and remove both names from
  `[mcp_servers.mdq].tool_names` (line 387-388).
- Remove the `"stats", "fts_consistency_check", "fts_rebuild"` branch's two
  admin-tool names from `scripts/mcp_servers/mdq/audit_target.py:30` (keep
  `"stats"` since it remains a live tool).
- Update tests (see Affected areas) that assert the current 9-tool /
  admin-tool state.
- Update docs: `docs/04_mcp_04_04_mdq.md` (tool count 9→7, drop admin-tool
  mention), `docs/04_mcp_03_01_dispatch-and-routing.md` (`MDQ_TOOLS` list),
  `docs/90_shared_02_03_types_and_protocols-reference.md` (`MDQ_TOOLS` list),
  `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md` (replace the "known
  issue" bullet describing the drift with a resolution note, matching the
  pattern already used for the resolved `concurrency_limit` item).

**Out of scope**

- Restoring/wiring the two tools into `_DISPATCH_TABLE` (the requirement's
  alternative path — not selected; see Design).
- Any change to `search_docs`, `get_chunk`, `outline`, `index_paths`,
  `refresh_index`, `stats`, `grep_docs` — the 7 tools that remain unaffected.
- Adding the drift-guardrail tests that prevent this class of bug from
  recurring — that is `requires/20260716_03_require.md`, a separate
  requirement in this batch.

## Assumptions

1. There is no current operational requirement to expose FTS maintenance
   through the MCP layer — the requirement's own "Recommended direction"
   states removal unless such a need exists, and no caller (agent CLI, docs,
   runbook) invokes these tools today (verified: no dispatch entry has ever
   existed, per `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md`'s existing
   "既知の課題" bullet).
2. `scripts/mcp_servers/mdq/db_fts.py`'s two functions have zero callers
   outside `MdqService.fts_consistency_check`/`fts_rebuild` — verified via
   `rg -n "fts_consistency_check|fts_rebuild" scripts/` (only `service.py`,
   `db_fts.py`, `tools.py`, `audit_target.py` reference the names).
3. No dedicated unit test exercises `db_fts.py` functions directly (verified —
   no `test_mdq_db_fts*.py` or similar file exists); tests reference the tool
   names only through `TOOL_LIST` / `MDQ_TOOLS` / metadata-consistency
   assertions, all of which are already identified in Affected areas.
4. `tests/test_tool_scheduler.py`, `tests/test_tool_executor.py`,
   `tests/test_tool_runner.py` use the string `"fts_rebuild"` only as a
   representative example tool name to exercise generic executor/scheduler
   logic (constructing tool metadata from literal dicts) — they do not query
   `MDQ_TOOLS` or the live registry, so they are unaffected by this removal.
   This should be re-confirmed by running the full suite in Implementation
   step 5 rather than assumed silently.

## Design

Two outcomes were possible per the requirement: full removal, or formal
restoration (dispatch wiring + safety tier + serialization + audit + tests).
Removal is selected because:

- The requirement's own "Recommended direction" and "Intent" sections state a
  preference for removal absent an active operational need.
- No caller has ever invoked these tools through MCP (the dispatch table
  never had entries for them — this is not a regression, it is a
  never-finished feature).
- `db_fts.py`'s underlying SQL (`PRAGMA integrity_check`, FTS5 rebuild via
  `INSERT INTO chunks_fts(chunks_fts) VALUES('rebuild')`) remains useful as
  an operational technique but is better run as an ad hoc SQL command or a
  future dedicated admin CLI command than as an unfinished MCP tool surface —
  restoring it properly (safety tier, serialization, audit enrichment, tests)
  is out of scope for a compatibility-cleanup pass and was not requested.

No new abstraction is introduced; this is a subtractive change only.

## Implementation steps

1. **Remove schema layer** — delete the two `TOOL_LIST` entries in
   `tools.py` and the three model classes in `models.py`. Independently
   testable: `tests/test_mdq_metadata_consistency.py::test_total_tool_count`
   should still be updated in the same commit since it will otherwise fail
   immediately.
2. **Remove service layer** — delete `MdqService.fts_consistency_check()`,
   `MdqService.fts_rebuild()`, the `db_fts` import, and delete
   `scripts/mcp_servers/mdq/db_fts.py`.
3. **Remove registry/config layer** — update `MDQ_TOOLS`/`MDQ_WRITE_TOOLS` in
   `tool_constants.py`, `config/agent.toml` safety tiers and `tool_names`,
   and `audit_target.py`'s admin-tool branch.
4. **Update tests** — `test_mdq_routing.py`, `test_mdq_metadata_consistency.py`,
   `test_tool_constants.py` per Affected areas.
5. **Full-suite sanity check** — run `uv run pytest tests/test_tool_scheduler.py
   tests/test_tool_executor.py tests/test_tool_runner.py -v` to confirm
   Assumption 4 (these tests use `"fts_rebuild"` as a literal example name and
   are unaffected).
6. **Update documentation** — the four doc files listed in Affected areas.
7. **Deployment/verification** — check `deploy/deploy.sh` for an explicit
   `db_fts.py` copy line and remove it if present; restart `mdq-mcp` in a dev
   environment and confirm `GET /v1/tools` now returns 7 tools and matches
   `_DISPATCH_TABLE` exactly.

## Risks

- **Risk**: Deleting `db_fts.py` loses the only tested implementation of
  FTS5 consistency-check/rebuild logic, which could be needed for a future
  operator runbook.
  **Mitigation**: the SQL is simple (2 statements) and is preserved verbatim
  in this plan's Design section and in git history; a future admin-tool
  restoration can recover it from `git log -- scripts/mcp_servers/mdq/db_fts.py`
  without needing the MCP-facing wrapper.
- **Risk**: `config/agent.toml` is a production config file — an incorrect
  edit could break MDQ startup if `tool_names` and the live server's actual
  tool set diverge.
  **Mitigation**: Implementation step 7 explicitly restarts `mdq-mcp` in dev
  and checks `/v1/tools` against `_DISPATCH_TABLE`; `check-mcp-docs` also
  validates tool-count consistency.
- **Risk**: Overlap with `requires/20260716_03_require.md` (adds drift
  guardrail tests) if implemented out of order — the guardrail tests should
  be added against the post-removal (7-tool) state, not before.
  **Mitigation**: implement this plan (02) before `03_require.md`'s plan, per
  the calling workflow's sequential per-file processing order.
