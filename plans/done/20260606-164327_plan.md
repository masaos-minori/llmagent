# Work Plan for Markdown Context Compression Engine MCP Server

## Goal
Implement a dedicated MCP server that compresses large Markdown while preserving meaning and retrieves only the necessary parts for injection.

## Scope
- Implement `mdq-mcp` server with Markdown-specific indexing and retrieval capabilities
- Enable heading-level local search and local retrieval for large Markdown documents
- Integrate with existing MCP operating model (HTTP, OpenRC, `/mcp`, `/v1/tools`, watchdog)
- Update Agent usage rules to prefer `search_docs` over `read_file` for Markdown documents

## Assumptions
- Current ingestion pipeline includes `web_crawler.py`, `chunk_splitter.py`, `rag_ingester.py`
- `chunk_splitter.py` already supports Markdown snippet splitting by heading boundary via `md_index_enable`
- Current Agent assumes HTTP MCP servers using `/v1/call_tool` and `/v1/tools` endpoints
- Current `file-mcp` provides search-oriented access to large documents but lacks Markdown heading structure support
- Target documents include local Markdown files, ADRs, specifications, and SKILL.md files
- Index storage will be in a dedicated local DB equivalent to `.mdq/index.sqlite`
- Agent usage rule will be: use `search_docs` before `read_file`

## Unknowns
- Specific implementation details of heading-aware parsing in Phase 1
- Exact schema and indexing approach for `md_chunks` table
- How to integrate with existing `chunk_splitter.py` logic
- Specific requirements for the MCP server base implementation
- How to handle incremental indexing and content hash comparison
- Details of the FTS5 implementation for Japanese search
- Whether the database directory `/opt/llm/db/` needs to be created (it does)
- Whether the config directory `/opt/llm/config/` needs to be created (it does)
- Whether the scripts directory `/opt/llm/scripts/` needs to be created (it does)

## Affected areas
- New files: `scripts/mdq_mcp_server.py`, `scripts/mdq_indexer.py`, `scripts/mdq_parser.py`, `scripts/mdq_search.py`, `scripts/mdq_models.py`, `config/mdq_mcp_server.json`, `init.d/mdq-mcp`
- Modified files: `tool_executor.py`, `agent_repl.py`, `agent_commands.py`, `deploy/deploy.sh`, `deploy/setup_services.sh`
- Existing code reuse: `chunk_splitter.py`, `SQLiteHelper`, `logger`, `config_loader`, shared MCP server base

## Design
- Use FastAPI + Uvicorn for MCP server implementation to align with existing HTTP MCP pattern
- Implement heading-aware parsing leveraging existing `chunk_splitter.py` logic
- Use SQLite with FTS5 for indexing, similar to current RAG implementation
- Create dedicated database `.mdq/index.sqlite` separate from `rag.sqlite`
- Implement tools: `search_docs`, `get_chunk`, `outline`, `index_paths`, `refresh_index`, `stats`, `grep_docs`
- Follow existing MCP server base structure for consistency

## Validation plan
- Unit tests for heading extraction, outline correctness, incremental indexing, grep/BM25/hybrid search
- Integration tests for `index_paths -> search_docs -> get_chunk` flow
- Performance tests with large SKILL.md and multiple ADR files
- Verify `/v1/tools` definition diff check works
- Confirm watchdog/OpenRC startup verification

## Risks
- Incomplete Markdown parser coverage - addressed by using heading-aware parse in Phase 1
- SQLite index growth - addressed by incremental indexing, content hash comparison, snippet length limits
- Agent keeps using `read_file` - addressed by adding explicit rule to tool_definitions and system prompt
- Variability in Japanese search precision - addressed by using normalized FTS fields like current RAG system
