# Plan: Generation-Aware Invalidation for SemanticCache

**Requirement:** `requires/20260624_10_require.md`
**Task size:** Path B (extend SemanticCache + pipeline integration)

---

## Goal

Add a `generation` counter to `SemanticCache` so all entries can be invalidated atomically when the indexed corpus or retrieval config changes.

---

## Current State

- `cache.py:31` — `SemanticCache`: in-memory FIFO; keyed by (embedding, history_context); no generation concept
- `CacheEntry` holds: `embedding`, `context_str`, `history_context` — no `generation` field
- No invalidation hook after ingestion or config reload

---

## Scope

**In:**
- `scripts/rag/cache.py` — add `_generation: int` counter; `invalidate()` method bumps it; `CacheEntry` gets `generation` field; `lookup()` skips stale entries
- `scripts/rag/pipeline.py` — call `cache.invalidate()` after ingestion operations (if pipeline holds the cache)
- `scripts/shared/config_loader.py` or reload path — call `cache.invalidate()` on reload if embedding config changes

**Out:**
- Replacing FIFO with LRU
- Redesigning the entire cache implementation

---

            self._entries.clear()  # simplest: clear all; no stale reads possible

**Decision**: use `clear()` on invalidation (simplest; avoids partial-staleness bugs). The `generation` field on `CacheEntry` becomes documentation-level only — entries beyond the cleared state are never present.
