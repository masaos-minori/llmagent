## 1. Goal

Implement production-safe `grep_docs` in mdq-mcp with `paths` filtering, result caps, context line controls, `chunk_id` in matches, regex validation, and unbounded-scan prevention.

## 2. Scope

- **In-Scope**:
  - Apply `paths` filter as path-prefix or exact-path match against `chunks.source_path`
  - Enforce `max_matches` (per-request override of `max_grep_matches`, default 200)
  - Enforce `max_chars_per_match` (per-request override, default 500)
  - Context-line controls (`context_before`, `context_after`) — already in model; wire them correctly into `_find_grep_match`
  - Ensure each match includes `chunk_id`, `heading_path`, and `source_path` in output
  - Raise `MdqValidationError` (→ `is_error=true`) on invalid regex
  - Fix current `paths` filter: currently uses `source_path IN (...)` (exact only); spec requires prefix matching (`LIKE ?%` for directory paths) plus exact matching for file paths
  - Add request-level `max_matches` field alias in `GrepDocsRequest` if needed (model already has `max_grep_matches`)
  - Add test coverage for: paths filter, max_matches cap, invalid regex, chunk_id in output

- **Out-of-Scope**:
  - DB schema changes (no new columns or tables needed)
  - index_root_jail / path allowlist redesign (dependency listed in require but separate task)
  - MDQ SQLite schema redesign (separate dependency)
  - Timeout-based scan abort (regex pre-compilation already prevents catastrophic backtracking; no timeout thread added)
  - Hybrid search changes

## 3. Assumptions

- `paths` filter semantics: a path ending in `/` or pointing to a directory → prefix match (`source_path LIKE 'path%'`); a path pointing to a file → exact match (`source_path = 'path'`). Mixed list supported.
- `max_grep_matches` is the canonical request field name (already in `GrepDocsRequest`); no alias change needed.
- `_find_grep_match` returns only the first match per chunk (current behavior); this is acceptable. Requirement says "return chunk IDs with matches," not "all matches within a chunk."
- `context_before` and `context_after` are wired into `_find_grep_match` but `_context_lines` is currently unused in output (dead code). The context lines should be prepended/appended to `match_text` in the formatted output.
- No DB migration needed since no schema change.
- Tests use `tmp_path` SQLite; no prod DB dependency.

## 8. Risks & Mitigations

- **Risk**: Path-prefix LIKE query (`source_path LIKE 'path%'`) may match unintended paths if path string is not normalized (e.g., missing trailing slash) → **Mitigation**: Normalize path inputs — strip trailing slash for file check, preserve trailing slash for dir prefix, then apply deterministic rule; add unit test with edge-case paths.
- **Risk**: SQL f-string for dynamic WHERE clause construction introduces injection risk → **Mitigation**: The f-string only interpolates the count of placeholders (`? for _ in req.paths`), not user data; user data goes into `params` list. Existing pattern from `grep_docs` is safe; preserve it.
- **Risk**: Existing tests for `grep_docs_returns_matches_after_indexing` may break if context-line output format changes → **Mitigation**: Test asserts `"Chunk" in result or "Content" in result` — this is loose enough to survive format changes; confirm before modifying output format.
- **Risk**: `_find_grep_match` complexity increase from context-line inclusion → **Mitigation**: Extract context formatting into a `_format_context_match` helper to keep `_find_grep_match` at CC <= 5.
