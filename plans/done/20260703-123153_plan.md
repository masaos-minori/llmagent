## Assumptions

1. The `ToolRegistry` routing authority model is fully implemented in code (`shared/tool_registry.py`, `shared/route_resolver.py`) — the requirement calls for docs to match code, not to change code.
2. Live `/v1/tools` at startup is used only for drift-validation comparison, not to override routing at call time — this is the implemented behavior as of the latest commits.
3. The shared audit log at `/opt/llm/logs/audit.log` is now exclusively JSON-lines format written by `scripts/agent/tool_audit.py`; the MCP server-side key=value `AUDIT session=...` format from `mcp/audit.py` still exists but is written to a separate sink or is being removed.
4. `HttpTransport` already raises `TransportError` for transport failures (confirmed in `04_mcp_03` line 330 and `04_mcp_02` lines 259-261).
5. `ToolExecutor` already maps transport failures to `ToolCallResult(error_type="transport")` (confirmed in `04_mcp_02` lines 259-261).
6. The watchdog does not restart on dependency/config failures (e.g., missing `github_token`) unless the HTTP health response returns non-200 — this is confirmed by MCP-08 in `04_mcp_90` and `04_mcp_02` §Watchdog behavior.
7. `deploy/deploy.sh` does not reference any of the affected doc files (verified: no matches found).
