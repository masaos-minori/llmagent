# Plan: Improve ingestion and query diagnostics for fallback and quality analysis

## Goal

Add structured diagnostics for RAG fallback behavior and retrieval quality by persisting pipeline metrics to `DiagnosticStore` and enriching session summaries with retrieval signals.

## Scope

**In scope:**
- Persist per-query RAG diagnostics (stage outcomes, fallback reasons, hit counts) to `DiagnosticStore`
- Enrich session summary with retrieval metrics (chunk counts, score distributions, search cardinality)
- Add structured fallback reason tracking for refiner and remote RAG
- Capture skipped chunk counts and consistency warning counts

**Out of scope:**
- Changing the RAG pipeline logic itself
- Adding new diagnostic storage tables (use existing `session_diagnostics`)
- Modifying the diagnostic store schema (use existing `kind`/`content` fields)
- Adding real-time alerting or monitoring (beyond diagnostics recording)

## Assumptions

1. The existing `DiagnosticStore.save()` method is sufficient; no schema changes needed.
2. Diagnostics should be stored as JSON in the `content` field with a structured `kind` prefix.
3. Session summary enrichment should be backward compatible (new fields optional).

| Unknown | Evidence missing | Resolution | Blocking |

## Implementation Steps

1. **Add `RagPipeline.get_diagnostics()` method**
   - Return a dict with structured RAG metrics:
     ```python
     {
         "stage_results": [...],  # from self.last_stage_results
         "timings": {...},        # from self.last_timings
         "fetch_result": {...} if self.last_fetch_result else None,
         "fallback_count": len([r for r in stage_results if r["status"] == "fallback"]),
         "fallback_reasons": [r["fallback_reason"] for r in stage_results if r.get("fallback_reason")],
     }
     ```
   - Include hit counts: vector search count, FTS count, merged count, reranked count

2. **Persist diagnostics to DiagnosticStore**
   - Add call in orchestrator/turn handler after pipeline execution
   - Use `kind="rag_query"` or `kind="rag_diagnostics"` prefix
   - Store as JSON in `content` field
   - Example:
     ```json
     {
       "kind": "rag_query",
       "query_length": 50,
       "stage_results": [
         {"stage_name": "MQEStage", "status": "success", "fallback_reason": null},
         {"stage_name": "SearchStage", "status": "success", "fallback_reason": null},
         {"stage_name": "FusionStage", "status": "success", "fallback_reason": null},
         {"stage_name": "RerankStage", "status": "fallback", "fallback_reason": "refiner_disabled"}
       ],
       "timings": {"MQEStage": 0.01, "SearchStage": 0.5, "FusionStage": 0.001},
       "hit_counts": {"vector": 20, "fts": 15, "merged": 25, "reranked": 10},
       "fallback_reasons": ["refiner_disabled"]
     }
     ```

3. **Enrich session summary with retrieval metrics**
   - Add to session summary dict in REPL:
     - `rag_chunks_returned`: int (total chunks in final context)
     - `rag_search_hit_counts`: {"vector": int, "fts5": int}
     - `rag_stage_outcomes`: list of `{stage, status, fallback_reason}`
     - `rag_avg_rrf_score`: float (if RRF used)
     - `rag_avg_rerank_score`: float (if reranker used)

4. **Capture refiner fallback reason**
   - When `_augment_refiner()` returns None, log the specific reason:
     - "refiner_disabled" (config `use_refiner=False`)
     - "refiner_empty_output" (LLM returned empty string)
     - "refiner_http_error" (HTTP status error)
     - "refiner_network_error" (transport error)
   - Store in `fallback_reasons` list

5. **Capture remote RAG fallback reason**
   - When `_augment_http()` returns None, log the specific reason:
     - "http_4xx_client_error"
     - "http_5xx_server_error" (after retries exhausted)
     - "http_transport_error"
     - "http_parse_error"
   - Store in `fallback_reasons` list

6. **Capture skipped chunk counts**
   - Track chunks that were filtered out at each stage:
     - MQE: queries filtered/expanded count
     - Search: results below threshold
     - Fusion: deduplicated count
     - Rerank: results below score threshold
   - Store as `skipped_counts` in diagnostics
