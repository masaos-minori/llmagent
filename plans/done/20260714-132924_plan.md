# Work Plan: Remove Obsolete Direct-Execution Fallback Documentation

## Goal

Update all documentation to consistently state that workflow support is mandatory. Remove all references to direct-execution fallback, degraded workflow mode, `workflow_mode="disabled"`, or `workflow_mode="auto"`.

## Scope

This plan covers documentation updates only. No source code changes are required because the codebase has already been updated to enforce workflow requirements unconditionally.

## Assumptions

1. The requirement `requires/20260714_01_require.md` is the canonical specification for this task.
2. The validation sequence defined in `rules/toolchain.md` applies after documentation changes.
3. All affected documentation sections must be reviewed even if they do not contain obsolete content.

## Unknowns

1. Whether any non-documentation artifacts (e.g., deployment scripts, CI configs) also reference the old workflow modes.
2. Whether the `workflow_status()` return value contract in the API reference needs to be updated to match the new reality.
3. Whether MCP server documentation outside the agent category references the old workflow modes.

## Design

### Change Strategy

1. **Remove** all text describing direct-execution fallback paths.
2. **Simplify** descriptions of `workflow_status()` return values to reflect that workflow is always enabled.
3. **Condense** references to deleted `workflow_mode` fields — keep only that the fields are rejected via `_FORBIDDEN_KEYS`, remove elaboration on what those modes were.
4. **Add** explicit notes where the current behavior differs from the old documented behavior (e.g., the removed fallback paragraph).

### Consistency Rules

- After changes, every mention of workflow enforcement must state it is unconditional and mandatory.
- No document should describe a scenario where workflow is absent, disabled, or degraded.
- If a document previously described both the workflow path and the fallback path, retain only the workflow path.
