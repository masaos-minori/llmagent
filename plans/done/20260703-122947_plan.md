# Plan: Make ToolRegistry the Routing Authority and Use Live /v1/tools Only for Validation

## Goal

Remove live `/v1/tools` discovery as a routing override in `ToolRouteResolver`, wire live-vs-registry drift validation into startup, and update tests and documentation to match the new single-authority model.

---

## Scope

### In-Scope

- `scripts/shared/route_resolver.py`: Remove `_discovery_map` from `resolve()`; repurpose `build_discovery_map()` as validation helper; update docstrings.
- `scripts/shared/tool_registry.py`: Update module-level docstring and `validate_routing_against_live()` to support per-tool server_key mismatch detection; clarify routing priority comment.
- `scripts/agent/repl_health.py`: Add a new `check_routing_drift_vs_live()` function (or extend `check_routing_drift()`) that calls `validate_routing_against_live()` with live `/v1/tools` data; wire it into startup in strict mode.
- `tests/test_route_resolver.py`: Replace `TestLiveDiscoveryRouting.test_discovery_wins_over_registry` with tests proving discovery does NOT override registry; update `_log_routing_coverage` coverage test.
- `tests/test_tool_registry.py`: Add tests for the four strict-mode failure conditions listed in the requirement (live tool not in registry, registry tool not in live, cross-server key mismatch, duplicate live ownership).
- `docs/04_mcp_03_routing_lifecycle_and_execution.md`: Remove all "discovery map has highest priority / overrides registry" language; rewrite to describe `ToolRegistry` as sole routing authority and discovery as validation only.
- `docs/04_mcp_06_configuration_and_operations.md`: Update step 4 of the New Tool Registration Procedure; remove "enables priority-1 discovery routing" framing.
- `docs/04_mcp_90_inconsistencies_and_known_issues.md`: Move or add an entry that records the resolution of the discovery-override routing model (analogous to the resolved MCP-02 entry).

### Out-of-Scope

- `scripts/shared/tool_executor.py`: the `discovery_map` constructor parameter is retained for integration-test compatibility (`test_agent_mcp_integration.py` passes a discovery_map to route non-production tool names); only routing priority in `resolve()` changes.
- `tests/integration/test_agent_mcp_integration.py`: uses `discovery_map` to route synthetic tool names that are not in `tool_constants.py`; this use case requires a separate decision on whether to register integration-test tools in the registry.
- MCP server `/v1/tools` endpoint implementations (unchanged; they remain the source of validation data).
- `deploy/deploy.sh`: copies `scripts/` wholesale; no per-file references to affected modules; no change required.

---

## Assumptions

1. The `discovery_map` constructor parameter on `ToolRouteResolver` and `ToolExecutor` will be retained but not consulted during `resolve()`. Passing a non-empty `discovery_map` will no longer affect routing results.
2. Strict-mode startup failure for live-vs-registry drift is gated on the existing `tool_definitions_strict` config flag (same flag as `_check_tool_definitions`), because no separate `routing_strict` flag exists in the current config schema.
3. The `_collect_server_tool_names()` helper in `repl_health.py` already gathers live tool names per server; it will be reused or refactored to supply data to `validate_routing_against_live()`.
4. Duplicate live tool ownership detection (same tool name returned by two different servers) is implemented inside `build_discovery_map()` already; the new startup validation will call `build_discovery_map()` and inspect duplicates before invoking `validate_routing_against_live()`.
5. The integration test (`test_agent_mcp_integration.py`) uses `discovery_map` for synthetic tool names not in the registry; after this change those tests will break unless the synthetic tools are registered or an alternative routing path is provided. This is a known blast-radius item (see Risks).

## Implementation Steps

### Step 1 — Remove discovery routing from `ToolRouteResolver.resolve()`

**File:** `scripts/shared/route_resolver.py`

Changes:
- In `resolve()`, delete the `Priority 1: discovery map` block (lines 103-104).
- Update the module docstring to say: "Routing priority: 1. Tool registry (canonical source of truth). Unknown tools raise ValueError."
- Update the `ToolRouteResolver` class docstring similarly.
- Remove the "Discovery map from live /v1/tools metadata (highest priority)" comment on `self._discovery_map`.
- Retain `self._discovery_map` as an attribute and `_log_routing_coverage()` check (so the param still works for integration tests; it just no longer affects `resolve()`).
- Rename the internal attribute comment to: "Validation data from live /v1/tools (not used for routing)."
- Update `_raise_strict_error()` message to remove "not in discovery map or".
- Update warning log message in `resolve()` to remove "not in discovery map or".

**Completion criteria:** `resolve()` returns registry result for a tool in the registry even when `discovery_map` maps that tool to a different server key. `test_discovery_wins_over_registry` must be deleted or replaced (see Step 3).

### Step 2 — Wire live-vs-registry validation into startup

**File:** `scripts/agent/repl_health.py`

Changes:
- Add a new async function `check_routing_drift_vs_live(ctx: AgentContext, *, strict: bool = False) -> HealthCheckResult`.
- Inside it:
  1. Call `_collect_server_tool_names(ctx)` to get per-server live tool lists.
  2. Call `build_discovery_map(server_tool_lists)` to detect duplicate live ownership.
  3. Call `validate_routing_against_live(registry, live_tool_lists)` from `shared.tool_registry` to detect drift.
  4. For strict mode: if any drift is detected or duplicates found, raise `RuntimeError`.
  5. For non-strict mode: return `HealthCheckResult` with warnings.
- Collect the per-server tool list as `dict[str, list[dict]]` from `_collect_server_tool_names`; refactor that helper if needed to return the raw tool dicts (currently returns just names).
  - If refactoring is too invasive, a simpler approach: directly call `/v1/tools` per server inside `check_routing_drift_vs_live()` to get tool dicts needed for `build_discovery_map()`.
- Call `check_routing_drift_vs_live(ctx, strict=strict_flag)` from `_check_services()` in `scripts/agent/startup.py` after the existing `check_routing_drift(ctx)` call.

**Completion criteria:** At agent startup, a tool returned by a live server but absent from the registry produces a warning (non-strict) or `RuntimeError` (strict). Log message includes server key and tool name.

### Step 3 — Replace tests asserting discovery wins over registry

**File:** `tests/test_route_resolver.py`

- Add a new test: `test_discovery_map_does_not_override_registry` — construct a resolver with `discovery_map={"read_text_file": "custom_server"}` and assert `resolve("read_text_file") == "file_read"` (registry wins).

**Completion criteria:** `uv run pytest tests/test_route_resolver.py -v` passes with no test asserting discovery overrides registry.

### Step 4 — Add strict-mode startup validation tests

**File:** `tests/test_tool_registry.py`

Changes:
- Add a new test class `TestStartupValidationStrictMode`.
- Add tests for each of the four strict-mode conditions from the requirement:
  1. `test_live_returns_tool_not_in_registry` — call `validate_routing_against_live()` with a live tool not in the registry; verify mismatch is returned.
  2. `test_live_omits_registry_tool_for_server` — call with live list missing a registry-owned tool; verify mismatch is returned.
  3. `test_live_returns_tool_under_wrong_server` — registry maps `tool_x` to `server_a`; live returns `tool_x` under `server_b`; verify mismatch is detected.
  4. `test_duplicate_live_ownership_detected` — call `build_discovery_map()` with same tool under two servers; verify warning is logged.
- Verify that in each case `validate_routing_against_live()` returns a non-empty drift dict.

**Completion criteria:** `uv run pytest tests/test_tool_registry.py -v` passes with all new tests green.
