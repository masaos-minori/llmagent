# Unify workflow and approval source of truth in durable storage

**Requirement:** `requires/20260624_60_require.md`

## Goal

Make durable `workflow.sqlite` the only authoritative source for approval state so that `/approve` and `/reject` work correctly after a REPL restart that creates a new session_id.

## Scope

**In:**
- `scripts/agent/workflow/state_store.py` — add `find_latest_pending_approval()` (global, cross-session search)
- `scripts/agent/commands/cmd_workflow.py` — change `_cmd_approve()` / `_cmd_reject()` to query DB for approval_id instead of reading `ctx.turn.pending_approval_id`
- `scripts/agent/startup.py` — change `_recover_pending_approvals()` to use global search instead of session-scoped search
- `scripts/agent/context.py` — document `WorkflowState.approval_pending` and `TurnState.pending_approval_id` as display cache, not authoritative source
- `tests/test_workflow_state_store.py` — add tests for `find_latest_pending_approval()`
- `tests/test_cmd_workflow.py` — new file; test approve/reject including cross-session scenario

**Out:**
- Full workflow phase redesign
- Repository write boundary refactor
- Removing `WorkflowState` from `AgentContext` entirely (kept as display cache)
- Changing the workflow engine, `WorkflowEngine`, or task/attempt/artifact lifecycle

## Assumptions

1. Each REPL restart creates a new `session_id` (integer auto-increment from `sessions` table). The old session's tasks remain in `workflow.sqlite` with the old `session_id`. This is the root cause of the restart-safety gap.
2. `StateStore.find_pending_approval_by_session(session_id)` is correct for in-session lookup but fails after restart because the new session_id doesn't match the stored session_id.
3. At most one pending approval exists at any given time — the `find_latest_pending_approval()` global query returning the most recent one is safe.
4. `ctx.workflow.approval_pending` and `ctx.turn.pending_approval_id` remain as display cache in `AgentContext`; the Orchestrator guard can continue using them as a performance optimization (avoids DB query per turn).
5. The `orchestrator.py` approval guard (`if ctx.workflow.approval_pending:`) is correctly populated at startup by `_recover_pending_approvals()`. No change needed there beyond fixing the recovery query.
6. After a successful `/approve` or `/reject`, the display cache is still cleared (`ctx.turn.pending_approval_id = None`, `ctx.workflow.approval_pending = False`).

## Design

### Root Cause

`_recover_pending_approvals()` in `startup.py` calls:
```python
store.find_pending_approval_by_session(str(ctx.session.session_id))
```
After restart, `ctx.session.session_id` is a NEW integer (new DB row). The pending approval was stored with the OLD session_id. The query returns `None`, so recovery silently fails.

### Fix

#### 1. `StateStore.find_latest_pending_approval()`

Add a global (cross-session) query method:

```python
def find_latest_pending_approval(self) -> tuple[str, ApprovalRecord] | None:
    """Return (task_id, approval) for the most recent globally pending approval, or None."""
    rows = self._db.fetchall(
        """
        SELECT t.task_id, a.approval_id, a.stage_id, a.reason, a.created_at, a.resolved_at
        FROM tasks t
        JOIN approvals a ON t.task_id = a.task_id
        WHERE t.status = 'pending_approval'
          AND a.status = 'pending'
        ORDER BY a.created_at DESC
        LIMIT 1
        """,
        (),
    )
    if not rows:
        return None
    r = rows[0]
    return r["task_id"], ApprovalRecord(
        approval_id=r["approval_id"],
        task_id=r["task_id"],
        stage_id=r["stage_id"],
        status="pending",
        reason=r["reason"],
        created_at=r["created_at"],
        resolved_at=r["resolved_at"],
    )
```

#### 2. `cmd_workflow.py._cmd_approve()` / `_cmd_reject()`

Replace `ctx.turn.pending_approval_id` lookup with DB query:

```python
def _cmd_approve(self, arg: str) -> None:
    from agent.workflow import StateStore
    store = StateStore()
    try:
        result = store.find_latest_pending_approval()
        if result is None:
            self._out.write_validation_error(
                "No pending approval. Run a workflow task first."
            )
            return
        _task_id, approval = result
        approval_id = approval.approval_id
        reason = arg.strip() or None
        store.resolve_approval(approval_id, "approved", reason)
    except RuntimeError as exc:
        self._out.write_validation_error(f"Failed to resolve approval: {exc}")
        return
    finally:
        store.close()
    self._ctx.turn.pending_approval_id = None
    self._ctx.workflow.approval_pending = False
    self._out.write(f"Approved: {approval_id}")
```

Same pattern for `_cmd_reject()`.

#### 3. `startup.py._recover_pending_approvals()`

```python
async def _recover_pending_approvals(self) -> None:
    ctx = self._ctx
    if ctx.workflow is None:
        return
    store = StateStore()
    result = store.find_latest_pending_approval()   # ← global, not session-scoped
    store.close()
    if result is None:
        return
    task_id, approval = result
    ctx.workflow.approval_pending = True
    ctx.turn.pending_approval_id = approval.approval_id
    logger.warning(...)
    self._view.write_warning(...)
```

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Multiple concurrent pending approvals: `find_latest_pending_approval()` picks wrong one | Low | Single-user REPL; at most one workflow runs at a time. LIMIT 1 + ORDER BY created_at DESC is deterministic. |
| `_recover_pending_approvals()` now restores approvals from entirely different prior workflows that were abandoned, not just the most recent session | Low | Acceptable: user sees the warning and can `/reject` the stale approval. Better than silently losing it. |
| `cmd_workflow.py` no longer checks `ctx.turn.pending_approval_id` — the Orchestrator guard still uses `ctx.workflow.approval_pending`, so a race exists if cache is cleared but DB not yet written | Very low | Both clear/write happen synchronously in `_cmd_approve()`. No async race. |
| `find_latest_pending_approval()` SQL joins `tasks` and `approvals`; performance regression if tables grow large | Very low | Indexed on `task_id`; single-row LIMIT 1 result. Acceptable for operator REPL use. |
