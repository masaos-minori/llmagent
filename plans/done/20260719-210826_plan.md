## Goal

`outline()` reports staleness using the same unit-correct comparison as `/health` and `stats()`, and `search_docs()` uses one clearly defined effective SQL limit plus honest result-count wording, so agents never mistake a limited fetch count for the true total number of matches.

## Scope

**In scope:**
- Fix the unit-mismatched stale comparison in `MdqService.outline()` (`scripts/mcp_servers/mdq/mdq_service.py`).
- Extract a single shared stale-detection definition used by `outline()`, `stats()`, and `health_check._check_stale_documents()`.
- Define and enforce one effective SQL result limit in `scripts/mcp_servers/mdq/search.py`, replacing the current "unbounded `req.limit` fetch, then post-hoc truncation against a separate `req.max_results_limit`" design.
- Decide the fate of the request-level `SearchDocsRequest.max_results_limit` field and document the decision.
- Replace `total = len(results)` (post-SQL-limit count) with either an exact matched-count query or reworded, honest "shown" language (Notes for implementer: prefer honest wording if exact counting is deemed too costly — see Design).
- Add/adjust tests per the requirement's target list; update `docs/04_mcp_04_04_mdq.md` to document the new limit/count semantics.

**Out of scope (explicitly, per requirement Notes):**
- No database schema/migration changes (`documents`/`chunks` table shapes are unchanged).
- No change to FTS5/BM25 search mode support (still `bm25` only).
- ~~No fix to the orphaned pre-rename duplicate files~~ — moot, already deleted by commit `abcf0820`; this cycle targets only the live `mdq_`-prefixed files.
- No change to `grep_docs`, `get_chunk`, `index_paths`, `refresh_index` limit logic (unaffected by this requirement).

## Design

### 1. Shared stale-detection helper (`mdq_models.py`)

```python
def is_stale(mtime_ns: int, indexed_at: float) -> bool:
    """Return True if the file's on-disk mtime (nanoseconds) is newer than
    the last indexed_at timestamp (seconds, from time.time())."""
    return mtime_ns > int(indexed_at * 1e9)


STALE_SQL_CONDITION = "mtime_ns > CAST(indexed_at * 1e9 AS INTEGER)"
```

- `MdqService.outline()` (currently `mdq_service.py` line ~212: `if doc_row["mtime_ns"] > doc_row["indexed_at"]:`) is rewritten to call `is_stale(doc_row["mtime_ns"], doc_row["indexed_at"])`. This is the actual bug fix — today it compares nanoseconds to raw seconds directly, so any file indexed even once ever satisfies `mtime_ns > indexed_at` (mtime_ns is astronomically larger), producing a spurious stale warning on effectively every `outline()` call for freshly indexed files.
- `MdqService.stats()` (currently inlines `"... WHERE mtime_ns > CAST(indexed_at * 1e9 AS INTEGER)"` at line ~314) and `health_check._check_stale_documents()` (line ~34, identical inline literal) both switch to `f"SELECT COUNT(*) as cnt FROM documents WHERE {STALE_SQL_CONDITION}"`, importing the constant from `mcp_servers.mdq.mdq_models`. This removes the duplicated magic string and guarantees the three call sites can never drift again.
- `health_check.py` gains one new import (`from mcp_servers.mdq.mdq_models import STALE_SQL_CONDITION`); no import cycle risk since `mdq_models.py` has zero intra-package imports (confirmed: only `pydantic`/`typing`).

### 2. Effective SQL limit (`search.py`)

Today `_search_docs_structured()` uses `limit = getattr(req, "limit", 10) or 10` unbounded directly in `LIMIT ?`, while `search_docs()` separately computes `max_results = min(req.max_results_limit, service.max_results_limit)` and applies it only as a post-fetch Python-side truncation (`result["results"][:max_results]`). This means a request with a large `limit` (e.g. 10000) but no `max_results_limit` bypasses the config cap at the SQL layer entirely — the full row set is fetched into memory before any capping happens.

Fix: compute the effective SQL limit **inside** `_search_docs_structured()`:

```python
effective_limit = min(getattr(req, "limit", 10) or 10, service.max_results_limit)
```

and use `effective_limit` in the SQL `LIMIT ?` parameter, so the database itself never returns more rows than the config allows. `search_docs()`'s existing post-fetch logic for `req.max_results_limit` remains as a secondary, tighter display cap on top of `effective_limit` (per UNK-02 recommendation) — it now operates on an already-bounded result set instead of an unbounded one.

### 3. Honest / exact count reporting

Replace `SearchResultResult(query=req.query, results=results, total=len(results))` (currently reports post-SQL-limit row count as "total") with a structure that separates the two concepts, e.g.:

```python
class SearchResultResult(BaseModel):
    query: str
    results: list[SearchResultItem]
    matched_count: int   # exact count via COUNT(*) over same WHERE clause, no LIMIT
    shown_count: int     # len(results) after effective_limit
```

`_search_docs_structured()` runs a second query reusing `_build_search_where()`'s `where_clause`/`params` with `SELECT COUNT(*) ... ` (no `LIMIT`) to populate `matched_count`. `search_docs()`'s header changes from `f"Search results for: {req.query!r} ({original_total} found)"` to something like `f"Search results for: {req.query!r} ({matched_count} matched, {shown_count} shown)"` when `matched_count != shown_count`, or the simpler existing phrasing when they're equal (no matches were dropped). Exact wording is an implementation-time detail; the plan's binding requirement is that the header must never claim an exact "found" total that is actually just the retrieved/limited count.

If Option A's extra COUNT query proves to add meaningful latency in implementation (see UNK-03), fall back to Option B: drop the exact count entirely and always render `f"Showing {shown_count} matching results for: {req.query!r}"` plus the existing truncation note when `shown_count < matched-so-far`. This requires no structural change beyond what Option A already introduces (the `SearchResultResult` fields would collapse to just `shown_count` plus a boolean `possibly_more: bool`).

### 4. Documentation (`docs/04_mcp_04_04_mdq.md`)

Add a short subsection under "設定フィールド" or a new "検索結果件数と制限" note describing:
- precedence: `effective_limit = min(request limit, service.max_results_limit)`, applied at the SQL layer
- role of the request-level `max_results_limit` field (secondary display cap)
- matched vs. shown count wording, and that MDQ does not claim an exact total when results are limited unless the exact count was computed

## Implementation steps

1. **Stale detection fix** — add `is_stale()` + `STALE_SQL_CONDITION` to `mdq_models.py`; update `mdq_service.py::outline()` and `mdq_service.py::stats()` to use them; update `health_check.py::_check_stale_documents()` to use `STALE_SQL_CONDITION`. Independently committable and testable via `tests/test_mdq_stale_detection.py` (new) plus existing `tests/test_mdq_health_stale.py` (already covers health_check's SQL condition and should keep passing unchanged, since the SQL text is unchanged, only its origin becomes a shared constant).
2. **Effective SQL limit** — in `search.py`, compute `effective_limit` inside `_search_docs_structured()` and use it in the `LIMIT ?` clause; leave `search_docs()`'s existing `max_results_limit` post-fetch logic in place (operating on the now-bounded set). Covered by `tests/test_mdq_search_limits.py` (new): default limit, explicit limit, limit > config cap.
3. **Matched vs shown count** — extend `SearchResultResult` with `matched_count`/`shown_count` (or the Option B equivalent), add the exact-count query in `_search_docs_structured()`, update `search_docs()`'s header/truncation wording. Covered by `tests/test_mdq_search_counts.py` (new): matches > displayed, header wording, truncation message consistency.
4. **Update existing test file** `tests/test_mdq_search_modes.py` — adjust the three `TestResultLimitBehavior` assertions (`"3 found"`, `"5 found"`) to match the new header wording introduced in step 3, without weakening their original intent (default-limit-honored, override-below-cap, override-above-cap-bounded).
5. **Tool schema + docs** — update `mdq_tools.py`'s `search_docs` `inputSchema` descriptions for `limit`/`max_results_limit` to state the effective-limit precedence; update `docs/04_mcp_04_04_mdq.md` per Design §4.
6. **Full validation pass** — run the validation sequence in `rules/toolchain.md` (see Validation Plan below) before considering the change complete.
