# Plan: [Medium][eventbus][offsets] ハイブリッド replay+push モデルでのオフセット語義再定義

## Goal

ハイブリッド subscribe モデル (replay=SQLite / live=broker queue) でのオフセット更新語義を明確に定義し、再接続時に正しい durable seq から再開することを保証する。

## Assumptions

1. req #29 (ack ベースオフセット) が実装済み: オフセットは `POST /events/{id}/ack?consumer_id=X` でのみ更新される
2. subscribe() の disconnect 時 write_offset は req #29 で削除済み
3. mid-stream checkpoint は req #29 で削除済み
4. ハイブリッドモデルでのオフセット語義: 「最後に ack されたイベントの seq」=「次回再接続時の開始 seq」
5. リプレイフェーズ開始 seq は `read_offset(consumer_id)` から取得 (req #37 の設計を継承)

## Design

**ハイブリッドモデルでのオフセット完全仕様:**

```
オフセット更新の唯一のパス:
  POST /events/{event_id}/ack?consumer_id=X
  → write_offset(offsets_dir, consumer_id, seq_of_acked_event)

再接続時のリカバリー:
  GET /subscribe?consumer_id=X&since_seq=0
  → read_offset(offsets_dir, X) → start_seq
  → SQLite リプレイ: seq > start_seq の全イベントをバッチ取得
  → broker queue ライブ配信へ切替

切断時の動作:
  broker から unsubscribe → queue は GC される
  オフセットは ack 済み最終 seq のまま変わらない

再接続後の保証:
  最後に ack したイベントの次から再配信される (at-least-once)
  未 ack イベントは再配信される (コンシューマーの責務で冪等処理)
```

**テスト追加 (`tests/test_eventbus_offsets.py` または `test_eventbus_subscribe.py` 内):**

1. publish 3件 → ack 2件 → 切断 → 再接続 → 3件目から再配信されること
2. ack なしで切断 → 再接続 → 全件再配信されること
3. 全件 ack → 切断 → 再接続 → 新規イベントのみ配信されること
