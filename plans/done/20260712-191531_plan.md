# Plan: Guard shutdown_all() against a second SIGINT during cleanup

Source requirement: `requires/20260712_15_require.md`

## Goal

`HttpServerLifecycleManager.shutdown_all()` completes termination of every managed MCP
subprocess even if a second Ctrl-C (SIGINT) arrives while cleanup is already in progress.

## Scope

**In scope:**
- `scripts/agent/http_lifecycle.py` — `shutdown_all()` method only
- `tests/test_lifecycle.py` — new unit test(s) simulating a mid-loop interrupt

**Out of scope:**
- `scripts/agent/repl.py` SIGTERM handler registration (`run()`, lines 469-480) — not touched
- `issues/20260709_00_issue.md` section B (`os.killpg(1, sig)` broadcast fix) — already fixed, not touched
- `issues/20260709_00_issue.md` section C (thread/process leak) — already fixed, not touched
- `asyncio.gather()` parallelization of per-server termination — explicitly deferred in the requirement (out of scope)
- `prctl(PR_SET_PDEATHSIG)` OS-level safety net — explicitly deferred in the requirement (Linux-only, out of scope)
- Real-signal integration tests reproducing the double-interrupt end-to-end — explicitly deferred in the requirement (CI stability risk); only unit-level simulation is in scope

## Assumptions

1. `shutdown_all()` always runs on the main thread (called from `AgentREPL.run()` via `asyncio.run()`), so `signal.signal()` can be used to temporarily override the SIGINT handler — `signal.signal()` raises `ValueError` if called outside the main thread, which would surface immediately in tests if this assumption were wrong.
2. The existing SIGTERM handler registered in `repl.py:476` (`loop.add_signal_handler(signal.SIGTERM, ...)`) is independent of SIGINT handling and does not need to change.
3. No other code path calls `shutdown_all()` from a non-main-thread context (confirmed: only call site is `_close_resources()` in `repl.py:290`, itself only called from the `finally` block of `_run_repl_loop()`, which runs under `asyncio.run()` on the main thread).

## Design

Wrap the existing `shutdown_all()` loop body in a temporary SIGINT override, restored in a
`finally` block — the standard "ignore repeated Ctrl-C during cleanup" pattern used by many
CLI tools, and the only option from the requirement's "採用: 対策1" that stays entirely inside
`shutdown_all()` (satisfying the requirement's risk mitigation: no changes to `repl.py`'s
SIGTERM registration).

```python
def shutdown_all(self) -> None:  # (async, unchanged signature)
    old_sigint = signal.getsignal(signal.SIGINT)

    def _absorb_sigint(signum, frame):
        logger.warning(
            "Lifecycle: SIGINT received during shutdown_all(); "
            "ignoring until cleanup completes"
        )

    try:
        signal.signal(signal.SIGINT, _absorb_sigint)
    except ValueError:
        # Not on the main thread (should not happen in practice; see Assumption 1) —
        # proceed without the guard rather than fail shutdown outright.
        old_sigint = None

    try:
        # ... existing per-server termination loop, unchanged ...
    finally:
        if old_sigint is not None:
            signal.signal(signal.SIGINT, old_sigint)
```

Key properties:
- The guard is scoped to the body of `shutdown_all()` only — restored in `finally`, so a
  second Ctrl-C after cleanup completes behaves normally (satisfies acceptance criterion
  "ガードの解除… を確認する").
- Existing per-server timeout escalation (5s SIGTERM wait + 5s SIGKILL wait via
  `_terminate_with_timeout`) is untouched, so a genuinely hung subprocess still gets
  force-killed on its own timeout — the guard does not create a new way to hang forever
  (addresses the risk "割り込みを無視する実装が過剰になり…" from the requirement).
- `ValueError` guard on `signal.signal()` handles the off-main-thread case defensively
  without silently swallowing unrelated exceptions.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Absorbing SIGINT makes a truly hung `shutdown_all()` uninterruptible by the user | Medium | Existing 5s/5s timeout escalation in `_terminate_with_timeout` is unchanged — cleanup always progresses or force-kills within a bounded time per server; the guard only prevents *abandoning* the loop, not the loop's own timeouts |
| `signal.signal()` call fails or behaves unexpectedly if `shutdown_all()` is ever invoked off the main thread in the future | Low | Guarded with `try/except ValueError`; falls back to unguarded behavior (today's status quo) rather than raising, so it degrades safely if the calling context changes |
| New test is flaky if `signal.raise_signal` interacts oddly with pytest-asyncio's event loop | Low | Prefer invoking the currently-installed handler directly (`signal.getsignal(signal.SIGINT)()`) over `os.kill`/`signal.raise_signal`, avoiding real OS-level signal delivery in the test |
