## Goal

Make `McpServerHealthRegistry`'s state transitions (`HEALTHY`/`DEGRADED`/`UNAVAILABLE`/`HALF_OPEN`) deterministic, fully tested, and documented — specifically fixing `record_degraded()` so it cannot silently clobber `UNAVAILABLE` or `HALF_OPEN` and bypass dispatch gating.

## Scope

**In-Scope:**
- `scripts/shared/mcp_health.py`: guard `record_degraded()` against overwriting `UNAVAILABLE`/`HALF_OPEN`; expand docstrings for `record_degraded()`, `record_success()`, `is_unavailable()`
- `tests/test_mcp_health_degraded.py`: add direct unit tests for every state transition listed in the requirement's Tasks, including the new guard behavior
- `docs/04_mcp_03_03_transport-and-health.md`: update the `record_degraded()` method-table row and state-transition diagram to reflect the new guard and document `is_unavailable()`'s state-changing side effect explicitly

**Out-of-Scope:**
- Watchdog probe parsing (`_watchdog_check_http()`'s decision logic for reachable/restart_recommended) — explicitly excluded per the requirement's Notes ("Watchdog probe parsing is handled separately")
- Any change to `record_failure()`'s threshold/cooldown constants or `ToolExecutor`/`HttpTransport` retry logic
- `docs/04_mcp_06_13_watchdog-health-reasons-scheduling-part1.md`/`-part2.md` — these document watchdog *reason strings*, not `HealthRegistry` state-machine internals; out of scope unless they turn out to describe the now-fixed `record_degraded()` behavior incorrectly (checked in Step 3 of implementation)

## Assumptions

1. `record_degraded()` is called only from `agent/repl_health.py::_watchdog_check_http()` (line 583), on the "reachable but not restart-recommended" branch — confirmed via `grep -rn "record_degraded"` across `scripts/`; no other caller exists today.
2. Dispatch gating is enforced exclusively through `is_unavailable()`, called from `shared/tool_executor.py:229` and `shared/tool_transport_invoker.py:123` — confirmed via `grep -rn "is_unavailable"`. `DEGRADED` state does **not** block dispatch; only `UNAVAILABLE` does.
3. The correct design decision for the requirement's first Task ("Decide whether `record_degraded()` may overwrite `UNAVAILABLE`") is **no** — allowing it creates a verified, real dispatch-gating bypass: a periodic watchdog probe reporting "degraded" on a server that real dispatch failures already pushed to `UNAVAILABLE` would silently reset it to `DEGRADED`, and `is_unavailable()` would then return `False`, routing tool calls to a server the circuit breaker had just opened against. This directly contradicts the requirement's own Acceptance Criteria ("Watchdog cannot unintentionally bypass dispatch gating"), so this is resolved by analysis, not a blocking unknown.
4. `HALF_OPEN` must be protected by the same guard as `UNAVAILABLE` (not just `UNAVAILABLE` as literally named in the requirement's Task list) — `HALF_OPEN` represents an in-flight trial-dispatch window (`is_unavailable()`'s side effect, consumed once by `tool_executor.py:226`/`tool_transport_invoker.py:120`); a watchdog probe landing between the `HALF_OPEN` transition and the trial dispatch attempt would otherwise silently downgrade it to `DEGRADED`, which does not gate dispatch, undermining the single-trial circuit-breaker semantics the same way. This is a generalization of Assumption 3, not a separate design question.
5. No existing test in `tests/test_mcp_health_degraded.py` or elsewhere exercises `record_failure()`'s own state transitions (`HEALTHY→DEGRADED`, `DEGRADED→UNAVAILABLE`, `HALF_OPEN→UNAVAILABLE` on failure) or `is_unavailable()`'s cooldown-elapsed `→HALF_OPEN` side effect directly against `McpServerHealthRegistry` — confirmed by reading `tests/test_mcp_health_degraded.py` in full and grepping `tests/test_tool_executor*.py` (those only assert `record_failure()` call counts via a `ToolExecutor`-level spy, not the registry's resulting state).

## Unknowns

No unknowns remain — the one design decision named in the requirement (whether `record_degraded()` may overwrite `UNAVAILABLE`) is resolved by direct analysis in Assumption 3, using the requirement's own Acceptance Criteria as the deciding evidence. No user input is needed.

## Design

### `record_degraded()` guard (scripts/shared/mcp_health.py)

```python
def record_degraded(self, server_key: str, reason: str | None = None) -> None:
    """Record a reachable-but-degraded server without triggering UNAVAILABLE.

    Does not downgrade UNAVAILABLE or HALF_OPEN: a periodic "degraded but
    reachable" watchdog probe must never silently clear the circuit-breaker
    state that dispatch gating (is_unavailable()) and the single-trial
    HALF_OPEN window depend on. Only HEALTHY/DEGRADED accept the transition.
    """
    current = self.get_state(server_key)
    if current in (McpServerHealthState.UNAVAILABLE, McpServerHealthState.HALF_OPEN):
        logger.debug(
            "Health: %r degraded probe ignored — currently %s; "
            "dispatch gating/trial window preserved",
            server_key,
            current.value,
        )
        return
    self._states[server_key] = McpServerHealthState.DEGRADED
    if reason is not None:
        self._degraded_reasons[server_key] = reason
    logger.warning(
        "Health: %r is DEGRADED (reason=%s)", server_key, reason or "unknown"
    )
```

This is the only behavior change. `record_failure()`, `record_success()`, `is_unavailable()`, and `get_state()` are unchanged in logic — only their docstrings are expanded per the requirement's Tasks (documenting `is_unavailable()`'s state-mutating side effect explicitly, since it is easy to miss that a getter-shaped method mutates state).

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Suppressing the `DEGRADED` transition entirely while `UNAVAILABLE`/`HALF_OPEN` (rather than still recording the reason for operator visibility) could hide a real, additional degradation reason from `cmd_mcp.py`'s `/mcp` status display (`commands/cmd_mcp.py:95` reads `get_state()==DEGRADED` to decide whether to show a reason) | Low | Accepted: while `UNAVAILABLE`, the server is already fully gated and the operator's actionable signal is the `UNAVAILABLE` state itself (already surfaced elsewhere); a suppressed degraded-reason string during that window is not a regression since no degraded-reason UI path exists for `UNAVAILABLE` today either. Documented explicitly in the new docstring so a future reader understands this is deliberate, not an oversight. |
| `test_is_unavailable_transitions_to_half_open_after_cooldown` could become flaky if it uses a real `time.sleep()` for the cooldown window | Low | Use `half_open_cooldown_sec=0.0` (or a monkeypatched `time.monotonic`) in the registry constructor for this test, avoiding any real wall-clock wait — consistent with how fast, deterministic unit tests are already written elsewhere in this file |
| The generalization to also guard `HALF_OPEN` (Assumption 4) goes slightly beyond the requirement's literal wording ("Decide whether `record_degraded()` may overwrite `UNAVAILABLE`") | Low | The requirement's Acceptance Criteria ("Watchdog cannot unintentionally bypass dispatch gating") applies with equal force to `HALF_OPEN`, since bypassing the trial-dispatch window has the same effective risk (an untested server treated as available); flagged explicitly in Assumption 4 rather than silently expanding scope, so this is visible to reviewers |
