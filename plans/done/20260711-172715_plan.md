## Assumptions

1. **`retry` is not a real workflow stage today.** Confirmed by direct read of `scripts/agent/workflow/workflow_engine.py::run()` (lines 118-130): the only stages ever passed to `_run_stage()` are `"plan"`, `"execute"` (via `_run_execute_with_retry()`), and `"verify"`. No code path anywhere calls `_run_stage(task, "retry", ...)` or `self._wdef.get_stage("retry")`. Retry-on-failure is instead handled entirely inside `_run_execute_with_retry()` (lines 170-210), which loops the **execute** stage itself and never touches a `retry`-labeled `StageDefinition`. `workflow_loader.py::_REQUIRED_STAGES = {"plan", "execute", "verify"}` (line 47) already excludes `retry` from required stages, so a `default.json` without it will still load validly.
2. **`exponential` backoff is accepted but has zero effect.** Confirmed by direct read of `_run_execute_with_retry()` (workflow_engine.py:194): `wait = policy.backoff_sec` is used directly with no branch on `policy.backoff` anywhere in the file (confirmed via full-file read ŌĆö no other reference to `.backoff` exists in `workflow_engine.py`). `workflow_loader.py::_SUPPORTED_BACKOFF = {"fixed", "exponential"}` (line 48) accepts both values at validation time, but the value is never read after loading except for storage in `RetryPolicy.backoff`. A workflow author setting `"backoff": "exponential"` today gets silently identical behavior to `"fixed"` ŌĆö this is the exact "config values that silently behave differently from their names" the requirement's Notes section calls out.
3. **The `plan` stage callback is a genuine, correct no-op; the `execute` stage callback does the memory-injection/mode-classification work.** Confirmed by direct read of `scripts/agent/orchestrator.py`: `plan_fn()` (line 195-196) returns `None` immediately, with a comment stating `_handle_turn_start` already ran; `execute_fn()` (line 198-212) calls `self._process_turn(line, ctx, turn_started_at)`, and `_process_turn()` (lines 423-449) is confirmed to call `_handle_memory_injection()` and `classify_and_inject_mode()` at lines 436-437 ŌĆö both inside the execute-stage callback. `default.json`'s current `plan` stage `description` ("Prepare RAG context and memory injection") is therefore factually wrong about which stage does this work; `docs/05_agent_03_03_...part1.md:59` ("`plan` ŌĆö LLMŃüīÕłØµ£¤Ķ©łńö╗ŃéÆńö¤µłÉ; Õ┐ģķĀł") is independently wrong in a different way (no LLM call happens in the plan stage at all).
4. Per the requirement's own "Notes for AI Implementer" ("Prefer removing unused configuration surface unless a real runtime feature is required. Avoid accepting config values that silently behave differently from their names."), and consistent with this session's established pattern of resolving genuinely-analyzable design choices without a user prompt (reserving `AskUserQuestion` for premise-level conflicts as in requirement 26): this plan selects **Option A (remove `retry` as a stage)** for the retry-stage mismatch and **Option B (remove `exponential`)** for the backoff mismatch. Both directly implement dead/no-op-code weight instead of adding new runtime behavior (real exponential backoff, a real retry-stage callback) that no current caller needs ŌĆö `default.json` already uses `"fixed"` exclusively, and no code anywhere constructs a workflow JSON with `"backoff": "exponential"` (confirmed via `grep -r "exponential" config/ scripts/`).
5. `config/workflows/default.json` is the only workflow definition file in the repository (confirmed via `ls config/workflows/`), so both edits (remove `retry` stage entry, keep `retry_policy.backoff: "fixed"`) affect a single file with no other workflow JSON to keep in sync.

## Design

### 1. `config/workflows/default.json`

Remove the `retry` stage entry from `stages`. Correct `plan` and `execute` descriptions:

```json
{
  "name": "default",
  "version": "1.0.0",
  "stages": [
    {
      "id": "plan",
      "description": "Idempotency/bookkeeping stage only; no LLM call. Turn-start audit logging happens before the engine runs; memory injection and mode classification happen in the execute stage.",
      "timeout_sec": 30,
      "retryable": false
    },
    {
      "id": "execute",
      "description": "Memory injection, mode classification, LLM call, and tool execution loop",
      "timeout_sec": 120,
      "retryable": true
    },
    {
      "id": "verify",
      "description": "Result validation and loop guard check",
      "timeout_sec": 10,
      "retryable": false
    }
  ],
  "retry_policy": {
    "max_attempts": 3,
    "backoff": "fixed",
    "backoff_sec": 1
  }
}
```

Bumping `version` is not required ŌĆö `WorkflowLoadError`/schema shape is unaffected (removing an already-non-required stage, and a value from an already-existing enum, is backward compatible for any loader that has always accepted `_REQUIRED_STAGES = {"plan", "execute", "verify"}"`). Confirm at implementation time whether `02_deployment-part1.md`'s deployment checklist or schema-version tracking (added by this session's earlier workflow-deployment-hardening work, commit `29562c46`) treats a `stages` array shrink as requiring a version bump; if so, bump `version` to `"1.0.1"` and note it in that doc.
