## Goal
Redesign the MDQ SQLite schema for production indexing with stable chunk IDs, document metadata, and FTS5 synchronization.

## Scope
**In-Scope**:
- Create `documents` table with source_path, mtime_ns, size_bytes, content_hash, indexed_at
- Create `chunks` table with chunk_id, doc_id, heading fields, content, line ranges, char_count, token_count, content_hash, tags_json, indexed_at
- Create `chunks_fts` FTS5 table synchronized with chunks
- Add migration path from existing `sections`/`sections_fts` schema
- Update `_init_db()` to create new tables and handle migration

**Out-of-Scope**:
- Adding new tools or features
- Token counting implementation (placeholder for now)

## Assumptions
1. Migration should preserve existing data when possible
2. `token_count` will be computed by the agent at indexing time (not in DB)
3. `tags_json` stores a JSON array of tags as TEXT column

- **Blast Radius**: High — changes the database schema. All indexing and querying code must be updated.

## Implementation Steps
1. **Phase 1: Define new tables**
   - Create documents table with doc_id (TEXT, PK), source_path, mtime_ns, size_bytes, content_hash, indexed_at
   - Create chunks table with chunk_id (TEXT, PK), doc_id (FK), heading fields, content, line ranges, char_count, token_count, content_hash, tags_json, indexed_at
   - Create chunks_fts FTS5 table
2. **Phase 2: Add migration logic**
   - Check if sections table exists (old schema)
   - If yes, migrate data to new schema
   - Drop old tables after migration
3. **Phase 3: Update indexing code**
   - Update indexer.py to use documents/chunks tables
   - Compute content_hash for both documents and chunks
4. **Phase 4: Update stats endpoint**
   - Update stats() to report document, chunk, FTS, and stale counts
