# Plan: Prevent /reload from mutating ctx.cfg.mcp.mcp_servers

Source requirement: `requires/20260708_24_require.md` (H-2)

## Goal

Guarantee that `/reload` never mutates `ctx.cfg.mcp.mcp_servers` (the dict or
any `McpServerConfig` object inside it), and add exhaustive per-field tests
proving it, so watchdog, lifecycle, and `/config` continue to observe the
startup-time MCP snapshot after every reload.

## Scope

In scope:
- `scripts/agent/services/config_reload.py`: same production change as
  `plans/20260709-094640_plan.md` (H-1) — this requirement is the
  mutation-safety half of that same change.
- `tests/test_config_reload.py` (or a new dedicated test module): add a
  parametrized test asserting all 11 `McpServerConfig` fields listed in the
  requirement are byte-for-byte unchanged after `/reload`.

Out of scope:
- `scripts/agent/services/mcp_status.py`, `scripts/agent/repl_health.py`,
  `scripts/agent/commands/cmd_mcp.py` — these already read
  `ctx.cfg.mcp.mcp_servers` directly (verified below); no change needed once
  mutation stops.
- `scripts/shared/mcp_config.py` (comparison source, unmodified).

## Assumptions

1. This requirement and H-1 (`requires/20260708_23_require.md`, already
   planned in `plans/20260709-094640_plan.md`) target the same function and
   are satisfied by the same production code change. H-1's design
   (`_classify_mcp_server_changes()`, whole-object `!=` comparison, zero
   mutation) already meets every acceptance criterion listed here.
2. No code outside `config_reload.py` writes to `ctx.cfg.mcp.mcp_servers` —
   verified via `grep -rn "mcp_servers\[.*\]\s*=\|mcp_servers\.\(pop\|update\)"
   scripts/` (no matches besides validation code in `mcp_config.py` that
   operates on raw TOML dicts, not the live config object).
3. Watchdog (`repl_health.py`), MCP status (`mcp_status.py`), and `/config`
   display (`cmd_mcp.py`) all read `ctx.cfg.mcp.mcp_servers` directly
   (`ctx.cfg.mcp.mcp_servers.items()` / membership checks) — verified by
   `grep -n "mcp_servers" scripts/agent/repl_health.py
   scripts/agent/services/mcp_status.py scripts/agent/commands/cmd_mcp.py`.
   Once mutation stops, they automatically see the unchanged startup snapshot;
   no code change is required in these files.

## Design

No new production design beyond H-1's `_classify_mcp_server_changes()`
(see `plans/20260709-094640_plan.md` § Design, revised while planning H-3 to
emit field-qualified entries such as `mcp/<key>.url` instead of a bare key).
This plan adds the specific test the requirement calls for:

```python
import copy

import pytest


@pytest.mark.parametrize(
    "field_name,new_value",
    [
        ("transport", TransportType.STDIO),
        ("url", "http://changed:9999"),
        ("startup_mode", StartupMode.SUBPROCESS),
        ("healthcheck_mode", HealthcheckMode.NONE),
        ("call_timeout_sec", 999.0),
        ("startup_timeout_sec", 999),
        ("tool_names", ["changed_tool"]),
        ("auth_token", "changed_token"),
        ("role", "changed_role"),
        ("cmd", ["changed", "cmd"]),
        ("env", {"CHANGED": "1"}),
    ],
)
def test_reload_never_mutates_mcp_server_field(field_name, new_value):
    old_srv = McpServerConfig(transport=TransportType.HTTP, url="http://localhost:8080", cmd=[])
    before = copy.deepcopy(old_srv)
    new_srv = copy.deepcopy(old_srv)
    setattr(new_srv, field_name, new_value)

    svc, ctx = _make_svc_with(old_srv)
    with patch("agent.config_builders._build_mcp_servers", return_value={"svc": new_srv}):
        outcome = svc._classify_mcp_server_changes(ctx, {})

    assert old_srv == before  # unchanged in place
    assert f"mcp/svc.{field_name}" in outcome.needs_restart
    assert outcome.applied == []
    assert outcome.deferred == []
```

## Implementation steps

1. Implement `_classify_mcp_server_changes()` exactly as designed in the H-1

   before editing.
2. Add the parametrized test above (or equivalent) to `tests/test_config_reload.py`,

3. Add one integration-style test that reloads with several fields changed at

   the original `McpServerConfig` object is unchanged and the key appears
   exactly once in `needs_restart`.
4. Run the full validation sequence (`rules/toolchain.md`) and fix any findings.

## Risks

- **Risk**: This plan is a near-duplicate of the H-1 plan; implementing both
  independently could produce two slightly different versions of the same
  function if worked on out of order.
  → **Mitigation**: Step 1 explicitly checks whether H-1's change already
  landed before touching production code again; this plan's unique
  contribution is the exhaustive field-level test suite, not a second
  implementation.
- **Risk**: The parametrized test's per-field mutation (e.g. changing only
  `role`) will trigger `needs_restart` under the H-1 design's whole-object
  `!=` comparison, which is broader than the three fields the original code
  checked. This is intentional (see H-1 plan risks) but should be called out
  again here since this requirement's test plan is what will surface it.
  → **Mitigation**: Assert `needs_restart` inclusion for every field, not just
  the three legacy ones — already reflected in the Design test above.
