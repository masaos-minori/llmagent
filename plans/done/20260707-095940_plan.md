## Goal

Remove the `workflow_mode="disabled"` direct execution code path so that every user turn unconditionally goes through the WorkflowEngine with task creation, stage tracking, idempotency, artifacts, approvals, and audit correlation.

## Scope

**In**:
- Remove `if not self._policy.is_workflow_enabled()` branch from `Orchestrator.handle_turn()` in `scripts/agent/orchestrator.py` (line 195–196)
- Remove the fallback direct-call block (`await self._process_turn(...)`, `await self._handle_turn_end(...)`) that runs when workflow is disabled or unavailable (lines 203–208)
- Remove `is_workflow_enabled()` from `WorkflowExecutionPolicy` (class deletion covered by plan 20260707-095938)
- Remove `logger.info("Workflow mode=disabled — direct execution")` log line
- Ensure "workflow unavailable" path fails fast via `_log_fallback()` → `RuntimeError` instead of falling through to direct execution
- Remove direct-execution tests in `tests/test_orchestrator.py`
- Add tests asserting: workflow unavailable → no LLM call, no tool call

**Out**:
- `WorkflowExecutionPolicy` class deletion (plan 20260707-095938)
- `workflow_mode` config field removal (plan 20260707-095938)
- Startup fallback removal (plan 20260707-095941)
- Approval changes (plan 20260707-095942)

**Note**: This plan (req 03) covers the same code site as plan 20260707-095939 (req 02). Execute these together; req 03 is the authoritative, narrower statement.

## Assumptions

1. `_handle_workflow_engine()` is the only valid turn execution path after this change.
2. `_log_fallback()` already raises unconditionally; no further change needed to it.
3. `_process_turn()` continues to be called from within stage callbacks inside `_handle_workflow_engine()`; only the top-level direct call is removed.
4. Static scan CI check (req 14) can verify absence of disabled-mode patterns post-change.
