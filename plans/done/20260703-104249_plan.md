# Plan: Clarify plugin registry repeated-load behavior and test isolation

**Source requirement:** `requires/20260703_07_require.md`

---

## Goal

Enforce and document safe plugin registry usage by adding docstrings clarifying single-load semantics, requiring `_reset_for_testing()` in all tests that manipulate the plugin registry, adding missing isolation tests, and confirming no production code calls `_reset_for_testing()`.

---

## Scope

### In-Scope

- Add/update docstring in `plugin_registry.py` to clarify: `load_plugins()` is for startup use only; repeated calls accumulate registrations; `_reset_for_testing()` is the only supported way to clear registries in tests; non-test code must not call `_reset_for_testing()`.
- Update `tests/test_plugin_registry.py` to verify that the existing `autouse` reset fixture adequately isolates all test classes (no-op if already correct, but verify explicitly).
- Fix `tests/test_tool_executor_routing.py` to use a `pytest.fixture` with autouse or explicit per-class setup instead of ad-hoc inline `_reset_for_testing()` calls around individual test methods.
- Fix `tests/integration/test_agent_mcp_integration.py` test `test_a11_plugin_tool_error_does_not_propagate` to use `_reset_for_testing()` instead of directly mutating the private `_tools` dict, and ensure cleanup uses `_reset_for_testing()`.
- Add one or two tests that explicitly verify registry state does not leak across multiple sequential plugin loads within the same test process.
- Update `docs/05_agent_11_extension-points.md` and `docs/90_shared_03_runtime_and_execution.md` to document `_reset_for_testing()` restriction.
- Confirm no production path (scripts, not tests) calls `_reset_for_testing()`.

### Out-of-Scope

- Implementing plugin hot reload
- Changing production plugin loading semantics
- Changing plugin conflict policy
- Changing the signature or behavior of `load_plugins()` or `_reset_for_testing()`

## Assumptions

1. `test_plugin_registry.py` already has a correct `autouse` fixture (`reset_registry`) that calls `_reset_for_testing()` before and after each test — confirmed by reading the file. No behavior change is needed there; only documentation alignment may be required.

4. The docstring in `plugin_registry.py` module header (lines 36–51) already contains most of the required lifecycle documentation; the task requires verifying completeness and potentially expanding the `_reset_for_testing()` function docstring.
5. No production code (in `scripts/`) calls `_reset_for_testing()` — confirmed by grep; this only needs to be documented as a verification step.
6. `deploy/deploy.sh` does not reference test files or docs and will not be affected.

## Implementation Steps

### Step 1 — Verify and tighten `plugin_registry.py` docstrings

**Files:** `scripts/shared/plugin_registry.py`

1. Confirm module-level docstring (lines 36–51) contains all four points from the requirement:
   - `load_plugins()` is for startup use
   - Repeated calls accumulate registrations
   - `_reset_for_testing()` is the only supported way to clear registries in tests
   - Non-test code must not call `_reset_for_testing()`
2. Expand the `_reset_for_testing()` function docstring (currently one line) to:
   ```
   Clear all registries and reset module-level state.

   For test use only. Do not call from production code.
   Call at the start (and optionally end) of each test that loads plugins
   or registers commands/tools directly. This is the only supported way
   to clear global registry state between tests.
   ```
3. Add an inline comment above `_commands`, `_tools`, and `_pipeline_post` declarations if the existing comments do not already reference `_reset_for_testing()` as the clearing mechanism (they currently do).

**Completion criteria:**
- All four required documentation points appear verbatim or in substance in the module docstring.
- `_reset_for_testing()` has a multi-line docstring that explicitly forbids non-test use.
- No behavior change; tests continue to pass.

### Step 5 — Update `docs/05_agent_11_extension-points.md`

**Files:** `docs/05_agent_11_extension-points.md`

1. Add a "Test Isolation" subsection after the "Registry API" table:

   ```markdown
   ### Test Isolation

   `_reset_for_testing()` is the **only** supported way to clear global registry state.

   Rules:
   - Tests that call `load_plugins()` or any `@register_*` decorator **must** call
     `_reset_for_testing()` in a `pytest.fixture(autouse=True)` before (and optionally after)
     each test function.
   - Production code (non-test modules) must **never** call `_reset_for_testing()`.
   - Direct mutation of `_commands`, `_tools`, or `_pipeline_post` is also forbidden in tests;
     use `_reset_for_testing()` + public decorators instead.

   Example:
   ```python
   import pytest
   import shared.plugin_registry as plugin_registry

   @pytest.fixture(autouse=True)
   def reset_registry():
       plugin_registry._reset_for_testing()
       yield
       plugin_registry._reset_for_testing()
   ```
   ```

**Completion criteria:**
- "Test Isolation" subsection present under "Registry API".
- The four documentation points from the requirement are all covered.
