# Plan: Resolve SemanticCache Eviction-Policy Inconsistency (req_34)

## 1. Goal

Align `SemanticCache` documentation to the actual FIFO eviction implementation;
eliminate all "LRU" references in SemanticCache-specific docs and the class
docstring, and expose the policy clearly to operators.

## 2. Scope

**In-Scope:**
- Confirm actual eviction policy from code (FIFO — confirmed: see §3)
- Update `SemanticCache` class docstring to explicitly state FIFO
- Fix `docs/03_rag_01_system_overview.md:115-116` ("LRU cache" → "FIFO cache")
- Fix `docs/05_agent_08_configuration.md:168` ("Max LRU cache entries" → "Max FIFO cache entries")
- Add a one-line note explaining FIFO semantics to the config table
  ("oldest entries pruned first on overflow")
- Verify `docs/03_rag_03_query_pipeline.md:216` already says FIFO (no change needed)

**Out-of-Scope:**
- Changing eviction policy to LRU (requires `OrderedDict` or `list` mutation on hit — unnecessary complexity)
- Changing cache hit semantics
- Replacing `SemanticCache` architecture

## 3. Confirmed Facts

**Implementation (FIFO — confirmed):**
- `self._entries: list[CacheEntry]` — insertion order, `append()` to tail
- `put()` calls `self.prune()` after append
- `prune()`: `self._entries = self._entries[-self._max_size:]`
  — drops the **oldest** (head) entries: this is FIFO eviction
- `lookup()`: iterates `_entries` but does **not** move hit entries to tail
  — access does not update recency → NOT LRU

**Documentation inconsistency (confirmed):**
- `docs/03_rag_01_system_overview.md:115-116`: "LRU cache" (WRONG)
- `docs/05_agent_08_configuration.md:168`: "Max LRU cache entries" (WRONG)
- `docs/03_rag_03_query_pipeline.md:216`: "Remove oldest entries (FIFO)" (CORRECT)

**`tool_cache` / MCP cache references are separate caches — unrelated to SemanticCache:**
- `docs/04_mcp_03_routing_lifecycle_and_execution.md:19`: uses `functools.lru_cache` or TTL+LRU — legitimately LRU
- These must NOT be changed

## 4. Assumptions

1. Keeping FIFO is intentional: SemanticCache is a simple embedding cache
   where recency-weighting would add complexity without proven benefit.
   FIFO is correct and simpler.
2. The `tool_cache` and MCP route cache LRU references are accurate —
   those are separate caches and must not be touched.
3. `semantic_cache_max_size` config key itself does not need renaming
   — the parameter name is neutral; only the description changes.
