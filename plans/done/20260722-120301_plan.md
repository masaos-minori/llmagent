## Goal
Distinguish between "never seen" and "unknown state" in LifecycleManager to prevent false state transition validation failures.

## Scope
**In**: `scripts/agent/factory.py` — refactor `_set_state()` method
**Out**: Changes to state machine definition or other lifecycle methods

## Assumptions
1. A missing key means "never existed," while UNKNOWN means "exists but state is unknown"
2. First-time servers should be allowed to initialize to any valid starting state

## Unknowns
1. ~~Is there a specific scenario where UNKNOWN → STARTING fails?~~ Resolved: yes, if the state machine doesn't explicitly allow UNKNOWN → STARTING, the assertion fails and the server remains uninitialized.

## Affected areas
- Server lifecycle management
- State machine transitions
- Initialization logic

## Design
Add explicit check for `server_key not in self._states` before accessing. Treat missing keys as "first time seeing this server" and allow any valid initial state. Keep existing validation for known servers.

## Implementation steps
1. Replace `.get(server_key, LifecycleState.UNKNOWN)` with explicit key existence check
2. Add info-level logging when initializing a new server's state
3. Run tests to verify no regressions
