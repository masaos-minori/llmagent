## Goal

Wrap the `StateStore` usage in `startup.py:_recover_pending_approvals()` with `try/finally` to guarantee `store.close()` is called even when `find_latest_pending_approval()` raises an exception, bringing this method in line with the defensive pattern used throughout `orchestrator.py`.

---

## Scope

**In-Scope**
- `scripts/agent/startup.py`: method `_recover_pending_approvals()` (lines 173–196)
- Add `try/finally` block around `store.find_latest_pending_approval()` call only
- Update test in `tests/test_startup.py` if needed to verify `store.close()` is called on exception path

**Out-of-Scope**
- `scripts/agent/workflow/state_store.py` — no API changes
- `scripts/agent/orchestrator.py` — already has correct pattern; no changes needed
- `scripts/agent/commands/cmd_workflow.py` — also calls `find_latest_pending_approval()` but is not part of this requirement (lines 44, 76)
- Exception handling / catch clauses — only `finally` is added; exceptions propagate unchanged
- DB schema changes

---

## Assumptions

1. `result` returned by `find_latest_pending_approval()` is a plain Python data object (`tuple[str, ApprovalRecord] | None`) with no open DB dependency, so it is safe to use after `store.close()` is called. (Confirmed in `state_store.py:328` — returns data extracted from SQLite rows.)
2. The `StateStore.__init__` opens a SQLite connection immediately (`state_store.py:28-29`). A partially-constructed instance on `__init__` failure is not possible in the normal error path; however `store.close()` in `finally` is safe to call even if the connection was never opened — `SQLiteHelper.close()` is idempotent.
3. Exception propagation to `StartupOrchestrator.run()` is the correct behavior. The require file explicitly confirms: "If this method raises, startup will abort — this is the correct behavior."
