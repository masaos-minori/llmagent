# Plan: [High][eventbus][db] スレッドプール実行に向けた SQLite 接続所有権の再設計

Source: `requires/20260625_16_require.md`

## Goal

スレッドプール実行を前提とした明示的・安全な SQLite 接続モデルを定義し、既存の「シングルスレッド・イベントループ」前提を廃止する。

## Scope

**In-Scope**
- `scripts/eventbus/db.py` の接続モデル変更
- `scripts/eventbus/app.py` のライフサイクル初期化・クリーンアップ更新
- 選択した接続モデルのドキュメント化 (docstring)
- 並列 publish/subscribe アクセスのテスト

**Out-of-Scope**
- DB スキーマ変更
- ファサード関数の追加 (req #15 の対象)
- ファイル I/O のオフロード (req #18 の対象)

## Assumptions

1. Python 3.13 / SQLite WAL モードを使用
2. 3つの選択肢 (per-call / thread-confined / shared+lock) から1つを選択する
3. 選択は実装容易性・テスト容易性・パフォーマンスのバランスで判断する

## Design

**選択肢の比較:**

| 案 | 説明 | メリット | デメリット |
|---|---|---|---|
| A: per-call 接続 | 各 DB 操作で `sqlite3.connect()` を開いて閉じる | 接続競合ゼロ、最もシンプル | 接続オーバーヘッド、WAL checkpoint が分散 |
| B: thread-confined 接続 | `threading.local()` で スレッドごとに接続を管理 | per-call より高速、スレッドセーフ | ライフサイクル管理が複雑 |
| C: shared + explicit lock | 現行の shared connection に `threading.Lock` を追加 | 実装変更が最小 | ロック競合によるスループット低下リスク |

**採用: 案 C (shared connection + explicit Lock)**

ユーザー決定。変更を最小限に抑えつつ、スレッドセーフを明示的に確保する。

実装イメージ:
```python
# db.py
import threading

_db_lock = threading.Lock()

def get_db_lock() -> threading.Lock:
    """Return the global lock that must be held for all DB operations."""
    return _db_lock
```

各スレッドオフロード関数内でロックを取得してから DB 操作を行う:
```python
# app.py (offload wrapper 内)
def _do_insert():
    with get_db_lock():
        cur = _db.execute("INSERT ...", (...))
        _db.commit()
        return cur
cur = await asyncio.to_thread(_do_insert)
```

既存の `_db` グローバル変数と `open_db()` は維持し、`check_same_thread=False` の justification を「Lock によりシリアライズ済み」に更新する。

## Implementation Steps

1. `db.py` に `_db_lock = threading.Lock()` と `get_db_lock()` を追加
2. `db.py` の `open_db()` の docstring を「Lock によりシリアライズ済み」に更新
3. `app.py` の各 DB 操作をラップする関数を `get_db_lock()` で保護
4. req #14 のスレッドオフロードと統合 (同一コミットまたは直後)
5. 並列 publish/subscribe アクセステストを追加
6. テスト通過確認

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| ロック競合でスループットが低下する | Medium | スレッドプールの同時実行数が少ない (FastAPI デフォルト = CPU*5) ため実用上問題は少ないが、high-load テスト (req #17) で検証する |
| ロック取得漏れで未保護の DB 呼び出しが残る | Medium | req #14 完了後に `rg "_db\.execute\|_db\.commit"` でグローバル _db への直接アクセスがないことを確認 |
