# Plan: Enforce MDQ / RAG Data Ownership Boundaries (20260620_03)

## Goal

## Scope
- Add a validation script (grep-based or Python-based).
- Update CI/toolchain to run the check.
- Document allowed vs forbidden access paths.
- No code changes to existing layers.

## Task Size
Small-Medium (Path A/B boundary): new script + CI integration + docs. 2-3 new/modified files.

## Design

### Validation script: `scripts/tests/test_mdq_rag_boundary.py`
A lightweight test that:
1. Scans `mcp/mdq/` source files for strings containing `rag.sqlite` → should find zero matches.
2. Scans `mcp/rag_pipeline/` source files for strings containing `mdq.sqlite` → should find zero matches.
3. Scans `agent/` source files for `SQLiteHelper("rag")` or `SQLiteHelper("mdq")` or `sqlite3.connect` with MDQ/RAG paths → should only find matches in `db_maintenance_service.py`.

### CI integration
Add to the lint/validation step (see `rules/toolchain.md` for existing toolchain).

### Documentation
- Add a "Boundary Enforcement" section to `docs/04_mcp-mdq.md` describing:
  - The automated check (script name, what it verifies).
  - Allowed vs forbidden access paths (table).
  - How to handle false positives (needs confirmation process).

## Risks
- **Low**: Comments/docstrings may contain references to the other DB name (e.g., "not rag.sqlite"). The script should scan only string literals/code, not comments, or use a whitelist approach for known safe references.
- **Medium**: Future admin maintenance tools might need direct DB access. The allowed-paths table must be easy to update, and new paths require a "needs confirmation" review.
