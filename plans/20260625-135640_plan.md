# Plan: [High][eventbus][architecture] DB アクセスファサードの導入

Source: `requires/20260625_15_require.md`

## Goal

`scripts/eventbus/db.py` に名前付き DB 関数を追加し、`app.py` のルートハンドラから生 SQL 呼び出しを排除することで、SQL ロジックを一箇所に集約する。

## Scope

**In-Scope**
- `scripts/eventbus/db.py` への DB ファサード関数追加
- `scripts/eventbus/app.py` のルートハンドラから生 SQL を除去し、ファサード関数に置換
- スレッドオフロードポリシーをファサード内に集約 (req #14 のオフロードと統合)

**Out-of-Scope**
- SQLite 接続戦略の変更 (req #16 の対象)
- DB スキーマ変更
- テスト追加 (req #17 の対象)
- ファイル I/O のオフロード (req #18 の対象)

## Assumptions

1. `app.py` の全 SQL 呼び出しは `db.py` 内の関数として命名できる
2. `dlq.py` の `promote_to_dlq()` は既に分離されているため対象外
3. `db.py` のファサード関数はコネクションを引数で受け取る (グローバル状態に依存しない設計)
4. req #14 と同時または直後に実施することを想定し、ファサード関数内にオフロードラッパーを持たせる

## Unknowns

| `scripts/eventbus/app.py` | 生 SQL 呼び出しをファサード関数に置換 | 中 (全ルートハンドラ) |

## Design

`db.py` に追加する関数群:

```python
def insert_event(conn, event_id, topic, payload_str, producer, published_at) -> tuple[int, bool]:
    """INSERT OR IGNORE。(rowid, inserted) を返す"""

def get_seq(conn, event_id) -> int:
    """event_id から seq を取得"""

def fetch_events_since(conn, since_seq, topics=None) -> list[sqlite3.Row]:
    """seq > since_seq の events を返す"""

def fetch_dlq(conn) -> list[sqlite3.Row]:
    """dlq_at IS NOT NULL の events を返す"""

def requeue_event(conn, event_id) -> bool:
    """retry_count + 1, dlq_at = NULL。rowcount > 0 なら True"""

def check_db(conn) -> bool:
    """SELECT 1 の成否を返す"""
```

## Implementation Steps

1. `db.py` に `check_db()` 関数を追加 → `health()` で使用
2. `db.py` に `insert_event()` / `get_seq()` を追加 → `publish()` で使用
3. `db.py` に `fetch_events_since()` を追加 → `replay()` / `subscribe()` で使用
4. `db.py` に `fetch_dlq()` / `requeue_event()` を追加 → `dlq_list()` / `dlq_requeue()` で使用
5. `app.py` の各ハンドラを順に置換、各ステップでテスト通過確認
