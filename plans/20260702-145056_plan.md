# Plan: Align DocumentStore.chunk_insert() with chunk metadata schema

**Timestamp:** 20260702-145056
**Requirement:** requires/20260702_43_require.md
**Decision: Option A — Extend the protocol and implementation**

---

## 1. Goal

Extend `DocumentStore.chunk_insert()` and `SQLiteDocumentStore.chunk_insert()` to accept
`chunk_type` and `source_file`, eliminating the gap between the DB store abstraction and
the canonical `chunks` schema.

---

## 2. Scope

**In-Scope:**
- `scripts/db/store_protocols.py` — add `chunk_type: str = ""` and `source_file: str = ""` to `DocumentStore.chunk_insert()` signature
- `scripts/db/store_impl.py` — add the same parameters to `SQLiteDocumentStore.chunk_insert()` and update the INSERT SQL
- `docs/90_shared_05_db_api_and_operations.md` — update `DocumentStore` protocol reference table
- Tests for `SQLiteDocumentStore.chunk_insert()` — add/update test cases to verify the new parameters are stored

**Out-of-Scope:**
- DB schema migrations (columns already exist via `_migrate_rag_schema()` in `create_schema.py`)
- Backward-compatible API shims
- Changing `RagIngester._insert_chunk()` call path (it already passes these fields correctly via direct SQL)
- Any change to `RagIngester` call sites (no caller currently uses `DocumentStore.chunk_insert()` in the RAG layer)

---

## 3. Assumptions

1. `chunk_type` and `source_file` columns already exist in `chunks` table in production, confirmed by `_migrate_rag_schema()` in `create_schema.py` (lines 33-39).
2. No code outside `db/` currently calls `DocumentStore.chunk_insert()` — confirmed by grep: zero hits in `scripts/mcp/`, `scripts/agent/`, `scripts/rag/`.
3. `RagIngester._insert_chunk()` bypasses `DocumentStore` intentionally (uses `SQLiteHelper.execute()` directly); this is an acceptable dual-path today but will become redundant after Option A.
4. Default values of `""` (empty string) for both new parameters preserve backward compatibility for any future callers.

## Rationale for Option A over Option B

**Option A is selected** because:
- The schema already has `chunk_type` and `source_file` (confirmed in `create_schema.py`).
- No external callers use `chunk_insert()` today, so the signature change has zero blast radius.
- The `DocumentStore` protocol exists to be the stable abstraction; leaving it inconsistent with the schema is a doc debt that would mislead future implementers.
- Option B (document bypass) would lock in the dual-path indefinitely and make `DocumentStore` a misleading abstraction — acceptable only if the schema fields were RAG-specific and never needed via the protocol.

Option B would be correct **only if** chunk_type/source_file were RAG-pipeline-exclusive and no future caller (e.g., a test fixture, MCP sqlite tool, or admin script) would ever insert chunks via `DocumentStore`. That is not the case here.
