## Assumptions

1. `validate_routing_against_config()` は設定ファイルのみを参照し、ネットワーク通信不要で startup 時に常に実行可能。
2. config の `tool_names` は optional (空リストが許容値) — drift 検出の警告は空リストのサーバーはスキップする。
3. `_log_routing_coverage()` は `self._registry` を持っているため、`_lookup_registry()` を再利用すれば registry チェックを追加できる。
4. drift 検出は warning-only (RuntimeError にしない) — `tool_definitions_strict` モードとは独立した概念であり、routing drift はオペレーターへの通知が目的。
5. `ToolRegistry` singleton が初期化失敗した場合は `repl_health` 側でも graceful fallback (警告ログのみ)。

## Design

### 1. `check_routing_drift()` (repl_health.py 追加)

```python
def check_routing_drift(ctx: AgentContext) -> list[str]:
    """Check config tool_names against ToolRegistry. Returns list of warning messages."""
    from shared.tool_registry import validate_routing_against_config
    try:
        server_configs = ctx.cfg.mcp.mcp_servers
        drift = validate_routing_against_config(server_configs=server_configs)
        warnings: list[str] = []
        for server_key, messages in drift.items():
            for msg in messages:
                full_msg = f"Routing drift [{server_key}]: {msg}"
                logger.warning(full_msg)
                warnings.append(full_msg)
        return warnings
    except Exception as exc:
        logger.warning("Routing drift check failed: %s", exc)
        return []
```

### 3. `_log_routing_coverage()` の修正

現在の実装はディスカバリーマップとコンフィグマップのみチェックし、registry (Priority 2) をスキップしている。

修正前:
```python
if tool_name in self._discovery_map or tool_name in self._config_map:
    mapped.append(tool_name)
    continue
try:
    self._fallback_route(tool_name)
    mapped.append(tool_name)
except ValueError:
    unmapped.append(tool_name)
```

修正後:
```python
if tool_name in self._discovery_map or tool_name in self._config_map:
    mapped.append(tool_name)
    continue
if self._lookup_registry(tool_name) is not None:
    mapped.append(tool_name)
    continue
try:
    self._fallback_route(tool_name)
    mapped.append(tool_name)
except ValueError:
    unmapped.append(tool_name)
```

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| `validate_routing_against_config()` で既存の drift warning が大量に出て startup テストが壊れる | Med | **実際の drift が存在する (確認済み):** `file_read` の config tool_names は 5 ツールだが registry は 9 ツール (`list_directory_with_sizes`, `read_media_file`, `read_multiple_files`, `get_file_info` が未記載)。初回 startup でこれらの warning が出る。**緩和:** `check_routing_drift()` は warning-only で RuntimeError を raise しない。テストは `MagicMock` で config を制御するため実 config の drift は影響しない。実 config の drift は別タスクで修正。 |
| `_log_routing_coverage()` 修正で registry チェック失敗時に warning が変わる | Low | `_lookup_registry()` は既存メソッドで None を返すだけ。例外を throw しない。 |
| startup フローに同期関数 `check_routing_drift()` を追加することで型エラーが出る | Low | `_check_services()` は sync コンテキストで呼ばれるため、非同期呼び出しは不要。`validate_routing_against_config()` は sync 関数。 |
