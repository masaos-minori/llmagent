## Goal

Clean up stale `_http_procs` entries after health check timeouts and other failure paths where `_terminate_with_timeout()` is called without subsequent dictionary cleanup.

## Scope

**In:**
- `scripts/agent/http_lifecycle.py`: Add cleanup of `_http_procs[server_key]` after health check timeout

**Out:**
- Modifying the health check logic itself
- Changing subprocess termination behavior
- Any other file modifications

## Assumptions

1. The `_terminate_with_timeout()` method terminates and kills the process but does NOT clean up `_http_procs` — confirmed by code review at line 111-147.
2. After `_terminate_with_timeout()`, the process is guaranteed to be dead (poll() returns non-None), so keeping a reference is useless.
3. The early exit path (line 263-280) already handles cleanup correctly — no change needed there because it raises before reaching the timeout path.
4. The restart path (line 322) already pops the entry before calling `_terminate_with_timeout()` — no change needed.
5. The shutdown_all path (line 368) already pops the entry before calling `_terminate_with_timeout()` — no change needed.

## Unknowns

Resolved: No unresolved unknowns remain.

- **UNK-01** (resolved): Whether `_terminate_with_timeout()` should be modified to also clean up `_http_procs`. Analysis shows that `_terminate_with_timeout()` is a generic method used in multiple contexts; adding dictionary-specific cleanup would couple it to the caller's internal state. Better to add cleanup at each call site.
- **UNK-02**: Whether the cleanup should happen before or after raising `HttpStartupError`. Since the error is raised immediately after, order doesn't matter functionally. Placing cleanup before the error makes the intent clearer.

## Design decisions

- Add `self._http_procs.pop(server_key, None)` directly after `_terminate_with_timeout()` on line 300, before raising `HttpStartupError`.
- Do NOT extract into a helper method — the cleanup is a single-line operation and extracting it would add unnecessary indirection for such a small scope.
- Also clean up `_http_pgids` since the process group ID is now meaningless for a terminated process.

## Alternatives considered

- Extract cleanup into `_cleanup_server_entry(server_key)` method: would reduce duplication if more call sites are added later, but adds complexity for a single fix.
- Modify `_terminate_with_timeout()` to accept an optional cleanup flag: couples the method to the caller's internal state, violates separation of concerns.
- Rely on eventual cleanup via `verify_running()` or `shutdown_all()`: leaves the stale entry in place until those methods are called, which could be indefinitely delayed.

## Risks & Mitigations

- **Risk**: Cleanup removes an entry that was re-added between `_terminate_with_timeout()` and the pop call → **Mitigation**: This race condition is impossible because the exception is raised immediately after, preventing any concurrent operations.
- **Risk**: `_http_pgids` cleanup removes data needed elsewhere → **Mitigation**: The process group ID is only useful while the process is alive; after termination, it has no meaning.
