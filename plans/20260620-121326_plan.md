# Plan: Refactor AgentSession to remove direct RAG-layer responsibility (requirement 20260619_01)

## Goal

Establish a strict responsibility boundary where `AgentSession` owns only `session.sqlite` persistence and all RAG document maintenance operations are routed through dedicated service boundaries or MCP call paths.

## Scope

**In scope:**
- Audit `agent/session.py` and confirm zero direct RAG table dependencies
- Verify `/db clean`, `/db urls`, `/db rebuild-fts`, `/db recover`, `/db consistency` are routed through `DbMaintenanceService` or `rag-pipeline-mcp` (not direct DB access)
- Separate `DbMaintenanceService` into session-only and RAG-only service boundaries (or clearly mark mixed responsibilities)
- Update `/db help` labels so each subcommand shows a single target DB ("Session" or "RAG"), not "RAG + Session"
- Update module docstrings and comments to reflect the new responsibility boundary
- Update affected documentation files

**Out of scope:**
- Redesigning the full RAG pipeline
- Changing the functional behavior of RAG indexing/search
- Modifying `db/maintenance.py` core functions (they are implementation details)
- Adding new MCP servers

## Assumptions

1. `fecfdb1` commit ("move RAG DB operations out of AgentSession") completed the primary refactor; remaining work is cleanup and boundary clarification
2. `DbMaintenanceService` can remain as a single class but should be refactored so RAG and session methods are clearly separated (e.g., via internal delegation or method grouping)
3. The `/db urls` and `/db clean` commands already route through `rag-pipeline-mcp` via `tool_executor` — this is the correct pattern to preserve
4. Documentation updates should target: `docs/05_agent_01_system-overview.md`, `docs/05_agent_04_state-and-persistence.md`, `docs/04_mcp_04_server_catalog.md`

## Current State Analysis

### What's already done (post fecfdb1)

1. **`AgentSession` is clean of RAG dependencies:**
   - All DB operations use `SQLiteHelper("session")` — points to `session.sqlite` only
   - No references to `documents`, `chunks`, `chunks_fts`, `chunks_vec`, or `rag.sqlite`
   - Delegates to `SessionMessageRepository`, `NoteRepository`, `DiagnosticStore`

2. **`/db clean` and `/db urls` route through MCP:**
   - Both call `self._ctx.services.tools.execute("rag_delete_document"/"rag_list_documents", ...)` via `rag-pipeline-mcp`
   - This is the correct pattern per the requirement

3. **`DbMaintenanceService` exists as a service boundary:**
   - Wraps RAG and session maintenance operations
   - Extracted from `cmd_db._DbMixin` for testability

### Remaining issues

1. **`DbMaintenanceService` mixes RAG and session operations:**
   - `stats()` reads from both `rag` and `session` DBs in one call
   - `rebuild_fts()`, `consistency()`, `recover()` operate on `rag.sqlite`
   - `health()`, `checkpoint()`, `vacuum()`, `purge()` operate on `session.sqlite`
   - Single class with no clear separation

2. **`/db help` shows "RAG + Session" for stats subcommand:**
   - Line 71: `"stats", "RAG + Session", "", "Record counts"` — blurs the boundary
   - Other subcommands correctly show single target ("RAG" or "Session")

3. **Docstrings in `cmd_db.py` reference RAG table names:**
   - Line 154: `"Rebuild the FTS5 chunks_fts index in rag.sqlite."` — exposes RAG schema details
   - Line 211: `"Run RAG consistency check: chunks vs FTS vs vec index sync."` — exposes RAG schema details
