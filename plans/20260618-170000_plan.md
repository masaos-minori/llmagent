# Plan: Show watchdog state in operational status output

## Goal

Make watchdog enabled/disabled state, interval, and restart policy visible in startup logs and `/mcp` status output so operators can quickly see whether MCP auto-recovery is active.

## Scope

**In scope:**
- `/mcp` status display
- Startup logs
- Watchdog operational visibility
- `scripts/agent/repl_health.py` — `watchdog_loop()` function

**Out of scope:**
- Changing watchdog behavior (enable/disable logic, restart policy)
- Modifying the watchdog loop implementation
- Changes to MCP server lifecycle management
- Adding new watchdog features

## Assumptions

1. Watchdog is controlled by `mcp_watchdog_interval` and `mcp_watchdog_max_restarts` config values.
2. In local mode, watchdog is disabled by default — operators may not notice this.
3. The `watchdog_loop()` function already uses these config values but does not log its own state at startup.
4. There is no visible indicator that watchdog is running or stopped.

## Implementation Steps

1. **Add startup log for watchdog state**
   - When `watchdog_loop()` starts, log:
     ```
     Watchdog: enabled (interval=30s, max_restarts=5)
     ```
     or
     ```
     Watchdog: disabled (interval=0) — failed servers will not be auto-restarted
     ```
   - Completion criterion: Startup logs clearly show watchdog state

2. **Add watchdog status to `/mcp` endpoint**
   - If `/mcp` exists and aggregates server states, include a `watchdog` section:
     ```json
     {"watchdog": {"enabled": true, "interval_sec": 30, "max_restarts": 5}}
     ```
   - Completion criterion: /mcp output includes watchdog state

3. **Document watchdog behavior in ops docs**
   - Clarify local vs production defaults
   - Explain that failed MCP subprocesses will not be auto-restarted when watchdog is disabled
   - Completion criterion: Docs accurately describe watchdog behavior
