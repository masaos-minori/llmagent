# Plan: Make /db command scope easier to understand

## Goal

Clarify which database each `/db` subcommand targets so users can instantly understand scope. Fix discrepancies between help text and actual behavior. Consider whether namespacing is needed if more databases are added later.

## Scope

**In scope:**
- Update `/db help` table with clearer DB labels (e.g., "RAG", "Session" instead of "rag+session")
- Update all subcommand output to explicitly state target DB in messages
- Fix discrepancies: `health` says "rag+session" but only checks session; `recover` says "session" but defaults to "rag"
- Update `SubcommandSpec` descriptions in command metadata for clarity
- Consider renaming `/db` to `/db-rag` and `/db-sessions` if scope separation is needed

**Out of scope:**
- Adding `/db` subcommands for workflow.sqlite (future feature, not part of this requirement)
- Refactoring the command dispatch mechanism
- Changing help text formatting style (keep existing table format)

## Assumptions

1. The platform has 3 databases: `rag.sqlite`, `session.sqlite`, `workflow.sqlite`. Currently `/db` only manages rag and session.
2. Users expect `/db` to be a general database command, but it is RAG/session-specific — this mismatch is the core problem.
3. Renaming to `/db-rag` and `/db-sessions` would be a breaking change for existing users; if adopted, an alias or redirect should be maintained.
4. The help table format (`write_table`) is already used and clear — only content needs updating.

## Implementation Steps

1. **Fix discrepancies first**
   - `health()`: either add rag health check or update help text to say "session only"
   - `recover()`: update help text to say "rag (default)" or make session recover configurable
   - Decision: update help text to match actual behavior (minimal code change)

2. **Update `/db help` table**
   - Change `[rag+session]` → `[RAG + Session]` (capitalized, clearer)
   - Change `[rag]` → `[RAG]` and `[session]` → `[Session]`
   - Ensure consistent formatting across all SubcommandSpec entries

3. **Update subcommand output messages**
   - Every success/error/no-data message from /db commands should include target DB in brackets
   - Example: `write_success("Deleted document [RAG]")` instead of just the URL
   - Multi-DB commands (stats): show each DB's results with clear header

4. **Update command metadata**
   - `command_defs.py`: update SubcommandSpec descriptions to match updated help text
   - Ensure consistency between `cmd_db.py` help output and `SubcommandSpec` labels

5. **Consider namespacing (optional, deferred)**
   - If workflow.sqlite maintenance is planned, rename to `/db-rag`, `/db-sessions` with `/db` as alias
   - For now: just clarify scope without renaming (lower risk)

6. **Update tests**
   - Update assertion strings in `test_db_commands.py` for new output format
   - Add test case verifying help text mentions both RAG and Session databases
