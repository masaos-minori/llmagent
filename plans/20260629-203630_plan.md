## Goal
MDQ ドメイン例外（Validation, Authorization, NotFound 等）が `call_tool` エンドポイントで `CallToolResponse(is_error=True)` として返却されるよう `server.py` を修正し、ツールレベルエラーとトランスポートエラーを明確に区別する。

## Scope
**In-Scope**:
- `call_tool` エンドポイントで MDQ ドメイン例外を捕捉し `is_error=True` を返す
- `MdqDatabaseError`, `MdqIndexNotReadyError`, `MdqConsistencyError` の HTTP 5xx マッピングを維持（トランスポートエラーとして区別）
- `tests/test_mdq_error_taxonomy.py` に `call_tool` レベルの `is_error=True` 確認テストを追加

**Out-of-Scope**:
- `models.py` の例外クラス定義変更（すでに実装済み）
- FastAPI exception handler の削除（トランスポートエラー用は維持）
- DB スキーマの変更

## Assumptions
1. `MdqValidationError`, `MdqAuthorizationError`, `MdqNotFoundError`, `MdqIndexNotReadyError` はツールレベルエラー → `is_error=True`
2. `MdqDatabaseError`, `MdqConsistencyError` はインフラ障害 → HTTP 5xx（トランスポートエラー）として維持
3. `MdqServiceError` の基底クラスは両方のカテゴリを含む（サブクラスで分類）
4. 現在の FastAPI exception handler (118-158行) はトランスポート用として残す

## Unknowns
| Unknown | Evidence missing | Resolution | Blocking |
|---|---|---|---|
| HTTP exception handler と call_tool 内 try/except の優先順位 | FastAPI の exception handler は `call_tool` ハンドラ内の例外よりも先に捕捉するか不明 | FastAPI の挙動を確認 (exception handler はルートハンドラ内の例外には適用される) | False |

**分析結果**: FastAPI の exception handler はルートハンドラ外で raise された例外を捕捉する。`call_tool` 内で try/except して自分で処理すれば exception handler は呼ばれない。

## Implementation Steps
1. **Phase 1: call_tool エンドポイントの修正**
   - [ ] `call_tool` 内の `_dispatch_mdq_tool(req.name, req.args)` 呼び出しを try/except で囲む
   - [ ] `MdqValidationError`, `MdqAuthorizationError`, `MdqNotFoundError`, `MdqIndexNotReadyError` は `CallToolResponse(is_error=True, result=str(exc))` を返す
   - [ ] 例外種別を audit ログの `error_kind` に記録
   - [ ] `MdqDatabaseError`, `MdqConsistencyError` は re-raise（HTTP 5xx exception handler に委ねる）
