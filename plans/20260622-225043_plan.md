## Goal

Add tests for concurrent upsert conflict scenarios in `tests/test_memory_store.py` to detect race condition behavior in the MemoryStore.upsert method.

## Scope

**In-Scope**:
- Add test for concurrent upsert with same ID verifying last-write-wins behavior
- Add test for concurrent upsert with embedding parameter
- Add test for concurrent upsert with BEGIN IMMEDIATE conflict (SQLite busy error)

**Out-of-Scope**:
- Modifying the MemoryStore implementation
- Adding tests for other methods or modules
- Any changes outside `tests/test_memory_store.py`

## Assumptions

1. SQLite's `BEGIN IMMEDIATE` prevents concurrent writes to the same database — one will succeed, others will get a busy error
2. The `_ConcurrentHelper` class in the existing test already uses separate connections per thread, which is necessary for testing concurrency
3. When two threads upsert the same memory_id concurrently, the last commit wins (SQLite's default behavior)

## Design

The existing `_ConcurrentHelper` class and `_make_concurrent_store()` function provide the infrastructure for concurrent testing. The plan is to add three new test methods:

1. **test_concurrent_upsert_same_id_last_write_wins**: Verify that when two threads upsert the same memory_id concurrently, exactly one wins (last-write-wins), and the content reflects the winning write
2. **test_concurrent_upsert_with_embedding**: Verify concurrent upsert with embedding parameter does not corrupt the vec table
3. **test_concurrent_upsert_busy_error_handling**: Verify that when SQLite returns a busy error due to BEGIN IMMEDIATE contention, the store handles it gracefully (either retries or propagates the error)

## Risks

- **Risk**: SQLite `BEGIN IMMEDIATE` may cause one thread to get a busy error, making the test flaky — **Mitigation**: Use `pytest.approx` for timing-sensitive assertions and add retry logic in test setup
- **Risk**: Concurrent upsert with embedding may expose vec table inconsistency — **Mitigation**: Add explicit assertion on vec table row count after concurrent upsert
- **Risk**: Test may be non-deterministic due to thread scheduling — **Mitigation**: Use large enough content differences to make the winning write detectable, and verify only structural properties (row count, not specific content)
