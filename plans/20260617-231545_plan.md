## Design

### Current state

```python
# events.py line 2-6
"""shared/events.py
Typed event definitions for agent lifecycle and artifact notifications.

ArtifactEvent is a pure data definition — no event bus yet.
"""
```

The phrase "no event bus yet" implies future work and may mislead readers into thinking delivery exists or is imminent.

### Proposed design

**Updated module docstring:**

```python
"""shared/events.py
Typed event definitions for agent lifecycle and artifact notifications.

ArtifactEvent is a pure data structure (TypedDict). It has no delivery system,
no event bus, and no consumers. It exists solely as a type annotation for code
that may emit artifact events in the future. Do not assume that creating an
ArtifactEvent instance triggers any action.
"""
```

**Update `docs/06_shared_90_inconsistencies_and_known_issues.md` EVENTS-01:**

Change from:
```
- **Recommended action:** Implement event bus before using ArtifactEvent in production workflows.
```

To:
```
- **Resolution (Option A selected):** Module docstring now explicitly states ArtifactEvent is data-only with no delivery system, no event bus, and no consumers. The phrase "no event bus yet" was replaced with clear documentation that creating an ArtifactEvent triggers no action.
- **Recommended action:** None — use ArtifactEvent only as a type annotation. If an event bus is needed in the future, implement it as a separate feature.
```

### Why Option A over Option B?

| Factor | Option A (document) | Option B (event bus) |
|---|---|---|
| Implementation effort | ~10 lines of doc text | 200+ lines (publisher, subscriber, routing, error handling) |
| Current benefit | Eliminates ambiguity | No consumers to deliver to |
| Risk | None | New code path, new surface area for bugs |
| Future extensibility | Type exists; bus can be added later | Bus exists but may need changes when actual consumers appear |
