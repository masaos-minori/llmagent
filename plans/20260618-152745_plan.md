# Plan: Add Workflow-Level Observability

Derived from `01_plan.md` using requirement `requires/20260618_25_require.md`.

---

## Goal

Execution can be traced at workflow/task level — workflow ID, task ID, retry, approval, and artifact events are captured in OTel spans, audit log, and optionally diagnostics.

---

## Scope

**In scope:**
- `scripts/agent/orchestrator.py` — add OTel spans for workflow phases (plan/execute/verify)
- `scripts/agent/workflow/engine.py` — add tracer to `WorkflowEngine.run()`, propagate span context
- `scripts/agent/workflow/models.py` — ensure `task_id`, `workflow_version` are serializable to observability
- `scripts/agent/tool_audit.py` — add workflow/task event types (e.g. `workflow_start`, `task_created`, `stage_completed`)
- `scripts/agent/services/` — optionally create `observability_service.py`
- `docs/05_agent_10_operations-and-observability.md` — document new event types, span schema

**Out of scope:**
- `diagnostics.jsonl` schema changes (session-level only)
- Grafana / dashboard setup
- Alert rules

---

## Assumptions

1. `WorkflowEngine` currently has no tracer parameter; one can be added without breaking existing callers.
2. OTel `TracerProtocol` is the same type already injected into `Orchestrator` — no new dependency.
3. Audit log events are JSON-lines written via `audit_logger.info()` — new event types follow the same pattern.

## Implementation Steps

1. **Read orchestration loop thoroughly**
   - Find where plan/execute/verify phases dispatch in `Orchestrator`
   - Identify where `StateStore.task_id` and `workflow_version` are accessible
   - Criterion: exact span insertion points known

2. **Read `WorkflowEngine` public API**
   - Confirm `run()` signature can accept a tracer without breaking tests
   - Criterion: minimal API change

3. **Add OTel spans**
   - `WorkflowEngine.run()` span: `"workflow_run"` with `workflow_id`, `task_id`, `version`
   - Per-stage spans: `"stage_plan"`, `"stage_execute"`, `"stage_verify"` with `stage_id`, `attempt_count`
   - Criterion: spans visible in ConsoleSpanExporter output

4. **Add audit events**
   - `workflow_start`, `task_created`, `stage_completed`, `approval_requested`, `artifact_created`
   - Include `workflow_id`, `task_id`, `stage_id` in each record
   - Criterion: events visible in audit log

5. **Document schema**
   - Update `docs/05_agent_10_operations-and-observability.md` with span and event schemas
   - Criterion: any engineer can write a query against audit log by reading the doc

6. **Lint, type, arch validation**
   - `ruff check scripts/` — 0 errors
   - `mypy scripts/` — no new errors
   - `lint-imports` — 0 violations
