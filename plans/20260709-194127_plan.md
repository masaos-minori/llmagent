## Goal

Add a dedicated `ProductionConfigValidator` that detects unsafe or compatibility-oriented settings and makes them fatal in production (`security_profile == "production"`) while preserving warning-only behavior for local/development.

## Scope

- **In-Scope**: ProductionConfigValidator creation and integration; bidirectional `tool_safety_tiers` validation; GitHub `fail_open` production prohibition; legacy `use_tool_dag=false` production audit; `allowed_tools=[]` visibility; startup integration; tests; documentation updates.
- **Out-of-Scope**: Removing `allowed_repos_mode="fail_open"`; removing `use_tool_dag`; changing runtime approval semantics; routing plugin tools through MCP; removing local/development compatibility modes.

## Assumptions

1. `security_profile` is the canonical discriminator between production and local modes.
2. The existing `RagConfigValidator` must not be modified; production checks go into a separate validator.
3. All target source files exist under `scripts/` as Python package paths.

## Design

### Existing state

`scripts/shared/production_config_validator.py` already exists (39 lines) with:
- `ConfigValidationResult` dataclass (`errors`, `warnings`)
- `validate()` method checking 4 strict keys (`plugin_strict`, `tool_definitions_strict`, `routing_drift_strict`, `use_tool_dag`) — but current logic `not config.get(key, False)` incorrectly flags `use_tool_dag=True` as error
- `validate_unknown_tool_safety_tiers()` method for unknown tier keys

Already integrated in `scripts/agent/repl_health.py:763` within `audit_security_defaults()`.

### Gap analysis vs requirement

| Req # | Check | Status |
|---|---|---|
| 2 | `plugin_strict=false` in production → fatal | Partial: exists but logic needs fix (uses `not config.get(key, False)`) |
| 2 | `tool_definitions_strict=false` in production → fatal | Partial: same fix needed |
| 2 | `routing_drift_strict=false` in production → fatal | Partial: same fix needed |
| 3 | Missing `tool_safety_tiers` entry → fatal (prod) / warning (local) | Missing: bidirectional check not implemented |
| 3 | Unknown `tool_safety_tiers` key → fatal (prod) / warning (local) | Partial: `validate_unknown_tool_safety_tiers` exists but always returns errors |
| 4 | GitHub `allowed_repos_mode="fail_open"` in production → fatal | Missing: handled in `repl_health.py:902` but not in validator |
| 5 | `use_tool_dag=false` in production → fatal or startup audit error | Partial: exists in `validate()` but incorrect logic |
| 6 | `allowed_tools=[]` → emit explicit audit message | Missing in validator; exists as warning in `repl_health.py:847` |
| 1 | Validator dedicated (not merged into RagConfigValidator) | Done |
| 7 | Startup integration | Done |

### Recommended changes

1. Refactor `ProductionConfigValidator.validate()` to accept `AgentConfig` (or expose config fields individually) and `security_profile: SecurityProfile`
2. Fix `use_tool_dag` logic: error only when `use_tool_dag=false`
3. Add bidirectional `tool_safety_tiers` validation using existing `check_tool_safety_tiers()` and `check_unknown_tool_safety_tiers()` from `tool_routing_validation.py`
4. Add GitHub `allowed_repos_mode="fail_open"` check via `load_github_audit_config()` from `security_audit_config.py`
5. Add `allowed_tools=[]` visibility check (distinct warning vs policy-violation)
6. Keep `audit_security_defaults()` as the integration point; production errors already raise `RuntimeError`
