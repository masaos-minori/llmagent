## 1. Goal

Replace the MDQ chunk ID generation formula with a stable, deterministic hash derived from normalized source path, heading path, ordinal, and content hash so that IDs are preserved across re-indexing of unchanged content.

## 2. Scope

- **In-Scope**:
  - New `generate_chunk_id()` helper in `scripts/mcp/mdq/indexer.py`
  - Update chunk ID generation in `_index_single_file()` and `_migrate_from_legacy()` inside `indexer.py`
  - No schema changes required — `chunks.chunk_id TEXT UNIQUE NOT NULL` already exists
  - No changes to `search_docs` return format — `chunk_id` is already included in `SearchResultItem`
  - No changes to `get_chunk` input — `GetChunkRequest.chunk_id: str` already accepts string IDs
  - No changes to `grep_docs` return format — `GrepDocMatch.chunk_id` is already returned
  - Tests verifying chunk ID stability across `refresh_index` / `index_paths`
  - Tests verifying search-result `chunk_id` can be passed directly to `get_chunk`

- **Out-of-Scope**:
  - DB schema migration (no schema change needed; `chunk_id` column is already TEXT)
  - Hybrid/vector search changes
  - `outline` tool changes (already uses `chunk_id` from DB)
  - Backward-compatibility shim for old integer row IDs (requirement says to prefer stable IDs)
  - `chunk_summaries` table chunk_id — same generation logic will apply automatically

## 3. Assumptions

- The current production `mdq.sqlite` at `/opt/llm/db/mdq.sqlite` can be wiped and rebuilt via `refresh_index` after this change (IDs are not persisted externally).
- `ParsedSection["heading_path"]` is already populated by `parse_markdown()` — confirmed by `ParsedSection` TypedDict definition and `parser.py` usage.
- `ParsedSection["ordinal"]` is already populated — confirmed by TypedDict and `indexer.py` reading `section.get("ordinal", 0)`.
- Normalizing source path means resolving to an absolute POSIX string (e.g. `Path(path).resolve().as_posix()`), consistent with how `doc_id` is currently computed from `str(path)`.
- Input for `content_hash` in the new ID formula is the SHA-256 of the raw section content (same as `content_hash` already stored per chunk).
- The FTS5 triggers use `chunks.id` (INTEGER AUTOINCREMENT rowid) for FTS5 sync — this is unaffected by the stable `chunk_id` TEXT column change.

## 5. Affected Areas & Tool Evidence

- **Affected Files**:
  - `scripts/mcp/mdq/indexer.py` — chunk_id generation in `_index_single_file()`, `_migrate_from_legacy()`, and `_generate_summaries()` (summary uses same hash formula)
  - `tests/test_mdq_service.py` — add stability test cases
  - `tests/test_mdq_incremental_refresh.py` — add chunk_id stability assertion after refresh

- **Blast Radius**:
  - `scripts/mcp/mdq/service.py` — `get_chunk()`, `grep_docs()`, `outline()` all read `chunk_id` from DB; no code change needed, but behavior changes at runtime because IDs in DB will be different after re-index
  - `scripts/mcp/mdq/search.py` — returns `chunk_id` from DB rows; no code change needed
  - `tests/test_mdq_schema_migration.py` — verifies legacy migration; may need chunk_id format update in assertions
  - `tests/test_mdq_summary_cache.py` — uses chunk IDs; may need updated fixture IDs

- **Risk Metrics**:
  - `refresh_paths` (radon grade E, complexity 34) is the highest-risk function; new formula only changes the hash input string — low code risk
  - `parse_markdown` (grade D, complexity 29) is unchanged
  - bandit: 3 medium issues exist in the module (pre-existing, unrelated to this change)

- **Deploy Impact**:
  - After deploying, all existing `chunk_id` values in `/opt/llm/db/mdq.sqlite` will be stale — a full `refresh_index` (force=True) or DB wipe+rebuild is required
  - No service restart required beyond the normal deploy flow
  - `mdq.sqlite` is not locked at time of planning (`lsof` returned no lock)

## 6. Implementation Steps

1. **Phase 1: Core Logic Change**
   - [ ] Add `generate_chunk_id(normalized_path: str, heading_path: str, ordinal: int, content_hash: str) -> str` function at the top of `scripts/mcp/mdq/indexer.py`
     - Input: `normalized_path = Path(path).resolve().as_posix()`, `heading_path = section["heading_path"]`, `ordinal = section.get("ordinal", 0)`, `content_hash = sha256(section["content"].encode()).hexdigest()`
     - Formula: `sha256(f"{normalized_path}|{heading_path}|{ordinal}|{content_hash}".encode()).hexdigest()`
     - Use `|` as delimiter (not `:`) to reduce collision risk with paths containing colons
   - [ ] Replace the inline `hashlib.sha256(f"{doc_id}:{section['heading']}:{section['start_line']}".encode()).hexdigest()` call in `_index_single_file()` with `generate_chunk_id(...)`
   - [ ] Replace the inline chunk_id formula in `_migrate_from_legacy()` with `generate_chunk_id(...)` using the legacy `file_path` as normalized path
   - [ ] Replace the inline chunk_id formula in `_generate_summaries()` (same pattern) with `generate_chunk_id(...)`

2. **Phase 2: Test Coverage**
   - [ ] Add `TestChunkIdStability` class to `tests/test_mdq_service.py`:
     - `test_chunk_id_stable_across_reindex`: index file, record chunk IDs, re-index same file, assert IDs are identical
     - `test_chunk_id_changes_on_content_edit`: index file, edit content, re-index, assert ID changed
     - `test_search_chunk_id_passthrough`: run `search_docs`, take first result `chunk_id`, pass to `get_chunk`, assert no `MdqNotFoundError`
   - [ ] Add `test_chunk_id_stable_after_refresh` to `tests/test_mdq_incremental_refresh.py`:
     - Force-index, record IDs, force-index again (unchanged), assert IDs equal

3. **Phase 3: Deployment & Verification**
   - [ ] Run full validation: `uv run ruff check scripts/mcp/mdq/`, `uv run mypy scripts/mcp/mdq/`, `uv run pytest tests/test_mdq_service.py tests/test_mdq_incremental_refresh.py tests/test_mdq_schema_migration.py tests/test_mdq_summary_cache.py -v`
   - [ ] On production deploy: run `refresh_index` with `force=True` on all indexed paths, or wipe `/opt/llm/db/mdq.sqlite` and re-index

## 8. Risks & Mitigations

- **Risk**: Existing production chunk IDs (stored in agent memory, audit logs, or client caches) become invalid after re-index → **Mitigation**: Document in deploy notes that a full `refresh_index --force` is required; grep agent memory tables for MDQ chunk ID references before deploying
- **Risk**: `heading_path` is empty string for root sections (content before first heading) → ID still unique because `content_hash` differentiates content; `ordinal` differentiates multiple root-level sections in same file → **Mitigation**: Add test case with a file that has content before first heading
- **Risk**: Two files with identical content, heading_path, and ordinal produce the same chunk_id → **Mitigation**: `normalized_path` is the first component, making IDs per-file unique
- **Risk**: `_generate_summaries()` called inside the per-section loop in `_index_single_file()` generates a chunk_id independently; if formula drifts from the insert formula, summary lookup breaks → **Mitigation**: Both call the same `generate_chunk_id()` helper; enforce via single code path
