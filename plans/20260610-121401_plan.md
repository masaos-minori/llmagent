# DB モジュール改修計画

instruction: `instructions/20260610_01_instruction.md`

## Goal

`db/` 配下の 6 モジュール (config.py / helper.py / create_schema.py / store.py / maintenance.py / tool_results.py) を fail-fast・明示的トランザクション・最新スキーマ単一化の原則に従って全面改修し、fallback・silent failure・migration 救済ロジックをゼロにする。

---

## Scope

**In**:
- `scripts/db/config.py` — `embed_url` 削除、`sqlite_vec_so` をオプション分離、DI 化
- `scripts/db/helper.py` — `_ensure_config` 削除・DI 化、`open()` の vec 条件ロード、`assert` → `RuntimeError`、クラス変数キャッシュ削除
- `scripts/db/create_schema.py` — migration 系コード全削除 (`_*_MIGRATE_SQL`, `_SAFE_MIGRATION_ERRORS`, `_run_migrations`)、`IF NOT EXISTS` 方針整理、`schema_version` 判断
- `scripts/db/store.py` — `get_embedding_dims()` fallback 削除、`doc_upsert()` を真の UPSERT に変更、`SQLiteMemoryDeleteStore` 部分成功廃止
- `scripts/db/maintenance.py` — VACUUM silent failure 廃止、`shutil.copy2` バックアップ禁止・checkpoint 方式に変更、`recover_corruption()` DB 対象引数化、config fallback `{}` 削除
- `scripts/db/tool_results.py` — exception 再 throw、`list_recent` から `full_text` 除外
- `tests/test_sqlite_helper.py`, `tests/test_create_schema.py`, `tests/test_db_maintenance.py`, `tests/test_tool_result_store.py` — 変更に合わせて更新

**Out**:
- `rag/` 層・`agent/` 層の変更
- DB スキーマの実カラム追加・削除 (DDL 文字列の内容変更はスコープ内だが、既存データを壊す変更は除く)
- `deploy/deploy.sh` の変更 (ファイル追加・削除なし)

## Assumptions

1. SQLite >= 3.25 (RENAME COLUMN サポート済み)。本番環境で確認済み。
2. `build_db_config()` は廃止方向。`SQLiteHelper` は `DbConfig` をコンストラクタで受け取るよう変更する。既存の呼び出し箇所 (`agent/factory.py` 等) も同時に修正が必要だが、今回は `db/` 内のみ対応し、外部呼び出し側への影響は Unknowns として整理する。
3. `_SESSION_MIGRATE_SQL` / `_RAG_MIGRATE_SQL` を削除すると、古い DB への適用ができなくなる。本番 DB は常に最新スキーマが適用済みであることを前提とする。
4. `shutil.copy2` バックアップは WAL 整合性が保証されないため廃止する。checkpoint 後に OS コピーするか、SQLite backup API を使う方式に変更する。
5. `tool_results.py` の `list_recent` から `full_text` を除外すると、呼び出し元 (`cmd_tooling.py`) も修正が必要。

## Design

### D-1. `SQLiteHelper` の DI 化方針

**現状**: `SQLiteHelper("rag")` のように文字列でターゲットを指定し、`_ensure_config()` クラスメソッドが初回に `build_db_config()` を呼ぶ。config ロード失敗は `warning` で継続。

**方針** (段階的移行):
- `SQLiteHelper` に `db_path` を直接渡せる第 3 の init 形式を追加:
  `SQLiteHelper(db_path="/opt/llm/db/rag.sqlite", load_vec=True)`
- 既存の `SQLiteHelper("rag")` / `SQLiteHelper("session")` は当面維持するが、config ロード失敗は `RuntimeError` を投げる (warning → exception)。
- クラス変数キャッシュ (`_RAG_PATH` 等) を削除し、`__init__` でインスタンス変数に保持する。

この設計により、既存呼び出し元 (`agent/factory.py` 等) を即時変更せずに fail-fast の原則を適用できる。

### D-2. `open()` の sqlite-vec 条件ロード

**現状**: `open()` は常に `_load_vec_extension()` を呼ぶ。session DB 操作で vec 不要な場合も拡張ロードが走る。

**方針**: `open(load_vec: bool = True)` パラメータを追加し、`load_vec=False` の場合はスキップ。`SQLiteHelper("session")` のデフォルトを `load_vec=False` に変更し、既存コードの動作を改善する。

### D-3. `create_schema.py` — migration 完全削除

**削除対象**:
- `_SAFE_MIGRATION_ERRORS` タプル
- `_RAG_MIGRATE_SQL` リスト
- `_SESSION_MIGRATE_SQL` リスト
- `_run_migrations()` 関数
- `schema_version` テーブル定義

**残置**: `CREATE TABLE IF NOT EXISTS` は初回作成を許容するためそのまま維持。instruction の「IF NOT EXISTS を削除」は新規 DB に対して二重実行を起こさないための修正だが、production で idempotent を保つため維持する。`create_rag_schema` / `create_session_schema` から migration 呼び出しを除去するだけで大半の要件を満たす。

### D-4. `maintenance.py` — backup 方式変更

**現状**: `shutil.copy2(db_path, dest)` でライブ DB をコピー。WAL 側ファイルも個別コピーするが、原子性なし。

**方針**: `sqlite3.Connection.backup(target_conn)` を使用する online backup API に変更。WAL が存在しても整合性が保たれる。

```python
import sqlite3

def _backup_db_file(src_path: Path, dest: Path) -> None:
    src_conn = sqlite3.connect(str(src_path))
    dst_conn = sqlite3.connect(str(dest))
    src_conn.backup(dst_conn)
    dst_conn.close()
    src_conn.close()
```

### D-5. `tool_results.py` — `list_recent` 変更

**現状**: `list_recent` の SELECT に `full_text` が含まれる。

**方針**: `full_text` を SELECT から除外し `{id, tool_name, summary, is_error}` のみ返す。呼び出し元 `cmd_tooling.py` の `/tool list` 表示部分も更新する。

### D-6. `store.py` — `get_embedding_dims()` fail-fast

**現状**: `except Exception: return 384` の fallback がある。

**方針**: fallback 削除。`build_db_config()` が失敗した場合は例外を伝播させる。呼び出し元 (`create_schema.py`) が例外をキャッチして適切に処理する責務を持つ。
