# Work Plan: Remove Obsolete Direct-Execution Fallback Documentation

## Goal

Ensure all documentation consistently states that workflow support is mandatory at startup, with no direct-execution fallback path.

## Scope

**In-Scope:**
- Update documentation to remove all references to direct-execution fallback when workflow loading fails
- Remove all references to `workflow_mode="disabled"` or `workflow_mode="auto"` as valid runtime options
- Ensure consistent wording across turn flow, configuration, operations, and reference docs
- Verify against implementation code for accuracy

**Out-of-Scope:**
- Code changes (this is documentation-only)
- Reintroducing direct execution fallback
- Adding a config toggle to disable workflows
- Changing workflow runtime behavior

## Assumptions

1. The confirmed specification is: agent requires `config/workflows/default.json` at startup; missing or invalid workflow definition causes startup failure with RuntimeError; no direct-execution fallback exists.
2. Implementation code already enforces this behavior (confirmed via inspection of `agent/startup.py`, `agent/orchestrator.py`, `agent/repl_health.py`).
3. `workflow_mode` is already removed from `AgentConfig` dataclass and rejected via `_FORBIDDEN_KEYS`.
