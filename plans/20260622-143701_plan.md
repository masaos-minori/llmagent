# Plan: upsert 同時実行テストの追加 — test_memory_store.py

## Goal

`tests/test_memory_store.py` に `TestUpsertConcurrency` クラスを新設し、
`ThreadPoolExecutor` による並行 `upsert()` 呼び出し下での整合性を検証する。
異なる memory_id の並行挿入（全件保持）と、同一 memory_id の並行 upsert（最終1件）を確認する。

## Scope

**In-Scope:**
- `tests/test_memory_store.py` — `TestUpsertConcurrency` クラスの新設

**Out-of-Scope:**
- `scripts/agent/memory/store.py` の実装変更
- `_FakeSQLiteHelper` の変更
- 他のテストクラスの変更

## Assumptions

1. `upsert()` は `begin_immediate()` で `BEGIN IMMEDIATE` を発行するため、
   SQLite レベルでライターがシリアライズされる。
2. `:memory:` DB は接続ごとに独立するため、並行テストでは `check_same_thread=False` で
   単一接続を複数スレッドで共有する必要がある。
3. GIL により Python レベルの競合は起きないが、`BEGIN IMMEDIATE` のロック競合が
   スレッド間で発生する可能性がある。`timeout=5` を `sqlite3.connect` に指定する。
4. 既存 `_FakeSQLiteHelper` は `begin_immediate()` を実装済みのため流用可能。
5. 新クラスは `_FakeSQLiteHelper` を `check_same_thread=False, timeout=5` の
   専用接続でインスタンス化するヘルパー関数を使う。

## Design

### 共有接続ヘルパー

```python
def _make_concurrent_store() -> tuple[MemoryStore, sqlite3.Connection]:
    conn = sqlite3.connect(":memory:", check_same_thread=False, timeout=5)
    conn.executescript(_SCHEMA_SQL)
    conn.commit()
    fake = _FakeSQLiteHelper(conn)
    with patch("agent.memory.store.SQLiteHelper", return_value=fake):
        store = MemoryStore()
    return store, conn
```

`patch` はコンテキストマネージャ外で使うため、`patch.start()` / `patch.stop()` パターンを使用。

### 追加するテストクラス

**TestUpsertConcurrency**

1. **`test_concurrent_upsert_different_ids_all_persisted`**
   - 異なる memory_id を持つ10エントリを `ThreadPoolExecutor(max_workers=4)` で並行 upsert
   - 完了後: `SELECT count(*) FROM memories` == 10

2. **`test_concurrent_upsert_same_id_single_row`**
   - 同一 memory_id で content だけ異なる5エントリを並行 upsert
   - 完了後: `SELECT count(*) FROM memories WHERE memory_id = ?` == 1
   - (最後の書き込みが勝つ; content の具体値は不問)

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| `BEGIN IMMEDIATE` ロック競合で `OperationalError` 発生 | medium | `timeout=5` で待機、スレッド数を 4 に制限、GIL により実質シリアライズ |
| `patch.start()` / `patch.stop()` のリークで他テストに影響 | low | `try/finally` で必ず `patch.stop()` を呼ぶ |
| 同一 ID の並行 upsert で ROLLBACK が失敗してデータ不整合 | low | `begin_immediate()` の `except BaseException` ブロックが ROLLBACK を試みる; GIL によりほぼシリアライズ |
