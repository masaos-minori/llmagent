## Goal

Improve visibility and policy clarity for Refiner fallback behavior when `use_refiner=true` — add counters/diagnostics for refiner failures, clarify retry/no-retry policy in docs, and improve `/rag search --debug` visibility.

## Scope

**In-Scope**:
- Add counters or diagnostics for `refiner_returned_empty` and `refiner_exception`
- Clarify in docs why retry is not performed, or add configurable retry if desired
- Improve `/rag search --debug` visibility for refiner fallback

**Out-of-Scope**:
- Replacing the Refiner model
- Redesigning the entire Augment stage

## Assumptions

1. No retry is currently performed — confirmed by pipeline_refiner.py:76 "No retry is performed"
2. Refiner fallback is already tracked in `StageResult` with status="fallback" and fallback_reason set
3. The `/rag search --debug` output shows stage results but doesn't have explicit refiner fallback frequency tracking

## Affected Areas & Tool Evidence

- **Affected Files**:
  - `scripts/rag/pipeline_refiner.py` — already has good error handling documentation (lines 49-76); no retry policy documented
  - `scripts/rag/pipeline.py` — AugmentStage tracks refiner fallback in StageResult (lines 522-531); no specific refiner counter
  - `scripts/agent/commands/cmd_ingest.py` — shows refiner fallback warning (lines 188-200) but not frequency tracking
  - `docs/03_rag_03_query_pipeline.md` — needs clarification on retry policy
  - `docs/03_rag_05_configuration_and_operations.md` — needs refiner config documentation

- **Blast Radius**: Low-Medium — changes to diagnostics model and documentation. No runtime behavior changes unless retry is added.

## Implementation Steps

1. **Phase 1: Add refiner fallback diagnostics**
   - Add `refiner_fallback_count` field to SearchDiagnostics or create a separate refiner diagnostics section in get_diagnostics()
   - Track `refiner_returned_empty` vs `refiner_exception` separately

2. **Phase 2: Improve /rag search --debug visibility**
   - In cmd_ingest.py, enhance debug output to show refiner fallback frequency when multiple queries are processed
   - Show explicit "[refiner] fallback: N times" in debug summary

3. **Phase 3: Document retry/no-retry policy**
   - Clarify in docs why retry is not performed (non-critical failure — raw chunks are acceptable degraded behavior)
   - Document that `use_refiner=false` disables refiner entirely (no fallback possible)

4. **Phase 4: Add refiner config documentation**
   - Document existing refiner config fields (refiner_max_tokens, refiner_max_chars_per_chunk, refiner_timeout)
   - If retry is added later, add `refiner_retry` config with clear limits
