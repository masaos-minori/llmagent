# Plan: Document Minimal Agent DB Initialization Path

**Requirement**: `requires/20260702_65_require.md`
**Requirement ID**: H-5
**Created**: 2026-07-02

---

## Goal

Add a "Minimal Agent DB initialization" section to `docs/05_agent_10_operations-and-observability.md` so that a new developer can initialize `session.sqlite` (and optionally `workflow.sqlite`) before running `python -m agent` without needing sqlite-vec or the full RAG schema.

---

## Scope

**In-Scope**:
- Add a new section to `docs/05_agent_10_operations-and-observability.md` describing the minimal DB initialization path for local development and first-time startup
- Document the `create_session_schema()` / `create_workflow_schema()` invocation using `PYTHONPATH=scripts uv run python`
- Document table verification commands for `sessions`, `messages`, and `tasks`
- Add a re-run safety (idempotency) note
- Optionally update the module docstring in `scripts/db/create_schema.py` to reference the docs section

**Out-of-Scope**:
- Changes to `create_schema.py` logic or function signatures
- sqlite-vec path handling or `create_rag_schema()` changes
- `/exit` handling
- `deploy/init_db.sh` changes (that script already calls `create_schema()` for full initialization)
- Any new CLI commands or agent features

---

## Assumptions

1. The target audience is a developer running the agent locally from the `scripts/` directory with `uv run`, not a production deployment user.
2. `create_session_schema()` and `create_workflow_schema()` are idempotent (they use `IF NOT EXISTS` guards and migration guards), so re-run safety holds.
3. `workflow.sqlite` initialization is optional when workflow tracking is disabled; the documentation should note this clearly.
4. The recommended invocation uses `PYTHONPATH=scripts uv run python -` (stdin heredoc) as specified in the requirement.
5. `docs/05_agent_10_operations-and-observability.md` is the correct target file — it already contains `session.sqlite` and `workflow.sqlite` inspection commands (around line 84-90).
6. `create_session_schema()` calls `get_embedding_dims()` for the `memories_vec` table; if sqlite-vec is absent, this call may fail. This must be verified in Phase 1.

## Implementation Steps

### Phase 1: Verify sqlite-vec dependency in create_session_schema()

- [ ] Read `scripts/db/store_protocols.py` to check `get_embedding_dims()` behavior when sqlite-vec is absent (raises vs. returns fallback).
- [ ] If it raises without sqlite-vec: update the documented command to skip `memories_vec` creation, or document sqlite-vec as a prerequisite for the minimal path.
- [ ] Confirm `create_workflow_schema()` has no sqlite-vec dependency (it uses `build_workflow_schema_sql()` which is pure SQL, no vectors).

### Phase 2: Write the documentation section

- [ ] Open `docs/05_agent_10_operations-and-observability.md`.
- [ ] Locate the existing DB inspection block (around the `session.sqlite` / `workflow.sqlite` queries, near line 84).
- [ ] Add a new section "Minimal Agent DB initialization" before the existing inspection commands, with the following subsections:
  1. **When to use**: first-time local development, after wiping `session.sqlite` or `workflow.sqlite`.
  2. **Session and workflow schema initialization command**:
     ```bash
     PYTHONPATH=scripts uv run python - <<'PY'
     from db.create_schema import create_session_schema, create_workflow_schema
     create_session_schema()
     create_workflow_schema()
     PY
     ```
  3. **Table verification commands**:
     - `sqlite3 /opt/llm/db/session.sqlite "SELECT name FROM sqlite_master WHERE type='table' AND name='sessions';"`
     - `sqlite3 /opt/llm/db/session.sqlite "SELECT name FROM sqlite_master WHERE type='table' AND name='messages';"`
     - `sqlite3 /opt/llm/db/workflow.sqlite "SELECT name FROM sqlite_master WHERE type='table' AND name='tasks';"`
  4. **Re-run safety note**: all schema functions use `IF NOT EXISTS`; safe to re-run.
  5. **workflow.sqlite note**: only required when `workflow_db_path` is configured in `common.toml`.
  6. **Error context**: missing `sessions` table produces `sqlite3.OperationalError: no such table: sessions` — this section resolves that error.

### Phase 3: Optional docstring update in create_schema.py

- [ ] Add a one-line note to the module docstring in `scripts/db/create_schema.py` pointing to `docs/05_agent_10_operations-and-observability.md` for the minimal initialization path.

### Phase 4: Validation

- [ ] Run the documented command in the local dev environment.
- [ ] Confirm `cd scripts && uv run python -m agent` starts without `no such table: sessions`.
- [ ] Run `uv run pytest` to ensure no tests are broken.
