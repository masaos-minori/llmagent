# Plan: Add detailed API documentation for the memory layer

## Goal

Add detailed documentation for each internal memory sub-module so developers can understand responsibilities, module interactions, and activation/disabled behavior without reading source code.

## Scope

**In scope:**
- Update `docs/05_agent_12_memory.md` with:
  - Module interaction diagram (text-based) showing data flow between store, retriever, injection, ingestion, embedding_client, jsonl_store, extract
  - Detailed responsibility descriptions for each of the 14 modules (expand beyond method tables)
  - Clear activation/deactivation flow (3-layer gate: use_memory_layer → memory_embed_enabled → lifecycle hook calls)
  - Disabled behavior section expanded with per-module details
  - Data model explanation (MemoryEntry fields, how they map to DB tables)
  - Search strategy comparison (FTS vs KNN vs Hybrid, when each is used)

**Out of scope:**
- Creating new documentation files (all content goes into existing `docs/05_agent_12_memory.md`)
- Documenting the memory CLI commands (`cmd_memory.py`) — covered in `05_agent_07_cli-and-commands.md`
- Documenting unit tests for memory modules

## Assumptions

1. The existing `docs/05_agent_12_memory.md` structure (module tables) should be preserved and expanded, not replaced.

## Implementation Steps

1. **Add module interaction overview**
   - Add text-based diagram showing: session_start → injection → retriever → store; session_stop → extract → ingestion → jsonl_store + store + embedding_client
   - Explain the data flow: MemoryEntry flows from extract → ingestion → (jsonl_store + store), retrieval flows from store → retriever → injection

2. **Expand each module's responsibility description**
   - For each of the 14 modules, add a 2-3 sentence paragraph explaining:
     - What problem this module solves
     - How it fits into the overall memory pipeline
     - Key design decisions (e.g., why jsonl_store exists alongside SQLite)
   - Preserve existing method tables

3. **Document the 3-layer activation gate**
   - Document each layer clearly:
     - Layer 1: `use_memory_layer` — factory-level None pattern
     - Layer 2: `memory_embed_enabled` — EmbeddingClient disabled behavior
     - Layer 3: Lifecycle hook conditional calls (startup.py, repl.py)
   - Include a decision tree or flow chart (text-based)

4. **Expand disabled behavior section**
   - Per-module disabled behavior: what each module does (or doesn't do) when memory is disabled
   - Clarify that no individual module implements its own enabled/disabled gate — factory-level None covers all

5. **Add data model explanation**
   - Explain MemoryEntry fields and their DB column mapping
   - Explain how memories, memories_fts, memories_vec tables relate to each other
   - Explain the role of memory_links table for dedup

6. **Add search strategy comparison**
   - FTS5 BM25: text search, fast, no embedding needed
   - KNN (sqlite-vec): semantic similarity, requires embedding
   - Hybrid (RRF merge): combines both, primary external interface
   - When each is used (on_session_start → top_semantic; on_user_prompt → hybrid search)

7. **Add JSONL format specification**
   - One JSON object per line
   - Fields match MemoryEntry dataclass
   - Append-only; read by jsonl_store.read_all()
