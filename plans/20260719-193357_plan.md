# Plan: web_search provider execution semantics, empty-result handling, timeout, error classification

Source requirement: `requires/20260717_22_require.md` (moved to `requires/done/` in Step 7 of this cycle).

## Goal

`search_web` treats an empty DuckDuckGo result set as a successful zero-result response (not an
upstream error), enforces an explicit, configurable MCP-level timeout around the provider call, and
raises classified domain exceptions (`WebSearchTimeoutError`, `WebSearchNetworkError`,
`WebSearchProviderError`, `WebSearchParseError`) instead of a single generic
`WebSearchUpstreamError`, with tests covering each new code path.

## Scope

**In scope:**
- `scripts/mcp_servers/web_search/search_provider.py` — remove empty-list-as-error, add
  `asyncio.wait_for` timeout wrapping, classify DuckDuckGo/provider failures, validate raw result
  shape before constructing `SearchResult`.
- `scripts/mcp_servers/web_search/formatters.py` — remove the `raise WebSearchUpstreamError("No
  results returned...")` branch in `search_web()`; empty results flow through to
  `fdisp_search_web()`'s existing `"No search results found."` branch unchanged.
- `scripts/mcp_servers/web_search/web_search_models.py` — add `search_timeout_sec` to
  `WebSearchConfig` (with validation: numeric, > 0, bounded upper limit) and add the new exception
  classes, all subclassing `WebSearchUpstreamError`.
- `config/web_search_mcp_server.toml` — add `search_timeout_sec = 10.0`.
- New tests: `tests/test_web_search_provider.py`, `tests/test_web_search_formatters.py`,
  `tests/test_web_search_server.py`.
- One necessary adjacent fix (see Design/Risks): `scripts/mcp_servers/web_search/server.py`'s import
  of `WebSearchUpstreamError`/`WebSearchConfig` from the orphaned duplicate `models.py` must be
  repointed to `web_search_models.py` so the registered FastAPI exception handler actually matches
  the exception classes the rest of the change set raises.

**Out of scope:**
- No new search provider (Brave/Bing/etc.) — DuckDuckGo remains the sole provider.
- No change to `SearchRequest`/`SearchResponse`/`SearchResult` public field shapes.
- No fix to the broader duplicate-module leftover pattern from commit `ca6b7bfe` in the other 7
  affected MCP servers (browser, shell, mdq, cicd, git, github, rag_pipeline) — flagged as a
  separate follow-up risk below, not part of this plan's implementation steps.
- No change to `docs/04_mcp_04_01_web-search-file-read-github.md` structure beyond documenting the
  new config key and error taxonomy (see Implementation Steps).

## Assumptions

1. The requirement's target file `models.py` refers to
   `scripts/mcp_servers/web_search/web_search_models.py`, not the byte-identical, orphaned duplicate
   `scripts/mcp_servers/web_search/models.py` — verified: `search_provider.py` and `formatters.py`
   (the requirement's other two source targets, both single-copy files) already import exclusively
   from `web_search_models` (`from mcp_servers.web_search.web_search_models import ...`), and
   `tests/test_web_search_models.py` (existing, passing) already imports the same module. Falsifiable:
   would be wrong if any currently-shipping code path imported domain types from the old `models.py`
   in a way this plan doesn't also fix — checked via `rg`, no such path exists other than the
   `server.py` import addressed above.
2. `asyncio.wait_for(..., timeout=search_timeout_sec)` wrapping the whole
   `asyncio.to_thread(_sync_search)` call (mirroring `scripts/mcp_servers/mdq/search.py`'s
   `search_docs()` pattern) is an acceptable timeout boundary, even though cancelling
   `asyncio.to_thread` does not kill the underlying OS thread immediately (explicitly called out in
   the requirement's "Notes for implementer") — the MCP request still returns promptly, matching mdq's
   existing accepted behavior for the same tradeoff.
3. `search_timeout_sec` defaults to `10.0` (requirement's recommended default) and validation follows
   the existing `WebSearchConfig.from_dict()` coercion style (`float(d.get(...))`), with an explicit
   raise (`ValueError`) if `<= 0` or above a fixed upper bound; `60.0` is a reasonable upper bound
   (config's health-check/timeout conventions elsewhere in the repo, e.g. `shell_mcp_server.toml`'s
   `max_timeout_sec = 300`, `mdq_mcp_server.toml`'s `search_timeout_sec = 30`, use similarly modest
   caps relative to their own domains; DuckDuckGo scraping has no legitimate reason to run longer than
   ~60s for an MCP tool call).
4. New exception classes subclass `WebSearchUpstreamError` (per requirement item 3), so the existing
   `@app.exception_handler(WebSearchUpstreamError)` registered in `server.py` (and duplicated
   identically in `web_search_server.py`) continues to catch them via Starlette's
   inheritance-aware exception middleware, once the import-path fix in Assumption/Risk item below
   lands — no new `@app.exception_handler` registration is required.
5. Malformed provider result handling (missing title/url/body) is lenient by default today
   (`r.get("title", "")` etc. in `search_provider.py` already default missing string fields to `""`);
   the requirement's "validate provider result shape" item is interpreted as: raise
   `WebSearchParseError` only for structurally invalid entries (e.g. `raw` item is not a dict / is
   `None`), not for individual missing string sub-fields, which remain defaulted to `""` as today —
   this preserves the existing "successful search formatting remains unchanged" acceptance criterion
   for well-formed partial results.

## Design

1. **`web_search_models.py`** — add below `WebSearchUpstreamError`:
   ```python
   class WebSearchTimeoutError(WebSearchUpstreamError):
       """Raised when the provider call exceeds search_timeout_sec."""

   class WebSearchNetworkError(WebSearchUpstreamError):
       """Raised on a network-level provider failure (connection, DNS, etc.)."""

   class WebSearchProviderError(WebSearchUpstreamError):
       """Raised on a non-network provider-side failure."""

   class WebSearchParseError(WebSearchUpstreamError):
       """Raised when provider response data cannot be parsed into SearchResult."""
   ```
   Add `search_timeout_sec: float = 10.0` to `WebSearchConfig`, parsed/validated in `from_dict()`:
   reject (`ValueError`) if not coercible to `float`, if `<= 0`, or if `> 60.0` (see Assumption 3 /
   UNK-03 default).

2. **`search_provider.py`** — wrap the existing `_sync_search` thread call:
   ```python
   try:
       raw = await asyncio.wait_for(
           asyncio.to_thread(_sync_search), timeout=cfg.search_timeout_sec
       )
   except TimeoutError as e:
       raise WebSearchTimeoutError(f"DuckDuckGo search timed out after {cfg.search_timeout_sec}s") from e
   except (RuntimeError, OSError) as e:
       raise WebSearchNetworkError(f"DuckDuckGo search failed: {e}") from e
   except Exception as e:  # fallback classification per UNK-01 default
       raise WebSearchProviderError(f"DuckDuckGo search failed: {e}") from e
   ```
   `search_duckduckgo()` needs `cfg: WebSearchConfig` (or just `search_timeout_sec: float`) threaded
   in as a parameter — call site is `formatters.py:search_web()`, which already has access to the
   module-level `_cfg` via `web_search_models`. Add shape validation before building each
   `SearchResult`: if an item in `raw` is not a `dict`, raise `WebSearchParseError` (do not silently
   skip — requirement asks for it to be "handled", tested explicitly).
   Return `[]` (not an error) when `raw` is legitimately empty — remove no special-casing needed here
   since the function already returns `[r for r in raw]`; empty `raw` naturally produces `[]`.

3. **`formatters.py`** — delete the `if not results: raise WebSearchUpstreamError(...)` block inside
   `search_web()`. `SearchResponse(query=..., results=[], provider="duckduckgo")` flows through
   normally; `fdisp_search_web()`'s pre-existing `if not result.results: return "No search results
   found."` already produces the required formatted output with no further change.

4. **`server.py` import fix** — change two import lines to source from `web_search_models`/
   `web_search_tools` instead of the orphaned `models`/`tools` duplicates, so
   `@app.exception_handler(WebSearchUpstreamError)` is registered against the same class object that
   `search_provider.py`/`formatters.py` actually raise. (`web_search_server.py`, the parallel
   new-named file, already does this correctly and needs no change — it is currently dead/unused per
   `config/agent.toml`'s cmd path, out of scope to reconcile further in this plan.)

5. **`config/web_search_mcp_server.toml`** — add:
   ```toml
   # search_timeout_sec: MCP-level timeout (seconds) wrapped around the DuckDuckGo provider call
   search_timeout_sec = 10.0
   ```
