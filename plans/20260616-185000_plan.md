# Plan: Prevent plugin tools from shadowing MCP tools (req 23)

## Goal

Reject plugin tool registration when a plugin tool name conflicts with a known MCP tool by default, while allowing explicit overrides via config flag.

## Scope

**In scope**:
- `shared/plugin_registry.py` — add conflict detection in `register_tool()` and `load_plugins()`
- `shared/tool_constants.py` — expose a function to get all known MCP tool names
- `agent/config_dataclasses.py` — add `plugin_tool_override` field to `ToolConfig`
- `agent/factory.py` — pass override policy to plugin loading
- `config/agent.toml` — add `plugin_tool_override = false` default
- `tests/test_plugin_registry.py` — add tests for conflict rejection
- `docs/05_agent_11_extension-points.md` — document precedence rule and override policy

**Out of scope**:
- Changing the plugin command override behavior (only tool names)
- Modifying `tool_executor.execute()` dispatch order
- Adding per-plugin override configuration

## Assumptions

1. All known MCP tool names are centralized in `tool_constants.py` as frozensets — this is the source of truth for conflict checking.
2. The override flag is a simple boolean (`plugin_tool_override`), not a list of allowed overrides, to keep it simple.
3. In strict mode (no override), `register_tool()` raises `ValueError` on conflict. In permissive mode, it logs a warning and allows registration.

## Implementation Steps

1. **Add `get_all_mcp_tool_names()` to `tool_constants.py`**
   - Return a `frozenset` of all tool names from READ_TOOLS, WRITE_TOOLS, DELETE_TOOLS, RAG_TOOLS, CICD_TOOLS, MDQ_TOOLS, GIT_TOOLS, SQLITE_TOOLS
   - Add comment noting this is the source of truth for conflict checking

2. **Modify `register_tool()` in `plugin_registry.py`**
   - Accept optional `known_tools: frozenset[str] = frozenset()` parameter
   - Before registering, check if name is in `known_tools`
   - If conflict and no override flag: raise `ValueError(f"Plugin tool {name!r} conflicts with MCP tool")`
   - If conflict and override enabled: log warning and allow registration

3. **Modify `load_plugins()` in `plugin_registry.py`**
   - Accept `override_policy: str = "reject"` parameter ("reject" or "allow")
   - Import `get_all_mcp_tool_names()` from `tool_constants`
   - For each plugin file, pass the known tools set and override policy to `register_tool()`
   - If a conflict is rejected in "reject" mode: log error and skip that plugin file (don't abort all plugins)
   - If override allowed: log warning per conflict

4. **Add `plugin_tool_override` field to `ToolConfig` in `config_dataclasses.py`**
   - Add `plugin_tool_override: bool = False`
   - Add validation in `__post_init__` (must be boolean)

5. **Update `_init_plugin_registry()` in `factory.py`**
   - Read `plugin_tool_override` from `ctx.cfg.tool.plugin_tool_override`
   - Pass override policy to `plugin_registry.load_plugins(plugin_dir, override_policy=...)`
   - Log: `"Plugin tool override policy: {reject|allow}"`

6. **Add config entry in `agent.toml`**
   - Add `plugin_tool_override = false` under `[tool]` section

7. **Add tests for conflict detection** (`test_plugin_registry.py`)
   - Test: register_tool with MCP tool name → ValueError raised (no override)
   - Test: register_tool with MCP tool name + override → warning logged, allowed
   - Test: register_tool with unique name → always succeeds
   - Test: load_plugins with conflicting plugin → skipped in reject mode
   - Test: load_plugins with conflicting plugin → allowed in allow mode

8. **Update documentation** (`docs/05_agent_11_extension-points.md`)
   - Document tool name precedence rules (plugin tools shadow MCP tools by default)
   - Document the override policy and configuration
