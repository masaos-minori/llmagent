# Plan: Define schema change, rebuild, and migration policy for DB-affecting work

Source requirement: `requires/20260716_10_require.md`

## Goal

`docs/90_shared_04_03_db_architecture_and_schema-migration-and-scaling.md`
documents all three existing DB migration/recreation patterns in this
codebase — including MDQ's own auto-detect-and-rebuild pattern, which is
currently undocumented anywhere — and a new schema-changing-issue checklist
exists so future schema work cannot omit "how do existing DBs get this
change" from its scope.

## Scope

**In scope**

- Add a new `mdq.sqlite` row/section to
  `docs/90_shared_04_03_db_architecture_and_schema-migration-and-scaling.md`'s
  §10 "正典（Source of Truth）" table and §8 migration-policy section,
  documenting `scripts/mcp_servers/mdq/db_schema.py::create_production_tables()`
  as a third, distinct migration pattern: automatic legacy-schema detection
  or an old `chunks` table shape (`id INTEGER PRIMARY KEY` +
  `chunk_id TEXT UNIQUE`) and an automatic `DROP`+recreate to the current
  shape (`chunk_id TEXT PRIMARY KEY`) — different from both rag/session/
  eventbus's "manual recreation only" policy and workflow's ALTER-TABLE
  incremental migration mechanism.
- Add a new "スキーマ変更チェックリスト" (schema-change checklist) section to
  the same doc, using this repository's established checklist format (seen
  in `docs/04_mcp_06_15_new-mcp-server-addition-checklist.md`), built
  directly from this requirement's own "Required checklist for future schema
  issues" list.
- Resolve the doc's own existing self-flagged inconsistency at §8a
  ("矛盾（要修正）" — workflow.sqlite's incremental migration is called out
  as contradicting the "no compat migration" principle stated in §8) by
  rewording §8's opening statement to say rag/session/eventbus lack
  incremental migration, rather than stating it as a blanket rule that §8a
  then contradicts.
- Add one clarifying sentence to §8/§10 confirming RAG/session vector tables
  (`chunks_vec`, `memories_vec` in `db/schema_sql.py`) are unrelated to and
  unaffected by MDQ schema/hybrid-search cleanup work — the requirement's
  explicit "Scope protection" ask, and consistent with what was already
  independently confirmed true during the 2026-07-16 MDQ cleanup batch
  (`plans/done/20260716-131500_plan.md`).

**Out of scope**

- Implementing a new migration framework — explicit non-goal.
- Removing or altering any vector table — explicit non-goal.
- Any actual DB schema change — this requirement is documentation/policy
  only; `create_schema.py`, `schema_sql.py`, and `mdq/db_schema.py` are
  read-only references for this plan, not edit targets.

## Design

Three migration patterns will be documented side by side in §8, each named
and scoped to its DB file(s):

1. **rag.sqlite / session.sqlite / eventbus.sqlite** — no incremental
   migration; DDL is `IF NOT EXISTS`-idempotent only; schema changes require
   archive → delete → `create_schema()` recreation (already documented).
2. **workflow.sqlite** — `_WORKFLOW_MIGRATIONS` + `apply_workflow_migrations()`
   ALTER-TABLE-based incremental migration, tracked via
   `workflow_schema_version` (already documented in §8a).
3. **mdq.sqlite** — automatic legacy-schema detection
   (`PRAGMA table_info(chunks)`) and unconditional `DROP` + recreate of the
   affected table(s) at every `create_production_tables()` call, with no
   version tracking and no explicit migration list (new — not documented
   anywhere today).

§8's opening sentence changes from an unqualified "互換マイグレーションは非対応"
(incompatible migration unsupported, stated as a blanket rule) to explicitly
scope that claim to rag/session/eventbus, removing the need for §8a to be
framed as a "矛盾（要修正）" — it becomes a named exception in a list of
three patterns rather than a contradiction of a blanket rule.

The new checklist section is built directly from the requirement's own
"Required checklist for future schema issues" list, formatted as Markdown
checkboxes matching the existing `04_mcp_06_15_...` convention.

## Risks

- **Risk**: Reframing §8's opening claim (from a blanket rule to a scoped
  one) could be read as loosening the "no migration support" guarantee for
  rag/session/eventbus if not worded carefully.
  **Mitigation**: keep the substantive guidance for rag/session/eventbus
  completely unchanged — only the scoping qualifier changes, and the
  existing §11 recreation-procedure cross-reference stays intact.
- **Risk**: A future schema change could still skip the new checklist if
  it isn't wired into an actual enforced process (e.g., a PR template check)
  — this plan only adds documentation, not automated enforcement.
  **Mitigation**: explicitly out of scope per this requirement's own
  "Non-goals" (do not implement a new migration framework); the checklist is
  a documented practice, not a CI gate, matching the same enforcement level
  as the existing MCP server addition checklist.
- **Risk**: Overlap with any other schema-documentation work in this
  MDQ-adjacent requirement batch (`requires/20260716_11-14`, not yet read at
  the time this plan was written) if one of them also touches
  `docs/90_shared_04_03_...md`.
  **Mitigation**: to be checked when processing those requirements next, per
  this workflow's sequential per-file processing order.
