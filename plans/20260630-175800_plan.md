# Plan: Document `/replay` pagination semantics

## Goal
`docs/06_eventbus_02_http_api_and_runtime.md` と `docs/06_eventbus_06_reference_api.md` に `GET /replay` のクエリパラメータ・バリデーション・レスポンス形状・SSE挙動を正確に記述し、クライアントがページネーション付きreplayを正しく実装できる状態にする。

## Scope
**In-Scope**:
- `docs/06_eventbus_02_http_api_and_runtime.md` の `GET /replay` セクション更新
- `docs/06_eventbus_06_reference_api.md` の `/replay` エンドポイント説明更新
- クエリパラメータ (`since_seq`, `format`, `limit`, `offset`) の明文化
- バリデーション制約 (`limit >= 1`, `limit <= 1000`, `offset >= 0`) の明文化
- JSON レスポンス形状 (`total`, `limit`, `offset`, `items`) の明文化
- SSE フォーマット使用時のページネーション動作の説明（`format=json` を推奨する旨の補足）

**Out-of-Scope**:
- 実装コードの変更
- テストの追加・変更
- 他エンドポイントのドキュメント更新
- `since_seq` の上限バリデーション追加（現状未実装）

## Assumptions
- 現行実装のバリデーションルールは `limit >= 1`, `limit <= 1000`, `offset >= 0` であり変更予定はない
- `total` はページネーションの `limit`/`offset` を無視したマッチイベント総数を返す
- SSEフォーマット時は `limit`/`offset` パラメータをSQLレベルで適用するが、ストリームの継続再開はサポートしない
- 既存テストが paginated response shape を検証済みであり、ドキュメントはそのテストに基づく
