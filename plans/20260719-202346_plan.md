# Plan: Verify and clean up registry/safety-tier/documentation consistency for `search_web`

Source requirement: `requires/20260717_25_require.md` (moved to `requires/done/` in Step 7 of this cycle).

## Goal

`search_web` is verified consistent across `tool_constants.py` (`WEB_SEARCH_TOOLS`), `ToolRegistry`,
the deployed server's `/v1/tools` response, `web_search_tools.py::TOOL_LIST`, and
`config/agent.toml` (tool definition + safety tier); `tool_constants.py` gains a routing-model
comment matching its sibling modules (`tool_registry.py`, `route_resolver.py`); new automated tests
close the four drift gaps the requirement calls out (registry-vs-`/v1/tools`, missing safety tier,
schema divergence, TOOL_LIST-vs-`WEB_SEARCH_TOOLS`); and documentation is corrected wherever it
already diverges from actual behavior — with no routing-behavior change except the one concrete
schema inconsistency this planning cycle already found (see Design §2).

## Scope

**In scope:**
- `scripts/shared/tool_constants.py` — add a routing-model comment above `WEB_SEARCH_TOOLS` (or at
  module top) stating: ToolRegistry (`tool_registry.py`) is the routing authority; live `/v1/tools`
  discovery is drift-validation only (`McpToolDiscoveryService` / `tool_routing_validation.py`); this
  module only supplies the frozenset seed data. Match the phrasing already used in
  `tool_registry.py`'s and `route_resolver.py`'s module docstrings so all three are consistent.
- New test `tests/test_tool_safety_tiers.py` — verifies every tool in `tool_constants.py`'s frozensets
  (starting with `search_web`, generalizable to all tools) has a `[tool_safety_tiers]` entry in
  `config/agent.toml`, and that no `[tool_safety_tiers]` entry references an unknown tool name.
- New test `tests/test_web_search_tool_schema.py` — verifies `search_web`'s
  `config/agent.toml` `[[tool_definitions]]` parameter set is a subset of (or matches, per the fix in
  Design §2) `web_search_tools.py::TOOL_LIST`'s `inputSchema.properties`, catching the schema
  divergence already found (`max_results` present in `TOOL_LIST` but absent from `agent.toml`).
- Extend `tests/test_tool_server_layer_consistency.py` (or add a focused new test) to cover
  registry-vs-live-`/v1/tools` drift for `search_web`, using the existing (but currently unwired)
  `shared/tool_routing_validation.py::validate_routing_against_live` — see Design §3 for whether this
  is added as a unit test against a stubbed discovery map or deferred.
- `config/agent.toml` — add `max_results` (optional integer) to `search_web`'s
  `[tool_definitions.function.parameters]` to close the schema-divergence gap (see Design §2). This is
  the one behavior-adjacent change in this plan, justified because a new test would otherwise fail
  and because `docs/04_mcp_04_01_web-search-file-read-github.md` already documents `max_results` as
  part of the tool's public input contract.
- Documentation: `docs/04_mcp_07_tool_schema_export_policy.md` (lines 31–39) — its per-server
  canonical-filename table still lists bare `tools.py`/`server.py` for `web_search` (and 7 other
  servers) without noting the `ca6b7bfe` rename left both old and new files on disk; add a short
  cross-reference to the already-tracked `issues/20260719-193357_risks.md` instead of re-describing
  the duplicate-file problem here (avoids duplicate tracking, per Assumption 3).
- `docs/04_mcp_03_02_tool-registry.md` — confirm/update wording so it explicitly names `search_web`
  as a worked example of the registry/live-discovery split, if the doc's existing generic explanation
  is judged insufficient at implementation time (see UNK-01).

**Out of scope:**
- Deleting the orphaned duplicate modules (`models.py`/`tools.py`/`server.py` vs
  `web_search_models.py`/`web_search_tools.py`/`web_search_server.py`) — already the explicit subject
  of `plans/20260719-195330_plan.md` (requirement 24) and tracked in
  `issues/20260719-193357_risks.md`. This plan only notes where that split affects consistency
  testing (Assumption 3) and does not re-plan the cleanup.
- Wiring `validate_all_routing()` / `validate_routing_against_live()` into `agent/startup.py` as a
  live startup check — these functions exist in `shared/tool_routing_validation.py` but are not
  called anywhere in `scripts/` today (confirmed via `rg`); making them a startup gate is a larger
  behavior change than this "guardrail and documentation cleanup" requirement calls for. This plan
  only adds a test that exercises them directly (see Design §3), it does not wire them into
  production startup.
- Any change to `search_provider.py`, `formatters.py`, `service`/`health`/`metrics` layering, error
  classification, or timeout behavior — those belong to the sibling requirements 21–24 whose plans
  already exist (`plans/20260719-192933_plan.md`, `plans/20260719-193357_plan.md`,
  `plans/20260719-193902_plan.md`, `plans/20260719-195330_plan.md`).
- Any of the other 8 MCP servers' registry/safety-tier consistency — this plan is `search_web`-scoped
  only, per the requirement.

## Assumptions

1. "Runtime request schema" in the requirement's acceptance criteria refers to
   `web_search_tools.py::TOOL_LIST`'s `inputSchema` (the schema actually sent to the LLM via
   `/v1/tools` and used for MCP tool dispatch), not `SearchRequest` the Pydantic model — falsifiable
   by checking `MCPServer.list_tools_with_server_key()` in `scripts/mcp_servers/server.py`, which
   serves `/v1/tools` directly from `TOOL_LIST` (confirmed in Step 3 via
   `docs/04_mcp_07_tool_schema_export_policy.md` line 51's description of that method).
2. `config/agent.toml`'s `[tool_safety_tiers]` table (line 209 onward) is the sole source of truth for
   safety tiers — no `SAFETY_TIER` constant exists in Python (`rg -rn "safety_tier|SAFETY_TIER"
   scripts/` returns nothing outside `agent.toml` parsing code) — falsifiable by that same grep.
3. Because the `models.py`/`tools.py` vs `web_search_*.py` duplicate-file split (from `ca6b7bfe`) is
   already being resolved by a separate, already-existing plan (`plans/20260719-195330_plan.md`), any
   new test this plan adds should import from whichever module `config/agent.toml`'s `cmd` line
   actually deploys today (`scripts/mcp_servers/web_search/server.py`, importing old-named
   `tools.py`), not the aspirational `web_search_tools.py`, so the test validates the real deployed
   artifact — falsifiable by `config/agent.toml:316`'s `cmd` entry (confirmed unchanged as of Step 3).
   Note `tests/test_tool_server_layer_consistency.py:179` already imports the new
   `web_search_tools.py` instead; since both files are byte-identical today this causes no test
   failure, but it is a latent false-pass risk this plan's new tests should not repeat (see Risk 3).
4. The `max_results` addition to `agent.toml`'s `search_web` tool definition (Design §2) is a safe,
   backward-compatible, optional-parameter addition — falsifiable by confirming
   `web_search_tools.py::TOOL_LIST`'s `max_results` is not in the `required` array (confirmed in Step
   3: only `query` is required) and that `SearchRequest` (Pydantic) already defaults it.
5. Single bus factor: `git blame` on `scripts/shared/tool_constants.py` shows one contributor across
   three name-casing variants (`Masao Sugimoto` / `Masao SUGIMOTO` / `Sugimoto Masao`) — no
   bus-factor mitigation needed.

## Design

1. **`tool_constants.py` comment update** — insert, directly above `WEB_SEARCH_TOOLS` (or as a short
   module-level note near the top, consistent with how the file already groups tool sets by server),
   a comment such as: `# Routing authority: shared.tool_registry.ToolRegistry (populated from these`
   `# frozensets). Live /v1/tools discovery (agent.services.mcp_tool_discovery) is drift-validation`
   `# only — see shared.tool_registry and shared.route_resolver module docstrings.` This makes
   `tool_constants.py` state the same model its two sibling modules already state, closing the
   ambiguity the requirement's "Reason for change" section describes.
2. **Schema divergence fix** — `web_search_tools.py::TOOL_LIST`'s `inputSchema` (and
   `docs/04_mcp_04_01_web-search-file-read-github.md`'s tool table) both describe `search_web` as
   accepting `{query: str, max_results?: int}`, but `config/agent.toml`'s `[[tool_definitions]]` entry
   for `search_web` (lines 402-413) currently defines only `query`. This is a real, verifiable
   inconsistency (not a hypothetical one), so per the requirement's acceptance criterion "No runtime
   behavior changes unless an actual inconsistency is found," this plan adds the missing
   `[tool_definitions.function.parameters.properties.max_results]` block (`type = "integer"`,
   optional, not in `required`) so the LLM-facing definition matches the MCP server's actual accepted
   schema. This is additive (a new optional parameter) and does not change existing behavior for
   callers that omit it.
3. **Registry-vs-live-`/v1/tools` drift test** — add a test using
   `shared.tool_routing_validation.validate_routing_against_live()` with a stubbed
   `live_tool_lists={"web_search": ["search_web"]}` (matching real shape) plus a deliberately-broken
   variant (e.g. `{"web_search": ["other_tool"]}`) to assert the function reports `search_web` as
   missing from live discovery when the stub omits it. This exercises the currently-untested function
   directly without requiring the web_search HTTP server to be running (per UNK-02's resolution).
4. **Safety-tier test** — `tests/test_tool_safety_tiers.py` parses `config/agent.toml`'s
   `[tool_safety_tiers]` table (same TOML-loading approach as existing config tests) and asserts every
   tool name returned by `tool_registry.ToolRegistry`'s populated registry (or, at minimum, every name
   in each `tool_constants.py` frozenset) has a corresponding entry, and that `search_web` in
   particular resolves to `"READ_ONLY"`.
5. **Schema-consistency test** — `tests/test_web_search_tool_schema.py` loads
   `config/agent.toml`'s `search_web` tool definition and `web_search_tools.py::TOOL_LIST`'s
   `inputSchema`, and asserts the `agent.toml` parameter set is a subset of the `TOOL_LIST` parameter
   set (post-fix, this will be an equality check since both will have `query`+`max_results`). This
   test is written to fail against pre-fix `agent.toml` (proving it actually catches the drift class
   the requirement describes) and to pass after Design §2's fix is applied.
6. **Documentation cross-reference** — rather than re-explaining the `ca6b7bfe` duplicate-file
   situation in `docs/04_mcp_07_tool_schema_export_policy.md`, add one line near its per-server table
   pointing to `issues/20260719-193357_risks.md` (already-tracked) so a future reader does not treat
   the table's old-named files as unambiguously canonical without knowing about the tracked
   discrepancy. Per `rules/coding.md`'s "Current behavior" classification table, this is the "Issue
   already tracked" category — cross-reference, do not duplicate.
