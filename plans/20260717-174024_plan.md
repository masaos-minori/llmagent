# Plan: Add runtime availability metadata (enabled / disabled_reason) to /v1/tools

Source requirement: `requires/20260717_15_require.md` (moved to `requires/done/` on
completion of this workflow)

## Goal

Make every MCP server's `GET /v1/tools` response distinguish "implemented" from
"currently executable" by adding two fields — `enabled: bool` and
`disabled_reason: str` — to every tool dict returned, computed at request time from
each server's live config state (`_cfg`). Disabled tools must still be listed (never
omitted), so the Agent's future `RuntimeToolRegistry` can tell "not implemented" apart
from "implemented but disabled".

**In scope** (matches requirement's Target files, corrected to the real
`scripts/`-prefixed paths — verified all four exist):

- `scripts/mcp_servers/file/write_server.py` — `/v1/tools` handler (line 144-148)

  `tests/mcp_servers/test_tools_endpoint.py` do not exist; see Assumptions/Unknowns).

Rules to implement, verbatim from the requirement:

- File servers (read/write/delete): `_cfg.allowed_dirs` empty -> every tool
  `enabled: false, disabled_reason: "allowed_dirs is empty"`; non-empty -> every tool
  `enabled: true, disabled_reason: ""`.
- Git server: `_cfg.allowed_repo_paths` empty -> every tool disabled with
  `"allowed_repo_paths is empty"` (this takes precedence over the read_only check).
  Otherwise, if `_cfg.read_only` is true, only the 5 write tools
  (`git_add`, `git_commit`, `git_checkout`, `git_pull`, `git_push`) are disabled with
  `"read_only=true"`; all other git tools stay enabled with `disabled_reason: ""`.

## Design

- Do not mutate `TOOL_LIST` (module-level constant, loaded once at import time and
  shared by `MCPServer.mcp_tools` for other purposes). Instead, compute
  `enabled`/`disabled_reason` inside each `/v1/tools` handler at request time, reading
  the module's existing `_cfg` object — same lifecycle as the already-working
  `server_key` injection.
- File servers (read/write/delete) share identical logic:
  ```python
  @app.get("/v1/tools")
  async def list_tools() -> dict[str, Any]:
      enabled = bool(_cfg.allowed_dirs)
      reason = "" if enabled else "allowed_dirs is empty"
      return {
          "tools": [
              {**t, "server_key": "file_read", "enabled": enabled, "disabled_reason": reason}
              for t in TOOL_LIST
          ],
      }
  ```
  (`server_key` value differs per server: `file_read` / `file_write` / `file_delete`.)
  This is a 3-line change per handler; no new helper function needed since the logic
  is a single boolean check reused identically in three files — extracting a shared
  helper (e.g. into `scripts/mcp_servers/file/common.py`, which all three already
  import from) is a reasonable optional simplification but not required to satisfy the
  requirement's acceptance criteria. Given the low duplication (3 lines x 3 files), and
  to keep the change minimal/low-risk, this plan proposes a small shared helper in
  `scripts/mcp_servers/file/common.py`:
  ```python
  def availability_flags(allowed_dirs: list[str]) -> tuple[bool, str]:
      """Return (enabled, disabled_reason) for a file-{read,write,delete} server."""
      if allowed_dirs:
          return True, ""
      return False, "allowed_dirs is empty"
  ```
  and each server's handler calls `enabled, reason = availability_flags(_cfg.allowed_dirs)`.
  This avoids drift across the three near-identical handlers while staying inside a
  file all three already depend on (no new import edges, no `.importlinter` risk).
- Git server logic (single tool loop, precedence rule respected — `allowed_repo_paths`
  empty wins over `read_only`):
  ```python
  from shared.tool_constants import GIT_WRITE_TOOLS

  @app.get("/v1/tools")
  async def list_tools() -> dict[str, list[dict[str, object]]]:
      tools: list[dict[str, object]] = []
      for t in TOOL_LIST:
          if not _cfg.allowed_repo_paths:
              enabled, reason = False, "allowed_repo_paths is empty"
          elif _cfg.read_only and t["name"] in GIT_WRITE_TOOLS:
              enabled, reason = False, "read_only=true"
          else:
              enabled, reason = True, ""
          tools.append({**t, "server_key": "git", "enabled": enabled, "disabled_reason": reason})
      return {"tools": tools}
  ```
  `mcp_servers.git.server` importing `shared.tool_constants.GIT_WRITE_TOOLS` is
  layer-compliant (`mcp_servers -> db, shared` per `.importlinter`).
- Reason strings use the exact standard values from the requirement:
  `"allowed_dirs is empty"`, `"allowed_repo_paths is empty"`, `"read_only=true"`.
  These are hardcoded string literals at the three/four call sites — no new constants
  module is introduced since the requirement doesn't ask for a shared reason-string
  registry and only 3 distinct strings are needed in this scope.

## Implementation steps

1. Add `availability_flags(allowed_dirs: list[str]) -> tuple[bool, str]` to
   `scripts/mcp_servers/file/common.py` (all three file servers already import from
   this module).
2. Update `scripts/mcp_servers/file/read_server.py`'s `list_tools()` (lines 234-238)
   to call `availability_flags(_cfg.allowed_dirs)` and merge `enabled`/`disabled_reason`
   into each returned tool dict.
3. Apply the same change to `scripts/mcp_servers/file/write_server.py` (lines 144-148)
   and `scripts/mcp_servers/file/delete_server.py` (lines 106-110).
4. Update `scripts/mcp_servers/git/server.py`'s `list_tools()` (lines 72-76): import
   `GIT_WRITE_TOOLS` from `shared.tool_constants`, and rewrite the tool loop to apply
   the precedence rule (`allowed_repo_paths` empty check first, then `read_only` +
   write-tool-membership check, else enabled).
5. Add tests to `tests/test_mcp_tools_validation.py` (or a clearly-named sibling file
   if it grows too large) using `TestClient` against each server's `app`, with
   `monkeypatch.setattr(<server_module>, "_cfg", <Config instance>)` to cover:
   - file server, `allowed_dirs=[]` -> all tools `enabled=False`,
     `disabled_reason="allowed_dirs is empty"`.
   - file server, `allowed_dirs=["/tmp"]` -> all tools `enabled=True`,
     `disabled_reason=""`.
   - git server, `allowed_repo_paths=[]` -> all tools disabled with
     `"allowed_repo_paths is empty"`, regardless of `read_only`.
   - git server, `allowed_repo_paths=["/tmp"], read_only=True` -> only the 5 write
     tools disabled with `"read_only=true"`; read tools enabled with `""`.
   - git server, `allowed_repo_paths=["/tmp"], read_only=False` -> all tools enabled
     with `""`.
   - Every tool in every response has boolean `enabled` and string `disabled_reason`
     (schema-shape assertion, matches Acceptance criteria literally).
6. Run the Validation plan below.
7. `grep -rn '"enabled"\|disabled_reason' scripts/mcp_servers/` to confirm the new
   fields appear only in the 4 intended `server.py` files (plus the new
   `common.py` helper) and nowhere unexpected.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Precedence rule (`allowed_repo_paths` empty beats `read_only=true`) implemented backwards, silently disabling read tools too when both conditions hold | Medium (easy to swap `if`/`elif` order) | Test suite (Implementation step 5) explicitly asserts read tools stay enabled when `allowed_repo_paths` is non-empty and `read_only=True`, and asserts ALL tools (including reads) are disabled when `allowed_repo_paths=[]` even with `read_only=False` — this pins the precedence order as a regression gate |
| Divergent copy-pasted logic across read/write/delete `server.py` handlers drifts over time (e.g. one gets fixed for a future disabled-reason and the others don't) | Medium (3 near-identical call sites) | Extracted `availability_flags()` into `scripts/mcp_servers/file/common.py` (Implementation step 1) so all three call one function instead of duplicating the boolean check three times |
| `disabled_reason` string literal typo'd differently from the requirement's exact standard value (e.g. `"read_only is true"` vs `"read_only=true"`) breaks Agent-side parsing if it does exact-string matching | Low-medium | Tests assert exact string equality (not substring/regex) against the requirement's literal values (`"allowed_dirs is empty"`, `"allowed_repo_paths is empty"`, `"read_only=true"`) |
| Existing callers of `GET /v1/tools` (if any exist today, e.g. a future `RuntimeToolRegistry` client or the `test_v1_tools_each_has_name_and_description` test) treat the response as a fixed shape and choke on two new keys | Low | Response is a strict superset (existing keys `name`/`description`/`inputSchema`/`status`/`requires_config`/`server_key` are untouched); confirmed no test in `tests/test_mcp_tools_validation.py` asserts an exact/closed key set for `/v1/tools` tool dicts (only presence checks on `name`/`description`) |
| Merge friction with the concurrently-planned `plans/20260717-173602_plan.md` (`requires_config` -> `config_dependent` rename) touching the same `tools.py`/`server.py` files in the same area of the codebase | Low (no line-level overlap, per Scope's SHARED WITH note) | Apply the two plans as separate, sequential commits and re-run `git diff` before each commit to confirm no accidental clobbering; both plans' Validation plans independently re-run the full test suite, which would catch a bad merge |
