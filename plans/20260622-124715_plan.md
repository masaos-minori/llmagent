## Assumptions

1. `resolve_safe()` は `Path(raw_path).resolve()` でシンボリックリンクを解決し、
   `relative_to(allowed.resolve())` で allowed_dirs 外を `FileAuthorizationError` で弾く。
2. `/etc/passwd` は Linux 環境に存在するが、`/proc/self/environ` も同様。
   テスト環境がこれらのパスを持たない場合でも `resolve_safe()` は `FileAuthorizationError` を投げる
   (パスが allowed_dirs 外であることには変わりない)。
3. `../../etc/shadow` は `Path.resolve()` で絶対パスに正規化されるため、
   path traversal 攻撃は `resolve()` で無効化される。
4. シンボリックリンクテストは `tmp_path` フィクスチャで allowed_dirs 外へのリンクを作成して実施。

## Design

`resolve_safe()` のセキュリティ動作:
```
Path(raw_path).resolve()  # symlink 解決 + 絶対パス正規化
    ↓
resolved.relative_to(allowed.resolve())  # allowed_dirs 外なら ValueError → FileAuthorizationError
```

追加するテストケース:
1. `/proc/self/environ` — Linux proc filesystem; allowed_dirs 外
2. `../../etc/shadow` (tmp_path からの相対) — path traversal; `resolve()` で絶対パス化後に弾かれる
3. シンボリックリンク: allowed_dirs 外を指すシンボリックリンク → FileAuthorizationError
4. シンボリックリンク: allowed_dirs 内を指すシンボリックリンク → 正常に解決

**除外: null byte パス** — Python 3.13 では `Path("\x00").resolve()` が `ValueError` を投げるが
`resolve_safe()` は `except OSError` でキャッチするため `ValueError` がそのまま伝播する。
`FileValidationError` に変換されない実装の問題であり、別 issue として管理する。
