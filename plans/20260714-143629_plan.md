## Design

### Actual workflow approval gate mechanism (confirmed via source inspection):

1. **Trigger condition**: `WorkflowDef.require_approval = true` in the workflow JSON definition (e.g., `config/workflows/default.json`)
2. **Gate insertion**: Between execute stage and verify stage in `WorkflowEngine.run()` (L129: `if self._wdef.require_approval:`)
3. **State persistence**: Approval records written to `workflow.sqlite` → `approvals` table via `request_approval()` in `approval_ops.py`
4. **Resolution**: `/approve <approval_id>` or `/reject <approval_id>` commands update the approval record status
5. **Default state**: Default workflow has no `require_approval` field (defaults to `False`), so gates don't fire in standard deployment

### Key distinction from old design:

Old design: `AgentConfig.workflow_require_approval = true` enabled workflow-level approval globally.
Current design: Each workflow definition independently controls whether its own gates fire via `require_approval` field.
