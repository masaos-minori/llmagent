# Plan: Add schema tests for MCP runtime availability metadata

Source requirement: `requires/20260717_18_require.md` (moved to `requires/done/` on
completion of this workflow)

## Goal

Add automated tests that validate the shape and invariants of the `enabled` /
`disabled_reason` / `config_dependent` runtime-availability metadata that sibling
requirements 14-16 are adding to the file (read/write/delete) and git MCP servers'
`/v1/tools` responses ā€” covering both static per-tool schema fields
(`name`, `description`, `inputSchema`, `status`, `config_dependent`) and the
request-time-computed `enabled`/`disabled_reason` pair, in both enabled and disabled
server states, so malformed metadata cannot silently reach Agent-side tool discovery.

## Scope

**In scope** (matches requirement's Target files, corrected to real `scripts/`-prefixed
paths and real `tests/` flat layout ā€” the requirement's own
`tests/mcp_servers/test_tool_schema.py` / `tests/mcp_servers/test_tools_endpoint.py`
paths do not exist as a directory; see Assumptions):

- New `tests/test_tool_schema.py` ā€” static schema validation of each server's
  `TOOL_LIST` (`mcp_servers.file.read_tools`, `.write_tools`, `.delete_tools`,
  `mcp_servers.git.tools`): required keys/types (`name`, `description`, `inputSchema`,
  `status`, `config_dependent`), and the "`requires_config` must not be present" rule.
- New `tests/test_tools_endpoint.py` ā€” endpoint-level tests (FastAPI `TestClient`)
  against each of the four servers' live `GET /v1/tools` handler, asserting the merged
  response for every tool includes `enabled: bool` and `disabled_reason: str`, and the
  enabled/disabled_reason correlation rule (`enabled=True` -> `disabled_reason == ""`;
  `enabled=False` -> `disabled_reason != ""`). Covers both enabled and disabled server
  states per the requirement's acceptance criteria (file server enabled/disabled; git
  server enabled, and disabled via both empty allowlist and read-only mode).
- Read-only, no production code changes.

**Out of scope**:

- Implementing `config_dependent` (requirement 14, `plans/20260717-173602_plan.md`)
  or `enabled`/`disabled_reason` (requirement 15, `plans/20260717-174024_plan.md`) in
  production code ā€” verified neither exists yet (see Assumptions). This plan writes
  tests against the contract those two requirements define; the tests cannot pass
  until that production code lands (see Risks).
- `/v1/call_tool` disabled-tool rejection (requirement 16,
  `plans/20260717-174848_plan.md`) and its own new test file
  `tests/test_call_tool_validation.py` ā€” that covers the *call* path; this plan covers
  only the *listing* path (`/v1/tools`).
- Agent-side `RuntimeToolRegistry` consumption of these fields (requirement 17,
  `plans/20260717-175327_plan.md`) ā€” out of this requirement's Target files entirely.
- `shell`, `cicd`, `github`, `rag_pipeline`, `mdq`, `web_search` servers ā€” not named in
  the requirement's Target files; requirements 14-16 likewise scope only file+git.

**SHARED WITH `plans/20260717-174024_plan.md`** (requirement 15) and
**`plans/20260717-174848_plan.md`** (requirement 16): both of those plans already
commit to writing their own `TestClient` + `monkeypatch.setattr(<module>, "_cfg", ...)`
tests exercising the same enabled/disabled server states (empty `allowed_dirs`,
`allowed_repo_paths` empty, `read_only=True`) against the same four servers, for their
own purposes (validating `/v1/tools` field computation in 15; validating
`/v1/call_tool` rejection in 16). This plan's `tests/test_tools_endpoint.py` targets
the same state-construction pattern for the same servers, but asserts on `/v1/tools`
response shape specifically. To avoid three near-duplicate state-construction helpers,
whichever of {15, 18} lands first should factor the `monkeypatch.setattr(module,
"_cfg", ...)` fixture/helper so the other can reuse it (mirrors the same
whichever-lands-first precedent requirement 16 already set with requirement 15 for
`availability_flags()`/`_git_tool_availability()`).

## Assumptions

1. `tests/mcp_servers/` does not exist anywhere in the repo (verified via `find`); the
   requirement's two named test paths are aspirational shorthand. This plan follows
   the existing flat `tests/` layout (same precedent used by requirements 14 and 16's
   plans) and creates `tests/test_tool_schema.py` and `tests/test_tools_endpoint.py`.
2. None of `config_dependent`, `enabled`, `disabled_reason` exist in `scripts/` today ā€”
   verified via `grep -rn "config_dependent\|disabled_reason" scripts/` (zero matches,
   excluding `__pycache__`). Every current `TOOL_LIST` entry (confirmed by reading
   `scripts/mcp_servers/file/read_tools.py`) still has the old `"requires_config"` key
   (e.g. lines 54/61/68/75/82/89/96/103/110), not `config_dependent`. This confirms
   requirements 14 and 15 are still plan-only, not implemented ā€” this is the central
   dependency risk for this plan (see Risks).
3. `enabled`/`disabled_reason` are designed (per requirement 15's plan,
   `plans/20260717-174024_plan.md`) to be computed **at request time** inside each
   server's `GET /v1/tools` handler (from live `_cfg` state) and merged into the
   response dict ā€” they are *not* stored statically in `TOOL_LIST`. Confirmed by
   reading `scripts/mcp_servers/file/read_server.py:234-238` and
   `scripts/mcp_servers/git/server.py:72-76`: today's handlers only do
   `{**t, "server_key": ...}` for `t in TOOL_LIST}` ā€” no per-request config lookup yet.
   Consequence: static-schema testing (`tests/test_tool_schema.py`, importing
   `TOOL_LIST` directly, no server process) can only meaningfully cover
   `name`/`description`/`inputSchema`/`status`/`config_dependent`; `enabled`/
   `disabled_reason` require the live endpoint (`tests/test_tools_endpoint.py`), since
   they don't exist on the module-level `TOOL_LIST` object at all.
4. Each of the four servers exposes a module-level `_cfg` object that can be
   monkeypatched to construct enabled/disabled states, confirmed by direct read:
   `_cfg = FileReadConfig.load()` (`read_server.py:66`), `FileWriteConfig.load()`
   (`write_server.py:53`), `FileDeleteConfig.load()` (`delete_server.py:47`),
   `GitConfig.load()` (`git/server.py:41`). Config fields confirmed:
   `FileReadConfig.allowed_dirs: list[str]` (`read_models.py:28`),
   `GitConfig.allowed_repo_paths: list[str]` / `read_only: bool = True`
   (`git/models.py:27-28`). Same fixture pattern requirement 15's and 16's plans both
   already commit to.
5. No existing plan/implementation already writes these two test files ā€” checked
   `plans/`, `plans/done/`, `implementations/`, `implementations/done/` for
   `test_tool_schema`, `test_tools_endpoint`, `config_dependent`, `disabled_reason`:
   only the four sibling plans (14/15/16/17) reference the underlying fields, none
   creates a dedicated schema-validation test file. No direct duplicate-effort overlap
   found, only the shared-fixture overlap noted above under "SHARED WITH".
6. `status` field's only observed value today is `"production"` (confirmed via read of
   `read_tools.py`); the requirement only requires `status` be a non-empty string, not
   a specific enum, so the test asserts type/non-emptiness only, not a fixed value set.

## Design

- **`tests/test_tool_schema.py`** (fast, no server process, mirrors
  `tests/test_mcp_tool_schema_exports.py`'s `_TOOL_MODULES` parametrization):
  - `_SCHEMA_MODULES = [("mcp_servers.file.read_tools", "TOOL_LIST"), ("mcp_servers.file.write_tools", "TOOL_LIST"), ("mcp_servers.file.delete_tools", "TOOL_LIST"), ("mcp_servers.git.tools", "TOOL_LIST")]`
  - For every tool dict in every listed module: assert `name` is a non-empty `str`;
    `description` is a non-empty `str`; `inputSchema` is a `dict`; `status` is a
    non-empty `str`; `config_dependent` is present and `isinstance(..., bool)`;
    `"requires_config" not in tool`.
  - These assertions are written against the **post-requirement-14** contract, so they
    will fail (missing `config_dependent`, stray `requires_config`) until requirement
    14 lands ā€” this is intentional/expected per the Risks mitigation below, not a bug
    in the test.
- **`tests/test_tools_endpoint.py`** (uses `TestClient`, monkeypatches `_cfg`):
  - File servers (read/write/delete), parametrized over the three modules: build a
    `TestClient(app)`, `monkeypatch.setattr(<server_module>, "_cfg", <ConfigClass>(allowed_dirs=[]))`
    for the disabled case and `allowed_dirs=["/tmp"])` for the enabled case; `GET
    /v1/tools`; for every tool in the response assert `enabled` and `disabled_reason`
    are present with correct types, and the correlation rule holds
    (`enabled=False` <=> non-empty `disabled_reason`, `enabled=True` <=> empty
    `disabled_reason`).
  - Git server: three states ā€” `allowed_repo_paths=[]` (all tools disabled,
    `disabled_reason` mentions the empty-allowlist condition), `allowed_repo_paths=["/tmp"], read_only=True`
    (only the 5 write tools disabled, reads stay enabled), `allowed_repo_paths=["/tmp"], read_only=False`
    (all tools enabled). This satisfies the requirement's explicit acceptance
    criterion "disabled git states for empty allowlist and read-only mode" as two
    distinct disabled-state test cases.
  - Same correlation assertion (`enabled` <=> `disabled_reason` non-emptiness) applied
    uniformly across all endpoint responses.
- Both files assert on the *response contract*, not on internal helper functions
  (`availability_flags()` / `_git_tool_availability()`) that requirements 15/16 may or
  may not have factored out yet ā€” this keeps the tests stable regardless of which
  internal implementation detail those plans choose, per requirement 18's own framing
  ("prevent invalid runtime availability metadata from entering Agent discovery",
  i.e. contract-level, not implementation-level).

## Implementation steps

1. Before writing any test code, `grep -rn "config_dependent\|disabled_reason" scripts/mcp_servers/` to re-confirm requirements 14 and 15 have landed in production code. **If either is still absent, do not proceed to write assertions that require it as passing** ā€” instead implement the tests but mark the assertions that depend on the missing field with `pytest.mark.xfail(reason="depends on requirement 14/15 landing", strict=True)` at the parametrize-case level, so the test suite stays green today and automatically flips to a hard failure (via `strict=True`) the moment the dependency lands with a differently-shaped contract, or to a pass once it lands correctly. Remove the `xfail` marker once both dependencies are confirmed implemented and the assertions pass unconditionally.
2. Add `tests/test_tool_schema.py` per the Design section's static-schema checks, parametrized over the four file/git `TOOL_LIST` modules.
3. Add `tests/test_tools_endpoint.py` per the Design section's `TestClient` + `monkeypatch(_cfg)` checks, covering: file-server enabled state, file-server disabled state, git-server enabled state, git-server disabled-via-empty-allowlist state, git-server disabled-via-read-only state (5 parametrized cases total, matching the requirement's acceptance criteria one-to-one).
4. Cross-check against `plans/20260717-173602_plan.md` (requirement 14)'s own planned test addition to `tests/test_mcp_tools_validation.py` ā€” if that lands first with a similar `config_dependent`-presence assertion, keep both (they are not contradictory, just overlapping coverage) but note in a code comment in `tests/test_tool_schema.py` that `tests/test_mcp_tools_validation.py` also covers a subset of this, to avoid a future contributor being confused about which file is authoritative.
5. Run the Validation plan below.
6. Once requirements 14 and 15 have landed and all `xfail` markers from step 1 have been removed, re-run the full suite once more to confirm the tests pass unconditionally (not just because `xfail` masked a failure).

## Risks

| Implementation order: if this plan is implemented before requirement 14 (rename) or 15 (add fields), and the `xfail` markers are removed prematurely or written without `strict=True`, the test suite silently reports false confidence | Medium | Step 6 explicitly requires a final unconditional run after both dependencies land, and the Validation plan's residual grep check for lingering `xfail` markers is a hard gate before considering this requirement fully done |
| Duplicate/overlapping `TestClient` + `monkeypatch(_cfg)` state-construction logic across this plan, requirement 15's plan, and requirement 16's plan, if all three are implemented independently without checking each other first | Medium (explicitly the reason for the "SHARED WITH" note in Scope) | Whichever of {15, 18} lands first should factor the state-construction fixture so the other reuses it (same precedent requirement 16 already set for `availability_flags()`); a post-implementation repo-wide `grep -n "monkeypatch.setattr.*_cfg"` across all three files should show a shared helper, not three independent re-derivations |
| `tests/test_tool_schema.py`'s coverage overlaps with requirement 14's own planned `config_dependent`-presence assertion in `tests/test_mcp_tools_validation.py`, risking two near-duplicate tests maintained independently and drifting apart | Low | Implementation step 4 requires an explicit cross-check and a code comment cross-referencing the other file; both assertions passing is not harmful (belt-and-suspenders), only silent duplication-without-awareness would be a maintenance risk |
| Git server's three-state parametrization (empty allowlist, read-only, fully enabled) could accidentally test the wrong precedence order (e.g. asserting read-only disables reads too, when the requirement says only writes are disabled under read-only) | Low | Design section explicitly states the expected per-state tool sets (all disabled for empty allowlist; only the 5 write tools disabled for read-only); the test itself pins this as a named assertion per state, catching an inverted precedence immediately as a test failure rather than silently accepting either behavior |
