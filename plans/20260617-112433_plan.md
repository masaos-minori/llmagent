# Plan: Req 31 — Make workflow engine mode explicit instead of implicit fallback

## Goal

Replace the implicit workflow activation logic (file existence + DB availability) with an explicit `workflow_mode` configuration enum (`auto`, `required`, `disabled`). Report the active mode at startup. Log fallback conditions clearly when in `auto` mode. Remove the open question from documentation guide.

## Scope

### In scope
- `agent/config_dataclasses.py`: Add `WorkflowConfig` dataclass with `mode: str` field (default `"auto"`), validation
- `agent/orchestrator.py`: Replace implicit `_workflow_def is None` and `StateStore` failure logic with explicit mode-driven behavior
- `agent/repl.py` / `agent/cli_view.py`: Report workflow mode at startup
- `docs/05_agent_90_inconsistencies_and_known_issues.md`: Resolve OQ-02 (workflow fallback conditions)
- `tests/test_orchestrator.py`: Add tests for each workflow mode

### Out of scope
- Changes to `agent/workflow/` package internals (models, engine, loader, state_store)
- New config file format — uses existing TOML merge pipeline via `AgentConfig` composition
- Migration of existing configs — default `"auto"` preserves current behavior
- Workflow mode hot-reload — only applied at startup

## Assumptions
- Three modes suffice: `auto` (current implicit behavior), `disabled` (always direct execution), `required` (hard error if workflow unavailable)
- Default mode is `"auto"` to preserve backward compatibility with existing deployments
- The mode is read from a new `workflow_mode` field in `AgentConfig` (not nested under a sub-config, keeping it simple)
- Startup reporting uses the existing `_print_startup_banner()` infrastructure or `_check_services()` warning path

## Unknowns (resolved through analysis)

| Unknown | Resolution |
|---------|------------|
| Where to place workflow_mode in config hierarchy? | Add as top-level field on `AgentConfig` (not nested). Workflow mode is a cross-cutting concern that affects the Orchestrator directly, not a domain-specific sub-config. |
| How does existing config pipeline handle new fields? | `AgentConfig` uses dataclass defaults; TOML keys map to dataclass fields via `config_builders.py`. New field with default="" preserves backward compatibility. |
| What log level for fallback in auto mode? | `logger.warning()` — operators should be aware workflow tracking is disabled, but it's not an error. |
| Does required mode halt on StateStore failure too? | Yes — both workflow JSON absence AND state_store unavailability are fatal in `required` mode. |
| Should startup banner show workflow status? | Yes, alongside existing "DB: X chunks | Tools: N" line. Format: "Workflow: auto (tracking enabled)". |

## Design

### New config field
```python
@dataclass
class AgentConfig:
    # ... existing fields ...
    workflow_mode: str = "auto"  # "auto" | "required" | "disabled"

    def __post_init__(self):
        self._validate_workflow_mode()

    def _validate_workflow_mode(self) -> None:
        valid = {"auto", "required", "disabled"}
        if self.workflow_mode not in valid:
            raise ValueError(
                f"workflow_mode must be one of {valid}, got {self.workflow_mode!r}"
            )
```

### Orchestrator mode-driven logic
```python
class Orchestrator:
    def __init__(self, ctx, ..., workflow_mode: str = "auto"):
        self._workflow_mode = workflow_mode
        self._workflow_def: WorkflowDef | None = None
        # ... existing loader logic unchanged ...

    async def handle_turn(self, line):
        if self._workflow_mode == "disabled":
            logger.info("Workflow mode=disabled — direct execution")
            return await self._process_turn_direct(line)

        # auto or required: attempt workflow
        if self._workflow_def is None:
            self._log_fallback("workflow definition not loaded")
            return await self._process_turn_direct(line)

        try:
            store = StateStore()
        except RuntimeError as exc:
            self._log_fallback(f"state_store unavailable: {exc}")
            if self._workflow_mode == "required":
                raise RuntimeError("Workflow mode=required but state_store unavailable") from exc
            return await self._process_turn_direct(line)

        # ... existing workflow execution ...
```

## Risks

| Risk | Mitigation |
|------|-----------|
| Default `"auto"` preserves current implicit behavior — operators won't notice change | Startup banner explicitly reports workflow mode. Migration guide in docs if needed. |
| `required` mode could break existing deployments that rely on fallback | Default is `"auto"`; operators must explicitly set `"required"` to get hard error behavior. |
| Existing tests mock WorkflowLoader silently — adding mode parameter changes constructor signature | Add default value `workflow_mode: str = "auto"` to constructor; existing test code continues to work without changes. |
| Startup reporting adds complexity to `_print_startup_banner()` | Keep it simple: single line appended after existing banner. Use existing warning path if workflow unavailable in auto mode. |
| Config loading may not populate `workflow_mode` from existing TOML files | Default `"auto"` ensures backward compatibility. New config entries can be added as needed without migration. |
