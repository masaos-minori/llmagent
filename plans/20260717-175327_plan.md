# Plan: RuntimeToolRegistry — represent disabled MCP tools without exposing them to the LLM

Source requirement: `requires/20260717_17_require.md` (moved to `requires/done/` on completion of this workflow).

## Goal

Introduce a `RuntimeToolRegistry` concept on the agent side that keeps a discovered-but-disabled
MCP tool visible for diagnostics (server key, `config_dependent`, `enabled`, `disabled_reason`,
`enabled_for_llm`) while guaranteeing it is never sent to the LLM as a callable tool and can never
be executed through the registry's own routing path. This separates four states the requirement
names explicitly: discovered tools, MCP-server-enabled tools, agent-policy-enabled tools, and
LLM-visible tools.

## Scope

In scope (per requirement's "Target files" and Scope/Acceptance criteria):

- New module `scripts/shared/runtime_tool.py` — a `RuntimeTool` dataclass (or similar) carrying
  `name`, `server_key`, `config_dependent`, `enabled`, `disabled_reason`, `enabled_for_llm`.
- New module `scripts/shared/runtime_tool_registry.py` — `RuntimeToolRegistry` class: register
  discovered tools (including disabled ones), expose LLM-visible subset, reject execution
  attempts against disabled tools, and provide a diagnostics dump.
- New module `scripts/agent/services/mcp_tool_discovery.py` — builds `RuntimeTool` entries from
  live `/v1/tools` responses (per-server `enabled`/`disabled_reason`, once those fields exist —
  see Unknowns/Risks) and feeds them into `RuntimeToolRegistry`.
- `scripts/agent/llm_turn_runner.py` — source the LLM-visible tool list from
  `RuntimeToolRegistry` instead of (or in addition to) the static
  `ctx.cfg.tool.tool_definitions`, filtering out disabled tools.
- `scripts/agent/commands/cmd_mcp.py` — extend `/mcp status` (or a new subcommand) to print, per
  tool: tool name, server key, `config_dependent`, `enabled`, `disabled_reason`,
  `enabled_for_llm`.
- New/updated tests: `tests/shared/test_runtime_tool_registry.py`,
  `tests/agent/services/test_mcp_tool_discovery.py`, and additions to the existing
  `tests/test_cmd_mcp.py` (the requirement's listed path `tests/agent/commands/test_cmd_mcp.py`
  does not match this repo's actual flat `tests/` layout — see Design note below).

Out of scope:

- Implementing the MCP-server-side `enabled`/`disabled_reason` computation and merge into
  `/v1/tools` responses — that is sibling requirement 15's job
  (`plans/20260717-174024_plan.md`, targeting `scripts/mcp_servers/file/read_server.py`,
  `scripts/mcp_servers/git/server.py`). **SHARED WITH `plans/20260717-174024_plan.md`**: this
  plan's `mcp_tool_discovery.py` is a pure consumer of the `enabled`/`disabled_reason` fields that
  plan produces. As of this writing that plan is not yet implemented in the tree (verified below),
  so this plan cannot be executed to completion until it lands — see Unknowns.
- Implementing the `/v1/call_tool` disabled-tool rejection inside each MCP server — that is
  sibling requirement 16's job (`plans/20260717-174848_plan.md`, same 4 file/git servers). This
  plan's own "reject execution attempts" acceptance criterion is scoped to
  `RuntimeToolRegistry`'s own routing (agent side), not to re-implementing the MCP server's
  `/v1/call_tool` gate. No file overlap with 174848's target files
  (`scripts/mcp_servers/file/*.py`, `scripts/mcp_servers/git/server.py`).
- Any change to `scripts/shared/tool_registry.py` (`ToolRegistry`, static tool-ownership registry)
  — it stays as the compile-time ownership/routing source of truth; `RuntimeToolRegistry` is a
  separate, request-time-refreshable structure layered on top, not a replacement.
- Any change to `scripts/mcp_servers/**` (server-side) code.

## Assumptions

- The requirement's file paths (`shared/runtime_tool.py`, `agent/services/mcp_tool_discovery.py`,
  `agent/llm_turn_runner.py`, `agent/commands/cmd_mcp.py`) omit the `scripts/` prefix that this
  repo actually uses for all Python packages (`scripts/shared/`, `scripts/agent/`). Verified by
  `ls scripts/agent/llm_turn_runner.py scripts/agent/commands/cmd_mcp.py` (both exist) versus
  `ls agent/llm_turn_runner.py` (does not exist, confirmed via `ls` error). This plan uses the
  real `scripts/`-prefixed paths throughout.
- The requirement's test path `tests/agent/commands/test_cmd_mcp.py` and
  `tests/agent/services/test_mcp_tool_discovery.py` imply a nested `tests/agent/...` layout, but
  the actual `tests/` directory is flat (`tests/test_cmd_mcp.py` exists; verified via
  `find tests -iname test_cmd_mcp.py`; only `tests/shared/`, `tests/integration/`, `tests/docs/`
  are subdirectories). Assumption: new tests should follow the repo's actual convention —
  `tests/shared/test_runtime_tool_registry.py` (subdirectory exists, matches requirement path
  verbatim) but `tests/test_mcp_tool_discovery.py` (flat, not
  `tests/agent/services/test_mcp_tool_discovery.py`) and additions live inside the existing
  `tests/test_cmd_mcp.py` rather than a new nested file.
- "RuntimeToolRegistry" is a new concept, not a rename of the existing `ToolRegistry` in
  `scripts/shared/tool_registry.py`. Verified via `grep -rn "RuntimeToolRegistry"` across the repo
  — zero matches anywhere (docs, code, tests) before this plan. The existing `ToolRegistry` is a
  frozen, compile-time, ownership-only structure (`ToolDefinition(name, server_key, description,
  input_schema)`, populated once from `tool_constants.py` frozensets at import time via
  `_populate_default_registry()`); it has no `enabled`/`disabled_reason` concept and is described
  in its own docstring as "not a schema/description registry" — a different responsibility than
  what this requirement asks for. Building a second, runtime-refreshable registry is therefore an
  additive change, not a modification of `ToolRegistry`.
- LLM-visible tool definitions currently come from `ctx.cfg.tool.tool_definitions` (static config
  loaded from `config/tools_definitions.toml`), consumed directly in
  `scripts/agent/llm_turn_runner.py:_stream_llm()` (line 213:
  `ctx.services_required.llm.stream(llm_url, ctx.conv.history, ctx.cfg.tool.tool_definitions)`).
  There is currently no request-time or startup-time step that fetches live `/v1/tools` and uses
  it to build the LLM-facing tool list — the only existing live-`/v1/tools` consumer is
  `scripts/agent/repl_health.py`'s `_check_tool_definitions()` / `_validate_registry_against_live()`,
  which is a diagnostic drift-check (warns on mismatch) and does not feed back into what is sent
  to the LLM. Wiring `RuntimeToolRegistry` into `llm_turn_runner.py` is therefore new plumbing, not
  a small filter added to an existing dynamic path.

## Design

1. **`RuntimeTool` (in `scripts/shared/runtime_tool.py`)**: a frozen dataclass with fields `name:
   str`, `server_key: str`, `config_dependent: bool`, `enabled: bool`, `disabled_reason: str`
   (empty string when enabled — matches this requirement's own acceptance criterion "Enabled
   tools show empty disabled reason"), and a derived/settable `enabled_for_llm: bool` (true only
   when `enabled` is true AND agent policy does not further restrict it — see policy note below).
   Shared layer — no imports from `agent`, per `.importlinter` (`shared → external only`).

2. **`RuntimeToolRegistry` (in `scripts/shared/runtime_tool_registry.py`)**: mirrors the existing
   `ToolRegistry`'s API shape (register/get/list) but keyed on `RuntimeTool` instead of
   `ToolDefinition`, and is instance-scoped (constructed per session/reload) rather than a global
   singleton, since discovery output can change across MCP config reloads. Methods:
   - `register(tool: RuntimeTool) -> None` — register or update (unlike `ToolRegistry.register`,
     re-registration on rediscovery must be allowed, since server state changes across restarts;
     this deliberately does NOT raise on duplicate name the way `ToolRegistry.register` does).
   - `get_llm_visible_definitions() -> list[dict]` — returns only `enabled_for_llm=True` tools,
     shaped for `llm.stream()`'s `tool_definitions` parameter (same shape as
     `ctx.cfg.tool.tool_definitions` entries).
   - `is_executable(name: str) -> bool` and `check_executable(name: str) -> None` (raises a new
     `DisabledToolError` when the tool is known-but-disabled, distinct from "unknown tool" —
     needed so `tool_runner.py`'s dispatch path can distinguish "not found" from "found but
     disabled" per the requirement's "reject execution attempts" criterion).
   - `diagnostics() -> list[dict]` — one row per tool with all six fields the requirement lists
     for `/mcp status` display.

3. **`mcp_tool_discovery.py` (in `scripts/agent/services/`)**: an `McpToolDiscoveryService`
   parallel in shape to `McpStatusService` (`scripts/agent/services/mcp_status.py`) — fetches
   `/v1/tools` per configured server (reusing the same `httpx.AsyncClient` probe pattern already
   used in `repl_health.py` / `mcp_status.py`), reads each tool dict's `enabled` /
   `disabled_reason` (defaulting `enabled=True, disabled_reason=""` when the server predates
   requirement 15's rollout, so this stays backward compatible), and populates a
   `RuntimeToolRegistry` instance. `config_dependent` is derived from whether the server's own
   `_cfg` has empty required config (mirrors the same signal requirement 15 already defines
   server-side — this module trusts the server's `enabled` bit rather than re-deriving it, to
   avoid duplicating config-dependency logic in two layers).

4. **`llm_turn_runner.py` integration**: `_stream_llm()` gains a `RuntimeToolRegistry` lookup
   (via `ctx.services_required`, following this codebase's existing service-locator pattern seen
   throughout `services/*.py`) and passes `registry.get_llm_visible_definitions()` in place of
   (or merged with, if plugin-provided tools must still flow through
   `ctx.cfg.tool.tool_definitions`) the raw config list. Exact merge strategy vs. plugin tools
   (see `factory.py`'s `_init_plugin_registry`) is an implementation-time decision, not a design
   blocker — plugin tools are policy-checked against `ToolRegistry` (the static one), a separate
   mechanism from MCP tool disabling.

5. **`cmd_mcp.py` diagnostics**: add a new `_format_tool_diagnostics_table()` (or a `/mcp tools`
   subcommand, mirroring the existing `/mcp status` dispatch in `_cmd_mcp()`) that renders
   `RuntimeToolRegistry.diagnostics()` rows: NAME, SERVER, CONFIG_DEP, ENABLED, DISABLED_REASON,
   LLM_VISIBLE. Appended after the existing server table in `_cmd_mcp_status()`, per the
   requirement's "`/mcp status` or equivalent diagnostics" acceptance criterion (either location
   satisfies it; appending to the existing command avoids adding a new subcommand surface).

## Implementation steps

1. Add `scripts/shared/runtime_tool.py`: `RuntimeTool` frozen dataclass + any small helper for
   computing `enabled_for_llm` from `enabled` and an optional policy predicate.
2. Add `scripts/shared/runtime_tool_registry.py`: `RuntimeToolRegistry` class, `DisabledToolError`
   exception, and the methods in Design step 2. No global singleton (unlike `ToolRegistry`) —
   constructed and owned by whichever service performs discovery.
3. Add `tests/shared/test_runtime_tool_registry.py`: register enabled/disabled tools; assert
   `get_llm_visible_definitions()` excludes disabled ones; assert `check_executable()` raises
   `DisabledToolError` for disabled tools and succeeds for enabled ones; assert `diagnostics()`
   includes all six required fields with correct values, including empty `disabled_reason` for
   enabled tools.
4. Add `scripts/agent/services/mcp_tool_discovery.py`: `McpToolDiscoveryService.discover() ->
   RuntimeToolRegistry`, fetching `/v1/tools` per server (reuse `httpx.AsyncClient` pattern),
   defaulting missing `enabled`/`disabled_reason` fields for backward compatibility with
   not-yet-updated servers.
5. Add `tests/test_mcp_tool_discovery.py`: mock `/v1/tools` responses (with and without
   `enabled`/`disabled_reason` present, to cover the backward-compat default path) and assert the
   resulting `RuntimeToolRegistry` state.
6. Wire `RuntimeToolRegistry` into `scripts/agent/llm_turn_runner.py::_stream_llm()` per Design
   step 4; add/adjust unit tests in the existing `tests/test_llm_turn_runner.py` (found via
   `find`, already covers `llm_turn_runner.py`) to assert disabled tools are excluded from the
   `tool_definitions` payload sent to `llm.stream()`.
7. Extend `scripts/agent/commands/cmd_mcp.py::_cmd_mcp_status()` with the new per-tool diagnostics
   table per Design step 5; extend `tests/test_cmd_mcp.py` to assert the new table's presence and
   field values for a disabled tool and an enabled tool.
8. Re-run `grep -rn "disabled_reason" scripts/mcp_servers/` before merging this work — if sibling
   requirement 15's plan (`plans/20260717-174024_plan.md`) has landed by then, drop the
   backward-compat defaulting fallback's test-only status and add an integration-style test
   against the real server response shape; if it has not landed, keep the defaulting path as the
   documented behavior and record the dependency in code comments referencing this plan.

## Risks

- **Cross-requirement dependency risk (primary)**: this plan's runtime behavior is only as good as
  the server-side `enabled`/`disabled_reason` data from requirement 15
  (`plans/20260717-174024_plan.md`), which is not yet implemented in the tree. Mitigation: the
  defaulting fallback in `mcp_tool_discovery.py` (missing fields → `enabled=True,
  disabled_reason=""`) keeps this plan buildable and testable independently today; Implementation
  step 8 explicitly re-checks before merge. Residual risk (that the two plans' field names/shapes
  drift apart before both land, or that requirement 15 never lands, leaving this feature
  permanently in fallback-only mode) is not fully mitigable from this plan alone — filed as
  `issues/20260717-175327_risks.md`.
- **Duplicate-registry confusion risk**: having both `ToolRegistry` (static, `shared/tool_registry.py`)
  and `RuntimeToolRegistry` (dynamic, `shared/runtime_tool_registry.py`) risks future maintainers
  conflating the two. Mitigation: both new modules' docstrings must explicitly cross-reference
  `ToolRegistry` and state the ownership split (ownership/routing vs. runtime enabled-state),
  mirroring the cross-referencing style already used in `tool_registry.py`'s own docstring.
- **LLM tool-list merge risk**: `llm_turn_runner.py` currently sources tool defs solely from
  static config; switching to `RuntimeToolRegistry` risks silently dropping plugin-provided tools
  (see `factory.py::_init_plugin_registry`) if the merge in Design step 4 is implemented
  carelessly. Mitigation: Implementation step 6 requires explicit test coverage asserting
  non-MCP/plugin tools remain visible; the design defers the exact merge strategy to
  implementation time but flags it as a required decision, not an afterthought.
- **Diagnostics table growth risk**: adding a per-tool table to `/mcp status` output could make it
  unreadable for deployments with many tools (this repo already lists ~9 servers,
  `tool_constants.py` shows dozens of tools per server in aggregate). Mitigation: Implementation
  step 7 should group the new table by server_key and consider truncating/summarizing enabled
  tools (one line: "N enabled") while always listing every disabled tool by name — full detail is
  most valuable exactly where something is disabled.

All risks except the primary cross-requirement dependency risk are adequately mitigated within
this plan; the primary risk's residual, unmitigable-from-this-plan-alone portion is filed at
`issues/20260717-175327_risks.md` (see Report).
