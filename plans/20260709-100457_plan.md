# Plan: Add field-level MCP server config diff helper

Source requirement: `requires/20260708_33_require.md` (M-1)

## Goal

Extract the per-field comparison inside `_classify_mcp_server_changes()`
into a standalone, directly unit-testable helper,
`_diff_mcp_server_config(old, new) -> list[str]`.

## Scope

In scope:
- `scripts/agent/services/config_reload.py` — add the module-level helper;
  this is a pure refactor extraction, already incorporated into the H-1
  plan's canonical Design (`plans/20260709-094640_plan.md`).
- A new, focused unit-test module/class for the helper itself, independent
  of `ConfigReloadService`/`ctx`.

Out of scope:
- `shared/mcp_config.py` — the requirement lists this as optional
  (“optionally … if helper placement is preferred there”); per the same
  single-consumer reasoning already recorded in the H-6 plan
  (`plans/20260709-095414_plan.md` § Design), the helper stays in
  `config_reload.py` alongside its only caller.
- Server add/remove detection — explicitly out of scope per the requirement
  ("Handle server add/remove outside the field diff helper"); that logic
  stays in `_classify_mcp_server_changes()`.

## Assumptions

1. The two candidate return shapes offered by the requirement ("field names"
   vs. "fully formatted restart-required entries") are mutually exclusive
   given the literal 2-argument signature stub
   (`_diff_mcp_server_config(old, new)`): formatting `mcp/<server_key>.<field>`
   requires the server key, which is not a parameter. Therefore the helper
   returns bare field names, and `_classify_mcp_server_changes()` (the only
   caller, which already has `key` in scope) performs the
   `f"mcp/{key}.{field_name}"` formatting — already reflected in the H-1
   plan's Design.
2. This is a behavior-preserving refactor: the set and order of
   `needs_restart` entries produced by `_classify_mcp_server_changes()` is
   unchanged before and after extracting the helper, since the helper's body
   is a verbatim extraction of the existing inline loop.

## Unknowns

None. The signature ambiguity in Assumption 1 was resolved by analysis, not
left open — the two options offered by the requirement are only compatible
in combination (bare names in, formatted names out, formatting done by the
caller), not as independent free choices.

## Design

See the updated `_classify_mcp_server_changes()` listing in
`plans/20260709-094640_plan.md` § Design, which now defines:

```python
def _diff_mcp_server_config(old: McpServerConfig, new: McpServerConfig) -> list[str]:
    """Return names of McpServerConfig fields that differ between old and new.

    Pure comparison — never mutates either argument. Field order follows
    _MCP_SERVER_FIELDS, so output is deterministic for a given pair of inputs.
    """
    return [
        field_name
        for field_name in _MCP_SERVER_FIELDS
        if getattr(old, field_name) != getattr(new, field_name)
    ]
```

Deterministic-order note: since `_MCP_SERVER_FIELDS` is a fixed tuple and the
comprehension iterates it in order, the returned list's order is always the
same for a given pair of differing fields — satisfies "deterministic
field-level differences."

Value-comparison note: `tool_names: list[str]`, `cmd: list[str]`, and
`env: dict[str, str]` are compared with plain `!=`, which for `list`/`dict`
is a structural value comparison (not identity) — satisfies "treat
tool_names, cmd, and env as value comparisons" without special-casing.

## Implementation steps

1. Add `_diff_mcp_server_config()` to `scripts/agent/services/config_reload.py`
   as a module-level function, placed after `_MCP_SERVER_FIELDS` and before
   `ConfigReloadOutcome`/`ConfigReloadService`.
2. Update `_classify_mcp_server_changes()` to call it (already reflected in
   the H-1 plan's canonical listing) instead of inlining the field loop.
3. Add direct unit tests for the helper:
   - identical configs → `[]`.
   - each scalar field (`transport`, `url`, `startup_mode`, `healthcheck_mode`,
     `call_timeout_sec`, `startup_timeout_sec`, `auth_token`, `role`) changed
     one at a time → `[<that field>]`.
   - `tool_names`, `cmd` changed → `[<that field>]` each.
   - `env` changed → `["env"]`.
   - assert the input `old`/`new` objects are unchanged after the call
     (no mutation).
4. Run the full validation sequence (`rules/toolchain.md`) and fix any findings.

## Risks

- **Risk**: Extracting the loop into a separate function could accidentally
  change iteration order or introduce a subtle behavior difference (e.g. if
  `getattr` fails on a missing attribute for one type).
  → **Mitigation**: The helper is a verbatim extraction of the existing
  inline comparison (same field tuple, same `!=` operator); the regression
  test suite in Validation plan (full `test_config_reload.py` run) catches
  any behavioral drift.
- **Risk**: `McpServerConfig` fields with custom `__eq__`-sensitive types
  (e.g. enums) could compare unequal in unexpected ways if a future field is
  added to the dataclass without updating `_MCP_SERVER_FIELDS`.
  → **Mitigation**: Out of scope for this refactor; already flagged as a
  standing consideration in the H-2 plan (exhaustive per-field test acts as a
  regression guard if the tuple and the dataclass drift apart).
