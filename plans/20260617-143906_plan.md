# Work Plan: Stage-level observability for pipeline fallback behavior

## Goal
- expose stage-level success/failure/fallback information in RAG pipeline
- record elapsed time per stage (extend existing `last_timings`)
- make degraded behavior visible in logs or debug output

## Scope
- `scripts/rag/pipeline.py` — `RagPipeline.run()` method, stage execution loop
- `scripts/rag/stage.py` — `PipelineContext`, `PipelineStage` Protocol
- `scripts/rag/stages/mqe.py` — MqeStage
- `scripts/rag/stages/search.py` — SearchStage
- `scripts/rag/stages/rerank.py` — RerankStage
- `scripts/rag/stages/fusion.py` — FusionStage (if exists)
- `scripts/rag/stages/augment.py` — AugmentStage

## Assumptions
- Existing `last_timings: dict[str, float]` in `RagPipeline` is the canonical timing storage
- `PipelineContext` can be extended with stage-level metadata without breaking callers
- Logging via `logging.getLogger(__name__)` is sufficient; no new metrics infrastructure required
- Fallback detection relies on catching exceptions or checking empty/None results per stage

## Unknowns
- [ ] What constitutes "fallback" for each stage? (e.g. MQE returns `[query]`, search returns `[]`, rerank uses RRF order)
- [ ] Should fallback be logged at WARNING level or a new DEBUG/INFO level?
- [ ] Does `PipelineContext` need a new field like `stage_results: list[StageResult]` or is per-stage logging sufficient?
- [ ] Are there existing tests that validate stage behavior and can be extended for observability assertions?

## Affected areas
| File | Change type |
|---|---|
| `scripts/rag/pipeline.py` | Extend `RagPipeline.run()` to track stage outcomes; add logging |
| `scripts/rag/stage.py` | Optionally extend `PipelineContext` with `stage_results` field |
| `scripts/rag/stages/mqe.py` | Add fallback detection logic (e.g. MQE disabled → returns `[query]`) |
| `scripts/rag/stages/search.py` | Add fallback logging when embedding fails or search returns empty |
| `scripts/rag/stages/rerank.py` | Add fallback logging when `use_rerank=False` or rerank exception |
| `scripts/rag/stages/fusion.py` | Check if RRF merge has fallback paths |
| `scripts/rag/stages/augment.py` | Check if augment has fallback paths |

## Design
### Option A: Logging-only (minimal)
- Add structured logging in each stage's `run()` method
- Log at INFO level for success, WARNING for fallback, ERROR for failure
- Record timing in existing `last_timings` dict (already done per step)
- No changes to `PipelineContext`

### Option B: Context metadata (recommended)
- Add `stage_results: list[StageResult]` to `PipelineContext`
- Define `StageResult` TypedDict with: `stage_name`, `status` (success/failure/fallback), `elapsed_seconds`, `fallback_reason?`
- Each stage appends its result to `ctx.stage_results` after `run()`
- `RagPipeline.run()` logs summary of all stage results at end
- Enables programmatic inspection by callers (e.g. debug_fn)

### Option C: OTel spans
- Wrap each stage in OpenTelemetry span
- Requires existing OTel integration (`shared/otel_tracer.py`)
- More overhead; may be overkill for current needs

**Recommendation: Option B** — balances visibility with minimal complexity and enables both logging and programmatic access.

## Implementation steps
1. Define `StageResult` TypedDict in `scripts/rag/stage.py`:
   ```python
   from typing import TypedDict

   class StageResult(TypedDict):
       stage_name: str
       status: str  # "success" | "failure" | "fallback"
       elapsed_seconds: float
       fallback_reason: str | None
   ```
2. Add `stage_results: list[StageResult] = dataclasses.field(default_factory=list)` to `PipelineContext`
3. Modify `RagPipeline.run()` in `scripts/rag/pipeline.py`:
   - After each stage execution, append `StageResult` to `ctx.stage_results`
   - Detect fallback/failure by checking:
     - Exception caught → status="failure"
     - Stage completed but result unchanged (e.g. MQE returns `[query]`, search returns empty) → status="fallback"
     - Normal completion → status="success"
   - Log summary after all stages complete
4. Update each stage module (`mqe.py`, `search.py`, `rerank.py`, etc.) to:
   - Detect fallback conditions internally (or rely on caller inspection)
   - Optionally set `fallback_reason` via kwargs or return value
5. Add debug output in `RagPipeline.augment()` to show stage outcomes when `debug_fn` is provided
6. Add logging for HTTP mode delegation fallback (`_augment_http` returns None)

## Validation plan
- Unit tests: Verify `ctx.stage_results` populated correctly after `run()`
  - Test MQE disabled → fallback status
  - Test search with empty results → fallback status
  - Test rerank disabled → fallback status
  - Test normal path → success status
- Integration test: Verify log output contains stage-level details
- Manual test: Run pipeline with degraded embedding service; confirm fallback logged

## Risks
| Risk | Mitigation |
|---|---|
| Breaking `PipelineContext` API for existing callers | Add new field with `default_factory=list`; backward-compatible |
| Fallback detection logic too fragile | Define clear criteria per stage; document in code comments |
| Performance overhead from logging | Use structured logging; only log at INFO/WARNING level (not DEBUG) |
| OTel integration conflict if Option C chosen | Stick to Option B; OTel spans can be added later as separate task |
| Missing fallback scenarios in some stages | Review each stage individually during implementation; add tests |
