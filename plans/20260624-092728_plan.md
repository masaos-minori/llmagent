# Plan: Implement retry_count Update Path for DLQ Promotion

**Requirement:** `requires/20260623_19_require.md`
**Task size:** Path B (Large — new retry increment path, tests, docs update)

---

## Goal

Implement an explicit `retry_count` increment mechanism in `eventbus/app.py` so that events naturally reach DLQ via retry exhaustion (`retry_count >= max_retry`).

---

## Scope

**In:**

**Out:**

## Assumptions

1. `promote_to_dlq()` SQL is correct and functional (`retry_count >= max_retry AND dlq_at IS NULL`) — confirmed by code.
2. `/dlq/{event_id}/requeue` resets `retry_count = 0` (confirmed: `app.py:193`).
3. Retry ownership: requeue-side — incrementing on each requeue call before reset makes "retry exhausted" reachable.

## Design

```python
# app.py — /dlq/{event_id}/requeue
# Before resetting retry_count to 0, the DLQ promotion already happened
# The requeue path: event is in DLQ → requeue → reset retry_count to 0
# The issue: retry_count never reaches max_retry through normal use
#
# Fix: On each requeue, do NOT reset to 0. Instead increment:
# UPDATE events SET retry_count = retry_count + 1, dlq_at = NULL WHERE event_id = ?
# This way: 3 requeues with max_retry=3 → DLQ promotion on next dlq_loop run
```
