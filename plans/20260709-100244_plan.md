# Plan: Remove github_server_url legacy configuration

Source requirement: `requires/20260708_32_require.md` (H-10)

## Goal

Remove the legacy `MCPConfig.github_server_url` scalar entirely so
`mcp_servers.github.url` is the single canonical source for the GitHub MCP
endpoint, and reject the removed key at config-load time.

## Scope

In scope:
- `scripts/agent/config_dataclasses.py` — remove `MCPConfig.github_server_url`.
- `scripts/agent/config_builders.py` — remove the field wiring in
  `build_agent_config()`; add `github_server_url` to the removed-key rejection.
- `scripts/agent/services/config_reload.py` — remove the
  `_apply_llm_prompt_params()` handling.
- `scripts/agent/commands/cmd_config_display.py` — remove the `/config`
  display line.
- `config/agent.toml` — remove the sample key.
- `docs/05_agent_08_configuration.md` — remove the table row.
- 13 test files (enumerated below) — remove now-invalid fixture data /
  assertions.

Out of scope:
- `tests/test_mcp_config.py` — verified its one `github_server_url` reference
  (line 137) is passed to `_build_mcp_servers()` directly, not
  `build_agent_config()`; that function never reads this key, so it is inert
  test data, not a fixture that needs removal for correctness. Left as-is to
  minimize unrelated diff.
- Any change to how the GitHub MCP endpoint is actually resolved at runtime
  — verified below that `cfg.mcp.github_server_url` has zero runtime readers
  outside the four production files being changed; the real GitHub MCP calls
  already route through `ctx.cfg.mcp.mcp_servers["github"].url` via the
  standard MCP transport path, unaffected by this change.

## Assumptions

1. `cfg.mcp.github_server_url` is read nowhere in `scripts/` besides
   `config_dataclasses.py` (definition), `config_builders.py` (construction),
   `config_reload.py` (reload wiring), and `cmd_config_display.py` (display)
   — verified by
   `grep -rn "github_server_url" scripts/ --include=*.py` and confirming no
   hits outside those four files. It is provably dead configuration with no
   other runtime consumer.
2. `build_agent_config()` (`scripts/agent/config_builders.py:253-271`)
   already has a `_FORBIDDEN_KEYS` mechanism used for prior removed keys
   (`workflow_mode`, `workflow_require_approval`, `use_tool_summarize`,
   `tool_summarize_threshold`), raising `ConfigLoadError` with the message
   `f"Config contains removed keys: {sorted(found)}. Remove them from your
   config file."` — verified by reading that function. This is the
   established precedent for "removed key → startup validation error."
3. `/reload`'s code path (`cmd_config.py:_cmd_reload()` →
   `ConfigLoader().load_all()` → `ConfigReloadService.apply_config_dict()`)
   never calls `build_agent_config()`, so the `_FORBIDDEN_KEYS` check does
   not run during reload — verified by reading
   `scripts/agent/commands/cmd_config.py:44-51`. `ConfigReloadService` has no
   general forbidden-key validation mechanism today (only field-level type
   checks via `ConfigReloadValidationError` for `masked_fields`).
4. Given assumption 3, adding reload-time *rejection* for this key would
   require introducing a new general validation mechanism in
   `ConfigReloadService` — a larger architectural change not explicitly
   requested. The requirement's acceptance criteria accepts either "rejected
   or explicitly handled as a removed key"; removing the reload-handling
   code path (so the key is silently ignored during `/reload`, with no
   effect) satisfies "explicitly handled as a removed key" for the reload
   path, while startup load gets the stronger "rejected" behavior via
   `_FORBIDDEN_KEYS`.
5. Of the 14 files that reference `github_server_url` in `tests/`, 13 pass
   a dict containing this key through `build_agent_config()` (directly or
   via a local `_cfg()`/`base = build_agent_config(...)` helper) — verified
   individually via `grep -n "build_agent_config" tests/<file>.py` for each.
   Once `github_server_url` is added to `_FORBIDDEN_KEYS`, all 13 will raise
   `ConfigLoadError` unless the key is removed from their fixture dicts.

## Unknowns

None. The reload-vs-startup rejection asymmetry (assumption 4) was resolved
by analysis, not left open, since introducing new reload validation
machinery is a larger change than this requirement asks for and the
acceptance criteria's "or explicitly handled" wording accommodates the
narrower fix.

## Design

**`config_builders.py`** — extend the existing forbidden-key mechanism with a
dedicated message for this key (kept separate from the generic
`_FORBIDDEN_KEYS` set so the actionable replacement guidance the requirement
recommends is preserved verbatim):

```python
    found = _FORBIDDEN_KEYS & set(cfg.keys())
    if found:
        raise ConfigLoadError(
            f"Config contains removed keys: {sorted(found)}. "
            "Remove them from your config file."
        )
    if "github_server_url" in cfg:
        raise ConfigLoadError(
            "Config contains removed key: github_server_url. "
            "Use [mcp_servers.github].url instead."
        )
```

And remove `github_server_url=cfg.get(...)` from the `MCPConfig(...)`
construction.

**`config_dataclasses.py`** — delete line 353
(`github_server_url: str = "http://127.0.0.1:8006"`).

**`config_reload.py`** — delete the `_apply_str(new_cfg, "github_server_url",
...)` block in `_apply_llm_prompt_params()`. No replacement rejection logic
is added here (see Assumption 4) — the key becomes silently inert during
`/reload`, matching "no longer applies."

**`cmd_config_display.py`** — delete line 38
(`self._out.write(f"  github_server_url   : {ctx.cfg.mcp.github_server_url}")`).

**`config/agent.toml`** — delete line 25.

**`docs/05_agent_08_configuration.md`** — delete the `github_server_url` row
from the config table; confirm the `mcp_servers.github.url` entry (or add
one) documents it as the canonical GitHub MCP endpoint.

2. In `config_builders.py`: remove the field from `MCPConfig(...)`
   construction; add the dedicated `ConfigLoadError` check per Design.
3. In `config_reload.py`: remove the `_apply_str(..., "github_server_url",
   ...)` line from `_apply_llm_prompt_params()`.
4. In `cmd_config_display.py`: remove the display line.
5. In `config/agent.toml`: remove the sample key.
6. In `docs/05_agent_08_configuration.md`: remove the table row; verify

7. Update the 13 test files listed in Affected areas: remove the
   `"github_server_url": "..."` fixture key (11 files) or the

8. Add a new test asserting `build_agent_config({"github_server_url": "x", ...})`

   `tests/test_config_builders.py` or alongside the existing

## Risks

- **Risk**: Missing one of the 13 test files in the sweep would leave a
  fixture that fails once `_FORBIDDEN_KEYS`-equivalent rejection is added.
  → **Mitigation**: Full-suite run (`uv run pytest`) in the validation plan
  will surface any missed file as a `ConfigLoadError` failure, not a silent
  gap.
- **Risk**: Startup rejects the key but `/reload` silently ignores it —
  asymmetric behavior could confuse an operator who fixes their config file
  but still has a stale in-memory reload path (though reload reads from the
  same files, so this is only surprising, not incorrect).
  → **Mitigation**: Documented explicitly in Assumption 4 and Design; flag in
  the PR description. If reviewers want symmetric rejection, that implies
  adding general forbidden-key validation to `ConfigReloadService`, which
  should be scoped as a separate, explicit follow-up requirement rather than
  folded into this one.
- **Risk**: `docs/05_agent_08_configuration.md` may have other passages
  (beyond the single table row found) referring to `github_server_url` in
  prose.
  → **Mitigation**: Implementation step 6 explicitly instructs re-reading
  the file's surrounding context, not just deleting the matched line.
