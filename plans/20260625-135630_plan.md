# Plan: [High][eventbus][runtime] SQLite操作をスレッドプールへオフロード

Source: `requires/20260625_14_require.md`

## Goal

`async def` エンドポイントおよびバックグラウンドタスク内で直接呼び出されているすべての同期 SQLite 操作を `asyncio.to_thread()` 経由でスレッドオフロードし、イベントループのブロッキングを排除する。

## Scope

**In-Scope**
- `scripts/eventbus/app.py` 内の全 SQLite 呼び出し箇所 (health, publish, replay, subscribe, dlq_list, dlq_requeue, _get_seq)
- `_dlq_loop()` 内の `promote_to_dlq()` 呼び出し

**Out-of-Scope**
- DB スキーマ変更
- 接続管理戦略の変更 (req #16 の対象)
- DB ファサードの導入 (req #15 の対象)
- ファイル I/O のオフロード (req #18 の対象)

## Assumptions

1. FastAPI / Starlette 環境なので `asyncio.to_thread()` (Python 3.9+) が利用可能
2. 既存の `check_same_thread=False` 設定はスレッドプール利用後も暫定的に維持する (接続戦略再設計は req #16 で行う)
3. `promote_to_dlq()` は `_dlq_loop()` からのみ呼ばれるため、並列呼び出しは発生しない

## Affected Areas

| File | Change | Blast Radius |
|---|---|---|
| `scripts/eventbus/app.py` | 全 SQLite 呼び出しを `asyncio.to_thread(lambda: ...)` でラップ | 中 (全エンドポイント) |

## Design

各 SQLite 呼び出しを次のパターンに置換する:

```python
# Before
cur = _db.execute("INSERT ...", (...))
_db.commit()

# After
def _do_insert() -> sqlite3.Cursor:
    cur = _db.execute("INSERT ...", (...))
    _db.commit()
    return cur
cur = await asyncio.to_thread(_do_insert)
```

`health()` の SELECT 1 もオフロード対象。
`_dlq_loop()` の `promote_to_dlq()` は `await asyncio.to_thread(promote_to_dlq, _db, ...)` に変更。

## Implementation Steps

1. `app.py` の `health()` に `asyncio.to_thread` ラッパーを追加してテスト通過を確認
2. `publish()` の INSERT/commit/SELECT をオフロード
3. `replay()` の SELECT をオフロード
4. `subscribe()` の SELECT ループをオフロード (ポーリング毎に `asyncio.to_thread` 呼び出し)
5. `dlq_list()` の SELECT をオフロード
6. `dlq_requeue()` の UPDATE/commit をオフロード
7. `_dlq_loop()` の `promote_to_dlq()` 呼び出しをオフロード
8. テスト実行・通過確認
