# Work Plan: Invalidate rag-pipeline MCP Server's Semantic Cache After CLI Ingestion

## Goal

Enable the `rag-ingest` CLI to invalidate the semantic cache held by the long-running `rag-pipeline` MCP server process after ingestion completes, preventing stale cached query results.

## Scope

**In-Scope:**
- Add HTTP endpoint to `rag-pipeline` MCP server to invalidate process-local semantic cache
- Add new config key to `config/ingester.toml` for MCP server base URL
- Update `RagIngester` to call the new endpoint via existing `httpx.Client` after `ingest_all()`
- Define failure behavior when MCP server is unreachable (log WARNING, continue normally)
- Update `docs/03_rag_90_inconsistencies_and_known_issues-part2.md` OPEN-01 to reflect resolved mechanism

**Out-of-Scope:**
- Changing the semantic cache's TTL/eviction policy
- Introducing EventBus-based notification path
- Modifying the `on_ingest_complete` callback's existing in-process contract
- Addressing DESIGN-2, DESIGN-3 in `docs/03_rag_90_inconsistencies_and_known_issues-part1.md`

## Assumptions

1. The confirmed specification is: CLI and MCP server are separate OS processes; an in-process callback cannot reach an object living in the MCP server process.
2. The MCP server already exposes `invalidate_cache()` via `_pipeline_or_raise().invalidate_cache()` in `service.py`.
3. The CLI already owns a reusable `httpx.Client` (`self._client`) used for embedding requests.
4. The existing plan format (see `plans/20260714-143629_plan.md`) is the canonical template.

## Design

### Architecture

```
CLI Process                          MCP Server Process (port 8010)
────────────                         ────────────────────────────────
rag-ingest                           /rag_invalidate_cache (POST)
    │                                    │
    │  HTTP POST /rag_invalidate_cache   │
    ├───────────────────────────────────►│
    │                                    ├── _pipeline_or_raise()
    │                                    ├── .invalidate_cache()
    │                                    └── semantic_cache.clear()
    │                                    │
    │  200 OK                            │
    │◄───────────────────────────────────┤
    │
    │  Continue normally (success or failure)
    │
    └── Log WARNING if unreachable
```

### Change Strategy

1. **Add endpoint**: POST `/rag_invalidate_cache` in `server.py`, delegating to `service.invalidate_cache()`.
2. **Expose method**: Ensure `RagPipelineMCPService.invalidate_cache()` is publicly accessible.
3. **Add config key**: `rag_pipeline_service_url = "http://127.0.0.1:8010"` in `config/ingester.toml`.
4. **Call endpoint**: After `ingest_all()` completes successfully, make HTTP POST to the new endpoint.
5. **Handle failures**: If the MCP server is unreachable, log WARNING and continue — do not raise.
6. **Update docs**: Replace OPEN-01 "Recommended action" with new HTTP-based invalidation description.

### Failure Behavior Decision

**Decision: Log WARNING and continue.**

Rationale:
- Local/dev environments may not have the MCP server running
- Ingestion success/failure reporting should not depend on cache invalidation
- A failed invalidation does not indicate data corruption — it indicates a transient state that will resolve naturally

### Endpoint Design

```python
# server.py - New route
@app.post("/rag_invalidate_cache")
async def rag_invalidate_cache():
    try:
        service = get_service()  # Get the MCP service instance
        await service.invalidate_cache()
        return {"status": "ok", "message": "Semantic cache invalidated"}
    except Exception as e:
        logger.error(f"Failed to invalidate cache: {e}")
        raise HTTPException(status_code=500, detail=str(e))
```

### Consistency Rules

- The invalidation endpoint must follow the existing MCP server HTTP pattern.
- The CLI call must use the existing `httpx.Client` — no new client creation.
- The config key must follow the existing style in `config/ingester.toml`.
