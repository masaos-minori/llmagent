## Goal

`scripts/mcp_servers/mdq/mdq_tools.py::TOOL_LIST[*]["inputSchema"]` exposes `enum`, `minimum`/`maximum`, and `items.type` constraints that match the runtime validation/config caps already enforced in `mdq_models.py` and `mdq_service.py`, without changing any tool name, dispatch behavior, or write/serial flag.

- New test file `tests/test_mdq_tool_schema.py` asserting the new constraints.

## Assumptions

1. The requirement's "Target files" list uses pre-rename names (`tools.py`, `models.py`, `service.py`); per the batch-wide context from cycles 26-28, the live files to edit are `mdq_tools.py`, `mdq_models.py`, `mdq_service.py`. Falsifiable via `rg "mcp_servers.mdq.(tools|models|service)\b" scripts/mcp_servers/mdq/__init__.py scripts/mcp_servers/mdq/__main__.py` — confirmed these import only `mdq_`-prefixed modules.
2. `tests/test_mdq_tool_schema.py` does not yet exist (confirmed by `find`/`ls`) — this is a new file, not an edit.
3. Config caps in `config/mdq_mcp_server.toml` (e.g. `max_results_limit=100`, `max_chars_per_chunk=10000`, `max_total_result_chars=100000`, `max_outline_items=500`, `max_grep_matches=200`) are operator-editable, not compile-time constants. Therefore schema `maximum` values must be read from the live `MdqService` instance's config attributes at schema-build/test time (or documented as "informational current defaults"), not hardcoded literals baked into `mdq_tools.py` — otherwise the schema will silently drift again the moment an operator edits the TOML. Falsifiable: `ConfigLoader().load("mdq_mcp_server.toml")` reflects the same values found in `mdq_service.py.__init__` defaults.
4. `mode` is runtime-hardcoded to BM25 only; `Literal["bm25"]` already exists in `SearchDocsRequest.mode` (`mdq_models.py`) but is absent from the JSON schema `enum`. Adding `"enum": ["bm25"]` is a pure schema-tightening, not a behavior change, since `mode` is already validated to `bm25` by Pydantic and ignored functionally beyond that.
5. `context_before`/`context_after` minimums should be `0` per the requirement text ("context_before.minimum = 0"), not derived from config (no config lower bound exists for these). Upper bounds for these two fields are not specified in the requirement's explicit list — the plan (Step covering grep_docs) will leave them unbounded above unless implementation review decides a sane cap is needed (recorded as a non-blocking Unknown below).
6. `OutlineRequest.max_depth` exists in the model (default 6) but is currently absent from the `outline` tool's `inputSchema.properties` entirely (only `path` and `max_outline_items` are exposed). The requirement text says "include `max_depth` if it is intended to be public" — treated as conditional. Default assumption: yes, expose it, since (a) the model already accepts and uses it (`mdq_service.py` outline logic falls back to `self.max_outline_depth` only when absent), and (b) leaving a functioning, model-level parameter fully undocumented in the public schema is the kind of drift this requirement exists to close. Falsifiable by checking whether any caller relies on `max_depth` being unreachable via the public API — none found in `mdq_server.py` or tests.
7. No Pydantic `Field(ge=/le=)` constraints will be added to `mdq_models.py` in this change — the requirement's target files list `models.py` only because the batch's pre-rename target list is generic; the requirement's "Required changes" section is scoped entirely to `inputSchema` (JSON Schema), not to Pydantic model validators. Model-level bound enforcement (defense in depth against a non-schema-respecting caller) is a separate, larger change and is noted as an out-of-scope follow-up in Risks, not implemented here.

## Design

All changes are additive JSON-Schema-dict edits inside `TOOL_LIST` in `mdq_tools.py`. No new imports, no control-flow changes, no risk of import-linter violations (file already sits in the `mcp_servers` layer and adds no new cross-layer import).

### search_docs
```python
"mode": {"type": "string", "enum": ["bm25"]},
"limit": {"type": "integer", "minimum": 1, "maximum": 100},
"max_results_limit": {"type": "integer", "minimum": 1, "maximum": 100},
"max_total_result_chars": {"type": "integer", "minimum": 1, "maximum": 100000},
"tag_filter": {"type": "array", "items": {"type": "string"}},
```
(100, 100000 sourced from `mdq_service.py` `max_results_limit`/`max_total_result_chars` config defaults — see Assumption 3 re: drift-guard test.)

### get_chunk
```python
"max_chars_per_chunk": {"type": "integer", "minimum": 1, "maximum": 10000},
```

### outline
```python
"max_outline_items": {"type": "integer", "minimum": 1, "maximum": 500},
"max_depth": {"type": "integer", "minimum": 1, "maximum": 6},
```
(Add `max_depth` as a new property per Assumption 6; bound 1-6 sourced from `max_outline_depth` config default.)

### index_paths / refresh_index
```python
"paths": {"type": "array", "items": {"type": "string"}, "minItems": 1},
```
(applies to both tool schemas; `refresh_index` keeps its existing `force: {"type": "boolean"}` unchanged)

### grep_docs
```python
"paths": {"type": "array", "items": {"type": "string"}},
"max_grep_matches": {"type": "integer", "minimum": 1, "maximum": 200},
"max_chars_per_match": {"type": "integer", "minimum": 1, "maximum": 500},
"context_before": {"type": "integer", "minimum": 0},
"context_after": {"type": "integer", "minimum": 0},
```
(No `minItems` requested for `grep_docs.paths` in the requirement text, unlike `index_paths`/`refresh_index` — kept as difference is intentional per requirement's explicit per-tool wording: index/refresh say "optionally minItems=1", grep_docs section does not mention minItems at all.)

### Drift-guard test design (`tests/test_mdq_tool_schema.py`)

Follow the dict-field-inspection style already used in `test_tool_server_layer_consistency.py` (per sub-agent finding item 5): import `TOOL_LIST` from `mcp_servers.mdq.mdq_tools`, index into `["inputSchema"]["properties"][field]`, and assert:
1. `search_docs.mode.enum == ["bm25"]`.
2. Each listed integer field has `minimum`/`maximum` keys present with the literal values above.
3. Each listed array field has `items == {"type": "string"}` (and `minItems` where applicable).
4. A drift-guard: load `config/mdq_mcp_server.toml` via the same `ConfigLoader` used by `mdq_service.py`, and assert the schema's literal `maximum` values equal the config's live defaults for `max_results_limit`, `max_chars_per_chunk`, `max_total_result_chars`, `max_outline_items`, `max_grep_matches`, `max_chars_per_match` — this is the mitigation for UNK-01/Assumption 3 (catches config/schema drift in CI rather than at runtime).

## Risks

- **Risk:** A downstream MCP client (LLM caller) that has cached/learned the looser pre-change schema may generate a request now rejected by a stricter published schema (e.g. `tag_filter` item not a string) even though runtime previously coerced/ignored it silently. **Likelihood:** low. **Mitigation:** the tightened schema strictly matches existing runtime behavior (per sub-agent finding item 2-3, values are already clamped via `min(request, config)` server-side) — no previously-succeeding request becomes newly invalid at the runtime layer; only the *advertised* schema becomes more informative. Documented in Assumption 7 that no Pydantic-level enforcement is added, so even a client that ignores the schema and sends out-of-range values continues to be served (clamped), not rejected.
- **Risk:** Config drift — if a future operator edits `config/mdq_mcp_server.toml` caps (e.g. raises `max_grep_matches` above 200), the static schema literals in `mdq_tools.py` become stale (schema `maximum` now understates the real cap). **Likelihood:** medium (config is operator-editable and separate from code). **Mitigation:** Phase 2's drift-guard test (`test_mdq_tool_schema.py` assertion group 4) fails CI the moment `mdq_tools.py` literals and `config/mdq_mcp_server.toml` defaults diverge, forcing a manual sync. This converts a silent-drift risk into a loud CI failure, which is an accepted mitigation for a planning-only static-schema design (full dynamic schema generation is out of scope — see UNK-01).
- **Risk:** Exposing `outline.max_depth` in the public schema (previously undocumented) could be interpreted as a behavior change beyond "schema precision," since it makes a previously-obscure parameter easily discoverable/callable. **Likelihood:** low. **Mitigation:** the model (`OutlineRequest.max_depth`) and service logic already accept and honor this field today — this is purely a documentation/discoverability change, not new functionality it is bounded 1-6 in the new schema, matching the existing config default ceiling (`max_outline_depth`), so no new out-of-range values become possible.
- **Risk:** This plan does not add Pydantic-level (`Field(ge=/le=)`) enforcement matching the new schema bounds, so a client bypassing schema validation (e.g. a non-conforming MCP client or direct HTTP call) could still submit out-of-schema values; runtime would silently clamp them via existing `min()`/`or` logic rather than reject them, which is consistent with today's behavior but leaves the "schema says X, runtime enforces something looser" gap only half-closed (bounds are now correctly *advertised* but still not *enforced* at the model layer). **Likelihood:** medium — this is an intentional scope boundary, not an oversight, but worth flagging since it's the most natural follow-up. **Mitigation:** documented as an explicit out-of-scope follow-up (Assumption 7); if desired, a future requirement should add `Field(ge=1, le=<cap>)` to `mdq_models.py` request models, which is a strictly larger change than the current requirement's stated scope (JSON schema `inputSchema` only) and was not requested. No unmitigated-risk issue file is warranted for this one since it is a stated, deliberate scope decision rather than an unresolved problem — recorded here for traceability only.

No risk above is left fully unmitigated without a concrete mitigation already in the plan; therefore no `issues/*_risks.md` file is written for this cycle.
