## Design

### Phase 1: 後方互換レイヤ確認・削除

1. `rg "repl_debug|context_detection|rag_debug" scripts/` で参照確認
2. `grep -n "# compat\|# legacy\|# deprecated\|# flat" scripts/agent/config.py scripts/agent/context.py` で互換コード特定
3. 存在しない場合はスキップ。存在する場合は削除して呼び出し側を `cfg.llm.*` / `ctx.conv.*` に統一

### Phase 2: hidden global state 排除

主な対象:
- `db/helper.py`: `build_db_config()` を `__init__` の引数で受け取る構造に変更
- `db/tool_results.py`: モジュールレベルの初期化を確認し、DI 化
- `_audit_logger_instance` が存在する場合: `AppServices.audit_logger` (既に DI 済み) へ統合

### Phase 3: 責務分離

`tool_executor.py` (597行, 34 def/class) の分割案:
- `shared/tool_routing.py` — `ToolRouteResolver` 移動
- `shared/tool_cache.py` — `_CacheEntry`, `_execute_with_cache` 移動
- `shared/tool_executor.py` — `ToolExecutor`, transport classes のみ残す

`llm_client.py` (622行, 27 def/class) の分割案:
- `shared/llm_transport.py` — HTTP transport, retry ロジック
- `shared/llm_sse.py` — `RobustSSEParser` 移動
- `shared/llm_client.py` — `LLMClient` のみ残す

### Phase 4: エラー処理統一

- `orchestrator.py` の partial completion / mid-turn transport failure を共通ポリシー化
- `format_transport_error()` は既に `TransportErrorInfo(summary, detail, retryable, partial, source, phase)` 構造を持つ — 拡充が必要な箇所を確認

### Phase 5: 型・スキーマ強化

```python
# 変更前
class LLMMessage(TypedDict):
    role: str  # 任意文字列

# 変更後
from typing import Literal
class LLMMessage(TypedDict):
    role: Literal["system", "user", "assistant", "tool"]
```

`mcp_config.py`:
```python
# 変更前
transport: str  # "http" | "stdio"

# 変更後
from enum import Enum
class TransportType(str, Enum):
    HTTP = "http"
    STDIO = "stdio"
```

### Phase 6: ルーティング単一ソース化

- `WRITE_TOOLS | DELETE_TOOLS | {"shell_run"}` の固定集合を `ToolSpec.capability` に統合
- `route_resolver.py` の static fallback に strict mode を追加

### Phase 7: ToolSpec 正式化・DAG 実行

- `resource_scope`, `requires_serial` は `tool_runner.py` に既に実装済み
- `ToolSpec` dataclass を `shared/tool_spec.py` に作成して正式化
- 同一 resource への write は自動直列化する実行計画生成

## Implementation Steps

各 Phase は独立してコミット可能。Phase 1 → 2 → 5 → 4 → 6 → 3 → 7 の順を推奨
（影響範囲が小さい順）。

1. **Phase 1** — 後方互換レイヤ確認・削除
   - `rg` で削除対象ファイルと参照を確認
   - flat 互換コードを削除し、呼び出し側を統一 API に修正
   - 完了条件: `uv run pytest -v` でリグレッションなし

2. **Phase 2** — hidden global state 排除
   - `db/helper.py`, `db/tool_results.py` の module-global 状態を DI 化
   - 完了条件: `uv run mypy scripts/` + `uv run pytest` pass

3. **Phase 5** — 型・スキーマ強化
   - `LLMMessage.role` を Literal 化
   - `McpServerConfig` の `transport`, `startup_mode`, `healthcheck_mode` を enum 化
   - 完了条件: `uv run mypy scripts/` でエラー減少

4. **Phase 4** — エラー処理統一
   - `orchestrator.py` の LLM 失敗処理を一元化
   - `git_helper.py` の `except Exception: return None` を reason-coded result に変更
   - 完了条件: `uv run pytest tests/test_orchestrator.py` pass

5. **Phase 6** — ルーティング単一ソース化
   - `ToolSpec` メタモデルに side_effect / capability を統合
   - `route_resolver.py` strict mode 追加
   - 完了条件: `uv run pytest tests/test_tool_*.py` pass

6. **Phase 3** — 責務分離（最大変更）
   - `tool_executor.py` を routing / cache / executor に分割
   - `llm_client.py` を transport / parser / client に分割
   - 完了条件: import graph チェック + 全テスト pass

7. **Phase 7** — ToolSpec 正式化・DAG 実行
   - `shared/tool_spec.py` 作成、`ToolSpec` dataclass 定義
   - 完了条件: `uv run pytest tests/test_tool_runner.py` pass

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Phase 3 (責務分離) で import 循環が発生する | 中 | 分割前に `pydeps` / `lint-imports` でグラフ確認。一方向に保つ |
| Phase 5 の enum 化で `str` 比較箇所が漏れる | 中 | `rg '\.transport\b\|startup_mode' scripts/` で全参照を列挙してから変更 |
| Phase 2 の DI 化で既存 factory (`factory.py`) の変更が波及する | 中 | `db/helper.py` 利用者を `rg "SQLiteHelper\|build_db_config"` で列挙してから変更 |
| 要望書に記載されたファイル (`repl_debug.py` 等) が存在せず Phase 1 が空振り | 低 | Phase 1 実装前に存在確認を徹底し、不要なら skip してドキュメントに記録 |
| 大規模変更により既存テスト 200+ 件が連鎖失敗する | 高 | Phase ごとに独立コミット。各 Phase 後に `uv run pytest` を必ず実行 |
