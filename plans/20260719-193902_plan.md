# Plan: Audit, health state, and lightweight metrics for web-search-mcp

Source requirement: `requires/20260717_23_require.md` (moved to `requires/done/` at end of this cycle)

## Goal

`web-search-mcp` unconditionally emits a classified audit record for every tool call
(success, zero-result, validation error, unknown tool, timeout, provider/network/parse
error, unexpected error), tracks recent DuckDuckGo provider health so `/health` reflects
degraded state after repeated failures, and maintains lightweight in-process metrics
(counts, average latency, last success/failure) — all without a heavy observability
framework and without ever storing full query text in metrics.

## Scope

**In scope:**
- `scripts/mcp_servers/web_search/server.py` — refactor `call_tool()` to
  try/except/finally so audit always fires; wire health state into `/health`.
- `scripts/mcp_servers/web_search/health.py` (new) — provider health state
  (last success/failure timestamp, last error type, consecutive failure count).
- `scripts/mcp_servers/web_search/metrics.py` (new) — process-local counters
  (queries_total/succeeded/failed, average latency, last_success_at, last_failure_at,
  last_error_type). No full query text stored.
- `scripts/mcp_servers/web_search/formatters.py` — ensure `search_web()` /
  `fdisp_search_web()` propagate enough information (latency, result count, exception)
  for `server.py` to record health/metrics and audit outcome without duplicating the
  DuckDuckGo call.
- `scripts/mcp_servers/web_search/search_provider.py` — hook point for recording a
  provider success/failure into the new health module (or return enough info for the
  caller to do so); no change to the DuckDuckGo call itself.
- `tests/test_web_search_audit.py`, `tests/test_web_search_health.py`,
  `tests/test_web_search_metrics.py`, `tests/test_web_search_server.py` (all new —
  none of these four files currently exist; only `tests/test_web_search_models.py`
  exists today).

**Out of scope:**
- The 8-server orphaned-duplicate-file cleanup (`models.py`/`web_search_models.py`,
  `tools.py`/`web_search_tools.py`, `server.py`/`web_search_server.py` and the same
  pattern in `browser`, `cicd`, `git`, `github`, `mdq`, `rag_pipeline`, `shell`) — this
  is already tracked in `issues/20260719-193357_risks.md`. This plan makes only the two
  minimal import-path corrections inside `server.py` described in Assumption 2 below,
  because they are required for the new error classification to work correctly; it does
  not delete the orphaned files or touch the other 7 servers.
- Adding new search providers or provider fallback/retry logic.
- Persisting metrics/health across process restarts (in-memory only, matches "lightweight").
- Live external health probing on every `/health` call (explicitly excluded by the
  requirement).
- Prometheus/OpenTelemetry or any external metrics backend.

## Assumptions

1. **`server.py` (old-named file) is the live, deployed entry point**, not
   `web_search_server.py`. Verified via `config/agent.toml:314`
   (`cmd = [..., "/opt/llm/scripts/mcp_servers/web_search/server.py"]`) and
   `WebSearchMCPServer.app_module = "mcp_servers.web_search.server:app"`
   (self-referential, matches how it is launched). This matches the requirement's
   target-file list, so all target files are confirmed to be the live files being
   asked for, not the stale duplicates.

2. **`server.py` currently imports `WebSearchConfig`/`WebSearchUpstreamError` from the
   orphaned old `mcp_servers.web_search.models` module**, while `formatters.py` and
   `search_provider.py` (and the existing test file) import the same-named classes from
   the canonical `mcp_servers.web_search.web_search_models`. `models.py` and
   `web_search_models.py` are byte-identical source but load as two distinct Python
   modules/classes at runtime (confirmed via `diff`, both files present in `git ls-files`).
   This is a pre-existing latent bug, already filed as
   `issues/20260719-193357_risks.md` by a concurrent planning cycle: `server.py`'s
   `@app.exception_handler(WebSearchUpstreamError)` does not actually catch exceptions
   raised through `formatters.py`/`search_provider.py`, because they are different class
   objects (`issubclass()` false). **This plan's `call_tool()` refactor must import
   `WebSearchUpstreamError` (and `WebSearchConfig`) from `web_search_models`, not
   `models`**, otherwise the new except-clause classification for "provider/network/parse
   error" will silently fail to match the exception actually raised, and it will fall
   through to the generic "unexpected error" bucket instead. This is a 2-line import-path
   fix inside a file already being edited for this task — not the broader 8-server
   cleanup tracked in the referenced issue, which stays out of scope.
3. Similarly, `server.py`'s `from mcp_servers.web_search.tools import TOOL_LIST` should
   be switched to `from mcp_servers.web_search.web_search_tools import TOOL_LIST` for
   the same identity-correctness reason (content is byte-identical today so this is
   low-risk, but leaves the file internally consistent with formatters.py/search_provider.py).
4. `health.py` and `metrics.py` do not exist yet (confirmed via `ls`) — both are new
   modules. The existing `scripts/mcp_servers/mdq/health_check.py` is the closest
   in-repo precedent: a module-level function returning a `JSONResponse` built via
   `mcp_servers.health_response.make_health_response(deps, details)`. The new
   `health.py` follows the same shape but tracks in-memory provider state instead of
   querying a DB.
5. `uvicorn.run()` in `MCPServer.run_http()` (scripts/mcp_servers/server.py:216) is
   called with no `workers=` argument, i.e. single-process. A module-level singleton
   in `health.py`/`metrics.py` is therefore safe — no cross-process consistency problem
   to solve.
6. Query hash uses stdlib `hashlib.sha256` over a normalized (stripped, lowercased)
   query string; no new dependency required. `orjson` is used for any JSON serialization
   per `rules/coding.md` (not needed for `health.py`/`metrics.py` internals since
   `JSONResponse`/FastAPI handle serialization, but if any raw JSON dump is added it
   must use `orjson`, not stdlib `json`).
7. "Repeated provider failures" degrading health, and "successful call resets/improves"
   health, will use a simple consecutive-failure counter with a small fixed threshold
   (e.g. 3, mirroring the existing `MCPServer._record_tool_error` pattern in
   `scripts/mcp_servers/server.py:191-205` which already uses `_error_threshold = 3`
   over a 300s window) — reusing an established threshold convention rather than
   inventing a new one.

## Design

1. **`health.py`** — a small dataclass-backed singleton:
   ```python
   @dataclasses.dataclass
   class ProviderHealth:
       provider: str = "duckduckgo"
       last_success_at: float | None = None
       last_failure_at: float | None = None
       last_error_type: str = ""
       consecutive_failures: int = 0

   def record_success() -> None: ...   # resets consecutive_failures to 0
   def record_failure(error_type: str) -> None: ...  # increments consecutive_failures
   def is_degraded() -> bool: ...      # consecutive_failures >= 3 (UNK-2 default)
   def health_details() -> dict[str, object]: ...  # for /health `details["provider"]`
   ```
   Mirrors `mcp_servers/mdq/health_check.py`'s shape (module function returning data
   consumable by `make_health_response`), but keeps state in-process instead of
   querying a DB. `server.py`'s `/health` route calls `health_details()` and folds the
   result into the existing `details` dict; if `is_degraded()`, add an entry to `deps`
   so `make_health_response` returns 503/degraded (reusing the existing shared
   contract in `mcp_servers/health_response.py` — no shape change needed there).

2. **`metrics.py`** — module-level counters, updated from the same call site as
   `health.py`:
   ```python
   @dataclasses.dataclass
   class WebSearchMetrics:
       queries_total: int = 0
       queries_succeeded: int = 0
       queries_failed: int = 0
       _latency_sum_ms: float = 0.0
       last_success_at: float | None = None
       last_failure_at: float | None = None
       last_error_type: str = ""

       @property
       def average_latency_ms(self) -> float: ...  # 0.0 when queries_total == 0

   def record_query(success: bool, latency_ms: float, error_type: str = "") -> None: ...
   def snapshot() -> dict[str, object]: ...  # no full query text, ever
   ```
   `snapshot()` never accepts or stores the query string — only latency/count/error
   type — satisfying "Metrics do not expose full query text" directly at the API
   boundary (impossible to violate by construction, not just by convention).

3. **`server.py::call_tool()` refactor** — replace the current
   `r = await _dispatch_web_tool(...)` + unconditional `_audit_log(...)` with:
   ```python
   t0 = time.perf_counter()
   error_type = ""
   outcome = "ok"
   try:
       r = await _dispatch_web_tool(req.name, req.args)
       outcome = r.outcome
       if outcome == "error":
           error_type = classify_dispatch_error(r.output)  # validation vs unknown-tool vs provider
       else:
           metrics.record_query(success=True, latency_ms=...)
           health.record_success()
       return _to_call_tool_response(r)
   except WebSearchUpstreamError as e:          # imported from web_search_models, see Assumption 2
       outcome, error_type = "error", classify_upstream_error(e)   # UNK-1 default: substring match
       metrics.record_query(success=False, latency_ms=..., error_type=error_type)
       health.record_failure(error_type)
       raise   # preserve existing HTTP 502 behavior via the existing exception_handler
   except Exception as e:
       outcome, error_type = "error", "unexpected_error"
       metrics.record_query(success=False, latency_ms=..., error_type=error_type)
       raise
   finally:
       _audit_log(logger, ..., outcome=outcome, error_type=error_type, server_key="web_search", ...)
   ```
   This satisfies "use try/except/finally or equivalent" and "preserve existing HTTP
   and MCP error behavior" (the `raise` after recording keeps the existing
   `@app.exception_handler(WebSearchUpstreamError)` → 502 behavior; unexpected errors
   still propagate to FastAPI's default 500 handler, unchanged). `_audit_log` already
   accepts `error_type` as a keyword (`scripts/mcp_servers/audit.py:26`), so no shared
   module needs to change — only `server.py`'s call site needs to pass it.
   Dispatch-layer errors (validation error, unknown tool, zero-result) are already
   surfaced as `DispatchResult(is_error=True, output=...)` by `dispatch_tool()`
   (`scripts/mcp_servers/dispatch.py`) without raising, so those are classified from
   `r.output`/`r.is_error` inside the `try` block, not via `except`.

4. **Audit fields** — extend the `_audit_log(...)` call site in `call_tool()` with the
   fields the requirement lists that `_audit_log` already supports as parameters
   (`server_key`, `action`/tool name, `outcome`, `error_type`, `target`) and add the two
   new ones the requirement asks for that aren't yet passed: requested `max_results`,
   result count, latency ms, query preview (first 80 chars, already the convention used
   in `formatters.py`'s own log line), and optional query hash (`hashlib.sha256` over
   the normalized query) — passed via `_audit_log`'s existing `detail` free-text
   parameter as a compact string (or, if cleaner, as an additional keyword argument on
   `_audit_log` in `scripts/mcp_servers/audit.py` — a small, backward-compatible
   addition since all existing keyword args have defaults). Confirm the minimal
   viable approach during implementation: prefer reusing `detail` first to avoid
   touching `audit.py` (not in the requirement's target-file list), only extend
   `audit.py`'s signature if `detail` proves awkward for structured fields.

5. **`formatters.py`/`search_provider.py`** — no behavior change; `search_web()`
   already measures latency (`time.perf_counter()`) and knows the result count. The
   minimal change needed is exposing/returning that data (or letting `server.py`
   re-measure latency itself around `_dispatch_web_tool`, which is simpler and avoids
   plumbing an extra return value through `DispatchResult`). Recommended default:
   measure latency in `server.py::call_tool()` around the `_dispatch_web_tool` call
   directly (already have `t0`/`time.perf_counter()` idiom used elsewhere in this
   codebase) rather than changing `DispatchResult`'s shape, keeping `formatters.py`
   and `search_provider.py` changes to zero or near-zero (they remain listed as
   "affected" mainly because tests for them may need light updates, and because the
   requirement lists them as target files — re-verify during implementation whether
   any change to these two files ends up necessary at all, or whether `server.py`-only
   changes suffice).

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Reusing `WebSearchUpstreamError` imported from the wrong module (`models` instead of `web_search_models`) silently reintroduces the exception-handler mismatch documented in `issues/20260719-193357_risks.md`, causing "provider/network/parse error" audits to instead be misclassified as "unexpected_error" | Medium (easy mistake to repeat since `server.py` currently has the wrong import) | Implementation step 1 fixes the import first, before any new logic is added, and a test in `tests/test_web_search_server.py` should assert that a simulated `WebSearchUpstreamError` (raised via a monkeypatched `search_web`) is both audited with `error_type` classified as provider/timeout AND still produces the existing HTTP 502 response, closing the loop on both symptoms of the bug in one assertion |
| Timeout vs. generic provider error classification (UNK-1) is a best-effort substring match on the exception message, which is a fragile heuristic that will silently misclassify if the wrapped message text changes in `search_provider.py` | Low-medium | Add a unit test in `tests/test_web_search_health.py`/`tests/test_web_search_audit.py` that pins the current message format so any future change to `search_provider.py`'s error message triggers a visible test failure instead of a silent classification regression; document the fragility inline as a comment where the substring check lives |
| Adding a `try/except/finally` around the full dispatch path in `call_tool()` could accidentally change existing HTTP/MCP error behavior (e.g. by catching and not re-raising, or altering which status code a client sees) | Medium (behavior-preservation is an explicit acceptance criterion) | `tests/test_web_search_server.py` must include an explicit "unchanged error behavior" test asserting the existing 502-on-upstream-error and whatever the current unknown-tool/validation-error HTTP status codes are, run once *before* the refactor (as a characterization test) and again after, per `rules/toolchain.md`'s standard validation sequence |
| In-memory metrics/health singletons reset on every process restart, so `/health` could misleadingly report "healthy" immediately after a restart that followed a real outage (consecutive_failures resets to 0) | Low | Acceptable per requirement's "lightweight, no persistence" framing and explicitly out of scope (see Scope); document this behavior in a code comment in `health.py` so it isn't mistaken for a bug later |
| New files `health.py`/`metrics.py` are forgotten in `deploy/deploy.sh`'s copy list, causing an ImportError only in production (not caught by local tests, which run against the working tree, not the deployed copy) | Low-medium | Implementation steps 2–3 explicitly call out the `deploy/deploy.sh` update; Implementation step 7 makes it a mandatory gate before considering the change ship-ready, matching `rules/toolchain.md`'s completion checklist ("deploy/deploy.sh updated if a file under scripts/ was added or removed") |
