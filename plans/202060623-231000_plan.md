## Goal

MCP インフラストラクチャ (`mcp/server.py`, `shared/tool_executor.py`, `shared/route_resolver.py`, `shared/mcp_config.py`) のコードレビューで発見された Critical/High 問題を修正し、ゾンビプロセス防止・レスポンス正確性・キャッシュ安全性・接続回復力を改善する。

---

## Scope

**In-Scope:**
- `scripts/shared/tool_executor.py`
  - `StdioTransport.stop()`: `kill()` 後に `await self._proc.wait()` を追加してゾンビプロセスを排除
  - `StdioTransport.call()`: タイムアウト後に `_restart_after_timeout()` でサブプロセスを再起動; バッファ汚染を防止
  - `StdioTransport._parse_stdio_response()`: `expected_id` 引数を追加してレスポンス ID を検証
  - `ToolExecutor._execute_with_cache()`: `_inflight` dict + `asyncio.Future` でキャッシュスタンピードを防止
  - `HttpTransport.call()`: `timeout=self._timeout` を `post()` に追加; 429/502/503/504 に対する指数バックオフリトライを実装
  - `ToolExecutor.execute()`: プラグイン型バリデーション例外 (`ValueError`/`TypeError`) を `try/except` 内に移動
  - `ToolExecutor._check_health()`: `error_type` を `"transport"` に修正
- `scripts/mcp/server.py`
  - `MCPServer.run_stdio()`: `asyncio.StreamWriter` を使った非同期 stdout 書き込みへの変更
  - `MCPServer.run_stdio()`: `asyncio.CancelledError` をキャッチしてエラーレスポンスを書いてから再 raise
  - `MCPServer.run_stdio()`: ループ先頭で `name = ""` を明示初期化
- `scripts/shared/mcp_config.py`
  - `_build_single_server()`: stdio サーバーに `HealthcheckMode.HTTP` が設定されるバグを修正 (`HealthcheckMode("")` を渡して `__post_init__` に解決させる)
  - `McpServerConfig`: `call_timeout_sec: float = 60.0` フィールドを追加
- `scripts/agent/lifecycle.py` (既存の lifecycle manager)
  - SIGTERM ハンドラを追加して全 `StdioTransport` を graceful shutdown
- `tests/test_tool_executor.py`: キャッシュスタンピード防止・レスポンス ID 検証・ゾンビ回収の新規テスト追加

**Out-of-Scope:**
- `McpServerHealthRegistry` の HALF_OPEN 実装 (別 plan: `20260623-092105_plan.md`)
- `ToolRouteResolver` の ToolRegistry 失敗時の強化 (別 PR 候補)
- uvicorn の graceful shutdown 設定 (インフラ設定層の変更)
- HTTP/2 対応やコネクションプーリング設定の変更

## Assumptions

1. `asyncio.loop.connect_write_pipe()` は Python 3.10+ で安定して使用できる。WSL2 (Linux 6.6.x) 環境で利用可能であることを前提とする。使用不可の場合は `loop.run_in_executor(None, sys.stdout.write, ...)` にフォールバックする。
2. `StdioTransport._restart_after_timeout()` はタイムアウト後にサブプロセスを kill→wait→start する。`start()` の再実行は `lifecycle.ensure_ready()` と競合しない (lifecycle が start を管理している場合は別途調整が必要)。
3. `_inflight` dict を使ったスタンピード防止は asyncio シングルスレッドモデルを前提とする。`ThreadPoolExecutor` 経由の呼び出しは対象外。
4. `HttpTransport` の指数バックオフリトライは最大 3 回とする。外部 API (GitHub, web_search) のレートリミット応答 (429) が主な対象。
5. `McpServerConfig.call_timeout_sec` のデフォルトは `60.0s` とする。長時間コマンドを実行する shell/cicd は TOML で個別オーバーライドする。

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| `asyncio.loop.connect_write_pipe()` が WSL2 で未サポートまたは動作不安定 | Med | UNK-01 でスモークテストを Phase 1 で実施。失敗時は `asyncio.get_event_loop().run_in_executor(None, sys.stdout.buffer.write, data)` にフォールバックする代替実装を準備 |
| `_restart_after_timeout()` が lifecycle と二重管理になり、lifecycle が把握していないサブプロセスが起動する | Med | UNK-02 で lifecycle.py を確認。lifecycle が `start()` を管理している場合は `_restart_after_timeout()` から lifecycle の `ensure_ready()` を呼ぶか、lifecycle にコールバックを通知する設計に変更 |
| `_inflight` + `asyncio.Future` 導入で、Future の例外が複数のウェイターに伝播し想定外のキャンセルが起きる | Low | `asyncio.shield(existing)` で各ウェイターを独立させる。Future の `set_exception` は全ウェイターに同じ例外を伝播するため、TransportError は正しく各呼び出し元に返る |
| HTTP リトライの指数バックオフ (最大 4s+2s+1s=7s) がエージェントのターンタイムアウトを超過する | Low | デフォルト最大待機は 4s (2^2)。`max_tool_turns` × リトライ待機がターンタイムアウトを超えないよう `call_timeout_sec` の TOML 設定ガイドを docs に追記 |
| `_parse_stdio_response` の ID 検証追加で、サーバー側が `id` フィールドを返さない古い実装と非互換になる | Low | `run_stdio` は `req["id"]` を必ずレスポンスに含める実装になっている (server.py L268-277 確認済み)。独自プロトコルのサードパーティサーバーは対象外 |
| `error_type="transport"` への変更が `_record_success` の `stat_tool_errors` インクリメント条件に影響する | Low | `_record_success` は `result.error_type == "tool"` の場合のみ `stat_tool_errors` を増やす。`_check_health` の結果は `_record_success` を経由しないため実質的に無影響。テストで確認 |
