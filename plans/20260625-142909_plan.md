# Plan: [Medium][eventbus][health] リトライバックログ・DLQ ラグ・コンシューマー失敗のヘルス指標追加

Source: `requires/20260625_31_require.md`

## Goal

`/health` エンドポイントに配信語義の劣化を表す指標 (リトライバックログ件数・DLQ オーファン件数・累計配信失敗件数) を追加し、オペレーターがサービスが「生存しているが配信が壊れている」状態を検出できるようにする。

## Scope

**In-Scope**
- `scripts/eventbus/app.py` の `health()` 拡張
- リトライバックログ件数: `delivery_failure_count > 0 AND dlq_at IS NULL AND acked_at IS NULL`
- DLQ オーファン件数: `delivery_failure_count >= max_retry AND dlq_at IS NULL` (通常は 0)
- 累計配信失敗件数: `SUM(delivery_failure_count)` (レート近似)
- `degraded_reasons` に配信劣化指標を追加

**Out-of-Scope**
- Prometheus / OpenTelemetry インストルメンテーション
- 時間窓ベースの失敗レート (スキーマに failure timestamp がないため不可)
- /health 以外のエンドポイント変更

## Assumptions

1. req #19 (health 拡張: db_latency_ms, dlq_last_run_age_s) が実装済みであること
2. req #28 (加算的スキーマ: `delivery_failure_count`) が実装済みであること
3. 失敗 "レート" は時間窓なしの累計件数を代理指標とする (タイムスタンプ付き失敗記録は本タスク対象外)
4. `/health` の DB クエリは既存の `_check_db()` callable 内に追加し、Lock を保持したまま実行する

## Design

**`health()` レスポンス拡張例:**
```json
{
  "status": "degraded",
  "db": "ok",
  "db_latency_ms": 12.3,
  "dlq_task": "running",
  "dlq_last_run_age_s": 45.0,
  "event_count": 1200,
  "retry_backlog": 5,
  "dlq_orphan_count": 0,
  "total_delivery_failures": 23,
  "degraded_reasons": ["retry_backlog_high"]
}
```

**新しい劣化判定ロジック:**
- `retry_backlog > N` (提案初期値 N=10) → `degraded_reasons` に `"retry_backlog_high"` を追加
- `dlq_orphan_count > 0` → `degraded_reasons` に `"dlq_orphan_detected"` を追加 (インライン昇格のバグ示唆)

**DB callable 内への追加クエリ (既存の Lock を保持したまま):**
```python
def _check_db() -> tuple[str, float, int, int, int, int]:
    t0 = time.monotonic()
    try:
        _db.execute("SELECT 1")
        count_row   = _db.execute("SELECT COUNT(*) FROM events").fetchone()
        backlog_row = _db.execute(
            "SELECT COUNT(*) FROM events"
            " WHERE delivery_failure_count > 0 AND dlq_at IS NULL AND acked_at IS NULL"
        ).fetchone()
        orphan_row  = _db.execute(
            "SELECT COUNT(*) FROM events"
            " WHERE delivery_failure_count >= ? AND dlq_at IS NULL",
            (_cfg.max_retry,),
        ).fetchone()
        failure_row = _db.execute(
            "SELECT COALESCE(SUM(delivery_failure_count), 0) FROM events"
        ).fetchone()
        latency_ms  = (time.monotonic() - t0) * 1000
        return "ok", latency_ms, int(count_row[0]), int(backlog_row[0]), int(orphan_row[0]), int(failure_row[0])
    except Exception:
        ...
```

`_cfg` は `_check_db` 内では直接参照できないため、max_retry をクロージャーで渡す。

## Implementation Steps

1. `health()` の `_check_db` callable に 3 クエリを追加
2. `_cfg.max_retry` を `_check_db` のクロージャーで渡す
3. `degraded_reasons` に `retry_backlog_high` / `dlq_orphan_detected` の判定を追加
4. レスポンスに `retry_backlog`, `dlq_orphan_count`, `total_delivery_failures` フィールドを追加
5. `tests/test_eventbus_health.py` を更新して新フィールドを検証

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| `_check_db` に追加クエリが増えると health チェック自体が遅くなる | Low | 4 クエリは単純な COUNT で高速。`db_latency_ms` で計測しているので劣化を検出できる |
| `_cfg` が None の場合に max_retry 参照でクラッシュ | Medium | `_cfg` は lifespan 後は必ず非 None。health() 呼び出し時は lifespan が完了しているはずだが `assert _cfg is not None` を追加 |
