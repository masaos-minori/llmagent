# Plan: Document MCP runtime availability metadata and disabled tool behavior

Source requirement: `requires/20260717_19_require.md` (moved to `requires/done/` on
completion of this workflow)

## Goal

Document the new MCP tool availability metadata contract — `config_dependent`
(replacing `requires_config`), `enabled`, and `disabled_reason` on `/v1/tools`
responses — and the rules around it (always-return-all-tools, disabled-not-omitted,
disabled-not-LLM-visible, disabled-rejected-before-dispatch, RuntimeToolRegistry
handling), so MCP server authors and agent-side implementers have a single
authoritative reference. This is documentation only — no source code, tests, or
config changes.

## Design

- New file `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` follows the
  existing `04_mcp_*` frontmatter convention (`title`, `category: mcp`, `tags`,
  `related`) seen in `04_mcp_03_02_tool-registry.md`. Content sections:
  1. **`config_dependent` (static)** — per-tool boolean in `TOOL_LIST`, replaces
     `requires_config` with no compatibility shim (per requirement 14's plan: pure
     rename, same boolean values). State explicitly that `requires_config` is
     removed and any doc/code reference to it describes obsolete behavior.
  2. **`enabled` / `disabled_reason` (runtime, request-time-computed)** — added to
     each tool dict in the live `/v1/tools` response, computed from each server's
     current config state; `enabled=True` <-> `disabled_reason == ""`,
     `enabled=False` <-> `disabled_reason` is a non-empty standard string.
  3. **Standard `disabled_reason` values** — table: `"allowed_dirs is empty"` (file
     read/write/delete servers), `"allowed_repo_paths is empty"` (git, takes
     precedence), `"read_only=true"` (git write tools only, when allowlist is
     non-empty), and reserved/forward-looking `"command_allowlist is empty"` /
     `"workflow_allowlist is empty"` (shell/cicd — explicitly marked "not yet
     implemented, reserved for consistency" per requirement 15's plan's own
     Out-of-scope note).
  4. **`/v1/tools` behavioral rules** — always returns every implemented tool
     (disabled tools are never omitted); example JSON response block showing one
     enabled and one disabled tool side by side.
  5. **Dispatch rule** — disabled tools must be rejected by `/v1/call_tool` before
     reaching the dispatch table (server-side gate; references requirement 16's
     plan for the exact response shape,
     `CallToolResponse(result="Tool disabled: <reason>", is_error=True)`).
  6. **RuntimeToolRegistry (agent-side)** — disabled tools are tracked for
     diagnostics (`enabled_for_llm` derived field) but never included in the
     LLM-facing tool list and never dispatchable through the registry's own routing
     path; references requirement 17's plan for the four states (discovered /
     MCP-server-enabled / agent-policy-enabled / LLM-visible).
  7. **Implementation status note** (per Assumption 2) — explicit "as of 2026-07-17
     this contract is documented as target design; production code still uses
     `requires_config` and has no `enabled`/`disabled_reason`/`RuntimeToolRegistry`
     — see `plans/20260717-173602_plan.md` through `plans/20260717-175630_plan.md`"
     callout box at the top of the file, so a reader isn't misled into thinking the
     contract is already live.
- The 4 existing catalog docs (`04_mcp_04_01/_02/_03/_05`) get a one-line
  `config_dependent`/`enabled`/`disabled_reason` note per server plus a link to the
  new `04_mcp_03_06` file, rather than duplicating the full spec four times.
- `04_mcp_00_document-guide.md` gets: a new File Index row for `04_mcp_03_06`, an

  (`_01〜_02/_03a/_03b/_04/_05` becomes `...⁄_05/_06`), and (optionally) a new AI

- `05_agent_08_04` gets a short paragraph under MCPConfig (or a new subsection)
