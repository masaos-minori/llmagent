## Goal

Rename the misleadingly-named `approval_ops.get_pending_approval()` (confirmed to return the *latest* approval record regardless of status, not filtered to `status='pending'`) to `get_latest_approval()`, document the intentional defensive duplication between `/reject`'s immediate task-halt and `WorkflowEngine._gate_approval()`'s own rejected-status handling, and close the small number of test gaps this investigation found — while leaving untouched the substantial majority of this requirement's asks, which direct code/test investigation confirmed are **already correctly implemented**.

## Out-of-Scope / Already Implemented (negative findings)

This requirement's Background and Tasks describe several behaviors as open questions or gaps; direct code and test investigation confirmed the large majority are **already correct and already tested**, requiring no change:

- **Resume timing decision** — already resolved as "resume on next user turn," not immediately. Confirmed via `scripts/agent/commands/cmd_workflow.py::_cmd_approve()` (lines 118-123): sets `ctx.turn.pending_approval_task_id = task_id` and emits exactly `"Approved: {approval_id} — workflow will resume on next turn"` — this is precisely the message wording the requirement's Task asks for ("update message to: workflow will resume on next user turn"), already in place. The method's own docstring (line 80) already states "After approval, the workflow engine will auto-resume on the next turn," and the class docstring (lines 40-51) already documents startup recovery. **No message or docstring change needed here.**
- **`/approve` sets `pending_approval_task_id`** — already tested: `tests/test_cmd_workflow.py::test_approve_sets_pending_approval_task_id` (lines 149-162) asserts exactly this.
- **Idempotent resume: `plan`/`execute` skipped, `verify` runs** — already tested end-to-end: `tests/test_workflow_engine.py::test_resume_does_not_rerun_plan_or_execute` (lines 265-310) asserts `plan_calls`/`execute_calls` stay empty and `verify_calls == 1` after an approve-then-resume cycle. `tests/test_workflow_engine.py::test_startup_recovered_approval_can_resume` (lines 313-329) additionally covers the startup-recovery variant of the same resume path. The underlying mechanism (confirmed by direct read of `agent/workflow/workflow_engine.py::_run_stage()` lines 226-239 and `agent/workflow/idempotency_ops.py::begin_stage_if_new()`) is `event_id = f"{task_id}:{stage_id}:{attempt}"` — identical across the pre-gate run and the resume run for the same task, so `is_event_processed()` correctly short-circuits `plan`/`execute` re-execution. **No behavior or test change needed here.**
- **No duplicate task created during resume, at the engine level** — confirmed via `agent/workflow/workflow_engine.py::run()`, which is always invoked with the *same* `TaskRecord` object across suspend/resume in the existing tests; the actual task-reuse decision happens one layer up, in `Orchestrator._init_workflow_task()` (see Design §3 — this is the one genuine remaining test gap, at the orchestrator layer, not the engine layer).
- **Rejection halts immediately and is tested** — `scripts/agent/commands/cmd_workflow.py::_cmd_reject()` (lines 159-160) already calls `resolve_approval(..., "rejected", ...)` followed immediately by `update_task_status(store._db, task_id, "halted")`. `tests/test_cmd_workflow_approval.py::test_reject_halts_task` (line 208) and `tests/test_workflow_engine.py::test_rejected_task_halts` (line 252) already cover both the CLI-level immediate halt and the engine-level `_gate_approval()` halt-on-rejected-status path independently. **No test change needed for the halt behavior itself** — only the *documentation* explaining that these are two independent, intentionally-redundant halt paths (Design §2) is missing.
- **`05_99.md`** (named in Scope) — confirmed non-existent, consistent with every other plan in this requirement batch naming this same file; not targeted.

## Assumptions

1. **`get_pending_approval()` is genuinely misleadingly named.** Confirmed by direct read of `scripts/agent/workflow/approval_ops.py:52-70`: the query is `SELECT * FROM approvals WHERE task_id=? ORDER BY created_at DESC LIMIT 1` — no `WHERE status='pending'` filter at all. Its sole production caller, `WorkflowEngine._gate_approval()` (`workflow_engine.py:143`), relies on this exact behavior — it fetches the row and then branches on `existing.status` being `"approved"`, `"pending"`, or `"rejected"` (lines 158-168), which would be impossible if the function only returned pending rows. The name actively contradicts the function's real, load-bearing behavior.
2. **Option A (rename) is lower-risk than it first appears.** Confirmed via `grep -rn "get_pending_approval"`: exactly one production call site (`workflow_engine.py:143`) and eight test references across three files (`tests/test_workflow_state_store.py`, `tests/test_approval_ops.py`, `tests/test_workflow_engine.py`). A mechanical rename to `get_latest_approval()` is fully enumerable and low-blast-radius. No backward-compatible alias is added — consistent with this session's established convention (e.g. requirement 20's plan removing an unused backward-compat re-export block) of not adding compatibility shims for an internal, single-caller function with no external/plugin consumers.
3. **Rejected-status duplication between `/reject` and `_gate_approval()` is intentional defensive redundancy, not dead/conflicting logic.** The two paths serve different scenarios: `/reject` (`cmd_workflow.py:159-160`) halts a task the *moment* the user rejects it — this is the common case. `WorkflowEngine._gate_approval()`'s own `if existing.status == "rejected"` branch (`workflow_engine.py:166-168`) is a **fallback** for the case where the engine re-evaluates a task whose approval was already resolved as `"rejected"` through some path other than `/reject` calling `update_task_status` (e.g., a future direct DB edit, an external tool, or a resume attempt that races with the halt). Confirmed no test or code path currently exercises a case where these two disagree (rejected-in-DB but task not yet halted) — this is genuinely defensive, not evidence of an actual behavioral conflict, matching the requirement's own hedge ("This may be intentional defensive duplication, but it should be documented").
4. `agent/workflow/task_ops.py` (named in Scope) requires **no code change** — confirmed by full read of `get_task_by_id()`/`create_task()`/`update_task_status()`; these are already correctly used by `Orchestrator._init_workflow_task()` for the resume-reuse logic (Out-of-Scope section above), and this requirement's Tasks list does not call for any task_ops.py-specific change beyond what orchestrator-level tests will exercise.

Update the one production call site (`workflow_engine.py:18,143`) and all eight test references (`tests/test_workflow_state_store.py`, `tests/test_approval_ops.py`, `tests/test_workflow_engine.py`) — including their test *names* where they encode the old function name (e.g. `test_get_pending_approval_returns_latest` → `test_get_latest_approval_returns_latest`) for consistency, though renaming test function names is cosmetic and not required for correctness.

### 2. Document the rejected-status defensive duplication

`scripts/agent/workflow/workflow_engine.py::_gate_approval()` — add a short comment above the rejected branch:

```python
if existing.status == "rejected":
    # Defensive fallback: /reject (cmd_workflow.py) already halts the task
    # immediately when the user rejects. This branch only fires if the
    # engine re-evaluates a task whose approval was resolved as "rejected"
    # through some other path before the halt was applied.
    self._store.update_task_status(task.task_id, "halted")
    raise WorkflowHaltError(f"approval rejected: {existing.reason}")
```

`scripts/agent/commands/cmd_workflow.py::_cmd_reject()` docstring — add one line noting the engine-side fallback exists:

```python
def _cmd_reject(self, arg: str) -> None:
    """Reject the pending workflow-level approval gate (approvals table only).

    Does not affect per-tool interactive approval (tool_approval.run_approval_checks).
    Immediately marks the task as halted. WorkflowEngine._gate_approval() also
    halts on a rejected-status approval record as a defensive fallback for
    resume paths that don't go through this command.
    """
```

`docs/05_agent_03_03_turn-processing-flow-workflow-engine-part1.md`'s existing "### 承認ゲート" section (lines 88-104) — add one sentence after the existing rejected-branch bullet (around line 101) clarifying that `/reject` performs the same halt immediately, and the engine's own check is a defensive fallback, not the primary path.

### 3. Orchestrator-level resume test (the one genuine test gap)

Add to `tests/test_orchestrator.py`:
- A test that sets `ctx.turn.pending_approval_task_id` to an existing task's ID, calls `Orchestrator._init_workflow_task()` (or drives it via `_handle_workflow_engine()` with a mocked `WorkflowEngine`), and asserts:
  - `create_task()` is **not** called (no duplicate task created) — confirmed by direct read that `_init_workflow_task()`'s `existing_task_id is not None` branch (`orchestrator.py:257-264`) calls `get_task_by_id()` instead of `create_task()`.
  - The returned task's `task_id` matches the pre-existing one.
  - `ctx.turn.pending_approval_task_id` is reset to `None` after use (confirmed by direct read of `orchestrator.py:186-187`, immediately after `_init_workflow_task()` is called in `_handle_workflow_engine()`).
