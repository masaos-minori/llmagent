# Work Plan: RAG/Ingestion/Retrieval 層 DTO統一・fail-fast・後方互換除去

## Goal

RAG パイプライン・取り込みパイプライン・検索層の全ファイルから `dict[str, Any]` / `except Exception` / `return None on failure` / backward-compat re-exports を排除し、typed DTO・enums・dedicated exceptions による strict な型付き API に統一する。

## Scope

**In scope:**

## Assumptions

1. `rag/models.py` / `rag/enums.py` / `rag/exceptions.py` はすべて新規ファイル (既存なし確認済み)。
2. `RawHit`/`MergedHit`/`RankedHit` を TypedDict → dataclass に変換する場合、既存の `hit["chunk_id"]` スタイルのアクセスを `hit.chunk_id` に全て変換する必要がある。影響範囲は pipeline.py, repository.py, llm.py, stages/*.py に及ぶ。
3. `ChunkDocument` は `pipeline_utils.py` が返す parsed JSON の typed 表現。`url`, `title`, `lang`, `content`, `code_blocks` フィールドを持つ。
4. config DTO (MqeConfig, FusionConfig 等) は `rag/models.py` の Config セクションとして定義する。
5. `mcp/mdq/search.py` の FTS5 実装は `mcp/mdq/` 内の既存 DB ヘルパーを使用する。実装内容は `mcp/mdq/service.py` のインタフェースに合わせる。
6. `.importlinter` では `rag` → `shared`, `db` は許可されており、`rag/models.py` 等の新規 rag 内ファイルは既存の layer contract に違反しない。
