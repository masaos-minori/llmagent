# Plan: Treat all McpServerConfig field changes as restart-required

Source requirement: `requires/20260708_28_require.md` (H-6)

## Goal

Confirm and finalize that every `McpServerConfig` field is compared and, if
changed, reported as `mcp/<server_key>.<field>` in
`ConfigReloadOutcome.needs_restart` — never mutated, never applied, never
deferred.

## Scope

This requirement is the umbrella/generalization of H-1 through H-5
(`plans/20260709-094640_plan.md` through `plans/20260709-095322_plan.md`),
all of which target the same function and field set. No new production
behavior is introduced beyond what those plans already specify.

In scope:
- Confirming `_classify_mcp_server_changes()` (H-1 design) iterates all 11
  fields the requirement lists.
- The exhaustive parametrized test already specified in the H-2 plan
  (`plans/20260709-094926_plan.md` § Design) — this requirement's test plan
  is identical to H-2's.
- A one-time decision on helper placement (`config_reload.py` vs.
  `shared/mcp_config.py`), which the requirement leaves optional
  ("if helper placement is preferred there").

Out of scope: no additional files beyond what H-1..H-5 already scope.

## Assumptions

1. The `_MCP_SERVER_FIELDS` tuple defined in the H-1 plan
   (`transport`, `url`, `startup_mode`, `healthcheck_mode`,
   `call_timeout_sec`, `startup_timeout_sec`, `tool_names`, `auth_token`,
   `role`, `cmd`, `env`) is identical, field-for-field, to the list in this
   requirement and in H-2 — verified by direct comparison of the three
   requirement files.
2. `shared/mcp_config.py` has no existing field-list constant to reuse
   (verified: `grep -n "FIELDS" scripts/shared/mcp_config.py` — no match), so
   there is no pre-existing convention pulling the helper toward that module.

## Unknowns

None.

## Affected areas

No new files beyond those already listed in `plans/20260709-094640_plan.md`
(`scripts/agent/services/config_reload.py`, `tests/test_config_reload.py`,
`tests/test_config_reload_classification.py`).

## Design

Decision: keep `_MCP_SERVER_FIELDS` and `_classify_mcp_server_changes()` in
`scripts/agent/services/config_reload.py`, not `shared/mcp_config.py`.

Rationale: the field tuple has exactly one consumer (the reload classifier).
`shared/mcp_config.py` defines `McpServerConfig` itself but has no existing
per-field iteration helper, and introducing one there for a single caller
would add a cross-module dependency without a second consumer to justify it.
If a future requirement needs the same field list elsewhere (e.g. a config
diff/display command), revisit this placement then.

No new code beyond the H-1 design (`plans/20260709-094640_plan.md` § Design)
and the H-2 exhaustive test (`plans/20260709-094926_plan.md` § Design) is
required to satisfy this requirement's acceptance criteria.

## Implementation steps

1. Implement `_classify_mcp_server_changes()` per the H-1 plan (if not
   already done).
2. Add the exhaustive parametrized test per the H-2 plan (if not already
   done) — this single test satisfies H-6's test plan verbatim (it already
   asserts `needs_restart` contains the field-qualified entry, `applied` and
   `deferred` contain no MCP entry, and the running config is unchanged, for
   every one of the 11 fields).
3. No further implementation step is needed specifically for H-6.

## Validation plan

Identical to `plans/20260709-094926_plan.md` § Validation plan (H-2) — the
same test run and toolchain sequence verifies both requirements together.

## Risks

- **Risk**: Treating H-6 as fully covered by H-1/H-2 could cause it to be
  skipped if H-1/H-2 are implemented with scope drift (e.g. someone drops a
  field from `_MCP_SERVER_FIELDS` during implementation).
  → **Mitigation**: The H-2 exhaustive parametrized test enumerates all 11
  fields explicitly, so a dropped field fails that test directly — the
  acceptance criteria are self-verifying, not just documentation.
- **Risk**: Deferring the "helper placement" decision could be revisited
  inconsistently by a future contributor who doesn't see this plan.
  → **Mitigation**: Rationale recorded above in Design; low likelihood of
  conflict since it's a non-breaking internal refactor if changed later.
