# Plan: Introduce RuntimeTool and RuntimeToolRegistry

Source requirement: `requires/20260717_02_require.md`

## Goal

A new `RuntimeToolRegistry` exists as the single, normalized runtime source for tool execution metadata (routing, LLM-visible schema, scheduler metadata, side-effect detection, operation-type classification, safety-tier lookup, approval requirement, policy filtering) — replacing today's fragmented model where five+ different modules independently derive overlapping classifications from the same static `tool_constants.py` frozensets and `config/agent.toml`'s `[[tool_definitions]]` block. This requirement only *introduces* the new abstraction; wiring existing consumers to it is scoped to requirements 04-07 (separately tracked).

## Scope

**In scope**
- New `scripts/shared/runtime_tool.py`: a `RuntimeTool` dataclass with the 13 fields listed in the requirement (`name`, `server_key`, `server_url`, `description`, `input_schema`, `raw_definition`, `status`, `is_write`, `requires_serial`, `resource_scope`, `agent_safety_tier`, `requires_approval`, `enabled_for_llm`).
- New `scripts/shared/runtime_tool_registry.py`: a `RuntimeToolRegistry` class with the 9 methods listed in the requirement (`resolve`, `get`, `all_tools`, `llm_tool_definitions`, `tool_spec_map`, `tool_spec_for_call`, `is_side_effect`, `classify_operation_type`, `apply_policy`).
- Safe-default logic for unknown/insufficiently-annotated tools (`agent_safety_tier = "WRITE_DANGEROUS"`, `requires_approval = True`, `enabled_for_llm = False`, `requires_serial = True` when parallel-execution uncertainty exists).
- New `tests/shared/test_runtime_tool.py` and `tests/shared/test_runtime_tool_registry.py` covering normalization, defaults, lookup, schema generation, and policy application.
- `scripts/agent/context.py`: add a field to hold a `RuntimeToolRegistry` reference (`AgentContext` today has no such slot — see Assumptions).

**Out of scope (deferred to later requirements in this batch)**
- Wiring `RuntimeToolRegistry` into actual routing (`route_resolver.py`) — requirement 04.
- Wiring it into LLM tool schema / help output — requirement 05.
- Wiring it into scheduler metadata / side-effect detection call sites — requirement 06.
- Wiring it into policy/approval classification call sites — requirement 07.
- Building the actual MCP tool-discovery mechanism that populates `RuntimeToolRegistry` at startup (`resolve()`/`get()` need data to operate on) — requirement 03.
- Deleting or modifying `scripts/shared/tool_constants.py`, `scripts/agent/tool_policy.py`, `scripts/shared/tool_executor_helpers.py`, or the existing `scripts/shared/tool_registry.py`/`ToolRegistry` — this requirement only adds the new classes; it does not yet replace any existing consumer (per the requirement's own target-file list, which lists only the two new modules + `tool_spec.py` + `context.py` + their tests).
- Any change to `scripts/shared/tool_spec.py`'s existing `ToolSpec` dataclass shape — see Assumption 3 on why `tool_spec_map()`/`tool_spec_for_call()` can produce `ToolSpec` instances without modifying that file.

## Assumptions

1. **`RuntimeToolRegistry` is a distinct concept from the existing `shared/tool_registry.py`'s `ToolRegistry`/`ToolDefinition`**, which the requirement does not mention at all. `ToolRegistry` (confirmed by direct read) is explicitly scoped as "routing authority only... not a schema/description registry" (`tool_registry.py:13-16`), populated statically from `tool_constants.py` frozensets at import time (`_populate_default_registry()`, lines 181-206), with `description`/`input_schema` fields on `ToolDefinition` explicitly documented as "reserved for future use... never populated... not read by any caller today" (lines 43-46, 51, 54). The new `RuntimeToolRegistry` is a superset in intent (routing + schema + scheduler + policy, all from live-discovered data) but this requirement does not ask to merge, replace, or delete the existing `ToolRegistry` — that decision, if any, belongs to requirement 04 (routing) or 09 (startup validation consolidation), not here. This plan creates `RuntimeToolRegistry` as an additive, currently-unused (until wired) new module, coexisting with the old one for now.
2. **No existing construction site needs modification** — `RuntimeToolRegistry` starts out unpopulated; requirement 03 (MCP tool discovery) is responsible for building it at startup. This plan's job is only to define the shape (`RuntimeTool` fields, `RuntimeToolRegistry` methods) and their normalization/default-application logic, testable against hand-constructed fixture data, not live MCP responses.
3. `RuntimeToolRegistry.tool_spec_map()`/`tool_spec_for_call()` can return the *existing* `ToolSpec` dataclass (`shared/tool_spec.py`, 6 fields: `call_id`, `name`, `args`, `resource_scope`, `requires_serial`, `is_write`) unmodified — `RuntimeTool` already has `is_write`/`requires_serial`/`resource_scope` fields (a superset), so `RuntimeToolRegistry` can construct a `ToolSpec` per call by copying those 3 fields plus the call-specific `call_id`/`args`. No new field is needed on `ToolSpec` itself. (Listed as a target file in the requirement, but confirmed via direct read that no structural change to `ToolSpec` is required — only a new construction path from `RuntimeTool` data, living in `runtime_tool_registry.py`.)
4. `RuntimeTool.agent_safety_tier`'s string values (`"WRITE_DANGEROUS"` per the requirement's safe-default) must align with the existing `_TIER_TO_RISK` mapping's key set in `scripts/agent/tool_policy.py:33-38` (`"READ_ONLY"`, `"WRITE_SAFE"`, `"WRITE_DANGEROUS"`, `"ADMIN"`) and `config/agent.toml`'s `[tool_safety_tiers]` values (lines 207-224) — confirmed these are the same 4-tier vocabulary the requirement's default value draws from, so `RuntimeTool.agent_safety_tier` should be typed as a value restricted to these 4 strings (a `Literal` or small enum), not an arbitrary string, to prevent silent typos.
5. `RuntimeToolRegistry.llm_tool_definitions()` must produce the same JSON-schema shape currently read directly from `config/agent.toml`'s `[[tool_definitions]]` blocks (`name`, `description`, `parameters` — confirmed via `config_dataclasses.py:194`'s `ToolConfig.tool_definitions: list[dict]` and its TOML source) — this is the exact shape the LLM client sends as the tool-calling schema; `RuntimeTool.input_schema` (sourced from MCP's `inputSchema` field, confirmed at `scripts/mcp_servers/mdq/tools.py:21-36` and the base class's `list_tools_with_server_key()`) must be re-keyed to `parameters` for LLM compatibility, since MCP's field name (`inputSchema`) differs from the LLM schema's expected field name (`parameters`).
6. `RuntimeTool.raw_definition` holds the exact dict as received from a server's `/v1/tools` response (`{name, description, inputSchema, server_key}`, confirmed at `scripts/mcp_servers/server.py:128-136`'s `list_tools_with_server_key()`) — `RuntimeTool`'s other fields (`is_write`, `requires_serial`, `resource_scope`, `agent_safety_tier`, `requires_approval`, `enabled_for_llm`) are *not* part of that raw response and must be derived/annotated separately (from `tool_constants.py`-equivalent classification data or explicit per-tool config, itself likely a requirement-03/07 concern) — for requirement 02's scope, these fields simply need to exist and default safely per the "Safe defaults" section when no annotation is supplied.
7. `tests/shared/test_runtime_tool.py` and `tests/shared/test_runtime_tool_registry.py` do not exist yet — confirmed via `ls tests/shared/ | grep runtime_tool` (no output). Both are genuinely new test files, not extensions of existing ones.
8. `scripts/agent/context.py`'s `AgentContext` (195 lines, read in full) currently has no tool-registry-shaped field — tool metadata today lives entirely under `ctx.cfg.tool` (a `ToolConfig`) and `ctx.services_required.tools` (a `ToolExecutor` instance). Adding a `RuntimeToolRegistry` reference requires a new field/property on `AgentContext` or `AppServices` (a design decision to make explicit — see Unknowns).

## Design

### Relationship to the existing `ToolRegistry` (critical clarification not in the requirement)

`shared/tool_registry.py`'s `ToolRegistry` is a narrow, routing-only, statically-seeded registry (tool_name → server_key, from `tool_constants.py` frozensets). The new `RuntimeToolRegistry` is a broader, dynamically-populated (by requirement 03), execution-metadata-rich registry. They are not the same class and this plan does not merge them — `RuntimeToolRegistry.resolve(tool_name) -> server_key` duplicates `ToolRegistry.get_server_for_tool()`'s *purpose* but not its *data source* (static seed vs. live discovery). This duplication is intentional and temporary: once requirements 03-04 land, `RuntimeToolRegistry` becomes the routing authority and the old `ToolRegistry` likely becomes redundant — but that transition is explicitly out of scope here per the Out-of-scope list.

### `RuntimeTool` construction

`RuntimeTool` instances are built by normalizing an MCP `/v1/tools` entry (`{name, description, inputSchema, server_key}`, per Assumption 6) plus annotation data (is_write/requires_serial/resource_scope/agent_safety_tier/requires_approval/enabled_for_llm) from a source not yet defined (requirement 03/07's concern). For requirement 02's scope, the constructor/factory function takes these as explicit parameters with the documented safe defaults applied when omitted — this makes `RuntimeTool` fully testable today with hand-built fixture dicts, independent of any real MCP server being running.

### `RuntimeToolRegistry` method behavior

- `resolve(tool_name) -> server_key`: dict lookup into the registry's internal `{name: RuntimeTool}` map, returning `.server_key`; raises or returns `None` for unknown names (mirror `ToolRegistry.get_server_for_tool()`'s `None`-on-unknown contract for consistency, per Assumption 1's stated duplication).
- `get(tool_name) -> RuntimeTool`: direct lookup, raising a clear error (not silently returning a default-safe stub) for a truly unregistered name — a *registered-but-under-annotated* tool gets safe defaults (per the Safe Defaults section), but a name with zero registry entry is a different failure mode and should not be silently papered over.
- `all_tools()`: return all registered `RuntimeTool` instances (list or iterator — decide based on test-file conventions during implementation).
- `llm_tool_definitions()`: filter to `enabled_for_llm=True` tools, re-key `input_schema` → `parameters` (per Assumption 5) to match the LLM schema shape.
- `tool_spec_map()`: build `dict[str, ToolSpec]` mirroring `tool_runner.py`'s current `_build_tool_meta()` shape, sourced from `RuntimeTool.is_write`/`requires_serial`/`resource_scope` (per Assumption 3).
- `tool_spec_for_call(call_id, name, args)`: single-call variant of the above, filling in call-specific `call_id`/`args`.
- `is_side_effect(tool_name)`: `self.get(tool_name).is_write` — direct field read, replacing `tool_executor_helpers.py`'s frozenset-membership check (once wired by a later requirement).
- `classify_operation_type(tool_name)`: derive an `OperationType` from `RuntimeTool.is_write`/a delete-specific flag if one exists, or fall back to a simple write/read distinction if `RuntimeTool` has no explicit delete flag (flag this gap explicitly in the new module's docstring if `RuntimeTool`'s 13 fields don't include one — the requirement's field list has no `is_delete`; confirm during implementation whether operation-type nuance beyond write/read is achievable from the stated fields alone, or needs a 14th field).
- `apply_policy(cfg)`: re-derive `agent_safety_tier`/`requires_approval`/`enabled_for_llm` for all registered tools from `cfg` (e.g. `cfg.approval.tool_safety_tiers`, `cfg.tool.allowed_tools`) without needing to re-run MCP discovery — this is the "reapply Agent policy after `/reload` without rediscovery" capability requirement 08 will consume.

### Safe defaults

Applied whenever a `RuntimeTool` is constructed without explicit annotation data: `agent_safety_tier = "WRITE_DANGEROUS"`, `requires_approval = True`, `enabled_for_llm = False`, `requires_serial = True` "when uncertainty affects safe parallel execution" (interpreted as: `requires_serial` defaults to `True` whenever `is_write` is also unannotated/defaulted, since an unclassified write tool cannot be assumed safe to run concurrently — this mirrors the existing `write_first` conservative-batching logic recently investigated in `tool_scheduler.py`).

## Implementation steps

1. **Resolve Unknown #1**: read `AppServices`'s full definition to decide whether `RuntimeToolRegistry` lives there or directly on `AgentContext`; follow the existing `ToolExecutor` placement pattern.

3. **Create `scripts/shared/runtime_tool_registry.py`**: define `RuntimeToolRegistry` with an internal `dict[str, RuntimeTool]`, implement the 9 methods per the Design section, and a construction/factory helper that applies the documented safe defaults when annotation data is omitted.

6. **Write `tests/shared/test_runtime_tool_registry.py`**: cover `resolve`/`get`/`all_tools`/`llm_tool_definitions`/`tool_spec_map`/`tool_spec_for_call`/`is_side_effect`/`classify_operation_type`/`apply_policy`, using hand-built `RuntimeTool` fixtures (no live MCP server needed, per Assumption 2).

8. **Verification**: run the standard validation sequence (`rules/toolchain.md`) scoped to the new files, then the full suite to confirm the additive change introduces no regression (nothing consumes the new classes yet, so risk is limited to import/syntax correctness).

## Risks

- **Risk**: Introducing `RuntimeToolRegistry` alongside the existing `ToolRegistry` (two classes with overlapping names/purposes) could confuse future contributors about which is authoritative during the transition window before requirements 04/09 land.
  **Mitigation**: `runtime_tool_registry.py`'s module docstring must explicitly state the relationship (per this plan's Design section) and that `ToolRegistry` remains the routing authority until a later requirement says otherwise — do not let the new module's docstring imply it's already wired in.
- **Risk**: `agent_safety_tier`'s string-literal typing (Assumption 4) could drift from `tool_policy.py`'s `_TIER_TO_RISK` dict if either is updated independently before they're unified by a later requirement.
  **Mitigation**: import the tier vocabulary from a single shared location if one already exists (check `tool_enums.py` first, since `RiskLevel`/`OperationType` already live there) rather than re-declaring the 4 strings inline in `runtime_tool.py`.
- **Risk**: Because this requirement's scope explicitly excludes wiring anything in yet, `RuntimeToolRegistry` (unused in production) risks bit-rotting or being deprioritized before requirements 03-07 actually consume it, becoming dead code that `vulture` or a future cleanup pass flags for removal.
  **Mitigation**: this plan's implementation should proceed together with (or immediately before) requirement 03's, minimizing the window where the new classes sit unused; flag to the user if requirement 03 isn't planned/implemented promptly after this one.
- **Risk**: `classify_operation_type()`'s design (per the Design section) may not be fully achievable from `RuntimeTool`'s 13 stated fields if a `DELETE`/`API_WRITE` distinction (per the existing `OperationType` enum) requires more granularity than a single `is_write` boolean provides.
  **Mitigation**: flag this explicitly during Implementation Step 3 rather than silently collapsing `DELETE`/`API_WRITE` into generic `WRITE` — if the requirement's field list is insufficient, surface this as a follow-up requirement rather than under-implementing the method silently.
