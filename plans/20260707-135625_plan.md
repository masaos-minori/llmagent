# Plan: M-4 — Extend McpProbeResult with lifecycle and health detail fields

## Goal

Add optional lifecycle and health detail fields to `McpProbeResult`, populate them in `McpStatusService`, and display the most important new fields in the `/mcp status` table renderer, enabling process-level observability without breaking existing construction.

## Scope

**In**: `scripts/agent/services/models.py` — add 11 optional fields to `McpProbeResult`; `scripts/agent/services/mcp_status.py` — populate new fields in `_probe_single_server()`; `scripts/agent/commands/cmd_mcp.py` — update `_format_mcp_table()` to display key new fields.

**Out**: No changes to `mcp_health.py`, `factory.py`, `lifecycle.py`, config files, or DB schema. No changes to `McpProbeResult` frozen constraint.

## Assumptions

1. `McpProbeResult` is `frozen=True` — all new fields require defaults (`None` or `False`) for backward compatibility.
2. Lifecycle state is available via `ctx.services_required.lifecycle.get_transport_state(server_key)` (returns `LifecycleState`).
3. Process PID and PGID require access to `_http_procs` / `_http_pgids` in `HttpServerLifecycleManager`; `_ServerLifecycleRouter` does not currently expose them. A read-only snapshot method must be added to `_ServerLifecycleRouter` (or `LifecycleManagerProtocol`) to expose PID/PGID.
4. `last_exit_code`, `last_shutdown_result`, and `stderr_log` are best-effort; `None` when unavailable.
5. Table renderer width may need adjustment; display only `lifecycle_state`, `pid`, `running`, `restart_recommended` to keep the table readable.

## Implementation Steps

1. **Add optional fields to `McpProbeResult` in `models.py`**
   - Add to `McpProbeResult` dataclass (all `= None` or `= False` defaults):
     ```python
     managed: bool = False
     pid: int | None = None
     pgid: int | None = None
     running: bool | None = None
     lifecycle_state: str = ""
     last_exit_code: int | None = None
     last_shutdown_result: str = ""
     restart_recommended: bool = False
     operator_action_required: bool = False
     health_reason: str = ""
     stderr_log: str = ""
     ```
   - Criterion: existing `McpProbeResult(key=..., ...)` construction without new fields still works.

2. **Add `get_process_snapshot()` to `HttpServerLifecycleManager`**
   - Returns `dict` with `{pid: int | None, pgid: int | None, running: bool}` for a given `server_key`.
   - `running`: `proc.poll() is None` if process exists, else `False`.
   - `pid`/`pgid`: from `_http_procs` and `_http_pgids` dicts (H-8 adds `_http_pgids`).
   - Criterion: returns `None` values for unknown keys.

3. **Surface snapshot in `_ServerLifecycleRouter`**
   - Add `def get_process_snapshot(self, server_key: str) -> dict | None` delegating to `_http_mgr.get_process_snapshot(server_key)`.
   - Criterion: callable from `McpStatusService` via `ctx.services_required.lifecycle`.

4. **Populate new fields in `McpStatusService._probe_single_server()`**
   - Get `lifecycle_state` from `ctx.services_required.lifecycle.get_transport_state(server_key).value`.
   - Get snapshot from `ctx.services_required.lifecycle.get_process_snapshot(server_key)` if method exists.
   - Populate `managed`, `pid`, `pgid`, `running` from snapshot.
   - `restart_recommended`: True if `lifecycle_state == "failed"`.
   - Criterion: `McpProbeResult` populated with non-None values for subprocess-mode servers.

5. **Update `_format_mcp_table()` in `cmd_mcp.py`**
   - Add `LIFECYCLE` column displaying `r.lifecycle_state or "-"` and `PID` column with `r.pid or "-"`.
   - Keep table width manageable; display only high-value fields.
   - Criterion: `/mcp status` shows lifecycle state and PID for subprocess servers.

6. **Add tests**
   - Test: `McpProbeResult` construction with and without new fields.
   - Test: `McpStatusService` populates `lifecycle_state` from router.
   - Test: `McpStatusService` populates `pid`/`running` from snapshot.
   - Test: table renderer includes lifecycle column.
   - Criterion: `uv run pytest tests/ -k "mcp_status or cmd_mcp" -v` passes.
