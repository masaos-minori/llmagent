## 1. Goal
- Align `RagConsistencyReport`, `summarize_issues()`, `/db consistency` CLI output, and the operations documentation so that affected identifiers and repair hints match what the doc promises.

## 2. Scope
- **In-Scope**:
  - Fix `summarize_issues()` logic: `fts_orphan_count` branch incorrectly reuses `affected_doc_ids` from the FTS-gap path; it should report that no chunk-level IDs are available for orphan FTS entries and guide operators to use `/db rag rebuild-fts`.
  - Add `affected_orphan_chunk_ids` and `affected_orphan_urls` to the `RagConsistencyReport` field table in `docs/03_rag_05_configuration_and_operations.md`.
  - Update the doc's CLI failure example to show a realistic issue line that includes affected identifiers.
  - Expand `tests/test_rag_consistency.py`: add test cases for vec/chunk mismatch repair guidance containing "ingester.py --force" if not already present (currently exists), and verify that `summarize_issues()` for `fts_orphan_count` does not spuriously include FTS-gap doc_ids.
  - Ensure `test_db_consistency_detail.py` covers CLI output that contains affected identifiers in issue lines.
- **Out-of-Scope**:
  - Automatic repair execution in `check_rag_consistency()`.
  - Background self-healing.
  - Changing ingestion pipeline behavior.
  - DB schema changes.

## 3. Assumptions
- The existing `affected_chunk_ids`, `affected_doc_ids`, `affected_orphan_chunk_ids`, `affected_orphan_urls` fields in `RagConsistencyReport` are correct and sufficient; no new fields are needed.
- `_db_consistency()` in `cmd_db.py` already passes issue strings (which contain identifiers) to `write_error()`; no structural change to the CLI handler is needed beyond confirming the issue strings are complete.
- URL resolution via the `documents` table JOIN is feasible for `affected_orphan_urls`; the existing query is correct (returns `None` when chunks are truly orphaned and no document record exists, which is expected).
- The "Needs confirmation" marker for URL resolution is not needed because the existing query handles the NULL case gracefully.

## 8. Risks & Mitigations
- **Risk**: Changing `summarize_issues()` string content could break tests that do exact string matching on issue messages. **Mitigation**: Review all assertions in `test_rag_consistency.py` and `test_db_consistency_detail.py` that match on the `fts_orphan_count` branch output before modifying; update test assertions in the same commit.
- **Risk**: The `fts_orphan_count` branch currently references `affected_doc_ids`, which is None in the pure-orphan case — the `if report.affected_doc_ids:` guard means no incorrect detail line is actually emitted today. The fix is therefore purely cosmetic / correctness rather than a behavior change in the common case. **Mitigation**: Low risk; add a regression test to lock in the correct behavior.
- **Risk**: Documentation update might diverge from code if fields are renamed in future refactoring. **Mitigation**: The field table in the doc maps 1:1 to the `@dataclass(frozen=True) RagConsistencyReport` fields; enforced by the existing docstring and field comments in `maintenance.py`.
