# Plan: Remove or harden unsafe direct-execution fallback from workflow mode

Source: `requires/20260625_03_require.md`

---

## Goal

workflow 実行が利用不可のときに発生する direct-execution fallback を audit イベントに記録し、protected 環境では hard error にし、CLI に fallback 状態を表示することで、safety feature のサイレント bypass を排除する。

---

## Scope

**In-Scope**
- `agent/orchestrator.py`: `_log_fallback()` を audit イベント出力に置き換え、`workflow_mode=required` 以外でも fallback を明示化
- `agent/config.py` または `shared/mcp_config.py`: `workflow_mode` 設定の確認・`protected` モード追加検討
- `config/common.toml`: `workflow_mode` のデフォルト値確認
- CLI 表示: fallback 時に REPL で通知
- docs: fallback policy の文書化

**Out-of-Scope**
- workflow engine 本体の変更
- agent/workflow/ の内部ロジック変更

---

## Assumptions

1. `orchestrator.py` の `_log_fallback()` は現行 `workflow_mode=required` のみ RuntimeError を raise し、`auto` は warning ログのみ。
2. `workflow_mode` の値は `"auto"` / `"required"` / `"disabled"` の 3 択。
3. fallback audit イベントは現状ログにのみ出力され、audit_logger には送られていない。

## Implementation Steps

### Phase 1: Audit
- [ ] 1-1. `orchestrator.py` の全 fallback パスを列挙: `_workflow_def is None`, `WorkflowLoadError`, `_log_fallback()` 呼び出し元
- [ ] 1-2. `factory.py` で `workflow_mode` が config から渡される箇所を確認

### Phase 2: 実装
- [ ] 2-1. `_log_fallback()` に `audit_logger` への fallback イベント送出を追加
- [ ] 2-2. fallback 発生時に `on_error` コールバックまたは `emit_*` で CLI に fallback 通知を出力
- [ ] 2-3. `workflow_mode=required` の動作を「protected mode」として docs に明記

### Phase 3: ドキュメント
- [ ] 3-1. docs に fallback が許可される条件・拒否される条件を記述

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| fallback 通知を on_error に流すと既存エラーハンドラが二重反応する | 低 | fallback 専用の emit 関数を追加して on_error とは分離する |
| `workflow_mode=required` の意味が変わることで既存設定が壊れる | 低 | 既存 required モードの動作は変えず、新規 audit イベントを追加するのみ |
