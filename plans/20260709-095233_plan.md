# Plan: Remove deferred auth_token reload for MCP servers

Source requirement: `requires/20260708_26_require.md` (H-4)

## Goal

Stop `/reload` from assigning `old_srv.auth_token = new_srv.auth_token` or
reporting auth token changes as deferred; report them as
`mcp/<server_key>.auth_token` in `ConfigReloadOutcome.needs_restart` instead.

  H-1/H-2/H-3 (`plans/20260709-094640_plan.md`,

  requirement is the `auth_token`-specific slice.

  behavior, not for reload classification; none reference
  `config_reload.py` or deferred/needs_restart auth token handling.

## Design

No new production design beyond H-1's `_classify_mcp_server_changes()`
(`plans/20260709-094640_plan.md` § Design). That design already:
- never assigns `old_srv.auth_token = new_srv.auth_token`,
- emits `mcp/<key>.auth_token` into `needs_restart` when `auth_token` differs,
- never appends to `deferred` for any MCP field.

Test addition:

```python
def test_auth_token_change_reports_restart_not_deferred(self) -> None:
    svc, old_srv = self._make_svc(old_auth="old_token")
    new_srv = McpServerConfig(
        transport=TransportType.HTTP,
        url="http://localhost:8080",
        cmd=[],
        auth_token="new_token",
    )
    result = self._run(svc, {"svc": new_srv})

    assert "mcp/svc.auth_token" in result.needs_restart
    assert not any("auth_token" in item for item in result.deferred)
    assert not any("auth_token" in item for item in result.applied)
    assert old_srv.auth_token == "old_token"  # unchanged
```

This replaces the old `test_auth_token_change_populates_deferred` test in
`tests/test_config_reload.py::TestDeferredReload` (see H-1 plan step 3),
which asserted the exact opposite (deferred, mutated) behavior.

## Risks

- **Risk**: Removing the deferred bucket for `auth_token` changes the
  user-visible `/reload` message from "deferred to next connection" to
  "requires restart" — a stronger, more disruptive-sounding message for what
  was previously treated as a soft update.
  → **Mitigation**: This is the explicit intent of the requirement
  (`auth_token` changes require full agent restart per Background:
  `HttpTransport` already snapshotted the old token) — not an unintended
  regression.
- **Risk**: If any external tooling or documentation describes `/reload`
  auth-token updates as "deferred," it will become stale.
  → **Mitigation**: Confirmed stale in `docs/05_agent_08_configuration.md`
  (lines 70, 72, 87-89) and `docs/05_agent_07_cli-and-commands.md` (lines
  255-256) while planning `requires/20260708_34_require.md` (M-2); the fix
  is scoped there (`plans/20260709-100635_plan.md` § Design), not in this
  plan.
