# Plan: Align plugin prefix command boundary matching with built-in commands

## Goal

Fix `CommandRegistry._dispatch_plugin()` to use word-boundary prefix matching (`line == cmd_name or line.startswith(cmd_name + " ")`) instead of bare `startswith`, preventing false-positive dispatch when a plugin command name is a prefix of a longer unrelated command.

## Scope

### In-Scope

- Change the prefix-match condition in `_dispatch_plugin()` from `line.startswith(cmd_name)` to `line == cmd_name or line.startswith(cmd_name + " ")`
- Change args extraction to use `.strip()` on the suffix
- Add pytest cases covering the boundary-matching behavior to the existing plugin dispatch tests

### Out-of-Scope

- Plugin command shadowing policy
- Plugin reload behavior
- Built-in command dispatch logic (`dispatch()` method) ‚Äî already correct
- Any changes to `shared/plugin_registry.py`

## Assumptions

1. The only code path that dispatches plugin commands is `CommandRegistry._dispatch_plugin()` in `scripts/agent/commands/registry.py`; no other caller performs its own prefix matching.
2. `plugin_registry.iter_commands()` returns a snapshot dict of `{cmd_name: (handler, is_prefix)}`; the iteration contract is stable.
3. Adding `.strip()` to args is safe ‚Äî existing plugin handlers that receive args already expect stripped strings, consistent with how built-in prefix commands expose args (see `dispatch()` line 123).
4. No plugin in production is intentionally relying on the loose `startswith` behavior (i.e., no plugin command is designed to match `/foobar` when registered as `/foo`).

## Implementation Steps

### Step 1 ‚Äî Fix `_dispatch_plugin()` boundary condition

**File:** `scripts/agent/commands/registry.py`, method `_dispatch_plugin()` (lines 140-155)

Replace:
```python
if is_prefix and line.startswith(cmd_name):
    args = line[len(cmd_name):]
```
With:
```python
if is_prefix and (line == cmd_name or line.startswith(cmd_name + " ")):
    args = line[len(cmd_name):].strip()
```

The `elif not is_prefix` branch is unchanged.

**Completion criteria:**
- `_dispatch_plugin()` no longer matches `/foobar` when only `/foo` is registered with `prefix=True`
- `/foo` (exact) and `/foo bar` (with trailing args) still dispatch and pass the correct args

### Step 2 ‚Äî Add targeted dispatch boundary tests

**File:** `tests/test_dispatch_plugin_boundary.py` (new file)

Write a `TestDispatchPluginBoundary` class using `pytest.mark.asyncio`. Use `shared.plugin_registry._reset_for_testing()` as an `autouse` fixture. Register a `/foo` prefix command and an `/exact` non-prefix command, then verify:

| Scenario | Input | Expected result |
|----------|-------|-----------------|
| Exact match of prefix command | `/foo` | dispatched, args = `""` |
| Prefix match with space | `/foo bar` | dispatched, args = `"bar"` |
| Prefix match with extra spaces | `/foo  bar` | dispatched, args = `"bar"` (stripped) |
| False-positive guard | `/foobar` | NOT dispatched (returns `False`) |
| Exact command exact match | `/exact` | dispatched |
| Exact command with trailing text | `/exact bar` | NOT dispatched (returns `False`) |
| Unknown command | `/unknown` | returns `False` |

**Completion criteria:**
- All 7 scenarios pass
- No existing tests regress (`uv run pytest` clean)
