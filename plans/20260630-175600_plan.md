# Plan: Clarify and test `consumer_id` sanitization for offset files

## Goal
Documentation and implementation of `_sanitize_consumer_id` agree on canonical behavior, and all required edge-case tests exist and pass.

## Scope
**In-Scope**:
- Align `docs/06_eventbus_04_dlq_offsets_and_delivery_semantics.md` with the actual sanitization logic in `scripts/eventbus/offsets.py`
- Add any missing test cases to `tests/test_eventbus_offsets.py` for the cases listed in the requirement

**Out-of-Scope**:
- Changing the sanitization algorithm itself (implementation is correct and all tests pass)
- Adding input validation at the HTTP layer (consumer_id length limit, charset enforcement)
- Windows-style backslash handling

## Assumptions
- The current implementation (`replace("..", "_").replace(".", "_").replace("/", "_")`) is the intended canonical behavior; no algorithmic change is required.
- The doc's phrase "replacing `.`, `/`, and `..` with `_`" is an imprecise summary; the precise order-dependent behavior (`.` included, `..` first to avoid double-replacement) needs to be documented explicitly.
- The empty-string fallback to `"default"` is intentional and must be documented.
- Long consumer IDs (>255 chars) are accepted as-is; the filesystem enforces name length limits. This is a known limitation but is out of scope for this ticket.
