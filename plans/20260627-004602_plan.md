## Goal

Add operational regression tests for Event Bus delivery, health, offsets, and DLQ behavior to lock down invariants discovered during require processing.

## Scope

**In-Scope**:
- Add tests for gaps identified in current test coverage:
  - crash-before-ack replay
  - replay-to-live transition race
  - slow consumer queue overflow and replay recovery
  - unsafe public bind guard
  - concurrent publish/replay/ack stress behavior
- Keep tests deterministic and isolated using temporary directories
- Encode delivery and recovery invariants directly

**Out-of-Scope**:
- Large-scale load testing
- Distributed multi-node tests
- Benchmarking throughput

## Implementation Steps

1. **Phase 1: Add crash-before-ack replay test**
   - Test: Consumer connects, receives events, disconnects without acking → on reconnect with same consumer_id, replays from seq=0 (offset not found)
   - Verify: offset file not written for unacked events

2. **Phase 2: Add replay-to-live transition race test**
   - Test: Event published during replay phase is delivered via live push (not lost)
   - Test: Event with seq <= replay_ceil is deduped against replay ceiling
   - Use deterministic synchronization (asyncio.Queue, asyncio.Event) instead of timing-based assertions

3. **Phase 3: Add slow consumer queue overflow test**
   - Test: Slow consumer with queue depth >= 100 triggers slow_consumer_count in /health
   - Test: Event published when queue is full is dropped (WARNING logged)
   - Verify: queue maxsize=1000, slow threshold=100

4. **Phase 4: Add unsafe public bind guard test**
   - Test: Start Event Bus with bind address 0.0.0.0 and allow_public_bind=False → fail-fast or critical warning
   - Test: Start Event Bus with bind address 127.0.0.1 → startup succeeds

5. **Phase 5: Add concurrent publish/replay/ack stress test**
   - Test: Concurrent publish requests (10+ simultaneous) → no SQLite lock errors
   - Test: Concurrent ack requests on same event_id → idempotent, no corruption
   - Test: Concurrent replay with overlapping time ranges → no duplicate delivery

6. **Phase 6: Add DLQ re-promotion edge case tests**
   - Test: Requeue of event at delivery_failure_count >= max_retry → requeue succeeds but next DLQ loop tick will re-promote
   - Test: dlq_requeue_count increments on each requeue (not reset)
