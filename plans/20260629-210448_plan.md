## Goal
consumer ID の安定性モデルを明確化し、「PID ベース ID はプロセス再起動後に変わるため offset resume できない」という事実を docs に明示する。サーバーが consumer ID を自動生成しないことを確認する。

## Scope
**In-Scope**:
- `scripts/eventbus/offsets.py:_make_consumer_id()` が Deprecated 関数であり、サーバーで使用されていないことの確認
- `docs/06_eventbus_04_dlq_offsets_and_delivery_semantics.md:L50` の stability requirement 記述確認
- `docs/06_eventbus_03_persistence_schema_and_replay.md` の consumer ID 説明確認
- restart resume テストのカバレッジ確認
- sha256(name:hostname:pid) 形式の注意点（プロセス再起動後の PID 変化）が docs に明示されているかの確認

**Out-of-Scope**:
- multi-node consumer group coordination
- exactly-once デリバリー
- サーバーサイドの consumer registry 実装

## Assumptions
1. `offsets.py:L12-20`: `_make_consumer_id()` は "Deprecated: ... never auto-generated by the server" と docstring に記述済み（確認済み）
2. `offsets.py:L17`: "Consumer IDs are always client-supplied via the `consumer_id` query parameter — never auto-generated by the server" — サーバーは自動生成しない
3. `06_eventbus_04:L50`: "Consumer ID stability requirement: Consumer IDs must be stable across restarts... Recommended: use application-level identifiers" — 既に正しい記述
4. `tests/test_eventbus_restart_resume.py` が再起動後の resume シナリオをカバーしている
