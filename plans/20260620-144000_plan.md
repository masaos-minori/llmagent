# Plan: Update Agent Routing to Explicitly Separate MDQ and RAG Usage (20260620_05)

## Goal
Implement an Agent-side routing policy that chooses MDQ or RAG based on query type, with explicit override support and documented fallback behavior.

## Scope
- New module: `scripts/agent/mdq_rag_classifier.py` — lightweight query classifier.
- Modify: `scripts/agent/orchestrator.py` — integrate classifier into turn flow.
- New config option: `mcp_servers.mdq_rag_mode` (override: `mdq`, `rag`, `auto`).
- Docs: update `docs/04_mcp-mdq.md` with routing policy section.
- Estimated 3-4 files modified/created.

## Task Size
Medium (Path B): new module + orchestrator integration + config + docs.

## Current State Analysis

### How tool selection works today
1. User sends a message → AgentSession processes it.
2. LLM generates response with optional `tool_calls`.
3. `ToolRouteResolver` routes each tool call to the correct server (mdq, rag_pipeline, etc.) based on tool name.
4. Tool results are returned to the LLM.

**Gap**: There is no classifier or policy that guides *which* system (MDQ vs RAG) should handle a given query. The LLM decides based on tool descriptions and prompt context alone.

### Existing infrastructure
- `ToolRouteResolver` (shared): routes tool_name → server_key. Already has MDQ_TOOLS and RAG_TOOLS sets.
- System prompts: configurable via `ctx.conv.system_prompt_content`.
- Config: `mcp_servers` config has `tool_names` per server, but no routing policy.

## Design

### Query classifier: `agent/mdq_rag_classifier.py`
A lightweight, deterministic classifier that returns `MdqRagMode` (`mdq`, `rag`, or `auto`).

**Heuristics for MDQ detection:**
- Query contains Markdown-specific terms: "heading", "outline", "hierarchy", "section", "markdown structure", "TOC", "table of contents"
- Query references `.md` files explicitly
- Query asks about document structure/formatting

**Default to RAG:**
- All other queries (semantic search, general retrieval, multi-format content).

**Override mechanism:**
- Config option `mcp_servers.mdq_rag_mode`:
  - `"auto"` (default): use classifier heuristics.
  - `"mdq"`: force MDQ for all retrieval queries.
  - `"rag"`: force RAG for all retrieval queries.

### Integration point: `agent/orchestrator.py`
- Before the LLM turn, run the classifier on the user's query.
- If mode is determined (mdq or rag), inject a system prompt hint that restricts tool selection guidance.
- Example injection: `"For this query, use Markdown-structural tools (mdq-mcp) for outline/heading queries."` or `"Use RAG tools for semantic/general retrieval."`

### Fallback behavior
- If MDQ is selected but mdq-mcp server is unavailable → log warning, fall back to RAG.
- If RAG is selected but rag-pipeline-mcp is unavailable → return error (no fallback).
- Override mode bypasses classifier; if the forced server is unavailable → return error.

## Implementation Steps
1. Create `scripts/agent/mdq_rag_classifier.py`:
   - `MdqRagMode` enum: `AUTO`, `MDQ`, `RAG`.
   - `classify_query(query: str) -> MdqRagMode`: heuristic-based classifier.
   - `resolve_mode(query: str, config_mode: str | None) -> MdqRagMode`: combines classifier with config override.
2. Modify `scripts/agent/orchestrator.py`:
   - Add `_classify_and_inject_mode(query: str) -> None` method.
   - Call before LLM turn in the main loop.
3. Add config option in `config/agent.toml` (or relevant config):
   - `[mcp_servers] mdq_rag_mode = "auto"` (default).
4. Update `docs/04_mcp-mdq.md`:
   - Add "Agent Routing Policy" section describing the classifier, override, and fallback behavior.

## Validation Plan
- Test classifier with sample queries:
  - MDQ-triggering: "show me the outline of this file", "what are the headings in X.md"
  - RAG-default: "search for information about X", "find similar documents"
- Verify override mode works (config-driven).
- Verify fallback logging when MDQ server is unavailable.
- Unit tests for `classify_query()` and `resolve_mode()`.

## Risks
- **Medium**: Heuristic classifier may produce false positives/negatives. The override mechanism mitigates this. Classifier heuristics should be documented and tunable.
- **Low**: System prompt injection increases token count slightly (~20-40 tokens per turn). Acceptable given the routing clarity benefit.
- **Medium**: If the LLM ignores the system prompt hint, the classifier has limited effect. This is a known limitation of LLM-driven tool selection; the override mode is the reliable path.

## Routing Priorities (per requirement)
1. Correctness of responsibility boundary — MDQ for Markdown structural, RAG for general/semantic.
2. Predictability of behavior — deterministic classifier + explicit override.
3. Safe and documented fallback — log warnings, not silent failures.
