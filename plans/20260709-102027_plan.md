## Risks

- **Risk**: Same as L-1 — this plan performs no changes of its own, relying
  entirely on M-2 and H-10; if either is implemented only partially, this
  requirement's acceptance criteria could go unverified.
  → **Mitigation**: The grep-based test plan above is a cheap, requirement-
  specific final check, matching this requirement's own test plan verbatim
  (`[DEFER] - mcp`, `github_server_url` searches) — run it independently of
  whether M-2/H-10 report themselves complete.
- **Risk**: The three example lines added to the sample output
  (`docs/05_agent_07_cli-and-commands.md`) are illustrative, not derived
  from an actual `/reload` run; if the real rendered order differs (e.g.
  fields sorted alphabetically vs. `_MCP_SERVER_FIELDS` order), the example
  could look inconsistent with real output.
  → **Mitigation**: Low impact — the example's purpose is to show the
  format and category, not to be byte-exact; `_MCP_SERVER_FIELDS` order
  (`transport, url, ..., auth_token, ..., cmd, env`) already places `url`
  before `auth_token` before `startup_mode`... note `startup_mode` actually
  precedes `auth_token` in `_MCP_SERVER_FIELDS` — reorder the doc's example
  to `url`, `startup_mode`, `auth_token` to match real iteration order
  exactly if byte-level consistency is desired during implementation.
