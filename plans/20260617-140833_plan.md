# Plan: [RAG][DB] Add consistency checks for chunks_fts and chunks_vec

## Goal

Add automated consistency checks for RAG derived tables (`chunks_fts`, `chunks_vec`) so that trigger-based sync failures and orphan records are detected early.

## Scope

**In scope:**
- `scripts/db/maintenance.py` — add RAG consistency check functions
- New file: `tests/test_rag_consistency.py` (or add to existing test file)
- Read-only reference: `scripts/db/schema_sql.py` — FTS5 trigger definitions (`chunks_ai`, `chunks_ad`, `chunks_au`)
- Documentation update in `03_rag_05_configuration_and_operations.md`

**Out of scope:**
- Auto-repair logic (detection only; repair is manual or via separate task)
- Changes to existing memory consistency check (`agent/memory/store.py:check_consistency()`) — RAG and memory use different DBs
- Schema changes to chunks/chunks_fts/chunks_vec tables

## Assumptions

1. `chunks_fts` is a FTS5 virtual table synced via triggers (`chunks_ai`, `chunks_ad`, `chunks_au`) — no direct INSERT/UPDATE from application code.
2. `chunks_vec` has NO foreign key constraint (sqlite-vec virtual table limitation) — orphan records can exist if delete triggers fail.
3. The existing memory store already has a `check_consistency()` pattern (`agent/memory/store.py:258`) returning a `ConsistencyReport` dataclass — RAG can follow the same pattern.
4. `maintenance.py` already handles WAL checkpoint, VACUUM, rotation, and corruption recovery — consistency checks fit as another maintenance operation.
5. FTS5 virtual tables support `SELECT count(*) FROM chunks_fts` (returns rowid count).

## Implementation Steps

1. **Add dataclass** for RAG consistency report:
   - `@dataclass(frozen=True) class RagConsistencyReport:` with fields: `chunks`, `fts`, `vec`, `orphan_vec_count`, `fts_gap`
   - `chunks`: count of rows in `chunks` table
   - `fts`: count of indexed rows in `chunks_fts`
   - `vec`: count of rows in `chunks_vec`
   - `orphan_vec_count`: chunks_vec rows whose chunk_id has no matching row in chunks
   - `fts_gap`: difference between chunks and fts counts (should be 0)

2. **Add `check_rag_consistency(db)` function** to `maintenance.py`:
   - Query: `SELECT COUNT(*) FROM chunks` → chunks count
   - Query: `SELECT COUNT(*) FROM chunks_fts` → fts count
   - Query: `SELECT COUNT(*) FROM chunks_vec` → vec count
   - Query: `SELECT COUNT(*) FROM chunks_vec WHERE chunk_id NOT IN (SELECT chunk_id FROM chunks)` → orphan vec count
   - Return `RagConsistencyReport` with all counts and gap calculation

3. **Add helper to detect issues**:
   - `is_consistent(report) -> bool` — returns True when `fts == chunks`, `orphan_vec_count == 0`, and `vec == chunks`
   - `summarize_issues(report) -> list[str]` — human-readable issue descriptions

4. **Create consistency tests**:
   - `test_consistency_report_counts_match` — insert known rows, verify counts match
   - `test_orphan_vec_detected` — manually insert into chunks_vec without chunks entry, verify orphan count > 0
   - `test_fts_gap_detected` — delete from chunks (trigger should remove fts entry), verify gap = 0 after clean delete
   - `test_fts_gap_after_broken_delete` — simulate broken trigger: delete from chunks but not fts, verify gap detected
   - `test_consistency_after_force_reinsert` — full force-reinsert cycle (delete order: vec → chunks → docs → re-insert), verify consistency restored

5. **Update documentation** in `03_rag_05_configuration_and_operations.md`:
   - Add section on consistency checks under maintenance operations
   - Document when to run checks (after large ingestion, after force reinsert, during diagnostics)

6. **Validation:** run `uv run pytest tests/test_rag_consistency.py -v` and confirm all tests pass.
