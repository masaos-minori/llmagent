# Plan: [High][eventbus][delivery] 通常デリバリー失敗のカウント追加

Source: `requires/20260625_24_require.md`

## Goal

コンシューマーによる配信失敗 (nack / タイムアウト) を `delivery_failure_count` でカウントし、DLQ 昇格判定の根拠を実際の配信失敗に切り替える。

## Scope

**In-Scope**
- `scripts/eventbus/dlq.py` の昇格判定を `delivery_failure_count >= max_retry` に変更
- `scripts/eventbus/app.py` に nack/fail ハンドラーを追加し `delivery_failure_count` をインクリメント
- `scripts/eventbus/schema.sql` への `delivery_failure_count` フィールド追加 (req #28 と共有)
- 通常配信失敗カウントのテスト追加

**Out-of-Scope**
- consumer outcome API のエンドポイント設計 (req #25 の対象)
- DLQ のインライン昇格 (req #26 の対象)
- スキーマの全面再設計 (req #28 の対象)

## Assumptions

1. req #25 で定義された consumer outcome API (nack/fail エンドポイント) が実装済みであること
2. req #28 でスキーマに `delivery_failure_count` フィールドが追加済みであること
3. `max_retry` は `delivery_failure_count` に対して適用する (DLQ requeue count とは別)

## Design

```python
# dlq.py — 判定条件変更
rows = db.execute(
    "SELECT ... FROM events WHERE delivery_failure_count >= ? AND dlq_at IS NULL",
    (max_retry,),
).fetchall()

# app.py — nack ハンドラー (req #25 エンドポイントから呼ばれる)
def _increment_failure(event_id: str) -> int:
    cur = conn.execute(
        "UPDATE events SET delivery_failure_count = delivery_failure_count + 1 WHERE event_id = ?",
        (event_id,),
    )
    conn.commit()
    return cur.rowcount
```

## Implementation Steps

1. (req #28 完了後) `schema.sql` に `delivery_failure_count INTEGER NOT NULL DEFAULT 0` 追加
2. (req #25 完了後) `app.py` の nack ハンドラー内で `delivery_failure_count` インクリメント
3. `dlq.py` の昇格条件を `delivery_failure_count >= max_retry` に変更
4. `dlq_requeue` の `retry_count + 1` ロジックを維持 (DLQ requeue は別カウント)
5. テスト追加: nack → failure count 増加 → DLQ 昇格の一連フローを検証
