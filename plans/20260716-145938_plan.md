# Plan: Enforce mandatory workflow execution and startup validation severity policy

Source requirement: `requires/20260716_14_require.md`

## Goal

Close the small number of concrete gaps between this requirement's
documented invariants and the current implementation — most of which
already exist and work correctly — by adding the missing explicit
regression tests, documenting `plan_fn`'s current no-op status, simplifying
one piece of dead defensive code, and publishing a single-source-of-truth
severity table for startup validation checks.

## Scope

**In scope**

- Add an explicit regression test asserting `Orchestrator.handle_turn()`
  always invokes `WorkflowEngine.run()` — currently only implicitly
  exercised via `tests/test_orchestrator.py`'s autouse
  `_patch_workflow_loader` fixture, never asserted directly.
- Document `plan_fn`'s current no-op behavior (`agent/orchestrator.py:199-200`:
  `async def plan_fn() -> str | None: return None`) — either as an explicit
  code comment cross-referencing this decision, or in
  `docs/05_agent_03_01_turn-processing-flow-overview.md` (the existing turn
  -flow doc), per the requirement's own Design intent instruction to
  "document the current no-op behavior... or separate it as future planning
  expansion."
- Simplify or document `Orchestrator.handle_turn()`'s dead
  `if self._workflow_def is None: self._log_fallback(...)` branch
  (`agent/orchestrator.py:169-170`) — this code path is unreachable today
  because `__init__` already raises `RuntimeError` if `WorkflowLoader().load()`
  fails, meaning `self._workflow_def` can never be `None` by the time
  `handle_turn()` runs (construction itself would have already failed).
  Either remove the dead check with a comment explaining why (constructor
  guarantee makes it unreachable) or convert it to an `assert` for defensive
  clarity — this is a design decision, not a bug, so document the choice.
- Publish a single severity-mapping reference table (check name → severity
  → rationale) in `docs/05_agent_10_01_operations-and-observability-startup-and-health.md`
  (the existing startup/health doc — confirmed to not already contain this
  table), built directly from `agent/startup.py`'s actual `add_fatal`/
  `add_warning`/`add_ok` call sites.
- Add regression tests for any severity classification not already covered
  by `tests/test_startup.py` (to be enumerated during implementation by
  cross-referencing the doc table against existing test classes).

**Out of scope**

- Redesigning the workflow engine — explicit non-goal; `WorkflowEngine.run()`'s
  plan → execute (retry) → [approval gate] → verify sequencing already
  matches the requirement's Design intent exactly and needs no change.
- Implementing advanced planning behavior — explicit non-goal; `plan_fn`
  stays a no-op, only its status becomes explicitly documented.
- Adding unrelated startup checks — explicit non-goal.
- Hiding fatal failures behind warnings — explicit non-goal; this plan does
  not change any existing severity classification, only documents the
  classifications that already exist.

## Assumptions

1. `Orchestrator.__init__` already enforces "workflow definition load is
   mandatory" — verified by direct read of `agent/orchestrator.py:127-133`:
   `WorkflowLoader().load()` failure raises `RuntimeError`, which
   necessarily aborts `Orchestrator` construction (no partially-constructed
   instance with `self._workflow_def is None` can exist afterward). This is
   also already tested: `tests/test_startup.py::TestStartupWorkflowPreflight`
   covers `test_aborts_on_missing_workflow_definition`,
   `test_aborts_on_invalid_workflow_json`,
   `test_aborts_on_missing_workflow_schema`.
2. `handle_turn()` always calls `_handle_workflow_engine()`, which always
   calls `await engine.run(task, plan_fn, execute_fn, verify_fn)` —
   verified by direct read; there is no code path in `handle_turn()` that
   bypasses this call once the approval-pending guard passes.
3. `LLMTurnRunner` (the object that actually streams from the LLM) has
   exactly one instantiation site in the codebase
   (`agent/orchestrator.py:122`, inside `Orchestrator.__init__`), and its
   `.run()` method is only ever invoked from `_handle_llm_turn()`, which is
   only ever called from `_process_turn()`, which is only ever called from
   `execute_fn()` inside `_handle_workflow_engine()`'s `engine.run(...)` call
   — verified via `rg -n "LLMTurnRunner\("` finding a single call site. No
   direct fallback path exists structurally today.
4. `ctx.workflow.approval_pending` blocking already exists and is already
   tested — verified by direct read of `handle_turn()`'s guard clause
   (lines 151-164) and `tests/test_orchestrator.py::TestApprovalPendingGuard`
   (`test_handle_turn_blocked_when_approval_pending`,
   `test_handle_turn_not_blocked_when_approval_not_pending`).
5. `agent/shared/health_models.py::StartupCheckStatus` already defines all
   four severities (`OK`, `WARNING`, `FATAL`, `SKIPPED`) with a structured
   `StartupCheckOutcome(source, status, message, remediation)` — this
   already satisfies "each validation result includes check name, severity,
   and reason" at the data-model level; the gap is that no doc currently
   presents this as a single reference table (verified via
   `grep -n "FATAL\|WARNING\|SKIPPED\|severity"
   docs/05_agent_10_01_operations-and-observability-startup-and-health.md`
   returning no matches).
6. `agent/startup.py` already assigns explicit, differentiated severities
   per check — verified by direct read of every `add_fatal`/`add_warning`/
   `add_ok` call site: `security_audit` (warning/ok, plus a fatal path),
   `embedding_dimensions` (ok/fatal — schema mismatch is fatal),
   `readiness` (warning/fatal/ok — some MCP failures warning, others fatal,
   condition to be confirmed during implementation), `tool_definitions`
   (warning/ok), `routing_drift` (warning-only, both success and failure
   paths), `routing_safety_tiers` (warning), `routing_drift_live`
   (fatal/warning/ok), `rag_consistency` (ok/warning). This plan documents
   these existing decisions; it does not change any of them without
   separately flagging a reason to.
7. `tests/test_startup.py::TestStartupRollback` already covers rollback
   shutdown behavior comprehensively (`rollback_on_check_services_failure`,
   `rollback_on_recover_pending_failure`,
   `rollback_shutdown_failure_preserves_original_error`,
   `no_rollback_on_initialize_failure`, `no_rollback_on_start_servers_failure`)
   — no new rollback test is needed unless the doc-table cross-reference
   (Implementation step 4) finds an uncovered severity/check combination.

## Design

This requirement is **already ~80% satisfied by the existing
implementation** — the workflow-mandatory execution model, approval
blocking, retry/halt logic, and 4-severity startup validation model are all
already built and mostly already tested. This plan's design is therefore
narrow and additive: close the specific, concrete gaps found (one missing
explicit test, two small documentation gaps, one dead-code judgment call),
rather than a redesign.

The severity-mapping doc table's format:

```markdown
| Check (`source`) | Severity | Condition | Rationale |
|---|---|---|---|
| `security_audit` | WARNING / FATAL | ... | ... |
| `embedding_dimensions` | OK / FATAL | dimension mismatch is FATAL | schema-level config, cannot be silently wrong |
| `readiness` | OK / WARNING / FATAL | [condition to confirm] | required vs. optional MCP servers |
| `tool_definitions` | OK / WARNING | ... | ... |
| `routing_drift` | OK / WARNING | never FATAL | drift is surfaced but does not block startup |
| `routing_safety_tiers` | OK / WARNING | ... | ... |
| `routing_drift_live` | OK / WARNING / FATAL | ... | ... |
| `rag_consistency` | OK / WARNING | ... | ... |
```

(Exact condition/rationale text to be filled in from full reads of each
check function during implementation — this plan establishes the table's
existence and structure, not its final prose.)

3. **Resolve the dead-code Unknown** with the reviewer, then apply the

   `agent/orchestrator.py:169-170`.
4. **Read every `agent/startup.py` check function in full** (not just the
   `add_fatal`/`add_warning`/`add_ok` call sites already surveyed) to
   extract the exact branching condition and rationale for each severity
   decision, resolving the `readiness` Unknown in the process.
5. **Publish the severity-mapping table** in
   `docs/05_agent_10_01_operations-and-observability-startup-and-health.md`
   using the format above, filled in with step 4's findings.
6. **Cross-reference the table against `tests/test_startup.py`'s existing

7. **Deployment/verification** — documentation and test-only changes plus
   one small orchestrator comment/simplification; no service restart
   needed. Run the full test suite to confirm no regression.

## Risks

- **Risk**: Converting the dead `_workflow_def is None` check to an
  `assert` could be stripped in optimized (`-O`) Python execution, silently
  reintroducing the exact ambiguity this requirement worries about (a
  direct-execution path with no guard) if the codebase or its deployment
  ever runs with assertions disabled.
  **Mitigation**: confirmed no `-O` flag or `PYTHONOPTIMIZE` usage exists in
  `deploy/` scripts before choosing `assert` over a plain `if: raise`;
  prefer the explicit-comment-only resolution if any doubt remains, since
  it carries zero behavioral risk.
- **Risk**: Documenting the severity-mapping table from current code
  behavior risks "locking in" a classification that was actually accidental
  (e.g., `routing_drift` never being FATAL might be an oversight, not a
  deliberate choice).
  **Mitigation**: Implementation step 4 reads each check function's full
  context (not just the call site) specifically to distinguish deliberate
  design from accidental omission; if any classification looks accidental
  during that read, it should be flagged to the user before being written
  into the doc as if it were an intentional decision, rather than silently
  codifying a possible bug as documented behavior.
- **Risk**: This plan intentionally does not re-verify every one of
  `agent/workflow/*`'s other modules (`state_store.py`, `task_ops.py`,
  `attempt_ops.py`, `idempotency_ops.py`, `artifact_ops.py`,
  `workflow_loader.py`, `validate.py`) in this pass — only
  `workflow_engine.py` and `orchestrator.py` were read in full.
  **Mitigation**: none of this requirement's acceptance criteria point at
  those specific modules beyond what `workflow_engine.py`'s already-read
  `run()` orchestrates; if implementation later surfaces a gap in one of
  them, treat it as new evidence requiring a plan update rather than an
  unplanned scope expansion.
