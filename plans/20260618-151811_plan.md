# Plan: Reduce AgentREPL to a UI and Command Adapter Only

Derived from `01_plan.md` using requirement `requires/20260618_21_require.md`.

---

## Goal

`AgentREPL` owns only input loop, command dispatch, and output display; all startup orchestration and runtime coordination logic is extracted into separate modules.

---

## Scope

**In scope:**
- `scripts/agent/repl.py` — strip startup/runtime methods; keep only UI and dispatch
- `scripts/agent/factory.py` — accept new bootstrap orchestration steps
- New module `scripts/agent/startup.py` — host extracted startup orchestration
- `scripts/agent/http_lifecycle.py` — may need minor interface adjustments
- `scripts/agent/stdio_lifecycle.py` — may need minor interface adjustments
- `scripts/agent/lifecycle.py` — may need expansion for new orchestration state
- `tests/test_repl.py` — update tests to match new boundary
- `deploy/deploy.sh` — add new module if needed (already covered by `scripts/` cp-all pattern)

**Out of scope:**
- RAG pipeline internals
- MCP server implementations (server catalog, protocol)
- Configuration format changes (`agent.toml`, `mcp_servers.toml`)
- LLM client or session DB changes
- AgentContext DI hub interface changes

---

## Assumptions

1. `factory.build_agent_context()` already performs most DI wiring; the missing step is calling it and chaining server start / health checks.
2. The watchdog loop can be moved into the startup module as an asyncio task spawn, separate from REPL input.
3. Shutdown coordination (`_close_resources`, `_persist_session_memories`) can stay in REPL as thin delegates to services.
4. Existing test coverage in `tests/test_repl.py` provides a behavior lock for the current REPL responsibilities.

## Implementation Steps

1. **Read and trace `AgentREPL.run()` call chain**
   - List every method called, in order
   - Categorize: (a) must move, (b) can stay, (c) needs interface change
   - Criterion: move is complete when a diff shows `run()` as ~5 lines

2. **Design `startup.py` module**
   - Define `class StartupOrchestrator` with `async startup(ctx, config) -> tuple[Orchestrator, ...]`
   - Extract: MCP server start (`_start_mcp_servers`, `_start_subprocess_servers`), watchdog spawn, health checks, session init, initial prompt, tool definition check
   - Criterion: module passes `ruff` and `mypy`

3. **Update `factory.py`**
   - Add `build_startup_orchestrator(...)` or integrate into `build_agent_context`
   - Criterion: existing tests for factory still pass

4. **Strip `repl.py`**
   - Delete moved methods; replace with calls to `StartupOrchestrator`
   - Keep: `_repl_loop`, `_read_input`, `_should_exit`, `_dispatch_line`, `_run_repl_loop`, `_prompt`, `_n_tools`, `_get_chunk_count`, `_print_startup_banner`, `_close_resources` (thin delegate), `_persist_session_memories` (thin delegate), `_persist_session_diagnostics` (thin delegate)
   - Criterion: `AgentREPL` class shrinks by >50% of method count

5. **Update `lifecycle.py` / `http_lifecycle.py` / `stdio_lifecycle.py`** (if needed)
   - Adjust interfaces if `StartupOrchestrator` needs different lifecycle hooks
   - Criterion: one adjustment per interface gap found in Step 1

6. **Update tests**
   - Refactor `tests/test_repl.py` mocks to target new boundaries
   - Add `tests/test_startup.py` for `StartupOrchestrator`
   - Criterion: `pytest` passes; diff-cover ≥ 90%

7. **Lint, type, arch, security validation**
   - `ruff check scripts/` — 0 errors
   - `mypy scripts/` — no new errors
   - `lint-imports` — 0 violations

8. **Update documentation**
   - Update `docs/05_agent_02_runtime-architecture.md` to reflect new startup boundary
   - Update `docs/05_agent_12_reference-api.md` if public API changes
   - Criterion: docs mention `StartupOrchestrator` and state REPL as UI-only

9. **Commit and deploy**
   - Independent commit per step (or grouped if small)
   - `deploy/deploy.sh` change only if new file needs explicit cp line

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Startup orchestration tightly coupled to REPL internals → extraction requires interface changes downstream | medium | Step 1 (full call-chain trace) surfaces all coupling before coding |
| Tests heavily mock REPL internals → refactoring breaks many tests | high | Run `pytest` after each step; update mocks incrementally |
| New `StartupOrchestrator` duplicates logic from `_ServerLifecycleRouter` / `HttpServerLifecycleManager` | low | Reuse those classes directly; do not reimplement lifecycle management |
| Unknown integration ordering (e.g. session init before/after MCP start) causes regression | medium | Step 1 documents exact order; startup module preserves it |
