# Plan: Document DB store public API boundaries

## Goal
Clearly document in all relevant doc files that callers must import from `db.store`, that `store_protocols.py` is the extension point for new storage contracts, and that `store_impl.py` is the SQLite implementation layer — so future implementors never import from internal modules unintentionally.

## Scope
**In-Scope**:
- `docs/90_shared_05_db_api_and_operations.md` — strengthen §1a with extension workflow and anti-pattern warning
- `docs/90_shared_04_db_architecture_and_schema.md` — add import-boundary note to §2 DB layer structure
- `docs/90_shared_00_document-guide.md` — add "Where are DB store module boundaries?" entry to AI Query Routing Table

**Out-of-Scope**:
- Source code changes (no Python file modifications)
- Schema or DDL documentation changes
- New test files
- `store.py` / `store_protocols.py` / `store_impl.py` docstring changes (already complete)

## Assumptions
- `db/store.py` docstring already contains the full boundary description; doc files should reference the same rules
- §1a in `90_shared_05_db_api_and_operations.md` exists and covers the basics; it needs an explicit extension workflow (add protocol → add impl → export) and a concrete bad-import example
- `90_shared_04` §2 only shows the directory structure without boundary semantics; a brief note is sufficient
- `90_shared_00_document-guide.md` AI Query Routing Table is the primary entry point for AI agents; a single routing entry is the minimal required change
