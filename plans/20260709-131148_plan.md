# Plan: Clarify local/development no-auth behavior for MCP auth_token=""

Source requirement: `requires/20260709_06_require.md`

## Goal

Make the security docs explicitly state that `auth_token=""` is
local/development compatibility behavior, that production requires a
non-empty `auth_token`, and that empty tokens must not be used in
production — with `04_mcp_05_security_and_safety_model.md` and
`04_mcp_06_configuration_and_operations.md` consistent with each other.
No code changes.

## Scope

In scope:
- `docs/04_mcp_05_security_and_safety_model.md`: expand the
  "Authentication (`auth_token`)" section to explicitly frame the empty
  default as local/dev compatibility and add a production-prohibition
  statement.
- `docs/04_mcp_06_configuration_and_operations.md`: add a brief
  cross-referencing note near the `auth_token` field row so this file is
  consistent with (not silent about) the local/production distinction
  `04_mcp_05` already documents in more detail.

Out of scope (per requirement's own "Optional"/"Not required" framing and
lack of a technical justification found during research):
- `/mcp status` "AUTH: none" / "AUTH: bearer" display relabeling
  (`agent/services/mcp_status.py`, `agent/commands/cmd_mcp.py`) — the
  requirement frames this as "Optional Runtime Visibility... Consider
  adding." Research found `/mcp status` **already displays an AUTH column**
  today (`cmd_mcp.py:32` header, `mcp_status.py:75` `auth = bool(cfg.auth_token)`,
  rendered as `"yes"`/`"no"` in `_format_mcp_table`) — see Assumption 2.
  Relabeling `"yes"/"no"` to `"bearer"/"none"` is a cosmetic judgment call
  with no acceptance-criterion forcing it ("If `/mcp status` display is
  changed, tests cover it" — conditional, not required). Not implemented by
  this plan.
- The "Future Consideration" migration to `auth_mode="none"` /
  `allow_no_auth=true` — explicitly "not required in this task" per the
  requirement itself.
- `agent/repl_health.py`, `shared/mcp_config.py` — read for evidence
  (confirmed `auth_token` field/validation and the production enforcement
  point already referenced by `04_mcp_05`) but not modified; no code
  behavior is changing.

## Assumptions

1. `docs/04_mcp_05_security_and_safety_model.md:159-193` already documents
   the `auth_token` field and the local/production enforcement distinction
   reasonably thoroughly (including the enforcement point and the reload
   boundary added in a prior session), but does not explicitly use the
   phrase "local/development compatibility behavior" nor a standalone
   "must not be used in production" warning — verified by reading the
   full section.
2. `docs/04_mcp_06_configuration_and_operations.md` currently mentions
   `auth_token` only in the field-reference table (line 82: `` `auth_token`
   | `str` | `""` | Bearer token for auth (empty = no auth) ``) and the
   Pre-Production checklist (line 837, about *storing* the key via env
   vars, not about *whether* it can be empty) — it has no statement about
   the local/production distinction at all. This is an information gap, not
   a contradiction; "ensure consistency" means adding a cross-reference so
   both docs agree, not resolving a conflict between them.
3. `/mcp status` already renders an AUTH column as `"yes"`/`"no"` —
   verified by reading `scripts/agent/commands/cmd_mcp.py` (`_format_mcp_table`,
   `"yes" if r.auth else "no"`) and `scripts/agent/services/mcp_status.py:75`
   (`auth = bool(cfg.auth_token)`).
4. No code in `shared/mcp_config.py` or `agent/repl_health.py` needs to
   change — the production enforcement behavior itself
   (`audit_security_defaults()` raising on empty `auth_token` in production)
   already exists and already matches what this requirement asks to
   *document*, not change (per Acceptance Criteria: "Production behavior
   remains unchanged. Local/development behavior remains unchanged.").

## Design

**`docs/04_mcp_05_security_and_safety_model.md`** — expand the
"Authentication (`auth_token`)" section (after line 168):

```markdown
**Local/development compatibility:** `auth_token=""` (no Bearer
authentication) is intentional local/development compatibility behavior,
not an oversight. **Empty `auth_token` must not be used in production** —
see [Security Profile](#security-profile-security_profile) below for the
startup-time enforcement that rejects it.
```

**`docs/04_mcp_06_configuration_and_operations.md`** — add immediately
after the `auth_token` field row (line 82):

```markdown

> `auth_token=""` (no Bearer auth) is allowed only in
> `security_profile="local"`; it is rejected at startup in
> `security_profile="production"`. See
> [04_mcp_05 §Authentication](04_mcp_05_security_and_safety_model.md#authentication-auth_token)
> and [§Security Profile](04_mcp_05_security_and_safety_model.md#security-profile-security_profile)
> for the full local/production distinction and enforcement point.
```

## Risks

- **Risk**: This plan intentionally does not implement the "Optional
  Runtime Visibility" `/mcp status` relabeling; if the requirement's author
  actually wanted it done (despite the "Optional"/"Consider" wording), this
  plan under-delivers.
  → **Mitigation**: The requirement's own Acceptance Criteria only makes it
  conditionally required ("If ... changed, tests cover it"); low risk, and
  easy to add as a follow-up if requested.
- **Risk**: Anchor-based cross-references between the two docs could break
  if either file's headings are renamed later.
  → **Mitigation**: Low likelihood within this batch's timeframe; verified
  once at implementation time per Implementation step 3.
