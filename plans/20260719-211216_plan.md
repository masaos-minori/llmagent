## Goal

Replace the fragile, regex/substring-based audit-detail inference in
`scripts/mcp_servers/mdq/mdq_server.py::call_tool()` with values read from structured
execution metadata that each MDQ tool handler produces internally, so that audit fields
(`result_count`, `indexed_count`, `skipped_count`, `deleted_count`, `failed_count`,
`match_count`, `truncated`, `duration_ms`) are always accurate regardless of how the
user-facing formatted text happens to be worded.

## Scope

**In scope:**
- `scripts/mcp_servers/mdq/mdq_server.py` — `call_tool()` audit-detail enrichment block
  (lines ~254–282), replacing string-parsing (`_re.search(r"Indexed:\s*(\d+)", ...)`,

**Out of scope:**

## Design

**Structured metadata wrapper.** Add per-tool `TypedDict`s to `mdq_models.py`, following
the existing style of `SearchResultResult`/`GrepDocMatch`:

```python
class SearchDocsMetadata(TypedDict):
    query_preview: str
    result_count: int       # total matches before truncation
    shown_count: int        # results actually included in output text
    truncated: bool
    total_count: int        # same as result_count today; kept distinct per requirement wording
    duration_ms: float

class IndexPathsMetadata(TypedDict):
    input_path_count: int
    indexed_count: int
    skipped_count: int
    failed_count: int
    duration_ms: float

class RefreshIndexMetadata(TypedDict):
    indexed_count: int
    skipped_count: int
    deleted_count: int
    failed_count: int
    elapsed_seconds: float

class GrepDocsMetadata(TypedDict):
    pattern_preview: str
    path_filter_count: int
    match_count: int
    truncated: bool
    grep_enabled: bool
```

`RefreshIndexMetadata` mirrors the field set of the existing `RefreshSummary` TypedDict
in `indexer.py`; the plan is to reuse `RefreshSummary` directly as the metadata for
`refresh_index` rather than duplicating it, since it already carries exactly the
required fields (`indexed_count`, `skipped_count`, `deleted_count`, `failed_count`,
`elapsed_seconds`) — `refresh_paths()` already returns this; the only change needed is
for `MdqService.refresh_index()` to hand the summary back up instead of discarding it
after `_format_refresh_summary()` builds the text.

**Threading metadata through the tool handlers (resolves UNK-02).** `mdq_server.py`
defines `_DISPATCH_TABLE` and `_handle_*` wrapper functions itself (mdq-local, not
shared). Recommended approach:

1. Each `MdqService` method that needs audit metadata (`search_docs`, `index_paths`,
   `refresh_index`, `grep_docs`) returns a small tuple `(text: str, metadata: <TypedDict>)`
   instead of bare `str`.
2. Each `_handle_*` function in `mdq_server.py` is updated to unpack `(text, metadata)`
   and — since these functions are the ones registered in `_DISPATCH_TABLE`, whose
   declared contract (`Callable[[ToolArgs], Awaitable[str]]`) is enforced by the shared
   `dispatch_tool()` — return only `text` from the handler (preserving today's contract
   with `mcp_servers/dispatch.py`), while stashing `metadata` in a small mdq-local,
   per-call side channel that `call_tool()` reads immediately after `await _dispatch_mdq_tool(...)`.
3. Concretely, the side channel is a plain local variable, not a global: refactor
   `call_tool()` to call the relevant `MdqService` method **once** (as it already does,
   indirectly, via `_dispatch_mdq_tool` → `dispatch_tool` → handler → service method) and
   have the mdq-local dispatch step return both the `DispatchResult` (for the existing
   error-handling/HTTP-response path) and the metadata dict together — e.g. change
   `_dispatch_mdq_tool()`'s return type from `DispatchResult` to a small mdq-local
   `MdqDispatchResult` (`output: str`, `is_error: bool`, `metadata: dict[str, Any]`) that
   wraps a `DispatchResult` plus metadata, without touching the shared `DispatchResult`
   class or `dispatch_tool()` signature itself. `_dispatch_mdq_tool()` is defined in
   `mdq_server.py` (not `mcp_servers/dispatch.py`), so this is a local, zero-blast-radius
   change; it still calls the shared `dispatch_tool()` for the actual dispatch/error
   semantics, then separately retrieves metadata (e.g. handlers write metadata to a
   `contextvars.ContextVar[dict]` set immediately before returning `text`, which
   `_dispatch_mdq_tool` reads right after `dispatch_tool()` returns — safe under asyncio's
   single-threaded-per-task model since each request awaits its own handler to
   completion before the ContextVar is read).
4. `call_tool()`'s audit-detail block (today's lines ~254–282) is replaced with a direct
   read of the metadata dict's fields per tool name, with no regex/`.count()` calls.

**Error-path audit (resolves UNK-04).** Switch `_audit_log(... detail=f"duration_ms={ms:.0f}, error_kind={error_kind}")`
to `_audit_log(... detail=f"duration_ms={ms:.0f}", error_type=error_kind)`, using the
already-existing `error_type` parameter of `_audit_log()` instead of embedding it in the
free-text `detail` string — consistent with `tests/test_audit_log_format.py`'s existing
`error_type` assertions for other MCP servers.

**`index_paths()` counting (new logic, not just wiring).** Add `indexed_count`,
`skipped_count` (non-`.md` or non-existent paths), `failed_count` (exceptions caught
per-path) counters inside `indexer.py::index_paths()`, mirroring the counting pattern
`refresh_paths()` already uses, and return them via the metadata alongside the unchanged
`"Indexing complete"` text.

## Implementation steps

1. **Add metadata types** — add `SearchDocsMetadata`, `IndexPathsMetadata`,
   `GrepDocsMetadata` TypedDicts to `mdq_models.py` (reuse existing `RefreshSummary` for
   refresh_index rather than adding a duplicate type). Independently committable/testable
   (no behavior change yet).
2. **`search.py`** — change `search_docs()` to return `(str, SearchDocsMetadata)`;
   populate from values already computed (`original_total`, `shown`, `truncated`,
   `max_chars`/`chars_used`). Update the one call site in `mdq_service.py`.
3. **`indexer.py`** — add indexed/skipped/failed counters to `index_paths()`; change
   return type to `(str, IndexPathsMetadata)`. Change `refresh_paths()` caller in
   `mdq_service.py` to keep the `RefreshSummary` alongside the formatted text instead of
   discarding it.
4. **`db_grep.py`** — change `grep_docs()` to return `(str, GrepDocsMetadata)` built from
   the already-computed `matches`/`truncated` locals. Update the one call site in
   `mdq_service.py`.
5. **`mdq_service.py`** — update `search_docs`, `index_paths`, `refresh_index`,
   `grep_docs` wrapper methods to propagate `(text, metadata)` tuples instead of bare
   `str` (keep the `[WARNING: Index is being updated...]` suffix logic in `search_docs`
   text-only, unaffected).
6. **`mdq_server.py`** — introduce the mdq-local `MdqDispatchResult`/ContextVar-based
   metadata capture (Design section) around the existing `_DISPATCH_TABLE`/`_handle_*`
   functions; rewrite the `call_tool()` audit-detail block (lines ~254–282) to read
   metadata fields directly, removing `_re.search(...)` and `.count("---")` entirely.
   Switch error-path `_audit_log()` calls to use `error_type=` per UNK-04.
7. **Tests** — extend `tests/test_mdq_service.py` for the new method return shapes;
   create `tests/test_mdq_audit.py` covering: audit detail for `search_docs`,
   `index_paths`, `refresh_index`, `grep_docs`, authorization error, validation error,
   and an explicit assertion that audit detail does **not** depend on output text markers
   (e.g. run a tool whose output text is rephrased/monkeypatched and confirm audit
   fields stay correct — the core regression test for this requirement).
8. **Documentation** — no `docs/04_mcp_04_04_mdq.md` changes expected (doc does not
   describe audit-detail derivation mechanics); verify during implementation that no
   doc references the old parsing behavior.
9. **Deployment/verification** — no `deploy/deploy.sh` change needed (no new files); run
   the full validation sequence below before commit.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Adding real indexed/skipped/failed counting to `index_paths()` (currently absent) changes observable behavior subtly (e.g. a path that previously silently no-op'd now increments a counter) even though the external text stays `"Indexing complete"` | Medium | Keep the change additive-only: counters are new fields that did not exist before, so no existing assertion can regress on them; add `tests/test_mdq_audit.py` cases that pin down expected counts for mixed existing/missing/skip scenarios before merging. |
| The ContextVar-based (or equivalent) metadata side-channel could leak stale metadata across concurrent requests if implemented incorrectly (e.g. using a plain module-level dict keyed by nothing, instead of a proper per-task ContextVar or direct return-value threading) | Medium | Prefer direct return-value threading (`_handle_*` functions return `(text, metadata)` and `call_tool()` is refactored to call them without going through the generic `dispatch_tool()` str-only contract for the success path) over any shared mutable state; if a ContextVar is used, add a concurrency test (`asyncio.gather` of two `search_docs` calls with different queries) asserting each audit record matches its own request. |
| `indexer.py` has the highest recent churn (8 commits) among affected files, raising merge-conflict/regression risk if another concurrent requirement-cycle also touches it | Low–Medium | This plan is planning-only (no code written yet); when implementation starts, re-run `git log --oneline -- scripts/mcp_servers/mdq/indexer.py` and `git diff` against `main` immediately before editing to catch any concurrent changes from sibling requirement cycles in this batch. |
| Changing `MdqService` method return types (`str` → tuple) is a breaking internal-interface change; any caller outside `mdq_server.py`'s `_handle_*` functions that assumes a bare `str` will break silently until `mypy`/tests catch it | Low | Confirmed via `rg` that `mdq_service.py` methods are only called from `mdq_server.py`'s handler functions (mdq-local); `mypy scripts/` in the validation sequence will surface any missed call site since the return type changes from `str` to a tuple type. |
| Reusing the existing `RefreshSummary` TypedDict as `refresh_index` audit metadata couples the audit schema to the indexer's internal summary shape; a future unrelated change to `RefreshSummary` could silently alter audit field names | Low | Acceptable per the requirement's own suggestion to avoid introducing unnecessary duplicate types; document the coupling with a code comment at the audit-emission call site referencing `RefreshSummary`'s definition in `indexer.py`. |
