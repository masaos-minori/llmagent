## Goal
Implement a production-grade /health endpoint for mdq-mcp that accurately reflects server readiness.

## Scope
**In-Scope**:
- Rewrite `/health` endpoint in `scripts/mcp/mdq/server.py` with full health checks
- Add details to health response (db path, document count, chunk count, FTS row count, etc.)

**Out-of-Scope**:
- Changing health check logic for other MCP servers
- Adding new tools or features to mdq-mcp

## Assumptions
1. The required tables are: `sections` (regular table) and `sections_fts` (FTS5 virtual table)
2. Required triggers: `sections_ai`, `sections_ad`, `sections_au`
3. Stale documents = sections where file_mtime doesn't match the current file's mtime

## Implementation Steps
1. **Phase 1: Implement health check logic**
   - Check db_path is configured
   - Try to open SQLite connection
   - Verify sections table exists
   - Verify sections_fts FTS5 table exists
   - Verify triggers exist (sections_ai, sections_ad, sections_au)
   - Execute a basic FTS5 query to verify it works
   - Read index statistics (chunk count, doc count, FTS row count)

2. **Phase 2: Add details to health response**
   - database path
   - document count
   - chunk count
   - FTS row count
   - last indexed timestamp (from MAX(file_mtime))
   - stale document count (sections where file_mtime differs from current file mtime)

3. **Phase 3: Remove stub status from config**
   - Update `config/mdq_mcp_server.toml` — change `status = "stub"` to `status = "production"`
