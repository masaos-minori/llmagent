## 1. Goal

Fix the `mcp.mdq` package import collision with the external `mcp` SDK and the
`vec0.so` path mismatch so that `python -m mcp.mdq.server` starts cleanly and
all MDQ tests pass.

## 2. Scope

- **In-Scope**:
  - Add `scripts/mcp/mdq/__main__.py` to enable `python -m mcp.mdq.server` in
    the dev/CI environment (`uv run python -m mcp.mdq.server`).
  - Fix `vec0.so` path mismatch: `config/common.toml` and
    `scripts/mcp/mdq/service.py` reference `/opt/llm/sqlite-vec/vec0.so` but
    the actual file is at `/opt/llm/sqlite-vec/dist/vec0.so`.
  - Verify all 7 MDQ files are syntax-clean (confirmed clean by this analysis).
  - Fix the pre-existing `test_mdq_hybrid_search.py::TestHybridSearchVectorTable::test_vector_table_created_when_enabled`
    failure caused by the `.so.so` double-extension bug in sqlite3's
    `load_extension` call (the path `/opt/llm/sqlite-vec/vec0.so` is expanded
    to `vec0.so.so` by the SQLite extension loader on this OS).
  - Confirm `/health`, `/v1/tools`, `/v1/call_tool` endpoints are already
    implemented and no stub code remains.

- **Out-of-Scope**:
  - Renaming `scripts/mcp/` to `scripts/mcpx/` or any package-level rename
    (high blast radius; all 11 MCP servers and dozens of tests would break).
  - Implementing hybrid/vector search (MDQ-02 — planned, not in this task).
  - Deploying to `/opt/llm/`.
  - Adding new test cases beyond fixing the existing failure.

## 3. Assumptions

- The production venv (`/opt/llm/venv/`) will have `scripts/` on its
  `PYTHONPATH` (via a `.pth` file or `PYTHONPATH` env var) so
  `mcp.mdq.server` resolves to `scripts/mcp/mdq/server.py` rather than the
  external `mcp` SDK — same as the dev `.venv/` which uses `conftest.py`
  `sys.path.insert`.
- The `.so` double-extension issue only affects the `_init_db` path when
  `use_embedding=True`; the main FTS5 path is unaffected.
- `conftest.py` already adds `scripts/` to `sys.path` for all tests; no
  changes to conftest are required.
- The require description's `**future**` / `**name**` / `**main**` patterns
  were Markdown bold notation describing the errors symbolically — the actual
  source files already have correct `__future__` / `__name__` / `__main__`
  syntax (confirmed by `ast.parse` over all 7 files).

## 8. Risks & Mitigations

- **Risk**: Changing `sqlite_vec_so` path in `config/common.toml` breaks RAG
  pipeline or memory embedding. → **Mitigation**: Only change `service.py` to
  read from config (DRY fix); do not change `common.toml` unless the path is
  confirmed wrong. If the path is correct as `/opt/llm/sqlite-vec/vec0.so`
  (symlink exists in production), the double-extension issue is a dev-env
  artifact only — use `try/except` with fallback to strip `.so` before retry.

- **Risk**: Adding `__main__.py` to `mcp/mdq/` conflicts with the external
  `mcp` SDK's own namespace. → **Mitigation**: The external `mcp` SDK does not
  expose a `mcp.mdq` subpackage (confirmed by inspecting
  `/home/masaos/llmagent/.venv/lib/python3.13/site-packages/mcp/`); the only
  collision is at the top-level `mcp` package. The `__main__.py` will not be
  discoverable as `python -m mcp.mdq.server` unless `scripts/` is first on
  `sys.path` — this is guaranteed in tests (conftest) and production (pth file
  / PYTHONPATH).

- **Risk**: `refresh_paths` and `_delete_file_from_index` in `indexer.py` have
  high cyclomatic complexity (E/D rating). → **Mitigation**: Out of scope for
  this ticket; flag for a dedicated refactoring task.
