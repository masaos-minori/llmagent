## Design

**提案する新スキーマフィールド:**

| Field | Type | Default | 用途 |
|---|---|---|---|
| `delivery_failure_count` | INTEGER NOT NULL | 0 | コンシューマー nack/タイムアウトによる失敗回数 |
| `dlq_requeue_count` | INTEGER NOT NULL | 0 | DLQ リキュー回数 (旧 `retry_count` の DLQ 後インクリメント部分) |
| `acked_at` | TEXT | NULL | コンシューマー ack 時刻 (現在は未使用 → 実際に使用) |

`retry_count` の扱い:
- **選択肢 A (加算的)**: `retry_count` を残して `DEFAULT 0` のまま deprecated とし、新フィールドを追加。既存データを壊さない。
- **選択肢 B (名称変更)**: `ALTER TABLE` で `retry_count` を `delivery_failure_count` に改名。SQLite は `ALTER TABLE RENAME COLUMN` を 3.25.0+ でサポート。既存テストの更新が必要。

## Implementation Steps

採用戦略: **加算的 (選択肢 A)**。`retry_count` は deprecated として残す。

1. `schema.sql` に `delivery_failure_count INTEGER NOT NULL DEFAULT 0` を追加
2. `schema.sql` に `dlq_requeue_count INTEGER NOT NULL DEFAULT 0` を追加
3. `_init_schema()` の `executescript` により既存 DB は `ALTER TABLE ADD COLUMN` が不要 (`CREATE TABLE IF NOT EXISTS` に列を追加するだけで新 DB に反映)
4. 既存の本番 DB 用に migration SQL を `schema.sql` の末尾に追加:
   ```sql
   ALTER TABLE events ADD COLUMN IF NOT EXISTS delivery_failure_count INTEGER NOT NULL DEFAULT 0;
   ALTER TABLE events ADD COLUMN IF NOT EXISTS dlq_requeue_count INTEGER NOT NULL DEFAULT 0;
   ```
5. `dlq.py` の昇格条件を `delivery_failure_count >= max_retry` に変更
6. `app.py` の `/dlq/{event_id}/requeue` で `dlq_requeue_count += 1` に変更 (`retry_count` の変更は廃止)
7. `db.py` の `requeue_event()` facade を更新
8. テスト更新 (`retry_count` 直接 UPDATE を `delivery_failure_count` に変更)
