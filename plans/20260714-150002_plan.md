# Work Plan: Clarify Responsibility Boundary Between config/agent.toml and config/*_mcp_server.toml

## Goal

Clarify the configuration ownership boundary between the agent process and MCP server-local configuration across all documentation. Eliminate ambiguity about which file owns which settings.

## Scope

**In-Scope:**
- Replace overly broad statements such as "all configuration keys are defined in config/agent.toml"
- Introduce clear two-layer model: agent-side MCP lifecycle/transport vs MCP server-local application config
- State that MCP servers read only their own `*_mcp_server.toml` files
- State that `agent.toml` does not own MCP server-local policy settings
- Document that MCP server lifecycle/transport changes require full agent restart
- Ensure process isolation documentation remains consistent

**Out-of-Scope:**
- Moving server-local application config into `agent.toml`
- Allowing MCP servers to read `agent.toml`
- Changing process isolation rules

## Assumptions

1. The confirmed specification is: `config/agent.toml` contains only agent-side MCP lifecycle and transport entries; `config/*_mcp_server.toml` contains each MCP server's own application configuration.
2. Agent-side MCP fields in `agent.toml`: `startup_mode`, `transport`, `url`, `healthcheck_mode`, `cmd`.
3. Server-local application config includes: allowlists, denylists, resource limits, audit paths, GitHub `allowed_repos`, shell `command_allowlist`, file server `allowed_dirs`.
4. The existing plan format (see `plans/20260714-143629_plan.md`) is the canonical template.

## Design

### Two-Layer Model

```
Layer 1 - Agent Process Configuration (config/agent.toml)
├── mcp_servers.<key>.startup_mode    # subprocess / persistent / none
├── mcp_servers.<key>.transport       # stdio / http
├── mcp_servers.<key>.url             # HTTP endpoint (for http transport)
├── mcp_servers.<key>.healthcheck_mode # probe / manual
└── mcp_servers.<key>.cmd             # Command to start subprocess (for stdio)

Layer 2 - MCP Server-Local Application Configuration (config/*_mcp_server.toml)
├── allowlists / denylists
├── resource limits
├── audit paths
├── allowed_repos / allowed_repos_mode (GitHub-specific)
├── command_allowlist (shell-specific)
├── allowed_dirs (file server-specific)
└── auth_token_env / auth_token_file (secret references)
```

### Change Strategy

1. **Remove** all overly broad statements claiming "all configuration keys are defined in config/agent.toml".
2. **Introduce** explicit two-layer model statement at the top of every affected document.
3. **Restrict** `agent.toml` field lists to the confirmed five agent-side fields only.
4. **Document** that MCP servers read only their own `*_mcp_server.toml` files — no cross-boundary reading.
5. **Update** operational procedures to use two-layer terminology consistently.
6. **Replace** raw `auth_token` with secret-based references (`auth_token_env`, `auth_token_file`).

### Consistency Rules

- After changes, every mention of MCP config must specify which layer it refers to.
- No document should imply that MCP servers can read `agent.toml`.
- No document should imply that `agent.toml` owns MCP server-local policy settings.
