# Plan: Mark mdq-mcp as stub / non-production server

## Goal

Ensure users and AI systems do not mistake `mdq-mcp` for a production-ready search/indexing server by marking its tools as stub/experimental, exposing stub status in `/health`, and aligning documentation with actual behavior.

## Scope

**In scope:**
- `scripts/mcp/mdq/tools.py` — change `"status": "production"` to `"status": "stub"` for all 7 tools
- `scripts/mcp/mdq/server.py` — update `/health` endpoint to include stub/non-production indicator
- `docs/04_mcp_04_server_catalog.md` — align mdq-mcp section with actual code behavior (FTS5 is implemented; doc claims placeholder)
- `docs/04_mcp_90_inconsistencies_and_known_issues.md` — update MISSING-01 entry to reflect current state
- `config/mdq_mcp_server.toml` — add `status = "stub"` or `status = "experimental"` config key (optional, for consistency)

**Out of scope:**
- Implementing real search/indexing functionality (FTS5 is already functional)
- Removing mdq-mcp from routing (`route_resolver.py`)
- Modifying `scripts/shared/tool_constants.py` (MDQ_TOOLS set)
- Modifying `config/agent.toml` mdq server config block
- Modifying `init.d/mdq-mcp` OpenRC service
- Changes to `scripts/mcp/mdq/models.py`, `parser.py`, `indexer.py`, `service.py` behavior

## Assumptions

1. `"status"` field in tool definitions is **not consumed by any code** (confirmed by analysis). It serves as machine-readable metadata for external consumers and documentation generators. Changing it to `"stub"` will not affect agent behavior but will accurately signal non-production status.
2. The `/health` endpoint response format should follow existing patterns (e.g., github-mcp: `{"status":"ok","github_token":"set"}`), adding a `stub` boolean field. Additive changes are safe across all MCP servers.
3. The documentation update in `04_mcp_04_server_catalog.md` and `04_mcp_90_inconsistencies_and_known_issues.md` is required to close the inconsistency between doc claims ("placeholder implementation") and actual code (functional FTS5 search/indexing).
4. Changing tool status to `"stub"` will not break existing tests — `test_mdq_service.py` tests the service layer directly, not the tool metadata definitions.
