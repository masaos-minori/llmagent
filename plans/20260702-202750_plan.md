# Plan: Clarify non-destructive compatibility repair policy in schema creation

## Goal

Clarify in module docstring and documentation that `create_schema.py` applies only non-destructive, idempotent compatibility repairs, not a full migration framework, so readers understand why `_migrate_*` helpers exist without being misled about the scope of migration support.

## Scope

**In**:
- Update module-level docstring in `scripts/db/create_schema.py` to explicitly describe the compatibility repair policy
- Update `docs/90_shared_04_db_architecture_and_schema.md` to document the schema initialization policy and `_migrate_*` helper semantics

**Out**:
- Removing or renaming existing `_migrate_*` helpers
- Adding destructive migrations
- Changing schema DDL or runtime behavior
- Modifying `scripts/db/schema_sql.py` or other DB modules

## Assumptions

1. The current module docstring (Japanese, top of `create_schema.py`) partially captures intent but does not provide enough policy detail for readers unfamiliar with the codebase.
2. `docs/90_shared_04_db_architecture_and_schema.md` currently documents table schemas and `SQLiteHelper` but does not have a dedicated section explaining the `create_schema.py` initialization and compatibility repair policy.
3. No runtime behavior change is required; this is documentation-only.
4. The four `_migrate_*` helpers (`_migrate_rag_schema`, `_migrate_add_undone_column`, `_migrate_session_schema`, `_migrate_workflow_schema`) are the complete set and no new ones will be added in this task.

## Implementation Steps

1. **Phase 1: Update `scripts/db/create_schema.py` docstring**
   - [ ] Expand the module-level docstring to state:
     - `create_schema()` creates current schema objects using `IF NOT EXISTS` (idempotent, non-destructive)
     - Limited compatibility repairs may add missing columns or rebuild specific local tables when safe
     - Destructive data migrations (DROP TABLE, DELETE data, column rename) are not performed
     - Large schema transitions must be handled as explicit maintenance tasks outside this module
   - [ ] Add a brief `Compatibility repairs:` subsection in the docstring listing the four `_migrate_*` functions with one-line purpose each

2. **Phase 2: Update `docs/90_shared_04_db_architecture_and_schema.md`**
   - [ ] Add a new subsection (e.g., `## 2a. Schema Initialization Policy`) between the overview table and the `DbConfig` section
   - [ ] Document the four policy points from the requirement:
     - `create_schema()` uses `IF NOT EXISTS` — safe to re-run at any time
     - `_migrate_*` helpers perform non-destructive, idempotent repairs only (add missing column, rebuild table with corrected DDL when no `ALTER TABLE ADD COLUMN` alternative exists)
     - Destructive data migrations are not in scope
     - Large schema transitions are explicit maintenance tasks (reference `db/maintenance.py`)
   - [ ] List each `_migrate_*` helper with its target table and the column/constraint it repairs
