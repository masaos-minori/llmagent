Source requirement: `requires/20260717_24_require.md` (moved to `requires/done/` in Step 7 of this cycle).

## Goal

`scripts/mcp_servers/web_search/server.py` is reduced to HTTP endpoints, exception mapping, and
audit only; a new `service.py` owns request validation, provider invocation, latency measurement,
empty-result semantics, and health/metrics updates; `search_provider.py` remains the only
DuckDuckGo-specific layer; `formatters.py` is reduced to pure `SearchResponse` → text conversion;
`health.py`/`metrics.py` hold provider health state and lightweight counters — each layer
independently testable, no circular imports, and the pre-existing orphaned-duplicate-module problem
in this directory (left by commit `ca6b7bfe`) is closed out as part of the same restructuring.

## Design

1. **`health.py`** (new, or reuse if Issue 3 landed it first) — a small in-memory singleton tracking
   consecutive-failure count / last-success timestamp / last-failure reason, exposing a
   `record_success()` / `record_failure(reason: str)` / `is_healthy() -> bool` API. `service.py` is
   the only caller.
2. **`metrics.py`** (new, or reuse if Issue 3 landed it first) — module-level counters (`total_calls`,
   `total_errors`, running average latency), updated only from `service.py`.
3. **`service.py`** (new) — the orchestration function that today lives inside
   `formatters.search_web()`:
   - builds/accepts `SearchRequest`
   - calls `search_provider.search_duckduckgo`
   - measures latency (`time.perf_counter`)
   - handles the empty-result branch as a single named helper (see UNK-04)
   - calls `metrics.py` and `health.py` update hooks
   - logs via the existing `Logger`/`fmt_kvlog` pattern currently in `formatters.py`
   - returns a `SearchResponse`
4. **`formatters.py`** (slimmed) — keeps only `fmt_search_result` and `fdisp_search_web`;
   `fdisp_search_web` calls `service.search_web(...)` instead of containing the orchestration itself;
   `dispatch_web_tool`/`_WEB_DISPATCH` stay here since they are formatting-adjacent dispatch, not
   provider orchestration (matches the requirement's "keep formatter responsibilities focused on
   converting structured search responses to text" while leaving `dispatch_web_tool` as the thin
   glue `server.py` already expects).
5. **`web_search_server.py`** (the sole remaining server file) — already imports only from
   `web_search_models.py` and `web_search_tools.py` (no stale import to fix), plus
   `formatters.dispatch_web_tool` (unchanged call site) and, transitively via `formatters.py`,
   `service.py`. No new responsibility is added; it keeps `/health`, `/v1/tools`, `/v1/call_tool`, the
   `WebSearchUpstreamError` exception handler, and `_audit_log` call — exactly the "HTTP endpoints /
   exception handling / audit" role the requirement specifies.
6. ~~**Dead-file cleanup**~~ — **already done** by commit `abcf0820`: `models.py`, `tools.py`, and the
   bare `server.py` were all deleted (not `web_search_server.py` — the opposite of this plan's original
   §5/§6 assumption). Nothing left to `git rm` in this plan.
7. **No circular imports**: dependency direction stays
   `web_search_server.py → formatters.py → service.py → {search_provider.py, health.py, metrics.py, web_search_models.py}`,
   consistent with the existing header comments in `formatters.py`/`search_provider.py`
   ("Dependency direction: ... → shared.formatters, mcp_servers.web_search.models") which should be
   updated to reference `service.py` and `web_search_models.py` accurately.
