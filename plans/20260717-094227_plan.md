# Plan: Add documentation-to-code consistency checks and triage current-behavior notes

Source requirement: `requires/20260716_17_require.md`

## Goal

Agent documentation gains the same class of automated, CI-enforced doc-vs-code consistency checks that MCP documentation already has (broken internal links, removed-legacy-file references, slash-command drift, DB-schema drift where feasible, obsolete diagnostics references), and all 31 existing `Current behavior` notes across Agent/MCP docs are triaged into one of five explicit classifications with follow-up issues/doc-fixes filed for anything not already resolved.

## Scope

**In scope**

- A new Agent-docs consistency checker (`tools/check_agent_docs_consistency.py`), mirroring `tools/check_mcp_docs_consistency.py`'s design (typed `Issue`/`DocFile` dataclasses, one function per check, `--skip` flag registry, ERROR/WARNING severity, non-zero exit on ERROR), implementing:
  - Broken internal Markdown link detection (`[text](path)` / `[text](#anchor)` resolution against actual files/headings) — genuinely new capability; neither existing checker does this today.
  - Prohibition of references to removed legacy doc files (the split-doc convention already documents that monolithic predecessors "should no longer be used" — but see Assumption 2: those monolithic files never existed in this repo's git history, so this check targets any *other* removed/renamed doc file, not `05_agent_all.md`/`04_mcp_all.md` specifically).
  - Slash-command documentation drift: extract command names from `CommandDef(...)` calls in `scripts/agent/commands/command_defs_list.py` (`_COMMANDS` list, 20 entries today) and flag any doc-referenced `/command` not in that list, or any command in that list with no doc mention.
  - DB schema documentation drift, best-effort: regex-extract `CREATE TABLE`/`CREATE VIRTUAL TABLE` names from `scripts/db/schema_sql.py`'s template strings (`_RAG_SCHEMA_TEMPLATE`, `_SESSION_SCHEMA_TEMPLATE`, workflow/eventbus templates) and cross-check against table names mentioned in `docs/90_shared_04_*` and `docs/05_agent_09_*` (data-layer docs). Documented as "feasible-effort" per the requirement's own wording — not a full schema-diff.
  - Obsolete diagnostics references: reuse `check_mcp_docs_consistency.py`'s existing pattern-based approach (e.g. its `watchdog-restart` check already caught a since-removed feature) generalized to a small set of known-removed Agent diagnostics commands/flags, seeded from `docs/05_agent_90_inconsistencies_and_known_issues.md`.
- **Fix a pre-existing CI break found during investigation** (not in the original requirement, but blocking): `.github/workflows/mcp-docs-consistency.yml` and `.github/workflows/rag-docs-quality.yml` both invoke `python scripts/checks/check_*_consistency.py`, but the actual files live at `tools/check_*_consistency.py` — `scripts/checks/` does not exist in the working tree (confirmed: only a stale `scripts/__pycache__/check_mcp_docs_consistency.cpython-313.pyc` remains, and `git log --all -- 'scripts/checks/*'` shows no such path in history under that exact name). These two CI checks are very likely broken today. This plan corrects both workflow file paths to `tools/...` as a prerequisite, since adding a third (Agent) workflow with the same broken pattern would compound the problem.
- A new `.github/workflows/agent-docs-consistency.yml`, modeled on the (path-corrected) `mcp-docs-consistency.yml`, triggered on changes to `docs/05_agent_*.md` and `tools/check_agent_docs_consistency.py`.
- A `pyproject.toml` `[project.scripts]` entry point (e.g. `check-agent-docs = "check_agent_docs_consistency:main"`) matching the existing `check-mcp-docs` pattern.
- **Fix a second pre-existing break found during investigation**: `uv run check-mcp-docs` itself is currently non-functional (`ModuleNotFoundError: No module named 'check_mcp_docs_consistency'`) because `pyproject.toml`'s `[tool.setuptools.packages.find]` only sets `where = ["scripts"]`, and `check_mcp_docs_consistency.py` lives under `tools/`, outside that package path. This plan corrects the package/entry-point configuration (e.g. adding `tools` to the package-find `where` list, or moving/symlinking the checkers under `scripts/`) so both `check-mcp-docs` and the new `check-agent-docs` resolve correctly — required before either entry point can be considered functional.
- Current-behavior triage: extract all 31 `Current behavior` notes (Japanese and English heading variants: `### Current behavior (...)`, `**Current behavior:**`, `> **Current behavior**:`, `### 実装上の補足（Current behavior）`, `**現在の動作:**`) from the 20 files that contain them, produce a triage table (file, section, short summary, classification), and act on each per its classification (promote / file implementation issue / file doc-fix task / cross-reference already-tracked issue / remove after verification).
- Add a short guidance note (in `rules/coding.md` or a new small doc section) on how future `Current behavior` notes should be written/classified, per the requirement's "Add guidance for future Current behavior notes" instruction.

**Out of scope**

- Rewriting the documentation set — explicit non-goal.
- Semantic verification of every paragraph — explicit non-goal.
- Automatically deleting `Current behavior` notes without review — explicit non-goal; every removal requires an explicit verification step in Implementation.
- Resolving every implementation mismatch discovered during triage inside this same work — mismatches become tracked issues, not inline fixes, matching the requirement's own non-goal.
- Treating `05_agent_all.md` / `04_mcp_all.md` as real target files — see Assumption 1; the requirement's file list is corrected to the actual split-doc files.
- A full, column-level DB schema documentation diff — the requirement itself scopes this to "where feasible."

## Design

Two independent mechanisms, matching the requirement's own "Design intent" framing:

**1. Objective CI checks (mechanical, no judgment required)**

`tools/check_agent_docs_consistency.py` follows `check_mcp_docs_consistency.py`'s architecture exactly: a `DocFile` dataclass built once via a recursive `discover_md_files(docs_dir)` pass scoped to `05_agent_*.md` (and optionally `04_mcp_*.md` if MCP-side link-checking is folded in rather than duplicated — a decision to make explicit before implementation, since `check_mcp_docs_consistency.py` doesn't currently do link-checking either), an `Issue(file, line, severity, message)` dataclass, one `check_*()` function per concern, and a `--skip <name>` argparse registry mirroring the existing pattern. Severity: broken links and removed-file references are ERROR (fail CI); command/DB-schema drift and diagnostics references are WARNING initially (to avoid a noisy first rollout), promotable to ERROR later once the checker has run clean for a period — this staged-severity approach is not in the original requirement but is a reasonable operational safeguard against an immediate false-positive-driven CI failure storm on first merge.

**2. Subjective triage (judgment required, one-time cleanup + ongoing guidance)**

A triage pass reads all 31 notes, assigns one of the five classifications, and acts per classification:
- *Accepted current specification* → note is reworded to remove "Current behavior" framing and merged into normal prose (it's not a discrepancy, it's just how the system works).
- *Implementation fix required* → file a GitHub-issue-template Markdown under `issues/`, leave the note in place with a cross-reference to the new issue.
- *Documentation fix required* → fix the documentation directly in this same pass (small, low-risk, docs-only changes) and remove the note.
- *Issue already tracked* → cross-reference the existing issue/known-issues entry, remove the redundant inline note.
- *Obsolete and removable* → verify against current code (confirm the discrepancy no longer exists) then delete the note.

Guidance for future notes (short addition to `rules/coding.md` or a small doc section): a `Current behavior` note must always state which of the five categories it is, at the time it's written, rather than being left ambiguous — closing the exact gap this requirement identifies.

## Risks

- **Risk**: The new Agent-docs checker's first run surfaces a large number of pre-existing findings (broken links, command drift accumulated over time), causing CI to fail immediately on merge.
  **Mitigation**: Implementation Step 5 explicitly requires fixing real findings *before* wiring the checker into required CI, and Design §1's staged-severity approach (WARNING-first for the two more heuristic checks) reduces first-rollout noise.
- **Risk**: Fixing two independent, previously-silent CI/tooling breaks (`scripts/checks/` path mismatch, `check-mcp-docs` package resolution) as part of this documentation-consistency requirement could be seen as scope creep beyond the requirement's literal text.
  **Mitigation**: Both breaks share the same root cause (checkers living under `tools/` while CI/packaging config assumes `scripts/`) and directly block this requirement's own goal of "CI checks for documentation consistency" — an Agent-docs checker added on top of an already-broken CI/entry-point pattern would just be a third broken thing. Both fixes are called out explicitly as prerequisite, not hidden inside unrelated commits.
- **Risk**: Regex-based DB-schema-drift detection against `schema_sql.py`'s template strings produces false positives/negatives due to placeholder substitution (`{DIMS}`) or `CREATE VIRTUAL TABLE ... USING vec0(...)` syntax the extractor doesn't handle.
  **Mitigation**: UNK-04's prototype-and-verify step happens before the check is wired into CI; if precision is too low, Design allows falling back to a hand-maintained table-name allowlist rather than blocking the rest of this plan on perfect automated extraction.
- **Risk**: Current-behavior triage is manual, judgment-heavy work across 20 files/31 notes — misclassifying an "implementation fix required" note as "accepted specification" would silently hide a real bug.
  **Mitigation**: Each classification decision should be traceable (recorded in the triage table with a one-line rationale); ambiguous cases default to "implementation fix required" (file an issue) rather than "accepted" when in doubt, since filing an unnecessary issue is cheaper to undo than silently accepting a real discrepancy.
- **Risk**: Adding a third documentation-consistency GitHub Actions workflow increases CI runtime and maintenance surface.
  **Mitigation**: The new workflow is scoped narrowly (path-triggered on `docs/05_agent_*.md` + the checker file only, matching the existing MCP/RAG workflows' pattern), so it only runs when relevant files change.
