## Goal

Verify and enforce that `ToolRouteResolver` uses only live discovery and `ToolRegistry` as runtime routing sources. Rename `tool_routing.py` to `tool_routing_validation.py` to make its validation-only purpose explicit. Fix import discrepancy between callers expecting validation functions from `shared.tool_registry` and actual definitions in `shared.tool_routing`.

## Scope

**In-Scope:**
- Verify `ToolRouteResolver` uses no config `tool_names` or `tool_constants.py` fallback (inspection only — no code change expected)
- Rename `scripts/shared/tool_routing.py` → `scripts/shared/tool_routing_validation.py`
- Add re-export stubs to `scripts/shared/tool_registry.py` for backward compatibility: `validate_routing_against_config`, `validate_routing_against_live`, `validate_all_routing` (currently imported from `shared.tool_registry` by `repl_health.py` and `tests/test_tool_registry.py` but not defined there)
- Add tests to `tests/test_route_resolver.py` confirming:
  - Config `tool_names` mismatch does not affect `ToolRouteResolver.resolve()`
  - `tool_constants.py` frozensets are not consulted directly by the resolver

**Out-of-Scope:**
- Changing `ToolRegistry` seed data
- Changing live discovery protocol
- Changing `tool_constants.py`

## Pre-Analysis Findings

### Finding 1: `ToolRouteResolver` already satisfies runtime routing requirements

Inspection of `scripts/shared/route_resolver.py`:
- `resolve()` checks `self._discovery_map` first (live discovery), then `_lookup_registry()` (ToolRegistry)
- No reference to config `tool_names` or `tool_constants.py` frozensets
- Module docstring explicitly: "Config `tool_names` is NOT a routing input; it is validation metadata only."
- No code change required in `route_resolver.py`

### Finding 2: Import discrepancy — validation functions defined in wrong module

`scripts/agent/repl_health.py:272` imports `validate_routing_against_config` from `shared.tool_registry`:
```python
from shared.tool_registry import validate_routing_against_config
```

`tests/test_tool_registry.py:17-19` imports all three functions from `shared.tool_registry`:
```python
from shared.tool_registry import validate_all_routing, validate_routing_against_config, validate_routing_against_live
```

But these functions are defined in `shared/tool_routing.py`, NOT in `shared/tool_registry.py`. `tool_registry.py` does not re-export them (confirmed by inspection).

**Conclusion:** Either these imports fail at runtime (import error), or `tool_registry.py` previously contained these functions and they were moved to `tool_routing.py` without updating the re-export. The fix is to add re-exports from `tool_registry.py` pointing to `tool_routing_validation.py` (after rename).

```python
def test_config_tool_names_do_not_affect_routing(self):
    """Config tool_names mismatch does not change ToolRouteResolver.resolve() behavior."""
    # Build a resolver with a config that has a tool_names list containing a tool
    # that is also in the registry. Then check the tool resolves via registry regardless.
    # (Currently config tool_names is never passed to ToolRouteResolver at all.)
    from shared.mcp_config import McpServerConfig, TransportType, StartupMode
    cfg = McpServerConfig(
        TransportType.HTTP, "http://localhost", startup_mode=StartupMode.PERSISTENT,
        tool_names=["read_text_file"],  # config metadata — not routing input
    )
    resolver = ToolRouteResolver({"file_read": cfg})
    assert resolver.resolve("read_text_file") == "file_read"

def test_constants_not_used_directly_by_resolver(self):
    """ToolRouteResolver does not hold or consult tool_constants frozensets at resolve time."""
    # Create a resolver with NO registry and NO discovery map — it should raise
    # ValueError for an unknown tool, NOT silently fall back to constants.
    resolver = ToolRouteResolver({})
    with pytest.raises(ValueError, match="Unknown tool"):
        resolver.resolve("nonexistent_tool_xyz")
```
