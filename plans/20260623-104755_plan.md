# Plan: ツール呼び出しスケジューリングの精細化

**Source:** `requires/20260622_58_require.md`
**Created:** 2026-06-23 10:47:55

## Goal

DAG スケジューラー (`_execute_with_dag`) において、2つの問題を修正することで「reads が writes の完了を待たなくて済む状況」と「`shell_run` の安全な扱い」を実現する。

1. **read-write 並列化**: write tools に `resource_scope` を設定して `write_first` sequential barrier から `resource_groups` concurrent batch に移行させる。これにより reads と writes が同時に実行可能になる。
2. **`shell_run` シリアライゼーション修正**: `_execute_standard()` は `shell_run` を side-effect tool として直列化するが、`_execute_with_dag()` では `is_write=False` のため `parallel` グループ扱いになっている。`requires_serial=True` を設定して一貫性を保つ。
3. **トレース可視化**: 各ツールのグループ割り当て（write_first / serial / resource_scope / parallel）をデバッグログに追加する。

## Background: 現状分析

### DAG モードのスケジューリングパス (verified from source)

`scripts/agent/tool_runner.py` L377: `use_tool_dag=True` (config/agent.toml L36) かつ `serial_tool_calls=False` のとき `_execute_with_dag()` を使用。

`_execute_with_dag()` (L192-203) の ToolSpec 構築:
```python
tool_meta[name] = ToolSpec(
    resource_scope=fn.get("resource_scope", ""),  # config に resource_scope なし → 常に ""
    requires_serial=fn.get("requires_serial", False),  # config に設定なし → 常に False
    is_write=name in WRITE_TOOLS or name in DELETE_TOOLS,  # 定数から正しく設定
)
```

`build_execution_groups()` (tool_scheduler.py L38-97) の振り分けルール:
- `requires_serial=True` → 個別 serial barrier (1ツールずつ)
- `is_write=True` + `resource_scope` あり → `resource_groups[scope]` → concurrent batch
- `is_write=True` + `resource_scope` なし → `write_first` → **独自の sequential phase**
- その他 → `parallel` → concurrent batch

**実行フェーズ:**
1. `serial_barrier` ツール (1個ずつ、順次)
2. `write_first` phase: write ツール全員が並行実行 → **reads はここで待つ**
3. `concurrent_batch`: resource_groups + parallel が全グループ同時実行

### 現状の問題点

| 問題 | 箇所 | 影響 |
|---|---|---|
| config に `resource_scope` なし | tools_definitions.toml, agent.toml | 全 write ツールが `write_first` に分類 → reads が待機 |
| `shell_run` が `parallel` に分類 | tool_runner.py L202 | reads と同時実行 (unsafe; standard mode と不一致) |
| グループ割り当てのトレース不足 | tool_runner.py | scheduling 理由がデバッグ困難 |

### resource_scope 追加後の動作

Round: `[write_file, create_directory, read_text_file1, read_text_file2]`

**現状 (write_first barrier):**
- Phase 1: `write_file` + `create_directory` が並行実行
- Phase 2: `read1` + `read2` が並行実行 (writes 完了待ち)

**改善後 (resource_scope あり):**
- Phase 1 (single concurrent batch): `[write_file]`, `[create_directory]`, `[read1, read2]` が全グループ同時実行
- reads が writes を待たない

### 安全性の考慮

同じパスに対する write + read の同時実行は read が stale データを見る可能性がある。これはパスレベルの conflict detection なしには防げない。本実装では:
- 同一 `resource_scope` を持つ writes は同一グループに入り並行実行される (現状の write_first と同様)
- 異なる tool タイプの write と read の並行実行はベストエフォート (path-level conflict detection は out-of-scope)
- `shell_run` は `requires_serial=True` で完全シリアル化 (任意の副作用を持つため)

## Design

### Step 1: tool_runner.py — ToolSpec 構築の修正

**対象:** `scripts/agent/tool_runner.py` `_execute_with_dag()` L192-203

**変更前:**
```python
from shared.tool_constants import DELETE_TOOLS, WRITE_TOOLS
...
tool_meta[name] = ToolSpec(
    call_id="",
    name=name,
    resource_scope=fn.get("resource_scope", ""),
    requires_serial=fn.get("requires_serial", False),
    is_write=name in WRITE_TOOLS or name in DELETE_TOOLS,
)
```

**変更後:**
```python
from shared.tool_constants import DELETE_TOOLS, SHELL_TOOLS, WRITE_TOOLS
...
_is_write = name in WRITE_TOOLS or name in DELETE_TOOLS
# config explicit value overrides the default; default = tool name for write tools (enables resource-scoped concurrency)
_default_scope = name if _is_write else ""
tool_meta[name] = ToolSpec(
    call_id="",
    name=name,
    resource_scope=fn.get("resource_scope", _default_scope),
    requires_serial=fn.get("requires_serial", False) or name in SHELL_TOOLS,
    is_write=_is_write,
)
```

**効果:**
- `write_file` → `resource_scope="write_file"`, `is_write=True` → `resource_groups["write_file"]` → concurrent batch
- `shell_run` → `requires_serial=True` → serial barrier (standard mode と一貫)
- reads → `resource_scope=""`, `is_write=False` → `parallel` → concurrent batch
- `write_first` グループは空になり、reads が writes を待たなくなる

### Step 2: tool_runner.py — グループ割り当てデバッグログ追加

`_execute_with_dag()` で `build_execution_groups()` 呼び出し直後にグループ割り当てサマリーを追加:

```python
_groups, metadata = build_execution_groups(approved_calls, tool_meta)
if logger.isEnabledFor(logging.DEBUG):
    for tc in approved_calls:
        n = tc["function"]["name"]
        m = tool_meta.get(n)
        if m is not None and m.requires_serial:
            bucket = "serial_barrier"
        elif m is not None and m.is_write and m.resource_scope:
            bucket = f"resource_scope:{m.resource_scope}"
        elif m is not None and m.is_write:
            bucket = "write_first"
        else:
            bucket = "parallel"
        logger.debug("DAG_BUCKET: %s → %s", n, bucket)
```

def test_write_file_gets_resource_scope_from_constant():
    """write_file has no resource_scope in config → defaults to tool name."""
    # tool_definition with no resource_scope field
    td = {"function": {"name": "write_file"}}
    # Build tool_meta as _execute_with_dag() does

    name = fn.get("name", "")
    _is_write = name in WRITE_TOOLS or name in DELETE_TOOLS
    _default_scope = name if _is_write else ""
    spec = ToolSpec(
        call_id="",
        name=name,
        resource_scope=fn.get("resource_scope", _default_scope),
        requires_serial=fn.get("requires_serial", False) or name in SHELL_TOOLS,
        is_write=_is_write,
    )
    assert spec.resource_scope == "write_file"
    assert spec.is_write is True
    assert spec.requires_serial is False

def test_shell_run_gets_requires_serial():
    """shell_run has no requires_serial in config → defaults to True via SHELL_TOOLS."""
    fn = {"name": "shell_run"}
    name = fn.get("name", "")
    _is_write = name in WRITE_TOOLS or name in DELETE_TOOLS
    _default_scope = name if _is_write else ""
    spec = ToolSpec(
        call_id="",
        name=name,
        resource_scope=fn.get("resource_scope", _default_scope),
        requires_serial=fn.get("requires_serial", False) or name in SHELL_TOOLS,
        is_write=_is_write,

    assert spec.requires_serial is True
    assert spec.is_write is False

**3b. スケジューリング動作の検証:**
```python
def test_write_and_read_in_same_concurrent_batch():
    """With resource_scope set, write_file and read_text_file share concurrent_batch."""
    tool_meta = {
        "write_file": ToolSpec(call_id="", name="write_file", resource_scope="write_file", is_write=True),
        "read_text_file": ToolSpec(call_id="", name="read_text_file", resource_scope="", is_write=False),
    }
    calls = [
        {"id": "1", "function": {"name": "write_file", "arguments": "{}"}},
        {"id": "2", "function": {"name": "read_text_file", "arguments": "{}"}},
    ]
    _, metadata = build_execution_groups(calls, tool_meta)
    # write_first must be empty; all groups in one concurrent batch
    assert len(metadata.concurrent_groups) == 1  # single concurrent phase
    assert len(metadata.concurrent_groups[0]) == 2  # write_file group + parallel group
```

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| `resource_scope` 追加により read と write が同時実行されるため、同一パスの read が stale データを取得する | Low-Med | 本実装は「path-level conflict detection なし」と文書化。write + read が同一パスを使用するのは稀なケース。許容できない場合は config で `requires_serial = true` を個別ツールに指定可能。 |
| `shell_run` に `requires_serial=True` を追加することで既存テストが壊れる | Low | `shell_run` が現在 `parallel` グループに分類されることを前提としたテストがあれば修正が必要。`test_tool_scheduler*.py` を事前確認する。 |
| `write_file` + `write_file` の 2 呼び出しが同一スコープで並行実行されることで同一ファイルへの同時書き込みが発生する | Low | スコープが同じなので同一グループ内で `asyncio.gather` により並行実行される。2 つの write_file が同じパスを対象とするケースはエージェントの設計上発生しにくい。発生しても最終状態は一方の結果になる (ファイル書き込みはアトミック)。 |
