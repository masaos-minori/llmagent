# Plan: Make `ToolResultStore.store()` return a non-optional integer ID

## Goal
`ToolResultStore.store()` の戻り値型を `int`（非Optional）に確定させ、callerの `int | None` ガードを除去し、テストで全パスを網羅する。

## Scope
**In-Scope**:
- `scripts/db/tool_results.py` — `store()` 戻り値型アノテーション・docstring の確認と残存 `int | None` 表記の除去
- `scripts/agent/tool_runner.py` — `_apply_turn_char_limit(result_id: int | None)` シグネチャを `int` に変更し、`None` ガード分岐を削除
- `tests/test_tool_result_store.py` — 要件テスト（成功時 `int` 返却・IDによるfetch・`lastrowid=None` で RuntimeError）が既に存在することの確認

**Out-of-Scope**:
- DB スキーマ変更（tool_results テーブルに変更なし）
- `store()` の引数シグネチャ変更
- `get()` / `list_recent()` の変更
- 他モジュールへの波及（`tool_runner.py` 以外のcallerは存在しない）

## Assumptions
- `db/tool_results.py` の `store()` は既に `-> int` アノテーションと `lastrowid is None` → `RuntimeError` ロジックを持っている（読み取り済み、事実）
- `tests/test_tool_result_store.py` に要件テスト3件が既に実装済み（読み取り済み、事実）
- `_apply_turn_char_limit` の呼び出し元は `_collect_tool_result_msgs` 1箇所のみ
- `_apply_turn_char_limit` 内の `result_id is not None` ガードは、`store()` が `int` を保証した後も動作に影響しないが、型シグネチャを `int` に揃えることで型安全性が向上する
