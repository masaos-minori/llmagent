# Work Plan: Normalize shared tool constants and side-effect helper behavior

Source requirement: `requires/20260715_19_require.md` (moved to `requires/done/` after this plan was created)

## Goal

Make `shared/tool_constants.py` the single, granular source of read/write/admin
classification for CICD, RAG, and MDQ tools, and make `is_side_effect()` in
`shared/tool_executor_helpers.py` correctly flag all known mutation tools
(`trigger_workflow`, `rag_delete_document`, `index_paths`, `refresh_index`,
`fts_rebuild`) as side-effecting, without changing runtime routing behavior.

## Scope

**In scope:**
- `shared/tool_constants.py`: add `CICD_READ_TOOLS`, `CICD_WRITE_TOOLS`,
  `RAG_READ_TOOLS`, `RAG_WRITE_TOOLS`, `MDQ_READ_TOOLS`, `MDQ_WRITE_TOOLS`,
  `MDQ_ADMIN_TOOLS`, `FILE_WRITE_TOOLS` (alias of `WRITE_TOOLS`); correct the
  module docstring's routing-authority wording.
- `shared/tool_executor_helpers.py`: expand `_SIDE_EFFECT_TOOLS` to include
  `CICD_WRITE_TOOLS`, `RAG_WRITE_TOOLS`, `MDQ_WRITE_TOOLS`.
- `tests/test_tool_constants.py` and `tests/test_tool_executor_helpers.py`:
  add coverage for the new classifications, the `is_side_effect()` expansion,
  and the `FILE_WRITE_TOOLS` alias.

**Out of scope (per requirement's Non-Goals):**
- Any change to runtime routing behavior (`shared/route_resolver.py`,
  `shared/tool_registry.py` population logic).
- Removing or renaming existing aggregate sets (`RAG_TOOLS`, `CICD_TOOLS`,
  `MDQ_TOOLS`, `WRITE_TOOLS`).
- `tool_hash_key()` and `format_transport_error()` fixes — **already
  implemented** in the current codebase (see Unknowns §1). No code change
  needed for these two; only confirmed by the existing/passing test suite.
- `agent/tool_policy.py` risk tiers / `tool_safety_tiers` config — these are
  driven by `config/agent.toml`, not by `shared/tool_constants.py`, and are
  not named as required changes in the source issue.

## Assumptions

1. The six "Required Changes" sections in the requirement are the authoritative
   scope; the "Why This Needs to Be Fixed" background (scheduler/approval/
   dry-run/diagnostics) is motivation only — no other module needs a code
   change to consume the new sets in this task.
2. `rag_run_pipeline`, `rag_debug_pipeline`, `rag_list_documents` are read
   operations (search/retrieval only) — confirmed by reading
   `scripts/mcp_servers/rag_pipeline/tools.py` and `server.py`; none of them
   write to the RAG index. Only `rag_delete_document` mutates it.
3. `fts_rebuild` intentionally belongs to both `MDQ_WRITE_TOOLS` and
   `MDQ_ADMIN_TOOLS` (it rewrites index data and is documented as an "Admin
   operation" in `scripts/mcp_servers/mdq/tools.py`). This overlap is
   deliberate and mirrors the issue's own recommended classification; it does
   not need to be disjoint from `MDQ_ADMIN_TOOLS` the way top-level categories
   (`READ_TOOLS`/`WRITE_TOOLS`/... ) are disjoint from each other.
4. New read/write/admin subsets are additive frozensets (explicit literals,
   matching the existing `GIT_READ_TOOLS`/`GIT_WRITE_TOOLS` and
   `GITHUB_READ_TOOLS`/`GITHUB_WRITE_TOOLS`/`GITHUB_DANGEROUS_TOOLS` style),
   not computed via set difference, so future edits to the aggregate sets
   don't silently reclassify tools.
5. No new source files are created, so `deploy/deploy.sh` (which `rsync`s the
   whole `scripts/` tree) needs no update, and no new MCP server/config file
   is involved.
6. `tools/check_mcp_docs_consistency.py` only scans `docs/**/*.md`, not `.py`
   docstrings, so the `tool_constants.py` docstring fix is not enforced by
   `check-mcp-docs` — it is a manual consistency fix only, matched to the
   wording already used in `scripts/shared/tool_registry.py` (lines 19-20)
   and `scripts/shared/route_resolver.py` (lines 57, 137-139).

## Design

### 1. `shared/tool_constants.py`

Add after `CICD_TOOLS`:

```python
CICD_WRITE_TOOLS: frozenset[str] = frozenset({"trigger_workflow"})
CICD_READ_TOOLS: frozenset[str] = frozenset(
    {"get_workflow_runs", "get_workflow_status", "get_workflow_logs"},
)
```

Add after `RAG_TOOLS`:

```python
RAG_WRITE_TOOLS: frozenset[str] = frozenset({"rag_delete_document"})
RAG_READ_TOOLS: frozenset[str] = frozenset(
    {"rag_run_pipeline", "rag_debug_pipeline", "rag_list_documents"},
)
```

Add after `MDQ_TOOLS`:

```python
MDQ_WRITE_TOOLS: frozenset[str] = frozenset(
    {"index_paths", "refresh_index", "fts_rebuild"},
)
MDQ_ADMIN_TOOLS: frozenset[str] = frozenset(
    {"fts_consistency_check", "fts_rebuild"},
)
MDQ_READ_TOOLS: frozenset[str] = frozenset(
    {"search_docs", "get_chunk", "outline", "stats", "grep_docs"},
)
```

Add near `WRITE_TOOLS` definition (or immediately after it):

```python
# Alias: WRITE_TOOLS covers file-write tools specifically; this alias makes
# that scope explicit at call sites that only care about file writes.
FILE_WRITE_TOOLS: frozenset[str] = WRITE_TOOLS
```

Fix module docstring line 11, replacing:

```
Not a routing fallback source. Routing is driven by live `/v1/tools` discovery and ToolRegistry only.
```

with wording consistent with `shared/tool_registry.py`:

```
ToolRegistry is the sole routing authority. Live `/v1/tools` discovery is used
only for startup validation (drift detection); config `tool_names` are
validation metadata, not routing inputs. These frozensets are not a routing
fallback source.
```

`get_all_mcp_tool_names()` is unaffected — it aggregates the top-level
category sets (`RAG_TOOLS`, `CICD_TOOLS`, `MDQ_TOOLS`, `WRITE_TOOLS`, ...),
which are unchanged; the new read/write/admin subsets are strict subsets of
those and must not be added to that union (would double-count / break the
existing `test_no_overlapping_tools` total of 44).

## Risks

- **Risk:** New `MDQ_ADMIN_TOOLS`/`MDQ_WRITE_TOOLS` overlap (`fts_rebuild` in
  both) could look like a bug to a future reviewer and get "fixed" by
  accident, silently narrowing `_SIDE_EFFECT_TOOLS`.
  → **Mitigation:** Add an explicit test
  (`test_mdq_read_write_admin_tools`) asserting the overlap, with an inline
  comment explaining it is intentional (fts_rebuild both mutates the index
  and is an admin-tier operation).

- **Risk:** Making `trigger_workflow`, `rag_delete_document`, `index_paths`,
  `refresh_index`, `fts_rebuild` side-effecting changes `agent/tool_runner.py`
  scheduling from parallel to serial when they co-occur with other tool calls
  in one LLM turn — a latent behavior change outside this issue's stated
  files.
  → **Mitigation:** This is the explicit, intended fix per the issue's
  Acceptance Criteria; document it in Affected Areas/blast radius (already
  done above) so implementers and reviewers are not surprised. No mitigation
  beyond running the full test suite is needed since serial execution is
  strictly safer than parallel for mutating tools.

- **Risk:** Single-author bus factor (100% of history on `tool_constants.py`
  and `tool_executor_helpers.py` is one person under three name spellings)
  means no independent reviewer has prior context on these files.
  → **Mitigation:** Keep the diff small and additive (no removed/renamed
  symbols), and rely on the exhaustive test list in Design §3 as the
  effective review aid.

- **Risk:** The docstring fix in `tool_constants.py` is not covered by any
  automated check (`check-mcp-docs` only scans `docs/**/*.md`), so wording
  could drift again in the future without detection.
  → **Mitigation:** Add the `test_tool_constants_routing_docstring_does_not_reference_live_routing`
  test so at least a regression back to the stale wording fails CI; this is
  the pragmatic lightweight substitute for extending `check-mcp-docs` to
  `.py` files (out of scope for this issue).

All risks above have mitigations already folded into the plan; none require
a separate issue file.
