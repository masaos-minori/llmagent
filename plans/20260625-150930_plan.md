## Design

**追加するセクション構成:**

```markdown
## Push-Mode Operations

### Delivery model overview

Live subscribers receive events in two phases:
1. **Replay phase**: all events since last acked seq are read from SQLite and streamed immediately
2. **Live phase**: new events are pushed from the in-memory broker queue (no polling)

### Queue sizing

Each subscriber has a dedicated `asyncio.Queue(maxsize=1000)`. When the queue is full,
new events are dropped with a WARNING log. The dropped events are still in SQLite and will
be re-delivered on reconnect (at-least-once guarantee).

Default sizing of 1000 events is appropriate for:
- Consumers that process events in < 1 s per event
- Publish rates up to ~1000 events/burst

For higher sustained publish rates, reduce consumer processing time or increase batch ack frequency.
Note: queue maxsize is currently not configurable (hardcoded in broker.py).

### Slow-consumer policy

A subscriber is "slow" when its queue depth >= 100 (10% of maxsize). Slow subscribers are
counted in `GET /health` under `slow_consumers`. When `slow_consumers > 0`, the health status
shows `"slow_consumers_detected"` in `degraded_reasons`.

Actions when slow consumers are detected:
1. Check `slow_consumers` and `max_queue_depth` in `/health`
2. Profile the consumer's processing time per event
3. Consider increasing batch ack size or reducing per-event work
4. If the consumer is a replay client (large backlog on reconnect), wait for replay to complete

### Fallback and resync behavior

The broker runs entirely in memory. On process restart:
- All subscriber registrations and queued events are lost
- Consumers reconnect and receive a full SQLite replay from their last acked seq
- No events are lost as long as they are in SQLite

If a consumer misses events during a broker disruption (e.g., queue overflow):
- Disconnect and reconnect
- The reconnect replay from last acked seq will re-deliver any missed events

### Investigating push delivery issues

Use `GET /health` to inspect push-mode indicators:

| Field | Meaning | Action |
|---|---|---|
| `active_subscribers` | Current SSE subscriber count | Compare to expected client count |
| `max_queue_depth` | Largest queue depth across subscribers | >500 → `broker_queue_backlog_high` |
| `slow_consumers` | Subscribers at slow threshold (>=100) | Profile consumer processing time |

### Troubleshooting guide

| Symptom | Check | Likely cause | Fix |
|---|---|---|---|
| Events delayed to live subscribers | `max_queue_depth` high | Slow consumer blocking queue | Profile consumer; increase ack throughput |
| Events missing after reconnect | Replay covers last acked seq | Not acking events | Ensure consumer calls POST /ack |
| `broker_queue_backlog_high` in health | `max_queue_depth >= 500` | Burst publish rate exceeds consumer throughput | Throttle publisher or optimize consumer |
| `slow_consumers_detected` | `slow_consumers > 0` | Consumer processing too slow | See slow-consumer policy above |
| Subscriber count drops to 0 unexpectedly | `/health active_subscribers = 0` | Consumers disconnected | Check client logs for reconnect failures |
```
