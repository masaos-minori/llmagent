## Goal
DLQ プロモーションのトリガーが nack (`delivery_failure_count >= max_retry`) のみであること、`retry_count` フィールドが現在どのコードパスでも更新されていないことを確認し、ドキュメントの誤解を招く記述を修正する。

## Scope
**In-Scope**:
- `scripts/eventbus/db.py` の `nack_event()` が `delivery_failure_count` を増加させることの確認
- `scripts/eventbus/db.py` の `requeue_event()` が `dlq_requeue_count` のみ更新し `retry_count` を更新しないことの確認
- `retry_count` フィールドがどのコードパスでも更新されていないことの確認
- `docs/06_eventbus_90_inconsistencies_and_known_issues.md` の `retry_count` の説明の修正: 「DLQ requeue 時にのみ更新される」→「どのコードパスでも更新されない（deprecated）; `dlq_requeue_count` を使うこと」
- `docs/06_eventbus_04_dlq_offsets_and_delivery_semantics.md` の DLQ モデル説明確認
- 「通常のストリーミング失敗（unacked event）は `delivery_failure_count` を増加させない」という記述が docs にあることの確認

**Out-of-Scope**:
- full consumer group failure tracking の実装
- nack API の設計変更
- exactly-once デリバリー

1. `db.py:L117-133`: `nack_event()` が `delivery_failure_count` を increment — DLQ promotion のトリガー（確認済み）
2. `db.py:L225-234`: `requeue_event()` が `dlq_requeue_count` を increment し `dlq_at = NULL` に設定 — `retry_count` は更新しない（確認済み）
