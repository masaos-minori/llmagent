## Design

### Current implementation state (all acceptance criteria met)

**1. Strict mode exists** ✓
- `shared/plugin_registry.py:198` — `strict_mode: bool = False` parameter in `load_plugins()`
- `agent/config_dataclasses.py:226` — `plugin_strict: bool = False` config field
- `agent/config_dataclasses.py:244-248` — type validation for `plugin_strict`

**2. Startup can fail on plugin load errors in strict mode** ✓
- `shared/plugin_registry.py:225-227` — when `strict_mode=True`, raises the original exception
- `agent/factory.py:340-344` — passes `strict_mode=ctx.cfg.tool.plugin_strict` to `load_plugins()`
- No try/except wraps this call in `factory.py`, so the exception propagates and aborts startup
- `agent/factory.py:338` — audit log records the mode: `"strict"` or `"fail-open"`

**3. Non-strict fail-open mode still exists** ✓
- `strict_mode=False` (default) — logged as WARNING, plugin skipped, other plugins continue loading
- `shared/plugin_registry.py:228` — `logger.warning(error_msg)` in non-strict path
- Default config value is `False`: `plugin_strict: bool = False`
