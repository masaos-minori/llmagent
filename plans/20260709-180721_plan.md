# 作業計画書

## Goal

`docs/` 配下の全Markdownファイル(51ファイル)を、AIエージェント/RAG/Claude Code/OpenCodeが利用しやすいナレッジベースへ再構成する。

- 8KBを超えるファイルをH2境界(必要ならH3、さらにモジュール単位)で分割し、全ファイルを8KB以下にする
- 全ファイルにYAML Front Matter、Related Documentsセクション、Keywordsセクションを付与する
- `docs/index.md` を新設し、トップレベルのナビゲーション(Overview/Deployment/RAG/MCP/Agent/Event Bus/Shared・DB/Known Issues)を提供する
- 元の情報は一切削除・要約せず、リンク切れなくAI検索/RAG取り込みに適した粒度に分割する

## Scope

**In:**
- `docs/*.md` 51ファイルの構造分析(完了)、分割計画の策定(完了)、分割実行、Front Matter/Keywords/Related Documents付与、内部リンク修正、`docs/index.md` 新設
- memo2.md の計画にない既存 `06_eventbus_*`(8ファイル)を「06 Event Bus」として正式カテゴリ化する(ユーザー承認済み)

**Out:**
- ドキュメント内容の技術的な正確性の検証・修正(誤字脱字レベルを除く)
- `03_rag_90` / `04_mcp_90` / `90_shared_90` の既知課題スケルトンへの新規内容追記(正規化のみ、リライトはスコープ外)
- ソースコード(`scripts/`等)の変更

## Assumptions

1. `docs/` は現状フラット構造で、サブディレクトリ・`index.md` は存在しない。
2. 既存の番号体系(`01_overview` / `02_deployment` / `03_rag` / `04_mcp` / `05_agent` / `90_shared`)はmemo2.mdの想定構成とほぼ一致しており、これを土台として拡張する。
3. 全ファイルでYAML Front Matter・Keywordsセクションが皆無、Related Documentsセクションは一部ファイルのみ存在する。
4. 既存の内部リンクはすべて実在ファイルを指しており、破損リンクはない(分割後は再検証が必要)。
5. 一部サブエージェントが「複数H1見出しあり」と報告した箇所は、コードブロック内のPythonコメント(`# ...`)をH1として誤検出したものであり、実際は全ファイルでH1は1個のみ(検証済み)。

## Design

### 分割方針
- 分割単位はH2見出しを基本とする。H2単体が6KBを超える場合はH3で再分割し、H3でも8KBを超える場合は機能グループ(feature/API/workflow/procedure/configuration/troubleshooting/examples/reference)単位、最終手段としてsequential part file(`-part1`, `-part2`...)で分割する。
- 命名規則: memo2.mdが例示する `<prefix>_<NN>_<slug>.md` の2桁連番形式を採用。`01_overview-arch` / `01_overview-files` はハイフン形式の既存命名を維持し、その配下も同じ命名系(`01_overview-arch-process.md`等)で統一する。

### Front Matter テンプレート

```yaml
---
title: "<内容を表すタイトル>"
category: <overview|deployment|rag|mcp|agent|eventbus|shared>
tags:
  - <5-20個の検索キーワード>
related:
  - <関連Markdownファイル>
source:
  - <分割元の元ファイル名(分割時のみ)>
---
```

### ファイル末尾テンプレート

```markdown
## Related Documents

- `xxx.md`

## Keywords

keyword
keyword
```

### docs/index.md 構成
- ドキュメント概要
- カテゴリ一覧(Overview / Deployment / RAG / MCP / Agent / Event Bus / Shared・DB / Known Issues)
- 各カテゴリの `*_00_document-guide.md` 相当ファイルへのリンク
- 推奨読書順序
- `01_overview.md` は従来通りシステム概要インデックスとして維持し、役割を分離する(`index.md`=ドキュメント全体ナビ、`01_overview.md`=システム概要)

### Step 9: 検証
- 全生成/修正ファイルが8KB以下であることを実測確認
- 各ファイルにH1が1個のみ、Front Matter/Related Documents/Keywordsセクションが存在することを確認
- 内部リンクの到達可能性を確認(リンク先ファイルが実在するか)
- コードブロック・Markdownテーブルが分割によって破断していないことを確認

## Validation Plan

| Check | Method | Target |
|---|---|---|
| ファイルサイズ | `wc -c docs/*.md` | 全ファイル8192バイト以下 |
| H1見出し数 | `grep -c '^# ' docs/*.md`(コードブロック内コメントの誤検出に注意) | 各ファイル1個 |
| Front Matter | 各ファイル先頭が `---` で始まるYAMLブロックを持つか目視/スクリプト確認 | 全ファイルで存在 |
| Related Documents / Keywords セクション | 各ファイル末尾の見出し確認 | 全ファイルで存在 |
| 内部リンク | `docs/` 内の `[text](*.md)` 形式リンクのリンク先ファイル存在確認 | リンク切れ0件 |
| index.md 到達性 | `docs/index.md` から各カテゴリガイドファイルまで辿れるか確認 | 全カテゴリ到達可能 |
| コードブロック/テーブルの破断 | 分割前後で fenced code block とテーブルの行数・内容が分断されていないか目視確認 | 破断0件 |

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| H2単位の概算分割でも実測で8KBを超えるファイルが多数残る(`05_agent_12`のModule Reference等) | medium | Phase 3実行時に分割直後に都度サイズを実測し、超過分はH3/モジュール単位で即時に追加分割する運用とする |
| 分割によりファイル数が51→100件超に増加し、相互参照リンクの修正漏れが発生する | medium | 分割完了後に `docs/` 全体でリンク先ファイル存在チェックを機械的に実施し、修正漏れを検出する |
| `routing.md` が参照している `docs/05_agent_*.md` 等の固定パスが分割後に無効化し、他タスクのcontext loadingが壊れる | high | Step 8で `routing.md` のDocs→taskマッピングも分割後のファイル名に更新することを必須ステップとして明記済み |
| 大量の分割・生成によりレビューが困難になり、意図しない情報欠落が混入する | medium | 「元の情報を削除しない・要約しない」の原則を各ステップで厳守し、分割はコピー&境界調整のみとする。最終検証で元ファイルとの内容カバレッジを目視確認する |
