# Plan: M-7 — Migration policy for existing tool_results tables

## Goal

既存の `session.sqlite` に存在する `tool_results` テーブルに対するマイグレーションポリシーを
決定・文書化し、テーブルの有無にかかわらずアプリが正常動作することを保証する。

---

## Scope

- `docs/05_agent_09_data-layer.md` — tool_results テーブル記述の更新
- `docs/90_shared_04_db_architecture_and_schema.md` — session.sqlite スキーマ説明への移行ポリシー追記
- `docs/90_shared_05_db_api_and_operations.md` — ToolResultStore 節への注記追加
- オプション: `scripts/db/` 配下への opt-in DROP マイグレーションスクリプト追加 (将来用テンプレート)

ランタイムコードへの変更は **行わない**。既存の `ToolResultStore` は `tool_results` テーブルが
存在することを前提に動作しており、その前提は変わらない。
ただし、テーブルが存在しない新規 DB でも `create_schema.py` が `IF NOT EXISTS` で
テーブルを作成するため、すでに安全である。

---

## Assumptions

- ランタイムコードは `ToolResultStore` 経由でのみ `tool_results` にアクセスする (要確認済)。
- `create_schema.py` の `_SESSION_SCHEMA_TEMPLATE` は `CREATE TABLE IF NOT EXISTS tool_results` を
  含むため、新規 DB への起動は安全。
- 既存 DB に `tool_results` が存在しない場合、`ToolResultStore` の SQL が `no such table` で
  失敗する可能性がある (後述の Unknowns 参照)。

---

## Unknowns (分析前)

1. **既存 DB に `tool_results` がない場合、`ToolResultStore` は起動時/実行時に失敗するか?**
   → `tool_results` テーブルは `create_schema.py` で `IF NOT EXISTS` 作成されるため、
     通常の起動フローでは必ず存在する。ただし手動で削除された旧 DB の場合は失敗する。
   → 対処: `create_schema.py` が新規/既存 DB のどちらでも実行されることをドキュメントで明示。

2. **旧 DB (M-7 対象以前に作成) に `tool_results` が存在する場合、副作用はあるか?**
   → スキーマは `IF NOT EXISTS` のため再作成は行われない。既存データは保持される。
   → ランタイムコードは引き続き `tool_results` を使用するため、データ不整合はない。

3. **`tool_results` を「未使用」にするという要件の解釈**
   → require ファイルの "remove all runtime references" は将来フェーズの話であり、
     現時点では「ランタイム参照を持つまま、削除しない」が推奨ポリシー。
   → 現状のコード (`tool_runner.py`, `repl.py`, `cmd_context.py`) は
     `tool_results` を積極的に使用中。参照を削除する作業は別タスクになる。

## Design

### 決定ポリシー: Leave-in-place (現状維持)

現時点では以下を採用する:

1. **既存テーブルは残置** — 削除しない、スキーマ変更しない
2. **ランタイム参照は維持** — `ToolResultStore` は引き続き `tool_results` を使用
3. **自動的な破壊的マイグレーションは実行しない**
4. **テーブルが存在しない DB への対処** — `create_schema.py` を起動前に実行することを
   運用手順書に明記 (既存手順の再確認)
5. **将来の DROP マイグレーション** — `scripts/db/migrate_drop_tool_results.py` として
   opt-in スクリプトを用意 (実行は明示的な操作者判断による)

### テーブル不存在時の安全性

- 新規 DB: `create_schema.py` → `create_session_schema()` が `IF NOT EXISTS` で自動作成
- 既存旧 DB (手動削除等): 起動時に `ToolResultStore` が `no such table` エラー
  → 対処は `create_session_schema()` 再実行で解決。手順をドキュメント化する。

## Implementation Steps

### Step 1: ドキュメント更新 (3ファイル)

#### `docs/05_agent_09_data-layer.md`

`tool_results` テーブル行の下に移行ポリシー節を追加:

```
### tool_results Migration Policy (M-7)

- Policy: leave-in-place. Existing table data is retained; no automatic DROP runs.
- Runtime: ToolResultStore continues to reference tool_results. Removal of runtime
  references is deferred to a future task.
- Missing table: if tool_results does not exist in an existing DB, run
  create_session_schema() to recreate it.
- Future DROP: opt-in only. Run scripts/db/migrate_drop_tool_results.py explicitly.
  Archive session.sqlite first via rotate_session_db().
```

#### `docs/90_shared_04_db_architecture_and_schema.md`

Section 8 (Schema Generation and Migration Approach) に注記追加:

```
### tool_results Deprecation Policy

The tool_results table is not deprecated. Leave-in-place policy is in effect.
See M-7 plan for details.
```

#### `docs/90_shared_05_db_api_and_operations.md`

Section 5 (`ToolResultStore`) に注記追加:

```
**Migration policy (M-7):** The tool_results table is retained. No automatic DROP.
If the table is absent in an existing DB (e.g. manually deleted), run
create_session_schema() to restore it. A future opt-in DROP script exists at
scripts/db/migrate_drop_tool_results.py.
```

### Step 2: opt-in DROP スクリプト (将来用テンプレート)

`scripts/db/migrate_drop_tool_results.py` を新規作成:
- 実行前に `rotate_session_db()` でアーカイブ
- `DROP TABLE IF EXISTS tool_results` を実行
- 実行確認プロンプト付き (対話式 opt-in)

### Step 3: Validation

- `uv run pytest tests/test_tool_result_store.py tests/test_create_schema.py` — 既存テスト pass
- `sqlite3 <db_path> ".tables"` で `tool_results` の有無を確認
- 旧 DB (tool_results あり) / 新規 DB (tool_results なし+create_schema実行) の両方で
  アプリが起動・ターン実行できることを確認
