# Plan: Make consumer_id to offset-filename mapping deterministic and collision-resistant

Source: `requires/20260625_49_require.md`

---

## Goal

`offsets.py` の `consumer_id` → ファイル名変換を衝突耐性の高い決定論的なルールに統一し、コードと docs の一致を確保する。

---

## Scope

**In-Scope**
- `scripts/eventbus/offsets.py`: sanitization ルールの強化（衝突耐性向上）
- 後方互換性の確認または migration 戦略の記述
- edge case テスト追加

**Out-of-Scope**
- offset storage backend の変更（ファイルシステム維持）

---

## Assumptions

1. 現行実装: `consumer_id.replace("/", "_").replace("..", "_")` → 衝突リスクあり（例: `a/b` と `a_b` が同一ファイル名になる）。
2. 後方互換性: 既存の offset ファイル名を変更すると consumer が offset を失う。

## Implementation Steps

### Phase 1: 設計
- [ ] 1-1. 衝突耐性の高い mapping strategy を決定:
  - Option A: `sha256(consumer_id.encode())[:16].hex()` → 完全衝突フリー
  - Option B: URL エンコーディング（`urllib.parse.quote_plus`）→ 可読性あり、衝突フリー
- [ ] 1-2. 既存 offset ファイルの migration 要否を判断

### Phase 2: 実装
- [ ] 2-1. `offsets.py` の `safe_id` 計算を選択した strategy に変更
- [ ] 2-2. 後方互換性のある migration パスを実装（必要な場合）

### Phase 3: テスト・ドキュメント
- [ ] 3-1. edge case（`/`, `..`, 空文字, 同一 hash 衝突確認）テスト追加
- [ ] 3-2. docs に canonical rule を記述
