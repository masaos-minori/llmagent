# Plan: Separate /mcp status semantics from pending reload config

Source requirement: `requires/20260708_37_require.md` (M-5)

## Goal

Confirm `/mcp` / `/mcp status` always reflects the running (pre-restart) MCP
config, never a pending reload diff, and document this boundary explicitly
— mirroring the H-8 watchdog guarantee for a different consumer.

## Scope

In scope:
- `docs/05_agent_07_cli-and-commands.md` — expand the `/mcp` /
  `/mcp status` command description to state it shows running config only.
- `tests/test_agent_mcp_status.py` (or `test_cmd_mcp.py`) — add a reload-
  interaction test.

Out of scope:
- `scripts/agent/commands/cmd_mcp.py`, `scripts/agent/services/mcp_status.py`
  — verified below that `McpStatusService.probe_all()` already iterates
  `ctx.cfg.mcp.mcp_servers.items()` live, at call time, with no snapshot or
  overlay of pending config. No code change needed, by the same
  no-mutation guarantee established in the H-1 plan.

## Assumptions

1. `McpStatusService.probe_all()` (`scripts/agent/services/mcp_status.py:56-63`)
   iterates `ctx.cfg.mcp.mcp_servers.items()` fresh on every call and passes
   each `cfg` (a live `McpServerConfig` reference) into
   `_probe_single_server()`, which reads `cfg.url` and `cfg.auth_token`
   directly — verified by reading that file. There is no caching, snapshot,
   or "pending config" overlay anywhere in this path.
2. `_cmd_mcp_status()` (`scripts/agent/commands/cmd_mcp.py:70-100`) also
   reads `ctx.cfg.mcp.mcp_servers` directly for the degraded-servers list
   (line 94) — same live dict, same guarantee.
3. Given assumptions 1-2, and the H-1 design's guarantee that
   `_classify_mcp_server_changes()` never mutates
   `ctx.cfg.mcp.mcp_servers`, every acceptance criterion in this requirement
   is already true by construction:
   - new servers pending restart are absent from `ctx.cfg.mcp.mcp_servers`
     until an actual restart rebuilds it, so `/mcp` cannot show them,
   - removed servers remain present (not yet deleted) until restart, so
     `/mcp` cannot hide them,
   - `cfg.url` on an existing key is untouched, so `/mcp` always probes the
     running URL, never a pending one.
4. `docs/05_agent_07_cli-and-commands.md:98-99` currently describes `/mcp` /
   `/mcp status` only as "HTTP probe to all MCP servers" / "Displays health
   table" — it does not state that this reflects running config only, which
   is the one genuine documentation gap this requirement asks to fill.
5. Neither `tests/test_agent_mcp_status.py` nor `tests/test_cmd_mcp.py`
   references `reload` — verified by `grep -n "reload"` on both files (no
   matches); no existing test proves this boundary.

## Design

**Documentation** — replace the two `/mcp*` rows
(`docs/05_agent_07_cli-and-commands.md:98-99`) with a row plus a clarifying
note:

```markdown
| `/mcp` | HTTP probe to all MCP servers | Displays health table (running config only) |
| `/mcp status` | HTTP probe to all MCP servers | Displays health table (running config only) |

`/mcp` / `/mcp status` is a health view of the **currently running** MCP
server configuration — it is not a preview of pending `/reload` changes.
After `/reload` reports `[RESTART]` items, `/mcp` continues to show the
pre-reload servers, URLs, and auth state until the agent is actually
restarted. See
[Hot-Reload Scope](#hot-reload-scope-reload) for what `/reload` reports.
```

**Test** — add to `tests/test_agent_mcp_status.py`:

```python
async def test_probe_all_reflects_running_config_after_reload_classification() -> None:
    """probe_all() must not see servers/fields that only exist in a pending reload."""
    from agent.services.config_reload import ConfigReloadService
    from agent.services.mcp_status import McpStatusService
    from shared.mcp_config import McpServerConfig, TransportType

    old_srv = McpServerConfig(transport=TransportType.HTTP, url="http://localhost:8080", cmd=[])
    ctx = _make_ctx_with_mcp_servers({"svc": old_srv})  # existing test helper

    new_srv = McpServerConfig(transport=TransportType.HTTP, url="http://127.0.0.1:9999", cmd=[])
    reload_svc = ConfigReloadService(ctx)
    with patch("agent.config_builders._build_mcp_servers", return_value={"svc": new_srv, "new_one": new_srv}):
        outcome = reload_svc._classify_mcp_server_changes(ctx, {})

    status_svc = McpStatusService(ctx)
    rows = await status_svc.probe_all()

    assert {r.key for r in rows} == {"svc"}  # "new_one" not shown — pending restart
    assert all(r.endpoint != "http://127.0.0.1:9999" for r in rows)  # old URL still probed
    assert "mcp/svc.url" in outcome.needs_restart
    assert "mcp/new_one (new server)" in outcome.needs_restart
```

(Adapt the HTTP probe to a mocked transport per existing conventions in
`test_agent_mcp_status.py`.)

## Risks

- **Risk**: This requirement, like H-8, is satisfied entirely by the H-1
  no-mutation guarantee; if a future change to `config_reload.py`
  reintroduces mutation, both this test and the H-8 watchdog test should
  catch it independently.
  → **Mitigation**: Intentional redundancy — two independent consumers
  (`/mcp status`, watchdog) each have their own regression test, so a future
  regression is caught regardless of which consumer a reviewer happens to
  check first.
- **Risk**: The new doc cross-reference
  (`#hot-reload-scope-reload`) must match the actual heading anchor for
  "## Hot-Reload Scope (`/reload`)" in the same file.
  → **Mitigation**: Verify the anchor slug matches Markdown's auto-generated
  anchor for that exact heading text before finalizing the edit.
