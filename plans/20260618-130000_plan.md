# Plan: Simplify and standardize tool routing rules

## Goal

Define one clear primary routing method, reduce special-case tools, document how new tools are added to routing, and improve startup validation for routing gaps.

## Scope

**In scope:**
- `scripts/shared/route_resolver.py` â€” analyze and simplify routing logic
- `scripts/shared/tool_constants.py` â€” tool name sets used by route_resolver
- Tool routing config in `config/agent.toml` (tool_names lists)

- Startup validation for routing gaps

**Out of scope:**
- Modifying existing tool definitions in individual MCP server files
- Changing the tool_constants.py sets (READ_TOOLS, WRITE_TOOLS, etc.) unless simplification requires it
- Modifying the agent's tool execution layer (`tool_executor.py`)
- Changes to `/v1/tools` discovery endpoint behavior

## Assumptions

1. The three-tier routing priority (discovery â†’ config â†’ static fallback) is correct and should be preserved, but the fallback tier needs stricter controls.
2. `query_sqlite` and `shell_run` are the only tools that use `_EXACT_ROUTES` instead of set-based matching; they could potentially be moved to set-based routing if their server keys align with config.
3. GitHub tools use prefix matching (`github_` prefix) â€” this is a special case because GitHub has 21 tools with different prefixes but one server.
4. Adding a new tool requires: (a) adding it to a tool set in `tool_constants.py`, (b) configuring `tool_names` in the MCP server config, (c) potentially adding to `_SET_ROUTES` or `_EXACT_ROUTES`. This should be reduced to a single documented procedure.

## Implementation Steps

1. **Simplify `_EXACT_ROUTES` in `route_resolver.py`**
   - Move `shell_run` â†’ `shell` and `search_web` â†’ `web_search` to `_SET_ROUTES` as single-element sets
   - Remove `_EXACT_ROUTES` dict entirely if all entries can be set-based
   - Keep `_GITHUB_PREFIX` matching (prefix-based routing is appropriate for tools with shared prefix pattern)
   - Completion criterion: `_EXACT_ROUTES` is empty or removed; all routing uses `_SET_ROUTES` + prefix matching

2. **Add startup validation for routing gaps**
   - After building `_config_map`, compare against `tool_definitions` in config
   - Log warnings for tools in `tool_definitions` that have no routing entry (neither config map nor static fallback)
   - In strict_mode, raise error for unmapped tools (existing behavior for resolve())
   - Completion criterion: Startup logs show routing coverage status; unmapped tools are flagged

3. **Document the new tool registration procedure**
   - Add one clear section in `04_mcp_06_configuration_and_operations.md` or a new doc
   - Procedure: (1) add tool name to appropriate set in `tool_constants.py`, (2) add tool name to `tool_names` in MCP server config, (3) verify at startup via routing coverage log
   - Completion criterion: New developer can add a tool following documented steps

4. **Improve startup validation messaging**
   - Log summary of routed vs unrouted tools at startup
   - Example: "Routing: 52/53 tools mapped; 1 tool (example_tool) has no routing entry"
   - Completion criterion: Startup output includes routing coverage summary
