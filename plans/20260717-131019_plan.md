# Plan: Add MCP tool schema versioning and validation tests

Source requirement: `requires/20260717_12_require.md`

## Goal

MCP servers may optionally include a `schema_version` field in their `/v1/tools` response; the Agent tolerates its absence during migration; a shared validation-test suite checks every server's tool schema (non-empty name, description, object `inputSchema`, valid optional `status`/`is_write`/`requires_serial`/`resource_scope`) and fails with clear server+tool identification on violation.

## Scope

**In scope**
- `scripts/mcp_servers/server.py`: `MCPServer.list_tools_with_server_key()` (confirmed by direct read, lines 128-136: currently returns `[{**t, "server_key": server_key} for t in tools]`, no `schema_version`) â€” the `/v1/tools` endpoint's response wrapper (the `{"tools": [...]}` dict each server's own `list_tools()` route handler builds, e.g. `mdq/server.py:189-192`) gains an optional top-level `schema_version` key.
- Each MCP server's `server.py` (`mdq`, `cicd`, `github`, `git`, `web_search`, `shell`, `file_read`/`file_write`/`file_delete`): confirm/add `schema_version` to their `/v1/tools` response dict, following the proposed shape.
- Each MCP server's `tools.py` (the `TOOL_LIST` definitions, e.g. `mdq/tools.py:21-36`): no structural change required â€” `schema_version` is a response-wrapper concern, not a per-tool field; per-tool optional fields (`status`, `is_write`, `requires_serial`, `resource_scope`) may already exist on some tool definitions and should be validated, not necessarily added where absent (they're optional).
- `scripts/agent/services/mcp_tool_discovery.py` (requirement 03/09's module): tolerate a missing `schema_version` (migration-friendly); validate the fields listed in Scope against every fetched tool entry, reusing/extending the validation logic already planned there (requirement 03's "Discovery validation rules").
- New test suite covering the validation rules, targeting the real test-file location (`tests/test_mcp_tool_discovery.py`, per the now-consistently-confirmed path-mismatch pattern â€” the requirement's stated `tests/mcp_servers/` and `tests/agent/services/test_mcp_tool_discovery.py` paths don't exist).

**Out of scope**
- Enforcing `schema_version` as required â€” explicitly optional per the requirement's own acceptance criteria ("MCP servers *may* include," "Agent tolerates missing... during migration").
- Redesigning `TOOL_LIST` definitions across all 9 MCP servers â€” only adding `schema_version` to the response wrapper, not restructuring existing tool schemas.
- Version-specific validation logic (e.g. different rules for `schema_version: "2.0"` vs `"1.0"`) â€” this requirement introduces the *field*, not a versioned-validation-branching mechanism; that's a natural future extension if a real v2 shape ever emerges, not built speculatively now.

## Assumptions

1. `MCPServer.list_tools_with_server_key()` (`scripts/mcp_servers/server.py:128-136`, confirmed by direct read) is a base-class helper each server's own `/v1/tools` route handler calls to build its response â€” the exact response-wrapping (the outer `{"tools": [...]}` dict) happens in each server's own `server.py` (e.g. `mdq/server.py:188-192`'s `list_tools()` route function), not in the shared base class. Adding `schema_version` requires touching each server's route handler (or, more cleanly, adding a shared helper on `MCPServer` that wraps both `schema_version` and the tools list, which each server's route calls instead of hand-building the dict â€” see Design).
2. `tests/mcp_servers/` and `tests/agent/services/test_mcp_tool_discovery.py` (the requirement's stated paths) don't exist â€” confirmed via direct `ls`/`find`; the real, existing test file for MCP server base behavior is `tests/test_mcp_server_base.py`, and the discovery-service test file (from requirements 03/09) is `tests/test_mcp_tool_discovery.py` per the now-repeatedly-confirmed flat `tests/` layout.
3. The proposed `/v1/tools` response shape's per-tool optional fields (`status`, `is_write`, `requires_serial`, `resource_scope`) are the same fields requirement 03's discovery-validation design already plans to type-check when present (per that plan's "Discovery validation rules" section) â€” this requirement's test-writing scope reuses that validation logic rather than inventing new rules, adding the missing `schema_version`-tolerance test specifically.
4. This requirement's validation tests should run against **each server's actual `TOOL_LIST`/route handler** (a per-server test matrix), not just the shared discovery-service logic in isolation â€” "clear server and tool identification" in the acceptance criteria implies failures must name which server+tool violated which rule, which requires testing real server response data, not only synthetic fixtures.

## Design

### Shared `schema_version` constant

Rather than 9 independently-hardcoded `"schema_version": "1.0"` literals (Unknown's resolution), define `MCP_TOOL_SCHEMA_VERSION = "1.0"` once in `scripts/mcp_servers/server.py`, and extend `MCPServer` with a wrapping helper (e.g. `list_tools_response()`) that returns `{"schema_version": MCP_TOOL_SCHEMA_VERSION, "tools": self.list_tools_with_server_key()}` â€” each server's route handler calls this new helper instead of hand-building `{"tools": [...]}`, guaranteeing consistency and a single point of future version bumps.

### Discovery-side tolerance

`mcp_tool_discovery.py`'s response parsing treats `schema_version`'s absence as a normal, non-error case (per "Agent tolerates missing schema_version during migration") â€” simply reads it if present (for logging/future-compatibility branching) and proceeds identically whether present or absent for this version's validation rules.

### Test matrix with clear identification

Per Assumption 4, the validation tests should iterate every configured MCP server's actual `TOOL_LIST` (or a live/simulated `/v1/tools` response) and assert each rule per tool, with failure messages including both `server_key` and tool `name` â€” mirroring the existing style already used in `repl_health.py`'s drift-detection warnings (e.g. `f"[{server_key}] tool {t!r}..."`, confirmed in requirement 09's investigation) for consistency with this codebase's existing error-message conventions.

## Implementation steps

1. **Add `MCP_TOOL_SCHEMA_VERSION` constant and a response-wrapping helper** to `scripts/mcp_servers/server.py`.
2. **Update each of the 9 MCP servers' `server.py`** to call the new helper instead of hand-building their `/v1/tools` response dict.
3. **Update `mcp_tool_discovery.py`** to read (optionally) and tolerate-if-absent the `schema_version` field.
4. **Write the validation test matrix** in `tests/test_mcp_tool_discovery.py` (or split per-server cases into `tests/test_mcp_server_base.py` if that's a better fit given existing conventions there) â€” one parametrized test per validation rule (non-empty name, description present, `inputSchema` is an object, optional `status`/`is_write`/`requires_serial`/`resource_scope` type-checked when present), run against every server's actual tool list.
5. **Verification**: run the standard validation sequence scoped to changed files, then the full suite, including a live check that each of the 9 servers' `/v1/tools` endpoint now includes `schema_version` (integration-level, if such a test harness exists) or at minimum a unit-level check on each server's route handler.

## Risks

- **Risk**: Adding `schema_version` to 9 separate `server.py` files independently (if the shared-helper approach in Design isn't adopted) risks inconsistent literal values or partial rollout (some servers updated, others not).
  **Mitigation**: use the shared constant/helper (per Design) specifically to prevent this class of drift.
- **Risk**: The validation test matrix could surface pre-existing, previously-unnoticed schema quality issues in one or more of the 9 servers' actual `TOOL_LIST` definitions (e.g. a tool missing a description) â€” this requirement's tests would then fail on existing, real data, not just hypothetical cases.
  **Mitigation**: treat any such finding as a legitimate, in-scope fix (correct the offending server's tool definition) rather than weakening the test to accommodate a real quality gap â€” this is exactly the kind of issue this requirement exists to catch.
