# Refactoring Plan: RAG Search Quality and Retrieval Control Improvements

## Goal

RAGパイプラインの4つの改善点を実装する:
1. two-stage fetch の決定を structured response に変換する
2. `rag_min_score` と `max_chunks_per_doc` による最終採用制御を明示的にする
3. semantic cache キーに `history_context` を含める（現状 cache は初期化されているが未使用）
4. FTS5 クエリの reserved term を確実にエスケープし、有効な column filter (`content:`) を適用する — **注: `chunks_fts` には `passage` カラムが存在しないため `passage:` FTS5 フィルタは使用不可（確認済み）**

---

## Design

### Step 1: TwoStageFetchResult DTO (`scripts/rag/models.py`)

```python
@dataclass(frozen=True)
class TwoStageFetchResult:
    """Typed result of a two-stage fetch operation."""
    hits: list[RagHit]           # reranked and filtered hits
    min_score_applied: float     # rag_min_score used for filtering (0.0 = no filter)
    max_chunks_per_doc: int      # dedup limit applied
    total_before_filter: int     # count before rag_min_score filter
    total_after_filter: int      # count after filter (== len(hits))

    @property
    def was_filtered(self) -> bool:
        return self.total_before_filter > self.total_after_filter
```

`pipeline.py` の `last_reranked: list[RagHit]` を `last_fetch_result: TwoStageFetchResult | None = None` に変更し、フィルタ・dedup の結果をまとめて保持させる。

現状: `rerank_candidates()` が内部で `rag_min_score` フィルタと `deduplicate_chunks()` を適用して `list[RagHit]` を返す。

変更後:
```python
async def rerank_candidates(self, query: str, merged: list[RagHit]) -> TwoStageFetchResult:

    if not self._cfg.use_rerank:

            hits=hits,
            min_score_applied=0.0,
            max_chunks_per_doc=self._cfg.max_chunks_per_doc,
            total_before_filter=len(candidates),
            total_after_filter=len(hits),
        )
    # rerank あり: cross_encoder_rerank → rag_min_score フィルタ → dedup
    ...
    return TwoStageFetchResult(...)
```

呼び出し元 (`mcp/rag_pipeline/service.py`) では `result.last_fetch_result.hits` でアクセスする。

### Step 3: SemanticCache のキーに history_context を含める (`rag/cache.py`)

**現状の問題**: `SemanticCache.lookup(embedding)` は埋め込みベクトルのみでキャッシュヒットを判定。同一クエリで `history_context` が異なる場合、不適切なキャッシュヒットが発生する可能性がある。さらに、現状は `pipeline.py` でキャッシュが全く呼ばれていない。

**変更後シグネチャ:**
```python
class SemanticCache:
    def lookup(self, embedding: list[float], history_context: str = "") -> str | None:
        # embedding の近傍探索に加えて history_context の完全一致も条件とする
        for entry in self._entries:
            if entry.history_context != history_context:
                continue
            sim = cosine_sim(embedding, entry.embedding)
            if sim >= self._threshold:
                return entry.context_str
        return None

    def put(self, embedding: list[float], history_context: str, context_str: str) -> None:

        ...
```

`CacheEntry` (`rag/models.py`) に `history_context: str = ""` フィールドを追加する。

**`pipeline.py` でのキャッシュ有効化 (augment() 内):**
```python
async def augment(self, query: str, ..., history_context: str = "") -> str:
    if not self._cfg.use_search:
        return ""
    # Semantic cache lookup (embedding-based)
    if self._cfg.use_semantic_cache:
        emb = await get_embedding(query, self._http, self._embed_url)
        if emb:
            cached = self.semantic_cache.lookup(emb, history_context)
            if cached is not None:
                ctx_stats.stat_semantic_cache_hits += 1  # 既存のstat
                return cached
    # ... (既存の run() 呼び出し)
    # キャッシュ書き込み
    if self._cfg.use_semantic_cache and emb:
        self.semantic_cache.put(emb, history_context, context_block)
    return context_block
```

> **注意**: `ctx` (AgentContext) が `augment()` から直接アクセスできないため、`stat_semantic_cache_hits` のインクリメントは呼び出し元 (orchestrator) か callback 経由で行う。または `augment()` が `AugmentResult(context: str, cache_hit: bool)` を返す形も検討できる。

### Step 4: FTS5 クエリの reserved term エスケープと column filter 適用

**テーブル定義の確認結果（`scripts/db/create_schema.py`）:**
- `chunks_fts`: カラムは `content` のみ (`tokenize = 'unicode61'`)。`passage:` カラムは存在しない
- `memories_fts`: カラムは `memory_id`(UNINDEXED), `content`, `summary`, `tags`。`passage:` カラムは存在しない

**`passage:` / `query:` FTS5 フィルタについて:**
- FTS5 の column filter 構文 `<column>:<query>` は存在するカラム名のみ使用可能
- `passage:` は両テーブルで使用不可（`passage` カラムがない）
- 代替として `content:` column filter を使うことは可能（`chunks_fts` / `memories_fts` の両方に `content` カラムあり）

**`scripts/agent/memory/retriever.py:_build_fts_query()` の問題:**
- 現状: `" OR ".join(tokens)` のみ。AND/OR/NOT/NEAR が含まれていた場合にクエリ構文エラー
- 変更後: 全トークンを `"..."` で引用することで reserved terms を確実にエスケープし、`content:` column filter を適用する

```python
_FTS5_RESERVED = frozenset({"AND", "OR", "NOT", "NEAR"})

def _build_fts_query(text: str) -> str:
    """Build FTS5 MATCH query with content: column filter and reserved term escaping.

    Note: passage: column filter is not available (memories_fts has no passage column).
    All tokens are double-quoted to escape FTS5 reserved terms (AND/OR/NOT/NEAR).
    """
    tokens = re.findall(r"\w+", text)
    if not tokens:
        return '""'
    # Double-quote all tokens to escape AND/OR/NOT/NEAR as literals
    quoted = " OR ".join(f'"{t}"' for t in tokens)
    return f"content: {quoted}"
```

**`scripts/rag/repository.py:_build_fts_query()` の現状:**
- トークンはすでに `f'"{t}"'` で引用されており、reserved terms は事実上エスケープ済み
- 変更: `content:` column filter を前置するだけ（`passage:` は不可）

```python
def _build_fts_query(text: str) -> str:
    ...
    return "content: " + " ".join(f'"{t}"' for t in sanitized)
```
