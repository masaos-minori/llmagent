# Plan: L-2 â€” Classify startup messages as fatal or non-fatal

## Goal

Improve startup diagnostics by classifying each warning as either `[warn:non-fatal]` (degraded local services) or `[fatal]` (blocking initialization errors), so developers can immediately tell which messages require action and which are expected in local profile.

## Scope

**In**:
- Add `write_fatal(msg)` method to `CLIView` / `CLIViewProtocol` in `scripts/agent/cli_view.py`
- Update `StartupOrchestrator._check_services()` in `scripts/agent/startup.py` to route:
  - LLM/MCP unreachable in non-production mode â†’ `write_warning` with `[non-fatal]` prefix
  - Plugin skip in fail-open mode â†’ `write_warning` with `[non-fatal]` prefix
  - Fatal initialization errors â†’ `write_fatal` (new method, prints `[fatal]` prefix and actionable hint)
- Update `AgentSession.start()` in `scripts/agent/session.py` to catch `sqlite3.OperationalError` on missing `sessions` table and re-raise with an actionable message: `"Run: bash deploy/init_db.sh"`
- Update `repl.py._run_repl_loop()` to catch the new actionable error and display it via `write_fatal`
- Update message strings in `repl_health.py` (`check_service_health`, `check_readiness`) to include severity label in the warning text

**Out**:
- Changing readiness policy (what causes abort vs. continue)
- `/exit` handling
- sqlite-vec path handling
- Production mode fatal raise behavior (already correct)
- Changes to watchdog or healthcheck intervals

## Assumptions

1. `[warn]` prefix in `write_warning` will become `[warn:non-fatal]` for local-profile degraded service messages, and `[fatal]` will be a new prefix emitted by `write_fatal`.
2. The only fatal session schema error is `sqlite3.OperationalError` with "no such table: sessions" from `session.start()`.
3. The command to initialize the schema is `bash deploy/init_db.sh` (confirmed from `deploy/init_db.sh`).
4. Plugin skip in fail-open mode is already logged only as a warning; no behavior change needed, only message wording.
5. LLM/Embed unreachable in non-production mode is already non-fatal (confirmed in `check_readiness()`); only message wording needs updating.
