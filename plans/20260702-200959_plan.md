# Plan: Remove Migration Helpers from create_schema.py

## Goal

Remove all `_migrate_*` functions and their call sites from `create_schema.py` so that schema creation executes only canonical DDL from `schema_sql.py`.

## Scope

**In-Scope:**
- Delete `_migrate_rag_schema()` from `scripts/db/create_schema.py`
- Delete `_migrate_session_schema()` from `scripts/db/create_schema.py`
- Delete `_migrate_workflow_schema()` from `scripts/db/create_schema.py`
- Delete `_migrate_add_undone_column()` from `scripts/db/create_schema.py`
- Remove all call sites of the above functions inside `create_rag_schema()`, `create_session_schema()`, and `create_workflow_schema()`
- Remove `import sqlite3` if it becomes unused after the removal
- Remove stale `type: ignore[arg-type]` comments that guarded `db.conn` access in migration calls
- Update the module docstring to remove references to migration helpers
- Remove or update migration-related tests in `tests/test_create_schema.py`

**Out-of-Scope:**
- `mcp/mdq/service.py::_migrate_from_legacy()` — unrelated migration in a different module
- `scripts/db/schema_sql.py` — DDL definitions are not changed
- Data migration tooling
- Backward compatibility for old DB schemas
- Production DB file recreation

## Assumptions

1. All migration columns (`chunk_type`, `source_file`, `workflow_id`, `undone`) are already included in the canonical DDL in `schema_sql.py`; the migrations exist only for pre-DDL databases.
2. No other module outside `create_schema.py` calls `_migrate_rag_schema`, `_migrate_session_schema`, `_migrate_workflow_schema`, or `_migrate_add_undone_column` (confirmed: `grep -rn "_migrate_"` shows no external callers).
3. The project policy states existing DB migration is not supported; DBs must be recreated from latest DDL.
4. `sqlite3` is imported only for `sqlite3.Connection` type hints in the migration functions and for `sqlite3.OperationalError`/`sqlite3.DatabaseError` in exception handling. After removal, `sqlite3` is still needed for the exception types.
