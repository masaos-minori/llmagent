## Goal

Every `[mcp_servers.<name>]` `cmd` entry in `config/agent.toml` launches a script path that exists on
disk, for the 6 servers still broken (`shell`, `git`, `github`, `cicd`, `rag_pipeline`, `mdq`), and a
regression test locks this invariant for every subprocess-mode MCP server going forward.

## Scope

**In scope:**
- `config/agent.toml` — fix the `cmd` array's final path segment for exactly 6 `[mcp_servers.<name>]`
  blocks: `shell` (line 292), `git` (304), `github` (357), `cicd` (366), `rag_pipeline` (375), `mdq`
  (387). Change `<name>/server.py` → `<name>/<name>_server.py`. No other field changes.

  `startup_mode="subprocess"` MCP server's `cmd` script path exists on disk, built against the real
  `config/agent.toml` via `agent.config_builders.build_agent_config()` (no override, no mock).
- Update `issues/20260720-070458_mcp_server_agent_toml_cmd_points_to_deleted_files.md`'s `## Status`
  section to resolved once the fix and test land and pass.
- Final repo-wide gate: `grep -n 'models\.py\|tools\.py\|service\.py\|/server\.py"' config/agent.toml`
  returns zero matches after the fix.

**Out of scope:**
- `web_search` — already fixed (confirmed: `config/agent.toml:313` already reads
  `.../web_search/web_search_server.py`).

  '\[mcp_servers\.browser\]' config/agent.toml` has no match; only unrelated `browser_fetch` tool-name

- `file_delete` / `file_write` / `file_read` — different naming convention

  `app_module` self-references are already correct; only the config file is wrong.
- Any change to `startup_mode`, `url`, `auth_token`, `role`, or `tool_names` for any of the 6 entries.
- Actually redeploying the fix to `/opt/llm` — a separate, explicit-approval `deploy` step after this

## Design

1. **`config/agent.toml` fix** — for each of the 6 blocks, change only the last path segment of `cmd`:
   ```toml
   # before (representative; apply the same suffix change to all 6)
   cmd              = ["uv", "run", "--directory", "/opt/llm", "python", "/opt/llm/scripts/mcp_servers/shell/server.py"]
   # after
   cmd              = ["uv", "run", "--directory", "/opt/llm", "python", "/opt/llm/scripts/mcp_servers/shell/shell_server.py"]
   ```
   Applied identically (substituting the server-specific name) to `shell`, `git`, `github`, `cicd`,
   `rag_pipeline`, `mdq`. No other field in any of these 6 blocks changes.

2. **Regression test** (`tests/test_mcp_server_cmd_paths.py`) — call `cfg = build_agent_config()` (no
   override, loads the real `config/agent.toml` + associated `*_mcp_server.toml` files), iterate
   `cfg.mcp.mcp_servers.items()` **(corrected accessor — see Assumption 6; the prior implementation
   doc's sketch used the stale `cfg.mcp_servers` and must not be copied verbatim)**, filter to
   `server_cfg.startup_mode == StartupMode.SUBPROCESS`, and for each assert that `server_cfg.cmd[-1]`,
   with its `/opt/llm/scripts/` prefix replaced by this repo's own `scripts/` directory
   (`Path(__file__).resolve().parent.parent / "scripts"`), exists as a file. Failure message names the
   offending server key and missing path (e.g. `f"mcp_servers.{key}: cmd script {script_path} does not
   exist"`). Before finalizing, confirm the test fails against a temporarily-reverted config (one `cmd`
   entry hand-edited back to bare `server.py`) and passes once restored — proving it is not vacuous.

3. **No code change** — `scripts/shared/mcp_config.py`, `scripts/agent/http_lifecycle.py`, and every
   `<name>_server.py` are already correct; this plan touches only the config file and adds test coverage.

4. **Issue close-out** — replace `issues/20260720-070458_...md`'s `## Status` section with a resolved
   statement per `implementations/20260720-134259_...md.md`'s procedure, cross-referencing the landed
   commit and the new test, and explicitly noting the original 8-server scope narrowed to these final 6
   (the other 2 — `web_search`, `browser` — were already resolved by separate, unrelated changes).
   Explicitly retain the note that production redeployment is a separate follow-up step.
