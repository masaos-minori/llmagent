# Plan: Require auth_token in production MCP deployments

## Goal

Enforce Bearer-token authentication for HTTP MCP servers when the agent runs in production mode, while keeping auth optional for local/dev profiles.

## Scope

**In scope**:
- `scripts/shared/mcp_config.py` — add `SecurityProfile` enum
- `scripts/agent/config_dataclasses.py` — add `security_profile` to `MCPConfig`
- `scripts/agent/config_builders.py` — read `security_profile` from config
- `scripts/agent/repl_health.py` — hard-fail in production mode for HTTP servers without auth_token
- `scripts/agent/repl.py` — pass security profile to audit function
- `tests/test_repl_health.py` — add tests for production-mode auth enforcement
- `config/agent.toml` — add `security_profile` field (default: "local")
- `docs/04_mcp_05_security_and_safety_model.md` — document production vs local security profiles

**Out of scope**:
- Changing the auth middleware itself (`mcp/server.py`)
- Adding auth enforcement to stdio servers
- Modifying per-server auth_token defaults
- Changing deployment scripts or OpenRC services

## Assumptions

1. `security_profile` is a global agent-level setting (not per-server), because production vs local is an operational decision, not a server-specific one.
2. Default value is `"local"` to avoid breaking existing development workflows.
3. The existing `audit_security_defaults()` function is the right place to enforce this — it already iterates over all servers and checks auth_token.
4. "Hard failure" means raising `RuntimeError` during `_check_services()`, which will abort REPL startup.

| Whether `security_profile` should be a top-level AgentConfig field or nested under MCPConfig | Design decision | Nested under MCPConfig keeps it with related MCP settings; follows existing pattern (mcp_servers, mcp_watchdog_*) | No |

## Implementation Steps

1. **Add `SecurityProfile` enum to `mcp_config.py`**
   - Define `class SecurityProfile(StrEnum)` with values `LOCAL = "local"`, `PRODUCTION = "production"`
   - Default: `SecurityProfile.LOCAL`
   - Validation in `__post_init__`: cast str inputs, reject invalid values

2. **Add `security_profile` field to `MCPConfig` in `config_dataclasses.py`**
   - Add `security_profile: SecurityProfile = SecurityProfile.LOCAL`
   - Import `SecurityProfile` from `shared.mcp_config`
   - Add `__post_init__` validation for cross-field consistency (if production mode, at least one HTTP server should have auth_token — but this is a warning, not a hard fail; the hard fail lives in audit)

3. **Read `security_profile` from config in `config_builders.py`**
   - In `build_agent_config()`, read `cfg.get("security_profile", "local")` and pass to `MCPConfig()`
   - Convert str to `SecurityProfile` enum in the builder

4. **Modify `audit_security_defaults()` in `repl_health.py`**
   - Add parameter: `production_mode: bool = False` (derived from `ctx.cfg.mcp.security_profile == SecurityProfile.PRODUCTION`)
   - In production mode, when an HTTP server has empty `auth_token`: raise `RuntimeError` instead of appending a warning
   - Collect all violations first, then raise with a consolidated message listing all affected servers
   - Log startup behavior clearly: log which profile is active and whether auth is required

5. **Update `repl.py`**
   - In `_check_services()`, derive `production_mode` from `ctx.cfg.mcp.security_profile`
   - Pass it to `audit_security_defaults(self._ctx, production_mode=production_mode)`

6. **Add tests in `tests/test_repl_health.py`**
   - Test: production mode with all HTTP servers having auth_token → no warnings
   - Test: production mode with one HTTP server missing auth_token → RuntimeError raised
   - Test: local mode with HTTP servers missing auth_token → warnings only (existing behavior preserved)
   - Test: security_profile enum parsing (valid/invalid values)

7. **Update `config/agent.toml`**
   - Add top-level `security_profile = "local"` key near other MCP-related config

8. **Update documentation in `docs/04_mcp_05_security_and_safety_model.md`**
   - Document the security_profile field and its effect on auth enforcement

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Existing deployments break when they upgrade and set security_profile=production without configuring auth_token | Medium | Default is "local" — production mode must be explicitly opt-in; clear error message lists affected servers |
| `security_profile` config key conflicts with existing keys | Low | New key name is unlikely to collide; validation would catch it anyway |
| Hard-fail during startup may be too aggressive for some use cases | Low | Default is local (no hard-fail); production mode is explicit opt-in; per-server override can be added later if needed |
