## Goal

Verify that `tool_hash_key()` correctly includes the tool name in its hash input to prevent fingerprint collisions between different tools with identical arguments.

## Scope

**In-Scope:**
- Verify current implementation in `scripts/shared/tool_executor_helpers.py` satisfies the requirement
- Confirm existing tests in `tests/test_tool_executor_helpers.py` cover all acceptance criteria
- Add one explicit acceptance-criteria test if any gap is found

**Out-of-Scope:**
- Changing cache key format
- Replacing MD5 with a cryptographic hash

## Pre-Analysis Finding: Requirement Already Satisfied

Inspection of `scripts/shared/tool_executor_helpers.py:49-54` reveals the current implementation is:

```python
def tool_hash_key(name: str, args: dict[str, object]) -> str:
    return hashlib.md5(
        f"{name}:{_json_dumps(args)}".encode(),
        usedforsecurity=False,
    ).hexdigest()
```

This is identical to the expected behavior in the require:
```python
payload = f"{name}:{_json_dumps(args)}"
return hashlib.md5(payload.encode(), usedforsecurity=False).hexdigest()
```

The tool name `name` is already included in the hash input. The difference is only style (inline vs. extracted variable).
