# Plan: Reapply Agent policy to RuntimeToolRegistry after /reload

Source requirement: `requires/20260717_08_require.md`

## Goal

After a successful `/reload`, `RuntimeToolRegistry.apply_policy(ctx.cfg)` (requirement 02) updates only `enabled_for_llm`/`requires_approval`/`agent_safety_tier` on the already-discovered tool set — `/reload` continues to never call `/v1/tools` or rebuild the registry; MCP server changes remain restart-required.

## Scope

**In scope**
- `scripts/agent/services/config_reload.py`: `ConfigReloadService.apply_config_dict()` (or its `_sync_services()` helper) gains a new call to `ctx.services_required.runtime_tools.apply_policy(ctx.cfg)` after config is applied, alongside the existing `_reload_approval_settings()` call.
- `scripts/shared/runtime_tool_registry.py`: **no new method needed** — `apply_policy(cfg)` is already in requirement 02's planned 9-method interface; this requirement is purely about *calling* it from the reload path, not designing it.
- `scripts/agent/context.py`: **no change** — the `RuntimeToolRegistry` field/placement was already settled by requirements 02-03.
- `tests/test_config_reload.py` (real path — requirement's stated `tests/agent/services/test_config_reload.py` doesn't exist, per the now-repeated path-mismatch pattern from requirements 04-07): add a test asserting `apply_policy()` is called on reload, and a regression test asserting no `/v1/tools` fetch occurs.

**Out of scope**
- Building `apply_policy()`'s internals — requirement 02.
- Building `RuntimeToolRegistry`'s discovery/population mechanism — requirement 03.
- Any change to MCP server config reload behavior (still restart-required, unchanged) — explicitly preserved per this requirement's acceptance criteria.

## Assumptions

1. `ConfigReloadService.apply_config_dict()` (`scripts/agent/services/config_reload.py:120-144`, confirmed by direct read) already calls `self._reload_approval_settings(ctx, new_cfg)` (line 134) — the natural place for `runtime_tools.apply_policy(ctx.cfg)` is immediately after this call (or inside `_sync_services()`, line 166+, alongside the other service-sync calls), since approval settings (`tool_safety_tiers`, `approval_risk_rules`) must be applied to `ctx.cfg` *before* `apply_policy()` reads them.
2. Per the prior investigation of this same file (requirement 03's planning pass) and a direct grep for `/v1/tools`/`route_resolver`/`tool_registry`/`discover` in `config_reload.py`, **zero matches were found** — confirming `/reload` today has no MCP-discovery call at all, and this requirement's constraint ("must not rediscover tools") is preserved by simply not adding any such call, not by removing an existing one.
3. `apply_policy(cfg)` (requirement 02's planned method) is designed to "re-derive `agent_safety_tier`/`requires_approval`/`enabled_for_llm` for all registered tools from `cfg`... without needing to re-run MCP discovery" (per that plan's Design section) — this requirement is the actual consumer requirement 02 anticipated when designing that method's purpose.
4. `ctx.services_required.runtime_tools` (the field from requirements 02-03) is guaranteed populated by the time `/reload` can be invoked (reload is a REPL command, only reachable post-startup, well after requirement 03's discovery completes) — no null-check edge case needs special handling beyond what requirement 02's own field-access pattern already assumes.
5. **Test-file path mismatch pattern (now confirmed 4 times across requirements 04-08)**: `tests/agent/services/` does not exist (confirmed in requirement 03's investigation) — the real, existing file is `tests/test_config_reload.py` (flat `tests/` dir).

## Design

`apply_config_dict()`'s existing sequence (per Assumption 1): config validation → `_apply_rag_tool_params` → `_reload_approval_settings` (applies `tool_safety_tiers`/`approval_risk_rules` to `ctx.cfg`) → mask-fields → MCP-server-change classification (restart-required detection, unchanged) → LLM/SSE params → `_sync_services()`. The new `runtime_tools.apply_policy(ctx.cfg)` call is inserted right after `_reload_approval_settings()` (so `ctx.cfg` already reflects the new approval config when `apply_policy()` reads it) and before or alongside `_sync_services()`'s other service-sync calls — treating `RuntimeToolRegistry` policy reapplication as one more "service" being synced, consistent with the existing pattern (LLM client, etc.) already in that method.

The "do not update discovered tool set / server_key / server_url / raw schema / discovery metadata" constraint is satisfied by construction: `apply_policy()`'s own planned design (requirement 02) only touches `enabled_for_llm`/`requires_approval`/`agent_safety_tier` fields on already-existing `RuntimeTool` entries — it has no code path that adds/removes/re-fetches tools, so this constraint doesn't require new guard logic in `config_reload.py`, only correct scoping inside `apply_policy()` itself (already requirement 02's responsibility).
