# Implementation procedure: scripts/mcp_servers/web_search/service.py

Source plan: `plans/done/20260719-195330_plan.md` (Phase 2, Design items 1-3, 7).

Note: this plan's original file-naming assumption (UNK-01, keep bare `server.py`)
was inverted by commit `abcf0820` (deleted bare `server.py`/`models.py`/`tools.py`,
kept `web_search_server.py`/`web_search_models.py`/`web_search_tools.py`). That
inversion does not affect this document — `service.py` is a brand-new file whose
name and responsibility are unaffected by which server-entrypoint name survived.

## Goal

Introduce `scripts/mcp_servers/web_search/service.py` as the sole orchestration
layer for a web search request: build/accept a `SearchRequest`, call
`search_provider.search_duckduckgo`, measure latency, handle the empty-result
case as a single named branch, and update `health.py`/`metrics.py`. This moves
the orchestration body currently living inside
`formatters.py::search_web()` into its own module so `formatters.py` can be
slimmed to pure `SearchResponse` -> text formatting (a separate, dependent
change — see Scope/Assumptions below).

## Scope

**In scope:**
- New file `scripts/mcp_servers/web_search/service.py` exposing an async
  `search_web(args: dict[str, Any]) -> SearchResponse` function with the same
  external signature/behavior as today's `formatters.search_web()`, so
  `formatters.fdisp_search_web()` can call `service.search_web(...)` as a
  drop-in replacement.
- Latency measurement (`time.perf_counter`), the existing `fmt_kvlog`/`Logger`
  logging call (moved verbatim from `formatters.py`), and the empty-result
  branch, isolated into a single named helper (e.g. `_handle_empty_results`)
  per the plan's UNK-04 so the "raise on empty" (current, pre-Issue-2) vs.
  "return empty `SearchResponse`" (post-Issue-2) semantics are a one-line flip.
- Calls into `health.py`/`metrics.py` update hooks on success/failure, IF and
  only if those calls are not already wired elsewhere by the time this is
  implemented (see Assumptions #3 for the required pre-check).
- `deploy/deploy.sh` copy-list note: no per-file line needed — `deploy.sh:73`
  copies `scripts/` as a whole directory.

**Out of scope:**
- Any change to `search_provider.py`'s responsibility (stays the sole
  DuckDuckGo-specific layer; this doc does not alter its signature — if
  Issue 2's `search_timeout_sec` plumbing has landed by implementation time,
  `service.py` simply passes it through, it does not introduce it).
- Slimming `formatters.py` itself (tracked by the pre-existing
  `implementations/20260720-081048_formatters.py.md`, which currently only
  covers the Issue-2 empty-result-raise removal, not the full
  orchestration-to-`service.py` move this plan also requires — the delta is
  noted in this plan's completion report, not duplicated here; whoever
  implements `service.py` must also update `formatters.py` per this plan's
  Design item 4, i.e. delete `search_web()` from `formatters.py` and replace
  `fdisp_search_web()`'s body with a call to `service.search_web(...)`).
- Any change to `web_search_server.py`'s HTTP/exception-handling/audit
  responsibilities (tracked by
  `implementations/20260720-081553_web_search_server.py.md` — see Assumptions
  #3 for the integration risk between that doc and this one).

## Assumptions

1. Current `formatters.py::search_web()` (verified by direct read, 86-line
   file) is exactly:
   ```
   async def search_web(args: dict[str, Any]) -> SearchResponse:
       req = SearchRequest(**args)
       t0 = time.perf_counter()
       results = cast(list[SearchResult], await search_duckduckgo(req.query, req.max_results))
       if not results:
           from mcp_servers.web_search.web_search_models import WebSearchUpstreamError
           raise WebSearchUpstreamError("No results returned from DuckDuckGo")
       ms = (time.perf_counter() - t0) * 1000
       ... fmt_kvlog / Logger.info(...) ...
       return SearchResponse(query=req.query, results=results, provider="duckduckgo")
   ```
   This entire body (minus the `_WEB_DISPATCH`/`dispatch_web_tool` plumbing,
   which stays in `formatters.py`) is what moves into `service.py`.
2. As of this read, Issue 2 (`plans/done/20260719-193357_plan.md`'s described
   change) has **not yet landed** in `formatters.py` — the file still raises
   `WebSearchUpstreamError` on empty results and `search_duckduckgo` is still
   called with only `(query, max_results)`, no timeout parameter (confirmed:
   `search_provider.py`'s `search_duckduckgo(query: str, max_results: int)`
   signature, no `search_timeout_sec` argument). This matches the plan's
   UNK-02 finding. `service.py` must be written so its empty-result branch is
   an isolated, named helper (`_handle_empty_results`) precisely so this gap
   is a one-line change once Issue 2 lands, not a structural rewrite.
3. **Integration risk requiring a pre-implementation check**: the existing
   `implementations/20260720-081553_web_search_server.py.md` document (from a
   sibling plan, `plans/done/20260719-193902_plan.md`) wires
   `health.record_success()`/`health.record_failure()` and
   `metrics.record_query()` calls directly into
   `web_search_server.py::call_tool()`, not into `service.py` — because
   `service.py` did not exist when that document was written. This plan's
   Design item 3 states `service.py` "calls metrics.py and health.py update
   hooks" and Design item 1/2 state `service.py` is health.py/metrics.py's
   "only caller". If `web_search_server.py::call_tool()` has already been
   implemented per the sibling doc by the time `service.py` is built, the
   health/metrics calls must be *moved* from `call_tool()` into `service.py`

   Before writing `service.py`, run
   `grep -n "health\.\|metrics\." scripts/mcp_servers/web_search/web_search_server.py`
   to check which state applies, and update `web_search_server.py` accordingly
   as part of this same implementation step.
4. `health.py`/`metrics.py` APIs (per
   `implementations/20260720-081456_health.py.md` and
   `implementations/20260720-081528_metrics.py.md`) expose
   `health.record_success()` / `health.record_failure(error_type: str)` /

## Implementation

### Target file

`scripts/mcp_servers/web_search/service.py` (new)

### Procedure

1. Create the module with the standard header comment declaring dependency
   direction: `mcp_servers.web_search.service -> mcp_servers.web_search.{search_provider, health, metrics, web_search_models}, shared.formatters, shared.logger`.
2. Move `search_web()`'s body from `formatters.py` verbatim, with these
   changes:
   - Extract the empty-result check into `_handle_empty_results(query: str) ->
     SearchResponse` (or a raise, per current pre-Issue-2 behavior) — a single
     call site, so flipping semantics later is a one-function edit.
   - After computing `ms`/latency and before/after the existing log call, add
     `metrics.record_query(success=True, latency_ms=ms)` and
     `health.record_success()` on the success path (only reached when results
     is non-empty or once Issue 2 changes empty-to-success); on the exception
     path (a `try`/`except WebSearchUpstreamError` around the
     `search_duckduckgo` call), add `metrics.record_query(success=False,
     latency_ms=ms, error_type=...)` and `health.record_failure(error_type)`
     before re-raising.
3. Keep the existing `fmt_kvlog`/`Logger` logging call unchanged in content
   (only its location moves).
4. Export `search_web` as the module's public function; no other public
   surface needed (`formatters.py` is the only importer, per Design item 7's
   dependency direction).

### Method

Pure move-and-wrap: no new business logic beyond the two health/metrics call
sites and the named empty-result helper. Keep cyclomatic complexity low
(single `try`/`except` around the provider call, single `if` inside the named
helper) — verify with `radon cc` per the Validation plan.

### Details

Illustrative shape (pseudocode, not production code):

```python
# service.py
async def search_web(args: dict[str, Any]) -> SearchResponse:
    req = SearchRequest(**args)
    t0 = time.perf_counter()
    try:
        results = cast(list[SearchResult], await search_duckduckgo(req.query, req.max_results))
    except WebSearchUpstreamError as e:
        ms = (time.perf_counter() - t0) * 1000
        metrics.record_query(success=False, latency_ms=ms, error_type="provider_error")
        health.record_failure("provider_error")
        raise

    if not results:
        return _handle_empty_results(req.query)   # UNK-04: one-line flip point

    ms = (time.perf_counter() - t0) * 1000
    logger.info(fmt_kvlog("search", q=req.query[:80], provider="duckduckgo", n=len(results), ms=f"{ms:.0f}"))
    metrics.record_query(success=True, latency_ms=ms)
    health.record_success()
    return SearchResponse(query=req.query, results=results, provider="duckduckgo")


def _handle_empty_results(query: str) -> SearchResponse:
    # pre-Issue-2 default: raise WebSearchUpstreamError("No results returned from DuckDuckGo")
    # post-Issue-2 (once landed): return SearchResponse(query=query, results=[], provider="duckduckgo")
    raise WebSearchUpstreamError("No results returned from DuckDuckGo")
```
