# Implementation: M-3 — Add UNKNOWN state to McpServerHealthState (mcp_health.py)

## Goal

Add `UNKNOWN = "unknown"` to `McpServerHealthState` enum so that callers can
distinguish a server whose health is untracked (no registry) from a genuinely HEALTHY
server.

## Scope

**Target**: `scripts/shared/mcp_health.py`

**Step covered**: Plan M-3 step 1.

**Out of scope**: `mcp_status.py` (covered in separate doc), any other file.

## Assumptions

1. `McpServerHealthState` is a plain `Enum` (not `StrEnum`); confirmed by source.
2. Adding `UNKNOWN` is backward-compatible: existing code checks for specific states
   (HEALTHY, DEGRADED, UNAVAILABLE, HALF_OPEN); none of those checks are affected.
3. `UNKNOWN.value == "unknown"` renders as `"UNKNOWN"` when `.value.upper()` is called.

## Implementation

### Target file

`scripts/shared/mcp_health.py`

### Procedure

#### Step 1: Add UNKNOWN to McpServerHealthState

Current enum:
```python
class McpServerHealthState(Enum):
    HEALTHY = "healthy"
    DEGRADED = "degraded"
    UNAVAILABLE = "unavailable"
    HALF_OPEN = "half_open"
```

New enum:
```python
class McpServerHealthState(Enum):
    HEALTHY = "healthy"
    DEGRADED = "degraded"
    UNAVAILABLE = "unavailable"
    HALF_OPEN = "half_open"
    UNKNOWN = "unknown"
```

Add `UNKNOWN` at the end to avoid renumbering existing members.

### Method

- One-line addition; no logic changes.
- No import changes needed.

### Details

- `UNKNOWN` is intentionally listed last; enum members are ordered by definition order
  but not by integer value (plain `Enum`, not `IntEnum`).
- Callers that iterate `McpServerHealthState` members (e.g. tests) will now see 5
  members instead of 4; update any exhaustive match/switch if present.
- The value `"unknown"` is lower-case; `.value.upper()` renders as `"UNKNOWN"` in the
  `/mcp status` table.
