# Implementation: Fix /health HTTP status semantics in EventBus app.py

Steps covered: Plan 20260626-094720 — Steps 1, 2, 3

---

## Goal

Make the `/health` endpoint return proper HTTP status codes: `200 OK` when healthy, `503 Service Unavailable` when degraded or unhealthy. Current implementation returns `200` even when reporting unhealthy state.

---

## Scope

- **In scope**: `scripts/eventbus/app.py` — `/health` route handler
- **Out of scope**: docs (step 4)

---

## Assumptions

- `/health` currently returns `{"status": "degraded"}` with HTTP 200.
- Correct semantics: `{"status": "ok"}` → 200; `{"status": "degraded"}` → 503; `{"status": "unhealthy"}` → 503.
- FastAPI uses `JSONResponse` with `status_code` parameter.

---

## Implementation

### Target file
`scripts/eventbus/app.py`

### Procedure
1. Read `scripts/eventbus/app.py` — find `/health` route.
2. Step 1 (Determine health state): The handler already computes health state. No change.
3. Step 2 (Fix status code):
   ```python
   @app.get("/health")
   async def health():
       state = await compute_health_state()
       status_code = 200 if state.status == "ok" else 503
       return JSONResponse(content=state.dict(), status_code=status_code)
   ```
4. Step 3 (Add degraded state): If "degraded" is not yet a possible state, add it:
   - DB connected but slow → degraded (503)
   - DB disconnected → unhealthy (503)
   - All systems nominal → ok (200)

### Method
HTTP status code fix. Breaking change for any caller that checks HTTP status (monitoring tools should be using 200/non-200, not parsing the JSON body).
