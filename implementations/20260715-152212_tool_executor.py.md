## Goal

Make `ToolExecutor.execute()` route side-effecting tools directly to
`_raw_execute()` (bypassing the cache entirely), while read-only tools keep
existing TTL caching, LRU eviction, and stampede protection; and make
cache-hit results carry the `server_key` of the server that produced the
original result plus `source="cache"`.

## Scope

**In scope**
- Import `is_side_effect` from `shared.tool_executor_helpers`.
- `execute()` — after the plugin check, branch on `is_side_effect(tool_name)`:
  call `_raw_execute()` directly if `True`, else `_execute_with_cache()` as
  today.
- `_store_and_evict()` — pass `server_key=result.server_key` into the
  `CacheEntry(...)` construction.
- `_execute_with_cache()`'s cache-hit branch — return
  `server_key=cached.server_key` and `source="cache"` (success case only;
  `is_error` cache hits are not possible today since only non-error results
  are ever stored, see Assumptions).

**Out of scope**
- No change to `_raw_execute()`, `_check_startup_mode()`, `_check_health()`,
  `_execute_with_stampede_protection()` signatures or bodies.
- No change to `PluginToolInvoker` bypass (already unconditional, ahead of
  the new branch).
- No removal of caching — read-only tools keep full cache/stampede behavior.

## Assumptions

- `is_side_effect()` (from `implementations/20260715-152212_tool_executor_helpers.py.md`,
  once applied) is available and already covers `write_file`, `edit_file`,
  `create_directory`, `move_file`, `delete_file`, `delete_directory`,
  `shell_run`, `git_add`/`git_commit`/`git_checkout`/`git_pull`/`git_push`,

  `fts_rebuild`.
- `_execute_with_cache()` and `_execute_with_stampede_protection()` remain
  present and unchanged in signature — they simply become unreachable from
  `execute()` for side-effecting tool names; existing direct-call tests in
  `tests/test_tool_executor.py` that invoke them directly continue to work
  unmodified except where the plan explicitly asks for tool-name changes (see
  `implementations/20260715-152212_test_tool_executor.py.md`).
- `_store_and_evict()` (current lines 189-196) is only ever called from
  `_execute_with_cache()` after a non-error result — `result.server_key` is
  always a valid, already-resolved server key at that point (set by

### Details

```python
async def execute(
    self,
    tool_name: str,
    args: dict[str, Any],
) -> ToolCallResult:
    """Execute a tool. Plugin tools bypass cache and MCP routing; side-effecting
    tools bypass the cache and always re-execute; other tools use the cache."""
    plugin_result = await self._plugin_invoker.try_execute(tool_name, args)
    if plugin_result is not None:
        return plugin_result
    if is_side_effect(tool_name):
        return await self._raw_execute(tool_name, args)
    return await self._execute_with_cache(tool_name, args)
```

```python
def _store_and_evict(self, cache_key: str, result: ToolCallResult) -> None:
    """Store a non-error result in the cache and evict LRU entry if needed."""
    self._cache[cache_key] = CacheEntry(
        output=result.output,
        is_error=result.is_error,
        cached_at=time.time(),
        server_key=result.server_key,
    )
    if self._cache_max_size > 0 and len(self._cache) > self._cache_max_size:
        evicted_key, _ = self._cache.popitem(last=False)
        logger.debug("Tool cache LRU evict: %r", evicted_key)
```
