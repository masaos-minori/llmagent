# Implementation Procedure: scripts/mcp_servers/web_search/formatters.py

Source plan: `plans/20260719-193357_plan.md` (Phase 2, Implementation steps item
"Remove the empty-result-raises branch from `search_web()` in `formatters.py`").

Note: this is `scripts/mcp_servers/web_search/formatters.py`, distinct from
`scripts/shared/formatters.py` (already documented separately in
`implementations/done/20260608-212054_shared_formatters.py.md`, an unrelated
file — same basename, different package/path, not a duplicate of this item).

## Goal

`search_web()` treats a zero-result DuckDuckGo response as a successful
`SearchResponse(results=[], ...)`, not an error; `fdisp_search_web()`'s existing
`"No search results found."` branch keeps handling the empty case unchanged.
`search_web()` also threads the configured `search_timeout_sec` into
`search_duckduckgo()`'s now-required timeout parameter.

## Scope

**In scope:**
- Delete the `if not results: raise WebSearchUpstreamError("No results returned
  from DuckDuckGo")` block (currently lines 52-55) inside `search_web()`.
- Update the `search_duckduckgo(req.query, req.max_results)` call site to also
  pass `search_timeout_sec` (sourced from the module's `_cfg` /
  `WebSearchConfig`), since `search_provider.py`'s procedure document changes
  that function's signature to require it.
- No change to `fdisp_search_web()` — its existing
  `if not result.results: return "No search results found."` already produces
  the required behavior once `search_web()` stops raising on empty results.

**Out of scope:**
- No change to `fmt_search_result()`.
- No change to the `_WEB_DISPATCH` table or `dispatch_web_tool()`.
- No change to logging (`fmt_kvlog(...)` call) — it already logs `n=len(results)`
  which will correctly log `n=0` for empty results once the raise is removed.

## Assumptions

1. Current file (verified by direct read, 86 lines) has exactly one
   `WebSearchUpstreamError` raise site, inside `search_web()`, guarded by
   `if not results:`. Removing it is a pure deletion, not a replacement with
   different logic — the function should fall through directly to constructing
   `SearchResponse(query=..., results=results, provider="duckduckgo")` for the
   empty case exactly as it already does for the non-empty case.
2. `search_web()` currently accesses config only indirectly (no module-level
   `_cfg` reference in this file today) — it must import or reference
   `WebSearchConfig`'s loaded instance to obtain `search_timeout_sec` for the
   updated `search_duckduckgo()` call. The plan's Design section (item 2) notes
   `formatters.py` "already has access to the module-level `_cfg` via
   `web_search_models`" — verified: `web_search_models.py` exposes a
   module-level `_cfg: WebSearchConfig` singleton; `formatters.py` will import
   it (e.g. `from mcp_servers.web_search.web_search_models import _cfg` or via
   a `WebSearchConfig.load()` call, matching whichever style
   `web_search_server.py` uses — that file re-loads its own `_cfg` at module
   level rather than importing the models-module singleton, so consistency
   with the existing repo style should be decided at implementation time,
   favoring re-importing the singleton to avoid loading the TOML file twice
   per process).
3. The local, function-scoped `from mcp_servers.web_search.web_search_models
   import WebSearchUpstreamError` (currently inside `search_web()`, used only
   for the raise being deleted) becomes dead code and should be removed along
   with the raise — no other line in `search_web()` uses that locally-scoped
   import.

## Implementation

### Target file

`scripts/mcp_servers/web_search/formatters.py`

### Procedure

1. Delete the `if not results: ... raise WebSearchUpstreamError(...)` block
   (including its now-unused local import) from `search_web()`.
2. Add a way to obtain `search_timeout_sec` (via the shared `_cfg` singleton
   from `web_search_models`, imported at module level alongside the existing
   `SearchRequest`/`SearchResponse`/`SearchResult` import).
3. Update the `search_duckduckgo(req.query, req.max_results)` call to
   `search_duckduckgo(req.query, req.max_results, _cfg.search_timeout_sec)`
   (or equivalent, matching whatever parameter name/order
   `search_provider.py`'s updated signature settles on).
4. Leave the rest of `search_web()` (timing, logging, `SearchResponse`
   construction) unchanged.

### Method

Pure deletion + one call-site argument addition; no new functions, no new
classes, no change to the public dispatch surface (`_WEB_DISPATCH`,
`dispatch_web_tool`).

### Details

Illustrative before/after shape (pseudocode, not production code):

```python
# before (current, lines ~43-56)
async def search_web(args: dict[str, Any]) -> SearchResponse:
    req = SearchRequest(**args)
    t0 = time.perf_counter()
    results = cast(list[SearchResult], await search_duckduckgo(req.query, req.max_results))
    if not results:
        from mcp_servers.web_search.web_search_models import WebSearchUpstreamError
        raise WebSearchUpstreamError("No results returned from DuckDuckGo")
    ms = (time.perf_counter() - t0) * 1000
    ...
    return SearchResponse(query=req.query, results=results, provider="duckduckgo")

# after
async def search_web(args: dict[str, Any]) -> SearchResponse:
    req = SearchRequest(**args)
    t0 = time.perf_counter()
    results = cast(
        list[SearchResult],
        await search_duckduckgo(req.query, req.max_results, _cfg.search_timeout_sec),
    )
    ms = (time.perf_counter() - t0) * 1000
    ...
    return SearchResponse(query=req.query, results=results, provider="duckduckgo")
```

`fdisp_search_web()` is unchanged — its existing `if not result.results: return
"No search results found."` branch already handles the now-non-exceptional
empty-results path.
