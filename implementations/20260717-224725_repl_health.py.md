# Implementation procedure: `scripts/agent/repl_health.py` (function removal)

Source plan: `plans/20260717-130506_plan.md` ("Consolidate tool startup validation into MCP discovery
service", requirement `requires/20260717_09_require.md`), Implementation step 7.

No existing implementations doc (under `implementations/` or `implementations/done/`) matches this
specific change. Many `repl_health.py`-named docs exist under `implementations/done/` (confirmed via
`ls`), but all predate today's `RuntimeToolRegistry`/discovery-service design batch and address unrelated,
earlier topics (probe/watchdog logic, malformed-tools checks being *added*, safety-tier checks, security
audit, schema-version checks) — none proposes *removing* `check_routing_drift_vs_live()`,
`check_tool_definitions_startup()`, or `_validate_tools_response()` in favor of a consolidated service.
Flagged as checked, not a genuine overlap.

## Goal

Remove `check_tool_definitions_startup()` (`repl_health.py:345-348`) and `check_routing_drift_vs_live()`
(`repl_health.py:351-427`) entirely from `repl_health.py`, since their logic has moved into
`McpToolDiscoveryService` (`implementations/20260717-224511_mcp_tool_discovery.py.md`) and their only
caller was `startup.py` (`implementations/20260717-224630_startup.py.md` removes that call site). Remove
`_validate_tools_response()`'s use inside `_collect_server_tool_names_per_server()` only insofar as that
function itself becomes dead code (see Assumption 2) — leave `_validate_tools_response()` itself in place
if `_collect_server_tool_names()` (the *other* caller, used by `check_readiness`/other still-active checks)
still needs it.

## Scope

**In scope**
- Delete `check_tool_definitions_startup()` (lines 345-348) from `repl_health.py`.
- Delete `check_routing_drift_vs_live()` (lines 351-427) from `repl_health.py`.
- Evaluate whether `_collect_server_tool_names_per_server()` (the helper `check_routing_drift_vs_live()`
  used to fetch per-server tool lists) becomes dead code once its only caller is removed — if so, delete
  it too; if any other function still calls it, keep it (verify via grep before deleting, per Method below).
- `check_tool_definitions_runtime()` (`repl_health.py:340-342`) is **kept, unmodified** — it is out of this
  plan's migration list (confirmed zero callers exist anywhere, per the discovery-service doc's Assumption
  1, but the plan does not name it for removal, and removing genuinely-unused-but-not-explicitly-scoped
  code is not this requirement's job).
- `_validate_tools_response()` (`repl_health.py:130-158`): **keep** — it is still called from
  `_collect_server_tool_names()` (`repl_health.py:198`), which is a separate, still-active function used
  elsewhere (confirmed caller chain needs re-verification at implementation time — see Assumption 3).

**Out of scope**
- `check_routing_drift()` (`repl_health.py:496-524`) and `check_routing_safety_tiers()`
  (`repl_health.py:527-533`) — untouched, per plan's explicit exclusion.
- Any change to `_check_tool_definitions()` (the shared helper both `check_tool_definitions_runtime()` and
  the now-removed `check_tool_definitions_startup()` called) — it must remain importable, since
  `mcp_tool_discovery.py` now imports it directly (per
  `implementations/20260717-224511_mcp_tool_discovery.py.md`, Assumption 1) and
  `check_tool_definitions_runtime()` still uses it too.

## Assumptions

1. Confirmed by direct read: `check_tool_definitions_startup()` is a 2-line wrapper at
   `repl_health.py:345-348` (exact form not re-quoted here — already confirmed by the plan's own Assumption
   1 and independently by this cycle's investigation: it calls
   `_check_tool_definitions(ctx, strict=getattr(ctx.cfg.tool, "tool_definitions_strict", False))`).
   Deleting it does not affect `_check_tool_definitions()` itself, which stays.
2. `check_routing_drift_vs_live()` (`repl_health.py:351-427`) internally calls
   `_collect_server_tool_names_per_server(ctx)` (line 374) to get `(per_server, unreachable)`. This helper
   is distinct from `_collect_server_tool_names()` (singular, no `_per_server` suffix, used by
   `check_readiness()`/other callers per `repl_health.py:161-259`'s docstring). Before deleting
   `_collect_server_tool_names_per_server()`, grep for any other caller
   (`rg -n "_collect_server_tool_names_per_server" scripts/ tests/`) — if `check_routing_drift_vs_live()`
   is its only caller (expected, but not independently re-confirmed line-by-line in this pass beyond the
   name-distinction check), delete it as dead code alongside the two named functions; otherwise keep it.
3. `_validate_tools_response()` is called from two places today: `_collect_server_tool_names()`
   (`repl_health.py:198`) and `_collect_server_tool_names_per_server()` (`repl_health.py:245`). If
   Assumption 2's grep confirms `_collect_server_tool_names_per_server()` becomes dead and is deleted, the
   `repl_health.py:245` call site disappears with it — but the `repl_health.py:198` call site (inside
   `_collect_server_tool_names()`, used by other still-active checks such as `check_readiness`) remains,
   so `_validate_tools_response()` itself must **not** be deleted, only the now-orphaned call site removed
   along with its enclosing dead function.
4. No test file directly imports `check_routing_drift_vs_live` or `check_tool_definitions_startup` from
   `agent.repl_health` — confirmed via the investigation's grep: the only direct-import test is
   `tests/test_routing_duplicate_ownership.py` (lines 70, 93, 115: `from agent.repl_health import
   check_routing_drift_vs_live`). This file's tests **will break** on `ImportError` once the function is
   removed — this is exactly the risk the plan's own step 9 flags ("particular attention to any existing
   test that directly imports ... since those import paths will break if the functions are fully removed
   rather than shimmed"). Per Assumption 4 of the plan (no other caller besides `check_routing_drift_vs_live`
   for `validate_routing_against_live`, now doubly confirmed) and since this plan chooses full removal (not
   a shim, per Unknown #1's resolution finding zero live callers to preserve), `tests/
   test_routing_duplicate_ownership.py`'s 3 tests referencing `check_routing_drift_vs_live` directly
   (lines 65-131) must be migrated to test the equivalent logic inside
   `McpToolDiscoveryService`/`tests/agent/services/test_mcp_tool_discovery.py` instead (see the paired test
   doc), and the old direct-import tests removed from `test_routing_duplicate_ownership.py` (or that whole
   file retired if nothing else in it survives — verify remaining content before deciding, not assumed
   here).
5. `_validate_tools_response()`'s own dedicated unit tests
   (`tests/test_repl_health_malformed.py`, confirmed 10 direct-call unit tests at lines 10-72) are
   **unaffected** — since `_validate_tools_response()` itself is kept (Assumption 3), these tests need no
   change.
