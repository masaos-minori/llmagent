## Goal

Add one new test to `tests/test_mdq_service.py` that indexes a Markdown file containing a
fenced code block with a unique token, then asserts `search_docs()` does NOT return that
document when searching for the token — locking in that fence content is excluded from the
searchable/indexed section text end-to-end (parsing → indexing → FTS5 search), not merely from
heading detection. Per the plan's Assumption 5, this test is expected to **pass against current
code with zero code changes** (it documents/locks existing behavior, it does not fix a bug). If
it unexpectedly fails, per `rules/coding.md`'s classification table this becomes an
"Implementation fix required" case — stop and file an issue rather than adjusting the doc to
match a bug.

### Method

Pseudocode/signature for the new test (no production code, per this skill's rules — the actual
test body is straightforward pytest and will be written directly against the real
`index_paths`/`search_docs` signatures at implementation time):

```python
class TestSearchDocs:
    ...

    def test_fence_content_excluded_from_search_index(
        self, service: MdqService, tmp_path: Path
    ) -> None:
        """A unique token that exists only inside a fenced code block must not be
        returned by search_docs — fence content is excluded from the indexed
        section text, not just from heading detection."""
        f = tmp_path / "fenced.md"
        f.write_text(
            "# Title\n\nNormal body text.\n\n```text\nUNIQUE_FENCE_TOKEN_XYZ\n```\n",
            encoding="utf-8",
        )
        asyncio.run(index_paths(service, IndexPathsRequest(paths=[str(f)])))

        # Sanity check: normal body text IS indexed and searchable.
        normal_result = asyncio.run(
            search_docs(service, SearchDocsRequest(query="Normal"))
        )
        assert "Title" in normal_result

        # The unique fence-only token must NOT be found.
        fence_result = asyncio.run(
            search_docs(service, SearchDocsRequest(query="UNIQUE_FENCE_TOKEN_XYZ"))
        )
        assert "UNIQUE_FENCE_TOKEN_XYZ" not in fence_result
        assert "No results found" in fence_result  # or equivalent zero-match assertion
```

### Details

- Include a "sanity check" assertion (searching for ordinary body text and confirming it IS
  found) alongside the negative assertion — this guards against the test passing for the wrong
  reason (e.g. if indexing silently failed entirely, both the positive and negative search would
  return no results, and only the negative assertion would look like it "passed"). This mirrors
  the existing `test_returns_results_after_indexing` pattern (line 548) for the positive half.
- Use a token unlikely to collide with FTS5 tokenization quirks — an uppercase
  underscore-joined identifier (`UNIQUE_FENCE_TOKEN_XYZ`) avoids being split into common English
  words that might partially match other indexed content in the same test's temp DB.
- Assert on the exact "no results" wording already used by `test_returns_no_results_for_empty_
  query` (line 557-560, `"No results found" in result`) for consistency with the existing
  empty-result-string convention in this codebase, rather than inventing a new assertion style.
- Place the test in `TestSearchDocs` (not `TestParseMarkdown`) because the claim under test is
  about `search_docs`'s end-to-end output (parsing + indexing + FTS5 search), not merely
  `parse_markdown`'s section-list output — `TestParseMarkdown.test_code_fence_hash_not_treated_
  as_heading` already covers the narrower parser-level heading claim; this test covers the wider
  content-searchability claim that the plan's Assumption 5 identifies as the actual doc gap.
- Do not modify `test_code_fence_hash_not_treated_as_heading` — it remains valid and unmodified;
  this is a purely additive test.
