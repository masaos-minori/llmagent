# Implementation: Unify partial-completion persistence path in orchestrator.py

Steps covered: Plan 20260626-091911 — Steps 1-1, 2-1

---

## Goal

Ensure that when a workflow task partially completes (some steps succeed, then a failure occurs), the completed steps are persisted via `DiagnosticStore._handle_partial_completion()` canonical path — not via ad-hoc in-line state updates scattered through the orchestrator.

---

## Scope

- **In scope**: `scripts/agent/orchestrator.py` — locate partial-completion handling and unify to canonical path
- **Out of scope**: session.py (step 1-2); tests (step 2-2); docs (step 3-1)

---

## Assumptions

- A `_handle_partial_completion(task_id, completed_steps, error)` method exists or needs to be added in `DiagnosticStore` or `StateStore`.
- The orchestrator currently has one or more locations where partial completion state is written ad hoc.
- The canonical path means: one call to `_handle_partial_completion()` with all relevant context, rather than scattered writes.

---

## Implementation

### Target file
`scripts/agent/orchestrator.py`

### Procedure
1. Read `scripts/agent/orchestrator.py` — search for partial completion handling (keywords: `partial`, `completed_steps`, `DiagnosticStore`, error recovery path).
2. Step 1-1 (Audit): List all locations where partial-completion state is written.
3. Step 2-1 (Unify): Replace scattered writes with a single call:
   ```python
   await ctx.diagnostic_store._handle_partial_completion(
       task_id=task_id,
       completed_steps=completed_steps,
       error=exc,
   )
   ```
   This call should be made in the `except` block of the main workflow execution loop.
4. Remove any now-redundant inline state writes.

### Method
Refactor within orchestrator.py. The `_handle_partial_completion` interface must be stable before this is implemented.
