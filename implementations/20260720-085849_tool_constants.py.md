   reality, not the plan's Design §1 draft wording.** The plan's Design §1 draft text says
   "Routing authority: shared.tool_registry.ToolRegistry ... Live /v1/tools discovery ... is
   drift-validation only." Verified by direct read, `tool_registry.py`'s *current* docstring
   (lines 4-8) now says the opposite emphasis: `ToolRegistry` is explicitly "not the primary
   runtime routing authority once RuntimeToolRegistry (shared/runtime_tool_registry.py) supplies
   LLM-visible tool schemas via live /v1/tools discovery (requirement 04)" and, post-migration,
   serves as "(a) drift-detection input for McpToolDiscoveryService ... and (b) backward-compatible
   routing fallback." `route_resolver.py`'s current docstring (lines 4-10) is consistent with this:
   "Routing priority: 1. Tool registry (canonical source of truth ...)" plus "Live /v1/tools
   discovery is used for startup validation only, not routing." The new `tool_constants.py`
   comment must match *this current* framing (registry as canonical/fallback source; live
   discovery as validation-only), not the plan's now-superseded phrasing naming `ToolRegistry` as
   unconditional routing authority.
2. `McpToolDiscoveryService` (verified: `scripts/agent/services/mcp_tool_discovery.py:75`) is the
   live-discovery consumer referenced by `tool_registry.py`'s docstring; the new comment may name
