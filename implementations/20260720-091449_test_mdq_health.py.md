# Implementation: tests/test_mdq_health.py (new — `allowed_dirs_count`/`deny_all` health tests)

Source plan: `plans/20260719-205532_plan.md`, Implementation steps Phase 2, item 3.

## Goal

Create a new test file covering the two new `/health` response fields added in the companion
`health_check.py` implementation doc: `allowed_dirs_count` and `deny_all`, for both empty and
non-empty `allowed_dirs` configurations.

## Scope

**In scope:**
- New file `tests/test_mdq_health.py`.
- Tests exercising `GET /health` (via the FastAPI app, same transport approach as the existing
  `tests/test_mdq_health_endpoint.py`) asserting `body["details"]["allowed_dirs_count"]` and
  `body["details"]["deny_all"]` for: (a) `allowed_dirs = []` (deny-all), (b) `allowed_dirs`
  with one or more entries.

**Out of scope:**
- Duplicating `tests/test_mdq_health_endpoint.py`'s existing coverage (schema/trigger/FTS5
  checks, `ready`/`status` fields, stale-document counts) — this file is a narrow addition
  for the two new fields only, per the plan's note that `test_mdq_health_endpoint.py` "already
  exists and covers other health behavior; this is an additional file per the requirement, not
  a duplicate."
- `MdqService.__init__`'s startup warning log — that is covered indirectly by the companion
  `tests/test_mdq_config.py` doc / `mdq_service.py` doc validation, not by an HTTP `/health`
  test.

## Assumptions

1. `tests/test_mdq_health_endpoint.py` (confirmed by direct read) establishes the working
   test pattern for this server: `_create_test_db(tmp_path)` builds a minimal schema-valid
   SQLite DB (documents/chunks/chunks_fts + trigger stubs), `_mock_config(db_path)` returns
   `{"db_path": db_path}`, and `patch("mcp_servers.mdq.health_check.ConfigLoader")` with
   `MockConfig.return_value.load.return_value = <config dict>` mocks config loading; requests
   are made via `httpx.AsyncClient(transport=ASGITransport(app=app), ...)` against
   `from mcp_servers.mdq.mdq_server import app` and `GET /health`.
2. This new file reuses the same `_create_test_db()`-equivalent helper (either import it from
   `tests/test_mdq_health_endpoint.py` if it is designed to be shared, or duplicate a minimal
   local copy if that module does not export it as a public/importable helper — check at
   implementation time whether it is prefixed `_` (module-private) and decide accordingly;
   duplicating a small ~20-line schema-creation helper is acceptable to avoid cross-test-file
   coupling, matching this codebase's general preference for test isolation).
3. The mock config dict passed to `MockConfig.return_value.load.return_value` needs an
   `"allowed_dirs"` key added on top of the existing `"db_path"` key (`_mock_config`'s current
   shape is `{"db_path": db_path}` only) — extend or parametrize this per test case.
4. All test functions are `async def` (matching the `pytest.mark.asyncio` / anyio pattern
   already used by `tests/test_mdq_health_endpoint.py`'s `class TestHealthEndpointReady`) —
   confirm the exact async marker convention (pytest-asyncio mode, or anyio) used in that file
   before writing this one, and match it exactly.

## Implementation

### Target file

`tests/test_mdq_health.py`

### Procedure

1. Create the file with a module docstring: "New `/health` fields: `allowed_dirs_count` and
   `deny_all` (mdq-mcp deny-all visibility)."
2. Reuse or locally reimplement a schema-valid test DB builder (see Assumption 2).
3. Add a local config-dict helper, e.g.:
   ```
   def _mock_config_with_allowed_dirs(db_path: str, allowed_dirs: list[str]) -> dict:
       return {"db_path": db_path, "allowed_dirs": allowed_dirs}
   ```
4. Add test cases (pseudocode):
   ```
   class TestHealthAllowedDirsVisibility:
       async def test_deny_all_reports_zero_count_and_true(self, tmp_path): ...
           # allowed_dirs=[] -> details["allowed_dirs_count"] == 0, details["deny_all"] is True

       async def test_non_empty_allowed_dirs_reports_count_and_false(self, tmp_path): ...
           # allowed_dirs=["/some/dir", "/other/dir"] ->
           # details["allowed_dirs_count"] == 2, details["deny_all"] is False

       async def test_missing_allowed_dirs_key_treated_as_deny_all(self, tmp_path): ...
           # config dict has no "allowed_dirs" key at all (matches production behavior when
           # the TOML key is absent) -> details["allowed_dirs_count"] == 0, deny_all is True

       async def test_raw_allowed_dirs_paths_not_exposed(self, tmp_path): ...
           # allowed_dirs=["/secret/path"] -> assert the literal string "/secret/path" does
           # NOT appear anywhere in the JSON response body (security: only count/bool exposed)
   ```
5. Each test follows the exact `patch(...)` + `AsyncClient(...)` + `client.get("/health")`
   shape already used in `tests/test_mdq_health_endpoint.py`.

### Method

New pytest file mirroring the existing `tests/test_mdq_health_endpoint.py` fixture/mocking
pattern exactly, scoped narrowly to the two new response fields plus one explicit
non-leakage assertion.

### Details

- `test_raw_allowed_dirs_paths_not_exposed` is the direct verification of the companion
  `health_check.py` doc's "do not add the raw `allowed_dirs` paths themselves to `details`"
  constraint — use a distinctive path string (e.g. `/secret/path`) unlikely to appear
  incidentally elsewhere in the response, and assert its absence from the full serialized
  JSON body, not just from the `details` sub-dict (defense in depth against a future
  regression that leaks it elsewhere in the response).
- Keep DB-schema setup minimal (do not add unrelated columns/tables) — this file only needs
  enough schema for `check_health()` to reach the `details` assignments and return
  `status: "ok"`; it does not need to test degraded-schema paths (already covered by
  `tests/test_mdq_health_endpoint.py`).
