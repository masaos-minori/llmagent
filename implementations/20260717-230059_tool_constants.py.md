# Implementation procedure: `scripts/shared/tool_constants.py` (docstring compatibility relabeling)

Source plan: `plans/20260717-130629_plan.md` (requirement `requires/done/20260717_10_require.md`),
Implementation step 3.

## Goal

Update `scripts/shared/tool_constants.py`'s module docstring to explicitly state that its 3 named
frozenset consumers (registry seed data, side-effect detection, risk/policy classification) treat
these frozensets as fallback/compatibility data once requirements 04/06/07 make `RuntimeToolRegistry`
the primary source for each, rather than as the first-checked source.

## Scope

**In scope**
- The module docstring only (`scripts/shared/tool_constants.py` lines 1-14).
- Correcting a pre-existing, independent filename inaccuracy in the consumer list discovered during
  investigation (see Assumption 2), since the docstring is being touched anyway.

**Out of scope**
- Deleting any frozenset or changing consumer logic (requirements 06/07's job, not landed yet — see
  Assumption 1).
- Requirement 04's routing migration.

## Assumptions

1. Per `implementations/20260717-225949_requirements_04_09_landing_check.md`, requirements 06 and 07
   are only **partially landed** (enum/test scaffolding only — `implementations/20260717-220404_tool_enums.py.md`
   adds `OperationType.UNKNOWN`; `implementations/20260717-220433_test_tool_policy.py.md` and
   `implementations/20260717-220527_test_tool_approval_risk.py.md` are test-side additions). No
   production-code doc yet migrates `is_side_effect()`/`classify_operation_type()` to consult
   `RuntimeToolRegistry`. Confirmed by direct read: both functions today (`scripts/shared/tool_executor_helpers.py:47-50`,
   `scripts/agent/tool_policy.py:52-60`) reference only the static frozensets, zero
   `RuntimeToolRegistry` mention. This docstring edit is therefore forward-looking end-state text and
   must not be applied to real source until 06/07 land their production-code migration.
2. Current docstring (verbatim, read directly) names `shared/tool_executor.py` as the
   `is_side_effect()` consumer:
   ```
   These sets serve two purposes:
     shared/tool_registry.py   — registry seed data (auto-populates ToolRegistry at import time)
     shared/tool_executor.py   — side-effect detection (is_side_effect())
     agent/tool_policy.py      — risk classification and approval logic

   Not a routing fallback source. Routing is driven by live `/v1/tools` discovery and ToolRegistry only.
   ```
   Direct grep confirms `is_side_effect()` actually lives in `scripts/shared/tool_executor_helpers.py:47`,
   not `shared/tool_executor.py` — a pre-existing, independent inaccuracy unrelated to this
   requirement's migration concern, worth fixing in the same edit since the docstring is touched
   anyway (not a scope violation — it's the same consumer-list line being edited for the
   compatibility framing).
3. The existing line "Not a routing fallback source. Routing is driven by live `/v1/tools` discovery
   and ToolRegistry only." is itself now stale post-requirement-04 (it should name
   `RuntimeToolRegistry`, not `ToolRegistry`, as what drives routing) — corrected as part of this same
   edit, consistent with `tool_registry.py`'s own docstring correction
   (`implementations/20260717-230029_tool_registry.py.md`).

## Implementation

### Target file

`scripts/shared/tool_constants.py` — module docstring, lines 1-14.

### Procedure

1. Gate: confirm requirements 06 and 07 have landed their production-code migration (not just enum/test
   scaffolding) before applying this edit to source.
2. Fix the `shared/tool_executor.py` -> `shared/tool_executor_helpers.py` filename in the consumer
   list (independent, low-risk fix, may be done regardless of the gate since it's a pre-existing
   inaccuracy, not new compatibility-framing content).
3. Add fallback/compatibility framing to each of the 3 consumer lines.
4. Correct the "Not a routing fallback source... ToolRegistry only" line to name `RuntimeToolRegistry`.

### Method

Direct text edit of the module docstring string literal only; zero logic/behavior change to the
frozensets themselves; `ruff format`/`ruff check`/`mypy` should pass trivially (comment-only).

### Details

- Consumer list correction: `shared/tool_executor.py` → `shared/tool_executor_helpers.py`.
- Append fallback framing per consumer (paraphrase — exact wording is an implementation-time
  judgment call):
  - `shared/tool_registry.py` — registry seed data; `ToolRegistry` itself is now compatibility/fallback
    only (per `implementations/20260717-230029_tool_registry.py.md`), so this frozenset usage is
    twice-removed from any runtime-authoritative role.
  - `shared/tool_executor_helpers.py`'s `is_side_effect()` — fallback path used only when a tool is
    absent from `RuntimeToolRegistry`'s live-discovered side-effect metadata (requirement 06).
  - `agent/tool_policy.py`'s `classify_operation_type()`/risk classification — fallback path used only
    when a tool is absent from `RuntimeToolRegistry` (requirement 07); `OperationType.UNKNOWN`
    (already landed, `implementations/20260717-220404_tool_enums.py.md`) is the fail-safe result when
    neither source can classify a tool.
- Final line correction: replace "Routing is driven by live `/v1/tools` discovery and ToolRegistry
  only." with wording naming `RuntimeToolRegistry` (populated from live `/v1/tools`, requirement 03)
  as what drives routing, with `ToolRegistry`/these frozensets positioned as fallback/compat only.
