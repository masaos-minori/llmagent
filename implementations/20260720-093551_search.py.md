Source plan: `plans/20260719-211216_plan.md` ("Replace fragile MDQ audit detail parsing with
structured metadata"), Design section ("Structured metadata wrapper" /
"Threading metadata through the tool handlers"), Implementation step 2.

**Note on pre-existing docs for this file:** `search.py` already has two companion implementation
documents from sibling design cycles in the same MDQ batch:
`implementations/20260720-091517_search.py.md` (requirement 26: read-time authorization filter

`implementations/20260720-092134_search.py.md` (requirement 27: effective SQL `LIMIT` +
`SearchResultResult.total` → `matched_count`/`shown_count` rename). Both were read in full.

## Goal

`search_docs()` already computes `original_total`/`matched_count`, `shown`/`shown_count`,
`truncated`, and the query string internally while building the formatted text, then discards
them once the string is returned. Change it to return `(text: str, metadata: SearchDocsMetadata)`
so `mdq_service.py` (and ultimately `mdq_server.py::call_tool()`) can read exact audit fields
without re-parsing the text.

## Scope

**In scope:**
- `search_docs()` (`search.py:27-81`): change return type from `str` to
  `tuple[str, SearchDocsMetadata]`; populate `SearchDocsMetadata` from values already computed
  in the function body (`original_total`/`matched_count`, `shown`, `truncated`, `req.query`) plus
  a `duration_ms` measured around the `asyncio.wait_for(...)` call.
- Import `SearchDocsMetadata` from `mdq_models.py` (companion doc
  `20260720-093550_mdq_models.py.md`).
- Update the one call site in `mdq_service.py::search_docs()` (covered by its own document,
  `20260720-093554_mdq_service.py.md`) to unpack the new tuple.

**Out of scope:**
- `_search_docs_structured()` and `SearchResultResult` — the internal FTS5 row-fetch helper and
  its return type are unchanged by this document; the sibling doc
  `20260720-092134_search.py.md` may rename its `total` field to `matched_count`/`shown_count`,
  and this document's `search_docs()` reads whichever field name is live at implementation time
  (see Details).
- `_build_search_where()` — unrelated, unchanged.
- The read-time authorization filter (`20260720-091517_search.py.md`) — unchanged by this
  document; it only affects which rows appear in `result["results"]` before this document's
  counting logic runs, so it composes transparently (fewer authorized rows just means a smaller
  `result_count`/`shown_count`, no code interaction needed).

## Assumptions

1. Direct read of `search.py:27-81` (current, pre-sibling-cycle-implementation state) confirms:
   `search_docs()` computes `original_total = result["total"]` (line 42), `truncated` (line 57,
   set `True` if `original_total > max_results` or if the char-budget loop breaks early), and
   `shown = len(lines) - 1` (line 71, only reached inside the `if len(result["results"]) > 0:`
   branch). The three early-return branches (`if not result["results"]:` line 38-39; the
   `if truncated:` branch line 73-78; the final non-truncated return line 79; and the
   zero-results-but-nonzero-total path — actually there is no such path since `results` empty
   implies the `No results found` early return) must each build and return a `SearchDocsMetadata`
   alongside their existing string.
2. `req.query` is always a `str` (Pydantic-validated on `SearchDocsRequest`, `search.py:14-19`
   import), so `query_preview = req.query[:80]` is safe without a `None` check, consistent with
   `audit_target.py`'s existing `str(...)[:80]` truncation convention.
3. `duration_ms` for `search_docs()`'s metadata is measured from function entry to the point
   where the formatted text is finalized (wrapping the existing `asyncio.wait_for(...)` call plus
   the truncation logic that follows it) — a local `time.perf_counter()` pair, not reusing
   `mdq_server.py`'s outer per-request timer (that outer timer measures the whole `call_tool()`
   request including dispatch overhead; this one measures search-specific work only, per the
   plan's per-tool metadata fields list which asks for `duration_ms` as a `search_docs`-specific
   field).

## Implementation

### Target file

`scripts/mcp_servers/mdq/search.py`

### Procedure

1. Add `import time` (module currently has no `time` import — confirmed via `grep -n "^import\|^from" search.py`) and `SearchDocsMetadata` to the `from mcp_servers.mdq.mdq_models import (...)` block (`search.py:14-19`).
2. Change `search_docs()`'s signature return annotation from `-> str` to
   `-> tuple[str, SearchDocsMetadata]`.
3. Wrap the function body in a `t0 = time.perf_counter()` at entry; compute
   `duration_ms = (time.perf_counter() - t0) * 1000` immediately before each `return` statement.
4. At each of the four existing `return` points, build a `SearchDocsMetadata` dict alongside the
   existing string and return both as a tuple:
   - No-results early return (line 38-39): `result_count=0, shown_count=0, truncated=False,
     total_count=0`.
   - Truncated return (line 73-78): `result_count=original_total, shown_count=shown,
     truncated=True, total_count=original_total`.
   - Non-truncated, non-empty return (line 79): `result_count=original_total, shown_count=shown,
     truncated=False, total_count=original_total`.
   - Non-truncated, zero-results-after-limit-but-nonzero-original_total edge case (line 81 — this
     is actually only reachable if `result["results"]` became empty after the `max_results`
     slice, which cannot happen since slicing to a positive `max_results` from a nonempty list
     stays nonempty; confirm during implementation whether this branch is dead code and, if so,
     note it rather than silently deleting — out of scope to remove dead branches in this
     documentation-only cycle).
   - `query_preview=req.query[:80]` and `duration_ms=duration_ms` in every branch.

### Method

Single-function signature change (`str` → `tuple[str, SearchDocsMetadata]`) with metadata built
from already-computed local variables at each existing return point — no new control flow, no
new queries, no change to `_search_docs_structured()`.

### Details

- **Interaction with sibling doc `20260720-092134_search.py.md`:** if that document's rename
  (`SearchResultResult.total` → `matched_count`/`shown_count`) lands first or together, this
  document's `original_total` local variable becomes `result["matched_count"]` and the
  post-truncation `shown` local becomes reconciled against `result["shown_count"]` — the
  `SearchDocsMetadata.result_count`/`total_count` fields should read `matched_count` and
  `SearchDocsMetadata.shown_count` should read the final rendered-line count (`shown`, which may
  be smaller than `result["shown_count"]` if the char-budget cuts off first — same distinction
  that sibling doc's Details section already calls out). Implementer should land both documents'
  changes together and use whichever field name (`total` vs. `matched_count`) is present in
  `SearchResultResult` at that time; this document does not fix that field name, only its own new
  `SearchDocsMetadata` output.
- **Interaction with sibling doc `20260720-091517_search.py.md`:** no interaction — that document
  only affects which rows are present in `result["results"]` before this document's branches run;
  no code changes needed here to accommodate it.
- `mdq_service.py::search_docs()`'s existing `[WARNING: Index is being updated...]` text-suffix
  logic (appended after calling this function) is unaffected — it operates on the text component
  of the returned tuple only (see companion `mdq_service.py` doc).
