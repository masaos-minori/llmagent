# Design: Add deny-all detection with lockdown suppression

## Goal
Add `security_lockdown_enabled` config option and improve DENY-ALL warning format in `audit_security_defaults()`.

## Target Files
- `scripts/agent/config_dataclasses.py` — add `security_lockdown_enabled` to AgentConfig
- `scripts/agent/repl_health.py` — improve warning messages, add lockdown check
- `docs/04_mcp_05_security_and_safety_model.md` — document intentional lockdown procedure

## Current State

### config_dataclasses.py (lines 493-509)
AgentConfig has 7 sub-configs + workflow_mode. No security_lockdown field.

### repl_health.py audit_security_defaults() (lines 397-512)
Already checks empty allowlists and categorizes them as fail-closed/fail-open. Warning format:
```
Security: shell.command_allowlist is empty (fail-closed: all shell commands denied)
```
Plan wants:
```
DENY-ALL detected: {setting} is empty. {server} will reject ALL requests from this category. Verify this is intentional or add allowed values to config.
```

### security docs (04_mcp_05_security_and_safety_model.md)
Already documents fail-closed/fail-open behavior and startup audit. Missing: intentional lockdown procedure.

## Implementation Steps

### Step 1: Add security_lockdown_enabled to AgentConfig

File: `scripts/agent/config_dataclasses.py`, line 509 (after `workflow_mode`)

Add:
```python
    security_lockdown_enabled: bool = False
```

This allows operators to suppress deny-all warnings when they intentionally configured empty allowlists.

### Step 2: Improve warning messages in audit_security_defaults()

File: `scripts/agent/repl_health.py`

Change the deny-all warning format from:
```python
msg = "Security: shell.command_allowlist is empty (fail-closed: all shell commands denied)"
```
To:
```python
msg = "DENY-ALL detected: shell.command_allowlist is empty. shell-mcp will reject ALL requests from this category. Verify this is intentional or add allowed values to config."
```

Apply to all deny-all checks (lines 450-476):
- `shell.command_allowlist` → "shell-mcp will reject ALL commands"
- `sqlite.db_allowlist` → "sqlite-mcp will reject ALL queries"
- `git.allowed_repo_paths` → "git-mcp will reject ALL repo access"

For `github.allowed_workflows` (fail-open, not deny-all), keep existing format — it's a different concern.

### Step 3: Add lockdown suppression check

In audit_security_defaults(), wrap the deny-all warnings (lines 450-476) with a lockdown check:

```python
    # Check shell sandbox and command_allowlist
    try:
        shell_cfg = ShellConfig.load()
        if shell_cfg.shell_sandbox_backend == "none":
            msg = "Security: shell_sandbox_backend=none (no sandbox for shell commands)"
            logger.warning(msg)
            warnings.append(msg)
        if not shell_cfg.command_allowlist and not ctx.cfg.security_lockdown_enabled:
            fail_closed_empty.append("shell.command_allowlist")
            msg = "DENY-ALL detected: shell.command_allowlist is empty. shell-mcp will reject ALL commands. Verify this is intentional or add allowed values to config."
            logger.warning(msg)
            warnings.append(msg)
    except Exception:
        pass
```

Repeat for sqlite.db_allowlist (line 461) and git.allowed_repo_paths (line 472).

Fail-open checks (github.allowed_workflows, tool.allowed_tools) are NOT suppressed by lockdown — they indicate potential security risk, not intentional deny-all.

### Step 4: Update security docs

File: `docs/04_mcp_05_security_and_safety_model.md`

Add new section after "Startup audit" (around line 328):

```markdown
### Intentional Lockdown Procedure

To suppress DENY-ALL warnings during startup, set `security_lockdown_enabled = true` in `config/agent.toml`:

```toml
[agent]
security_lockdown_enabled = true
```

This is intended for security lockdown scenarios where empty allowlists are deliberate. When enabled:
- DENY-ALL warnings for fail-closed settings are suppressed
- Fail-open warnings (github.allowed_workflows, tool.allowed_tools) still appear
- The security posture summary line still appears with full detail

Use this only when you have intentionally configured deny-all states for security reasons.
```

## Completion Criteria
- `AgentConfig.security_lockdown_enabled` exists (default False)
- DENY-ALL warnings use new format with server name and action
- Lockdown mode suppresses fail-closed deny-all warnings
- Security docs document intentional lockdown procedure
