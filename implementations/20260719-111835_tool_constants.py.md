# Implementation procedure: `scripts/shared/tool_constants.py` (Browser MCP registration)

Source plan: `plans/20260719-101501_plan.md`, Implementation step 3 (Design/Affected areas).

`tool_constants.py` is a common target filename in this repo's history (e.g.
`implementations/20260717-230059_tool_constants.py.md`,
`implementations/done/20260704-075037_tool_constants_py.md`,
`implementations/done/20260716-131150_tool_constants.py.md`, several others). Checked by
content: every hit found addresses unrelated additions/removals (toolcount cleanup, prior tool
registrations for other servers, etc.) — none adds a `BROWSER_TOOLS` frozenset or registers
`browser_fetch`. Flagged as checked, not a genuine overlap.

## Goal

Add `browser_fetch` to `scripts/shared/tool_constants.py`'s canonical tool-classification
frozensets, following the exact per-server single-tool pattern already used for `SHELL_TOOLS`
and `WEB_SEARCH_TOOLS` (both verified, `tool_constants.py:108-111`), and include it in
`get_all_mcp_tool_names()`'s union (the "routing frozenset" referenced by the plan).

## Scope

**In scope**
- New `BROWSER_TOOLS: frozenset[str] = frozenset({"browser_fetch"})` constant, placed near
  `SHELL_TOOLS`/`WEB_SEARCH_TOOLS` (both single-tool frozensets, `tool_constants.py:108-111`).
- Add `BROWSER_TOOLS` to the `frozenset(...)` union inside `get_all_mcp_tool_names()`
  (`tool_constants.py:155-167`, verified).

**Out of scope**
- No `BROWSER_READ_TOOLS`/`BROWSER_WRITE_TOOLS` split — Browser MCP has exactly one tool and it
  is entirely read-only (per plan scope), so the single-frozenset pattern (`SHELL_TOOLS`,
  `WEB_SEARCH_TOOLS`) applies, not the read/write-split pattern used by `GIT_TOOLS`/
  `GITHUB_TOOLS`/`RAG_TOOLS`/`CICD_TOOLS`/`MDQ_TOOLS` (all of which have both read and write
  tools).
- No change to `READ_TOOLS`, `WRITE_TOOLS`, or `DELETE_TOOLS` (those are the filesystem-server
  frozensets specifically, per their docstring context — `browser_fetch` is not a
  filesystem-read tool and must not be added there; it needs its own dedicated frozenset, same
  as `search_web` is not folded into `READ_TOOLS` either).

## Assumptions

1. Verified directly (`tool_constants.py:108-111`):
   ```python
   # Shell execution tools (shell-mcp)
   SHELL_TOOLS: frozenset[str] = frozenset({"shell_run"})

   # Web search tools (web-search-mcp)
   WEB_SEARCH_TOOLS: frozenset[str] = frozenset({"search_web"})
   ```
   This exact single-line-comment + single-item-frozenset shape is the template for
   `BROWSER_TOOLS`.
2. Verified directly (`tool_constants.py:155-167`, `get_all_mcp_tool_names()`): the function
   returns `frozenset(READ_TOOLS | WRITE_TOOLS | DELETE_TOOLS | RAG_TOOLS | CICD_TOOLS |
   MDQ_TOOLS | GIT_TOOLS | SHELL_TOOLS | WEB_SEARCH_TOOLS | GITHUB_TOOLS)` — this is the
   "routing frozenset" the plan refers to (module docstring, `tool_constants.py:13`: "Not a
   routing fallback source. Routing is driven by live `/v1/tools` discovery and ToolRegistry
   only" — despite the docstring's caveat that this is not the *live* routing source, it is
   still the canonical name-registry used for conflict-checking, `is_side_effect()`, and
   `tool_policy.py`'s classification, so `BROWSER_TOOLS` must be added to this union for those
   consumers to recognize `browser_fetch`).
3. This file has no other place `browser_fetch` needs registering (confirmed by the module's
   own docstring, `tool_constants.py:1-16`: it serves `tool_registry.py` (registry seed data)
   and `tool_executor.py` (`is_side_effect()`) and `tool_policy.py` (risk classification) — all
   three consume the frozensets defined here, not a separate list).

## Implementation

### Target file

`scripts/shared/tool_constants.py`.

### Procedure

1. After the `WEB_SEARCH_TOOLS` definition (`tool_constants.py:110-111`), insert:
   ```python

   # Browser fetch tool (browser-mcp) — read-only page fetch + render-to-text
   BROWSER_TOOLS: frozenset[str] = frozenset({"browser_fetch"})
   ```
2. In `get_all_mcp_tool_names()`, add `BROWSER_TOOLS` to the union (exact insertion point:
   alongside `WEB_SEARCH_TOOLS`, since both are single-tool, non-split frozensets):
   ```python
   return frozenset(
       READ_TOOLS
       | WRITE_TOOLS
       | DELETE_TOOLS
       | RAG_TOOLS
       | CICD_TOOLS
       | MDQ_TOOLS
       | GIT_TOOLS
       | SHELL_TOOLS
       | WEB_SEARCH_TOOLS
       | GITHUB_TOOLS
       | BROWSER_TOOLS,
   )
   ```

### Method

Additive frozenset registration; no existing frozenset is modified.

### Details

No new types — `BROWSER_TOOLS` is a `frozenset[str]` literal, matching every sibling constant's
type annotation exactly.
