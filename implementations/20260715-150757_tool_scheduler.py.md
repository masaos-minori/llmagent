# Implementation Procedure: scripts/agent/tool_scheduler.py

Source plan: `plans/20260715-133548_plan.md`

## Goal

Make the `write_first` scheduled batch execute its tools serially instead of
concurrently, closing a safety gap where multiple write tools without an
explicit `resource_scope` were gathered with `asyncio.gather` (i.e. run
concurrently) inside `build_execution_groups()`.

## Scope

**In scope:**
- `build_execution_groups()` — change the `write_first` `ScheduledBatch`'s
  `serialize_flags` value from `[False]` to `[True]`.

**Out of scope:**
- No change to `serial_barrier` handling (already `serialize_flags=[False]`
  per one-element group, which is correct — a one-element group has nothing
  to serialize against itself).
- No change to the `resource_groups` / `parallel` combined-batch logic.
- No change to `_SerializationEvent`, `_GroupMetadata`, `ScheduledBatch`
  dataclass shapes.
- No change to any docstring beyond the one comment line noted in Details.

## Assumptions

- "Serialize `write_first` groups" means setting
  `ScheduledBatch(groups=[write_first], serialize_flags=[True])`, not
  splitting `write_first` into per-tool single-element batches. This is the
  smaller, more localized change and keeps `write_first` as one scheduled
  batch; the caller (`_run_group_calls()` in `tool_runner.py`) already
  executes a batch's group sequentially when `serialize=True` — verified by
  reading that function.
- This is the *only* line that changes in this file for this plan. The
  existing comment on line 99 ("write_first gets its own sequential batch (no
  scope — conservative)") already describes the *intended* behavior — the bug
  is that the code did not match the comment (`serialize_flags=[False]`
  instead of `[True]`). No other line needs to change to align code with that
  comment.

## Implementation

### Target file

`scripts/agent/tool_scheduler.py`

### Procedure

1. Open `build_execution_groups()`.
2. Locate the `if write_first:` block that appends to `cgr` (currently around
   line 105-106):
   ```python
   if write_first:
       cgr.append(ScheduledBatch(groups=[write_first], serialize_flags=[False]))
   ```
3. Change `serialize_flags=[False]` to `serialize_flags=[True]` on that line
   only.
4. Leave every other line in the function unchanged, including the
   `serial_barrier` batch construction (`serialize_flags=[False]` there is
   correct as-is — a one-element group needs no internal serialization) and
   the `has_concurrent` batch construction below it.

### Method

Single-line value change (`False` → `True`) inside an existing dataclass
constructor call. No new imports, no new branches, no refactor of
`build_execution_groups()`'s control flow (this function is already flagged
grade D / CC=23 by `radon`; keep the diff to this one line to avoid
compounding complexity risk, per the plan's Risks section).

### Details

Before:

```python
    cgr: list[ScheduledBatch] = []
    for tc in serial_barrier:
        cgr.append(ScheduledBatch(groups=[[tc]], serialize_flags=[False]))
    if write_first:
        cgr.append(ScheduledBatch(groups=[write_first], serialize_flags=[False]))
```

After:

```python
    cgr: list[ScheduledBatch] = []
    for tc in serial_barrier:
        cgr.append(ScheduledBatch(groups=[[tc]], serialize_flags=[False]))
    if write_first:
        cgr.append(ScheduledBatch(groups=[write_first], serialize_flags=[True]))
```

Effect: when `_run_group_calls()` (in `tool_runner.py`) processes this batch,
its `if serialize:` branch now applies to the `write_first` group, so the
tools inside it execute one at a time in list order, instead of via
`asyncio.gather`.
