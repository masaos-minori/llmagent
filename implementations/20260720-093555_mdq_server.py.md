# Implementation: scripts/mcp_servers/mdq/mdq_server.py (structured audit-detail block +
mdq-local dispatch-result wrapper + `error_type` audit fix)

Source plan: `plans/20260719-211216_plan.md` ("Replace fragile MDQ audit detail parsing with
structured metadata"), Design sections ("Threading metadata through the tool handlers (resolves
UNK-02)" and "Error-path audit (resolves UNK-04)"), Implementation step 6.

**Note on pre-existing docs for this file:** the only existing doc referencing this file,
`implementations/done/20260620-145231_mdq_server_and_rag_pipeline_server.py.md`, covers an
unrelated cosmetic change (the FastAPI `description=` string in the pre-rename `mdq/server.py`
class definition) and has no overlap with the audit-detail/dispatch-result concern here. This is
the first document targeting the live `mdq_server.py`'s `call_tool()` audit logic.

## Goal

Replace the fragile regex/substring-based audit-detail inference in `call_tool()`
(`mdq_server.py:254-282`, uses `_re.search(r"Indexed:\s*(\d+)", ...)` and `.count("---")`) with
direct reads from the structured metadata each `MdqService` method now returns (companion doc
`20260720-093554_mdq_service.py.md`), so audit fields are always accurate regardless of how the
formatted text happens to be worded. Also switch error-path `_audit_log()` calls to use the
existing `error_type` parameter instead of embedding `error_kind=` into the free-text `detail`
string (resolves plan UNK-04).

## Scope

**In scope:**
- Introduce a small mdq-local `MdqDispatchResult` dataclass (`output: str`, `is_error: bool`,
  `metadata: dict[str, Any]`) that wraps the shared `DispatchResult` plus a metadata dict, without
  touching `mcp_servers/dispatch.py`'s `DispatchResult` class or `dispatch_tool()` signature
  (resolves plan UNK-02; keeps zero blast radius on the 9 other MCP servers sharing
  `dispatch.py`).
- Change each `_handle_*` function (`mdq_server.py:147-186`) that now receives a `(text,
  metadata)` tuple from its `MdqService` call (`search_docs`, `index_paths`, `refresh_index`,
  `grep_docs`) to unpack the tuple, stash `metadata` via a `contextvars.ContextVar`, and return
  only `text` — preserving the existing `Callable[[ToolArgs], Awaitable[str]]` contract that
  `dispatch_tool()` (shared, unmodified) enforces.
- Change `_dispatch_mdq_tool()` (`mdq_server.py:200-202`) to call the shared `dispatch_tool()` as
  today, then read the ContextVar immediately after it returns, and return an `MdqDispatchResult`
  (wrapping the `DispatchResult` plus the retrieved metadata) instead of a bare `DispatchResult`.
- Rewrite `call_tool()`'s audit-detail block (`mdq_server.py:254-282`) to read metadata fields
  directly per tool name, removing `_re.search(...)` and `.count("---")` entirely.
- Switch both `_audit_log()` call sites that currently embed `error_kind=` in `detail`
  (`mdq_server.py:82` inside `_mdq_error_handler`, and `mdq_server.py:246` inside `call_tool()`'s
  exception branch) to pass `error_type=error_kind` and drop it from `detail`.

**Out of scope:**
- `mcp_servers/dispatch.py`'s `DispatchResult`/`dispatch_tool()` — unmodified (plan's explicit
  design tradeoff; shared by 9 other MCP servers).
- `_handle_get_chunk`, `_handle_outline`, `_handle_stats` — these three tools do not gain
  structured metadata per the plan's scope; `get_chunk`'s existing "truncated" detection stays a
  simple text-substring check (`"[Truncated" in r.output`, unchanged) since it needs no new
  metadata type per plan Scope.
- `MdqMCPServer.dispatch()` (`mdq_server.py:327-329`) — this method's own return type
  (`DispatchResult`) is part of the `MCPServer` base class contract used by other servers; it
  already calls `_dispatch_mdq_tool()` and returns its result directly. Since
  `_dispatch_mdq_tool()`'s return type changes to `MdqDispatchResult` in this document, verify
  during implementation whether `MdqMCPServer.dispatch()`'s declared return annotation
  (`-> DispatchResult`) needs updating to `-> MdqDispatchResult` or whether `MdqDispatchResult`
  should structurally subtype/wrap `DispatchResult` compatibly — see Details.

## Assumptions

1. Direct read of `mdq_server.py` (338 lines, current code) confirms the exact fragile-parsing
   code the plan describes, at lines 254-294 (plan's cited "~254-282" is close; actual end of the
   block including the final `_audit_log()` call is line 294):
   - `result_count = r.output.count("---") if r.output else 0` (line 259, `search_docs`)
   - `match_count = r.output.count("---") if r.output else 0` (line 265, `grep_docs`)
   - `_re.search(r"Indexed:\s*(\d+)", r.output)` (lines 269, 274, `index_paths`/`refresh_index`)
   - `_re.search(r"Skipped.*?:\s*(\d+)", r.output)` (line 275, `refresh_index`)
   - `_re.search(r"Deleted from index:\s*(\d+)", r.output)` (line 276, `refresh_index`)
   - `import re as _re` is a local import inside `call_tool()` (line 219) — removed entirely once
     no regex use remains in the function.
2. `_DISPATCH_TABLE`/`_handle_*` functions (`mdq_server.py:147-197`) are defined in this file
   (mdq-local), not in `mcp_servers/dispatch.py` — confirmed by direct read — so changing
   `_dispatch_mdq_tool()`'s return type is a local, contained change per plan UNK-02's resolution.
3. `asyncio`'s single-task-per-request model (FastAPI + Starlette request handling) means a
   `contextvars.ContextVar` set inside a `_handle_*` coroutine and read immediately after
   `await dispatch_tool(...)` returns, within the same `_dispatch_mdq_tool()` call, is safe: each
   incoming request's coroutine chain (`call_tool` → `_dispatch_mdq_tool` → `dispatch_tool` →
   `_handle_*`) runs to completion (or raises) before the next await point yields control back to
   a *different* request's coroutine at the same call depth — confirmed as the plan's own
   recommended approach with an explicit note that a concurrency test
   (`asyncio.gather` of two concurrent calls) should verify no cross-request leakage (see
   Validation plan).
4. `mcp_servers/audit.py::_audit_log()` already accepts an `error_type: str = ""` parameter
   (confirmed by the plan's Assumption 4 and cross-referenced against
   `tests/test_audit_log_format.py`'s existing `error_type` assertions for other servers) — no
   change needed to `_audit_log()` itself, only to what `mdq_server.py` passes into it.

## Implementation

### Target file

`scripts/mcp_servers/mdq/mdq_server.py`

### Procedure

1. **Imports:** add `import contextvars` and `from dataclasses import dataclass` (or reuse the
   `dataclass` pattern already used by `DispatchResult` in `mcp_servers/dispatch.py` for
   consistency); no longer need the function-local `import re as _re` inside `call_tool()` —
   remove it once step 5 lands.
2. **`MdqDispatchResult` + ContextVar** (module level, near `_DISPATCH_TABLE`):
   ```python
   _mdq_metadata_var: contextvars.ContextVar[dict[str, Any]] = contextvars.ContextVar(
       "mdq_metadata", default={}
   )

   @dataclass(frozen=True)
   class MdqDispatchResult:
       """Wraps the shared DispatchResult with mdq-local per-call metadata."""

       output: str
       is_error: bool
       metadata: dict[str, Any]

       @property
       def outcome(self) -> str:
           return "error" if self.is_error else "ok"
   ```
3. **`_handle_*` functions that gain metadata** — for `_handle_search_docs`, `_handle_index_paths`,
   `_handle_refresh_index`, `_handle_grep_docs` (all four, per companion `mdq_service.py` doc):
   ```python
   async def _handle_search_docs(args: ToolArgs) -> str:
       """Dispatch search_docs tool call to the mdq service."""
       text, metadata = await _service.search_docs(SearchDocsRequest(**args))
       _mdq_metadata_var.set(dict(metadata))
       return text
   ```
   (same pattern for the other three, substituting the request type and service method).
   `_handle_get_chunk`, `_handle_outline`, `_handle_stats` are unchanged (still return bare `str`,
   no metadata to stash).
4. **`_dispatch_mdq_tool()`** (`mdq_server.py:200-202`):
   ```python
   async def _dispatch_mdq_tool(name: str, args: ToolArgs) -> MdqDispatchResult:
       """Route a tool call through the shared dispatch mechanism, capturing mdq-local metadata."""
       token = _mdq_metadata_var.set({})
       try:
           result = await dispatch_tool(_DISPATCH_TABLE, name, args)
           metadata = _mdq_metadata_var.get()
       finally:
           _mdq_metadata_var.reset(token)
       return MdqDispatchResult(
           output=result.output, is_error=result.is_error, metadata=metadata
       )
   ```
   Resetting the ContextVar to a fresh `{}` at the start of each call (via `.set({})` before
   dispatch, then `.reset(token)` in `finally`) guards against any stale value leaking into a
   tool that does not itself call `.set(...)` (e.g. `get_chunk`, `outline`, `stats` — their
   `_handle_*` never call `_mdq_metadata_var.set(...)`, so `metadata` stays `{}` for those tools,
   which is correct: `call_tool()`'s per-tool branch for those three tools does not read from
   `metadata` at all).
5. **`call_tool()`'s audit-detail block** (`mdq_server.py:254-294`) — replace entirely:
   ```python
   detail_parts: list[str] = [f"duration_ms={ms:.0f}"]
   if r.is_error:
       detail_parts.append("error_kind=tool_error")
   elif req.name == "search_docs":
       md = r.metadata
       detail_parts.append(f"result_count={md.get('result_count', 0)}")
       detail_parts.append(f"shown_count={md.get('shown_count', 0)}")
       if md.get("truncated"):
           detail_parts.append("truncated=true")
   elif req.name == "get_chunk":
       if r.output and "[Truncated" in r.output:
           detail_parts.append("truncated=true")
   elif req.name == "grep_docs":
       md = r.metadata
       detail_parts.append(f"match_count={md.get('match_count', 0)}")
       if md.get("truncated"):
           detail_parts.append("truncated=true")
   elif req.name == "index_paths":
       md = r.metadata
       detail_parts.append(f"indexed_count={md.get('indexed_count', 0)}")
       detail_parts.append(f"skipped_count={md.get('skipped_count', 0)}")
       detail_parts.append(f"failed_count={md.get('failed_count', 0)}")
   elif req.name == "refresh_index":
       md = r.metadata
       detail_parts.append(f"indexed_count={md.get('indexed_count', 0)}")
       detail_parts.append(f"skipped_count={md.get('skipped_count', 0)}")
       detail_parts.append(f"deleted_count={md.get('deleted_count', 0)}")
       detail_parts.append(f"failed_count={md.get('failed_count', 0)}")
   ```
   `r` here is the `MdqDispatchResult` from `_dispatch_mdq_tool()` (variable name `r` already used
   at line 229: `r = await _dispatch_mdq_tool(req.name, req.args)` — type changes from
   `DispatchResult` to `MdqDispatchResult`, both expose `.output`/`.is_error`/`.outcome`, so
   `_to_call_tool_response(r)` at line 294 needs its input type widened or a small adapter — see
   Details).
6. **`error_type` fix** — two call sites:
   - `_mdq_error_handler()` (line 75-84): change
     `detail=f"error_kind={error_kind}"` → `detail=""`, add `error_type=error_kind` to the
     `_audit_log(...)` call.
   - `call_tool()`'s exception branch (line 239-248): change
     `detail=f"duration_ms={ms:.0f}, error_kind={error_kind}"` →
     `detail=f"duration_ms={ms:.0f}"`, add `error_type=error_kind`.

### Method

A module-level `ContextVar` + a thin mdq-local dataclass wrapping the shared `DispatchResult`,
set by `_handle_*` functions and read once per request by `_dispatch_mdq_tool()` — direct
return-value-style threading preferred by the plan's Risk mitigation over any cross-request
shared mutable state, since the ContextVar is request-task-scoped and reset via `token` in a
`finally` block every call. `call_tool()`'s per-tool `if/elif` chain keeps its existing shape
(same branches, same tool names) but every branch now reads `r.metadata` dict keys instead of
running `_re.search`/`.count("---")` against `r.output`.

### Details

- **`_to_call_tool_response(r)` type compatibility (flagged in Procedure step 5):** the shared
  `_to_call_tool_response()` (`mcp_servers/dispatch.py:36-40`) is typed to accept a
  `DispatchResult` and only reads `.output`/`.is_error`. Since `MdqDispatchResult` is a distinct
  dataclass (not a subclass of `DispatchResult`), passing it directly may fail a strict
  `mypy` check even though it is structurally compatible (duck-typed) at runtime. Resolve by
  either (a) defining `MdqDispatchResult` as a subclass of `DispatchResult` with an added
  `metadata` field (requires `DispatchResult` to not be `frozen`-incompatible with subclassing —
  it is `@dataclass(frozen=True)`, which does support inheritance in Python dataclasses as long as
  the subclass is also frozen), or (b) constructing a plain `DispatchResult(output=r.output,
  is_error=r.is_error)` inline at the `_to_call_tool_response(...)` call site instead of passing
  `r` directly. **Recommended: (b)** — avoids any inheritance coupling to the shared
  `mcp_servers/dispatch.py` module, keeping `MdqDispatchResult` fully independent per plan UNK-02's
  "without touching the shared `DispatchResult` class" goal. Concretely:
  `return _to_call_tool_response(DispatchResult(output=r.output, is_error=r.is_error))`.
- **`MdqMCPServer.dispatch()` (`mdq_server.py:327-329`):** its declared return type
  `-> DispatchResult` no longer matches what `_dispatch_mdq_tool()` returns
  (`MdqDispatchResult`). Two options: change the method's own return annotation to
  `MdqDispatchResult` (if the `MCPServer` base class's `dispatch()` abstract signature permits a
  covariant return — verify `mcp_servers/server.py`'s `MCPServer.dispatch()` signature during
  implementation), or have `MdqMCPServer.dispatch()` construct a plain `DispatchResult` from the
  `MdqDispatchResult`'s `.output`/`.is_error` (same pattern as option (b) above) to preserve the
  base class contract exactly. **Recommended:** construct a plain `DispatchResult` here too,
  keeping `MdqDispatchResult` entirely internal to `mdq_server.py`'s own `call_tool()` path and
  never exposed through the `MCPServer` base class interface.
- `contextvars.ContextVar` default value `{}` is a **mutable default** shared across all
  `ContextVar.get()` calls that never had `.set()` called in their context — since
  `_dispatch_mdq_tool()` always calls `.set({})` (a fresh dict) before dispatch and reads via
  `.get()` after, the shared default is never actually read or mutated in practice; still, avoid
  ever calling `.get()` without a preceding `.set()` in the same task to sidestep any accidental
  shared-default mutation risk (this document's `_dispatch_mdq_tool()` always sets first, so this
  is a design note rather than a required additional guard).
