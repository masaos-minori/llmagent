# Implementation procedure: health.py (web-search-mcp)

Source plan: `plans/20260719-193902_plan.md` (Implementation step 2, Design §1)

## Goal

Add a new module `scripts/mcp_servers/web_search/health.py` that tracks
in-process DuckDuckGo provider health (last success/failure timestamp, last
error type, consecutive failure count) and exposes a degraded/healthy
determination that `web_search_server.py`'s `/health` endpoint can fold into
its response — without any external health-probing or persistence.

## Scope

**In scope:**
- New file `scripts/mcp_servers/web_search/health.py`.
- Module-level singleton state (safe because `MCPServer.run_http()` calls
  `uvicorn.run()` with no `workers=` argument — single-process, confirmed at
  `scripts/mcp_servers/server.py:216` region).
- Functions to record a provider success, record a provider failure (with an
  error-type label), determine degraded status, and produce a details dict
  for `/health`.
- `deploy/deploy.sh` copy-list entry for the new file (per `rules/coding.md`
  "Module addition | update copy list in deploy/deploy.sh").

**Out of scope:**
- Any change to `web_search_server.py` itself (covered by a separate
  `web_search_server.py` implementation doc).
- Live external probing of DuckDuckGo on every `/health` call (explicitly
  excluded by the source requirement).
- Cross-process/shared state (no multi-worker deployment today).

## Assumptions

1. `scripts/mcp_servers/web_search/health.py` does not exist yet (confirmed
   via `ls scripts/mcp_servers/web_search/`: only `__init__.py`,
   `formatters.py`, `search_provider.py`, `web_search_models.py`,
   `web_search_server.py`, `web_search_tools.py` are present).
2. Closest in-repo precedent is `scripts/mcp_servers/mdq/health_check.py`: a
   module-level `check_health() -> JSONResponse` function that builds a
   `deps`/`details` pair and calls
   `mcp_servers.health_response.make_health_response(deps, details)`. The new
   `health.py` reuses the same *shape* (a details dict consumable by
   `make_health_response`) but tracks in-memory provider state instead of
   querying a DB — it does not itself build a `JSONResponse`; that stays the
   caller's (`web_search_server.py`'s) job, per Design §1 of the plan.
3. Consecutive-failure degraded threshold defaults to 3, mirroring
   `MCPServer._record_tool_error`'s `_error_threshold = 3` convention at
   `scripts/mcp_servers/server.py:191-205` (plan's UNK-2 default).
4. `make_health_response(deps, details)` (`scripts/mcp_servers/health_response.py`)
   treats a non-empty `deps` dict as "degraded" (503) and an empty dict as
   "ok" (200) — confirmed by reading its body. `health.py` itself does not
   call `make_health_response`; it only supplies the ingredients
   (`is_degraded()` bool + `health_details()` dict) for the caller to do so.

## Implementation

### Target file

`scripts/mcp_servers/web_search/health.py` (new)

### Procedure

1. Create the file with module docstring following the existing convention
   in this package (see `search_provider.py`'s docstring: module path,
   one-line purpose, "Dependency direction: ..." line, "Import from here: ..."
   line).
2. Define a frozen-free (mutable, since it tracks running state)
   `@dataclasses.dataclass` `ProviderHealth` with fields: `provider: str =
   "duckduckgo"`, `last_success_at: float | None = None`,
   `last_failure_at: float | None = None`, `last_error_type: str = ""`,
   `consecutive_failures: int = 0`.
3. Define a module-level singleton instance, e.g. `_health =
   ProviderHealth()`, following the same singleton pattern already used for
   `_cfg: WebSearchConfig = WebSearchConfig.load()` in both
   `web_search_server.py:38` and `web_search_models.py:65`.
4. Define `DEGRADED_THRESHOLD: int = 3` as a module constant (Assumption 3 /
   plan UNK-2), not a magic number inline.
5. Define `record_success() -> None`: sets `_health.last_success_at =
   time.time()`, resets `_health.consecutive_failures = 0`, clears
   `_health.last_error_type` unchanged (last error type is informational —
   keep it, only the counter resets; document this choice inline).
6. Define `record_failure(error_type: str) -> None`: sets
   `_health.last_failure_at = time.time()`, sets `_health.last_error_type =
   error_type`, increments `_health.consecutive_failures += 1`.
7. Define `is_degraded() -> bool`: returns `_health.consecutive_failures >=
   DEGRADED_THRESHOLD`.
8. Define `health_details() -> dict[str, object]`: returns a plain dict
   (e.g. `{"provider": ..., "last_success_at": ..., "last_failure_at": ...,
   "last_error_type": ..., "consecutive_failures": ..., "degraded":
   is_degraded()}`) suitable for merging into `web_search_server.py`'s
   `/health` `details` dict under a `"provider"` key.
9. Add a module-level `reset()` test helper (or accept that tests construct
   a fresh `ProviderHealth()` and monkeypatch the module singleton) — decide
   during implementation which is less invasive; a `reset()` function is
   simplest for test isolation between test cases in
   `tests/test_web_search_health.py`.

### Method

- Use `time.time()` (wall-clock, matches `mcp_servers/audit.py`'s own
  `time.time()` usage for `ts`), not `time.perf_counter()` (which is only
  meaningful for elapsed-duration measurement, used elsewhere for latency).
- No `orjson`/JSON serialization needed inside this module — `health_details()`
  returns a plain `dict[str, object]`; FastAPI's `JSONResponse` (built by the
  caller via `make_health_response`) handles serialization.
- No I/O, no async — this module is pure in-memory bookkeeping, callable from
  both the `/health` GET handler and `call_tool()`'s try/except/finally.

### Details

- Import surface: `from __future__ import annotations`, `dataclasses`,
  `time`. No dependency on `mcp_servers.web_search.web_search_models` or any
  other package module — keeps `health.py` a leaf module within
  `mcp_servers.web_search` (imported only by `web_search_server.py`, per the
  plan's "Affected areas" table: "Low — imported only by
  `web_search_server.py`").
- Do not import `fastapi` here; `health.py` must stay decoupled from the
  HTTP layer so it can be unit-tested without spinning up FastAPI.
- Document in a code comment (per the plan's Risks table) that in-memory
  state resets on process restart, so `/health` can misleadingly report
  "healthy" immediately after a restart that followed a real outage — this
  is accepted/out-of-scope behavior per the plan, not a bug.
