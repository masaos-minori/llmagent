# Implementation procedure: `scripts/shared/runtime_tool.py` (disabled-tool diagnostics variant)

Source plan: `plans/20260717-175327_plan.md` ("RuntimeToolRegistry — represent disabled MCP tools
without exposing them to the LLM", requirement 17), Implementation step 1.

## Goal

Create `scripts/shared/runtime_tool.py`, a new module defining `RuntimeTool` — a frozen dataclass
that represents one discovered MCP tool's diagnostic/visibility state: `name`, `server_key`,
`config_dependent`, `enabled`, `disabled_reason`, `enabled_for_llm`. This lets a discovered-but-
disabled tool stay visible for diagnostics while guaranteeing it is never sent to the LLM as a
callable tool. This is the data type `RuntimeToolRegistry` (separate implementation doc) stores.

**IMPORTANT — filename collision with an unrelated design (flag, do not merge)**: a prior doc in
this batch, `implementations/20260717-203121_runtime_tool.py.md`, ALSO targets
`scripts/shared/runtime_tool.py` but is sourced from a different plan/requirement
(`plans/done/20260717-124020_plan.md`, requirement 02) and defines a **different, incompatible**
13-field `RuntimeTool` (routing, LLM schema, scheduler metadata, side-effect detection, safety
tier, approval — fields `name, server_key, server_url, description, input_schema, raw_definition,
status, is_write, requires_serial, resource_scope, agent_safety_tier, requires_approval,
enabled_for_llm`). That doc's own `build_runtime_tool()` factory and its extensions
(`implementations/20260718-084710_runtime_tool.py.md`, adding a `capabilities` field) are for a
**different feature entirely** (general tool execution-metadata normalization for routing/policy),
not this requirement's disabled-tool-visibility concept. The two designs cannot both be implemented
against the same physical file path without a real conflict — this is a genuine cross-plan
collision in the batch, not a false-positive filename match to silently reuse. This doc's own
6-field shape below is what THIS plan (requirement 17) requires; whoever integrates both plans into
real source must reconcile or rename one of the two `RuntimeTool` concepts before implementation
(e.g. this doc's type could become `McpToolAvailability` or similar to avoid the clash) — flagged
here, not resolved, since resolving it is outside this documents-only workflow's scope for a single
plan.

## Scope

**In scope**
- New file `scripts/shared/runtime_tool.py` only (per this plan's own numbering; see Goal note on
  the physical-file collision with the other `RuntimeTool` design).
- `RuntimeTool` frozen dataclass, 6 fields (see Details).
- A small helper for computing `enabled_for_llm` from `enabled` plus an optional policy predicate.

**Out of scope**

- Any change to `scripts/shared/tool_registry.py` (`ToolRegistry` stays the compile-time
  ownership/routing source of truth; confirmed unmodified per this plan's Assumptions).
- Reconciling the file-path collision noted above with requirement 02's `RuntimeTool` design.

## Assumptions

- Per `.importlinter`'s `shared-is-leaf` contract (confirmed in `/home/sugimoto/llmagent/.importlinter`:
  `shared` may not import `agent`, `mcp_servers`, `rag`, or `db`), `runtime_tool.py` must not import
  anything from `agent` (e.g. no `agent.tool_policy.RiskLevel`/`OperationType`). The "policy predicate"
  used to compute `enabled_for_llm` is therefore a plain `Callable[[str], bool] | None` passed in by the
  caller (agent-layer code supplies the actual predicate; this module stays agent-agnostic), not an
  imported agent type.
- `disabled_reason` is a plain `str`, empty (`""`) when `enabled=True` — matches this plan's own
  Design step 1 and the requirement's acceptance criterion "Enabled tools show empty disabled reason."
  No `Enum`/`None` union is introduced; empty string is the single "no reason" sentinel.
- `config_dependent` mirrors the same signal sibling requirement 15 defines server-side (per-tool
  `config_dependent`/formerly `requires_config` boolean in each server's `TOOL_LIST`, confirmed still
  named `requires_config` in real source today: `scripts/mcp_servers/file/read_tools.py:54` etc., and
  `scripts/mcp_servers/git/tools.py:30` — e.g. `"requires_config": True` for git tools). This module
  does not compute `config_dependent` itself; it is a plain field populated by the caller
  (`mcp_tool_discovery.py`) from whatever the live response provides.

## Implementation

### Target file

`scripts/shared/runtime_tool.py` (new).

### Procedure

1. Module docstring: state this module defines the tool-visibility/diagnostics shape for
   requirement 17 ("RuntimeToolRegistry — represent disabled MCP tools without exposing them to the
   LLM"); explicitly cross-reference `scripts/shared/tool_registry.py` (ownership/routing authority,
   unchanged) and note this is a *separate* concept from that module's `ToolDefinition`; explicitly
   flag (in a comment, not just this doc) the physical-file collision with the other,
   already-documented 13-field `RuntimeTool` design (`implementations/20260717-203121_runtime_tool.py.md`)
   so a future integrator does not silently merge the two shapes.
2. Define `RuntimeTool` as `@dataclass(frozen=True)` with the 6 fields (see Details).
3. Define a module-level helper `compute_enabled_for_llm(enabled: bool, policy_check: Callable[[str],
   bool] | None, name: str) -> bool` that returns `False` immediately if `enabled` is `False`
   (disabled tools are never LLM-visible, regardless of policy), otherwise returns
   `policy_check(name)` if `policy_check` is provided, else returns `enabled` unchanged.
4. No runtime validation beyond the above; mypy enforces field types statically.

### Method

Plain dataclass + one module-level pure helper function. No `Protocol`/`ABC` — no polymorphism
needed (mirrors the "plain data container" style already used by `scripts/shared/tool_registry.py`'s
`ToolDefinition`).

enabled_for_llm: bool

Construction is via the plain dataclass constructor (`RuntimeTool(name=..., server_key=..., ...,
