# Implementation: Elevate RAG Consistency Check to Normal Operations Diagnostics

## Goal

Surface RAG consistency check results during normal operation, not only as an on-demand maintenance command.

## Scope

**In:**
- `scripts/db/maintenance.py` — add `issues` field to `RagConsistencyReport`; populate from `summarize_issues()` in `check_rag_consistency()`
- `scripts/rag/ingestion/ingester.py` — remove unused `summarize_issues` import; use `report.issues` instead of calling `summarize_issues()` separately
- `scripts/agent/services/ingest_workflow.py` — emit terminal warnings for each consistency issue after ingest
- `scripts/agent/commands/models.py` — add `rag_db_configured: bool = False` to `StatsViewModel`
- `scripts/agent/commands/cmd_config_stats.py` — add `_get_rag_db_configured()` helper; add hint line in `/stats` when RAG DB is configured

**Out:**
- Auto-repair of detected issues (out of scope per requirement)
- Running consistency check on every `/stats` call (too heavy for a frequent command)

## Assumptions

1. `RagConsistencyReport` is a frozen dataclass — adding `issues: tuple[str, ...] = ()` with a default is safe
2. `ingest_workflow.py` has access to `on_status` callback for emitting messages — same pattern as `n_embed_failed` warning
3. `check_rag_consistency()` in `ingester.py:131` already runs after ingestion — the issues need only be propagated up, not generated anew
4. The hint in `/stats` is one line and adds no DB read overhead

## Implementation

### Target file 1: `scripts/db/maintenance.py`

**Procedure:** Add `issues` field to `RagConsistencyReport`; populate it in `check_rag_consistency()`

**Method:**
- Add `issues: tuple[str, ...] = ()` to `RagConsistencyReport` dataclass (after `embed_failed` since it has a default)
- In `check_rag_consistency()`, create the report first, then use `dataclasses.replace(report, issues=tuple(summarize_issues(report)))` to populate issues

**Details:**
- Current `RagConsistencyReport` (lines 80-89): no `issues` field
- Added `import dataclasses` at top of file for `dataclasses.replace()` usage
- The `issues` field is populated automatically — callers no longer need to call `summarize_issues()` separately

### Target file 2: `scripts/rag/ingestion/ingester.py`

**Procedure:** Use `report.issues` directly; remove unused `summarize_issues` import

**Method:**
- In `ingest_all()`, replace the manual `summarize_issues()` call with `report.issues`
- Remove `summarize_issues` from the import since it's no longer needed

**Details:**
- Current code (lines 131-136): calls `summarize_issues(report)` and iterates over issues separately
- Changed to: iterate over `report.issues` directly — the issues are now part of the report

### Target file 3: `scripts/agent/services/ingest_workflow.py`

**Procedure:** Emit warnings for consistency issues from the report

**Method:**
- After `report = await loop.run_in_executor(None, ingester.ingest_all)` (line 190), check if report is not None and has issues
- Emit warnings for each consistency issue via `on_status`

**Details:**
- Current code (lines 190-201): only checks `embed_failed` from the report
- Added: after line 200, check `report.issues` and emit per-issue warnings

### Target file 4: `scripts/agent/commands/models.py`

**Procedure:** Add `rag_db_configured` field to `StatsViewModel`

**Method:**
- Add `rag_db_configured: bool = False` as the last field (has default, so order doesn't matter)

**Details:**
- Current `StatsViewModel` (lines 12-34): no RAG DB configuration indicator

### Target file 5: `scripts/agent/commands/cmd_config_stats.py`

**Procedure:** Add `_get_rag_db_configured()` helper; add hint line in `/stats`

**Method:**
- Add `_get_rag_db_configured(ctx)` function that tries to build the DB config and returns True if successful
- In `_collect_stats()`, pass `rag_db_configured=_get_rag_db_configured(ctx)` to `StatsViewModel`
- In `_cmd_stats()`, after the approval line, add hint: `Run /db rag consistency for index integrity status`

**Details:**
- Current code: no RAG consistency hint in `/stats` output
- Added: try/except around `build_db_config()` — if it raises ValueError/RuntimeError, RAG DB is not configured
