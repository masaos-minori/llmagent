# Implementation procedure: `scripts/mcp_servers/web_search/health.py` (per-tool keying, UNK-03)

Source plan: `plans/20260720-135137_plan.md`, Implementation step 4 (health/metrics half); resolves
UNK-03 per the plan's recommended default (two separate module-level singletons, not a generic
`dict[str, ProviderHealth]`).

Prior `health.py`-named docs under `implementations/done/` are all for unrelated modules
(`repl_health.py`, `agent_repl_health.py`, `mcp_health.py`, `test_mdq_health.py`, etc. ‚Äî different
subsystems). The one `web_search`-specific hit,
`implementations/done/20260720-081456_health.py.md` (+ its test counterpart
`20260720-081716_test_web_search_health.py.md`), belongs to today's earlier requirement batch ‚Äî read
in full, targets `DEGRADED_THRESHOLD`/error-type wiring for the DuckDuckGo provider only, no mention
of a second tool or `browser`. No overlap. New document.

## Goal

Extend `health.py` so `browser_fetch` failures/successes are tracked independently of
`search_web`'s DuckDuckGo-provider health state, preventing a `browser_fetch` outage from
spuriously marking `search_web` as degraded (plan's Risk #1).

## Scope

**In scope**: add a second `ProviderHealth` singleton (`_browser_health`) and its own
`record_browser_success()`/`record_browser_failure()`/`is_browser_degraded()`/
`browser_health_details()` public functions, in `scripts/mcp_servers/web_search/health.py` (current:
86 lines).
**Out of scope**: `ProviderHealth` dataclass shape, `DEGRADED_THRESHOLD`, and the existing
`record_success`/`record_failure`/`is_degraded`/`health_details`/`reset` functions for the
`_health` (DuckDuckGo) singleton ‚Äî untouched, external contract stable (per plan's Affected-areas
note "keep `is_degraded()`/`health_details()` external contract stable").

## Assumptions

1. Prefer **two separate singletons** over a `dict[str, ProviderHealth]` (plan's own "implementer's
   choice, pick the simpler of the two unless arbitrary future tool counts are needed" ‚Äî this
   package will realistically only ever have these two tools, per the plan's explicit out-of-scope
   note "no new tools beyond browser_fetch"). This keeps every existing call site
   (`record_success()`, `is_degraded()`, `health_details()`) byte-for-byte unchanged ‚Äî zero risk of
   an accidental signature change breaking `search_web`'s existing behavior.
2. `_browser_health = ProviderHealth(provider="browser_fetch")` reuses the existing
   `ProviderHealth` dataclass as-is (it already accepts an arbitrary `provider: str` field) ‚Äî no new
   dataclass needed.
3. `reset()` (currently a test helper resetting `_health` only) needs a `reset_browser()` sibling,
   or `reset()` itself could reset both singletons ‚Äî prefer a separate `reset_browser()` to let
   tests reset one without disturbing the other's recorded state (matches the "independent tools"
   spirit of this whole doc).

## Implementation

### Target file

`scripts/mcp_servers/web_search/health.py`

### Procedure

1. Add `_browser_health = ProviderHealth(provider="browser_fetch")` alongside the existing
   `_health = ProviderHealth()` (line 45).
2. Add `record_browser_success() -> None` / `record_browser_failure(error_type: str) -> None`,
   bodies identical to the existing `record_success`/`record_failure` but operating on
   `_browser_health` instead of `_health`.
3. Add `is_browser_degraded() -> bool` ‚Äî identical to `is_degraded()` but reading
   `_browser_health.consecutive_failures`.
4. Add `browser_health_details() -> dict[str, object]` ‚Äî identical shape to `health_details()` but
   reading `_browser_health`'s fields.
5. Add `reset_browser() -> None` (test helper, mirrors `reset()`'s `global`/reassignment pattern).
6. Update the module docstring's "Dependency direction" line only if needed (still stdlib-only, no
   new imports) ‚Äî otherwise leave as-is.

### Method

Straight duplication of the existing 4-function pattern against a second singleton ‚Äî no shared
helper/generalization introduced (plan explicitly favors "two separate singletons" simplicity over
a parameterized `dict`-keyed API, avoiding speculative generality per `rules/coding.md`'s "No
speculative over-engineering").

### Details

- Pseudocode for the new block (placed directly after the existing `_health`-based functions,
  before `reset()`):
  ```
  _browser_health = ProviderHealth(provider="browser_fetch")

  def record_browser_success() -> None: ...   # mirrors record_success(), targets _browser_health
  def record_browser_failure(error_type: str) -> None: ...  # mirrors record_failure()
  def is_browser_degraded() -> bool: ...       # mirrors is_degraded()
  def browser_health_details() -> dict[str, object]: ...  # mirrors health_details()
  def reset_browser() -> None: ...             # mirrors reset()
  ```
- `web_search_server.py`'s `/health` endpoint (separate doc) calls both `health.health_details()`
  and `health.browser_health_details()`, and folds `health.is_degraded() or
  health.is_browser_degraded()` into its `deps` degradation dict ‚Äî two independent entries, not one
  merged boolean, so an operator can tell which tool is unhealthy.
