# Implementation: scripts/mcp_servers/mdq/health_check.py (add `allowed_dirs_count`/`deny_all`
to `/health` details)

Source plan: `plans/20260719-205532_plan.md`, Design §2 ("Deny-all visibility"),
Implementation steps Phase 2, item 1.

## Goal

Make `check_health()`'s response surface whether MDQ is currently in deny-all state
(`allowed_dirs = []`) without exposing the actual allowed-directory paths, so operators can see
deny-all state directly from `/health` instead of having to inspect the TOML config file.

## Scope

**In scope:**
- `check_health()` (`health_check.py:41-117`): after loading `mdq_cfg`
  (line 47), read `allowed_dirs` from it and add two new keys to the existing `details` dict:
  `allowed_dirs_count` (int) and `deny_all` (bool).

**Out of scope:**
- Adding the raw `allowed_dirs` path list itself to `details` — explicitly forbidden per the
  plan's Design §2 ("Do not add the raw `allowed_dirs` paths themselves to `details`")
  matching existing security policy documented in
  `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md`.
- Instantiating `MdqService` inside `health_check.py` — the plan's Assumption 6 explicitly
  chooses reading `allowed_dirs` directly from `ConfigLoader`, matching this file's existing
  pattern for `db_path` (`health_check.py:48`), to avoid adding a service-object dependency to
  the standalone health function.
- `_check_stale_documents()` (`health_check.py:30-38`) and the degraded-response early-return
  branches (`health_check.py:51-53, 62-64, 66-68, 69-71, 76-81, 88-90`) — unchanged.
- The startup warning log for empty `allowed_dirs` — that lives in `MdqService.__init__`
  (companion `mdq_service.py` doc), not here; this file only affects the on-demand `/health`
  response, not startup-time logging.

## Assumptions

1. `mdq_cfg` (`health_check.py:47`, `ConfigLoader().load("mdq_mcp_server.toml")`) is a plain
   dict; `mdq_cfg.get("allowed_dirs")` may return `None` if the key is absent, matching the
   existing `mdq_cfg.get("db_path") or "/opt/llm/db/mdq.sqlite"` pattern immediately below it
   (line 48) — the new code must apply the same `or []` fallback style for consistency:
   `allowed_dirs = mdq_cfg.get("allowed_dirs") or []`.
2. This computation must happen unconditionally, right after `mdq_cfg` is loaded
   (inside the existing `try` block starting at line 46), so it is present in `details` on
   both the healthy and degraded response paths (all of the function's early `return
   _degraded_response(deps, details)` calls happen after line 47, so placing the new
   assignment immediately after line 49 — the existing `details["database"] = db_path` line —
   ensures `allowed_dirs_count`/`deny_all` appear in every returned response, not only the
   fully-healthy one).
3. `deny_all` is computed the same way `MdqService`'s deny-all concept works
   (`scripts/mcp_servers/mdq/auth.py:20-21`: `if not allowed_dirs: return False`) — i.e.
   `deny_all = len(allowed_dirs) == 0`, kept as a plain boolean, no further nuance.

## Implementation

### Target file

`scripts/mcp_servers/mdq/health_check.py`

### Procedure

1. Locate lines 47-49:
   ```python
   mdq_cfg = ConfigLoader().load("mdq_mcp_server.toml")
   db_path = mdq_cfg.get("db_path") or "/opt/llm/db/mdq.sqlite"
   details["database"] = db_path
   ```
2. Immediately after `details["database"] = db_path` (line 49), insert:
   ```python
   allowed_dirs = mdq_cfg.get("allowed_dirs") or []
   details["allowed_dirs_count"] = len(allowed_dirs)
   details["deny_all"] = len(allowed_dirs) == 0
   ```
3. Do not modify any of the subsequent `if not Path(db_path).is_file(): ...` degraded-return
   logic (lines 51-53) or anything after it — the new `details` keys are already set before
   this point, so they naturally propagate through every return path in the function,
   including the early degraded ones.

### Method

Two-line additive insertion immediately following an existing, structurally identical
`details[...]` assignment (`details["database"] = db_path`) — matches the file's established
pattern of reading one config value and mirroring it into `details` right after load, with no
control-flow changes.

### Details

- Do not read `allowed_dirs` from anywhere except `mdq_cfg` (the already-loaded config dict)
  — no new `ConfigLoader()` instantiation, no new file I/O.
- The existing outer `except (FileNotFoundError, PermissionError, KeyError, TypeError) as e:`
  handler (lines 113-114) already covers the case where `ConfigLoader().load(...)` itself
  fails; in that failure case `details["allowed_dirs_count"]`/`details["deny_all"]` are simply
  never set (matching how `details["database"]` is also never set in that same failure path
  today) — no special-casing needed for this new code.
