# Implementation procedure: metrics.py (web-search-mcp)

Source plan: `plans/20260719-193902_plan.md` (Implementation step 3, Design §2)

## Goal

Add a new module `scripts/mcp_servers/web_search/metrics.py` providing
lightweight, in-process counters for web-search-mcp: total/succeeded/failed
query counts, average latency, and last success/failure timestamps plus last
error type — with the API shape making it structurally impossible to record
the full query text (no query-string parameter anywhere in the public
surface).

## Scope

**In scope:**
- New file `scripts/mcp_servers/web_search/metrics.py`.
- Module-level singleton counters (single-process, same justification as
  `health.py`: `uvicorn.run()` has no `workers=` argument).
- A `record_query(success, latency_ms, error_type="")` entry point and a
  `snapshot()` read accessor.
- `deploy/deploy.sh` copy-list entry for the new file.

**Out of scope:**
- Any change to `web_search_server.py` itself (separate implementation doc).
- Persistence across restarts (explicitly out of scope per the plan).
- Prometheus/OpenTelemetry/external metrics backend (explicitly out of scope).
- Per-query text or per-query detail storage of any kind (must remain
  impossible to violate "by construction, not just by convention", per the
  plan's Design §2).

## Assumptions

1. `scripts/mcp_servers/web_search/metrics.py` does not exist yet (confirmed
   via `ls scripts/mcp_servers/web_search/`).
2. `queries_total`/`queries_succeeded`/`queries_failed` are simple integer
   counters; average latency is derived (`_latency_sum_ms / queries_total`,
   guarded against division by zero) rather than stored as a running average,
   for simplicity and testability.
3. Latency is measured by the caller (`web_search_server.py::call_tool()`)
   around `_dispatch_web_tool(...)`, using the same `time.perf_counter()`
   idiom already used in `formatters.py::search_web()` (`t0 =
   time.perf_counter()` ... `ms = (time.perf_counter() - t0) * 1000`) — this
   module only receives the already-computed `latency_ms` float, it does not
   measure time itself.
4. No new dependency required; stdlib `dataclasses`/`time` only.

## Implementation

### Target file

`scripts/mcp_servers/web_search/metrics.py` (new)

### Procedure

1. Create the file with the same module-docstring convention as
   `search_provider.py`/`health.py` (module path, purpose, dependency
   direction line, import-from-here line).
2. Define `@dataclasses.dataclass class WebSearchMetrics` with fields:
   `queries_total: int = 0`, `queries_succeeded: int = 0`,
   `queries_failed: int = 0`, `_latency_sum_ms: float = 0.0`,
   `last_success_at: float | None = None`, `last_failure_at: float | None =
   None`, `last_error_type: str = ""`.
3. Add a `@property average_latency_ms(self) -> float` on the dataclass:
   returns `0.0` when `queries_total == 0`, otherwise
   `self._latency_sum_ms / self.queries_total`.
4. Define a module-level singleton `_metrics = WebSearchMetrics()`.
5. Define `record_query(success: bool, latency_ms: float, error_type: str =
   "") -> None`:
   - increments `_metrics.queries_total`
   - adds `latency_ms` to `_metrics._latency_sum_ms`
   - if `success`: increments `queries_succeeded`, sets
     `last_success_at = time.time()`
   - else: increments `queries_failed`, sets `last_failure_at = time.time()`,
     sets `last_error_type = error_type`
   - **Signature never accepts a query string parameter** — this is the
     load-bearing part of Design §2's "impossible to violate by construction"
     requirement; do not add one later without re-reviewing this constraint.
6. Define `snapshot() -> dict[str, object]`: returns a plain dict of all
   public fields plus `average_latency_ms`, e.g. `{"queries_total": ...,
   "queries_succeeded": ..., "queries_failed": ..., "average_latency_ms":
   ..., "last_success_at": ..., "last_failure_at": ..., "last_error_type":
   ...}`. Never includes anything derived from a query string (there is none
   in scope to include).
7. Add a `reset()` test helper analogous to `health.py`'s, for test isolation
   in `tests/test_web_search_metrics.py`.

### Method

- Pure in-memory bookkeeping; no I/O, no async, no FastAPI import (same
  decoupling rationale as `health.py` — must be unit-testable standalone).
- Use `time.time()` for `last_success_at`/`last_failure_at` (wall-clock
  timestamps for display), consistent with `health.py`'s choice and
  `mcp_servers/audit.py`'s `ts` field.

### Details

- Import surface: `from __future__ import annotations`, `dataclasses`,
  `time`. No dependency on other `mcp_servers.web_search.*` modules — a leaf
  module, imported only by `web_search_server.py` (and possibly
  `formatters.py` if latency ends up measured there instead — plan's Design
  §5 recommends measuring in `web_search_server.py` directly instead, keeping
  `formatters.py` unchanged; confirm no import from `formatters.py` is
  actually needed once `web_search_server.py`'s refactor lands).
- Test-writer note for `tests/test_web_search_metrics.py`: assert that
  `snapshot()`'s dict values/keys never contain the literal query string used
  in the test call — this is the plan's explicit acceptance check for
  Design §2 ("Metrics do not expose full query text").
