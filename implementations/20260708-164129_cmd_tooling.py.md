# Implementation: H-7 — Replace ctx.tool_result_store with local ToolResultStore() in cmd_tooling.py

## Goal

Keep `/tool list` and `/tool show <id>` fully functional after `AgentContext` stops owning a
`tool_result_store` field, by having `_ToolingMixin` instantiate `ToolResultStore()` locally
instead of reading it off `ctx`.

## Scope

**Target**: `scripts/agent/commands/cmd_tooling.py`

**Depends on**: land together with `implementations/20260708-163935_context.py.md` (field
removal).

**Out of scope**: `db/tool_results.py` (`ToolResultStore` class itself, including its
`list_recent`/`get` methods — unchanged), `_cmd_plan()` and other unrelated mixin methods.

## Assumptions

1. `ToolResultStore()` has a zero-argument constructor (confirmed:
   `db/tool_results.py`'s `ToolResultStore` class has no `__init__` override, and its
   `_make_helper()` method internally opens `SQLiteHelper("session")` on each call — the class is
   stateless per-call, matching the plan's U-2 resolution: "インスタンスはメソッドローカルで生成
   (stateless なので問題なし)").
2. `agent` → `db` imports are permitted by the import-linter layer contract (`db` has no
   restriction against being imported by `agent`; only the reverse direction — `db` importing
   `agent` — is forbidden). Confirmed by the existing contract list
   (`db must not import from agent, mcp, or rag`).
3. `db.models.ToolResultRow` is already imported in this file (line 15) — no new import needed
   for that type.

## Implementation

### Target file

`scripts/agent/commands/cmd_tooling.py`

### Procedure

#### Step 1: Add the `ToolResultStore` import

Add to the existing import block (near line 15, alongside `from db.models import ToolResultRow`):

```python
from db.models import ToolResultRow
from db.tool_results import ToolResultStore
```

#### Step 2: Update `_tool_list()` to instantiate locally

Current (line 58-61):

```python
    def _tool_list(self) -> None:
        """Print stored tool results for the current session."""
        ctx = self._ctx
        entries = ctx.tool_result_store.list_recent(ctx.session.session_id)
```

Replace with:

```python
    def _tool_list(self) -> None:
        """Print stored tool results for the current session."""
        ctx = self._ctx
        entries = ToolResultStore().list_recent(ctx.session.session_id)
```

#### Step 3: Update `_tool_show()` to instantiate locally

Current (line 85):

```python
        raw = self._ctx.tool_result_store.get(tool_id)
```

Replace with:

```python
        raw = ToolResultStore().get(tool_id)
```

### Method

- Two call sites changed from `ctx.tool_result_store.<method>(...)` to
  `ToolResultStore().<method>(...)` — a fresh, stateless instance is created per call, matching
  how the class is designed to be used elsewhere (each method opens/closes its own
  `SQLiteHelper` connection internally).
- No change to `_cmd_tool()`, `_cmd_plan()`, `_decode_args()`, or `_to_tool_result_view()`.

### Details

- `ctx` remains used in `_tool_list()` for `ctx.session.session_id` — only the
  `.tool_result_store` attribute access is replaced.
- `self._ctx` is still referenced in `_tool_show()`'s usage-error path (`self._out.write(...)`)
  and is otherwise unaffected — only the one line reading `.tool_result_store` changes.
- This introduces a new dependency edge `agent.commands.cmd_tooling` → `db.tool_results`, which
  is architecturally sound: `agent` is already permitted to import from `db` per the existing
  import-linter contracts (only `db` → `agent` is forbidden, not the reverse).

## Risks

- `tests/test_agent_cmd_tooling.py`'s `_make_cmd()` helper currently injects a `MagicMock()` via
  `ctx = SimpleNamespace(tool_result_store=store, ...)` — this stops working once `_tool_list`/
  `_tool_show` no longer read `ctx.tool_result_store`. The companion test doc switches the
  injection strategy to `patch("agent.commands.cmd_tooling.ToolResultStore", return_value=store)`
  instead. See `implementations/*_test_agent_cmd_tooling.py.md`.
- After H-1/H-2/H-4 land, nothing in the codebase writes NEW rows to `tool_results` anymore —
  `/tool list`/`/tool show` become read-only views of whatever historical rows already exist in
  `session.sqlite` from before this rollout (or empty, on a fresh database). This is the expected,
  intentional end state per the plan's U-2 resolution — not a regression to fix here.
