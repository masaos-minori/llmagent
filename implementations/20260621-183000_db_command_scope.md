# Implementation: Explicit Database Scope for `/db` Commands

## Goal

Make the target database (RAG / Session) explicit in `/db` command structure so operators don't need implementation knowledge to understand which database an operation affects.

## Scope

**In-Scope:**
- Add scoped subcommand forms: `/db rag <subcmd>` and `/db session <subcmd>`
- Keep existing flat commands (`/db stats`, `/db urls`, etc.) as backward-compatible aliases
- Add `[RAG]` / `[Session]` label to all command output and help text
- `/db stats` continues to show both RAG and Session counts (combined); `/db rag stats` and `/db session stats` show per-DB counts
- Update `/db help` to show scoped forms prominently
- Update `docs/05_agent_04_state-and-persistence.md` (or relevant docs) for new command forms

**Out-of-Scope:**
- Removing any existing maintenance subcommand
- Replacing SQLite or the maintenance subsystem
- Adding new maintenance capabilities (e.g. `/db workflow` ‚Äî no workflow DB exists today)

## Assumptions

1. The current `/db help` table already has a "Target DB" column ‚Äî the structural split is already documented; this task makes it programmatically explicit.
2. "Workflow storage" referenced in the require maps to `session.sqlite` workflow tables ‚Äî there is no separate workflow database today; document this explicitly.
3. Backward compatibility: existing scripts or muscle memory that uses `/db stats` etc. must not break.
4. `/db rag stats` and `/db session stats` can share the same underlying service calls as the combined `/db stats`.

## Unknowns & Gaps

| ID | Unknown | Evidence Missing | Resolution | Blocking |
|---|---|---|---|---|
| UNK-01 | Is there a separate workflow SQLite file? | No `workflow.sqlite` in `/opt/llm/db/` from env.md | No separate workflow DB ‚Äî document that workflow data lives in session.sqlite | False |
| UNK-02 | Should `/db rag stats` and `/db session stats` replace or extend `/db stats`? | Require says "restructure or clarify" | Keep `/db stats` (both), add `/db rag stats` and `/db session stats` as explicit sub-forms | False |

## Implementation

### Target files

- `scripts/agent/commands/cmd_db.py` ‚Äî add scope-level routing (`rag`, `session`) and per-scope help; tag output with `[RAG]` / `[Session]`
- `docs/05_agent_04_state-and-persistence.md` ‚Äî update `/db` command reference table
- `tests/test_agent_cmd_db.py` ‚Äî add tests for scoped forms

### Procedure

#### Step 1: Add scope routing in `_cmd_db()`

Split args to detect scope prefix (`rag`, `session`) as first token. If first token is `rag`: dispatch to `_cmd_db_rag(rest)` with remaining args. If first token is `session`: dispatch to `_cmd_db_session(rest)` with remaining args. Otherwise: existing flat dispatch (backward compat).

#### Step 2: Implement `_cmd_db_rag()` dispatcher

Maps subcommands: `stats` ‚Üí `_db_rag_stats()`, `urls` ‚Üí `_db_list_urls`, `clean` ‚Üí `_db_clean`, `rebuild-fts` ‚Üí `_db_rebuild_fts`, `recover` ‚Üí `_db_recover`, `consistency` ‚Üí `_db_consistency`. Default (no sub): show RAG help table.

#### Step 3: Implement `_cmd_db_session()` dispatcher

Maps subcommands: `stats` ‚Üí `_db_session_stats()`, `health` ‚Üí `_db_health`, `checkpoint` ‚Üí `_db_checkpoint`, `vacuum` ‚Üí `_db_vacuum`, `purge` ‚Üí `_db_purge`. Default (no sub): show Session help table.

#### Step 4: Add `_db_rag_stats()` and `_db_session_stats()`

- `_db_rag_stats()`: show documents/chunks from `RagMaintenanceService().stats_rag()` with `[RAG]` label
- `_db_session_stats()`: show sessions/messages from `DbMaintenanceService().stats()` with `[Session]` label

#### Step 5: Tag existing output with `[RAG]` / `[Session]` labels

All `_db_rebuild_fts`, `_db_health`, `_db_checkpoint`, `_db_vacuum`, `_db_recover`, `_db_consistency` output already has target labels; verify consistency.

#### Step 6: Update `/db help`

- Show scoped forms prominently at top: `/db rag <subcmd>` and `/db session <subcmd>`
- Keep the existing flat commands section as "Compatibility aliases"
- Note: "workflow data lives in session.sqlite; no separate workflow DB"

#### Step 7: Update docs

- `docs/05_agent_04_state-and-persistence.md`: add new scoped command examples, update `/db` command reference table

#### Step 8: Tests

- Test: `/db rag stats` calls `_db_rag_stats()`
- Test: `/db session stats` calls `_db_session_stats()`
- Test: `/db stats` (flat) still works (backward compat)
- Test: `/db rag unknown` shows RAG help

### Method

- Additive routing layer (~30 lines in `cmd_db.py`)
- New per-scope subcommand handlers
- Documentation expansion for scoped forms

## Risks

- **Risk:** Adding scope routing breaks a test that calls `_cmd_db("stats")` expecting flat dispatch
  ‚Üí **Mitigation:** flat dispatch is the fallback; "stats" doesn't start with "rag" or "session"
- **Risk:** `/db rag` conflicts with a future MDQ-related command using "rag" prefix
  ‚Üí **Mitigation:** "rag" scope refers to rag.sqlite maintenance; scoping is within `/db` only ‚Äî no conflict
