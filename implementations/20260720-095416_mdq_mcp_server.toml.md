config documentation"), Design item 3 / Implementation step "Phase 1 — Config history

**Existing-doc check:** `ls implementations/ implementations/done/ | grep -F
"mdq_mcp_server.toml"` returns one hit, `implementations/20260618-150339_mdq_mcp_server.toml.md`

own content) since-superseded concern (the `status` key it added was itself removed on

targets the top of the file, near `db_path`; this doc targets lines 50-95 at the bottom). This is
a NEW, ADDITIONAL document for the same filename covering a disjoint concern.

## Goal

Replace the six historical `NOTE` comment blocks currently at `config/mdq_mcp_server.toml:50-95`
(~46 lines) with a short pointer comment referencing the new `docs/mdq_config_history.md`
(companion doc `implementations/20260720-095342_mdq_config_history.md.md`), shrinking the active
config file to only currently-relevant keys and comments while preserving full historical
rationale in the dedicated doc.

## Scope

**In scope:**
- Delete `config/mdq_mcp_server.toml:50-95` (the six `NOTE` blocks for `audit_log_path`,
  `concurrency_limit`, `max_search_results`, the four hybrid-search keys, the three
  summary-cache keys, `enable_refresh`, and `status`).
- Replace them with a short 2-3 line pointer comment naming all seven removed-key identifiers
  (for grep-ability) and referencing `docs/mdq_config_history.md`.

**Out of scope:**
- Any active (non-comment) config key or value in this file — no key is added, removed, or
  changed; only comment text is trimmed.
- Creating `docs/mdq_config_history.md` itself — covered by the companion doc
  `implementations/20260720-095342_mdq_config_history.md.md` (must land together with this
  change, or the pointer comment would reference a nonexistent file).
- Any code in `scripts/mcp_servers/mdq/*.py` — this config file is comment-only churn in the
  affected region; `MdqService`'s config loader does not parse comments.

## Assumptions

1. `config/mdq_mcp_server.toml:50-95` currently contains exactly the six NOTE blocks described
   in the plan, confirmed by direct read during this design cycle — line numbers match the
   plan's citations exactly (no drift since the plan was written).
2. `config/mdq_mcp_server.toml` is already covered by an existing `deploy/deploy.sh` copy line
   (per the plan's Assumption 6, line 88), so no `deploy/deploy.sh` edit is needed for this
   comment-only trim.
3. Removing these comments does not affect `MdqService`'s config parsing — TOML comments are
   not parsed as data; only active `key = value` lines are read. Confirmed no active key is
   touched (lines 50-95 are 100% `#`-prefixed comment lines and blank lines, per the direct
   read).
4. The original comment text remains recoverable via `git log -p -- config/mdq_mcp_server.toml`
   even after this trim, providing a fallback in addition to the new dedicated doc (per the
   plan's stated risk mitigation).

## Implementation

### Target file

`config/mdq_mcp_server.toml`

### Procedure

1. Open `config/mdq_mcp_server.toml` and locate the six `NOTE` blocks (currently lines 50-95;
   re-run `rg -n "^# NOTE:" config/mdq_mcp_server.toml` at implementation time in case line
   numbers have shifted, per the plan's own risk note).
2. Delete all six blocks (comment lines + their surrounding blank-line separators) in that
   range.
3. Insert a short replacement comment in their place, naming the seven removed-key identifiers
   and pointing to the new doc.
4. Leave every active `key = value` line elsewhere in the file untouched (e.g. `allowed_dirs`,
   `db_path`, `include_globs`, `max_results_limit`, `enable_grep`, etc. at lines 1-48 stay
   exactly as-is).

### Method

Comment-block replacement. Pseudocode for the replacement text (not a full file dump — only the
segment currently at lines 50-95 changes):

```toml
# Historical notes on config keys removed from this file (audit_log_path,
# concurrency_limit, max_search_results, use_embedding/embedding_dims/
# vector_table/embedding_model, summary_cache_enabled/summary_threshold/
# summary_model, enable_refresh, status) have moved to
# docs/mdq_config_history.md.
```

Everything before line 50 (`enable_grep = true` and earlier, lines 1-48) and everything after
line 95 (currently nothing — line 95/96 is the file's end) stays unchanged; if a trailing blank
line existed after the last NOTE block, preserve normal end-of-file formatting (single trailing
newline).

### Details

- Keep the pointer comment to 2-3 lines as specified by the plan (Design item 3) — it should be
  scannable, not a re-summary of each key's rationale (that content lives in the new doc, not
  here).
- List all seven key identifiers (not just "six blocks", since the hybrid-search group covers
  four keys and the summary-cache group covers three, for 1+1+1+4+3+1+1 = 12 total key names
  across 7 NOTE blocks/groups) so a future `grep` for any individual key name (e.g.
  `embedding_model`) still finds a hit in the config file pointing to the history doc.
- Re-verify via `rg -n "^# NOTE:" config/mdq_mcp_server.toml` before editing that no other NOTE
  block exists outside the 50-95 range that this change might inadvertently also need to touch
  (the plan's Affected areas table scopes this to exactly lines 50-95; confirmed by direct read
  that no other `# NOTE:` comment exists in the file today).
- This change must land in the same commit/PR as the companion `docs/mdq_config_history.md`
  creation (implementations doc `20260720-095342_mdq_config_history.md.md`), since the pointer
  comment references it by path.
