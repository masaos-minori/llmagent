# Implementation procedure: `scripts/mcp_servers/web_search/service.py` (add `fetch_browser` orchestration, UNK-01)

Source plan: `plans/20260720-135137_plan.md`, Implementation step 4 (service-layer half); resolves
UNK-01 (route `browser_fetch` through `service.py` for health/metrics parity) per the plan's
recommended default.

Prior docs for this filename among the many `service.py` hits under `implementations/done/` are for
unrelated modules entirely (undo/db-maintenance/pipeline/mdq/github services — different packages).
The one `web_search`-specific hit, `implementations/done/20260720-083212_service.py.md`, belongs to
today's earlier requirement batch (timeout/error-classification wiring for `search_web`) — read in
full, no mention of `browser`/`fetch_browser`. No overlap. New document.

## Goal

Add `service.fetch_browser()`, mirroring `service.search_web()`'s timing/validation/health/metrics
orchestration pattern exactly, so `browser_fetch` gains the same health/metrics tracking discipline
as `search_web` (closing the gap UNK-01 identifies: the requirement's literal sample code bypasses
`service.py` entirely, which would leave `/health` blind to `browser_fetch` failures).

## Scope

**In scope**: add one new public async function, `fetch_browser()`, plus a private
`_classify_browser_error` helper, to `scripts/mcp_servers/web_search/service.py` (current: 112
lines, one function `search_web` plus `_classify_upstream_error`).
**Out of scope**: `search_web()` and `_classify_upstream_error()` — untouched, still the sole
handler for `WebSearchUpstreamError` subclasses. Per-tool keying of `health.py`/`metrics.py`
internals is a separate concern documented in the `health.py`/`metrics.py` docs — this doc only
covers the *call sites* `fetch_browser()` makes into those two modules' public functions, however
they end up keyed.

## Assumptions

1. `search_web()`'s shape (lines 60-113): `t0 = time.perf_counter()`, build/validate the Pydantic
   request, call the provider, and on `ValueError` record a `metrics.record_query(success=False,
   error_type="validation_error")` (no `health.record_failure` — validation errors never reach the
   provider, so they are not provider-health signals); on `WebSearchUpstreamError` classify + record
   both `metrics.record_query(success=False, ...)` and `health.record_failure(error_type)`; on
   success, log + `metrics.record_query(success=True, ...)` + `health.record_success()`.
   `fetch_browser()` mirrors this exactly, substituting `BrowserFetchRequest`/`fetch_browser`
   (provider) / `BrowserValidationError`/`BrowserAuthorizationError` for the search-specific types.
2. Per UNK-03's per-tool-keying resolution (see `health.py`/`metrics.py` docs), `fetch_browser()`
   calls the **browser-specific** health/metrics accessors (e.g. `health.record_browser_success()`/
   `health.record_browser_failure()` and a `tool="browser_fetch"`-tagged `metrics.record_query`, or
   whatever concrete API the `health.py`/`metrics.py` docs land on) — not the bare `record_success`/
   `record_failure`/`record_query` names `search_web()` uses, to avoid cross-contaminating
   `search_web`'s degradation status (this is the plan's primary Risk #1, directly mitigated here).
3. `_cfg` (the module-level `WebSearchConfig` singleton imported from `web_search_models`) now
   carries the `browser_*` fields (per the `web_search_models.py` doc) — `fetch_browser()`
   constructs a `BrowserConfig` view via `BrowserConfig.from_web_search_config(_cfg)` once per call
   (cheap, 4-field dataclass) rather than caching it at import time, to stay consistent with how
   `search_web()` already reads `_cfg.search_timeout_sec` fresh (no caching) on every call.

## Implementation

### Target file

`scripts/mcp_servers/web_search/service.py`

### Procedure

1. Extend the existing `mcp_servers.web_search.web_search_models` import to include
   `BrowserAuthorizationError`, `BrowserConfig`, `BrowserFetchRequest`, `BrowserFetchResponse`,
   `BrowserValidationError`.
2. Extend the `search_provider` import to include `fetch_browser as _provider_fetch_browser`
   (aliased to avoid a name clash with this module's own new `fetch_browser` function).
3. Add `_classify_browser_error(exc: Exception) -> str`: returns `"authorization_error"` for
   `BrowserAuthorizationError`, `"validation_error"` for `BrowserValidationError`, else
   `"fetch_error"` (covers unclassified `httpx` exceptions propagating from the provider layer, per
   the `search_provider.py` doc's Method section — this module is where that "let it propagate
   un-classified" decision gets resolved into a metrics label).
4. Add:
   ```
   async def fetch_browser(args: dict[str, Any]) -> BrowserFetchResponse:
       t0 = time.perf_counter()
       browser_cfg = BrowserConfig.from_web_search_config(_cfg)
       try:
           req = BrowserFetchRequest(**args)
           result = await _provider_fetch_browser(req, browser_cfg)
       except (BrowserValidationError, BrowserAuthorizationError, Exception) as e:
           ms = (time.perf_counter() - t0) * 1000
           error_type = _classify_browser_error(e)
           # metrics/health calls per health.py/metrics.py doc's chosen per-tool API
           raise
       ms = (time.perf_counter() - t0) * 1000
       logger.info(fmt_kvlog("browser_fetch", url=req.url[:80], ms=f"{ms:.0f}"))
       # success metrics/health calls per health.py/metrics.py doc's chosen per-tool API
       return result
   ```
   Note: the bare `except (..., Exception)` clause above is pseudocode shorthand for "catch the two
   named exceptions plus anything else the provider raises" — at implementation time, split into
   `except (BrowserValidationError, BrowserAuthorizationError) as e: ... raise` (both re-raised
   after recording, so `web_search_server.py`'s 403/422 handlers still fire) and a narrower
   `except Exception` only if a genuinely unclassified-fetch-failure metrics bucket is wanted; do
   **not** introduce a bare `except:` (violates `rules/coding.md`'s ast-grep constraint check).

### Method

Directly mirrors `search_web()`'s try/except/timing structure; no new control-flow pattern
introduced. `fetch_browser()` re-raises every exception after recording (never swallows), exactly
like `search_web()`, so `web_search_server.py`'s exception handlers remain the single place that
turns exceptions into HTTP responses.

### Details

- This function does **not** call `search_duckduckgo` or touch anything `search_web`-specific —
  fully independent code path sharing only the module's `logger`/timing idiom.
- `formatters.fdisp_browser_fetch` (separate doc) calls `service.fetch_browser`, not
  `search_provider.fetch_browser` directly — this is the concrete mechanism satisfying UNK-01.
