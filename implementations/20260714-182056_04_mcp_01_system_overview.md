# Implementation Procedure: Clarify Responsibility Boundary Between Agent and MCP Server Config (System Overview)

## Goal

Introduce the two-layer configuration model in the MCP system overview documentation and clarify that `agent.toml` owns only lifecycle/transport settings, not application config.

## Scope

- `docs/04_mcp_01_system_overview.md` only
- Text addition and clarification; no new content creation beyond what's needed to introduce the two-layer model

## Assumptions

1. The requirement `requires/20260714_04_require.md` is the canonical specification for this task.
2. `config/agent.toml` contains only agent-side MCP lifecycle and transport entries.
3. `config/*_mcp_server.toml` contains each MCP server's own application configuration.
4. No source code changes are required — documentation updates only.

## Implementation

### Target file

`docs/04_mcp_01_system_overview.md`

### Procedure

1. **Locate config-related sections**: Find any statements claiming "all configuration keys are defined in config/agent.toml" or similar overly broad claims.
2. **Remove overly broad statements**: Delete claims implying a single-file config model.
3. **Introduce two-layer model**: Add explicit statement of the two-layer configuration model at the top of relevant sections.

### Method

- Pattern-based search followed by targeted text replacement via file edit.
- Preserve surrounding context and formatting.

### Details

- Remove any text claiming "all config in agent.toml" or equivalent.
- Introduce the two-layer model:
  ```
  Layer 1 - Agent Process Configuration (config/agent.toml)
  ├── mcp_servers.<key>.startup_mode    # subprocess / persistent / none
  ├── mcp_servers.<key>.transport       # stdio / http
  ├── mcp_servers.<key>.url             # HTTP endpoint (for http transport)
  ├── mcp_servers.<key>.healthcheck_mode # probe / manual
  └── mcp_servers.<key>.cmd             # Command to start subprocess (for stdio)

  Layer 2 - MCP Server-Local Application Configuration (config/*_mcp_server.toml)
  ├── allowlists / denylists
  ├── resource limits
  ├── audit paths
  ├── allowed_repos / allowed_repos_mode (GitHub-specific)
  ├── command_allowlist (shell-specific)
  ├── allowed_dirs (file server-specific)
  └── auth_token_env / auth_token_file (secret references)
  ```
- Ensure every mention of MCP config specifies which layer it refers to.

## Validation plan

1. Verify no remaining overly broad "all config in agent.toml" claims.
2. Confirm the two-layer model is explicitly stated in the system overview.
3. Verify no implication that MCP servers can read `agent.toml`.
4. Verify no broken cross-references from removed sections.
5. Run `pre-commit run --all-files` if markdown linting is configured.
