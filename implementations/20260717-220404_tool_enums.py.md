# Implementation procedure: `scripts/agent/tool_enums.py`

Source plan: `plans/20260717-130159_plan.md` (requirement `requires/20260717_07_require.md`),
Implementation step 3: "Add `UNKNOWN` to `OperationType` in `tool_enums.py`."

## Goal

Add a new `UNKNOWN` member to `agent.tool_enums.OperationType` so `classify_operation_type()`
(`scripts/agent/tool_policy.py`) can report a fail-safe classification for tools that
`RuntimeToolRegistry` (or, currently, the static frozenset checks) cannot identify, instead of
silently defaulting to `READ` â€” the safest-looking category â€” for an unclassified tool.

## Scope

**In scope**
- `scripts/agent/tool_enums.py`: `OperationType` enum only â€” add one member, `UNKNOWN = "unknown"`.

**Out of scope**
- Any change to `RiskLevel`, `ApprovalDecisionType`, `GuardDecisionType` (same file, unrelated enums).
- Wiring `OperationType.UNKNOWN` into `classify_operation_type()`/`classify_risk()` â€” that is
  `tool_policy.py`'s own implementation item (this plan's steps 4-5), tracked separately. This doc
  covers only the enum addition itself.
- `scripts/shared/runtime_tool_registry.py` / `scripts/shared/runtime_tool.py` â€” those files (per
  their own implementation docs, `implementations/20260717-203121_runtime_tool.py.md` and
  `implementations/20260717-203200_runtime_tool_registry.py.md`) live in the `shared` layer and
  **cannot import `agent.tool_enums`** at all (`.importlinter`'s `shared-is-leaf` contract forbids
  `shared` importing from `agent`). Those docs already resolved their own
  `classify_operation_type()`/operation-type field to a locally-declared `Literal["read", "write"]`,
  not `agent.tool_enums.OperationType`. This `UNKNOWN` addition is agent-layer-only and has no
  cross-layer effect.

## Assumptions

1. Confirmed by direct read (`scripts/agent/tool_enums.py:17-22`) â€” `OperationType` currently has
   exactly 5 members: `WRITE = "write"`, `DELETE = "delete"`, `EXECUTE = "execute"`,
   `API_WRITE = "api_write"`, `READ = "read"`. No `UNKNOWN` member exists today.
2. `OperationType` is a `StrEnum` (`from enum import StrEnum`, line 8) â€” the new member must follow
   the same `NAME = "lowercase_value"` convention as its siblings for consistency with existing
   string-comparison call sites (e.g. `tests/test_tool_policy.py` compares against plain strings like
   `"medium"`/`"high"` for `RiskLevel`; `OperationType` members are compared as enum values in
   `tests/test_tool_approval_risk.py::TestClassifyOperationType`, e.g.
   `_classify_operation_type("shell_run") == "execute"` â€” `StrEnum` members compare equal to their
   string value, so `UNKNOWN` must be `"unknown"` to support the same pattern).
3. This is a pure additive enum change â€” no existing call site matches on an exhaustive
   `if/elif` chain over `OperationType` that would silently mis-handle a new member as a fallthrough
   bug (verified: `tool_policy.py`'s `classify_operation_type()` builds and returns `OperationType`
   values via ordered membership checks, defaulting to `READ`; it does not switch *on* an existing
   `OperationType` value). No other file in `scripts/` was found importing `OperationType` besides
   `tool_policy.py` and its two consuming test files (`tests/test_tool_policy.py`,
   `tests/test_tool_approval_risk.py`) at the time of this investigation.
