# Implementation: req 22 — Add strict plugin loading mode

## Goal

Add a strict plugin loading mode that fails startup when a plugin import fails, while keeping the current fail-open behavior for development.

## Scope

**In scope**:
- `scripts/shared/plugin_registry.py` — add `strict_mode` parameter to `load_plugins()`, change error handling in strict mode
- `scripts/agent/config_dataclasses.py` — add `plugin_strict` field to `ToolConfig`
- `scripts/agent/config_builders.py` — read `plugin_strict` from config
- `scripts/agent/factory.py` — pass strict mode flag to `load_plugins()`
- `config/agent.toml` — add `plugin_strict = false` default
- `tests/test_plugin_registry.py` — add tests for strict mode behavior
- `docs/05_agent_11_extension-points.md` — document strict mode

**Out of scope**:
- Changing the plugin directory path (still hardcoded in factory.py)
- Adding plugin validation beyond import/syntax check
- Modifying decorator behavior (`@register_command`, `@register_tool`, etc.)

## Assumptions

1. `plugin_strict` is a top-level config key under `[tool]` section, defaulting to `false` for backward compatibility.
2. In strict mode, the first plugin import failure raises an exception that aborts agent startup.
3. The existing warning log message is improved in both modes to include file name and error summary.

**Step 1 — `plugin_registry.py:load_plugins()`:**
```python
def load_plugins(
    plugin_dir: str | Path,
    *,
    known_tools: frozenset[str] = frozenset(),
    override_policy: str = "reject",
    strict_mode: bool = False,  # NEW parameter
) -> int:
    """Import all *.py files from plugin_dir; returns count loaded.

    When *strict_mode* is True, the first plugin import failure raises
    the original exception instead of logging and continuing.

    When *known_tools* is non-empty and *override_policy* is ``"reject"``,
    plugin tools that conflict with known MCP tools are removed after loading.
    """
    plugin_path = Path(plugin_dir)
    if not plugin_path.is_dir():
        logger.debug("Plugin dir not found, skipping: %s", plugin_dir)
        return 0

    loaded = 0
    for py_file in sorted(plugin_path.glob("*.py")):
        try:
            spec = importlib.util.spec_from_file_location(py_file.stem, py_file)
            if spec and spec.loader:
                mod = importlib.util.module_from_spec(spec)
                spec.loader.exec_module(mod)
                loaded += 1
                logger.info("Plugin loaded: %s", py_file.name)
        except (ImportError, SyntaxError, AttributeError, RuntimeError) as e:
            # NEW: improved error format for both modes
            error_msg = f"Plugin load failed ({py_file.name}): {type(e).__name__}: {e}"
            if strict_mode:
                logger.error(error_msg)
                raise  # NEW: re-raise in strict mode
            logger.warning(error_msg)

    _validate_tool_conflicts(known_tools, override_policy)
    return loaded
```

def _init_plugin_registry(ctx: AgentContext, audit_logger: Logger) -> None:
    """Load and register plugins from the plugins/ directory."""
    from shared.tool_constants import get_all_mcp_tool_names

    plugin_dir = Path(__file__).parent.parent.parent / "plugins"
    override_policy = "allow" if ctx.cfg.tool.plugin_tool_override else "reject"

    if ctx.cfg.tool.plugin_tool_override:
        audit_logger.info(
            "Plugin tool override policy: allow (shadowing MCP tools permitted)"
        )

    known_tools = (
        get_all_mcp_tool_names() if override_policy == "reject" else frozenset()
    )
    
    # NEW: log loading mode
    mode_str = "strict" if ctx.cfg.tool.plugin_strict else "fail-open"
    audit_logger.info("Plugin loading mode: %s", mode_str)

    n_plugins = plugin_registry.load_plugins(
        plugin_dir,
        known_tools=known_tools,
        override_policy=override_policy,
        strict_mode=ctx.cfg.tool.plugin_strict,  # NEW arg
    )
    if n_plugins:
        audit_logger.info("Loaded %s plugin(s) from %s", n_plugins, plugin_dir)
```

```markdown
### Strict Plugin Loading Mode

When `plugin_strict = true` in `[tool]`, the first plugin import failure raises
an exception that aborts agent startup. This is useful for CI/CD pipelines where
plugin failures should be treated as build errors.

Default is `false` (fail-open): plugin import failures are logged as warnings
and other plugins continue loading.

Example configuration:
```toml
[tool]
plugin_strict = true  # Fail startup on first plugin import error
```

Error message format (both modes):
`Plugin load failed (<filename>): <error_type>: <message>`
```
