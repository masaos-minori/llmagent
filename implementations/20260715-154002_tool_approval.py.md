# Implementation: Shared GitHub mutation set + high-risk dry-run failure blocking in `scripts/agent/tool_approval.py`

Source plan: `plans/20260715-141245_plan.md` (Implementation steps 2 and 3)

## Goal

1. Replace the locally duplicated `_GITHUB_WRITE_TOOLS` frozenset in
   `scripts/agent/tool_approval.py` with a shared-set-derived,
   explicitly documented pair of constants
   (`_GITHUB_MUTATION_TOOLS`, `_GITOPS_BLOCKABLE_TOOLS`), and use the
   latter in the `gitops_push_blocked` guard so the excluded-tools scope
   (`github_create_issue`, `github_add_issue_comment`) is explicit and
   testable rather than implicit in an unrelated-looking local set.
2. Make dry-run failure handling explicit for HIGH-risk GitHub mutation
   tools: an explicit dry-run **error result** (`result.is_error is
   True`) on a HIGH-risk tool must deny the approval immediately instead
   of silently falling back to a text-only preview; dry-run
   **unsupported/connection-failure** (`RuntimeError`/`OSError`) must
   keep today's fallback-to-preview behavior unchanged.

## Assumptions

1. `_GITHUB_WRITE_TOOLS` in the current file (7 tools: `github_push_files`,
   `github_create_or_update_file`, `github_delete_file`,
   `github_merge_pull_request`, `github_create_pull_request`,
   `github_update_pull_request`, `github_create_branch`) already excludes
   `github_create_issue`/`github_add_issue_comment` — this is the
   existing, intentional scope for `gitops_push_blocked` and must be
   preserved exactly (plan Assumption 4 / UNK-03), just made explicit via
   a named constant with a doc comment instead of an unlabeled local set.
2. Per plan Assumption 3, "HIGH-risk GitHub mutation tools" = tools whose
   `classify_risk()` result is `RiskLevel.HIGH` per
   `_DEFAULT_APPROVAL_RISK_RULES` in `scripts/agent/config_builders.py`
   (lines 64-80): currently `github_push_files`,
   `github_create_or_update_file`, `github_delete_file`,
   `github_merge_pull_request`. This is a superset check performed at
   runtime via `classify_risk(ctx.cfg, tool_name, args) == RiskLevel.HIGH`
   in `check_approval()` — no new static set is needed for this part;
   the existing `risk` variable computed at line 134 is reused.
3. No GitHub tool is in `_DEFAULT_DRY_RUN_TOOLS` today (`config_builders.py`
   lines 107-114 lists only filesystem tools), so this new blocking
   branch is dormant until an operator opts a GitHub tool into
   `approval_dry_run_tools` — confirmed low likelihood of behavior
   change on merge (plan Risk #3).
4. `ApprovalPreviewBlockingError` will subclass `ApprovalPreviewError` (per
   plan Design §3: "raise `ApprovalPreviewBlockingError`" as a
   discriminator), so existing `except ApprovalPreviewError` handling
   elsewhere in the codebase that is not aware of the new subclass still
   catches it — the new `check_approval()` logic must catch
   `ApprovalPreviewBlockingError` **before** the generic
   `ApprovalPreviewError` branch to distinguish the two paths.

## Implementation

### Target file

`scripts/agent/tool_approval.py` (existing, 196 lines)

### Procedure

1. Add import: `from shared.tool_constants import GITHUB_DANGEROUS_TOOLS,
   GITHUB_WRITE_TOOLS`.
2. Add import of the new exception from `tool_exceptions`:
   `ApprovalPreviewBlockingError` (defined in the companion
   `tool_exceptions.py` doc), alongside the existing
   `ApprovalPreviewError, PolicyViolationError` import block (lines
   18-21).
3. Replace lines 38-48 (`_GITHUB_WRITE_TOOLS` definition) with:
   ```python
   _GITHUB_MUTATION_TOOLS: frozenset[str] = GITHUB_WRITE_TOOLS | GITHUB_DANGEROUS_TOOLS

   # gitops_push_blocked guards repository-content and PR mutations.
   # Issue-tracker mutations are intentionally excluded — see
   # docs/05_agent_06_01_tool-execution-and-approval-execution.md.
   _GITOPS_BLOCKABLE_TOOLS: frozenset[str] = _GITHUB_MUTATION_TOOLS - {
       "github_create_issue",
       "github_add_issue_comment",
   }
   ```
4. In `check_approval()` line 117, replace
   `tool_name in _GITHUB_WRITE_TOOLS` with
   `tool_name in _GITOPS_BLOCKABLE_TOOLS`.
5. In `_build_preview_with_dry_run()`, replace the `is_error` branch
   (lines 76-79):
   ```python
   if is_error:
       raise ApprovalPreviewBlockingError(
           f"Dry-run for {tool_name!r} returned an error: {dry_text[:200]}"
       )
   ```
   Keep the `except ApprovalPreviewError: raise` (line 81-82) — since
   `ApprovalPreviewBlockingError` subclasses `ApprovalPreviewError`, this
   re-raise still passes it through unchanged. Keep the
   `RuntimeError`/`OSError` branch (lines 83-86) raising the plain
   `ApprovalPreviewError`.
6. In `check_approval()`, restructure the `try/except` around
   `_build_preview_with_dry_run` (lines 140-144):
   ```python
   try:
       preview = await _build_preview_with_dry_run(ctx, tool_name, args)
   except ApprovalPreviewBlockingError as e:
       if risk == RiskLevel.HIGH:
           msg = f"  [DENIED] {tool_name}: dry-run reported an error: {e}"
           audit_approval(ctx, tool_name, risk, args, "denied_dry_run_error")
           emit_denied(tool_name, msg)
           return False
       logger.warning("Dry-run preview unavailable for %r: %s", tool_name, e)
       preview = build_preview(tool_name, args)
   except ApprovalPreviewError as e:
       logger.warning("Dry-run preview unavailable for %r: %s", tool_name, e)
       preview = build_preview(tool_name, args)
   ```
   Note: the `except ApprovalPreviewBlockingError` branch must be listed
   before `except ApprovalPreviewError` since the former is a subclass —
   Python evaluates `except` clauses in order and the first matching
   clause wins.
7. Confirm `risk` (computed at line 134, before this try/except block) is
   already available at this point — no reordering needed.

### Method

Two additive/substitutive changes, both confined to this file (plus the
new exception class in `tool_exceptions.py`):
- Set consolidation: rename + derive from shared constants (no behavior
  change for the `gitops_push_blocked` guard — same tools blocked as
  today).
- New conditional deny path keyed off `risk == RiskLevel.HIGH` and a new
  exception subtype; all other combinations (MEDIUM risk, or the
  RuntimeError/OSError "unsupported" path) retain the current
  fallback-to-preview behavior verbatim.

### Details

- `audit_approval(ctx, tool_name, risk, args, "denied_dry_run_error")` —
  use the risk actually computed (`RiskLevel.HIGH` in this branch), not a
  hardcoded literal, for consistency with the existing
  `denied_gitops_push_blocked` call site (line 119-121) which hardcodes
  `RiskLevel.HIGH` because that guard runs before risk classification;
  here `risk` is already known, so pass it directly.
- `emit_denied(tool_name, msg)` mirrors the existing pattern at line 122;
  reuse the same helper, no new output primitive needed.
- Do not catch `ApprovalPreviewBlockingError` inside
  `_build_preview_with_dry_run`'s own `except (RuntimeError, OSError)`
  clause — it is raised explicitly from the `is_error` branch, not from a
  caught runtime failure, so it must propagate past that clause
  untouched (already the case since it's raised, not caught, in that
  code path).
