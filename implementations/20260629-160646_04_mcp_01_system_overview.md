# Implementation: MCP Startup Mode Terminology Alignment

## Goal

Align MCP startup mode terminology between `McpServerConfig` implementation and all MCP documentation, add cross-field validation for all invalid startup_mode+transport combinations, and add test coverage for those validations.

## Scope

- **In-Scope**:
  - `scripts/shared/mcp_config.py`: add validation that `ondemand` is only valid for `stdio`
  - `scripts/agent/factory.py`: verify routing handles all mode+transport combos correctly
  - `docs/04_mcp_01_system_overview.md`: clarify `persistent` (http) meaning; standardize wording
  - `docs/04_mcp_02_protocol_and_transport.md`: same
  - `docs/04_mcp_03_routing_lifecycle_and_execution.md`: same
  - `docs/04_mcp_06_configuration_and_operations.md`: same
  - `tests/test_mcp_config.py`: add missing validation tests
- **Out-of-Scope**:
  - Introducing a new `external` startup mode
  - Redesigning lifecycle router
  - Changing transport behavior
  - Modifying `mcp/` server implementations

## Assumptions

- The string `"external"` is NOT used as a `startup_mode` value anywhere in existing docs (verified: zero occurrences). The requirement title overstates the problem; the actual issue is (a) missing cross-field validation for `ondemand`+`http` and (b) inconsistent prose describing `persistent` (http) across docs.
- `persistent` (http) canonically means "agent connects to a pre-existing HTTP server; no lifecycle action taken" — this is already the correct meaning per `04_mcp_01_system_overview.md` and `_validate_cross_fields`.
- `ondemand` is exclusively valid for `stdio` transport (per lifecycle code in `stdio_lifecycle.py`). Using `ondemand` with `http` silently has no effect (factory.py line 77 skips it), so a `ValueError` should be raised at config construction time.

## Implementation

### Target file: `scripts/shared/mcp_config.py`

#### Procedure

Add validation that raises `ValueError` when `startup_mode == StartupMode.ONDEMAND and transport == TransportType.HTTP`.

#### Method

Direct file edit — add new check in `_validate_cross_fields()` after the existing `subprocess`+`stdio` check.

#### Details

**After line 105 (after the subprocess+stdio validation), add:**
```python
        if (
            self.startup_mode == StartupMode.ONDEMAND
            and self.transport == TransportType.HTTP
        ):
            raise ValueError(
                "startup_mode='ondemand' is only valid for transport='stdio'; "
                "HTTP servers use 'persistent' or 'subprocess'"
            )
```

### Target file: `tests/test_mcp_config.py`

#### Procedure

Add 3 new test cases to the existing `TestMcpServerConfigValidation` class.

#### Method

Direct file edit — append new test methods.

#### Details

**Add after line 71 (after `test_idle_timeout_default_zero`):**
```python
    def test_ondemand_http_raises(self) -> None:
        """ondemand startup_mode with HTTP transport should raise ValueError."""
        with pytest.raises(ValueError, match="ondemand.*is only valid for transport='stdio'"):
            McpServerConfig("http", "http://127.0.0.1:8000", [], startup_mode="ondemand")

    def test_subprocess_stdio_raises(self) -> None:
        """subprocess startup_mode with stdio transport should raise ValueError."""
        with pytest.raises(ValueError, match="subprocess.*is only valid for transport='http'"):
            McpServerConfig("stdio", "", ["python", "s.py"], startup_mode="subprocess")

    def test_persistent_http_valid(self) -> None:
        """persistent startup_mode with HTTP transport should be valid (regression guard)."""
        cfg = McpServerConfig("http", "http://127.0.0.1:8000", [], startup_mode="persistent")
        assert cfg.startup_mode == "persistent"
```
