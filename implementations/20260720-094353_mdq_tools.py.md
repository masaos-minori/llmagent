# Implementation: scripts/mcp_servers/mdq/mdq_tools.py (align `TOOL_LIST` inputSchema with runtime constraints)

Source plan: `plans/20260719-211521_plan.md` ("Align MDQ public tool schema with runtime
constraints"), Implementation step Phase 1 / Design section.

No prior implementation document (in `implementations/` or `implementations/done/`) covers this
concern. Three other `mdq_tools.py` docs exist but address unrelated changes: removal of
`fts_consistency_check`/`fts_rebuild` admin entries (`implementations/done/20260716-131146_mdq_tools.py.md`),
removal of `stats`/hybrid-adjacent admin tools (`implementations/done/20260620-145229_mdq_tools.py.md`,
titled `scripts/mcp/mdq/tools.py` — pre-rename path, same concern-class: description text only), and a
`search_docs.limit`/`max_results_limit` *description string* rewrite
(`implementations/20260720-092231_mdq_tools.py.md`). None of the three add or touch `enum`,
`minimum`, `maximum`, `items.type`, or `minItems` JSON-Schema constraint keys — this document is a
new, additive schema-tightening pass and does not conflict or overlap with any of them (the
092231 doc's `description` string edits for `limit`/`max_results_limit` are compatible with, and
independent of, the `minimum`/`maximum` keys added here — both may land in the same dict block).

## Goal

Add `enum` / `minimum` / `maximum` / `items.type` / `minItems` constraints to
`TOOL_LIST[*]["inputSchema"]["properties"]` for `search_docs`, `get_chunk`, `outline`,
`index_paths`, `refresh_index`, and `grep_docs` so the publicly advertised JSON Schema matches the
runtime validation/config caps already enforced in `mdq_models.py` (`SearchDocsRequest.mode:
Literal["bm25"]`) and `mdq_service.py` (`MdqService.__init__`'s config-sourced caps). Purely
additive dict-literal edits — no tool add/remove/rename, no dispatch/handler change, no
`is_write`/`requires_serial` flag change.

## Scope

**In scope:**
- `TOOL_LIST` entries for `search_docs`, `get_chunk`, `outline`, `index_paths`, `refresh_index`,
  `grep_docs` (`scripts/mcp_servers/mdq/mdq_tools.py:23-164`) — add schema keys to existing
  `properties` entries; add one new property (`outline.max_depth`).
- `MCPToolSchema` TypedDict shape is unaffected — `inputSchema` is already typed as
  `dict[str, Any]`, so no TypedDict field changes are needed to hold the new keys.

**Out of scope:**
- `stats` tool (`mdq_tools.py:129-134`) — empty `properties`, nothing to add.
- Any `"name"`, `"description"`, `"status"`, `"is_write"`, `"requires_serial"` value.
- `mdq_models.py` Pydantic field validators (`Field(ge=/le=)`) — schema-only per plan Assumption 7;
  no model-level enforcement added in this change.
- `mdq_service.py` — no runtime validation/clamping logic changes; existing `min(request, config)`
  clamping behavior is unchanged and unaffected by tightening the advertised schema.
- `config/mdq_mcp_server.toml` — no default value changes.

## Assumptions

1. Current file confirmed by direct read (`scripts/mcp_servers/mdq/mdq_tools.py`, 167 lines,
   7-entry `TOOL_LIST`): none of the 6 in-scope tools' `inputSchema.properties` currently carry
   `enum`, `minimum`, `maximum`, `items`, or `minItems` keys — every property is currently only
   `{"type": ..., "description": ...}`. This is a pure addition, not a modification of existing
   constraint values.
2. `mdq_models.py:98` confirms `SearchDocsRequest.mode: Literal["bm25"] | None = "bm25"` already
   restricts `mode` to `"bm25"` at the Pydantic layer; the schema `enum` addition documents an
   existing runtime constraint, it does not introduce a new one.
3. `mdq_service.py.__init__` (lines 50-81) confirms the live config-sourced defaults used to derive
   `maximum` values below: `max_results_limit=100`, `max_chars_per_chunk=10000`,
   `max_total_result_chars=100000`, `max_outline_items=500`, `max_outline_depth=6`,
   `max_grep_matches=200`, `max_chars_per_match=500`. `config/mdq_mcp_server.toml` (read directly)
   declares the same five TOML-backed values (`max_results_limit`, `max_chars_per_chunk`,
   `max_total_result_chars`, `max_outline_items`, `max_grep_matches` all present with matching
   values); `max_outline_depth`, `max_chars_per_match`, `context_before`, `context_after` are not
   present as TOML keys — they only exist as Python-side `.get(key, default)` fallback literals in
   `mdq_service.py`, so their schema `maximum`/`minimum` values are sourced from the Python default,
   not a TOML value (see Details for how the drift-guard test in the companion test doc must handle
   this distinction).
4. `mdq_models.py:114-119` confirms `OutlineRequest` already has `max_depth: int | None = 6` as a
   model field, but `mdq_tools.py`'s `outline` `inputSchema.properties` (lines 86-91) currently
   exposes only `path` and `max_outline_items` — `max_depth` is entirely absent from the public
   schema today. Per plan Assumption 6, this document adds it as a new property.
5. `paths` fields (`index_paths`, `refresh_index`, `grep_docs`) and `tag_filter`
   (`search_docs`) are currently `{"type": "array", "description": ...}` with no `items` key at
   all — adding `"items": {"type": "string"}` is additive, not a replacement of an existing (wrong)
   `items` value.

## Implementation

### Target file

`scripts/mcp_servers/mdq/mdq_tools.py`

### Procedure

Six independent dict-literal edits inside the existing `TOOL_LIST` list, in file order. For each,
locate the property block by its `"name"` key, then edit only the listed sub-keys — do not touch
`"description"` strings (a separate, already-landed concern per
`implementations/20260720-092231_mdq_tools.py.md` for `search_docs.limit`/`max_results_limit`; other
tools' descriptions are untouched by this change).

1. **`search_docs`** (`inputSchema.properties`, currently lines ~30-56):
   - `mode`: add `"enum": ["bm25"]`.
   - `limit`: add `"minimum": 1, "maximum": 100`.
   - `max_results_limit`: add `"minimum": 1, "maximum": 100`.
   - `max_total_result_chars`: add `"minimum": 1, "maximum": 100000`.
   - `tag_filter`: add `"items": {"type": "string"}`.

2. **`get_chunk`** (currently lines ~67-75):
   - `max_chars_per_chunk`: add `"minimum": 1, "maximum": 10000`.

3. **`outline`** (currently lines ~87-91):
   - `max_outline_items`: add `"minimum": 1, "maximum": 500`.
   - Insert new property `max_depth`: `{"type": "integer", "description": "Max heading depth to include (default: use config value)", "minimum": 1, "maximum": 6}`.

4. **`index_paths`** (currently line ~103):
   - `paths`: add `"items": {"type": "string"}, "minItems": 1`.

5. **`refresh_index`** (currently line ~117):
   - `paths`: add `"items": {"type": "string"}, "minItems": 1`.
   - `force` unchanged.

6. **`grep_docs`** (currently lines ~141-158):
   - `paths`: add `"items": {"type": "string"}` (no `minItems` — optional filter, unlike the two
     write tools above, per plan's explicit per-tool wording).
   - `max_grep_matches`: add `"minimum": 1, "maximum": 200`.
   - `max_chars_per_match`: add `"minimum": 1, "maximum": 500`.
   - `context_before`: add `"minimum": 0` (no `maximum` — no config cap exists for this field).
   - `context_after`: add `"minimum": 0` (no `maximum`).

### Method

In-place dict-literal key additions using `Edit` on each property sub-dict — no structural
reshaping of `TOOL_LIST`, no new imports, no control-flow change. Each edit adds keys after the
existing `"type"`/`"description"` pair; key order within a dict is not semantically significant to
JSON Schema consumers, but for readability append new constraint keys after `"description"` to keep
each property block visually consistent across the file, e.g. (pseudocode, not to be pasted
verbatim — exact formatting must satisfy `ruff format`'s 120-char line width):

```
<field>: {
    "type": <existing>,
    "description": <existing, unchanged unless already covered by 092231 doc>,
    <new constraint keys>,
}
```

For the new `outline.max_depth` property, insert as a new dict entry inside
`outline`'s `"properties"` mapping, after `max_outline_items`.

### Details

- Do not change any `"type"` value — only add sibling keys (`enum`, `minimum`, `maximum`, `items`,
  `minItems`) or, for `outline`, one new property (`max_depth`).
- Do not add `minimum`/`maximum` to `context_before`/`context_after` beyond the `minimum: 0`
  specified — no config cap exists to source an upper bound from (plan Assumption 5 / UNK-02).
  Leave these two fields unbounded above.
- Do not add `minItems` to `grep_docs.paths` — intentional asymmetry vs `index_paths`/
  `refresh_index.paths`, per plan Design section's explicit note that only the two write-tools'
  `paths` fields require at-least-one-path.
- `TOOL_LIST`'s 7th entry (`stats`) is not touched.
- No `_WRITE_TOOLS` frozenset change — `is_write`/`requires_serial` flags on `index_paths`/
  `refresh_index` are unaffected by adding `items`/`minItems` to their `paths` property.
- This is pure dict-literal data — no unit test lives inside this file; correctness is verified by
  the companion `tests/test_mdq_tool_schema.py` document (see sibling implementation doc with the
  same batch context) and by `tests/test_tool_server_layer_consistency.py` (re-run only, no edit
  expected per plan Phase 3 / UNK-04).
