## Goal

`grep_docs()` already builds a `matches: list[GrepDocMatch]` and a `truncated: bool` internally
(`db_grep.py:76-85`) before formatting them into text. Change the function to return
`(str, GrepDocsMetadata)` so `mdq_server.py::call_tool()` can read `match_count`/`truncated`
directly instead of counting `"---"` separator occurrences in the formatted output text (today's
fragile approach, `mdq_server.py:265`).

## Scope

**In scope:**
- `grep_docs()` (`db_grep.py:51-106`): change return type from `str` to
  `tuple[str, GrepDocsMetadata]`; populate `GrepDocsMetadata` from `len(matches)` (→
  `match_count`), the existing `truncated` local (`db_grep.py:85`), `len(req_paths)` (→
  `path_filter_count`), and a `pattern_preview` built from the compiled pattern's source string.
  `grep_enabled` is **not** available inside `db_grep.py` (it is an `MdqService`-level config
  flag, `service.enable_grep`, checked by the caller *before* `db_grep.grep_docs()` is even
  invoked — see `mdq_service.py:328-329`); this document's `GrepDocsMetadata` sets
  `grep_enabled=True` unconditionally at this layer (since `db_grep.grep_docs()` is only ever
  reached when the config gate already passed) — the caller (`mdq_service.py`, companion doc)
  does not need to override this field for the success path; the `enable_grep=False` case never
  reaches `db_grep.py` at all (it raises `MdqValidationError` earlier), so `call_tool()`'s error
  path handles that case via the existing error-audit branch, not via this metadata.
- Import `GrepDocsMetadata` from `mdq_models.py` (companion doc
  `20260720-093550_mdq_models.py.md`).
- Update the one call site in `mdq_service.py::grep_docs()` (companion doc
  `20260720-093554_mdq_service.py.md`) to unpack the new tuple.

**Out of scope:**
- `find_grep_match()` (`db_grep.py:18-48`) — unchanged, no metadata concern (operates on a single
  already-fetched row).
- The `req_paths` authorization-list semantics from `20260720-091546_db_grep.py.md` — this
  document does not re-specify or alter that behavior; if that document's Option A (caller-side
  short-circuit, no `db_grep.py` change) is implemented, `path_filter_count` in this document
  simply reflects whatever `req_paths` list the caller passes (authorized subset or raw list,
  depending on which sibling change has landed) — no interaction requiring code changes here.
- `mdq_service.py`'s `enable_grep` config gate — unchanged, lives entirely in the caller.

## Assumptions

1. Direct read of `db_grep.py:51-106` (current code) confirms: `matches` (`list[GrepDocMatch]`,
   line 76) and `truncated` (`bool`, line 85, computed as `len(matches) >= max_matches`) are both
   already local variables available at the point the formatted string is built (lines 87-106) —
   no new computation needed, only exposing existing locals as metadata fields.
2. `compiled: re.Pattern[str]` is the function's second parameter; `compiled.pattern` (the
   original regex source string) is used for `pattern_preview` (truncated to 80 chars, matching
   `audit_target.py`'s existing truncation convention for `grep_docs`'s `pattern` field,
   `audit_target.py:29`).
3. The two early/late return points (`"No matches found."` at line 88, and the final formatted
   string at lines 100-106) both need to return a `GrepDocsMetadata` tuple — `match_count=0,
   truncated=False` for the no-matches case (since `truncated` is only ever `True` when
   `len(matches) >= max_matches` and `matches` is empty in that branch, `truncated` is trivially
   `False`).

## Implementation

### Target file

`scripts/mcp_servers/mdq/db_grep.py`

### Procedure

1. Add `GrepDocsMetadata` to the `from mcp_servers.mdq.mdq_models import GrepDocMatch` import
   (`db_grep.py:15`), changing it to a parenthesized multi-name import.
2. Change `grep_docs()`'s signature return annotation from `-> str` to
   `-> tuple[str, GrepDocsMetadata]`.
3. At the `"No matches found."` early return (line 88):
   ```
   return "No matches found.", GrepDocsMetadata(
       pattern_preview=compiled.pattern[:80],
       path_filter_count=len(req_paths),
       match_count=0,
       truncated=False,
       grep_enabled=True,
   )
   ```
4. At the final return (lines 100-106), keep the existing string-building logic unchanged and
   return it alongside metadata:
   ```
   return result, GrepDocsMetadata(
       pattern_preview=compiled.pattern[:80],
       path_filter_count=len(req_paths),
       match_count=len(matches),
       truncated=truncated,
       grep_enabled=True,
   )
   ```

### Method

Two-return-point signature change; both `GrepDocsMetadata` constructions read from already-
computed locals (`matches`, `truncated`, `req_paths`, `compiled.pattern`) — no new queries or
control flow.

### Details

- `path_filter_count=len(req_paths)` reports the count of the *input* path filter list as passed
  to this function (could be `0` if no filter was supplied, i.e. scanning all indexed chunks) —
  this is intentionally the raw input count, not a "matched paths" count (that information is not
  tracked separately; `match_count` already covers matched-row count).
- No exception-handling change: `grep_docs()` has no `try`/`except` in its current body (the
  caller, `mdq_service.py::grep_docs()`, wraps the whole call in its own connection-management
  `try`/`finally`) — this document does not add one.
