# Implementation procedure: `scripts/agent/services/config_reload.py`

Source plan: `plans/20260717-130341_plan.md` (requirement `requires/20260717_08_require.md`),
Implementation steps 1-2 ("Locate the exact insertion point" + "Add the call").

## Goal

After a successful `/reload`, re-apply the current `agent_safety_tier`/`requires_approval`/
`enabled_for_llm` policy derivation onto the already-discovered `RuntimeToolRegistry` tool set, by
calling its `apply_policy(...)` method from `ConfigReloadService._sync_services()`. `/reload` must
continue to never fetch `/v1/tools` or rebuild the registry — this change is additive only.

## Scope

**In scope**
- `scripts/agent/services/config_reload.py`: `ConfigReloadService._sync_services()` gains a new
  conditional block calling `ctx.services_required.runtime_tools.apply_policy(...)`, mirroring the
  existing `llm`/`hist_mgr`/`tools` null-check-and-sync pattern already in that method (lines 170-197).

**Out of scope**
- `apply_policy()`'s internals (`scripts/shared/runtime_tool_registry.py`) — requirement 02.
- `RuntimeToolRegistry`'s discovery/population and the `AppServices.runtime_tools` field itself —
  requirement 03 (see Assumption 1 below: **this field does not concretely exist in the repo yet**).
- Any change to MCP server config reload / restart-required classification — unchanged.

## Assumptions

1. **Critical, verified discrepancy vs. this plan's stated call signature**: the plan's Goal/Design
   sections say the new call is `ctx.services_required.runtime_tools.apply_policy(ctx.cfg)` (a single
   `cfg` argument). Direct read of `implementations/20260717-203200_runtime_tool_registry.py.md`
   (requirement 02's actual finalized implementation procedure for `apply_policy()`) shows the method's
   real designed signature is:
   ```
   def apply_policy(
       self,
       tier_map: Mapping[str, AgentSafetyTier],
       allowed_tools: Sequence[str] = (),
   ) -> None: ...
   ```
   Requirement 02 rejected a `cfg`-object parameter specifically because `scripts/shared/` (where
   `RuntimeToolRegistry` lives) cannot import agent-layer config dataclasses (`.importlinter`'s
   `shared-is-leaf` contract) — so this call site must extract plain primitives from `ctx.cfg` itself,
   not pass `ctx.cfg` through. **This doc uses the corrected, real signature; the plan's `apply_policy(ctx.cfg)`
   text is stale and must not be implemented literally.**
2. **`tier_map` source**: `ctx.cfg.approval.tool_safety_tiers: dict[str, str]` (confirmed at
   `scripts/agent/config_dataclasses.py:350`, inside `ApprovalConfig`) is the tool-name -> tier-string
   mapping (`"READ_ONLY"|"WRITE_SAFE"|"WRITE_DANGEROUS"|"ADMIN"`), already updated by
   `_reload_approval_settings()` -> `_reload_approval_config()` (`config_reload.py:414-415`,
   `approval.tool_safety_tiers = dict(d)`) earlier in `apply_config_dict()`'s sequence — confirmed by
   direct read of the current file (lines 119-143, 395-419). Its values are validated against the same
   4-value set by `ApprovalConfig.__post_init__` -> `validate_tool_safety_tiers()`
   (`scripts/agent/services/config_validators.py:178-187`) **only at `ApprovalConfig` construction
   time** — `_reload_approval_config()`'s direct attribute mutation on the live `approval` object does
   **not** re-run `__post_init__`, so a malformed `/reload` payload could in principle install an
   out-of-vocabulary tier string without validation. This is a pre-existing gap, out of scope to fix
   here, but it means the static type of `tool_safety_tiers` (`dict[str, str]`) does not literally match
   `apply_policy()`'s `Mapping[str, AgentSafetyTier]` parameter (a 4-value `Literal`) — a `typing.cast`
   is needed at the call site (see Details); it is a type-level assertion, not a runtime guarantee.
3. **`allowed_tools` source**: `ctx.cfg.tool.allowed_tools: list[str]` (confirmed at
   `scripts/agent/config_dataclasses.py:197`), updated by `_reload_approval_settings()`
   (`config_reload.py:447-448`, `ctx.cfg.tool.allowed_tools = list(lst)`) in the same pre-`_sync_services()`
   sequence. Empty list = all tools allowed, matching `apply_policy()`'s own documented convention
   (per requirement 02's doc, Assumption 2).
4. **`ctx.services_required.runtime_tools` does not exist in the repo today.** Direct grep of
   `scripts/agent/context.py` (`AppServices.__init__`, lines 134-156) confirms no `runtime_tools`
   field, and repo-wide grep for `RuntimeToolRegistry` returns zero matches in any `.py` file (only in
   design docs under `implementations/`/`plans/`). Requirement 02's and requirement 03's own
   implementation docs each state the `context.py` field is "already covered" by pre-existing,
   unrelated `implementations/done/*context.py*.md` docs (a filename-match false positive — those docs
   predate this batch and do not add a `runtime_tools` field). **This means the field this call depends
   on is not yet concretely designed anywhere in the repo as of this doc.** This doc proceeds on the
   working assumption (consistent naming across requirements 02/03's docs) that the field will be named
   `runtime_tools` on `AppServices` and will be `RuntimeToolRegistry | None` (nullable, per requirement
   03's plan Assumption 3 — populated post-startup, async, after `build_agent_context()` returns).
   **Flag for whoever implements requirement 03/context.py wiring: confirm the exact field name matches
   `runtime_tools` before this call is wired up; if it differs, this call site's attribute name must be
   updated to match.**
5. `_sync_services()` (`config_reload.py:165-203`) already runs strictly after
   `_reload_approval_settings()` (called at line 133, inside `apply_config_dict()`, before
   `_sync_services()` is invoked at line 139) — so by the time `_sync_services()` runs, `ctx.cfg.approval
   .tool_safety_tiers` and `ctx.cfg.tool.allowed_tools` already reflect the new reload payload. No
   additional ordering change is needed; the new block can be added directly inside `_sync_services()`,
   consistent with that method's existing `llm`/`hist_mgr`/`tools` service-sync blocks.
6. No rediscovery / no `/v1/tools` call is introduced by this change — confirmed by construction: the
   new block only reads `ctx.cfg.approval.tool_safety_tiers`/`ctx.cfg.tool.allowed_tools` and calls
   `apply_policy()`; it makes no HTTP calls and does not touch `ctx.cfg.mcp.mcp_servers` or
   `_classify_mcp_server_changes()`.

## Implementation

### Target file

`scripts/agent/services/config_reload.py`

### Procedure

1. Add imports at the top of the file (after existing `from typing import TYPE_CHECKING, Any` — extend
   to `from typing import TYPE_CHECKING, Any, cast`), plus `from collections.abc import Mapping` and,
   inside the existing `if TYPE_CHECKING:` block (alongside `from agent.context import AgentContext`),
   `from shared.runtime_tool import AgentSafetyTier` (type-only import — `AgentSafetyTier` is a `Literal`
   alias, needed only for the `cast(...)` annotation, so it belongs under `TYPE_CHECKING` to avoid a
   runtime import of a `shared`-layer module that may not exist yet at the time this file is edited,
   per Assumption 4 — if requirement 02's `runtime_tool.py` has landed by then, a normal top-level import
   is equally fine and preferred for clarity, since `agent` may import `shared` per the layer contract).
2. In `_sync_services()` (`config_reload.py:165-203`), add a new conditional block. Insert it after the
   existing `ctx.services_required.tools is not None` block (lines 193-197) and before the
   `system_prompt_tool` handling (lines 199-201), so it reads top-to-bottom as "one more service being
   synced," consistent with the existing pattern:
   ```
   if ctx.services_required.runtime_tools is not None:
       ctx.services_required.runtime_tools.apply_policy(
           tier_map=cast(Mapping[str, "AgentSafetyTier"], ctx.cfg.approval.tool_safety_tiers),
           allowed_tools=ctx.cfg.tool.allowed_tools,
       )
       result.applied.append("runtime_tools")
   ```
3. Do not modify `apply_config_dict()` itself, `_reload_approval_settings()`, `_reload_approval_config()`,
   or any MCP-classification method — this change is additive only inside `_sync_services()`.

### Method

Plain additive conditional block inside an existing method — no new class, no new public method, no
change to `ConfigReloadService`'s public surface (`apply_config`, `apply_config_dict` signatures
unchanged).

### Details

Pseudocode for the full amended `_sync_services()` body (no production code — illustrative only):

```
def _sync_services(self, new_cfg: dict[str, Any]) -> ConfigReloadOutcome:
    result = ConfigReloadOutcome()
    ctx = self._ctx

    if ctx.services_required.llm is not None:
        ... # unchanged
        result.applied.append("llm")

    if ctx.services_required.hist_mgr is not None:
        ... # unchanged
        result.applied.append("hist_mgr")

    if ctx.services_required.tools is not None:
        ... # unchanged
        result.applied.append("tools")

    # NEW block:
    if ctx.services_required.runtime_tools is not None:
        ctx.services_required.runtime_tools.apply_policy(
            tier_map=cast(Mapping[str, AgentSafetyTier], ctx.cfg.approval.tool_safety_tiers),
            allowed_tools=ctx.cfg.tool.allowed_tools,
        )
        result.applied.append("runtime_tools")

    if "system_prompt_tool" in new_cfg:
        ctx.conv.system_prompt_content = new_cfg["system_prompt_tool"]

    return result
```

Notes:
- `"runtime_tools"` is appended to `result.applied` so `/reload`'s CLI-facing report includes it,
  consistent with how `"llm"`/`"hist_mgr"`/`"tools"` are already reported.
- No `try`/`except` is added around this call — the existing three service-sync blocks above it also do
  not guard against exceptions from the underlying `apply_config()`/`apply_policy()` calls; consistency
  is preferred over introducing new error-handling asymmetry in this method (if `apply_policy()` itself
  should never raise for well-formed inputs per requirement 02's contract, no guard is needed here).
- This block is a no-op when `ctx.services_required.runtime_tools is None` (e.g., discovery has not
  completed yet, or requirement 03 has not landed) — same defensive shape as the other three blocks.
