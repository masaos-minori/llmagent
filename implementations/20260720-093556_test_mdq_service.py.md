## Goal

Once `search.search_docs()`, `indexer.index_paths()`, `MdqService.search_docs/index_paths/
refresh_index/grep_docs` change from returning bare `str` to `(str, <metadata>)` tuples (companion
docs), every existing test call site in this file that captures the return value and asserts on
it as a string breaks (either an `AssertionError` from comparing a tuple to a string with `==`, or
a `TypeError`/semantic-mismatch from `"substring" in result` treating `result` as an iterable of
tuple elements rather than a string). Migrate every such call site to unpack the tuple, and add
new assertions on the metadata component so the new fields have direct test coverage in this
file (in addition to the dedicated `tests/test_mdq_audit.py`, companion doc
`20260720-093557_test_mdq_audit.py.md`, which covers `mdq_server.py`'s audit-log integration
specifically).

## Implementation

### Target file

`tests/test_mdq_service.py`

### Procedure

For each call site listed in Scope above:
1. Change `result = asyncio.run(<call>)` to `text, metadata = asyncio.run(<call>)`.
2. Change every subsequent `result ==`/`in result` assertion to `text ==`/`in text` (same
   assertion text, only the variable name changes).
3. Add 1-3 new assertions on `metadata[...]` per the "New assertions to add" list above,
   immediately after the migrated text assertions in the same test function (no new test
   functions needed for this coverage — additive assertions in existing tests, since each test
   already sets up the exact scenario the metadata assertion needs).

Example migration (`test_index_paths_md_file_returns_complete`, lines 529-534):
```python
def test_index_paths_md_file_returns_complete(
    self, service: MdqService, md_file: Path
) -> None:
    req = IndexPathsRequest(paths=[str(md_file)])
    text, metadata = asyncio.run(index_paths(service, req))
    assert text == "Indexing complete"
    assert metadata["indexed_count"] == 1
    assert metadata["failed_count"] == 0
```

### Method

Mechanical two-part edit per call site: (1) unpack the tuple where the return value is captured,
renaming `result` to `text`/`metadata` (or keeping `result` as the tuple and indexing `result[0]`/
`result[1]` if a smaller diff is preferred — either is acceptable, this document recommends the
unpacking form for readability, matching the style already used in `search.py`'s own internal
`_search_docs_structured` call: `result = ...; result["results"]` uses dict-style, so this file's
convention should prefer named unpacking over dict-style since the new types are per-tool
`TypedDict`s, not a single shared result dict). (2) add metadata-content assertions using the new
field names defined in companion `mdq_models.py` doc.

### Details

- Do not change the `_slow_search` monkeypatch stub in `test_search_timeout_raises_
  mdq_consistency_error` — it patches `_search_docs_structured` (the inner FTS5 helper, unaffected
  by this plan) and the test never inspects the outer `search_docs()`'s return value (it's wrapped
  in `pytest.raises(...)`).
- Import additions: none needed — `IndexPathsMetadata`, `SearchDocsMetadata`, `GrepDocsMetadata`,
  `RefreshSummary` are only referenced via dict-key access on already-typed local variables
  (`metadata["indexed_count"]`, etc.); no new top-level imports required unless a test wants to
  annotate the unpacked variable explicitly (optional, not required for `mypy` since the tuple's
  element types are already inferred from the called function's return annotation).
