# Implementation: Add pagination to /replay and /dlq endpoints in app.py

Steps covered: Plan 20260626-094715 — Steps 2-1, 2-2, 2-3, 2-4

---

## Goal

Add `limit` and `offset` (or `cursor`) pagination parameters to the `/replay` and `/dlq` HTTP endpoints so that large event sets do not cause unbounded memory consumption or response timeouts.

---

## Scope

- **In scope**: `scripts/eventbus/app.py` — `/replay` and `/dlq` route handlers
- **Out of scope**: `db.py` pagination query (step 2-1 for db, covered separately); docs (step 3-2); tests (step 3-1)

---

## Assumptions

- `/replay` replays events from a stream. `/dlq` lists dead-letter queue entries.
- Current implementation returns all matching events without pagination.
- Pagination approach: `limit: int = 100, offset: int = 0` query parameters.
- Response should include `total`, `limit`, `offset`, `items` fields.

---

## Implementation

### Target file
`scripts/eventbus/app.py`

### Procedure
1. Read `scripts/eventbus/app.py` — find `/replay` and `/dlq` route definitions.
2. Step 2-1 (`/replay` params): Add `limit` and `offset`:
   ```python
   @app.get("/replay")
   async def replay(
       stream: str,
       limit: int = Query(default=100, ge=1, le=1000),
       offset: int = Query(default=0, ge=0),
   ) -> ReplayResponse:
       events = await db.get_events(stream=stream, limit=limit, offset=offset)
       total = await db.count_events(stream=stream)
       return ReplayResponse(total=total, limit=limit, offset=offset, items=events)
   ```
3. Step 2-2 (`/dlq` params): Same pattern for `/dlq`.
4. Step 2-3 (Response models): Define `ReplayResponse` and `DlqResponse` with `total`, `limit`, `offset`, `items` fields.
5. Step 2-4 (Validation): Enforce `limit <= 1000` to prevent unbounded responses.

### Method
FastAPI query parameter addition. Response model change (breaking: callers expecting a plain list now get an object).
