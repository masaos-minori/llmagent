## Goal

Create `tests/test_mdq_audit.py`, testing `mdq_server.py::call_tool()`'s audit-detail block
(companion doc `20260720-093555_mdq_server.py.md`) end-to-end via the FastAPI app, per the plan's
acceptance criteria: audit detail for `search_docs`, `index_paths`, `refresh_index`, `grep_docs`,
authorization error, validation error, and — the core regression test for this requirement — an
explicit assertion that audit detail does **not** depend on output-text wording (rephrasing the
tool's output text must not break audit fields, since they now come from structured metadata, not
text parsing).

## Scope

**In scope — new test file with these test classes/functions:**
- `TestSearchDocsAudit`: index a file, call `/v1/call_tool` with `search_docs`, capture the
  emitted audit log record (via caplog or a patched `_audit_log`), assert `detail` contains
  `result_count=`/`shown_count=` with values matching the actual indexed content (not `0` or a
  placeholder).
- `TestIndexPathsAudit`: call `index_paths` with a mix of one valid `.md` file, one nonexistent
  path, and one non-`.md` file in a single request; assert audit `detail` contains
  `indexed_count=1, skipped_count=2, failed_count=0` (pinning down the exact mixed-scenario
  counts per the plan's Risk mitigation: "add test cases that pin down expected counts for mixed
  existing/missing/skip scenarios before merging").
- `TestRefreshIndexAudit`: call `refresh_index`, assert audit `detail` contains
  `indexed_count=`/`skipped_count=`/`deleted_count=`/`failed_count=` matching the
  `RefreshSummary` the underlying `refresh_paths()` call produces.
- `TestGrepDocsAudit`: index a file with a known pattern, call `grep_docs`, assert audit `detail`
  contains `match_count=` matching the real match count (not derived from counting `"---"` in the
  text).
- `TestAuthorizationErrorAudit`: call a tool with a path outside `allowed_dirs`, assert the
  resulting `_audit_log()` call received `error_type="MdqAuthorizationError"` (or whatever
  `type(exc).__name__` resolves to) via the `error_type=` parameter, **not** embedded in `detail`
  (regression test for plan UNK-04's fix).
- `TestValidationErrorAudit`: call `grep_docs` with `enable_grep=False` (or an invalid regex
  pattern), assert the same `error_type=` parameter usage.
- `TestAuditDetailIndependentOfOutputText` (the core regression test for this requirement): patch
  the relevant `MdqService` method (e.g. `search_docs`) so it returns a *rephrased* text string
  (e.g. `"3 hits located"` instead of the real formatted text) while returning the **same,
  correct** metadata dict; call `/v1/call_tool`; assert the audit `detail`'s `result_count=` still
  reflects the real (metadata-sourced) count, proving the audit path never re-derives the count
  from `r.output` text.
- `TestConcurrentAuditIsolation` (plan Risk mitigation: ContextVar concurrency safety): issue two
  concurrent `search_docs` requests with different queries (via `asyncio.gather` against the
  ASGI app, or two sequential `TestClient` calls if true concurrency is impractical to assert
  against — see Details for the concrete mechanism) with different expected result counts; assert
  each request's audit record's `result_count` matches its own query, not the other's (regression
  guard against the mdq-local `ContextVar` metadata side-channel leaking across requests).

**Out of scope:**
- Re-testing `tests/test_mdq_service.py`'s tuple-unpacking migration (covered by its own companion
  doc, `20260720-093556_test_mdq_service.py.md`) — this file tests the `mdq_server.py` HTTP layer
  end-to-end, not `MdqService`'s internal method signatures directly.
- `tests/test_audit_log_format.py`'s existing shared-format assertions (`error_type` field
  presence/absence generically across all MCP servers) — this file adds MDQ-specific audit
  *content* tests, not a duplicate of the shared-format contract tests.
- Any other MCP server's audit behavior.

## Assumptions

1. `mcp_servers/audit.py::_audit_log()` is the single audit-emission point (confirmed via
   `mdq_server.py`'s two call sites, lines 75-84 and 284-293) — tests can either (a) capture
   emitted log lines via `caplog` and assert on the formatted message content (the pattern
   `_mdq_error_handler` and `call_tool()` already log through `Logger(__name__, ...)`), or (b)
   patch `mcp_servers.audit._audit_log` directly with a `Mock`/`AsyncMock` and assert on its
   call-site keyword arguments (`detail=`, `error_type=`, `outcome=`) — **recommended: (b)**,
   since asserting on structured kwargs is more precise and less brittle than parsing a formatted
   log line, and avoids coupling the test to `_audit_log`'s internal message format.
2. Existing sibling test file `tests/test_mdq_error_taxonomy.py` (lines 152-154, 176-178)
   patches `"mcp_servers.mdq.server._dispatch_mdq_tool"` — this appears to reference a stale
   pre-rename module path (`mcp_servers/mdq/server.py` was deleted per commit `abcf0820`, per
   this plan's revision note; only `mcp_servers/mdq/mdq_server.py` exists today). This document's
   new tests use the correct, current path `mcp_servers.mdq.mdq_server._dispatch_mdq_tool` (or
   `mcp_servers.mdq.mdq_server._audit_log` for the audit-patching approach) and do **not**
   replicate the apparently-stale `mcp_servers.mdq.server` reference — flagging this discrepancy
   for awareness, not fixing it (out of scope; likely already covered by the pre-existing
   `issues/20260719-205532_risks.md`/`issues/20260720-070458_...` cross-references this plan's
   header note describes, or a new issue if not).
3. `fastapi.testclient.TestClient` (already used by `tests/test_mdq_error_taxonomy.py`,
   `tests/test_mdq_routing.py`) is the right harness for exercising `call_tool()` end-to-end
   without running a live uvicorn process.
4. For `TestIndexPathsAudit`'s mixed-scenario test, per companion `indexer.py` doc's Assumption 2
   (known limitation: `indexed_count` counts "attempted and did not raise," not "chunks
   confirmed written"), the test's valid `.md` file must actually contain at least one heading/
   section (so `_index_single_file` doesn't silently no-op) to make `indexed_count == 1`
   unambiguous and not rely on the documented limitation's edge case.
5. `service.enable_grep = False` is the simplest way to trigger `MdqValidationError` for
   `TestValidationErrorAudit`, reusing the existing check at `mdq_service.py:328-329`
   (`if not self.enable_grep: raise MdqValidationError(...)`, unchanged by this plan).

## Implementation

### Target file

`tests/test_mdq_audit.py` (new)

### Procedure

1. Module setup: reuse the existing `service`/`md_file`/`md_dir` fixture patterns already defined
   in `tests/test_mdq_service.py` (either import/reuse via a shared `conftest.py` fixture if one
   already provides `MdqService` instances scoped to `tmp_path`, or duplicate the minimal fixture
   locally — check `tests/conftest.py` during implementation for a reusable `mdq_service`
   fixture before duplicating).
2. For each test class in Scope, follow this pattern:
   ```python
   def test_search_docs_audit_result_count(
       self, service: MdqService, md_file: Path, monkeypatch: pytest.MonkeyPatch
   ) -> None:
       from unittest.mock import MagicMock

       import mcp_servers.mdq.mdq_server as mdq_server_module

       mock_audit = MagicMock()
       monkeypatch.setattr(mdq_server_module, "_audit_log", mock_audit)
       monkeypatch.setattr(mdq_server_module, "_service", service)

       from fastapi.testclient import TestClient

       asyncio.run(index_paths(service, IndexPathsRequest(paths=[str(md_file)])))
       client = TestClient(mdq_server_module.app)
       response = client.post(
           "/v1/call_tool", json={"name": "search_docs", "args": {"query": "Content"}}
       )
       assert response.status_code == 200
       call_kwargs = mock_audit.call_args.kwargs
       assert "result_count=1" in call_kwargs["detail"]
   ```
3. `TestAuditDetailIndependentOfOutputText`: patch `MdqService.search_docs` (or the relevant
   method) with an `AsyncMock` returning `("completely rephrased wording with no numbers", {
   "query_preview": "Content", "result_count": 1, "shown_count": 1, "truncated": False,
   "total_count": 1, "duration_ms": 5.0})`, then assert the audit `detail` still contains
   `result_count=1` — proving the value came from the mock's metadata return, not from parsing
   the (deliberately numberless) rephrased text.
4. `TestConcurrentAuditIsolation`: use `httpx.AsyncClient(app=mdq_server_module.app,
   base_url="http://test")` with `asyncio.gather(client.post(...), client.post(...))` for two
   different queries against two differently-sized indexed corpora, capturing two separate
   `mock_audit.call_args_list` entries and asserting each's `result_count` matches its own
   request's expected count (not swapped).

### Method

HTTP-level integration tests against the FastAPI `TestClient`/`AsyncClient`, patching only
`_audit_log` (to capture its kwargs) and, where needed, `_service`/individual `MdqService`
methods (for the rephrased-text and concurrency tests) — no unit-level testing of
`search.py`/`indexer.py`/`db_grep.py` metadata construction directly (that is implicitly covered
by these end-to-end assertions plus the metadata-content assertions added to
`tests/test_mdq_service.py` per its companion doc).

### Details

- Keep assertions on `mock_audit.call_args.kwargs["detail"]` substring-based (`"result_count=1"
  in detail`) rather than exact-string-equality, since `detail`'s exact formatting (order of
  `duration_ms=`/other fields) is an implementation detail not pinned by the plan's acceptance
  criteria — only the *presence and correctness* of each field matters.
- For `TestAuthorizationErrorAudit`/`TestValidationErrorAudit`, assert
  `mock_audit.call_args.kwargs["error_type"] == "MdqAuthorizationError"` (or
  `"MdqValidationError"`) and assert `"error_kind="` is **absent** from `detail` — the negative
  assertion is the actual regression guard for UNK-04's fix (today's code embeds it in `detail`;
  after the fix it must not).
- Do not import from the stale `mcp_servers.mdq.server` path (see Assumption 2) — always
  `mcp_servers.mdq.mdq_server`.
