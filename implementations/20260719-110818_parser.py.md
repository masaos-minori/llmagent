# Implementation: `scripts/mcp_servers/mdq/parser.py` (YAML frontmatter `tags` extraction)

Source plan: `plans/20260719-100727_plan.md` ("Implement `tags_json`/`token_count` in mdq-mcp;
reconcile stale MDQ known-issue docs"), Implementation step 1.

No existing implementations doc (under `implementations/` or `implementations/done/`) targets
`parser.py` (confirmed via `ls implementations/ implementations/done/ | grep -i parser` — zero
matches). This is a new item, not a genuine-overlap skip.

## Goal

`parse_markdown()` in `scripts/mcp_servers/mdq/parser.py` currently detects a leading
`---`...`---` YAML frontmatter block only to skip it, discarding its content entirely. Change it
to capture that block, parse it as YAML, and return a normalized `list[str]` of `tags` alongside
the existing `list[ParsedSection]`, so the one caller (`indexer.py`) can persist real tag data
instead of the hardcoded `tags_json = ""`.

## Scope

**In scope**
- Modify `parse_markdown()`'s return type from `list[ParsedSection]` to
  `tuple[list[ParsedSection], list[str]]` (sections, document-level tags).
- Within the existing frontmatter-detection block, capture the lines between the delimiters,
  join them, and parse with `yaml.safe_load`.
- Extract a `tags` field from the parsed frontmatter (if present): accept either a YAML list
  (`tags: [a, b]`) or a comma-separated string (`tags: "a, b"`), normalizing both to
  `list[str]`.
- Guard against malformed frontmatter (invalid YAML, non-dict top-level result, or a `tags`
  value that is neither a list nor a string): fall back to an empty tags list, do not raise.
- Add `import yaml` to this file (not currently imported anywhere in
  `scripts/mcp_servers/mdq/`).
- Add unit tests for: no frontmatter (empty tags), list-form tags, comma-separated-string-form
  tags, and malformed frontmatter (parse failure does not crash, falls back to empty tags) — see
  the companion test-file doc for the exact test placement (this doc's own Validation plan only
  runs them, it does not duplicate their design).

**Out of scope**
- Any change to the one caller (`indexer.py`) beyond what's needed to keep it working after this
  signature change — that caller-side threading is covered by a separate doc targeting
  `indexer.py` (same plan, Implementation steps 2-3).
- Per-section/per-heading tags — frontmatter is file-level; there is no per-heading tag concept
  in this parser today, and this change does not add one.
- Any change to `ParsedSection`'s own fields (`heading`, `heading_level`, `heading_path`,
  `content`, `start_line`, `end_line`, `ordinal`, `parent_heading`) — unaffected.

## Implementation

### Target file

`scripts/mcp_servers/mdq/parser.py`

### Procedure

1. Add `import yaml` near the top of the file, alongside the existing `import logging` (line 35)
   — keep import-order (ruff `I` rules / isort-compatible): stdlib (`logging`), then third-party
   (`yaml`), then local (`from mcp_servers.mdq.models import ...`).
2. Change the return type annotation of `parse_markdown` (line 49) from `-> list[ParsedSection]`
   to `-> tuple[list[ParsedSection], list[str]]`.
3. Before the main parsing loop (before line 70, `i = 0`), initialize `frontmatter_tags:
   list[str] = []`.
4. Inside the frontmatter-detection block (lines 93-102), when the closing `---` is found (the
   `if j < len(lines) and lines[j].rstrip() == "---":` branch), before `i = j + 1; continue`:
   - Capture `frontmatter_text = "\n".join(lines[i + 1 : j])`.
   - Parse it: `try: fm_data = yaml.safe_load(frontmatter_text) except yaml.YAMLError:
     fm_data = None`.
   - If `fm_data` is a `dict`, extract `raw_tags = fm_data.get("tags")`; otherwise leave
     `raw_tags = None`.
   - Normalize `raw_tags` into `frontmatter_tags`:
     - If it is a `list`, keep only string-coercible entries: `[str(t).strip() for t in
       raw_tags if str(t).strip()]`.
     - If it is a `str`, split on `,` and strip each piece: `[t.strip() for t in
       raw_tags.split(",") if t.strip()]`.
     - Otherwise (missing, wrong type, or parse failure), leave `frontmatter_tags = []`.
   - Wrap the extraction in a broad `except Exception` only around the YAML-shape handling (not
     a bare `except:` — `rules/coding.md` and `ast-grep --pattern 'except: $$$'` both prohibit
     bare except), so a malformed/unexpected frontmatter shape degrades to "no tags" rather than
     propagating and breaking indexing of an otherwise-valid document (this is the plan's
     explicitly stated risk mitigation).
5. At the end of the function, change `return sections` (line 167) to
   `return sections, frontmatter_tags`.
6. Update the function's docstring (lines 50-54) to mention the second return value.

### Method

Direct extension of the existing procedural parser — no new classes, no `Protocol`/`ABC`. The
normalization logic (list-or-comma-string) is a small pure function inline in the loop; it does
not warrant extraction into a separate module given its size (a few lines) and single call site.

### Details

- Do not use a bare `except:` for the YAML-parsing guard — catch `yaml.YAMLError` for the
  `yaml.safe_load` call specifically, and validate the resulting shape (`isinstance(fm_data,
  dict)`, `isinstance(raw_tags, list)` / `isinstance(raw_tags, str)`) explicitly rather than
  relying on exceptions for type mismatches.
- Use `yaml.safe_load`, not `yaml.load` — `rules/toolchain.md`'s bandit step and the source
  plan's own Validation plan both call this out explicitly to avoid a bandit finding.
- Keep the frontmatter-detection block's existing control flow (the `i == 0` at-start-of-file
  check, the `j` closing-delimiter search) unchanged — only add the capture/parse logic inside
  the already-existing "closing delimiter found" branch, immediately before the existing
  `i = j + 1; continue`.
- No change to `ParsedSection`, `models.py`, or `db_schema.py` from this item.
