# Implementation procedure: `tests/test_cmd_mcp.py` (per-tool disabled-diagnostics table tests)

Source plan: `plans/20260717-175327_plan.md` ("RuntimeToolRegistry — represent disabled MCP tools
without exposing them to the LLM", requirement 17), Implementation step 7 (test portion, paired
with `implementations/20260718-094405_cmd_mcp.py.md`).

## Goal

Extend `tests/test_cmd_mcp.py` with test coverage for the new per-tool diagnostics table appended to
`_cmd_mcp_status()`'s output (per `implementations/20260718-094405_cmd_mcp.py.md`): the table's
presence and correct field values for both a disabled tool and an enabled tool, and its absence when
no `RuntimeToolRegistry` is populated.

**Filename check — false positive, do not skip**: `implementations/20260718-032956_test_cmd_mcp.py.md`
also targets `tests/test_cmd_mcp.py`, but its Goal (paired with the requirement-11 `cmd_mcp.py` doc)
tests a **new `/mcp tools` subcommand** with 7 different fields
(`server_key`/`name`/`enabled_for_llm`/`agent_safety_tier`/`requires_approval`/`is_write`/
`requires_serial`) and `UnknownSubcommandError`'s allowed-tuple change — a disjoint feature from
this plan's extension to the existing `status` output. Confirmed by reading its Goal in full before
concluding this is a different test target, not a duplicate.

## Assumptions

- Current test-fixture shape confirmed by reading `tests/test_cmd_mcp.py:1-90`: `_Ctx` (lines
  18-45) is a `MagicMock`-backed stub for `AgentContext`; `_Mcp(_McpMixin)` (lines 48-50) wraps it
  for calling `_cmd_mcp_status()` directly. `_Ctx.__init__` currently sets
  `self.services_required = MagicMock()` with several explicit attributes configured
  (`health_registry`, `audit_logger`, `lifecycle`, ...) — `runtime_tools` is not among them today
  (confirmed: not present in the class body), so on a bare `MagicMock()`,
  `ctx.services_required.runtime_tools` auto-vivifies as a `MagicMock` (truthy, not `None`), which
  would NOT match the paired implementation doc's `if registry is not None` guard behaving as
  intended for the "no registry populated" test case. This test doc must extend `_Ctx.__init__` (or
  set the attribute per-test via `ctx.services_required.runtime_tools = ...`) to explicitly control
  this value in each new test case — explicitly set to `None` for the "no table" case, and to a real
  (or `MagicMock`-wrapped) `RuntimeToolRegistry`-shaped object with a working `.diagnostics()` return
  value for the "table present" cases.
- Simplest fixture approach, consistent with this file's existing style (real dataclasses/configs
  are used elsewhere, e.g. `_http()`/`_stdio()` build real `McpServerConfig` instances rather than
  mocks): construct a real `shared.runtime_tool_registry.RuntimeToolRegistry`, `register()` real
  `shared.runtime_tool.RuntimeTool` instances into it, and assign that real object to
  `ctx.services_required.runtime_tools` — avoids mocking `.diagnostics()`'s return shape by hand and
  exercises the real registry/table integration end-to-end.

## Implementation

### Target file

`tests/test_cmd_mcp.py` (existing, 555 lines).

### Procedure

1. Add imports: `from shared.runtime_tool import RuntimeTool` and `from
   shared.runtime_tool_registry import RuntimeToolRegistry` at the top of the file, alongside the
   existing `shared.mcp_config`/`shared.mcp_health` imports (lines 14-15).
2. Add a small module-level helper `_registry_with(*tools: RuntimeTool) -> RuntimeToolRegistry` that
   constructs a registry and registers each given tool — reduces per-test boilerplate, mirroring
   this file's existing `_http()`/`_stdio()` helper-function style.
3. Add a new test class `TestCmdMcpStatusToolDiagnostics` after `TestCmdMcpStatusSerialization`
   (after line ~511, before `TestAppServicesMemoryOptional` at line 514):
   - `test_tool_diagnostics_table_shows_disabled_and_enabled_tools`: build a `_Ctx` with
     `mcp_servers={"file_read": _http()}`, set `ctx.services_required.runtime_tools =
     _registry_with(RuntimeTool(name="read_file", server_key="file_read",
     config_dependent=False, enabled=True, disabled_reason="", enabled_for_llm=True),
     RuntimeTool(name="git_push", server_key="git", config_dependent=True, enabled=False,
     disabled_reason="missing GIT_TOKEN", enabled_for_llm=False))`; call `_cmd_mcp_status()`; assert
     via `capsys` that the printed output contains both tool names, `"missing GIT_TOKEN"`, and
     correct `True`/`False` markers for each tool's `enabled`/`enabled_for_llm` columns.
   - `test_tool_diagnostics_table_absent_when_registry_none`: `ctx.services_required.runtime_tools =
     None`; call `_cmd_mcp_status()`; assert the output contains no `"Tools (RuntimeToolRegistry)"`
     section header (proves the "skip when unpopulated" behavior from the paired implementation
     doc).
   - `test_tool_diagnostics_table_absent_when_registry_empty`: registry with zero registered tools;
     assert same as above (empty diagnostics also suppresses the section, per Procedure step 3 of
     the paired `cmd_mcp.py` doc: "if ... `registry.diagnostics()` is non-empty").
4. Update `_Ctx.__init__` (lines 18-45) to add `self.services_required.runtime_tools = None` as an
   explicit default (so every *other* existing test in this file, which doesn't care about this new
   field, keeps getting the "no table" behavior deterministically instead of an auto-vivified
   `MagicMock` truthy value) — this is a small, backward-compatible fixture change, not a behavior
   change to any existing test's assertions.
