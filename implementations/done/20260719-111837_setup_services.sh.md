# Implementation procedure: `deploy/setup_services.sh` (Browser MCP port-range comment update)

Source plan: `plans/20260719-101501_plan.md`, Implementation step 4 (Design/Affected areas).

`setup_services.sh` is a common target filename in this repo's history (e.g.
`implementations/20260710-154825_setup_services_sh_workflow_preflight.md`,
`implementations/20260710-155505_setup_services_sh_schema_version_check.md`). Checked by
content: both existing docs address unrelated prior additions (workflow preflight checks, schema
version checks) — neither touches the MCP-server port-range comments or Event Bus port
references. Flagged as checked, not a genuine overlap.

## Goal

Update `deploy/setup_services.sh`'s descriptive comments that state the MCP-server port range
(currently "8004-8014") to include Browser MCP's new port, and confirm no literal per-service
startup block needs adding for Browser MCP — correcting a scope assumption in the plan.

## Scope

**In scope**
- Update the three comments that state the MCP port range as `8004-8014`
  (`deploy/setup_services.sh:6`, `:73`, `:99`, all verified by direct read) to `8004-8016` (or
  the equivalent phrasing chosen — see Assumption 2) to reflect Browser MCP's addition at 8016.

**Out of scope — verified drift from the plan's own wording.** The plan's Implementation step 4
says "Update ... `deploy/setup_services.sh`" and Design/Affected-areas says "add Browser MCP
startup steps," implying a per-server startup block similar to how the Event Bus is started
(lines 85-91: an explicit `python -m eventbus.app &` background-launch block). **Direct read of
the full file (114 lines) shows this is not how MCP servers are started at all**: line 6's
comment states "MCP サーバ (ports 8004-8014) はエージェント管理 subprocess として起動" (MCP
servers ports 8004-8014 are started as agent-managed subprocesses), and line 73's comment
states explicitly "MCP サーバ (8004-8014) はエージェントが subprocess として管理するため登録不要"
(MCP servers 8004-8014 require no registration here because the agent manages them as
subprocesses) — meaning **every one of the 10 existing MCP servers, including the ones added
across many prior plans, has zero explicit startup code in this file**; they are all started
by the agent process itself at runtime, driven by each server's `[mcp_servers.*].cmd` entry in
`config/agent.toml` (`startup_mode = "subprocess"`). Line 99's comment reinforces this: "※ MCP
サーバ (ports 8004-8014) はエージェント起動時に自動起動します" (MCP servers auto-start when the
agent starts). **Therefore Browser MCP needs no explicit startup block added to this script —
only the three descriptive port-range comments need updating** to stay accurate now that a
server exists at 8016. This is the corrected, verified scope; the plan's "add ... startup
steps" wording does not match how any MCP server (old or new) is actually started in this file.

## Assumptions

1. Verified directly (`deploy/setup_services.sh:6`): `# MCP サーバ (ports 8004-8014) はエージェント
   管理 subprocess として起動。`
2. Verified directly (`deploy/setup_services.sh:73`): `# MCP サーバ (8004-8014) はエージェントが
   subprocess として管理するため登録不要`
3. Verified directly (`deploy/setup_services.sh:99`): `echo "  ※ MCP サーバ (ports 8004-8014)
   はエージェント起動時に自動起動します"`
4. **Port drift, verified.** Line 85's comment `# ── Event Bus (port 8015) ──` confirms port
   8015 is already occupied by the Event Bus, not available for a new MCP server — this is the
   evidence basis for the `server.py`/`agent.toml` docs' conclusion that Browser MCP must use
   port **8016**, not 8015. Since Browser MCP is the 11th MCP server and its port (8016) falls
   immediately after the Event Bus's port (8015), the three "8004-8014" range comments in this
   file are the only textual references that become stale once Browser MCP exists; no other
   line in this file enumerates individual MCP server ports.
5. No health-check or startup-order dependency is introduced by Browser MCP for any other
   service in this script — Browser MCP's own health check (per the `server.py` doc) is a
   trivial, dependency-free liveness check, so it does not need to be added to the
   `--- ヘルスチェック ---` block (lines 93-100), which only checks `embed-llm`, `agent-llm`,
   and `eventbus` explicitly — none of the 10 existing MCP servers appear in that health-check
   block either (confirmed by direct read), so adding Browser MCP there would be an
   inconsistent, unrequested addition, not a gap to fill.

## Implementation

### Target file

`deploy/setup_services.sh`.

### Procedure

1. Line 6: change `# MCP サーバ (ports 8004-8014) はエージェント管理 subprocess として起動。` to
   `# MCP サーバ (ports 8004-8016) はエージェント管理 subprocess として起動。`
2. Line 73: change `# MCP サーバ (8004-8014) はエージェントが subprocess として管理するため登録不要`
   to `# MCP サーバ (8004-8016) はエージェントが subprocess として管理するため登録不要`
3. Line 99: change `echo "  ※ MCP サーバ (ports 8004-8014) はエージェント起動時に自動起動します"` to
   `echo "  ※ MCP サーバ (ports 8004-8016) はエージェント起動時に自動起動します"`
4. Do not add any explicit Browser MCP startup/launch block — per the verified Out-of-scope
   note, no MCP server has one in this file.
5. Do not add Browser MCP to the health-check echo block (lines 93-100) — per Assumption 5, no
   MCP server appears there.

### Method

Three-line comment-text edits only; no control-flow changes.

### Details

No code logic — these are descriptive `echo`/comment strings with no functional effect on the
script's behavior; the edit is purely to keep operator-facing documentation-in-script accurate.
