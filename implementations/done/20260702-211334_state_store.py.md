# Implementation: state_store.py — Update _now() to Z-suffix Format

**Plan source:** `plans/20260702-201038_plan.md` (Phase 2b)
**Target file:** `scripts/agent/workflow/state_store.py`

---

## Goal

Change `_now()` from `datetime.now(UTC).isoformat()` (which emits `+00:00` suffix) to `datetime.now(UTC).strftime('%Y-%m-%dT%H:%M:%SZ')` (which emits `Z` suffix). This aligns Python-side timestamp generation with the Z-suffix canonical format and the DDL DEFAULT added in Phase 2a.

---

## Scope

**In:**
- `_now()` function in `scripts/agent/workflow/state_store.py`

**Out:**
- Any callers of `_now()` — they are unchanged; only the output format changes
- Existing DB rows — not rewritten
- Event Bus, RAG, or Session timestamp logic
- Any other file

---

## Assumptions

1. The canonical format is `YYYY-MM-DDTHH:MM:SSZ` (Z suffix), as decided in plan Phase 1.
2. `_now()` is the sole source of Python-generated workflow timestamps. No other callers produce workflow timestamps outside this module (to be verified with grep before implementing).
3. `datetime.now(UTC).strftime('%Y-%m-%dT%H:%M:%SZ')` truncates microseconds. This is intentional and acceptable.
4. Existing tests may assert the `+00:00` format. Any such assertions must be updated to match the new `Z` suffix.
5. Existing workflow.sqlite rows retain `+00:00`; no conversion is performed.

---

## Implementation

### Target file

`scripts/agent/workflow/state_store.py`

### Procedure

1. **Pre-check:** run `grep -rn "datetime.now" scripts/agent/workflow/` to confirm no other callers exist outside `state_store.py`.
2. Locate `_now()` at the top of the file (line 20–21 as observed).
3. Replace the function body.
4. Verify no import changes are needed (`datetime`, `UTC` are already imported).

### Method

Single Edit tool call on the `_now` function body.

### Details

```
# Before
def _now() -> str:
    return datetime.now(UTC).isoformat()

# After
def _now() -> str:
    return datetime.now(UTC).strftime('%Y-%m-%dT%H:%M:%SZ')
```

No other lines in the file change.
