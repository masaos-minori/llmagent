# Implementation: H-9 — Remove tool_results table DDL from schema_sql.py

## Goal

Remove the `CREATE TABLE IF NOT EXISTS tool_results (...)` and
`CREATE INDEX IF NOT EXISTS idx_tool_results_session` statements from `_SESSION_SCHEMA_TEMPLATE`,
so freshly-created `session.sqlite` databases no longer include this table.

## Scope

**Target**: `scripts/db/schema_sql.py`

**Depends on**: no hard runtime dependency (schema DDL and Python code that reads the table are
independent), but should land in the same rollout as `db/tool_results.py`'s deletion for
conceptual consistency (no code left that could use this table even if it existed).

**Out of scope**: every other table/trigger/index definition in `_SESSION_SCHEMA_TEMPLATE`
(`sessions`, `messages`, `memories`, etc.) and every other template string in this file
(`_RAG_SCHEMA_TEMPLATE`, workflow/eventbus templates) — unchanged.

## Assumptions

1. `CREATE TABLE IF NOT EXISTS` means this DDL is idempotent — removing it from the template
   does NOT drop the `tool_results` table from any EXISTING database that already has it (per
   the plan's explicit assumption: "既存の DB ファイルは変更されない"). This change only affects
   what gets created in NEW/fresh databases going forward.
2. No other template (`_RAG_SCHEMA_TEMPLATE`, workflow, eventbus) references `tool_results` —
   confirmed by the earlier grep showing all `tool_results` DDL matches are within this one
   template block.

## Implementation

### Target file

`scripts/db/schema_sql.py`

### Procedure

#### Step 1: Confirm the exact block boundaries

```bash
grep -n "tool_results" scripts/db/schema_sql.py
```

Expected: matches only within `_SESSION_SCHEMA_TEMPLATE` (the `CREATE TABLE` block and the
`CREATE INDEX` statement immediately after it).

#### Step 2: Remove the table and index DDL

Current (within `_SESSION_SCHEMA_TEMPLATE`, between the `messages` table and the `memories`
table):

```sql
    CREATE TABLE IF NOT EXISTS tool_results (
        id         INTEGER PRIMARY KEY AUTOINCREMENT,
        session_id INTEGER REFERENCES sessions(session_id) ON DELETE CASCADE,
        turn       INTEGER NOT NULL,
        tool_name  TEXT    NOT NULL,
        args_masked  TEXT,
        full_text  TEXT    NOT NULL,
        summary    TEXT,
        is_error   INTEGER NOT NULL DEFAULT 0,
        undone     INTEGER NOT NULL DEFAULT 0,
        created_at TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))
    );
    CREATE INDEX IF NOT EXISTS idx_tool_results_session
        ON tool_results(session_id);
```

Remove this entire block. The `messages` table's closing `);` and the `memories` table's opening
`CREATE TABLE IF NOT EXISTS memories (` become adjacent within the template string (with normal
whitespace/indentation, exactly as if this block had never existed between them).

### Method

- Pure text removal from a module-level SQL template string; no change to `build_session_schema_sql()`'s
  function signature or `.format(...)`/interpolation logic (assuming `dims` substitution happens
  elsewhere in the template, unrelated to this block).

### Details

- This does not require a migration script — `CREATE TABLE IF NOT EXISTS` semantics mean the
  removed DDL simply stops being ISSUED for new databases; it never actively removes anything
  from existing ones. Per the plan's own explicit note, no migration/DROP TABLE is in scope.
- The `tool_results.undone` column's documentation in `docs/05_agent_09_data-layer.md` was
  already addressed by the H-7 documentation doc
  (`implementations/20260708-164730_h7_docs_updates.md`) — no further doc changes needed here,
  though that doc's wording assumed the table still exists (for historical rows); it remains
  accurate under H-9 as well since existing databases keep the table.
