# Implementation: Audit workflow factory for fallback creation paths in factory.py

Steps covered: Plan 20260626-091910 — Step 1-2

---

## Goal

Audit `factory.py` (workflow factory) for any path that can produce a workflow object that bypasses the standard plan/approval path. Ensure the factory raises rather than returning a degraded or unsafe workflow object.

---

## Scope

- **In scope**: `scripts/agent/workflow/factory.py` (or equivalent) — audit for unsafe creation paths
- **Out of scope**: orchestrator.py changes (step 2-x); docs

---

## Assumptions

- `factory.py` contains a `WorkflowFactory` or `create_workflow()` function.
- There may be a `create_default()` or fallback path that creates a workflow with `require_approval=False` regardless of config.
- The fix is to either remove the unsafe path or add a guard that reads `ctx.cfg.workflow_require_approval`.

---

## Implementation

### Target file
`scripts/agent/workflow/factory.py`

### Procedure
1. Read `scripts/agent/workflow/factory.py` fully.
2. Search for any path that creates a `WorkflowEngine` or `WorkflowDefinition` with hardcoded `require_approval=False` or that ignores the config.
3. If found:
   - Remove the hardcoded override.
   - Replace with: `require_approval=ctx.cfg.workflow_require_approval`.
4. If a `create_default()` or emergency-fallback factory exists that was intended as a bypass, remove it entirely.
5. Ensure the factory raises `WorkflowCreationError` (defined in step 2-3 of plan 03) if inputs are insufficient.

### Method
Audit + minimal code change. Primary risk is breaking existing tests that rely on the default factory.
