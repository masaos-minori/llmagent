# Implementation: Event Bus Infrastructure (config, schema, importlinter, deploy)

## Goal

Lay the non-Python foundation for the eventbus package: SQLite DDL, JSON Schema, TOML config,
.importlinter contract, and deploy.sh runtime directory creation. All subsequent phases depend on
these artifacts being in place before any Python code runs.

## Scope

**In-Scope:**
- `config/eventbus.toml` — runtime settings (port, DB path, MAX_RETRY, storage dirs)
- `schemas/event_envelope.json` — JSON Schema draft-07 for event envelope validation
- `.importlinter` — add `eventbus` root_package + isolated forbidden contract
- `scripts/eventbus/__init__.py` — empty package marker
- `scripts/eventbus/schema.sql` — `events` table DDL with WAL, UNIQUE, and indexes
- `deploy/deploy.sh` — add mkdir for storage/offsets/deadletter + cp for eventbus.toml + schemas/

**Out-of-Scope:**
- `scripts/eventbus/app.py` (Python application code — covered in eventbus_app_phase1)
- Any changes to existing scripts/ packages

## Assumptions

1. `scripts/eventbus/` placed inside `scripts/` is automatically included in
   `deploy.sh` rsync (`scripts/ → /opt/llm/scripts/`).
2. `tomllib` (Python 3.11+) or `tomli` (backport) is available for TOML parsing.
3. JSON Schema draft-07 `format: "date-time"` is validated by `jsonschema` with `format_checker`.
4. `.importlinter` forbidden contract for `eventbus` must exist before any eventbus Python
   code is written, so CI catches violations immediately.

## Implementation

### Target files

### Procedure

#### Step 1: `config/eventbus.toml`

Create `config/eventbus.toml`:

```toml
port = 8015
db_path = "/opt/llm/db/eventbus.sqlite"
storage_dir = "/opt/llm/storage"
offsets_dir = "/opt/llm/offsets"
deadletter_dir = "/opt/llm/deadletter"
max_retry = 3
poll_interval_ms = 500
```

All paths are absolute `/opt/llm/` production paths. Tests override via environment or
fixture-level config injection.

#### Step 2: `schemas/event_envelope.json`

Create `schemas/` directory and `schemas/event_envelope.json`:

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "EventEnvelope",
  "type": "object",
  "required": ["event_id", "topic", "payload", "producer", "published_at"],
  "additionalProperties": false,
  "properties": {
    "event_id": {
      "type": "string",
      "pattern": "^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$",
      "description": "UUID v4 idempotency key"
    },
    "topic": {
      "type": "string",
      "minLength": 1,
      "maxLength": 255
    },
    "payload": {
      "type": "object"
    },
    "producer": {
      "type": "string",
      "minLength": 1,
      "maxLength": 255
    },
    "published_at": {
      "type": "string",
      "format": "date-time"
    },
    "schema_version": {
      "type": "string",
      "default": "1.0"
    }
  }
}
```

#### Step 3: `.importlinter` update

Append to `root_packages` and add new contract section:

```ini
root_packages =
    agent
    db
    mcp
    rag
    shared
    eventbus

[importlinter:contract:eventbus-is-isolated]
name = eventbus must not import from agent, mcp, rag, db, or shared
type = forbidden
source_modules =
    eventbus
forbidden_modules =
    agent
    mcp
    rag
    db
    shared
```

#### Step 4: `scripts/eventbus/__init__.py`

Create empty file — Python package marker only. No imports.

```python
"""eventbus — HTTP publish / SSE subscribe / SQLite persistence / Replay / DLQ."""
```

#### Step 5: `scripts/eventbus/schema.sql`

```sql
PRAGMA journal_mode=WAL;

CREATE TABLE IF NOT EXISTS events (
    seq          INTEGER PRIMARY KEY AUTOINCREMENT,
    event_id     TEXT    NOT NULL UNIQUE,
    topic        TEXT    NOT NULL,
    payload      TEXT    NOT NULL,
    producer     TEXT    NOT NULL,
    published_at TEXT    NOT NULL,
    acked_at     TEXT,
    retry_count  INTEGER NOT NULL DEFAULT 0,
    dlq_at       TEXT
);

CREATE INDEX IF NOT EXISTS idx_events_topic ON events(topic);
CREATE INDEX IF NOT EXISTS idx_events_seq   ON events(seq);
```

`seq` is `AUTOINCREMENT` to guarantee monotonic increment even after DELETE (DLQ operations
must not reuse seq values for replay correctness).

#### Step 6: `deploy/deploy.sh`

Add after the existing `mkdir -p "${DEPLOY_LOGS}"` block:

```bash
mkdir -p /opt/llm/storage
mkdir -p /opt/llm/offsets
mkdir -p /opt/llm/deadletter
```

Add to the config copy section:

```bash
cp "${REPO_ROOT}/config/eventbus.toml"  "${DEPLOY_CONFIG}/"
```

Add after the config copy section:

```bash
echo "--- schemas/ → /opt/llm/schemas/ ---"
mkdir -p /opt/llm/schemas
cp -r "${REPO_ROOT}/schemas/." /opt/llm/schemas/
```

### Method

- `schema.sql` uses `PRAGMA journal_mode=WAL` as the first statement so the connection that
  runs the DDL already benefits from WAL mode before any INSERT.
- `AUTOINCREMENT` on `seq` is intentional (vs plain `INTEGER PRIMARY KEY`) because DLQ
  operations may DELETE rows; `AUTOINCREMENT` guarantees no seq reuse.
- `additionalProperties: false` in `event_envelope.json` rejects unexpected fields at the
  boundary; internal code only handles the declared fields.
- importlinter contract is added before any Python code so CI enforces isolation from day 1.

## Validation plan

| Target | Strategy | Command | Expected |
|---|---|---|---|
| `schema.sql` syntax | Manual | `sqlite3 :memory: < scripts/eventbus/schema.sql` | no error |
| JSON Schema validity | Manual | `python3 -c "import json,jsonschema; jsonschema.Draft7Validator.check_schema(json.load(open('schemas/event_envelope.json')))"` | no exception |
| importlinter contract | Static | `uv run lint-imports` | 0 violations |
| deploy.sh syntax | Manual | `bash -n deploy/deploy.sh` | no error |
| eventbus package import | Manual | `PYTHONPATH=scripts python3 -c "import eventbus"` | no error |

## Risks

- **Risk:** `AUTOINCREMENT` adds a hidden `sqlite_sequence` table write per INSERT, increasing
  write latency by ~5 µs.
  **Mitigation:** Acceptable at current traffic; if throughput becomes a concern, switch to
  plain `INTEGER PRIMARY KEY` and enforce non-reuse via application logic in DLQ cleanup.
- **Risk:** `schemas/` copy in deploy.sh uses `cp -r` which overwrites production schemas
  unconditionally; a broken schema deployed to prod could break all `/publish` validation.
  **Mitigation:** Schema changes must go through a PR review gate; deploy.sh does not run
  automatically without human trigger.
