# Goal

Add `"sandbox_backend": service.sandbox_backend` to the `/health` response in `scripts/mcp/shell/server.py`, exposing the resolved sandbox state to health check consumers.

# Scope

- **File:** `scripts/mcp/shell/server.py`
- **Change:** Update `/health` endpoint return dict to include `sandbox_backend`
- **Out of scope:** Other endpoints, sandbox selection logic

# Assumptions

1. `ShellService.sandbox_backend` property is available (prerequisite: service.py change).
2. The `service` object is accessible in the FastAPI app context (either as a module-level singleton or via dependency injection).
3. Additive field change — existing health consumers won't break.
4. Example result: `{"status": "ok", "ready": true, "sandbox_backend": "none", "dependencies": {}}`

# Implementation

## Target file

`scripts/mcp/shell/server.py`

## Procedure

1. Check how `service` is accessed in `server.py` (module-level singleton via `build_service()` or dependency injection).
2. In the `health()` function, add `"sandbox_backend": <service_ref>.sandbox_backend` to the return dict.

## Method

Current health endpoint:
```python
@app.get("/health")
async def health() -> dict[str, object]:
    deps: dict[str, str] = {}
    ...
    return {"status": "ok", "ready": ready, "dependencies": deps, "details": {}}
```

After:
```python
return {
    "status": "ok",
    "ready": ready,
    "sandbox_backend": _service.sandbox_backend,  # or however service is referenced
    "dependencies": deps,
    "details": {},
}
```

The `_service` reference depends on how server.py manages the service instance.

## Details

- Position `sandbox_backend` after `ready` for logical grouping (status → ready → sandbox → deps).
- Must read `scripts/mcp/shell/server.py` fully before editing to confirm how service is accessed.
