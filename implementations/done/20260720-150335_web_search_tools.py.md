## Goal

Add a `browser_fetch` entry to `TOOL_LIST`, ported field-for-field from
`scripts/mcp_servers/browser/browser_tools.py` (39 lines, read in full), preserving its
`"config_dependent": True` marker exactly (per plan Assumption 3 — a test class currently locks
this field in and will be relocated to import from this module instead of `browser_tools.py`).

## Scope

`scripts/mcp_servers/web_search/web_search_tools.py` (current: 39 lines, one entry `search_web`).
**Out of scope**: `search_web`'s existing entry and `get_max_results_limit()` import — untouched.

## Assumptions

1. Source `browser_tools.py`'s `TOOL_LIST[0]` has: `name="browser_fetch"`, a description string
   (lines 14-18), `inputSchema` with `url` (required, string) and `max_response_kb` (optional,
   integer) properties, `"status": "production"`, and `"config_dependent": True`. All fields are
   ported verbatim — no schema changes.
2. `web_search_tools.py`'s existing `search_web` entry has `"status": "production"` but no
   `config_dependent` key at all (i.e. it's absent, not `False`) — this asymmetry between the two
   entries in the same `TOOL_LIST` is intentional and matches the plan's Assumption 3 (`search_web`
   is not "config dependent" in the sense the field describes — it works with an empty/default
   config — whereas `browser_fetch`'s allowlist can be empty, making it meaningfully
   config-dependent). Do not add `"config_dependent": False` to `search_web`'s entry — leave it
   absent, unchanged.

## Implementation

### Target file

`scripts/mcp_servers/web_search/web_search_tools.py`

### Procedure

1. Append a second dict literal to `TOOL_LIST` (after the `search_web` entry, before the closing
   `]`):
   ```
   {
       "name": "browser_fetch",
       "description": (
           "Fetch a URL and return its visible text content (read-only; no JavaScript "
           "execution, no interactive actions). Host must be in the configured domain "
           "allowlist."
       ),
       "inputSchema": {
           "type": "object",
           "properties": {
               "url": {
                   "type": "string",
                   "description": "Absolute http(s) URL to fetch (host must be allowlisted)",
               },
               "max_response_kb": {
                   "type": "integer",
                   "description": (
                       "Output size limit in KB (default: server-configured; caller "
                       "value is clamped to server maximum)"
                   ),
               },
           },
           "required": ["url"],
       },
       "status": "production",
       "config_dependent": True,
   },
   ```
   (Ported verbatim from `browser_tools.py` lines 12-38 — only the enclosing list append changes,
   no field values change.)

### Method

Pure data addition — no function/import changes needed (this module has no logic beyond the
`TOOL_LIST` constant and the pre-existing `get_max_results_limit()` import for `search_web`'s
`maximum` bound).

### Details

- `TOOL_LIST` becomes a 2-element list; `web_search_server.py`'s `build_tools_response(TOOL_LIST,
  "web_search")` (separate doc) then reports both tools under `server_key="web_search"` at
  `/v1/tools` — this is the concrete mechanism that changes `browser_fetch`'s routed server key from
  `"browser"` to `"web_search"`, which the `tests/test_runtime_tool_routing_integration.py` doc's
  fixture updates depend on.
- `TestBrowserToolsConfigDependentMigration` (in the test file, currently imports from
  `mcp_servers.browser.browser_tools`) is relocated by the companion test doc to import `TOOL_LIST`
  from this module instead — no behavior change to the assertions themselves, only the import path.
