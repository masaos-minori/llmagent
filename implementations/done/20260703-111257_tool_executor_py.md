## Goal

Change the plugin return-value length guard in `ToolExecutor.execute()` from `< 2` (minimum-length) to `!= 2` (strict equality), so that 3+-element tuples are rejected with a `ValueError` just like 1-element tuples.

## Scope

- In-Scope:
  - Change `len(result_raw) < _PLUGIN_RESULT_TUPLE_LENGTH` to `len(result_raw) != _PLUGIN_RESULT_TUPLE_LENGTH` on line 570 of `scripts/shared/tool_executor.py`
  - Update the error message on lines 572-574 from `"must return tuple[str, bool]"` to `"must return exactly tuple[str, bool] (2 elements)"`
  - Update the module-level comment on line 37 to make clear this is a strict-equality sentinel
- Out-of-Scope:
  - Registration-time validation in `plugin_registry.py` (already strict `!= tuple[str, bool]`)
  - MCP tool result validation
  - Cache logic
  - Any other file

## Assumptions

1. No production plugin currently returns a 3+-element tuple; the registration-time annotation check (`!= tuple[str, bool]`) would have already rejected such code.
2. Raising `ValueError` for the `!= 2` case is correct — the existing `< 2` path already raises `ValueError`, so the exception type is unchanged.
3. The single-line diff is sufficient; no structural refactor of `execute()` is needed.
4. The `_PLUGIN_RESULT_TUPLE_LENGTH` constant itself does not need to change value (still `2`) — only the comparison operator and comment change.
