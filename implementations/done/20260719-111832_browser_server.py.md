# Implementation procedure: `scripts/mcp_servers/browser/server.py` (new file)

Source plan: `plans/20260719-101501_plan.md`, Implementation step 1 (Design step 3, Affected
areas, Assumption 3). Depends on the paired `models.py`/`service.py`/`tools.py` docs.

No existing implementations doc under `implementations/` or `implementations/done/` matches
this file. Many `server.py`-named docs exist (github, rag_pipeline, mdq, `read_server.py`,
`write_server.py`, `delete_server.py`, `git_server.py`, `mcp_servers_server.py` [the base
class], etc.), all for different, already-existing servers/modules — none for the not-yet-
existing `scripts/mcp_servers/browser/` package. Flagged as checked, not a genuine overlap.

## Goal

Create `scripts/mcp_servers/browser/server.py` — the FastAPI app, `BrowserMCPServer(MCPServer)`
subclass, and endpoint wiring for the new read-only Browser MCP server, following the same
overall shape as `scripts/mcp_servers/shell/server.py` (170 lines, verified) but wiring
`attach_auth_middleware()` per the plan's Assumption 3 (Browser performs outbound network
fetches, a materially different risk profile than shell's local-command execution), following
the `git`/`cicd` precedent instead of shell's precedent.

## Scope

**In scope**
- `app = FastAPI(title="browser-mcp", ...)`.
- `attach_auth_middleware(app, _cfg.auth_token or "")` call, mirroring
  `scripts/mcp_servers/git/server.py:50`.
- Exception handlers: `BrowserAuthorizationError` → 403 (mirrors
  `shell/server.py:53-56`/`ShellAuthorizationError`), `BrowserValidationError` → 422 (mirrors
  `shell/server.py:59-64`), plus a handler for outbound-fetch failures — an `httpx.HTTPError`
  (or its `TimeoutException`/`ConnectError` subclasses) should map to a 502/504-style response
  rather than an unhandled 500, since these are expected external-network failure modes, not
  server bugs (no existing server has an equivalent handler to copy verbatim — `cicd`'s
  `service_github_actions.py` catches `httpx` errors and re-raises as typed `Cicd*Error`s
  instead of letting `httpx` exceptions reach FastAPI directly; Browser should follow that
  same re-raise-as-domain-exception pattern for consistency, meaning `service.py` should catch
  `httpx.HTTPError` and re-raise e.g. a new `BrowserFetchError(RuntimeError)` from `models.py`,
  mapped here to 502 — flag this as a scope note for the `models.py`/`service.py` docs to pick
  up if not already covered there; this doc's `server.py` only needs the exception-handler
  registration once that type exists).
- `GET /health` — trivial liveness check with **no external dependency / no network call at
  healthcheck time** (per plan Risks: avoid the same startup-stall class of issue attributed to
  `cicd`/`rag_pipeline`'s slow-dependency health checks). Mirror the shape of
  `make_health_response(deps, details)` (`shell/server.py:91-107`) with `deps = {}` always
  (no liveness dependency to probe) and `details = {"service": "browser-mcp"}`.
- `GET /v1/tools` returning `{"tools": [{**t, "server_key": "browser"} for t in TOOL_LIST]}`,
  mirroring `shell/server.py:120-125`.
- `POST /v1/call_tool` dispatching via `dispatch_tool(_service.get_dispatch_table(), req.name,
  req.args)` and audit-logging via `mcp_servers.audit._audit_log`, mirroring
  `shell/server.py:128-145` (target field for the audit log should be `req.args.get("url",
  "")[:80]`, analogous to shell's `req.args.get("command", "")[:80]`).
- `class BrowserMCPServer(MCPServer)`: `server_name = "browser-mcp"`, `server_version = "1.0.0"`,
  `http_port = <next available port>` (see Assumption 2 below — **not** 8015), `own_config_file
  = "browser_mcp_server.toml"`, `app_module = "mcp_servers.browser.server:app"`, `mcp_tools =
  TOOL_LIST`, and `async def dispatch(...)` delegating to the same dispatch helper used by
  `/v1/call_tool`.
- `if __name__ == "__main__":` block instantiating `BrowserMCPServer()` and calling
  `.run_http()`, mirroring `shell/server.py:168-170`.

**Out of scope**
- No per-tool business logic (lives in `service.py`).
- No new middleware beyond `attach_auth_middleware` (no rate-limiting, no CORS — none of the
  10 existing servers have either).

## Assumptions

1. Verified directly: `scripts/mcp_servers/server.py:77-98` (`attach_auth_middleware`) takes
   `(app: _FastAPIApp, token: str)`; when `token` is empty, auth is skipped but an
   `X-Request-Id` header is still injected. Called at module level immediately after `app =
   FastAPI(...)` in both `git/server.py:50` and (per plan Assumption 3) `cicd/server.py` — this
   doc follows that exact call-site placement.
2. **Port assignment — verified with a re-check, and drift found.** All 10 current MCP servers'
   `http_port` values (grepped directly from each `server.py`): web_search=8004,
   file_read=8005, github=8006, file_write=8007, file_delete=8008, shell=8009,
   rag_pipeline=8010, cicd=8012, mdq=8013, git=8014 (8011 is an unused gap, most likely the
   former sqlite-mcp server's port, per the plan's own Non-Goals section on sqlite-mcp removal —
   not reused here to avoid any ambiguity with the removed server's old port). The highest
   currently-used MCP-server port is **8014** (git). However, **`deploy/setup_services.sh:85`
   already assigns port 8015 to the Event Bus** (`# ── Event Bus (port 8015)
   ──`, confirmed by direct read), which is not an MCP server but does occupy the next
   sequential port. Therefore the correct next-available port for Browser MCP is **8016**, not
   8015 — picking 8015 would collide with the already-running Event Bus service. This is a
   verified drift from what a naive "take the next integer after the highest MCP-server port"
   reading of the plan's Design step 3 would produce; `http_port = 8016` is the value to use.
3. Verified: `shell/server.py`'s comment at line 4 documents its own port inline
   (`"""... (port 8009)."""`) — Browser's module docstring should do the same:
   `"""... (port 8016)."""`.
4. Verified: none of `shell`/`git`/`cicd`/`web_search`'s `server.py` register FastAPI
   `@app.on_event("startup")`/lifespan hooks — `_service` is built at import time via a
   module-level `_service = build_service(BrowserConfig.load())` call (mirrors
   `shell/server.py:44`), not lazily. Browser follows the same eager-init pattern (no async
   resource needs pre-warming per the paired `service.py` doc's per-request
   `httpx.AsyncClient` design).
5. `mcp_servers.audit._audit_log` and `mcp_servers.dispatch.{DispatchResult,
   _to_call_tool_response, dispatch_tool}` and `mcp_servers.health_response.make_health_response`
   and `mcp_servers.models.{CallToolRequest, CallToolResponse}` are all shared, unmodified
   imports — verified present and used identically by `shell/server.py:27-31` and
   `git/server.py:30-36`; no changes needed to any of these shared modules for this plan.

## Implementation

### Target file

`scripts/mcp_servers/browser/server.py` (new file).

### Procedure

1. Write the module docstring: purpose, port (8016, per Assumption 2), security bullets
   (domain allowlist fail-closed, IP-literal/loopback rejection, Bearer-token auth via
   `attach_auth_middleware`, read-only/no-JS scope), and endpoint list — mirror
   `shell/server.py:1-17`'s docstring shape.
2. Imports: `logging`, `time`, `typing.Any`; `fastapi.{FastAPI, Request}`;
   `fastapi.responses.JSONResponse`; `shared.formatters.fmt_kvlog`;
   `mcp_servers.audit._audit_log`; `mcp_servers.dispatch.{DispatchResult,
   _to_call_tool_response, dispatch_tool}`; `mcp_servers.health_response.make_health_response`;
   `mcp_servers.models.{CallToolRequest, CallToolResponse}`;
   `mcp_servers.server.{MCPServer, ToolArgs, attach_auth_middleware}`;
   `mcp_servers.browser.models.{BrowserAuthorizationError, BrowserConfig,
   BrowserValidationError}` (plus `BrowserFetchError` if introduced per the Scope note above);
   `mcp_servers.browser.service.{BrowserService, build_service}`;
   `mcp_servers.browser.tools.TOOL_LIST`.
3. `_cfg = BrowserConfig.load()`; `_service: BrowserService = build_service(_cfg)`.
4. `app = FastAPI(title="browser-mcp", version="1.0.0", description="Read-only Browser MCP
   server: page fetch + render-to-text")`.
5. `attach_auth_middleware(app, _cfg.auth_token or "")` (per Assumption 1).
6. `@app.exception_handler(BrowserAuthorizationError)` → 403 JSONResponse.
7. `@app.exception_handler(BrowserValidationError)` → 422 JSONResponse.
8. (If introduced) `@app.exception_handler(BrowserFetchError)` → 502 JSONResponse — for
   network/timeout failures surfaced as a domain exception from `service.py`.
9. `GET /health` returning `make_health_response({}, {"service": "browser-mcp"})` — no
   dependency probing (per Scope/plan Risks).
10. `GET /v1/tools` returning `{"tools": [{**t, "server_key": "browser"} for t in TOOL_LIST]}`.
11. `POST /v1/call_tool(req: CallToolRequest, request: Request) -> CallToolResponse`: dispatch
    via `dispatch_tool(_service.get_dispatch_table(), req.name, req.args)`, then
    `_audit_log(logger, session_id=..., request_id=..., action=req.name,
    target=req.args.get("url", "")[:80], outcome=r.outcome, server_key="browser")`.
12. `class BrowserMCPServer(MCPServer)` per Scope, with `async def dispatch(self, name, args) ->
    DispatchResult` delegating to the same dispatch helper as `/v1/call_tool`.
13. `if __name__ == "__main__": BrowserMCPServer().run_http()`.

### Method

New-file creation adapting `shell/server.py`'s structure, with the `git/cicd`-style
`attach_auth_middleware` addition per plan Assumption 3. No new abstractions.

### Details

No new public types beyond `BrowserMCPServer`. Health-check response shape must conform to the
canonical dict documented at `scripts/mcp_servers/server.py:141-172`
(`status`/`ready`/`liveness`/`restart_recommended`/`operator_action_required`/`dependencies`/
`details`) — `make_health_response` already produces this shape; Browser's `/health` handler
only needs to supply `deps` and `details`, not hand-construct the envelope.
