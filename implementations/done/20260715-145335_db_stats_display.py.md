# Implementation Procedure: db_stats_display.py

## Goal

Remove the dead `_db_stats()` method and relocate `_db_session_stats()` so it is
consumable by `cmd_session.py`'s `_SessionMixin`, since `db_stats_display.py`'s sole
current mixer (`_DbMixin` in `cmd_db.py`) is being deleted.

## Scope

### In scope
- Delete `_db_stats()` (current lines 18-27) — confirmed dead, no caller anywhere in
  `scripts/` or `tests/` besides its own definition.
- Move `_db_session_stats()` (current lines 29-38) so `cmd_session.py`'s
  `_SessionMixin` can call it — either:
  (a) keep `DbStatsDisplay` as a class here containing only `_db_session_stats`, and
      compose it into `_SessionMixin` in `cmd_session.py`, or
  (b) inline `_db_session_stats`'s body directly as a method on `_SessionMixin` in
      `cmd_session.py` and delete this file entirely.
  Either is acceptable per the plan ("implementer's choice, not behavior-affecting").

### Out of scope
- `DbMaintenanceService` (`agent/services/db_maintenance_service.py`) — reused
  unchanged; `.stats()` return shape (`result.sessions`, `result.messages`) is not
  touched.
- Any other `_db_*` display logic — this file only ever contained these two methods.

## Assumptions

- `_db_stats()` and `_db_session_stats()` are byte-for-byte identical in body
  (verified by reading the full 39-line file: both call `DbMaintenanceService().stats()`
  and `self._out.write_kv([("sessions", ...), ("messages", ...), ("target", "Session")])`).
  This confirms `_db_stats` was already dead duplication before this plan — no
  behavior is lost by keeping only `_db_session_stats`.
- `DbStatsDisplay` is composed only via `_DbMixin(MixinBase, DbHelpDisplay, DbStatsDisplay)`
  in `cmd_db.py` (verified: `grep -rn "DbStatsDisplay" scripts/` shows only
  `db_stats_display.py`'s own definition and `cmd_db.py`'s import/composition).

## Implementation

### Target file

`scripts/agent/commands/db_stats_display.py`

### Procedure — recommended approach (option a: keep the class, retarget the mixin)

1. Delete `_db_stats()` (lines 18-27) in full.
2. Rename the class docstring/comment context from "Provides stats display methods
   for /db subcommands" to "Provides the session-stats display method for
   `/session stats`", to reflect the new sole caller.
3. Leave `_db_session_stats()` (lines 29-38) and the `_out: OutputPort` type-hint
   attribute (line 16) unchanged.
4. In `cmd_session.py`, compose `DbStatsDisplay` into `_SessionMixin`'s base classes:
   ```python
   from agent.commands.db_stats_display import DbStatsDisplay

   class _SessionMixin(MixinBase, DbStatsDisplay):
       ...
   ```
   (This is the minimal-diff option; if the implementer prefers option (b) — inlining
   the method body directly into `cmd_session.py` and deleting this file — do that
   instead and skip this file's retention.)

### Method

Single dead-method deletion plus either (a) a docstring/consumer retarget (file
survives) or (b) full-file deletion with method inlining elsewhere (file removed).
Pick (a) unless the reviewer prefers a stricter one-class-per-purpose layout; (a) is
lower-risk (smaller diff, `_db_session_stats`'s exact source text is untouched).

### Details

- Whichever option is chosen, `_db_session_stats`'s method body, docstring, and
  return type (`None`) must not change — only its class/module residency.
- If option (a): update the module-level docstring at the top of the file
  (`"""stats_display.py — Database stats display logic."""`) to reflect the narrower
  scope, e.g. `"""db_stats_display.py — Session stats display logic (used by /session stats)."""`.
