# Implementation: Unify GitHub mutation-tool set in `scripts/agent/tool_policy.py`

Source plan: `plans/20260715-141245_plan.md` (Implementation step 1)

## Goal

Remove the locally duplicated `_API_WRITE_TOOLS` frozenset from
`scripts/agent/tool_policy.py` and replace it with the shared union
`GITHUB_WRITE_TOOLS | GITHUB_DANGEROUS_TOOLS` imported from
`scripts/shared/tool_constants.py`, so `tool_policy.py` and
`tool_approval.py` (see companion doc for `tool_approval.py`) classify
GitHub mutation tools from a single source of truth. No observable
behavior change — the union is numerically identical to the current
local set.

## Scope

**In scope**
- `scripts/agent/tool_policy.py`: import `GITHUB_DANGEROUS_TOOLS`,
  `GITHUB_WRITE_TOOLS` from `shared.tool_constants`; replace
  `_API_WRITE_TOOLS` with a module-level constant derived from the union;
  update the two call sites that reference it
  (`classify_operation_type`, `check_allowed_repo`).
- `tests/test_tool_policy.py`, `tests/test_tool_policy_comprehensive.py`:
  add/confirm a regression assertion that classification is unchanged
  after the switch (mitigates the plan's Risk #1 — a future edit to
  `shared/tool_constants.py` silently changing classification with no
  test catching it).

**Out of scope**
- `check_allowed_repo()`'s allowlist comparison logic (`f"{owner}/{repo}"
  in allowed`) — plan Assumption 2 / UNK-02 confirms this is already
  correct; no code change.
- `classify_risk()`, `_special_case_risk()`, `_escalate_for_path()`,
  `_escalate_for_github_branch()` — untouched.
- `scripts/agent/tool_approval.py` — covered by its own implementation
  doc (`20260715-154002_tool_approval.py.md`).

## Assumptions

1. `GITHUB_WRITE_TOOLS | GITHUB_DANGEROUS_TOOLS` in
   `scripts/shared/tool_constants.py` (lines 128-145) is numerically
   equal to the current `_API_WRITE_TOOLS` (9 tools:
   `github_push_files`, `github_create_or_update_file`,
   `github_delete_file`, `github_merge_pull_request`,
   `github_create_branch`, `github_create_pull_request`,
   `github_update_pull_request`, `github_create_issue`,
   `github_add_issue_comment`) — confirmed by direct comparison of both
   sets.
2. `agent → shared` import direction is already used in this file
   (`from shared.tool_constants import DELETE_TOOLS, SHELL_TOOLS,
   WRITE_TOOLS` at line 16) and is permitted by `.importlinter`.
3. Renaming `_API_WRITE_TOOLS` to a new name (e.g.
   `_GITHUB_MUTATION_TOOLS`) is safe: it is a module-private constant
   with 2 internal call sites in this file only (verified via `grep -n
   "_API_WRITE_TOOLS" scripts/agent/tool_policy.py`).

## Implementation

### Target file

`scripts/agent/tool_policy.py` (existing, 216 lines)

### Procedure

1. Update the import at line 16:
   ```python
   from shared.tool_constants import (
       DELETE_TOOLS,
       GITHUB_DANGEROUS_TOOLS,
       GITHUB_WRITE_TOOLS,
       SHELL_TOOLS,
       WRITE_TOOLS,
   )
   ```
2. Replace the `_API_WRITE_TOOLS` block (lines 23-35) with:
   ```python
   _GITHUB_MUTATION_TOOLS: frozenset[str] = GITHUB_WRITE_TOOLS | GITHUB_DANGEROUS_TOOLS
   ```
3. Update `classify_operation_type()` (line 54): replace
   `if tool_name in _API_WRITE_TOOLS:` with
   `if tool_name in _GITHUB_MUTATION_TOOLS:`.
4. Update `check_allowed_repo()` (line 181): replace
   `if tool_name not in _API_WRITE_TOOLS:` with
   `if tool_name not in _GITHUB_MUTATION_TOOLS:`.
5. Confirm no other reference to `_API_WRITE_TOOLS` remains:
   `grep -n "_API_WRITE_TOOLS" scripts/agent/tool_policy.py` → no matches.
6. Add a regression assertion in `tests/test_tool_policy.py` (new test in
   the existing `classify_operation_type` test class) that iterates
   `GITHUB_WRITE_TOOLS | GITHUB_DANGEROUS_TOOLS` and asserts every member
   returns `OperationType.API_WRITE`. Confirm
   `tests/test_tool_policy_comprehensive.py` still passes unchanged (it
   already exercises the affected functions; no new test needed there
   unless it duplicates set membership expectations that must be updated
   to reference the shared constants instead of a hardcoded literal
   list).

### Method

Minimal rename-and-import substitution; no control-flow changes. The
diff is confined to lines 16, 23-35, 54, 181 of `tool_policy.py`, plus
one new test in `tests/test_tool_policy.py`.

### Details

- Keep the constant name module-private (leading underscore) —
  `_GITHUB_MUTATION_TOOLS` is not part of the public API of this module.
- Do not export `_GITHUB_MUTATION_TOOLS` from `tool_policy.py` for reuse
  in `tool_approval.py`; `tool_approval.py` derives its own
  `_GITHUB_MUTATION_TOOLS` directly from `shared.tool_constants` (see
  companion doc) to avoid a new `tool_policy → tool_approval` (or
  reverse) coupling. Both modules already depend on `shared.tool_constants`
  independently, which is the intended dependency direction.
- No change to `deploy/deploy.sh` (no module added/removed).
