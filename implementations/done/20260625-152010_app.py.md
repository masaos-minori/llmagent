# Implementation: app.py — offset verification in hybrid subscribe (req #41)

## Goal

Verify that `read_offset()` is called correctly at the start of `subscribe()` and that no mid-stream `write_offset()` calls exist. Confirm the ack-only offset model is correctly wired.

## Scope

- `read_offset()` called at connect to restore `start_seq` for `consumer_id`
- No `write_offset()` anywhere in `subscribe()` body
- Offset advances only via `POST /events/{id}/ack`
- Reconnect semantics: consumer_id + seq > last_acked replays missed events

## Assumptions

- req #29 removed mid-stream checkpointing
- req #37 subscribe() skeleton does not include any write_offset calls

## Implementation

### Target file

`scripts/eventbus/app.py`

### Procedure

1. Verify `read_offset` is imported (already in offsets.py)
2. Verify subscribe() start block has `start_seq = read_offset(...)` when `consumer_id` is present
3. Grep for `write_offset` in subscribe — should be 0 results

### Method

Code verification + grep check. No edits required if req #35 is correct.

### Details

**Expected subscribe() start block:**
```python
start_seq = since_seq
if consumer_id and start_seq == 0:
    start_seq = read_offset(_cfg.offsets_dir, consumer_id)
```

**Key semantics:**
- `since_seq` is explicit client override — takes precedence over stored offset
- `consumer_id` only affects `start_seq` when `since_seq == 0` (default)
- On reconnect: client passes `consumer_id` without `since_seq` → auto-resume from last acked seq
- Offset NOT advanced in subscribe body — only via `POST /events/{id}/ack`

**ack endpoint (unchanged from req #29):**
```python

async def ack_event(event_id: str, consumer_id: str = Query(...)) -> dict[str, Any]:
    ...
    write_offset(_cfg.offsets_dir, consumer_id, row["seq"])
    ...
```
