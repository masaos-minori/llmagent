## Goal

Improve RAG consistency diagnostics so operators can move from detection to repair without manual database investigation by adding affected identifiers and actionable fix hints to the consistency check output.

## Scope

**In-Scope**:
- Add recommended actions for `fts_gap`, `fts_orphan_count`, and `orphan_vec_count`
- Expose affected `chunk_id`, `doc_id`, and URL where feasible in `RagConsistencyReport`
- Add fix hints to `/db consistency` output
- Document when to use FTS rebuild vs `ingester.py --force`

**Out-of-Scope**:
- Automatic destructive repair
- Replacing consistency checks with a background job

## Assumptions

1. `check_rag_consistency()` must remain read-only — no mutations
2. Affected identifier resolution is feasible for `fts_gap` and `orphan_vec_count` but may be expensive/ambiguous for `fts_orphan_count`
3. The `/db consistency` command output is generated by `cmd_db.py` using the report from `RagMaintenanceService.consistency()`

## Implementation

### Target file: scripts/db/maintenance.py

**Procedure**: Enhance RagConsistencyReport with affected identifiers, improve summarize_issues() repair guidance.

**Method**: Modify RagConsistencyReport dataclass and check_rag_consistency() function.

**Details**:
1. Add optional fields to RagConsistencyReport (nullable to maintain backward compatibility):
   - `affected_chunk_ids: list[int] | None = None` — affected chunk IDs for fts_gap
   - `affected_doc_ids: list[int] | None = None` — affected doc IDs for fts_gap or orphan_vec_count
   - `affected_urls: list[str] | None = None` — affected URLs for orphan_vec_count
2. Populate these fields in check_rag_consistency() where feasible:
   - For fts_gap: query chunks table for missing chunk_ids, limit to top 10
   - For orphan_vec_count: query chunks_vec for orphan entries, include doc_id and URL
3. Improve summarize_issues() repair guidance:
   - For `fts_gap`: add specific action — "Run `/db rebuild-fts`" (already present, verify)
   - For `fts_orphan_count`: clarify severity and add action — "Run `/db rebuild-fts` immediately; orphan FTS entries indicate data loss risk" (already present, verify)
   - For `orphan_vec_count`: specify which URLs/IDs are affected if available

### Target file: scripts/agent/commands/cmd_db.py

**Procedure**: Add fix hints to /db consistency output.

**Method**: Modify /db consistency command output formatting in cmd_db.py.

**Details**:
1. Update cmd_db.py to display actionable fix hints from the report
2. Include affected identifiers when available (chunk_ids, doc_ids, URLs)
3. Format output for operator readability

### Target file: docs/03_rag_05_configuration_and_operations.md

**Procedure**: Document repair flow in RAG ops docs.

**Method**: Add new section documenting FTS rebuild vs ingester.py --force decision tree.

**Details**:
1. Add clear guidance on when to use FTS rebuild vs `ingester.py --force`
2. Document the decision tree for operators:
   - Use `/db rebuild-fts` when fts_gap > 0 or fts_orphan_count > 0
   - Use `ingester.py --force` when orphan_vec_count > 0 or chunks table mismatch
