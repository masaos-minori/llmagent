# Implementation: scripts/mcp_servers/mdq/mdq_service.py (MdqConfig wiring, deny-all startup
warning, read-time authorization in `get_chunk`/`grep_docs`)

Source plan: `plans/20260719-205532_plan.md` ("Strengthen MDQ read-time authorization, deny-all
visibility, and config validation"), Design §1 (`get_chunk`/`grep_docs` halves), Design §2
(startup warning half), Design §3 (`MdqConfig` wiring half). Implementation steps Phase 1 item 2,
Phase 2 item 2, Phase 3 items 2-3.

This document combines four distinct changes to the same file (all in `MdqService`) into one
procedure document, per the task instructions allowing multiple Design sub-items that share one
target file to be combined:
1. Replace ad hoc `mdq_cfg.get(...)` reads in `__init__` with `MdqConfig(**mdq_cfg)` validation.
2. Add a startup warning log when `allowed_dirs` is empty.
3. Add read-time authorization to `get_chunk()`.
4. Add read-time authorization to `grep_docs()` (both explicit-`paths` and no-filter cases).

`search.py`'s authorization change (`_search_docs_structured`) and `db_grep.py`'s signature
extension are covered by their own separate implementation documents, since those are different
target files.

## Goal

`MdqService` fails fast on invalid config values, warns operators at startup when
`allowed_dirs` is empty (deny-all), and its `get_chunk`/`grep_docs` methods re-check
authorization against the live `allowed_dirs` config before returning indexed content — closing
the gap where content indexed under a previously-broader `allowed_dirs` remained visible after
the config was narrowed.

## Scope

**In scope:**
- `MdqService.__init__` (`mdq_service.py:50-94`): replace the sequence of `mdq_cfg.get(...)`
  calls (lines 59-81) and the manual `isinstance(self._allowed_dirs, list)` check
  (lines 84-88) with `MdqConfig(**mdq_cfg)` validation; add the empty-`allowed_dirs` warning
  log.
- `MdqService.get_chunk()` (`mdq_service.py:138-166`): extend the `SELECT` to also fetch
  `source_path`; authorize it; raise `MdqAuthorizationError` if denied.
- `MdqService.grep_docs()` (`mdq_service.py:326-361`): authorize `req.paths` when supplied
  (reject the whole call on any unauthorized explicit path); when not supplied, resolve the
  authorized subset of all indexed `source_path` values and pass that into `db_grep.grep_docs()`
  via the (extended, per companion `db_grep.py` doc) `req_paths` parameter.

**Out of scope:**
- `search_docs()` (`mdq_service.py:129-136`) itself is a thin wrapper that delegates to
  `search.py::search_docs()` — the actual filtering logic change lives in `search.py`
  (companion doc), not here. This file's `search_docs()` method body is unchanged.
- `outline()` (`mdq_service.py:168-251`) and `_validate_paths()`
  (`mdq_service.py:278-288`) — already authorize via `authorize_path()`, no gap, unchanged.
- `stats()` (`mdq_service.py:300-324`) — returns aggregate counts only, no per-path content;
  explicitly out of scope per the plan.
- `db_path` handling (`mdq_service.py:59`) — stays a plain `db_path or mdq_cfg.get(...)`
  expression, not part of `MdqConfig` (see companion `mdq_models.py` doc, Out of scope).

## Assumptions

1. `MdqConfig` (companion `mdq_models.py` doc) exists and is importable from
   `mcp_servers.mdq.mdq_models` before this change lands — both must land together or
   `MdqService` fails to import.
2. `get_chunk`'s current query (`mdq_service.py:147-150`) is
   `"SELECT heading, content, content_hash FROM chunks WHERE chunk_id = ?"` — it does not
   fetch `source_path` today (Assumption 3 in the plan). Adding it is a non-breaking additive
   change to the `SELECT` column list; `row["content_hash"]` remains unused/available as
   before (confirmed: `content_hash` is selected but never read in the current method body —
   leave that pre-existing oddity untouched, not in scope for this plan).
3. Per the plan's Design §1, `get_chunk` raises `MdqAuthorizationError` (not
   `MdqNotFoundError`) on an unauthorized chunk, and the exception message must NOT include
   the chunk's `source_path` (avoid leaking the unauthorized path in an error string) —
   contrast with `MdqNotFoundError(f"Chunk {req.chunk_id} not found")` (`mdq_service.py:152`)
   which already includes `chunk_id` (not a filesystem path, so no equivalent leak concern).
4. `grep_docs`'s current signature call to `db_grep.grep_docs()`
   (`mdq_service.py:350-358`) passes `req.paths or []` as the `req_paths` positional
   parameter. The companion `db_grep.py` doc's signature stays backward compatible with this
   call shape — this file passes an authorized/resolved path list into the *same* parameter
   position, not a new one.
5. Resolving "all indexed `source_path` values" for the no-filter case requires a new query
   (`SELECT DISTINCT source_path FROM chunks`) executed on the same connection already opened
   via `self._get_db_connection()` in `grep_docs()` (`mdq_service.py:348`) — no second
   connection needed.
6. The startup warning (Design §2) belongs in `MdqService.__init__`, scoped to the
   module-local `logger` (`mdq_service.py:44`), not the global `audit_security_defaults()`
   (explicitly out of scope per the plan's Scope section).

## Implementation

### Target file

`scripts/mcp_servers/mdq/mdq_service.py`

### Procedure

**1. Config validation wiring (`__init__`, lines 50-94):**

1. Add `from mcp_servers.mdq.mdq_models import MdqConfig` to the existing multi-line
   `from mcp_servers.mdq.mdq_models import (...)` block (`mdq_service.py:29-41`), keeping
   alphabetical/isort ordering.
2. Replace lines 59-81 (`self.db_path = ...` through `self.sqlite_busy_timeout = ...`) and
   the `isinstance` check at lines 84-88 with:
   ```
   cfg = MdqConfig(**mdq_cfg)   # raises pydantic.ValidationError on invalid values — do not catch
   self.db_path = db_path or mdq_cfg.get("db_path", "/opt/llm/db/mdq.sqlite")
   self._allowed_dirs = cfg.allowed_dirs
   self.include_globs = cfg.include_globs
   self.exclude_globs = cfg.exclude_globs
   self.max_snippet_chars = cfg.max_snippet_chars
   self.max_chunk_chars = cfg.max_chunk_chars
   self.max_file_chars = cfg.max_file_chars
   self.search_timeout_sec = cfg.search_timeout_sec
   self.enable_grep = cfg.enable_grep
   self.max_grep_matches = cfg.max_grep_matches
   self.max_chars_per_match = cfg.max_chars_per_match
   self.context_before = cfg.context_before
   self.context_after = cfg.context_after
   self.max_results_limit = cfg.max_results_limit
   self.max_chars_per_chunk = cfg.max_chars_per_chunk
   self.max_total_result_chars = cfg.max_total_result_chars
   self.max_outline_items = cfg.max_outline_items
   self.max_outline_depth = cfg.max_outline_depth
   self.sqlite_busy_timeout = cfg.sqlite_busy_timeout
   ```
   Do NOT wrap `MdqConfig(**mdq_cfg)` in a `try/except` that swallows `ValidationError` — the
   plan requires this to fail fast/crash startup, not degrade.
3. Do not touch the outer `try/except (FileNotFoundError, KeyError, TypeError): mdq_cfg = {}`
   block (lines 52-57) — per plan UNK-05, a missing/unparseable TOML file still falls back to
   `{}` (all-defaults, which passes `MdqConfig` validation); only invalid *values* within a
   loaded dict should raise.

**2. Deny-all startup warning (still in `__init__`, immediately after the block above):**

```
if not self._allowed_dirs:
    logger.warning(
        "MDQ allowed_dirs is empty — all path-based operations will be denied. "
        "Configure allowed_dirs in mdq_mcp_server.toml to enable indexing and path operations."
    )
```

**3. `get_chunk()` authorization (lines 138-166):**

1. Extend the `SELECT` at lines 147-150 to:
   ```
   "SELECT heading, content, content_hash, source_path FROM chunks WHERE chunk_id = ?"
   ```
2. Immediately after the existing `if row is None: raise MdqNotFoundError(...)` check
   (line 151-152), insert:
   ```
   if not authorize_path(Path(row["source_path"]), self.allowed_dirs):
       raise MdqAuthorizationError(f"Access denied: chunk {req.chunk_id} is not authorized")
   ```
   (message includes `chunk_id`, not `source_path` — see Assumption 3.)
3. Leave the rest of the method (`content` truncation logic, lines 153-164) unchanged.

**4. `grep_docs()` authorization (lines 326-361):**

1. After the existing `enable_grep`/regex-compile checks (lines 328-333), before the
   size-limit computation (lines 335-346), insert path authorization:
   ```
   if req.paths:
       for p in req.paths:
           if not authorize_path(Path(p), self.allowed_dirs):
               raise MdqAuthorizationError(f"Access denied: {p} is outside allowed directories")
       authorized_paths = req.paths
   else:
       conn_check = self._get_db_connection()
       try:
           all_paths = [
               row["source_path"]
               for row in conn_check.execute("SELECT DISTINCT source_path FROM chunks").fetchall()
           ]
       finally:
           conn_check.close()
       authorized_paths = [p for p in all_paths if authorize_path(Path(p), self.allowed_dirs)]
   ```
2. Replace the existing call at lines 350-358 (`grep_docs(conn, compiled, req.paths or [], ...)`)
   so it passes `authorized_paths` instead of `req.paths or []`.

### Method

Four sequential, independently-testable edits within one file: (a) config-field assignment
block replaced by a validated-object read, (b) one new warning log statement, (c) one extra
SELECT column plus one authorization check plus one raise, (d) an authorization pre-pass before
the existing `db_grep.grep_docs()` call. No method is removed; `get_chunk`/`grep_docs`'s
existing truncation/formatting logic is untouched apart from the inserted authorization gate.

### Details

- `authorize_path` is already imported (`mdq_service.py:17`) — no new import needed for (c)/(d).
- `Path` is already imported (`mdq_service.py:13`) — no new import needed.
- `MdqAuthorizationError` is already imported (`mdq_service.py:33`) — no new import needed.
- After step 1, confirm no remaining reference to the old manual `isinstance(self._allowed_dirs, list)`
  warning-and-default pattern — it is superseded by `MdqConfig`'s type validation, which raises
  rather than silently defaulting (a deliberate behavior tightening per the plan).
- Two `_get_db_connection()` calls now occur inside `grep_docs()` (the new `conn_check` in the
  no-filter branch, and the existing `conn` used for the main grep query) — these are separate,
  short-lived connections; do not attempt to reuse one connection object across both, since the
  authorization pre-pass must complete (and its connection close) before the main query opens
  its own connection, keeping the two responsibilities cleanly separated.
