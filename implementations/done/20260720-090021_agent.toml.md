# Implementation procedure: `config/agent.toml` (add `max_results` to `search_web`'s tool definition)

Source plan: `plans/20260719-202346_plan.md`, Implementation step Phase 2 / Design §2.

`config/agent.toml` has many prior implementation docs. Checked the three most recent by content:
`implementations/done/20260620-145430_agent.toml.md` and `implementations/done/20260719-105935_agent.toml.md`
address unrelated topics (verified via grep — neither mentions `max_results` or `search_web`'s
tool-definition block); `implementations/done/20260719-111834_agent.toml.md` registers
`browser_fetch` (`[mcp_servers.browser]`, its own `[[tool_definitions]]` block, its own safety
tier) and only *references* `search_web`'s existing block as a shape precedent to copy — it does
not modify `search_web`'s own definition. None adds `max_results` to `search_web`'s parameters.
Flagged as checked, not a genuine overlap — this is a new document.

## Goal

Add `max_results` (optional integer) to `search_web`'s `[tool_definitions.function.parameters]`
block in `config/agent.toml`, closing the verified schema divergence between this config and
`web_search_tools.py::TOOL_LIST`'s actual `inputSchema`.

## Scope

**In scope**
- `config/agent.toml`'s `search_web` `[[tool_definitions]]` block only (verified at lines 400-413:
  `[[tool_definitions]]` / `type = "function"` / `[tool_definitions.function]` `name = "search_web"`
  / `[tool_definitions.function.parameters]` `type = "object"`, `required = ["query"]` /
  `[tool_definitions.function.parameters.properties.query]` `type = "string"`).
- One new sub-table: `[tool_definitions.function.parameters.properties.max_results]` with
  `type = "integer"` and a description, **not** added to `required`.

**Out of scope**
- No change to `query`'s existing sub-table.
- No change to `tool_safety_tiers` (`search_web = "READ_ONLY"` at line 222, verified — already
  correct, needs no edit for this item).
- No change to the unrelated, already-removed top-level `web_search_max_results` /
  `RAGConfig.web_search_max_results` knob (verified at `agent.toml` lines 29-34: a NOTE explains it
  was removed 2026-07-17 because it was parsed but never read — **this is a different setting from
  the one being added here**. This item's `max_results` is a per-tool-definition LLM-facing
  parameter inside `[[tool_definitions]]`, not a top-level `RAGConfig` field; it is not affected by,
  and does not reintroduce, that removed knob. An implementer should not confuse the two or treat
  the NOTE as blocking this change).
- No change to `config/web_search_mcp_server.toml`'s own `default_max_results`/`max_results_limit`
  settings (those are the MCP server's own enforced limits per the NOTE above; unrelated file, out
  of scope here).

## Assumptions

1. Verified directly, `web_search_tools.py::TOOL_LIST` (lines 11-30): `max_results` is present in
   `inputSchema.properties` with `type = "integer"`, and `required` is `["query"]` only — i.e.
   `max_results` is optional in the MCP server's own accepted schema. `agent.toml`'s LLM-facing
   definition should match this shape exactly (optional integer, not required) so the LLM-facing
   contract and the actual server-accepted schema agree.
2. Verified directly, `config/agent.toml` lines 400-413: `search_web`'s current
   `[[tool_definitions]]` block has only `query` — no `max_results` property exists yet, confirming
   the divergence the plan's Design §2 describes is still live (not already fixed by any other
   landed work).
3. This is additive/backward-compatible: existing callers that omit `max_results` are unaffected
   (it is optional both in `TOOL_LIST` and in the new `agent.toml` block); `SearchRequest` (Pydantic,
   `web_search_models.py`) already defaults the field per the plan's Assumption 4 — confirming
   Pydantic-side default behavior is unaffected by this LLM-schema-only addition.

## Implementation

### Target file

`config/agent.toml`, `search_web`'s `[[tool_definitions]]` block (currently lines 400-413).

### Procedure

1. Locate the `[tool_definitions.function.parameters.properties.query]` sub-table (line ~410-413
   currently).
2. Immediately after it (before the next `[[tool_definitions]]` block, i.e. before `list_directory`'s
   entry), add a new sub-table:
   `[tool_definitions.function.parameters.properties.max_results]` with
   `type = "integer"` and a `description` (Japanese, matching the file's existing convention for
   other tool parameter descriptions, e.g. `"取得する検索結果の最大件数（省略時はサーバー側の既定値）"`
   or similar — exact wording is an implementation-time judgment call, should read naturally next
   to `query`'s existing Japanese description).
3. Do **not** add `max_results` to the `required = ["query"]` array — it stays optional.

### Method

Pseudocode (TOML content sketch, not applied — documentation only):

```toml
[tool_definitions.function.parameters.properties.query]
type        = "string"
description = "検索クエリ文字列"

[tool_definitions.function.parameters.properties.max_results]
type        = "integer"
description = "取得する検索結果の最大件数（省略可、既定値はサーバー側設定に従う）"
```

### Details

- `required` stays `["query"]` — unchanged array, confirming `max_results` is optional at both the
  MCP server layer (`TOOL_LIST`) and the LLM-facing layer (`agent.toml`) after this fix.
- TOML table ordering: place `max_results` after `query` (matches `TOOL_LIST`'s own property
  ordering, `query` then `max_results`, for readability/consistency — not functionally required
  since TOML tables are unordered maps, but keeps the two schemas visually parallel for future
  reviewers doing a side-by-side diff).
- This is the one behavior-adjacent change in the whole plan (per the plan's own Risks table) —
  implement it as an independently revertable, separately committable change (Phase 2 in the
  plan's Implementation Steps), not bundled with the comment-only Phase 1 items.
- No `deploy/deploy.sh` change needed — this is a config content edit, not a new/removed file under
  `scripts/`; the existing whole-file config deploy/copy step picks it up.
