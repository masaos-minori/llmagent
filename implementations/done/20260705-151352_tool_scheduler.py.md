# Implementation: agent/tool_scheduler.py — Serialization metadata extension

## Goal

Add `resource_scope`, `is_write`, `requires_serial`, `scheduling_decision` fields to `_SerializationEvent` and add a `serialize_within` flag to resource-scope groups in `_GroupMetadata.concurrent_groups`, so tool_runner.py can distinguish groups that must run sequentially from those that can be gathered.

## Scope

**In**: Extend `_SerializationEvent` with four new fields. Add `serialize_within: bool` to group metadata. Populate new fields in `build_execution_groups()`.

**Out**: Changes to `tool_runner.py`, test files, documentation.

## Assumptions

1. `_SerializationEvent` is a `@dataclass`; adding fields with defaults is backward-compatible.
2. `_GroupMetadata.concurrent_groups` is currently `list[list[list[dict]]]`. Adding a parallel `serialize_flags` list (or switching to a named structure) is the cleanest way to carry the flag.
3. Resource-scope write groups (`resource_groups[scope]`) are the only groups requiring `serialize_within=True`. Serial barrier and write-first groups are already single-element or conservative.
4. The `scheduling_decision` field is a human-readable classification string: `"serial_barrier"`, `"resource_scope"`, `"write_first"`, `"parallel"`.

## Implementation

### Target file
`scripts/agent/tool_scheduler.py`

### Procedure
1. Add new fields to `_SerializationEvent`.
2. Add `ScheduledBatch` dataclass to replace `list[list[dict]]` in `concurrent_groups`.
3. Update `_GroupMetadata.concurrent_groups` type to `list[ScheduledBatch]`.
4. Populate new fields in `build_execution_groups()`.

### Method

**New `_SerializationEvent` fields:**
```python
@dataclass
class _SerializationEvent:
    trigger_tool: str
    reason: str
    tools_count: int
    resource_scope: str = ""          # NEW: scope key or "" if not applicable
    is_write: bool = False            # NEW: True if write/delete op
    requires_serial: bool = False     # NEW: True if requires_serial=True
    scheduling_decision: str = ""     # NEW: "serial_barrier"|"resource_scope"|"write_first"|"parallel"
```

**New `ScheduledBatch` dataclass:**
```python
@dataclass
class ScheduledBatch:
    groups: list[list[dict]]   # groups to execute concurrently (each group's tools run together)
    serialize_within: bool = False  # True: run tools within each group sequentially, not gathered
```

**Updated `_GroupMetadata`:**
```python
@dataclass
class _GroupMetadata:
    total_tools: int
    total_groups: int
    serialization_events: list[_SerializationEvent] = field(default_factory=list)
    concurrent_groups: list[ScheduledBatch] = field(default_factory=list)  # CHANGED type
```

**`build_execution_groups()` changes:**
- When creating `ScheduledBatch` for resource-scope groups: `serialize_within=True`.
- When creating `ScheduledBatch` for write-first and parallel: `serialize_within=False`.
- Populate `_SerializationEvent.resource_scope`, `is_write`, `requires_serial`, `scheduling_decision` for each event.

### Details

Serial barrier events:
```python
metadata.serialization_events.append(_SerializationEvent(
    trigger_tool=name,
    reason="requires_serial",
    tools_count=1,
    requires_serial=True,
    is_write=True,       # shell/serial tools are always write-intent
    scheduling_decision="serial_barrier",
))
```

Resource scope events:
```python
meta = tool_meta.get(scope_tcs[0]["function"]["name"])
metadata.serialization_events.append(_SerializationEvent(
    trigger_tool=trigger,
    reason="resource_scope_conflict",
    tools_count=len(scope_tcs),
    resource_scope=scope,
    is_write=True,
    scheduling_decision="resource_scope",
))
```

Write-first events:
```python
metadata.serialization_events.append(_SerializationEvent(
    trigger_tool=trigger,
    reason="is_write_overlap",
    tools_count=len(write_first),
    is_write=True,
    scheduling_decision="write_first",
))
```

`concurrent_groups` construction:
```python
cgr: list[ScheduledBatch] = []
for tc in serial_barrier:
    cgr.append(ScheduledBatch(groups=[[tc]], serialize_within=False))  # single tool, no inner concurrency needed
if write_first:
    cgr.append(ScheduledBatch(groups=[write_first], serialize_within=False))  # writes without scope: concurrent within batch is OK
if concurrent_batch_groups:
    # concurrent_batch_groups = list of scope groups + parallel group
    # serialize_within=True for scope groups; False for parallel
    # Package them: each scope group in its own ScheduledBatch with serialize_within=True,
    # then the parallel group in its own ScheduledBatch
    for scope_tcs in resource_groups.values():
        cgr.append(ScheduledBatch(groups=[scope_tcs], serialize_within=True))
    if parallel:
        cgr.append(ScheduledBatch(groups=[parallel], serialize_within=False))
```

Wait — this breaks the "different scopes run concurrently" property. The original design put all scope groups + parallel in ONE concurrent batch. With the above, each scope gets its own ScheduledBatch and they run sequentially. That's too conservative.

**Revised approach**: Keep different scopes concurrent but serialize WITHIN each scope's group:
```python
# concurrent_batch: all scope groups + parallel, all run as concurrent group-level tasks
# But each scope group needs serialize_within=True
# Solution: ScheduledBatch.serialize_within applies per-group within the batch
```

Change `ScheduledBatch` to carry per-group serialize flags:
```python
@dataclass
class ScheduledBatch:
    groups: list[list[dict]]
    serialize_flags: list[bool]  # parallel to groups; True = run tools sequentially within group
```

Then in `build_execution_groups()`:
```python
if concurrent_batch or parallel:
    groups_list = []
    flags_list = []
    for scope_tcs in resource_groups.values():
        groups_list.append(scope_tcs)
        flags_list.append(True)   # serialize within scope group
    if parallel:
        groups_list.append(parallel)
        flags_list.append(False)  # gather reads
    cgr.append(ScheduledBatch(groups=groups_list, serialize_flags=flags_list))
```

This preserves concurrent scope execution while serializing within each scope.
