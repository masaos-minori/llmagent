## Goal

Create `scripts/mcp_servers/browser/tools.py` — the `TOOL_LIST` MCP tool-schema export for the
single `browser_fetch` tool — following the exact `inputSchema`-dict format used by every other
server's `tools.py`, verified directly against `scripts/mcp_servers/shell/tools.py` (57 lines).

## Scope

**In scope**
- `TOOL_LIST: list[dict[str, Any]]` containing exactly one entry: `browser_fetch`, with
  `inputSchema.properties` for `url` (required) and `max_response_kb` (optional).
- `"status": "production"`, `"requires_config": True` metadata fields, matching every entry in
  `shell/tools.py` (lines 11-56).

**Out of scope**
- No `[[tool_definitions]]` TOML block (that is `config/agent.toml`, a separate file/doc — see
  the paired `agent.toml` doc).
- No `tool_safety_tiers`/`shared/tool_constants.py` registration (separate docs).

## Assumptions

1. Verified: `shell/tools.py:11-57` defines `TOOL_LIST` as a module-level `list[dict[str, Any]]`
   with one dict per tool: `name`, `description`, `inputSchema` (JSON-Schema-shaped:
   `type: "object"`, `properties: {...}`, `required: [...]`), `status`, `requires_config`. This
   is the canonical shape; Browser's `tools.py` follows it exactly for its one tool.
2. `shell_run`'s `max_output_kb` property (`shell/tools.py:43-46`) is documented as `"type":
   "integer"` with a description noting the default and that the server enforces a ceiling —
   Browser's `max_response_kb` property should follow the identical description style: "Output
   size limit in KB (default: server-configured; caller value is clamped to server maximum)".
3. `shell_run`'s `argv` property demonstrates a nested-array-items shape
   (`shell/tools.py:26-30`) — not needed for Browser (no array-typed parameters), noted only to
   confirm the schema-dict conventions used project-wide (plain `{"type": ..., "description":
   ...}` per property, `"required": [...]` listing only the mandatory top-level keys).

## Implementation

### Target file

`scripts/mcp_servers/browser/tools.py` (new file).

### Procedure

1. Create the file with the same header/import shape as `shell/tools.py:1-9`
   (`from __future__ import annotations`, `from typing import Any`).
2. Define:
   ```python
   TOOL_LIST: list[dict[str, Any]] = [
       {
           "name": "browser_fetch",
           "description": (
               "Fetch a URL and return its visible text content (read-only; no JavaScript "
               "execution, no interactive actions). Host must be in the configured domain "
               "allowlist."
           ),
           "inputSchema": {
               "type": "object",
               "properties": {
                   "url": {
                       "type": "string",
                       "description": "Absolute http(s) URL to fetch (host must be allowlisted)",
                   },
                   "max_response_kb": {
                       "type": "integer",
                       "description": (
                           "Output size limit in KB (default: server-configured; caller "
                           "value is clamped to server maximum)"
                       ),
                   },
               },
               "required": ["url"],
           },
           "status": "production",
           "requires_config": True,
       },
   ]
   ```
3. Confirm the exact wording chosen for `description` does not contradict the Japanese-language
   `[[tool_definitions]]` description written in `config/agent.toml` (see the paired `agent.toml`
   doc) — the two descriptions serve different audiences (`tools.py`'s is the MCP-protocol-level
   schema description; `agent.toml`'s `[[tool_definitions]].function.description` is what the LLM
   sees) and are not required to be identical, but should not describe contradictory scopes
   (e.g. one must not imply interactive browsing).

### Method

New-file creation; single dict-literal export, no logic, no classes.

### Details

No new types beyond the existing `list[dict[str, Any]]` convention. `status: "production"`
should only be used if the implementation is validated end-to-end before deploy (per the
existing convention across `shell`/`git`/`web_search` — all use `"production"` once merged);
implementers may use `"experimental"` initially if Browser MCP ships behind an empty
`allowed_domains` default and is not yet operator-configured — confirm the project's convention
for this distinction at implementation time by checking whether any existing `tools.py` entry
uses a value other than `"production"` (not found during this investigation — all sampled
`tools.py` files use `"production"` uniformly, so default to `"production"` unless a reviewer
requests otherwise).
