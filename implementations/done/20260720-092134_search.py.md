# Implementation: scripts/mcp_servers/mdq/search.py (effective SQL result limit + honest
matched/shown count reporting)

Source plan: `plans/20260719-210826_plan.md` ("Fix MDQ freshness detection and search result
correctness"), Design ┬¦2 ("Effective SQL limit") and Design ┬¦3 ("Honest / exact count
reporting"), Implementation steps 2 and 3.

**Note:** `search.py` already has a companion implementation document
(`implementations/20260720-091517_search.py.md`) from a sibling design cycle on
`plans/20260719-205532_plan.md` (requirement 26: read-time authorization filtering in
`_search_docs_structured()` ŌĆö dropping unauthorized rows before the `total`/count is computed).
That document is a different, but interacting, concern: it inserts an `authorize_path()` filter
step between the FTS5 row fetch and the `results = fts_results` assignment. This document's
changes (effective SQL `LIMIT`, matched/shown count split) sit around that same code region and
must be composed with it, not conflict with it ŌĆö see "Interaction with the authorization filter"
in Details below. This is a NEW, ADDITIONAL document covering this plan's distinct concern
(freshness/count correctness), combining Design ┬¦2 and ┬¦3 into one document per the task
instructions since both target the same file.

## Goal

1. Make `_search_docs_structured()` cap the SQL `LIMIT` itself (`effective_limit = min(req.limit
   or 10, service.max_results_limit)`), so a request with a large `limit` and no
   `max_results_limit` cannot bypass the config cap by having the database return an unbounded
   row set before any Python-side truncation happens.
2. Replace `SearchResultResult`'s single `total: int` field (== `len(results)`, i.e. the
   post-`LIMIT` row count mislabeled as a total) with `matched_count` (exact count of all rows
   matching the query, no `LIMIT`) and `shown_count` (`len(results)` after `effective_limit`), so
   `search_docs()`'s header text never claims an exact "found" total that is actually just the
   retrieved/limited count.

## Scope

**In scope:**

    populate `matched_count`.
  - Return `SearchResultResult(query=req.query, results=results, matched_count=matched_count,
    shown_count=len(results))` instead of the current `total=len(results)` (line 143). Also
    update the early-return empty-query branch (line 89:
    `return SearchResultResult(query=req.query, results=[], total=0)`) to the new field shape.
- `search_docs()` (`search.py:27-81`): rework the `original_total`-based header/truncation
  wording (lines 42-81) to use `matched_count`/`shown_count` instead of `total`, per Design ┬¦3's
  guidance: prefer honest "matched"/"shown" wording; only claim an exact match count when one was
  computed (which, per this document's Option A implementation, is always ŌĆö see Details for the
  Option B fallback path if COUNT profiling proves too costly).

**Out of scope:**
- `_build_search_where()` (`search.py:146-164`) ŌĆö unchanged; both the main query and the new
  COUNT query reuse its output as-is (same `where_clauses`/`params`).
- The `sqlite3.OperationalError`/`sqlite3.Error` exception handling (`search.py:131-139`) ŌĆö

- The read-time authorization filter itself (companion document
  `20260720-091517_search.py.md`) ŌĆö not re-specified here; see Details for how the two compose.
- `db_grep.py`, `mdq_service.py`'s `get_chunk`/`grep_docs` ŌĆö unrelated files/methods.

## Assumptions

1. Direct read of the current file confirms:
   - Line 93: `limit = getattr(req, "limit", 10) or 10` ŌĆö used unbounded in the SQL `LIMIT ?` at
     line 111 (`params + [limit]`). No reference anywhere to `service.max_results_limit` inside
     `_search_docs_structured()` today ŌĆö confirming the plan's stated bug: the SQL layer never
     sees the config cap.
   - Line 143: `return SearchResultResult(query=req.query, results=results, total=len(results))`
     ŌĆö the sole production call of `SearchResultResult(...)` (plus the empty-query early return
     at line 89) confirming this is a 2-call-site change.
   - `search_docs()`, lines 42-81, computes `original_total = result["total"]` (line 42) and
     uses it in three places: the truncation check (`if original_total > max_results:`, line 58),
     the truncation message (`f"[Truncated ŌĆö {original_total} results found, ..."`, line 75),
     and both non-truncated header lines (`f"Search results for: {req.query!r} ({original_total}
     found)"`, lines 64 and 81). All four must be updated to reference `matched_count`.
2. `SearchDocsRequest.max_results_limit` (per the plan's UNK-02 recommendation, "keep") continues
   to exist and continues to act as a secondary, tighter *display* cap applied in `search_docs()`
   on top of the now-SQL-bounded `effective_limit` ŌĆö i.e. `search_docs()`'s existing
   `max_results` computation (lines 43-49: `min(request_results, config_results)` where
   `request_results = req.max_results_limit`) is retained unchanged in shape; it now operates on
   an already-`effective_limit`-bounded `result["results"]` list instead of an unbounded one.
3. `service.max_results_limit` (an `int` attribute on `MdqService`, set from config in
   `__init__`) is already accessible from `_search_docs_structured()` since `service: MdqService`

4. Per plan Assumption 5 / UNK-03, FTS5 `COUNT(*)` over the same `MATCH` predicate (no `LIMIT`)
   is cheap enough at MDQ's expected corpus size to justify computing `matched_count` exactly
   (Option A) on every call, rather than dropping to honest-wording-only (Option B). This
   document implements Option A; if implementation-time profiling shows a meaningful regression,
   fall back to Option B per the plan's fallback note (Details below) without a call-site

## Implementation

### Target file

`scripts/mcp_servers/mdq/search.py`

### Procedure

**1. Effective SQL limit (replace line 93 and its use at line 111):**

```
effective_limit = min(getattr(req, "limit", 10) or 10, service.max_results_limit)
```

Use `effective_limit` in place of `limit` in the `LIMIT ?` query parameter list (`params +
[effective_limit]`, replacing `params + [limit]` at line 111).

**2. Exact matched-count query (add after the main FTS5 query, before or after the row-mapping
list comprehension at lines 114-127 ŌĆö either ordering is fine since they are independent
queries on the same open connection):**

```
matched_count_row = conn.execute(
    f"""SELECT COUNT(*) as cnt
        FROM chunks_fts f
        JOIN chunks c ON f.rowid = c.rowid
        WHERE {where_clause}""",
    params,
).fetchone()
matched_count = matched_count_row["cnt"] if matched_count_row is not None else 0
```

Note: reuses the same `where_clause`/`params` already built by `_build_search_where()` (line 97)
for the main query ŌĆö no new WHERE-building logic.

**3. Update `SearchResultResult` construction (two call sites):**

- Empty-query early return (line 89):
  ```
  return SearchResultResult(query=req.query, results=[], matched_count=0, shown_count=0)
  ```
- End-of-function return (line 143):
  ```
  return SearchResultResult(
      query=req.query,
      results=results,
      matched_count=matched_count,
      shown_count=len(results),
  )
  ```

**4. Rework `search_docs()`'s header/count wording (lines 38-81):**

Replace every reference to `result["total"]`/`original_total` with the new fields:
```
matched_count = result["matched_count"]
shown_count = result["shown_count"]
```
Truncation/header decision:
- If `matched_count == shown_count` (no rows were dropped by `effective_limit` or the secondary
  `max_results_limit` cap): keep the simple existing phrasing style, e.g.
  `f"Search results for: {req.query!r} ({matched_count} found)"` ŌĆö accurate as-is since nothing
  was hidden.
- If `matched_count != shown_count`: use honest wording that does not conflate the two, e.g.
  `f"Search results for: {req.query!r} ({matched_count} matched, {shown_count} shown)"`, and keep
  the existing `[Truncated ŌĆö ...]` trailer but phrase it in terms of `matched_count`/`shown_count`
  rather than the old `original_total`/`shown` pair (the old `shown` local variable at line 71,
  computed as `len(lines) - 1`, is the same concept as the new `shown_count`, but note: the
  *char-limit* truncation path can produce a `shown` value smaller than `result["shown_count"]`
  if the char budget cuts off before all `effective_limit`/`max_results_limit`-bounded rows are
  rendered ŌĆö keep both concepts distinct in the message, e.g. distinguish "results shown after
  count/size limits" from "results rendered before the character budget ran out" if the two
  diverge, to avoid reintroducing a different flavor of the same honesty bug this plan fixes).
  Exact wording is an implementation-time detail (per plan Design ┬¦3); the binding requirement is
  that the header must never claim an exact "found" total that is actually the retrieved/limited
  count.

### Method

Three composable edits to `_search_docs_structured()` (SQL limit change, one additional COUNT
query, two updated `SearchResultResult` construction call sites) plus a wording-only rework of
`search_docs()`'s existing truncation logic (no new control-flow branches beyond what already
exists ŌĆö same `if original_total/matched_count > max_results:` shape, just renamed/re-sourced
variables). No function is renamed, added, or removed.

### Details

- **Interaction with the authorization filter (companion document
  `20260720-091517_search.py.md`):** that document inserts an `authorize_path()` filtering step
  between the row-mapping list comprehension (`fts_results = [...]`) and the
  `results = fts_results` assignment, changing it to
  `results = [item for item in fts_results if authorize_path(...)]`. If both this document's and
  that document's changes land together, `matched_count` (this document, from the exact COUNT
  query with no `LIMIT`) and `shown_count` (`len(results)` after both `effective_limit` AND the
  authorization filter) will legitimately diverge even when no config-cap truncation occurred ŌĆö
  this is consistent with this plan's intent (the header should never overclaim), so no special
  handling is needed: `matched_count` measures "rows satisfying the SQL predicate";
  `shown_count` measures "rows actually returned to the caller" for any reason (limit or
  authorization). Do not attempt to make `matched_count` reflect authorization ŌĆö per the sibling
  document's Assumption 2, the count of hidden (unauthorized) results must not be surfaced.
- Add the new COUNT query inside the same `try` block (lines 96-141) so `sqlite3.OperationalError`
  /`sqlite3.Error` handling already in place also covers it ŌĆö do not wrap it in a separate
  `try`/`except`.
- **Option B fallback** (only if Option A's extra COUNT query is later found to add meaningful
  latency): drop `matched_count` entirely, keep only `shown_count` plus a `possibly_more: bool`
  field (`True` if the un-limited row count could exceed what was fetched ŌĆö approximated by
  checking whether the main query returned exactly `effective_limit` rows, a cheap signal that
  avoids a second query). This is a same-shape fallback ŌĆö the call site in `search_docs()` would
  read `possibly_more` instead of comparing `matched_count != shown_count`, no other structural
  change required. Not implemented in this document's primary procedure; documented here so the
  implementer can pivot without a design detour if profiling shows a problem.
- `getattr(req, "limit", 10) or 10` is retained verbatim (not changed to `req.limit`) since
  `req` is typed as `SearchDocsRequest` which does define `limit: int | None = 10` ŌĆö the
  `getattr` with a default is defensive but harmless; not in scope to simplify here.
