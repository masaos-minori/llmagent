# Implementation: app.py — update requeue endpoint for dlq_requeue_count (req #28)

## Goal

Update `POST /dlq/{event_id}/requeue` to increment `dlq_requeue_count` instead of `retry_count`, and update `GET /dlq` to expose the new delivery state fields in its response.

## Scope

- Update `dlq_requeue` handler: `retry_count + 1` → `dlq_requeue_count + 1`
- Update `dlq_list` SELECT to include `delivery_failure_count` and `dlq_requeue_count`
- Remove `retry_count` from the DLQ list response (deprecated)

## Assumptions

- req #28 schema.sql is applied first
- `dlq_requeue_count` column exists in the events table
- Existing tests that check `retry_count == 3` after requeue will need updating (see test_eventbus_dlq.py doc)

## Implementation

### Target file

`scripts/eventbus/app.py`

### Procedure

1. Update `dlq_requeue` SQL: change `retry_count = retry_count + 1` to `dlq_requeue_count = dlq_requeue_count + 1`
2. Update `dlq_list` SELECT to include new fields

### Method

Edit two existing handler functions.

### Details

**Current `dlq_requeue` SQL (line ~230):**
```python
"UPDATE events SET retry_count = retry_count + 1, dlq_at = NULL WHERE event_id = ?",
```

**Target:**
```python
"UPDATE events SET dlq_requeue_count = dlq_requeue_count + 1, dlq_at = NULL WHERE event_id = ?",
```

**Current `dlq_list` SELECT (line ~221):**
```python
"SELECT seq, event_id, topic, producer, published_at, retry_count, dlq_at"
" FROM events WHERE dlq_at IS NOT NULL ORDER BY seq"
```

**Target:**
```python
"SELECT seq, event_id, topic, producer, published_at,"
" delivery_failure_count, dlq_requeue_count, dlq_at"
" FROM events WHERE dlq_at IS NOT NULL ORDER BY seq"
```

This removes `retry_count` from the DLQ list response. If backward compatibility is needed, keep `retry_count` in the SELECT and filter it in `_row_to_dict` or similar. For a fresh design (no external consumers relying on `retry_count` in DLQ response), removal is clean.
