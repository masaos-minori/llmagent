# Implementation: scripts/mcp_servers/mdq/search.py (read-time authorization filtering in
`_search_docs_structured`)

Source plan: `plans/20260719-205532_plan.md`, Design §1 ("Read-time authorization"), first
bullet. Implementation steps Phase 3, item 1.

## Goal

`search_docs` must not return `SearchResultItem` rows whose `source_path` is outside the
*current* `allowed_dirs` config, so narrowing `allowed_dirs` after indexing immediately hides
previously-indexed content from search results without requiring a DB migration.

## Scope

**In scope:**
- `_search_docs_structured()` (`search.py:84-143`): filter `fts_results` by
  `authorize_path()` before the `results = fts_results` assignment (line 129), so the
  `total` count (`len(results)`, line 143) reflects only authorized rows.

**Out of scope:**
- `search_docs()` (`search.py:27-81`), the public async entry point — unchanged; it already
  operates purely on whatever `result["results"]`/`result["total"]` it receives from
  `_search_docs_structured()`, with no direct DB/path access of its own.
- `_build_search_where()` (`search.py:146-164`) — unrelated, builds the SQL `WHERE` clause,
  no path-authorization concern.
- The `sqlite3.OperationalError`/`sqlite3.Error` exception handling
  (`search.py:131-139`) — unchanged.
- `db_grep.py`'s authorization support and `mdq_service.py`'s `get_chunk`/`grep_docs`
  authorization — covered by their own separate implementation documents.

## Assumptions

1. Filtering must happen **before** truncation/size-limit math runs, per the plan's
   Assumption 2: `search_docs`'s existing size/char truncation logic in `search.py` runs on
   the already-filtered result set — otherwise reported "total" counts would include
   unauthorized (invisible) rows and read as inconsistent to the caller. Concretely, this
   means the filter must be applied inside `_search_docs_structured()` (which computes
   `total=len(results)`, consumed by `search_docs()`'s own truncation logic at
   `search.py:42-81`), not inside `search_docs()` itself.
2. Unauthorized rows are silently dropped, not counted anywhere in the response (matches the
   plan's Risk item: "no count of hidden results is surfaced, by design" — fail-closed,
   no metadata leak about the existence of unauthorized matches).
3. `service.allowed_dirs` (the `MdqService.allowed_dirs` property, already defined at
   `mdq_service.py:96-99`) is the source of truth for the current allowlist — `search.py`
   already receives `service: MdqService` as its first parameter in both
   `_search_docs_structured` and `search_docs`, so no new parameter threading is needed.
4. `authorize_path()` (`scripts/mcp_servers/mdq/auth.py:11-38`) takes a `Path` and
   `list[str]`, returning `bool`; it is not currently imported in `search.py` (confirmed:
   `search.py`'s imports at lines 9-19 include only `asyncio`, `logging`, `sqlite3`,
   `typing.TYPE_CHECKING`, and four names from `mcp_servers.mdq.mdq_models`) — a new import
   is required.

## Implementation

### Target file

`scripts/mcp_servers/mdq/search.py`

### Procedure

1. Add `from pathlib import Path` and `from mcp_servers.mdq.auth import authorize_path` to
   the import block at the top of `search.py` (currently lines 7-19), keeping ruff's `I`
   (isort) ordering — `pathlib` groups with stdlib imports near `sqlite3`; `auth` groups with
   the existing `mcp_servers.mdq.mdq_models` import.
2. Locate the row-construction block (`search.py:114-127`):
   ```python
   fts_results = [
       SearchResultItem(
           chunk_id=row["chunk_id"],
           source_path=row["source_path"],
           heading=row["heading"],
           heading_path=row["heading_path"],
           score=float(row["rank"]) if row["rank"] is not None else 0.0,
           start_line=row["start_line"],
           end_line=row["end_line"],
           token_count=row["token_count"],
           snippet=row["content"][: service.max_snippet_chars],
       )
       for row in rows
   ]
   ```
   and the line immediately following it (`search.py:129`): `results = fts_results`.
3. Replace `results = fts_results` (line 129) with a filtering step:
   ```python
   results = [
       item
       for item in fts_results
       if authorize_path(Path(item.source_path), service.allowed_dirs)
   ]
   ```
4. Leave the rest of the function (`total=len(results)` at line 143, already computed from
   the post-filter `results` variable) unchanged — it automatically reflects the filtered
   count since it reads `results`, not `fts_results`.

### Method

Insert one list-comprehension filtering step between the existing FTS5 row-to-model mapping
and the existing "assign to `results`" line — no change to the SQL query itself (all rows are
still fetched from the DB; filtering happens in Python after fetch, consistent with the plan's
Assumption 4 for `grep_docs`'s analogous "no stored per-row authorization flag exists" caveat,
here applied to search).

### Details

- Do not filter inside the SQL query (e.g. adding a `WHERE source_path IN (...)`) — unlike
  `grep_docs`'s no-filter case (companion `mdq_service.py`/`db_grep.py` docs), `search_docs`
  has no equivalent pre-existing `IN (...)` filter mechanism in its `WHERE` clause
  (`_build_search_where()` builds `path_prefix`/`heading_prefix`/`tag_filter` clauses, none of
  which express "authorized set membership"); post-fetch Python filtering is simpler and
  sufficient given MDQ's documented low-to-medium result-set sizes.
- `item.source_path` is a `str` (the `SearchResultItem.source_path: str` field,
  `mdq_models.py:156`) — wrap in `Path(...)` before calling `authorize_path()`, matching the
  call convention already used in `mdq_service.py`'s `outline()`/`_validate_paths()`.
- After this change, confirm `logger.info("MDQ search query: %s", req.query)` (line 91) is
  unaffected — it logs before the DB query runs, not after filtering; no change needed there.
