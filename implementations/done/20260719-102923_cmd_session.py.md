## Goal

Add a `/session rag-consistency` subcommand to `scripts/agent/commands/cmd_session.py` so operators
have an on-demand replacement for the removed `/db rag consistency` command, backed by the existing
`RagMaintenanceService().consistency()` check (already used by `agent/startup.py`'s automatic
startup check). No new business logic — this is a thin dispatch + print wrapper mirroring the
existing `_db_session_stats` pattern.

## Scope

**In scope**
- Add a module-level import of `RagMaintenanceService` from `agent.services.rag_maintenance_service`.
- Add a `_rag_consistency` method to the class containing `_cmd_session` that calls
  `RagMaintenanceService().consistency()` and prints `is_consistent` plus each string in `.issues`
  when not consistent.
- Add `"rag-consistency": self._rag_consistency` to the `db_dispatch` dict inside `_cmd_session`.
- Update the two subcommand-list usage strings in this file (the `_cmd_session` docstring and the
  `write_validation_error` fallback message) to include `rag-consistency`.

**Out of scope**
- Any change to `RagMaintenanceService.consistency()`, `RagConsistencyResult`, `is_consistent()`, or
  `summarize_issues()` in `scripts/db/rag_consistency.py` — reused as-is.
- Any change to `agent/startup.py`'s existing automatic consistency check.
- Any change to `DbSessionOps`/`db_session_ops.py` — RAG consistency targets `rag.sqlite`, not
  `session.sqlite`, so it does not belong there (same reasoning `_db_session_stats` already follows
  by calling `DbMaintenanceService()` directly instead of delegating to `DbSessionOps`).
- The module docstring at `cmd_session.py:7` (`_cmd_session — /session dispatcher (list/load/...)`)
  is a comment, not a runtime usage string; updating it is a nice-to-have but not required by the
  plan's Scope — see Assumption 4.

## Assumptions

1. `RagMaintenanceService` is not currently imported anywhere in `cmd_session.py` (confirmed:
   grep for `RagMaintenanceService` in this file returns no matches). It must be added as a new
   module-level import, alongside the existing `DbMaintenanceService` import at line 21
   (`from agent.services.db_maintenance_service import DbMaintenanceService`), consistent with that
   existing pattern rather than a function-local/lazy import.
2. `RagMaintenanceService.consistency()` (`scripts/agent/services/rag_maintenance_service.py:46-54`)
   takes no arguments beyond `self` and returns a `RagConsistencyResult` (frozen dataclass,
   `scripts/agent/services/models.py:165-171`) with fields `is_consistent: bool`, `issues: list[str]`,
   `report: RagConsistencyReport`. Verified by direct read — matches the plan's expectation exactly.
3. `RagConsistencyResult.issues` is a `list[str]` of presentation-ready, human-readable strings (each
   prefixed with a severity tag like `[WARNING]` or `[CRITICAL]` and including repair guidance), per
   `scripts/db/rag_consistency.py:88-148`'s `summarize_issues()`. This confirms the plan's Assumption 2
   — the strings are safe to print directly, one per line, with no further formatting needed. This
   resolves the plan's UNK-01.
   - Note (out of scope, flagged separately per the plan's own Risks section): some of these issue
     strings still say `Run '/db rag rebuild-fts' to repair` — a removed command. `_rag_consistency`
     prints `.issues` verbatim, so it will surface this same stale text when an inconsistency is
     found. This is a pre-existing condition (already true of `agent/startup.py`'s automatic check)
     and is tracked separately per the plan's Risks section (`issues/20260719-091341_...md`), not
     addressed by this procedure.
4. The plan's Scope text describes two usage strings to update ("the docstring and the
   `write_validation_error` fallback message"). Investigation found exactly two such occurrences in
   `_cmd_session` itself:
   - The method docstring, `cmd_session.py:158-160`.
   - The `write_validation_error` fallback message, `cmd_session.py:204-209`.
   A third, similar-looking string exists in the **module-level docstring** (`cmd_session.py:7`,
   a `# ...` comment listing `_cmd_session — /session dispatcher
   (list/load/rename/delete/export/stats/health/checkpoint/vacuum/purge/recover)`). The plan's Scope
   text does not name this third location, and it is a comment (not a runtime-visible usage string),
   so it is treated as optional/out-of-scope here — update it only as an incidental consistency
   improvement if convenient, not as a required step.
5. `_db_session_stats` (`cmd_session.py:127-136`) is the exact pattern to mirror for output style:
   ```python
   def _db_session_stats(self) -> None:
       """Print session/message counts from the Session database."""
       result = DbMaintenanceService().stats()
       self._out.write_kv(
           [
               ("sessions", f"{result.sessions:,}"),
               ("messages", f"{result.messages:,}"),
               ("target", "Session"),
           ]
       )
   ```
   `write_kv` is used for structured key-value output; `write_success`/`write_no_data` are used
   elsewhere in this file for pass/fail-style single-line messages (e.g. `cmd_session.py:70, 179, 222`
   and `cmd_session.py:72, 87, 226` respectively). Since `_rag_consistency` needs to print a
   pass/fail flag (`is_consistent`) plus a variable-length list of issue strings only when
   inconsistent, the closest style match is `write_kv` for the `is_consistent` line (matching
   `_db_session_stats`'s structured-line style) followed by plain `self._out.write(...)` calls (the
   same primitive already used for `cmd_config_stats.py`'s hint line) for each issue string — see
   Method below for the exact recommended shape.
6. `db_dispatch` (`cmd_session.py:188-198`) maps subcommand strings to zero-arg callables — either a
   bound method reference directly (e.g. `"stats": self._db_session_stats`) or a lambda closing over
   `rest` when an argument is needed (e.g. `"purge": lambda: self._db_session_ops.purge(rest)`).
   `_rag_consistency` takes no arguments, so it can be registered the same way as `"stats"`:
   `"rag-consistency": self._rag_consistency` (a direct bound-method reference, no lambda needed).

## Implementation

### Target file

`scripts/agent/commands/cmd_session.py` (226 lines total at time of investigation).

### Procedure

1. Add `from agent.services.rag_maintenance_service import RagMaintenanceService` to the import
   block at the top of the file, immediately after the existing
   `from agent.services.export_formatter import render_export, write_export` line (alphabetically
   between `export_formatter` and `models`, consistent with the existing `agent.services.*`
   import grouping at lines 21-23).
2. Add a new `_rag_consistency` method near `_db_session_stats` (lines 127-136), e.g. directly after
   it, with this shape:
   ```python
   def _rag_consistency(self) -> None:
       """Print RAG index consistency status (is_consistent + any issues)."""
       result = RagMaintenanceService().consistency()
       self._out.write_kv([("is_consistent", str(result.is_consistent))])
       for issue in result.issues:
           self._out.write(issue)
   ```
   (Exact output formatting is a judgment call within this shape — the requirement is: print
   `is_consistent`, then print each string in `.issues` verbatim, one per line, only useful when the
   list is non-empty; printing nothing extra when `.issues` is empty is correct since the list is
   empty exactly when consistent, per `is_consistent()`'s definition in `rag_consistency.py:77-85`.)
3. In the `db_dispatch` dict (`cmd_session.py:188-198`), add one line:
   `"rag-consistency": self._rag_consistency,` — placement within the dict does not affect behavior;
   append it after `"recover"` to match the plan's phrasing ("adding one dict key").
4. Update the `_cmd_session` docstring (`cmd_session.py:158-160`) to append `|rag-consistency` after
   `recover`, e.g. `... | stats|health|checkpoint|vacuum|purge|recover|rag-consistency.`
5. Update the `write_validation_error` fallback message (`cmd_session.py:204-209`) the same way, e.g.
   append `|rag-consistency` to the last line's `stats|health|checkpoint|vacuum|purge|recover` suffix.
6. Run `uv run ruff format scripts/agent/commands/cmd_session.py` and
   `uv run ruff check scripts/agent/commands/cmd_session.py --fix`.

### Method

Plain method addition + one dict entry + two string edits. No new abstractions, no `Protocol`/`ABC`
considerations, no signature changes to any existing method.

### Details

- New method: `_rag_consistency(self) -> None` — no parameters, no return value, matches the
  zero-arg-callable contract required by `db_dispatch`'s `dict[str, Callable[[], None]]` type
  (`cmd_session.py:188`).
- New import: `RagMaintenanceService` (module-level, eager — matches the existing
  `DbMaintenanceService` import style at line 21, not a function-local/lazy import).
- No new types, no new constants, no config changes.
