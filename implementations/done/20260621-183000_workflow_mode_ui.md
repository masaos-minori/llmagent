# Implementation: Expose Workflow Execution Mode in Normal Runtime UI

## Goal

Make the active workflow execution mode (`auto`, `required`, `disabled`) and approval-pending state visible in `/context` and `/stats` without requiring startup log inspection.

## Scope

**In-Scope:**
- The startup banner ALREADY shows workflow status via `_get_workflow_status()` — no change needed
- Add `workflow_mode: str` and `approval_pending: bool` fields to `ContextStateView`
- Pass `workflow_mode` and `approval_pending` from `collect_context_state()` in `context_view.py`
- Show both fields in the `/context` command output
- Add workflow mode to `/stats` output (if `/stats` goes through a separate code path)
- When `/approve` is expected: emit a visible prompt hint after workflow suspension (in `orchestrator.py` where `pending_approval_id` is set)

**Out-of-Scope:**
- Redesigning the workflow engine or approval flow
- Adding new workflow commands
- Changing how workflow mode is configured

## Assumptions

1. `ctx.cfg.workflow_mode` (or `self._orchestrator._workflow_mode`) provides the mode string.
2. `ctx.turn.pending_approval_id is not None` signals approval is pending.
3. `ContextStateView` is a plain dataclass — adding fields with defaults is backward-compatible.
4. The startup banner already shows workflow status; this task adds `/context` visibility.

## Implementation

### Target files

- `scripts/agent/services/models.py` — add `workflow_mode: str = ""` + `approval_pending: bool = False` to `ContextStateView`
- `scripts/agent/services/context_view.py` — pass `workflow_mode` + `approval_pending` from `ctx`
- `scripts/agent/orchestrator.py` — after setting `ctx.turn.pending_approval_id`, emit `logger.info` / print hint
- `tests/test_context_view.py` — update `ContextStateView` construction to include new fields

### Procedure

#### Step 1: Add fields to `ContextStateView`

- `models.py`: `workflow_mode: str = ""`; `approval_pending: bool = False`

#### Step 2: Pass values in `collect_context_state()`

```python
workflow_mode=getattr(ctx.cfg, "workflow_mode", ""),
approval_pending=ctx.turn.pending_approval_id is not None,
```

#### Step 3: Show workflow mode in `/context` output

The `/context` command renders `ContextStateView` — verify the writer includes all fields, or add a specific row for `workflow_mode` and `approval_pending`. Output example:
```
Workflow: auto (tracking enabled)
Approval pending: Yes → use /approve or /reject
```

#### Step 4: Add approval-pending hint in `orchestrator.py`

After `ctx.turn.pending_approval_id = exc.approval_id` (line 199):
```python
logger.info("[workflow] Approval pending: %s", exc.approval_id)
# emit user-visible hint via the view if accessible, or via print:
print(f"\n[workflow] Approval required. Use /approve [reason] or /reject [reason].\n")
```

#### Step 5: Add workflow mode to `/stats` output (if missing)

Check `cmd_config_stats.py` for existing stats rows; add workflow_mode if not shown.

#### Step 6: Tests

- Test: `collect_context_state()` with `pending_approval_id` set → `approval_pending=True`
- Test: `/context` shows workflow_mode field
- Test: `ContextStateView` backward-compatible (fields have defaults)

### Method

- Additive fields to `ContextStateView` dataclass with defaults (backward-compatible)
- Pass values from `collect_context_state()` in context_view.py
- New visibility hint in orchestrator.py

## Risks

- **Risk:** `ctx.cfg.workflow_mode` may not exist as a direct field (could be nested under `cfg.workflow`)
  → **Mitigation:** confirm field name before implementing; use `getattr(ctx.cfg, "workflow_mode", "")` as safe fallback
- **Risk:** Print-to-stdout hint in `orchestrator.py` bypasses the `_out` interface — may break tests
  → **Mitigation:** use `logger.warning` instead of `print`; the orchestrator doesn't hold a `_out` ref; rely on log routing
- **Risk:** Adding `approval_pending` to `ContextStateView` breaks tests that compare the full dict/repr
  → **Mitigation:** add `= False` default; check for `== repr` tests and update them
