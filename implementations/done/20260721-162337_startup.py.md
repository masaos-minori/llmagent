## Goal

When `McpToolDiscoveryService(ctx).discover_all()` itself raises (not per-server findings, the
whole discovery call failing), surface the failure as `FATAL` (raising `RuntimeError` at the end
of `_check_services()`) when `production_mode` is `True`, since a discovery-call failure means
every tool call will fail for the entire session — an outage-grade condition that today is silently
downgraded to a `SKIPPED` outcome regardless of security profile.

## Implementation

### Target file

`scripts/agent/startup.py`

### Procedure

1. Open `scripts/agent/startup.py`, locate step 4's except block (currently lines 245-248, inside
   `_check_services()`, immediately after the `try` block that calls
   `McpToolDiscoveryService(ctx).discover_all()`).
2. Replace the unconditional `add_skipped` call with a `production_mode` branch: `add_fatal` (with
   a `remediation=` string) when `production_mode` is `True`, otherwise keep `add_skipped` with
   upgraded message wording.
3. Do not touch any other line in `_check_services()`.

### Method

Direct text edit of the except block. No new function, no new parameter, no signature change.

### Details

Current code (lines 245-248):

```python
        except Exception as exc:  # noqa: BLE001
            pipeline.add_skipped(
                "mcp_tool_discovery", f"MCP tool discovery skipped: {exc}"
            )
```

Replace with:

```python
        except Exception as exc:  # noqa: BLE001
            msg = f"MCP tool discovery failed - ALL tool calls will fail this session: {exc}"
            if production_mode:
                pipeline.add_fatal(
                    "mcp_tool_discovery",
                    msg,
                    remediation="Investigate MCP server connectivity/discovery failure before restarting.",
                )
            else:
                pipeline.add_skipped("mcp_tool_discovery", msg)
```

Notes:
- Keep the existing `# noqa: BLE001` comment — the broad `except Exception` is intentional and
  already justified elsewhere in this method (see steps 1, 3, 5, 5b, 6 for the same pattern).
- `add_fatal`'s signature already supports a `remediation` kwarg (used identically at line 202-206
  for `security_audit`) — no signature change needed on `StartupValidationResult`.
- Message wording is a recommendation from the plan's UNK-03, not a hard requirement; keep both
  branches' message text identical except for the `add_fatal`/`add_skipped` call and the
  `remediation` argument, per the plan's Design section.
