## Goal

Wrap the `StateStore` usage in `startup.py:_recover_pending_approvals()` with `try/finally` to guarantee `store.close()` is called even when `find_latest_pending_approval()` raises an exception.

## Scope

**In**: `scripts/agent/startup.py` — `_recover_pending_approvals()` method, lines 178–180 only.

**Out**:
- `scripts/agent/workflow/state_store.py` — no API changes.
- `scripts/agent/orchestrator.py` — already has correct pattern.
- `scripts/agent/commands/cmd_workflow.py` — also uses `find_latest_pending_approval()` but is out of scope.
- No exception catching — exceptions must propagate unchanged.

## Assumptions

1. `result` returned by `find_latest_pending_approval()` is a plain Python data object with no open DB dependency. It is safe to use after `store.close()`. (Confirmed: `state_store.py:328` returns extracted data, not a live cursor.)
2. `SQLiteHelper.close()` is idempotent — calling it on a not-fully-opened connection is safe.
3. Exception propagation to `StartupOrchestrator.run()` is intentional: if `find_latest_pending_approval()` fails, startup should abort.
4. The code at lines 181–196 (`if result is None: return ...`) does **not** move inside the `try` block — it comes after `try/finally` since `result` has no DB dependency at that point.
