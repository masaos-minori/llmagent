Source plan: `plans/20260719-100727_plan.md` ("Implement `tags_json`/`token_count` in mdq-mcp;
reconcile stale MDQ known-issue docs"), Implementation step 1's test sub-bullet and

An existing doc, `implementations/done/20260716-130700_test_mdq_service.py.md`, also targets
`tests/test_mdq_service.py` — read and confirmed **not a genuine overlap**: that doc adds a
single regression test asserting `MdqService` has no `audit_log_path` attribute (an unrelated,

## Goal

Add test coverage for real `tags_json`/`token_count` (per the paired `parser.py` and `indexer.py`
docs), and migrate the 9 existing `TestParseMarkdown` call sites that call `parse_markdown`
directly, since its return signature changes from `list[ParsedSection]` to
`tuple[list[ParsedSection], list[str]]` in the paired `parser.py` doc
(`implementations/20260719-110818_parser.py.md`) — without this migration, all 9 existing tests
break immediately once that signature change lands.

## Scope

**In scope**
- Migrate all 9 existing `parse_markdown(...)` call sites in `TestParseMarkdown`
  (`tests/test_mdq_service.py:254,266,275,289,301,317,329,342,355,370`) to unpack the new
  2-tuple return (`sections, tags = asyncio.run(parse_markdown(...))` or equivalent), so the
  existing suite keeps passing after the paired `parser.py` change.
- Update `test_frontmatter_stripped` (`:296-305`) to also assert on the tags return value for its
  existing frontmatter-without-`tags`-key fixture (`title: Test` only — expect empty tags).
- Add 4 new tests to `TestParseMarkdown` for frontmatter tag extraction: no frontmatter (empty
  tags), list-form `tags: [a, b]`, comma-separated-string-form `tags: "a, b"`, and malformed
  frontmatter (invalid YAML or non-list/non-string `tags` value — parse failure/unexpected shape
  falls back to empty tags, does not raise).
- Add assertions to `TestIndexer`'s `test_index_single_file_stores_in_db`
  (`tests/test_mdq_service.py:381-394`) confirming the raw `chunks` DB row has a non-null
  `token_count` (a positive integer) and a real `tags_json` JSON array (via a new frontmatter
  fixture with a `tags:` key) rather than the old hardcoded `""`/`None`.
- Add a new test confirming `search_docs(..., tag_filter=[...])` matches a document indexed with
  that tag and does not match a document indexed without it (two-file `tag_filter` scenario).
- Add a new test confirming `_search_docs_structured(...)`'s `SearchResultItem.token_count` is
  non-null for a chunk indexed after this plan's change — following the existing pattern at
  `test_snippet_length_respects_max_snippet_chars` (`:474-487`), which already imports and calls
  `_search_docs_structured` directly and iterates `result["results"]` items by attribute
  (`item.snippet`).

**Out of scope**
- Testing `get_chunk`'s string response payload for `token_count` — **not achievable as the
  plan's Implementation step 4 literally describes it**. See Assumption 5 below (a discrepancy
  found during this investigation, carried over from the paired `indexer.py` doc's Assumption
  10): `service.py:138-165`'s `get_chunk` only `SELECT`s `heading, content, content_hash` from
  `chunks` and never includes `token_count` in its plain-string output
  (`f"## {row['heading']}\n\n{content}"`). This plan does not modify `service.py`, so no test can
  assert `token_count` appears in `get_chunk`'s output without a `service.py` code change that is
  out of scope here. Coverage is instead provided via the raw DB row (`TestIndexer`) and via
  `search_docs`'s structured payload (`TestSearchDocs`), which together satisfy the plan's actual
  underlying intent ("non-null `token_count` is really persisted and really returned by at least
  one read path") without asserting something the current code cannot produce.
- Creating a brand-new dedicated test file (e.g. `tests/test_mdq_tags_and_token_count.py`) — the
  plan explicitly leaves this choice open but names `tests/test_mdq_service.py`'s existing
  `TestParseMarkdown`/`TestIndexer` classes as "the natural home"; this doc follows that guidance
  since the new tests are small extensions of existing test classes, not a new subsystem.
- Any change to `tests/test_mdq_rag_boundary.py`, `tests/test_mdq_hybrid_search.py` (if it still
  exists), or any other MDQ test file — out of scope for this item.

## Implementation

### Target file

`tests/test_mdq_service.py`

### Procedure

1. Migrate the 9 existing `parse_markdown` call sites in `TestParseMarkdown`
   (lines 254, 266, 275, 289, 301, 317, 329, 342, 355, 370) from
   `sections = asyncio.run(parse_markdown(...))` to
   `sections, _tags = asyncio.run(parse_markdown(...))` (use `_tags` as the unused-variable name
   in the 8 tests that don't care about tags, to signal intentional non-use without triggering a
   ruff/unused-variable warning — the existing codebase convention for intentionally-unused
   loop/unpack variables prefixes with `_`).
2. In `test_frontmatter_stripped` (`:296-305`), change the unpack to
   `sections, tags = asyncio.run(parse_markdown(...))` (no underscore — this one uses the value)
   and add `assert tags == []` after the existing `headings` assertions (the fixture's
   frontmatter has `title: Test` but no `tags` key).
3. Add 4 new test methods to `TestParseMarkdown` (after `test_heading_with_no_content_body_omitted`,
   before the `class TestIndexer:` line, i.e. after line 374):
   ```python
   def test_frontmatter_tags_list_form(
       self, service: MdqService, tmp_path: Path
   ) -> None:
       """tags: [a, b] in frontmatter is returned as a normalized list."""
       f = tmp_path / "tags_list.md"
       f.write_text(
           "---\ntags: [alpha, beta]\n---\n\n# Title\n\nBody.", encoding="utf-8"
       )
       _sections, tags = asyncio.run(
           parse_markdown(service, ParseMarkdownRequest(path=str(f)))
       )
       assert tags == ["alpha", "beta"]

   def test_frontmatter_tags_comma_string_form(
       self, service: MdqService, tmp_path: Path
   ) -> None:
       """tags: "a, b" (comma-separated string) is normalized to a list."""
       f = tmp_path / "tags_string.md"
       f.write_text(
           '---\ntags: "alpha, beta"\n---\n\n# Title\n\nBody.', encoding="utf-8"
       )
       _sections, tags = asyncio.run(
           parse_markdown(service, ParseMarkdownRequest(path=str(f)))
       )
       assert tags == ["alpha", "beta"]

   def test_frontmatter_no_tags_key_returns_empty(
       self, service: MdqService, tmp_path: Path
   ) -> None:
       """Frontmatter present but with no tags key yields an empty tags list."""
       f = tmp_path / "no_tags.md"
       f.write_text("---\ntitle: Test\n---\n\n# Title\n\nBody.", encoding="utf-8")
       _sections, tags = asyncio.run(
           parse_markdown(service, ParseMarkdownRequest(path=str(f)))
       )
       assert tags == []

   def test_malformed_frontmatter_does_not_crash_indexing(
       self, service: MdqService, tmp_path: Path
   ) -> None:
       """Invalid YAML or a non-list/non-string tags value falls back to empty tags,
       not an exception — a malformed frontmatter block must not break indexing of an
       otherwise-valid document."""
       f = tmp_path / "malformed_fm.md"
       f.write_text(
           "---\ntags: {not: a, list: or, string: value}\n---\n\n# Title\n\nBody.",
           encoding="utf-8",
       )
       sections, tags = asyncio.run(
           parse_markdown(service, ParseMarkdownRequest(path=str(f)))
       )
       assert tags == []
       assert len(sections) == 1
       assert sections[0]["heading"] == "Title"
   ```
   (Note: `test_frontmatter_no_tags_key_returns_empty` duplicates coverage already added to
   `test_frontmatter_stripped` in step 2 with a dedicated fixture/name for clarity per the
   plan's Implementation step 1 wording, which names "no frontmatter (empty tags)" as one of the
   four required cases — keep both: `test_frontmatter_stripped` for the pre-existing heading
   behavior, this one for the tags-specific case with an unambiguous name.)
4. In `TestIndexer.test_index_single_file_stores_in_db` (`:381-394`), add after the existing
   `assert "Title" in row["heading"]`:
   - Extend the `SELECT` in the existing query from `"SELECT heading, content FROM chunks..."` to
     `"SELECT heading, content, token_count, tags_json FROM chunks..."`.
   - `assert row["token_count"] is not None and row["token_count"] > 0`.
   - `assert row["tags_json"] == "[]"` (the shared `md_file` fixture has no frontmatter, so tags
     should be an empty JSON array, not the old hardcoded `""`).
5. Add a new test method to `TestIndexer` (after `test_index_single_file_stores_in_db`):
   ```python
   def test_index_single_file_stores_real_tags_json(
       self, service: MdqService, tmp_path: Path
   ) -> None:
       f = tmp_path / "tagged.md"
       f.write_text(
           "---\ntags: [urgent, draft]\n---\n\n# Title\n\nBody.", encoding="utf-8"
       )
       asyncio.run(_index_single_file(service, f))
       conn = service._get_db_connection()
       try:
           row = conn.execute(
               "SELECT tags_json FROM chunks WHERE source_path = ?", (str(f),)
           ).fetchone()
           assert row is not None
           assert row["tags_json"] == '["urgent", "draft"]'
       finally:
           conn.close()
   ```
   (Adjust the exact JSON string literal to match `json.dumps`'s actual default separator output
   — verify with a quick local check during implementation rather than assuming
   `orjson`/`json`-specific spacing; `rules/coding.md` mandates `orjson` for new JSON
   serialization in `scripts/`, but the paired `indexer.py` doc uses stdlib `json.dumps` per the
   source plan's own Design section wording ("`json.dumps(tags)`") — if the implementer follows
   `rules/coding.md`'s `orjson` preference instead when writing the actual code, this test's
   expected string must match `orjson.dumps(tags).decode()`'s exact output instead, which has no
   space after `,`/`:` by default (e.g. `'["urgent","draft"]'`) — confirm the real separator used
   by the landed `indexer.py` change before finalizing this exact string, or assert via
   `json.loads(row["tags_json"]) == ["urgent", "draft"]` to be separator-agnostic, which is the
   safer, recommended assertion form).
6. Add a new test method to `TestSearchDocs` (after `test_snippet_length_respects_max_snippet_chars`):
   ```python
   def test_search_docs_tag_filter_matches_real_tags(
       self, service: MdqService, tmp_path: Path
   ) -> None:
       """search_docs(..., tag_filter=[...]) matches docs indexed with that tag only."""
       tagged = tmp_path / "tagged.md"
       tagged.write_text(
           "---\ntags: [important]\n---\n\n# Title\n\nShared keyword content.",
           encoding="utf-8",
       )
       untagged = tmp_path / "untagged.md"
       untagged.write_text("# Other\n\nShared keyword content.", encoding="utf-8")
       asyncio.run(
           index_paths(service, IndexPathsRequest(paths=[str(tagged), str(untagged)]))
       )
       req = SearchDocsRequest(query="keyword", tag_filter=["important"])
       result = asyncio.run(search_docs(service, req))
       assert "Title" in result
       assert "Other" not in result
   ```
7. Add a new test method to `TestSearchDocs` (after the tag_filter test):
   ```python
   def test_search_docs_structured_token_count_non_null(
       self, service: MdqService, md_file: Path
   ) -> None:
       """SearchResultItem.token_count is a positive int after indexing (API response
       payload, not just the raw DB row)."""
       from mcp_servers.mdq.search import _search_docs_structured

       asyncio.run(index_paths(service, IndexPathsRequest(paths=[str(md_file)])))
       result = _search_docs_structured(service, SearchDocsRequest(query="Content"))
       assert len(result["results"]) > 0
       for item in result["results"]:
           assert item.token_count is not None
           assert item.token_count > 0
   ```

### Method

Pure test additions plus a mechanical signature-migration edit (unpacking a 2-tuple instead of a
1-value return at 9 existing call sites) — no new fixtures, no new mocking beyond what
`test_snippet_length_respects_max_snippet_chars` already established as this file's pattern for
structured-result assertions.

### Details

- Use `_tags` (leading underscore) for unpacked-but-unused tags values in the 8 migrated tests
  that don't assert on tags, to keep ruff/mypy quiet about unused locals while making the
  intentional non-use explicit (matches this file's own existing convention of prefixing
  intentionally-unused variables, e.g. `_service`/`_req` in `test_search_timeout_raises_...`
  at `:502`).
- Keep all new test docstrings in English (`rules/coding.md`: "comments and log output: English
  only").
- Do not modify the `service`/`md_file`/`md_dir` fixtures themselves (`:38-68`) — all new tests
  either reuse them as-is or write their own `tmp_path`-scoped fixture files inline, matching
  the existing style of tests like `test_snippet_length_respects_max_snippet_chars` that build a
  custom file under `tmp_path` rather than extending a shared fixture.
- Prefer `json.loads(row["tags_json"]) == [...]` over an exact string-literal comparison
  wherever feasible, to stay robust to whichever JSON encoder (`json.dumps` vs `orjson.dumps`)
  the landed `indexer.py` change actually uses (see Procedure step 5's note).
