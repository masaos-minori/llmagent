# Implementation: Add explicit scope comment to /approve and /reject in cmd_workflow.py

Steps covered: Plan 20260626-090724 — Phase 2, Step 2-3

---

## Goal

Make it explicit in `cmd_workflow.py` that `/approve` and `/reject` resolve **workflow-level approval only** (the `workflow_approvals` DB record), and do not interact with tool-level (`run_approval_checks`) approval.

---

## Scope

- **In scope**: `scripts/agent/commands/cmd_workflow.py` — update docstrings and/or add inline comments
- **Out of scope**: runtime behavior changes

---

## Assumptions

- The current docstrings ("Approve the pending workflow task") are correct but lack the scope boundary.
- Tool-level approval is resolved interactively via stdin — `/approve`/`/reject` should not affect it.
- A one-line scope clarification is sufficient; no new logic is needed.

---

## Implementation

### Target file
`scripts/agent/commands/cmd_workflow.py`

### Procedure
1. Read `scripts/agent/commands/cmd_workflow.py` fully.
2. Update `_cmd_approve` docstring:
   ```python
   """Approve the pending workflow-level approval gate (workflow_approvals table only).
   Does not affect per-tool interactive approval (tool_approval.run_approval_checks).
   """
   ```
3. Update `_cmd_reject` docstring similarly:
   ```python
   """Reject the pending workflow-level approval gate (workflow_approvals table only).
   Does not affect per-tool interactive approval (tool_approval.run_approval_checks).
   """
   ```
4. Update the class docstring of `_WorkflowMixin`:
   ```python
   """Workflow approval slash-command handlers.
   
   These commands resolve workflow-level approval gates only (DB-persisted,
   per-task). Per-tool interactive approval (run_approval_checks) is handled
   separately via stdin prompts and is not affected by these commands.
   """
   ```

### Method
Docstring-only change. No runtime logic changes.

### Details

The distinction to make clear:
- Workflow-level: `StateStore.find_latest_pending_approval()` → `resolve_approval()` → DB write.
- Tool-level: `run_approval_checks()` → stdin read → returns approved/denied list.

These two flows are completely independent. `/approve`/`/reject` touch only the DB record.
