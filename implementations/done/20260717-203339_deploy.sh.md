# Implementation procedure: `deploy/deploy.sh` (verification-only, no change expected)

Source plan: `plans/done/20260717-124020_plan.md` (requirement `requires/20260717_02_require.md`), Implementation step 7.

## Goal

Resolve the plan's step 7 ("Update `deploy/deploy.sh`: add copy-list entries for the 2 new
`scripts/shared/*.py` files") against `deploy/deploy.sh`'s actual current mechanism, and record whether
any change is genuinely required once `scripts/shared/runtime_tool.py` and
`scripts/shared/runtime_tool_registry.py` exist.

## Scope

**In scope**: read-only verification of `deploy/deploy.sh`'s deployment mechanism for files under
`scripts/`; a decision on whether an edit is needed.

**Out of scope**: any other part of `deploy/deploy.sh` (config file copy list, top-level project file
list) — those are unrelated to this requirement's 2 new files.

## Assumptions

1. The plan's "Affected areas" table states: *"Deploy impact: 2 new `scripts/shared/*.py` modules —
   `deploy/deploy.sh`'s copy-list needs 2 new entries (confirm exact list format during implementation)."*
   `rules/coding.md`'s general convention likewise states *"Module addition | update copy list in
   `deploy/deploy.sh`."* Both describe a **per-file copy-list** mechanism.
2. **Direct read of `deploy/deploy.sh` contradicts this assumption.** The section that deploys
   `scripts/` (confirmed at lines 68-73) is a **wholesale directory `rsync`**, not a per-file copy-list:
   ```
   68: rsync -av --delete \
   69:   --exclude="__pycache__/" \
   70:   --exclude="*.pyc" \
   71:   --exclude="*.pyo" \
   72:   "${REPO_ROOT}/scripts/" "${DEPLOY_SCRIPTS}/"
   ```
   This syncs the entire `scripts/` tree (minus bytecode artifacts) to the deploy target. Any new file
   placed under `scripts/shared/` — including `runtime_tool.py` and `runtime_tool_registry.py` — is
   **automatically included** the next time this rsync runs, with zero edits to `deploy/deploy.sh`
   itself. The only `cp`-style individual-file entries in `deploy.sh` (elsewhere in the file, e.g. near
   lines 64-65 and 84-101 per this doc's investigation) are for top-level project files and
   `config/*.toml` files, not anything under `scripts/`.
3. Per `rules/coding.md`'s own "Current behavior classification" framework, this is a **documentation
   fix required** case for the plan text and the general `rules/coding.md` convention line — both
   describe a copy-list mechanism that does not exist for `scripts/` in the current `deploy.sh`. Fixing
   `rules/coding.md`/the (now-archived) plan text is out of scope for this documents-only task (this task
   only produces per-item implementation-procedure docs); this doc simply records the corrected finding
   so the implementer does not spend effort searching for a non-existent copy-list.

## Implementation

### Target file

`deploy/deploy.sh` — **no modification expected.**

### Procedure

1. Before starting the actual code implementation pass for `runtime_tool.py`/`runtime_tool_registry.py`,
   re-read `deploy/deploy.sh` lines 60-80 to confirm the `rsync` block shown in Assumption 2 is still
   present and unchanged (in case `deploy.sh` was refactored between this design pass and implementation).
2. If the `rsync`-of-`scripts/` mechanism is still in place: make **no edit** to `deploy/deploy.sh` for
   this requirement. Note in the implementation PR/commit message that deploy impact was verified
   nil, referencing this doc, so a reviewer doesn't flag the missing "copy-list update" as an omission.
3. If `deploy.sh` has since changed to a per-file/explicit list mechanism (unexpected, but possible if
   another requirement in this batch touched it first): add the 2 new file paths
   (`scripts/shared/runtime_tool.py`, `scripts/shared/runtime_tool_registry.py`) to whatever list
   mechanism is found, following the exact format of neighboring entries at that time.

### Method

Verification + conditional no-op. Not a code-design task — no dataclasses/interfaces involved.
