## Assumptions

1. `_Ctx` (`tests/test_cmd_mcp.py:18-45`, confirmed by direct read) is a hand-built stub for
   `AgentContext`, not the real class — it sets `self.services_required = MagicMock()` with only
   `health_registry`, `audit_logger`, `lifecycle` explicitly configured (lines 36-43). It does **not**
   currently set `.runtime_tools`, so `ctx.services_required.runtime_tools` on an unmodified `_Ctx`
   instance auto-resolves to a `MagicMock()` (truthy, not `None`) rather than the real nullable
   `RuntimeToolRegistry | None` default. Tests for `_cmd_mcp_tools()` must explicitly set
   `ctx.services_required.runtime_tools` to either `None` (registry-unavailable path) or a real (or
   fake) `RuntimeToolRegistry`-shaped object exposing `.all_tools() -> list[RuntimeTool]` — relying on the
   MagicMock auto-attribute default would silently exercise neither real path.
2. `RuntimeTool` (`implementations/20260717-203121_runtime_tool.py.md`) is a frozen dataclass; tests can
   construct fixture instances directly (e.g. via its factory `build_runtime_tool(...)`, once
   `scripts/shared/runtime_tool.py` exists) or a minimal local stub object exposing the same 7 attributes if
   the real module is not yet importable at test-authoring time (see the cmd_mcp.py doc's Assumption 1 on
   the `AppServices.runtime_tools` prerequisite not yet landing in real source).
3. `UnknownSubcommandError.__str__` (`agent/commands/exceptions.py:25-29`) formats as `f"Unknown subcommand
   {sub!r}. Valid: {', '.join(valid)}"`. The two existing tests that construct this error
   (`test_unknown_subcommand_raises_error` line 411, `test_invalid_subcommand_shows_usage` line 420) match
   only on the `"Unknown subcommand 'invalid'"` / `"'install'"` prefix via `pytest.raises(...,
   match=...)`the — regex `match` performs a substring search, so widening the allowed tuple from
   `("status",)` to `("status", "tools")` does not change whether these two existing assertions pass.
   **No edit needed to these two tests**, but a new positive test should confirm the widened tuple
   explicitly (Details below).
4. `TestMcpCommandDefinition` (`tests/test_cmd_mcp.py:536-555`) already asserts `/mcp`'s `CommandDef` has no
   `"install"` in its help text and is a prefix+async command — these assertions are unaffected by adding a
   `"tools"` subcommand (no `CommandDef` change, per the cmd_mcp.py doc's Scope). No edit needed there.

## Implementation

### Target file

`tests/test_cmd_mcp.py`.

### Procedure

1. Add a small test-local fixture for `RuntimeTool`-shaped fake tools (a `SimpleNamespace` or a minimal
   local dataclass mirroring the 7 rendered fields, to avoid a hard dependency on
   `shared.runtime_tool.RuntimeTool` existing at test-write time — swap to a real import once that module
   lands, per this doc's Assumption 2).
2. Add `class TestCmdMcpTools:` with:
   - `test_tools_subcommand_dispatches`: `ctx.services_required.runtime_tools` set to a fake registry whose
     `.all_tools()` returns 2+ fixture tools (at least one with `enabled_for_llm=False` /
     `requires_approval=True`, i.e. disabled/policy-filtered); call `await mcp._cmd_mcp("tools")`; assert
     each of the 7 field values appears in `capsys` output for each tool, including the disabled one (direct
     verification of the plan's "disabled tools are visible" acceptance criterion).
   - `test_tools_subcommand_no_registry`: `ctx.services_required.runtime_tools = None`; call `await
     mcp._cmd_mcp("tools")`; assert it does not raise and writes an explanatory line (no crash on the
     nullable-service path).
   - `test_tools_subcommand_excludes_raw_schema`: fixture tool includes a non-trivial `input_schema`/
     `raw_definition` dict; assert none of that dict's content leaks into the rendered output (direct
     verification of the plan's "output concise and safe for logs" requirement).
   - `test_unknown_subcommand_allows_tools`: assert `"tools"` no longer raises
     `UnknownSubcommandError` (i.e. `await mcp._cmd_mcp("tools")` completes without raising) — a direct,
     explicit check of the widened allowed-tuple, complementing the existing prefix-only assertions in
     Assumption 3.
3. Do not modify `_Ctx`'s `__init__` signature; instead set `ctx.services_required.runtime_tools` per-test
   directly on the constructed `_Ctx`/`MagicMock`, following the existing convention already used elsewhere
   in this file for other per-test service overrides (e.g. `ctx.services_required.health_registry` set
   directly in several `TestCmdMcpStatus*` tests).

### Method

Plain `unittest.mock`-based unit tests, `pytest.mark.asyncio`, `capsys` for output capture — same toolkit
already used throughout this file (no new test infra needed).
