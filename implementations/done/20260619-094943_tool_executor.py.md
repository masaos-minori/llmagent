# Implementation: Distinguish Transport vs Tool Errors in Metrics

## Goal

Add an `error_type` discriminator to tool execution results so transport-level failures (`error_type=transport`) and tool-level errors (`error_type=tool`) are distinguishable in log output and any existing tracing spans.

## Scope

- `scripts/shared/tool_executor.py`
  - After `_raw_execute()` returns, detect `is_error=True` and log with `error_type=tool`
  - Transport errors already logged as transport failures; add `error_type=transport` attribute to the existing warning log

Out of scope:
- OTel span attribute changes (additive logging is sufficient without a running OTel backend)
- Changing HTTP response format
- Per-server counters (Step 2)

## Assumptions

1. `tool_executor.py` already distinguishes transport errors (raises `TransportError`, line 522) from tool errors (`is_error=True` in response, line 514-521).
2. The conceptual distinction exists in comments (line 516-518) but is not surfaced in log output with a structured field.
3. `stat_cache_hits: int` is the only existing stat counter (line 403); the pattern can be extended.
4. No OTel span is currently attached to `_raw_execute()`; adding a log field is the minimal change.

## Implementation

### Target file

`scripts/shared/tool_executor.py`

### Procedure

1. In `_raw_execute()`, after a successful return from `_do_call()`, log at `DEBUG` level when `result.is_error` is True, with `error_type=tool`.
2. In the `except TransportError` handler (line 522-534), add `error_type=transport` to the existing `logger.warning(...)` call.

### Method

Two targeted log statement changes. No new classes or fields needed.

### Details

**After line 521 (successful transport return), add:**
```python
if result.is_error:
    logger.debug(
        "tool error from %r: %s (error_type=tool)",
        server_key,
        result.output[:120],
    )
```

**In the except TransportError handler (line 526), update warning:**
```python
logger.warning(
    "transport failure for %r: %s (state=%s, error_type=transport)",
    server_key,
    e,
    state.value,
)
```

These two changes make `error_type=tool` and `error_type=transport` grep-able in logs without changing any data structure or response format.

**Optional OTel extension:** If `otel_tracer.py` is used in this code path, add span attributes:
```python
span.set_attribute("error.type", "tool")   # or "transport"
```
Check whether a span context is available in `_raw_execute()` before adding.
