## Goal

Add an optional `capabilities: tuple[str, ...] = ()` field to the planned `RuntimeTool` frozen dataclass
(`scripts/shared/runtime_tool.py`, per `implementations/20260717-203121_runtime_tool.py.md`), and thread a
matching optional `capabilities` parameter through `build_runtime_tool()`'s factory signature, defaulting
to an empty tuple when omitted. This requirement only adds the field's existence and safe-default
behavior — no real MCP tool populates non-empty capability values as part of this change (that is
requirement 13's own explicitly out-of-scope future adoption work).

## Scope

**In scope**
- `scripts/shared/runtime_tool.py`: add the `capabilities` field to `RuntimeTool` and the corresponding
  parameter to `build_runtime_tool()`.

**Out of scope**
- `scripts/shared/runtime_tool_registry.py` — no method signature changes. `RuntimeToolRegistry`'s 9
  methods (`resolve`, `get`, `all_tools`, `llm_tool_definitions`, `tool_spec_map`, `tool_spec_for_call`,

  instances or copy a fixed subset of fields (`resource_scope`/`requires_serial`/`is_write` into
  `ToolSpec`); none of them need to read or expose `capabilities` for this requirement's acceptance
  criteria, and adding an unused pass-through would be speculative. If a future requirement needs
  capability-based filtering, that is out of scope here (per the plan's own Out-of-scope section).
- Any actual capability string values on any real MCP tool's `TOOL_LIST` entry.
- The `operation_type` field mentioned in the plan's Risk #2 (a second, independent extension to
  `RuntimeTool` proposed by a different, not-yet-designed requirement 07) — confirmed via
  `grep -rl "operation_type" implementations/*.md` returning zero hits at investigation time, so there is
  no prior design to coordinate against yet. This doc adds only `capabilities`; if requirement 07's

  into one final field list before real implementation, per the plan's Risk #2 mitigation.

## Assumptions

1. **The base doc's 13-field list and factory signature are taken as given** (per
   `implementations/20260717-203121_runtime_tool.py.md`'s Details section): `name`, `server_key`,
   `server_url`, `description`, `input_schema`, `raw_definition`, `status`, `is_write`, `requires_serial`,
   `resource_scope`, `agent_safety_tier`, `requires_approval`, `enabled_for_llm`. This doc adds a 14th
   field, `capabilities`, without altering any of the 13 existing ones or their safe-default rules.
2. **`capabilities` needs no sentinel-`None` factory parameter**, unlike `is_write`/`requires_serial`
   (which encode "was this explicitly annotated" via a `None` sentinel per the base doc's Assumption 5).
   `capabilities` has no such derived-default coupling to another field — an omitted `capabilities`
   argument simply means "empty", so the factory parameter can default directly to `None` and resolve to
   `()` (mirroring `input_schema`/`raw_definition`'s existing `dict | None = None` -> `{}` pattern from the
   base doc, applied to a tuple instead of a dict) rather than needing a three-state sentinel.
3. **Empty-mutable-default handling does not apply to tuples** — `tuple` is immutable, so
   `capabilities: tuple[str, ...] = ()` is safe as a literal dataclass field default in principle; however,
   this doc still routes construction through the `build_runtime_tool()` factory (not a bare dataclass
   default) for consistency with every other field's construction path in the base doc, and to allow
   normalizing a caller-supplied `list[str]` into a `tuple[str, ...]` at the boundary (a `/v1/tools` JSON
   response naturally decodes `capabilities` as a `list`, not a `tuple`).
4. **No runtime validation of individual capability strings against the `domain.action` convention is
   added** (consistent with the base doc's Assumption/Details note that `agent_safety_tier`'s `Literal` is
   enforced statically only, not at runtime) — the convention document
   (`implementations/20260718-084628_tool_capability_naming_convention.md`) defines the shape as
   documentation guidance, not a runtime-enforced schema. Adding a regex guard here would be speculative,
   unrequested validation beyond this requirement's acceptance criteria.

### Procedure

1. In the `RuntimeTool` frozen dataclass body (per the base doc's Procedure step 3), add a 14th field
   after `enabled_for_llm`: `capabilities: tuple[str, ...] = ()`.
2. In `build_runtime_tool()`'s parameter list (per the base doc's Procedure step 4), add:
   `capabilities: Sequence[str] | None = None` (accepting any sequence — including a JSON-decoded `list`
   — not narrowly `tuple | None`, since callers normalizing a `/v1/tools` response will most naturally
   have a `list[str]`).
3. Inside the factory body, resolve: `resolved_capabilities = tuple(capabilities) if capabilities is not
   None else ()` — placed alongside the existing `input_schema`/`raw_definition` `None`-to-empty
   resolution lines (per the base doc's Details), and pass `capabilities=resolved_capabilities` into the
   `RuntimeTool(...)` constructor call.
4. Update the module docstring's field-list summary (if the base doc's docstring enumerates all fields) to
   mention `capabilities` is optional, additive metadata with no consumer yet — mirroring how the base doc
   already flags `agent_safety_tier`'s duplication-not-import rationale in its own docstring.
5. No change to `Sequence` import scope beyond adding `from collections.abc import Sequence` if not already
   imported by the base doc's implementation (needed only for the factory's parameter type annotation, not
   the frozen dataclass field itself, which stays a concrete `tuple[str, ...]`).

### Method

Same as the base doc: plain dataclass field + factory-function parameter, no `Protocol`/`ABC`. This is a
strictly additive field append, not a redesign of the existing 13 fields' shapes or defaults.
