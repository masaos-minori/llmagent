Source plan: `plans/20260717-130506_plan.md` ("Consolidate tool startup validation into MCP discovery
service", requirement `requires/20260717_09_require.md`), Implementation step 6.

No existing implementations doc (under `implementations/` or `implementations/done/`) matches this
specific change. All `startup.py`-named docs under `implementations/done/` predate today's
`RuntimeToolRegistry`/`mcp_tool_discovery.py` design batch (confirmed: none reference `RuntimeToolRegistry`

## Goal

Replace `_check_services()`'s 3 separate calls to `check_tool_definitions_startup()`,
`check_routing_drift()` (static, untouched — stays as-is), and `check_routing_drift_vs_live()` with a
single call into the consolidated `McpToolDiscoveryService.discover_all()` (per
`implementations/20260717-224511_mcp_tool_discovery.py.md`), folding its returned
`list[StartupCheckOutcome]` directly into `pipeline` via `add_fatal`/`add_warning`/`add_ok` — no
`try/except Exception` swallowing a deliberate fatal into a "skipped" outcome (see the discovery-service
doc's Assumption 4 for why the current exception-based path is being retired).

### Procedure

1. Edit the import block (lines 19-27): remove `check_routing_drift_vs_live` and
   `check_tool_definitions_startup` from the `from agent.repl_health import (...)` tuple; add
   `from agent.services.mcp_tool_discovery import McpToolDiscoveryService` near the existing
   `from agent.services.rag_maintenance_service import RagMaintenanceService` import (line 28) to keep
   `agent.services.*` imports grouped.
2. Replace step 4 (lines 224-234) and step 6 (lines 252-266) with one consolidated step (keep it between
   the old step-3/step-5 positions, i.e. directly after step 3 "Service readiness" and before step 5
   "Routing drift (static)", to preserve the existing step ordering/log sequence as closely as possible —
   renumber the inline comments accordingly, e.g. "# 4. MCP tool discovery and validation").
3. New step body (pseudocode, no production code):
   ```
   # 4. MCP tool discovery and validation (consolidated)
   discovery_result = await McpToolDiscoveryService(ctx).discover_all()
   for outcome in discovery_result.findings:
       if outcome.status is StartupCheckStatus.FATAL:
           pipeline.add_fatal("mcp_tool_discovery", outcome.message)
       elif outcome.status is StartupCheckStatus.WARNING:
           pipeline.add_warning("mcp_tool_discovery", outcome.message)
   if not discovery_result.findings:
       pipeline.add_ok("mcp_tool_discovery")
   ```
   No `try/except Exception` wrapper is added around this call — per the discovery-service doc's design,
   `discover_all()` does not raise for validation findings (only a genuine bug/unexpected error should
   propagate, which is consistent with how steps 1-3 already let unexpected exceptions surface rather than
   downgrading them silently, except where the existing code explicitly chose not to, as in step 3's
   `except Exception` at line 221 — mirror that same broad catch here **only** if `discover_all()`'s own
   error-handling doesn't already cover HTTP/parse failures internally; per the discovery-service doc,
   per-server fetch failures are already captured as warning `StartupCheckOutcome`s inside `discover_all()`,
   so no additional catch is needed at this call site for the common failure modes).
4. Renumber the remaining inline step comments (old step 5 becomes step 5 still since it's untouched;
   old step 7 stays step 7) — only the comment text for the replaced step needs updating, not the
   surrounding steps' numbers, since this consolidation replaces exactly steps 4 and 6 with one step
   inserted at the step-4 position (i.e., what was step 6 no longer exists as a separate step).
5. Update the three test files' patch targets per Scope's "Note for the implementer" (this is source-adjacent
   test work required for this file's change to be verifiable, tracked here since the plan's own
   Affected-areas table does not list `test_startup.py`/`test_startup_consistency.py`/
   `test_startup_validation_pipeline.py` as separate rows — treat their patch-target updates as part of
   this implementation step, not a separate document):
   - `tests/test_startup.py`: change `"check_tool_definitions_startup": AsyncMock(...)` /
     `"check_routing_drift_vs_live": AsyncMock(...)` default mock entries (lines 479, 482) and the 4
     dedicated test methods (lines 641-793) to patch `agent.startup.McpToolDiscoveryService` (constructing
     a `MagicMock` whose `.return_value.discover_all` is an `AsyncMock` returning a `DiscoveryResult` with
     the desired `findings` list) instead of patching the two removed function names directly.
   - `tests/test_startup_consistency.py` (lines 32, 59, 84) and
     `tests/test_startup_validation_pipeline.py` (lines 92, 99, 123, 130, 153, 160, 189, 196, 220, 227, 247,
     254): same patch-target substitution.
