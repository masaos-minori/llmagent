# Implementation Procedure: scripts/agent/http_lifecycle.py

Source plan: `plans/20260712-191531_plan.md` (Implementation Steps §1: Guard implementation)
Source requirement: `requires/done/20260712_15_require.md`

## Goal

`HttpServerLifecycleManager.shutdown_all()` absorbs a second `SIGINT` that arrives while
cleanup is already in progress, so every managed MCP subprocess still gets terminated,
instead of the loop being aborted mid-iteration by a propagated `KeyboardInterrupt`.

## Scope

**In scope:** `shutdown_all()` method body only, in `scripts/agent/http_lifecycle.py`.

**Out of scope:**
- `restart()`, `start()`, `_terminate_with_timeout()`, `_wait_exited()` — unchanged
- `scripts/agent/repl.py` SIGTERM handler registration (`run()`, lines 469-480) — not touched
- Any parallelization of per-server termination (`asyncio.gather()`) — explicitly deferred by the requirement

## Assumptions

1. `shutdown_all()` always executes on the main thread (only call site: `_close_resources()`
   in `scripts/agent/repl.py:290`, itself only reached from `_run_repl_loop()`'s `finally`
   block under `asyncio.run()`) — `signal.signal()` requires the main thread, and this design
   defends against the assumption being wrong via a `try/except ValueError` fallback rather
   than assuming it holds unconditionally.
2. `signal` is already imported at module level in `scripts/agent/http_lifecycle.py` (line 14:
   `import signal`) — no new import needed.
3. `logger` is already defined at module level (line 26: `logger = logging.getLogger(__name__)`)
   — reused as-is for the new warning log line.

## Implementation

### Target file

`scripts/agent/http_lifecycle.py`

### Procedure

1. Locate `shutdown_all()` (currently lines 310-335).
2. At the top of the method body, capture the current SIGINT handler and attempt to install
   a temporary absorbing handler, guarding against the off-main-thread case:
   ```python
   old_sigint = None
   try:
       old_sigint = signal.getsignal(signal.SIGINT)
       signal.signal(signal.SIGINT, self._absorb_sigint_during_shutdown)
   except ValueError:
       # Not on the main thread — proceed without the guard rather than fail shutdown.
       old_sigint = None
   ```
3. Add a small private handler method (or module-level function) used above:
   ```python
   @staticmethod
   def _absorb_sigint_during_shutdown(signum: int, frame: object) -> None:
       logger.warning(
           "Lifecycle: SIGINT received during shutdown_all(); "
           "ignoring until cleanup completes"
       )
   ```
4. Wrap the existing per-server termination loop (the `for key in keys:` block and the
   trailing `self._stderr_log_paths.clear()`) in a `try/finally`, restoring the original
   handler in `finally`:
   ```python
   try:
       # ... existing loop body, unchanged ...
       self._stderr_log_paths.clear()
   finally:
       if old_sigint is not None:
           signal.signal(signal.SIGINT, old_sigint)
   ```
5. Do not change the method's signature (`async def shutdown_all(self) -> None`), its
   existing per-key `try/except (OSError, TimeoutError)` block, or any other method.

### Method

Direct in-place edit of the existing method body — no new class, no new module. This is an
additive wrap (install guard → run existing logic unchanged → restore guard), matching the
"standard CLI tool" absorb-and-restore pattern described in the plan's Design section.

### Details

- The handler installation must happen **before** the loop starts and the restoration must
  happen in a `finally` so it runs even if `_terminate_with_timeout` raises an exception type
  not already caught by the existing per-key `except (OSError, TimeoutError)` (defense in
  depth — restoring the handler must never be skipped).
- `signal.getsignal(signal.SIGINT)` returns the handler that was active before this call —
  typically Python's `signal.default_int_handler`, but capturing it dynamically (rather than
  hardcoding `default_int_handler`) is required so repeated calls to `shutdown_all()`
  (`test_shutdown_all_twice_is_safe` in the existing test suite) restore correctly regardless
  of what was previously installed.
- The absorbing handler only logs; it does not set any flag or attempt to influence the loop —
  the loop's own per-server timeout escalation (5s SIGTERM wait + 5s SIGKILL wait, unchanged)
  remains the sole mechanism for progress, so a truly hung subprocess still gets force-killed
  on schedule rather than blocking forever.
