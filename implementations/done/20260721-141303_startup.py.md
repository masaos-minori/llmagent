## Goal

`routing_drift_strict = true` (in `[tool]` section of `config/agent.toml`, backed by
`ToolConfig.routing_drift_strict: bool = False` in `scripts/agent/config_dataclasses.py:164`) actually
causes `StartupOrchestrator._check_services()` to abort startup with a fatal outcome when routing drift is
detected, instead of only ever logging a warning regardless of the flag's value.

## Scope

**In scope:**
- `scripts/agent/startup.py` line 252: pass `strict=ctx.cfg.tool.routing_drift_strict` into the
  `check_routing_drift(ctx)` call.
- `scripts/agent/startup.py` lines 254-255: restructure the `except Exception` block so a `RuntimeError`
  raised by `check_routing_drift(..., strict=True)` is caught separately and routed to
  `pipeline.add_fatal(...)`, while all other exception types still fall through to the existing
  `pipeline.add_warning(...)` behavior.

**Out of scope:**
- The other 7 checks in `_check_services()` (security audit, embedding dimensions, service readiness, MCP
  tool discovery, routing safety tiers, RAG consistency) — each has its own independent try/except block;
  none are touched.
- `check_routing_drift_vs_live()` / `McpToolDiscoveryService` (steps 4/5b/6) — untouched.
- `check_routing_drift()`'s internal implementation in `scripts/agent/repl_health.py` — already correctly
  raises `RuntimeError` when `strict=True` and drift is found (lines 383-387); no change needed there.
- Adding `routing_drift_strict` to `config/agent.toml` itself — the dataclass default (`False`) is a safe

## Implementation

### Target file

`scripts/agent/startup.py`

### Procedure

1. Locate the "5. Routing drift (static)" block inside `_check_services()` (current file: lines 250-255,
   immediately after the "4. MCP tool discovery and validation" block and before "5b. Routing safety
   tiers").
2. Replace:
   ```python
           # 5. Routing drift (static)
           try:
               for msg in check_routing_drift(ctx):
                   pipeline.add_warning("routing_drift", msg)
           except Exception as exc:  # noqa: BLE001
               pipeline.add_warning("routing_drift", f"Routing drift check failed: {exc}")
   ```
   with:
   ```python
           # 5. Routing drift (static)
           try:
               for msg in check_routing_drift(ctx, strict=ctx.cfg.tool.routing_drift_strict):
                   pipeline.add_warning("routing_drift", msg)
           except RuntimeError as exc:
               pipeline.add_fatal("routing_drift", str(exc))
           except Exception as exc:  # noqa: BLE001
               pipeline.add_warning("routing_drift", f"Routing drift check failed: {exc}")
   ```
3. Re-run `rg "check_routing_drift\(" scripts/` to confirm no other call site is affected by widening the
   exception handling at this one site.

### Method

Minimal, localized edit: one added keyword argument, one added `except RuntimeError` clause inserted before
the existing broad `except Exception` clause (Python evaluates `except` clauses in order, so the narrower
`RuntimeError` clause must come first or it will never be reached — the existing `except Exception` would
otherwise shadow it).

### Details

- `ctx.cfg.tool.routing_drift_strict` is the existing `ToolConfig` field (default `False`) — no new config
  plumbing needed; `ctx.cfg.tool` is already accessed elsewhere in `_check_services()`
  (`ctx.cfg.mcp.security_profile` pattern confirms `ctx.cfg.<section>` access is standard here).
  This mirrors how the "4. MCP tool discovery" step already uses `StartupCheckStatus.FATAL` /
  `pipeline.add_fatal` for its own fatal path.
- No changes to imports — `check_routing_drift` is already imported (line 23 in the current import block);
  no new symbols needed since `RuntimeError` is a builtin.
- This is a narrowly-scoped, low-blast-radius change: each of the other checks in `_check_services()` has
  its own independent `try/except`, confirmed by direct read of the current file (steps 1-3, 4, 5b, 6 all
  have separate `try`/`except` blocks) — narrowing exception handling for step 5 only cannot affect them.
- `noqa: BLE001` (broad exception justification) stays on the remaining `except Exception` clause since it
  is unchanged in purpose (catching non-`RuntimeError` failures from the drift check, e.g. unexpected
  attribute errors); the new `except RuntimeError` clause does not need a `noqa` since it is a narrow,
  specific exception type.
