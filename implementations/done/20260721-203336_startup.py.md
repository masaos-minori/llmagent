# Implementation Procedure: Make MCP tool discovery failures fatal under PRODUCTION security profile

## Goal

Make MCP tool discovery failures fatal under PRODUCTION security profile, preventing the agent from starting silently crippled when all tools will fail.

## Scope

- `scripts/agent/startup.py` — modify `_check_services()` step 4 exception handler to check security profile and add FATAL vs WARNING accordingly
- Tests — add unit/integration tests for discovery failure under both PRODUCTION and LOCAL profiles

## Assumptions

1. Option A (changing the exception handler in `_check_services()`) is preferred over Options B/C because it mirrors existing patterns in `startup.py`
2. Under LOCAL profile, WARNING level is sufficient — the agent can still start but the warning should be unmistakable
3. No existing tests directly cover the discovery failure path in `_check_services()`

## Implementation

### Target file

`scripts/agent/startup.py`

### Procedure

1. Locate the bare `except Exception` handler in `_check_services()` at approximately lines 245-248
2. Replace with profile-aware logic:
   ```python
   except Exception as exc:  # noqa: BLE001
       if self._ctx.cfg.mcp.security_profile == SecurityProfile.PRODUCTION:
           pipeline.add_fatal(
               "mcp_tool_discovery", f"MCP tool discovery failed: {exc}"
           )
       else:
           pipeline.add_warning(
               "mcp_tool_discovery", f"MCP tool discovery skipped: {exc}"
           )
   ```

3. Verify `pipeline.add_fatal()` causes startup to halt (confirm against existing usage patterns in `_check_services()` at lines 202-206, 213, 225)

4. Add tests covering:
   - PRODUCTION profile: Mock discovery raising exception → assert pipeline outcome is FATAL → assert startup abort
   - LOCAL profile: Same mock setup → assert pipeline outcome is WARNING (not SKIPPED) → assert startup proceeds
   - Integration test: Fully unreachable MCP server cluster causes startup failure under PRODUCTION profile

### Method

Use targeted edit to replace the bare `except Exception` handler with profile-aware logic. This mirrors existing patterns in `startup.py` where `SecurityProfile.PRODUCTION` triggers fatal-on-subprocess-failure behavior.

### Details

- Blast radius: Medium — changes startup behavior for discovery failures; agents that previously started with silent tool-call failures will now fail fast with clear error messages
- Zero churn on this code path (no existing tests cover discovery failure); the bug has been present since sole-authority routing was implemented
- Deploy impact: None — no config changes needed; behavior change takes effect immediately upon deployment
