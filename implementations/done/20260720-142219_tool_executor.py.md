# Implementation procedure: `scripts/shared/tool_executor.py` (`set_runtime_registry()` mutates, not replaces)

Source plan: `plans/done/20260720-134821_plan.md`, Implementation step Phase 2 (item 2).

`implementations/` and `implementations/done/` contain ~11 prior docs whose filename matches this
target (`tool_executor.py`, `shared_tool_executor.py.md`, `test_tool_executor.py.md` variants). All
read (headers) and confirmed to target unrelated, already-applied changes: cache-bypass routing for
side-effecting tools (`20260715-152212`), `apply_config()` docstring narrowing (`20260709-103724`),
`ensure_ready` failure wrapping (`20260707-143159`), and other earlier cache/lifecycle work. None
touches `set_runtime_registry()`'s resolver-replacement behavior. This is a new document.

## Goal

Change `ToolExecutor.set_runtime_registry()` from constructing a brand-new `ToolRouteResolver` (which
discards `self._resolver`'s identity and resets `discovery_map` to `{}`) to delegating to the
resolver's own new `set_runtime_registry()` mutator (added in the companion `route_resolver.py` doc),
so the same `ToolRouteResolver` instance persists across the call.

## Scope

**In scope**: `scripts/shared/tool_executor.py`, `set_runtime_registry()` method only (current lines
76-84).

**Out of scope**:
- `__init__`'s `ToolRouteResolver(...)` construction (lines 67-69) â€” unchanged; still the one and
  only place the resolver is first created.
- `apply_config()`, `_check_startup_mode()`, and all execution-path methods â€” untouched.
- `scripts/shared/route_resolver.py` â€” separate doc (this doc assumes
  `ToolRouteResolver.set_runtime_registry()` already exists per that doc; sequence these two edits
  together in the same implementation pass since this file's edit calls a method that does not exist
  in the resolver until that edit lands).

## Implementation

### Target file

`scripts/shared/tool_executor.py`

### Procedure

1. Replace the method body:
   - Before:
     ```
     def set_runtime_registry(self, registry: RuntimeToolRegistry) -> None:
         """Wire RuntimeToolRegistry into the resolver after discovery completes."""
         from shared.route_resolver import ToolRouteResolver

         self._resolver = ToolRouteResolver(
             self._server_configs,
             discovery_map={},
             runtime_registry=registry,
         )
     ```
   - After (pseudocode):
     ```
     def set_runtime_registry(self, registry: RuntimeToolRegistry) -> None:
         """Wire RuntimeToolRegistry into the existing resolver after discovery completes."""
         self._resolver.set_runtime_registry(registry)
     ```
2. Remove the now-unused local `from shared.route_resolver import ToolRouteResolver` import inside
   the method. Confirm `ToolRouteResolver` is still imported at module level (line 30,
   `from shared.route_resolver import ToolRouteResolver`) for `__init__`'s use â€” that import stays.
3. No change to the method's public signature (`registry: RuntimeToolRegistry`) or return type.

### Method

Single-method body replacement; no new imports added, one import removed. Behavior change: the
side effect of resetting `discovery_map` to `{}` on every call is eliminated â€” `discovery_map` now
retains whatever value was passed to `__init__` (default `{}` anyway per current call sites, so this
is a no-observable-difference case in practice, but it is a real semantic difference the plan calls
out explicitly: "no more `discovery_map={}` reset side effect").

### Details

- After this edit, `self._resolver` object identity is preserved across `set_runtime_registry()`
  calls â€” verify via `id(ex._resolver)` before/after in the rewritten integration test (see
  `test_runtime_tool_routing_integration.py` implementation doc, which renames
  `test_runtime_registry_set_runtime_registry_replaces_resolver` to assert `ex._resolver is
  resolver_before`, the opposite of its current assertion).
- `RuntimeToolRegistry` import at the top of `tool_executor.py` (used only for the type hint on this
  method's parameter) is untouched â€” still needed.
- No change to `__init__`'s `ToolRouteResolver(server_configs, discovery_map=discovery_map or {})`
  call (line 67-69) â€” that remains the sole construction site for the resolver, matching the plan's
  explicit statement that the resolver is "mutated, not replaced" after that point.
