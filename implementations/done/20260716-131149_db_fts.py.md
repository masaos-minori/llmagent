# Implementation: scripts/mcp_servers/mdq/db_fts.py (delete file)

Source plan: `plans/20260716-123031_plan.md`

## Goal

Delete `scripts/mcp_servers/mdq/db_fts.py` in full — both functions it
defines (`fts_consistency_check`, `fts_rebuild`) have no callers once
`MdqService.fts_consistency_check()`/`fts_rebuild()` are removed (companion
`service.py` doc).

## Scope

**In:**
- Delete `scripts/mcp_servers/mdq/db_fts.py` (the entire file, 55 lines).

**Out:**
- `MdqConsistencyError` (imported from `models.py` by this file) — the class
  itself lives in `models.py` and is used elsewhere (`search.py`,
  `server.py`); only this file's *import* of it disappears along with the
  file, no other action needed.
- Any other `db_*.py` sibling module in `scripts/mcp_servers/mdq/`
  (`db_schema.py`, `db_grep.py`) — unaffected.

## Assumptions

1. `rg -n "fts_consistency_check|fts_rebuild" scripts/` shows the only
   callers of these two functions are `scripts/mcp_servers/mdq/service.py`
   (being removed in the companion doc) — no other file imports from
   `mcp_servers.mdq.db_fts`.
2. This deletion must land in the same commit as the `service.py` edit
   (companion doc) that removes the `from mcp_servers.mdq.db_fts import
   fts_consistency_check, fts_rebuild` line — deleting the file first
   without updating `service.py` breaks the import at load time.
3. No test file directly imports from `mcp_servers.mdq.db_fts` — confirmed,
   no `test_mdq_db_fts*.py` file exists in `tests/`.

## Implementation

### Target file

`scripts/mcp_servers/mdq/db_fts.py`

### Procedure

1. Confirm no remaining importer: run
   `rg -n "from mcp_servers.mdq.db_fts\|mcp_servers\.mdq\.db_fts" scripts/ tests/`
   and verify the only hit besides the file itself is the `service.py`
   import line (which the companion doc removes in the same change).
2. Delete the file: `git rm scripts/mcp_servers/mdq/db_fts.py` (or
   equivalent file removal), not a manual truncation to empty — the module
   should not exist at all post-change.

### Method

Whole-file deletion — no code from this file is preserved elsewhere in the
codebase (the SQL logic is preserved only in git history and in this plan's
Design section, per the plan's own risk mitigation).

### Details

- Do not leave an empty `db_fts.py` stub — the plan's intent is full
  removal, not deprecation.
- Check `deploy/deploy.sh` for an explicit copy line naming `db_fts.py` by
  filename (as opposed to a directory glob that copies all of
  `scripts/mcp_servers/mdq/`) — if an explicit per-file line exists, remove
  it in the same change (see Implementation step 7 / Deployment note in the
  source plan). If the deploy script copies the whole
  `scripts/mcp_servers/mdq/` directory as a unit, no `deploy.sh` change is
  needed.
