## Goal

Create `scripts/mcp_servers/browser/models.py` — the config-loading, Pydantic schema, and
domain-exception module for the new read-only Browser MCP server — following the exact
structural role that `scripts/mcp_servers/shell/models.py` (144 lines, verified) plays for
shell-mcp: typed config dataclass + `.load()` classmethod, domain exceptions, and Pydantic
request/response models for the server's one tool (`browser_fetch`).

## Scope

**In scope**
- `BrowserAuthorizationError(RuntimeError)` — raised when the domain-allowlist check fails
  (mapped to HTTP 403 in `server.py`, mirroring `ShellAuthorizationError`
  at `scripts/mcp_servers/shell/models.py:20-21`).
- `BrowserValidationError(ValueError)` — raised on malformed input (mapped to HTTP 422,
  mirroring `ShellValidationError` at `shell/models.py:24-25`).
- `BrowserConfig` dataclass (`@dataclasses.dataclass`) with `from_dict()` and `load()`,
  mirroring `ShellConfig` (`shell/models.py:33-79`).
- `BrowserFetchRequest` / `BrowserFetchResponse` Pydantic `BaseModel`s, mirroring
  `ShellRunRequest`/`ShellRunResponse` (`shell/models.py:113-145`).

**Out of scope**
- No `ShellPolicy`-style separate `shared/protocols/browser.py` dataclass. `shell/models.py`
  delegates to `shared.protocols.shell.ShellPolicy` (`shell/models.py:15,87-105`) via a
  `load_shell_policy()` builder function, but this indirection exists in shell-mcp because
  `ShellPolicy` is a generic, reusable dataclass shape; no other module needs a shared
  `BrowserPolicy` type for this plan's single-tool server. Per the plan's own Affected-areas
  list (only `models.py` is named — no `shared/protocols/browser.py`), `BrowserConfig` is used
  directly by `service.py` with no extra indirection layer. This is a deliberate simplification
  versus the shell template, not an oversight — note it explicitly if a reviewer expects a
  1:1 structural match with shell.
- No allowlist-check logic itself (`_check_domain`-equivalent) — that belongs in `service.py`
  per Design step 1.
- No FastAPI app wiring — that belongs in `server.py`.

## Assumptions

1. Verified by direct read: `shell/models.py` (144 lines) has exactly this four-part shape:
   domain exceptions (lines 20-25), `ShellConfig` dataclass with `from_dict`/`load`
   (lines 33-79), a policy-builder function (lines 87-105, **not reused** here per Scope), and
   Pydantic request/response models (lines 113-145). Browser's `models.py` follows the same
   four-part shape minus the policy-builder indirection.
2. `ConfigLoader().load("shell_mcp_server.toml")` (`shell/models.py:79`) is the config-loading
   entry point used by every server; `BrowserConfig.load()` calls
   `ConfigLoader().load("browser_mcp_server.toml")` identically. `ConfigLoader` itself is not
   modified by this doc (out of scope; shared infra).
3. `ShellConfig.from_dict()` (`shell/models.py:52-74`) uses `d.get(key) or default` (not
   `d.get(key, default)`) specifically to avoid `str(None)` producing `"None"` and `int(None)`
   raising `TypeError` when a key is present with an explicit `null`/empty value in TOML.
   `BrowserConfig.from_dict()` must use the same `or`-based defaulting pattern for every field.
4. Per the plan's Design step 1 and Risks section, `BrowserConfig` needs, at minimum:
   `allowed_domains: list[str] = []` (fail-closed default), `max_response_kb: int`,
   `timeout_sec: int`, `auth_token: str = ""`. The Risks section also calls for defense-in-depth
   IP-literal/loopback rejection "regardless of the domain allowlist" — this is a boolean/no-arg
   behavior in `service.py` (always-on), not a configurable field, so no extra config key is
   needed for it here (confirm this reading holds at implementation time; if a toggle is wanted,
   add `reject_ip_literals: bool = True` to `BrowserConfig`).
5. `mypy`'s `from __future__ import annotations` + `ClassVar`-free dataclass style used by
   `shell/models.py` (no explicit `ClassVar` annotations needed since none of its fields are
   class-level constants) applies identically here.

## Implementation

### Target file

`scripts/mcp_servers/browser/models.py` (new file; package `scripts/mcp_servers/browser/` does
not exist yet — `__init__.py` for the package is a trivial empty file, not separately documented
here, matching `shell/__init__.py`'s 0-line content, confirmed by `wc -l`).

### Procedure

1. Create `scripts/mcp_servers/browser/__init__.py` as an empty file (matches every existing
   server package's `__init__.py`, e.g. `scripts/mcp_servers/shell/__init__.py`, 0 lines).
2. Create `scripts/mcp_servers/browser/models.py` with:
   - Module docstring identifying it as the browser-mcp config/models module (mirror
     `shell/models.py:1-5`'s docstring style).
   - `from __future__ import annotations`, `dataclasses`, `logging`, `typing.Any` imports;
     `from pydantic import BaseModel, Field`; `from shared.config_loader import ConfigLoader`.
   - `class BrowserAuthorizationError(RuntimeError): """Raised when a domain-allowlist check
     fails (HTTP 403)."""`
   - `class BrowserValidationError(ValueError): """Raised on invalid input (HTTP 400/422)."""`
   - `@dataclasses.dataclass class BrowserConfig` with fields: `allowed_domains: list[str] =
     dataclasses.field(default_factory=list)`, `max_response_kb: int = 256`,
     `timeout_sec: int = 15`, `auth_token: str = ""`. Include `from_dict(cls, d: dict[str, Any])
     -> BrowserConfig` classmethod using `d.get(key) or default` per Assumption 3, and
     `load(cls) -> BrowserConfig` classmethod calling
     `cls.from_dict(ConfigLoader().load("browser_mcp_server.toml"))`.
   - `class BrowserFetchRequest(BaseModel)`: `url: str = Field(..., description=...)`,
     `max_response_kb: int | None = Field(default=None, ge=1, le=<server max>, description=...)`
     (caller override clamped server-side, mirroring `ShellRunRequest.max_output_kb`'s
     `Field(default=512, ge=1, le=65536)` pattern at `shell/models.py:129` — Browser's field
     is optional/`None`-defaulted since the clamp-to-server-max logic lives in `service.py`,
     matching the plan's Design step 2 wording "caller override, clamped to server max").
   - `class BrowserFetchResponse(BaseModel)`: `text: str`, `truncated: bool`, `url: str`,
     `status_code: int`, `elapsed_sec: float`.

### Method

New-file creation; no refactor. Follow `shell/models.py`'s existing conventions exactly
(dataclass + Pydantic split, `or`-based `from_dict` defaulting, module-level `logger =
logging.getLogger(__name__)` even if unused directly in this file, matching `shell/models.py:17`
for consistency — confirm at implementation time whether an unused logger triggers a vulture/ruff
finding; if so, only declare it if actually used, e.g. in a config-validation warning path).

### Details

Type contracts:
```python
@dataclasses.dataclass
class BrowserConfig:
    allowed_domains: list[str] = dataclasses.field(default_factory=list)
    max_response_kb: int = 256
    timeout_sec: int = 15
    auth_token: str = ""

    @classmethod
    def from_dict(cls, d: dict[str, Any]) -> "BrowserConfig": ...
    @classmethod
    def load(cls) -> "BrowserConfig": ...


class BrowserFetchRequest(BaseModel):
    url: str
    max_response_kb: int | None = None


class BrowserFetchResponse(BaseModel):
    text: str
    truncated: bool
    url: str
    status_code: int
    elapsed_sec: float
```
No `Protocol`/`ABC` abstractions introduced (matches shell's plain-dataclass-plus-Pydantic
style; no speculative interfaces).
