# Implementation procedure: `config/agent.toml` (Browser MCP registration)

Source plan: `plans/20260719-101501_plan.md`, Implementation step 3 (Design/Affected areas).

`config/agent.toml` is an extremely common target filename in this repo's implementation
history (e.g. `implementations/20260719-105935_agent.toml.md` and many others under
`implementations/done/`). Checked by content: every hit found via
`ls`/`grep -l "agent.toml"` addresses unrelated topics (memory config, CLI commands, tool
availability metadata, etc. — none registers a `[mcp_servers.browser]` section, a
`browser_fetch` tool definition, or a `browser_fetch` safety tier). Flagged as checked, not a
genuine overlap.

## Goal

Register Browser MCP in `config/agent.toml`: a new `[mcp_servers.browser]` transport block, a
new `[[tool_definitions]]` entry for `browser_fetch`, and a new `tool_safety_tiers` entry —
per the canonical checklist (`docs/04_mcp_06_15_new-mcp-server-addition-checklist.md:20-28`,
verified).

## Scope

**In scope**
- `[mcp_servers.browser]` block, placed after `[mcp_servers.mdq]` (the current last entry,
  `config/agent.toml:377-387`, verified), following the common `shell`/`git`/`web_search` shape:
  `transport = "http"`, `url = "http://127.0.0.1:8016"` (port per the `server.py` doc's
  Assumption 2 — **8016**, not 8015, since 8015 is already Event Bus), `startup_mode =
  "subprocess"`, `auth_token = ""`, `role = "..."` (see Assumption 2 below), `tool_names =
  ["browser_fetch"]`, `cmd = ["uv", "run", "--directory", "/opt/llm", "python",
  "/opt/llm/scripts/mcp_servers/browser/server.py"]`.
- One new `[[tool_definitions]]` block for `browser_fetch` (LLM-facing function-calling schema,
  Japanese-language description per the existing convention), placed near the end of the
  existing `[[tool_definitions]]` sequence.
- One new `tool_safety_tiers` entry: `browser_fetch = "READ_ONLY"`.

**Out of scope**
- No change to any existing `[mcp_servers.*]` block, `[[tool_definitions]]` entry, or
  `tool_safety_tiers` entry.
- No `healthcheck_mode` field — verified this field was removed project-wide on 2026-07-17
  (`config/agent.toml:277-283`'s NOTE, confirmed by direct read: "healthcheck_mode was removed
  ... from every `[mcp_servers.*]` block ... HTTP is the only supported transport"). Do not
  re-add it for `[mcp_servers.browser]`.
- No `startup_mode = "persistent"` — see Assumption 1 (drift note) below; every current server
  uses `"subprocess"`.

## Assumptions

1. **Plan drift, verified and corrected.** The source plan's Assumption 7 states that
   `cicd`/`rag_pipeline` "carry `healthcheck_mode='http'` and use `startup_mode='persistent'`
   instead of `'subprocess'`, per a documented fix for a startup-stall issue," and that
   `[mcp_servers.browser]` should follow the "more common" `shell`/`git`/`web_search` shape
   unless a similar stall risk is discovered. Direct re-verification of the current file
   (`grep -n "startup_mode" config/agent.toml`) shows **all ten** `[mcp_servers.*]` blocks —
   including `cicd` (line 362) and `rag_pipeline` (line 371) — currently use `startup_mode =
   "subprocess"`, and **no block anywhere in the file uses `healthcheck_mode` at all** (the field
   was fully removed 2026-07-17, per the NOTE at lines 277-283, which post-dates whatever state
   the plan's Assumption 7 was written against). This is a real drift between the plan's
   Assumption 7 and the file's current content — not something to silently "fix by matching the
   plan," but to flag explicitly: `[mcp_servers.browser]` uses `startup_mode = "subprocess"` and
   no `healthcheck_mode` key, which is now simply what *every* server does, not a choice made
   "unless a stall risk is discovered."
2. `role` field values observed across existing entries: `"execute"` (shell), `"file_write"`
   (git, file_write), `"search"` (web_search, github), `"file_read"`, `"file_delete"`, `"query"`
   (cicd, rag_pipeline, mdq) — verified via direct read of `config/agent.toml:285-387`. None of
   these exactly fits "fetch a URL and extract text" semantics. `"search"` is the closest
   existing precedent for a read-only, information-retrieval tool that is not local-filesystem-
   scoped (matches `web_search`'s and `github`'s role) — recommend `role = "search"` for
   `[mcp_servers.browser]` unless implementation discovers a `role`-driven behavior branch
   elsewhere in the codebase that would make `"search"` semantically wrong for a page-fetch tool
   (not found during this investigation — `role` appears to be routing/classification metadata
   only, per the surrounding schema; verify at implementation time with `rg -n "\.role\b"
   scripts/` before finalizing).
3. `tool_safety_tiers` naming convention (verified, `config/agent.toml:209-267`): tiers used are
   `READ_ONLY`, `WRITE_SAFE`, `WRITE_DANGEROUS`, and `ADMIN` (the last used only for
   `shell_run`, line 221). `search_web` (a read-only, external-network tool with no local
   side effects, the closest structural analog to `browser_fetch`) is tiered `READ_ONLY`
   (line 222, verified). `browser_fetch` should be tiered `READ_ONLY` for the same reason: it
   has no side effects, only reads and returns externally-fetched content.
4. `[[tool_definitions]]` block shape (verified, `config/agent.toml:391-405`, the `search_web`
   entry): `[[tool_definitions]]` / `type = "function"` / `[tool_definitions.function]` with
   `name`/`description` (Japanese) / `[tool_definitions.function.parameters]` with
   `type = "object"`, `required = [...]` / one
   `[tool_definitions.function.parameters.properties.<field>]` sub-table per parameter with
   `type`/`description`. `browser_fetch`'s block follows this exact shape with `required =
   ["url"]` and two properties (`url`, `max_response_kb`).
5. Per Assumption 1's drift finding, no `[mcp_servers.browser]` field list needs to diverge from
   the common 6-field shape (`transport`, `url`, `startup_mode`, `auth_token`, `role`,
   `tool_names`, `cmd` — 7 fields counting `cmd`) used by `shell`/`git`/`web_search`/
   `file_*`/`github` — `cicd`/`rag_pipeline`/`mdq` are also confirmed to use this same 7-field
   shape now (the plan's claimed divergence for those two no longer exists in the current file).

## Implementation

### Target file

`config/agent.toml`.

### Procedure

1. After the `[mcp_servers.mdq]` block (ends at line 387, verified — followed by the
   `# ── Tool definitions ──...` section header at line 389), insert:
   ```toml
   [mcp_servers.browser]
   transport        = "http"
   url              = "http://127.0.0.1:8016"
   startup_mode     = "subprocess"
   auth_token       = ""
   role             = "search"
   tool_names       = ["browser_fetch"]
   cmd              = ["uv", "run", "--directory", "/opt/llm", "python", "/opt/llm/scripts/mcp_servers/browser/server.py"]
   ```
2. Append a new `[[tool_definitions]]` block (after the last existing one — locate via
   `grep -n "\[\[tool_definitions\]\]" config/agent.toml` and insert after the final match,
   re-verify the exact line number at implementation time since earlier edits in this same
   plan cycle do not touch this section, but other concurrent plans might):
   ```toml
   [[tool_definitions]]
   type = "function"

   [tool_definitions.function]
   name        = "browser_fetch"
   description = "指定した URL のページを取得し、可視テキストを抽出して返す(読み取り専用; JavaScript は実行しない; クリック・フォーム入力・ナビゲーション等の対話操作は行わない)。アクセス可能なドメインは allowlist 設定による。"

   [tool_definitions.function.parameters]
   type     = "object"
   required = ["url"]

   [tool_definitions.function.parameters.properties.url]
   type        = "string"
   description = "取得する絶対 URL (http/https のみ; ホストは allowlist 設定が必要)"

   [tool_definitions.function.parameters.properties.max_response_kb]
   type        = "integer"
   description = "レスポンステキストのサイズ上限 (KB)。サーバ設定の上限値でクランプされる。"
   ```
3. Add to `[tool_safety_tiers]` (section starts at line 209, verified — insert alongside the
   other `READ_ONLY` entries for locality, e.g. near `search_web` at line 222):
   ```toml
   browser_fetch            = "READ_ONLY"
   ```
4. Do **not** add a `browser_fetch` entry to `approval_high_risk_tools`/any approval-gating
   list (verified: only `shell_run` and the GitHub write/dangerous tools appear in the
   `high`/`medium` approval-risk maps around `config/agent.toml:184-193` — a `READ_ONLY` tool
   has no approval-gating entry anywhere in the file, matching `search_web`'s precedent of
   having no approval-map entry either).

### Method

Additive TOML edits only — no existing block is modified. Insert new blocks at the structural
positions identified above (end of `[mcp_servers.*]` sequence, end of `[[tool_definitions]]`
sequence, alongside other `READ_ONLY` tiers).

### Details

No code; TOML config additions consumed by `shared/mcp_config.py` (`McpServerConfig` /
`StartupMode` parsing — unmodified by this plan) and by the agent's tool-definition loader
(unmodified).
