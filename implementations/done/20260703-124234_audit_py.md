# Step 1 — Fix `scripts/mcp/audit.py`: rename fields and always emit `error_type`/`server_key`

## Goal

Rename `"session"` to `"session_id"` and `"request"` to `"request_id"` in the `_audit_log` record dict, and unconditionally include `"error_type"` and `"server_key"` fields (using empty string as default) so the schema is always complete.

## Scope

- In-Scope:
  - Rename `"session"` key to `"session_id"` in the `record` dict inside `_audit_log`
  - Rename `"request"` key to `"request_id"` in the `record` dict inside `_audit_log`
  - Move `error_type` and `server_key` out of their conditional blocks; include them unconditionally in the `record` dict
  - Remove the `if detail:` conditional block; include `"detail"` unconditionally (empty string when not provided), OR keep `detail` conditional — per the plan, only `error_type` and `server_key` must become unconditional
- Out-of-Scope:
  - Per-server audit logs in shell/file-delete/github/mdq (separate log files, not `mcp/audit.py`)
  - Changing the function signature of `_audit_log`
  - Modifying any MCP server callers of `_audit_log` (they already pass `session_id=` and `request_id=` as kwargs)
  - Tests (covered by Step 4)

## Assumptions

1. All callers of `_audit_log()` in all MCP servers (cicd, github, shell, mdq, file-read, file-write, etc.) already use the keyword argument names `session_id=` and `request_id=`, so renaming only the dict keys does not break callers.
2. The required unified schema specifies that `"error_type"` and `"server_key"` are always present as top-level keys (empty string when not applicable), consistent with the agent-side `ToolExecEvent` pattern.
3. The `"detail"` field remains conditional (only included when non-empty) — the plan only explicitly requires `error_type` and `server_key` to become unconditional.
4. `test_error_type_absent_when_ok` in `TestMcpAuditLogFormat` (line 193) currently asserts `"error_type" not in parsed` for a successful call. After making `error_type` unconditional, this test must also be updated (covered in Step 4).
