# Implementation: retriever.py â€” Split into FtsRetriever / VectorRetriever / HybridRetriever

## Goal

Split `scripts/agent/memory/retriever.py` into three focused classes, making `_vec_search()`
a public `knn_search()` API on `VectorRetriever`, and exposing `HybridRetriever` as the
primary external interface with a `MemoryRetriever` compatibility alias.

## Scope

- `scripts/agent/memory/retriever.py`: refactor in-place (do not split into separate files;
  keep all three classes in one module to minimize import changes).
- Add `FtsRetriever`, `VectorRetriever`, `HybridRetriever` classes.
- Keep `MemoryRetriever = HybridRetriever` alias until `layer.py` is deleted (step 8).
- `tests/test_memory_retriever.py`: update imports if private helper functions move.

## Assumptions

1. All three classes stay in `retriever.py` to minimize import-path changes across the codebase.
2. Module-level helper functions (`_build_fts_query`, `_recency_boost`, `_rrf_merge`, `_score`) remain at module level in `retriever.py` for backward compatibility with `test_memory_retriever.py`.
3. `VectorRetriever.knn_search()` replaces `MemoryRetriever._vec_search()` as the public dedup API.
4. `HybridRetriever` composes `FtsRetriever` and `VectorRetriever`; it is the primary search interface.
5. `ingestion.py` will receive a `VectorRetriever` instance for dedup (or `HybridRetriever` with `knn_search` delegation).
6. Constructor signatures of `FtsRetriever`, `VectorRetriever`, and `HybridRetriever` accept the same `fts_limit`, `rrf_k`, `recency_days` parameters as the current `MemoryRetriever`.
