## Goal

Add a retry cooldown mechanism to `_ServerLifecycleRouter` that prevents rapid subprocess start retries for failed MCP servers.

## Scope

**In:**
- `scripts/agent/factory.py`: Add `_failed_starts` dict and cooldown logic to `_ServerLifecycleRouter`

**Out:**
- Modifying production mode behavior (already treats startup failures as fatal)
- Changing the exception propagation path in `ensure_ready()`
- Any other file modifications

## Assumptions

1. The cooldown should apply regardless of whether the failure occurred during initial startup or a later tool call — any failed subprocess start triggers the cooldown.
2. The cooldown duration is configurable via a module-level constant (`COOLDOWN_SECONDS`) rather than a runtime configuration option.
3. The cooldown applies only to HTTP subprocess-mode servers (the same subset affected by the current `ensure_ready()` logic).
4. Successful subprocess start clears the cooldown entry, allowing the next failure to trigger a new cooldown period.

## Design decisions

- Cooldown is applied at the `_ServerLifecycleRouter` level rather than inside `HttpServerLifecycleManager.start()` to avoid modifying the lifecycle protocol interface contract.
- Using `time.monotonic()` instead of `time.time()` for monotonic clock semantics — immune to system clock changes.
- Cooldown applies equally to all modes (production and non-production) since production mode already fails fast on startup, making the cooldown practically irrelevant there.

## Alternatives considered

- Moving cooldown into `HttpServerLifecycleManager.start()`: Would centralize cooldown logic but requires modifying the lifecycle protocol interface contract and affects all callers including those outside this router.
- Configurable cooldown per-server: Adds complexity without clear benefit given the transient nature of most failures.
- Exponential backoff: Overkill for a simple retry storm prevention use case; linear cooldown is sufficient.

```python
def _in_cooldown(self, server_key: str) -> bool:
    """Return True if the server is within its cooldown window."""
    last_failure = self._failed_starts.get(server_key, 0)
    if last_failure == 0:
        return False
    elapsed = time.monotonic() - last_failure
    if elapsed < _COOLDOWN_SECONDS:
        _logger.info(
            "Lifecycle: %r still in cooldown (%.1fs remaining)",
            server_key,
            _COOLDOWN_SECONDS - elapsed,
        )
        return True
    # Cooldown expired — remove stale entry
    del self._failed_starts[server_key]
    return False
```

### Details

- The cooldown check uses `time.monotonic()` which is immune to system clock changes and provides a reliable way to measure elapsed time.
- Stale cooldown entries are automatically cleaned up when the cooldown expires — no background cleanup task needed.
- On successful start, the cooldown entry is removed so the next failure triggers a fresh cooldown period.
- All three methods (`ensure_ready()`, `start_http_subprocess()`, `restart()`) share the same cooldown logic to prevent bypass via direct calls.
- The cooldown applies to both production and non-production modes. Since production mode already treats startup failures as fatal (fail-fast), the cooldown has minimal practical impact there.

## Compatibility considerations

- No public API changes — `_ServerLifecycleRouter` is an internal class.
- The cooldown adds in-memory state; on process restart, it resets naturally.
- If the server config changes while the process is running, the cooldown would need to be cleared manually — document this limitation.
