# Implementation: tests/test_mdq_config.py (new — `MdqConfig` validation tests)

Source plan: `plans/20260719-205532_plan.md`, Implementation steps Phase 1, item 3.

## Goal

Create a new test file covering the `MdqConfig` Pydantic model (companion `mdq_models.py` doc)
and its wiring into `MdqService.__init__` (companion `mdq_service.py` doc): invalid numeric
fields (zero/negative), invalid list fields (wrong container type), invalid `enable_grep` type,
and confirmation that existing valid config still constructs successfully.

## Scope

**In scope:**
- New file `tests/test_mdq_config.py`.
- Tests targeting `mcp_servers.mdq.mdq_models.MdqConfig` directly (construction-level unit
  tests) and `mcp_servers.mdq.mdq_service.MdqService.__init__` (integration-level: confirm a
  malformed on-disk TOML config raises at service construction time, not later).

**Out of scope:**
- Read-time authorization tests (`search_docs`/`get_chunk`/`grep_docs`) — covered by the
  companion `tests/test_mdq_read_authorization.py` doc.
- Health `allowed_dirs_count`/`deny_all` tests — covered by the companion
  `tests/test_mdq_health.py` doc.
- Modifying `tests/test_mdq_service.py` — this plan does not require changes there beyond
  what Phase 1's own validation step (`uv run pytest tests/test_mdq_config.py
  tests/test_mdq_service.py -v`) implies as a regression check; no new test cases are added to
  that existing file by this document.

## Assumptions

1. `MdqConfig` (companion `mdq_models.py` doc) is a Pydantic `BaseModel` importable as
   `from mcp_servers.mdq.mdq_models import MdqConfig`.
2. Existing test files for this package use `PYTHONPATH=scripts`-relative imports, e.g.
   `from mcp_servers.mdq.mdq_service import MdqService` — this new file follows the same
   import convention (verified against sibling test file naming, e.g.
   `tests/test_mdq_service.py`).
3. `MdqService.__init__` accepts an optional `db_path` and internally loads
   `mdq_mcp_server.toml` via `ConfigLoader` (`mdq_service.py:53-57`) — to test config-driven
   validation failures at the service-construction level without mutating the real TOML file
   on disk, tests should monkeypatch `shared.config_loader.ConfigLoader.load` (or the
   equivalent patch point already used by `tests/test_mdq_service.py` for config-dependent
   tests — check that file's existing fixture/monkeypatch pattern first and reuse it for
   consistency rather than inventing a new one).
4. `pydantic.ValidationError` is the expected exception type for both direct `MdqConfig(...)`
   construction failures and the propagated failure inside `MdqService.__init__` (per the
   plan's "fail fast" requirement — not caught/translated into an `Mdq*Error` subclass).

## Implementation

### Target file

`tests/test_mdq_config.py`

### Procedure

1. Create the file with a module docstring identifying it as covering `MdqConfig`
   construction/validation (both directly and via `MdqService.__init__`).
2. Add a test class/group for direct `MdqConfig` construction, e.g.:
   ```
   class TestMdqConfigDefaults:
       def test_all_defaults_construct_successfully(self): ...
           # MdqConfig() with no kwargs succeeds; assert default values match
           # mdq_service.py's pre-existing hardcoded literals (max_snippet_chars=500, etc.)

   class TestMdqConfigNumericValidation:
       @pytest.mark.parametrize("field", [
           "max_snippet_chars", "max_chunk_chars", "max_file_chars",
           "search_timeout_sec", "max_results_limit", "max_chars_per_chunk",
           "max_total_result_chars", "max_outline_items", "max_grep_matches",
           "max_chars_per_match", "max_outline_depth", "sqlite_busy_timeout",
       ])
       @pytest.mark.parametrize("bad_value", [0, -1])
       def test_rejects_non_positive(self, field, bad_value): ...
           # MdqConfig(**{field: bad_value}) raises pydantic.ValidationError

       @pytest.mark.parametrize("field", ["context_before", "context_after"])
       def test_rejects_negative_but_allows_zero(self, field): ...
           # MdqConfig(**{field: -1}) raises; MdqConfig(**{field: 0}) succeeds (ge=0, not gt=0)

   class TestMdqConfigListValidation:
       @pytest.mark.parametrize("field", ["allowed_dirs", "include_globs", "exclude_globs"])
       def test_rejects_non_list(self, field): ...
           # MdqConfig(**{field: "not-a-list"}) raises pydantic.ValidationError

   class TestMdqConfigBoolValidation:
       def test_rejects_non_bool_enable_grep(self): ...
           # MdqConfig(enable_grep="not-a-bool") raises, or document coercion behavior if
           # pydantic's default coercion accepts a particular value — assert the actually
           # observed behavior, do not assume without running it first
   ```
3. Add a service-construction integration test group, e.g.:
   ```
   class TestMdqServiceConfigValidation:
       def test_valid_config_constructs_service(self, tmp_path, monkeypatch): ...
           # patch ConfigLoader.load to return a valid dict; MdqService(db_path=...) succeeds

       def test_invalid_config_value_fails_construction(self, tmp_path, monkeypatch): ...
           # patch ConfigLoader.load to return {"max_snippet_chars": -5, ...};
           # MdqService(db_path=...) raises pydantic.ValidationError (not caught/swallowed)

       def test_missing_config_file_falls_back_to_defaults(self, tmp_path, monkeypatch): ...
           # patch ConfigLoader.load to raise FileNotFoundError;
           # MdqService(db_path=...) still constructs successfully (defaults are valid)
   ```
4. Use `tmp_path` fixture for `db_path` to avoid touching `/opt/llm/db/mdq.sqlite` during
   tests, matching the isolation pattern already used elsewhere in `tests/test_mdq_service.py`
   (check that file's fixtures for the exact `db_path`/tmp-dir convention and reuse it).

### Method

New pytest file, no production code changes. Pure unit tests against `MdqConfig` plus a small
number of integration tests exercising `MdqService.__init__`'s validation-failure path via
monkeypatched config loading.

### Details

- Do not assert on `pydantic`'s exact error message text (fragile across pydantic versions);
  assert only that `pydantic.ValidationError` (or `pydantic_core.ValidationError`, whichever
  is the actually-raised type in the installed pydantic version — verify with a quick
  interactive check before writing the assertion) is raised.
- If `TestMdqConfigBoolValidation.test_rejects_non_bool_enable_grep` reveals that pydantic's
  default (non-strict) coercion silently accepts some non-bool value (e.g. `"true"` → `True`
  or `1` → `True`), adjust the test to assert the actually-observed coercion behavior and add
  a corresponding note in the companion `mdq_models.py` doc's Details section about whether a
  `field_validator` was ultimately added to tighten it — do not leave the test silently wrong
  relative to actual runtime behavior.
