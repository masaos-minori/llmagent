# Implementation Procedure: cmd_db.py

## Goal

Delete `scripts/agent/commands/cmd_db.py` in full — `/db` (and its remaining
`session`/`help` subcommands) is removed entirely; equivalent functionality is
reachable only via `/session export|stats|health|checkpoint|vacuum|purge|recover`.

## Scope

### In scope
- Delete the file `scripts/agent/commands/cmd_db.py` (71 lines, current content:
  `_DbMixin(MixinBase, DbHelpDisplay, DbStatsDisplay)` with `_cmd_db` and
  `_cmd_db_session`; note the `/db rag` family and `_DbMixin._rag_ops` instantiation
  have already been removed by a prior, unrelated change — see Assumptions).
- Remove `_DbMixin` import/composition from `registry.py` (cross-referenced; see that
  file's own implementation doc).

### Out of scope
- `db_session_ops.py` (`DbSessionOps`) — reused verbatim by `cmd_session.py`'s new
  branches; not modified.
- `db_stats_display.py` — its `_db_session_stats` method is relocated (see that file's
  own implementation doc); `_db_stats` (dead) is deleted there, not here.
- `db_help_display.py` — deleted in its own step (this file was its sole consumer).

## Assumptions

- Verified by reading the current file in full: `_DbMixin` today only composes
  `MixinBase`, `DbHelpDisplay`, `DbStatsDisplay`, and only instantiates
  `DbSessionOps` (`self._session_ops = DbSessionOps(self._ctx, self._out)` in
  `__init__`). The `/db rag` subcommand family and `DbRagOps` are **already absent**
  from this file (removed by an earlier, unrelated change documented in
  `implementations/done/20260714-213701_04_mcp_02_03_dead-rag-settings-and-unused-commands.md`).
  `_cmd_db` today only dispatches `session` → `_cmd_db_session` and `help` → `_db_help`;
  anything else falls through to `write_validation_error("/db session <subcmd>")`.
  This narrows this plan's actual code deletion to the session+help dispatch shown
  below (no `/db rag` code remains to delete here).
- `DbHelpDisplay` (`db_help_display.py`) has no consumer besides this file (verified
  via `grep -rn "DbHelpDisplay\|_db_help\b\|_db_help_session" scripts/ tests/` — only
  hits in `cmd_db.py` and its own module/tests).
- `DbStatsDisplay` (`db_stats_display.py`) is mixed in here only for
  `_db_session_stats`/`_db_stats`; after this file is deleted, `_db_session_stats`
  needs a new home (moved into `cmd_session.py`'s mixin — see that file's own doc)
  and `_db_stats` (already dead, no caller) is deleted outright.

## Implementation

### Target file

`scripts/agent/commands/cmd_db.py`

### Procedure

1. Confirm current dispatch surface matches the Assumptions section above by
   re-reading the file immediately before deleting (guards against drift since this
   design doc was written).
2. Ensure `cmd_session.py`'s new `export`/`stats`/`health`/`checkpoint`/`vacuum`/
   `purge`/`recover` branches (see that file's own implementation doc) are implemented
   and passing tests **before** deleting this file, so `/session ...` functional
   parity is proven first.
3. Delete the file: `git rm scripts/agent/commands/cmd_db.py`.
4. In `registry.py`:
   - Remove `from agent.commands.cmd_db import _DbMixin`.
   - Remove `_DbMixin` from `CommandRegistry`'s base-class tuple.
   - Update the module docstring's mixin list to drop the `cmd_db.py — _DbMixin: /db` line.
   (Cross-referenced — belongs to `registry.py`'s own implementation step, must land
   in the same change-set.)

### Method

Whole-file deletion via `git rm`, coordinated with `command_defs_list.py`'s removal
of the `CommandDef("/db", ...)` entry (so no `CommandDef` ever points at a deleted
`_cmd_db` handler) and with `registry.py`'s removal of `_DbMixin` from its
composition.

### Details

- Deletion order within a single commit does not matter to Python (it's all staged
  together), but for incremental local testing, land `cmd_session.py`'s additions
  first, run its tests, then delete `cmd_db.py` + update `registry.py` +
  `command_defs_list.py` together, then run the full suite.
- `deploy/deploy.sh`'s `rsync -av --delete` over `scripts/` requires no edit (file
  removal is synced automatically).

## Validation plan

- `uv run pytest tests/test_agent_cmd_session.py -v` — new migrated tests (health,
  checkpoint, vacuum, purge, session-scope, backward-compat, export) pass **before**
  this file is deleted (proves parity first).
- After deletion: `uv run pytest tests/` — `tests/test_agent_cmd_db.py` must already
  be deleted in the same change-set (see its own implementation doc) or every test
  in it will fail with `ModuleNotFoundError` on `from agent.commands.cmd_db import _DbMixin`.
- `uv run pytest tests/test_command_def_sync.py -v` — passes once
  `INTENTIONAL_HANDLER_EXCLUSIONS` drops `_cmd_db_rag`/`_cmd_db_session` (they cease
  to exist entirely, so they must not be referenced as exclusions either — see that
  test file's own implementation doc).
- `grep -rn "cmd_db\b\|_DbMixin\|_cmd_db\b" scripts/ tests/` returns no matches except
  possibly historical doc/comment mentions outside code (documentation updates are a
  separate step in this plan).
- `PYTHONPATH=scripts uv run lint-imports` — no broken import.
- `uv run vulture scripts/agent/commands/ --min-confidence 80` — no new dead-code hit.
