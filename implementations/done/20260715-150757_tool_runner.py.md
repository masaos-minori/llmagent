# Implementation Procedure: scripts/agent/tool_runner.py

Source plan: `plans/20260715-133548_plan.md`

## Goal

Broaden `_build_tool_meta()`'s `is_write` / `requires_serial` classification
beyond `WRITE_TOOLS` / `DELETE_TOOLS` / `SHELL_TOOLS` so that Git write,
GitHub write/dangerous, CI/CD `trigger_workflow`, RAG `rag_delete_document`,
and MDQ `index_paths` / `refresh_index` / `fts_rebuild` are never silently
placed in the default parallel read-only group when `tool_definitions` lacks
explicit metadata — while still respecting explicit `resource_scope` /
`requires_serial` / `is_write` keys present in a tool definition's `function`
dict.

## Scope

**In scope:**
- `_build_tool_meta()` — replace the narrow
  `name in WRITE_TOOLS or name in DELETE_TOOLS` computation with one based on
  the extended `is_side_effect()` (from
  `implementations/20260715-150757_tool_executor_helpers.py.md`), and add
  `MDQ_SERIAL_TOOLS` to the `requires_serial` default check alongside the
  existing `SHELL_TOOLS` check.
- Import cleanup: add `MDQ_SERIAL_TOOLS` import; drop `WRITE_TOOLS` /
  `DELETE_TOOLS` imports from `shared.tool_constants` if (and only if) no
  other usage remains in the file after the edit.

**Out of scope:**
- No change to `fn.get("resource_scope", _default_scope)` / `fn.get(...)`
  override shape — explicit per-definition metadata continues to win.
- No change to `_compute_serial_overhead()` or any other function in this
  file.
- No change to `_execute_with_dag()` or `_execute_standard()` control flow
  (only the metadata *values* they consume, via `_build_tool_meta()`,
  change).
- Depends on `implementations/20260715-150757_tool_constants.py.md` and
  `implementations/20260715-150757_tool_executor_helpers.py.md` being applied
  first (this file imports `MDQ_SERIAL_TOOLS` from `tool_constants.py` and
  relies on the extended `is_side_effect()`).

## Assumptions

- `_build_tool_meta()` is the sole production caller that must reflect the
  new classification; no other code path builds `ToolSpec` from
  `tool_definitions` — confirmed via `ToolSpec(` grep across `scripts/`
  (only `tool_runner.py` and test helper files construct it).
- `is_side_effect()` (after the `tool_executor_helpers.py` edit) already
  includes `WRITE_TOOLS | DELETE_TOOLS`, so
  `name in WRITE_TOOLS or name in DELETE_TOOLS` is fully subsumed by
  `is_side_effect(name)` — no double logic needed.
- `fts_rebuild` gets `requires_serial=True` via an explicit `MDQ_SERIAL_TOOLS`
  membership check (mirroring how `SHELL_TOOLS` is already handled — a
  name-set check for a barrier condition distinct from write-classification),
  rather than relying solely on `is_write=True` from `is_side_effect()`.
- `SHELL_TOOLS` import stays (still used for the `requires_serial` check);
  only `WRITE_TOOLS` / `DELETE_TOOLS` are candidates for import removal, and
  only if grep confirms no other in-file usage.

    tool_definitions: list[dict],
) -> dict[str, ToolSpec]:
    """Build tool metadata map from tool_definitions for DAG execution."""
    tool_meta: dict[str, ToolSpec] = {}
    for td in tool_definitions:
        fn = td.get("function", {})

New `_build_tool_meta()` body:

```python
def _build_tool_meta(
    tool_definitions: list[dict],
) -> dict[str, ToolSpec]:
    """Build tool metadata map from tool_definitions for DAG execution."""
    tool_meta: dict[str, ToolSpec] = {}
    for td in tool_definitions:
        fn = td.get("function", {})
        name = fn.get("name", "")
        if name:
            _is_write = is_side_effect(name)
            _requires_serial = (
                fn.get("requires_serial", False)
                or name in SHELL_TOOLS
                or name in MDQ_SERIAL_TOOLS
            )
            _default_scope = name if _is_write else ""
            tool_meta[name] = ToolSpec(
                call_id="",
                name=name,
                resource_scope=fn.get("resource_scope", _default_scope),
                requires_serial=_requires_serial,
                is_write=fn.get("is_write", _is_write),
            )
    return tool_meta
```

Notes:
- `is_side_effect` is already imported in this file
  (`from shared.tool_executor_helpers import is_side_effect, tool_hash_key`)
  — no new import needed for it.
- `fn.get("is_write", _is_write)` is a *new* override point: previously
  `is_write` was always computed inline with no way for an explicit
  `tool_definitions` entry to override it. This preserves explicit `is_write`
  from `tool_definitions` (requirement item 3: "Preserve explicit metadata
  where safe") while still defaulting unclassified-but-known-side-effecting
  tools to `True`.
- `_default_scope` is now populated for the newly-classified families too
  (e.g. `trigger_workflow` → default scope `"trigger_workflow"`), so
  `trigger_workflow` calls land in a resource-scope group (serialized among
  themselves) rather than the generic `write_first` group.
- `fts_rebuild` is written to `requires_serial=True` via the explicit
  `MDQ_SERIAL_TOOLS` check, independent of whatever `is_side_effect()`
  returns for it (it will also be `True` there, but `requires_serial` is a
  separate field).
