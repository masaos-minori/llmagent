# Implementation procedure: `deploy/deploy.sh` (Browser MCP config copy)

Source plan: `plans/20260719-101501_plan.md`, Implementation step 4 (Design/Affected areas).

`deploy.sh` is a common target filename in this repo's history (e.g.
`implementations/done/20260717-203339_deploy.sh.md`). Checked by content: that doc addresses an
unrelated prior change; no existing doc under `implementations/` or `implementations/done/`
adds a `browser_mcp_server.toml` copy line. Flagged as checked, not a genuine overlap.

## Goal

Update `deploy/deploy.sh` so `config/browser_mcp_server.toml` is copied to
`/opt/llm/config/` during deployment, matching the pattern used for every other server's
per-server TOML.

## Scope

**In scope**
- One new `cp` line for `config/browser_mcp_server.toml`, added to the existing MCP-server
  config copy block (`deploy/deploy.sh:79-89`, verified).

**Out of scope — verified drift from the plan's own wording.** The plan's Design/Affected-areas
text says: "add `scripts/mcp_servers/browser/*.py` to the copy list" and Implementation step 4
says "Update `deploy/deploy.sh` ... (per checklist steps)," implying a per-file `.py` copy list
similar to the config-file list. **Direct read of the full file (138 lines) shows this is not
how `scripts/` is deployed**: `deploy/deploy.sh:68-73` copies the entire `scripts/` tree in one
shot via `rsync -av --delete --exclude="__pycache__/" --exclude="*.pyc" --exclude="*.pyo"
"${REPO_ROOT}/scripts/" "${DEPLOY_SCRIPTS}/"` — there is no per-file or per-server `.py` copy
list anywhere in this script; every `.py` file under `scripts/`, including the four new
`scripts/mcp_servers/browser/*.py` files, is picked up automatically by this `rsync` once they
exist in the repo, with **no edit to `deploy.sh` required for the `.py` files themselves**. The
only actual `deploy.sh` edit this plan requires is the explicit per-server `cp` line for the new
**config TOML** (`config/browser_mcp_server.toml`), since config files (unlike `scripts/`) are
copied individually, one `cp` line per file (`deploy/deploy.sh:76-95`, verified — every MCP
server's own `*_mcp_server.toml` has its own explicit `cp` line; there is no wildcard/rsync for
`config/*.toml`). This doc's Procedure reflects this corrected, verified scope — not the plan's
literal "copy list" wording for `.py` files.

## Assumptions

1. Verified directly (`deploy/deploy.sh:67-73`): the `scripts/` copy step is a single `rsync`
   invocation with `--delete`, meaning any file present in the deploy target but removed from
   the source repo is also removed on deploy — this is already correct, generic behavior that
   requires zero changes for a new server package; only new *config* files need an explicit
   line.
2. Verified directly (`deploy/deploy.sh:76-95`): every existing MCP server has exactly one `cp`
   line under the `# MCP サーバー (各サーバー専用設定; agent.toml は読み込まない)` comment
   block (lines 79-89), in the same order as roughly their historical addition (not
   alphabetical, not port-ordered) — `web_search`, `github`, `file_read`, `file_write`,
   `file_delete`, `shell`, `rag_pipeline`, `cicd`, `mdq`, `git`. The new `browser` line can be
   appended after the last one (`git_mcp_server.toml`, line 89) without needing to match any
   particular ordering convention.
3. `tests/` deployment (`deploy/deploy.sh:120-126`) is also a whole-directory `rsync`, so the new
   `tests/test_browser_mcp_service.py` (see paired doc) requires no `deploy.sh` edit either —
   confirmed by the same reasoning as Assumption 1.

## Implementation

### Target file

`deploy/deploy.sh`.

### Procedure

1. After line 89 (`cp "${REPO_ROOT}/config/git_mcp_server.toml"            "${DEPLOY_CONFIG}/"`),
   insert:
   ```bash
   cp "${REPO_ROOT}/config/browser_mcp_server.toml"        "${DEPLOY_CONFIG}/"
   ```
   (align the trailing whitespace before `"${DEPLOY_CONFIG}/"` to match the existing column
   alignment used by the surrounding lines — verify the exact spacing at implementation time
   since it is cosmetic-but-conventional in this file, not enforced by any linter).
2. Do not add any line for `scripts/mcp_servers/browser/*.py` — covered automatically by the
   existing `rsync` step per the Scope note above.
3. Do not add any line for `tests/test_browser_mcp_service.py` — covered automatically by the
   existing `rsync` step per Assumption 3.

### Method

Single-line additive edit; no restructuring of the script.

### Details

No code logic beyond a `cp` invocation; this is a deployment shell script, not a Python module.
