# Implementation: Reconcile SQLite vs JSONL divergence in EventBus

Steps covered: Plan 20260626-094718 — Steps 1-1, 2-1, 2-2, 2-3

---

## Goal

Identify and resolve the divergence between SQLite (authoritative event store) and JSONL (log/backup format) in the EventBus. Ensure that the canonical read path always uses SQLite, and that JSONL is only used for archiving/debugging.

---

## Scope

- **In scope**: `scripts/eventbus/app.py` and/or `scripts/eventbus/db.py` — eliminate any code that reads from JSONL as a primary data source
- **Out of scope**: docs (step 3-1)

---

## Assumptions

- Both SQLite and JSONL exist; JSONL was an earlier storage format.
- Some endpoints may still read from JSONL when SQLite has the same data.
- The fix: audit all read paths, remove JSONL reads where SQLite is authoritative.

---

## Implementation

### Target files
`scripts/eventbus/app.py`, `scripts/eventbus/db.py`

### Procedure
1. Step 1-1 (Audit): `grep -rn "jsonl\|JSONL\|\.jsonl" scripts/eventbus/` to find all JSONL read paths.
2. For each JSONL read:
   - Is the same data available in SQLite? → Yes: remove JSONL read, use SQLite.
   - Is JSONL the only store for this data? → Document gap and plan SQLite migration.
3. Step 2-1 (`app.py`): Remove any route handler that reads from JSONL for primary response data.
4. Step 2-2 (`db.py`): Remove any JSONL fallback in query methods.
5. Step 2-3 (Write path): Ensure that if JSONL is still written (for archiving), it is never read back. Add a comment: "JSONL is write-only; canonical reads use SQLite."

### Method
Audit + targeted removal. Risk: if JSONL has data not yet in SQLite (schema migration gap), removing JSONL reads causes data loss. Investigate before removing.
