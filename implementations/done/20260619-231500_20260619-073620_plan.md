## Step 4: Add signature validation at registration time

**Status:** Partially implemented

**Current code** (`plugin_registry.py:75-91`):
- Already checks return type annotation and warns if wrong type
- Does NOT warn if return annotation is missing entirely

**Changes:**

In `register_tool()` decorator, add check before the existing type check:

```python
def decorator(fn: _F) -> _F:
    _tools[name] = fn
    hints = typing.get_type_hints(fn)
    return_hint = hints.get("return")
    if return_hint is None:
        logger.warning("[plugin] warn: tool '%s' missing return type annotation", name)
    elif return_hint != tuple[str, bool]:
        logger.warning(
            "[plugin] warn: tool '%s' expected return type 'tuple[str, bool]', got %s",
            name,
            return_hint,
        )
    logger.debug("[plugin] tool registered: %s", name)
    return fn
```

**Completion criterion:**
- Tools without return annotations produce a startup warning with `[plugin]` prefix

## Implementation Notes

### Data structure changes

The main structural change is storing `(fn, module_name)` tuples in `_tools` and `(fn, prefix, module_name)` in `_commands`. This requires updating:

1. `register_tool()` - store tuple
2. `register_command()` - store tuple
3. `get_tool()` - return `fn` from tuple (or update callers)
4. `iter_tools()` - return dict with `fn` values extracted
5. `get_command()` - return `(fn, prefix)` from tuple (or update callers)
6. `iter_commands()` - return dict with `(fn, prefix)` values extracted

Since `get_tool()` and `get_command()` are public accessors used by other modules, the safest approach is to keep their return signatures unchanged and extract the function/prefix from the stored tuples internally.

### Backward compatibility

- `_reset_for_testing()` must clear the registries (already does via `.clear()`)
- Existing tests that use `get_tool()` and `get_command()` should continue to work if accessor methods extract values from tuples
- Factory.py `_init_plugin_registry()` needs to handle new return types from validation functions

### Logging prefix convention

All new log messages use `[plugin]` prefix for easy grep/awk filtering:
- `[plugin] loaded: <module>`
- `[plugin] skipped: <module> (<reason>)`
- `[plugin] loaded=N, skipped=M`
- `[plugin] conflict: tool '<name>' in '<module>' shadows MCP tool — <reject/allow>`
- `[plugin] tool conflicts: shadowed=N, allowed=M`
- `[plugin] command shadow: '<name>' in '<module>' shadows built-in`
- `[plugin] warn: tool '<name>' missing return type annotation`
