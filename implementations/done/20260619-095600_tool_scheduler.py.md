# Implementation: Log Serialization Events from build_execution_groups

## Goal

Log when `build_execution_groups()` produces serial groups (requires_serial, resource_scope write conflict, write-first grouping), with the trigger tool name and reason, so every serialization event is visible in logs.

## Scope

- `scripts/agent/tool_scheduler.py` — add logger and emit per-group serialization event
- `scripts/agent/tool_runner.py` — in `_execute_with_dag()`, log group structure after `build_execution_groups()` returns

Out of scope:
- `_execute_standard()` path (already has "Side-effect tool detected" log at line 207-210)
- Changing grouping logic
- Metrics aggregation (Step 2)

## Assumptions

1. `tool_scheduler.py` is a pure function module with no logger (confirmed — no `import logging`).
2. The plan says to log from `tool_scheduler.py`, but logging from `_execute_with_dag()` in `tool_runner.py` is cleaner (avoids injecting a logger into the pure function). Either approach is valid; this doc chooses `tool_runner.py` as the logging point.
3. Reason values must be deterministic from the group structure:
   - Single-element groups with `requires_serial=True` → reason: `requires_serial`
   - Multi-element groups with same `resource_scope` → reason: `resource_scope_conflict`
   - `write_first` group (no scope, is_write) → reason: `is_write_overlap`
4. The prescribed log format: `"ROUND_SERIALIZATION: triggered by {tool_name} ({reason}) — {N} tools serialized in this round"`

## Implementation

### Target file

`scripts/agent/tool_runner.py` (logging site); `scripts/agent/tool_scheduler.py` (optional: add reason to return value)

### Procedure

**Option A (minimal — log in tool_runner.py from group structure):**

After `groups = build_execution_groups(approved_calls, tool_meta)` (line 178), iterate groups and classify reason by inspecting each non-final group.

**Option B (preferred — return reason from tool_scheduler.py):**

Modify `build_execution_groups()` to return `list[tuple[list[dict], str]]` where the str is the reason tag. Caller (`_execute_with_dag`) logs the reason. More explicit; avoids re-inferring reason from group structure.

### Method

Use Option A first (no interface change) to keep blast radius low. Reason is inferred from `tool_meta`:
- If group has 1 tool and `tool_meta[name].requires_serial` → reason `requires_serial`
- If group has multiple tools and they share a `resource_scope` → reason `resource_scope_conflict`
- If group has multiple tools with `is_write=True` and no scope → reason `is_write_overlap`

### Details

**tool_runner.py — add import:**
```python
import logging
logger = logging.getLogger(__name__)
```
(if not already present — verify)

**tool_runner.py — after line 178 (`groups = build_execution_groups(...)`):**
```python
for group in groups:
    if len(group) == 1:
        name = group[0]["function"]["name"]
        meta = tool_meta.get(name)
        if meta is not None and meta.requires_serial:
            logger.info(
                "ROUND_SERIALIZATION: triggered by %s (requires_serial)"
                " — 1 tool serialized in this round",
                name,
            )
    elif len(group) > 1:
        names = [tc["function"]["name"] for tc in group]
        first_meta = tool_meta.get(names[0])
        scope = first_meta.resource_scope if first_meta else ""
        reason = "resource_scope_conflict" if scope else "is_write_overlap"
        logger.info(
            "ROUND_SERIALIZATION: triggered by %s (%s)"
            " — %d tools serialized in this round",
            names[0],
            reason,
            len(group),
        )
```

Note: The final `parallel` group is not logged (it is not a serialization event).
