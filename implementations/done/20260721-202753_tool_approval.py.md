# Implementation Procedure: Restrict approval dry-run execution to dry-run-capable tools

## Goal

Restrict approval dry-run execution to tools known to support `dry_run`, preventing HIGH-risk false denials when unsupported tools are listed in `approval_dry_run_tools`.

## Scope

- `scripts/agent/tool_approval.py` — define dry-run-capable tool set, modify `_build_preview_with_dry_run()` to check capability before executing
- `scripts/agent/config_dataclasses.py` — add dry-run capability metadata or validation hook
- `scripts/agent/config_builders.py` — add startup validation for `approval_dry_run_tools`
- `scripts/agent/services/config_validators.py` — add dry-run tool validation function
- Tests: `tests/test_tool_approval_preflight.py`, `tests/test_tool_approval_repos.py`, `tests/test_tool_approval_risk.py`

## Assumptions

1. Only file-oriented tools (`write_file`, `edit_file`, `create_directory`, `delete_file`, `delete_directory`, `move_file`) currently support `dry_run`
2. Warning + ignoring unsupported entries is preferred over strict rejection unless strict config mode is enabled
3. Dry-run failures from transport/server errors should be treated as unavailable (downgrade to text-only preview), not as blocking errors

## Implementation

### Target file

`scripts/agent/tool_approval.py`

### Procedure

1. In `tool_approval.py`, create a canonical frozenset of dry-run-capable tool names based on `_DEFAULT_DRY_RUN_TOOLS`:
   ```python
   _DRY_RUN_CAPABLE_TOOLS = frozenset([
       "write_file", "edit_file", "create_directory",
       "delete_file", "delete_directory", "move_file"
   ])
   ```

2. In `_build_preview_with_dry_run()`, before calling `ctx.services_required.tools.execute(..., dry_run=True)`, add capability check:
   ```python
   if tool_name not in _DRY_RUN_CAPABLE_TOOLS:
       logger.warning("Tool %s does not support dry_run; using static preview", tool_name)
       return build_preview(tool_name, args)
   ```

3. Distinguish dry-run failure categories:
   - Unsupported dry-run argument → treat as unavailable (same as transport error)
   - Transport/server failure → treat as unavailable (current behavior)
   - Valid dry-run result with blocking error → keep current HIGH-risk denial behavior

4. In `config_builders.py` or `config_validators.py`, add startup validation:
   ```python
   def _validate_dry_run_tools(tools: list[str]) -> list[str]:
       """Validate and filter dry-run tools at startup."""
       supported = _DRY_RUN_CAPABLE_TOOLS
       filtered = []
       for tool in tools:
           if tool not in supported:
               logger.warning(
                   "approval_dry_run_tools contains unsupported tool: %s; dry-run will use static preview",
                   tool
               )
           else:
               filtered.append(tool)
       return filtered
   ```

5. Call this validation after loading `approval_dry_run_tools` — warn but do NOT reject startup.

6. Update tests to cover:
   - Supported dry-run tool appends dry-run output to preview
   - Unsupported dry-run tool uses static preview (no execute call)
   - HIGH-risk unsupported tool is not denied solely due to missing dry-run support
   - HIGH-risk supported tool with real dry-run blocking error IS denied

### Method

Use targeted edits to add the dry-run capability check in the existing code path. Use startup-time validation to warn on misconfiguration rather than failing fast.

### Details

- Default dry-run tools are defined in both `config_builders.py` (line 107-114) and `config_dataclasses.py` (line 331-340) — consistent but no validation currently exists
- Dry-run failures raise `ApprovalPreviewBlockingError` for HIGH risk or `ApprovalPreviewError` for MEDIUM risk — this behavior remains unchanged for capable tools
- Blast radius: Medium — affects approval flow for all tools listed in `approval_dry_run_tools`; HIGH-risk denial behavior changes for unsupported tools
