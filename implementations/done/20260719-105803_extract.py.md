# Implementation procedure: `scripts/agent/memory/extract.py` (chunking stage)

Source plan: `plans/20260719-095637_plan.md` ("Enable the persistent memory layer by default and add the
missing chunking step", requirement `requires/done/20260714_15_require.md`), Implementation steps 1-2.

No existing implementations doc (under `implementations/` or `implementations/done/`) matches this
specific change. `implementations/done/20260626-100122_extract.py.md`, `implementations/done/20260610-114915_extract_py.md`,
`implementations/done/20260606-194532_extract.md` and several other `extract`-named docs exist under
`implementations/done/`, but all predate this batch and address unrelated earlier topics (initial
extraction-policy design, compat removal, unrelated refactors) — none proposes replacing the hard
truncation with a chunking stage or changing the return type of `_try_extract_from_assistant`/
`_try_extract_from_user`/`extract_memories`. Flagged as checked, not a genuine overlap.

## Goal

Replace the hard truncation in `_try_extract_from_assistant` (and add equivalent size-bounding to
`_try_extract_from_user`, which currently has none) with a chunking stage, so over-long extracted
content is split into multiple `MemoryEntry` rows instead of being cut and discarded. `extract_memories`
must collect all chunk-entries from every source message instead of at most one entry per message.

## Scope

**In scope**
- Add a new private helper `_split_content(content: str, max_chars: int) -> list[str]` to `extract.py`.
- Change `_try_extract_from_assistant` (`extract.py:218-257`) to stop truncating and instead split
  `content` into chunks, returning one `MemoryEntry` per chunk.
- Change `_try_extract_from_user` (`extract.py:260-293`) to accept a new `max_content_chars: int`
  parameter, apply the same chunking helper, and return one `MemoryEntry` per chunk.
- Change both functions' return type from `MemoryEntry | None` to `list[MemoryEntry] | None`.
- Update `extract_memories` (`extract.py:296-361`) call sites: replace the shared
  `entry = ...; if entry is not None: candidates.append(entry)` pattern with
  `chunks = ...; if chunks: candidates.extend(chunks)`, and pass `max_content_chars` to the
  `_try_extract_from_user` call site (which does not receive it today).

**Out of scope**
- `scripts/agent/memory/ingestion.py` — no change; its `for entry in entries:` loop
  (`ingestion.py:90-91`) already handles a longer `entries` list correctly (re-verified directly, see
  Assumptions).
- `scripts/rag/ingestion/chunk_splitter.py` — not reused (file-in/file-out, RAG-metadata-coupled, no
  plain string-splitting public function; confirmed by the plan's own investigation, not re-verified
  line-by-line in this pass).
- `scripts/db/schema_sql.py`, `scripts/agent/memory/retriever.py`, `scripts/agent/memory/rrf.py` — untouched.
- The importance/classification heuristics themselves (`_classify_content`, `_importance_from_content`)
  are not changed in logic — only the content they receive changes (full content instead of
  pre-truncated content; see Assumption 3).

## Assumptions

1. Confirmed by direct read (`scripts/agent/memory/extract.py`): `_try_extract_from_assistant` spans
   lines 218-257, `_try_extract_from_user` spans lines 260-293, and `extract_memories` spans lines
   296-361. `_make_entry` (113-144) and `_make_entry_with_importance` (147-178) are the two entry
   builders; both build a **new** `MemoryEntry(memory_id=str(uuid.uuid4()), ...)` per call, and
   `MemoryEntry` is `@dataclass(frozen=True)` (`types.py:82-83`) — confirming that producing N chunk
   entries requires N separate builder calls (no in-place mutation possible), which is exactly what the
   per-chunk loop below does.
2. The truncation being replaced is at `extract.py:234-235`:
   `if max_content_chars > 0 and len(content) > max_content_chars: content = content[:max_content_chars]`
   — inside `_try_extract_from_assistant`, applied **before** classification (`_classify_content`,
   line 238) and importance scoring (`_importance_from_content`, line 242-244).
3. **Correction to the plan's framing**: the plan's Design step 2 describes replacing truncation "in
   `_try_extract_from_assistant`/`_try_extract_from_user`" as if both functions currently truncate
   symmetrically. Direct read of `_try_extract_from_user` (`extract.py:260-293`) shows it has **no**
   `max_content_chars` parameter and **no** truncation logic at all today — user-message content is
   currently extracted unbounded. Adding chunking to `_try_extract_from_user` is therefore not "replacing
   an existing truncation" but introducing a new size-bounding parameter to that function, per the
   plan's Design intent (Design step 2's literal text: build one `MemoryEntry` per chunk in both
   functions). The Implementation step 2 below reflects this: `_try_extract_from_user` gains a new
   keyword parameter, and its `extract_memories` call site (currently `extract.py:337-347`, which does
   **not** pass `max_content_chars`) must be updated to pass it.
4. **Classification/importance now run on full (untruncated) content** rather than truncated content,
   since the truncate-then-classify order in `_try_extract_from_assistant` is being replaced by
   classify-then-split. This is an intentional, minor behavior improvement (classification keyword
   search previously missed any keyword occurring after the truncation cutoff; it no longer does) and is
   consistent with the plan's Risk mitigation ("no regression versus current behavior, only strictly
   better — nothing is discarded"). Explicitly call this out in the PR/commit description at
   implementation time so it is not mistaken for an unintended side effect.
5. `extract_memories`'s current per-message handling (`extract.py:323-351`) computes a single `entry`
   per message (from either branch or `None`) then does one shared
   `if entry is not None: candidates.append(entry)`. This becomes `if chunks: candidates.extend(chunks)`
   using the same variable name pattern (rename `entry` to `chunks` at both call sites) — a minimal,
   localized diff.
6. `ingestion.py`'s `on_session_stop` (`ingestion.py:67-96`) calls `extract_memories(...)` and then
   `for entry in entries: await self._persist_entry_with_dedup(entry)` (`ingestion.py:87-91`, exact
   lines re-verified in this cycle) — this loop is agnostic to how many entries came from a single
   source message, so it requires **no change** once `extract_memories` returns more entries per
   over-long input. Matches the plan's Assumption 2; independently re-confirmed here, not contradicted.
7. `memory_max_content_chars` (`config_dataclasses.py:220`, default `500`, confirmed by direct read) is
   read live from `config/agent.toml` (`config_builders.py:205`,
   `memory_max_content_chars=int(cfg.get("memory_max_content_chars", 500))` — not on any
   removed/forbidden-key list) and becomes, per this change, the **per-chunk** size limit rather than a
   total-content cap. No config key changes are needed for this file's edit.

### Method

Pure function addition (`_split_content`) plus a signature/return-type change on two existing private
functions and their call sites. No new classes, no `Protocol`/`ABC`, no new module. Kept in `extract.py`
itself rather than a new `chunk.py` module (implementer's choice per the plan) to avoid an unnecessary
new file and barrel-export churn (`__init__.py`, `docs/05_agent_12_03...` line 37 lists `extract_memories`/
`ExtractionPolicy` as the barrel's public "抽出" surface; `_split_content` stays private and is not
re-exported).

### Details

`_split_content` pseudocode (boundary-preserving, paragraph-first, hard-cut fallback — reuses the
existing truncation's own fallback semantics per the plan's Design step 1 and Assumption 3 in the
source plan):

```
def _split_content(content: str, max_chars: int) -> list[str]:
    if max_chars <= 0 or len(content) <= max_chars:
        return [content]
    chunks: list[str] = []
    current = ""
    for para in content.split("\n\n"):
        candidate = f"{current}\n\n{para}" if current else para
        if len(candidate) <= max_chars:
            current = candidate
            continue
        if current:
            chunks.append(current)
            current = ""
        if len(para) <= max_chars:
            current = para
        else:
            # hard-cut fallback — same character-slicing semantics as today's
            # truncation, applied per max_chars-sized slice instead of once.
            for start in range(0, len(para), max_chars):
                chunks.append(para[start : start + max_chars])
    if current:
        chunks.append(current)
    return chunks
```

Edge cases to cover in the procedure: `max_chars <= 0` (today's guard `max_content_chars > 0` is
preserved — non-positive means "no limit", single chunk); content exactly at the limit (single chunk,
no split); content with no `"\n\n"` boundaries at all that exceeds the limit (falls straight to the
hard-cut branch — same as today's behavior, just chunked instead of truncated).

`MemoryEntry` field reuse per chunk: all fields except `content`/`summary`/`memory_id` (and
`created_at`/`updated_at`, both set to the same `now` for all chunks from one source message — chunks
share a timestamp since they originate from one extraction pass) are identical across chunks; `summary`
is recomputed per chunk via `_make_summary(chunk)` inside the existing builder (no separate summary
logic needed).
