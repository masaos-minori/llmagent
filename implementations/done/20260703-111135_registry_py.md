## Goal

Fix `CommandRegistry._dispatch_plugin()` to use word-boundary prefix matching so that a plugin command `/foo` with `prefix=True` no longer false-positively dispatches `/foobar`.

## Scope

- In-Scope:
  - Replace `line.startswith(cmd_name)` with `line == cmd_name or line.startswith(cmd_name + " ")` inside the `is_prefix` branch of `_dispatch_plugin()`
  - Change `args = line[len(cmd_name):]` to `args = line[len(cmd_name):].strip()` in that same branch
- Out-of-Scope:
  - Plugin command shadowing policy
  - Plugin reload behavior
  - `dispatch()` method (built-in commands) — already uses correct boundary matching at line 122
  - `shared/plugin_registry.py` — no changes needed
  - Any other method in `CommandRegistry`

## Assumptions

1. `_dispatch_plugin()` is the sole code path that performs plugin prefix matching; no caller outside this method does its own prefix check against plugin command names.
2. `plugin_registry.iter_commands()` returns a `dict[str, tuple[Callable, bool]]` snapshot; iteration order and contract are stable.
3. Existing plugin handlers that receive `args` expect a stripped string — consistent with how built-in prefix commands expose args via `dispatch()` (line 123: `args = line[len(cmd.name):]` without strip, but the fix aligns plugin behavior to the spec requirement).
4. No deployed plugin in `/opt/llm/plugins/` intentionally relies on the current loose `startswith` behavior to match `/foobar` when registered as `/foo`.

### Method

- Pattern to use: same word-boundary guard already present in `dispatch()` at line 122:
  ```python
  if line == cmd.name or line.startswith(cmd.name + " "):
  ```
- `str.startswith(cmd_name + " ")` — ensures a space separator exists between the command name and any arguments, preventing substring collisions.
- `.strip()` — removes leading/trailing whitespace from the extracted args substring, consistent with the spec requirement and test expectations.
