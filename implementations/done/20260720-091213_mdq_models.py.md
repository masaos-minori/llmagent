# Implementation: scripts/mcp_servers/mdq/mdq_models.py (add `MdqConfig` validation model)

Source plan: `plans/20260719-205532_plan.md` ("Strengthen MDQ read-time authorization, deny-all
visibility, and config validation"), Design §3 ("Config validation (`MdqConfig`)"),
Implementation steps Phase 1, item 1.

## Goal

Add a new Pydantic `BaseModel` named `MdqConfig` to `mdq_models.py` that validates every
type/range-constrained config field currently read ad hoc via `mdq_cfg.get(...)` in
`MdqService.__init__`, so invalid values (wrong type, non-positive numeric bounds) raise a
`pydantic.ValidationError` instead of silently defaulting or being read unvalidated.

## Scope

**In scope:**
- Add one new class, `MdqConfig(BaseModel)`, to `scripts/mcp_servers/mdq/mdq_models.py`.
- Cover every numeric/list/bool field currently read in `MdqService.__init__`
  (`scripts/mcp_servers/mdq/mdq_service.py:59-81`): `allowed_dirs`, `include_globs`,
  `exclude_globs`, `max_snippet_chars`, `max_chunk_chars`, `max_file_chars`,
  `search_timeout_sec`, `enable_grep`, `max_grep_matches`, `max_chars_per_match`,
  `context_before`, `context_after`, `max_results_limit`, `max_chars_per_chunk`,
  `max_total_result_chars`, `max_outline_items`, `max_outline_depth`, `sqlite_busy_timeout`.
- Add `field_validator`/`Field` constraints per Design §3's pseudocode: `gt=0` for all
  "must be positive" numeric fields, `ge=0` for `context_before`/`context_after` (zero is a
  valid "no context lines" setting), plain `list[str]`/`bool` typing for the rest.

**Out of scope:**
- `db_path` — deliberately excluded from `MdqConfig` (Design §3: "`db_path` is not in the
  validated list"); it is a plain string with no meaningful range/type constraint beyond
  "is a string", and `MdqService.__init__` already handles it via a separate
  `db_path or mdq_cfg.get("db_path", ...)` expression outside the `MdqConfig` object.
  Do not add a `db_path` field to `MdqConfig`.
  See the companion `mdq_service.py` implementation doc for how `MdqConfig` is consumed —
  wiring `MdqConfig(**mdq_cfg)` into `MdqService.__init__` is out of scope for *this* file's
  document.
- Any change to existing request/response models (`SearchDocsRequest`, `GetChunkRequest`,
  `GrepDocsRequest`, etc.) — untouched.
- Any change to the domain exception classes (`MdqServiceError` and subclasses,
  `mdq_models.py:59-84`) — untouched; `MdqConfig` validation failures surface as
  `pydantic.ValidationError`, not a custom `Mdq*Error` subclass (this is deliberate per the
  plan's "fail-fast on invalid config values" framing — startup should crash loudly, not be
  caught and translated into a domain error a caller might handle gracefully).

## Assumptions

1. Current `mdq_models.py` (248 lines) imports `from pydantic import BaseModel` only
   (`mdq_models.py:11`) — `Field` and `field_validator` are not yet imported; both must be
   added to this import line.
2. Default values must exactly match the literals currently hardcoded in
   `MdqService.__init__` (`mdq_service.py:59-81`), confirmed by direct read:
   `include_globs=["*.md"]`, `exclude_globs=[".git/**", "__pycache__/**"]`,
   `max_snippet_chars=500`, `max_chunk_chars=10000`, `max_file_chars=100000`,
   `search_timeout_sec=30`, `enable_grep=True`, `max_grep_matches=200`,
   `max_chars_per_match=500`, `context_before=2`, `context_after=2`,
   `max_results_limit=100`, `max_chars_per_chunk=10000`, `max_total_result_chars=100000`,
   `max_outline_items=500`, `max_outline_depth=6`, `sqlite_busy_timeout=5000`,
   `allowed_dirs=[]` (via `mdq_cfg.get("allowed_dirs") or []`).
3. Per the plan's UNK-03, `max_outline_depth` and `sqlite_busy_timeout` — not explicitly
   named in the source requirement's validation list — are included under the same `gt=0`
   constraint for consistency, since both are already read as ints with no validation today.
4. Per the plan's UNK-04, a Pydantic `BaseModel` (not a hand-rolled validator function) is
   used, matching the file's existing convention of defining all request/response contracts
   as `BaseModel` subclasses.
5. Plain pydantic v2 type coercion (no `strict=True`) is used first; a companion note in the
   plan flags that if a `field_validator` proves necessary for stricter list/bool type
   rejection (e.g. rejecting `"true"` string → bool coercion), it should be added during
   actual implementation and verified with a unit test — the exact strictness knob is an
   implementation-time decision, not fixed in this design document.

## Implementation

### Target file

`scripts/mcp_servers/mdq/mdq_models.py`

### Procedure

1. Update the import line at `mdq_models.py:11` from:
   ```python
   from pydantic import BaseModel
   ```
   to:
   ```python
   from pydantic import BaseModel, Field
   ```
   (add `field_validator` here too only if step 4 below is actually needed.)
2. Insert the new `MdqConfig` class. Placement: after the domain-exception block
   (`mdq_models.py:55-84`, ending with `MdqConsistencyError`) and before
   `ParseMarkdownRequest` (`mdq_models.py:87`) — grouping it with other
   non-request/response "service-level" pydantic contracts, ahead of the per-tool
   request/response models that follow.
3. Signature/shape (pseudocode, not production code):
   ```
   class MdqConfig(BaseModel):
       allowed_dirs: list[str] = []
       include_globs: list[str] = ["*.md"]
       exclude_globs: list[str] = [".git/**", "__pycache__/**"]
       max_snippet_chars: int = Field(default=500, gt=0)
       max_chunk_chars: int = Field(default=10000, gt=0)
       max_file_chars: int = Field(default=100000, gt=0)
       search_timeout_sec: int = Field(default=30, gt=0)
       max_results_limit: int = Field(default=100, gt=0)
       max_chars_per_chunk: int = Field(default=10000, gt=0)
       max_total_result_chars: int = Field(default=100000, gt=0)
       max_outline_items: int = Field(default=500, gt=0)
       max_grep_matches: int = Field(default=200, gt=0)
       max_chars_per_match: int = Field(default=500, gt=0)
       context_before: int = Field(default=2, ge=0)
       context_after: int = Field(default=2, ge=0)
       enable_grep: bool = True
       max_outline_depth: int = Field(default=6, gt=0)
       sqlite_busy_timeout: int = Field(default=5000, gt=0)
   ```
4. If, during implementation-time unit testing (companion `tests/test_mdq_config.py` doc),
   plain pydantic coercion turns out to accept invalid shapes that should be rejected (e.g.
   a bare string coerced element-wise into `list[str]`, or `"yes"` coerced into `bool`), add
   a `field_validator("allowed_dirs", "include_globs", "exclude_globs", mode="before")`
   that raises `ValueError` unless the input is already a `list`, and/or set
   `model_config = ConfigDict(strict=False)` explicitly with a comment noting why strict
   mode was not chosen (pydantic already coerces int-like strings for numeric fields, which
   is acceptable per the plan; only container/bool coercion is the concern).

### Method

Additive-only: one new class inserted into an existing file. No existing class, field, or
import is removed or renamed. `Field`/`field_validator` join the existing `pydantic` import
rather than replacing it.

### Details

- Do not add this model to `__all__` or any export list — `mdq_models.py` has no `__all__`
  today (confirmed by reading the full 248-line file); nothing to update there.
- `MdqConfig` has no consumer defined in this file — the companion `mdq_service.py`
  implementation doc covers `MdqService.__init__`'s `MdqConfig(**mdq_cfg)` call site. Land
  both changes together; `MdqConfig` alone (unused) is inert and imports cleanly on its own.
- Keep field order in `MdqConfig` matching the order the plan's Design §3 pseudocode lists
  them in, for straightforward side-by-side review against the plan.
