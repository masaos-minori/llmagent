# Implementation: tool_spec.py (Phase 7 — ToolSpec formalization)

## Goal

Create `scripts/shared/tool_spec.py` with a `ToolSpec` dataclass that formalizes the
`resource_scope`, `requires_serial`, and `is_write` metadata currently scattered across
`tool_runner.py`'s inline dict construction. This makes tool execution metadata
inspectable, typed, and testable.

## Scope

**In**:
- New file `scripts/shared/tool_spec.py`
- `scripts/agent/tool_runner.py` — replace inline dict with `ToolSpec` instances

**Out**: No changes to `ToolExecutor`, `route_resolver.py`, or MCP server protocol.
The DAG execution logic in `tool_runner.py` is adjusted to read from `ToolSpec` fields
rather than dict keys — behavior is unchanged.

## Assumptions

1. `tool_runner.py` L152-163 builds inline dicts:
   ```python
   {
       "name": name,
       "args": args,
       "resource_scope": fn.get("resource_scope", ""),
       "requires_serial": fn.get("requires_serial", False),
       "is_write": name in WRITE_TOOLS or name in DELETE_TOOLS,
   }
   ```
2. These dicts are consumed within `_execute_with_dag()` only. Converting to `ToolSpec`
   is an internal refactor with no external API change.
3. `tool_spec.py` must stay within `shared/` — no `agent/` imports.

## Implementation

### Target file

`scripts/shared/tool_spec.py` (new), `scripts/agent/tool_runner.py` (modify)

### Procedure

1. Create `scripts/shared/tool_spec.py` with `ToolSpec` dataclass.
2. In `tool_runner.py`: import `ToolSpec`; replace the dict construction and access.

### Method

**`scripts/shared/tool_spec.py`**:

```python
#!/usr/bin/env python3
"""shared/tool_spec.py
Typed metadata for one tool call in the execution DAG.
"""

from __future__ import annotations

from dataclasses import dataclass, field


@dataclass(frozen=True)
class ToolSpec:
    """Execution metadata for a single approved tool call.

    Fields:
        call_id:        LLM-assigned tool call id (from tool_calls[].id)
        name:           Tool function name
        args:           Parsed argument dict
        resource_scope: Resource path/branch string for conflict detection ("" if none)
        requires_serial: True when the tool must not run concurrently with others
        is_write:       True when the tool has write/delete side effects
    """

    call_id: str
    name: str
    args: dict = field(default_factory=dict)
    resource_scope: str = ""
    requires_serial: bool = False
    is_write: bool = False
```

**`tool_runner.py`** change (in `_execute_with_dag()`):

```python
# Before (inline dict):
tc_spec = {
    "name": name,
    "args": args,
    "resource_scope": fn.get("resource_scope", ""),
    "requires_serial": fn.get("requires_serial", False),
    "is_write": name in WRITE_TOOLS or name in DELETE_TOOLS,
}

# After (ToolSpec):
from shared.tool_spec import ToolSpec
tc_spec = ToolSpec(
    call_id=tc.get("id", ""),
    name=name,
    args=args,
    resource_scope=fn.get("resource_scope", ""),
    requires_serial=fn.get("requires_serial", False),
    is_write=name in WRITE_TOOLS or name in DELETE_TOOLS,
)
```

Update all dict key accesses (`tc_spec["name"]`, etc.) to attribute access
(`tc_spec.name`, etc.) in the same function.

### Details

`ToolSpec` is intentionally simple — it does NOT incorporate `WRITE_TOOLS` / `DELETE_TOOLS`
sets internally. That logic stays in `tool_runner.py` for now to keep this change minimal.
Future Phase 6 can move capability metadata into `ToolSpec` when `ToolSpec` is registered
in a schema/registry.
