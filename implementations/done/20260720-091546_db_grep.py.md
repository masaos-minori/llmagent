# Implementation: scripts/mcp_servers/mdq/db_grep.py (accept pre-authorized `source_path`
allowlist via existing `req_paths` parameter)

Source plan: `plans/20260719-205532_plan.md`, Design §1 ("Read-time authorization"),
`grep_docs` bullet. Implementation steps Phase 3, item 3 (the `db_grep.py` signature-extension
half — the caller-side change lives in `mdq_service.py`, covered by its own document).

## Goal

Confirm/adjust `db_grep.grep_docs()` so that the authorized `source_path` list resolved by
`MdqService.grep_docs()` (companion `mdq_service.py` doc) — for both the explicit-`paths` and
no-filter cases — can be threaded through the existing `req_paths` parameter to filter the SQL
query, without a signature change.

## Scope

**In scope:**
- Verify `grep_docs()`'s existing `req_paths: list[str]` parameter (`db_grep.py:51-58`) and
  its `WHERE source_path IN (...)` construction (`db_grep.py:64-67`) already support being
  called with a *resolved, pre-authorized* path list (as opposed to only ever receiving the
  raw, user-supplied `req.paths`) — this is a **no functional code change** verification
  document unless the review below finds a gap.
- If (and only if) verification finds a gap — e.g. the function assumes `req_paths` is always
  either empty or the literal user-supplied list and this assumption breaks for a
  server-resolved list — make the minimal adjustment needed.

**Out of scope:**
- `find_grep_match()` (`db_grep.py:18-48`) — no path-authorization concern; operates on
  already-fetched rows.
- Any change to the calling convention beyond what `mdq_service.py`'s companion doc already
  specifies (passing `authorized_paths` into the existing `req_paths` positional argument).
- Any change to `db_grep.py`'s public API surface (function names, parameter names/order,
  return type) — the plan's Assumption 4 explicitly reuses the existing `req_paths` parameter
  "which already supports this via the `WHERE source_path IN (...)` clause" rather than adding
  a new parameter.

## Assumptions

1. Per the plan's Assumption 4: "enumerating all indexed distinct `source_path` values and
   authorizing each is the only structural option available (no stored per-row authorization
   flag exists), so this is a scan-then-authorize-then-filter implementation even though the
   requirement frames it as 'limit scanning'" — the scan-and-authorize step happens in
   `MdqService.grep_docs()` (companion doc), not in `db_grep.py`; this file only needs to
   accept whatever final path list is handed to it.
2. Direct read of `db_grep.py:51-74` confirms `grep_docs(conn, compiled, req_paths, max_matches,
   max_chars, ctx_before, ctx_after)` already builds `WHERE source_path IN (?,?,...)` from
   `req_paths` when non-empty, and omits the clause entirely (scans all rows) when
   `req_paths` is empty/falsy (`if req_paths:` at line 64). This is exactly the mechanism
   needed: passing the authorized-subset list (from either the explicit-`paths` or no-filter
   case in `mdq_service.py`) produces the desired `IN (...)` restriction with zero changes
   to `db_grep.py` itself.
3. The one behavior to double check: in the **no-filter** case, `mdq_service.py`'s companion
   doc resolves `authorized_paths` to the full authorized subset of *all* indexed
   `source_path` values (never an empty list unless zero paths are authorized) — if
   `authorized_paths` ends up empty (deny-all, `allowed_dirs = []`), `db_grep.py`'s current
   `if req_paths:` check (line 64) would treat an empty list the same as "no filter supplied"
   and fall through to scanning *all* rows unfiltered — which is the **wrong** behavior for a
   deny-all config (it would leak all content instead of returning zero matches). This is a
   real gap requiring a fix (see Implementation step 2 below) — flagged during doc-writing
   grounding, not assumed away.

## Implementation

### Target file

`scripts/mcp_servers/mdq/db_grep.py`

### Procedure

1. Verify no changes needed to `grep_docs()`'s signature (`db_grep.py:51-58`) —
   `req_paths: list[str]` stays as-is; the caller (`mdq_service.py`) is responsible for
   resolving/authorizing before calling.
2. **Fix the empty-list-vs-no-filter ambiguity** identified in Assumption 3: today,
   `req_paths=[]` (empty list) and "not supplied" are indistinguishable at this layer — both
   currently fall through to unfiltered scanning (`if req_paths:` is falsy for `[]`, so the
   `WHERE` clause is omitted). Because `MdqService.grep_docs()`'s companion doc now always
   calls this function with an explicit, already-resolved `authorized_paths` list (never
   "no filter" in the old sense — the caller has always already decided the exact path set),
   `db_grep.py`'s contract changes subtly: an empty list from this caller means "zero
   authorized paths," which must produce zero matches, not an unfiltered scan.
   - Option A (recommended, minimal): keep `db_grep.grep_docs()`'s signature/behavior
     unchanged, and make `MdqService.grep_docs()` (companion doc) short-circuit and return
     `"No matches found."` directly when `authorized_paths == []`, without calling
     `db_grep.grep_docs()` at all — avoids touching `db_grep.py`'s existing, still-correct
     "empty/no-filter = scan all" semantics for its *other*, non-authorization-related
     callers (there are none today, but keeping the function's existing contract stable is
     lower risk).
   - Option B (alternative): change `db_grep.py`'s `if req_paths:` line 64 to a
     three-state parameter (`req_paths: list[str] | None`, where `None` means "no filter" and
     `[]` means "filter to nothing") — a wider, more invasive signature change touching the
     one existing call site in `mdq_service.py` too.
   - **Recommended: Option A.** It requires zero changes to `db_grep.py`'s code and keeps this
     document's actual "Implementation" footprint at verification-only; the fix lives entirely
     in the companion `mdq_service.py` doc's `grep_docs()` procedure (add an early-return
     check: `if req.paths is None and not authorized_paths: return "No matches found."` before
     calling `db_grep.grep_docs()`). Cross-reference this decision in that companion doc during
     implementation.
3. If Option A is chosen (recommended), `db_grep.py` requires **no source changes at all** —
   this document's role is the verification and the explicit call-out of the edge case for the
   companion `mdq_service.py` doc to handle. If a future implementer instead chooses Option B,
   update this doc's Method/Details accordingly before implementing.

### Method

Verification-first: read the existing `req_paths` handling in full, confirm it already
supports the intended call pattern for the non-empty-authorized-set case, and explicitly
document the one edge case (fully-empty authorized set) that needs a decision — resolved via
a caller-side short-circuit (Option A) rather than a `db_grep.py` signature change, to keep
this module's contract minimal and stable.

### Details

- Do not conflate this with `enable_grep`'s existing config gate (`mdq_service.py:328-329`,
  `if not self.enable_grep: raise MdqValidationError(...)`) — that is a separate,
  already-implemented, unrelated check that runs before any path authorization logic.
- If Option B is ever chosen instead, the `WHERE` clause construction at
  `db_grep.py:64-67` needs a third branch: `req_paths is None` → no clause (scan all, current
  behav7ior for a hypothetical unauthenticated caller — not expected to occur in production
  once `mdq_service.py` always resolves an explicit list); `req_paths == []` → a clause that
  matches zero rows (e.g. `WHERE 1=0`) instead of omitting the clause. This alternative is
  noted for completeness but not the recommended path.
