# Implementation: scripts/mcp_servers/mdq/mdq_tools.py (`search_docs` tool schema description
text for `limit`/`max_results_limit`)

Source plan: `plans/20260719-210826_plan.md` ("Fix MDQ freshness detection and search result
correctness"), Implementation step 5. No existing implementation document (in `implementations/`
or `implementations/done/`) references `mdq_tools.py` — this is a new file for this filename.

## Goal

Update the `search_docs` tool's `inputSchema` description strings for `limit` and
`max_results_limit` so external callers (agents reading the MCP tool schema) understand the new
effective-limit precedence introduced by the companion `search.py` document
(`implementations/20260720-092134_search.py.md`): `limit` is now capped at the SQL layer by
`service.max_results_limit` before any rows are fetched, and `max_results_limit` is a secondary,
tighter display cap applied on top of that.

## Scope

**In scope:**
- `mdq_tools.py`'s `TOOL_LIST` entry for `"name": "search_docs"` (`mdq_tools.py:24-60`):
  update the `description` strings for the `"limit"` (line 31-34) and `"max_results_limit"`
  (line 48-51) properties only.

**Out of scope:**
- No field add/remove — `inputSchema["properties"]` keeps the same set of keys
  (`query`, `limit`, `mode`, `path_prefix`, `tag_filter`, `heading_prefix`, `max_results_limit`,
  `max_total_result_chars`), `required: ["query"]`, and `"status": "production"` all unchanged.
- No change to any other tool's schema (`get_chunk`, `outline`, `index_paths`, `refresh_index`,
  `stats`, `grep_docs`) — untouched.
- No change to `MCPToolSchema`'s `TypedDict` shape (`mdq_tools.py:12-20`) — untouched.
- Per `rules/toolchain.md`'s MCP documentation consistency check, this is a description-text-only
  change; it does not alter tool count, tool names, or `status` values, so `uv run
  check-mcp-docs --skip toolcount` is not even necessary — the standard `check-mcp-docs` run
  should pass unaffected either way.

## Implementation

### Target file

`scripts/mcp_servers/mdq/mdq_tools.py`

### Procedure

1. Replace the `"limit"` property block (lines 31-34):
   ```python
   "limit": {
       "type": "integer",
       "description": "Max results (default: 10)",
   },
   ```
   with an updated description, e.g.:
   ```python
   "limit": {
       "type": "integer",
       "description": (
           "Max results to fetch (default: 10). Bounded at the database layer by the "
           "server's configured max_results_limit — the effective SQL limit is "
           "min(limit, configured max_results_limit)."
       ),
   },
   ```
2. Replace the `"max_results_limit"` property block (lines 48-51):
   ```python
   "max_results_limit": {
       "type": "integer",
       "description": "Override max results from config (default: use config value)",
   },
   ```
   with an updated description, e.g.:
   ```python
   "max_results_limit": {
       "type": "integer",
       "description": (
           "Secondary display cap applied after the SQL-layer limit; if smaller than the "
           "effective limit derived from 'limit', further restricts how many results are "
           "shown (default: use server config value)."
       ),
   },
   ```
3. Exact wording is an implementation-time detail; the binding requirement is that both
   descriptions must, together, communicate the precedence (`limit` capped by config at the SQL
   layer; `max_results_limit` as an additional, tighter display-only cap) — see companion
   `docs/04_mcp_04_04_mdq.md` document for the longer-form human-readable explanation this schema
   text summarizes.

### Method

Two string-literal replacements inside an existing dict literal — no structural change to
`TOOL_LIST`, `MCPToolSchema`, or any other tool entry. Multi-line string descriptions should use
parenthesized implicit string concatenation (as sketched above) to stay under the 120-char line
length limit (`rules/coding.md`), consistent with how other long strings are already wrapped
elsewhere in the codebase.

### Details

- Do not change `"type": "integer"` for either field — only the `"description"` value strings
  change.
- Do not touch the `"required": ["query"]` list or `"status": "production"` — both unchanged.
- Since this file is pure data (`TOOL_LIST: list[MCPToolSchema] = [...]`), there is no logic to
  unit-test beyond confirming the dict still parses/imports and the schema still validates
  against whatever schema-shape test exists (see `tests/test_mdq_search_modes.py`'s neighbors —
  none currently assert on `mdq_tools.py`'s description text specifically; no new test is
  required by this plan beyond the general `check-mcp-docs` gate).
