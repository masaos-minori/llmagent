## Goal

Make `GET /v1/tools` on the git server merge `enabled: bool` and `disabled_reason: str`
into every returned tool dict, with a two-tier precedence rule: (1) if
`_cfg.allowed_repo_paths` is empty, ALL tools are disabled with
`"allowed_repo_paths is empty"` (this wins regardless of `read_only`); (2) otherwise, if
`_cfg.read_only` is true, only the 5 write tools (from `GIT_WRITE_TOOLS`) are disabled with
`"read_only=true"`, all other tools stay enabled with `""`.

## Implementation

### Procedure

1. Add `from shared.tool_constants import GIT_WRITE_TOOLS` to the import block (near
   lines 30-37).
2. Rewrite the `list_tools()` handler body (lines 73-76) to build the tools list via an
   explicit loop (not a one-line comprehension, since the precedence branching needs
   multiple lines per tool), applying:
   - `not _cfg.allowed_repo_paths` → `(False, "allowed_repo_paths is empty")` (checked
     first, unconditionally overrides everything else).
   - else `_cfg.read_only and t["name"] in GIT_WRITE_TOOLS` → `(False, "read_only=true")`.
   - else → `(True, "")`.
3. Return type annotation `dict[str, list[dict[str, object]]]` stays unchanged — `object`
   already covers the new `bool`/`str` values.

### Method

Pseudocode (no production code blocks per design-doc convention):

```
@app.get("/v1/tools")
async def list_tools() -> dict[str, list[dict[str, object]]]:
    tools: list[dict[str, object]] = []
    for t in TOOL_LIST:
        if not _cfg.allowed_repo_paths:
            enabled, reason = False, "allowed_repo_paths is empty"
        elif _cfg.read_only and t["name"] in GIT_WRITE_TOOLS:
            enabled, reason = False, "read_only=true"
        else:
            enabled, reason = True, ""
        tools.append({**t, "server_key": "git", "enabled": enabled, "disabled_reason": reason})
    return {"tools": tools}
```

### Details

- Precedence order is load-bearing: the `allowed_repo_paths` empty check MUST be the
  first `if` (not `elif` after the `read_only` check), so that when both
  `allowed_repo_paths=[]` and `read_only=False` hold, ALL tools are still disabled with
  `"allowed_repo_paths is empty"` — this is the exact regression the plan's Risks table
  calls out ("Precedence rule... implemented backwards").
  Corresponding regression test (see the `test_mcp_tools_validation.py` doc): assert that
  with `allowed_repo_paths=[], read_only=False`, every tool (including read tools) is
  still disabled.
  Corresponding regression test 2: with `allowed_repo_paths=["/tmp"], read_only=True`,
  read tools stay enabled and only the 5 `GIT_WRITE_TOOLS` names are disabled.
- `t["name"] in GIT_WRITE_TOOLS` — `GIT_WRITE_TOOLS` is a `frozenset[str]`, so membership
  check is O(1) and type-safe against `t["name"]` (a `str` per tool schema).
- No change to `TOOL_LIST` import (line 34) or any other handler.
- No new exception handling needed — all branches are pure boolean/membership checks.
