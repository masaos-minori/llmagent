# Implementation: scripts/mcp_servers/mdq/indexer.py (`index_paths()` real counters +
`(str, IndexPathsMetadata)` return)

Source plan: `plans/20260719-211216_plan.md` ("Replace fragile MDQ audit detail parsing with
structured metadata"), Design section ("`index_paths()` counting (new logic, not just wiring)"),
Implementation step 3.

**Note on pre-existing docs for this file:** `implementations/done/20260719-110900_indexer.py.md`
also targets `indexer.py`, but covers an unrelated, already-different concern: real `tags_json`
and `token_count` values inside `_index_single_file()` (a different function than the one this
document changes). Read in full and confirmed no overlap: that document does not touch
`index_paths()`, does not add any counters, and does not change the function's return type. This
is a NEW document covering `index_paths()`'s counting/metadata concern only.

## Goal

`index_paths()` currently indexes/skips files with no counting at all and unconditionally returns
the fixed string `"Indexing complete"`. Add real `indexed_count`/`skipped_count`/`failed_count`
counters (mirroring the pattern `refresh_paths()` already uses, `indexer.py:252-386`) and change
the return type to `(str, IndexPathsMetadata)` so `mdq_server.py::call_tool()` can report accurate
audit fields for the `index_paths` tool. Also thread `refresh_paths()`'s existing
`RefreshSummary` return value through `MdqService.refresh_index()` unchanged in shape (that half
is a `mdq_service.py`-only change — see companion doc — since `refresh_paths()` already returns
the structured summary and needs no change here).

## Scope

**In scope:**
- `index_paths()` (`indexer.py:220-238`): add local counters `indexed_count`, `skipped_count`,
  `failed_count` (all starting at `0`); increment them at the existing per-path branch points
  (`_index_single_file` call → indexed; `continue` on missing path → skipped; unsupported
  path/suffix `else` branch → skipped; any exception during indexing of a given path → failed —
  requires wrapping the per-path indexing calls in `try`/`except` where none exists today, see
  Details for the exact exception-catching approach). Change return type from `str` to
  `tuple[str, IndexPathsMetadata]`; the returned string stays exactly `"Indexing complete"`
  (unchanged, out of scope per plan) while the metadata carries the new counts plus
  `input_path_count=len(req.paths)` and `duration_ms`.

**Out of scope:**
- `refresh_paths()` (`indexer.py:241-386`) — already returns a structured `RefreshSummary`; no
  change needed in this file for that function. The only change on the `refresh_index` path is in
  `mdq_service.py` (companion doc), which currently discards `summary` after formatting text and
  must instead keep it.
- `_index_single_file()`, `_index_directory()`, `_iter_indexable_files()`,
  `generate_chunk_id()`, `_estimate_token_count()`, `_matches_any_glob()` — unchanged.
- The external text `"Indexing complete"` — the plan explicitly allows this to stay fixed; only
  internal metadata is new.
- `implementations/done/20260719-110900_indexer.py.md`'s `tags_json`/`token_count` concern
  inside `_index_single_file()` — untouched by this document.

## Assumptions

1. Direct read of `index_paths()` (`indexer.py:220-238`, current code) confirms the plan's
   Assumption 5: there is genuinely no counting logic today — the function only logs
   (`logger.warning`) on a missing path or unsupported path/suffix and calls
   `_index_single_file`/`_index_directory` with no return-value capture (both are `-> None`
   coroutines) and no `try`/`except` around them. Implementing counters requires new logic, not
   wiring up existing data.
2. `_index_single_file()` and `_index_directory()` both return `None` unconditionally and already
   swallow their own internal errors (`_index_single_file` catches `FileNotFoundError` at
   `indexer.py:133-135` and `sqlite3.Error` at `indexer.py:204-209`, logging and returning without
   raising) — so today, "success" vs. "failure" for a single file is not observable from
   `index_paths()`'s call site at all. To produce a real `failed_count`, either (a) change
   `_index_single_file`/`_index_directory` to signal failure (raising or returning a bool), which
   is a wider blast-radius change since `_index_directory` is also called from
   `refresh_paths()` (`indexer.py:297`, inside its own `try`/`except Exception` block that already
   handles failures at the `refresh_paths()` level), or (b) treat `index_paths()`'s
   `failed_count` conservatively as "paths whose type didn't match a known case" (the existing
   `else: logger.warning("Skipping non-Markdown path...")` branch, `indexer.py:236-237`) plus any
   exception `index_paths()` itself can catch by wrapping its own per-path dispatch in a local
   `try`/`except Exception`, without modifying `_index_single_file`/`_index_directory`'s existing
   swallow-and-log behavior. **This document takes approach (b)** — see Risk note below,
   consistent with the plan's Risk table ("Keep the change additive-only ... no existing
   assertion can regress").
3. `skipped_count` counts two distinct existing log-only paths that currently produce no return
   signal: (i) `if not p.exists(): ... continue` (`indexer.py:225-227`) and (ii) the `else` branch
   for a path that is neither a file-with-`.md`-suffix nor a directory
   (`indexer.py:236-237`) — both become `skipped_count += 1` under approach (b) above, since
   neither actually attempted an index write.

## Implementation

### Target file

`scripts/mcp_servers/mdq/indexer.py`

### Procedure

1. Add `IndexPathsMetadata` to the `from mcp_servers.mdq.mdq_models import (...)` block
   (`indexer.py:21-26`); add `import time` if not already present (module already imports `time`
   at `indexer.py:13` for `_index_single_file`'s `path.stat().st_mtime_ns`/`time.time()` — reuse
   it, no new import needed).
2. In `index_paths()`, before the loop: `t0 = time.perf_counter()`,
   `indexed_count = skipped_count = failed_count = 0`.
3. Inside the loop (`indexer.py:223-237`), at each existing branch:
   - `if not p.exists(): logger.warning(...); skipped_count += 1; continue` (was: `continue` with
     no counting).
   - The `authorize_path` check (`indexer.py:228-231`) is unchanged — it raises
     `MdqAuthorizationError`, which propagates out of `index_paths()` entirely (no counter
     update needed; the whole call fails).
   - `if p.is_file() and p.suffix == ".md":` — wrap the `await _index_single_file(service, p)`
     call in a local `try/except Exception as e:` that logs
     `logger.error("Failed to index %s: %s", path_str, e)` and increments `failed_count`, else
     (no exception) increments `indexed_count`. Note: per Assumption 2, `_index_single_file`
     today does not raise on its own internal errors (it catches and logs internally) — this
     `try/except` at the `index_paths()` level is future-proofing/defensive and will, in
     practice, almost always increment `indexed_count` even for files `_index_single_file`
     silently skipped internally (e.g. the `file_size > max_file_chars` early return at
     `indexer.py:120-127`, or `not sections` at `indexer.py:137-139`). This is a known, documented
     limitation — see Risks in the plan and Details below — not silently hidden.
   - `elif p.is_dir():` — same pattern: wrap `await _index_directory(service, p)` in
     `try/except Exception`, increment `failed_count` or `indexed_count` accordingly.
   - `else: logger.warning(...); skipped_count += 1` (was: `logger.warning(...)` only).
4. After the loop: `duration_ms = (time.perf_counter() - t0) * 1000`. Return
   `"Indexing complete", IndexPathsMetadata(input_path_count=len(req.paths),
   indexed_count=indexed_count, skipped_count=skipped_count, failed_count=failed_count,
   duration_ms=duration_ms)`.

### Method

Additive counters threaded through the existing per-path `if`/`elif`/`else` chain in
`index_paths()`, plus a `try`/`except Exception` wrapper around the two indexing calls (currently
absent) to observe exceptions that escape `_index_single_file`/`_index_directory` (both of which
mostly swallow their own errors internally today, per Assumption 2) — no change to
`_index_single_file`/`_index_directory`/`refresh_paths()` themselves.

### Details

- **Known limitation to document, not silently fix:** because `_index_single_file()` swallows
  its own file-size-exceeded and no-sections-found cases internally (logging only, no
  return signal), `indexed_count` under this document's implementation will count "attempted and
  did not raise" rather than "actually wrote at least one chunk." This is consistent with the
  plan's Risk mitigation ("counters are new fields that did not exist before, so no existing
  assertion can regress") and is a strictly more accurate signal than today's total absence of
  counting — but it is not a fully precise "chunks written" count. If a future cycle wants
  `_index_single_file` to return a success/failure/skip enum instead of `None`, that is a larger,
  separate change out of scope here (would also require updating its `refresh_paths()` call site
  at `indexer.py:297`, `320`, `344`).
- Do not change `_index_directory()`'s or `_index_single_file()`'s signatures or return types in
  this document — only `index_paths()`'s own local `try/except` wrapping and counters.
- `refresh_paths()` needs **no changes in this file** — its `RefreshSummary` return value already
  has the exact shape needed; the discard happens in `mdq_service.py::refresh_index()`, covered
  by that file's companion document.
