`_cfg.allowed_dirs` state at request time, using the new shared
`availability_flags()` helper from `scripts/mcp_servers/file/common.py`.

### Procedure

1. Add `availability_flags` to the existing `from mcp_servers.file.common import (...)`

2. Rewrite the `list_tools()` handler body (lines 235-238) to call
   `availability_flags(_cfg.allowed_dirs)` once, unpack `enabled, reason`, and merge both
   keys into each tool dict alongside the existing `server_key` merge.
3. Return type annotation `dict[str, Any]` (line 235) stays unchanged — `Any` already
   covers the new `bool`/`str` values inside the tool dicts.

### Method

Pseudocode (no production code blocks per design-doc convention):

```
@app.get("/v1/tools")
async def list_tools() -> dict[str, Any]:
    enabled, reason = availability_flags(_cfg.allowed_dirs)
    return {
        "tools": [
            {**t, "server_key": "file_read", "enabled": enabled, "disabled_reason": reason}
            for t in TOOL_LIST
        ],
    }
```

### Details

- `_cfg.allowed_dirs` is read once per request (not cached across requests), matching the
  plan's Assumption 2: "computed per-request... reflect current `_cfg` state."
- Every tool in `TOOL_LIST` gets the identical `enabled`/`disabled_reason` pair (file
  servers have no per-tool distinction, unlike git) — this matches the plan's rule: "every
  tool `enabled: false, disabled_reason: 'allowed_dirs is empty'`" / "every tool
  `enabled: true, disabled_reason: ''`".
- No change to `TOOL_LIST` import (line 60) or any other handler (`health`, `call_tool`,
  the dispatch helper).
- No new exception handling — `availability_flags` cannot raise.
