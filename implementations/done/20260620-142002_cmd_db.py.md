# Implementation: cmd_db.py (update — help labels and docstrings)

## Goal

Update `cmd_db.py` to:
1. Replace "RAG + Session" label with "Session" in `/db help` and `_db_stats` output (Step 3)
2. Remove RAG internal schema references from docstrings (Step 4)
3. Import `RagMaintenanceService` and route RAG subcommands (`rebuild-fts`, `consistency`,
   `recover`) through it instead of `DbMaintenanceService`

## Scope

**In:**
- `_db_help()`: change `["stats", "RAG + Session", ...]` → `["stats", "Session", ...]`
- `_db_stats()`: change `("target", "RAG + Session")` → `("target", "Session")`
- `_db_rebuild_fts()` docstring: remove FTS5 schema detail, replace with abstraction
- `_db_consistency()` docstring: remove table name references
- `_db_rebuild_fts()`, `_db_consistency()`, `_db_recover()`: delegate to `RagMaintenanceService`
  instead of `DbMaintenanceService`

**Out:**
- Changing functional behavior of any subcommand
- Adding or removing subcommands
- Modifying help descriptions other than "Target DB" column

## Assumptions

- `RagMaintenanceService` exists (created in step `20260620-142000`)
- `DbMaintenanceService` no longer exposes `rebuild_fts()`, `consistency()`, `recover()`
  (removed in step `20260620-142001`)
- `/db stats` continuing to show combined RAG+Session data is acceptable if `_db_stats`
  calls both services; the label change only reflects the primary owner (session)

## Implementation

### Details

- `/db stats` output continues to show both docs/chunks and sessions/messages;
  the label change only reflects that the `/db stats` command's primary responsibility
  is "Session" (the combined view is an operational convenience, not a mixed boundary)
- No change to the functional behavior of `_db_clean` or `_db_list_urls`
  (they already route through `rag-pipeline-mcp`)
- After this change, `DbMaintenanceService` is only used for:
  `stats()` (sessions/messages), `health()`, `checkpoint()`, `vacuum()`, `purge()`
