# Goal

Update the shell-mcp section in `docs/04_mcp_04_server_catalog.md` to clearly warn operators that:
1. The default `sandbox_backend` is `"none"` (no sandboxing)
2. Enabling sandbox requires firejail to be installed
3. The `/health` endpoint now exposes the active sandbox backend

# Scope

- **File:** `docs/04_mcp_04_server_catalog.md`
- **Lines:** Shell-mcp section (~150-182)
- **Out of scope:** Other server sections

# Assumptions

1. Current docs describe shell-mcp as "sandboxed" in the server description line (port 8009).
2. The config table already shows `shell_sandbox_backend | "none"` but has no warning.
3. The new `/health` endpoint exposes `sandbox_backend` (after server.py change).

# Implementation

## Target file

`docs/04_mcp_04_server_catalog.md`

## Procedure

1. Add a warning block after the config table in the shell-mcp section.
2. Update the **Purpose** or description line to clarify sandbox is optional.

## Method

After the config table (after `| audit_log_path | ... |`), add:

```markdown
> **Security note — Sandbox is disabled by default:** `sandbox_backend` defaults to `"none"`.
> Shell commands run with the agent process's OS user and permissions — no container, no namespace isolation.
> To enable sandboxing, install firejail and set `sandbox_backend = "firejail"` in
> `config/shell_mcp_server.toml`. The active backend is visible in the `/health` response
> (`"sandbox_backend": "none"` or `"sandbox_backend": "firejail"`).
```

## Details

- Do not change the config table entries.
- The warning should be visually prominent (blockquote).
