# Implementation Procedure: Restrict approval dry-run tools to dry-run-capable tools

## Goal

Prevent dry-run execution for tools that don't support the `dry_run` parameter, avoiding false denials for HIGH-risk tools during approval preview.

## Scope

- `scripts/agent/tool_approval.py` — add dry-run capability check before injecting `dry_run=True`
- `scripts/agent/config_builders.py` — validate `approval_dry_run_tools` at startup, warn on unsupported entries
- `scripts/agent/config_dataclass.py` — may need validation addition
- `tests/test_tool_approval_preflight.py` — add/update tests for dry-run capability filtering

## Assumptions

- Tool capabilities can be determined by checking against a known set of dry-run-capable tools
- The default dry-run tools list (`write_file`, `edit_file`, `create_directory`, `delete_file`, `delete_directory`, `move_file`) is authoritative
- Custom dry-run tools added via config should be validated against known capabilities

## Implementation

### Target file

`scripts/agent/tool_approval.py`

### Procedure

1. In `tool_approval.py`, define `_DRY_RUN_CAPABLE_TOOLS` constant containing all known dry-run-capable tool names (include current defaults from config_builders.py line 107-114 and config_dataclasses.py line 331-340).

2. Add `_is_dry_run_capable()` helper function:
   ```python
   def _is_dry_run_capable(tool_name: str) -> bool:
       return tool_name in _DRY_RUN_CAPABLE_TOOLS
   ```

3. In `_build_preview_with_dry_run()`, before injecting `dry_run=True` (line 72), add capability check:
   ```python
   if not _is_dry_run_capable(tool_name):
       logger.warning("Tool %s does not support dry_run; using static preview", tool_name)
       return build_preview(tool_name, args)
   ```

4. In `config_builders.py`, add startup validation for `approval_dry_run_tools`:
   ```python
   def _validate_dry_run_tools(tools: list[str]) -> list[str]:
       supported = {t for t in _DEFAULT_DRY_RUN_TOOLS} | _KNOWN_DRY_RUN_CAPABLE_TOOLS
       filtered = []
       for tool in tools:
           if tool not in supported:
               logger.warning("Dry-run tool '%s' is not supported; ignoring", tool)
           else:
               filtered.append(tool)
       return filtered
   ```

5. Call `_validate_dry_run_tools()` after loading `approval_dry_run_tools` in `_build_approval_config()`.

6. Update HIGH-risk denial logic to only trigger for meaningful dry-run blocking results.

### Method

Use targeted edits to add the dry-run capability check in the existing code path. Use startup-time validation to warn on misconfiguration rather than failing fast.

### Details

- Recommend hardcoded whitelist (Option A) over dynamic capability discovery (Option B) — explicit is better than implicit
- Default dry-run tools are defined in both `config_builders.py` (line 107-114) and `config_dataclasses.py` (line 331-340) — consistent but no validation currently exists
- Dry-run failures raise `ApprovalPreviewBlockingError` for HIGH risk or `ApprovalPreviewError` for MEDIUM risk — this behavior remains unchanged for capable tools
