# Implementation: Add `ApprovalPreviewBlockingError` to `scripts/agent/tool_exceptions.py`

Source plan: `plans/20260715-141245_plan.md` (Implementation step 3)

## Goal

Add a distinct exception subtype so "dry-run returned an explicit error
result" can be handled differently in `tool_approval.py` from "dry-run
unsupported / connection failure", without changing how any existing
`except ApprovalPreviewError` call site behaves.

## Scope

**In scope**
- `scripts/agent/tool_exceptions.py`: add
  `class ApprovalPreviewBlockingError(ApprovalPreviewError)`.

**Out of scope**
- All other exception classes in this file — untouched.
- Any change to `tool_approval.py`'s use of the new exception — covered
  by the companion doc `20260715-154002_tool_approval.py.md`.

## Assumptions

1. `ApprovalPreviewError(RuntimeError)` already exists at the current end
   of the file (line 24-25 of `scripts/agent/tool_exceptions.py`):
   ```python
   class ApprovalPreviewError(RuntimeError):
       """Raised when dry-run preview execution fails."""
   ```
2. Subclassing (rather than adding a sibling exception or an attribute
   flag) is the correct approach because
   `_build_preview_with_dry_run()`'s existing
   `except ApprovalPreviewError: raise` re-raise clause (line 81-82 of
   `tool_approval.py`) must continue to pass the new error through
   unchanged even if some future call site only knows about the base
   class — subclassing satisfies this without any change to that
   re-raise clause.
3. No constructor logic is needed beyond the inherited
   `RuntimeError.__init__` — the message string constructed in
   `tool_approval.py` (`f"Dry-run for {tool_name!r} returned an error:
   {dry_text[:200]}"`) is sufficient; no additional structured fields
   (e.g. `tool_name`, `dry_text`) are required by the plan's Design
   section, unlike `PolicyViolationError` which does carry an
   `audit_decision` attribute for a different purpose (feeding
   `audit_approval`'s `decision` argument). This plan's design instead
   passes a hardcoded string literal `"denied_dry_run_error"` directly at
   the `tool_approval.py` call site, so no attribute is needed on the
   exception itself.

## Implementation

### Target file

`scripts/agent/tool_exceptions.py` (existing, ~35 lines)

### Procedure

1. Locate the existing `ApprovalPreviewError` class definition.
2. Add immediately after it:
   ```python
   class ApprovalPreviewBlockingError(ApprovalPreviewError):
       """Raised when dry-run preview returns an explicit error result.

       Distinct from the base ApprovalPreviewError (which also covers
       dry-run being unsupported or unreachable) so callers can deny
       approval outright for HIGH-risk tools on an explicit dry-run
       failure, while still falling back to a text-only preview for the
       unsupported/connection-failure case.
       """
   ```
3. No other lines in the file change.

### Method

Single-class addition; no constructor override, no new imports needed
(`from __future__ import annotations` is already present at the top of
the file).

### Details

- Placement: directly after `ApprovalPreviewError` in the file, to keep
  related exceptions adjacent (matches the existing grouping style, e.g.
  `PolicyViolationError` followed immediately by its related
  `AuditUnavailableError`/`LifecycleConfigurationError` neighbors is not
  strictly grouped by relation in the current file, but placing the new
  subclass right after its parent class is the clearest signal of the
  inheritance relationship for a reader).
- Do not export this exception anywhere else — it is imported directly
  by `scripts/agent/tool_approval.py` via
  `from agent.tool_exceptions import ApprovalPreviewBlockingError,
  ApprovalPreviewError, PolicyViolationError` (see companion doc).
