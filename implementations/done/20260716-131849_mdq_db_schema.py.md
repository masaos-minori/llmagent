# Implementation: scripts/mcp_servers/mdq/db_schema.py (remove embedding params and vec0 table-creation block)

Source plan: `plans/20260716-131500_plan.md`

## Goal

Simplify `create_production_tables()`'s signature by removing the
`use_embedding`, `vector_table`, `embedding_dims` parameters and the
conditional `vec0` extension-loading / `CREATE VIRTUAL TABLE ... USING
vec0(...)` block that depended on them.

## Scope

**In:**
- `create_production_tables()` signature (`scripts/mcp_servers/mdq/db_schema.py:21-28`):
  remove the `use_embedding: bool`, `vector_table: str`,
  `embedding_dims: int` parameters, leaving `conn`, `db_path`,
  `sqlite_busy_timeout`.
- Delete the "Create vector table conditionally based on use_embedding
  config" block (lines 135-158): the `if use_embedding:` branch including
  `vec0_path`, `conn.enable_load_extension(...)`, `conn.load_extension(...)`,
  the `.so.so` workaround, and the `CREATE VIRTUAL TABLE IF NOT EXISTS
  {vector_table} USING vec0(...)` statement.

**Out:**
- Every other table/trigger creation statement in this function
  (`documents`, `chunks`, `chunks_fts`, `chunks_ai`/`chunks_ad`/`chunks_au`
  triggers, `index_state`, `chunk_summaries`) — all unrelated, unchanged.
- The old-schema migration/rebuild logic at the top of the function
  (lines 37-47) — unrelated, unchanged.
- `conn.commit()` at line 159 — keep, now runs immediately after the
  `chunk_summaries` table creation instead of after the vec0 block.

## Assumptions

1. This function's only caller is `MdqService._init_db()`
   (`service.py:112-127`), which passes these three parameters — this
   change must land in the same commit as the companion `service.py` doc,
   which updates the call site to the simplified 3-argument form (`conn`,
   `db_path`, `sqlite_busy_timeout`).
2. Per the source plan's Affected-areas note on database impact: existing
   `mdq.sqlite` databases that previously had `use_embedding=true` may
   already contain a `chunks_vec` virtual table (created via `CREATE
   VIRTUAL TABLE IF NOT EXISTS`, an idempotent/additive statement) — removing
   the creation code here does **not** drop any pre-existing table; it only
   stops creating new ones going forward. No migration/DROP logic is added
   as part of this change (out of scope; the source plan's Implementation
   step 5 only asks to *check* whether any live database has this table,
   as an informational step, not to clean it up).
3. No test directly exercises the `vec0` extension-loading code path with
   `use_embedding=True` in a way that would break silently if left
   unverified — check `tests/test_mdq_hybrid_search.py` (deleted in a
   companion doc) for any such coverage before finalizing; per the source
   plan, all of that file's tests exercise config/schema/merge behavior
   that no longer exists after this change, confirming no orphaned
   dependency on this removed code path.

## Implementation

### Target file

`scripts/mcp_servers/mdq/db_schema.py`

### Procedure

1. Open `scripts/mcp_servers/mdq/db_schema.py`.
2. Change the function signature (current lines 21-28):
   ```python
   def create_production_tables(
       conn: sqlite3.Connection,
       db_path: str,
       use_embedding: bool,
       vector_table: str,
       embedding_dims: int,
       sqlite_busy_timeout: int,
   ) -> None:
   ```
   to:
   ```python
   def create_production_tables(
       conn: sqlite3.Connection,
       db_path: str,
       sqlite_busy_timeout: int,
   ) -> None:
   ```
3. Delete the block at lines 135-158:
   ```python
   # Create vector table conditionally based on use_embedding config
   if use_embedding:
       vec0_path: str = "/opt/llm/sqlite-vec/vec0.so"
       try:
           conn.enable_load_extension(True)
           try:
               conn.load_extension(vec0_path)
           except sqlite3.OperationalError as inner:
               # Some SQLite builds auto-append .so, causing .so.so
               if ".so.so" in str(inner) and vec0_path.endswith(".so"):
                   conn.load_extension(vec0_path[:-3])
               else:
                   raise
           conn.enable_load_extension(False)
       except sqlite3.OperationalError as e:
           conn.enable_load_extension(False)
           logger.error("Failed to load sqlite-vec extension: %s", e)
           raise
       conn.execute(f"""
           CREATE VIRTUAL TABLE IF NOT EXISTS {vector_table} USING vec0(
               chunk_id TEXT PRIMARY KEY,
               embedding float[{embedding_dims}]
           )
       """)
   ```
   leaving `conn.commit()` as the line immediately following the
   `chunk_summaries` table creation block.

### Method

Signature simplification (drop 3 parameters) plus deletion of the
conditional block that consumed them — no other function behavior changes.

### Details

- Do not touch the module-level docstring
  (`"""mcp_servers/mdq/db_schema.py ... Dependency direction: db_schema →
  models ..."""`) — still accurate, this module has no other dependency
  change.
- After this edit, `db_schema.py` no longer references
  `/opt/llm/sqlite-vec/vec0.so` or `enable_load_extension`/`load_extension`
  at all — confirm via `grep -n "vec0\|load_extension" scripts/mcp_servers/mdq/db_schema.py`.
