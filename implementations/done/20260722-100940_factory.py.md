## Goal
Clarify lifecycle state management by distinguishing "never seen" servers from "unknown state" servers, allowing first-time server initialization without validation errors.

## Scope
**In**: `scripts/agent/factory.py` — refactor `_set_state()` to handle first-time servers explicitly
**Out**: Changes to `LifecycleState` enum values, `assert_valid_transition()` logic, or other lifecycle methods

## Assumptions
1. A missing key in `_states` always means "never existed" rather than "state is unknown"
2. Any initial state is valid for a first-time server (STARTING, HEALTHY, etc.)
3. Info-level logging is sufficient for first-time initialization events

## Implementation

### Target file
`scripts/agent/factory.py`

### Procedure
1. Add explicit check for `server_key not in self._states` in `_set_state()`
2. Allow any initial state for first-time servers without validation
3. Add info-level log when initializing a new server's state
4. Remove default value `LifecycleState.UNKNOWN` from `.get()` call

### Method
Refactor `_set_state()` to skip validation when the server has never been tracked.

### Details
Current code in `factory.py` line 65-77:
```python
def _set_state(self, server_key: str, new_state: LifecycleState) -> None:
    """Transition the server state with validation."""
    current = self._states.get(server_key, LifecycleState.UNKNOWN)
    try:
        assert_valid_transition(current, new_state)
    except ValueError:
        _logger.warning(
            "Lifecycle: invalid state transition %r -> %r for %r",
            current,
            new_state,
            server_key,
        )
    self._states[server_key] = new_state
```

Problem: When a server is first seen, `_states.get(server_key, LifecycleState.UNKNOWN)` returns `LifecycleState.UNKNOWN` as the default. While `lifecycle.py` defines `UNKNOWN` as allowing any transition (`frozenset(LifecycleState)`), this masks the semantic difference between "first-time server" and "server whose state became unknown." The ambiguity makes debugging harder and obscures the intent.

Fix:
```python
def _set_state(self, server_key: str, new_state: LifecycleState) -> None:
    """Transition the server state with validation."""
    # First-time server: no prior state to validate against.
    if server_key not in self._states:
        _logger.info(
            "Lifecycle: initializing state for %r -> %r",
            server_key,
            new_state,
        )
        self._states[server_key] = new_state
        return

    current = self._states[server_key]
    try:
        assert_valid_transition(current, new_state)
    except ValueError:
        _logger.warning(
            "Lifecycle: invalid state transition %r -> %r for %r",
            current,
            new_state,
            server_key,
        )
    self._states[server_key] = new_state
```

Key points:
- Check `server_key not in self._states` BEFORE attempting any transition validation
- Log INFO level on first-time initialization for observability
- Use direct dictionary access `self._states[server_key]` instead of `.get()` since we've already confirmed the key exists
- Remove `LifecycleState.UNKNOWN` as a sentinel — it was only used as a fallback default

Also update `get_state()` method at line 163:
```python
# Current:
return self._states.get(server_key, LifecycleState.UNKNOWN)

# Fix:
if server_key not in self._states:
    return None  # or keep UNKNOWN if callers depend on non-None return
return self._states[server_key]
```

Note: If callers expect a non-None return value, keep returning `LifecycleState.UNKNOWN` but add a docstring clarifying its meaning.
