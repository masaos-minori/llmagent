## Goal

Clarify consumer ID stability for reliable offset resume by removing misleading documentation about auto-generated consumer IDs and ensuring durable consumers can resume offsets after restart.

## Scope

**In-Scope**:
- Remove misleading docs that say "Consumer IDs are generated as `sha256(name:hostname:pid)[:16]`" — this is never used in app.py, consumer_ids are always client-supplied via query parameters
- Clarify that consumer_ids are client-supplied and must be stable across restarts for offset resume to work
- Define collision detection behavior for duplicate consumer_ids
- Update docs to distinguish process-lifetime stability vs restart-stable durability
- Add tests for restart/resume behavior

**Out-of-Scope**:
- Multi-node consumer group coordination
- Exactly-once delivery

## Assumptions

1. Consumer IDs are always client-supplied via `consumer_id` query parameter — no auto-generation in app.py (confirmed: lines 257, 347, 382 all use `Query(default="")`)
2. `_make_consumer_id()` in offsets.py is never called from app.py at runtime — only used in tests (test_eventbus_offsets.py:160-164)
3. For durable consumers to resume offsets after restart, they must use the same consumer_id across restarts
4. If a consumer changes its ID between restarts, it will start from seq=0 (offset not found)

## Verification of plan claims

- **Claim 1**: `_make_consumer_id` is dead code — Partially true. The function exists in offsets.py:12 and is tested in test_eventbus_offsets.py:158-164, but it is never called from app.py at runtime. It serves as a test utility for verifying consumer ID stability properties.
- **Claim 2**: sha256 text in docs is misleading — Confirmed. `06_eventbus_03_persistence_schema_and_replay.md:49` and `06_eventbus_05_configuration_deploy_and_operations.md:61` both contain "Consumer IDs are generated as `sha256(name:hostname:pid)[:16]`" which is false — app.py never generates consumer IDs, they are always client-supplied.
- **Claim 3**: Collision detection for duplicate consumer_ids — Unconfirmed. No collision detection mechanism exists in read_offset/write_offset functions. Last write wins behavior applies silently.
