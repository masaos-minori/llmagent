## Goal

Add an optional `capabilities: tuple[str, ...] = ()` field to the planned `RuntimeTool` frozen dataclass
(`scripts/shared/runtime_tool.py`, per `implementations/20260717-203121_runtime_tool.py.md`), and thread a
matching optional `capabilities` parameter through `build_runtime_tool()`'s factory signature, defaulting
to an empty tuple when omitted. This requirement only adds the field's existence and safe-default
behavior â€” no real MCP tool populates non-empty capability values as part of this change (that is
requirement 13's own explicitly out-of-scope future adoption work).

## Assumptions

1. **The base doc's 13-field list and factory signature are taken as given** (per
   `implementations/20260717-203121_runtime_tool.py.md`'s Details section): `name`, `server_key`,
   `server_url`, `description`, `input_schema`, `raw_definition`, `status`, `is_write`, `requires_serial`,
   `resource_scope`, `agent_safety_tier`, `requires_approval`, `enabled_for_llm`. This doc adds a 14th
   field, `capabilities`, without altering any of the 13 existing ones or their safe-default rules.
2. **`capabilities` needs no sentinel-`None` factory parameter**, unlike `is_write`/`requires_serial`
   (which encode "was this explicitly annotated" via a `None` sentinel per the base doc's Assumption 5).
   `capabilities` has no such derived-default coupling to another field â€” an omitted `capabilities`
   argument simply means "empty", so the factory parameter can default directly to `None` and resolve to
   `()` (mirroring `input_schema`/`raw_definition`'s existing `dict | None = None` -> `{}` pattern from the
   base doc, applied to a tuple instead of a dict) rather than needing a three-state sentinel.
3. **Empty-mutable-default handling does not apply to tuples** â€” `tuple` is immutable, so
   `capabilities: tuple[str, ...] = ()` is safe as a literal dataclass field default in principle; however,
   this doc still routes construction through the `build_runtime_tool()` factory (not a bare dataclass
   default) for consistency with every other field's construction path in the base doc, and to allow
   normalizing a caller-supplied `list[str]` into a `tuple[str, ...]` at the boundary (a `/v1/tools` JSON
   response naturally decodes `capabilities` as a `list`, not a `tuple`).
4. **No runtime validation of individual capability strings against the `domain.action` convention is
   added** (consistent with the base doc's Assumption/Details note that `agent_safety_tier`'s `Literal` is
   enforced statically only, not at runtime) â€” the convention document
   (`implementations/20260718-084628_tool_capability_naming_convention.md`) defines the shape as
   documentation guidance, not a runtime-enforced schema. Adding a regex guard here would be speculative,
   unrequested validation beyond this requirement's acceptance criteria.
