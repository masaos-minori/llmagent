# Implementation Procedure: Documentation Set Validation

## Goal

Validate all documentation files after splitting and normalization operations to ensure compliance with size limits, structural requirements, and link integrity.

## Scope

### In scope

- Verify all generated/modified files are ≤ 8KB
- Verify each file has exactly one H1 heading
- Verify Front Matter exists in all files
- Verify Related Documents and Keywords sections exist in all files
- Verify internal link reachability (all link targets exist)
- Verify fenced code blocks and Markdown tables are not broken across splits

### Out of scope

- Content accuracy validation beyond structural checks
- Technical correctness of documentation content
- Performance testing of RAG ingestion

## Assumptions

1. All previous steps (splitting, normalization, link correction) have completed successfully.
2. The validation methods listed in the plan are sufficient to detect common issues.
3. Manual spot-checks will complement automated checks for edge cases.

## Implementation

### Target files

All markdown files under `docs/` directory:

- Overview files (`docs/01_overview*.md`, `docs/02_deployment.md`)
- RAG files (`docs/03_rag_*.md`)
- MCP files (`docs/04_mcp_*.md`)
- Agent files (`docs/05_agent_*.md`)
- Event Bus files (`docs/06_eventbus_*.md`)
- Shared/DB files (`docs/90_shared_*.md`)
- Index file (`docs/index.md`)

### Procedure

#### Step 1: File size validation

Run the following command to verify all files are ≤ 8KB:

```bash
wc -c docs/*.md | awk '$1 > 8192 {print $2, $1}'
```

Expected result: No output (no files exceeding 8192 bytes).

If any file exceeds 8KB:
1. Identify which file exceeds the limit
2. Determine which section caused the overflow
3. Further split at H3 boundaries or functional group boundaries
4. Repeat this validation step until all files pass

#### Step 2: H1 heading count validation

Run the following command to verify each file has exactly one H1 heading:

```bash
grep -c '^# ' docs/*.md | awk -F: '$2 != 1 {print $1, $2}'
```

Note: Be careful about false positives from Python comments (`# ...`) inside code blocks. Exclude lines within fenced code blocks.

Expected result: No output (each file has exactly one H1 heading).

If any file has zero or multiple H1 headings:
1. Review the file manually
2. Fix incorrect H1 headings or remove duplicate H1 headings
3. Re-run this validation step

#### Step 3: Front Matter validation

Verify each file starts with `---` YAML block:

```bash
for file in docs/*.md; do
    head -1 "$file" | grep -q '^---' || echo "Missing Front Matter: $file"
done
```

Expected result: No output (all files have Front Matter).

If any file is missing Front Matter:
1. Add YAML Front Matter following the template defined in the plan
2. Include appropriate title, category, tags, related documents, and source fields

#### Step 4: Related Documents / Keywords section validation

Verify each file has both `## Related Documents` and `## Keywords` sections at the end:

```bash
for file in docs/*.md; do
    tail -20 "$file" | grep -q '^## Related Documents' || echo "Missing Related Documents: $file"
    tail -20 "$file" | grep -q '^## Keywords' || echo "Missing Keywords: $file"
done
```

Expected result: No output (all files have both sections).

If any file is missing either section:
1. Add the missing section(s) at the end of the file
2. Populate with appropriate content based on related files and keywords

#### Step 5: Internal link reachability validation

Verify all internal links point to existing files:

```bash
for file in docs/*.md; do
    grep -oP '\[([^\]]+)\]\(\K([^)]+\.md)' "$file" | while read -r link; do
        dir=$(dirname "$file")
        target="$dir/$link"
        if [ ! -f "$target" ]; then
            echo "Broken link in $file: $link"
        fi
    done
done
```

Expected result: No output (no broken links).

If any broken links are found:
1. Identify the correct target file for each broken link
2. Update the link to point to the correct file
3. Re-run this validation step

#### Step 6: Code block / table breakage validation

Manually verify that fenced code blocks and Markdown tables are not broken across file splits:

1. For each split file pair (original → new files):
   - Compare fenced code block start/end markers (```)
   - Compare Markdown table row counts before and after splitting
   - Verify no partial code blocks or truncated tables exist
2. Pay special attention to:
   - Code blocks near split boundaries
   - Tables near split boundaries
   - Multi-line content that may have been split incorrectly

Expected result: No broken code blocks or tables detected.

If any breaks are found:
1. Move the affected content to the appropriate file
2. Ensure fenced code block markers are properly paired
3. Ensure table rows are complete (not split mid-row)

#### Step 7: index.md reachability validation

Verify `docs/index.md` can reach all category guide files:

1. Extract all links from `docs/index.md`
2. Verify each link target exists
3. Verify all 8 categories are linked (Overview, Deployment, RAG, MCP, Agent, Event Bus, Shared·DB, Known Issues)

Expected result: All category guide files are reachable from `index.md`.

### Method

- **Automated checks**: Steps 1-5 use automated commands to detect common issues quickly
- **Manual checks**: Steps 6-7 require manual inspection for edge cases that automated tools cannot reliably detect
- **Iterative validation**: Run all validation steps after each major operation (splitting, normalization, link correction) rather than waiting until the end

### Details

#### Validation prioritization

1. **Critical** (must fix immediately):
   - Files exceeding 8KB (breaks RAG chunking assumptions)
   - Missing Front Matter (breaks metadata extraction)
   - Broken internal links (causes navigation failures)

2. **Important** (fix before completion):
   - Multiple H1 headings (confuses document structure)
   - Missing Related Documents / Keywords sections (incomplete metadata)
   - Broken code blocks/tables (corrupted content)

3. **Nice to have** (improve over time):
   - Consistent naming conventions across categories
   - Complete tag coverage in Front Matter
   - Comprehensive cross-references between related files
