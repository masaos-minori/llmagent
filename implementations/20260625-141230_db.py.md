# Implementation: Add threading.Lock for shared connection protection

Source plan: `plans/20260625-135650_plan.md` (req #16, db.py portion)

## Goal

Add a module-level `threading.Lock` to `scripts/eventbus/db.py` and update the `open_db()` docstring to reflect the new multi-thread safety model (shared + explicit Lock), replacing the old single-event-loop-thread justification.

## Scope

- Add `_db_lock: threading.Lock` module-level variable
- Add `get_db_lock() -> threading.Lock` accessor function
- Update `open_db()` docstring
- Out of scope: applying the lock in `app.py` callables (covered in companion app.py doc), DB facade functions (req #15)

## Assumptions

1. req #15 (DB facade) is implemented before or concurrently with this change
2. The lock is owned by `db.py` and retrieved by `app.py` via `get_db_lock()`
3. `check_same_thread=False` remains; the lock provides the explicit serialization previously assumed from single-thread execution

## Implementation

### Target file

`scripts/eventbus/db.py`

### Procedure

1. Add `import threading` at the top
2. Add `_db_lock = threading.Lock()` at module level
3. Add `get_db_lock() -> threading.Lock` function
4. Update `open_db()` docstring

### Method

The lock is a simple module-level `threading.Lock`. It serializes all DB access from `asyncio.to_thread` callables in `app.py`. The lock object is retrieved by `app.py` via `get_db_lock()` rather than importing the private `_db_lock` directly.

### Details

```python
import threading

_db_lock = threading.Lock()


def get_db_lock() -> threading.Lock:
    """Return the lock that must be held for all DB operations.

    All asyncio.to_thread callables in app.py must acquire this lock before
    executing any sqlite3 operation on the shared connection.
    """
    return _db_lock
```
