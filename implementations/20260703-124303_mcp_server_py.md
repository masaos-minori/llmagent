## Goal

Extend `MCPServer.health()` in `mcp/server.py` to return `liveness`, `restart_recommended`, and `operator_action_required` fields in the base health response dict.

## Scope

- In-Scope:
  - Add three new keys to the dict returned by `MCPServer.health()`: `liveness` (bool), `restart_recommended` (bool), `operator_action_required` (bool)
  - Update the docstring to describe all seven returned fields and the new watchdog semantics
  - Healthy default: `liveness=True, restart_recommended=False, operator_action_required=False`
  - (The base class `deps` is always empty, so healthy is the only reachable state; the degraded defaults are documented for subclass guidance)
- Out-of-Scope:
  - Changes to any concrete MCP server health functions (covered in Step 2)
  - Changes to `attach_auth_middleware`, `_truncate_with_meta`, or any other helper
  - Changes to the watchdog itself

## Assumptions

1. `MCPServer.health()` in the base class always produces an empty `deps` dict, so the method always returns `ready=True`; the new fields will always reflect the healthy defaults.
2. The `test_mcp_server_base.py` `TestHealth` class asserts an exact dict equality on the return value; those assertions must be updated alongside this change (or the test already expects the new fields — it currently does not, so the test will break and must be updated in this same step).
3. The existing `test_health_response_shape_when_healthy` test in `test_mcp_server_health_status.py` checks only key presence, not exact value, so it will pass as long as all expected keys are present.
4. No concrete server calls `super().health()` — they each build their own dicts independently — so adding fields to the base class does not automatically propagate to concrete servers.

### Method

- Plain dict literal update; no new imports required.
- The `liveness` field is always `True` from the base class because the base class only runs if the process is alive. Subclasses may override with `liveness=False` when they detect a fatal internal state.
- Return type remains `tuple[dict[str, object], int]`.
