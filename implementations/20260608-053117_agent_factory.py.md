# Implementation: agent/factory.py — lifecycle wiring change

## Goal

Replace the `ServerLifecycleManager` facade usage with direct wiring of `HttpServerLifecycleManager` / `StdioServerLifecycleManager` based on `startup_mode`.

## Scope

- `scripts/agent/factory.py`
- `_build_tool_executor()` return type changes from `tuple[ToolExecutor, ServerLifecycleManager]` to `tuple[ToolExecutor, LifecycleManagerProtocol]`.

## Assumptions

- `startup_mode` is available from each `McpServerConfig`.
- For mixed-mode setups, a single lifecycle manager cannot represent both HTTP and stdio servers. The plan assumes `ServerLifecycleManager` handled both types; after removal, the factory may need to create both managers and wrap them.

## Implementation

### Target file

`scripts/agent/factory.py`

### Procedure

1. Replace `from agent.lifecycle import ServerLifecycleManager` with:
   - `from agent.lifecycle_protocol import LifecycleManagerProtocol`
   - `from agent.http_lifecycle import HttpServerLifecycleManager`
   - `from agent.stdio_lifecycle import StdioServerLifecycleManager`
2. Change `_build_tool_executor` return type annotation from `ServerLifecycleManager` to `LifecycleManagerProtocol`.
3. In the method body, replace `ServerLifecycleManager(...)` construction with conditionally creating the appropriate concrete manager. Since `ServerLifecycleManager` wraps both types, the factory may need to:
   - Group `server_configs` by transport type
   - Create `HttpServerLifecycleManager()` for HTTP servers
   - Create `StdioServerLifecycleManager(server_configs, tool_executor, stdio_procs, last_called)` for stdio servers
   - But `ToolExecutor.set_lifecycle()` expects a single lifecycle manager. This is a design tension — see Details.
4. Run `ruff format`.
5. Run `mypy`.

### Method

Refactoring of `_build_tool_executor()` to use concrete lifecycle managers.

### Details

The challenge: `ToolExecutor.set_lifecycle()` receives a single lifecycle object. If `ServerLifecycleManager` is removed, the factory needs to either:
- (a) Create a simple wrapper/adapter that holds both managers and delegates, or
- (b) Modify `ToolExecutor` to accept multiple lifecycle managers.

Check `ToolExecutor.set_lifecycle()` signature to determine the approach. If (a), the wrapper can be a simple class in `factory.py` or a separate module. If (b), scope expands beyond this plan.
