# Implementation procedure: `scripts/agent/commands/cmd_mcp.py`

Source plan: `plans/done/20260717-130901_plan.md` (requirement `requires/20260717_11_require.md`),
Implementation steps 1, 2, 4.

## Goal

Add a new `/mcp tools` subcommand that renders a concise table of every tool known to
`RuntimeToolRegistry` — `server_key`, `name`, `enabled_for_llm`, `agent_safety_tier`, `requires_approval`,
`is_write`, `requires_serial` — including disabled/policy-filtered tools, as a sibling to the existing
`/mcp status` health-probe subcommand. No change to `/mcp status`'s own output.

## Scope

**In scope**
- `scripts/agent/commands/cmd_mcp.py`: new `_cmd_mcp_tools()` method; new `_format_runtime_tools_table()`
  formatting helper; a `"tools"` branch in `_cmd_mcp()`'s dispatch; update the `UnknownSubcommandError`
  allowed-subcommand tuple; update the `"Usage: /mcp [status]"` line printed when `/mcp` is called with no
  subcommand.

**Out of scope**
- `/mcp tools dump <name>` (the plan's optional sub-subcommand) — explicitly deferred per the plan's own
  Implementation step 3 wording ("only implement if the base table view is confirmed insufficient during
  review — do not build speculatively"). Not built by this doc.
- `/mcp status`'s existing rendering path (`_cmd_mcp_status()`, `_format_mcp_table()`) — untouched.
- `scripts/agent/commands/registry.py` / `scripts/agent/commands/command_defs_list.py` — confirmed by direct
  read (`registry.py:43,62` already imports/composes `_McpMixin`; `command_defs_list.py:70-76`'s `/mcp`
  `CommandDef` is already a prefix command whose help text — `"MCP server status, tool list, connectivity
  check"` — already generically covers this feature). No change needed to either file.
- `scripts/shared/runtime_tool_registry.py` — already designed (see Assumption 1), not modified here.

## Assumptions

1. **`AppServices.runtime_tools` does not exist in real source yet — but `RuntimeToolRegistry` itself
   now does.** **Status update (post-review, 2026-07-18)**: `scripts/shared/runtime_tool_registry.py`
   now exists (172 lines, confirmed by direct read), matching the 13-field/9-method design this doc
   references. `AppServices.runtime_tools` is still absent, though (confirmed by direct grep:
   `scripts/agent/context.py`'s `AppServices.__init__` has no `runtime_tools` parameter/attribute). This
   mirrors the already-documented gap in `implementations/20260717-223600_config_reload.py.md`
   (Assumption 4): the field is designed (`implementations/20260717-203200_runtime_tool_registry.py.md`)
   as `AppServices.runtime_tools: RuntimeToolRegistry | None`, populated post-startup, nullable until MCP
   discovery completes, and is now owned by
   `implementations/20260718-111454_context_py_appservices_runtime_tools.md`. This doc's plan text
   ("`agent/context.py`: no change — `ctx.services_required.runtime_tools` is already available per
   requirements 02-03") remains **not accurate** — the field itself still does not exist in real source,
   only the registry class it would point to. This doc proceeds on the same working assumption as the
   config_reload doc (field name `runtime_tools`, type `RuntimeToolRegistry | None`) and is written so it
   applies mechanically once that field lands. **Blocking prerequisite**: this change cannot be exercised
   end-to-end until `AppServices.runtime_tools` actually exists — see the new owning doc above.
2. `RuntimeTool` (`implementations/20260717-203121_runtime_tool.py.md`) is a frozen dataclass with (among
   its 13 fields) exactly the 7 scalar fields this table needs: `server_key: str`, `name: str`,
   `enabled_for_llm: bool`, `agent_safety_tier: AgentSafetyTier` (a `Literal["READ_ONLY", "WRITE_SAFE",
   "WRITE_DANGEROUS", "ADMIN"]`, a plain string at runtime — **not** the same enum as `_format_mcp_table()`'s
   `TIER_LABELS`/`r.tier` in the existing `/mcp status` path, which is `McpProbeResult.tier`, an unrelated
   type; do not reuse `TIER_LABELS` for `agent_safety_tier`), `requires_approval: bool`, `is_write: bool`,
   `requires_serial: bool`.
3. `RuntimeToolRegistry.all_tools()` (`implementations/20260717-203200_runtime_tool_registry.py.md:129-130`)
   returns `list[RuntimeTool]` — a plain list, unsorted, unfiltered (includes disabled/policy-filtered
   tools, satisfying the plan's "disabled tools are visible" acceptance criterion with no extra flag).

## Implementation

### Target file

`scripts/agent/commands/cmd_mcp.py`.

### Procedure

1. Add a new module-level helper `_format_runtime_tools_table(tools: list[RuntimeTool]) -> str`, sibling to
   the existing `_format_mcp_table()` (line 26), following the same fixed-width-columns style. Sort input
   by `(server_key, name)` before rendering, per the plan's Design section ("stable, scannable output").
2. Add `_cmd_mcp_tools(self) -> None` to `_McpMixin`, sibling to `_cmd_mcp_status()`:
   - Read `ctx.services_required.runtime_tools`.
   - If `None` (registry not yet populated — e.g. discovery has not completed, or the field itself does not
     exist yet in a given deployment), write a single explanatory line (e.g. `"RuntimeToolRegistry not
     available."`) and return — do not raise, mirroring `_cmd_mcp_status()`'s defensive `registry is not
     None` checks (lines 99-100, 112-113) for the same nullable-service pattern.
   - Otherwise call `.all_tools()`, pass to `_format_runtime_tools_table()`, write the result.
3. In `_cmd_mcp()`:
   - Change the dispatch guard from `if sub in ("status", ""):` to a 3-way branch: `""`/`"status"` →
     existing status path (unchanged behavior); `"tools"` → `await self._cmd_mcp_tools()`; else → raise.
   - Update the allowed-subcommand tuple passed to `UnknownSubcommandError` from `("status",)` to
     `("status", "tools")`.
   - Update the empty-args usage line from `"Usage: /mcp [status]"` to `"Usage: /mcp [status|tools]"`.
4. No change to `_cmd_mcp_status()` itself, `_format_mcp_table()`, or any import already present other than
   adding `RuntimeTool` to the type-checking imports (for the new helper's type hint) — import from
   `shared.runtime_tool` (once that module exists; see Assumption 1), guarded the same way other
   forward-dependent imports in this batch are guarded (a plain top-level import is fine once the module
   exists; this doc does not invent a conditional-import workaround since `shared.runtime_tool` is
   unconditionally available once requirement 02 lands).

### Method

Plain function + method addition, no new class/protocol. Mirrors the existing `_format_mcp_table()` /
`_cmd_mcp_status()` pair structurally, so the two subcommands stay visually and structurally consistent.
