## Goal

Add documentation for the new effective-limit precedence (`min(request limit, service config
cap)` applied at the SQL layer) and the matched-vs-shown count wording introduced by the
companion `search.py`/`mdq_tools.py` documents, so `docs/04_mcp_04_04_mdq.md` accurately
describes `search_docs`'s result-count and limiting behavior instead of leaving readers to infer
it from source code.

  04_mcp_04_04_mdq.md:81-101`) describing:

## Implementation

### Target file

`docs/04_mcp_04_04_mdq.md`

### Procedure

1. Locate the end of the `### 検索モード` section, immediately after the existing blockquote
   notes (`docs/04_mcp_04_04_mdq.md:99-101`):
   ```
   > **注記:** MDQ はハイブリッド/セマンティック検索をサポートしない（FTS5/BM25のみ）。セマンティック検索が必要な場合は RAG パイプラインを使用すること。
   >
   > **注記:** MDQ 対 RAG の境界の詳細な定義については、[04_mcp_05 §MDQ vs RAG Boundary](04_mcp_05_04_mdq-rag-boundary.md#mdq-vs-rag-boundary) を参照。
   ```
2. Insert a new subsection immediately after (and before the blank line + `---` separator at
   line 103):
   ```
   #### 検索結果件数と制限

   `search_docs` の結果件数と上限は次の優先順位で決まる:

   - **有効な SQL 上限**: `effective_limit = min(リクエストの limit, サーバー設定の
     max_results_limit)` を SQL の `LIMIT` 句に直接適用する。これにより、リクエストが大きな
     `limit` を指定しても、設定上限を超える行がデータベースから取得されることはない。
   - **`max_results_limit`（リクエストパラメータ）**: 上記の有効な SQL 上限が適用された後、
     さらに表示件数を絞る二次的な表示上限として機能する（`effective_limit` 以下の値を指定した
     場合のみ意味を持つ）。
   - **件数表記**: レスポンスヘッダーは、絞り込みが発生しない場合（一致件数と表示件数が一致する
     場合）にのみ「N found」のような確定件数を表示する。絞り込みが発生した場合は「matched」
     （クエリに一致した正確な件数）と「shown」（実際に表示された件数）を区別して表記し、
     取得・制限された件数をあたかも正確な合計であるかのように主張しない。
   ```
3. Do not modify the file's existing `---`/`## Related Documents`/`## Keywords` trailer
   (lines 103-118) — the new subsection is inserted before it, as a sibling of `### 検索モード`.

### Method

Direct Markdown insertion of one new `####`-level subsection at the end of an existing `###`

  "matched"/"shown" map directly to `SearchResultResult`'s `matched_count`/`shown_count` fields

  a third, differently-worded terminology set in this doc.
  should this doc be read together with the companion パス制御 table update, note the two are
  about different concerns (authorization filtering vs. limit/count semantics) — do not merge
  the two subsections or cross-reference one from the other beyond what's naturally implied.

  file's convention uses dated notes for *behavior-change disclosures relative to the doc's own

  existing content is written (no dated note there either).
