# Implementation procedure: `tests/test_tool_policy.py`

Source plan: `plans/20260717-130159_plan.md` (requirement `requires/20260717_07_require.md`),
Implementation step 7 (test-file portion covering `tests/test_tool_policy.py`).

## Goal

Extend `tests/test_tool_policy.py` with test cases for the `OperationType.UNKNOWN` classification
path and for `RuntimeToolRegistry`-sourced classification, once `scripts/agent/tool_policy.py`'s
`classify_operation_type()`/`classify_risk()` are migrated to consult `RuntimeToolRegistry` as their
primary source (this plan's steps 4-5). Acceptance target: an unregistered/unannotated tool name
classifies as `OperationType.UNKNOWN` via `classify_operation_type()` and as `RiskLevel.HIGH` via
`classify_risk()`.

## Scope

**In scope**
- `tests/test_tool_policy.py` only — new test classes/cases, plus any fixture helper needed to
  construct/inject a `RuntimeToolRegistry` (or equivalent test double) into `classify_operation_type()`/
  `classify_risk()`.

**Out of scope**
- `scripts/agent/tool_policy.py` itself — this plan's own filename-match check
  (`implementations/done/20260715-154002_tool_policy.py.md`) found an existing implementation doc for
  this file and treated it as already covered per this workflow's skip rule. **Flag, do not silently
  trust**: direct read of the current `tool_policy.py` source during this investigation
  (`scripts/agent/tool_policy.py:14-24, 33-34, 52-62, 126-155`) shows the file still uses the *old*
  frozenset-membership implementation (`WRITE_TOOLS`/`DELETE_TOOLS`/`_EXEC_TOOLS`/
  `_GITHUB_MUTATION_TOOLS`, `_TIER_TO_RISK` fallback to `tool_constants.py`) with **no
  `RuntimeToolRegistry` import and no `OperationType.UNKNOWN` handling at all**. The existing "done"
  doc almost certainly documents an earlier, unrelated `tool_policy.py` change (e.g. the MDQ/RAG/
  CICD/Git write-tool-set fix visible in `tests/test_tool_policy.py::TestClassifyOperationType
  ::test_mdq_rag_cicd_git_write_tools`), not this plan's `RuntimeToolRegistry` migration. **This test
  doc's test cases are therefore written against the target end-state described in the plan's Design
  section, not against today's actual `tool_policy.py` behavior** — they will fail until a
  `tool_policy.py` change matching this plan's steps 4-5 actually lands. Do not implement
  `tool_policy.py` itself under this task (documents-only); this gap should be reported so a human can
  confirm whether the RuntimeToolRegistry migration for `tool_policy.py` still needs its own
  implementation pass despite the filename match.
- `scripts/shared/runtime_tool_registry.py` / `runtime_tool.py` — not yet implemented as real source
  (confirmed: `scripts/shared/runtime_tool_registry.py` and `scripts/shared/runtime_tool.py` do not
  exist on disk at all yet; only their own design docs exist under `implementations/`). Those docs
  (`implementations/20260717-203121_runtime_tool.py.md`,
  `implementations/20260717-203200_runtime_tool_registry.py.md`) resolve `classify_operation_type()`
  at the **shared-layer registry** to return only `Literal["read", "write"]` — **not** a full
  `OperationType` enum — because `agent.tool_enums.OperationType` cannot be imported into `shared/`
  under `.importlinter`'s `shared-is-leaf` contract. This is a direct conflict with this plan's own
  Design section, which recommends resolving Unknown #1 by adding an `operation_type: OperationType`
  field to `RuntimeTool` — that recommendation is not implementable as literally stated, since
  `RuntimeTool` lives in `shared/` and `OperationType` lives in `agent/`. **This conflict is
  unresolved and blocking**: it must be reconciled (e.g. `RuntimeTool` gains a locally-declared
  `Literal["read", "write", "delete", "execute", "api_write", "unknown"]` field instead, which
  `tool_policy.py` then wraps into `agent.tool_enums.OperationType` on the agent side) before
  `tool_policy.py`'s actual migration — and therefore before these test cases can pass for real. This
  test doc documents the target test *shape* assuming that reconciliation happens; it does not resolve
  it.

4. `OperationType.UNKNOWN` (added by the paired `tool_enums.py` implementation doc,
   `implementations/20260717-220404_tool_enums.py.md`) is available for import once that change lands.

## Implementation

### Target file

`tests/test_tool_policy.py`

### Procedure

1. Add a `_registry_with(...)` (or similarly named) test-local helper that builds a minimal
   `RuntimeToolRegistry`-shaped test double containing a small, explicit set of known tool names (e.g.
   `write_file` → write, `delete_file` → delete, `shell_run` → execute, `github_push_files` →
   api_write) and omits any name not explicitly listed — used to construct the "known vs. unknown tool"
   contrast the new tests need. Keep this local to the test file (do not import a partially-implemented
   `scripts/shared/runtime_tool_registry.py` if it does not exist yet at implementation time — this
   file does not exist on disk as of this investigation).
2. Add a new test class `TestClassifyOperationTypeUnknown` with:
   - `test_unregistered_tool_returns_unknown`: a tool name absent from both the registry double and
     every static frozenset (`_ALL_WRITE_TOOLS`, `DELETE_TOOLS`, `_EXEC_TOOLS`,
     `_GITHUB_MUTATION_TOOLS`) → `classify_operation_type(...) == OperationType.UNKNOWN` (not `READ` —
     this is the core fail-safe behavior change from today's `return OperationType.READ` default at
     `tool_policy.py:62`).
3. Add a new test class `TestClassifyRiskUnknown` with:
   - `test_unregistered_tool_returns_high_risk`: same unregistered tool name →
     `classify_risk(cfg, name, {}) == RiskLevel.HIGH` (string-compares as `"high"`, consistent with
     existing assertions elsewhere in this file that compare against plain strings).
   - `test_unregistered_tool_high_risk_not_overridden_by_default_tier_fallback`: confirms the
     unknown-tool path does not fall through to the current `WRITE_DANGEROUS` → `MEDIUM` mapping (per
     the plan's Design section flag that `_TIER_TO_RISK["WRITE_DANGEROUS"] == RiskLevel.MEDIUM` today,
     which must not silently apply to the unknown-tool default).
4. **Update** (not just add) `TestCfgHelperDefaults::test_unknown_tool_defaults_to_medium_risk`
   (current lines 171-173): rename to `test_unknown_tool_defaults_to_high_risk` and change the
   asserted value from `"medium"` to `"high"`, per Assumption 2 above. Add a one-line comment noting
   this is a deliberate behavior change from the pre-migration default, not a typo.
5. Do not modify `TestClassifyRiskConstantsFallback` (lines 302-333) — those cases test *known*,
   explicitly-classified tools (`delete_file`, `shell_run`, `write_file`) falling back to
   `tool_constants.py`-derived tiers when absent from `tool_safety_tiers`; that fallback tier remains
   valid post-migration per the plan's Step 4 ("static frozensets demoted to fallback ... for tools not
   yet in the registry") — only the *fully unregistered* case changes.

### Method

Plain pytest function-based test classes, consistent with the file's existing style (no new test
framework, no parametrize-heavy rewrite — mirror the existing one-assertion-per-test convention).

### Details

Pseudocode sketch (no production code, illustrative only):

```
class TestClassifyOperationTypeUnknown:
    def test_unregistered_tool_returns_unknown(self) -> None:
        assert classify_operation_type("totally_unregistered_tool_xyz") == OperationType.UNKNOWN


class TestClassifyRiskUnknown:
    def test_unregistered_tool_returns_high_risk(self) -> None:
        cfg = _cfg()
        assert classify_risk(cfg, "totally_unregistered_tool_xyz", {}) == "high"

    def test_unregistered_tool_high_risk_not_overridden_by_default_tier_fallback(self) -> None:
        cfg = _cfg()
        result = classify_risk(cfg, "totally_unregistered_tool_xyz", {})
        assert result != "medium"  # must not silently take the WRITE_DANGEROUS->MEDIUM path
        assert result == "high"


class TestCfgHelperDefaults:
    def test_unknown_tool_defaults_to_high_risk(self) -> None:
        # was: test_unknown_tool_defaults_to_medium_risk asserting "medium" -- now "high"
        # per this plan's fail-safe acceptance criterion for unregistered tools.
        cfg = _cfg()
        assert classify_risk(cfg, "unknown_tool", {}) == "high"
```

Use a tool-name string that is verifiably absent from every existing static set
(`_ALL_WRITE_TOOLS`, `DELETE_TOOLS`, `_EXEC_TOOLS`, `_GITHUB_MUTATION_TOOLS` in `tool_policy.py`) —
e.g. a clearly synthetic name like `"totally_unregistered_tool_xyz"` — to avoid an accidental false
pass if the name happens to collide with a real tool added later.
