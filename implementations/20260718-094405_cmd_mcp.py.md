# Implementation procedure: `scripts/agent/commands/cmd_mcp.py` (per-tool disabled-diagnostics table)

Source plan: `plans/20260717-175327_plan.md` ("RuntimeToolRegistry — represent disabled MCP tools
without exposing them to the LLM", requirement 17), Implementation step 7.

## Goal

Extend `scripts/agent/commands/cmd_mcp.py::_cmd_mcp_status()` with a new per-tool diagnostics table
(NAME, SERVER, CONFIG_DEP, ENABLED, DISABLED_REASON, LLM_VISIBLE), sourced from this plan's 6-field
`RuntimeToolRegistry.diagnostics()`, appended after the existing per-server table — per the
requirement's "`/mcp status` or equivalent diagnostics" acceptance criterion.

**Filename check — false positive, do not skip**: `implementations/20260718-032912_cmd_mcp.py.md`
also targets `scripts/agent/commands/cmd_mcp.py`, but its Goal is a **different feature**: it adds a
**new `/mcp tools` subcommand** (sourced from requirement 11, `plans/done/20260717-130901_plan.md`)
rendering 7 different fields (`server_key`, `name`, `enabled_for_llm`, `agent_safety_tier`,
`requires_approval`, `is_write`, `requires_serial` — the *other* 13-field `RuntimeTool` lineage's
fields) as a sibling command to `/mcp status`, explicitly stating "No change to `/mcp status`'s own
output." This plan's requirement instead extends the *existing* `_cmd_mcp_status()` output itself,
with a different, disjoint field set (`config_dependent`, `enabled`, `disabled_reason` have no
equivalent in the requirement-11 doc). Read both Goals in full before concluding — they are close
enough in wording ("per-tool diagnostics table") to look like a match at a glance, but target
different subcommands, different field sets, and different source plans. Not skipped.

**In scope**
- `scripts/agent/commands/cmd_mcp.py::_cmd_mcp_status()` — append a new per-tool table after the
  existing server table.
- A new helper function `_format_tool_diagnostics_table()` (module-level, alongside the existing
  `_format_mcp_table()`).

**Out of scope**

- Any change to `_format_mcp_table()` itself (per-server, unchanged) or to `_cmd_mcp()`'s dispatch
  logic (no new subcommand added by this plan; the requirement's acceptance criterion is satisfied
  by extending the existing `status` output, per this plan's Design step 5).

## Assumptions

  - `_format_mcp_table()` at lines 26-66 is a module-level function taking `rows:
    list[McpProbeResult]`, per-server, unrelated in shape to what this new table needs
    (`McpProbeResult` has no per-tool fields).
  - `_cmd_mcp_status()` at lines 75-150 calls `svc.probe_all()` (line 83), writes the server table
    (line 84), then a servers-summary line, degraded/unavailable server sections, and a
    serialization-stats block (lines 122-150). The new per-tool table is appended after all of this
    existing output, at the very end of `_cmd_mcp_status()`.
  - `_cmd_mcp()` at lines 152-162 dispatches `""`/`"status"` to `_cmd_mcp_status()`; no other
    subcommand exists in this file today (confirmed: `UnknownSubcommandError(sub, ("status",))` at
    line 162 is the only allowed-tuple).

  companion `llm_turn_runner.py` doc's Procedure step 4, which adds this `AppServices` field); if
  `None` (registry not yet populated — e.g. discovery hasn't run, or in tests that construct a bare
  context), `_cmd_mcp_status()` must skip the new table section entirely (write nothing extra)
  rather than error or print an empty table with a misleading header — consistent with how the
  degraded/unavailable-servers sections above it are already conditionally written only when
  non-empty (lines 102, 115).
- `diagnostics()` returns `list[dict[str, object]]` with keys `name`, `server_key`,
  `config_dependent`, `enabled`, `disabled_reason`, `enabled_for_llm` (per
  `implementations/20260718-094055_runtime_tool_registry.py.md`); the new table renders one row per
  dict, sorted by name (registry's `diagnostics()` already sorts).

## Implementation

### Target file

`scripts/agent/commands/cmd_mcp.py` (existing, 162 lines).

### Procedure

1. Add import: `from shared.runtime_tool_registry import RuntimeToolRegistry` — used only for a
   type-checking `isinstance`/type-hint reference, not instantiated in this file (module already
   imports service/model types the same lightweight way, e.g. `McpProbeResult`).
2. Add module-level `_format_tool_diagnostics_table(rows: list[dict[str, object]]) -> str`, mirroring
   `_format_mcp_table()`'s column-format-string style (fixed-width columns): NAME, SERVER,
   CONFIG_DEP, ENABLED, DISABLED_REASON, LLM_VISIBLE.
3. In `_cmd_mcp_status()`, after the existing serialization-stats block (after line 150), add: fetch
   `registry = ctx.services_required.runtime_tools`; if not `None` and `registry.diagnostics()` is
   non-empty, write a blank line, a section header (e.g. `"  Tools (RuntimeToolRegistry):"`), then
   `_format_tool_diagnostics_table(registry.diagnostics())`. Per the Risks note below (table-growth
   risk), group/order rows so disabled tools sort first or are visually distinguished (e.g. a
   leading `[DISABLED]` marker in the NAME column) — full detail matters most for disabled tools,
   per this plan's own Risks section mitigation.

### Method

Plain module-level formatting function + a few new lines inside the existing `_cmd_mcp_status()`
method — no new class, no new subcommand routing.
