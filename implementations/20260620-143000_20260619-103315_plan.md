# Implementation: Add integration tests for FTS5 fallback when normalized_content is NULL

**Plan:** `plans/20260619-103315_plan.md`
**Created:** 2026-06-20 14:xx

---

## Goal

Add end-to-end integration tests for FTS5 fallback behavior when `normalized_content` is NULL. Verify that the COALESCE trigger correctly falls back to `content` and that search results are accurate.

## Scope

**In scope:**
- New test file `tests/test_fts_fallback.py` with test classes: `TestEnglishFtsFallback`, `TestCodeFtsFallback`, `TestNormalizedContentEdgeCases`
- English chunk FTS fallback tests (single/multi-word queries, BM25 score ordering)
- Code chunk FTS fallback tests (symbols, original content preservation)
- Edge case tests (empty string vs NULL, mixed Japanese/English documents)
- Full pipeline integration test

**Out of scope:**
- Modifying the FTS5 schema or trigger logic
- Adding new search algorithms or tokenization strategies
- Testing the full ingestion pipeline

## Assumptions

1. Existing tests in `test_rag_repository.py` and `test_fts_japanese.py` provide a good foundation for patterns.
2. The COALESCE trigger logic is correct; tests should verify behavior, not fix bugs.
3. Tests should use in-memory SQLite with real FTS5 (not mocks).

## Implementation

### Step 1: Create new test file `tests/test_fts_fallback.py`

Use the same schema setup pattern as `test_fts_japanese.py`:
- Import `SQLiteHelper`, schema DDL from `scripts/db/schema_sql.py`
- Use in-memory SQLite database
- Set up triggers via schema DDL

### Step 2: Add English chunk FTS fallback tests

Class: `TestEnglishFtsFallback`

| Test | Description |
|---|---|
| `test_english_chunk_indexed_on_content_when_normalized_null` | Insert chunk with NULL normalized_content, search for keyword, verify match via COALESCE fallback to `content` |
| `test_english_multi_word_query` | Verify multi-word English queries (e.g., "machine learning") correctly match via FTS5 |
| `test_english_bm25_score_ordering` | Verify more relevant matches have better BM25 scores |

### Step 3: Add code chunk FTS fallback tests

Class: `TestCodeFtsFallback`

| Test | Description |
|---|---|
| `test_code_chunk_with_symbols` | Insert Python code with symbols (`->`, `===`, backticks), verify alphanumeric tokens are extracted correctly |
| `test_code_search_returns_original_content` | Verify returned content is original text (not normalized) |
| `test_code_chunk_indexed_on_content_when_normalized_null` | Insert code chunk with NULL normalized_content, search for keyword, verify match |

### Step 4: Add edge case tests

Class: `TestNormalizedContentEdgeCases`

| Test | Description |
|---|---|
| `test_empty_string_vs_null_normalized` | Verify behavior when `normalized_content=""` vs `NULL` (empty string returns empty in FTS due to COALESCE semantics) |
| `test_mixed_japanese_english_document` | Insert both JA and EN chunks in same document, verify each is indexed correctly |

### Step 5: Add full pipeline integration test

Class: `TestEnglishFtsFallback` (or new `TestFullPipeline`)

| Test | Description |
|---|---|
| `test_end_to_end_english_ingestion_search` | Write chunk JSON with NULL normalized_content -> insert via DB -> search via `fts_search()` -> verify results |
