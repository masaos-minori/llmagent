# Implementation procedure: `scripts/shared/tool_registry.py` (docstring compatibility relabeling)

Source plan: `plans/20260717-130629_plan.md` (requirement `requires/done/20260717_10_require.md`),
Implementation step 2.

## Goal

Correct `scripts/shared/tool_registry.py`'s module docstring so it no longer claims `ToolRegistry` is
the sole/primary runtime routing authority — once requirement 04 (`RuntimeToolRegistry`-based routing)
supersedes it — while accurately preserving what the module *does* still do afterward (registry seed
data and, per requirement 09's consolidation, a drift-detection comparison input inside
`McpToolDiscoveryService`).

## Scope

**In scope**
- The module docstring only (`scripts/shared/tool_registry.py` lines 1-25, specifically line 4's
  "Primary source of truth..." claim and lines 18-19's "Routing authority: ToolRegistry is the sole
  routing authority" claim).

**Out of scope**
- Any code/behavior change to `ToolRegistry` itself, or its removal (explicitly out of scope per this
  requirement's own Scope section).
- Requirement 04's actual routing migration (`shared/route_resolver.py`, `shared/tool_executor.py`) —
  not this requirement's job.

## Assumptions

1. Per the companion doc `implementations/20260717-225949_requirements_04_09_landing_check.md`,
   requirement 04 has **not landed** yet (no implementation procedure doc exists for
   `shared/route_resolver.py`/`shared/tool_executor.py`'s migration to `RuntimeToolRegistry`-based
   routing). This docstring correction therefore describes a target end-state and must not be applied
   to real source until requirement 04 lands — applying it earlier would make the docstring describe
   behavior the code doesn't have (this plan's own first Risk).
2. Requirement 09's consolidation doc (`implementations/20260717-224511_mcp_tool_discovery.py.md`)
   shows `McpToolDiscoveryService` absorbing `check_routing_drift_vs_live()`'s logic, which compares
   live-discovered tool lists against the static `ToolRegistry` for drift — so `ToolRegistry`'s
   frozenset-seeded data continues to serve a real purpose (drift-detection input) even after
   requirement 04 lands. The corrected docstring must state this, not claim `ToolRegistry` has zero
   remaining purpose (this plan's third Risk / Design step 2's "confirm against requirement 09" note).
3. Current docstring, verbatim, confirmed by direct read of `scripts/shared/tool_registry.py`:
   - Line 4: `Primary source of truth for MCP tool ownership and routing.`
   - Lines 18-19: `Routing authority:\n  ToolRegistry is the sole routing authority. Live /v1/tools is used only for startup\n  validation (drift detection), not for routing decisions.`
   - Lines 13-16 (module scope statement, unaffected by this edit): `This module owns tool-to-server ownership and routing only; it is not a schema/description registry. LLM-visible tool schemas come from each server's own` `tools.py` `TOOL_LIST`.

## Implementation

### Target file

`scripts/shared/tool_registry.py` — module docstring, lines 1-25 (no other lines change).

### Procedure

1. Gate: confirm requirement 04 has landed (re-check `implementations/` for a
   `route_resolver.py`/`tool_executor.py` production-code doc) before applying this edit to source.
2. Replace line 4's claim.
3. Replace the "Routing authority:" block (lines 18-19).
4. Add an explicit pointer to `scripts/shared/runtime_tool_registry.py`'s `RuntimeToolRegistry` as the
   current runtime routing authority.
5. Leave the "Ownership model" (lines 6-11) and "Drift detection" (line ~23+) sections intact —
   still accurate post-migration.

### Method

Direct text edit of the module docstring string literal only; zero logic/behavior change; no
import/signature change; `ruff format`/`ruff check`/`mypy` should pass trivially since only a comment
string changes.

### Details

- Line 4 replacement (paraphrase, not verbatim-mandated wording — implementer should keep the plan's
  intent, exact phrasing is a judgment call at implementation time):
  "Compatibility/fallback registry: seeds `tool_constants.py`-derived tool-to-server ownership data
  and serves as a drift-detection comparison input for `McpToolDiscoveryService`
  (`agent/services/mcp_tool_discovery.py`); no longer the runtime routing authority."
- Lines 18-19 ("Routing authority:") replacement: state that `RuntimeToolRegistry`
  (`scripts/shared/runtime_tool_registry.py`), populated from live `/v1/tools` at startup (requirement
  03), is now the runtime routing authority; `ToolRegistry`'s frozenset-seeded map remains as (a)
  startup drift-detection input (requirement 09), and (b) a compatibility/fallback source only — not
  a routing input for any residual runtime fallback path. Whether such a runtime fallback path exists
  at all is resolved separately — see
  `implementations/20260717-230218_startup_static_fallback_unknown_resolution.md` (this plan's step 6,
  addressed in a companion doc) — do not assume one exists without checking that doc first.
- Add a one-line pointer: "See `scripts/shared/runtime_tool_registry.py`'s `RuntimeToolRegistry` for
  the current runtime routing authority."
- Do not touch: "Ownership model" bullets (lines 6-11), "Config `tool_names` is NOT a routing input"
  line, "Drift detection: Canonical validation module: `shared.tool_routing_validation`" line — all
  still accurate.
