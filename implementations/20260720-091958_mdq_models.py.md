## Goal

Give `outline()`, `stats()`, and `health_check._check_stale_documents()` a single shared,
unit-correct staleness definition (`mtime_ns` in nanoseconds vs. `indexed_at` in seconds), so the
three call sites can never drift out of sync again; and give `search.py`'s structured search
result an honest distinction between "how many rows matched the query" and "how many rows are
shown in this response" instead of silently reporting the post-`LIMIT` row count as if it were
the true total.

## Scope

**In scope:**
- Add `is_stale(mtime_ns: int, indexed_at: float) -> bool` â€” a plain module-level function.
- Add `STALE_SQL_CONDITION: str` â€” a module-level string constant holding the SQL fragment

- Extend `SearchResultResult` (`mdq_models.py:243-248`, currently a `TypedDict` with fields
  `query`, `results`, `total`) to add `matched_count: int` and `shown_count: int`, replacing the
  single `total: int` field per this plan's Design Â§3.

- Any change to per-tool request models (`SearchDocsRequest`, `GetChunkRequest`, etc.,
  `mdq_models.py:87-150`) â€” untouched by this document (see the companion `search.py` document

## Implementation

### Target file

`scripts/mcp_servers/mdq/mdq_models.py`

### Procedure

1. **Stale-detection helper** â€” insert near the end of the file, after `SearchResultResult`'s
   current position or immediately before it (either placement is fine; keep it visually close
   to the file's other small free-function/constant additions, not buried mid-class-block).
   Pseudocode/signature:
   ```
   def is_stale(mtime_ns: int, indexed_at: float) -> bool:
       """True if the file's on-disk mtime (ns) is newer than the last indexed_at (s, time.time())."""
       return mtime_ns > int(indexed_at * 1e9)


   STALE_SQL_CONDITION: str = "mtime_ns > CAST(indexed_at * 1e9 AS INTEGER)"
   ```
   No new imports required â€” both symbols use only builtins.

2. **`SearchResultResult` field rename** â€” replace the `total: int` field with two fields:
   ```
   class SearchResultResult(TypedDict):
       """Search result wrapper containing query and result items."""

       query: str | None
       results: list[SearchResultItem]
       matched_count: int   # exact count of rows matching the query, no LIMIT applied
       shown_count: int     # len(results) â€” rows actually returned after effective_limit
   ```
   Every construction site (`search.py::_search_docs_structured()`, the early-return empty-query
   branch in the same function) must be updated to supply both new keys instead of `total` â€” see
   the companion `search.py` document for those call-site edits; this document only covers the
   `mdq_models.py` type definition itself.

### Method

Two independent, additive-in-spirit edits in one pass: (a) two new free-standing symbols with no
dependency on anything else in the file, (b) a field-shape change to an existing `TypedDict`
whose only producer/consumers are covered by companion documents (`search.py`,
`tests/test_mdq_search_counts.py`, `tests/test_mdq_search_limits.py`,
`tests/test_mdq_search_modes.py`). Both edits must land together with their respective
consumer-side updates, or `mypy`/`ruff` will immediately flag the mismatched `TypedDict` keys at
`search.py`'s construction site.

### Details

- `is_stale()` takes `mtime_ns: int` â€” callers passing a SQLite `Row` value must ensure it is
  already an `int` (SQLite integer columns deserialize to Python `int` via the default adapter;
  no explicit cast needed at the call site beyond what `mdq_service.py::outline()` already does
  today).
- `STALE_SQL_CONDITION` is a bare string, not an f-string â€” it has no interpolated variables, so
  callers embed it directly into a larger f-string (e.g.
  `f"SELECT COUNT(*) as cnt FROM documents WHERE {STALE_SQL_CONDITION}"`) rather than calling it.
- Do not add either symbol to an `__all__` list â€” `mdq_models.py` has no `__all__` today
  (confirmed by the sibling `mdq_models.py` document's Assumption 2); nothing to update there.
- Keep `matched_count` before `shown_count` in field order (matches the plan's Design Â§3
  ordering and the natural "count everything, then count what's shown" reading order).
