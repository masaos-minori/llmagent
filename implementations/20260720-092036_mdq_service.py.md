# Implementation: scripts/mcp_servers/mdq/mdq_service.py (fix `outline()` stale-comparison bug;
switch `stats()` to shared `STALE_SQL_CONDITION`)

## Goal

Fix the actual reported bug: `outline()` compares `mtime_ns` (nanoseconds) directly against
`indexed_at` (seconds since epoch) without unit conversion, so any successfully indexed file
satisfies `mtime_ns > indexed_at` (nanosecond values are ~10^9 times larger than second values),
producing a spurious "file has been modified since last indexing" warning on effectively every
`outline()` call. Bring `outline()` into line with the already-correct comparison used by
`stats()` and `health_check._check_stale_documents()`, and remove the duplicated inline SQL
literal from `stats()` in favor of the new shared `STALE_SQL_CONDITION` constant (see companion
`mdq_models.py` document, `implementations/20260720-091958_mdq_models.py.md`).

## Scope

**In scope:**
- `outline()` (`mdq_service.py:168-251`): replace the buggy direct-comparison at line 212
  (`if doc_row["mtime_ns"] > doc_row["indexed_at"]:`) with a call to
  `is_stale(doc_row["mtime_ns"], doc_row["indexed_at"])`, imported from `mdq_models`.
- `stats()` (`mdq_service.py:300-324`): replace the inline SQL literal at line 314
  (`"SELECT COUNT(*) as cnt FROM documents WHERE mtime_ns > CAST(indexed_at * 1e9 AS INTEGER)"`)
  with `f"SELECT COUNT(*) as cnt FROM documents WHERE {STALE_SQL_CONDITION}"`, importing
  `STALE_SQL_CONDITION` from `mdq_models`.

**Out of scope:**
- Everything the sibling document (`20260720-091248_mdq_service.py.md`) already covers:
  `__init__`'s `MdqConfig` wiring, the deny-all startup warning, `get_chunk()`/`grep_docs()`
  authorization changes — untouched by this document.
- `search_docs()` (`mdq_service.py:129-136`) — thin delegating wrapper to `search.py`, unaffected
  by the stale-detection fix; see companion `search.py` document for its own separate changes.
- Any change to `outline()`'s truncation/heading logic (`mdq_service.py:218-246`) — unrelated,
  untouched.

## Implementation

### Target file

`scripts/mcp_servers/mdq/mdq_service.py`

### Procedure

1. Add `is_stale` and `STALE_SQL_CONDITION` to the existing
   `from mcp_servers.mdq.mdq_models import (...)` block, in alphabetical order alongside the
   other imported names, to satisfy ruff's `I` (isort) rule.
2. Replace line 212:
   ```python
   if doc_row["mtime_ns"] > doc_row["indexed_at"]:
   ```
   with:
   ```python
   if is_stale(doc_row["mtime_ns"], doc_row["indexed_at"]):
   ```
   Leave the surrounding guard (lines 207-211) and the `stale_warning` message construction
   (lines 213-216) unchanged — only the condition itself changes.
3. Replace lines 313-315:
   ```python
   stale_count = conn.execute(
       "SELECT COUNT(*) as cnt FROM documents WHERE mtime_ns > CAST(indexed_at * 1e9 AS INTEGER)"
   ).fetchone()["cnt"]
   ```
   with:
   ```python
   stale_count = conn.execute(
       f"SELECT COUNT(*) as cnt FROM documents WHERE {STALE_SQL_CONDITION}"
   ).fetchone()["cnt"]
   ```

### Method

Two isolated single-line/single-expression substitutions in two different methods of the same
class (`outline()` and `stats()`) — no control-flow, signature, or surrounding-logic changes in
either method. Each substitution is independently testable: `outline()`'s fix is a genuine
behavior change (removes the spurious warning); `stats()`'s substitution is a no-op behaviorally
(same SQL text, different origin) and exists purely to eliminate the duplicated magic string.

### Details

- After this change, `mdq_service.py` has zero remaining occurrences of the raw literal
  `"mtime_ns > CAST(indexed_at * 1e9 AS INTEGER)"` — both call sites reference
  `STALE_SQL_CONDITION`/`is_stale` from `mdq_models` instead.
- Do not change the `stale_warning` message text (lines 213-216) — it still reports the raw
  `mtime_ns`/`indexed_at` values for operator diagnostics; only the *comparison* that decides
  whether to emit the message is fixed.
- `outline()`'s existing docstring, authorization check (`authorize_path`, lines 179-183), and
  heading-truncation logic are all untouched — confirmed no other line in the method references
  `mtime_ns`/`indexed_at` besides lines 203-216.
