# Implementation Procedure: web_search_tools.py

Source plan: `plans/20260719-192933_plan.md` ("Validate WebSearchConfig and align search_web input schema")

## Goal

Make the `search_web` MCP tool's LLM-facing `inputSchema` in `web_search_tools.py`
match the runtime `SearchRequest` constraints (query length, max_results bounds), so
the LLM cannot construct tool calls that pass JSON-Schema validation but fail
Pydantic runtime validation.

## Scope

**In scope**
- `TOOL_LIST[0]["inputSchema"]` for `search_web`: add `query.minLength`,
  `query.maxLength`, `max_results.minimum`, `max_results.maximum`.
- Import the bound values from `web_search_models.py` rather than duplicating
  literals.

**Out of scope**
- The validation/normalization logic itself, which lives in `web_search_models.py`
  (separate target file/doc: `implementations/20260720-080006_web_search_models.py.md`).
- `web_search_server.py` dispatch/audit mechanics.

## Assumptions

1. Per the plan's Assumption 2: the schema stays hand-written with shared constants
   imported from `web_search_models.py`, rather than generated from
   `SearchRequest.model_json_schema()` — `web_search_tools.py` currently has zero
   imports from `web_search_models.py`, and the MCP `inputSchema` format is not a
   literal Pydantic JSON schema dump, so hand-written + shared constants is the
   lower-risk increment.
2. Per the plan's Design note (item 2): the schema's `maximum` should reflect the
   live, TOML-loaded `_cfg.max_results_limit` value (via an accessor or by importing
   the singleton), not the static `MAX_RESULTS_LIMIT` module constant — because the
   static constant does not reflect a TOML override, and `SearchRequest`'s own Field
   bound is already startup-time-bound to `_cfg`. This keeps the schema and the
   runtime model consistent within a single process lifecycle.

## Implementation

### Target file

`scripts/mcp_servers/web_search/web_search_tools.py`

Current shape (verified by reading the live file):
- L1-9: module docstring, `from __future__ import annotations`, `from typing import Any`.
  No imports from `web_search_models.py`.
- L11-30: `TOOL_LIST: list[dict[str, Any]] = [...]` — single entry, `name: "search_web"`.
- L17-27: `inputSchema`:
  - L20: `"query": {"type": "string", "description": "Search query string"}` — no
    `minLength`/`maxLength`.
  - L21-24: `"max_results": {"type": "integer", "description": "Maximum number of
    results (default: configured)"}` — no `minimum`/`maximum`.
  - L26: `"required": ["query"]`.

### Procedure

1. Add an import from `web_search_models.py`. Two options, per the plan's stated
   tradeoff (record the choice made, do not silently pick one):
   a. Import the module-level singleton value directly:
      `from mcp_servers.web_search.web_search_models import _cfg` and reference
      `_cfg.max_results_limit` — simplest, but reaches into a "private" (underscore)
      name across a module boundary.
   b. Add a small public accessor `get_max_results_limit() -> int` to
      `web_search_models.py` returning `_cfg.max_results_limit`, and import that
      function instead — cleaner public surface, no underscore-name reach-through.
   Recommendation: option (b), since it avoids importing a private-by-convention
   name into another module and is a trivial addition to
   `web_search_models.py` (would need to be folded into that file's own doc/PR if not
   already covered there).
2. Update `inputSchema.properties.query` (L20) to add `"minLength": 1, "maxLength": 500`
   — these two values are Pydantic `Field` constants already in
   `SearchRequest.query` (`web_search_models.py` L73-74) and can be hand-copied as
   literals since they are not expected to vary at runtime (unlike
   `max_results_limit`, which is config-driven).
3. Update `inputSchema.properties.max_results` (L21-24) to add `"minimum": 1` (a
   Pydantic `Field` constant, `ge=1` at `web_search_models.py` L79) and
   `"maximum": <imported max_results_limit value>` (the config-driven bound).
4. Do not change `"required": ["query"]` or the tool's `"description"`/`"status"`.

### Method

Pseudocode only (per `skills/python-design/SKILL.md` — no production code blocks):

```
from mcp_servers.web_search.web_search_models import get_max_results_limit  # or _cfg, per chosen option

TOOL_LIST = [
    {
        "name": "search_web",
        "description": "...",
        "inputSchema": {
            "type": "object",
            "properties": {
                "query": {
                    "type": "string",
                    "description": "Search query string",
                    "minLength": 1,
                    "maxLength": 500,
                },
                "max_results": {
                    "type": "integer",
                    "description": "Maximum number of results (default: configured)",
                    "minimum": 1,
                    "maximum": get_max_results_limit(),
                },
            },
            "required": ["query"],
        },
        "status": "production",
    },
]
```

### Details

- Importing `web_search_models` at module-import time captures whatever
  `max_results_limit` was in `config/web_search_mcp_server.toml` at process
  startup — same lifecycle as `SearchRequest`'s own `Field` bound (both are
  startup-time-bound to `_cfg`), so a config change requires a server restart to
  be reflected in `/v1/tools`. This is documented as an accepted tradeoff in the
  plan's Risks table, not a bug — confirm the docs update
  (`docs/04_mcp_04_01_web-search-file-read-github.md`, see its own doc:
  `implementations/` entry for that file) states this explicitly.
- Verify the import direction does not violate the layer contract in
  `.importlinter`: both `web_search_models.py` and `web_search_tools.py` live in
  `scripts/mcp_servers/web_search/`, i.e. within the same `mcp_servers` layer, so
  `web_search_tools.py` importing `web_search_models.py` is intra-layer and not a
  boundary violation (`mcp_servers → db, shared` contract is unaffected).
- This module is imported by `web_search_server.py` (for `TOOL_LIST` at server
  startup) and by three test files (`tests/test_tool_server_layer_consistency.py`,
  `tests/test_mcp_tool_schema_exports.py`, `tests/agent/services/test_mcp_tool_discovery.py`)
  — confirm none of them assert the *exact* prior schema shape (missing
  `minLength`/`maxLength`/`minimum`/`maximum` keys) in a way that would now fail;
  if they do, those are expected/acceptable schema-shape updates, not regressions.
