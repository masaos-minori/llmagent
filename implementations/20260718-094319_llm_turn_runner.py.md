# Implementation procedure: `scripts/agent/llm_turn_runner.py` (filter disabled MCP tools from LLM tool list)

Source plan: `plans/20260717-175327_plan.md` ("RuntimeToolRegistry — represent disabled MCP tools
without exposing them to the LLM", requirement 17), Implementation step 6.

## Goal

Wire `RuntimeToolRegistry` (this plan's 6-field variant,
`implementations/20260718-094055_runtime_tool_registry.py.md`) into
`scripts/agent/llm_turn_runner.py::_stream_llm()` so that disabled MCP tools are excluded from the
`tool_definitions` payload sent to the LLM, while any non-MCP tool definitions remain visible (see the
post-review note in Scope below: the plugin subsystem this was originally framed around has since been
removed entirely).

`scripts/agent/llm_turn_runner.py` for this RuntimeToolRegistry batch specifically — the only
`llm_turn_runner*` matches in `implementations/done/` (`20260608-053617_agent_llm_turn_runner.py.md`,

RuntimeToolRegistry batch (which started 2026-07-17) — unrelated historical work on this same file
(module extraction/orchestrator refactors), not this feature. No skip; this is a genuinely new item.

**Review note (post-review, 2026-07-18) — this plan's "6-field variant" is superseded, not a
separate design**: a cross-batch review found this plan's `RuntimeToolRegistry` design (referenced
above as `implementations/20260718-094055_runtime_tool_registry.py.md`, a doc that in fact was never
written) is incompatible with the 13-field/9-method `RuntimeTool`/`RuntimeToolRegistry` lineage used
by requirements 02/03/04-08/11/13 (`implementations/20260718-084710_runtime_tool.py.md` /
`implementations/20260717-203200_runtime_tool_registry.py.md`). **Decision: the 13-field lineage is
the adopted baseline.** Everywhere below that says "this plan's 6-field variant," read instead: the
disabled-tool-visibility fields (`config_dependent`/`enabled`/`disabled_reason`) and methods
(`get_llm_visible_definitions()`/`known_names()`/`diagnostics()`) this plan needs must be added to
the existing 13-field `RuntimeTool`/9-method `RuntimeToolRegistry`, not built as a second, disjoint
class. `_filter_disabled_tool_definitions()`'s logic below is otherwise unaffected by this
correction — only the registry it reads from changes from "a separate 6-field class" to "the
13-field class, extended."

## Scope

**In scope**
- `scripts/agent/llm_turn_runner.py::_stream_llm()` — filter `ctx.cfg.tool.tool_definitions` through
  a `RuntimeToolRegistry` lookup before passing to `llm.stream()`.
- The minimal `AppServices`/`AgentContext` plumbing needed to make a populated
  `RuntimeToolRegistry` reachable from `_stream_llm()` (see Assumptions — this plan's own Affected

**Out of scope**

- Any change to `execute_all_tool_calls()` / `tool_runner.py`'s dispatch path (the "reject execution
  attempts" behavior is `RuntimeToolRegistry.check_executable()`'s own concern, wired at the
  execution call site in a later step not covered by this plan's Implementation steps list).
- **Superseded (post-review, 2026-07-18)**: `scripts/agent/factory.py::_init_plugin_registry` no
  longer exists — the plugin subsystem has been fully removed from real source (confirmed: no
  `plugin_registry.py` on disk, no plugin-related function in `factory.py`; recorded in
  `implementations/done/20260717-224511_plugin_registry_py_removal.md`). There is no plugin-tool

  unchanged" branch below is still worth keeping (it protects against any future non-MCP tool source

  read every "plugin" reference below as "any non-MCP tool definition, if one exists," not as a
  reference to the now-removed plugin subsystem.

  discovered gap, not silently worked around: `_stream_llm()` must treat
  `ctx.services_required.runtime_tools` as `None`-safe (registry may be `None` if discovery hasn't
  run yet, e.g. very early startup or in unit tests that don't wire it), falling back to the
  unfiltered `ctx.cfg.tool.tool_definitions` list when `None` — this also keeps existing tests that
  don't set up a registry passing unmodified.
- Per `get_llm_visible_definitions()`'s corrected return shape (per the registry doc's Assumptions:
  `list[str]` of enabled tool *names*, not full definition dicts, since this plan's 6-field
  `RuntimeTool` has no `description`/`input_schema` of its own), the actual filtering in
  `_stream_llm()` is: keep every entry in `ctx.cfg.tool.tool_definitions` whose
  `entry["function"]["name"]` is either (a) in the registry's LLM-visible name set, or (b) not
  known to the registry at all (i.e. not an MCP tool the registry tracks — covers plugin-provided
  tool definitions, which never went through MCP discovery and must not be silently dropped, per
  this plan's own "LLM tool-list merge risk").  Only names the registry *does* track and marks
  disabled are excluded. This directly satisfies the plan's Risk mitigation ("required decision, not
  an afterthought") for the plugin-tool-visibility concern.
- `ctx.cfg.tool.tool_definitions` entries have shape `{"type": "function", "function": {"name":
  ..., "description": ..., "parameters": ...}}` (confirmed by cross-referencing
  `scripts/agent/repl_health.py:291-295`'s existing parsing: `td["function"]["name"] for td in
  ctx.cfg.tool.tool_definitions if "function" in td`) — the same defensive `"function" in td` guard
  should be reused here for entries lacking that key (skip filtering logic for those, pass through
  unchanged, since they're not shaped as expected and shouldn't be silently dropped either).

## Implementation

### Target file

`scripts/agent/llm_turn_runner.py` (existing, 216 lines; lines 202-215 are the integration point).

### Procedure

1. Add a private helper method `_filter_disabled_tool_definitions(self, tool_definitions:
   list[dict[str, Any]]) -> list[dict[str, Any]]` on `LLMTurnRunner` (near `_stream_llm`, after
   `_finalize_answer_text`).
2. In `_stream_llm()`, replace the direct `ctx.cfg.tool.tool_definitions` argument at line 213 with
   `self._filter_disabled_tool_definitions(ctx.cfg.tool.tool_definitions)`.
3. `_filter_disabled_tool_definitions()`: read `ctx.services_required.runtime_tools`; if `None`,
   return the input list unchanged (no registry populated yet — same behavior as today); otherwise
   compute `visible_names = set(registry.get_llm_visible_definitions())` and return
   `[td for td in tool_definitions if "function" not in td or td["function"].get("name") not in
   registry_known_names or td["function"].get("name") in visible_names]` — i.e. keep everything
   except entries whose name the registry *tracks and marks not-LLM-visible*. This requires the
   registry to also expose (or this helper to derive from `diagnostics()`) the set of *known*
   (tracked) tool names, not just the visible subset, to correctly distinguish "not an MCP tool at
   all" from "an MCP tool the registry knows is disabled." If `RuntimeToolRegistry` does not expose
   a "known names" accessor, add one (`known_names() -> set[str]`, trivial `set(self._tools)`) as a
   small addition to the registry doc's method list — flagged here since the registry doc's method
   list (register/get_llm_visible_definitions/is_executable/check_executable/diagnostics) does not
   currently include it but this integration needs it.
4. Add `runtime_tools: RuntimeToolRegistry | None = None` to `AppServices.__init__()`
   (`scripts/agent/context.py:134-156`) as a new optional constructor parameter/attribute, mirroring
   how `gateway: RepositoryGateway | None = None` is already an optional, late-bound field there.
   Wire `factory.py::build_agent_context()` to construct an `McpToolDiscoveryService(ctx)` and call
   `.discover()` to populate it — the exact startup timing (blocking startup vs. best-effort
   background task) is an implementation-time decision for whoever wires this; this doc does not
   prescribe it further since `factory.py`/`startup.py` are not in this plan's Affected areas table.

### Method

Small pure-ish filtering helper (reads `ctx.services_required.runtime_tools`, returns a new
filtered list) plus one call-site substitution. No new class or protocol needed.
