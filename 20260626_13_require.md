# [High] Implement production-grade /health endpoint for mdq-mcp

## Priority
High

## Labels
`mcp`, `mdq`, `healthcheck`, `operations`, `production-readiness`

## Dependencies
- Fix mdq-mcp syntax errors and incomplete server entry point

## Background
The current `/health` implementation is incomplete. Production MCP servers require reliable health checks for startup verification, watchdog monitoring, and operator diagnostics.

## Task
Implement a complete `/health` endpoint for `mdq-mcp`.

## Implementation Requirements
The health response must include:
- `status`
- `ready`
- `dependencies`
- `details`

The endpoint must verify:
- MDQ database path is configured.
- SQLite connection can be opened.
- Required tables exist.
- Required FTS5 table exists.
- Required triggers exist.
- A basic FTS5 query can be executed.
- Index statistics can be read.

Add useful `details`, such as:
- database path
- document count
- chunk count
- FTS row count
- last indexed timestamp, if available
- stale document count, if available

## Acceptance Criteria
- `GET /health` returns `ready: true` only when MDQ is actually usable.
- Broken DB schema results in `ready: false`.
- Missing FTS table results in `ready: false`.
- The response no longer contains `stub: true`.
- `/mcp` status can accurately reflect MDQ server readiness.
