# Decide and document `notes` table schema policy

## Priority
Low

## Labels
database, schema, notes, cleanup, migration

## Dependencies
- Delete `NoteRepository`
- Remove note delegation APIs from `AgentSession`

## Background
Removing the note feature eliminates runtime access to the `notes` table. The project must decide whether to leave the existing table unused or remove it from new schemas.

## Task
Define the schema policy for the legacy `notes` table.

## Implementation Requirements
Choose one policy:

Option A:
- Leave existing `notes` tables in old databases untouched.
- Stop using them from runtime code.
- Remove the table from documentation as an active feature.

Option B:
- Remove `notes` from new database schema creation.
- Add an explicit migration to drop `notes` if desired.

If backward compatibility is intentionally ignored, Option B is allowed, but it should still be explicit.

## Acceptance Criteria
- The selected policy is documented.
- Runtime code does not depend on the `notes` table.
- If new schemas are changed, schema tests are updated.

## Source Context
`NoteRepository` is the current runtime access layer for the `notes` table. Once removed, the table becomes unused unless kept for legacy data.
