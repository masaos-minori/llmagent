# Normalize `None` message content at the session persistence boundary

## Priority
High

## Labels
- db
- session
- persistence
- compatibility
- high-priority

## Dependencies
None

## Background
`LLMMessage.content` may be `None` for assistant messages that contain only tool calls. However, the session messages table currently stores `content` as `TEXT NOT NULL`, and the existing tests use an empty string for assistant tool-call-only messages.

The current preferred behavior is to keep the database schema unchanged and normalize `None` to an empty string at the repository boundary.

## Scope
Modify:
- `agent/session_message_repo.py`
- `tests/test_session_message_repo.py`

Do not modify:
- `db/schema_sql.py`
- `messages.content` nullability
- Low-level DB schema migrations

## Required Changes
- Update `SessionMessageRepository.save()` so `content` accepts `str | None`.
- Normalize `content` before insert:
  - `None` must be stored as `""`.
  - Existing string content must be preserved.
- Update `SessionMessageRepository.save_many()` so message tuples may contain `content: str | None`.
- Normalize `None` values in `save_many()` before `executemany()`.
- Keep `fetch_messages()` returning string content.
- Keep `messages.content TEXT NOT NULL`.

## Tests
Update `tests/test_session_message_repo.py`.

Add tests for:
- Saving an assistant message with `content=None` and `tool_calls`.
- Saving multiple messages where one message has `content=None`.
- Verifying restored content is `""`, not `None`.
- Verifying existing behavior for normal string content remains unchanged.

## Acceptance Criteria
- `repo.save("assistant", None, tool_calls=[...])` succeeds.
- The stored message has `content == ""`.
- `repo.save_many()` handles `None` content safely.
- Existing session message tests continue to pass.
- The database schema remains unchanged.
