# Title
Unify the memory JSONL configuration name and ownership

## Priority
Medium

## Labels
- bug
- agent

## Dependencies
- None

## Summary
Resolve the mismatch between `memory_jsonl_dir` and `memory_jsonl_path` so configuration, implementation, and docs use one consistent contract.

## Background
Configuration docs describe `memory_jsonl_dir` as the JSONL source directory.
Memory docs describe the JSONL file path as being controlled by `memory_jsonl_path`.
This suggests either a naming mismatch or an incomplete ownership model between config and implementation.

## Problem
Operators and AI implementers may misconfigure the memory JSONL location because docs and implementation expectations are not aligned.

## Required Changes
- Choose one canonical configuration model:
  - directory-based, or
  - file-path-based
- Align docs, config loading, status output, and errors with that model.
- If backward compatibility is needed, document it explicitly.
- Add tests for correct JSONL path/dir resolution.

## Acceptance Criteria
- Docs and runtime agree on how the JSONL location is configured.
- Operators can configure memory storage without ambiguity.
- If compatibility aliases remain, they are intentional and documented.

## Out of Scope
- Replacing JSONL as the canonical source
- Refactoring the entire memory storage subsystem
