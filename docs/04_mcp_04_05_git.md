# MCP Server Catalog: git-mcp

## git-mcp（ポート 8014）

**目的:** 2段階の安全ガードを備えたローカル git リポジトリ操作。
**起動モード:** persistent（HTTP）
**設定:** `config/git_mcp_server.toml`
**認証:** `GITHUB_TOKEN` は不要; ローカルの git 認証情報を使用する

**ツール:**

全ツールとも config が必須（`config_dependent: true`）。

gitサーバーの`enabled`/`disabled_reason`の計算：`allowed_repo_paths`が空の場合、優先的に無効化（理由 `"allowed_repo_paths is empty"`）。それ以外の場合、`read_only=true`なら書き込みツールのみ無効化（理由 `"read_only=true"`）。詳細は[04_mcp_03_06_tool-runtime-availability-metadata.md](04_mcp_03_06_tool-runtime-availability-metadata.md)。

| ツール | ティア | `read_only` ガード | `dry_run` | `config_dependent` |
|---|---|---|---|---|
| `git_status` | READ_ONLY | — | — | yes |
| `git_log` | READ_ONLY | — | — | yes |
| `git_diff` | READ_ONLY | — | — | yes |
| `git_branch` | READ_ONLY | — | — | yes |
| `git_show` | READ_ONLY | — | — | yes |
| `git_add` | WRITE_SAFE | `read_only=true` の場合ブロック | yes | yes |
| `git_commit` | WRITE_SAFE | ブロック | yes | yes |
| `git_checkout` | WRITE_DANGEROUS | ブロック | yes | yes |
| `git_pull` | WRITE_DANGEROUS | ブロック | yes | yes |
| `git_push` | WRITE_DANGEROUS | ブロック | yes | yes |

「ティア」列は `scripts/mcp_servers/git/` 自体には存在しない分類であり、`config/agent.toml` の `[tool_safety_tiers]`（agent 層の承認ポリシー設定）に由来する値である。git-mcp サーバー実装が内部でティアを判定・保持しているわけではない。(Explicit in code)

**ヘルス:** git が見つかる場合は `{"status":"ok","ready":true,"liveness":true,"restart_recommended":false,"operator_action_required":false,"dependencies":{},"details":{}}`; 見つからない場合は `"status":"degraded","ready":false,"dependencies":{"git":"git not found in PATH"/"check failed"}}` — ready 時は HTTP 200、degraded 時は 503。
**設定:**

| キー | デフォルト | 備考 |
|---|---|---|
| `allowed_repo_paths` | `[]` | fail-closed; 空 = 全て拒否; パスは `Path.resolve()` で解決される |
| `read_only` | `true` | 明示的に `false` にしない限り、全ての書き込みツールは `[DENIED]` を返す |
| `max_log_entries` | `50` | `git_log` のエントリ数上限 |
| `auth_token` | `""` | MCP サーバー呼び出し認証用の Bearer トークン |

**注記（2026-07-13）:** `audit_log_path` は `config/git_mcp_server.toml` から削除された。実装で一切参照されない未使用キーであり、設定ファイルに残すと「git 操作がこのパスに監査ログとして書き込まれる」という誤った期待を招くため。`GitConfig.audit_log_path` フィールド自体（デフォルト `""`）は `models.py` に残っているが、キー未設定でも正常動作する。

**注記:** `git_show` は8000文字で切り詰められる。`git_log` の inputSchema はデフォルトで `max_entries=20`; config の上限 `max_log_entries` のデフォルトは `50`。

### 実装上の補足

- `audit_log_path`（設定ファイルからは削除済み、上記注記参照）に対応する `GitConfig.audit_log_path` フィールドは `GitService`/`server.py` のどこからも参照・書き込みに使われていない。実際の呼び出しログは `server.py::call_tool` が `mcp_servers.audit._audit_log` 経由で標準の `logging.getLogger(__name__)`（モジュールロガー）に JSON 行として出力するのみであり、指定パスへ直接書き込む処理は実装されていない。github-mcp（`service_security.py`）や shell-mcp（`service.py`）、file-delete-mcp（`delete_service.py`）には `audit_log_path` を実際にオープンして追記する専用ロジックがあるが、git-mcp には同等の実装がない。(Explicit in code)
- 書き込み系5ツール（`git_add`/`git_commit`/`git_checkout`/`git_pull`/`git_push`）はすべて `GitService._validate_repo` 内で `allowed_repo_paths` チェック後に `_check_write()`（`read_only` ガード）を通す共通経路であり、`_WRITE_TOOLS` フローズンセット（`service.py`）で判定される。`git_checkout`/`git_pull`/`git_push` を他の書き込みツールと区別する追加ガード（例: 保護ブランチ・force push 制限）は実装されていない。(Explicit in code)
- `git_commit` は `dry_run=false` かつステージ済み変更が無い場合 `GitServiceError("nothing staged to commit")` を送出する。他の書き込みツールにはこの種の「実行前提条件チェック」は無い。(Explicit in code)
- `_open_repo` は `search_parent_directories=False` で `git.Repo` を開くため、`repo_path` はリポジトリのルート（`.git` を直接含むディレクトリ）を指す必要があり、サブディレクトリを渡した場合は `git.InvalidGitRepositoryError` として `GitServiceError` にラップされる。(Explicit in code)
