# MCP Server Catalog: mdq-mcp

## mdq-mcp（ポート 8013）

**目的:** Markdown ドキュメントのインデックス化とコンテキスト圧縮。
**起動モード:** persistent（HTTP）
**設定:** `config/mdq_mcp_server.toml`

**ツール:** `search_docs`, `get_chunk`, `outline`, `index_paths`, `refresh_index`, `stats`, `grep_docs`
**ツールステータス:** 7ツール全てが `production`（`search_docs`, `get_chunk`, `outline`, `index_paths`, `refresh_index`, `stats`, `grep_docs`）。**本番稼働範囲:** FTS5（BM25）ベースのMarkdown構造的検索のみをサポートする。ハイブリッド/セマンティック検索の設定・コードは削除済み（下記参照）。

**設定フィールド:** `allowed_dirs`, `db_path`, `include_globs`, `exclude_globs`, `max_snippet_chars`, `max_chunk_chars`, `max_file_chars`, `max_results_limit`, `max_chars_per_chunk`, `max_total_result_chars`, `max_outline_items`, `max_grep_matches`, `search_timeout_sec`, `enable_grep`, `max_chars_per_match`（デフォルト 500）, `context_before`（デフォルト 2）, `context_after`（デフォルト 2）, `max_outline_depth`（デフォルト 6）, `sqlite_busy_timeout`（デフォルト 5000）（`audit_log_path`は設定ファイルから2026-07-13に削除、コード側の読み込み(`MdqService.audit_log_path`)も2026-07-16に削除。`concurrency_limit`は実装で未使用のため2026-07-13に設定ファイルから削除。`max_search_results`は実装で未使用のため2026-07-16に削除。`use_embedding`・`embedding_dims`・`vector_table`・`embedding_model`はハイブリッド検索の削除に伴い2026-07-16に削除。`summary_cache_enabled`・`summary_threshold`・`summary_model`はsummaryキャッシュが実際には要約を生成しないスタブだったため2026-07-16に削除。`enable_refresh`は`refresh_index()`で一度も強制されなかったため2026-07-16に削除。`status`はどのコードからも読まれない自由記述ラベルだったため2026-07-17に削除）

**注記(2026-07-17):** `include_globs` / `exclude_globs` / `max_snippet_chars` / `max_chunk_chars` / `max_file_chars` / `search_timeout_sec` の6キーは、これまで`MdqService`に読み込まれるだけで実際には未使用（インデックス化は常に`rglob("*.md")`ハードコード、スニペットは常に`[:150]`固定、ファイル/チャンクサイズの上限enforcementなし、検索にタイムアウトなし）だったが、この日付以降は実際に enforced される:
- `include_globs` / `exclude_globs` — インデックス化走査関数が4箇所のインデックス化走査すべてでこれらのglobパターンを適用する
- `max_file_chars` — ファイルインデックス処理がファイルサイズ超過時にskip-and-warn（インデックスに一切含めない）
- `max_chunk_chars` — ファイルインデックス処理がセクションごとに`content_hash`計算前に切り詰める（ハッシュは保存内容と一致）
- `max_snippet_chars` — `search.py`のスニペット処理が`snippet=row["content"][:150]`の代わりにこの値でスニペットを切り詰める
- `search_timeout_sec` — `search_docs()`が`asyncio.wait_for(asyncio.to_thread(...), timeout=...)`で検索をラップし、タイムアウト時に`MdqConsistencyError`を送出する（バックグラウンドスレッド自体を強制中断するものではない点に注意）

**ヘルス:** `{"status":"ok"/"degraded","ready":bool,"liveness":true,"restart_recommended":false,"operator_action_required":bool,"dependencies":{...},"details":{"service":"mdq-mcp",...}}` — 基本レスポンスよりも多くのフィールドを返す（[04_mcp_06 §Health probes](04_mcp_06_06_verification-methods.md#health-probes) を参照）

**DB パス:** `/opt/llm/db/mdq.sqlite`（`config/mdq_mcp_server.toml`: `db_path`）
**ログ:** `/opt/llm/logs/mdq-mcp.log`
**Audit:** Layer1 (Agent/MCP共有): tool_exec / Layer2 (共有MCP): mcp_tool_exec / Layer3 (専用): なし — `_audit_log()` 経由で共有audit log(`/opt/llm/logs/audit.log`)にJSON-linesで記録

**注記(2026-07-20):** 専用の MDQ audit log（`mdq_audit.log` / `audit_log_path`
設定キー）の導入を検討したが、既存の共有 audit log
（`_audit_log()` / `server_key="mdq"` 経由で `/opt/llm/logs/audit.log` に記録）
で十分と判断し、見送った。具体的な実装ニーズが生じない限り、専用ログ機構は
再導入しない。

**使用場面:** Markdown ドキュメントのインデックス化とコンテキスト圧縮。本番用 RAG 検索には `rag-pipeline-mcp` を使用する。

**注記(2026-07-19):** `chunks` テーブルの `tags_json`（YAML frontmatter の `tags:` フィール
ドから抽出した JSON 配列。リスト形式・カンマ区切り文字列形式のどちらも受け付ける）と
`token_count`（ローカルなヒューリスティック `len(content) // 4` による概算値。正確なトーク
ナイザーではない）は、これまで常に `""`/`None` のハードコードされたプレースホルダーだった
が、実データを格納するようになった。`search_docs` の `tag_filter` はこの実データに対して照
合される。

### パス制御

パスを受け取る全ツールは、`allowed_dirs` を介した fail-closed な認可を強制する。

| ツール | パス入力 | 認可 |
|------|------------|---------------|
| `index_paths` | インデックス対象のディレクトリ/ファイル | `authorize_path()` — `allowed_dirs` 外の場合 `MdqAuthorizationError` を発生させる |
| `refresh_index` | 更新対象のパス | パス認可 — いずれかのパスが拒否された場合 `MdqAuthorizationError` を発生させる |
| `outline` | outline 対象のファイル | `authorize_path()` — `allowed_dirs` 外の場合 `MdqAuthorizationError` を発生させる |
| `search_docs` | インデックス済みチャンクの `source_path`（読み取り時に再チェック） | `authorize_path()` — `allowed_dirs` 外のチャンクは結果から無音で除外される（件数にも計上されない） |
| `get_chunk` | 取得対象チャンクの `source_path`（読み取り時に再チェック） | `authorize_path()` — `allowed_dirs` 外の場合 `MdqAuthorizationError` を発生させる |
| `grep_docs` | 明示指定パス、または全インデックス済み `source_path` のうち認可済みの集合（読み取り時に再チェック） | `authorize_path()` — 明示指定パスが未認可の場合は `MdqAuthorizationError` を発生させ全体を拒否；フィルタ未指定時は未認可パスを無音で除外 |

- `..` トラバーサル: `authorize_path()` 内の `Path.resolve()` によりブロックされる
- シンボリックリンクによる脱出: `authorize_path()` 内の `Path.resolve()` によりブロックされる
- 空の `allowed_dirs = []`（デフォルト）: 全てのパスを拒否する（fail-closed）

インデックス化ツールを使用する前に、`config/mdq_mcp_server.toml` で `allowed_dirs` を設定すること。

### Markdown 互換性の範囲

| Markdown 機能              | サポート | フォールバックの挙動                          |
|---------------------------|---------|---------------------------------------------|
| ATX 見出し（H1〜H6）      | Yes     | —                                           |
| フェンス付きコードブロック        | Yes     | フェンス内の `#` は見出しとして扱われない。フェンス内容自体もセクションのインデックス対象テキストから除外され、`search_docs`/`grep_docs` では検索されない（`parser.py`が `continue` でフェンス行をセクション本文へ追加する前にスキップするため） |
| YAML frontmatter          | Yes     | ファイル先頭で解析され除去される             |
| H1 より前のコンテンツ         | Yes     | `<root>` セクションとして格納される                  |
| 重複する見出し        | Yes     | ordinal により個別の chunk ID を付与              |
| Setext 見出し（===,---) | No      | プレーンテキストとして扱われる                       |
| HTML ブロック                | No      | プレーンテキストとして扱われる                       |
| MDX                       | No      | インデックス対象外（.mdx は glob で除外）         |
| GFM テーブル                | No      | 親セクション内のプレーンテキストとして格納される      |
| インライン HTML タグ          | No      | プレーンテキストとして扱われる                       |

> フロントマターの `tags`/`categories` の検索対応については本表の対象外
> （見出し/構造機能の一覧のため）。frontmatter `tags` の実データ格納と
> `search_docs` の `tag_filter` 対応については上記の注記(2026-07-19)を参照。

### 検索モード

| モード | 説明 | 設定 |
|---|---|---|
| FTS5（BM25） | FTS5 による全文検索; 唯一のサポートされる検索モード | `search_docs`の`mode`パラメータ（`bm25`のみ） |

**ハイブリッド検索は削除済み:**

ハイブリッド/セマンティック検索の設定項目（`use_embedding`, `embedding_dims`, `vector_table`, `embedding_model`）およびコード（`_search_vector()`, `_merge_hybrid()`）は2026-07-16に完全に削除された。実装されたことのない機能であり（`_search_vector()`は常に空リストを返す永続的なプレースホルダーだった）、`search_docs`の`mode`パラメータは`bm25`のみをサポートする。

**MDQ FTS5対 RAG の判断基準:**

| 使用場面 | 推奨 | 理由 |
|---|---|---|
| Markdown の構造的クエリ（見出し、セクション、outline） | MDQ FTS5 | MDQ は Markdown ドキュメント構造を理解する; FTS5 はドキュメント内のキーワードマッチングに高精度 |
| 全インデックス済みコンテンツに対する汎用的なセマンティック検索 | RAG パイプライン | RAG はより広いコーパスカバレッジと成熟した埋め込みモデル統合を持つ |
| ドキュメント間の構造比較 | MDQ FTS5 | MDQ の chunk_id には見出しの階層情報が含まれる（level, parent_path, ordinal） |

> **注記:** MDQ はハイブリッド/セマンティック検索をサポートしない（FTS5/BM25のみ）。セマンティック検索が必要な場合は RAG パイプラインを使用すること。
>
> **注記:** MDQ 対 RAG の境界の詳細な定義については、[04_mcp_05 §MDQ vs RAG Boundary](04_mcp_05_04_mdq-rag-boundary.md#mdq-vs-rag-boundary) を参照。

#### 検索結果件数と制限

`search_docs` の結果件数と上限は次の優先順位で決まる:

- **有効な SQL 上限**: `effective_limit = min(リクエストの limit, サーバー設定の
  max_results_limit)` を SQL の `LIMIT` 句に直接適用する。これにより、リクエストが大きな
  `limit` を指定しても、設定上限を超える行がデータベースから取得されることはない。
- **`max_results_limit`（リクエストパラメータ）**: 上記の有効な SQL 上限が適用された後、
  さらに表示件数を絞る二次的な表示上限として機能する（`effective_limit` 以下の値を指定した
  場合のみ意味を持つ）。
- **件数表記**: レスポンスヘッダーは、絞り込みが発生しない場合（一致件数と表示件数が一致する
  場合）にのみ「N found」のような確定件数を表示する。絞り込みが発生した場合は「matched」
  （クエリに一致した正確な件数）と「shown」（実際に表示された件数）を区別して表記し、
  取得・制限された件数をあたかも正確な合計であるかのように主張しない。

#### 監査ログ（detail フィールド）の構造化メタデータ

`call_tool()` の監査ログ（`_audit_log()` 経由、上記「Audit」参照）の `detail` フィールドは、
各ツールの出力テキストを正規表現/部分文字列で解析するのではなく、`MdqService` の各メソッド
が返す構造化メタデータ（`(text, metadata)` のタプル）から直接構築される。出力テキストの文言
が変わっても監査値は変化しない。

- `search_docs`: `result_count=`, `shown_count=`, （絞り込み時のみ）`truncated=true`
- `grep_docs`: `match_count=`, （絞り込み時のみ）`truncated=true`
- `index_paths`: `indexed_count=`, `skipped_count=`, `failed_count=`
- `refresh_index`: `indexed_count=`, `skipped_count=`, `deleted_count=`, `failed_count=`
- `get_chunk`: `truncated=true`（出力テキスト中の `"[Truncated"` 部分文字列判定のみ、変更なし）

ツールレベルの例外（`MdqValidationError` / `MdqAuthorizationError` / `MdqNotFoundError` /
`MdqIndexNotReadyError`）およびFastAPI例外ハンドラ経由のエラー（`MdqDatabaseError` /
`MdqConsistencyError` / `MdqServiceError`）は、エラー種別を `detail` 文字列に埋め込まず
`_audit_log()` の `error_type=` パラメータとして渡す。
