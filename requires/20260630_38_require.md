# [Low] Audit and classify `fallback` terminology

## Priority
Low

## Background
The word `fallback` is used for different concepts:
- backward compatibility fallback that should be removed
- degraded runtime behavior that should remain
- estimation paths that are acceptable

RAG command code still uses valid degraded behavior such as FTS-only fallback and refiner fallback, while routing fallback is a backward compatibility path that should be removed. 【10-99a595】【1-acccf4】

## Goal
Classify fallback references and rename or remove them where appropriate.

## Scope
Affected files:
- `docs/`
- `agent/`
- `shared/`
- `mcp/`
- `tests/`

## Implementation Tasks
- Audit all `fallback` references.
- Remove or rename compatibility fallback references:
  - routing fallback
  - GitHub prefix fallback
  - config `tool_names` fallback
  - backward-compatible command alias fallback
- Keep valid degraded behavior, but rename where clearer:
  - `degraded mode`
  - `estimation path`
  - `diagnostic path`
- Ensure stdio is not described as fallback transport.
- Document which fallback-like behaviors are intentionally retained.

## Acceptance Criteria
- Backward compatibility fallback terminology is removed from active specs.
- Valid degraded behavior remains documented.
- Future readers can distinguish degraded behavior from backward compatibility.

## Validation
- Run: `rg -n "fallback" docs agent shared mcp tests`
