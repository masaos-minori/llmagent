# Title
Unify config loading, reload behavior, and restart-required rules

## Summary
Create a single authoritative behavior model for configuration loading and runtime reload behavior.

## Background
Configuration behavior is currently documented across multiple sections with overlapping but differently phrased explanations.
Some settings are hot-reloadable, some require restart, and some are partially reloadable.

## Problem
It is too easy to misunderstand whether a configuration change:
- is loaded at startup only
- is applied by `/reload`
- requires a process restart
- is partially applied

## Required Changes
- Define one authoritative configuration matrix for all config files and fields.
- Classify each setting as one of:
  - startup-only
  - hot-reloadable
  - restart-required
  - partially reloadable
- Make `/reload` output explicitly report:
  - applied
  - deferred
  - restart required
  - ignored / unchanged
- Clarify the ownership and reload semantics of `common.toml`.

## Acceptance Criteria
- Operators can determine runtime behavior for any config field from one place.
- `/reload` output clearly communicates what actually changed at runtime.
- Documentation and implementation agree on reload semantics.
- No ambiguous overlap remains between config-related docs.

## Out of Scope
- Redesigning the entire config file layout.
- Migrating all config files into one file.

## Notes
Optimize for operational clarity and machine-aided maintenance.
