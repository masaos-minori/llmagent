# [Medium] Connect RAG consistency checks to DESIGN-3 responsibility boundaries

## Goal

Clarify that RAG consistency checks validate synchronization between canonical chunk data and derived indexes.

## Background

The operations docs describe `check_rag_consistency()` and `/db consistency`, including `fts_gap`, `fts_orphan_count`, and `orphan_vec_count`. These checks should be explicitly connected to DESIGN-3: `chunks` is canonical, while `chunks_fts` and `chunks_vec` are derived indexes. 【5-63b173】【1-54c80f】

## Scope

Documentation only, unless implementation contradicts the documented model.

- Update `03_rag_05_configuration_and_operations.md`.

- Document that consistency checks compare:
  - canonical `chunks`
  - derived `chunks_fts`
  - derived `chunks_vec`
- Define:
  - `fts_gap`: chunks missing from `chunks_fts`
  - `fts_orphan_count`: extra FTS entries without matching chunks
  - `orphan_vec_count`: vector rows without matching chunks
- Explain that repair operations rebuild or refresh derived indexes from canonical data.
- Ensure the repair decision tree aligns with canonical/derived responsibilities.
