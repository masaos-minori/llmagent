## Issue 3: Filter ephemeral markers before persisting compressed history to DB

## Title
Filter `_memory_injected`/`_ephemeral` system messages before persisting compressed history snapshot

## Summary
- Memory injection and MDQ/RAG routing hint messages are added to `ctx.conv.history` with custom markers (`_memory_injected=True`, `_ephemeral=True`).
- When history compression runs, `replace_messages()` persists the full history snapshot to `session.sqlite` without filtering these markers.
- After session reload, these ephemeral messages reappear in the LLM context, polluting future turns with stale memory fragments.

## Background
`_handle_memory_injection()` appends `{"role": "system", ..., "_memory_injected": True}` to history. `_classify_and_inject_mode()` appends `{"role": "system", ..., "_ephemeral": True}`. These are designed to be transient per-turn. `_sync_system_prompt()` removes `_ephemeral` markers at each turn start but does NOT remove `_memory_injected` markers. When `replace_messages()` is called after compression, both types are saved to `messages` table and retrieved by `fetch_messages()` on reload.

## Problem
`session_message_repo.py:replace_messages()` does not filter messages with `_memory_injected=True` or `_ephemeral=True`. After `/session load <id>`, `restore_session()` calls `fetch_messages()` which returns all messages including these ephemeral system rows. The `_memory_injected` ones are not removed by `_sync_system_prompt()`, so they persist indefinitely in the reloaded context.

## Required Changes
- `scripts/agent/orchestrator.py` — `_handle_history_compression()`: Before calling `ctx.session.replace_messages(ctx.conv.history)`, filter out messages where `m.get("_memory_injected")` or `m.get("_ephemeral")` is truthy.
- `scripts/agent/orchestrator.py` — `_sync_system_prompt()`: Add removal of messages where `m.get("_memory_injected")` is truthy, mirroring the existing `_ephemeral` removal.
- Do NOT change `undo_service.py` — its `_memory_injected` handling is already correct.

## Acceptance Criteria
- After history compression, `session_diagnostics` (via query on `messages` table) contains no rows with `_memory_injected` or `_ephemeral` content.
- After `/session load <id>` on a compressed session, the LLM context does not contain memory injection messages from the previous session's turns.
- `/undo` behavior is unaffected.
- A test covering compress → reload → verify no ephemeral messages in history is added.

## Out of Scope
- Changes to `_memory_injected` / `_ephemeral` marker naming conventions.
- Changes to how `undo_service.py` handles these markers.
- Changes to `DiagnosticStore` or `session_diagnostics` table.

## AI Implementation Instruction
**What to change:**
1. File: `scripts/agent/orchestrator.py`
   - Method: `_handle_history_compression()`: The line `ctx.session.replace_messages(ctx.conv.history)` should receive a filtered copy: `replace_messages([m for m in ctx.conv.history if not m.get("_memory_injected") and not m.get("_ephemeral")])`. Note: `ctx.conv.history` itself should retain these messages for the current turn; only the DB snapshot should be filtered.
   - Method: `_sync_system_prompt()`: The existing list comprehension `[m for m in ctx.conv.history if not m.get("_ephemeral")]` should be extended to also filter `_memory_injected`: `[m for m in ctx.conv.history if not m.get("_ephemeral") and not m.get("_memory_injected")]`.

2. Do NOT modify `session_message_repo.replace_messages()` itself — keep the filter at the call site in `orchestrator.py` so the marker-filtering semantics are visible in orchestration context.

**What NOT to change:**
- Do not change `undo_service.py` — it correctly handles `_memory_injected`.
- Do not change how `_handle_memory_injection()` builds the marker.
- Do not change `fetch_messages()` — it correctly returns all persisted rows.

**Boundary awareness:**
- The in-memory `ctx.conv.history` should still contain `_memory_injected` messages during the current turn (LLM needs them for the current request). Only the DB snapshot must exclude them.
- `_sync_system_prompt()` runs at the start of each turn, before `_handle_memory_injection()`. Adding `_memory_injected` filtering there handles the reload scenario (where old markers are loaded from DB).

**Docs/code consistency:**
- Update `docs/05_agent_04_state-and-persistence.md` §Message save rules to document that `replace_messages()` receives a filtered list excluding `_memory_injected`/`_ephemeral` messages.

**Needs confirmation:** Whether the filtering should also apply to `save()` calls in `_append_user_message()` — currently those do not save system messages directly, so this should not be needed. Confirm before adding further filter logic.
