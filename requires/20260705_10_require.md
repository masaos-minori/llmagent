# Strengthen consistency between history compression, database persistence, and undo

## Priority

Medium

## Type

Design / Implementation / Test

## Background

Conversation history compression replaces old messages with a compressed summary. The compressed snapshot is persisted back to `session.sqlite` so that `/session load` restores the same context state.

Tool outputs are stored separately in `tool_results`, and undone tool results are retained for audit using an `undone` marker.

## Problem

After compression, the database no longer contains the original message rows as normal conversation history. `/undo` may therefore operate on compressed rows rather than the original turn boundaries.

This can create inconsistencies between:

- `ctx.conv.history`
- `messages`
- `tool_results`
- `/undo`
- `/session load`
- audit expectations

## Goal

Make compression, persistence, and undo behavior explicit and internally consistent.

## Scope

Affected areas:

- `HistoryManager`
- `AgentSession.replace_messages()`
- `/compact`
- `/undo`
- `messages` table persistence
- `tool_results` retention
- Session reload tests

## Implementation Tasks

- Verify that automatic compression persists the compressed snapshot.
- Verify that `/compact` persists the compressed snapshot.
- Verify that fallback truncation persists the snapshot.
- Add tests for `/session load` after compression.
- Add tests for `/undo` after compression.
- Add tests for `tool_results.undone` after undo on compressed history.
- Consider storing compression metadata:
  - `compression_id`
  - `source_message_ids`
  - `compressed_range_start`
  - `compressed_range_end`
- Consider retaining original messages in a separate audit-only archive.

## Acceptance Criteria

- Reloading a compressed session restores the compressed history exactly.
- `/undo` behavior after compression is deterministic and tested.
- `tool_results` rows remain retained and marked correctly after undo.
- Fallback truncation does not leave DB state inconsistent.
- Compression metadata is available if implemented.
- No partial or stale original rows remain in normal LLM-visible history.

## Validation Plan

Add tests for:

- Automatic compression followed by reload.
- Manual `/compact` followed by reload.
- Fallback truncation followed by reload.
- `/undo` after automatic compression.
- `/undo` after manual compression.
- Tool result retention after compressed undo.
- Message count and role sequence after `replace_messages()`.

## Risks

Adding compression metadata may require schema changes. Prefer additive schema changes and backward-compatible migrations.

## Out of Scope

- Removing compression.
- Changing summarization prompt behavior.
- Documentation-only updates.
