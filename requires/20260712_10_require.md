# [Agent][Commands] `/skill` コマンドによるSkills読込機構の追加

## 背景

定型手順（コードレビュー、テスト生成、移行ガイドなど）を実行するたびに、会話内で毎回同じ指示を書き起こしている。これは指示の反復とトークン消費のコストを生んでいる。

`skills/*/SKILL.md` は既にドキュメントとして存在し、`routing.md` から人間・エージェントが参照する運用になっている。しかし `scripts/` 配下のPythonランタイムには `SKILL.md` を読み込む実装が存在しない（`grep -r "SKILL.md" scripts --include="*.py"` はヒットなし）。手順テンプレートを呼び出すには、毎回ファイル内容を会話に貼り付けるほかない。

## 目的

`skills/<name>/SKILL.md` の内容を、スラッシュコマンド `/skill <name> [args]` から一時的なsystemメッセージとしてLLMに渡せるようにする。

探索対象は `repo_root/skills/` のみとする。個人スコープ用に別ディレクトリを設ける要望はヒアリングの結果なし（個人スコープも同じ `repo_root/skills/` を指す）。したがって `config/agent.toml` への検索パス追加や複数パスの優先順位付けは本タスクのスコープ外とする。

## 現状（調査結果）

- スラッシュコマンドのディスパッチ本体は `scripts/agent/commands/registry.py` の `CommandRegistry.dispatch()`（118〜148行）。組み込みコマンドは prefix/完全一致でハンドラーへ振り分けられ、一致しない場合は `_dispatch_plugin()`（150〜165行）が `shared.plugin_registry.iter_commands()` に委譲する。
- コマンド一覧の単一情報源は `scripts/agent/commands/command_defs_list.py` の `_COMMANDS: list[CommandDef]`。登録はデコレータではなく、この静的リストへの追加によって行う。
- `CommandDef` の定義は `scripts/agent/commands/command_defs.py`（30〜42行）。フィールドは `name, prefix, is_async, handler, help, subcommands`。
- ハンドラー本体は `_ContextMixin` 系クラスのメソッドとして各 `cmd_*.py` に実装され、`registry.py` が多重継承で束ねている（58〜72行）。
- **一時的なsystemメッセージ注入の前例が既にある：**
  - `scripts/agent/mode_classification.py`（21〜39行）`classify_and_inject_mode()` — `ctx.conv.history.append({"role": "system", "content": hint, "_ephemeral": True})`
  - `scripts/agent/orchestrator.py`（343〜360行）`_handle_memory_injection()` — `_memory_injected: True` フラグ付きで同様に注入
  - 除去ロジックは `scripts/agent/orchestrator.py` の `_sync_system_prompt()`（500〜519行）。毎ターン開始前に `_ephemeral` / `_memory_injected` フラグ付きメッセージを取り除いてから `history[0]` を再構築する
- 設定ファイルの実体は `config/agent.json` ではなく **`config/agent.toml`**（ローダー: `scripts/shared/config_loader.py` の `ConfigLoader`、既定探索先は `repo_root/config/`）。対応する dataclass は `scripts/agent/config_dataclasses.py` の `AgentConfig`（413行〜）。

以上より、元の提案にあった「`agent_repl.py` への `extra_system_prompt` 経路新設」は不要であり、既存の ephemeral message パターン（`_ephemeral: True`）を再利用すれば足りる。

## 提案設計

1. `/skill` を引数なしで実行した場合、`repo_root/skills/` 配下の**ディレクトリのみ**を対象に、その名前一覧を返す。`skills/DESIGN.md` のようなファイル単体のエントリは一覧・`<name>` 解決の両方から除外する（ディレクトリでない場合は無視）。一覧表示にdescriptionは含めず、ディレクトリ名のみとする。LLM呼び出しは行わない。
2. `/skill <name> [args]` 実行時、`repo_root/skills/<name>/SKILL.md` を読み込む。`<name>` が `repo_root/skills/` 直下のディレクトリでない場合（例: `DESIGN`）は、5. のエラー処理と同様に扱う。
3. 読み込む前に、`ctx.conv.history` 内に既存の `/skill` 由来メッセージ（後述の `_skill_ephemeral: True` フラグ付き）があれば全て削除してから、新しい内容を追記する（＝前回分を除去して置き換え）。他の `_ephemeral: True`（`mode_classification.py` 由来）や `_memory_injected: True` のメッセージには影響しない。
4. 追記するメッセージは `{"role": "system", "content": <SKILL.md本文>, "_ephemeral": True, "_skill_ephemeral": True}` とする（`_ephemeral: True` を付けることで既存の `_sync_system_prompt()` の除去対象にも入る。`_skill_ephemeral: True` は `/skill` 由来メッセージを識別するための追加フラグ）。
5. `args` が渡された場合は、SKILL.md本文の末尾に「呼び出し引数: {args}」などの形で付加するのみとする。`/skill` 自体はLLM応答をその場ではトリガーしない。ユーザーが続けて発言した際に、注入済みのsystemメッセージを踏まえた応答が返る。
6. 指定した `<name>` が存在しない、またはディレクトリでない場合はエラーメッセージを返し、LLM呼び出しは行わない。
7. `<name>` ディレクトリは存在するが `SKILL.md` が無いケースは、現時点の `skills/` 配下では発生しない（全ディレクトリに `SKILL.md` が存在することを確認済み）ためガードしない。将来ディレクトリのみ追加された場合は例外が自然に発生する想定でよく、本タスクのスコープ外とする。

## 効果

- 定型手順の再利用により、会話内での指示の反復とトークン消費を削減する。
- 手順の標準化により、実行結果の品質のばらつきを低減し再現性を向上させる。
- `/skill` 単体実行での一覧表示により、利用可能な手順テンプレートを毎回 `skills/` を直接見に行かずに確認できる。

個人スコープ／プロジェクトスコープの分離（別ディレクトリの追加）は今回の要望確認の結果、対象外と判断した（「現状」節を参照）。将来的に個人用skillディレクトリが必要になった時点で別途要望を起票する。

## 受け入れ基準

- [ ] `/skill`（引数なし）実行で `repo_root/skills/` 配下のディレクトリのみのskill名一覧が表示され、`DESIGN.md` は含まれず、LLM呼び出しは発生しない。
- [ ] `/skill DESIGN` のようにファイル単体のエントリ名を指定した場合、`<name>` 未存在時と同じエラー扱いになる。
- [ ] `/skill <name>` 実行後、次のLLM呼び出しで `skills/<name>/SKILL.md` の内容がsystemメッセージとして送信される。
- [ ] 同一会話内で `/skill` を別のskill名で再度実行すると、前回分のメッセージは履歴から取り除かれ、新しい内容のみが送信される（積み重ならない）。
- [ ] 次のターン以降、注入した内容が `_sync_system_prompt()` によって自動的に除去される（履歴に永続しない）。
- [ ] 存在しない `<name>` を指定した場合、LLM呼び出しを行わずにエラーメッセージが返る。
- [ ] 既存のスラッシュコマンド・ephemeral注入（`/system`、`mode_classification`、`memory injection`）の挙動に regression がない。

## リスク

| リスク | 可能性 | 対策 |
|---|---|---|
| `_ephemeral` フラグ付きメッセージが複数同時に存在する場合の順序・競合 | 中 | `_sync_system_prompt()` の既存除去ロジックの挙動を確認し、`/skill` 由来のメッセージも同一ルールで除去されることをテストで確認する |
| `_skill_ephemeral` の置き換え処理が `mode_classification` / `memory injection` 由来の `_ephemeral` メッセージまで誤って削除する | 中 | 削除条件を `_skill_ephemeral: True` の有無のみで判定し、他のフラグのメッセージには触れないことをテストで確認する |
| SKILL.md本文が長大な場合のトークン消費増大 | 低 | 当面は対象外（運用でSKILL.mdのサイズを管理する） |
| `<name>` ディレクトリはあるが `SKILL.md` が無い場合に未処理の例外（`FileNotFoundError`等）がユーザーにそのまま表示される | 低 | 現状のskills/配下では発生しないため意図的にガードしない（「提案設計」7.を参照）。発生した場合はスタックトレースでなくエラーとして扱われるかは実装時のコマンドディスパッチ層の共通例外処理に委ねる |


## AI実装者への注記

- `extra_system_prompt` のような新規経路は追加しないこと。既存の `_ephemeral: True` パターン（`mode_classification.py`、`orchestrator.py`）を再利用する。
- `config/agent.json` は実在しない。設定ファイルは `config/agent.toml` であり、`ConfigLoader`（`scripts/shared/config_loader.py`）経由で読み込む。
- `config/agent.toml` への変更・複数パス対応は行わないこと。探索対象は `repo_root/skills/` 固定でよい。
- 置き換え処理は `_skill_ephemeral: True` という専用フラグで判定すること。既存の `_ephemeral: True` だけで判定すると `mode_classification.py` / `memory injection` 由来のメッセージまで削除してしまう。
