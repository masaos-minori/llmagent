# Title
Unify Event Bus host binding configuration ownership

## Priority
Medium

## Labels
- bug
- eventbus
- configuration
- deployment
- security

## Dependencies
- Issue 9 recommended

## Summary
Clarify whether the Event Bus bind host is controlled by TOML configuration or by the uvicorn start command.

## Background
Operations docs show `host = "127.0.0.1"` in `config/eventbus.toml`, but `EventBusConfig` does not include a `host` field. Startup examples also pass `--host 127.0.0.1` directly to uvicorn.

## Problem
Operators cannot tell whether host binding is configured in TOML or only at process startup. This is especially risky because the Event Bus has no authentication layer.

## Required Changes
- Decide one canonical host configuration model:
  - TOML config field `host`, or
  - uvicorn CLI / service manager only.
- If TOML owns host:
  - add `host` to `EventBusConfig`
  - update start command to use config-derived host
- If uvicorn owns host:
  - remove `host` from TOML examples
  - document host as deployment/runtime process configuration
- Log the effective bind address at startup.
- Add config validation tests.

## Acceptance Criteria
- `host` ownership is unambiguous.
- `EventBusConfig` and configuration docs agree.
- Startup logs show effective host and port.
- Operators can audit whether the service is bound to loopback.

## Out of Scope
- Adding authentication.
- Replacing uvicorn deployment.
