## Title
[High] Restrict DLQ requeue to events currently in the DLQ

## Priority
High

## Labels
bug, eventbus, dlq, api, data-integrity

## Dependencies
None

## Background
`requeue_event()` currently updates rows by `event_id` only. This allows non-DLQ events to be treated as successfully requeued and increments `dlq_requeue_count`.

`POST /dlq/{event_id}/requeue` should only succeed for events where `dlq_at IS NOT NULL`.

## Scope
Update:
- `scripts/eventbus/db.py`
- `scripts/eventbus/app.py`
- DLQ tests under `tests/test_eventbus_dlq*.py`
- Concurrent DLQ requeue tests

## Required Changes
- Update `requeue_event()` to require:
  - `event_id = ?`
  - `dlq_at IS NOT NULL`
- Return failure when the event exists but is not currently in the DLQ.
- Decide and implement response behavior for non-DLQ events:
  - preferred: HTTP 409 Conflict
  - acceptable: HTTP 404 Not Found
- Keep existing behavior for unknown events as HTTP 404.
- Ensure `dlq_requeue_count` increments only on successful DLQ requeue.
- Preserve the behavior that `delivery_failure_count` is not reset on requeue.

## Tests to Add or Update
- Add `test_requeue_non_dlq_event_fails`.
- Add `test_requeue_unknown_event_returns_404`.
- Update concurrent requeue test:
  - only one concurrent request should successfully requeue the DLQ event
  - later requests should fail because the event is no longer in DLQ
- Keep tests verifying:
  - successful requeue clears `dlq_at`
  - successful requeue increments `dlq_requeue_count`
  - `delivery_failure_count` is unchanged
  - `dlq_imminent` is returned when applicable

## Acceptance Criteria
- Non-DLQ events cannot be requeued.
- DLQ events can be requeued exactly once per DLQ state.
- Concurrent requeue does not create multiple successful requeue operations for the same DLQ state.
- DLQ API documentation matches the new behavior.
