# Title
Add operator-facing consistency check and rebuild flows for the memory layer

## Summary
Expose memory-layer consistency checking and index rebuild operations so operators can detect and repair JSONL-to-SQLite divergence.

## Background
The memory layer uses JSONL as the append-only source of truth and SQLite as the searchable index.
Consistency checking exists, but operational access paths are weak or unclear.

## Problem
When JSONL and SQLite drift or indexing becomes inconsistent, operators need a supported way to diagnose and repair the issue.

## Required Changes
- Add an operator-facing consistency-check flow, such as `/memory check-consistency`.
- Add an operator-facing rebuild or reindex flow from JSONL to SQLite.
- Expose mismatch signals through observability or diagnostics.

## Acceptance Criteria
- Operators can check JSONL / SQLite consistency without custom scripts.
- Operators can trigger a supported rebuild or reindex flow.
- Consistency mismatch is visible through CLI or observability tooling.

## Out of Scope
- Rewriting extraction heuristics
- Changing the core memory schema

## AI Implementation Instruction
Implement this as an operability and recovery-path task.

Strict rules:
- Preserve JSONL as the canonical source of truth unless another issue explicitly changes that design.
- Do not add silent self-healing without visibility.
- Prefer explicit operator-invoked checks and rebuilds.
- If rebuild semantics are uncertain, mark them as: Needs confirmation

Deliverables:
1. Operator-facing consistency check
2. Operator-facing rebuild/reindex path
3. Visible diagnostics for consistency problems
