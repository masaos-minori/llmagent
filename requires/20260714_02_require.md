# [High] Remove obsolete workflow_require_approval configuration references

## Labels
priority/high, area/workflow, area/approval, area/config, area/docs

## Summary
Remove or rewrite obsolete documentation that describes `workflow_require_approval` as a valid `AgentConfig` field.

The confirmed specification is:

- `workflow_require_approval` is not a valid current config key.
- It must not be documented as a way to enable workflow-level approval.
- `/approve` and `/reject` remain workflow approval resolution commands, but their activation conditions must be described based on the current workflow implementation.

## Background
The canonical approval documentation still describes workflow-level approval as controlled by `AgentConfig.workflow_require_approval`.

However, the current configuration documentation states that `workflow_require_approval` has been removed and is rejected as a forbidden key.

## Required Changes

- Remove all text that says `workflow_require_approval` is a valid config key.
- Remove all text that says workflow-level approval is enabled by `workflow_require_approval=true`.
- Keep `/approve` and `/reject` documentation, but describe them as commands that resolve existing workflow approval records.
- Re-document the actual condition under which a workflow approval gate is created.
- If current implementation does not create workflow approval gates, state that workflow-level approval records are currently not generated by config.
- Ensure forbidden-key documentation remains consistent.

## Acceptance Criteria
- `workflow_require_approval` is documented only as a removed/forbidden key, if mentioned at all.
- No current behavior depends on `workflow_require_approval`.
- `/approve` and `/reject` documentation remains accurate.
- Workflow-level approval behavior is described from current implementation, not old config.
- Configuration docs and approval docs no longer contradict each other.

- Do not change tool-level approval behavior.
- Do not remove `/approve` or `/reject`.
