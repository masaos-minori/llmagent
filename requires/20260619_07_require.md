# Title
Improve tool-call scheduling to avoid unnecessary round-wide serialization

## Summary
Refine tool execution scheduling so that non-conflicting read-only tool calls are not unnecessarily serialized when a side-effect tool is present in the same round.

## Background
Current behavior serializes the entire tool-call round if any side-effect tool is present.

## Problem
This is safe but overly coarse.
Independent read-only calls may be delayed even when they do not logically depend on writes or destructive actions.

## Required Changes
- Introduce a more dependency-aware scheduling policy.
- Separate independent read-only tool calls from side-effect tool calls where safe.
- Use or extend existing DAG-oriented execution support if available.
- Make scheduling decisions explainable via logs or debug output.

## Acceptance Criteria
- Independent read-only tools are not always forced into sequential execution.
- Side-effect tools still execute under safe ordering constraints.
- Execution ordering is traceable for debugging and review.
- Existing safety guarantees are not weakened.

## Out of Scope
- Full distributed workflow orchestration.
- Cross-turn planning beyond the current tool loop.

## Notes
Prefer safety-preserving granularity over global serialization.
