# Title
Make CI/CD and GitHub write-operation defaults safer for production

## Summary
Harden default settings for workflow triggering and GitHub writes so production deployments start from a safer baseline.

## Background
Current defaults allow too much by default:
- CI/CD workflow allowlists are permissive when empty
- GitHub write defaults allow force push and do not require PR review by default

## Problem
These defaults increase the risk of unintended or overly powerful write actions in production automation.

## Required Changes
- Change production behavior so an empty `workflow_allowlist` does not imply allow-all.
- Change GitHub defaults to:
  - `allow_force_push = false`
  - `require_pr_review = true`
  or a clearly safer equivalent
- Ensure startup audits surface these settings clearly.
- Preserve explicit opt-in for more permissive behavior where needed.

## Acceptance Criteria
- Production workflow triggering is not implicitly allow-all when unconfigured.
- GitHub write defaults are safer than the current baseline.
- Operators can see these settings clearly at startup or in config status.

## Out of Scope
- Replacing GitHub or CI/CD server architecture
- Building a full policy engine

## AI Implementation Instruction
Implement this as a default-hardening and startup-safety task.

Strict rules:
- Do not remove explicit override capability.
- Make the safer behavior the default, not the optional path.
- Keep configuration explicit and operator-visible.
- If the exact production-only enforcement point is unclear, mark it as: Needs confirmation

Deliverables:
1. Safer production defaults
2. Startup/config visibility for those defaults
3. Documentation updated to reflect the hardened baseline
