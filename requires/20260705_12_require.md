# Harden malformed /v1/tools response handling

## Priority

High

## Type

Implementation / Validation / Startup Safety

## Background

Startup validation probes each MCP server's `/v1/tools` endpoint to compare live tool names with configured and registered tool names.

The response is expected to contain a JSON object with a `tools` list, where each tool entry contains a non-empty `name`.

## Problem

Malformed `/v1/tools` responses can currently cause unclear startup failures or be misclassified as reachability problems.

Malformed examples include:

- Response is not valid JSON.
- Response JSON is not an object.
- `tools` is missing.
- `tools` is not a list.
- A tool entry is not an object.
- A tool entry has no `name`.
- A tool name is empty or not a string.

## Goal

Classify malformed `/v1/tools` responses explicitly and handle them consistently in strict and non-strict modes.

## Scope

Affected areas:

- `agent/repl_health.py`
- `/v1/tools` startup validation
- `HealthCheckResult`
- Startup validation tests

## Implementation Tasks

- Add schema validation for `/v1/tools` responses.
- Add a structured error type for malformed tool list responses.
- Include server key and URL in validation warnings or errors.
- In strict mode, treat malformed `/v1/tools` as fatal.
- In non-strict mode, return warnings through `HealthCheckResult`.
- Avoid raw `KeyError`, `TypeError`, or JSON parse exceptions escaping without context.

## Acceptance Criteria

- Invalid JSON response is reported as malformed tool list.
- Missing `tools` field is reported as malformed tool list.
- Non-list `tools` field is reported as malformed tool list.
- Tool entry without valid `name` is reported as malformed tool list.
- Strict mode raises `RuntimeError`.
- Non-strict mode logs warning and continues.
- The error message identifies the affected server.

## Validation Plan

Add tests for:

- Valid `/v1/tools` response.
- Invalid JSON response.
- Missing `tools`.
- `tools` is not a list.
- Tool entry is not a dict.
- Tool entry has no `name`.
- Tool entry name is empty.
- Strict and non-strict behavior.

## Risks

Existing invalid server responses may now fail startup in strict mode. This is desirable because tool discovery must be reliable.

## Out of Scope

- Changing the `/v1/tools` public protocol.
- Changing routing behavior.
- Documentation-only updates.
