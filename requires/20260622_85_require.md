# Title
Make RAG quality degradation visible instead of silently degraded

## Summary
Improve observability so operators can distinguish “no relevant results” from “retrieval or indexing degraded because of internal failure.”

## Background
Several failure paths currently degrade behavior but allow execution to continue:
- FTS errors may become empty result sets
- embedding failures may skip chunks
- consistency problems may be logged only as warnings

## Problem
These behaviors improve availability, but they also make it easy to miss that retrieval quality has degraded.

## Required Changes
- Add operator-visible diagnostics for degraded retrieval/indexing conditions.
- Distinguish between:
  - legitimate zero-result searches
  - search-path failures
  - chunk-ingestion skips caused by embedding failure
  - consistency mismatch warnings
- Expose these signals through runtime stats, status output, or equivalent diagnostics.
- Preserve graceful degradation where appropriate, but make it visible.

## Acceptance Criteria
- Operators can tell whether a zero-result outcome is normal or degraded.
- Retrieval/indexing degradation is visible without deep trace logs only.
- Existing graceful-degradation paths remain documented and observable.

## Out of Scope
- Replacing graceful degradation with hard failure everywhere
- Redesigning all error-policy decisions

## AI Implementation Instruction
Implement this as an observability and diagnosability task.

Strict rules:
- Do not remove graceful degradation blindly.
- Focus on visibility and diagnosability, not on changing every fallback.
- Preserve user-facing availability where possible.
- Add explicit indicators for degraded states.
- If a degraded state cannot yet be classified reliably, mark it as: Needs confirmation

Deliverables:
1. Operator-visible degraded-state indicators
2. Clear distinction between normal empty results and degraded retrieval
3. Updated docs for visible failure/degradation states
