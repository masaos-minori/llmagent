# [Low] Clarify local/development no-auth behavior for MCP auth_token=""

## Labels
priority/low, area/security, area/mcp, documentation

## Summary
Clarify that empty MCP `auth_token` is local/development compatibility behavior.

Production already requires non-empty `auth_token` for HTTP MCP servers. This task documents and optionally improves visibility of no-auth local mode.

## Background
`auth_token=""` means no Bearer token authentication.

In production, HTTP MCP servers without `auth_token` must fail startup. In local/development, this remains allowed but should be explicit and visible.

## Required Changes

### Documentation
- Update security documentation to clearly state:
  - `auth_token=""` is local/development compatibility behavior.
  - production requires non-empty `auth_token`.
  - empty token must not be used in production.
- Ensure `04_mcp_05_security_and_safety_model.md` and `04_mcp_06_configuration_and_operations.md` are consistent.

### Optional Runtime Visibility
Consider adding no-auth visibility to `/mcp status`.

Example display field:

- `AUTH: none`
- `AUTH: bearer`

If implemented, add tests for the status display.

### Future Consideration
Document a possible migration from implicit empty token to explicit configuration such as:

- `auth_mode="none"`
- `allow_no_auth=true`

This migration is not required in this task.

## Acceptance Criteria
- Documentation clearly distinguishes local no-auth from production auth requirement.
- Production behavior remains unchanged.
- Local/development behavior remains unchanged.
- If `/mcp status` display is changed, tests cover it.

## Target Files
- `agent/repl_health.py`
- `agent/services/mcp_status.py`
- `shared/mcp_config.py`
- `04_mcp_05_security_and_safety_model.md`
- `04_mcp_06_configuration_and_operations.md`

## Non-Goals
- Do not remove `auth_token=""` local support.
- Do not introduce a new auth mode unless separately approved.
- Do not change Bearer token validation semantics.
