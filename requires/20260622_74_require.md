# Title
Harden plugin loading and plugin tool shadowing behavior

## Summary
Make plugin loading fail fast in controlled environments and prevent accidental MCP tool shadowing by plugin tools.

## Background
Plugins currently load in fail-open mode by default.
Import failures can be logged and skipped.
Plugin tools are resolved before MCP tools and may shadow MCP behavior depending on configuration.

## Problem
This weakens production reproducibility and creates a hidden control-plane split between plugin tools and MCP tools.

## Required Changes
- Make strict plugin loading the default or strongly enforced mode in production/CI-like environments.
- Reject MCP-name conflicts by default.
- Allow plugin override of MCP tools only through explicit opt-in.
- Expose plugin load results (loaded / failed / skipped / overridden) in an operator-visible status path.

## Acceptance Criteria
- Plugin import failure is not silently ignored in strict environments.
- MCP tool shadowing cannot happen accidentally.
- Operators can inspect plugin load status without relying only on raw logs.

## Out of Scope
- Removing the plugin architecture
- Replacing the decorator-based extension model

## AI Implementation Instruction
Implement this as a safety and operability hardening task.

Strict rules:
- Do not weaken MCP tool visibility or safety behavior.
- Preserve plugin support, but make override and failure behavior explicit.
- Favor fail-fast in controlled environments over silent partial startup.
- If compatibility trade-offs are unclear, mark them as: Needs confirmation

Deliverables:
1. Safer plugin-loading behavior
2. Explicitly controlled override behavior
3. Operator-visible plugin load status
