# Title
Unify the responsibility boundary between AgentSession and RAG maintenance

## Summary
Make `AgentSession` responsible only for session/message/note persistence in `session.sqlite` and ensure RAG maintenance behavior is handled through dedicated RAG service boundaries.

## Background
The documentation set previously contained mixed descriptions about whether `AgentSession` accessed RAG-layer data directly.
The latest API reference now describes `AgentSession` as session persistence only, with `/db`-style RAG operations delegated to `rag-pipeline-mcp`.

## Problem
If the boundary is not fully enforced in code, docs, and command behavior, future maintenance will reintroduce coupling between the session layer and the RAG layer.

## Required Changes
- Audit `agent/session.py` and confirm it does not directly depend on RAG schema details.
- Route all RAG maintenance behavior through dedicated maintenance services or MCP calls.
- Update command/help/documentation paths so they do not imply that session persistence owns RAG maintenance responsibilities.
- Make the boundary explicit in data-layer and API docs.

## Acceptance Criteria
- `AgentSession` owns only session/message/note persistence.
- RAG maintenance is handled outside `AgentSession`.
- Documentation, command behavior, and code all describe the same boundary.

## Out of Scope
- Redesigning the RAG pipeline itself
- Removing `/db` maintenance operations entirely

## AI Implementation Instruction
Implement this as a boundary-hardening task.

Strict rules:
- Do not add new direct RAG-table access to `AgentSession`.
- Do not move session persistence responsibilities out of `AgentSession`.
- If any existing behavior still mixes session persistence and RAG maintenance, move the RAG side outward, not inward.
- Update docs and code together.
- If you find ambiguous behavior that cannot be safely resolved from the docs alone, mark it as: Needs confirmation

Deliverables:
1. Refactored boundary in code
2. Updated help/docs describing the final boundary
3. A short note listing any remaining ambiguous cases
