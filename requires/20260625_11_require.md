## [High][config][security] Make production config templates safe-by-default instead of relying on post-load audit

### Summary
The documented production safety posture exists, but some server-level defaults still start from dangerous values and rely on later validation.

### Problem
Dangerous defaults should not survive into production templates or generated configs.
Safe-by-default is better than dangerous-by-default plus audit.

### Scope
- config/*.toml
- deployment templates
- production sample configs
- config generation process

### Task
- review all production-facing MCP config templates
- make safe values the default in production templates
- require explicit allowlists for workflow-capable servers
- make sandboxed shell execution the production baseline
- document the difference between local dev defaults and production defaults

### Expected Result
Production-oriented config artifacts start from safe values without relying on later warning-based review.

### Acceptance Criteria
- production templates are safe-by-default
- workflow-related controls require explicit allowlists
- shell sandbox is enabled in production defaults
- documentation reflects the new template policy
