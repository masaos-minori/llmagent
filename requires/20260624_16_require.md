# Unify strict tool-definition validation behavior across MCP docs

## Summary
Consolidate the behavior of `tool_definitions_strict` and live `/v1/tools` validation into one coherent specification.

## Background
The known-issues document and the operations document describe related parts of startup validation behavior, but the normative behavior is split across files. This makes startup failure conditions harder to understand.

## Objective
Document one authoritative behavior model for:
- tool name mismatch handling
- partial server unreachability
- all-server unreachability
- strict vs non-strict startup behavior

## In Scope
- `04_mcp_90_inconsistencies_and_known_issues.md`
- `04_mcp_06_configuration_and_operations.md`
- `04_mcp_03_routing_lifecycle_and_execution.md`

## Required Changes
- Move the normative startup-validation behavior into one primary document
- Update the known-issues file so it points to the authoritative behavior after resolution
- Ensure strict-mode semantics are consistent across all affected sections
- Clearly distinguish:
  - mismatch
  - partial unreachable
  - all unreachable
  - validation skipped

## Acceptance Criteria
- A reader can determine strict-mode startup behavior from one primary source
- Known issues no longer duplicate normative behavior once clarified
- The operations document and the known-issues document do not contradict each other

## Out of Scope
- Rewriting the tool-definition validation implementation
- Adding new validation modes

## Notes
Use explicit behavior tables and avoid ambiguous wording such as "may fail" where the behavior is intended to be deterministic.
