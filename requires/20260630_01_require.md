# [High] Remove module-level `rag.pipeline._cfg` override from RAG MCP service

## Priority
High

## Difficulty
Medium

## Background
`RagPipelineMCPService.start()` currently injects configuration into `RagPipeline` via `RagPipeline(self._http, rag_cfg)`, but it also mutates `rag.pipeline._cfg` directly. This looks like a leftover compatibility path from the old module-level config cache design. The canonical runtime boundary should be explicit config injection through `RagConfig`.

Source evidence: `service.py`, `shared/types.py` 【1-dea44a】【3-6a074a】

## Goal
Remove process-scoped module-level config override and make injected `RagConfig` the only runtime configuration path for the RAG pipeline.

## Scope
Affected files:
- `scripts/mcp/rag_pipeline/service.py`
- `scripts/rag/pipeline.py`
- `scripts/rag/llm_client.py`
- related tests

## Implementation Tasks
- Remove direct assignment to `rag.pipeline._cfg` in `RagPipelineMCPService.start()`.
- Search for remaining reads of `rag.pipeline._cfg`.
- Replace any remaining module-level config access with values from injected `RagConfig`.
- Ensure `RagPipeline`, `RagLLM`, search stages, rerank stage, and refiner receive configuration through constructor arguments or explicit parameters.
- Add regression tests proving `RagPipelineMCPService` works without mutating module-level config.

## Acceptance Criteria
- No runtime code writes to `rag.pipeline._cfg`.
- RAG MCP service starts and runs pipeline using only injected config.
- Existing RAG MCP tools continue to work:
  - `rag_run_pipeline`
  - `rag_debug_pipeline`
  - `rag_list_documents`
  - `rag_delete_document`
- Tests fail if module-level config override is reintroduced.

## Non-goals
- Do not change MCP tool names.
- Do not change `RagConfig` Protocol fields unless required by existing runtime usage.

## Validation
- Run RAG MCP service startup test.
- Run `/rag search <query> --debug` through Agent if available.
- Run unit tests for `RagPipelineMCPService.start()`.
