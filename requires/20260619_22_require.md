# Title
Remove cache-key ambiguity between documentation and implementation

## Summary
Resolve the remaining ambiguity around MCP tool cache key generation by aligning code and documentation and removing unused alternate logic.

## Background
Documentation previously described an MD5-based cache key, while the implementation uses a plain string concatenation of tool name and serialized args.

## Problem
Even if behavior is currently known, retaining multiple cache-key concepts increases maintenance confusion.

## Required Changes
- Choose one canonical cache-key strategy.
- Remove or deprecate unused helper code if it is not part of the active implementation.
- Update docs to match the actual code path exactly.
- Add a small regression test for cache key behavior if one does not already exist.

## Acceptance Criteria
- Documentation and code describe the same cache-key behavior.
- No unused alternate cache-key path remains silently in the codebase.
- Future readers can determine cache-key behavior unambiguously.

## Out of Scope
- Changing cache eviction policy.
- Redesigning the full caching subsystem.

## Notes
Prefer one clearly documented implementation path.
