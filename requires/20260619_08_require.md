# Title
Redesign `/db` commands to make database targets explicit

## Summary
Refactor the database command interface so that each command clearly specifies which database it operates on.

## Background
The system uses multiple SQLite databases:
- session.sqlite
- rag.sqlite
- workflow.sqlite

Current `/db` documentation and command grouping make the target database ambiguous in some cases.

## Problem
Operators and automated agents should not need implicit knowledge to know whether a DB command affects session, RAG, or workflow data.

## Required Changes
- Redesign DB command structure so that the database target is explicit.
- Consider subcommands such as:
  - `/db rag ...`
  - `/db session ...`
  - `/db workflow ...`
- Align implementation, help text, and documentation with the new structure.
- Keep RAG maintenance responsibilities out of session persistence code.

## Acceptance Criteria
- Every DB command clearly identifies its target database.
- Help text and docs are consistent with actual behavior.
- Responsibility boundaries between agent/session/RAG/workflow operations become clearer.
- Existing critical maintenance functions remain available.

## Out of Scope
- Replacing SQLite with another storage engine.
- Rebuilding the full database maintenance subsystem.

## Notes
Optimize for explicitness, not shorthand convenience.
