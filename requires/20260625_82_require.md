## [High][rag][tests] Remove backward-compatibility test coverage for `rag.llm` and `PipelineStageResult`, and replace it with canonical-path coverage only

### Summary
Backward-compatibility surfaces should not remain preserved indirectly through the test suite.
The RAG docs explicitly identify both the `rag.llm` re-export and `PipelineStageResult` as compatibility-retained constructs. 【1-f0779f】【2-222f47】

### Problem
If tests still validate old import paths or old legacy types, the backward-compatibility layer remains implicitly protected and can be reintroduced accidentally.

### Scope
- RAG tests
- type-related tests
- import-path tests
- diagnostics/debug-output tests

### Task
- remove tests that import from `rag.llm`
- remove tests that reference `PipelineStageResult`
- replace them with tests that use:
  - the real LLM implementation modules
  - `StageResult` only
- add negative tests or static import checks where practical to prevent reintroduction of removed compatibility paths

### Expected Result
The test suite protects only the canonical current RAG interfaces, not deleted compatibility layers. 【1-f0779f】【2-222f47】

### Acceptance Criteria
- no test imports from `rag.llm`
- no test references `PipelineStageResult`
- canonical-path coverage exists for the same functionality
- removed compatibility paths are not preserved by tests
