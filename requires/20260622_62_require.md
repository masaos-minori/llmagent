# Title
Remove cache-key ambiguity between documentation and implementation

## Summary
Resolve the remaining ambiguity around MCP tool cache key generation by aligning code, docs, and helper usage to one canonical strategy.

## Background
The docs previously described an MD5-based cache key, while the implementation uses a plain string key built from tool name and serialized arguments.

## Problem
Even after documentation correction, leaving multiple cache-key concepts in the codebase creates future maintenance confusion.

## Required Changes
- Choose one canonical cache-key strategy.
- Remove or deprecate unused helper code if it is not part of the active implementation.
- Update docs so they describe the actual code path exactly.
- Add a small regression test if appropriate.

## Acceptance Criteria
- Documentation and implementation describe the same cache-key behavior.
- No unused alternate cache-key strategy remains silently in the active path.
- Future readers can determine cache-key behavior unambiguously.

## Out of Scope
- Redesigning cache eviction policy
- Replacing the cache subsystem
