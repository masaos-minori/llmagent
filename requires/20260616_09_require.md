# [MCP][Startup] Clarify and harden tool_definitions_strict startup behavior

## Summary
Strict tool-definition validation is documented, but the behavior for server-unreachable cases is not clear enough.

## Problem
It is hard to tell when startup should:
- fail because of tool mismatch
- fail because of server unreachability
- skip validation because all servers are unreachable

## Scope
- startup validation logic
- startup logs
- related docs

## Task
- separate failure cases clearly:
  - server unreachable
  - `/v1/tools` fetch failed
  - tool mismatch
  - all servers unreachable → skip
- make strict-mode logs explicit
- document exact strict-mode behavior

## Acceptance Criteria
- [ ] startup logs explain the exact reason for failure
- [ ] docs define all strict-mode outcomes clearly
- [ ] strict-mode behavior is deterministic and easy to diagnose
