# H-5: Remove deferred startup_mode reload for MCP servers

## Priority

High

## Background

`startup_mode` controls whether an MCP server is externally managed or managed as a subprocess by the agent.

Changing this at runtime is unsafe because lifecycle routing, subprocess management, command, environment, URL, and startup health polling are all startup-time concerns.

## Target files

- `agent/services/config_reload.py`
- lifecycle reload tests

## Implementation tasks

- Remove runtime assignment of `startup_mode` during reload.
- Do not execute behavior equivalent to:
  - `old_srv.startup_mode = new_srv.startup_mode`
- Remove deferred reporting for MCP startup mode changes.
- Detect `startup_mode` differences.
- Add `mcp/<server_key>.startup_mode` to `ConfigReloadOutcome.needs_restart`.
- Treat both directions as restart-required:
  - `persistent -> subprocess`
  - `subprocess -> persistent`
- Do not add startup mode changes to `applied` or `deferred`.

## Acceptance criteria

- `startup_mode` changes require full agent restart.
- `/reload` reports startup mode changes as `[RESTART]`.
- `/reload` never reports startup mode changes as `[DEFER]`.
- Running config startup mode remains unchanged after `/reload`.
- Lifecycle behavior cannot change mid-process due to reload.

## Test plan

- Reload with `startup_mode` changed from `persistent` to `subprocess`.
- Reload with `startup_mode` changed from `subprocess` to `persistent`.
- Verify both changes are restart-required.
- Verify running config is unchanged.
- Verify no deferred entry is emitted.
