# Title
Consolidate MCP tool routing and tool-definition ownership

## Summary
Reduce routing and definition drift by clarifying the single source of truth for MCP tool names and route resolution.

## Background
Tool routing currently combines:
- config-driven `tool_names`
- static fallback tables
- live `/v1/tools` validation
- separate tool-definition config for LLM exposure

## Problem
This multi-source structure increases the risk of implementation/config/document drift when adding or changing tools.

## Required Changes
- Define a clear source-of-truth strategy for tool ownership and routing.
- Minimize duplicated declarations where possible.
- Add validation to detect drift between:
  - server implementation
  - config route mapping
  - tool-definition exposure
- Document the intended ownership model.

## Acceptance Criteria
- Tool additions require fewer duplicated declarations.
- Drift between implementation and config is easier to detect automatically.
- Route resolution behavior is easier to reason about and document.

## Out of Scope
- Removing all static fallback logic if it is still required for compatibility.
- Rewriting all MCP servers.

## Notes
Prefer explicit ownership over parallel partial ownership.
