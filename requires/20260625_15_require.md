## [High][eventbus][architecture] Introduce a DB access facade and remove direct sqlite3 calls from route handlers

### Summary
Route handlers currently use the shared SQLite connection directly.

### Problem
Direct DB calls inside route handlers make it easy to miss offload boundaries and hard to change connection strategy later.

### Scope
- app.py
- db.py
- all Event Bus route handlers
- DLQ logic entry points

### Facts
- `app.py` stores a module-global shared `_db` connection. 【1-6bdda3】【2-f75a0d】
- Route handlers call `_db.execute(...)` and `_db.commit()` directly. 【1-6bdda3】
- The document guide states code is the canonical source of truth and docs must follow code. 【3-219e10】

### Task
- create a DB access facade in `db.py`
- move all SQL statements behind named DB functions
- make handlers call DB facade functions only
- centralize thread-offload policy inside the DB access layer
- remove raw SQLite calls from `app.py`

### Expected Result
Event Bus route handlers depend on a DB facade, not on raw `sqlite3.Connection`.

### Acceptance Criteria
- no route handler executes raw SQL directly
- SQL logic is centralized in the DB access layer
- the DB access layer owns thread-offload policy
- code becomes ready for later connection-strategy changes
