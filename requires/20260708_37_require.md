# M-5: Separate /mcp status semantics from pending reload config

## Priority

Medium

## Background

`/mcp` status is a health view of currently running MCP servers. It must not be interpreted as a view of pending restart-required config changes.

## Target files

- MCP status command implementation if needed
- documentation
- tests if status output is covered

## Implementation tasks

- Keep `/mcp` and `/mcp status` based on running config.
- Do not display pending restart config inside `/mcp` status.
- Keep pending MCP config differences limited to `/reload` output.
- Document that `/mcp` is not a pending config inspection command.
- Ensure reload classification does not affect subsequent `/mcp` status behavior.

## Acceptance criteria

- `/mcp` shows currently configured running MCP servers.
- `/mcp` does not show newly added servers until restart.
- `/mcp` does not hide removed servers until restart.
- `/mcp` does not probe changed pending URLs until restart.
- Documentation clearly separates runtime health and pending config diff.

## Test plan

- Reload with changed MCP URL.
- Run `/mcp` status.
- Verify it still uses the running URL.
- Reload with new server.
- Run `/mcp` status.
- Verify new server is not shown until restart.
