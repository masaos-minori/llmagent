# Consolidate MCP observability into a single operator diagnostic flow

## Summary
Reorganize observability guidance so operators can diagnose MCP failures using one coherent flow instead of multiple scattered sections.

## Background
The documents separately describe watchdog behavior, health states, transport errors, tool errors, repeated failures, and serialization events. The information is valuable but fragmented.

## Objective
Create a single diagnostic workflow for MCP operations.

## In Scope
- `04_mcp_03_routing_lifecycle_and_execution.md`
- `04_mcp_06_configuration_and_operations.md`

## Required Changes
- Add a unified "How to diagnose MCP failures" section
- Cover at least:
  - transport failure
  - tool-level failure
  - server health state transitions
  - repeated-failure warnings
  - serialization-related slowdown
- Include decision logic such as:
  - what causes a restart
  - what does not cause a restart
  - what to inspect first in logs
- Cross-link existing detail sections instead of duplicating all content

## Acceptance Criteria
- Operators can follow one documented flow to classify a failure
- The flow distinguishes restart-worthy failures from normal tool errors
- Serialization is documented as a safety behavior, not automatically as a defect

## Out of Scope
- Changing watchdog policy
- Adding new log sinks

## Notes
Optimize for incident response readability.
