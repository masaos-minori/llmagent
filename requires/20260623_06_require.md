## Title: [MEDIUM] Enhance `/reload` Command Scope and Fix Configuration Shadowing

### Context
Currently, the `/reload` command only loads `common.toml` and `agent.toml`. However, configuration is fragmented across roughly 10 files (e.g., `security.toml`, `tools.toml`). Additionally, `http` fields are duplicated in `common.toml`, `llm.toml`, and `http.toml`.

### Problem Statement
1. Changes to security policies (`security.toml`) or tool lists cannot be hot-reloaded via `/reload`, requiring a full application restart.
2. Duplicated `http` config blocks lead to shadowing issues, where it's unclear which configuration takes precedence.

### Requirements
- **Full Scope Reload**: Update the `/reload` implementation to invoke `ConfigLoader.load_all()`, parsing all TOML files inside the `config/` directory at runtime.
- **De-duplicate / Define Precedence**: Explicitly define the resolution hierarchy for `http` configurations in `agent/config.py`. Document this hierarchy clearly or eliminate the duplication by centralizing it into a single file.

### Constraints
- Do NOT introduce race conditions; apply a thread-safe update or context swap when replacing the active `AgentConfig` instance.
- Properties requiring a hard restart (e.g., DB paths) must continue to trigger the `[RESTART]` warning.

### Acceptance Criteria (Definition of Done)
- [ ] Running `/reload` after changing a value in `security.toml` successfully reflects the changes in runtime behavior without restarting.
- [ ] Duplicated `http` configuration resolution is backed by explicit code fallback logic or a consolidated configuration structure.
