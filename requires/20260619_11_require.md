# Title
Expose `dry_run` in MCP tool input schemas for all supported tools

## Summary
Update MCP tool definitions so that every tool which already supports `dry_run` in its model or service layer also exposes `dry_run` in its `inputSchema`.

## Background
Several MCP tools already implement `dry_run` behavior internally, but this capability is not visible in the published MCP tool schema.
As a result, the LLM cannot reliably discover or use the safer preview path.

## Problem
The implementation supports `dry_run`, but the schema does not advertise it.
This creates a mismatch between runtime capability and LLM-visible capability.

## Required Changes
- Add `dry_run` to the tool `inputSchema` for all affected tools.
- Ensure tool descriptions clearly explain that `dry_run` performs a preview with no side effects.
- Verify that schema, validation model, and service behavior are aligned.

## Affected Tools
- `write_file`
- `move_file`
- `delete_file`
- `delete_directory`
- `trigger_workflow`

## Acceptance Criteria
- All affected tools expose `dry_run` in `/v1/tools` metadata or equivalent schema output.
- LLMs can discover `dry_run` directly from tool definitions.
- No tool supports `dry_run` only implicitly through implementation while hiding it from schema.

## Out of Scope
- Changing the actual `dry_run` semantics for tools that already support it.
- Adding approval-policy changes unrelated to schema visibility.

## Notes
This task is about schema correctness and LLM discoverability, not about introducing new safety concepts.
