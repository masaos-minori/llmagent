# Title
Add half-open recovery behavior to MCP server health management

## Summary
Improve `McpServerHealthRegistry` and tool dispatch behavior so that temporarily failed MCP servers can recover without requiring only manual restart or external watchdog intervention.

## Background
The current health model marks a server as `UNAVAILABLE` after repeated failures and blocks dispatch to that server.
This is good for protection, but recovery paths are weak once dispatch is fully blocked.

## Problem
A server may recover externally, but the agent may continue treating it as unavailable without a controlled re-entry path.

## Required Changes
- Introduce a half-open or trial-reconnect behavior after a cooldown period.
- Allow limited recovery probes for previously unavailable servers.
- Ensure successful health probes or successful trial calls can restore healthy state.
- Keep transport failures isolated and auditable.

## Acceptance Criteria
- A server can recover from temporary failure without requiring a full agent restart in all cases.
- Recovery does not remove existing protections against repeated failing calls.
- Health transitions remain observable and documented.

## Out of Scope
- Replacing the watchdog subsystem.
- Full redesign of HTTP vs stdio lifecycle management.

## Notes
Use a conservative circuit-breaker style design: fail safely, recover deliberately.
