# [High] Fix regression and stale-artifact fallout from commit 12f8e8c9 (config schema normalization)

## Priority

High

## Type

Bug Fix / Test Maintenance / Documentation Sync / Regression Prevention

## Target files

- `scripts/agent/tool_scheduler.py`
- `tests/test_tool_scheduler_serialization.py`
- `tests/test_removed_commands.py`
- `docs/05_agent_01_system-overview.md`
- `tests/docs/test_command_docs_sync.py`
- (reference only, no change expected) `scripts/agent/commands/cmd_session.py`, `scripts/agent/commands/registry.py`

## Background

A full-suite `uv run pytest -q` run (2026-07-16, immediately after completing an
unrelated tool-count-literal removal batch across 12 files) reported:

```
6 failed, 4036 passed, 13 skipped, 1 warning in 160.38s
```

All 6 failures were confirmed unrelated to the tool-count-removal batch (verified: none
of the 6 failing test files or their production dependencies were touched by that
change). Root-cause investigation (3 parallel sub-agent investigations, one per
distinct failure symptom) traced all 6 failures to a single prior commit:

**`12f8e8c9`** — "Normalize config schema: remove deprecated Memory and GitOps
security keys" (2026-07-15 13:54).

This commit's stated purpose (config schema normalization) does not match its actual
diffstat, which also includes an unrelated 1-line production change to
`scripts/agent/tool_scheduler.py`, and omits necessary follow-up updates in a test file
and a documentation file whose subjects (`cmd_db.py` removal, `/db` → `/session`
command migration) were otherwise correctly and intentionally handled elsewhere in the
same commit (`scripts/agent/commands/cmd_session.py` gained the migrated
functionality; `docs/05_agent_07_08_cli-and-commands-slash-commands-session-mcp.md`
correctly documents the migration).

## Problem

Three independent, differently-natured defects, all traceable to commit `12f8e8c9`:

### P-1: Real regression — `tool_scheduler.py`'s write-first batch serialization flag flipped

`scripts/agent/tool_scheduler.py:105-106` currently reads:

```python
if write_first:
    cgr.append(ScheduledBatch(groups=[write_first], serialize_flags=[True]))
```

This directly contradicts the function's own inline comment (lines 99-101 of the same
file): "write_first gets its own sequential batch (no scope — conservative)...
serialize_flags=True for same-scope write groups, False for reads" — a scope-less
write-first batch should get `False`, matching the pre-`12f8e8c9` behavior.
`tests/test_tool_scheduler_serialization.py::TestSerializeFlags::test_write_first_batch_has_serialize_false`
fails with `assert True is False` as a direct consequence. All 11 other tests in the
same file still pass, isolating the defect to this one line. `git show 12f8e8c9 --
scripts/agent/tool_scheduler.py` confirms this is the *only* change to this file in
that commit, and it has no relationship to the commit's stated Memory/GitOps
config-key-removal purpose — it appears to be an accidental inclusion, not an
intentional behavior change.

### P-2: Stale test — `tests/test_removed_commands.py::TestRemovedDbAliases` imports a deleted module

Commit `12f8e8c9` deleted `scripts/agent/commands/cmd_db.py`,
`db_help_display.py`, and `db_stats_display.py` (1004+ lines removed across those
files plus the old `tests/test_agent_cmd_db.py`) as part of an intentional, documented
migration of `/db session <subcmd>` functionality into `/session <subcmd>`
(`scripts/agent/commands/cmd_session.py` gained a `db_dispatch` code path; the
migration is explicitly documented in
`docs/05_agent_07_08_cli-and-commands-slash-commands-session-mcp.md:46`).

However, `tests/test_removed_commands.py::TestRemovedDbAliases` (added earlier, in
commit `6f31e877`, 2026-07-01 — predating the `cmd_db.py` deletion by two weeks) was
not updated in the same commit. Its 4 methods
(`test_db_urls_rejected`, `test_db_clean_rejected`, `test_db_rebuild_fts_rejected`,
`test_db_recover_rejected`) each do `from agent.commands.cmd_db import _DbMixin`,
which now raises `ModuleNotFoundError: No module named 'agent.commands.cmd_db'` at
collection/execution time — not because the underlying behavior these tests intended
to verify (flat `/db <subcmd>` aliases must be rejected) is wrong, but because the
verification mechanism (importing `_DbMixin` directly) no longer has a target to
import. `rg -n "_DbMixin" scripts/ tests/` confirms zero production references remain
outside a single historical code comment in
`scripts/agent/services/db_maintenance_service.py:5`.

### P-3: Documentation drift — `/db` command row not removed from system overview doc

`docs/05_agent_01_system-overview.md:129` still contains a command-reference table row:

```
| DB | `/db rag stats|urls|clean|rebuild-fts|vec-rebuild|reconcile-url|recover|consistency; session stats|health|checkpoint|vacuum|purge|recover` |
```

`tests/docs/test_command_docs_sync.py::TestCommandDocsSync::test_active_docs_match_registered_commands`
scans this doc (and one other, `docs/05_agent_07_cli-and-commands.md`, which no
longer exists post-split and is silently skipped by the test's own
`if not filepath.exists(): continue` guard) for command-name references and compares
them against the single source of truth,
`scripts/agent/commands/command_defs_list.py`'s `_COMMANDS` list. It fails with:

```
AssertionError: .../docs/05_agent_01_system-overview.md contains unregistered
command references: ['/db']
```

The correct migration note already exists elsewhere
(`docs/05_agent_07_07_cli-and-commands-migration-notes.md:42`, stating `/db`
command is "完全削除" / fully removed), but `05_agent_01_system-overview.md`'s
command table was never updated to match, and the sync test only checks the two
hardcoded filenames above — one of which is stale (the file it names was split into
11 separate files, `05_agent_07_01_...` through `05_agent_07_11_...`, none of which
the test currently scans).

## Reason for change

All three defects erode confidence in `uv run pytest -q` as a clean regression gate:
a real behavior regression (P-1) is currently masked by two unrelated pre-existing
failures (P-2, P-3), making it easy for a future contributor to see "6 failures,
probably pre-existing, ignore" and miss that one of them is an actual bug introduced
by a real, reachable code path. Additionally, commit `12f8e8c9`'s pattern — bundling
multiple completed, previously-reviewed implementation plans into one commit whose
message describes only one of them — is itself a process risk worth flagging, since
it makes it easy for an unrelated one-line change (P-1) to slip in unnoticed and for
required companion updates (P-2, P-3) to be dropped silently.

## Design intent

- P-1: Restore `serialize_flags=[False]` for the write-first batch, matching the
  function's own documented invariant (scope-less write-first batches are not
  serialized against each other; only same-scope write groups are). Do not change any
  other line in `build_execution_groups()`.
- P-2: Update `TestRemovedDbAliases` to verify the *currently correct* place where
  flat `/db <subcmd>` aliases must be rejected — most likely via
  `CommandRegistry`/`agent.commands.cmd_session`'s `_SessionMixin` dispatch path (the
  actual current home of the migrated functionality), or via whatever public
  entry point a user-facing `/db ...` input now resolves through. Do not resurrect
  `cmd_db.py` or `_DbMixin` — the migration itself is correct and intentional; only the
  test's import target is stale.
- P-3: Remove the stale `/db` row from `docs/05_agent_01_system-overview.md`'s command
  table (or update it to point to the `/session` equivalents, matching the wording
  already used in `05_agent_07_08_...md`). Separately, update
  `tests/docs/test_command_docs_sync.py`'s hardcoded doc-file list to scan the actual
  current 11-file split (`05_agent_07_01_...md` through `05_agent_07_11_...md`)
  instead of (or in addition to) the no-longer-existing `05_agent_07_cli-and-commands.md`,
  so the sync check has real, current coverage rather than silently skipping a
  nonexistent file.

### Process note (no code change required, informational)

- Consider whether future commits like `12f8e8c9` (bundling multiple completed,
  previously-reviewed implementation docs into a single commit) should instead be
  split into one commit per implementation doc, to reduce the chance of an unrelated
  change slipping in unnoticed and to make "which commit changed X" traceable per
  logical change rather than per batch. This is a process suggestion for the human
  maintainer, not an automated enforcement mechanism to build.

## Non-goals

- Do not resurrect `cmd_db.py`, `db_help_display.py`, `db_stats_display.py`, or
  `_DbMixin` — the `/db` → `/session` migration itself is correct, intentional, and
  out of scope to reverse.
- Do not re-litigate or redesign the `/session` command's db-related subcommands
  (`stats`, `health`, `checkpoint`, `vacuum`, `purge`, `recover`) — only fix the stale
  test/doc references to them.
- Do not attempt to split or rewrite the historical commit `12f8e8c9` itself (e.g. via
  rebase) — address its fallout forward, not retroactively.
- Do not expand this requirement into a general audit of every commit for similar
  bundling risk — the process note above is informational only.
