## Title

Clean up config lifecycle fields and deprecated GitOps approval settings

## Priority

High

## Target files

- `agent/config_dataclasses.py`
- `agent/config_builders.py`
- `agent/services/config_reload.py`
- `agent/tool_policy.py`
- `agent/tool_approval.py`
- `tests/test_config_reload.py`
- `tests/test_config_reload_classification.py`
- `tests/test_tool_approval_repos.py`
- `tests/test_tool_approval_risk.py`
- Documentation files related to config and approval behavior

## Background

Several config fields currently have inconsistent lifecycle behavior.

`use_memory_layer` exists in `MemoryConfig` and is read at startup, but reload classification contains a removed-comment path and does not report changes as startup-only.

Separately, `ApprovalConfig` still contains deprecated GitOps-related fields:

- `gitops_force_push_blocked`
- `gitops_protected_branches`

However, these fields are not read by the builder. Branch risk escalation currently uses `approval_high_risk_branches`.

## Problem

There are two related issues:

1. `use_memory_layer` is a real startup field but is not reported as startup-only during reload.
2. Deprecated GitOps fields remain visible on the dataclass while being disconnected from config loading.

This creates ambiguity for operators and future implementers.

## Reason for change

Config fields should have clear lifecycle semantics:

- runtime reloadable
- startup-only
- removed
- deprecated but intentionally retained

No field should appear as active configuration if it is no longer loaded or enforced.

## Implementation intent

Clarify config lifecycle behavior and remove misleading dead configuration paths.

## Implementation instructions

### Part A: `use_memory_layer`

- Restore detection of `use_memory_layer` changes in `_detect_startup_only()`.
- If the new config contains `use_memory_layer` and it differs from the running config, add it to `startup_only`.
- Do not hot-apply this setting unless memory service reconstruction is implemented.
- Remove or correct the stale comment that says the key was removed from schema.
- Align tests with the intended startup-only behavior.

### Part B: deprecated GitOps fields

Choose one design.

Preferred option: remove deprecated fields.

- Remove `gitops_force_push_blocked` from `ApprovalConfig`.
- Remove `gitops_protected_branches` from `ApprovalConfig`.
- Remove stale removed comments in the builder.
- Keep `approval_high_risk_branches` as the single branch-risk mechanism.
- Update docs and tests.

Alternative option: reconnect the fields.

- Read the fields in `_build_approval_config()`.
- Define clear enforcement behavior.
- Add tests for force-push blocking and protected branch handling.

## Acceptance criteria

- `use_memory_layer` changes appear under `/reload` startup-only output.
- Runtime memory service is not silently changed by reload.
- Removed GitOps fields are either fully removed or fully reconnected.
- Dataclass fields and builder fields are consistent.
- Branch high-risk behavior remains covered by tests.

## Tests

Add or update tests for:

- `use_memory_layer` true-to-false change appears in `startup_only`.
- `use_memory_layer` false-to-true change appears in `startup_only`.
- Same `use_memory_layer` value does not appear in `startup_only`.
- Deprecated GitOps fields are no longer present if removal is chosen.
- `approval_high_risk_branches` still escalates GitHub branch risk.
