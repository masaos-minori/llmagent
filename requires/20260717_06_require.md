# Use RuntimeToolRegistry for scheduler metadata and side-effect detection

## Priority

High

## Target files

- agent/tool_runner.py
- agent/tool_scheduler.py
- shared/tool_spec.py
- shared/tool_executor_helpers.py
- shared/runtime_tool_registry.py
- tests/agent/test_tool_runner.py
- tests/agent/test_tool_scheduler.py
- tests/shared/test_runtime_tool_registry.py

## Reason

DAG scheduling and side-effect detection currently depend on static tool_definitions and static tool constants.

Scheduler metadata should come from runtime-discovered MCP tool metadata such as `is_write`, `requires_serial`, and `resource_scope`.

## Intent

Make RuntimeToolRegistry the source for ToolSpec and side-effect decisions.

## Scope

Replace ToolRunner metadata construction from static tool_definitions with RuntimeToolRegistry-derived ToolSpec.

Replace primary side-effect detection based on static constants with:

- `runtime_tools.is_side_effect(tool_name)`

## ToolSpec requirements

RuntimeToolRegistry should provide ToolSpec metadata for each approved tool call:

- `call_id`
- `name`
- `args`
- `resource_scope`
- `requires_serial`
- `is_write`

## Safe defaults

For unknown or ambiguous tools:

- side-effecting = true
- requires serial = true
- requires approval = true

## Acceptance criteria

- ToolRunner does not create ToolSpec from static tool_definitions.
- Scheduler receives RuntimeToolRegistry-derived metadata.
- Serialization behavior remains correct.
- Static constants are not the primary source of side-effect detection.
