## [worker-runtime][design] Design a reusable WorkerRuntime for role execution

### Summary

現在の Agent 実装は対話型 REPL に最適化されているが、新システムでは Planner / Retriever / PatchWorker / Integrator / Validator / Publisher を共通枠組みで起動できる reusable WorkerRuntime が必要である。

### Problem

role ごとに実行制御を複製すると、tool policy / input-output schema / timeout / logging / health handling が分散し、保守性と安全性が低下する。一方で WorkerRuntime が状態正本を持つ設計も避ける必要がある。

### Scope
- shared worker runtime design
- role-specific runtime settings
- stateless execution rule
- UI-specific code の分離

### Task
- WorkerRuntime の責務を定義する
- 全 role で共有する部分を定義する
- role ごとに差し替える部分を定義する
- runtime と state_manager / retry_controller / barrier の接続点を定義する
- UI 依存コードを runtime から分離する

### WorkerRuntime Concept

WorkerRuntime は、role-specific な prompt / tool policy / MCP binding / schema を受け取り、1 task または 1 phase 単位の処理を実行する stateless executor である。

#### Shared parts across all roles
- MCP endpoint access through mcp_registry.py
- health / capability pre-check
- input schema validation
- output schema validation
- timeout / cancellation handling
- event emission to state_manager / audit_logger
- standard worker logging

#### Role-specific parts
- prompt configuration
- allowed / denied tool lists
- input / output schema
- LLM endpoint or model setting
- worktree isolation requirement
- concurrency limit
- task batching policy

#### Explicit non-responsibilities
- workflow orchestration
- retry policy decision
- barrier state decision
- approval state ownership
- state source-of-truth ownership

### Integration Rules
- 永続 state 読み書きは state_manager API 経由のみ
- retry 可否判定は retry_controller が担当する
- phase 遷移判定は orchestrator / workflow_engine が担当する
- worktree 作成 / cleanup は worktree_manager が担当する
- write 操作は Publisher + repository_gateway.py に限定する

### Policy Alignment Notes

This requirement must follow the Phase 1 integration policy below.

1. **Do not create `orchestrator-mcp`.**
   - The central Orchestrator is an internal Agent component, not an MCP server.
   - Role MCP services may be added and monitored via MCP health/capability checks, but the Orchestrator itself must not be registered as an MCP server.
2. **Add internal Orchestrator APIs.**
   - REPL / command layers should call internal APIs such as `start_workflow`, `resume_workflow`, `cancel_workflow`, `get_workflow_status`, `get_workflow_artifacts`, and `retry_phase`.
3. **Use `state_manager.py` as the source of truth.**
   - Workflow / task / artifact / retry / approval state must be persisted and read through `state_manager.py`.
   - REPL, `AgentContext`, WorkerRuntime, JSONL logs, and UI code must not own canonical workflow state.
4. **Extend `workflow.sqlite`; event messages stay in the existing Event Bus.**
   - Phase 1 must reuse and extend `workflow.sqlite` for workflow metadata, task state, artifact metadata references, retries, and approvals.
   - Do not introduce a separate `orchestrator.sqlite` in Phase 1.
   - Event messages themselves are NOT duplicated into `workflow.sqlite`. They are already persisted by the existing Event Bus (`eventbus.sqlite`, managed by `scripts/eventbus/`), which provides publish/subscribe, replay, ack/nack, and DLQ over HTTP (see `docs/06_eventbus_*.md`). `workflow.sqlite` only tracks which Event Bus `event_id`s a workflow has already consumed, via the `processed_events` table (idempotency — see `scripts/agent/workflow/idempotency_ops.py`).

### Expected Result

role-specific worker を共通 runtime で起動しつつ、状態正本の分散や責務混濁を防げる。

### Acceptance Criteria
- shared worker runtime design が定義されている
- role-specific settings が定義されている
- runtime が stateless executor であることが明記されている
- UI-dependent parts が runtime logic から分離されている
