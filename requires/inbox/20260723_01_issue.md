# Document MCP Tool Ownership, Service Boundaries, and Capability Mapping

## Summary

Create a consolidated MCP documentation update that defines tool ownership, MCP service boundaries, and capability-level dependencies.

This issue replaces separate documentation tasks for:

- MCP Tool Ownership Matrix
- MCP Service Boundaries
- MCP Dependency and Capability Diagrams

## Reason for Change

The current documentation describes MCP servers and runtime tool discovery, but it does not provide a single canonical view of which tool belongs to which MCP server.

This creates several maintenance risks:

- Developers cannot quickly identify the owning MCP server for a tool.
- Tool ownership may become ambiguous as new tools are added.
- Responsibility boundaries between similar MCP servers may erode over time.
- Workflow and approval behavior cannot be easily traced back to specific MCP tools.
- Runtime discovery explains how tools are discovered, but not who owns each tool.

The most important unclear areas are:

- Local Git operations versus GitHub API operations
- GitHub repository operations versus CI/CD operations
- File read, file write, and file delete responsibilities
- RAG versus MDQ responsibilities
- Tool risk classification and approval requirements

## Intent

The intent is to make MCP ownership explicit and reviewable.

After this change, a maintainer should be able to answer the following questions from documentation alone:

- Which MCP server owns a given tool?
- Which capability depends on a given MCP server?
- Which workflow stage normally invokes the tool?
- Which tools require approval?
- Which operations are explicitly outside the responsibility of a given MCP server?
- Where should a new tool be added?

This is not intended to change runtime behavior.
This is a documentation and architecture clarification task.

## Scope

Create or update the following documentation:

- `04_mcp_01_tool_ownership_matrix.md`
- `04_mcp_02_service_boundaries.md`
- MCP document guide or index
- Architecture overview links where appropriate

## Required Content

### 1. Tool Ownership Matrix

Add a canonical matrix with the following fields:

- Tool name
- Owning MCP server
- Capability group
- Risk tier
- Approval required
- Typical workflow stage
- Notes

The matrix must cover all built-in tools exposed by MCP servers.

Suggested capability groups:

- File Management
- Repository Operations
- GitHub Operations
- CI/CD Operations
- Internet Search
- RAG Retrieval
- Markdown Query
- Shell Execution

### 2. MCP Service Boundary Definitions

For each MCP server, document:

- Responsibilities
- Explicit non-responsibilities
- Allowed operation types
- Forbidden operation types
- Ownership rationale

At minimum, define boundaries for:

- `web-search-mcp`
- `file-read-mcp`
- `file-write-mcp`
- `file-delete-mcp`
- `shell-mcp`
- `git-mcp`
- `github-mcp`
- `cicd-mcp`
- `rag-pipeline-mcp`
- `mdq-mcp`

### 3. Boundary Clarification Rules

Document rules such as:

- Local repository operations belong to `git-mcp`.
- GitHub remote API operations belong to `github-mcp`.
- GitHub Actions operations belong to `cicd-mcp`.
- File reading, writing, and deletion remain separate services.
- RAG handles semantic retrieval.
- MDQ handles Markdown-structured querying.

### 4. Capability Dependency Mapping

Create a capability mapping that links:

- Capability
- Required MCP server
- Optional MCP server
- Degraded behavior
- Unavailable behavior

Example capabilities:

- Repository Operations
- GitHub Operations
- CI/CD Operations
- File Management
- Internet Search
- RAG Retrieval
- Markdown Query
- Shell Execution

### 5. Mermaid Diagrams

Add Mermaid diagrams showing:

- Tool to MCP ownership
- Capability to MCP dependency
- Workflow stage to tool to MCP relationship

Use GitHub-compatible Mermaid syntax.

## Implementation Notes for AI Agent

- Do not invent tool names if the exact tool list is unavailable.
- If a tool list is incomplete, mark missing tool names as `Unknown` and add a TODO.
- Prefer documentation updates over code changes in this issue.
- Keep ownership statements strict and unambiguous.
- Do not include EventBus-related content in this issue.
- Preserve existing terminology such as `RuntimeToolRegistry`, `ToolExecutor`, and `RouteResolver`.

## Acceptance Criteria

- A canonical MCP Tool Ownership Matrix exists.
- Every known built-in MCP tool is mapped to exactly one owning MCP server.
- Every MCP server has a responsibility and non-responsibility section.
- Capability-level dependencies are documented.
- Mermaid diagrams render correctly in GitHub Markdown.
- MCP documentation index links to the new documents.
- Ambiguous or unknown tool ownership is explicitly marked as `Unknown`, not guessed.
