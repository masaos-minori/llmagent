# Needs confirmation 箇所を一覧化する

## 概要

設計書群に含まれる `Needs confirmation` の箇所を抽出し、確認待ち項目として一覧化する。

## 修正理由

`Needs confirmation` は、設計意図、実装現状、文書記述の関係が未確定であることを示す重要なラベルである。

しかし、これらが本文中に分散したままだと、どの未確認事項が残っているのか、どれを優先して確認すべきかが把握しにくい。

## 修正意図

この修正の意図は、`Needs confirmation` の本文をすぐに修正することではなく、まず確認待ち項目として一覧化し、追跡可能にすることである。

判断、修正、Known Issues への移行は別 Issue で行う。

## 対象ファイル

- `docs/00_governance_07_needs-confirmation-inventory.md`

新規作成する。

## 実施内容

`docs/` 以下の Markdown ファイルから `Needs confirmation` を含む箇所を抽出し、一覧化する。

## 一覧項目

各項目には以下を含める。

- ID
- Source File
- Section
- Quoted Summary
- Question
- Evidence
- Impact
- Required Action
- Target Document
- Status
- Review Timing

## ID 形式

以下の形式を使用する。

- `NC-001`
- `NC-002`
- `NC-003`

## Status Values

以下を使用する。

- open
- investigating
- resolved
- moved-to-known-issues
- deprecated

## AI 実装時の共通指示

- この Issue の対象範囲のみを実施すること。
- 実装コードは変更しないこと。
- 既存ドキュメントの移動、統合、削除、リネームは行わないこと。
- `docs/` 以下に新規サブディレクトリを作成しないこと。
- Markdown の構造を壊さないこと。
- コードブロック、リンク、見出し階層を正しく維持すること。
- 判断に迷う場合は推測で修正せず、未解決事項として報告すること。
- EventBus と Workflow の関係整理は本 Issue では行わないこと。
- ドキュメント分割サイズを変更しないこと。
- 作業後、変更ファイル、変更概要、未解決事項を報告すること。

## 作業禁止事項

- 抽出元の記述を削除しない。
- `Needs confirmation` を `Explicit in code` に変更しない。
- 実装の正誤を断定しない。
- Known Issues 既存エントリを変更しない。

## 検証観点

- `Needs confirmation` 箇所が抽出されていること。
- 各項目に確認すべき Question があること。
- 影響と対応先が分かること。
- 一覧が今後の Known Issues 移行に使えること。

## 受け入れ条件

- [ ] `docs/00_governance_07_needs-confirmation-inventory.md` が作成されている。
- [ ] `Needs confirmation` 箇所が一覧化されている。
- [ ] 各項目に ID が付与されている。
- [ ] 各項目に Source File がある。
- [ ] 各項目に Question / Impact / Required Action がある。
- [ ] 抽出元文書を変更していない。

## 作業後の報告内容

1. 作成したファイル
2. 抽出した件数
3. 代表的な確認待ち項目
4. Known Issues へ移行が必要そうな項目数
