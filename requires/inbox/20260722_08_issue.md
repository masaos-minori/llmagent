# Align startup validation severity documentation and FATAL output behavior

## Priority

Medium

## Scope

Documentation and small implementation alignment if desired.

## Target documentation sections

- `04_mcp_06_11_startup-validation-behavior-tool_definitions_strict.md`
- Agent CLI output or startup display documentation
- Any section describing `[fatal]`, `[non-fatal]`, or startup validation messages

## Problem

Startup validation uses statuses such as `WARNING`, `FATAL`, and `SKIPPED`.

However, current startup display code sends FATAL outcomes through the warning output path while using a fatal tag. At the same time, `CLIView` exposes a dedicated `write_fatal()` method.

This creates ambiguity between semantic severity and display method.

## Required decision

Choose one of the following:

### Option A: Change implementation

Use `write_fatal()` for `StartupCheckStatus.FATAL`.

### Option B: Document current behavior

Explicitly state that startup pipeline messages use `write_warning()` for display, and severity is indicated by `OutputTag.FATAL`.

## Recommended approach

Option A is preferred because it matches the CLI abstraction and makes fatal startup outcomes visually and semantically distinct.

## Required documentation updates

Document:

- Meaning of `WARNING`.
- Meaning of `FATAL`.
- Meaning of `SKIPPED`.
- Production versus local behavior for MCP discovery.
- How fatal startup outcomes are displayed.

## Reason for change

Operators should not have to infer severity only from embedded text. Fatal startup validation outcomes should be visually distinct and consistently documented.

## Acceptance criteria

- Startup validation severity documentation matches implementation.
- FATAL display behavior is explicit.
- Tests are updated if output method or text changes.
- The docs clarify that `mcp_tool_discovery` skipped in local mode can still indicate a full-session tool-call outage.
