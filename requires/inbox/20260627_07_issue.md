## [orchestrator][retry] Define retry and abort rules

### Summary

validation failure や worker crash が発生した場合に、安全に再実行するか中断するかを機械的に判断できる retry / abort policy が必要である。

### Problem

retry 条件が抽象的だと deterministic failure を無駄に再試行し、publish の安全性やコストを損なう。逆に abort 条件が曖昧だと workflow が stuck しやすい。

### Scope
- retry policy
- abort policy
- stuck detection
- cleanup after abort

### Task
- failure classification を定義する
- retry 可能条件を定義する
- max retry を phase / role / error_class ごとに定義する
- abort 条件を定義する
- stuck detection ルールを定義する
- abort 後 cleanup を定義する

### Failure Classes
- **transient**: transport error, temporary MCP unhealth, timeout with retryable evidence, SQLITE_BUSY など
- **deterministic**: syntax error, lint failure, unit test assertion failure, policy deny, approval deny, hash mismatch など
- **fatal**: corrupted persistent state, unrecoverable schema mismatch, mandatory artifact loss など

### Retry Rules
- 自動 retry 対象は transient failure のみ
- deterministic failure は原則 retry しない（明示的 retry_phase を除く）
- fatal failure は即 abort 候補とする
- retry unit は (workflow_id, phase, role, error_class)

### Example Max Attempts
- Planner: 1
- Retriever: 2
- PatchWorker: 2
- Integrator: 1
- Validator: 1（自動）
- Publisher: 0 or 1（policy/approval 成立後の transient のみ）

### Abort Conditions
- fatal failure が発生した場合
- 同一 retry unit が max_attempts を超過した場合
- approval deny / policy deny / hash mismatch で publish 前提が崩れた場合
- mandatory artifact を復元できない場合
- barrier が timeout を超えて解放されない場合

### Stuck Detection Rules
- phase heartbeat が一定時間更新されない
- worker が running のまま timeout を超過する
- barrier が blocker_reason 不明のまま継続する
- state_manager 更新が連続失敗する

### Cleanup After Abort
- running tasks を cancel する
- orphaned worktrees 候補を記録し cleanup queue に渡す
- workflow state を failed または cancelled に遷移する
- audit log / workflow event に abort reason を記録する
- partially generated artifacts を quarantine または terminal mark する

### Policy Alignment Notes

This requirement must follow the Phase 1 integration policy below.

1. **Do not create `orchestrator-mcp`.**
   - The central Orchestrator is an internal Agent component, not an MCP server.
   - Role MCP services may be added and monitored via MCP health/capability checks, but the Orchestrator itself must not be registered as an MCP server.
2. **Add internal Orchestrator APIs.**
   - REPL / command layers should call internal APIs such as `start_workflow`, `resume_workflow`, `cancel_workflow`, `get_workflow_status`, `get_workflow_artifacts`, and `retry_phase`.
3. **Use `state_manager.py` as the source of truth.**
   - Workflow / task / artifact / retry / approval state must be persisted and read through `state_manager.py`.
   - REPL, `AgentContext`, WorkerRuntime, JSONL logs, and UI code must not own canonical workflow state.
4. **Extend `workflow.sqlite`; event messages stay in the existing Event Bus.**
   - Phase 1 must reuse and extend `workflow.sqlite` for workflow metadata, task state, artifact metadata references, retries, and approvals.
   - Do not introduce a separate `orchestrator.sqlite` in Phase 1.
   - Event messages themselves are NOT duplicated into `workflow.sqlite`. They are already persisted by the existing Event Bus (`eventbus.sqlite`, managed by `scripts/eventbus/`), which provides publish/subscribe, replay, ack/nack, and DLQ over HTTP (see `docs/06_eventbus_*.md`). `workflow.sqlite` only tracks which Event Bus `event_id`s a workflow has already consumed, via the `processed_events` table (idempotency — see `scripts/agent/workflow/idempotency_ops.py`).

### Expected Result

orchestrator は再試行可能な失敗と中断すべき失敗を区別し、安全に recovery または stop を実行できる。

### Acceptance Criteria
- failure classification が定義されている
- retry rules が定義されている
- max retry rules が定義されている
- abort conditions が定義されている
- stuck detection rules が定義されている
- cleanup rules が定義されている
