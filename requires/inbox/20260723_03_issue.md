# Document Failure Modes, Degradation Behavior, and Operational Readiness

## Summary

Create a centralized operations document that explains failure modes, degradation behavior, and capability-oriented readiness.

This issue combines the previous tasks for:

- Failure modes and degradation documentation
- Operational readiness model

## Reason for Change

Normal operation is documented in detail, but failure and degradation behavior is distributed across multiple documents.

Operators need one place to understand what happens when parts of the system are unavailable.

Important examples include:

- MCP server startup failure
- Workflow definition missing or invalid
- Workflow database schema mismatch
- RAG database failure
- Embedding service failure
- Memory layer degraded mode
- Tool discovery failure
- Partial system startup in local mode

The current documentation describes some of these behaviors individually, but it does not provide a single operational model that explains system readiness from the user's point of view.

## Intent

The intent is to move from service-oriented status to capability-oriented readiness.

Instead of only showing that a service is unavailable, the documentation should explain what user-visible capability is affected.

For example:

- `git-mcp` unavailable means Repository Operations may be unavailable.
- `rag-pipeline-mcp` unavailable means RAG Retrieval may be unavailable.
- Embedding service unavailable may cause semantic search degradation.
- Memory embedding failure may degrade memory retrieval to FTS-only mode.

This issue does not require implementing readiness reporting.
It prepares the operational model that implementation can follow.

## Scope

Create a document:

- `05_failure_modes_and_operational_readiness.md`

Update operations documentation and indexes to link to it.

## Required Sections

### 1. Failure Mode Overview

Document failure categories:

- Startup failure
- Runtime service failure
- Configuration failure
- Database failure
- Tool discovery failure
- LLM service failure
- Embedding service failure
- Degraded memory mode
- RAG search degradation

### 2. MCP Failure Behavior

Document:

- `startup_mode = none`
- `startup_mode = persistent`
- `startup_mode = subprocess`
- fail-fast behavior in production mode
- fail-open behavior in local mode
- expected operator action

### 3. Workflow Failure Behavior

Document:

- Missing workflow definition
- Invalid workflow definition
- Missing workflow database
- Workflow schema mismatch
- Required table missing
- Expected startup behavior
- Recovery action

### 4. RAG Failure Behavior

Document:

- Embedding API unavailable
- RAG database unavailable
- Vector extension unavailable
- FTS search failure
- Rerank failure
- Refiner fallback
- Expected degraded behavior

### 5. Memory Layer Failure Behavior

Document the known memory activation modes:

- `disabled`
- `fts-only`
- `degraded`
- `hybrid`

Explain what each mode means for user-visible behavior.

### 6. Capability Readiness Model

Define capability readiness states:

- `healthy`
- `degraded`
- `unavailable`
- `unknown`

For each capability, document:

- Required services
- Optional services
- Degraded conditions
- Unavailable conditions
- Recovery actions

Suggested capabilities:

- Workflow Execution
- File Management
- Repository Operations
- GitHub Operations
- CI/CD Operations
- RAG Retrieval
- Markdown Query
- Memory Services
- Internet Search
- Shell Execution

### 7. Operator-Facing Examples

Include user-facing examples such as:

- "Repository Operations unavailable because git-mcp is unavailable."
- "RAG Retrieval degraded because embedding service is unavailable."
- "Memory Services running in fts-only mode."
