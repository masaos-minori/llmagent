## Problem

git-mcp implements runtime availability metadata in `/v1/tools`, but the documentation must be updated to reflect this as current behavior.

Also, the disabled `/v1/call_tool` response appears to omit the disabled reason even though it is computed.

## Current implementation behavior to document

For `/v1/tools`:

- If `allowed_repo_paths` is empty:
  - All git tools are disabled.
  - `disabled_reason` is `allowed_repo_paths is empty`.

- If `read_only=true`:
  - Git write tools are disabled.
  - `disabled_reason` is `read_only=true`.

- Read-only git tools remain enabled when `allowed_repo_paths` is non-empty.

## Required implementation check

Verify and fix git-mcp `/v1/call_tool` disabled handling.

Expected response:

- `Tool disabled: allowed_repo_paths is empty`
- `Tool disabled: read_only=true`

Current response appears to be:

- `Tool disabled: `

Also verify validation error handling. If the common contract expects `Validation error: <details>`, git-mcp should include the actual validation error details.

## Required documentation changes

Update git-mcp documentation to state:

- git-mcp currently returns `enabled` and `disabled_reason` from `/v1/tools`.
- `allowed_repo_paths is empty` takes precedence over `read_only=true`.
- Disabled direct tool calls should return `Tool disabled: <reason>`.

If implementation is not changed immediately, document the current behavior as a known deviation.

## Reason for change

The server already exposes useful availability metadata, but if direct disabled calls hide the reason, operators lose the most important diagnostic detail. The documentation and implementation should agree on the disabled-tool contract.

## Acceptance criteria

- git-mcp catalog accurately describes current `/v1/tools` availability metadata.
- Disabled git tool calls include the concrete reason, or the deviation is documented as known.
- Validation errors either include details or the docs explicitly state that details are omitted.
- Tests cover:
  - Empty `allowed_repo_paths`.
  - `read_only=true` for write tools.
  - Enabled read-only tools when repository paths are configured.
