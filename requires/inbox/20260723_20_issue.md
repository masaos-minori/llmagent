# Compress Implementation-derived File Structure Listings

## Priority

High

## Summary

Compress detailed file structure listings that duplicate the implementation tree, and replace them with concise responsibility-oriented summaries.

## Reason for Change

Several overview documents contain long directory trees and file-by-file explanations for implementation packages.

These lists are easy to verify from the repository itself. They also become stale when files are renamed, added, removed, or reorganized.

A design document should explain why a package or layer exists, what it owns, what boundaries it enforces, and what other components depend on it. It should not duplicate the entire file tree.

## Implementation Intent

The intent is to keep the overview documents useful as architecture documents, not source tree catalogs.

After this change, the documentation should explain:

- Major package responsibilities
- Layer boundaries
- Runtime or process boundaries
- Dependency direction
- Configuration ownership
- Security-relevant boundaries
- Operationally important exceptions
- Links to more specific documents

The documentation should no longer contain exhaustive lists of every implementation file.

## Required Changes

For each target document:

1. Identify file-by-file listings that merely mirror the repository.
2. Remove or compress detailed implementation file lists.
3. Replace them with Japanese responsibility-oriented summaries.
4. Preserve major directory and package responsibilities.
5. Preserve layer boundaries and ownership descriptions.
6. Preserve configuration ownership and runtime boundaries.
7. Preserve security-relevant or operationally relevant notes.
8. Add source references where appropriate.

Suggested replacement sections:

- `## 主要ディレクトリと責務`
- `## レイヤー境界`
- `## 変更時の注意点`
- `## 実装詳細の参照先`

Example replacement style:

- `scripts/agent/` — Agent REPL、ターン制御、ツール実行、メモリ、Workflow 制御を担当する。
- `scripts/mcp_servers/` — 単一責務 MCP サーバ群を提供する。
- `scripts/rag/` — RAG 取込・検索パイプラインを提供する。
- `scripts/shared/` — 全層共通の設定、HTTP、LLM、MCP、型、ログ、ユーティリティを提供する。
- `scripts/db/` — SQLite スキーマ、接続、保守、復旧処理を提供する。

Add a note such as:

- 完全なファイル一覧はリポジトリの実装ツリーを参照する。

## Content to Remove or Compress

Remove or compress:

- Complete source file lists
- One-line descriptions for every source file
- Repeated `__init__.py` explanations
- Long nested directory trees that mirror the repository
- Tables that only map file names to implementation descriptions

## Content to Keep

Keep:

- Package-level responsibilities
- Architectural layer boundaries
- Process boundaries
- Runtime ownership
- Dependency direction
- Configuration ownership
- Security-relevant boundaries
- Operational constraints
- Links to more specific documents
- Known Issues
- Needs Confirmation items

## Acceptance Criteria

- Detailed file-by-file listings are significantly reduced.
- Each document still explains what the major directory or package is responsible for.
- Design intent and responsibility boundaries are preserved.
- Existing useful links are preserved.
- No Known Issues or Needs Confirmation items are removed.
- New documentation text is written in Japanese.
- Markdown renders correctly.

## Definition of Done

- The edited documents are shorter and more design-focused.
- The source tree is no longer duplicated in full.
- Each edited document still provides enough architectural context.
- No source code is modified.
- No ADR documents are created.
- `00_index.md` is not modified.

## Out of Scope

- Do not remove design intent.
- Do not remove responsibility boundaries.
- Do not remove invariants.
- Do not remove failure behavior.
- Do not remove operational constraints.
- Do not remove Known Issues.
- Do not modify source code.
- Do not compress API signature tables in this task.
- Do not compress DTO or JSON field listings in this task.

## AI Implementation Instruction

Work document by document.

For each target file, first identify implementation-derived file listings.

Compress them into Japanese responsibility-oriented summaries.

If a listing contains design-relevant information, preserve that information in a concise form.

If unsure whether content is design-relevant, keep it and mark it for later review rather than deleting it.
