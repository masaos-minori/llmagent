## 重点確認項目

以下を重点的に確認してください。

- RuntimeToolRegistry が実行時ルーティングの唯一の正本として記述されているか
- ToolRegistry が runtime routing source と誤読される記述が残っていないか
- `tool_constants.py` が routing input と誤解される記述が残っていないか
- `/v1/tools` の response shape が現行 discovery 実装と一致しているか
- `enabled` / `disabled_reason` / `enabled_for_llm` の説明が実装と一致しているか
- `disabled_reason` が RuntimeTool の直接 field であるかのような説明が残っていないか
- Git MCP の availability metadata と disabled call response の仕様差が整理されているか
- web-search / browser_fetch の availability metadata が実装確認なしに断定されていないか
- LLM-visible と runtime-routable の違いが説明されているか
- DAG scheduling metadata の入力元が RuntimeToolRegistry と誤記されていないか
- 新規ツール追加手順が live `/v1/tools` discovery 前提になっているか
- 削除済み MCP 機能、設定、エンドポイント、旧 routing model の説明が残っていないか
- 同じ MCP routing 説明が複数章に重複していないか

## 修正方針

- routing の詳細説明は `04_mcp_03_01_dispatch-and-routing.md` に集約する。
- ToolRegistry の説明は `04_mcp_03_02_tool-registry.md` に集約する。
- `/v1/tools` と availability metadata は `04_mcp_03_06_tool-runtime-availability-metadata.md` に集約する。
- 新規ツール追加手順は `04_mcp_03_05_lifecycle-and-new-server.md` または `04_mcp_06_14_new-tool-registration-procedure.md` に集約する。
- サーバー固有の差分は各サーバーカタログ章へ記載する。
- 未解決の仕様差は `04_mcp_90_inconsistencies_and_known_issues.md` に移す。

基本方針は以下です。

1. 実装とドキュメントが矛盾する場合は、実装を正本としてドキュメントを修正する。
2. 現行仕様に影響しない過去経緯・移行履歴・旧仕様説明は削除する。
3. 実装で削除済みの機能、API、CLIコマンド、設定キー、DBカラム、型、関数、ファイルの説明は削除する。
4. 同一内容の説明が複数箇所にある場合は、正典となる1箇所に集約し、他は削除または参照リンクだけにする。
5. 判断できない場合は推測せず、必要な実装ファイル・設定ファイル・テストファイルを列挙する。
6. Known Issues に残すのは、現行仕様に影響する未解決事項だけにする。
7. 修正後は File Index、AI Query Routing Table、Related Documents、内部リンクも整合させる。
