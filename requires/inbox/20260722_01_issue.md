# Normalize MCP runtime routing and new-tool registration docs around RuntimeToolRegistry

## Priority

High

## Scope

Documentation-only, but high priority because it affects how future MCP tools and servers are implemented.

## Target documentation sections

- `04_mcp_03_01_dispatch-and-routing.md`
  - `ToolRouteResolver`
  - `ルーティングの信頼できる情報源`
  - `Tool Registry`
- `04_mcp_03_02_tool-registry.md`
  - `Tool Registry`
  - `ドリフト検証`
  - `RuntimeToolRegistry とライブ検出`
- `04_mcp_03_05_lifecycle-and-new-server.md`
  - `Adding a new tool`
  - `新しいツールを安全に追加する方法`
  - `ルーティング優先度の要約`
  - `新規サーバー/ツール登録チェックリスト`
- `04_mcp_06_14_new-tool-registration-procedure.md`

## Problem

The documentation still mixes the old static routing model with the current live-discovery routing model.

The current implementation uses `RuntimeToolRegistry` as the sole runtime routing authority. `ToolRegistry` and `tool_constants.py` still matter, but they are not runtime routing fallback sources.

This creates a serious implementation risk: a future developer may add a tool only to `tool_constants.py` or `ToolRegistry`, assume it is routable, and miss the required live `/v1/tools` discovery path.

## Current implementation behavior

The current model is:

- MCP servers expose tool metadata through `/v1/tools`.
- `McpToolDiscoveryService.discover_all()` fetches live `/v1/tools` responses.
- The discovery service builds a `RuntimeToolRegistry`.
- `startup.py` stores the registry in `ctx.services_required.runtime_tools`.
- `startup.py` also calls `ToolExecutor.set_runtime_registry(discovery.registry)`.
- `ToolRouteResolver.resolve()` uses `RuntimeToolRegistry` only.
- If a tool is missing from `RuntimeToolRegistry`, routing fails with `Unknown tool`.

## Required documentation change

Rewrite the relevant routing and registration sections so the source-of-truth model is:

- `RuntimeToolRegistry`
  - Sole runtime routing authority.
  - Populated from live `/v1/tools` discovery.

- `ToolRegistry`
  - Static ownership and drift-validation input.
  - Not a runtime routing fallback.

- `tool_constants.py`
  - Seed data for static ownership.
  - Side-effect classification and policy support.
  - Not sufficient to make a tool routable.

- `config [mcp_servers.*].tool_names`
  - Drift-validation metadata only.
  - Not a routing input.

## Required new-tool registration wording

Update the new-tool registration checklist to require all of the following:

1. Add the tool schema to the owning server's `TOOL_LIST`.
2. Ensure `/v1/tools` exposes a valid entry for the tool.
3. Add the server-side dispatch or service handler.
4. Add static ownership and policy classification in `tool_constants.py`.
5. Add or update `config/agent.toml` `tool_definitions` if the tool should be visible to the LLM.
6. Add safety-tier and approval-policy entries.
7. Verify startup discovery registers the tool in `RuntimeToolRegistry`.
8. Treat `tool_names` as drift-validation metadata, not routing data.

## Reason for change

The documentation must prevent future implementation mistakes. Runtime routing is no longer based on static registry fallback. A tool must be discovered live and inserted into `RuntimeToolRegistry` to be executable.

## Acceptance criteria

- No documentation section states or implies that `ToolRegistry` is the runtime routing fallback.
- New-tool registration docs clearly state that `/v1/tools` discovery is required for runtime routing.
- The docs distinguish runtime routing, static ownership, policy classification, LLM visibility, and drift validation.
- The failure mode "tool exists in static data but is not in RuntimeToolRegistry" is explicitly documented.
