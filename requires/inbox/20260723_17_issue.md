# Migrate All Area Known Issues Documents to the Common Template

## Priority
High

## Summary
Migrate all area-specific Known Issues documents to the common template defined in `00_governance_04_known-issues-template.md`.

This task covers:
- `05_agent_90_inconsistencies_and_known_issues.md`
- `04_mcp_90_inconsistencies_and_known_issues.md`
- `03_rag_90_inconsistencies_and_known_issues.md`
- `06_eventbus_90_inconsistencies_and_known_issues.md`
- `90_shared_90_inconsistencies_and_known_issues.md`

## Background
The governance documentation already defines a common Known Issues template with required fields, status values, type values, severity values, area values, and lifecycle rules.

A migration plan also exists for moving area Known Issues documents toward the common template.

The suggested migration order prioritizes Agent and MCP first, followed by RAG, EventBus, and Shared/DB.

## Problem
Each area currently uses a different Known Issues format.

This creates several problems:
- cross-area comparison is difficult,
- issue severity and status are inconsistent,
- AI-generated implementation tasks are harder to derive,
- unresolved documentation/code mismatches are harder to track consistently.

## Rationale
Known Issues documents are central to documentation quality and implementation planning.

Because the common template already exists, the right next step is not to design another format, but to migrate existing area documents to the shared structure.

Agent and MCP should be handled first because they have strong cross-area dependencies.

RAG, EventBus, and Shared/DB should be migrated in the same task if feasible, because the operation is repetitive and template-driven.

## Intent
The intent is to normalize issue tracking format across all major areas without changing the technical meaning of any existing issue.

This is a formatting and governance-alignment task, not a bug-fixing task.

## Required Changes

For each target Known Issues document:

1. Preserve all existing entries.
2. Preserve existing IDs, or explicitly map old IDs to new IDs.
3. Preserve the original technical meaning.
4. Convert each entry to the common template fields.
5. Add missing fields where possible.
6. Add `Status`, `Severity`, `Type`, `Area`, and `Owner`.
7. Use `未確認` where a required field cannot be confidently filled.
8. Add a migration note with migration date and source format.
9. Preserve links to related governance documents.

Recommended order:
1. Agent
2. MCP
3. RAG
4. EventBus
5. Shared/DB

## Acceptance Criteria
- All five target Known Issues documents are migrated or explicitly marked as partially migrated with remaining work listed.
- All existing entries remain present.
- Existing IDs are preserved or explicitly mapped.
- No technical meaning is changed.
- No issue is resolved as part of migration.
- No new issue is added unless needed only to record migration status.
- Each migrated entry follows the common template as closely as possible.
- Missing fields are filled with `未確認` rather than guessed.
- Markdown renders correctly.

## Definition of Done
- Documentation is written in Japanese.
- All migrated documents link to `00_governance_04_known-issues-template.md`.
- Migration notes are added.
- No source code is modified.
- No issue resolution work is performed.
- No new governance model is introduced.
- Existing historical context is preserved.

## Out of Scope
- Do not fix any Known Issues.
- Do not add new technical Known Issues.
- Do not modify source code.
- Do not change the common template itself.
- Do not introduce ADRs.
- Do not change the substance of existing entries.

## AI Implementation Instruction
Use `00_governance_04_known-issues-template.md` as the target format.

Proceed area by area in the recommended order.

If the full migration is too large for one pass, complete Agent and MCP first, then leave a clear migration status section for the remaining area files.

Do not alter the technical meaning of any issue.
