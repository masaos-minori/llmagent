# Add diagnosis docs for discovery failure and LLM-visible but unroutable tools

## Priority

Medium

## Scope

Documentation-only.

## Target documentation sections

- `04_mcp_06_09_mcp-failure-diagnosis.md`
- `04_mcp_03_01_dispatch-and-routing.md`
- `04_mcp_06_11_startup-validation-behavior-tool_definitions_strict.md`

## Problem

A tool can be visible to the LLM but still fail at execution time with `Unknown tool`.

This can happen if `RuntimeToolRegistry` is missing or incomplete. In that case, LLM tool payload filtering may not remove the tool, but runtime routing still cannot resolve it.

## Required documentation changes

Add a diagnosis section for:

`LLM called a tool, but execution failed with Unknown tool`

Recommended diagnosis steps:

1. Check startup output for `mcp_tool_discovery` outcomes.
2. Check whether discovery was `FATAL`, `WARNING`, or `SKIPPED`.
3. Check whether the owning server's `/v1/tools` response includes the tool.
4. Check whether duplicate tool name detection excluded the tool.
5. Check whether `ctx.services_required.runtime_tools` was populated.
6. Check whether `ToolExecutor.set_runtime_registry()` was called.
7. Check whether the tool exists in `RuntimeToolRegistry`.
8. Do not rely only on `config/agent.toml` `tool_definitions`.

## Required startup status clarification

Document that in local mode, MCP discovery failure may appear as `SKIPPED`, but it can still mean all tool calls will fail for that session.

## Reason for change

This failure mode is confusing because it looks like an LLM or approval problem, but the true cause is missing runtime routing metadata.

## Acceptance criteria

- Failure diagnosis includes the LLM-visible but unroutable case.
- Docs explain why configured tool definitions alone are insufficient.
- Startup discovery outcomes are linked to runtime routing failures.
- `SKIPPED` discovery in local mode is described as operationally high impact.
