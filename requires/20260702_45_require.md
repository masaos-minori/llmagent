# Add ChunkSplitter-to-DB field mapping documentation

## Priority

Medium

## Summary

Document how chunk JSON fields produced by `ChunkSplitter` map to DB tables and columns.

## Background

`ChunkSplitter.py` outputs fields such as:

- `url`
- `title`
- `lang`
- `source_file`
- `chunk_index`
- `chunk_type`
- `content`

`pipeline_utils.py` reads these fields.

The DB schema should clearly document which fields are persisted and where.

## Goal

Make ingestion field ownership and persistence explicit.

## Target files

- `90_shared_04_db_architecture_and_schema.md`
- Optionally `ingester.py`
- Optionally `pipeline_utils.py`

## Scope

### In scope

- RAG ingestion input field mapping.
- `documents` table mapping.
- `chunks` table mapping.
- Unknown or non-persisted fields.

### Out of scope

- Changing ingestion behavior unless mapping reveals a mismatch.

## Implementation tasks

Document these mappings:

- `url` -> `documents.url`
- `title` -> `documents.title`
- `lang` -> `documents.lang`
- `etag` -> `documents.etag`
- `last_modified` -> `documents.last_modified`
- `chunking_strategy` -> `documents.chunking_strategy`
- `chunk_index` -> `chunks.chunk_index`
- `content` -> `chunks.content`
- `normalized_content` -> `chunks.normalized_content`
- `chunk_type` -> `chunks.chunk_type`
- `source_file` -> `chunks.source_file`
- `artifact_type` -> not persisted or TBD
- `fetched_at` -> not persisted or TBD

## Acceptance criteria

- Every `ChunkJsonRaw` field is either mapped to a DB column or explicitly marked as not persisted or TBD.
- `chunk_type` and `source_file` are mapped to `chunks`.
- `artifact_type` status is explicitly documented.
- `fetched_at` status is explicitly documented.

## Validation

- Review `ChunkJsonRaw` in `pipeline_utils.py`.
- Review `ChunkSplitter` output contract.
- Review `schema_sql.py` DDL.

## Confirmation required

- Confirm whether `artifact_type` should be persisted.
- Confirm whether input `fetched_at` should be persisted or ignored.
