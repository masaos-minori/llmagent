# Clarify plugin registry repeated-load behavior and test isolation

## Priority

Medium

## Labels

plugin, tests, lifecycle, documentation

## Background

Plugin registry loading is cumulative.

Calling `load_plugins()` more than once does not clear existing registrations. Duplicate command or tool names registered by later loads silently overwrite earlier entries.

This is acceptable for process-start loading, but it can cause test pollution or future hot-reload issues if not clearly documented and tested.

## Goal

Clarify and enforce safe plugin registry usage across runtime and tests.

## Scope

### In scope

- Document that plugin loading is intended to happen once per process startup
- Document that repeated loads accumulate registrations
- Require `_reset_for_testing()` in tests that load plugins
- Add tests if missing

### Out of scope

- Implementing plugin hot reload
- Changing production plugin loading semantics
- Changing plugin conflict policy

## Required documentation

Document:

- `load_plugins()` is intended for startup use
- Repeated calls accumulate registrations
- `_reset_for_testing()` is the only supported way to clear registries in tests
- Non-test code must not call `_reset_for_testing()`

## Acceptance criteria

- Test documentation explains plugin registry isolation
- Tests that load plugins call `_reset_for_testing()`
- No production code calls `_reset_for_testing()`
- Existing plugin load behavior is preserved

## Validation

Run plugin-related tests repeatedly in the same process if possible.

Expected result:

- Tests do not leak plugin registrations across test cases
- Plugin load result remains deterministic

## Affected files

- `scripts/shared/plugin_registry.py`
- Plugin tests
- Plugin documentation

## Dependencies

None
