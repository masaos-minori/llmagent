# Title
Clarify the contract between SQLite persistence and JSONL append in `/publish`

## Summary
Make the persistence contract explicit when SQLite commit succeeds but JSONL append may fail afterward.

## Background
`/publish` commits the event into SQLite first and then appends the event to `storage/events.jsonl`.
The implementation plan describes SQLite as the trusted primary store, but the runtime error contract of JSONL append is not clearly documented.

## Problem
If JSONL append fails after DB commit, the system may have already persisted the event while the API path appears to fail or behave inconsistently.

## Required Changes
- Document that SQLite is the source of truth if that is the intended model.
- Decide whether JSONL append failure is:
  - best-effort warning only, or
  - part of publish failure semantics.
- Align exception handling, logging, and API behavior to that decision.
- Add tests for “SQLite committed but JSONL append failed”.

## Acceptance Criteria
- The system clearly defines which store is authoritative.
- JSONL append failure behavior is predictable and documented.
- Publish success/failure semantics are understandable to operators.

## Out of Scope
- Removing JSONL support entirely
- Replacing the storage model wholesale

- `scripts/eventbus/app.py`
