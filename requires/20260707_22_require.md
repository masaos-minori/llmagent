# H-2: Inject one shared MCP health registry into ToolExecutor and AppServices

## Priority

High

## Background

`ToolExecutor` supports `set_health_registry()` and uses the registry for:

- dispatch gating
- recording successful transport calls
- recording transport failures

`AppServices` also has a `health_registry` field, and `/mcp status` reads from `ctx.services_required.health_registry`.

Currently, the same `McpServerHealthRegistry` instance must be injected into both places so dispatch behavior and status reporting remain consistent.

## Target files

- `agent/factory.py`
- `agent/context.py` if needed

## Implementation tasks

- Create one `McpServerHealthRegistry` instance during service construction.
- Inject the registry into `ToolExecutor` with `tools.set_health_registry(health_registry)`.
- Pass the same registry to `AppServices(..., health_registry=health_registry, ...)`.
- Ensure `/mcp status` and `ToolExecutor` observe the same server health state.
- Keep the registry optional only for test or intentionally degraded construction paths.

## Acceptance criteria

- `ctx.services_required.health_registry` is non-null after `build_agent_context()`.
- `ToolExecutor._health_registry` uses the same object as `ctx.services_required.health_registry`.
- Transport failures recorded by `ToolExecutor` are visible in `/mcp status`.
- Successful calls reset the same registry used by status display.

## Test plan

- Unit-test `build_agent_context()` and assert the same registry instance is wired into `ToolExecutor` and `AppServices`.
- Simulate `ToolExecutor._record_transport_error()` and verify `/mcp status` reads the updated state.
- Simulate `ToolExecutor._record_success()` and verify the registry state returns to healthy.
