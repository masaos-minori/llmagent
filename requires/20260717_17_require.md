# Update RuntimeToolRegistry to handle disabled MCP tools

## Priority

Medium

## Target files

- shared/runtime_tool.py
- shared/runtime_tool_registry.py
- agent/services/mcp_tool_discovery.py
- agent/llm_turn_runner.py
- agent/commands/cmd_mcp.py
- tests/shared/test_runtime_tool_registry.py
- tests/agent/services/test_mcp_tool_discovery.py
- tests/agent/commands/test_cmd_mcp.py

## Reason

MCP servers will return all tools from `/v1/tools`, including disabled tools.

RuntimeToolRegistry must preserve disabled tools for diagnostics, but must not expose them to the LLM or allow them to execute.

This separates:

- discovered tools
- MCP-server-enabled tools
- Agent-policy-enabled tools
- LLM-visible tools

## Intent

Represent disabled MCP tools as discovered but not executable and not LLM-visible.

## Scope

When a discovered tool has `enabled: false`, RuntimeToolRegistry should:

- register the tool
- preserve `disabled_reason`
- exclude it from LLM tool definitions
- reject execution attempts
- expose it in diagnostics

Update MCP diagnostics to display:

- tool name
- server key
- config_dependent
- enabled
- disabled_reason
- enabled_for_llm

## Acceptance criteria

- Disabled tools are present in RuntimeToolRegistry.
- Disabled tools are absent from LLM-visible tool definitions.
- Disabled tools cannot be executed through RuntimeToolRegistry routing.
- RuntimeToolRegistry preserves `disabled_reason`.
- `/mcp status` or equivalent diagnostics show disabled tools and reasons.
- Enabled tools show empty disabled reason.
