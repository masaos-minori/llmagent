# [High] Align HttpTransport error-handling documentation with the current implementation

## Goal

Update MCP documentation to match the current `HttpTransport` and `ToolExecutor` error boundary.

## Background

The current implementation uses `TransportError` for transport-level failures.

Current implementation behavior:

- `HttpTransport.call()` returns `ToolCallResult` for valid HTTP 200 responses.
- HTTP 200 with `is_error=true` is a tool-level error.
- Timeouts, HTTP status failures, request errors, malformed responses, and retry exhaustion raise `TransportError`.
- `ToolExecutor._raw_execute()` catches `TransportError`.
- `ToolExecutor._record_transport_error()` converts it into `ToolCallResult(error_type="transport")`.

Some MCP docs still imply that `HttpTransport` catches all HTTP/request errors and returns `is_error=True` directly. That wording is incorrect.

## Scope

Documentation and tests.

Update:

- `04_mcp_02_protocol_and_transport.md`
- `04_mcp_03_routing_lifecycle_and_execution.md`
- `04_mcp_06_configuration_and_operations.md`
- `04_mcp_90_inconsistencies_and_known_issues.md`
- `tests/test_tool_executor.py`

Do not change `HttpTransport` behavior unless tests reveal an implementation mismatch.

## Required Error Model

Tool-level error:

- server responded
- HTTP 200
- response has `is_error=true`
- `HealthRegistry.record_success()`
- `stat_tool_errors` increments
- `error_type="tool"`

Transport-level error:

- server unreachable, timeout, HTTP non-success, malformed response, retry exhausted
- `HttpTransport` raises `TransportError`
- `ToolExecutor` catches it
- `HealthRegistry.record_failure()`
- `stat_transport_errors` increments
- returned `ToolCallResult.error_type == "transport"`

## Implementation Tasks

- Remove documentation that says `HttpTransport` returns `is_error=True` for transport errors.
- Document that `HttpTransport` raises `TransportError`.
- Document that `ToolExecutor` converts `TransportError` to `ToolCallResult(error_type="transport")`.
- Document that tool-level errors still count as transport success.
- Add or strengthen tests in `tests/test_tool_executor.py`.

## Acceptance Criteria

- MCP docs consistently describe the `TransportError` boundary.
- Tool-level and transport-level errors are clearly distinguished.
- HealthRegistry behavior is documented correctly.
- Tests cover both error classes.

## Validation

Run:

    uv run pytest tests/test_tool_executor.py -v

Expected test coverage:

- HTTP 200 + `is_error=true` increments tool error counters.
- HTTP 500 raises `TransportError` internally and increments transport error counters.
- timeout is classified as transport error.
- malformed response is classified as transport error.
- retryable HTTP 503 exhausts retries and becomes transport error.
- tool-level error calls `record_success`.
- transport error calls `record_failure`.
