# Title
Update Agent routing to explicitly separate MDQ and RAG usage

## Labels
agent
routing
mdq
rag
integration
design

## Priority
High

## Dependencies
- Formalize the MDQ / RAG responsibility boundary
- Reflect the MDQ / RAG responsibility split in MCP tool descriptions and docs
- Enforce MDQ / RAG data ownership boundaries with automated checks (recommended)

## Components
- Agent routing policy
- Tool selection logic
- Query classifier
- Explicit MDQ / RAG override handling

## Summary
Implement an Agent-side routing policy that chooses MDQ or RAG intentionally based on query type and system responsibility.

## Background
The intended split is:
- MDQ for Markdown structural queries
- RAG for general-purpose, semantic, production retrieval

Without Agent-side routing policy, the architectural split may exist only on paper while actual runtime behavior remains mixed and inconsistent.

## Problem
The Agent needs a predictable way to distinguish:
- Markdown structure-centric queries
- general retrieval requests
- semantic retrieval needs
- production-oriented search requests

Because MDQ is experimental and RAG is the production retrieval path, fallback and override behavior also need to be explicit.

## Required Changes
- Define a routing policy such as:
  - use MDQ for Markdown-only, heading/outline/section/hierarchy-oriented queries
  - use RAG for multi-format, semantic, general-purpose, or production retrieval queries
- Add a lightweight query classifier to support this routing decision.
- Add an explicit override path so higher-level logic or users can force:
  - `mode=mdq`
  - `mode=rag`
- Define and document fallback behavior if MDQ is selected but unavailable or unsuitable.

## Acceptance Criteria
- Markdown structural queries route to MDQ by default.
- General or semantic retrieval routes to RAG by default.
- Explicit override is supported.
- Routing rules and fallback behavior are documented clearly.

## Out of Scope
- Improving search quality itself
- Implementing automatic migration from MDQ to RAG

## AI Implementation Instruction
Implement this as a routing-policy and integration task, not as a search-engine rewrite.

Strict rules:
- Do not merge MDQ and RAG into one undifferentiated path.
- Do not add direct DB access as a shortcut.
- Preserve the architectural split:
  - MDQ for Markdown structural queries
  - RAG for general-purpose and semantic retrieval
- Keep classifier logic lightweight, explicit, and explainable.
- Add an override mechanism so humans or higher-level logic can force MDQ or RAG when needed.
- Do not rely only on heuristic guessing if an explicit override is available.

Routing priorities:
1. correctness of responsibility boundary
2. predictability of behavior
3. safe and documented fallback behavior

If fallback policy is ambiguous:
- do not invent a hidden fallback chain,
- document the gap,
- mark it as: Needs confirmation

Deliverables:
1. Routing policy implementation
2. Explicit MDQ / RAG override path
3. Documentation of routing rules and fallback behavior
