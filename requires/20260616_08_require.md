# [MCP][Health] Expand /health to include readiness and dependency state

## Summary
Current `/health` responses are mostly liveness-only and do not show whether dependencies or runtime prerequisites are actually ready.

## Problem
A server can be alive but still unusable because tokens, DB paths, indexes, or external dependencies are missing or broken.

## Scope
- `mcp/server.py`
- per-server health overrides

## Task
- expand `/health` to distinguish liveness from readiness
- include dependency state where relevant
- standardize the response shape across servers

## Suggested Fields
- `status`
- `ready`
- `dependencies`
- `details`

## Acceptance Criteria
- [ ] `/health` can distinguish running vs ready
- [ ] dependency state is included for key servers
- [ ] response format is reasonably consistent across MCP servers
