# [High] Clean up or formalize stale MCP transport and startup config fields

## Goal

Resolve stale or unsupported MCP configuration fields in `McpServerConfig`.

## Background

`McpServerConfig` still includes:

- `TransportType.STDIO`
- `StartupMode.ONDEMAND`
- `cmd`
- `env`

However, active MCP docs describe HTTP-based `persistent` and `subprocess` modes as the supported startup model. Stdio and ondemand behavior are not fully documented as active runtime modes.

Additionally, `McpServerConfig` validates that `cmd` must be non-empty when `startup_mode="subprocess"`, but `_build_single_server()` does not currently pass `cmd` or `env` into `McpServerConfig`. This can make valid subprocess configs fail validation.

## Scope

Update:

- `shared/mcp_config.py`
- `config/mcp_servers.toml`
- `tests/test_mcp_config.py`
- MCP docs:
  - `04_mcp_01_system_overview.md`
  - `04_mcp_02_protocol_and_transport.md`
  - `04_mcp_03_routing_lifecycle_and_execution.md`
  - `04_mcp_05_security_and_safety_model.md`
  - `04_mcp_06_configuration_and_operations.md`

## Required Decision

Choose and implement one of the following:

### Option A: HTTP-only cleanup

- Remove `TransportType.STDIO`.
- Remove stdio references from docs.
- Remove stdio auth exemptions.
- Remove stdio timeout documentation.

### Option B: Formal stdio support

- Keep `TransportType.STDIO`.
- Document stdio protocol and lifecycle.
- Add tests for stdio routing and execution.
- Ensure security profile behavior is defined.

Preferred option: Option A, unless stdio is actively supported.

Also choose one of the following for `StartupMode.ONDEMAND`:

### Option A: Remove ONDEMAND

- Remove enum value.
- Remove stale docs if any.

### Option B: Formalize ONDEMAND

- Implement lifecycle behavior.
- Document startup semantics.
- Add tests.

Preferred option: Option A, unless ondemand is actively supported.

## Implementation Tasks

- Decide whether stdio is active or removed.
- Decide whether ondemand is active or removed.
- If subprocess mode remains supported:
  - parse `cmd` from config
  - parse `env` from config
  - pass both to `McpServerConfig`
  - document `cmd` and `env`
- Add config tests for:
  - HTTP persistent
  - HTTP subprocess with cmd/env
  - subprocess without cmd should fail
  - unsupported transport should fail
  - unsupported startup mode should fail
- Update docs to match the final supported configuration surface.

## Acceptance Criteria

- Active docs and `McpServerConfig` expose the same supported modes.
- No unsupported stdio or ondemand behavior appears as active guidance.
- Subprocess config validation works correctly.
- `_build_single_server()` handles all active fields.
- Tests cover config parsing and validation.

## Validation

Run:

    uv run pytest tests/test_mcp_config.py -v

Also run:

    rg -n "stdio|STDIO|ondemand|ONDEMAND|cmd|env" shared config docs tests

Remaining references must be intentional and documented.
