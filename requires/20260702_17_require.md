# [Medium] Clarify and optionally add strict artifact validation for RAG ingestion

## Priority
Medium

## Background
The RAG artifact format defines `schema_version`, `artifact_type`, and `created_by`, but `RagIngester._validate_artifact()` is documented as lenient for backward compatibility when `artifact_type` is missing.

Evidence:
- Ingestion docs define JSON fields such as `schema_version`, `artifact_type`, and `created_by`.
- Data model docs define crawler/chunk JSON formats with metadata fields.
- Ingestion docs describe `_validate_artifact()` as lenient for backward compatibility.

## Goal
Clarify current compatibility behavior and prepare an optional strict mode.

## Scope
Affected files:
- `scripts/rag/ingestion/ingester.py`
- `scripts/rag/ingestion/pipeline_utils.py`
- `docs/03_rag_02_ingestion_pipeline.md`
- `docs/03_rag_04_data_model_and_interfaces.md`
- `docs/03_rag_05_configuration_and_operations.md`

## Implementation Tasks
- Inspect `_validate_artifact()` implementation.
- Document legacy acceptance rules explicitly.
- Add `strict_artifact_validation` design if needed.
- In strict mode, reject missing:
  - `schema_version`
  - `artifact_type`
  - `created_by`
- Add validation failure logs with path, expected type, actual type, and missing fields.

## Open Questions / Unknowns
- Need full `_validate_artifact()` implementation.
- Need to decide default value for `strict_artifact_validation`.
- Need migration guidance for legacy artifact files.

## Acceptance Criteria
- Current lenient behavior is documented accurately.
- Strict behavior is either implemented or explicitly deferred.
- New artifact metadata requirements are clear.

## Validation
- Run ingestion tests for both legacy and metadata-complete artifacts.
