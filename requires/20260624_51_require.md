# Route workflow schema initialization through the shared SQLiteHelper policy path

## Summary
Refactor workflow schema initialization so it uses the same DB-opening policy model as the rest of the DB layer.

## Background
`workflow_schema.py` currently opens SQLite directly via `sqlite3.connect()`, while the rest of the DB layer centralizes connection setup in `SQLiteHelper`. This makes workflow schema initialization an exception to the normal DB policy path.

## Objective
Apply consistent connection-policy handling across workflow schema setup.

## In Scope
- `workflow_schema.py`
- `helper.py` if helper support needs minor extension

## Required Changes
- Replace or encapsulate direct `sqlite3.connect()` usage behind the shared helper path
- Ensure workflow schema init benefits from the same connection-policy expectations where appropriate
- Preserve current DDL behavior and migration behavior unless explicitly refactored together

## Acceptance Criteria
- Workflow DB initialization no longer bypasses the shared connection-management path without reason
- Connection-policy handling is more consistent across all DB setup operations
- Workflow initialization remains idempotent

## Out of Scope
- Full migration framework work
- Redesigning workflow schema

## Implementation Notes
Consistency is the main goal. Reuse the common DB-opening path unless there is a documented reason not to.
