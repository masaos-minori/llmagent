## [High][eventbus][api] Add consumer outcome APIs for ack, nack, and failure reporting

### Summary
The current Event Bus can stream events to consumers, but it has no protocol for the consumer to report success or failure back to the bus.

### Problem
Without a consumer outcome API, the Event Bus cannot detect whether an event was processed successfully, failed, or timed out.
This prevents reliable retry, timeout handling, and DLQ promotion.

### Scope
- app.py
- subscribe flow
- delivery lifecycle
- consumer feedback protocol
- schema and API design

### Task
- design and add a consumer outcome API such as `/ack`, `/nack`, or `/fail`
- make the Event Bus record delivery outcomes explicitly
- use the outcome signal to drive retry and DLQ logic
- define the request/response schema for the new API
- add tests for ack/nack/failure reporting

### Expected Result
The Event Bus can observe consumer success and failure explicitly instead of inferring state from stream position only.

### Acceptance Criteria
- there is a supported API path for consumer success reporting
- there is a supported API path for consumer failure reporting
- delivery outcome is persisted or otherwise reflected in retry/DLQ logic
- tests cover the new consumer outcome flow
