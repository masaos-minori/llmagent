# Title
`deploy/init_db.sh` への SQL 統合と `docs/02_deployment.md` の実装整合リファクタリング

## Summary
DB 初期化用 SQL と sqlite-vec ビルド手順をドキュメントから実行スクリプトへ集約し、`docs/02_deployment.md` を現行実装に合わせて整理・更新する。

## Background
現在、DB 初期化や sqlite-vec ビルドに関する情報が、以下のように複数の場所へ分散している。

- `db/rrf.sql`
- `scripts/eventbus/schema.sql`
- `deploy/init_db.sh`
- `deploy/build_sqlite_vec.sh`
- `docs/02_deployment.md`
- `docs/03_rag_04_data_model_and_interfaces.md`

この状態だと、実行スクリプトとドキュメントの責務が曖昧で、将来的に内容の不整合が起きやすい。

## Problem
現状には、以下の問題がある。

- DB 初期化手順が standalone SQL ファイルと shell script に分散している
- sqlite-vec ビルド手順が実行スクリプトではなくドキュメントに依存している
- `docs/02_deployment.md` に古い内容・重複内容・実装とずれた説明が残っている
- スキーマ概要がデプロイ文書とデータモデル文書にまたがっており、責務が分散している
- モデル取得に関する説明が現在の運用・実装方針と一致していない

## AI Implementation Instruction
これは **デプロイ資産の集約とドキュメント整合作業** として実装すること。

### Strict rules
- 実行可能な手順は、可能な限り shell script 側を正本とすること
- SQL を standalone ファイルに残したまま、`deploy/init_db.sh` にも重複保持しないこと
- `db/` ディレクトリ自体は削除しないこと
- `docs/02_deployment.md` には、現行実装に存在しない手順や古い説明を残さないこと
- ドキュメント更新時に、歴史的経緯ではなく **現在の実装フロー** を優先して記述すること
- `docs/03_rag_04_data_model_and_interfaces.md` へ移管する内容は、単純削除ではなく、移管後に文脈が通るように統合すること
- 実装と文書が食い違う箇所があれば、推測で埋めずに `Needs confirmation` を明記すること

### Deliverables
1. `deploy/init_db.sh` へ統合された SQL 初期化処理
2. `deploy/build_sqlite_vec.sh` へ統合された sqlite-vec ビルド手順
3. 更新済みの `docs/02_deployment.md`
4. 更新済みの `docs/03_rag_04_data_model_and_interfaces.md`
5. 削除済みの `db/rrf.sql`
6. 削除済みの `scripts/eventbus/schema.sql`
7. 変更サマリ（何を統合・削除・移管したか）
