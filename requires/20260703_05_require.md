# Align plugin prefix command boundary matching with built-in commands

## Priority

Medium

## Labels

bug, plugin, cli, command-registry

## Background

Built-in commands use boundary-aware prefix matching:

- `line == cmd.name`
- `line.startswith(cmd.name + " ")`

This prevents false positives such as matching `/foo` against `/foobar`.

Plugin prefix commands currently use a looser prefix check. This may incorrectly dispatch a plugin command registered as `/foo` when the input is `/foobar`.

## Goal

Make plugin command prefix matching use the same boundary rule as built-in commands.

## Scope

### In scope

- Update plugin dispatch matching in `CommandRegistry._dispatch_plugin()`
- Normalize plugin args with `.strip()`
- Preserve built-in command priority over plugin commands

### Out of scope

- Plugin command shadowing policy
- Plugin reload behavior
- Built-in command dispatch behavior

## Required behavior

For a plugin command named `/foo` with `prefix=True`:

- `/foo` should match
- `/foo bar` should match and pass `bar`
- `/foobar` should not match

## Suggested implementation

Use boundary matching:

- `line == cmd_name or line.startswith(cmd_name + " ")`

Then compute args with:

- `args = line[len(cmd_name):].strip()`

## Acceptance criteria

- Plugin prefix commands do not match longer unrelated command names
- Plugin exact commands continue to match only exact input
- Built-in commands still take precedence over plugin commands
- Existing plugin tests pass

## Validation

Add or update tests with plugin commands:

- Register `/foo` with `prefix=True`
- Verify `/foo bar` dispatches
- Verify `/foobar` does not dispatch as `/foo`

## Affected files

- `scripts/agent/commands/registry.py`
- Plugin command tests

## Dependencies

None
