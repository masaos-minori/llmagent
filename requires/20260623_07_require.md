## Title: [LOW] Clean Up Layering Violations in `db/maintenance.py`

### Context
`db/maintenance.py` is primarily responsible for `session.sqlite` maintenance. However, it currently contains RAG-specific file utility functions like `rotate_rag_db` and `vacuum_db`, although it has zero explicit `rag/` module imports.

### Problem Statement
Having RAG maintenance logic mixed inside the session storage utility module breaks domain boundaries and clean layering principles.

### Requirements
- **Refactor Domain Logic**: Move `rotate_rag_db` and `vacuum_db` out of `db/maintenance.py` and relocate them into `RagMaintenanceService` or a dedicated shared storage utility layer.
- **Clean Interface Boundary**: Ensure `db/maintenance.py` interacts exclusively with session-scoped storage concerns, keeping the RAG layer decoupled.

### Constraints
- Do NOT introduce circular dependencies between the agent layer and the RAG layer.

### Acceptance Criteria (Definition of Done)
- [ ] `db/maintenance.py` contains zero functions handling `rag.sqlite`.
- [ ] All commands routed via the REPL (`/db rag` operations) continue to execute successfully without regressions.
