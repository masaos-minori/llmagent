# Add DB recreation procedure to operations documentation

## Priority

Medium

## Summary

Document the standard procedure for applying schema changes by recreating DB files.

## Background

The project no longer supports existing DB-compatible schema migration.

Schema changes are applied by recreating DB files from the latest DDL.

`maintenance.py` includes `rotate_all_dbs()` for archiving all DBs.

## Goal

Provide a clear operational workflow for schema updates.

## Target files

- `90_shared_05_db_api_and_operations.md`
- `90_shared_04_db_architecture_and_schema.md`

## Scope

### In scope

- Archive existing DB files.
- Remove DB files.
- Recreate schema.
- Clarify that data migration is out of scope.

### Out of scope

- Automated migration.
- Data export/import tooling.

## Implementation tasks

- Add a DB recreation section.
- Recommend archiving existing DBs before deletion.
- Include all three DB files:
  - `rag.sqlite`
  - `session.sqlite`
  - `workflow.sqlite`
- State that recreated DBs are empty and use the latest DDL.
- State that existing records are not migrated automatically.

## Suggested operation flow

- Run archive:
```bash
python -c "from db.maintenance import rotate_all_dbs; rotate_all_dbs()"
```
- Remove existing DB files.

- Recreate schema:

    python -c "from db.create_schema import create_schema; create_schema()"

## Acceptance criteria

- Operations docs describe DB recreation as the official schema update method.
- Docs explicitly state that automatic migration is unsupported.
- Docs recommend `rotate_all_dbs()` before DB recreation.

## Validation

- Review docs for consistency with `create_schema.py`.
- Confirm `rotate_all_dbs()` covers all three DB files.

## Confirmation required

- Confirm whether DB paths should be hardcoded in docs or described via `common.toml`.
