# H-8: Stop HTTP MCP subprocesses by process group

## Priority

High

## Background

HTTP MCP subprocesses are currently started without `start_new_session=True`, and shutdown uses PID-level `terminate()` and `kill()`. If an MCP server launches child processes, PID-only termination can leave orphaned children.

## Target files

- `agent/http_lifecycle.py`

## Implementation tasks

- Start subprocesses with `start_new_session=True`.
- Capture or calculate `pgid` after process creation.
- Update termination logic to send `SIGTERM` to the process group.
- If the process does not exit before timeout, send `SIGKILL` to the process group.
- Fall back to `proc.terminate()` and `proc.kill()` when process group signaling fails.
- Treat `ProcessLookupError` as already stopped.

## Acceptance criteria

- Subprocesses are created in their own process group.
- Shutdown sends signals to the process group where possible.
- Fallback PID-level termination still works.
- Child processes are not left behind during normal shutdown.

## Test plan

- Unit-test process group kill path with mocks.
- Unit-test fallback path when `os.killpg()` raises.
- Integration-test with a subprocess that spawns a child process, if feasible.
