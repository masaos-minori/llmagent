# Title
Standardize cicd-mcp workflow_allowlist as fail-closed

## Priority
High

## Summary
Standardize `workflow_allowlist` behavior for cicd-mcp as fail-closed: an empty list must deny all workflow trigger requests.

## Background
The server catalog describes `workflow_allowlist` as fail-open in one place, while the security model describes it as fail-closed. CI/CD workflow triggering is a dangerous write operation.

## Problem
Ambiguous fail-open/fail-closed behavior can lead to unintended GitHub Actions workflow execution or unexpected operational lockout.

## Required Changes
- Inspect cicd-mcp implementation.
- Confirm behavior when `workflow_allowlist=[]`.
- Enforce fail-closed behavior:
  - empty list denies all `trigger_workflow` requests.
- Decide production startup behavior:
  - either RuntimeError on unsafe config, or
  - startup warning plus runtime deny.
- Use one consistent behavior across code and docs.
- Clarify dry-run behavior and whether allowlist checks happen before or after dry-run.
- Remove fail-open wording from docs.
- Add tests for:
  - empty allowlist
  - allowed workflow
  - disallowed workflow
  - dry_run with allowed/disallowed workflow

## Acceptance Criteria
- Empty `workflow_allowlist` never allows workflow triggers.
- Server catalog, security model, and operations docs describe the same policy.
- Production preflight cannot silently permit unrestricted workflow triggering.
- Tests verify fail-closed behavior.

## Out of Scope
- Adding support for non-GitHub CI backends.
- Changing `repo_allowlist` semantics.
