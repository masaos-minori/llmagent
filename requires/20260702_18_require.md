# [Medium] Confirm and document semantic cache invalidation after ingestion

## Priority
Medium

## Background
`RagIngester.ingest_all()` accepts `on_ingest_complete`, which can invalidate the semantic cache. `SemanticCache` has `invalidate()` and `generation`, but the `CacheService` Protocol does not include those members.

Evidence:
- `RagIngester.ingest_all()` is documented with `on_ingest_complete`.
- `SemanticCache` exposes `invalidate()` and `generation`; `CacheService` does not include them.

## Goal
Ensure stale RAG context is not served from semantic cache after ingestion.

## Scope
Affected files:
- `scripts/rag/cache.py`
- `scripts/rag/pipeline.py`
- `scripts/rag/ingestion/ingester.py`
- `scripts/mcp/rag_pipeline/service.py`
- `docs/03_rag_03_query_pipeline.md`
- `docs/03_rag_04_data_model_and_interfaces.md`

## Implementation Tasks
- Confirm whether `on_ingest_complete` is used by CLI ingestion.
- Confirm whether MCP ingestion invalidates the pipeline cache.
- Add or update tests for cache invalidation after ingestion.
- Decide whether `CacheService` Protocol should include:
  - `invalidate()`
  - `generation`
- Add debug output for cache generation if useful.

## Open Questions / Unknowns
- Need `scripts/rag/cache.py` full implementation.
- Need `scripts/mcp/rag_pipeline/service.py` ingestion/maintenance paths.
- Need to identify whether multiple RagPipeline instances can exist and hold separate caches.

## Acceptance Criteria
- Ingestion invalidates relevant semantic caches.
- Cache invalidation behavior is documented.
- Tests prevent stale context after re-ingestion.

## Validation
- Run semantic cache tests.
- Manually ingest changed content and verify repeated query does not return stale context.
