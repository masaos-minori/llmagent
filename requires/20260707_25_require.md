# H-5: Convert lifecycle ensure_ready failures into ToolCallResult errors

## Priority

High

## Background

After `ensure_ready()` starts missing subprocess servers, it may raise startup or OS errors. `ToolExecutor._raw_execute()` currently catches `TransportError` from the transport call, but lifecycle failures can occur before the transport call.

Lifecycle startup failures should be returned as structured tool errors rather than escaping the tool loop.

## Target files

- `shared/tool_executor.py`

## Implementation tasks

- Wrap `await self._lifecycle.ensure_ready(server_key)` in `try/except`.
- Catch `OSError` and `RuntimeError`.
- On failure:
  - record health failure when `_health_registry` exists
  - return `ToolCallResult` with `is_error=True`
  - use `error_type="transport"`
  - include a clear message such as `Lifecycle ensure_ready failed for '<server_key>': <error>`
- Do not proceed to transport execution after lifecycle startup failure.

## Acceptance criteria

- `ensure_ready()` startup failure does not escape `_raw_execute()`.
- Tool callers receive a `ToolCallResult` error.
- Health registry failure count is incremented when available.
- Existing transport error handling continues to work.

## Test plan

- Mock lifecycle `ensure_ready()` to raise `RuntimeError`.
- Assert `_raw_execute()` returns `ToolCallResult(is_error=True, error_type="transport")`.
- Assert health registry `record_failure()` is called.
- Verify transport is not called after lifecycle failure.
