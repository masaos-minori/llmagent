# H-4: Make ensure_ready actually start missing subprocess MCP servers

## Priority

High

## Background

`ToolExecutor._raw_execute()` calls `lifecycle.ensure_ready(server_key)` before resolving and invoking the transport.

Currently, `_ServerLifecycleRouter.ensure_ready()` only calls `verify_running(server_key)` and ignores the return value. `verify_running()` does not restart or start the process.

`ensure_ready()` should guarantee readiness for agent-managed HTTP subprocess MCP servers.

## Target files

- `agent/factory.py`
- `agent/http_lifecycle.py` if helper methods are needed

## Implementation tasks

- Update `_ServerLifecycleRouter.ensure_ready()` so subprocess-mode HTTP servers are started when missing or not running.
- Keep persistent-mode HTTP servers externally managed; do not start them.
- Respect the `_shutting_down` guard.
- Use enum comparisons for `TransportType.HTTP` and `StartupMode.SUBPROCESS`.
- Preserve no-op behavior for unknown server keys.

## Acceptance criteria

- For a subprocess-mode HTTP MCP server with no running process, `ensure_ready()` starts it.
- For a subprocess-mode HTTP MCP server with an already running process, `ensure_ready()` does nothing.
- For a persistent-mode server, `ensure_ready()` does not start a subprocess.
- During shutdown, `ensure_ready()` does not start a subprocess.

## Test plan

- Mock `HttpServerLifecycleManager.verify_running()` returning false and assert `start()` is called.
- Mock `verify_running()` returning true and assert `start()` is not called.
- Verify persistent server mode does not call `start()`.
- Verify shutdown guard prevents `start()`.
