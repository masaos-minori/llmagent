## Background

Multiple docs and tests hardcode the total number of MCP servers and/or MCP
tools, either globally or per server:

- `docs/04_mcp_01_system_overview.md:30` states "10個のMCPサーバ、合計65個のtool"
  (global totals).
- Each MCP server catalog doc (`docs/04_mcp_04_01..05_*.md`) has a
  `**ツール（N個）:**` heading per server.
- `tests/test_tool_registry_counts.py`'s `TestRegistryTotalCounts` asserts the
  registry's global tool count (63) and per-server counts (GitHub 21,
  rag_pipeline 4).
- `tests/test_tool_constants.py`'s `TestToolConstants` asserts both an exact
  expected-name set and `len() == N` for every `shared/tool_constants.py`
  frozenset, plus a global union count (42) in `test_no_overlapping_tools`.
- `tests/test_mdq_metadata_consistency.py::test_total_tool_count` and
  `tests/test_mdq_routing.py::TestMdqToolsCount` each separately hardcode the
  MDQ tool count (7) and an exact expected-name set.
- `tools/check_mcp_docs_consistency.py`'s Check 5 ("Tool count consistency")
  parses `ツール（N個）` doc headings and compares them against a hand-maintained
  `_SERVER_TOOLS_MAP` mirror of `shared/tool_constants.py`.
- `tests/test_check_mcp_docs_consistency.py::TestCheckToolCounts` (7 methods)
  is a synthetic-fixture unit test suite for `check_tool_counts()` itself,
  separate from the doc files it validates.

Note: `.github/workflows/mcp-docs-consistency.yml` invokes this script at
`scripts/checks/check_mcp_docs_consistency.py`, a path that does not exist in
the current tree (the real file is `tools/check_mcp_docs_consistency.py`,
importable as top-level module `check_mcp_docs_consistency` per
`pyproject.toml`'s `[project.scripts]` entry). This CI workflow is already
broken independently of this requirement — out of scope to fix here, but
means Check 5's removal carries no CI-breakage risk today since the workflow
that would run it cannot currently locate the script at all.

These counts have already drifted at least once: `docs/04_mcp_04_04_mdq.md`
documents 9 MDQ tools (including `fts_consistency_check`/`fts_rebuild`) while
`shared/tool_constants.py`'s `MDQ_TOOLS` has 7 (per the removal completed
under `requires/done/20260716_02_require.md`), and
`docs/04_mcp_01_system_overview.md`'s "65個" total does not match the
registry's actual 63.

## Problem

Every time a tool is added to or removed from any MCP server (the MDQ
resurfacing work under `requires/20260716_08_require.md` and
`requires/20260716_13_require.md` is expected to keep doing this), every
hardcoded count above must be updated in lockstep across 2 doc files, 4 test
files, and 1 CI script, or a test/doc fails for a reason unrelated to
correctness. This is pure maintenance overhead: the count itself carries no
information a reviewer needs that the tool-name list doesn't already convey.

## Reason for change

Per user direction, total-count tracking (global MCP server count, global MCP
tool count, and per-server tool count) is out of scope for docs and tests
going forward. Only two classes of check remain valuable and must be
preserved:

1. **Name-level drift detection between independently-maintained
   implementation artifacts** (e.g. live `/v1/tools` response vs.
   `MDQ_TOOLS`, `TOOL_LIST` vs. `_DISPATCH_TABLE`, `ToolRegistry` vs.
   `tool_constants.py`) — these catch a real bug (a tool silently
   unroutable/undispatchable) and do not hardcode a literal count or a
   duplicate name list in the test itself.
2. **Structural/classification invariants** that don't depend on how many
   tools exist (is-a-frozenset, all-elements-are-strings, no cross-category
   overlap, read/write disjointness, side-effect classification).

Any assertion of the form `len(X) == N` (a literal count) or
`X == {"a", "b", ...}` (a hardcoded expected-name literal that duplicates the
implementation) is explicitly in scope for removal, even where it currently
also happens to catch drift — the disambiguating rule is: **if the check
compares two independent implementation sources against each other, keep it;
if it compares an implementation source against a number or name-list
literal written into the test/doc itself, remove it.**

## Implementation instructions

### Docs

- `docs/04_mcp_01_system_overview.md:30`: remove the "10個のMCPサーバ、合計65個のtool"
  total-count sentence; replace with a description that doesn't require
  updating on every tool change (e.g. point to `config/agent.toml`'s
  `[mcp_servers.*]` sections and `shared/tool_constants.py` as the sources of
  truth for server/tool inventory).
- `docs/04_mcp_04_01_web-search-file-read-github.md`,
  `docs/04_mcp_04_02_file-write-file-delete-shell.md`,
  `docs/04_mcp_04_03_rag-pipeline-and-cicd.md`, `docs/04_mcp_04_04_mdq.md`,
  `docs/04_mcp_04_05_git.md`: change every `**ツール（N個）:**` heading to
  `**ツール:**` (drop the parenthetical count; keep the tool-name listing that
  follows it unchanged).

### Tests

- `tests/test_tool_registry_counts.py`: remove `TestRegistryTotalCounts`
  in full (`test_registry_total_tool_count`, `test_github_tools_count`,
  `test_mcp_tools_module_count`) and remove
  `TestRegistryRagPipelineCounts.test_registry_rag_pipeline_tool_count`.
  Keep `test_rag_tools_all_in_registry`, `test_rag_list_documents_registered`,
  `test_rag_delete_document_registered` (server-key mapping checks, not
  counts).
- `tests/test_tool_constants.py`: remove the 7 test methods in
  `TestToolConstants` that assert a hardcoded expected-name set and/or
  `len() == N` (`test_read_tools`, `test_write_tools`, `test_delete_tools`,
  `test_rag_tools`, `test_cicd_tools`, `test_mdq_tools`, `test_git_tools`).
  In `test_no_overlapping_tools`, remove only the `assert len(all_tools) == 42`
  line; keep the overlap-detection loop. Leave
  `test_all_tools_are_strings`, `TestGitToolClassification`,
  `TestGithubToolClassification`, `TestCicdToolClassification`,
  `TestRagToolClassification`, `TestMdqToolClassification` unchanged (they
  assert unions/disjointness/side-effect classification, not counts or
  literal name lists).
- `tests/test_mdq_metadata_consistency.py`: remove `test_total_tool_count`.
  Remove the "Total tool count is 7" line from the module docstring and the
  "7" reference in `test_production_tool_statuses`'s and
  `test_all_tools_have_status_field`'s docstrings (docstring wording only —
  their assertion logic does not hardcode a count and is unaffected).
- `tests/test_mdq_routing.py`: remove `TestMdqToolsCount` in full
  (`test_mdq_tools_count`, `test_mdq_production_tools_count`,
  `test_mdq_tools_exact_names`). In `test_v1_tools_returns_all_tools`, remove
  `assert len(body["tools"]) == 7`; keep `assert "tools" in body`. Leave
  `TestMdqNoUnmappedTools`, `TestMdqSafetyTiers`,
  `TestMdqMCPServerConformance`, `test_v1_tools_names_match_mdq_tools`, and
  `test_mdq_tools_match_server_definition` unchanged — these compare two
  independent implementation sources against each other (registry vs.
  `MDQ_TOOLS`, live endpoint vs. `MDQ_TOOLS`, `TOOL_LIST` vs. `MDQ_TOOLS`),
  not a literal count or a hardcoded name list.

### CI script

- `tools/check_mcp_docs_consistency.py`: remove Check 5 ("Tool count
  consistency") in full — `_TOOL_COUNT_RE`, `_TOOL_COUNT_CATALOG_FILES`,
  `_SERVER_TOOLS_MAP`, `check_tool_counts()`, `_check_tool_counts_for_file()`,
  `_find_server_section_for_line()` — plus its call site in `main()` and the
  `"toolcount"` entry in `skip_choices`/the module docstring's `--skip`
  listing.
- `tests/test_check_mcp_docs_consistency.py`: remove
  `TestCheckToolCounts` in full (7 methods) and drop `check_tool_counts` from
  the `from check_mcp_docs_consistency import (...)` block at the top of the
  file.

## Required tests

- After removal, `uv run pytest tests/test_tool_registry_counts.py
  tests/test_tool_constants.py tests/test_mdq_metadata_consistency.py
  tests/test_mdq_routing.py -v` passes with the remaining (non-count) tests
  green.
- `uv run python tools/check_mcp_docs_consistency.py` runs without a
  `toolcount`-related `--skip` choice available and without emitting any
  tool-count warnings.
- No new test is added that reintroduces a literal `len(X) == N` or
  hardcoded expected-name-set assertion for MCP server/tool inventory.

## Acceptance criteria

- No doc under `docs/04_mcp_*.md` states a total MCP server count or a total
  MCP tool count (global or per-server).
- No test in `tests/` asserts `len(<mcp tool collection>) == <literal int>`
  or compares an MCP tool collection to a hardcoded literal name-set.
- `tools/check_mcp_docs_consistency.py` no longer parses or validates
  `ツール（N個）`-style doc headings.
- Name-level drift checks between independently-maintained sources (registry
  vs. constants, dispatch table vs. schema, live endpoint vs. constants)
  remain intact and continue to pass.
- Structural/classification tests (frozenset membership type, no-overlap,
  read/write disjointness, side-effect tier) remain intact and continue to
  pass.

## Non-goals

- Do not change which tools are exposed by any MCP server (no functional
  change to `mcp_servers/*/tools.py` or `_DISPATCH_TABLE`).
- Do not remove or weaken the name-level drift/consistency checks listed
  under "Required tests" retention — only literal counts and hardcoded
  expected-name-set literals are in scope.
- Do not address `requires/20260716_08_require.md`'s MDQ tool-surface
  consolidation decision (Option A vs. B) — this requirement is independent
  of which tools MDQ ultimately exposes.
