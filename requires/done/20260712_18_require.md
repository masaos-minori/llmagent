# [Tooling][MCP] check_mcp_docs_consistency.py の stale-language チェックが日本語の否定表現でfalse positiveを出す

## 背景

`tools/check_mcp_docs_consistency.py` の `check_transport_error_is_error()`(Check 10:
HttpTransport is_error=True、724-761行目)は、docsが「HttpTransportがトランスポート障害に
対して`is_error=True`を直接返す」という誤った(stale)記述をしていないかを検出する仕組みである。

しかし現状、`docs/04_mcp_02_03_audit-logging-and-errors.md:56` に対して誤って警告を
発火させている。実行結果:

```
uv run python tools/check_mcp_docs_consistency.py
→ Found 2 warning(s).
[WARNING] 04_mcp_04_01_web-search-file-read-github.md:95: Tool count mismatch for github-mcp: documented 21, expected 11
[WARNING] 04_mcp_02_03_audit-logging-and-errors.md:56: Possible stale language: HttpTransport raises TransportError, not is_error=True, for transport failures.
```

このうち後者(56行目)はfalse positiveである(前者のgithub-mcpツール数不一致は別件・
対応不要と判断済み、本要望のスコープ外)。

## 目的

false positiveの原因(日本語の否定表現を認識できない)を修正し、正しく記述された
ドキュメントに対して誤った警告が出続ける状態を解消する。

## 現状(調査結果)

- `docs/04_mcp_02_03_audit-logging-and-errors.md:56` の実際の記述:
  `> **注記:** HttpTransport.call() はトランスポート障害に対して is_error=True を直接返すことはない。`
  — これは「直接返すことは**ない**」という否定文であり、内容自体は正しい(stale ではない)。
  続く57-58行目で `TransportError` を発生させ、ハンドラがこれを捕捉する旨も正しく説明されている。
- `check_transport_error_is_error()` の誤検知回避ロジック(`tools/check_mcp_docs_consistency.py:748`)は
  以下の条件でのみ警告をスキップする:
  ```python
  if "never" in line.lower() or "ToolCallResult" in line:
      continue
  ```
  この行(56行目)には英語の `"never"` も `"ToolCallResult"`(58行目にあるが56行目にはない)も
  含まれないため、スキップ条件に該当せず誤って警告が発火する。
- 検知対象の正規表現(727-730行目)`_TRANSPORT_IS_ERROR_RE` 自体は、単に
  `"HttpTransport...is_error=True"` または `"is_error=True...transport"` という語の
  近接のみを見ており、否定/肯定の判定はしていない。誤検知回避は「skip条件」側でのみ
  行われる設計であり、この skip 条件が英語限定になっている。

## 提案設計

1. `check_transport_error_is_error()` の skip 条件に、日本語の否定表現(「ことはない」
   「しない」「直接返さない」等)を追加する。
   - 個別の否定語を列挙するアプローチは将来また別の言い回しで漏れる可能性があるため、
     以下のいずれかを検討する:
     a. 日本語の代表的な否定パターン(`ことはない`, `しない`, `返さない` 等)を
        skip条件に追加する(現状の実装方針に最も近い、最小差分の対応)。
     b. `> **注記:**` で始まる引用ブロック(既に訂正済みの注記であることを示す慣習)を
        丸ごとスキップする。
   - 本ドキュメントでは a を推奨する(bは注記の中身を見ずに一律スキップするため、
     本当にstaleな内容が注記内に紛れ込んだ場合に検出漏れとなるリスクがある)。
2. 修正後、意図的にstaleな記述(否定なしで「is_error=Trueを返す」と書いたテスト用docs)を
   用意し、正しく警告が発火することを確認する。

## 効果

- 正しく記述されたドキュメントに対する誤警告を解消し、チェックツールの信頼性を回復する。
- 誤警告に慣れて本当のstale記述を見逃す「オオカミ少年」状態を防ぐ。

## リスク

| リスク | 可能性 | 対策 |
|---|---|---|
| 日本語否定表現の列挙が不十分で、別の言い回しで再度false positiveが起きる | 中 | 完全な自然言語否定検出は範囲外とし、実際に遭遇したパターンを都度追加する運用とする(既存の`"never"`判定と同じ思想) |
| skip条件を緩めすぎて、本当にstaleな記述を見逃すようになる | 低 | 否定表現の判定は行全体ではなく該当語の直接的な否定文脈に限定し、`> **注記:**` 記法の有無だけでは判定しない(提案設計1のa案採用) |
