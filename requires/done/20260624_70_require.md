# Clarify when stdio transport is acceptable and when HTTP is preferred

## Summary
Document stdio transport as a supported but constrained transport, and make the production-preferred role of HTTP more explicit.

## Background
The MCP docs already explain stdio transport and its serialization constraints, while also stating that all current production servers use HTTP.

The intended use boundary can be documented more directly.

## Objective
State clearly:
- when stdio is acceptable
- when stdio should be avoided
- when HTTP is the preferred operational path

## In Scope
- `04_mcp_01_system_overview.md`
- `04_mcp_02_protocol_and_transport.md`
- `04_mcp_03_routing_lifecycle_and_execution.md`

## Required Changes
- Add an explicit “When to use stdio” subsection
- Add an explicit “When not to use stdio” subsection
- Re-state that current production deployments use HTTP
- Keep future stdio scaling ideas clearly labeled as non-committed planning notes

## Acceptance Criteria
- Readers can identify whether stdio is appropriate for a local low-concurrency tool
- HTTP is clearly documented as the production-default transport path
- stdio serialization limits are represented as an operational constraint, not a minor footnote

## Out of Scope
- Implementing stdio multiplexing
- Removing stdio support

## Notes
Do not over-promise future stdio improvements. Keep the guidance practical and current-state focused.
