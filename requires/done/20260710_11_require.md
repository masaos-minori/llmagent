# [High] Define HealthRegistry state transition semantics and add complete tests

## Background

`McpServerHealthRegistry` controls dispatch gating and is updated by both ToolExecutor and watchdog. Its state transitions must be deterministic and well-tested.

Important behaviors include:

- Failure threshold transitions to `UNAVAILABLE`.
- Cooldown transitions `UNAVAILABLE` to `HALF_OPEN`.
- Successful trial dispatch restores `HEALTHY`.
- Failed trial dispatch returns to `UNAVAILABLE`.
- `record_degraded()` currently records reachable-but-degraded state without increasing failure count.
- `record_degraded()` interaction with existing `UNAVAILABLE` state must be explicitly defined.

## Goal

Make HealthRegistry state transitions explicit, safe, documented, and fully tested.

## Scope

- `shared/mcp_health.py`
- HealthRegistry tests
- Watchdog tests where state interaction is involved
- MCP lifecycle documentation

## Tasks

- [ ] Decide whether `record_degraded()` may overwrite `UNAVAILABLE`.
- [ ] If not allowed, add a guard so `UNAVAILABLE` is preserved.
- [ ] If allowed, document the expected transition and operational reason.
- [ ] Add tests for `HEALTHY -> DEGRADED`.
- [ ] Add tests for `DEGRADED -> UNAVAILABLE`.
- [ ] Add tests for `UNAVAILABLE -> HALF_OPEN` after cooldown.
- [ ] Add tests for `HALF_OPEN -> HEALTHY` on success.
- [ ] Add tests for `HALF_OPEN -> UNAVAILABLE` on failure.
- [ ] Add tests for `record_degraded()` when the current state is `UNAVAILABLE`.
- [ ] Add tests for `record_success()` clearing failure count, unavailable timestamp, and degraded reason.
- [ ] Expand docstrings for `record_degraded()`, `record_success()`, and `is_unavailable()`.
- [ ] Document that `is_unavailable()` has a state-changing side effect when cooldown has elapsed.

## Acceptance Criteria

- [ ] `UNAVAILABLE` handling is deterministic.
- [ ] Watchdog cannot unintentionally bypass dispatch gating.
- [ ] All core state transitions are covered by tests.
- [ ] `record_degraded()` behavior is documented and tested.
- [ ] Maintainers can understand state transition behavior without tracing all call sites.

## Notes

This issue owns HealthRegistry semantics. Watchdog probe parsing is handled separately.
