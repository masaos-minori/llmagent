# [Low] Improve disabled-state visibility for memory layer and embedding

## Labels
priority/low, area/memory, area/observability, documentation

## Summary
Improve visibility and documentation for disabled memory layer and disabled embedding states.

## Background
The memory layer has multiple disabled or degraded states:

- `use_memory_layer=false`
  - memory services are not constructed
  - injection, ingestion, and retrieval are skipped
- `memory_embed_enabled=false`
  - embedding is disabled
  - retrieval may use FTS-only behavior when memory is enabled
- embedding circuit open
  - hybrid retrieval degrades to fallback behavior

Current configuration has `use_memory_layer=false` and `memory_embed_enabled=false`, so memory is disabled in the provided config. 【1-14e820】

The memory documentation describes the activation gate and disabled behavior by module. 【5-639074】

## Required Changes

### Runtime Visibility
Improve startup banner or `/memory status` display to clarify:

- memory layer disabled
- memory layer enabled but embedding disabled
- FTS-only mode
- degraded mode
- hybrid mode

### Documentation
Update memory and operations docs:

- explain `use_memory_layer=false`
- explain `memory_embed_enabled=false`
- explain difference between disabled, FTS-only, degraded, and hybrid modes
- align production checklist with real config examples

### Tests
If display behavior changes, add or update tests for memory status output.

## Acceptance Criteria
- Operators can distinguish memory layer disabled from embedding disabled.
- `/memory status` or startup output is clearer.
- Documentation matches runtime states.
- No memory persistence behavior is changed.
