# [Critical] `enabled_for_llm` is never set True for discovered tools — filters ALL tools out of the LLM payload once discovery succeeds

## Priority

Critical

## Summary

Re-investigating `issues/20260718-122348_mcp_runtime_availability_metadata_landing_risks.md`'s flagged
gap ("`mcp_tool_discovery.py` doesn't feed live `enabled`/`disabled_reason` into
`RuntimeTool.enabled_for_llm`") reveals the actual impact is worse than that note described: this is not
merely a case of an individually-disabled tool staying visible — it is a systemic bug that hides **every
successfully-discovered tool** from the LLM.

Chain of evidence, verified directly in current source:

1. `scripts/shared/runtime_tool.py::build_runtime_tool()` defaults `enabled_for_llm` to `False` when the
   caller doesn't pass it explicitly (`resolved_enabled_for_llm = enabled_for_llm if enabled_for_llm is
   not None else False`).
2. `scripts/agent/services/mcp_tool_discovery.py`'s only call site
   (`built[name] = build_runtime_tool(name=..., server_key=..., ..., capabilities=...)`) never passes
   `enabled_for_llm` — confirmed by reading the full call (lines ~311-322). Every tool built from a live
   `/v1/tools` discovery response therefore gets `enabled_for_llm=False`.
3. `scripts/shared/runtime_tool_registry.py::llm_tool_definitions()` returns only tools where
   `tool.enabled_for_llm` is true.
4. `scripts/agent/llm_turn_runner.py::_filter_disabled_tool_definitions()` computes
   `visible_names = {td["name"] for td in registry.llm_tool_definitions()}` and
   `known_names = {tool.name for tool in registry.all_tools()}`, then keeps a config-defined tool
   definition only if `name not in known_names or name in visible_names`. Since every discovered tool is
   in `known_names` and none are in `visible_names` (per points 1-2), **every tool whose name matches a
   discovered tool gets filtered out** of the payload sent to the LLM.
5. `scripts/agent/startup.py:230` sets `ctx.services_required.runtime_tools = discovery.registry` after
   a **successful** discovery — i.e. this is the normal, common-case code path, not an edge case.

Net effect: in the ordinary case where MCP discovery succeeds (which, since a recent change made
`RuntimeToolRegistry` the sole routing authority, is now required for any tool call to route at all),
the LLM receives **zero tool definitions** for any tool that was actually discovered. This is
plausibly a full-agent tool-calling outage, not a partial/cosmetic gap.

Confirmed this is untested: `tests/agent/services/test_mcp_tool_discovery.py` has zero references to
`enabled_for_llm`, and every test elsewhere that constructs a `RuntimeTool` directly (there are ~13 call
sites across `tests/`) passes `enabled_for_llm=True` explicitly — meaning existing tests mask this by
never exercising the real discovery-service construction path end-to-end against
`_filter_disabled_tool_definitions()`.

## Target files

- `scripts/agent/services/mcp_tool_discovery.py` (the `build_runtime_tool()` call site)
- Possibly `scripts/agent/llm_turn_runner.py::_filter_disabled_tool_definitions()` (if the fix belongs on
  the consumer side instead of/in addition to the producer side — see Unknowns)
- `tests/agent/services/test_mcp_tool_discovery.py`
- `tests/test_tool_executor_routing.py` / an integration test exercising the full
  discovery → registry → `_filter_disabled_tool_definitions()` chain (currently no such test exists)

## Reason for change

Silently hiding all tools from the LLM is a severe functional regression with no error, warning, or log
line to indicate it happened — it would present as "the agent doesn't call any tools anymore" with no
diagnostic trail pointing at the cause.

## Intent

After this change:

- `enabled_for_llm` on a `RuntimeTool` built from a live discovery response reflects the tool's actual
  intended visibility (true for a normally-available tool; false only for a tool a server explicitly
  reports as unavailable via its own `enabled`/`disabled_reason` fields, per requirement 15's schema).
- `_filter_disabled_tool_definitions()` continues to correctly hide genuinely-disabled tools (e.g.
  `write_file` when `allowed_dirs` is empty) without hiding normally-available ones.
- A test exercises the full chain end-to-end (real `build_runtime_tool()` call inside discovery, not a
  hand-built `RuntimeTool(enabled_for_llm=True)` fixture) so this class of bug cannot regress silently
  again.

## Required changes

### 1. Fix `mcp_tool_discovery.py`'s tool-building step

Pass `enabled_for_llm` to `build_runtime_tool()` derived from the discovered tool entry's own
availability signal. Read `entry.get("enabled", True)` (per requirement 15's `enabled`/`disabled_reason`
schema — default to `True` for servers/tools that don't report it at all, since most tools have no
config-dependent gating) and pass it through as `enabled_for_llm=entry.get("enabled", True)`.

### 2. Decide how `disabled_reason` factors in (if at all)

Check whether `RuntimeTool` or its consumers need the actual `disabled_reason` string surfaced anywhere
(e.g. in `/mcp status` diagnostics) beyond the boolean gate — `cmd_mcp.py:94` already reads
`row.get("enabled_for_llm", "-")` for display; consider whether `disabled_reason` deserves a similar
diagnostics column.

### 3. Add end-to-end test coverage

Add a test that builds a `RuntimeToolRegistry` via the real `McpToolDiscoveryService` construction path
(a fake/mocked `/v1/tools` HTTP response is fine, but the tool objects must flow through the actual
`build_runtime_tool()` call site, not a hand-constructed `RuntimeTool`), then asserts
`_filter_disabled_tool_definitions()` keeps a normally-discovered tool's config-defined definition
visible. Also add a case with `entry["enabled"] = False` and confirm that specific tool is filtered while
others remain visible.

## Unknowns to resolve during planning

- Whether `entry.get("enabled", True)` is the right default, or whether a tool entirely absent an
  `enabled` key (i.e. a server that hasn't adopted requirement 15's schema at all) should be treated
  differently from one explicitly reporting `enabled: true` — resolve by checking what the 6 servers that
  don't yet report `enabled`/`disabled_reason` (only `file/{read,write,delete}` and `git` do today) return
  in their `/v1/tools` entries, and confirm they don't include a bare `enabled: false` by accident.

## Acceptance criteria

- After a successful discovery cycle, a tool with no explicit `enabled: false` in its discovery entry is
  visible to the LLM (present in the filtered `tool_definitions` payload).
- A tool a server explicitly reports as `enabled: false` (e.g. `write_file` with empty `allowed_dirs`) is
  still correctly hidden from the LLM.
- A new or extended test exercises the real `build_runtime_tool()` call site inside
  `McpToolDiscoveryService`, not a hand-built `RuntimeTool` fixture, and would have failed against the
  pre-fix code.
- Full test suite passes with no new failures.

## Notes for implementer

### Severity note

Treat this as the highest-priority item among the three requirements filed alongside it
(`requires/20260721_01_require.md`, `requires/20260721_02_require.md`) — a stale `cmd` path breaks one
server; a discovery-failure visibility gap degrades one session; this bug, if the analysis above is
confirmed correct at implementation time, silently breaks tool-calling for **every** session where
discovery succeeds, which is the common case. Verify the full chain (points 1-5 above) against current
source before starting, in case something has changed since this was filed.

### Out of scope

- Any change to `ToolRouteResolver`/routing itself — `enabled_for_llm` does not gate routing
  (`RuntimeToolRegistry.resolve()` is independent of this field); this is purely about what tool
  definitions are sent to the LLM's function-calling schema.
