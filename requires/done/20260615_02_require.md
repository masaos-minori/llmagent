# 改善要望

## 0. 可観測性の底上げ

- malformed request、ignored input、plugin load 結果、migration 結果、recovery 結果、reload 結果、memory persistence 失敗、tool-definition fallback 使用などを、機械可読レポートまたは監査ログで残す。単なる log 出力のみで終えない。
- logging strategy は string-based kvlog と JSON logging の二重運用をやめ、structured logging に統一する。context は shared filter ではなく task-local で付与する。

## 1. 後方互換レイヤと façade の削除

- `config.py` の `DbConfig / build_db_config` re-export、`lifecycle.py` の `ServerLifecycleManager` façade、`session.py` の repository re-export、`llm_turn_runner.py` の call-site 無変更前提 adapter など、互換維持だけを目的とした導線を削除すること。依存入口の二重化と責務の曖昧化を解消するための改善点。
- `registry.py` の `mask_args` backward-compatible re-export、`tool_result_formatter.py` と `registry.py` の多重公開、`tool_runner.py` の legacy DAG 実装など、旧経路の痕跡を除去して現在の正規コンポーネントへ統一すること。
- Memory 系では `MemoryLayer` 互換 façade を廃止し、`MemoryInjectionService / MemoryIngestionService / MemoryRetriever` へ直接依存する構成へ移行すること。

## 2. CLI / command 層の責務縮小

- command mixin が `ctx.conv.history`、`ctx.stats`、`ctx.session` などを直接更新している状態をやめ、History/Session/DB/Memory の service に処理を移すこと。CLI を「引数解析 + service 呼出 + 表示」に縮退させる改善点。
- `cmd_context.py` の undo、context 集約、system prompt 更新、`cmd_session.py` の session restore / title generation、`cmd_db.py` の maintenance 操作、`cmd_memory.py` の pin/delete/prune、`cmd_tooling.py` の `/tool` 表示と `/plan` 切替などを、それぞれ専用 service と formatter に分離すること。
- `cmd_mcp.py` の status 表示と install wizard、`cmd_ingest.py` の export / ingest / compact、`cmd_debug.py` の監査 tail / logger level / debug toggle など、多責務コマンドを分割すること。

## 3. dispatch・引数解析・usage 生成の共通化

- `registry.py` の `sync_cmds / async_cmds / prefix_cmds / help` を宣言的コマンド定義へ統合し、help / usage / dispatch の情報源を 1 か所にまとめること。
- `/session`、`/db`、`/memory`、`/note`、`/tool`、`/mcp` などで散在する `split()`、`isdigit()`、手書きフラグ解析を共通 parser に統一し、未知オプション、不足引数、usage 表示の不統一を解消すること。
- formatter / presenter / safe JSON decode helper を共通化し、list/detail/table/export の表示経路を標準化すること。

## 4. Orchestrator / LLM turn 実行の責務再分離

- `orchestrator.py` を turn lifecycle 管理に限定し、prompt assembly、audit、turn start/end、error policy を専用 service へ切り出すこと。`handle_turn()` の肥大化リスクへの改善点。
- `llm_turn_runner.py` の monolithic loop を streaming、tool turn loop、span/context、error injection に分割し、文字列戻り値ではなく構造化 outcome を返すこと。
- `orchestrator.py` と `llm_turn_runner.py` に分散する LLM transport error 処理、partial completion、synthetic history injection の契約を 1 つに統一すること。

## 5. 設定反映の単一入口化と差分適用化

- `config_reload.py` は `apply_config_dict()` を唯一の公開入口にし、`sync_services()` の legacy direct-call 前提を削除すること。
- `bool() / int() / float()` の ad-hoc 変換や、未指定項目への既定値強制上書きをやめ、差分反映と検証付き変換へ変更すること。部分 reload 時の意図しない設定変化リスクへの改善点。
- LLM、MCP、Approval、Tool/RAG、Conversation Prompt の reload をドメイン別 handler に分割し、before/after を持つ差分レポートを返すこと。

## 6. ライフサイクル・監視・起動処理の整理

- `repl.py` から bootstrapping、health check、watchdog、MCP 起動管理を切り出し、REPL 自体は入出力 loop に専念させること。
- `repl_health.py` を health probe、tool definition validator、watchdog に分割し、startup 時と runtime 時で重複する tool definition 検証を統一すること。
- `lifecycle.py / http_lifecycle.py / stdio_lifecycle.py` の状態表現を共通 `LifecycleState` や `TransportHandle` に統一し、HTTP と stdio で非対称な state 返却をやめること。
- `factory.py` を bootstrapping、service factory、logger factory へ分割し、plugin path や logger 構築、cleanup/rollback 契約を明示化すること。

## 7. DB 層の責務再分離と silent fallback 撤廃

- DB 層を 設定管理、接続管理、スキーマ管理、永続化 API、保守運用 に再分離し、`ConfigLoader` 直読込、`SQLiteHelper` の過重責務、module/class-level 状態依存を廃止すること。
- `helper.py` の設定キャッシュや private/class-level 状態、`maintenance.py` の `SQLiteHelper._ensure_config()` 依存、`tool_results.py` の DB 異常時 fallback を削除すること。
- `create_schema.py` を versioned migration 前提へ再設計し、`IF NOT EXISTS` や safe migration error 依存をやめ、schema version 管理と migration artifact 管理へ移行すること。
- `store.py` / `tool_results.py` / `maintenance.py` の戻り値を typed DTO や構造化結果へ統一し、「データなし」と「DB 異常」を区別可能にすること。

## 8. ツール結果全文の永続化と再参照性強化

- `tool_result_store` の FIFO メモリ保持だけでなく、ツール結果全文を `tool_results` テーブルへ永続化し、`/tool list` と `/tool show` を DB ベース参照へ変更すること。セッション再開後の追跡性向上の改善点。
- 保存項目として `session_id`、`turn_id`、`tool_name`、`args_masked`、`full_text`、`summary_text`、`created_at` を持たせ、監査再利用性を高めること。

## 9. ツール実行基盤の統一

- `tool_runner.py` の `_execute_with_dag_legacy()` を削除し、`tool_scheduler.py` の `build_execution_groups()` を execution order の唯一の正とすること。
- 承認 UI と policy 判定を分離し、repo/path/branch/deny/escalation の判断は `tool_policy.py` に一元化し、`tool_approval.py` は UI と flow のみを担当させること。
- `tool_audit.py` は approval / decision / exec で schema を揃え、masked args、resource scope、operation type、decision を対称に記録すること。
- `tool_loop_guard.py` を per-turn guard 実装の唯一の正とし、caller 側に残る重複 guard を削除すること。

## 10. MCP 基盤の責務分離と入力契約厳格化

- `server.py` を HTTP transport と stdio transport に分離し、`run()` alias と二重 truncate API を削除すること。
- `dispatch.py` の `(str, bool)` 戻り値を `DispatchResult` のような構造化結果へ置換し、HTTPException duck typing を廃止すること。
- `models.py` の `validate_args()` 手動呼出し前提をやめ、tool-specific args 検証を自動バリデーションへ統合すること。validator 未登録 no-op も廃止対象。
- `audit.py` を正式な監査 API へ昇格させ、server/dispatch/role-specific server から共通 schema で使うこと。

## 11. MCP installer / status / workflow の表示責務分離

- `mcp_install.py` から `print_next_steps()` を削除し、ScaffoldResult 返却に専念させること。OpenRC 固定の手順文言や固定パス前提は Presenter / deployment adapter へ移管すること。
- `mcp_status.py` は probe と table formatting を分離し、`status = "availability/health"` の連結文字列ではなく、availability / health / transport detail の構造化フィールドを返すこと。
- `ingest_workflow.py` は lazy import と包括 `except Exception` をやめ、crawl / split / ingest 各段の原因分類済み例外、構造化イベント、DI ベース依存へ変更すること。

## 12. Memory 層の永続化入口統一

- `ingestion.py` の `on_session_stop()` と `_persist_entry()` の保存経路差分をなくし、dedup 判定、JSONL 書き込み、SQLite upsert、duplicate link 記録を 1 本の保存パイプラインへ統一すること。
- `jsonl_store.py` の `write()` と `append()` の二重 API を解消し、JSONL 保存経路を単一化すること。排他制御ルールの一意化の改善点。
- JSONL と SQLite のどちらを source of truth とするか、片系失敗時の扱いを明確化し、`check_consistency()` や再同期方針を定義すること。

## 13. Memory 型・復元・バリデーションの一元化

- `mapper.py` の `row_to_entry()` と `jsonl_store.py` の `_entry_from_dict()` に分散する復元ロジックを 1 箇所に集約し、`MemoryEntry` 生成・検証・既定値補完を `types.py` または専用 codec/factory に寄せること。
- `source_type` 不正時に `CONVERSATION` へ無言 fallback する処理、tags 変換失敗時に空配列へ落とす処理など、データ異常隠蔽を廃止し、型検証失敗として明示化すること。
- `memory_type`、`source_type`、`importance`、`created_at`、`updated_at` の検証を型側へ集約し、mapper/store 側の場当たり的正規化を削除すること。

## 14. Memory 検索・注入・埋め込みの改善

- `retriever.py` は FTS、KNN、RRF、boost、FTS query builder を分解し、`FtsRetriever / VectorRetriever / HybridRetriever` などへ責務分離すること。
- KNN score を「負距離」として呼び出し側が解釈する暗黙契約をやめ、距離または類似度を型・契約として明示すること。
- `EmbeddingClient` の失敗表現を `None` 返却から結果型へ変更し、通信失敗、応答異常、空埋め込みを区別可能にすること。
- `embedding_client.py` では実入力 text が POST 本文に反映されていない疑いがあり、text 利用の明示修正が必要である。
- `injection.py` の `on_session_start()` と `on_user_prompt()` の接頭辞、整形規則、summary 優先ルールを統一し、未使用引数 `session_id` の削除または利用方針確定が必要である。

## 15. 抽出ポリシーと dedup ポリシーの設定化

- `extract.py` に埋め込まれた semantic / episodic 判定キーワード、閾値、summary 規則、importance 規則を設定へ外出しすること。英語キーワード依存や固定閾値依存の改善点。
- `DedupAction` の多モード運用や `InjectionPolicy.dedup_window` の未使用設定は整理し、不要モード・未使用設定を削除すること。

## 16. テストカバレッジと characterization tests の追加

- 低カバレッジな `cmd_session.py`、`cmd_memory.py`、`cmd_notes.py`、`cmd_db.py`、`cmd_context.py` に characterization tests を追加し、Phase 2 gate の `coverage >= 80%` を満たすこと。
- `rag/stages/search.py`、`rerank.py`、`mqe.py`、`fusion.py`、`augment.py` の各 `run()` を `tests/test_rag_stages.py` で単体テストすること。
- `tool_scheduler.py`、`shared/formatters.py`、`shared/git_helper.py`、`mcp/dispatch.py` の未テスト経路を補強し、後続リファクタ前に観測面を固めること。
- 非同期テストのモック設定不備、全体テスト timeout、373 個の async 関数を抱える状態に対して、非同期テスト管理の標準化と timeout 見直しが必要である。

## 17. 廃止スタブ・未使用コードの削除

- `scripts/agent/repl_debug.py`、`rag_debug.py`、`context_detection.py`、`agent/memory/scoring.py` など、参照元ゼロ・カバレッジ 0 のスタブ/未使用ファイルを削除または本体へ再統合すること。
- 削除時は `deploy/deploy.sh` のコピー処理も合わせて更新し、デプロイ資産との不整合を残さないこと。

## 18. 型注釈・内部依存・品質改善

- `rag/stages/*` の `__init__` シグネチャに型注釈を付け、`assert isinstance()` の unsafe assertion を警告 + 制御フローへ置換すること。
- `cmd_session.py` から `agent.config._CONFIG_DIR` のような private symbol 参照を除去し、公開プロパティ経由へ切り替えること。
- logging の f-string を `%s` 形式へ揃え、不要な文字列構築を抑えること。優先度は低いが、横断的一貫性向上の改善点。
