# 要件定義書 — Metadata DB / Workflow 定義 / 冪等性管理の整備

作成日: 2026-06-15

---

## Goal

plan / execute / verify / retry の段階モデルをフォーマルなワークフロー定義に昇格し、タスク状態・試行履歴・処理済みイベント・アーティファクト参照を Metadata DB で一元管理することで、冪等な実行と障害後の再開を保証する。

---

## Scope

**In**:
- `workflows/default.json` — ワークフロー定義ファイル（ステージ・タイムアウト・リトライポリシーの宣言）
- `orchestrator/state/` — ワークフロー実行状態の永続化レイヤー
- Metadata DB スキーマ — task / attempt / event / artifact テーブルの設計と DDL

**Out**:
- 既存の `agent/orchestrator.py` のターンレベル実行ロジックの変更
- UI / CLI 表示レイヤーの変更
- MCP サーバー群の変更

---

## Assumptions

1. ワークフローエンジンと Metadata DB は同一プロセス内またはローカル SQLite で動作する（分散システムは対象外）。
2. ワークフロー定義は JSON で記述し、起動時にロードして検証する。
3. plan / execute / verify / retry の各ステージは冪等であり、再実行可能な単位として設計できる。
4. `idempotency_key` はワークフローインスタンスとステージの組み合わせから生成する。
5. SQLite の WAL モードを使用し、ロールバック不能な副作用（外部 API 呼び出し等）はステージ完了後にのみ記録する。

## Design

### ワークフロー定義（`workflows/default.json`）

```json
{
  "version": "1.0",
  "name": "default",
  "stages": [
    {
      "id": "plan",
      "timeout_sec": 60,
      "retry": { "max_attempts": 3, "backoff_sec": 2 }
    },
    {
      "id": "execute",
      "timeout_sec": 300,
      "retry": { "max_attempts": 2, "backoff_sec": 5 }
    },
    {
      "id": "verify",
      "timeout_sec": 60,
      "retry": { "max_attempts": 1, "backoff_sec": 0 }
    },
    {
      "id": "retry",
      "timeout_sec": 30,
      "retry": { "max_attempts": 1, "backoff_sec": 0 }
    }
  ],
  "on_failure": "halt"
}
```

### Metadata DB スキーマ

**tasks テーブル** — ワークフローインスタンス

```sql
CREATE TABLE IF NOT EXISTS tasks (
    task_id          TEXT PRIMARY KEY,
    workflow_name    TEXT NOT NULL,
    idempotency_key  TEXT NOT NULL UNIQUE,
    status           TEXT NOT NULL CHECK(status IN ('pending','running','completed','failed','retrying')),
    created_at       TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now')),
    updated_at       TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
);
```

**attempts テーブル** — ステージ試行履歴

```sql
CREATE TABLE IF NOT EXISTS attempts (
    attempt_id       TEXT PRIMARY KEY,
    task_id          TEXT NOT NULL REFERENCES tasks(task_id) ON DELETE CASCADE,
    stage_id         TEXT NOT NULL,
    attempt_number   INTEGER NOT NULL,
    status           TEXT NOT NULL CHECK(status IN ('running','completed','failed','timeout')),
    started_at       TEXT NOT NULL,
    finished_at      TEXT,
    error_detail     TEXT
);
```

**processed_events テーブル** — 重複排除用イベント台帳

```sql
CREATE TABLE IF NOT EXISTS processed_events (
    event_id         TEXT PRIMARY KEY,
    task_id          TEXT NOT NULL REFERENCES tasks(task_id) ON DELETE CASCADE,
    stage_id         TEXT NOT NULL,
    processed_at     TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
);
```

**artifacts テーブル** — ステージ出力の参照

```sql
CREATE TABLE IF NOT EXISTS artifacts (
    artifact_id      TEXT PRIMARY KEY,
    task_id          TEXT NOT NULL REFERENCES tasks(task_id) ON DELETE CASCADE,
    stage_id         TEXT NOT NULL,
    artifact_type    TEXT NOT NULL,
    uri              TEXT NOT NULL,
    created_at       TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
);
```

### 冪等性管理

1. ステージ実行前に `processed_events(event_id)` を確認する。
2. `event_id` が存在する場合はスキップし、前回の artifact を返す。
3. 存在しない場合はステージを実行し、完了後に `processed_events` と `artifacts` を INSERT する。
4. INSERT は同一トランザクションで行い、失敗時はロールバックして再試行対象にする。

### `orchestrator/state/` 構成案

```
orchestrator/state/
  __init__.py
  workflow_loader.py   # workflows/*.json のロードと検証
  workflow_engine.py   # ステージ遷移・タイムアウト・リトライ制御
  state_store.py       # Metadata DB への CRUD
  models.py            # TaskRecord / AttemptRecord / ArtifactRef dataclass
```

## Implementation Steps

1. **Metadata DB スキーマ定義**
   - `db/workflow_schema.py` に DDL を実装
   - `deploy/init_db.sh` に `workflow.sqlite` 初期化ステップを追加
   - 完了条件: `uv run python -m db.workflow_schema` で DB が生成される

2. **`workflows/default.json` 作成**
   - ステージ・タイムアウト・リトライポリシーを記述
   - JSON Schema による検証スクリプトを追加
   - 完了条件: スキーマバリデーションが通る

3. **`orchestrator/state/models.py` 実装**
   - `TaskRecord`, `AttemptRecord`, `ArtifactRef` dataclass を定義
   - 完了条件: `uv run mypy orchestrator/` が 0 errors

4. **`orchestrator/state/state_store.py` 実装**
   - Metadata DB への CRUD 操作と冪等性チェックを実装
   - 完了条件: 単体テスト（重複 event_id でスキップ、新規 event_id で実行）pass

5. **`orchestrator/state/workflow_engine.py` 実装**
   - plan → execute → verify → (retry) のステージ遷移ロジックを実装
   - タイムアウトとリトライを `workflows/default.json` から読み込む
   - 完了条件: 統合テストで失敗→リトライ→完了シナリオが pass

6. **`agent/orchestrator.py` との統合**
   - WorkflowEngine を参照する最小限の接続を実装
   - 完了条件: 既存の `uv run pytest` でリグレッションなし

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| SQLite のロック競合（agent ターンと workflow state 書き込みの競合） | 中 | `workflow.sqlite` を独立 DB にする。WAL モードを強制する |
| ワークフロー定義変更時の既存 task との不整合 | 低 | `workflow_name` + `version` フィールドを tasks テーブルに持たせる |
| 冪等性保証の抜け（artifact INSERT 後にクラッシュ） | 低 | artifact と processed_events を同一トランザクションで INSERT し、partial commit を防ぐ |
| plan.md → execute.md → implementation.md の既存フローとの二重管理 | 中 | 既存フローを workflow エンジンの「plan ステージの実行物」として扱い、重複を回避する設計とする |
