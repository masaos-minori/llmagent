## Background

Several overview documents contain long directory trees and file-by-file explanations for implementation packages. These lists are easy to verify from the repository itself and become stale when files are renamed, added, removed, or reorganized.

## Problem

Design documents that duplicate the entire file tree create several problems:
1. They require frequent updates when implementation changes
2. They increase cognitive load for readers who must distinguish design-relevant from implementation-referential content
3. AI agents may treat copied implementation details as authoritative even when the implementation has changed
4. Long reference tables make it harder to find actual design decisions

## Reason for change

A design document should explain why a package or layer exists, what it owns, what boundaries it enforces, and what other components depend on it. It should not duplicate the entire file tree. This task compresses implementation-derived file listings into responsibility-oriented summaries while preserving architectural context.

## Implementation intent

Keep the overview documents useful as architecture documents, not source tree catalogs. Compress detailed file listings into Japanese responsibility-oriented summaries. Do not remove design intent, responsibility boundaries, invariants, failure behavior, or operational constraints.

## Implementation instructions

For each target document:

1. Identify file-by-file listings that merely mirror the repository
2. Remove or compress detailed implementation file lists
3. Replace them with Japanese responsibility-oriented summaries using suggested section headings:
   - `## 主要ディレクトリと責務`
   - `## レイヤー境界`
   - `## 変更時の注意点`
   - `## 実装詳細の参照先`
4. Preserve major directory and package responsibilities
5. Preserve layer boundaries and ownership descriptions
6. Preserve configuration ownership and runtime boundaries
7. Preserve security-relevant or operationally relevant notes
8. Add source references where appropriate
9. Add a note such as "完全なファイル一覧はリポジトリの実装ツリーを参照する"

Content to remove or compress:
- Complete source file lists
- One-line descriptions for every source file
- Repeated `__init__.py` explanations
- Long nested directory trees that mirror the repository
- Tables that only map file names to implementation descriptions

Content to keep:
- Package-level responsibilities
- Architectural layer boundaries
- Process boundaries
- Runtime ownership
- Dependency direction
- Configuration ownership
- Security-relevant boundaries
- Operational constraints
- Links to more specific documents
- Known Issues
- Needs Confirmation items

Work document by document. If unsure whether content is design-relevant, keep it and mark for later review rather than deleting it.
