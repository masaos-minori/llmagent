# Agent Startup Processing — MCP Subprocess Start Ignored in Non-Production Mode

## Summary

When an HTTP subprocess MCP server fails to start in non-production mode, the error is logged and a warning is displayed, but execution continues without raising. The server key remains untracked in `_http_procs`, so subsequent `ensure_ready()` calls will attempt to start it again. No backoff mechanism exists.

## Severity

Low-Medium

## Files

- `scripts/agent/startup.py:140-152`

## Details

In `StartupOrchestrator._start_servers()`:

```python
for key, cfg in ctx.cfg.mcp.mcp_servers.items():
    if (
        cfg.startup_mode == StartupMode.SUBPROCESS
        and cfg.transport == TransportType.HTTP
    ):
        try:
            await ctx.services_required.lifecycle.start_http_subprocess(key, cfg)
        except (OSError, RuntimeError) as e:
            if ctx.cfg.mcp.security_profile == SecurityProfile.PRODUCTION:
                msg = f"{OutputTag.FATAL} MCP subprocess {key!r} failed to start: {e}"
                logger.error(msg)
                raise RuntimeError(msg) from e
            logger.error("Failed to start HTTP subprocess MCP server %r: %s", key, e)
            self._view.write_warning(f"{OutputTag.NON_FATAL} HTTP subprocess MCP server {key!r} failed to start: {e}")
```

In production mode, the failure is treated as fatal and startup aborts. In non-production mode, the error is logged and a warning is shown, but the loop continues to the next server. The failed server's process object is never stored in `_http_procs`, so when a tool call later triggers `_ServerLifecycleRouter.ensure_ready()`:

```python
async def ensure_ready(self, server_key: str) -> None:
    if self._shutting_down:
        return
    cfg = self._server_configs.get(server_key)
    if cfg is None:
        return
    if not self._http_mgr.verify_running(server_key):
        self._set_state(server_key, LifecycleState.STARTING)
        try:
            await self._http_mgr.start(server_key, cfg)
            self._set_state(server_key, LifecycleState.RUNNING)
        except Exception:
            self._set_state(server_key, LifecycleState.FAILED)
            raise
```

This will retry starting the same failing subprocess on every tool call, with no exponential backoff or circuit breaker. A transient port conflict could cause repeated failures until the port becomes available.

## Reproduction Steps

1. Configure an HTTP subprocess MCP server with an invalid command or conflicting port
2. Start the agent in non-production mode (security_profile = "local")
3. Observe that startup succeeds despite the MCP server failure
4. Trigger a tool call that routes to the failed server
5. Observe repeated subprocess start attempts without backoff

## Expected Behavior

Non-production mode should still track failed servers and implement some form of rate limiting or circuit breaker for retry attempts, rather than attempting to restart the subprocess on every tool call.

## Proposed Fix

Track failed servers in `_ServerLifecycleRouter` and implement a simple retry cooldown:

```python
class _ServerLifecycleRouter:
    def __init__(self, ...):
        self._failed_starts: dict[str, float] = {}  # server_key -> last_failure_time
    
    async def ensure_ready(self, server_key: str) -> None:
        if self._shutting_down:
            return
        cfg = self._server_configs.get(server_key)
        if cfg is None:
            return
        
        # Check if we're in cooldown period after a recent failure
        last_failure = self._failed_starts.get(server_key, 0)
        if time.monotonic() - last_failure < COOLDOWN_SECONDS:
            logger.warning(
                "Lifecycle: skipping ensure_ready(%r) — cooldown active",
                server_key
            )
            return
        
        if not self._http_mgr.verify_running(server_key):
            self._set_state(server_key, LifecycleState.STARTING)
            try:
                await self._http_mgr.start(server_key, cfg)
                self._set_state(server_key, LifecycleState.RUNNING)
                self._failed_starts.pop(server_key, None)  # Clear cooldown on success
            except Exception:
                self._set_state(server_key, LifecycleState.FAILED)
                self._failed_starts[server_key] = time.monotonic()  # Set cooldown
                raise
```
