# Remove or formally restore partially wired MDQ FTS admin tools

## Priority

High

## Target files

- `mcp_servers/mdq/server.py`
- `mcp_servers/mdq/tools.py`
- `mcp_servers/mdq/service.py`
- `mcp_servers/mdq/models.py`
- `shared/tool_constants.py`
- `shared/tool_registry.py`
- `tests/test_mdq_*.py`
- `tests/test_tool_registry*.py`
- `tests/test_tool_routing*.py`

## Type

Backward compatibility removal / routing consistency

## Background

The MDQ FTS admin tools are partially present but not fully wired.

The following tools appear in some layers:

- `fts_consistency_check`
- `fts_rebuild`

Current state:

- `tools.py` exposes both tools in `TOOL_LIST`
- `models.py` defines `FtsConsistencyCheckRequest`, `FtsConsistencyCheckResponse`, and `FtsRebuildRequest`
- `service.py` implements `MdqService.fts_consistency_check()` and `MdqService.fts_rebuild()`
- `server.py` does not include these tools in `_DISPATCH_TABLE`
- `/v1/tools` returns `TOOL_LIST`, so these tools may be advertised even though they cannot be dispatched

This is a classic compatibility-removal candidate: a feature is present in schema/model/service layers but absent from the actual dispatch path.

## Problem

The system exposes tools that are not callable through the MCP dispatch endpoint.

This creates the following issues:

- LLM-visible tool schema may contain unusable tools
- Startup drift validation may produce confusing results
- ToolRegistry may route stale tools if constants still include them
- Operators may attempt admin operations that fail as `Unknown tool`
- Tests may pass at one layer while runtime behavior fails at another

## Intent

Make the MDQ tool surface internally consistent.

There are two valid outcomes:

1. Remove the FTS admin tools completely
2. Formally restore them by wiring dispatch, safety tiers, serialization, audit, and tests

For backward compatibility removal, the preferred direction is complete removal.

## Tasks

### Decision

- [ ] Decide whether `fts_consistency_check` should be removed or formally restored
- [ ] Decide whether `fts_rebuild` should be removed or formally restored
- [ ] Document the decision in a short ADR or maintenance note

### If removal is selected

- [ ] Remove `fts_consistency_check` from `mcp_servers/mdq/tools.py`
- [ ] Remove `fts_rebuild` from `mcp_servers/mdq/tools.py`
- [ ] Remove `FtsConsistencyCheckRequest` from `models.py`
- [ ] Remove `FtsConsistencyCheckResponse` from `models.py`
- [ ] Remove `FtsRebuildRequest` from `models.py`
- [ ] Remove `MdqService.fts_consistency_check()` from `service.py`
- [ ] Remove `MdqService.fts_rebuild()` from `service.py`
- [ ] Remove unused imports from `service.py`
- [ ] Remove related references from `shared/tool_constants.py`
- [ ] Confirm `ToolRegistry` no longer registers the removed tools
- [ ] Remove related tool safety tier entries
- [ ] Remove related config/tool definition examples
- [ ] Update MDQ tool documentation to list only supported tools

### If restoration is selected

- [ ] Add dispatch handlers to `server.py`
- [ ] Add admin safety tier classification
- [ ] Add serialization/concurrency protection for `fts_rebuild`
- [ ] Add audit enrichment for both admin tools
- [ ] Add tests for successful dispatch
- [ ] Add tests for failure behavior
- [ ] Add docs for admin tool usage and operational risk

## Acceptance criteria

- `/v1/tools` and `_DISPATCH_TABLE` are consistent
- Every MDQ tool exposed through schema is dispatchable
- `ToolRegistry` contents match supported MDQ runtime tools
- Routing drift validation passes
- No stale FTS admin compatibility path remains
- Tests fail if schema-only tools are introduced again

## Recommended direction

Remove the admin tools unless there is a current operational requirement to expose FTS maintenance through MCP.
