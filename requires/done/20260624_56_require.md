# Reduce maintenance-policy dependence on broad load_all() config merges

## Summary
Limit maintenance-policy config reads to the minimum necessary source instead of depending broadly on merged global config.

## Background
The maintenance layer currently reads some policy values through `ConfigLoader().load_all()`. This inherits the same broad-merge and silent-skip risk profile as the global config loader path.

## Objective
Make maintenance behavior depend on the smallest necessary configuration surface.

## In Scope
- `maintenance.py`
- `config_loader.py` if needed for helper support

## Required Changes
- Review maintenance functions that consume merged global config
- Narrow each config read to the minimum necessary source where practical
- Keep behavior unchanged unless a deliberate compatibility change is documented

## Acceptance Criteria
- Maintenance-policy code no longer depends on a broader config merge than necessary
- Silent config-skip risk is reduced for maintenance operations
- Relevant policy values remain easy to locate and reason about

## Out of Scope
- Redesigning all maintenance operations
- Changing maintenance scheduling policy

## Implementation Notes
Focus on dependency reduction and clarity. Maintenance code should not need unrelated config context.
