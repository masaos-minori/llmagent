# Decide and align `DocumentStore.chunk_insert()` with chunk metadata schema

## Priority

Medium

## Summary

Decide whether `SQLiteDocumentStore.chunk_insert()` should support `chunk_type` and `source_file`, and implement or document the decision.

## Background

The canonical `chunks` schema should include `chunk_type` and `source_file`.

The ingestion implementation inserts these fields directly.

However, `SQLiteDocumentStore.chunk_insert()` currently accepts only:

- `doc_id`
- `chunk_index`
- `content`
- `normalized_content`

## Goal

Make the DB store abstraction consistent with the schema, or clearly document why ingestion bypasses it.

## Target files

- `store_impl.py`
- `store_protocols.py`
- `ingester.py`
- `90_shared_05_db_api_and_operations.md`

## Scope

### In scope

- Review `DocumentStore` protocol.
- Review `SQLiteDocumentStore`.
- Review RAG ingestion call sites.
- Update protocol and implementation if selected.

### Out of scope

- Backward-compatible API shims.
- Existing DB migration.

## Implementation options

### Option A: Extend the protocol and implementation

Update `chunk_insert()` arguments to include:

- `chunk_type: str = "text"`
- `source_file: str = ""`

### Option B: Keep the current store API and document bypass

If `RagIngester._insert_chunk()` is the supported insertion path for full RAG metadata, document that `SQLiteDocumentStore.chunk_insert()` is a minimal adapter and does not cover full ingestion metadata.

## Acceptance criteria

- Either `DocumentStore.chunk_insert()` supports `chunk_type` and `source_file`, or documentation explains why it does not.
- No ambiguity remains between schema fields and store API fields.
- Tests are updated for the selected option.

## Validation

- Run DB store tests.
- Run RAG ingestion tests if available.

## Confirmation required

- Confirm whether all chunk insertion should go through `DocumentStore`.
- Confirm whether direct SQL insertion in `RagIngester` is intentional.
