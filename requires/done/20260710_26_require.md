# [High] Align workflow-required startup, always-on approval gate, and workflow approval CLI behavior

## Background

The current implementation requires a valid workflow definition and workflow schema during startup.

Startup performs workflow checks before Orchestrator initialization:

- `config/workflows/default.json` must exist.
- `workflow.sqlite` must contain the required workflow schema.
- `Orchestrator.__init__()` loads the workflow definition and fails if it cannot be loaded.

The current `WorkflowEngine.run()` executes:

1. `plan`
2. `execute`
3. workflow-level approval gate
4. `verify`

There is no active `workflow_require_approval` switch in the confirmed implementation. The approval gate is always reached after `execute` and before `verify`.

The CLI implementation also requires explicit approval IDs:

- `/approve <approval_id> [reason]`
- `/reject <approval_id> [reason]`

However, documentation and error messages still contain stale or inconsistent references:

- direct execution fallback when workflow definition is missing
- `workflow_require_approval`
- `/approve [reason]`
- `/reject [reason]`
- `/workflow status` even though the command is not confirmed in the registry

## Goal

Make startup behavior, approval gate behavior, CLI help, command errors, and documentation match the actual workflow implementation.
