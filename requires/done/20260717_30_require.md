# [Medium] Expand MDQ concurrency and boundary guardrail tests

## Priority

Medium

## Summary

Expand tests around MDQ concurrency and MDQ/RAG boundary enforcement.

This issue consolidates:

- index operation serialization test coverage
- MDQ/RAG database boundary tests
- agent-layer direct database access guardrails

## Target files

- `tests/test_mdq_index_serialization.py`
- `tests/test_mdq_rag_boundary.py`
- `tests/test_tool_server_layer_consistency.py`
- `scripts/mcp_servers/mdq/`
- `scripts/mcp_servers/rag_pipeline/`
- `agent/`
- `shared/`
- `docs/04_mcp_04_04_mdq.md`
- related RAG documentation files

## Background

MDQ indexing operations use an internal `_index_lock`.

Existing tests verify concurrent `refresh_index` calls are serialized.

MDQ write tools are also declared as write and `requires_serial`, and cross-layer tests verify basic write/serial flags.

Separately, an existing boundary test verifies that MDQ code does not reference `rag.sqlite`.

These are good guardrails, but coverage can be expanded.

## Reason for change

Concurrency guardrails should cover all mutation combinations, not only `refresh_index` with `refresh_index`.

Boundary tests should be bidirectional:

- MDQ should not depend on RAG internals.
- RAG should not depend on MDQ internals.
- agent runtime code should not directly access MDQ/RAG SQLite databases.

This prevents future architectural drift.

## Intent

Preserve existing concurrency and boundary design.

This issue is mainly about test coverage and guardrails, not changing runtime behavior.

## Required changes

### 1. Expand serialization tests

Add tests for:

- concurrent `index_paths` with `index_paths`
- concurrent `index_paths` with `refresh_index`
- `_is_indexing` warning visibility during active indexing, if practical

Use the same style as the existing serialization test:

- monkeypatch the protected section
- record start and end events
- verify operations do not interleave

### 2. Strengthen MDQ/RAG boundary tests

Add source scanning tests for:

- no `rag.sqlite` reference in MDQ code
- no `mdq.sqlite` reference in RAG pipeline code
- no direct MDQ/RAG DB access from agent runtime code
- no direct MDQ/RAG DB access from shared runtime code unless explicitly allowed

### 3. Use narrow allowlists

If legitimate exceptions exist, define a narrow allowlist with comments explaining why each exception is allowed.

## Acceptance criteria

- `index_paths` calls are verified to serialize.
- `index_paths` and `refresh_index` are verified to serialize with each other.
- MDQ code does not reference RAG DB internals.
- RAG code does not reference MDQ DB internals.
- agent runtime code does not directly access MDQ/RAG SQLite files.
- Test failure messages include violating file paths.
- Existing cross-layer tests continue to pass.

## Test plan

Add tests for:

- `index_paths` with `index_paths`
- `index_paths` with `refresh_index`
- scanning MDQ directory
- scanning RAG directory
- scanning agent directory
- scanning shared directory if applicable

## Notes for implementer

Do not change the concurrency model unless tests reveal an actual bug.

Do not unify `requires_serial` and `is_side_effect()` in this issue.

Those mechanisms are intentionally separate today.
