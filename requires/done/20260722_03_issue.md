# Update /v1/tools documentation to match current discovery expectations

## Priority

High

## Scope

Documentation-first. Some server implementations may need follow-up if their `/v1/tools` responses do not match the documented contract.

## Target documentation sections

- `04_mcp_02_01_endpoints-and-transport.md`
  - `HTTP ツール一覧: /v1/tools`
- `04_mcp_03_06_tool-runtime-availability-metadata.md`
- `04_mcp_06_14_new-tool-registration-procedure.md`
- Server catalog sections where `/v1/tools` behavior is described

## Problem

Some documentation still describes `/v1/tools` as a minimal list of tool names and descriptions.

The current discovery service expects richer metadata and validates more fields.

If server authors follow the minimal documentation, new tools may fail live discovery or lose important metadata.

## Current discovery expectations

Required per-tool fields:

- `name`
- `description`
- `inputSchema` or `input_schema`

Optional per-tool fields:

- `status`
- `is_write`
- `requires_serial`
- `resource_scope`
- `enabled`
- `capabilities`

Common top-level fields may include:

- `schema_version`
- `tools`

Server-specific responses may also include:

- `server_key`
- `config_dependent`
- `disabled_reason`

## Required documentation changes

Update the `/v1/tools` endpoint documentation to show the current expected response shape.

Include a full example with:

- `schema_version`
- `tools`
- `name`
- `description`
- `inputSchema`
- `status`
- `server_key`
- `config_dependent`
- `enabled`
- `disabled_reason`
- `is_write`
- `requires_serial`
- `resource_scope`
- `capabilities`

Clearly mark which fields are required and which are optional.

## Important clarification

Do not document deferred fields such as `disabled_code` as currently implemented.

## Reason for change

`/v1/tools` is no longer just an informational endpoint. It is the source used to build `RuntimeToolRegistry`. Incorrect documentation here directly causes routing, visibility, and diagnostics problems.

## Acceptance criteria

- `/v1/tools` docs match the validation behavior of `McpToolDiscoveryService`.
- Required and optional fields are clearly separated.
- New server authors can implement a valid `/v1/tools` endpoint from the docs alone.
- Deferred fields are clearly separated from current implementation.
