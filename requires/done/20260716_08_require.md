## Background

The MDQ tool definitions are currently spread across multiple layers: the LLM-visible schema in `tools.py`, runtime dispatch in `server.py`, service methods in `service.py`, static tool ownership sets in `tool_constants.py`, and registry population in `tool_registry.py`.

`server.py` documents and dispatches 7 MDQ tools. `tools.py` defines additional admin tools, including `fts_consistency_check` and `fts_rebuild`. `tool_constants.py` also includes those admin tools in `MDQ_TOOLS` and `MDQ_WRITE_TOOLS`. `tool_registry.py` states that `ToolRegistry` is the primary source of truth for MCP tool ownership and routing, and that it is populated from `tool_constants.py`. 【1-bfd1ab】【2-03d74e】【3-c9d102】【4-2af89e】

## Problem

The current implementation can create a tool-surface drift where a tool may be:

- exposed in `/v1/tools` but not dispatchable,
- registered in `ToolRegistry` but not executable,
- classified as write/admin in `tool_constants.py` but missing from runtime dispatch,
- documented as unsupported or absent while still visible to routing or validation layers.

This is more serious than a documentation mismatch. Since `ToolRegistry` is the routing authority and is populated from `tool_constants.py`, stale entries in `MDQ_TOOLS` can affect ownership and routing assumptions even if `server.py` does not dispatch them. 【4-2af89e】【3-c9d102】

## Reason for change

The system must not expose or route tools that cannot be executed. A mismatch among `TOOL_LIST`, `_DISPATCH_TABLE`, `MDQ_TOOLS`, and `MDQ_WRITE_TOOLS` makes the MCP tool contract unreliable. It also weakens startup drift validation because the expected source of truth is not aligned with the live server surface.

This change is needed to make the MDQ tool surface deterministic, auditable, and safe for AI tool routing.

## Design intent

Make one canonical MDQ tool set visible across all relevant layers:

- LLM-visible tool schema
- Runtime dispatch
- Tool ownership registry
- Write/admin classification
- Startup drift validation
- Tests

After this issue is complete, adding or removing an MDQ tool should require updating all relevant layers deliberately, and CI should fail if drift is introduced.

## Implementation instructions

Choose one of the following approaches and apply it consistently.

### Option A: Remove the admin tools

Use this option if `fts_consistency_check` and `fts_rebuild` are not intended to be publicly callable MDQ tools.

- Remove `fts_consistency_check` from `mcp_servers/mdq/tools.py`.
- Remove `fts_rebuild` from `mcp_servers/mdq/tools.py`.
- Remove both tools from `MDQ_TOOLS` in `shared/tool_constants.py`.
- Remove `fts_rebuild` from `MDQ_WRITE_TOOLS` in `shared/tool_constants.py`.
- Remove related service methods and imports from `mcp_servers/mdq/service.py` if they are no longer used.
- Update `server.py` documentation to reflect the final supported tool count.
- Ensure config references in `config/agent.toml` do not mention removed tools.

### Option B: Fully restore the admin tools

Use this option only if the admin tools are intentionally supported.

- Add handler functions in `mcp_servers/mdq/server.py`.
- Register them in `_DISPATCH_TABLE`.
- Keep them in `TOOL_LIST`.
- Keep them in `MDQ_TOOLS`.
- Ensure `fts_rebuild` remains in `MDQ_WRITE_TOOLS`.
- Add audit logging for both admin operations.
- Ensure safety tier, approval policy, and write-risk classification are explicit.
- Update documentation to distinguish production tools from admin tools.

## Required tests

Add or update tests to validate the following invariants:

- The set of names in `TOOL_LIST` equals the set of keys in `_DISPATCH_TABLE`.
- The set of names in `TOOL_LIST` equals `MDQ_TOOLS`.
- `MDQ_WRITE_TOOLS` is a subset of dispatchable MDQ tools.
- Removed tools cannot remain in `tool_constants.py`.
- Startup/live validation detects drift between ToolRegistry and `/v1/tools`.
- Test failure messages include the missing or extra tool names.

## Acceptance criteria

- No MDQ tool is exposed unless it is dispatchable.
- No MDQ tool is registered in `ToolRegistry` unless it is intentionally owned by MDQ.
- No MDQ write/admin tool exists without explicit classification.
- The MDQ server docstring matches the final tool surface.
- CI fails on future drift between `TOOL_LIST`, `_DISPATCH_TABLE`, and `MDQ_TOOLS`.
- The final behavior is documented.

## Non-goals

- Do not change RAG/session vector schema in this issue.
- Do not implement new admin tools beyond resolving the existing `fts_consistency_check` and `fts_rebuild` mismatch.
- Do not change unrelated MCP server tool ownership.
