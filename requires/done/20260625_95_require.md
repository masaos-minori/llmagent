# Title
Optimize dedup, retention, and rebuild policies for long-term memory use

## Priority
Medium

## Labels
- enhancement
- memory

## Dependencies
- Issue 2 recommended

## Summary
Tune dedup, retention, and rebuild behavior so the memory layer remains useful over long-term accumulation. The current system already has `DedupPolicy`, retention days, `/memory prune`, `/memory rebuild`, and uses JSONL as the canonical source.

## Background
The current memory design is append-only in JSONL and rebuilds SQLite indices from JSONL when necessary. Retention and dedup mechanisms exist, but they are not strongly differentiated by memory type or source type.

## Problem
If all memory types are treated identically over long periods, useful rule/decision memory and noisy conversational memory can accumulate in ways that reduce retrieval quality.

## Required Changes
- Review dedup policy per memory type / source type.
- Review retention policy per memory type / source type.
- Strengthen docs around JSONL as canonical source and operational rebuild flow.
- Keep `/memory prune` and `/memory rebuild` understandable as long-term maintenance tools.

## Acceptance Criteria
- Long-term accumulation behavior is easier to control.
- Dedup / prune / rebuild usage is explainable in docs.
- Important source types are less likely to be lost or drowned in noise.

## Out of Scope
- Replacing JSONL as canonical source
- Replacing the storage backend
