# Simplify Command Examples and JSON Examples in Design Documents

## Priority

Medium

## Summary

Simplify long command examples and JSON examples in design documents.

Keep only representative examples that explain design intent, operational boundaries, failure behavior, compatibility, or data lifecycle.

Move, compress, or replace procedural details that are better confirmed through execution guides, command help, source code, generated schemas, or actual runtime files.

## Reason for Change

Some design documents contain long command sequences, repeated execution examples, full JSON samples, and payload examples that closely mirror implementation structures.

These examples can be useful for operators, but they can also make design documents harder to read and maintain.

Long examples create several problems.

First, command examples become stale when scripts, arguments, paths, or deployment steps change.

Second, full JSON examples become stale when DTOs, schemas, or payload fields change.

Third, repeated examples make it harder to identify the actual design point being explained.

Fourth, AI agents may incorrectly treat an old example as an authoritative operational command or schema.

A design document should explain why a workflow exists, what boundary it represents, what can fail, what operators must be careful about, and which behavior is intentionally constrained.

It should not copy long procedural manuals or full implementation-shaped payloads unless they are necessary to understand design or operational safety.

## Implementation Intent

The intent is not to remove all examples.

The intent is to keep examples only when they clarify one of the following:

- design intent
- required execution order
- operational boundary
- failure behavior
- compatibility requirement
- data lifecycle
- irreversible or dangerous operation
- local file versus HTTP behavior
- workflow deployment requirement
- database initialization responsibility

Examples should be reduced to minimal representative examples.

The documentation should avoid duplicating long command manuals, full CLI tutorials, or full DTO-shaped JSON payloads.

## Required Changes

For each target document:

1. Identify long command examples and JSON examples.
2. Classify each example before editing.
3. Use the following classification categories:
   - required for design understanding
   - required for operational safety
   - representative minimal example
   - procedural detail better suited for an execution guide
   - implementation-derived payload copy
   - removable duplicate
4. Keep examples that clarify design or operational behavior.
5. Compress repeated examples.
6. Replace full copied JSON payloads with minimal examples where possible.
7. Preserve compatibility-sensitive fields where necessary.
8. Preserve operationally important failure-mode instructions.
9. Preserve required deployment, initialization, and validation sequences.
10. Ensure remaining command examples are formatted as valid Markdown code blocks in the source document.
11. Ensure remaining JSON examples are formatted as valid Markdown code blocks in the source document.
12. Do not place commands on the same line as prose.

## Content to Remove or Compress

Remove or compress:

- repeated health-check commands
- multiple similar CLI invocation examples
- long command sequences that do not add design insight
- full JSON payloads that simply mirror DTOs
- full output examples mechanically generated by implementation
- command examples duplicated across multiple documents
- inline commands embedded directly in prose
- overly detailed examples that belong in command help or an operations runbook

## Content to Keep

Keep:

- minimal representative command examples
- startup-critical commands
- initialization sequence requirements
- workflow definition validation requirements
- database initialization responsibilities
- RAG file lifecycle examples where design-relevant
- `file://` versus `http(s)://` behavior differences
- JSON fields that affect compatibility or persistence
- failure-mode examples that operators must understand
- warnings for dangerous or irreversible operations
- examples that explain process boundaries or responsibility boundaries

## Suggested Replacement Style

Use concise Japanese explanations.

For command examples, keep only minimal representative examples.

For JSON examples, keep only minimal representative examples.

In the actual source document, remaining shell examples should be fenced as `bash`.

In the actual source document, remaining JSON examples should be fenced as `json`.

Avoid placing commands or JSON fragments on the same line as prose.

Suggested Japanese wording:

- 詳細な引数は各コマンドのヘルプまたは実行ガイドを参照する。
- ここでは設計上重要な実行順序と失敗ポイントのみを扱う。
- 完全な JSON フィールド一覧は実装上の DTO またはスキーマ定義を参照する。
- この例は操作手順の完全な一覧ではなく、責務境界を示すための代表例である。

## Acceptance Criteria

- Long command sequences are reduced where they do not add design value.
- Repeated command examples are removed or consolidated.
- JSON examples are reduced to design-relevant minimum examples.
- Startup-critical and failure-mode commands remain documented.
- Compatibility-sensitive JSON fields remain documented.
- Remaining examples are properly formatted in Markdown.
- Prose and commands are not joined on the same line.
- New or rewritten documentation text is written in Japanese.
- Markdown renders correctly.

## Definition of Done

- Target documents are more concise and easier to read.
- Design documents no longer duplicate long procedural manuals.
- Remaining examples serve a clear design or operational purpose.
- No operationally critical warning is removed.
- No source code is modified.
- No ADR documents are created.
- `00_index.md` is not modified.
- No unrelated document cleanup is performed.

## Out of Scope

- Do not remove required deployment steps.
- Do not remove Workflow deployment validation steps.
- Do not remove database initialization requirements.
- Do not remove operational failure-mode guidance.
- Do not remove compatibility-sensitive JSON fields.
- Do not modify source code.
- Do not migrate Known Issues documents.
- Do not resolve Needs Confirmation items.
- Do not bulk-format unrelated documents.
- Do not create new governance documents.

## AI Implementation Instruction

Work document by document.

Before removing an example, decide whether it exists for design understanding, operational safety, compatibility, or simple implementation reference.

Remove or compress only simple implementation examples and repeated procedural examples.

Keep representative minimal examples.

Use Japanese for all new or rewritten documentation text.

Ensure remaining command and JSON examples are valid Markdown and do not break the document.

If an example appears operationally important but too long for a design document, compress it and add a reference to the relevant execution guide or operations document.
