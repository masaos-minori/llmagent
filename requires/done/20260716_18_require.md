## Consolidated from

- Add MCP watchdog settings to documented operational constraints

## Background

The Agent documentation notes that MCP watchdog behavior is controlled by watchdog-related settings, including an interval and maximum restarts. It also notes that these settings are not included in the main constraints table.

This is primarily a documentation and operations issue, not a runtime behavior change.

## Problem

MCP watchdog behavior is operationally important because it affects subprocess lifecycle, health monitoring, and recovery behavior. If watchdog settings are only mentioned as a current behavior note, operators may miss them when reviewing configuration and operational constraints.

## Reason for change

Operational settings should be discoverable in the formal configuration and operations reference. Watchdog interval, disabled behavior, and restart limits are part of runtime reliability and should not remain as incidental notes.

## Design intent

Document MCP watchdog configuration as an official operational constraint.

The documentation should make clear:

- which settings control watchdog behavior,
- what their defaults are,
- how to disable the watchdog,
- how maximum restart behavior works,
- how watchdog state appears in startup or health output if applicable.

## Implementation instructions

- Confirm the setting names for watchdog interval and maximum restarts.
- Confirm default values from config dataclasses.
- Confirm disabled behavior from runtime implementation.
- Add the settings to the main constraints or configuration table.
- Add operational notes explaining how watchdog affects MCP subprocess lifecycle.
- Ensure documentation matches `repl_health.py` and configuration dataclasses.
- Update any related operations documentation.

## Acceptance criteria

- Watchdog interval is documented with default and disabled behavior.
- Watchdog maximum restart behavior is documented.
- Operations/configuration documentation includes watchdog behavior.
- Documentation matches implementation.
- No runtime behavior changes are introduced.

## Non-goals

- Do not change watchdog runtime behavior.
- Do not change server lifecycle management.
- Do not modify restart policy unless separately scoped.
- Do not add new watchdog features.

## Disposition: REJECTED (2026-07-17)

Rejected without a work plan, in favor of `requires/done/20260716_20_require.md`
("Remove the MCP watchdog feature"), which directly contradicts this requirement's
premise. `requires/done/20260716_20_require.md`'s Background section states
explicitly: "The user has decided this feature is unnecessary and requested its
complete removal" — an explicit, higher-priority (Medium vs. this requirement's Low)
user decision, versus this requirement's own origin as a general documentation-audit
observation (not a direct user request).

Since the watchdog feature itself is slated for complete removal
(`implementations/20260717-001510_mcp_watchdog_removal.md`, not yet implemented as of
this rejection), formally documenting its settings as an "official operational
constraint" would produce documentation that is immediately obsolete once the removal
lands. If the removal plan is ever reversed, this requirement should be reopened
rather than resurrected from this rejected copy, since the surrounding doc structure
may have changed by then.
