# Title
Replace or harden the global shared SQLite connection model

## Summary
Review the use of a single global SQLite connection shared by publish, replay, subscribe, and the DLQ background task, and move to a clearer connection strategy.

## Background
`app.py` stores a global `_db` connection initialized in FastAPI lifespan.
The connection is opened with `check_same_thread=False`, and multiple paths use that shared handle.

## Problem
The current shared-connection model is simple, but it makes concurrency behavior, failure isolation, and runtime debugging harder as publish, polling, and DLQ tasks all depend on the same handle.

## Required Changes
- Document the current connection-lifecycle model explicitly.
- Decide whether to keep one shared connection or move to separated read/write or per-request/task connections.
- Add diagnostics that make DB failures traceable to the calling path.
- Preserve WAL-based behavior and schema initialization semantics.

## Acceptance Criteria
- The DB connection strategy is explicit and justified.
- Runtime behavior under concurrent publish/poll/DLQ activity is easier to reason about.
- DB-related failures are easier to diagnose.

## Out of Scope
- Replacing SQLite with another database
- Redesigning the API surface

## Components
- `scripts/eventbus/app.py`
- `scripts/eventbus/db.py`
- `scripts/eventbus/schema.sql`
- Event Bus operations / runtime docs
