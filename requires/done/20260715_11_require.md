# Normalize configuration schema and reload boundaries for removed Memory keys and GitOps security fields

## Priority

High

## Type

Configuration / Reload / Validation

## Primary Files

- `agent/config_dataclasses.py`
- `agent/config_builders.py`
- `agent/services/config_reload.py`

## Do Not Modify in This Issue

- `agent/factory.py`
- `agent/memory/*`
- `agent/commands/*`
- `agent/tool_policy.py`
- `agent/tool_approval.py`
- Documentation files
- Test files, except config-specific tests if required

## Background

Legacy Memory configuration fields are still present in the schema and builder/reload logic:

- `use_memory_layer`
- `memory_jsonl_dir`
- `memory_embed_enabled`

GitOps-related fields also have unclear or incomplete runtime and reload behavior:

- `gitops_push_blocked`
- `gitops_force_push_blocked`
- `gitops_protected_branches`

## Problem

The legacy Memory keys keep the following deprecated behaviors alive:

- Memory disabled mode
- JSONL runtime persistence
- Embedding-disabled / FTS-only behavior

GitOps fields are security-sensitive and must have clear load, reload, and runtime semantics.

## Implementation Tasks

### Removed Memory keys

- [ ] Remove `use_memory_layer` from `MemoryConfig`.
- [ ] Remove `memory_jsonl_dir` from `MemoryConfig`.
- [ ] Remove `memory_embed_enabled` from `MemoryConfig`.
- [ ] Remove these fields from `_build_memory_config()`.
- [ ] Remove cross-field validation that depends on these fields.
- [ ] Remove `use_memory_layer` from startup-only reload detection.
- [ ] Add removed-key validation for:
  - `use_memory_layer`
  - `memory_jsonl_dir`
  - `memory_embed_enabled`

### GitOps security fields

- [ ] Decide whether `gitops_force_push_blocked` and `gitops_protected_branches` should be retained or removed.
- [ ] If retained, keep them in the config schema and define explicit reload behavior.
- [ ] If removed, reject them through removed-key validation.
- [ ] Ensure `gitops_push_blocked` has explicit reload behavior.
- [ ] Clearly classify each field as one of:
  - hot-reloadable
  - restart-required
  - removed
  - invalid

### Error quality

- [ ] Include the original exception message in `ConfigLoadError`.
- [ ] Add reload preflight validation before mutating runtime config where feasible.

## Acceptance Criteria

- [ ] Removed Memory keys are rejected at both load time and reload time.
- [ ] `use_memory_layer` is no longer reported as startup-only.
- [ ] `memory_jsonl_dir` is no longer accepted.
- [ ] `memory_embed_enabled` is no longer accepted.
- [ ] GitOps security fields have clear schema and reload semantics.
- [ ] Config load failures include the original error reason.
- [ ] Failed reload validation does not partially mutate runtime config.

## Dependencies

- Runtime Memory construction cleanup is handled in a separate issue.
- Tool policy enforcement is handled in a separate issue.
