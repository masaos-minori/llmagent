# Fix side-effect scheduling and ToolSpec metadata generation

## Priority

High

## Type

Implementation / Safety / Testing

## Target Files

- `agent/tool_scheduler.py`
- `agent/tool_runner.py`
- `shared/tool_spec.py`
- `tests/test_tool_scheduler.py`
- `tests/test_tool_runner.py` if available

## Summary

Fix the scheduler and ToolSpec metadata generation so that tools with side effects are not accidentally executed in unsafe parallel groups.

This issue covers two closely related problems:

1. `write_first` groups are currently scheduled before read-only groups, but the tools inside the group may still run concurrently.
2. ToolSpec metadata generation must use shared side-effect classification so that non-file side-effecting tools, such as Git, GitHub, CI/CD, RAG delete, and MDQ admin/write tools, are treated safely even when explicit metadata is missing from `tool_definitions`.

## Background

The scheduler uses `ToolSpec` metadata to decide how tool calls are grouped:

- `requires_serial=True` tools become serial barriers.
- Write tools with the same `resource_scope` are grouped and serialized within that scope.
- Write tools without `resource_scope` go into a `write_first` group.
- Remaining tools are treated as read-only and can run in parallel.

This model is reasonable, but it depends entirely on correct `ToolSpec` metadata.

There are two safety gaps:

### Gap 1: `write_first` is not actually serial inside the group

Write tools without `resource_scope` are placed into a `write_first` group, but that group is currently scheduled with `serialize_flags=[False]`.

That means the group runs before read-only tools, but the tools inside the group can still be executed concurrently.

For tools that mutate state and do not have a precise resource scope, concurrent execution is unsafe. If the resource scope is unknown, the system should be conservative.

### Gap 2: ToolSpec generation is too narrow

The current metadata generation path primarily recognizes file write/delete tools and shell tools. Other tool families also have side effects:

- Git write tools
- GitHub write/dangerous tools
- CI/CD workflow trigger tools
- RAG deletion tools
- MDQ index/admin tools

If these tools are not marked as write or serial candidates, the scheduler may place them in the default parallel group.

## Why This Needs to Be Fixed

The scheduler is part of the safety boundary for tool execution. If write/admin tools are incorrectly treated as read-only, the agent can perform concurrent mutations that were never intended to run in parallel.

Concrete examples:

- Two Git operations may mutate the same repository state concurrently.
- A GitHub merge and file update may be issued at the same time.
- `trigger_workflow` may dispatch multiple external workflows concurrently.
- `fts_rebuild` may run while another MDQ index operation is active.
- `rag_delete_document` may run in parallel with RAG query/debug operations.

These are not purely performance concerns. They can cause inconsistent state, hard-to-debug failures, or unsafe external side effects.

## Fix Intent

The intent is to make the scheduler safe by default:

- If a tool is known to mutate state, treat it as write-capable even if the tool definition lacks metadata.
- If a write tool has no resource scope, serialize it conservatively.
- If a tool is admin-like or operationally dangerous, allow it to become a serial barrier.
- Keep read-only tools parallel where safe.

This should preserve concurrency for read-only calls while making mutation handling deterministic and safer.

## Required Changes

### 1. Serialize `write_first` groups

Update `agent/tool_scheduler.py`.

Current unsafe behavior:

- `write_first` scheduled with `serialize_flags=[False]`

Required behavior:

- `write_first` must use `serialize_flags=[True]`

Alternative acceptable design:

- Split each write-first tool into a single-tool scheduled batch.

The selected implementation should be simple and clearly tested.

### 2. Use shared side-effect classification when building ToolSpec

Update `_build_tool_meta()` in `agent/tool_runner.py`.

The metadata builder should use shared classification, such as `is_side_effect()` and/or shared constants, as a safe fallback.

At minimum, these tools/tool families must be treated as side-effecting:

- File write tools
- File delete tools
- Shell tools
- Git write tools
- GitHub write tools
- GitHub dangerous tools
- CI/CD write tools, especially `trigger_workflow`
- RAG write tools, especially `rag_delete_document`
- MDQ write/admin tools, especially `index_paths`, `refresh_index`, and `fts_rebuild`

### 3. Preserve explicit metadata where safe

If `tool_definitions` explicitly provides:

- `resource_scope`
- `requires_serial`
- `is_write`

then that metadata should still be respected.

However, explicit metadata should not make a known side-effecting tool less safe unless there is a deliberate and documented reason.

## Acceptance Criteria

- `write_first` groups are serialized inside the group.
- `trigger_workflow` does not fall into the default read-only parallel group.
- `rag_delete_document` does not fall into the default read-only parallel group.
- `fts_rebuild` does not fall into the default read-only parallel group.
- Git write tools are treated as write tools.
- GitHub write and dangerous tools are treated as write tools.
- Shell tools remain serial barriers or equivalent safe execution units.
- Read-only tools remain eligible for parallel execution.
- Existing scheduler tests continue to pass after updating expectations.
- New regression tests cover write-first serialization and side-effect ToolSpec generation.

## Suggested Tests

- `test_write_first_group_is_serialized`
- `test_trigger_workflow_generates_write_toolspec`
- `test_rag_delete_document_generates_write_toolspec`
- `test_fts_rebuild_generates_write_or_serial_toolspec`
- `test_github_write_tools_do_not_enter_parallel_read_group`
- `test_git_write_tools_do_not_enter_parallel_read_group`
- `test_read_only_tools_remain_parallel`

## Non-Goals

- Do not disable all parallel tool execution.
- Do not require every tool definition to manually declare scheduling metadata.
- Do not make read-only tools serial unless there is a clear conflict reason.

## Implementation Notes

Prefer centralizing side-effect classification rather than duplicating tool name sets in scheduler-related code.

If additional constants are required, add them in `shared/tool_constants.py` and consume them through `shared/tool_executor_helpers.is_side_effect()` or an equivalent shared helper.
