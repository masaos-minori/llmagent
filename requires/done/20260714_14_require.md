# [Medium] Add a `/diff` slash command to review files the agent wrote/edited during the session

## Labels
priority/medium, area/agent, area/cli, feature

## Summary
There is currently no way for a user to review, in one place, which files the agent wrote or
edited during the current session and what changed. Investigation shows the two building
blocks already exist (a diff-generation routine in the file-write MCP server and a
`git_diff` MCP tool), but nothing wires them into a slash command, and no component
persists a list of touched file paths across the session. This requirement proposes adding
a `/diff` command, following the existing `CommandDef` / `CommandRegistry` mixin pattern
used by `/history`, `/export`, and `/compact`.

## Background
- Slash commands are registered in a single source of truth,
  `scripts/agent/commands/command_defs_list.py` (`_COMMANDS: list[CommandDef]`, comment at
  lines 8-17 spells out the add-a-command procedure: append a `CommandDef` entry, implement a
  `_cmd_<name>` handler in the appropriate mixin, do not add `CommandDef` entries elsewhere).
  Handlers live in mixins listed in `scripts/agent/commands/registry.py` (lines 10-24), e.g.
  `cmd_context.py` (`_ContextMixin`: `/context`, `/clear`, `/undo`, `/history`, `/system`) and
  `cmd_rag_export.py` (`_RagExportMixin`: `/export`, `/compact`, `/rag`).
- `write_file` (`scripts/mcp_servers/file/write_service.py`, `write_file()` around lines
  121-146) always returns `diff=""` on a real (non-dry-run) write — the unified diff is only
  computed on the dry-run path (`_write_file_dry_run()`, lines 155-179, via
  `_generate_diff()`, lines 68-83, using `difflib.unified_diff`). `edit_file()` does compute
  and return a real diff, but in both cases the diff string is returned to the LLM as the
  tool response and is not persisted or aggregated anywhere.
- The audit log (`scripts/agent/tool_audit.py`, `audit_tool_exec()`, lines 158-197) records
  `resource_scope` and a masked `args_preview` per tool call but not diff bodies or
  post-write file content. Its `artifact_uri` field is only populated by
  `scripts/agent/workflow/workflow_engine.py` for workflow-approval runs, not by ordinary
  REPL tool calls, so it cannot be relied on as a general-purpose change log.
- No `scripts/eventbus/` event type exists for file-content changes.
- The conversation history, `ctx.conv.history`, holds every tool call (including
  `write_file`/`edit_file` calls and their arguments) and is persisted to the DB via
  `scripts/agent/session.py` (`messages.tool_calls` column, `scripts/db/schema_sql.py:80-94`).
  However, `/compact` compresses this history and `/clear` resets it, so it is not a durable
  source of truth for "everything the agent changed this session" once either command has
  run.
- A `git_diff` MCP tool already exists end-to-end: schema in
  `scripts/mcp_servers/git/tools.py` (lines 58-83: `repo_path`, `staged`, `commit` args) and
  implementation in `scripts/mcp_servers/git/service.py` (`git_diff()`, lines 150-157, via
  `format_diff()`). Commands already call MCP tools directly through
  `ctx.services_required.tools.execute(...)`, e.g. `scripts/agent/commands/cmd_mdq.py` and
  `scripts/agent/commands/db_rag_ops.py:36-38` — the same pattern `/diff` would use to invoke
  `git_diff`.
- `docs/05_agent_07_10_cli-and-commands-slash-commands-workflow-debug.md:70-71` documents
  `/compact` and `/export` but has no equivalent entry for a diff/change-review command.

## Required Changes
- Add a `CommandDef("/diff", ...)` entry to `_COMMANDS` in
  `scripts/agent/commands/command_defs_list.py`, and implement `_cmd_diff` in a mixin (e.g.
  `cmd_context.py`, alongside `/history`, given the similar "inspect recent session
  activity" purpose — final placement is an implementation decision, not fixed by this
  requirement).
- `_cmd_diff` must:
  1. Scan `ctx.conv.history` for `write_file`/`edit_file` tool calls made during the current
     session and extract the distinct file paths touched.
  2. For each path, invoke the existing `git_diff` MCP tool
     (`ctx.services_required.tools.execute("git_diff", {"repo_path": ..., "commit": ""})`)
     to obtain a working-tree diff, and print a per-file summary.
  3. Define and implement explicit behavior for paths that are not inside a git repository,
     or inside a repository with no tracked history for that path (e.g. skip with a clear
     one-line notice per file, rather than raising) — this must not abort the whole command
     because one path is untrackable.
- Document the known limitation that this approach only sees tool calls still present in
  `ctx.conv.history`: paths touched before a `/compact` or `/clear` in the same session will
  not appear. Whether this limitation is acceptable, or whether a durable tracking
  mechanism (e.g. a new DB table recording path/diff/timestamp on every `write_file`/
  `edit_file` call, independent of git and of conversation-history retention) is required
  instead, is an open decision left to the design phase (see Unknowns) — this requirement
  does not mandate a new persistence layer.
- Update `docs/05_agent_07_10_cli-and-commands-slash-commands-workflow-debug.md` to add a
  `/diff` entry alongside the existing `/compact`/`/export` documentation.

## Acceptance Criteria
- `/diff` is registered in `command_defs_list.py` and dispatches through `CommandRegistry`
  like any other built-in command.
- Running `/diff` after the agent has written or edited one or more files in the current
  session lists each touched file path and shows its current working-tree diff.
- Running `/diff` when no files were written/edited in the session (or all touched paths are
  outside any git repository) produces a clear "nothing to show" message, not an error or
  stack trace.
- Paths outside a git repository are skipped with an explicit per-file notice rather than
  failing the whole command.
- `docs/05_agent_07_10_cli-and-commands-slash-commands-workflow-debug.md` documents `/diff`.

## Non-Goals
- Do not implement a new DB-backed change-tracking table as part of this requirement; that
  is an open design question (see Unknowns), not a mandated deliverable.
- Do not surface workflow-approval diffs (the `approvals` table / `WorkflowMixin`,
  `/approve`, `/reject`) through this command — that is a separate, already-covered flow.
- Do not add any git write operations (commit, stage, push) to `/diff` — it is a read-only,
  no-side-effect command.

## Unknowns
- How often, in real usage, the agent writes to paths outside a git repository — this
  affects how significant the git-unmanaged-path limitation of the history-scan approach is.
- Whether users need `/diff` to still show changes made before a `/compact` or `/clear` in
  the same session — if yes, a durable tracking mechanism independent of
  `ctx.conv.history` is required; if no, the simpler history-scan approach is sufficient.
