# Clarify the boundary between runtime architecture and reference API chapters

## Summary
Reduce navigation ambiguity by explicitly distinguishing “where a class lives and who calls it” from “what role the component plays in runtime architecture”.

## Background
Both the runtime-architecture chapter and the reference-API chapter are useful for implementation work, but their intended roles overlap enough to cause navigation friction.

## Objective
Document the intended reader use-case for each chapter.

## In Scope
- `05_agent_02_runtime-architecture.md`
- `05_agent_13_reference-api.md`
- `05_agent_00_document-guide.md` if needed for routing updates

## Required Changes
- Add a short “When to use this chapter” note to each file
- Clarify that:
  - runtime architecture = responsibility boundaries and dependency relationships
  - reference API = module location, callers, callees, failure surface
- Update the guide if needed so routing matches the intended usage

## Acceptance Criteria
- Readers can quickly choose the right chapter for lookup vs architecture reasoning
- The guide routes class-lookup questions to the correct chapter
- The two chapters no longer feel redundant

## Out of Scope

- API signature expansion

## Notes
Keep the distinction operational and concrete.
