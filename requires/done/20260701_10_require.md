## Issue 10: [Medium] Resolve workflow database location inconsistency

### Goal

Fix the inconsistency between `/db help` output and the documented workflow database model.

### Background

`cmd_db.py` help text says workflow data lives in `session.sqlite` and that there is no separate workflow DB.
However, the schema defines a workflow database and schema creation uses `SQLiteHelper("workflow")`.

### Scope

- Investigate and fix documentation or help text.
- Prefer aligning CLI help with the actual `workflow.sqlite` model if confirmed.

### Implementation Tasks

- Review `cmd_db.py` `_db_help()` output.
- Review `05_agent_09_data-layer.md`.
- Review `05_agent_04_state-and-persistence.md`.
- Review `05_agent_10_operations-and-observability.md`.
- If `workflow.sqlite` is canonical, update `/db help` note to say:
  - workflow data lives in `workflow.sqlite`
  - `/db` does not expose direct workflow maintenance commands
- If implementation has changed and workflow really lives in `session.sqlite`, update schema/docs accordingly.

### Acceptance Criteria

- CLI help and documentation agree on workflow DB location.
- If `workflow.sqlite` remains canonical, docs explain that `/db session` does not manage workflow data.
- No active documentation says both contradictory things.

### Validation

- Run or inspect `/db help` output.
- Confirm `SQLiteHelper("workflow")` usage.
- Confirm workflow schema creation path.
