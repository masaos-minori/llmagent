# Title
Fail fast in strict mode when all MCP servers are unreachable

## Summary
Change startup validation so strict tool-definition mode no longer skips validation silently when every MCP server is unreachable.

## Background
Current startup behavior skips tool-definition validation entirely when all MCP servers are unreachable, even in strict mode.

## Problem
Strict mode should provide strong startup integrity guarantees.
Skipping validation when zero MCP servers are reachable weakens that guarantee.

## Required Changes
- In strict mode, treat “all MCP servers unreachable” as a startup failure or equivalent hard error.
- Preserve current non-strict behavior unless there is a separate decision to change it.
- Ensure error messages clearly distinguish:
  - all unreachable
  - partial unreachable
  - tool mismatch

## Acceptance Criteria
- Strict mode no longer starts successfully when all MCP servers are unreachable.
- Errors clearly distinguish reachability failure from tool mismatch.
- Documentation reflects the new strict-mode behavior.

## Out of Scope
- Redesigning all readiness checks
- Changing non-strict mode behavior without explicit approval

## AI Implementation Instruction
Implement this as a fail-fast integrity task.

Strict rules:
- Do not weaken strict mode semantics.
- Keep non-strict mode behavior stable unless explicitly changed.
- Error messages must be operator-actionable.
- If the exact interaction with other startup checks is unclear, mark it as: Needs confirmation

Deliverables:
1. Strict-mode fail-fast behavior for all-unreachable case
2. Clear error reporting
3. Updated documentation
