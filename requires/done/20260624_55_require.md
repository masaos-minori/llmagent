# Clarify store.py as the stable DB-store import surface

## Summary
Improve module-level guidance in `db/store.py` so it is unmistakably clear that this file is the stable public import surface, not the place where protocol or implementation logic lives.

## Background
`store.py` already acts as a compatibility re-export layer, but readers can still be unsure whether to modify `store.py`, `store_protocols.py`, or `store_impl.py` when extending the DB store surface.

## Objective
Make the public API boundary of the DB store modules easier to understand.

## In Scope
- `store.py`

## Required Changes
- Strengthen module-level comments explaining:
  - `store.py` = public import surface
  - `store_protocols.py` = abstract contracts
  - `store_impl.py` = SQLite implementations
- Keep backward-compatibility behavior unchanged

## Acceptance Criteria
- A maintainer can tell where to modify code vs where to import from
- `store.py` reads clearly as a façade / compatibility surface
- Public import stability is easier to preserve

## Out of Scope
- Reorganizing the DB-store package
- Changing import paths across the codebase

## Implementation Notes
This is a clarity task. The goal is to reduce maintenance ambiguity, not to change runtime behavior.
