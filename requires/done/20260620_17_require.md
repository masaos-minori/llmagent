# Title
Improve tool scheduling to become dependency-aware instead of round-wide serialization

## Summary
Refine tool execution so that independent read-only tools are not unnecessarily serialized just because one side-effect tool appears in the same round.

## Background
Current behavior serializes the entire tool-call round when any side-effect tool is present.

## Problem
This is safe but too coarse.
It increases latency and reduces concurrency even when some tool calls are logically independent.

## Required Changes
- Distinguish independent read-only tool calls from side-effect tool calls where safe.
- Preserve ordering guarantees for side-effect tools.
- Extend or refine DAG-oriented execution support if available.
- Make scheduling decisions visible in debug or trace output.

## Acceptance Criteria
- Independent read-only tools are not always forced into sequential execution.
- Safety constraints for side-effect tools remain intact.
- Execution order is explainable for debugging.

## Out of Scope
- Replacing the workflow engine
- Building a distributed orchestration system

## AI Implementation Instruction
Implement this as a scheduling-granularity improvement.

Strict rules:
- Do not weaken side-effect safety.
- Do not sacrifice correctness for concurrency.
- Prefer explicit dependency-aware grouping over hidden heuristics.
- Make the resulting execution behavior explainable.
- If a dependency rule is ambiguous, mark it as: Needs confirmation

Deliverables:
1. Improved execution scheduling strategy
2. Debug/trace visibility into execution order decisions
3. Preserved safety behavior for side-effect tools
