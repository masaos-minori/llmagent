# Introduce RuntimeTool and RuntimeToolRegistry

## Priority

High

## Target files

- shared/runtime_tool.py
- shared/runtime_tool_registry.py
- shared/tool_spec.py
- agent/context.py
- tests/shared/test_runtime_tool.py
- tests/shared/test_runtime_tool_registry.py

## Reason

The Agent should no longer own static tool definitions. MCP servers own their tool definitions and expose them through `/v1/tools`.

The Agent needs a normalized runtime representation and a single runtime index for discovered tools. This replaces the current fragmented model where routing, LLM-visible schemas, side-effect classification, scheduler metadata, and policy are derived from different sources.

## Intent

Create RuntimeToolRegistry as the single runtime view for tool execution metadata.

RuntimeToolRegistry should become the shared source for:

- routing
- LLM-visible schema generation
- scheduler metadata
- side-effect detection
- operation type classification
- safety tier lookup
- approval requirement
- policy filtering

## Scope

Add a `RuntimeTool` dataclass with fields such as:

- `name`
- `server_key`
- `server_url`
- `description`
- `input_schema`
- `raw_definition`
- `status`
- `is_write`
- `requires_serial`
- `resource_scope`
- `agent_safety_tier`
- `requires_approval`
- `enabled_for_llm`

Add `RuntimeToolRegistry` with methods such as:

- `resolve(tool_name) -> server_key`
- `get(tool_name) -> RuntimeTool`
- `all_tools()`
- `llm_tool_definitions()`
- `tool_spec_map()`
- `tool_spec_for_call(call_id, name, args)`
- `is_side_effect(tool_name)`
- `classify_operation_type(tool_name)`
- `apply_policy(cfg)`

## Safe defaults

For unknown or insufficiently annotated tools:

- `agent_safety_tier = "WRITE_DANGEROUS"`
- `requires_approval = True`
- `enabled_for_llm = False`
- `requires_serial = True` when uncertainty affects safe parallel execution

## Acceptance criteria

- RuntimeTool can represent MCP `/v1/tools` entries.
- RuntimeToolRegistry resolves tool names to server keys.
- RuntimeToolRegistry can generate LLM-compatible tool schemas.
- RuntimeToolRegistry can generate ToolSpec metadata.
- RuntimeToolRegistry can reapply Agent-side policy without rediscovery.
- Unit tests cover normalization, defaults, lookup, schema generation, and policy application.
