## Goal

Create a migration plan for transitioning existing Known Issues / Inconsistencies documents across all areas to the new common template, without modifying any existing content directly.

## Scope

Create a governance document (`docs/00_governance_08_known-issues-migration-plan.md`) that defines:
- Which existing Known Issues documents are in scope
- Current format summary for each area's Known Issues
- Migration policy emphasizing gradual transition
- Priority criteria for ordering migrations
- Suggested migration order
- Risks and mitigation strategies

## Assumptions

- Five areas have Known Issues / Inconsistencies documents: RAG, MCP, Agent, EventBus, Shared/DB.
- Existing entries must not be modified during planning phase.
- Migration will be split into separate follow-up issues per area.
- Existing IDs and history must be preserved during migration.

## Unknowns

- None identified. All scope items are defined in the issue.

## Affected areas

- `docs/00_governance_08_known-issues-migration-plan.md` — new file

## Design

### Target Files

Investigate the following (existence to be confirmed):
- RAG Known Issues / Inconsistencies
- MCP Known Issues / Inconsistencies
- Agent Known Issues / Inconsistencies
- EventBus Known Issues / Inconsistencies
- Shared/DB Known Issues / Inconsistencies

### Current Format Summary

For each area, record:
- Approximate number of entries
- Whether severity classification exists
- Whether type classification exists
- Whether status tracking exists
- Any unique formatting conventions

### Migration Policy

- No bulk changes to existing documents.
- Each area migrated separately via its own follow-up issue.
- Preserve existing IDs during migration.
- Map old classifications to new values where possible.
- Add migration notes to each entry explaining the change.

### Priority Criteria

Order migrations by:
1. Security impact — safety-related inconsistencies first
2. Operator impact — operational confusion next
3. Document-code mismatch — clear contradictions
4. Implementation bug — code does not match design intent
5. Deprecated description — obsolete documentation
6. Low-risk formatting only — cosmetic inconsistencies last

### Suggested Migration Order

Based on priority criteria, suggest an order (to be refined after reviewing actual files):
1. Agent Known Issues (likely highest security impact)
2. MCP Known Issues (MCP tool specifications affect routing)
3. RAG Known Issues (RAG behavior affects user-facing output)
4. Shared/DB Known Issues (infrastructure-level concerns)
5. EventBus Known Issues (lower direct user impact)

### Risks

- Existing IDs lost during migration → preserve original IDs in mapping table
- Context loss when reformatting → keep original descriptions alongside new format
- Inconsistent migration quality across areas → define minimum acceptance criteria per migration

### Non-Goals

- Modifying existing Known Issues entries during this planning phase.
- Consolidating multiple areas' Known Issues into a single file.
- Resolving individual Known Issues as part of this plan.

### Acceptance Criteria for Future Migration

Each future migration issue should verify:
- All entries from the source document are accounted for
- Original IDs are preserved in a mapping table
- New format fields are populated for every entry
- No content was deleted during migration

## Implementation steps

1. Create `docs/00_governance_08_known-issues-migration-plan.md` with the sections listed below.
2. Investigate each area's Known Issues document to record current format summary.
3. Define priority criteria based on the six categories listed in the issue.
4. Propose suggested migration order based on priority.
5. Document risks and non-goals.

## Validation plan

- Verify all five areas are listed as investigation targets.
- Verify priority criteria cover all six categories from the issue.
- Verify migration order rationale is documented.
- Verify no existing Known Issues documents were modified.

## Risks

- Medium risk: Migration order may need adjustment after reviewing actual files. Mitigation: Make order a suggestion, not a requirement; allow flexibility per area.
- Low risk: Some areas may have no Known Issues documents. Mitigation: Report missing documents rather than assuming they exist.
