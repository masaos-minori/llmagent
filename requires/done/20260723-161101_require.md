## Background

Some design documents contain long command sequences, repeated execution examples, full JSON samples, and payload examples that closely mirror implementation structures. These examples can be useful for operators, but they can also make design documents harder to read and maintain. Long examples create several problems: they become stale when scripts, arguments, paths, or deployment steps change; full JSON examples become stale when DTOs, schemas, or payload fields change; repeated examples make it harder to identify the actual design point being explained; and AI agents may incorrectly treat an old example as an authoritative operational command or schema.

## Problem

Design documents that copy long procedural manuals or full implementation-shaped payloads reduce their value as architectural references. A design document should explain why a workflow exists, what boundary it represents, what can fail, what operators must be careful about, and which behavior is intentionally constrained — not duplicate full command manuals or full DTO-shaped JSON payloads.

## Reason for change

Keep examples only when they clarify one of the following: design intent, required execution order, operational boundary, failure behavior, compatibility requirement, data lifecycle, irreversible or dangerous operation, local file versus HTTP behavior, workflow deployment requirement, or database initialization responsibility. Reduce examples to minimal representative examples. Avoid duplicating long command manuals, full CLI tutorials, or full DTO-shaped JSON payloads.

## Implementation intent

Simplify long command examples and JSON examples in design documents. Keep only representative examples that explain design intent, operational boundaries, failure behavior, compatibility, or data lifecycle. Move, compress, or replace procedural details that are better confirmed through execution guides, command help, source code, generated schemas, or actual runtime files.

## Implementation instructions

For each target document:

1. Identify long command examples and JSON examples
2. Classify each example before editing using the following categories:
   - required for design understanding
   - required for operational safety
   - representative minimal example
   - procedural detail better suited for an execution guide
   - implementation-derived payload copy
   - removable duplicate
3. Keep examples that clarify design or operational behavior
4. Compress repeated examples
5. Replace full copied JSON payloads with minimal examples where possible
6. Preserve compatibility-sensitive fields where necessary
7. Preserve operationally important failure-mode instructions
8. Preserve required deployment, initialization, and validation sequences
9. Ensure remaining command examples are formatted as valid Markdown code blocks in the source document
10. Ensure remaining JSON examples are formatted as valid Markdown code blocks in the source document
11. Do not place commands on the same line as prose

Content to remove or compress:
- Repeated health-check commands
- Multiple similar CLI invocation examples
- Long command sequences that do not add design insight
- Full JSON payloads that simply mirror DTOs
- Full output examples mechanically generated by implementation
- Command examples duplicated across multiple documents
- Inline commands embedded directly in prose
- Overly detailed examples that belong in command help or an operations runbook

Content to keep:
- Minimal representative command examples
- Startup-critical commands
- Initialization sequence requirements
- Workflow definition validation requirements
- Database initialization responsibilities
- RAG file lifecycle examples where design-relevant
- `file://` versus `http(s)://` behavior differences
- JSON fields that affect compatibility or persistence
- Failure-mode examples that operators must understand
- Warnings for dangerous or irreversible operations
- Examples that explain process boundaries or responsibility boundaries

Suggested replacement style — use concise Japanese explanations. For command examples, keep only minimal representative examples. For JSON examples, keep only minimal representative examples. In the actual source document, remaining shell examples should be fenced as `bash`. In the actual source document, remaining JSON examples should be fenced as `json`. Avoid placing commands or JSON fragments on the same line as prose.

Suggested Japanese wording:
- 詳細な引数は各コマンドのヘルプまたは実行ガイドを参照する。
- ここでは設計上重要な実行順序と失敗ポイントのみを扱う。
- 完全な JSON フィールド一覧は実装上の DTO またはスキーマ定義を参照する。
- この例は操作手順の完全な一覧ではなく、責務境界を示すための代表例である。

Work document by document. If an example appears operationally important but too long for a design document, compress it and add a reference to the relevant execution guide or operations document.
