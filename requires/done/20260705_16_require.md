# Include duplicate live tool ownership in structured health results

## Priority

Medium

## Type

Implementation / Validation / Observability

## Background

Live `/v1/tools` responses are used to validate routing drift against `ToolRegistry`.

A tool should not be exposed by multiple MCP servers unless explicitly designed, because `ToolRegistry` assigns each tool to exactly one owner.

## Problem

Duplicate live tool ownership may be logged but not returned as structured startup validation output.

If duplicate ownership is not part of `HealthCheckResult`, callers and tests may miss this drift condition.

## Goal

Make duplicate live tool ownership visible, testable, and fatal in strict mode.

## Scope

Affected areas:

- `shared/route_resolver.py`
- `agent/repl_health.py`
- `HealthCheckResult`
- Routing drift validation tests

## Implementation Tasks

- Modify `build_discovery_map()` or related validation logic to return duplicate ownership information.
- Include duplicate ownership warnings in `HealthCheckResult`.
- In strict mode, raise `RuntimeError` for duplicate live tool ownership.
- Log tool name, all live server keys, and registry owner.
- Add tests covering duplicate ownership with reachable servers.

## Acceptance Criteria

- Duplicate live tool ownership appears in structured validation results.
- Strict mode fails startup on duplicate ownership.
- Non-strict mode logs warning and returns warning.
- Error output identifies tool name and involved servers.
- Registry owner is included when available.

## Validation Plan

Add tests for:

- No duplicate ownership.
- Same tool exposed by two servers.
- Duplicate tool where registry owner is one of the live servers.
- Duplicate tool where registry has no owner.
- Strict mode behavior.
- Non-strict mode behavior.

## Risks

Some existing servers may expose duplicated tools unintentionally. Such drift should be fixed rather than ignored.

## Out of Scope

- Supporting multi-owner routing.
- Changing ToolRegistry ownership model.
- Documentation-only updates.
