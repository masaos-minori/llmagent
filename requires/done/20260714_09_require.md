# [High] Normalize HttpTransport retry and watchdog restart semantics

## Labels
priority/high, area/transport, area/watchdog, area/health, documentation, tests

## Summary
Normalize documentation and tests for HTTP transport retry behavior and watchdog restart decisions.

## Background
The documentation contains contradictions:

- one section says HTTP timeout is non-retryable
- another classifies timeout as retryable
- one section says `restart_recommended=true` can trigger restart
- another implies degraded HTTP 503 cases only warn and never restart
- `operator_action_required=true` is described as if it only applies to HTTP 200 cases, but examples show it in degraded HTTP 503 responses

## Required Changes

### 1. HttpTransport retry behavior
Inspect implementation and document the actual behavior for:

- HTTP 429
- HTTP 502
- HTTP 503
- HTTP 504
- timeout
- connection refused
- malformed response

Clarify whether timeout is retryable.

If timeout is retryable:

- document retry count
- document delay behavior
- add tests

If timeout is not retryable:

- fix the error classification table
- add tests

If delay behavior is decreasing delay, document it as decreasing delay, not exponential backoff.

### 2. Watchdog restart decision model
Document the watchdog decision order as:

1. Server unreachable:
   - subprocess mode: restart candidate
   - persistent or external mode: warning only

2. Reachable and `operator_action_required=true`:
   - no restart
   - warning only

3. Reachable and `restart_recommended=true`:
   - subprocess mode: restart candidate
   - persistent or external mode: warning only

4. Reachable and degraded but `restart_recommended=false`:
   - no restart
   - warning only

5. Healthy:
   - no action

### 3. Health status edge cases
Explicitly document behavior for:

- HTTP 503 + `operator_action_required=true`
- HTTP 503 + `restart_recommended=true`
- HTTP 503 + both flags false
- unreachable subprocess server
- unreachable persistent server

### 4. Status display versus action
Clarify the difference between:

- `/mcp status` display fields
- lifecycle state
- watchdog restart action

## Acceptance Criteria
- Timeout retry behavior is documented consistently.
- Error classification table matches implementation.
- Watchdog restart decision rules are consistent across protocol, lifecycle, and monitoring docs.
- `operator_action_required=true` is documented for degraded responses too.
- HTTP 503 + `restart_recommended=true` behavior is unambiguous.
- Tests cover timeout retry behavior and watchdog decision branches.
