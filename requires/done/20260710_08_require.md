# [Medium] Document deployment responsibility split for workflow artifacts

## Background

Workflow deployment involves multiple phases: file copy, DB initialization, and service startup. These responsibilities should be clearly separated.

## Goal

Define and document responsibility boundaries for workflow deployment.

## Scope

- Deployment documentation
- Runbook
- Script comments
- Optional README updates

## Tasks

- [ ] Document that `deploy.sh` checks workflow definition existence.
- [ ] Document that `deploy.sh` validates workflow definition.
- [ ] Document that `deploy.sh` copies workflow definition.
- [ ] Document that `deploy.sh` prints workflow metadata and checksum.
- [ ] Document that `init_db.sh` creates `workflow.sqlite`.
- [ ] Document that `init_db.sh` applies workflow schema migrations.
- [ ] Document that `init_db.sh` verifies workflow tables.
- [ ] Document that `setup_services.sh` revalidates deployed workflow definition.
- [ ] Document that `setup_services.sh` checks workflow DB schema.
- [ ] Document that services start only after workflow checks pass.
- [ ] Add a deployment checklist.
- [ ] Add failure and remediation examples.

## Acceptance Criteria

- [ ] Documentation clearly states that workflow definition is a required deployment artifact.
- [ ] Each deployment phase has explicit workflow responsibilities.
- [ ] Operators can identify which script to run for each failure mode.
- [ ] No documentation suggests workflow can be disabled or bypassed.

## Notes

Keep the direction strict: mandatory workflow, no fallback mode.
