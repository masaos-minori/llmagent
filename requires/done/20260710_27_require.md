# [High] Align workflow definition stages, retry policy, and runtime execution behavior

## Background

`config/workflows/default.json` defines the stages:

- `plan`
- `execute`
- `verify`
- `retry`

But `WorkflowEngine.run()` executes only:

- `plan`
- `execute`
- approval gate
- `verify`

The `retry` stage is not executed as a real workflow stage. Retry behavior is implemented internally by `_run_execute_with_retry()`.

In addition, `WorkflowLoader` accepts both `fixed` and `exponential` backoff values, but the engine currently uses `policy.backoff_sec` directly and does not branch on `policy.backoff`.

There is also a stage description mismatch: `default.json` describes `plan` as preparing RAG context and memory injection, but the confirmed Orchestrator implementation performs memory injection and mode classification inside turn processing, not inside a real plan-stage callback.

## Goal

Make workflow definition JSON, loader validation, engine runtime, and documentation consistent.

## Scope

- `config/workflows/default.json`
- `agent/workflow/workflow_loader.py`
- `agent/workflow/workflow_engine.py`
- `agent/workflow/models.py`
- `agent/orchestrator.py`
- `05_99.md`
- workflow loader tests
- workflow engine tests

## Tasks

### Resolve `retry` stage mismatch

Choose one option:

#### Option A: Remove `retry` as a workflow stage

- [ ] Remove `retry` from `default.json`.
- [ ] Document retry as an internal execute-stage retry policy.
- [ ] Keep required stages as `plan`, `execute`, and `verify`.
- [ ] Update docs that currently describe `retry` as a workflow stage.
- [ ] Add tests proving workflow stages match engine runtime behavior.

#### Option B: Implement `retry` as a real workflow stage

- [ ] Define when `retry` stage should run.
- [ ] Add callback support for `retry`.
- [ ] Persist retry-stage attempts consistently.
- [ ] Update `default.json`.
- [ ] Add tests proving `retry` stage executes when expected.

### Resolve `exponential` backoff mismatch

Choose one option:

#### Option A: Implement exponential backoff

- [ ] In `_run_execute_with_retry()`, branch on `policy.backoff`.
- [ ] For `fixed`, use `policy.backoff_sec`.
- [ ] For `exponential`, use a documented deterministic formula.
- [ ] Add tests for `fixed` delay.
- [ ] Add tests for `exponential` delay.
- [ ] Document the formula.

#### Option B: Remove `exponential` as a supported value

- [ ] Remove `exponential` from `WorkflowLoader` supported values.
- [ ] Update documentation to list only `fixed`.
- [ ] Add a loader test that rejects `exponential`.
- [ ] Update example workflow files.

### Fix stage descriptions

- [ ] Update `default.json` stage descriptions to match real runtime behavior.
- [ ] Either move memory injection/mode classification into the plan stage or correct the plan-stage description.
- [ ] Ensure documentation explains what each stage actually does.

## Acceptance Criteria

- [ ] `default.json` stages match actual engine behavior.
- [ ] No documentation describes a workflow stage that never executes.
- [ ] Retry behavior is accurately documented and tested.
- [ ] Accepted backoff config values match runtime behavior.
- [ ] Stage descriptions match implementation responsibilities.

## Notes for AI Implementer

Prefer removing unused configuration surface unless a real runtime feature is required. Avoid accepting config values that silently behave differently from their names.
