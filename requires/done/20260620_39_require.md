# Title
Elevate RAG consistency checking from maintenance-only to normal operational diagnostics

## Summary
Make RAG consistency checks a normal operational diagnostic path rather than a rarely used maintenance-only tool.

## Background
Consistency checking already exists and can detect:
- chunk / FTS gaps
- orphaned vectors
- index mismatch conditions

However, this is still primarily framed as a maintenance-time utility.

## Problem
Index inconsistency can silently reduce retrieval quality if operators do not regularly run consistency checks or see their results.

## Required Changes
- Integrate consistency checking into normal operational workflows.
- Expose consistency summary signals in runtime diagnostics or regular maintenance output.
- Consider threshold-based warnings or alerts for FTS gaps and orphaned vectors.
- Preserve the existing read-only nature of the consistency check unless a separate issue adds repair behavior.

## Acceptance Criteria
- Consistency status is visible in normal operations.
- Operators can detect FTS/vector consistency issues early.
- Existing maintenance commands remain supported.

## Out of Scope
- Automatic repair of all detected consistency issues
- Replacing SQLite triggers

## AI Implementation Instruction
Implement this as an operational-diagnostics enhancement.

Strict rules:
- Do not silently auto-repair unless explicitly approved elsewhere.
- Keep the consistency check trustworthy and explainable.
- Surface actionable warnings without creating excessive noise.
- If threshold policy is unclear, mark it as: Needs confirmation

Deliverables:
1. Better visibility of consistency status
2. Operational diagnostics or alerts for mismatch conditions
3. Updated docs reflecting the normal-operations role of consistency checks
