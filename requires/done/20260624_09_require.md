# Add structured diagnostics for remote RAG HTTP mode

## Summary
Enhance HTTP-mode observability by recording structured diagnostics for remote calls, fallback paths, and empty-result handling.

## Background
The current design distinguishes remote empty results from in-process fallback, but its diagnostic model can be made more explicit for operators and debuggers.

## Objective
Expose enough structured metadata to distinguish normal empty results, remote failures, and successful remote retrieval.

## In Scope
- `scripts/rag/pipeline.py`
- `scripts/rag/pipeline_service.py`
- MCP or debug output surfaces if they expose pipeline diagnostics
- `03_rag_03_query_pipeline.md`

## Required Changes
- Add structured diagnostic fields such as:
  - `result_source`
  - `http_result_kind`
  - `remote_status_code`
  - `remote_latency_ms`
  - `fallback_reason`
- Ensure diagnostics are accessible via existing debug/reporting paths
- Update documentation with the final field definitions

## Acceptance Criteria
- Operators can distinguish:
  - successful remote non-empty result
  - successful remote empty result
  - HTTP failure with in-process fallback
- Diagnostic fields are documented and testable
- Debug output and logs remain internally consistent

## Out of Scope
- Redesigning remote retry policy
- Implementing distributed tracing

## Implementation Notes
Prefer explicit enum-like values over free-form strings where possible.
