# [Medium] Align MDQ public tool schema with runtime constraints

## Priority

Medium

## Summary

Improve `tools.py::TOOL_LIST.inputSchema` so the public MDQ tool schema reflects runtime models and configuration caps.

## Target files

- `scripts/mcp_servers/mdq/tools.py`
- `scripts/mcp_servers/mdq/models.py`
- `scripts/mcp_servers/mdq/service.py`
- `tests/test_mdq_tool_schema.py`
- `tests/test_tool_server_layer_consistency.py`

## Background

Cross-layer tests already verify that tool names, dispatch tables, registry entries, write flags, and serial flags are consistent.

However, `inputSchema` still lacks detailed constraints for many fields.

Examples:

- `search_docs.mode` is runtime-limited to `bm25`, but schema does not expose an enum.
- integer fields lack `minimum` and `maximum`.
- array fields often lack `items.type`.
- request overrides bounded by config caps are not reflected in schema.

## Reason for change

The LLM-facing schema should be as close as practical to runtime behavior.

If the published schema is looser than runtime validation and service caps, the LLM may generate requests that are syntactically accepted by the schema but rejected or silently capped at runtime.

This increases avoidable errors and makes tool behavior harder to understand.

## Intent

Reduce tool schema drift.

Make `/v1/tools` more precise while preserving existing tool names and dispatch behavior.

## Required changes

### search_docs schema

Add or update:

- `mode.enum = ["bm25"]`
- `limit.minimum`
- `limit.maximum`
- `max_results_limit.minimum`
- `max_results_limit.maximum`
- `max_total_result_chars.minimum`
- `max_total_result_chars.maximum`
- `tag_filter.items.type = string`

### get_chunk schema

Add or update:

- `max_chars_per_chunk.minimum`
- `max_chars_per_chunk.maximum`

### outline schema

Add or update:

- `max_outline_items.minimum`
- `max_outline_items.maximum`
- include `max_depth` if it is intended to be public

### index_paths and refresh_index schema

Add or update:

- `paths.items.type = string`
- optionally `paths.minItems = 1`

### grep_docs schema

Add or update:

- `paths.items.type = string`
- `max_grep_matches.minimum`
- `max_grep_matches.maximum`
- `max_chars_per_match.minimum`
- `max_chars_per_match.maximum`
- `context_before.minimum = 0`
- `context_after.minimum = 0`

## Acceptance criteria

- Public schema exposes important runtime constraints.
- Existing cross-layer tool-name consistency tests continue to pass.
- New schema drift tests cover bounds and enum values.
- No circular imports are introduced.
- `/v1/tools` remains stable except for more precise constraints.

## Test plan

Add tests for:

- `search_docs.mode.enum`
- integer minimums and maximums
- array item types
- config-capped maximums where applicable
- schema and runtime model consistency

## Notes for implementer

This issue is not about adding or removing tools.

It only improves the accuracy of public tool schemas.
