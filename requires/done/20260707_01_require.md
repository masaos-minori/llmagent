# Remove workflow_mode and make workflow execution always required

## Priority

High

## Type

Breaking Change / Workflow / Configuration

## Background

The agent currently supports `workflow_mode` with compatibility values such as `auto`, `required`, and `disabled`.

The target architecture no longer supports workflow mode switching. Workflow execution must always be required.

## Problem

Keeping `workflow_mode` allows compatibility behavior to remain in the system:

- `disabled` can bypass WorkflowEngine.
- `auto` can allow startup degradation.
- `required` becomes the only meaningful mode.

If only one valid mode remains, the configuration field itself is misleading.

## Goal

Remove `workflow_mode` entirely and make workflow execution an unconditional runtime invariant.

## Scope

Affected areas:

- `agent/config_dataclasses.py`
- `agent/config_builders.py`
- `agent/workflow_execution_policy.py`
- `agent/orchestrator.py`
- Config reload validation
- `/config`, `/context`, and workflow status display
- Tests referencing workflow modes

## Implementation Tasks

- Remove `AgentConfig.workflow_mode`.
- Remove `AgentConfig._validate_workflow_mode()`.
- Remove `WorkflowExecutionPolicy`.
- Remove `workflow_mode` from `Orchestrator.__init__()`.
- Reject config files that still contain `workflow_mode`.
- Remove mode display from status output.
- Remove tests that rely on `workflow_mode="auto"` or `workflow_mode="disabled"`.
- Update tests to assume workflow is always required.

## Acceptance Criteria

- `workflow_mode` is no longer part of `AgentConfig`.
- Any config containing `workflow_mode` fails config loading.
- There is no runtime branch based on workflow mode.
- Workflow execution is always required.
- Status output does not mention workflow mode.
- Tests pass with no workflow mode configuration.

## Validation Plan

Add or update tests for:

- Config containing `workflow_mode="auto"` fails.
- Config containing `workflow_mode="disabled"` fails.
- Config containing `workflow_mode="required"` fails if the field is removed entirely.
- Config without `workflow_mode` succeeds.
- Orchestrator construction does not accept a `workflow_mode` argument.
- Workflow status output has no mode field.

## Risks

This is a breaking change. Existing configs must remove `workflow_mode`.

## Out of Scope

- Supporting legacy workflow modes.
- Adding a compatibility shim.
- Migration or backfill handling.
- Documentation-only updates.
