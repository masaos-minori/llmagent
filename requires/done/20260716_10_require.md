# [High] Define schema change, rebuild, and migration policy for DB-affecting work

## Priority

High

## Type

Documentation / Migration Safety / Operational Readiness / Scope Control

## Target files

- `create_schema.py`
- `db/schema_sql.py`
- `mcp_servers/mdq/db_schema.py`
- Documentation related to DB initialization and upgrades
- Issue templates or developer checklist if present

## Consolidated from

- Add DB rebuild or migration guidance to schema-changing tasks
- Keep RAG/session vector schema separate from MDQ hybrid cleanup

## Background

`create_schema.py` states that the current schema is created from DDL only and that the DB should be recreated when schema changes are required. 【8-76187d】

`schema_sql.py` defines schema builders for RAG, session, workflow, and eventbus databases. It also includes workflow-specific migrations through `apply_workflow_migrations()`, while the provided file does not show a general incremental migration mechanism for every DB. 【7-409e83】

`schema_sql.py` also defines vector tables for RAG/session, including `chunks_vec` and `memories_vec`. These are part of the generic schema and should not be confused with the unfinished MDQ hybrid search path. 【7-409e83】

## Problem

Schema-changing tasks can be completed at the DDL level without specifying how existing databases should be handled. This creates ambiguity for:

- local developer upgrades,
- production upgrades,
- test database rebuilds,
- data retention expectations,
- compatibility with existing installations.

The risk is higher because workflow has an explicit migration mechanism, while other schemas appear to rely on recreation or separate handling.

## Reason for change

Any issue that changes schema must state whether existing DBs are migrated, rebuilt, or unsupported. Otherwise, developers may assume that updating DDL is enough even when existing databases retain the old structure.

This is especially important for MDQ cleanup work that may remove or alter summary-cache or embedding-related schema paths.

## Design intent

Create a clear operational rule:

- Workflow DB changes should use or extend the existing migration mechanism.
- Non-workflow DB changes must explicitly state whether DB recreation is required.
- MDQ-specific schema changes must identify the MDQ schema source and upgrade path.
- Vector schema changes must identify whether they affect RAG, session, or MDQ.

## Implementation instructions

- Document which DBs currently support incremental migrations.
- Document which DBs are DDL/recreation based.
- Add a required "Existing DB handling" section to schema-changing issue templates or developer checklist.
- For workflow DB changes, require updates to migration definitions when appropriate.
- For RAG/session/eventbus/MDQ schema changes, require an explicit rebuild or migration plan.
- Document that RAG/session vector tables must not be removed as part of MDQ hybrid search cleanup.
- Confirm whether MDQ has a separate schema module and how it should be upgraded.

## Required checklist for future schema issues

Each schema-changing issue must answer:

- Which DB is affected?
- Which schema source file is affected?
- Is this a new install-only DDL change?
- Is an existing DB migration required?
- If no migration is provided, is DB recreation required?
- Is data loss possible?
- Are tests updated to reflect the schema behavior?
- Does the change affect RAG, session, workflow, eventbus, or MDQ?

## Acceptance criteria

- Documentation clearly distinguishes DDL creation from incremental migration.
- Workflow migration behavior is documented separately from recreation-only schemas.
- Future schema-changing issues cannot omit existing DB handling.
- MDQ schema changes require explicit upgrade/rebuild guidance.
- RAG/session vector schema is explicitly out of scope for MDQ hybrid cleanup unless separately stated.

## Non-goals

- Do not implement a new migration framework in this issue.
- Do not remove vector tables in this issue.
- Do not change DB schema unless required for documenting the policy.
