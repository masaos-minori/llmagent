# Add guard checks against compatibility-only stubs and shims

## Priority

Medium

## Summary

Add guard checks to prevent reintroducing compatibility-only stubs and shims.

## Background

The following compatibility-only mechanisms are being removed:

- `agent/config.py` re-export stub
- `McpServerConfig._cast_enums()` compatibility shim
- `mcp/github/service.py` re-export stub
- `mcp/github/__init__.py` service symbol re-export

A guard check should prevent similar patterns from being reintroduced accidentally.

## Goal

Detect compatibility-only stubs and shims during validation.

## Target files

- Test files
- Lint scripts
- Documentation validation scripts
- CI configuration if applicable

## Scope

### In scope

- Add grep-based or test-based checks.
- Detect compatibility-only keywords.
- Add an allowlist only if strictly necessary.

### Out of scope

- Building a complex static analyzer.
- Blocking legitimate historical documentation unless intentionally configured.

## Implementation tasks

- Add a check for these terms:

  - `backward-compatible`
  - `re-export stub`
  - `compatibility shim`
  - `existing imports continue to work`
  - `_cast_enums`

- Ensure the removed compatibility mechanisms are not allowlisted.
- If unavoidable documentation mentions these terms as historical notes, isolate them behind an explicit allowlist.

## Acceptance criteria

- CI or local validation can detect reintroduced compatibility stubs.
- The removed targets do not appear in grep results.
- Any remaining occurrence is intentional and documented.

## Validation

- Run:
```bash
grep -R "backward-compatible\|re-export stub\|compatibility shim\|existing imports continue to work\|_cast_enums" -n scripts tests docs
```

## Undecided items / confirmation required

- Confirm whether historical docs are allowed to mention removed compatibility stubs.
- Confirm where this guard should live:
  - unit test
  - lint script
  - CI-only check
