# Title
Unify MCP tool routing authority around ToolRegistry and tool_constants

## Priority
High

## Labels
- bug
- mcp
- routing
- tool-registry
- documentation
- architecture

## Dependencies
- None

## Summary
Clarify and enforce a single source of truth for MCP tool routing. Runtime routing should not be described inconsistently as discovery-first, config-driven, registry-driven, and static fallback at the same time.

## Background
The routing documentation currently contains conflicting descriptions. One section describes a priority order using discovery map, registry, config-driven routing, and static fallback. Another section says the ToolRegistry is the routing authority, `tool_constants.py` is the static input, and config/live discovery are validation-only.

## Problem
AI implementers may incorrectly treat live `/v1/tools` or config `tool_names` as routing authorities. This can cause routing drift and broken new-tool registration.

## Required Changes
- Inspect `shared/route_resolver.py`, `shared/tool_registry.py`, and `shared/tool_constants.py`.
- Confirm the actual runtime routing source.
- Make documentation state one canonical model:
  - `shared/tool_constants.py` frozensets are the static input.
  - `ToolRegistry` is the runtime routing authority.
  - config `tool_names` is validation-only.
  - live `/v1/tools` is validation-only unless implementation proves otherwise.
- Remove contradictory wording such as:
  - "Discovery map highest priority"
  - "Config-driven routing"
  - "Static fallback" if no runtime fallback exists.
- Keep future live-drift validation clearly separate from routing.
- Update routing tests to reflect the canonical behavior.

## Acceptance Criteria
- There is exactly one documented runtime routing authority.
- New tool registration instructions point to the correct required files.
- Config and live discovery are documented as validation-only if they are not used for routing.
- Tests prove routing works without config `tool_names` when the registry contains the tool.

## Out of Scope
- Adding dynamic runtime routing from live discovery unless explicitly required.
- Changing the tool naming convention.

## Components
- `shared/route_resolver.py`

## AI Implementation Instruction
Implement this as a routing source-of-truth cleanup.

Strict rules:
- Do not keep contradictory routing descriptions.
- Do not make config `tool_names` required for routing if the implementation routes through the registry.
- Do not treat `/v1/tools` discovery as authoritative unless runtime code actually does so.
- If implementation and docs disagree, trust implementation first, then update docs.
