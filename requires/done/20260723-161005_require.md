## Background

Some design and operations documents contain complete CLI argument lists, configuration key lists, default values, and type descriptions. These details are often implementation-derived and can usually be confirmed from source code, configuration dataclasses, command help output, actual TOML configuration files, or deployment scripts. Keeping complete copies of these details in design documents creates several maintenance problems: they become stale when CLI options, configuration keys, default values, or implementation types change; they make it harder to distinguish architectural intent from implementation reference material; AI agents may treat copied configuration tables as the canonical source even when the implementation has already changed; and long reference tables increase the amount of context required to understand a document.

## Problem

Exhaustive CLI argument tables and configuration key tables that can be mechanically confirmed from source code, command help, configuration dataclasses, or actual configuration files bloat design documents and reduce their value as architectural references.

## Reason for change

Separate two types of information: (1) information that belongs in implementation, command help, or configuration files, and (2) information that belongs in design documentation. The resulting documents should explain which process owns which configuration file, which configuration values affect runtime behavior, which configuration changes require restart, which settings are hot-reloadable, which settings can cause startup failure, which settings affect security posture, which settings affect fail-fast or fail-open behavior, which settings are deprecated, which settings affect data compatibility or persistence, and which settings remain unresolved or require confirmation. The documentation should no longer attempt to duplicate full CLI help output, full configuration files, or full configuration schemas.

## Implementation intent

Make configuration documentation useful for architectural understanding and operational decision-making, while leaving mechanically discoverable details to the implementation and actual configuration files. Compress exhaustive CLI argument tables and configuration key tables into concise Japanese summaries and source references.

## Implementation instructions

For each target document:

1. Identify exhaustive CLI argument lists and configuration key tables
2. Classify each table or list before editing using the following categories:
   - implementation-derived detail
   - design-relevant configuration behavior
   - operationally important setting
   - security-sensitive setting
   - persistence-sensitive setting
   - deprecated item
   - unresolved or Needs Confirmation item
3. Remove or compress only implementation-derived details
4. Preserve design-relevant configuration behavior
5. Preserve process ownership and configuration ownership
6. Preserve restart and hot-reload boundaries
7. Preserve fail-fast and fail-open behavior
8. Preserve security-sensitive settings and operational warnings
9. Preserve deprecated settings and references to deprecated-item governance
10. Preserve unresolved configuration questions and link them to Needs Confirmation where applicable
11. Replace exhaustive tables with concise Japanese summaries and source references

Content to remove or compress:
- Complete CLI argument tables
- Complete configuration key tables
- Default-value-only tables
- Type-only tables
- Tables that directly duplicate configuration dataclasses
- Tables that directly duplicate TOML files
- Long CLI help output copied into design documents
- Repetitive descriptions that do not explain design or operational impact
- Configuration tables that only restate names, types, and defaults

Content to keep:
- Process-local configuration ownership
- The rule that each process reads only its own configuration file
- The rule that shared common configuration files should not be introduced
- Hot-reloadable versus restart-required boundaries
- Startup-failure conditions
- Workflow definition requirements
- Absence of workflow optional mode where currently documented
- `startup_mode` operational impact
- `security_profile` fail-fast and fail-open behavior
- Security-sensitive configuration behavior
- Data compatibility or persistence-sensitive configuration behavior
- Deprecated configuration items
- Known Issues
- Needs Confirmation items
- Behavior verified by tests or observed operationally

Suggested replacement structure — use Japanese section headings such as:
- `## 設定の責務境界`
- `## 変更時の運用影響`
- `## 再起動が必要な設定`
- `## ホットリロード可能な範囲`
- `## 起動失敗につながる設定`
- `## セキュリティに関わる設定`
- `## 実装詳細の参照先`

Suggested Japanese wording:
- 完全な設定キー、型、デフォルト値は、実装上の設定定義および実際の設定ファイルを参照する。
- この設計書では、設定の所有者、変更時の影響、再起動要否、失敗時の挙動、運用上の注意のみを扱う。
- 設定値そのものではなく、設定変更がどの責務境界や運用挙動に影響するかを記載する。

Work document by document. If unsure whether a configuration entry is design-relevant, keep it and add a short note for later review instead of deleting it.
