# Reject disabled tools before dispatch in /v1/call_tool and clarify validation policy

## Priority

High

## Target files

- mcp_servers/file/read_server.py
- mcp_servers/file/write_server.py
- mcp_servers/file/delete_server.py
- mcp_servers/git/server.py
- mcp_servers/dispatch.py
- mcp_servers/models.py
- mcp_servers/tool_validators.py
- tests/mcp_servers/test_call_tool_contract.py
- tests/mcp_servers/test_call_tool_validation.py

## Reason

RuntimeToolRegistry will prevent disabled tools from being exposed to the LLM, but `/v1/call_tool` may still be called directly over HTTP.

MCP servers must enforce the same runtime availability state before dispatch. Otherwise, direct callers may bypass `enabled=false` metadata.

Also, `CallToolRequest` exposes `validate_args()`, but current server implementations do not clearly call it before dispatch. Since `/v1/call_tool` is the standard execution endpoint, argument validation behavior must be explicit and consistent.

## Intent

Make `/v1/tools` availability metadata and `/v1/call_tool` execution behavior consistent.

Disabled tools must be rejected before reaching service dispatch handlers.

Argument validation policy must be deterministic across MCP servers.

## Scope

Before calling `dispatch_tool()`, check whether the requested tool is enabled under current server configuration.

If the tool is disabled, return a normal `CallToolResponse` with:

- `result: "Tool disabled: <disabled_reason>"`
- `is_error: true`

Do not call the service dispatch table for disabled tools.

Clarify argument validation behavior by choosing and implementing one policy.

Recommended policy:

- Every `/v1/call_tool` endpoint calls `req.validate_args()` before checking or dispatching the tool.

Alternative policy:

- Remove or deprecate `validate_args()` if validation is fully handled by request model construction inside handlers.

## Expected disabled tool cases

Add tests for:

- file tools when `allowed_dirs` is empty
- git tools when `allowed_repo_paths` is empty
- git write tools when `read_only` is true

## Acceptance criteria

- Disabled tools are rejected before dispatch.
- Disabled tool response uses `is_error: true`.
- Disabled tool response includes the corresponding `disabled_reason`.
- Disabled tools do not reach service dispatch handlers.
- Direct HTTP callers cannot bypass disabled state.
- Argument validation policy is documented.
- All reviewed MCP servers follow the same validation policy.
- Invalid arguments produce a consistent error response.
