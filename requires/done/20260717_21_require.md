# [High] Validate WebSearchConfig and align search_web input schema with runtime constraints

## Priority

High

## Summary

Validate `WebSearchConfig` values at startup and align the public `search_web` tool schema with the runtime `SearchRequest` model.

This issue combines the following work:

- Fail-fast validation for `WebSearchConfig`
- Synchronization between `tools.py::TOOL_LIST.inputSchema` and `SearchRequest`
- Query whitespace and control-character handling
- Schema drift tests

## Target files

- `scripts/mcp_servers/web_search/models.py`
- `scripts/mcp_servers/web_search/tools.py`
- `config/web_search_mcp_server.toml`
- `tests/test_web_search_models.py`
- `tests/test_web_search_tool_schema.py`

## Reason for change

`SearchRequest` enforces runtime validation for `query` and `max_results`, but `tools.py::TOOL_LIST.inputSchema` currently exposes a looser schema.

The runtime model validates constraints such as:

- query must not be empty
- query must not exceed 500 characters
- max_results must be at least 1
- max_results must not exceed the configured limit

However, the LLM-facing schema does not expose equivalent constraints.

This creates schema drift. The LLM may generate tool calls that appear valid according to `/v1/tools`, but fail during runtime validation.

`WebSearchConfig.from_dict()` also currently converts values to integers but does not fully validate relationships such as `default_max_results <= max_results_limit`.

## Intent

Create one consistent contract across:

- configuration
- runtime request validation
- MCP tool schema exposed to the LLM
- tests

Invalid configuration should fail fast during server startup. Invalid request shapes should be visible in the tool schema before execution.

## Required changes

### 1. Validate WebSearchConfig

Add explicit validation for:

- `default_max_results >= 1`
- `max_results_limit >= 1`
- `default_max_results <= max_results_limit`
- `max_results_limit <= HARD_MAX_RESULTS_LIMIT`

Add a hard safety cap such as `HARD_MAX_RESULTS_LIMIT = 100`.

Do not silently clamp invalid values. Raise `ValueError` with an actionable message.

### 2. Align tools.py inputSchema with SearchRequest

Update the `search_web` schema in `tools.py` so it includes:

- `query.minLength = 1`
- `query.maxLength = 500`
- `max_results.minimum = 1`
- `max_results.maximum = configured max_results_limit`

Avoid unchecked duplication of magic numbers. Prefer shared constants or generated schema.

### 3. Define query normalization policy

Define and test behavior for:

- leading whitespace
- trailing whitespace
- whitespace-only query
- NUL character
- newline-only query
- other control characters

Recommended behavior:

- trim leading and trailing whitespace
- reject empty-after-trim query
- reject NUL characters
- reject or normalize other control characters consistently

Ensure the normalized query is used consistently for:

- provider call
- response
- audit
- metrics

### 4. Add schema drift tests

Add tests that compare the runtime `SearchRequest` constraints with the public `TOOL_LIST.inputSchema`.

## Acceptance criteria

- Invalid config fails during `WebSearchConfig.load()`.
- `default_max_results = 0` is rejected.
- `max_results_limit = 0` is rejected.
- `default_max_results > max_results_limit` is rejected.
- Excessive `max_results_limit` is rejected.
- Existing valid config remains valid.
- `/v1/tools` exposes `search_web` with matching schema constraints.
- Tool schema and runtime model constraints are covered by tests.
- Whitespace-only queries are not passed to the provider.
- NUL/control-character behavior is documented and tested.

## Test plan

Add or update tests for:

- default valid config
- custom valid config
- invalid default below minimum
- invalid limit below minimum
- invalid default greater than limit
- invalid excessive limit
- `TOOL_LIST.inputSchema` matching `SearchRequest`
- whitespace-only query
- query with leading/trailing whitespace
- query containing NUL
- query containing newline or control characters

## Notes for implementer

Prefer a single source of truth.

Possible approaches:

- define constants in `models.py` and import them from `tools.py`
- generate `inputSchema` from `SearchRequest.model_json_schema()`
- keep manual schema but add strict drift tests

Avoid circular imports between `models.py` and `tools.py`.
