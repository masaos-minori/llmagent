# [High] Implement RAG corpus-change cache freshness contract for MCP deletion and CLI ingestion

## Background

`SemanticCache` is an in-memory cache owned by a `RagPipeline` instance.

MCP `rag_delete_document` runs in the same service process as the pipeline, so it can directly invalidate the cache after successful deletion.

CLI ingestion is different. `RagIngester.ingest_all()` accepts an `on_ingest_complete` callback and forwards it to post-ingest consistency checking, but the CLI entry point currently calls `ingester.ingest_all(args.force)` without passing a callback. Also, a CLI process cannot directly access the MCP service's in-memory `RagPipeline` instance.

Therefore, MCP deletion and CLI ingestion require different cache freshness strategies.

## Goal

Ensure RAG corpus changes do not leave stale semantic cache entries, while documenting the cross-process limitation of CLI ingestion.

## Acceptance Criteria

- [ ] Successful MCP `rag_delete_document` clears the semantic cache.
- [ ] Not-found MCP deletion does not clear the semantic cache.
- [ ] MCP service does not directly access `pipeline.semantic_cache`.
- [ ] Cache invalidation occurs through a public pipeline method.
- [ ] CLI ingestion cache behavior is explicitly documented.
- [ ] Cross-process cache invalidation is not treated as if it were a local callback problem.
- [ ] Tests prove stale cached context is not returned after MCP deletion.
- [ ] Operators know what to do after CLI ingestion if immediate freshness is required.

## Notes for AI Implementer

Do not assume the CLI ingestion process can access the MCP service's in-memory cache.

Keep MCP delete invalidation and CLI ingestion freshness related but clearly separated in implementation and documentation.
