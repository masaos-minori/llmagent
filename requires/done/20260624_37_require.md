# Cross-link `/reload` output categories with config ownership and hot-reload rules

## Summary
Make the `/reload` UX easier to interpret by directly connecting CLI output categories to the underlying config ownership and reload rules.

## Background
The CLI chapter documents the visible reload outcome tags, while the configuration chapter documents ownership and field-level reload classification. The relationship is correct but not easy to follow during operations.

## Objective
Allow an operator to understand *why* a change is `[OK]`, `[DEFER]`, `[RESTART]`, `[STARTUP-ONLY]`, or `[SKIP]` without reading multiple chapters deeply.

## In Scope
- `05_agent_07_cli-and-commands.md`
- `05_agent_08_configuration.md`

## Required Changes
- Add direct cross-references from `/reload` output categories to the config ownership section
- Add a concise explanation of why a field lands in each category
- Ensure the file-ownership table and the CLI outcome tags use the same terminology
- Add one worked example of a mixed reload outcome

## Acceptance Criteria
- The `/reload` category meanings are easy to trace from the CLI chapter
- Operators can understand deferred vs restart-required vs startup-only from the docs alone
- The terminology is identical across both chapters

## Out of Scope
- Changing reload behavior
- Redesigning config files

## Notes
Keep the CLI chapter concise, but do not force operators to infer category meaning from implementation details.
