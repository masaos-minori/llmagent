## [Medium][eventbus][health] Extend /health to report stall-related degradation, not just binary availability

### Summary
The current `/health` endpoint checks DB reachability and DLQ task liveness only.

### Problem
It cannot express “the service is alive but badly stalled,” which is exactly the failure mode caused by event-loop blocking.

### Scope
- /health endpoint
- event-loop stall metrics
- DB/query latency visibility
- operational health model

### Facts
- `/health` currently returns only `status`, `db`, and `dlq_task`, and always uses HTTP 200. 【1-2f69b2】【1-6bdda3】
- Known issues explicitly mention uncertainty about degraded-state behavior and FastAPI thread-pool usage. 【5-bd74c0】
- Current health logic runs a synchronous `SELECT 1` directly. 【1-6bdda3】

### Task
- add degraded indicators for DB query latency
- add degraded indicators for SSE lag or backlog
- add degraded indicators for DLQ loop delay
- optionally review degraded HTTP status behavior
- document health semantics for availability vs performance degradation

### Expected Result
Operators can distinguish total failure from severe runtime degradation.

### Acceptance Criteria
- health output includes latency/stall-related degradation indicators
- health semantics are documented clearly
- health checks no longer rely only on binary DB availability
- degraded runtime states are observable
