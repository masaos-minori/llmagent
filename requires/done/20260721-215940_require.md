# Require: Increase health check request timeout for slow-starting MCP servers

## Priority

Medium

## Target files

- `scripts/agent/http_lifecycle.py`

## Background

HTTP MCP server health check polling uses an `httpx.AsyncClient` with `timeout=2.0` per request. The overall startup timeout is controlled by `cfg.startup_timeout_sec`. Each health check attempt has only 2 seconds before timing out.

## Problem

When an MCP server takes longer than 2 seconds to start, the first few health checks will always time out with `httpx.HTTPError`, even though the server is actually starting. With a polling interval of 0.5 seconds, a server that starts in 2.5 seconds would have its first 5 health checks all fail due to timeout — creating a false negative where the server is healthy but the health check cannot reach it fast enough.

## Reason for change

The 2-second per-request timeout is unreasonably short for HTTP health checks on servers that may take several seconds to initialize. This causes unnecessary failures during startup and misleading error messages.

## Implementation intent

Increase the per-request timeout to a reasonable value (5–10 seconds) so that individual health check requests do not timeout before the server has had time to respond. Also upgrade the timeout exception log level from DEBUG to INFO for better observability.

## Implementation instructions

In `scripts/agent/http_lifecycle.py`, around line 249, replace:

```python
async with httpx.AsyncClient(timeout=2.0) as client:
```

with:

```python
async with httpx.AsyncClient(timeout=httpx.Timeout(connect=5.0, read=5.0)) as client:
```

Also update the timeout exception logging at approximately the same location — change `logger.debug(...)` for timeout exceptions to `logger.info(...)` so operators can see them without enabling debug logging.

## Acceptance criteria

- Health check requests use a 5-second connect/read timeout instead of 2 seconds
- Timeout exceptions are logged at INFO level instead of DEBUG
- Servers that start within 5 seconds pass their first health check without spurious timeouts
- Overall startup timeout (`cfg.startup_timeout_sec`) continues to work correctly
- No regression in fast-starting servers (they should still be detected quickly)

## Tests

- Unit test: Mock server that responds after 3 seconds, verify health check succeeds with new timeout
- Unit test: Mock server that never responds, verify overall startup timeout still triggers
- Integration test: Start a real MCP server with a simulated delay, verify health check passes
