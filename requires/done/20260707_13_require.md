# Delete WorkflowExecutionPolicy

## Priority

Low

## Type

Cleanup / Refactoring / Workflow

## Background

`WorkflowExecutionPolicy` exists to centralize workflow mode decisions.

After removing `workflow_mode`, there are no workflow modes left to decide.

## Problem

Keeping `WorkflowExecutionPolicy` after workflow becomes always-required creates unnecessary abstraction and makes future reintroduction of mode branching easier.

## Goal

Remove `WorkflowExecutionPolicy` entirely.

## Scope

Affected areas:

- `agent/workflow_execution_policy.py`
- Orchestrator imports
- Startup checks
- Tests referencing the policy

## Implementation Tasks

- Delete `WorkflowExecutionPolicy`.
- Remove imports.
- Remove policy unit tests.
- Replace any remaining policy checks with workflow-required invariants.
- If a helper is needed, create a narrow `assert_workflow_ready()` helper instead.
- Ensure no code references mode-based policy methods.

## Acceptance Criteria

- `agent/workflow_execution_policy.py` is removed or empty.
- No production code imports `WorkflowExecutionPolicy`.
- No tests depend on workflow modes.
- No mode decision methods remain.
- Workflow readiness is enforced through startup validation.

## Validation Plan

Add or update tests for:

- Import scan has no `WorkflowExecutionPolicy`.
- Static scan has no `allow_startup_fallback`.
- Static scan has no `is_workflow_enabled`.
- Startup validation still enforces workflow readiness.

## Risks

Some tests may need simple rewrites to remove policy construction.

## Out of Scope

- Mode compatibility.
- Replacing policy with another mode abstraction.
- Migration support.
