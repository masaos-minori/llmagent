## [High][workflow][safety] Remove or harden unsafe direct-execution fallback from workflow mode

### Summary
The current design allows fallback from workflow execution to direct execution when workflow execution is unavailable or fails.

### Problem
Direct fallback can bypass workflow-level approval, artifact tracking, retry tracking, and idempotency guarantees.

### Scope
- agent/config.py
- agent/orchestrator.py
- agent/workflow/*
- config/common.toml
- workflow-mode documentation

### Task
- audit all direct fallback paths from workflow execution
- add explicit audit events for workflow fallback
- surface fallback state in CLI and logs
- add a strict policy to deny fallback in non-local or protected environments
- document exactly when fallback is allowed and when it is denied

### Expected Result
Workflow safety features cannot be bypassed silently by implicit fallback.

### Acceptance Criteria
- fallback events are logged and visible
- protected environments can run fail-closed without direct fallback
- CLI surfaces indicate when execution is not using workflow mode
- workflow fallback policy is documented clearly
