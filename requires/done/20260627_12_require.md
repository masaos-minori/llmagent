Unify MCP HTTP startup mode terminology and configuration values

High

## Summary
Unify MCP HTTP startup mode terminology across implementation and documentation. The docs currently use both `persistent` and `external`, but `McpServerConfig.startup_mode` only documents `persistent`, `ondemand`, and `subprocess`.

## Background
Some documents recommend `startup_mode = "external"` for production HTTP servers, while the configuration reference only lists `"persistent"`, `"ondemand"`, and `"subprocess"` as valid startup modes.

## Problem
Operators may copy a non-supported startup mode into configuration, causing startup failure or undefined behavior. It is also unclear whether `persistent` means externally managed HTTP servers or agent-managed persistent servers.

## Required Changes
- Inspect `shared/mcp_config.py` and startup validation logic.
- Decide the canonical model:
  - Option A: use `persistent` for externally managed HTTP servers and remove `external` from docs.
  - Option B: implement `external` as a real startup mode and document precedence/behavior.
- Update all MCP docs to use only supported startup mode names.
- Define behavior for:
  - HTTP `persistent`
  - HTTP `subprocess`
  - stdio `persistent`
  - stdio `ondemand`
- Update examples in configuration and operations docs.
- Add validation tests for valid and invalid startup mode combinations.

## Acceptance Criteria
- Documentation and `McpServerConfig` use the same startup mode values.
- Production recommended configuration can be copied directly and starts successfully.
- `external` and `persistent` are not used ambiguously.
- Invalid startup mode + transport combinations are tested.

## Out of Scope
- Redesigning the lifecycle router.
- Changing transport behavior beyond terminology/config validation alignment.
