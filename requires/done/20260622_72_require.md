# Title
Fix the `use_memory_layer` default inconsistency and expose memory state clearly

## Summary
Determine the real implementation default for `use_memory_layer`, align all active documentation to that value, and make memory enabled/disabled state visible in normal runtime views.

## Background
The memory-layer docs do not agree on the default state of `use_memory_layer`.
One document implies memory is enabled by default, while the latest API reference says memory is disabled by default and `ctx.services.memory` is `None` unless enabled.

## Problem
This inconsistency makes configuration reasoning unreliable for operators and AI implementation agents.

## Required Changes
- Confirm the real implementation default for `use_memory_layer`.
- Update all memory/config/API docs to that confirmed value.
- Expose memory enabled/disabled state in normal runtime surfaces such as startup banner, `/config`, or `/stats`.
- Keep the existing graceful-degradation model explicit.

## Acceptance Criteria
- All active docs agree on the default value.
- Operators can see whether memory is currently enabled or disabled without reading source code.
- The nullability of `ctx.services.memory` is documented clearly and consistently.

## Out of Scope
- Redesigning memory retrieval logic
- Changing embedding-client behavior

## AI Implementation Instruction
Implement this as a correctness-and-visibility task.

Strict rules:
- Do not guess the default. Verify it from implementation.
- Do not silently change runtime behavior unless the issue explicitly requires it.
- Update documentation only after confirming actual behavior.
- Expose state in a normal operator-facing place, not only in deep debug output.
- If startup/runtime behavior still remains ambiguous after inspection, mark it as: Needs confirmation

Deliverables:
1. Corrected documentation
2. Runtime visibility for memory state
3. A short note naming the actual source of truth for the default
