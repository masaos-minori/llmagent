# [Agent][Workflow] Make workflow engine mode explicit instead of implicit fallback

## Summary
Workflow execution currently depends on whether the workflow JSON exists and whether the workflow DB is available.
If not, the agent falls back to direct execution.

## Problem
This fallback is documented, but the exact runtime mode is too implicit and is also listed as an open question in the documentation guide.

## Scope
- workflow engine activation logic
- startup mode reporting
- workflow-related runtime docs

## Task
- define explicit workflow modes, such as:
  - `auto`
  - `required`
  - `disabled`
- show the active workflow mode at startup
- make fallback behavior explicit in logs and docs

## Expected Behavior
- operators know whether workflow execution is active
- fallback to direct execution is intentional and visible
- workflow-related behavior is easier to reason about

## Acceptance Criteria
- [ ] workflow mode is explicitly configurable
- [ ] startup reports the active workflow mode
- [ ] fallback conditions are clearly logged
- [ ] docs stop treating workflow fallback as an open question
