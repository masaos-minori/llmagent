## [startup][reliability] _recover_pending_approvals の StateStore を try/finally で保護する

### Summary
`startup.py:_recover_pending_approvals()` で `StateStore` インスタンスを作成後、`find_latest_pending_approval()` が例外を投げた場合に `store.close()` が呼ばれない。`orchestrator.py` の他の StateStore 使用箇所では `try/finally` で保護されているが、この箇所だけ保護が欠けている。

### Background
`StateStore` は `workflow.sqlite` への接続を保持する。接続リークを避けるため、全ての `StateStore` インスタンスは `close()` が確実に呼ばれる必要がある。`orchestrator.py` では `_handle_workflow_engine()` (行 210, 263) と `_init_workflow_task()` (行 275–284, 292–299) の両方で `try/finally: store.close()` パターンが採用されている。しかし `startup.py:_recover_pending_approvals()` には同様の保護がない。

### Problem
`startup.py:178–179`:
```python
store = StateStore()
result = store.find_latest_pending_approval()
store.close()
```
`find_latest_pending_approval()` が SQLite エラー等で例外を投げた場合、`store.close()` が実行されず DB 接続がリークする。スタートアップ時の 1 回のみの呼び出しであり実害は限定的だが、他の箇所との一貫性の観点から修正が必要。

### Scope
- `startup.py:_recover_pending_approvals()` のみ
- StateStore の API 変更は範囲外

### Task
1. `startup.py:_recover_pending_approvals()` の `store = StateStore()` 以降のコードを `try/finally: store.close()` で囲む
2. 修正パターン:
   ```python
   store = StateStore()
   try:
       result = store.find_latest_pending_approval()
   finally:
       store.close()
   ```
3. `result` の後続処理 (`if result is None: return` および task_id/approval の取り出し) は `try` ブロック内、または `store.close()` 後のコードに配置する (後者が読みやすい)

### Acceptance Criteria
- `find_latest_pending_approval()` が例外を投げた場合も `store.close()` が呼ばれる
- スタートアップフローの正常動作が変わらない
- 既存テストが通過する

### AI Implementation Instruction
**What to change:**
- File: `scripts/agent/startup.py`
- Method: `_recover_pending_approvals()` (approximately lines 173–196)
- The current code pattern:
  ```python
  store = StateStore()
  result = store.find_latest_pending_approval()
  store.close()
  if result is None:
      return
  task_id, approval = result
  ...
  ```
  Should be refactored to:
  ```python
  store = StateStore()
  try:
      result = store.find_latest_pending_approval()
  finally:
      store.close()
  if result is None:
      return
  task_id, approval = result
  ...
  ```
  The `result` handling after `store.close()` is fine since `result` is a plain Python object that does not require the connection to remain open.

**What NOT to change:**
- Do not change the logic after `store.close()` (the `if result is None: return` branch and the subsequent warning/display code).
- Do not change `StateStore` class itself.
- Do not add exception handling (catching) — the existing `try/finally` pattern is sufficient; exceptions should still propagate to the caller.

**Boundary awareness:**
- `_recover_pending_approvals()` is called from `StartupOrchestrator.run()` at line ~54. If this method raises, startup will abort — this is the correct behavior (same as other startup errors).
- `result` from `find_latest_pending_approval()` is a tuple `(task_id, ApprovalRecord)` or `None`. It is a data object with no DB dependency, so using it after `store.close()` is safe.
