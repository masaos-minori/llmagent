# [High] Remove stale `rag.llm` and module-level config cache comments

## Priority
High

## Difficulty
Low

## Background
`service.py` still mentions overriding module-level `_cfg` caches in `rag.pipeline`, `rag.llm`, and `db.helper`, while the same file also states that `rag.llm` no longer has a module-level `_cfg` cache. This stale wording can mislead future implementation and reintroduce removed compatibility behavior.

Source evidence: `service.py` 【1-dea44a】

## Goal
Update RAG MCP service documentation and comments to describe the current config injection model only.

## Scope
Affected files:
- `scripts/mcp/rag_pipeline/service.py`
- `scripts/rag/pipeline.py`
- `scripts/rag/llm_client.py`
- RAG documentation if it repeats the stale wording

## Implementation Tasks
- Remove references to `rag.llm` module-level config caches.
- Remove comments suggesting that RAG submodules should read from process-global `_cfg`.
- Replace with a clear statement:
  - `RagPipelineMCPService` loads `rag_pipeline_mcp_server.toml`.
  - It builds a `RagConfig` adapter.
  - It injects that adapter into `RagPipeline`.
- Search for stale strings:
  - `rag.llm`
  - `_cfg cache`
  - `module-level config`
  - `override module-level`

## Acceptance Criteria
- No active source comment implies that `rag.llm` exists as a config-cached module.
- No active source comment recommends module-level config override.
- Documentation and comments consistently describe explicit config injection.
