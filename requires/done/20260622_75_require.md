# Title
Add fallback truncation when LLM-based history compression fails

## Summary
When LLM-based history compression fails, apply a deterministic fallback truncation strategy so context growth remains bounded.

## Background
History compression currently depends on LLM summarization.
If summarization fails, the history is returned unchanged.

## Problem
This allows context growth to continue unchecked during LLM instability and increases the risk of a negative feedback loop.

## Required Changes
- Add a fallback truncation path for compression failure.
- Preserve `history_protect_turns`.
- Prefer removing or shrinking lower-priority content first:
  - tool-role messages
  - temporary reasoning artifacts
  - low-value older content
- Make fallback usage visible through `/context`, `/stats`, diagnostics, or logs.
- Distinguish successful compression from fallback truncation in stats.

## Acceptance Criteria
- Context size remains bounded even when LLM summarization fails.
- Protected recent turns remain intact.
- Operators can tell whether compression succeeded normally or fell back to truncation.

## Out of Scope
- Replacing LLM-based compression
- Full redesign of history scoring/selection

## AI Implementation Instruction
Implement this as a resilience-improvement task.

Strict rules:
- Do not remove LLM-based compression.
- Keep protected recent turns intact.
- Prefer deterministic and explainable truncation rules.
- Make fallback behavior observable.
- If truncation priority order is uncertain, mark it as: Needs confirmation

Deliverables:
1. Fallback truncation path
2. Visibility of fallback usage
3. Stats/diagnostics distinguishing compression paths
