# [shared][events] Clarify that ArtifactEvent is a data-only payload and prepare future event-bus extension points

## Summary
`ArtifactEvent` exists as a shared type, but there is no event bus, no delivery mechanism, and no consumers today.

## Problem
The current name and placement can make it look like creating an `ArtifactEvent` triggers some runtime behavior.
In reality, it is only a data shape.

At the same time, there may be a future need to implement an event bus or artifact notification flow.

## Scope
- `shared/events.py`
- shared event documentation
- design notes for future event delivery

## Task
- make it clearer that `ArtifactEvent` is a data-only payload today
- reduce the chance that developers mistake it for a real emitted event
- document that a future event bus may be implemented separately
- if useful, define a minimal future event-envelope direction, such as:
  - `event_id`
  - `source`
  - `timestamp`
  - `correlation_id`

## Expected Result
Developers understand that `ArtifactEvent` does nothing by itself today,
while the codebase still leaves a clean path for a future event bus implementation.

## Acceptance Criteria
- [ ] docs clearly state that `ArtifactEvent` is data-only today
- [ ] developers are less likely to assume an event bus already exists
- [ ] future event-bus extension points are documented at a high level
