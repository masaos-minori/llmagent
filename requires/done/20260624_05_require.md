# Formalize PluginHooks as a first-class pipeline contract

## Summary
Remove the special-case handling of PluginHooks by defining an explicit interface and lifecycle contract.

## Background
The pipeline documentation describes PluginHooks as a post-rerank step that is not a `PipelineStage`. This weakens abstraction consistency and makes extension behavior harder to reason about.

## Objective
Define PluginHooks as a first-class and well-typed execution contract within the pipeline model.

## In Scope
- `scripts/rag/stage.py`
- `scripts/rag/pipeline.py`
- Hook-related type definitions and diagnostics
- `03_rag_03_query_pipeline.md`

## Required Changes
- Define a clear interface for plugin hook execution
- Specify:
  - input data contract
  - output data contract
  - error handling behavior
  - strict vs non-strict execution semantics
- Decide whether hooks become:
  - full `PipelineStage` implementations, or
  - a separate but formally typed stage-like interface
- Update diagnostics and stage result reporting accordingly
- Update documentation to match the final model

## Acceptance Criteria
- Hook execution no longer relies on undocumented special-case behavior
- Strict and non-strict hook failure behavior is explicitly defined
- The pipeline execution model is internally consistent
- The documentation explains exactly where hook execution occurs and what it can mutate

## Out of Scope
- Implementing new plugins
- Changing plugin business logic

## Implementation Notes
Preserve the current runtime ordering unless there is a strong reason to change it. Focus on contract clarity and observability.
