## Title
[High] Enforce tool approval checks inside the tool execution path

## Priority
High

## Labels
bug, security, tool-execution, approval, agent

## Dependencies
None

## Background
Tool approval logic exists in `agent/tool_approval.py`, but the main tool execution path currently allows tool calls to reach execution without mandatory approval checks. Existing tests indicate that without a gateway, all tool calls execute directly.

This creates an approval bypass risk for write, delete, shell, and GitHub write tools.

## Scope
Update the following areas:
- `agent/tool_runner.py`
- `agent/llm_turn_runner.py`
- `agent/tool_approval.py`
- Relevant tests under `tests/`

## Required Changes
- Call `run_approval_checks()` inside `execute_all_tool_calls()` before any DAG or standard execution.
- Execute only `approved_calls`.
- Pass `denied_ids` into `_build_denied_messages(denied_ids)`.
- Persist denied tool results as `tool` role messages.
- Ensure plan-mode blocked tools are denied through the same path.
- Ensure direct calls to `execute_all_tool_calls()` cannot bypass approval.
- Keep workflow-level approval separate from per-tool approval.

## Tests to Add or Update
- Update `tests/test_tool_runner.py`.
- Replace the current expectation that write tools execute without gateway approval.
- Add:
  - `test_write_tool_requires_approval_without_gateway`
  - `test_denied_tool_call_is_returned_as_tool_message`
  - `test_plan_mode_blocked_tool_is_not_executed`
  - `test_execute_all_tool_calls_does_not_bypass_approval`

## Acceptance Criteria
- Write/delete/shell/GitHub write tools do not execute without approval.
- Denied tool calls are returned to the LLM as tool messages.
- Plan-mode blocked tools are not executed.
- Approval audit events are emitted.
- Existing approval unit tests still pass.
