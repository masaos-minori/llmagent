## Summary
Replace smoke-test assertions in RAG quality regression tests with meaningful checks for ranking, RRF behavior, semantic cache behavior, fallback diagnostics, HTTP mode, and Refiner fallback.

## Background
The current test documentation still describes assertions such as `len(result.reranked) >= 0`, which are always true and cannot detect retrieval-quality regressions.

## Problem
The tests may pass even when ranking, RRF fusion, fallback behavior, semantic cache behavior, or diagnostics are broken.

## Required Changes
- Remove always-true assertions such as `len(result.reranked) >= 0`.
- Use deterministic fixtures with known documents and known queries.
- Assert expected `chunk_id`, URL, title, or ranking order for known queries.
- Verify `use_rrf=True` and `use_rrf=False` produce distinguishable ranking behavior.
- Verify diagnostics for:
  - `fusion_mode="rrf"`
  - `fusion_mode="dedup_only"`
  - embed failure counts
  - FTS error counts
  - semantic cache hits
  - HTTP `remote_empty`
  - HTTP `in_process_fallback`
  - Refiner fallback reasons
- Add tests for semantic cache generation and invalidation behavior.

## Acceptance Criteria
- Ranking regressions fail the test suite.
- No-RRF mode is tested as a quality-degraded mode, not only as a no-crash path.
- Semantic cache tests validate actual cache behavior, not just non-exception behavior.
- Embed failure and HTTP fallback tests validate diagnostics.
- Refiner fallback tests validate reason and counters.

## Out of Scope
- Building a large benchmark or evaluation framework.
- Changing RRF, rerank, or Refiner algorithms unless a test exposes a real bug.
