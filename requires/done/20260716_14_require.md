# [High] Enforce mandatory workflow execution and startup validation severity policy

## Priority

High

## Type

Architecture Invariant / Startup Reliability / Validation Policy / Regression Prevention

## Consolidated from

- Treat WorkflowEngine as the mandatory turn execution path
- Specify and test startup validation severity policy

## Background

The Agent documentation states that each user turn is executed through `WorkflowEngine.run()` using plan, execute, and verify stages. It also states that workflow definition load failure stops construction or startup, and that there is no direct fallback path outside the workflow engine.

The documentation also describes startup validation results such as OK, WARNING, FATAL, and SKIPPED. It states that startup stops when any FATAL result exists and that already-started MCP subprocesses are shut down during rollback.

These are both startup and lifecycle invariants. Workflow loading, validation, and turn execution must be treated as one coherent runtime contract.

## Problem

If the code or tests preserve older assumptions, the system can become ambiguous in two ways.

First, direct turn execution may accidentally reappear as a fallback if workflow loading fails or if `WorkflowEngine.run()` is bypassed.

Second, startup validation severity may be applied inconsistently. For example, routing drift, schema mismatch, embedding dimension mismatch, optional memory failure, or MCP readiness failure may be classified differently depending on the code path.

This can lead to unsafe startup, inconsistent dev/prod behavior, or hidden direct execution paths that contradict the documented architecture.

## Reason for change

The workflow engine is the documented mandatory execution path. Startup validation is the gate that determines whether the runtime is safe to start.

These must be enforced together because startup constructs the workflow-backed runtime, validates services, starts MCP subprocesses, and rolls back on fatal failures.

Without this issue, tests may pass even when the runtime violates its documented architecture.

## Design intent

Make the following behavior explicit and testable:

- Workflow definition load is mandatory.
- `Orchestrator.handle_turn()` always executes via `WorkflowEngine.run()`.
- The turn lifecycle always goes through plan, execute, and verify stages.
- Approval-pending state blocks new turns through workflow state.
- Startup validation has documented severities.
- Fatal validation stops startup.
- Startup rollback shuts down already-started subprocesses.
- Optional capabilities degrade only when explicitly allowed.

## Implementation instructions

### Workflow invariant

- Add tests that workflow definition load failure stops startup or orchestrator construction.
- Add tests that `handle_turn()` always invokes `WorkflowEngine.run()`.
- Search for direct `LLMTurnRunner` fallback paths and remove or explicitly reject them.
- Document the current no-op behavior of `plan_fn`, or separate it as future planning expansion.
- Add regression tests for `approval_pending=True` blocking new user turns.
- Verify pending approval recovery restores workflow state consistently.

### Startup validation severity

- Extract all startup validation checks from implementation.
- Define severity for each check: OK, WARNING, FATAL, or SKIPPED.
- Define local/dev versus production behavior where such distinction exists.
- Make routing drift, schema mismatch, and unsafe configuration severity explicit.
- Decide how optional memory layer failures and optional MCP server failures are classified.
- Ensure FATAL validation shuts down already-started MCP subprocesses.
- Improve startup output so each validation result includes check name, severity, and reason.
- Add tests for severity classification and rollback behavior.

## Acceptance criteria

- No direct user-turn execution path exists outside `WorkflowEngine`.
- Missing or invalid workflow definition causes deterministic startup failure.
- Approval-pending state blocks new turns until approval or rejection.
- Every startup validation check has documented severity.
- FATAL validation reliably stops startup.
- Startup rollback shuts down already-started MCP subprocesses.
- Tests cover workflow invariants, validation severity, and rollback behavior.
- Documentation and implementation describe the same startup and turn execution model.

## Non-goals

- Do not redesign the workflow engine.
- Do not implement advanced planning behavior unless separately scoped.
- Do not add unrelated startup checks.
- Do not hide fatal safety failures behind warnings without explicit design approval.
