# Task: Add Connection Reuse / Pooling Option to SQLiteHelper

```markdown
## Overview and Context
The operational foundation of `SQLiteHelper` is designed around opening and closing database connections on a per-query basis. While this configuration benefits isolation and pairs with WAL mode concurrency rules, high-frequency operations—such as rapid ingestion, conversational message history logging, and real-time vector or FTS5 indices synchronization—incur massive file system I/O connection overhead. This can lead to intermittent database lock contention ("Database is locked").

## Acceptance Criteria (AC)
- [ ] Introduce an optional connection-reuse parameter (e.g., `reuse_connection: bool = False`) to the `SQLiteHelper.open()` initialization context pattern.
- [ ] When enabled, retain the connection instance across consecutive operations inside the same thread or execution lifecycle scope without degrading thread-safety rules.
- [ ] Maintain seamless backward compatibility with all existing `with SQLiteHelper().open(write_mode=True)` context blocks across the repository.

## Technical Specifications and Implementation Steps
1. Target Files: `shared/db/helper.py`
2. Step 1 (Connection Mapping): Implement a thread-local storage or reference pointer inside `SQLiteHelper` to map active connection hooks when `reuse_connection=True` is provided.
3. Step 2 (Context Management modification): Alter `__enter__` and `__exit__` methods to skip calling `.close()` if the connection reuse flag is active, deferring closure until the overarching session context explicitly completes.

## Validation and Testing Plan
- Command: `pytest tests/shared/test_db_performance.py`
- Required Test Cases:
  - `test_connection_reuse_same_instance`: Asserts that consecutive operations return the identical underlying connection memory ID.
  - `test_backward_compatibility_per_query`: Ensures standard calls without the flag continue to open and close connections on a per-query basis.

## Constraints and Core Rules
- Ensure thread-local boundaries are never breached; different execution threads must never share an underlying raw SQLite connection pointer.
