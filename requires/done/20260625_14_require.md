## [High][eventbus][runtime] Offload all SQLite operations from async endpoints to a threadpool

### Summary
The Event Bus currently executes synchronous `sqlite3` operations directly inside `async def` FastAPI endpoints and async background tasks.

### Problem
Synchronous SQLite calls block the event loop.
This can delay publish requests, stall SSE subscriptions, and cause background task interference under load.

### Scope
- app.py
- health()
- publish()
- replay()
- subscribe()
- dlq_list()
- dlq_requeue()
- _get_seq()
- _dlq_loop()

### Facts
- Event Bus endpoints are implemented as `async def`. 【1-6bdda3】【1-2f69b2】
- `sqlite3.Connection.execute()`, `fetchall()`, and `commit()` are called directly inside those async handlers. 【1-6bdda3】
- The DLQ background loop also calls synchronous DB logic directly. 【1-6bdda3】【1-835298】

### Task
- move all SQLite operations behind a single DB execution wrapper
- execute that wrapper via `run_in_threadpool` or an equivalent thread-offload mechanism
- ensure every endpoint and background task uses the offload path consistently
- add tests that fail if direct event-loop-thread DB execution is reintroduced

### Expected Result
No synchronous SQLite operation runs directly on the FastAPI event loop.

### Acceptance Criteria
- all SQLite execute/fetch/commit operations are offloaded
- no endpoint performs direct SQLite I/O on the event loop
- the DLQ loop also uses the same offload path
