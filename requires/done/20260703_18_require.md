# [Medium] Align strict tool-definition validation docs with implementation

## Goal

Resolve the mismatch between startup tool-definition validation documentation and implementation.

## Background

The current implementation raises `RuntimeError` in strict mode when all MCP servers are unreachable and tool definitions cannot be validated.

Some existing documentation may state that validation is skipped even in strict mode when all servers are unreachable. That does not match the implementation.

## Scope

Docs first. Implementation changes only if the product decision is to restore skip behavior.

Update:

- `agent/repl_health.py`
- `tests/test_repl_health.py`
- `04_mcp_06_configuration_and_operations.md`
- `04_mcp_90_inconsistencies_and_known_issues.md`

## Decision Required

Choose one canonical behavior:

### Option A: Keep current fail-fast behavior

When `tool_definitions_strict=true` and all MCP servers are unreachable, startup fails with `RuntimeError`.

### Option B: Restore skip behavior

When all MCP servers are unreachable, validation is skipped even in strict mode.

Recommended option: Option A, because strict mode should fail when validation cannot be performed.

## Implementation Tasks

If choosing Option A:

- Update docs to state strict mode fails when all servers are unreachable.
- Add tests for strict + all unreachable -> `RuntimeError`.
- Add tests for non-strict + all unreachable -> warning or skip behavior.
- Ensure error message lists unreachable servers.

If choosing Option B:

- Change implementation to skip validation even in strict mode.
- Update tests accordingly.
- Document why strict mode allows skip.

## Acceptance Criteria

- Docs and implementation agree.
- Strict-mode behavior is explicitly documented.
- Tests cover:
  - all reachable
  - partial unreachable
  - all unreachable
  - mismatch in strict mode
  - mismatch in non-strict mode

## Validation

Run:

    uv run pytest tests/test_repl_health.py -v
