# Clarify schema_version, capabilities, and deferred /v1/tools fields

## Priority

Medium

## Scope

Documentation-only unless missing behavior is intentionally implemented.

## Target documentation sections

- `04_mcp_02_01_endpoints-and-transport.md`
  - `/v1/tools`
- `04_mcp_03_06_tool-runtime-availability-metadata.md`
  - `Future / deferred design options`
  - `include_disabled query parameter`
  - `disabled_code structured field`

## Problem

The documentation needs a sharper boundary between current `/v1/tools` fields and deferred future options.

Current implementation appears to support:

- Top-level `schema_version` as discovery-time metadata.
- Per-tool `capabilities` as optional metadata.
- `enabled` as input to `RuntimeTool.enabled_for_llm`.

Current implementation does not appear to support:

- `include_disabled` query parameter.
- `disabled_code`.
- Top-level `capabilities`, unless separately implemented.

## Required documentation changes

Document current fields separately from future fields.

### Current fields

- `schema_version`
- `tools`
- `name`
- `description`
- `inputSchema`
- `status`
- `enabled`
- `disabled_reason`
- `is_write`
- `requires_serial`
- `resource_scope`
- per-tool `capabilities`

### Deferred fields / options

- `include_disabled`
- `disabled_code`
- top-level capabilities, if not implemented

## Reason for change

API consumers and server authors need to know which fields are safe to rely on today. Deferred design options should not be confused with implemented API behavior.

## Acceptance criteria

- Implemented fields and deferred fields are clearly separated.
- `include_disabled` is not presented as available unless implemented.
- `disabled_code` is not presented as available unless implemented.
- Per-tool `capabilities` behavior is documented.
- Top-level capabilities are either documented as unsupported or implemented and documented.
