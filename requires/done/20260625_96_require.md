# Title
Extend memory observability to show memory quality, not just memory status

## Priority
Medium

## Labels

## Dependencies
- Issue 2 recommended
- Issue 3 recommended

## Summary
Improve observability so operators can see not only whether memory is enabled, but also what kind of memory is being extracted, injected, skipped, deduplicated, and used. Current docs already describe `/memory status`, diagnostics, startup display, and memory consistency checks.

## Background
The current observability surface can show enabled/disabled state and retrieval mode, but it is weaker at showing the quality and composition of stored memories.

## Problem
Persistent Semantic Memory is valuable only if operators can tell whether the system is storing and restoring the right things.

## Required Changes
- Extend `/memory status` with counts and useful quality signals.
- Consider adding:
  - semantic / episodic counts
  - RULE / DECISION / FAILURE / CONVERSATION counts
  - embed-skip counts
  - last retrieval mode
  - last ingest summary
- Consider adding memory-related summary fields to diagnostics and logs.

## Acceptance Criteria
- Memory quality is easier to inspect operationally.
- `/memory status` and diagnostics complement each other.
- The memory layer’s useful behavior is more visible in production.

## Out of Scope
- Full metrics platform / dashboard rollout
- Redesigning the extraction algorithm itself

## Components
- `/memory status` implementation
- diagnostics / observability output
- memory ingestion log summaries
- `05_agent_10_operations-and-observability.md`

## AI Implementation Instruction
Implement this as a **memory observability task**.

Strict rules:
- Do not overload status output with low-value noise.
- Distinguish extraction, dedup, and injection metrics clearly.
- Preserve the meaning of existing diagnostics.
- If a metric is ambiguous, mark it as: `Needs confirmation`
