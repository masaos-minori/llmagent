# MCP Server Process Startup: stderr File Leak on getpgid Failure

## Priority

Medium

## Target Files

- `scripts/agent/http_lifecycle.py` (lines 240-243)

## Background

When starting an MCP server via `McpServerLifecycle.start_server()`, the process is spawned using `subprocess.Popen()` and its process group ID is captured via `os.getpgid()`. If `getpgid()` fails after `Popen()` succeeds, stderr resources are not cleaned up.

## Problem

In `start_server()` at lines 240-243:

```python
try:
    self._http_pgids[server_key] = os.getpgid(proc.pid)
except OSError:
    pass
self._http_procs[server_key] = proc
```

The sequence is:
1. `Popen()` succeeds, `proc` is created
2. `_stderr_files[server_key]` and `_stderr_log_paths[server_key]` entries exist
3. `os.getpgid(proc.pid)` raises `OSError` (process may have already exited)
4. The `except OSError:` block catches but does nothing (`pass`)
5. `self._http_procs[server_key] = proc` stores the process reference
6. Later, `_terminate_with_timeout()` calls `proc.terminate()` but stderr files remain leaked

The `except Exception:` block at lines 235-239 only handles `Popen()` failures, not `getpgid()` failures.

## Reason for Change

- Long-running deployments accumulate leaked file descriptors
- A transient race condition (process exits between Popen and getpgid) causes silent resource leak
- No logging or warning about the failure, making debugging difficult

## Implementation Intent

Move the `getpgid()` call into the same exception handler that cleans up stderr resources, ensuring consistent cleanup regardless of which operation fails.

## Implementation Instructions

Refactor lines 240-244 to handle `getpgid()` failure with proper cleanup:

```python
try:
    self._http_pgids[server_key] = os.getpgid(proc.pid)
except OSError:
    logger.warning(
        "Failed to get pgid for %s (pid=%d); process may have exited",
        server_key, proc.pid
    )
    # Clean up stderr resources since process may be gone
    stderr_fh = self._stderr_files.pop(server_key, None)
    if stderr_fh is not None:
        stderr_fh.close()
    self._stderr_log_paths.pop(server_key, None)
    self._http_procs.pop(server_key, None)
    return  # Cannot proceed without valid process reference
```

Key points:
- Add logging to surface the issue during runtime
- Pop the process reference from `_http_procs` since we cannot manage a process whose pgid we cannot track
- Close and remove stderr file references
- Return early since we cannot perform health checks without a valid process

## Acceptance Criteria

1. When `os.getpgid()` fails after `Popen()` succeeds, stderr files are properly closed and removed from tracking dictionaries
2. A warning-level log message is emitted when this occurs
3. The process reference is removed from `_http_procs` to prevent later operations from failing silently
4. No new file descriptor leaks occur during startup

## Tests

- Unit test: mock `os.getpgid()` to raise `OSError` after `Popen()` succeeds, verify stderr files are cleaned up
- Integration test: start MCP server with a misconfigured `cmd` that exits immediately, verify no leaked file descriptors
