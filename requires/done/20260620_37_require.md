# Title
Improve result classification and observability in HTTP RAG mode (`rag_service_url`)

## Summary
Make it easier to distinguish remote-service failure fallback from valid empty remote results when using `rag_service_url`.

## Background
In HTTP mode:
- `None` means failure and triggers in-process fallback
- `""` is treated as a valid empty context result

## Problem
Operators may not be able to tell whether an empty response means “no relevant context exists” or “a remote path behaved unexpectedly but still returned empty content.”

## Required Changes
- Record whether a returned context came from:
  - remote success
  - remote empty result
  - remote failure with local fallback
- Surface this classification in debug output, diagnostics, or stage results.
- Keep current fallback behavior unless a separate issue changes the policy.

## Acceptance Criteria
- Operators can distinguish remote success, remote empty, and remote fallback cases.
- Diagnostics are visible without requiring deep internal tracing.
- HTTP fallback semantics remain documented.

## Out of Scope
- Replacing remote RAG mode
- Redesigning the HTTP protocol

## AI Implementation Instruction
Implement this as a result-classification and observability task.

Strict rules:
- Do not silently change the meaning of `None` vs `""`.
- Preserve existing fallback semantics unless explicitly changed elsewhere.
- Focus on observability, not protocol redesign.
- If classification remains ambiguous in edge cases, mark it as: Needs confirmation

Deliverables:
1. Clear classification of remote/local result paths
2. Visible diagnostics for HTTP-mode outcomes
3. Documentation aligned with observed behavior
