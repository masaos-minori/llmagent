# Agent Startup Processing — Embedding Dimension Mismatch Only Logged in Production Mode

## Summary

If the RAG database contains embeddings with a different dimension count than expected by the model, the mismatch is only detected and logged in production mode. In non-production mode, the mismatch goes undetected, potentially causing silent incorrect similarity search results.

## Severity

Low

## Files

- `scripts/agent/startup.py:171-187`

## Details

In `StartupOrchestrator._validate_rag_consistency()`:

```python
async def _validate_rag_consistency(self) -> None:
    ctx = self._ctx
    if ctx.services_required.rag_maintenance is None:
        return
    if ctx.cfg.mcp.security_profile != SecurityProfile.PRODUCTION:
        return  # <-- Skipped entirely in non-production
    
    db_path = ctx.services_required.rag_maintenance.db_path
    if db_path is None:
        return
    try:
        actual_dim = ctx.services_required.rag_maintenance.get_embedding_dimension(db_path)
        model_dim = ctx.services_required.rag_maintenance.model_embedding_dim
        if actual_dim != model_dim:
            msg = f"RAG DB vector dim ({actual_dim}) != model dim ({model_dim})"
            logger.error(msg)
            self._view.write_fatal(f"{OutputTag.FATAL} {msg}")
    except Exception as e:
        logger.error("Failed to validate RAG consistency: %s", e)
```

This early return means non-production deployments never validate RAG consistency. A common scenario where this matters:

1. Developer switches embedding models between sessions (e.g., `text-embedding-3-small` → `text-embedding-3-large`)
2. Old RAG database still has vectors from the old model with different dimensions
3. Similarity search returns meaningless results because cosine similarity across dimensions is invalid
4. No warning or error is emitted — the operator sees normal-looking behavior but gets wrong answers

## Reproduction Steps

1. Start with a RAG database populated using one embedding model
2. Change the config to use a different embedding model with different dimensions
3. Start the agent in non-production mode
4. Perform a query against the RAG database
5. Observe that no warning about dimension mismatch is shown

## Expected Behavior

Dimension validation should run regardless of security profile. At minimum, non-production mode should emit a warning instead of silently skipping.

## Proposed Fix

Remove the security profile guard and emit warnings based on severity level:

```python
async def _validate_rag_consistency(self) -> None:
    ctx = self._ctx
    if ctx.services_required.rag_maintenance is None:
        return
    
    db_path = ctx.services_required.rag_maintenance.db_path
    if db_path is None:
        return
    try:
        actual_dim = ctx.services_required.rag_maintenance.get_embedding_dimension(db_path)
        model_dim = ctx.services_required.rag_maintenance.model_embedding_dim
        if actual_dim != model_dim:
            msg = f"RAG DB vector dim ({actual_dim}) != model dim ({model_dim})"
            if ctx.cfg.mcp.security_profile == SecurityProfile.PRODUCTION:
                logger.error(msg)
                self._view.write_fatal(f"{OutputTag.FATAL} {msg}")
            else:
                logger.warning(msg)
                self._view.write_warning(f"{OutputTag.NON_FATAL} {msg}")
    except Exception as e:
        logger.error("Failed to validate RAG consistency: %s", e)
```
