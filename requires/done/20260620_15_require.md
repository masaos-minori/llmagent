# Title
Add fallback truncation when LLM-based history compression fails

## Summary
When LLM-based history compression fails, apply a deterministic fallback truncation strategy so that context growth remains bounded.

## Background
HistoryManager currently attempts LLM-based compression when context exceeds configured limits.
If compression fails, it returns the original history unchanged.

## Problem
This allows context growth to continue unchecked during LLM instability and may create a negative feedback loop in which large context makes later LLM calls even less reliable.

## Required Changes
- Add a fallback truncation path for compression failure.
- Preserve `history_protect_turns` semantics.
- Prefer trimming low-value content types first, such as:
  - tool-role message history
  - temporary reasoning artifacts
  - older low-priority content
- Make fallback usage visible through `/context`, `/stats`, audit output, or debug logs.
- Distinguish successful LLM compression from fallback truncation in stats.

## Acceptance Criteria
- Context size remains bounded even when LLM compression fails.
- Protected recent turns are preserved.
- Operators can tell whether compression succeeded normally or fell back to truncation.

## Out of Scope
- Full redesign of history selection policy
- Replacing LLM-based compression entirely

## AI Implementation Instruction
Implement this as a resilience improvement, not a change to the core compression concept.

Strict rules:
- Do not remove LLM-based compression.
- Keep recent protected turns intact.
- Prefer deterministic, explainable truncation over hidden heuristics.
- Make fallback behavior observable.
- If the correct ordering of truncation candidates is uncertain, mark it as: Needs confirmation

Deliverables:
1. Fallback truncation path
2. Visibility of fallback usage
3. Stats or diagnostics that distinguish compression paths
