# Strengthen McpServerConfig value validation

## Priority

High

## Type

Implementation / Configuration Safety

## Background

`McpServerConfig` controls MCP transport, lifecycle, authentication, timeouts, tool name metadata, and subprocess launch behavior.

It already validates enum types and some cross-field constraints.

## Problem

Additional invalid values may still be accepted and fail later at runtime.

Examples include:

- Negative call timeout
- Zero or negative startup timeout
- Empty tool names
- Duplicate tool names
- Non-string environment keys or values
- Invalid HTTP URL
- Non-string auth token

## Goal

Fail fast on invalid MCP server configuration.

## Scope

Affected areas:

- `shared/mcp_config.py`
- Config builder tests
- Startup validation tests

## Implementation Tasks

- Validate `call_timeout_sec >= 0`.
- Validate `startup_timeout_sec > 0`.
- Validate `tool_names` is a list of non-empty strings.
- Reject duplicate tool names.
- Validate `auth_token` is a string.
- Validate `env` is `dict[str, str]`.
- Validate HTTP URL scheme and netloc.
- Include the MCP server key in validation error messages.

## Acceptance Criteria

- Invalid timeout values fail config construction.
- Empty tool names fail config construction.
- Duplicate tool names fail config construction.
- Invalid environment values fail config construction.
- Invalid HTTP URLs fail config construction.
- Error messages identify the affected server key.
- Existing valid configs continue to load.

## Validation Plan

Add tests for:

- Valid persistent HTTP config.
- Valid subprocess HTTP config.
- Missing URL.
- Subprocess mode without command.
- Negative call timeout.
- Zero startup timeout.
- Empty tool name.
- Duplicate tool name.
- Invalid env key or value.
- Invalid URL.

## Risks

Stricter validation may reject previously accepted but invalid configs. This is intended to prevent runtime failures.

## Out of Scope

- Adding non-HTTP transports.
- Changing config file layout.
- Documentation-only updates.
