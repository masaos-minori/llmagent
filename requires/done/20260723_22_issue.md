# Compress CLI Argument and Configuration Key Reference Details

## Priority

Medium

## Summary

Compress exhaustive CLI argument tables and configuration key tables that can be mechanically confirmed from source code, command help, configuration dataclasses, or actual configuration files.

The updated documentation must retain information that is relevant to design, operation, ownership, failure behavior, security, compatibility, and maintenance.

## Reason for Change

Some design and operations documents contain complete CLI argument lists, configuration key lists, default values, and type descriptions.

These details are often implementation-derived and can usually be confirmed from:

- source code
- configuration dataclasses
- command help output
- actual TOML configuration files
- deployment scripts

Keeping complete copies of these details in design documents creates several maintenance problems.

First, the documentation can become stale when CLI options, configuration keys, default values, or implementation types change.

Second, exhaustive tables make it harder for readers to distinguish architectural intent from implementation reference material.

Third, AI agents may treat copied configuration tables as the canonical source even when the implementation has already changed.

Fourth, long reference tables increase the amount of context required to understand a document, even when the reader only needs design-level information.

Therefore, CLI and configuration documentation should be compressed so that design documents focus on meaning, ownership, constraints, and operational impact rather than reproducing full schemas.

## Implementation Intent

The intent is not to remove configuration knowledge from the documentation.

The intent is to separate two types of information:

1. Information that belongs in implementation, command help, or configuration files.
2. Information that belongs in design documentation.

The resulting documents should explain:

- which process owns which configuration file
- which configuration values affect runtime behavior
- which configuration changes require restart
- which settings are hot-reloadable
- which settings can cause startup failure
- which settings affect security posture
- which settings affect fail-fast or fail-open behavior
- which settings are deprecated
- which settings affect data compatibility or persistence
- which settings remain unresolved or require confirmation

The documentation should no longer attempt to duplicate full CLI help output, full configuration files, or full configuration schemas.

The design intent is to make configuration documentation useful for architectural understanding and operational decision-making, while leaving mechanically discoverable details to the implementation and actual configuration files.

## Required Changes

For each target document:

1. Identify exhaustive CLI argument lists and configuration key tables.
2. Classify each table or list before editing.
3. Use the following classification categories:
   - implementation-derived detail
   - design-relevant configuration behavior
   - operationally important setting
   - security-sensitive setting
   - persistence-sensitive setting
   - deprecated item
   - unresolved or Needs Confirmation item
4. Remove or compress only implementation-derived details.
5. Preserve design-relevant configuration behavior.
6. Preserve process ownership and configuration ownership.
7. Preserve restart and hot-reload boundaries.
8. Preserve fail-fast and fail-open behavior.
9. Preserve security-sensitive settings and operational warnings.
10. Preserve deprecated settings and references to deprecated-item governance.
11. Preserve unresolved configuration questions and link them to Needs Confirmation where applicable.
12. Replace exhaustive tables with concise Japanese summaries and source references.

## Content to Remove or Compress

Remove or compress:

- complete CLI argument tables
- complete configuration key tables
- default-value-only tables
- type-only tables
- tables that directly duplicate configuration dataclasses
- tables that directly duplicate TOML files
- long CLI help output copied into design documents
- repetitive descriptions that do not explain design or operational impact
- configuration tables that only restate names, types, and defaults

## Content to Keep

Keep:

- process-local configuration ownership
- the rule that each process reads only its own configuration file
- the rule that shared common configuration files should not be introduced
- hot-reloadable versus restart-required boundaries
- startup-failure conditions
- workflow definition requirements
- absence of workflow optional mode where currently documented
- `startup_mode` operational impact
- `security_profile` fail-fast and fail-open behavior
- security-sensitive configuration behavior
- data compatibility or persistence-sensitive configuration behavior
- deprecated configuration items
- Known Issues
- Needs Confirmation items
- behavior verified by tests or observed operationally

## Suggested Replacement Structure

Use Japanese section headings such as:

- `## 設定の責務境界`
- `## 変更時の運用影響`
- `## 再起動が必要な設定`
- `## ホットリロード可能な範囲`
- `## 起動失敗につながる設定`
- `## セキュリティに関わる設定`
- `## 実装詳細の参照先`

Suggested Japanese wording:

- 完全な設定キー、型、デフォルト値は、実装上の設定定義および実際の設定ファイルを参照する。
- この設計書では、設定の所有者、変更時の影響、再起動要否、失敗時の挙動、運用上の注意のみを扱う。
- 設定値そのものではなく、設定変更がどの責務境界や運用挙動に影響するかを記載する。

## Acceptance Criteria

- Exhaustive CLI argument tables are reduced where they only duplicate implementation or command help.
- Exhaustive configuration key tables are reduced where they only duplicate configuration files or code.
- Configuration ownership remains clear.
- Restart and hot-reload behavior remains documented where relevant.
- Fail-fast and fail-open behavior remains documented.
- Workflow-required behavior remains documented.
- Deprecated or unresolved settings are not removed.
- Known Issues and Needs Confirmation items are preserved.
- All new or rewritten documentation text is written in Japanese.
- Markdown renders correctly.

## Definition of Done

- Target documents are more concise and design-focused.
- Configuration documentation no longer reads like a copied schema.
- CLI documentation no longer duplicates full command help output.
- Important operational and architectural configuration behavior remains explicit.
- No source code is modified.
- No ADR documents are created.
- `00_index.md` is not modified.
- No unrelated document cleanup is performed.

## Out of Scope

- Do not modify source code.
- Do not remove process-local configuration rules.
- Do not remove Workflow-required behavior.
- Do not remove fail-fast or fail-open behavior.
- Do not remove deprecated configuration references without preserving them in the appropriate deprecated-items or known-issues context.
- Do not resolve Needs Confirmation items.
- Do not migrate Known Issues documents.
- Do not bulk-format unrelated Markdown files.
- Do not create new governance documents.

## AI Implementation Instruction

Work document by document.

Before removing a CLI or configuration detail, classify it.

Remove or compress only details that are mechanically recoverable from implementation, command help, or configuration files.

Keep any configuration information that affects design intent, responsibility boundaries, operational constraints, failure behavior, security, persistence, compatibility, or maintenance.

Use Japanese for all new or rewritten documentation text.

Prefer concise source references over exhaustive copied tables.

If unsure whether a configuration entry is design-relevant, keep it and add a short note for later review instead of deleting it.
