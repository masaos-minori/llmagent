## Background

A full audit of every leaf key across `config/*.toml`, `config/workflows/*.json` against the codebase (`scripts/`) found the following inventory:

- **19 files audited**: `agent.toml`, 10 `*_mcp_server.toml` files, `eventbus.toml`, `chunk_splitter.toml`, `crawler.toml`, `ingester.toml`, `workflows/default.json`.
- **1 orphaned key** (defined in TOML, no corresponding parse/lookup anywhere in Python): `config/mdq_mcp_server.toml`'s `status`.
- **12 parsed-but-unconsumed keys** (read into a Python field/variable at config-load time, then never referenced again — "dead config"):
  1. `config/agent.toml`'s top-level `web_search_max_results` (line 29) — its only read-site is `cmd_config_display.py:43`'s `/config` status printout; the actual `search_web` tool result cap comes from `config/web_search_mcp_server.toml`'s own (also-unused, see below) `default_max_results`/`max_results_limit`, which in turn is shadowed by hardcoded module constants (`DEFAULT_MAX_RESULTS = 5`, `MAX_RESULTS_LIMIT = 20` in `scripts/mcp_servers/web_search/models.py`).
  2. `config/agent.toml`'s `[mcp_servers.<name>].healthcheck_mode`, present in all 10 server blocks (`shell`, `git`, `web_search`, `file_delete`, `file_write`, `file_read`, `github`, `cicd`, `rag_pipeline`, `mdq`) — `scripts/shared/mcp_config.py::_derive_healthcheck_mode()` always returns `HealthcheckMode.HTTP` regardless of this value; the field round-trips through validation and reload-diffing (`config_reload.py`'s `_MCP_SERVER_FIELDS`) but never changes runtime behavior.
  3. `config/mdq_mcp_server.toml`'s `include_globs`, `exclude_globs`, `max_snippet_chars`, `max_chunk_chars`, `max_file_chars`, `search_timeout_sec` — all six are assigned to `self.<name>` in `MdqService.__init__` (`scripts/mcp_servers/mdq/service.py:53-60`) and never read again; `indexer.py` hardcodes `*.md` glob patterns instead of using `include_globs`/`exclude_globs`, and `search.py:114` hardcodes a `[:150]` snippet slice instead of using `max_snippet_chars`.
  4. `config/web_search_mcp_server.toml`'s `default_max_results`, `max_results_limit` — parsed into the `WebSearchConfig` singleton `_cfg` in `scripts/mcp_servers/web_search/server.py:34`, but `_cfg` has no read-sites anywhere in that package; the module's hardcoded `DEFAULT_MAX_RESULTS`/`MAX_RESULTS_LIMIT` constants (same file, `models.py`) do the actual enforcement, coincidentally defaulting to the same values as the config file — masking the disconnect.
  5. `config/workflows/default.json`'s `stages[].description` (plan/execute/verify) — parsed into `StageDefinition.description` (`workflow_loader.py:119`); referenced only in test fixtures, never read by `workflow_engine.py` or any other runtime code.
  6. `config/workflows/default.json`'s `stages[].retryable` (plan/execute/verify) — parsed into `StageDefinition.retryable` (`workflow_loader.py:121`); `workflow_engine.py::_run_execute_with_retry` retries the hardcoded `"execute"` stage unconditionally and never checks this field.
  7. `config/workflows/default.json`'s `retry_policy.backoff` — parsed into `RetryPolicy.backoff` (`workflow_loader.py:128`), validated at parse time against `_SUPPORTED_BACKOFF = {"fixed"}`, but never read again; only `retry_policy.backoff_sec` drives the actual `asyncio.sleep()` duration in `workflow_engine.py`.
- All other keys across all 19 files (including every key in `eventbus.toml`, `chunk_splitter.toml`, `crawler.toml`, `ingester.toml`, and 8 of the 10 `*_mcp_server.toml` files) were confirmed to have live read-sites driving real behavior — not in scope for this requirement.

## Problem

A config key that appears settable but has no effect on runtime behavior misleads an operator changing it — they will reasonably assume the change takes effect, when in fact:
- `healthcheck_mode` gives the false impression that a non-HTTP healthcheck transport is selectable today.
- `mdq_mcp_server.toml`'s `include_globs`/`exclude_globs`/`max_snippet_chars`/`max_chunk_chars`/`max_file_chars`/`search_timeout_sec` look like real indexing/search tuning knobs but are silently ignored — an operator trying to exclude a directory from MDQ indexing via `exclude_globs`, for example, would see no effect and have no error to explain why.
- `web_search_max_results` and its per-server counterparts create three independent-looking size knobs (`agent.toml`'s field, the web-search server's own config field, and that server's hardcoded constants) where only the hardcoded constants actually govern behavior.
- `workflows/default.json`'s `retryable`/`backoff`/`description` fields suggest per-stage retry control and stage documentation that isn't actually wired into the engine's decision logic.

This is technical debt that actively increases operational risk during config changes: nothing fails loudly when one of these keys is edited, so the gap is only found by reading source code, not by observing behavior.

## Reason for change

Every config key should either drive real behavior or not exist. Per-key, the right fix is either (a) delete the key because it documents an unfinished/abandoned idea, or (b) wire it into the implementation because it represents a real, intended feature that was left unconnected. This requirement resolves that ambiguity per key so the config surface accurately reflects what the system actually does.

## Implementation instructions

For each finding, choose Option A (remove) or Option B (wire up) per the guidance below; do not leave a key in the "silently ignored" state either way.

### 1. `web_search_max_results` (agent.toml) + `default_max_results`/`max_results_limit` (web_search_mcp_server.toml)

**Option A (recommended): consolidate on the web-search server's own config, remove the rest.**
- Remove `web_search_max_results` from `config/agent.toml` and its `RAGConfig.web_search_max_results` field (`config_dataclasses.py`), its builder line (`config_builders.py`), and its display line (`cmd_config_display.py:43`).
- In `scripts/mcp_servers/web_search/models.py`, replace the hardcoded `DEFAULT_MAX_RESULTS`/`MAX_RESULTS_LIMIT` module constants with reads from the already-parsed `WebSearchConfig` singleton (`default_max_results`, `max_results_limit`), so the config file's values actually take effect.

**Option B: keep `agent.toml`'s field, make it the single source of truth, remove the web-search server's own duplicate.**
- Remove `default_max_results`/`max_results_limit` from `config/web_search_mcp_server.toml` and `WebSearchConfig`; have the web-search server read the value from `agent.toml`'s `RAGConfig.web_search_max_results` at request time instead (requires wiring, since the web-search MCP server process may not currently load `agent.toml`).

### 2. `[mcp_servers.<name>].healthcheck_mode` (all 10 blocks)

**Option A (recommended): remove until a second transport exists.**
- Remove the `healthcheck_mode = "http"` line from all 10 `[mcp_servers.*]` blocks in `config/agent.toml`.
- Remove `McpServerConfig.healthcheck_mode` field, `_derive_healthcheck_mode()`, and the `"healthcheck_mode"` entry in `config_reload.py`'s `_MCP_SERVER_FIELDS`.
- Update any doc that lists `healthcheck_mode` as a configurable field to note it was removed (HTTP is currently the only supported healthcheck transport).

**Option B: keep as a documented placeholder for a planned second transport.**
- If a non-HTTP healthcheck transport is on the near-term roadmap, keep the field but add an explicit code comment at `_derive_healthcheck_mode()` and a doc note stating it is currently a no-op pending that work, so a reader doesn't need to trace the dead path themselves.

### 3. `mdq_mcp_server.toml`'s `status` (orphaned)

**Option A (recommended): remove.** No parse site exists anywhere; delete the line from `config/mdq_mcp_server.toml`.
**Option B:** if `status` was meant to gate something (e.g. a maintenance/read-only mode), specify what and wire it into `MdqService`.

### 4. `mdq_mcp_server.toml`'s `include_globs`, `exclude_globs`, `max_snippet_chars`, `max_chunk_chars`, `max_file_chars`, `search_timeout_sec`

**Option A: remove all six** and the corresponding `self.<name>` assignments in `MdqService.__init__`, since `indexer.py`/`search.py` already hardcode equivalent behavior.
**Option B (recommended — these look like real, useful knobs): wire each into the code path it was clearly meant to control.**
- `include_globs`/`exclude_globs` → `indexer.py`'s `rglob("*.md")` call sites (lines 138, 218, 256, 282): iterate `include_globs` instead of the hardcoded pattern, and filter out paths matching `exclude_globs`.
- `max_snippet_chars` → `search.py:114`'s hardcoded `row["content"][:150]` slice.
- `max_chunk_chars`/`max_file_chars` → wherever chunking/file-size limits are currently implicit or hardcoded in the indexing path (locate via the same read of `indexer.py` used above).
- `search_timeout_sec` → wrap the search execution path in `asyncio.wait_for(..., timeout=self.search_timeout_sec)`.
- This option is recommended because these keys represent genuinely useful operational controls (limiting indexed file types, bounding snippet/chunk size, bounding search latency) rather than abandoned/speculative config — removing them would be a regression in configurability, not just cleanup.

### 5. `workflows/default.json`'s `stages[].description`

**Option A (recommended): remove.** It's pure documentation metadata with no runtime role; if stage documentation is wanted, put it in `docs/` instead of a field that looks executable.
**Option B:** surface it in `/workflow status`-style CLI output or logs if per-stage human-readable description is actually wanted at runtime.

### 6. `workflows/default.json`'s `stages[].retryable`

**Option A: remove**, since `workflow_engine.py` already hardcodes retry-eligibility to the `"execute"` stage only.
**Option B (recommended if per-stage retry control is intended): wire it in.** In `workflow_engine.py::_run_execute_with_retry` (or its generalized equivalent), gate retry behavior on `stage_def.retryable` rather than a hardcoded stage-name check, so `plan`/`verify` could also become retryable via config without a code change.

### 7. `workflows/default.json`'s `retry_policy.backoff`

**Option A (recommended, given only one strategy exists today): remove**, since `_SUPPORTED_BACKOFF = {"fixed"}` means there's exactly one value this key can ever have — validating a single-valued enum client-side and then never using it is pure overhead. Keep `backoff_sec` (which does the real work).
**Option B:** if a second backoff strategy (e.g. exponential) is planned, keep the field and implement the branch in `workflow_engine.py` now rather than leaving it validated-but-inert.

## Required tests

- For every key resolved via removal (Option A anywhere above): confirm no test references the removed field/key (`rg` the field/key name across `tests/`); update or delete any test that does.
- For every key resolved via wiring (Option B anywhere above): add a test that changes the config value and asserts the resulting behavior differs (e.g. an `exclude_globs` value causes a matching file to be skipped during indexing; a non-default `search_timeout_sec` causes a slow search to raise/timeout; a `retryable: false` stage does not retry on failure).
- Re-run `tests/test_check_mcp_docs_consistency.py`/`tools/check_mcp_docs_consistency.py` and the full suite after any `config/agent.toml`/`config_dataclasses.py`/`config_builders.py` change (per `rules/toolchain.md`'s standard validation sequence).

## Acceptance criteria

- No key remains in config that is parsed into a Python value with zero downstream read-sites, and no key remains in any `*.toml`/`*.json` under `config/` without a corresponding parse site in the code that owns it.
- Every decision (A or B) made above is reflected consistently across the config file, the Python parsing code, and any doc that documents that config key.
- `config/mdq_mcp_server.toml`'s `status` key is either removed or given a documented purpose.

## Non-goals

- Do not implement a second MCP healthcheck transport in this requirement even if Option B is chosen for `healthcheck_mode` — that's a separate, larger feature; this requirement only decides whether the placeholder stays documented-as-pending or gets removed.
- Do not redesign the workflow engine's retry/backoff architecture beyond wiring the two named fields (`retryable`, `backoff`) — no new backoff strategies, no new stage types.
- Do not re-audit the `[[tool_definitions]]` block's ~300 individual JSON-schema property nodes in `config/agent.toml` — that data is forwarded wholesale as an LLM tool-calling contract, not read key-by-key by name, and was explicitly out of scope for the originating audit.
