## [High][eventbus][delivery] Count normal delivery failures instead of only counting DLQ requeues

### Summary
The current Event Bus design uses `retry_count` in a way that does not represent normal delivery failure.

### Problem
`retry_count` is currently incremented only when an event is requeued from the DLQ.
It does not increase when normal consumer delivery fails or times out.
This makes the retry counter unsuitable as the source of truth for retry and DLQ promotion decisions.

### Scope
- app.py
- dlq.py
- schema.sql
- delivery failure handling
- retry accounting semantics

### Task
- introduce a delivery-failure accounting path for normal consumer failures
- increment `retry_count` or a new `delivery_failure_count` on normal delivery failure and timeout
- stop using DLQ requeue count as the only retry signal
- make retry accounting reflect actual delivery failures
- add tests for normal delivery failure counting

### Expected Result
Retry accounting reflects real delivery failures and can be used for reliable retry and DLQ decisions.

### Acceptance Criteria
- normal consumer failure increments the failure counter
- delivery timeout increments the failure counter
- DLQ requeue is no longer the only source of retry increments
- tests verify the new retry accounting behavior
