# Clarify MDQ authorization behavior and exclude empty `auth_token` from compatibility cleanup

## Priority

Low

## Target files

- `mcp_servers/mdq/server.py`
- `config/mdq_mcp_server.toml`
- `docs/04_mcp_*.md`
- `docs/security*.md`

## Type

Documentation / scope clarification

## Background

`mdq-mcp` attaches auth middleware with an empty token. The code comment states that MDQ has its own authorization through `allowed_dirs`.

The current config also states that `allowed_dirs = []` means deny-all by default.

This behavior is not identified as deprecated or backward compatibility behavior.

Therefore, it should not be removed as part of backward compatibility cleanup.

## Problem

There is a risk that empty `auth_token` behavior may be mistaken for deprecated compatibility behavior and removed together with unrelated cleanup.

That would be a security model change, not a simple compatibility cleanup.

## Intent

Document the current MDQ authorization model clearly and keep it separate from compatibility-removal work.

## Tasks

- [ ] Document that MDQ path authorization is based on `allowed_dirs`
- [ ] Document that empty `auth_token` is current MDQ behavior
- [ ] Document that `allowed_dirs = []` is deny-all
- [ ] Confirm security audit behavior for MDQ HTTP auth
- [ ] Ensure empty `auth_token` is not grouped with removed config keys
- [ ] Add a note that changing MDQ HTTP auth requires a separate security design issue

## Acceptance criteria

- MDQ authorization behavior is documented
- Empty `auth_token` is not treated as deprecated in cleanup docs
- `allowed_dirs` is documented as the authorization boundary
- Any future auth-token change is tracked separately from compatibility cleanup

## Non-goals

- Do not change MDQ authentication behavior
- Do not replace `allowed_dirs` authorization
- Do not introduce a new auth scheme in this issue
