# [Medium] Strengthen workflow schema validation, migration tracking, and DB config error messages

## Background

The workflow schema includes:

- `tasks`
- `attempts`
- `processed_events`
- `artifacts`
- `approvals`
- `workflow_schema_version`

`create_workflow_schema()` creates the workflow schema, applies migrations, and records `WORKFLOW_SCHEMA_VERSION`.

Startup schema preflight validates required workflow tables and then reads `workflow_schema_version`, but `workflow_schema_version` is not listed in the explicit required-table map.

Workflow migrations are currently represented as a list of SQL statements. Migration execution catches `sqlite3.OperationalError` and assumes the column already exists, which can hide unexpected errors.

`SQLiteHelper` also contains a stale error message that references `common.toml`, even though current documentation points to current DB config / `agent.toml`.

## Goal

Improve workflow DB validation, migration safety, and operator-facing DB error messages.

## Scope

- `agent/repl_health.py`
- `db/schema_sql.py`
- `db/create_schema.py`
- `db/helper.py`
