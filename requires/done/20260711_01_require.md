# GitHub MCP Server Startup Failure — Root Cause Analysis and Fix Proposal

## 1. Objective

Resolve the GitHub MCP server startup failure caused by circular import dependency.

The GitHub MCP server exits immediately upon startup due to a circular import between `server.py` and `server_common.py`. This causes the server process to terminate before it can register any tools or respond to health checks, resulting in repeated watchdog restart attempts until max restarts (3) is reached.

---

## 2. Current-State Assessment

### 2.1 Symptom

- HTTP subprocess MCP server 'github' exits early
- Initial startup failures: 4 times
- Watchdog recovery attempts: 51 times (after reaching max restarts = 3, operator action required)

### 2.2 Error Log

```
Traceback (most recent call last):
  File "/opt/llm/scripts/mcp_servers/github/server.py", line 50, in <module>
    from mcp_servers.github.server_file import router as file_router
  File "/opt/llm/scripts/mcp_servers/github/server_file.py", line 22, in <module>
    from mcp_servers.github.server_common import _get_service, _info
  File "/opt/llm/scripts/mcp_servers/github/server_common.py", line 12, in <module>
    from mcp_servers.github.server import _service, logger  # noqa: PLC0415
ImportError: cannot import name 'router' from 'mcp_servers.github.server_file'
```

### 2.3 Root Cause

Circular import dependency chain:

```
server.py ──→ server_file.py ──→ server_common.py ──→ server.py (circular!)
```

1. `server.py` imports `router` from `server_file.py` at module level (line 50)
2. `server_file.py` imports `_get_service` and `_info` from `server_common.py` at module level (line 22)
3. `server_common.py` imports `_service` and `logger` from `server.py` at module level (line 12)

This creates a cycle: when Python tries to resolve `server.py` → `server_file.py` → `server_common.py` → back to `server.py`, the `server.py` module has not finished executing yet, so `_service` and `logger` are not available.

### 2.4 Affected Modules

All four server files under `scripts/mcp_servers/github/` are affected because they all depend on `server_common.py`:

| Module | Depends on `server_common.py` |
|--------|-------------------------------|
| `server_file.py` | Line 22: `_get_service, _info` |
| `server_issues.py` | Line 24: `_get_service, _info` |
| `server_pull_requests.py` | Line 26: `_get_service, _info` |
| `server_repository.py` | Line 26: `_get_service, _info` |

Each of these modules is imported by `server.py` at module level (lines 50–53), so the circular dependency is triggered regardless of which sub-router is loaded first.

### 2.5 Why PyGithub Is Not the Problem

PyGithub is installed correctly:

```bash
uv run --directory /opt/llm python -c "from github import Github; print('PyGithub OK')"
# Output: PyGithub OK
```

The error occurs before PyGithub is even referenced, during the import phase.

### 2.6 GITHUB_TOKEN Status

`GITHUB_TOKEN` is not set in the environment. While this would cause API rate limiting (60 requests/hour without authentication), it does not prevent the server from starting. The server fails before any GitHub API calls are made.

## 3. Improvement Principles

### 3.1 Core Principles

- **Eliminate the circular import at its source.** Do not work around it with lazy imports or try/except blocks.
- **Minimize changes to existing code.** The fix should be localized to the dependency boundary between `server_common.py` and `server.py`.
- **Preserve the current design intent.** `server_common.py` provides shared FastAPI route helpers (`_get_service()`, `_info()`). These should remain accessible to all server_* modules.

---

## 4. Target Architecture

### 4.1 Desired Dependency Graph

After the fix, the dependency graph should be acyclic:

```
server.py
├── server_file.py     → server_common.py
├── server_issues.py   → server_common.py
├── server_pull_requests.py → server_common.py
└── server_repository.py → server_common.py

server_common.py
└── (no import of server.py)
```

### 4.2 Resolution Strategy

Remove the direct import of `_service` and `logger` from `server_common.py`. Instead, use one of the following approaches:

#### Option A: Pass dependencies explicitly (Recommended)

Change `_get_service()` to accept the service instance as a parameter instead of importing it:

```python
# server_common.py
def _get_service(service: Any) -> GitHubService:
    """Dependency that returns the singleton GitHubService instance."""
    return service

# server_file.py
@router.post("/get_file_contents")
async def get_file_contents(
    req: GetFileContentsRequest,
    gh_service: GitHubService = Depends(lambda: _get_service(_service)),
):
    ...
```

This approach requires modifying every endpoint function signature across all four server_* modules, which is more invasive.

#### Option B: Lazy import inside the function (Minimal change)

Move the import of `_service` and `logger` inside the functions that need them:

```python
# server_common.py
def _get_service() -> GitHubService:
    from mcp_servers.github.server import _service  # noqa: PLC0415
    return _service

def _info(msg: str, **kwargs: Any) -> None:
    from mcp_servers.github.server import logger  # noqa: PLC0415
    from shared.formatters import fmt_kvlog  # noqa: PLC0415
    logger.info(fmt_kvlog(msg, **kwargs))
```

This is the minimal-change solution. The import happens at runtime (when the function is called), not at module load time, so the circular dependency is avoided.

#### Option C: Redesign `server_common.py` as a standalone module

Extract `_service` and `logger` into a separate module that both `server.py` and `server_common.py` import:

```
server.py          ← imports logger, _service
server_common.py   ← imports logger, _service (no circular dependency)
server_file.py     ← imports _get_service, _info from server_common.py
```

This is the cleanest architectural solution but requires the most code changes.

---

## 5. Recommended Fix

### 5.1 Chosen Approach: Option B (Lazy Import)

Option B is recommended because:

1. It requires the fewest code changes (only `server_common.py`).
2. It preserves the existing public API of `server_common.py` — no changes needed in `server_file.py`, `server_issues.py`, `server_pull_requests.py`, or `server_repository.py`.
3. The performance impact is negligible — the import only happens once per request due to Python's module caching.
4. It follows the same pattern already used elsewhere in the codebase (see `server_common.py` line 23 for `fmt_kvlog`).

### 5.2 Implementation

Modify `scripts/mcp_servers/github/server_common.py`:

```python
#!/usr/bin/env python3
"""mcp_servers/github/server_common.py
Shared FastAPI route helpers for github-mcp server endpoints.

This module provides:
  _get_service() — FastAPI dependency returning the singleton GitHubService
  _info(msg, **kwargs) — Structured logging helper
"""

from typing import Any


def _get_service() -> GitHubService:
    """Dependency that returns the singleton GitHubService instance."""
    from mcp_servers.github.server import _service  # noqa: PLC0415
    return _service


def _info(msg: str, **kwargs: Any) -> None:
    """Log a structured info message with kv-log formatting."""
    from mcp_servers.github.server import logger  # noqa: PLC0415
    from shared.formatters import fmt_kvlog  # noqa: PLC0415

    logger.info(fmt_kvlog(msg, **kwargs))
```

## 6. Alternative Long-Term Improvements

### 6.1 Option C: Extract Shared State into a Separate Module

If the team prefers a cleaner architecture, consider extracting `_service` and `logger` into a dedicated module:

**New file: `scripts/mcp_servers/github/shared_state.py`**
```python
#!/usr/bin/env python3
"""Module-level singleton state for GitHub MCP server."""

import logging

logger = logging.getLogger(__name__)

_service: "GitHubService | None" = None


def set_service(svc: "GitHubService") -> None:
    global _service
    _service = svc


def get_service() -> "GitHubService":
    if _service is None:
        raise RuntimeError("GitHubService not initialized")
    return get_service
```

Then update `server.py` to register the service:
```python
from .shared_state import set_service, logger
_service = build_service(cfg)
set_service(_service)
```

And update `server_common.py` to read from the shared module:
```python
from .shared_state import get_service, logger

def _get_service() -> GitHubService:
    return get_service()
```

This eliminates the circular dependency while keeping the code structure clean and explicit about shared mutable state.

### 6.2 Preventive Measure: Add Circular Import Detection to CI

Add a pre-commit hook or CI check to detect circular imports before they reach production. Tools like `pylint` with `--disable=I0011` can help, but a custom script using `ast` to detect cycles in the import graph would be more reliable.

## 8. Risks and Mitigations

### Risk 1: Lazy import masks other issues

**Mitigation**: If the lazy import still raises an error, it means there is another circular dependency deeper in the call stack. In that case, switch to Option C (shared_state.py).

### Risk 2: Performance degradation from repeated imports

**Mitigation**: Python caches imported modules after the first import. Subsequent imports within the same process are essentially free — just a dictionary lookup. No measurable performance impact.

### Risk 3: `_service` could be accessed before initialization

**Mitigation**: This risk exists today as well. If `_service` is accessed before `build_service()` runs, it will be `None` and cause an AttributeError. This is a pre-existing issue unrelated to the circular import fix. Consider adding a guard in `_get_service()`:

```python
def _get_service() -> GitHubService:
    from mcp_servers.github.server import _service  # noqa: PLC0415
    if _service is None:
        raise RuntimeError("GitHubService not initialized — server not started properly")
    return _service
```

---

## 9. Conclusion

The GitHub MCP server startup failure is caused by a straightforward circular import between `server.py` and `server_common.py`. The minimal fix is to move the problematic imports in `server_common.py` inside the functions that use them, converting them from module-level imports to lazy imports. This resolves the immediate issue with minimal code changes. For a longer-term improvement, extracting shared state into a dedicated `shared_state.py` module would provide a cleaner architecture and make the dependency graph explicitly acyclic.
