# MCP Tool Discovery Timeout Parameter Inconsistency

## Priority

Low-Medium

## Target Files

- `scripts/agent/services/mcp_tool_discovery.py` (line 123)
- `scripts/agent/http_lifecycle.py` (line 249)

## Background

`McpToolDiscovery._fetch_server_tools()` uses `timeout=5.0` directly when calling `http.get()`, while `McpServerLifecycle.start_server()` uses `httpx.Timeout(timeout=5.0)` for its health check client. Both should use the same timeout parameter format for consistency.

## Problem

In `_fetch_server_tools()` at line 123:

```python
resp = await self._ctx.services_required.http.get(
    f"{cfg.url}/v1/tools", timeout=5.0
)
```

The `timeout` parameter accepts either a float (seconds) or an `httpx.Timeout` object. While both work functionally, the semantics differ:
- `timeout=5.0`: applies 5 seconds to all operations (connect, read, write, idle)
- `httpx.Timeout(connect=5.0, read=5.0)`: explicitly sets connect and read timeouts (but requires all four parameters per httpx API)

Additionally, `http_lifecycle.py` line 249 has a related issue: it uses `httpx.Timeout(connect=5.0, read=5.0)` which violates httpx's requirement that all four parameters must be provided together. This raises `ValueError`.

## Reason for Change

- Code consistency across modules
- The `httpx.Timeout(connect=5.0, read=5.0)` pattern in `http_lifecycle.py` is actually broken per httpx API requirements
- Using explicit `httpx.Timeout(timeout=X)` makes intent clearer and avoids accidental misuse

## Implementation Intent

Standardize on `httpx.Timeout(timeout=X)` for all HTTP timeout usage in the codebase. Fix the broken `connect`/`read` only pattern in `http_lifecycle.py`.

## Implementation Instructions

Fix `mcp_tool_discovery.py` line 123:

```python
resp = await self._ctx.services_required.http.get(
    f"{cfg.url}/v1/tools", timeout=httpx.Timeout(timeout=5.0)
)
```

Fix `http_lifecycle.py` line 249:

```python
async with httpx.AsyncClient(timeout=httpx.Timeout(timeout=5.0)) as client:
```

Key points:
- Use `httpx.Timeout(timeout=X)` instead of bare float or incomplete parameter set
- Ensure all timeout values are consistent across the codebase
- Consider adding a constant for the common timeout value (e.g., `MCPSERVER_HEALTH_TIMEOUT = 5.0`)

## Acceptance Criteria

1. All HTTP timeout usage in the codebase uses `httpx.Timeout(timeout=X)` format
2. No `ValueError` from incomplete `httpx.Timeout` parameter sets
3. Timeout values are consistent across MCP server startup and tool discovery

## Tests

- Unit test: verify `httpx.Timeout` object is correctly passed to `http.get()` calls
- Integration test: start MCP server with slow `/v1/tools` endpoint, verify timeout behavior matches expectations
