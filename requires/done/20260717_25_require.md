# [Low] Verify registry, safety tier, and documentation consistency for search_web

## Priority

Low

## Summary

Verify and clean up consistency across tool registry, tool safety tiers, `/v1/tools`, and documentation for `search_web`.

## Target files

- `scripts/shared/tool_constants.py`
- `config/agent.toml`
- `tests/test_tool_server_layer_consistency.py`
- `tests/test_tool_safety_tiers.py`
- `tests/test_web_search_tool_schema.py`
- `docs/04_mcp_03_*.md`
- `docs/04_mcp_04_01_web-search-file-read-github.md`
- `docs/04_mcp_06_*.md`

## Reason for change

`search_web` is registered in `WEB_SEARCH_TOOLS`.

However, comments and documentation should consistently state the correct routing model:

- ToolRegistry is the routing authority.
- Live `/v1/tools` discovery is for drift validation only.
- `tools.py::TOOL_LIST` is the server-side LLM-facing tool schema.
- `config/agent.toml` must expose matching tool definitions and safety tier entries.

Ambiguous comments can lead to incorrect future changes.

## Intent

Make routing ownership and validation responsibilities unambiguous.

Ensure search_web remains consistent across:

- tool constants
- ToolRegistry
- `/v1/tools`
- `tools.py`
- agent tool definitions
- safety tiers
- documentation
- tests

## Required changes

### 1. Update comments in tool_constants.py

Clarify that routing is driven by ToolRegistry only.

Live `/v1/tools` discovery should be described as drift validation only.

### 2. Verify search_web consistency

Confirm:

- `WEB_SEARCH_TOOLS` contains `search_web`
- ToolRegistry maps `search_web` to `web_search`
- `/v1/tools` exposes `search_web`
- `tools.py::TOOL_LIST` contains `search_web`
- `config/agent.toml` has matching tool definition
- `config/agent.toml` has a safety tier for `search_web`

### 3. Add or update consistency tests

Tests should fail when:

- `search_web` exists in `TOOL_LIST` but not in `WEB_SEARCH_TOOLS`
- `search_web` exists in registry but not in `/v1/tools`
- `search_web` lacks a safety tier
- `search_web` schema diverges from runtime request schema

### 4. Update documentation

Update docs to match final behavior after Issues 1 to 4:

- config validation
- schema constraints
- zero-result behavior
- timeout behavior
- audit behavior
- health behavior
- error classification

## Acceptance criteria

- Comments match canonical routing model.
- `search_web` registration is consistent across layers.
- `search_web` has a safety tier.
- Tests detect registry/schema/safety drift.
- Docs describe actual behavior after implementation.
- No runtime behavior changes unless an actual inconsistency is found.
