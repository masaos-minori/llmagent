## Summary
Make `project`, `repo`, and `branch` real retrieval and injection boundaries so Persistent Semantic Memory does not leak unrelated knowledge across projects or branches. The current memory model already stores these fields, but branch-aware retrieval policy is not strong enough yet.

## Background
`MemoryEntry` already includes `project`, `repo`, and `branch`, and retrieval APIs already accept `project` and `repo` values. However, branch-aware filtering and policy are not clearly established as first-class behavior.

## Problem
If project, repo, and branch boundaries are weak, unrelated rules, past failures, or decisions can be injected into the wrong session context.

## Required Changes
- Add or formalize branch-aware filtering in retrieval and injection flows.
- Define how current project / repo / branch context is resolved.
- Distinguish global rule memory from repo/branch-specific episodic memory in ranking.
- Add tests proving unrelated repo/branch memory is not injected.

## Acceptance Criteria
- Retrieval and injection can explain how project/repo/branch filtering works.
- Unrelated project memory is less likely to leak into the current task.
- Docs describe context-boundary behavior clearly.

## Out of Scope
- Full git integration redesign
- Replacing memory metadata structure

## AI Implementation Instruction
Implement this as a **memory-boundary control task**.

Strict rules:
- Do not leave `branch` as a stored-but-unused field.
- Do not rank global rules and local episodic memories identically by default.
- Do not invent a hidden context source; define it explicitly.
- If the current project/repo/branch source is unclear, mark it as: `Needs confirmation`
