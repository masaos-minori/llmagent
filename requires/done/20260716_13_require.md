## Background

The Agent documentation describes `ToolExecutor` as the component responsible for MCP routing and tool execution. It also describes startup service checks that include readiness, tool definition validation, and routing drift validation.

The same documentation states that tool execution may run calls in parallel, except when side-effecting tools are present. It also states that workflow approval state can block new turns.

These concerns are tightly coupled. A tool must not only be visible to the model; it must also be routable, dispatchable, correctly classified, safe to execute, and covered by approval rules where needed.

## Problem

The current architecture has multiple potential sources of truth for tools:

- MCP server tool schemas
- Runtime dispatch tables
- ToolRegistry ownership
- Static tool constants
- Configured tool names
- Side-effect detection
- Approval policy
- Startup drift validation

If these layers drift, a tool can appear in one place but be absent or misclassified elsewhere. For example, a tool may be exposed to the LLM but not dispatchable, registered as owned by a server but missing from runtime dispatch, or treated as safe for parallel execution even though it performs write/admin actions.

This is not only a documentation issue. It can directly affect routing correctness, approval enforcement, execution ordering, and auditability.

## Reason for change

MCP tools are executable capabilities. The agent must guarantee that a tool visible to the model is actually callable and that its safety properties are enforced consistently.

Without this consolidation, AI implementation agents and runtime logic may rely on stale or partial definitions. That can cause runtime failures, unsafe execution ordering, missing approvals, or misleading startup validation.

This issue should establish a reliable MCP tool contract across all layers.

## Design intent

Define a single consistent lifecycle for each MCP tool:

- It is declared by the server schema.
- It is dispatchable by the server runtime.
- It is intentionally owned by a ToolRegistry entry.
- It has side-effect metadata if it can mutate state.
- It has approval policy if needed.
- It is included in startup validation.
- It is covered by tests.

Tool classification should be shared or cross-validated so that side-effect detection, serialization, approval requirement, and audit enrichment do not diverge.

## Implementation instructions

### Tool surface and routing consistency

- Enumerate all MCP servers and exported tool schemas.
- Compare exported tool names with runtime dispatch handlers for each server.
- Compare exported tool names with ToolRegistry ownership and static tool constants.
- Compare configured tool names with live discovery results.
- Improve routing drift validation so it reports server key, missing tool names, and unexpected tool names.
- Decide which drift cases must be fatal at startup.
- Add CI tests that fail when tool names drift across schema, config, registry, and dispatch layers.

### Side-effect and approval consistency

- Identify the current source of truth for write, delete, shell, and admin tool classification.
- Compare side-effect detection with approval policy.
- Compare approval policy with static tool constants or registry metadata.
- Ensure that a batch containing any side-effecting tool is serialized.
- Ensure approval-required tools cannot execute without approval.
- Ensure approval-pending workflow state blocks new user turns.
- Add audit fields where applicable: tool name, server key, side-effect flag, approval id, and outcome.

## Acceptance criteria

- Every MCP tool exposed by a server is dispatchable.
- Every dispatchable tool is intentionally represented in ToolRegistry ownership.
- Every write/admin/side-effecting tool has explicit classification.
- Side-effecting tool calls are serialized when mixed with other calls.
- Approval-required tools cannot run before approval.
- Startup validation detects tool surface drift and reports actionable details.
- CI fails on future drift across schema, registry, config, approval, and dispatch layers.
- Audit logs contain enough context to trace tool execution and approval outcomes.

## Non-goals

- Do not add new MCP tools in this issue.
- Do not redesign the entire approval UI.
- Do not modify individual MCP tool business behavior unless required to fix classification or dispatch drift.
- Do not change non-MCP command execution behavior.
