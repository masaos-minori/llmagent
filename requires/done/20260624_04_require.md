# Replace `RagPipeline.run()` tuple output with a typed result object

## Summary
Refactor `RagPipeline.run()` so that it returns a typed result object instead of a position-dependent tuple.

## Background
The current API returns a tuple whose meaning depends on index order. At the same time, stage execution mutates a shared `PipelineContext` in place. This reduces readability and makes future changes harder.

## Objective
Introduce a typed result object for pipeline execution and reduce reliance on positional tuple contracts.

## In Scope
- `scripts/rag/pipeline.py`
- `scripts/rag/stage.py`
- `scripts/rag/types.py`
- Related documentation:
  - `03_rag_03_query_pipeline.md`
  - `03_rag_04_data_model_and_interfaces.md`

## Required Changes
- Define a new result type, for example `PipelineRunResult`
- Include explicit named fields such as:
  - `queries`
  - `search_results`
  - `merged`
  - `reranked`
  - `stage_results`
  - `diagnostics`
  - `result_source`
- Update `RagPipeline.run()` to return the typed object
- Update all affected callers
- Update all documentation that describes the return type

## Acceptance Criteria
- No caller depends on tuple index positions from `RagPipeline.run()`
- The result object exposes named attributes for all previously returned data
- The documentation matches the final implementation contract
- Stage diagnostics remain accessible without reading mutable internal state directly

## Out of Scope
- Large behavior changes to ranking or retrieval logic
- Renaming external CLI commands

## Implementation Notes
Prefer an explicit dataclass or equivalent typed model. Keep backward compatibility only if it can be implemented cleanly and documented clearly.
