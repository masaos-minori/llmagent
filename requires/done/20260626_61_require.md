# Title
Unify `/health` HTTP status semantics to fail closed on degraded states

## Priority
High

## Labels
- bug
- eventbus
- healthcheck
- operations
- monitoring
- fail-closed

## Dependencies
- None

## Summary
Unify the `/health` endpoint behavior so it returns HTTP 200 only for `status="ok"` and HTTP 503 for `status="degraded"` or `status="unhealthy"`.

## Background
The current Event Bus documentation is inconsistent. One document states that `/health` always returns HTTP 200, while operations and known-issues documents state that degraded and unhealthy states must return HTTP 503.

## Problem
Monitoring tools may incorrectly treat a degraded Event Bus as healthy if they rely on HTTP status codes. This breaks fail-closed operational monitoring.

## Required Changes
- Verify the runtime behavior of `GET /health`.
- Return HTTP 200 only when all components are healthy.
- Return HTTP 503 when:
  - DB is unavailable
  - DLQ task is stopped
  - broker queue backlog is high
  - slow consumers are detected
  - any critical component is degraded or unhealthy
- Remove or correct documentation that says `/health` always returns HTTP 200.
- Update tests for all health states.

## Acceptance Criteria
- `GET /health` returns HTTP 200 only for `status="ok"`.
- `GET /health` returns HTTP 503 for degraded or unhealthy states.
- Monitoring guidance says to alert on HTTP status, not only JSON body.
- `06_eventbus_02_http_api_and_runtime.md`, `06_eventbus_05_configuration_deploy_and_operations.md`, and `06_eventbus_90_inconsistencies_and_known_issues.md` are consistent.

## Out of Scope
- Adding authentication to `/health`.
- Changing the health response JSON schema unless required for consistency.
