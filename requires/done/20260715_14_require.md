High

## Background

GitHub write operations are controlled by repository allowlists and GitOps safety flags. `gitops_push_blocked` is currently wired into approval, but other GitOps fields have unclear runtime behavior. GitHub write tool sets also appear to be duplicated.

## Problem

Security-sensitive policy must be explicit, consistent, and fail-closed.

Potential issues include:

- Repository allowlist checks must compare the target `owner/repo` against the configured allowlist.
- An empty allowlist must deny GitHub write tools.
- GitHub write tool sets must not drift across modules.
- `gitops_force_push_blocked` and `gitops_protected_branches` must either be wired into enforcement logic or removed from the config layer.

## Implementation Tasks

### Repository allowlist

- [ ] Ensure `check_allowed_repo()` compares `f"{owner}/{repo}"` against `approval_github_allowed_repos`.
- [ ] Reject missing or invalid `owner` / `repo`.
- [ ] Preserve fail-closed behavior when the allowlist is empty.

### Tool set consolidation

- [ ] Create or reuse a shared GitHub write tool constant.
- [ ] Use the shared constant in policy and approval modules.
- [ ] Decide whether issue/comment creation tools are GitHub write tools.

### GitOps approval behavior

- [ ] Ensure `gitops_push_blocked` blocks the full intended GitHub write tool set.
- [ ] If `gitops_force_push_blocked` is retained by the config issue, implement deny behavior for force push or `force=True`.
- [ ] If `gitops_protected_branches` is retained by the config issue, connect it to branch protection logic.
- [ ] Add audit decisions for GitOps denials.

### Workflow approval skip

- [ ] Review `skip_in_workflow_mode`.
- [ ] Confirm whether per-tool approval may be skipped under workflow-level approval.
- [ ] If this conflicts with the canonical approval model, remove or revise the skip path.

## Acceptance Criteria

- [ ] GitHub repository allowlist enforcement is correct and fail-closed.
- [ ] The GitHub write tool set is shared and consistent.
- [ ] No retained GitOps security field lacks runtime behavior.
- [ ] Per-tool approval is not accidentally bypassed.
- [ ] Denials are auditable.
