# Title
Make `retry_count` meaningful so DLQ promotion actually works

## Summary
Define and implement the official retry-count update path so the condition `retry_count >= max_retry` leads to real DLQ promotion in normal operation.

## Background
The DLQ promotion logic selects rows from `events` where `retry_count >= max_retry` and `dlq_at IS NULL`.
However, the current implementation does not increment `retry_count` during publish flow, and the implementation plan explicitly lists this as an unresolved discrepancy.

## Problem
As implemented today, events do not naturally reach DLQ through retry exhaustion, because the retry counter does not have an official update path.

## Required Changes
- Define where retry ownership lives: producer-side, consumer-side, requeue-side, or another explicit path.
- Implement a supported mechanism that updates `events.retry_count`.
- Clarify how `/dlq/{event_id}/requeue` interacts with retry counting after requeue resets `retry_count` to `0`.
- Add or update tests that prove “retry exhausted → DLQ promotion” works end to end.

## Acceptance Criteria
- `retry_count` is updated through an explicit, documented path.
- Events promoted to DLQ via retry exhaustion can be reproduced in tests.
- Documentation explains the relationship between retry, requeue, and DLQ promotion.

## Out of Scope
- Replacing DLQ storage format
- Replacing SQLite persistence
