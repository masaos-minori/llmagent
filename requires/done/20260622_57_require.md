# Title
Consolidate MCP tool routing and tool-definition ownership

## Summary
Reduce routing and definition drift by clarifying the single source of truth for tool ownership and route resolution.

## Background
Tool routing and tool exposure currently depend on multiple overlapping declarations:
- config-driven `tool_names`
- static fallback sets
- live server tool lists
- separate tool-definition config for LLM exposure

## Problem
This increases maintenance complexity and makes new-tool changes prone to drift.

## Required Changes
- Define a clear source-of-truth model for tool ownership and routing.
- Minimize duplicated declarations where possible.
- Add validation to detect drift between:
  - server implementation
  - route mapping
  - LLM-facing tool definitions
- Update docs to explain the intended ownership model.

## Acceptance Criteria
- Adding or changing a tool requires fewer duplicated edits.
- Drift between implementation and configuration is easier to detect automatically.
- The ownership model is explicit and documented.

## Out of Scope
- Removing all static fallback logic if still required for compatibility
- Rewriting all MCP servers

## AI Implementation Instruction
Implement this as a consistency and maintainability task.

Strict rules:
- Do not silently change routing behavior without documenting ownership changes.
- Prefer explicitness over hidden fallback magic.
- Keep validation practical and enforceable.
- If there are unavoidable multiple ownership points, document them clearly and mark unresolved ambiguity as: Needs confirmation

Deliverables:
1. Clear ownership model
2. Drift-detection validation
3. Updated documentation
