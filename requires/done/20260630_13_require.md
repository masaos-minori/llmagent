# [High] Remove config `tool_names` as a routing input

## Priority
High

## Background
`tool_names` in MCP server config is intended as drift validation metadata, but it is still described and partially tested as a routing fallback. This keeps an old config-driven routing path alive.

## Goal
Make `tool_names` validation-only and remove it from tool resolution.

## Scope
Affected files:
- `shared/route_resolver.py`
- `shared/mcp_config.py`
- `shared/tool_registry.py`
- `tests/test_route_resolver.py`
- `docs/04_mcp_03_routing_lifecycle_and_execution.md`
- `docs/04_mcp_06_configuration_and_operations.md`

## Implementation Tasks
- Ensure `ToolRouteResolver.resolve()` does not read `McpServerConfig.tool_names`.
- Keep `tool_names` only for drift validation.
- Keep `validate_routing_against_config()` if useful, but make it independent from routing.
- Update tests so that tools listed only in config `tool_names` do not route.
- Remove `Config tool_names` as a routing priority from docs.
- Clarify that missing or incomplete `tool_names` does not affect routing.

## Acceptance Criteria
- Changing `tool_names` does not change routing behavior.
- Tools not present in discovery or `ToolRegistry` fail to resolve.
- `tool_names` remains available for drift checks only.
- Documentation describes `tool_names` as validation metadata only.

## Validation
- Run `pytest tests/test_route_resolver.py -v`.
- Add or update a test proving that a config-only custom tool does not route.
- Run `rg -n "tool_names.*fallback|Config tool_names.*routing|last-resort fallback" shared docs tests`.
