# Fix ToolExecutor stampede protection exception handling

## Priority

High

## Goal

Keep stampede protection in `ToolExecutor`, but fix exception handling so concurrent waiters never hang.

## Background

`_execute_with_stampede_protection()` owns in-flight execution sharing for identical cache keys.

The current implementation creates an in-flight future and sets the result when `_raw_execute()` succeeds. However, if `_raw_execute()` raises an exception, waiting callers may await a future that is never resolved.

## Scope

Affected files:

- `shared/tool_executor.py`
- stampede protection tests

## Implementation requirements

Keep `_execute_with_stampede_protection()` in `ToolExecutor`.

Do not move `_inflight` into `ToolResultCache`.

Update exception handling so that `_raw_execute()` exceptions are propagated to all waiters:

    try:
        result = await self._raw_execute(tool_name, args)
    except Exception as e:
        if not inflight.done():
            inflight.set_exception(e)
        raise
    else:
        if not inflight.done():
            inflight.set_result(result)
        return result
    finally:
        self._inflight.pop(cache_key, None)

## Acceptance criteria

- Same `cache_key` concurrent misses call `_raw_execute()` only once.
- On success, all callers receive the same `ToolCallResult`.
- On exception, all callers receive the same exception.
- No concurrent caller hangs on `_raw_execute()` exception.
- `_inflight` entry is cleaned up on success and on exception.
- Failed `ToolCallResult` and exceptions are not cached.

## Validation plan

- Add `tests/shared/test_tool_executor_stampede.py`.
- Test concurrent success path.
- Test concurrent exception path.
- Test `_inflight` cleanup after success.
- Test `_inflight` cleanup after exception.
- Test failed `ToolCallResult` is not cached.

## Out of scope

- Moving stampede protection to `ToolResultCache`.
- Changing cache key format.
