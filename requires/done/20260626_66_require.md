Redefine consumer ID stability for reliable offset resume

## Summary
Clarify and fix consumer ID stability so offset resume behavior works as documented across consumer restarts.

## Background
Docs describe consumer IDs as `sha256(name:hostname:pid)[:16]` and also say the same process gets the same ID. Because this includes `pid`, a restarted process may get a different ID and fail to resume from the intended offset.

## Problem
The current design may only be stable within a process lifetime, not across restarts. This conflicts with the desired reconnect/recovery behavior for durable consumers.

## Required Changes
- Decide the canonical consumer identity model:
  - client-supplied stable `consumer_id`, or
  - server-generated process-lifetime ID, or
  - server-generated restart-stable ID without PID.
- If durable resume is required, remove PID from stable ID generation or require explicit client-provided IDs.
- Define collision detection behavior.
- Update docs to distinguish:
  - process-lifetime stability
  - restart-stable durability
- Add tests for restart/resume behavior.

## Acceptance Criteria
- Consumer ID stability is accurately documented.
- Durable consumers have a clear way to resume offsets after restart.
- Offset file names remain deterministic for stable consumers.
- Collision behavior is tested or documented.

## Out of Scope
- Multi-node consumer group coordination.
- Exactly-once delivery.
