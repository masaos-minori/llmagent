# Add referential integrity for tool_results.session_id

## Summary
Align `tool_results` with session lifecycle by introducing explicit referential integrity between `tool_results.session_id` and `sessions.session_id`.

## Background
The `tool_results` table stores session-scoped tool artifacts, and `ToolResultStore.list_recent()` queries by `session_id`. However, unlike `messages` and `session_diagnostics`, `tool_results.session_id` is not defined as a foreign key with cascade delete.

This can leave orphaned rows after session deletion.

## Objective
Make `tool_results` lifecycle consistent with session lifecycle.

## In Scope
- `schema_sql.py`
- `tool_results.py`
- Migration path if schema evolution is already supported elsewhere

## Required Changes
- Add a foreign-key relationship from `tool_results.session_id` to `sessions(session_id)`
- Define the intended delete behavior explicitly
- Ensure store behavior remains valid when `session_id` is nullable
- Review any listing or retrieval code that assumes orphan rows are acceptable

## Acceptance Criteria
- Session-scoped tool results are no longer structurally detached from sessions
- Session deletion behavior for `tool_results` is deterministic
- Schema and store semantics match the intended session ownership model

## Out of Scope
- Redesigning ToolResultStore API
- Moving tool results into the messages table

## Implementation Notes
If nullable `session_id` must remain supported, document the exact meaning of “sessionless” tool results and ensure the FK policy remains valid for those rows.
