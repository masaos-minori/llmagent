# Title
Event Bus の導入: HTTP publish / SSE subscribe / append-only log / SQLite 永続化 / Replay / DLQ

## Summary

エージェントシステムに Event Bus レイヤーを追加し、コンポーネント間のイベント配信を分離・非同期化する。
HTTP publish + SSE subscribe をトランスポートとし、SQLite による永続化・シーケンスベース Replay・Dead Letter Queue (DLQ) を備えた at-least-once 配信基盤を構築する。

## Background

現在のシステムは、エージェントの各コンポーネント（REPL、ワークフローエンジン、MCP サーバー）が同一プロセス内または HTTP RPC で直接通信する。
これにより以下の課題が生じている。

- コンポーネント間の結合度が高く、ファンアウト通知（複数コンシューマーへのブロードキャスト）が困難。
- クラッシュ後のイベント再送手段がない。
- 障害発生時の診断に使えるイベント履歴が存在しない。
- 将来的な複数ノード構成への拡張が困難。

## Problem

1. **ファンアウト不可** — 既存の直接呼び出しモデルは 1:1 通信のみ。1:N ブロードキャストには個別実装が必要。
2. **耐障害性の欠如** — publish 後に consumer がダウンすれば、イベントは消失する。
3. **Replay なし** — 再起動後に missed イベントを取り戻す手段がない。
4. **DLQ なし** — 処理失敗イベントの診断・再投入ができない。
5. **ノード間同期なし** — 将来の多ノード構成を想定した replication 機構がない。

## Required Changes

### 1. Event Envelope スキーマの定義

- `event_id` (UUID) — 冪等性キー
- `topic` (string) — イベント種別 (例: `tool.executed`, `session.started`, `workflow.step.completed`)
- `payload` (JSON object) — イベント本体
- `producer` (string) — 発行元識別子
- `published_at` (ISO 8601 string) — 発行時刻
- `seq` (INTEGER, autoincrement) — Replay 用グローバルシーケンス番号
- `schema_version` (string) — エンベロープスキーマバージョン

ファイル: `schemas/event_envelope.json`

### 2. SQLite 永続化スキーマ (`events` テーブル)

```sql
CREATE TABLE IF NOT EXISTS events (
    seq          INTEGER PRIMARY KEY AUTOINCREMENT,
    event_id     TEXT    NOT NULL UNIQUE,
    topic        TEXT    NOT NULL,
    payload      TEXT    NOT NULL,  -- JSON
    producer     TEXT    NOT NULL,
    published_at TEXT    NOT NULL,
    acked_at     TEXT,              -- NULL = 未 ACK
    retry_count  INTEGER NOT NULL DEFAULT 0,
    dlq_at       TEXT               -- NULL = DLQ 未移送
);
```

ファイル: `eventbus/schema.sql`

### 3. append-only JSONL ログ

- `storage/events.jsonl` — publish された全イベントをシーケンス順に追記。
- SQLite が唯一の信頼できるストアであり、JSONL は障害診断・外部連携用の補助ログ。
- ローテーション方針は運用ドキュメントで定義する。

### 4. HTTP publish / SSE subscribe エンドポイント

`eventbus/app.py` に FastAPI アプリを実装する。

| エンドポイント | メソッド | 説明 |
|---|---|---|
| `/publish` | POST | `event_envelope.json` に従う JSON ボディを受信し `events` テーブルへ INSERT |
| `/subscribe` | GET | SSE ストリームを開く。クエリパラメータ `topic` (複数可)・`since_seq` (省略可) でフィルタ |
| `/replay` | GET | `since_seq` 以降の全イベントを返す (SSE or JSON array, クエリパラメータで選択) |
| `/health` | GET | サービス稼働確認 |

### 5. since_seq ベース Replay

- consumer は最後に処理した `seq` を `offsets/<consumer_id>` ファイルへ永続化する。
- 再接続時に `?since_seq=<last_seq>` を付与し、未受信イベントを再取得する。
- `offsets/` ディレクトリの管理方針を運用ドキュメントで定義する。

### 6. Dead Letter Queue (DLQ)

- `retry_count >= MAX_RETRY`（設定値、デフォルト 3）のイベントを DLQ に移送する。
- DLQ 移送: `events.dlq_at` に移送時刻を記録し、`deadletter/<event_id>.json` へ書き出す。
- `/dlq` エンドポイントまたは CLI ツールで DLQ イベントの一覧・再投入を提供する。
- `deadletter/` ディレクトリの管理方針を運用ドキュメントで定義する。

### 7. at-least-once 配信保証

- publish は SQLite INSERT が成功した時点で確定とする。
- consumer は `acked_at` を更新することで処理完了を通知する（オプション）。
- リース / リトライは `retry_count` と `dlq_at` で追跡する。
- stale イベントのクリーンアップポリシーを設定値で定義する。

### 8. event_id 冪等性

- `events.event_id` に UNIQUE 制約を付与。
- 同一 `event_id` の再 publish は `INSERT OR IGNORE` または HTTP 409 を返す。
- producer は UUID v4 を生成して `event_id` に使用する。

### 9. 各ホストへの `eventbus-node` 配置 (将来フェーズ)

- 各ホストにローカル `eventbus-node` を配置し、ノード間の replication を実装する。
- replication の詳細設計は Phase 2 以降とする（本要望書のスコープ外）。

## Acceptance Criteria

1. `POST /publish` でイベントを受信し、`events` テーブルと `storage/events.jsonl` に永続化できる。
2. `GET /subscribe?topic=<t>` で SSE ストリームを受信できる。
3. `GET /replay?since_seq=<n>` で指定シーケンス以降のイベントを取得できる。
4. 同一 `event_id` を再 publish しても重複 INSERT が起きない（冪等性）。
5. `retry_count >= MAX_RETRY` のイベントが `deadletter/` へ移送される。
6. `offsets/<consumer_id>` にシーケンスが永続化される。
7. `GET /health` が稼働中に 200 を返す。
8. 既存 agent・MCP レイヤーのコードを変更せず Event Bus を導入できる（新規レイヤーとして分離）。

## Out of Scope

- 既存 agent / MCP / RAG レイヤーへの Event Bus 統合（Phase 2 以降）
- ノード間 replication の詳細設計（Phase 2 以降）
- メッセージブローカー（Kafka、RabbitMQ 等）の導入
- 認証・認可機構（初期実装では信頼ネットワーク内運用を前提とする）

## Implementation Targets

| ファイル | 役割 |
|---|---|
| `eventbus/app.py` | FastAPI アプリ (publish / subscribe / replay / health エンドポイント) |
| `eventbus/schema.sql` | `events` テーブル DDL |
| `schemas/event_envelope.json` | イベントエンベロープの JSON Schema |
| `storage/events.jsonl` | append-only ログ（実行時生成） |

## AI Implementation Instruction

実装はフェーズ分けで進める。

**Phase 1: コア基盤**

1. `eventbus/schema.sql` — `events` テーブル DDL を実装する。`event_id` UNIQUE 制約、`seq` AUTOINCREMENT、WAL モードを含める。
2. `schemas/event_envelope.json` — JSON Schema を定義する。必須フィールド: `event_id`, `topic`, `payload`, `producer`, `published_at`。
3. `eventbus/app.py` — `POST /publish`・`GET /health` を先行実装する。publish 時に `events` テーブルと `storage/events.jsonl` へ同時書き込みを行う。冪等性は `INSERT OR IGNORE` で実装する。

**Phase 2: Replay と SSE**

4. `GET /subscribe` SSE エンドポイントを実装する。`topic` フィルタと `since_seq` クエリパラメータに対応する。
5. `GET /replay` エンドポイントを実装する。`since_seq` 以降のイベントを返す。
6. `offsets/<consumer_id>` への書き込みユーティリティを実装する。

**Phase 3: DLQ とリトライ**

7. DLQ 移送ロジックを実装する。`retry_count >= MAX_RETRY` の検出・`deadletter/` 書き出し・`events.dlq_at` 更新。
8. DLQ イベント一覧・再投入 CLI または `/dlq` エンドポイントを実装する。

Strict rules:
- 既存 `scripts/` 以下のコードには触れない。`eventbus/` を新規ディレクトリとして追加する。
- SQLite の WAL モードを必ず有効にする。
- `event_id` の冪等性は DB 制約で担保する（アプリ層のみの重複チェックは不可）。
- テストは各 Phase 完了後に `tests/test_eventbus_*.py` として追加する。
- `MAX_RETRY` はコード上の定数ではなく設定値（環境変数または `config/` 下の TOML）から読む。
- DLQ ファイル書き出しは atomic write（temp file + rename）で実装する。
- Needs confirmation: ノード間 replication の設計は Phase 1 実装前に別途確認する。
