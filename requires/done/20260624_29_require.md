# Unify partial-completion and diagnostic persistence into one canonical storage model

## Summary
Define one canonical persistence model for partial LLM completions and diagnostic artifacts.

## Background
The current Agent documentation describes multiple possible persistence paths for interrupted LLM output and diagnostic data, including:
- diagnostic messages in `messages`
- a dedicated diagnostics store / table
- `tool_result_store`
- `diagnostics.jsonl`

These descriptions are not fully consistent across the document set.

## Objective
Produce one coherent design for:
- where partial completions are stored
- where non-history diagnostics are stored
- which artifacts are restored into history
- which artifacts are operator-facing only
- which commands surface which artifacts

## In Scope
- `05_agent_03_turn-processing-flow.md`
- `05_agent_04_state-and-persistence.md`
- `05_agent_09_data-layer.md`
- `05_agent_10_operations-and-observability.md`
- `05_agent_17_partial-llm-completion-visibility.md`

## Required Changes
- Compare all current descriptions of:
  - partial LLM completion persistence
  - diagnostic persistence
  - `/tool`, `/stats`, `/context` visibility
  - session restore behavior
- Choose one canonical storage model
- Rewrite all affected sections so they reflect exactly the same model
- Explicitly separate:
  - conversation history
  - operator diagnostics
  - tool-result artifacts
  - session-end summaries
- Add a short artifact-ownership table

## Acceptance Criteria
- A reader can identify one canonical storage path for partial completions
- A reader can identify one canonical storage path for diagnostics
- The docs explain whether each artifact is:
  - restored into history
  - visible in `/tool`
  - visible in `/stats`
  - visible in `/context`
  - persisted in DB
  - persisted in JSONL
- No contradictory persistence description remains

## Out of Scope
- LLM streaming logic changes
- Tool-result schema redesign
- Database migration implementation

## Notes
Do not leave ambiguous wording such as “may be stored” unless the behavior is truly unresolved. If unresolved, mark it as `Unknown` explicitly.
