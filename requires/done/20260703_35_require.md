# Add ApprovalPolicy based on tool_constants classification

## Priority

Medium

## Goal

Centralize approval risk classification so approval behavior does not depend only on per-tool explicit configuration.

## Background

`tool_constants.py` defines canonical tool classifications such as read, write, delete, shell, Git, GitHub, RAG, MDQ, and other tool groups.

Approval policy should use these classifications after explicit config-based overrides.

## Scope

Affected files:

- `agent/tool_policy.py` new or existing file
- `agent/tool_approval.py`
- `shared/tool_constants.py`
- config/security handling
- tests for approval policy

## Classification priority

Use this priority order:

1. `approval_risk_rules`
2. `tool_safety_tiers`
3. `tool_constants.py` classification
4. default fallback

## Implementation requirements

Create or update `ApprovalPolicy`.

Responsibilities:

- Determine risk level.
- Apply explicit `approval_risk_rules`.
- Apply `tool_safety_tiers`.
- Use `WRITE_TOOLS`, `DELETE_TOOLS`, `SHELL_TOOLS`, Git/GitHub classifications.
- Keep prompt handling separate from risk classification.
- Keep tool-level approval behavior unchanged.

## Acceptance criteria

- Explicit `approval_risk_rules` override all classification defaults.
- `tool_safety_tiers` are applied after explicit risk rules.
- Write tools classify as at least medium risk unless explicitly overridden.
- Delete and shell tools classify as high risk unless explicitly overridden.
- Risk classification does not perform interactive prompting.
- Existing approval behavior is preserved.

## Validation plan

- Add tests for risk rule override.
- Add tests for safety tier fallback.
- Add tests for constants-based fallback.
- Add tests for unknown tool default behavior.
- Add tests proving prompt logic is separate.

## Out of scope

- Changing approval prompt UX.
- Changing workflow-level approval.
