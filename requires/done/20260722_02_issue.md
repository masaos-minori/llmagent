# Clarify LLM visibility, runtime routability, and availability metadata behavior

## Priority

High

## Scope

Documentation-first, with a possible follow-up implementation task if diagnostics require first-class `disabled_reason` support.

## Target documentation sections

- `04_mcp_03_01_dispatch-and-routing.md`
  - `ツール呼び出しディスパッチフロー`
  - `ToolRouteResolver`
- `04_mcp_03_06_tool-runtime-availability-metadata.md`
  - `enabled / disabled_reason`
  - `/v1/tools behavioral rules`
  - `RuntimeToolRegistry (agent-side)`
- `04_mcp_06_09_mcp-failure-diagnosis.md`

## Problem

The documentation does not clearly separate three different concepts:

1. Tool visibility to the LLM.
2. Runtime routability through `ToolRouteResolver`.
3. Runtime availability metadata from `/v1/tools`.

These concepts are related, but they are not identical.

## Current implementation behavior

LLM visibility:

- `LLMTurnRunner._filter_disabled_tool_definitions()` uses `ctx.services_required.runtime_tools`.
- It calls `RuntimeToolRegistry.llm_tool_definitions()`.
- Tools not visible according to the registry are removed from the LLM payload.
- If `RuntimeToolRegistry` is `None`, the LLM payload is not filtered and configured tool definitions are passed through.

Runtime routing:

- `ToolRouteResolver.resolve()` uses `RuntimeToolRegistry` only.
- If a tool is not present in the registry, execution fails with `Unknown tool`.

Availability metadata:

- `/v1/tools` `enabled` is normalized into `RuntimeTool.enabled_for_llm`.
- `RuntimeTool` currently does not expose a first-class `disabled_reason` field.
- `disabled_reason` may remain available only via the raw tool definition.

## Required documentation changes

Add a clear two-stage explanation:

### LLM-visible

A tool is LLM-visible if it remains in the tool payload after `LLMTurnRunner._filter_disabled_tool_definitions()` runs.

### Runtime-routable

A tool is runtime-routable only if `ToolRouteResolver.resolve()` can find it in `RuntimeToolRegistry`.

### Important failure mode

If `RuntimeToolRegistry` is missing, a tool may still be visible to the LLM because filtering is skipped, but execution can still fail with `Unknown tool`.

## Required availability metadata clarification

Document the current field mapping:

- `/v1/tools.enabled` -> `RuntimeTool.enabled_for_llm`
- `/v1/tools.disabled_reason` -> part of the `/v1/tools` contract, but not currently a first-class `RuntimeTool` field

If direct diagnostic access is required, create a future implementation task to add:

- `RuntimeTool.disabled_reason`

## Reason for change

Operators and developers need to understand why a tool can appear in the LLM payload but fail at execution time. Without this distinction, routing failures may be misdiagnosed as LLM or approval issues.

## Acceptance criteria

- Documentation clearly separates LLM visibility from runtime routability.
- The failure mode "LLM can see a tool but execution fails with `Unknown tool`" is documented.
- The docs state that `enabled` maps to `enabled_for_llm`.
- The docs do not imply that `RuntimeTool.disabled_reason` exists today.
- A future task is documented if first-class disabled reason support is desired.
