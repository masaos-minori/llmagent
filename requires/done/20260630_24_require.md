# [High] Remove static fallback routing from ToolRouteResolver

## Priority
High

## Background
`shared/route_resolver.py` still documents static fallback routing and defines `_SET_ROUTES`. This is a backward-compatibility path and should be removed. Routing must rely only on live `/v1/tools` discovery and `ToolRegistry`. 【1-acccf4】

## Goal
Remove static fallback routing from `ToolRouteResolver`.

## Scope
Affected files:
- `shared/route_resolver.py`
- `shared/tool_registry.py`
- `shared/tool_constants.py`
- `tests/test_route_resolver.py`
- MCP/Agent routing documentation

## Implementation Tasks
- Remove `_SET_ROUTES` from `shared/route_resolver.py`.
- Remove routing-purpose imports from `shared.tool_constants`.
- Remove static fallback lookup from `ToolRouteResolver.resolve()`.
- Keep routing sources limited to:
  1. live `/v1/tools` discovery map
  2. `ToolRegistry`
- Make unknown tools fail with a clear `ValueError`.
- Remove static fallback wording from `route_resolver.py` and `tool_registry.py` docstrings.
- Update tests that assume static fallback routing.

## Acceptance Criteria
- `ToolRouteResolver` no longer imports tool sets from `tool_constants.py` for routing.
- `_SET_ROUTES` no longer exists.
- Unknown tools are not resolved through static fallback.
- Active docs no longer describe static fallback as a routing layer.
- Tests pass.

## Validation
- Run: `pytest tests/test_route_resolver.py -v`
- Run: `rg -n "Static fallback|compatibility/emergency|_SET_ROUTES|_fallback_route" shared docs tests`
- No active implementation or active specification should contain removed static fallback routing.
