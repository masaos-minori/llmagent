# Requirement: Install SIGINT Handler for Graceful Shutdown During Normal Operation

## Priority

Medium

## Target files

- `scripts/agent/repl.py`

## Background

The agent REPL currently handles SIGTERM gracefully via `loop.add_signal_handler(signal.SIGTERM, _sigterm_handler)`, which sets `ctx.conv.shutdown_requested = True` and triggers `_close_resources()`. However, no corresponding SIGINT handler is installed for normal REPL operation. During shutdown, `HttpServerLifecycleManager.shutdown_all()` temporarily replaces the SIGINT handler with `_absorb_sigint_during_shutdown` to prevent orphaned subprocesses from a second Ctrl-C, and restores the original handler afterward. But during normal operation, pressing Ctrl-C raises `KeyboardInterrupt` which is caught by `_read_input()` and silently returns `None`, exiting the REPL loop without triggering the graceful shutdown path.

## Problem

When the user presses Ctrl-C during normal REPL operation (not during shutdown), the default behavior raises `KeyboardInterrupt` which is caught by `_read_input()`:

```python
except KeyboardInterrupt:
    self._view.write_turn_end()
    return None
```

This causes the REPL loop to exit without calling `_close_resources()`, leaving MCP subprocesses, HTTP clients, and other managed resources in an inconsistent state. Resources may remain running after the REPL exits.

## Reason for change

Graceful shutdown must work consistently regardless of which interrupt signal the user sends. Without a SIGINT handler, Ctrl-C produces different behavior than SIGTERM — it bypasses resource cleanup entirely. This violates the principle of least surprise and can lead to orphaned processes, especially when MCP subprocesses are running.

## Implementation intent

Install a SIGINT handler alongside the existing SIGTERM handler in `AgentREPL.run()`, routing both signals to the same shutdown handler function. This ensures that Ctrl-C and SIGTERM produce identical shutdown behavior.

## Implementation instructions

1. In `AgentREPL.run()`, add `loop.add_signal_handler(signal.SIGINT, _sigterm_handler)` alongside the existing SIGTERM handler installation.
2. In the `NotImplementedError` fallback branch (for non-event-loop environments), also add `_signal.signal(signal.SIGINT, lambda *_: _sigterm_handler())`.
3. Ensure the handler function name is clear — consider renaming `_sigterm_handler` to `_shutdown_handler` since it now handles both signals, or keep the current naming if refactoring scope is out of bounds.
4. Do NOT modify the shutdown-time SIGINT absorption logic in `http_lifecycle.py` — that mechanism remains unchanged and serves a different purpose (preventing double-interrupt during cleanup).

## Acceptance criteria

- [ ] Pressing Ctrl-C during normal REPL operation triggers `_close_resources()` and cleans up all managed resources (MCP subprocesses, HTTP clients, etc.)
- [ ] SIGINT and SIGTERM produce identical shutdown behavior
- [ ] No orphaned MCP subprocesses remain after Ctrl-C shutdown
- [ ] The shutdown-time SIGINT absorption mechanism in `http_lifecycle.py` continues to function correctly (no interference between the two mechanisms)

## Tests

- Manual test: Start REPL, wait for full startup, press Ctrl-C, verify MCP subprocesses terminate cleanly
- Unit test: Mock event loop and verify SIGINT handler is registered alongside SIGTERM handler
- Integration test: Verify that during shutdown_all(), the SIGINT absorption mechanism still works without conflict
