# Add destructive-operation guard for memory rebuild

## Priority

Medium

## Type

Implementation / Safety / Operations

## Background

The memory subsystem stores current memory state in SQLite. JSONL is retained as an append-only archive and recovery source.

JSONL does not replay delete, pin, or unpin state changes. Rebuilding SQLite from JSONL may therefore resurrect deleted entries or old pin states.

## Problem

`/memory rebuild` can be destructive if executed during normal operation without clear confirmation.

## Goal

Make memory rebuild safe by default and explicit when destructive changes are requested.

## Scope

Affected areas:

- `/memory rebuild` command handler
- `agent/memory/import_ops.py`
- `agent/memory/rebuild_ops.py`
- `JsonlMemoryStore`
- Memory consistency checks
- Memory command tests

## Implementation Tasks

- Make `/memory rebuild` default to dry-run mode.
- Require explicit confirmation for actual rebuild.
- Before rebuild, display:
  - JSONL record count
  - Current SQLite memory count
  - Expected inserted count
  - Warning that delete, pin, and unpin are not replayed
- After rebuild, run memory consistency checks automatically.
- Return structured rebuild results for testability.
- Consider future support for JSONL tombstone and state-change events.

## Acceptance Criteria

- Running `/memory rebuild` without confirmation does not modify SQLite.
- Actual rebuild requires an explicit user action.
- User sees a clear warning about non-replayed state changes.
- Rebuild result includes before/after counts.
- Consistency check runs after rebuild.
- Tests cover dry-run and actual rebuild paths.

## Validation Plan

Add tests for:

- Default dry-run behavior.
- Confirmed rebuild behavior.
- JSONL count reporting.
- SQLite count reporting.
- Consistency check invocation.
- Entries deleted from SQLite but still present in JSONL.
- Pin/unpin state not replayed from JSONL.

## Risks

Changing default behavior may surprise operators who currently expect rebuild to execute immediately. This is acceptable because rebuild is potentially destructive.

## Out of Scope

- Implementing full event-sourced memory state.
- Changing memory retrieval logic.
- Documentation-only updates.
