# [High] Remove static fallback routing from ToolRouteResolver

## Priority
High

## Background
`ToolRouteResolver` still contains static fallback routing based on tool sets from `tool_constants.py`. The implementation describes this path as compatibility/emergency routing. This fallback should be removed so that MCP tool routing depends only on live discovery and `ToolRegistry`.

## Goal
Remove static fallback routing and make `ToolRegistry` the only non-discovery routing source.

## Scope
Affected files:
- `shared/route_resolver.py`
- `shared/tool_registry.py`
- `shared/tool_constants.py`
- `tests/test_route_resolver.py`
- `docs/04_mcp_03_routing_lifecycle_and_execution.md`
- `docs/04_mcp_06_configuration_and_operations.md`

## Implementation Tasks
- Remove `_SET_ROUTES` from `shared/route_resolver.py`.
- Remove routing-purpose direct imports of tool sets from `shared/tool_constants.py`.
- Update `ToolRouteResolver.resolve()` to use only:
  - live discovery map
  - `ToolRegistry`
- Make unknown tools fail immediately with a clear `ValueError`.
- Remove `Static fallback` from `shared/tool_registry.py` documentation.
- Remove static fallback references from MCP routing documentation.
- Update or remove tests that assume fallback routing exists.

## Acceptance Criteria
- `route_resolver.py` no longer imports tool sets for routing.
- `ToolRouteResolver` no longer has a static fallback path.
- Unknown tools are not resolved through `tool_constants.py`.
- Tests pass after fallback-based expectations are removed.
- Active docs no longer mention static fallback as part of routing.

## Validation
- Run `pytest tests/test_route_resolver.py -v`.
- Run `rg -n "Static fallback|compatibility/emergency|_SET_ROUTES|_fallback_route" shared docs tests`.
- The search must not return active source references except migration/archive notes if explicitly allowlisted.
