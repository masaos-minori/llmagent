# Requirement: Implement Retry Cooldown for Failed MCP Subprocess Starts in Non-Production Mode

## Priority

Low-Medium

## Target files

- `scripts/agent/factory.py`

## Background

In `StartupOrchestrator._start_servers()`, when an HTTP subprocess MCP server fails to start in non-production mode, the error is logged and a warning is displayed, but execution continues. The failed server's process object is never stored in `_http_procs`. When a tool call later triggers `_ServerLifecycleRouter.ensure_ready()` for that server, it attempts to start the subprocess again without any delay or rate limiting.

## Problem

When an HTTP subprocess MCP server fails to start during initial startup in non-production mode, subsequent tool calls that route to that server trigger repeated subprocess start attempts without any backoff mechanism. A transient port conflict or temporary resource unavailability causes the same failure to be retried on every tool call until the condition resolves. This produces noisy logs and wastes system resources.

## Reason for change

Without a retry cooldown, transient failures during startup lead to persistent retry storms. Each tool call triggers another subprocess start attempt, generating log noise and consuming CPU cycles. A simple cooldown mechanism prevents rapid retries while allowing eventual recovery once the underlying condition resolves.

## Implementation intent

Add a retry cooldown mechanism to `_ServerLifecycleRouter` that tracks the last failure time per server and skips `ensure_ready()` attempts within the cooldown window. On successful start, clear the cooldown entry.

## Implementation instructions

1. Add `_failed_starts: dict[str, float] = {}` field to `_ServerLifecycleRouter.__init__()` to store last failure timestamps per server key.
2. Define a constant `COOLDOWN_SECONDS = 30` (or similar) at module level for the cooldown duration.
3. In `_ServerLifecycleRouter.ensure_ready()`, before calling `verify_running()`, check if the server is in cooldown:
   - Get last failure time from `_failed_starts.get(server_key, 0)`
   - If `time.monotonic() - last_failure < COOLDOWN_SECONDS`, log a warning and return early
4. After a successful subprocess start in `ensure_ready()`, remove the server from `_failed_starts`: `self._failed_starts.pop(server_key, None)`
5. After a failed subprocess start in `ensure_ready()`, record the failure time: `self._failed_starts[server_key] = time.monotonic()`
6. Import `time` module if not already imported.

## Acceptance criteria

- [ ] When an MCP subprocess fails to start during initial startup in non-production mode, subsequent `ensure_ready()` calls within the cooldown window are skipped with a warning log
- [ ] After the cooldown expires, `ensure_ready()` retries the subprocess start
- [ ] Successful subprocess start clears the cooldown entry
- [ ] No regression in production mode behavior (production mode already treats startup failures as fatal)
- [ ] Multiple consecutive tool calls routing to the same failed server do not generate repeated subprocess start attempts

## Tests

- Unit test: Mock `_http_mgr.verify_running()` to return False, verify `_failed_starts` is populated after first failure
- Unit test: Verify `ensure_ready()` returns early when cooldown is active
- Unit test: Verify cooldown entry is cleared after successful start
- Integration test: Configure an MCP server with invalid command, start agent in non-production mode, trigger multiple tool calls to the failed server, observe only one subprocess start attempt per cooldown period
