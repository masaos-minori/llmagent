# [High] Make ToolRegistry the single source of MCP tool ownership

## Priority
High

## Background
`ToolRegistry` is described as the primary source of truth, but routing documentation and code still mention config and static fallback layers. Tool ownership should be centralized in `ToolRegistry`.

## Goal
Make `ToolRegistry` the only non-discovery source of MCP tool ownership.

## Scope
Affected files:
- `shared/tool_registry.py`
- `shared/tool_constants.py`
- `shared/route_resolver.py`
- `tests/test_tool_registry.py`
- `tests/test_route_resolver.py`
- `docs/04_mcp_03_routing_lifecycle_and_execution.md`

## Implementation Tasks
- Update `tool_registry.py` documentation to state:
  - `ToolRegistry` is the MCP tool ownership source.
  - `tool_constants.py` is only a registry seed and classification source.
  - config `tool_names` is validation metadata.
  - static fallback does not exist.
- Ensure `ToolRouteResolver` calls `ToolRegistry` for registry routing.
- Ensure all MCP tools, including GitHub tools, are registered.
- Add or update tests proving every callable MCP tool has exactly one owner.
- Remove any comments suggesting fallback ownership.

## Acceptance Criteria
- Tool ownership is defined in `ToolRegistry`.
- Each tool belongs to exactly one server key.
- Resolver does not directly route from `tool_constants.py`.
- Tests fail if a callable MCP tool is missing from `ToolRegistry`.

## Validation
- Run `pytest tests/test_route_resolver.py tests/test_tool_registry.py -v`.
- Run `rg -n "Static fallback|Config tool_names.*routing|tool_constants.py.*routing" shared docs tests`.
