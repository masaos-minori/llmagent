# [Medium] Remove agent.toml's dead/duplicated RAG pipeline settings and unused /db rag subcommands

## Labels
priority/medium, area/config, area/rag, area/mcp, area/cli, consistency, cleanup

## Summary
`config/agent.toml`'s "RAG pipeline" section (lines 32-56) contains 16 keys: 15 RAG-related
keys plus `max_tool_turns` (unrelated to RAG, see below). Investigation showed that none of
the 15 RAG-related keys affect the agent's normal RAG execution path (`rag_run_pipeline`
tool → `rag_pipeline` MCP server, governed entirely by `config/rag_pipeline_mcp_server.toml`).
`embed_url` is a separate, unrelated field declared elsewhere in `agent.toml` (not one of
these 15) that remains genuinely in use — see Background for the distinction. The 15
RAG-related keys must be deleted outright — not consolidated or re-pointed — from
`agent.toml`, from `AgentConfig`, and from every code path that currently reads them
(`/config` display, `/rag search` debug command).

Separately, the entire `/db rag <subcmd>` command family (`stats`, `urls`, `clean`,
`rebuild-fts`, `vec-rebuild`, `reconcile-url`, `recover`, `consistency`) must be removed. It
has no operational track record and only imposes maintenance cost. `/db session <subcmd>`
is unaffected and must be kept.

Separately again, the `/rag` slash command itself (`/rag search <query> [--debug]`,
`_cmd_rag()` in `cmd_rag_export.py`) must be removed entirely. This supersedes Bucket 3 and
Bucket 4 below with respect to `/rag`: since the command itself is gone, there is no more
code path reading `use_mqe`/`use_rrf`/`use_rerank`/`rag_top_k`/`rag_min_score`/
`mqe_n_queries`/`use_search`/`rag_service_url` from `agent.toml` at all, and the Bucket 3
"Implementation note" (what `/rag search` should fall back to) is now moot — there is no
`/rag search` left to need a fallback. `/export` and `/compact` (also in `cmd_rag_export.py`,
via `_RagExportMixin`) are unrelated commands and are out of scope.

## Background
Investigation of `scripts/agent/config_dataclasses.py` (`RAGConfig` dataclass, line 149),
`scripts/agent/config_builders.py` (`_build_rag_config()`, lines 153-169), and
`scripts/agent/commands/cmd_rag_export.py` (the `/rag search` debug command) shows the
following. `agent.toml`'s "RAG pipeline" section (lines 32-56) contains 16 keys, one of
which (`max_tool_turns`) is unrelated to RAG and handled separately below. The remaining 15
keys fall into three buckets:

- **Note on `embed_url`:** `embed_url` is *not* one of these 15 keys — it is declared
  elsewhere in `agent.toml`, under the "Infrastructure (common)" section (line 10), not
  under "RAG pipeline". It is mentioned here only for contrast: it is the sole RAG-related
  field of the `RAGConfig` dataclass (`config_dataclasses.py:157`) that is actually consumed
  — in `scripts/agent/factory.py:331` and `scripts/agent/repl_health.py:113` (embedding LLM
  URL construction and healthcheck). It is out of scope for this requirement and must be
  kept as-is, in its current section, unchanged.

**Bucket 2 — Held in `RAGConfig` but never read for RAG execution:** `top_k_search`,
`top_k_rerank`, `max_chunks_per_doc`, `rrf_k`, `use_semantic_cache`,
`semantic_cache_threshold`, `semantic_cache_max_size`. These exist as dataclass fields and
pass through `config_validators.py` validation (with caveats — see the validator note under
Bucket 2 in Required Changes), and most are shown by `/config` (`cmd_config_display.py`) —
except `rrf_k`, which `cmd_config_display.py` never displays at all. No code path passes any
of these 7 fields into an actual `RagPipeline` construction or execution.

**Bucket 3 — Not present in `AgentConfig` at all; read only by the `/rag search` debug
command:** `use_mqe`, `use_rrf`, `use_rerank`, `rag_top_k`, `rag_min_score`, `mqe_n_queries`,
`use_search`. `cmd_rag_export.py`'s `_cmd_rag()` calls `ConfigLoader().load_all()` to
re-read the raw merged TOML dict (bypassing `AgentConfig` entirely) and passes it to
`RagPipelineConfig.from_dict()` (`scripts/mcp_servers/rag_pipeline/models.py`). This is
the only place these 7 keys have any effect, and only when a developer runs
`/rag search <query>` in the REPL.

**Bucket 4 — Completely dead:** `rag_service_url`. Even in the one code path that reads it
   (`/rag search` via `RagPipelineConfig.from_dict()`), `build_rag_cfg_adapter()`
   (`scripts/mcp_servers/rag_pipeline/models.py:119`) unconditionally overwrites it to `""`
   with the comment "prevent HTTP loop when augment() is called in-process". The value
   configured in `agent.toml:52` (`http://127.0.0.1:8010`) is never actually used anywhere;
   the real, effective MCP connection URL is `agent.toml:386`'s
   `[mcp_servers.rag_pipeline].url` (same value, separately defined).

Separately, `max_tool_turns` (`agent.toml` line 38) sits inside the "RAG pipeline" section
header but is unrelated to RAG — it is `ToolConfig.max_tool_turns`
(`scripts/agent/config_dataclasses.py:210`), consumed by
`scripts/agent/llm_turn_runner.py:76` to cap tool-call loop iterations per turn. This is a
section-placement mistake: the key and its function must be preserved, only its location in
`agent.toml`'s comment-block layout is wrong.

This was surfaced while answering "should agent.toml's RAG pipeline config actually live in
the RAG MCP server's config?" — the conclusion was that most of it is dead weight to be
removed, not relocated.

### `/db rag` subcommand removal

`scripts/agent/commands/cmd_db.py`'s `_DbMixin` dispatches `/db <subcmd>` into two independent
families: `/db rag <subcmd>` (handled by `_cmd_db_rag()`, lines 54-75, dispatching to
`DbRagOps` in `scripts/agent/commands/db_rag_ops.py`: `clean`, `list_urls`, `rebuild_fts`,
`vec_rebuild`, `reconcile_url`, `recover`, `consistency`, plus a `stats` handler in
`DbStatsDisplay`) and `/db session <subcmd>` (handled by `_cmd_db_session()`, lines 77-98,
dispatching to `DbSessionOps`). These two families are structurally independent — `/db rag`
can be removed without touching `/db session`.

Reason for removal: no operational track record, ongoing maintenance cost only (per user
decision). This is a command-surface removal, not necessarily a removal of the underlying
RAG DB maintenance primitives themselves (`scripts/rag/maintenance.py`'s `rebuild_fts()`,
`scripts/db/recovery.py`'s `recover_corruption()`, etc.) — see Non-Goals.

### `/rag` command removal

`/rag` is registered in `scripts/agent/commands/command_defs_list.py` (`CommandDef("/rag", ...,
"_cmd_rag", "search <query> [--debug]  Search the RAG knowledge base")`) and implemented as
`_cmd_rag()` in `scripts/agent/commands/cmd_rag_export.py`'s `_RagExportMixin`, alongside the
unrelated `_cmd_export` (`/export`) and `_cmd_compact` (`/compact`) handlers in the same
mixin/file. `_cmd_rag()` is the sole reason `agent.toml`'s Bucket 3/4 keys have any effect at
all (via `ConfigLoader().load_all()` → `RagPipelineConfig.from_dict()` → in-process
`RagPipeline.augment()`); removing the command removes that entire code path along with it,
so Bucket 3/4 no longer need any replacement config source.

Reason for removal: per user decision, following the same "no operational track record,
maintenance cost only" reasoning as `/db rag`.

## Required Changes

**Bucket 2 — remove entirely (agent.toml + AgentConfig + `/config` display):**
- Remove `top_k_search`, `top_k_rerank`, `max_chunks_per_doc`, `rrf_k`, `use_semantic_cache`,
  `semantic_cache_threshold`, `semantic_cache_max_size` from `agent.toml`'s RAG pipeline
  section.
- Remove the corresponding fields from `RAGConfig` (`config_dataclasses.py`) and from
  `_build_rag_config()` (`config_builders.py`).
- Remove only the validators that exist for these specific fields —
  `_v_rag_tks` (`top_k_search`), `_v_rag_tkr` (`top_k_rerank`), `_v_rag_mcd`
  (`max_chunks_per_doc`), `_v_rag_rrf` (`rrf_k`), all called from `RAGConfig.__post_init__`
  (`config_dataclasses.py:169-175`). **Do not** remove `_v_rag_rmt`, `_v_rag_rt`, or
  `_v_rag_rmcc` from that same `__post_init__` — those validate `refiner_max_tokens`,
  `refiner_timeout`, and `refiner_max_chars_per_chunk` respectively, which are unrelated
  `RAGConfig` fields kept out of scope by this requirement. Note also that
  `use_semantic_cache`, `semantic_cache_threshold`, and `semantic_cache_max_size` have no
  dedicated validator function in `__post_init__` at all — there is nothing to remove for

  `semantic_cache_threshold`, and `semantic_cache_max_size`. `rrf_k` has no display line in

## Acceptance Criteria
- `config/agent.toml` no longer declares `top_k_search`, `top_k_rerank`, `max_chunks_per_doc`,
  `rrf_k`, `use_semantic_cache`, `semantic_cache_threshold`, `semantic_cache_max_size`,
  `use_mqe`, `use_rrf`, `use_rerank`, `rag_top_k`, `rag_min_score`, `mqe_n_queries`,
  `use_search`, or `rag_service_url`.
- `max_tool_turns` is documented under the "Tools" section comment block, not "RAG pipeline",
  and continues to function identically (same default, same validation, same effect on
  `llm_turn_runner.py`).
- `RAGConfig` (`config_dataclasses.py`) contains only fields with a real consumer
  (`embed_url` plus the non-RAG-pipeline fields already there, e.g. `web_search_url`,
  `web_search_max_results`, `use_refiner`, `refiner_*`).
- `/config` no longer displays any of the removed `RAGConfig` fields.
- `/rag` no longer exists as a command (`/rag search <query>` falls through to the unknown
  command path); `/export` and `/compact` continue to work identically to before.
- `docs/03_rag_05_1-configuration-reference.md` no longer lists removed keys as
  `agent.toml`/`AgentConfig` fields.
- `uv run pytest` passes with no new failures; existing tests referencing the removed
  `RAGConfig` fields, validators, or `cmd_rag_export.py`'s old config-loading behavior are
  updated.
- `/db rag` no longer exists as a command (`/db rag stats` etc. fall through to the
  dispatcher's error/help path); `/db session <subcmd>` continues to work identically to
  before.
- `DbRagOps` (`db_rag_ops.py`) is deleted; no remaining import references it.
- No documentation file describes `/db rag` as a currently-available command.
- `_cmd_rag()` no longer exists in `cmd_rag_export.py`; no `CommandDef("/rag", ...)` entry
  remains in `command_defs_list.py`; `tests/test_cmd_rag_refiner_hint.py` is deleted.
- No documentation file describes `/rag search` / `/rag <query>` as a currently-available
  command.

## Non-Goals
- Do not change the `rag_pipeline` MCP server's own behavior or `rag_pipeline_mcp_server.toml`
  schema — it remains the single source of truth for RAG pipeline tuning for the normal
  `rag_run_pipeline` tool path.
- Do not remove `embed_url` from `agent.toml` or `RAGConfig` — it has a real consumer.
- Do not address unrelated `agent.toml` sections in this task.
- Do not remove `/db session <subcmd>` or any of its underlying operations
  (`DbSessionOps`) — only the `/db rag` family is in scope.
- Do not remove the RAG MCP server tools `rag_delete_document` / `rag_list_documents` — those
  are a separate, MCP-facing interface and are out of scope here.
- Do not decide, in this requirement, whether `scripts/rag/maintenance.py` /
  `scripts/db/recovery.py` themselves should be deleted if they become unreferenced — flag it
  as a finding for the planning phase instead of deleting library code as a side effect of a
  command-removal task.
- Do not remove `/export` or `/compact` (`_cmd_export`, `_cmd_compact` in
  `cmd_rag_export.py`) — only `_cmd_rag()` is in scope; the other two handlers in the same
  mixin/file are unrelated and must keep working.
- Do not remove or change the `rag_run_pipeline` / `rag_debug_pipeline` MCP tools — those
  remain the supported way to invoke the RAG pipeline interactively (via the LLM), replacing
  only the REPL-local `/rag search` debug shortcut.
