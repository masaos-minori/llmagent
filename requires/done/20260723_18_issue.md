# Standardize Major Design Documents with Japanese Behavior, Intent, Boundary, Failure, and Unknown Sections

## Priority
Medium

## Summary
Standardize the structure of selected major design documents by separating current behavior, design intent, responsibility boundaries, failure behavior, and unresolved items.

## Background
The documentation already contains useful information such as:
- implementation notes,
- current behavior,
- inferred intent,
- responsibility boundaries,
- failure behavior,
- Needs confirmation items.

However, section names and structure differ across documents.

## Problem
When current behavior, inferred design intent, and unresolved uncertainty are mixed together, AI agents and reviewers may mistake uncertain interpretation for confirmed specification.

A consistent section structure will make the documents easier to read, review, and use for implementation planning.

## Rationale
This task improves clarity without changing technical content.

The documentation should clearly distinguish:
- what the implementation currently does,
- why the design appears to exist,
- what the component owns or does not own,
- how failures are handled,
- what still needs confirmation.

This supports safer AI-assisted implementation and documentation maintenance.

## Intent
The intent is to improve readability and reduce ambiguity in major design documents.

This is not a rewrite. It is a targeted restructuring of existing content where the separation improves clarity.

## Required Changes
Apply the following Japanese headings where appropriate:

- `## 現在の実装挙動`
- `## 設計意図`
- `## 責務境界`
- `## 失敗時の挙動`
- `## 未確認事項`

Do not force all five sections into every document if a section is not relevant.

Initial target documents:
- `01_overview-arch-01-process.md`
- `01_overview-arch-02-pipelines.md`
- `01_overview-arch-03-features.md`
- `03_rag_01_system_overview-part1.md`
- `03_rag_03_01_query_pipeline-overview.md`
- `04_mcp_00_document-guide.md`
- `05_agent_00_document-guide.md`
- `06_eventbus_00_document-guide.md`
- `90_shared_00_document-guide.md`

Use the existing evidence labels defined in `00_governance_03_evidence-labels.md`.

## Acceptance Criteria
- Selected major documents more clearly separate current behavior from design intent.
- Responsibility boundaries are explicit where relevant.
- Failure behavior is clearly separated where relevant.
- Unresolved or uncertain items are placed under `未確認事項` or linked to the Needs Confirmation inventory.
- Existing evidence labels are preserved or normalized.
- No new evidence label system is introduced.
- Existing useful content is preserved.
- No document is rewritten from scratch.

## Definition of Done
- Documentation is written in Japanese.
- Markdown renders correctly.
- Technical meaning is preserved.
- No source code is modified.
- No ADR documents are created.
- No Known Issues migration is performed in this task.
- No changes are made to `00_index.md`.

## Out of Scope
- Do not edit `00_index.md`.
- Do not introduce ADRs.
- Do not change system behavior.
- Do not resolve Needs Confirmation items.
- Do not migrate Known Issues documents.
- Do not force the template into short documents where it does not help.

## AI Implementation Instruction
Work document by document.

Preserve existing content and reorganize only where it improves clarity.

Use the Japanese section headings exactly as specified.

Do not over-template guide documents.

Keep edits small, reviewable, and evidence-aware.
