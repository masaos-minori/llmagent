# Improve Fail-Open Visibility and Runtime Tool Availability Handling

## Summary

Improve user and operator visibility when the system starts in fail-open mode, and ensure unavailable MCP tools are not exposed as usable runtime tools.

This issue combines the previous tasks for:

- Startup readiness report
- Fail-open visibility
- Runtime exclusion of tools from unavailable MCP servers

## Reason for Change

In local mode, the system may continue startup even when one or more MCP servers fail.

This behavior is useful for development, but it can mislead users because the agent appears to be running while some capabilities are unavailable.

Potential problems include:

- Users may not notice that important MCP servers failed to start.
- The LLM may attempt to use tools that are unavailable.
- Tool execution may fail repeatedly.
- Workflow tasks may fail after approval even though the missing dependency was known at startup.
- Tool loop guards mitigate repeated failures but do not solve the underlying availability mismatch.

Production mode already uses stricter fail-fast behavior, but local mode needs stronger visibility and safer runtime tool availability handling.

## Intent

The intent is not to remove fail-open behavior from local mode.

The intent is to make fail-open state explicit, visible, and operationally safe.

After this change:

- Users should clearly see which capabilities are unavailable.
- Startup output should summarize MCP health.
- A CLI command should show readiness state after startup.
- Runtime tool discovery should not expose tools from unavailable MCP servers.
- Planning context should not advertise tools that cannot be executed.

## Scope

Update startup, runtime registry, and documentation.

Likely affected areas:

- Startup orchestration
- MCP health tracking
- Runtime tool discovery
- Runtime tool registry construction
- CLI status command
- Operations documentation

Exact file names are implementation-dependent and should be verified before making changes.

## Functional Requirements

### 1. Startup Readiness Report

At startup, print a concise readiness report.

The report should include:

- MCP healthy count
- MCP degraded count
- MCP unavailable count
- Workflow readiness
- RAG readiness if available
- Memory readiness if available
- Capability-level summary where possible

Example output shape:

System Readiness Report

MCP:
  Healthy: 8
  Degraded: 1
  Unavailable: 2

Capabilities:
  Repository Operations: unavailable
  File Management: healthy
  RAG Retrieval: healthy
  Memory Services: degraded

### 2. Persistent Visibility

Expose the same readiness information through a CLI command.

Suggested command options:

- `/mcp status`
- `/health`
- `/status`

Use the existing command structure where appropriate.

### 3. Logging

Write readiness information to logs during startup.

The log should clearly state:

- Which MCP servers are unavailable
- Which capabilities are affected
- Whether the process is continuing because of local fail-open behavior

### 4. Runtime Tool Registration Rule

Runtime tool registry construction must follow this rule:

- Healthy MCP server: register discovered tools.
- Degraded MCP server: behavior must be explicitly defined.
- Unavailable MCP server: do not register its tools.

If degraded behavior is uncertain, implement the conservative behavior:

- Degraded MCP server tools are excluded unless explicit policy allows inclusion.

### 5. Planning Context Safety

Tools that cannot be executed must not be advertised to the LLM as available tools.

This prevents the model from selecting tools that are known to be unavailable.

### 6. Documentation

Update documentation to explain:

- Why local mode can continue after MCP startup failure
- How fail-open state is displayed
- How unavailable MCP tools are handled
- Difference between local and production behavior
- Operator action when required capabilities are unavailable

## Non-Goals

- Do not remove fail-open behavior from local mode.
- Do not change production fail-fast behavior unless required by existing tests.
- Do not introduce EventBus integration.
- Do not redesign workflow approval logic.

## Implementation Notes for AI Agent

- Inspect the current startup and MCP discovery flow before modifying code.
- Preserve production fail-fast semantics.
- Prefer conservative runtime safety: do not expose unavailable tools.
- Add tests for unavailable MCP server behavior.
- Add tests for startup readiness report generation if test infrastructure exists.
- If exact degraded-state semantics are unclear, document them as `Unknown` and implement only clearly safe behavior.
- Avoid duplicating readiness logic in multiple places; centralize if practical.

## Acceptance Criteria

- Startup output clearly reports degraded or unavailable MCP services.
- Local fail-open continuation is explicitly visible to the user.
- Readiness information is available after startup through a CLI command.
- Readiness information is written to logs.
- Tools from unavailable MCP servers are not registered in the runtime tool registry.
- Unavailable tools are not advertised to the LLM planning context.
- Production fail-fast behavior remains intact.
- Documentation is updated.
- Tests are added or updated for unavailable MCP tool registration behavior.
