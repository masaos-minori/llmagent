# Title
Add half-open recovery behavior to MCP server health management

## Summary
Improve MCP server health handling so temporarily failed servers can recover through controlled re-entry instead of depending only on watchdog or manual restart.

## Background
The current health model tracks failures and blocks dispatch when a server becomes unavailable.
This protects the system, but recovery paths are limited once dispatch is fully blocked.

## Problem
A server may recover externally, but the agent may continue treating it as unavailable without a controlled recovery probe path.

## Required Changes
- Introduce a half-open or trial-reconnect behavior after a cooldown period.
- Allow limited recovery probes for previously unavailable servers.
- Recover health state when trial probes or health checks succeed.
- Keep failure transitions and recovery transitions visible in logs or diagnostics.

## Acceptance Criteria
- Temporarily failed servers can recover without requiring a full agent restart in all cases.
- Recovery does not remove protection against repeated failure loops.
- State transitions remain observable and documented.

## Out of Scope
- Replacing the watchdog subsystem
- Redesigning HTTP vs stdio lifecycle management

## AI Implementation Instruction
Implement this as a circuit-breaker-style recovery task.

Strict rules:
- Do not weaken failure isolation.
- Recovery must be intentional, limited, and observable.
- Preserve existing healthy/degraded/unavailable semantics unless an explicit redesign is required.
- If the retry/cooldown policy is unclear, mark it as: Needs confirmation

Deliverables:
1. Half-open recovery path
2. Visible state-transition logging/diagnostics
3. Documentation of recovery behavior
