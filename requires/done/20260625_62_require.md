## [High][agent][cli] Remove the /db recover compatibility alias and require explicit DB scope

### Summary
The current recover command still has a compatibility alias behavior for backward compatibility.

### Problem
`/db recover` hides which database is being recovered and keeps legacy routing alive.
Recovery commands are destructive or maintenance-sensitive operations and should require explicit scope.

### Scope
- agent/commands/*
- recovery command routing
- docs/05_agent_90_inconsistencies_and_known_issues.md
- docs/05_agent_07_cli-and-commands.md

### Facts
- The known-issues doc explicitly states that `/db recover` is a `compatibility alias` and that `/db rag recover` targets `rag.sqlite`, while `/db session recover [backup-path]` exists separately for session DB recovery. 【3-4da049】
- The CLI docs include `/db recover [backup-path]` in the flat compatibility-alias command set. 【1-819e5c】

### Task
- remove `/db recover` as a valid command
- require explicit recovery scope:
  - `/db rag recover [backup-path]`
  - `/db session recover [backup-path]`
- return a clear validation error when `/db recover` is used
- update user-facing help, docs, and examples to use only scoped recovery commands
- remove or update tests that still expect `/db recover`

### Expected Result
Recovery commands are explicit about which database they operate on, and legacy alias routing is removed.

### Acceptance Criteria
- `/db recover` is no longer accepted
- only scoped recovery commands remain
- help text and docs show only scoped recovery commands
- tests no longer rely on `/db recover`
