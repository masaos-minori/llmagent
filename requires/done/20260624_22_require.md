# Improve startup visibility for fail-open MCP security settings

## Summary
Increase operator visibility for fail-open settings by documenting and summarizing them as one explicit startup review category.

## Background
The documentation clearly distinguishes fail-open and fail-closed controls, but fail-open settings remain distributed across multiple server-specific sections.

## Objective
Make all fail-open settings reviewable in one place during startup and operations.

## In Scope
- `04_mcp_05_security_and_safety_model.md`
- `04_mcp_06_configuration_and_operations.md`
- Startup-audit guidance

## Required Changes
- Add a consolidated fail-open review list covering at least:
  - `workflow_allowlist`
  - `allowed_repos_mode = "fail_open"`
  - empty `path_denylist`
  - empty `protected_branches`
- Document the startup warning posture for each
- Distinguish risk-bearing empty settings from intentional permissive configuration
- Add an operator checklist for pre-production review

## Acceptance Criteria
- All fail-open settings can be reviewed from one operations-oriented section
- Operators can identify which empty settings increase access scope
- The fail-open review guidance is consistent with server-specific security sections

## Out of Scope
- Changing current security defaults
- Adding new allowlist types

## Notes
Focus on visibility and reviewability rather than policy changes.
