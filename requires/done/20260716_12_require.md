## Background

`server.py` states that the MDQ server uses an empty `auth_token` and that MDQ has its own authorization through `allowed_dirs`. It also calls `attach_auth_middleware(..., "")`. 【1-bfd1ab】

The same file currently describes the MDQ server as exposing 7 tools, while `tools.py` has included additional admin tools. The final wording depends on the tool surface decision made in the higher-priority issue. 【1-bfd1ab】【2-03d74e】

## Problem

Some MDQ code comments and documentation may be misread as legacy leftovers rather than intentional design choices. In particular:

- `auth_token=""` could be mistaken for accidental insecure configuration,
- `allowed_dirs` authorization could be overlooked,
- tool count comments can become stale after tool-surface cleanup,
- MDQ search cleanup could be confused with RAG/session vector schema cleanup.

## Reason for change

After implementation cleanup, documentation must accurately explain the final architecture. This reduces the chance that future maintainers remove intentional design choices or reintroduce previously cleaned-up partial features.

## Design intent

Use documentation to lock in the intended boundaries:

- MDQ HTTP auth middleware behavior,
- MDQ path authorization via `allowed_dirs`,
- final MDQ tool surface,
- distinction between MDQ search and RAG/session vector storage,
- distinction between production tools and admin tools if admin tools remain.

## Implementation instructions

- Update `server.py` module docstring after final tool surface is decided.
- Document whether MDQ exposes only production tools or also admin tools.
- Document that `auth_token=""` is intentional under the current design.
- Document that MDQ path authorization is handled through `allowed_dirs`.
- Add a note that future bearer authentication hardening should be tracked separately.
- Add a note that MDQ hybrid search cleanup does not imply removal of RAG/session vector schemas.
- Ensure documentation does not contradict the final implementation from the High and Medium issues.

## Acceptance criteria

- `allowed_dirs` is documented as the MDQ authorization boundary.
- Future authentication hardening is not mixed into cleanup work.

- No behavioral changes are introduced by this issue.
