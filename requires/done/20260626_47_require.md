# Title
Improve diagnostics and warnings for `use_rrf=False`

## Priority
Medium

## Labels
- enhancement
- rag
- retrieval
- diagnostics
- quality

## Dependencies
- None

## Summary
Make the quality impact of `use_rrf=False` more visible in diagnostics, debug output, and configuration docs.

## Background
When `use_rrf=False`, fusion falls back to dedup-only mode. All hits receive `rrf_score=0.0`, and MQE no longer contributes rank-weighted benefit.

## Problem
This setting can significantly reduce retrieval quality, but it may look like a harmless fallback mode.

## Required Changes
- Ensure diagnostics include `fusion_mode="dedup_only"` when `use_rrf=False`.
- Add debug output that explicitly says rank signal is disabled.
- Add configuration documentation warning about retrieval quality degradation.
- Optionally emit startup/config warning when `use_rrf=False`.

## Acceptance Criteria
- Operators can see when dedup-only fusion is active.
- Debug output explains the quality tradeoff.
- Docs recommend `use_rrf=True` unless there is a clear reason to disable it.

## Out of Scope
- Removing `use_rrf=False`.
- Changing RRF scoring formula.

## Components
- `scripts/rag/stages/fusion.py`
- `scripts/rag/pipeline.py`
- `/rag search --debug`
- `03_rag_03_query_pipeline.md`
- `03_rag_05_configuration_and_operations.md`

## AI Implementation Instruction
Implement this as a retrieval-quality observability task.

Strict rules:
- Do not hide `use_rrf=False` as a normal success path without diagnostic context.
- Do not change RRF scoring unless separately requested.
- Make the quality tradeoff clear in docs and debug output.
- If warning behavior is undesirable in some environments, mark it as: `Needs confirmation`.
