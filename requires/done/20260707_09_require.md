# Treat tasks.workflow_id as a required invariant

## Priority

Medium

## Type

Workflow / Data Integrity / Observability

## Background

Workflow audit and observability depend on stable workflow identifiers.

`workflow_id` must remain consistent across:

- task creation
- stage execution
- approvals
- artifacts
- retries
- turn-end audit events

## Problem

If a task lacks `workflow_id`, code may generate a new workflow ID during resume.

That breaks audit correlation and makes the workflow trace inconsistent.

## Goal

Make missing `workflow_id` an invalid workflow state.

## Scope

Affected areas:

- Task creation
- Existing task resume
- Orchestrator workflow initialization
- Workflow state validation
- Audit and OTel context

## Implementation Tasks

- Ensure new tasks always require `workflow_id`.
- Remove fallback logic equivalent to `task.workflow_id or uuid4()`.
- Fail fast when resuming a task without workflow ID.
- Ensure audit events always use the stored workflow ID.
- Ensure OTel spans always use the stored workflow ID.
- Add tests for missing workflow ID failure.

## Acceptance Criteria

- New workflow tasks always have workflow ID.
- Existing tasks without workflow ID cannot resume.
- No code path generates a replacement workflow ID for an existing task.
- Audit correlation remains stable across approval suspend/resume.
- OTel spans use the same workflow ID across all stages.

## Validation Plan

Add tests for:

- New task has workflow ID.
- Resume task with workflow ID succeeds.
- Resume task without workflow ID fails.
- Approval resume keeps workflow ID.
- Stage audit uses stable workflow ID.
- Turn-end audit uses stable workflow ID.

## Risks

Old task rows without workflow ID become invalid. Migration compatibility is intentionally out of scope.

## Out of Scope

- Backfilling workflow IDs.
- Supporting legacy nullable workflow IDs.
- Migration scripts.
