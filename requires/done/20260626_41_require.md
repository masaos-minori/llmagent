# Title
Unify RAG artifact file extensions to `.json`

## Priority
High

## Labels
- bug
- rag
- ingestion
- documentation
- file-format
- consistency

## Dependencies
- None

## Summary
Unify all RAG ingestion artifact paths and documentation around `.json` files. Remove stale `.txt` references from implementation, CLI help, docs, glob patterns, sentinel checks, and examples.

## Background
The RAG ingestion flow currently describes crawler output, chunk output, and registered chunk files as JSON artifacts. However, some documentation and operational examples still refer to `.txt` files.

## Problem
Mixed `.json` and `.txt` references make the artifact format ambiguous. This can cause operators or AI implementers to reintroduce obsolete `.txt` behavior.

## Required Changes
- Ensure crawler output uses `rag-src/*.json`.
- Ensure chunk output uses `rag-src/chunk/*.json`.
- Ensure registered output uses `rag-src/registered/*.json`.
- Update `source_file` examples from `.txt` to `.json`.
- Update CLI help, docs, glob patterns, and sentinel checks to use `.json`.
- If temporary `.txt` compatibility remains, document it explicitly as temporary compatibility.

## Acceptance Criteria
- No current documentation describes active RAG artifacts as `.txt`.
- Crawler, chunk splitter, ingester, and pipeline utilities consistently use `.json`.
- Tests cover `.json` file discovery, chunk creation, registered file movement, and `source_file` values.

## Out of Scope
- Changing the JSON payload schema.
- Switching to `.jsonl` or another artifact format.
