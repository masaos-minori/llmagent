# MCP /v1/tools Endpoint: Update Documentation to Match Current Discovery Expectations

## Priority

High

## Target Files

- `docs/04_mcp_02_01_endpoints-and-transport.md` — update /v1/tools section
- `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` — clarify field mappings
- `docs/04_mcp_06_14_new-tool-registration-procedure.md` — add /v1/tools requirements

## Background

Some documentation still describes /v1/tools as a minimal list of tool names and descriptions. The current discovery service expects richer metadata and validates more fields.

## Problem

If server authors follow the minimal documentation, new tools may fail live discovery or lose important metadata. This is because /v1/tools is no longer just an informational endpoint — it is the source used to build RuntimeToolRegistry.

## Reason for Change

/v1/tools is no longer just an informational endpoint. It is the source used to build RuntimeToolRegistry. Incorrect documentation here directly causes routing, visibility, and diagnostics problems.

## Implementation Intent

Update the /v1/tools endpoint documentation to show the current expected response shape with all required and optional fields clearly marked.

## Implementation Instructions

### Step 1: Update endpoints-and-transport.md

In the HTTP ツール一覧: /v1/tools section, replace the minimal example with a full example containing all fields:

```json
{
    "schema_version": "1",
    "tools": [
        {
            "name": "example_tool",
            "description": "Example description",
            "inputSchema": {"type": "object", "properties": {}},
            "status": "active",
            "is_write": false,
            "requires_serial": false,
            "resource_scope": null,
            "enabled": true,
            "capabilities": [],
            "server_key": "example-server",
            "config_dependent": false,
            "disabled_reason": null
        }
    ]
}
```

Clearly mark which fields are required and which are optional:

**Required per-tool fields:**
- `name`
- `description`
- `inputSchema` or `input_schema`

**Optional per-tool fields:**
- `status`
- `is_write`
- `requires_serial`
- `resource_scope`
- `enabled`
- `capabilities`

**Common top-level fields:**
- `schema_version`
- `tools`

**Server-specific fields:**
- `server_key`
- `config_dependent`
- `disabled_reason`

### Step 2: Update tool-runtime-availability-metadata.md

Clarify that /v1/tools is the source for RuntimeToolRegistry construction.

### Step 3: Update new-tool-registration-procedure.md

Add /v1/tools requirements section referencing the updated documentation in endpoints-and-transport.md.

### Step 4: Document deferred fields separately

Do NOT document deferred fields such as `disabled_code` as currently implemented. Create a separate note indicating these are deferred.

## Acceptance Criteria

1. /v1/tools docs match the validation behavior of McpToolDiscoveryService
2. Required and optional fields are clearly separated
3. New server authors can implement a valid /v1/tools endpoint from the docs alone
4. Deferred fields are clearly separated from current implementation
