# Evaluate future options for disabled tool filtering and structured disabled codes

## Priority

Low

## Target files

- docs/mcp_tool_metadata.md
- shared/runtime_tool.py
- mcp_servers/*/server.py
- tests/mcp_servers/test_tools_endpoint.py

## Reason

The initial design uses `disabled_reason` as a human-readable string and returns all tools by default.

In the future, clients may want filtered tool discovery or structured disabled reason codes.

These are not required for the initial migration, but the design decision should be tracked.

## Intent

Document future options without changing the initial default behavior.

## Scope

Evaluate whether `/v1/tools?include_disabled=false` is useful.

If implemented later:

- default must remain `include_disabled=true`
- disabled tools are omitted only when explicitly requested

Evaluate whether `disabled_code` should be added later.

Possible examples:

- `EMPTY_ALLOWED_DIRS`
- `EMPTY_ALLOWED_REPO_PATHS`
- `READ_ONLY`
- `EMPTY_COMMAND_ALLOWLIST`
- `EMPTY_WORKFLOW_ALLOWLIST`

## Acceptance criteria

- No immediate implementation is required.
- Future options are documented.
- Default `/v1/tools` behavior remains unchanged.
- The initial migration does not depend on `include_disabled` or `disabled_code`.
