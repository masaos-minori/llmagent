# Fix SQLiteHelper default vec-loading behavior for DbTarget enum inputs

## Summary
Correct `db/helper.py::SQLiteHelper.__init__()` so that `"rag"` and `DbTarget.RAG` produce identical default `load_vec` behavior.

## Background
The constructor accepts either a `DbTarget` enum or a string literal. However, `_default_load_vec` is currently derived from the original `target` argument instead of the normalized resolved value.

As a result, `SQLiteHelper(DbTarget.RAG)` can behave differently from `SQLiteHelper("rag")`.

## Objective
Normalize target handling so the default vec-loading policy is consistent regardless of input form.

## In Scope
- `helper.py`

## Required Changes
- Base `_default_load_vec` on the normalized resolved target value
- Ensure:
  - `"rag"` and `DbTarget.RAG` default to vec-loading enabled
  - `"session"` / `DbTarget.SESSION` default to vec-loading disabled
  - `"workflow"` / `DbTarget.WORKFLOW` default to vec-loading disabled
- Add regression tests if a test suite exists for `SQLiteHelper`

## Acceptance Criteria
- Enum and string target forms are behaviorally equivalent
- Rag target defaults to the same vec-loading policy in all supported call forms
- No other target behavior regresses

## Out of Scope
- Refactoring SQLiteHelper into separate subclasses
- Changing explicit `load_vec=` override semantics

## Implementation Notes
Normalize first, then branch. Do not derive default behavior from a pre-normalized input variant.
