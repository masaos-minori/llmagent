# Update plugin registry documentation and tests for structured load results

## Priority
Medium

## Labels
- plugins
- documentation
- tests
- medium-priority

## Dependencies
None

## Background
`plugin_registry.load_plugins()` already returns `PluginLoadResult`, and failure details are represented by `PluginFailure`. The previous known issue stating that plugin failures are not machine-readable is outdated.

## Scope
Modify:
- `docs/90_shared_90_inconsistencies_and_known_issues.md`
- `docs/90_shared_03_runtime_and_execution.md`
- `tests/test_plugin_registry.py`

## Required Changes
- Mark the plugin load reporting issue as resolved.
- Update documentation so `load_plugins()` is documented as returning `PluginLoadResult`, not `int`.
- Document:
  - `PluginFailure`
  - `PluginLoadResult`
  - `PluginLoadError`
  - `get_last_load_result()`
- Add regression tests for structured plugin load reporting.

## Tests
Add or update `tests/test_plugin_registry.py`.

Test:
- Missing plugin directory returns `PluginLoadResult(loaded_count=0, failed=())`.
- Syntax error plugin is included in `failed`.
- `strict_mode=True` raises `PluginLoadError`.
- Known MCP tool conflict with reject policy increments conflict counters.
- Command shadow count is reported.
- `get_last_load_result()` returns the most recent result.

## Acceptance Criteria
- Documentation matches current implementation.
- Plugin load failures are testable without parsing logs.
- The outdated known issue is no longer misleading.
