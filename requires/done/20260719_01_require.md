# [Medium] `/stats` hint points to removed `/db rag consistency`; no on-demand replacement exists

## Priority

Medium

## Summary

`/stats` prints a hint recommending `Run /db rag consistency for index integrity
status` whenever a RAG DB is configured. `/db` (and specifically `/db rag
consistency` / `/db consistency`) was removed entirely and has no successor
slash command (`docs/05_agent_07_07_cli-and-commands-migration-notes.md`).
Running the suggested command is rejected as unrecognized.

Filed from `issues/20260717-133541_stale_db_rag_consistency_hint.md`, which
identified the defect but deferred the fix because it requires a product
decision. This document confirms the implementation state and recommends a
specific resolution.

## Target files

- `scripts/agent/commands/cmd_config_stats.py` (`_cmd_stats`, line 144-147)
- `scripts/agent/services/rag_maintenance_service.py` (`RagMaintenanceService.consistency`)
- `scripts/agent/commands/db_session_ops.py` (`DbSessionOps.health` — session-only, not RAG)
- `scripts/agent/startup.py` (line 271-282, step 7 "RAG consistency")
- `tests/test_rag_consistency.py`, `tests/test_startup_consistency.py` (existing coverage for the underlying check)
- new/updated test file for `/stats` hint behavior (none exists today)

## Background

Confirmed by reading the code:

- `cmd_config_stats.py:144` emits the stale hint whenever
  `stats.rag_db_configured` is true (set by `_get_rag_db_configured`, which
  just checks that `db.config.build_db_config()` succeeds — it does not
  check whether the index is actually consistent).
- `/db` is fully removed from `_COMMANDS`; `/db consistency` has "no successor
  command" per the migration notes table.
- `/session health` (the closest-sounding existing command) operates on
  `session.sqlite` via `DbMaintenanceService`, not on the RAG DB — it is not
  a substitute.
- The underlying capability is **not actually gone**: `RagMaintenanceService.
  consistency()` still exists, wraps `db.rag_consistency.check_rag_consistency()`,
  and is exercised by real tests (`tests/test_rag_consistency.py`,
  `tests/test_startup_consistency.py`, `tests/test_rag_index_integrity.py`).
- That method is called today from exactly one place:
  `agent/startup.py:273`, as step 7 of the startup validation pipeline. It
  runs automatically on every agent launch and reports issues via
  `pipeline.add_warning("rag_consistency", ...)`, displayed by
  `_display_pipeline_results`. There is currently no way to re-run this
  check on demand mid-session — a user must restart the agent to see fresh
  RAG consistency status.

So the choice isn't a blind pick between "remove the hint" and "invent a
brand-new feature." The check already exists, is tested, and already runs —
it's just not reachable on demand.

## Reason for change

- Telling users to run a command that the REPL will reject as unknown is a
  confusing, broken UX — this is a genuine (if minor) correctness defect, not
  cosmetic.
- Silently deleting the hint (option (a) from the original issue) would
  regress discoverability: a user mid-session currently has no way to check
  RAG index integrity without restarting the agent, and removing the hint
  makes that limitation invisible instead of fixing it.
- Since `RagMaintenanceService.consistency()` already exists and is tested,
  wiring it to an on-demand command is a small, low-risk addition — not a new
  feature built from scratch.

## Intent

Make `/stats`'s hint point at something that actually works, and close the
underlying gap (no on-demand RAG consistency check) that made the hint
impossible to satisfy correctly in the first place.

## Required changes

### 1. Decide where the on-demand RAG consistency check lives

Recommended: add a `rag-consistency` action to `/session` (parallel to the
existing `stats|health|checkpoint|vacuum|purge|recover` set), backed directly
by the existing `RagMaintenanceService().consistency()` — no new service
logic required, only a new dispatch entry (`cmd_session.py`) and output
formatting (mirroring `DbSessionOps.health`'s `write_kv` style, plus the
`issues` list from `RagConsistencyResult`).

Alternative (if a new top-level namespace is preferred over overloading
`/session` with a RAG-scoped action): a dedicated `/rag consistency`
subcommand. Either is acceptable; pick one and apply it consistently with
naming used elsewhere (`/db rag <subcmd>` was scoped under `/db`, and `/db
session <subcmd>` moved to `/session`, so `/session rag-consistency` follows
existing precedent most directly).

### 2. Update the `/stats` hint string

Change `cmd_config_stats.py:146` from:

```python
"  Hint          : Run /db rag consistency for index integrity status"
```

to reference whatever command is added in step 1, e.g.:

```python
"  Hint          : Run /session rag-consistency for index integrity status"
```

### 3. Add test coverage for the hint itself

No test currently pins this hint string (confirmed: grep for
`"Run /db rag consistency"` and `rag_db_configured` across `tests/` returns
no hits referencing the hint text). Add a regression test asserting `/stats`
output references only commands that exist in `_COMMANDS`, so this class of
drift is caught automatically going forward (this could also be a small
addition to `tools/check_agent_docs_consistency.py` if that tool already
scans for known commands).

### 4. If the on-demand command is not built (fallback)

If, on reflection, no on-demand RAG consistency command is wanted, at minimum
remove the hint and replace it with accurate text, e.g.:

```python
"  Hint          : RAG index integrity is checked automatically at agent startup"
```

This avoids recommending a dead command while still telling the user where
consistency status comes from.

## Acceptance criteria

- `/stats` never recommends a command that `_COMMANDS` does not contain.
- If a new on-demand RAG consistency command is added, it reuses
  `RagMaintenanceService.consistency()` rather than duplicating query logic.
- A regression test fails if the `/stats` hint is changed to reference a
  nonexistent command again.
- No change to `agent/startup.py`'s existing automatic consistency check
  behavior (step 7) unless explicitly decided as part of this work.

## Test plan

- Unit test: `/stats` with `rag_db_configured=True` prints a hint whose
  referenced command exists in `_COMMANDS` (parse the command token out of
  the hint and assert membership).
- If `/session rag-consistency` (or equivalent) is added:
  - unit test calling the new dispatch path with a consistent RAG DB fixture
    (expect "consistent" / no issues output)
  - unit test with an inconsistent fixture (reuse fixtures/patterns from
    `tests/test_rag_consistency.py`) expecting issue lines printed
  - update `docs/05_agent_07_07_cli-and-commands-migration-notes.md`'s
    `/db consistency` row from "後継コマンドなし" to the new command name, since
    a successor would now exist
- Existing tests (`tests/test_startup_consistency.py`,
  `tests/test_rag_consistency.py`) should be unaffected since
  `RagMaintenanceService.consistency()` itself is not being changed.

## Notes for implementer

Do not change `_get_rag_db_configured`'s semantics (it only checks that a RAG
DB *path is configured*, not that the index is consistent) — that's a
separate, correctly-named concern from the consistency check itself.
