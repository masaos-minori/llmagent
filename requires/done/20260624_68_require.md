# Declare the routing source-of-truth model explicitly

## Summary
Clarify the MCP routing ownership model by making the canonical routing source explicit and reclassifying all other routing inputs as validation or compatibility layers.

## Background
Tool routing currently combines:
- live discovery map
- tool registry
- config `tool_names`
- static fallback

The priority order is documented, but the maintenance model is still too complex.

## Objective
Define which routing input is:
- authoritative
- validation-only
- compatibility-only
- fallback-only

## In Scope
- `04_mcp_03_routing_lifecycle_and_execution.md`
- `04_mcp_06_configuration_and_operations.md`

## Required Changes
- Explicitly declare the routing source of truth
- Clarify whether `tool_names` is:
  - required metadata
  - optional validation data
  - compatibility material
- Clarify the role of static fallback
- Update “new tool” and “new server” procedures so they match the final ownership model

## Acceptance Criteria
- The routing source of truth is obvious
- `tool_names` has a clearly documented role
- Static fallback has a clearly documented role
- New tool registration instructions align with the final routing model

## Out of Scope
- Refactoring routing code
- Removing fallback behavior unless separately approved

## Notes
Optimize for low maintenance overhead. The final documentation should reduce manual synchronization points.
