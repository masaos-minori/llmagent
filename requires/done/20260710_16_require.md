# [High] Improve routing drift diagnostics and document config tool_names semantics

## Background

Live `/v1/tools` responses are used for startup drift validation only. They must not affect routing decisions.

Config `tool_names` is also validation metadata only. It is not a routing input. When `tool_names` is omitted or empty, config-vs-registry validation is skipped for that server.

Current diagnostics can be improved:

- A live tool that is unknown and a live tool owned by another server can look similar.
- Duplicate live ownership should be clearly reported.
- `tool_names=[]` can be misread as saying the server owns no tools.

## Goal

Make routing drift diagnostics precise and make `tool_names` semantics unambiguous.

## Scope

- `shared/tool_registry.py`
- `shared/tool_routing_validation.py`
- `shared/route_resolver.py`
- Startup routing drift tests
- MCP routing documentation
- MCP configuration documentation
- New server and new tool documentation

## Tasks

- [ ] Update live mismatch diagnostics to distinguish unknown tools from owner mismatches.
- [ ] If a tool is known but owned by another server, include the registry owner in the diagnostic message.
- [ ] Preserve existing detection for tools in registry but missing from live response.
- [ ] Ensure duplicate live ownership is detected and reported clearly.
- [ ] Document where duplicate live ownership is detected.
- [ ] Add tests for unknown live tool.
- [ ] Add tests for live tool claimed by the wrong server.
- [ ] Add tests for duplicate live tool ownership across multiple servers.
- [ ] Clarify that `tool_names` does not participate in routing.
- [ ] Clarify that `tool_names` is only used for config-vs-registry drift validation.
- [ ] Clarify that omitted or empty `tool_names` skips config-vs-registry validation for that server.
- [ ] Clarify that routing still uses ToolRegistry.
- [ ] Add examples for omitted `tool_names`, empty `tool_names`, and populated `tool_names`.
- [ ] Update new server and new tool procedures to reference the clarified behavior.

## Acceptance Criteria

- [ ] Unknown live tools produce an explicit unknown-tool message.
- [ ] Known tools claimed by the wrong live server report the registry owner.
- [ ] Duplicate live ownership reports all claiming server keys.
- [ ] Strict mode still raises on drift.
- [ ] Non-strict mode logs warnings and returns structured warning information.
- [ ] Documentation states that `tool_names=[]` does not disable routing.
- [ ] Documentation states that `tool_names=[]` skips config-vs-registry validation.
- [ ] No implementation uses live `/v1/tools` or config `tool_names` to make routing decisions.

## Notes

Routing must remain registry-driven. Live discovery and config `tool_names` are validation-only inputs.
