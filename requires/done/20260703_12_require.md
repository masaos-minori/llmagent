# [High] Make ToolRegistry the routing authority and use live /v1/tools only for validation

## Goal

Make `ToolRegistry` the authoritative source for MCP tool ownership and routing.

Live `/v1/tools` discovery must no longer override registry-based routing. It should be used only for availability, schema, and drift validation.

## Background

The current routing model allows live discovery metadata to override `ToolRegistry` ownership. This conflicts with the design statement that `ToolRegistry` is the single source of truth for tool ownership.

Current behavior to remove:

- `ToolRouteResolver` gives live discovery higher priority than `ToolRegistry`.
- Tests currently assert that discovery can override registry routing.
- Duplicate live tool ownership is only warned about and uses first-wins behavior.

This weakens the routing authority model and may allow a misconfigured server response to change runtime routing.

## Scope

Update:

- `shared/route_resolver.py`
- `shared/tool_registry.py`
- `agent/repl_health.py`
- `tests/test_route_resolver.py`
- `tests/test_tool_registry.py`
- MCP docs:
  - `04_mcp_03_routing_lifecycle_and_execution.md`
  - `04_mcp_06_configuration_and_operations.md`
  - `04_mcp_90_inconsistencies_and_known_issues.md`

## Required Behavior

Routing authority:

- `ToolRegistry`

Validation sources:

- live `/v1/tools`
- config `tool_names`

Config `tool_names` remains drift-validation metadata only.

## Implementation Tasks

- Update `ToolRouteResolver.resolve()` so it does not route using `discovery_map`.
- Keep or rename `build_discovery_map()` as a validation helper only.
- Ensure registry routing is always used for known tools.
- Unknown tools must still raise `ValueError`.
- Wire live-vs-registry validation into startup validation.
- In strict validation mode, fail startup when:
  - live `/v1/tools` returns a tool not in the registry
  - live `/v1/tools` omits a registry-owned tool for that server
  - live `/v1/tools` reports a tool under a different server key
  - duplicate live ownership is detected
- Replace tests that assert discovery wins over registry.
- Add tests proving discovery does not override registry.
- Update documentation to remove "live discovery override" as a routing rule.

## Acceptance Criteria

- `ToolRegistry` is the only routing authority.
- live `/v1/tools` cannot change the server selected for a registered tool.
- `config.mcp_servers.*.tool_names` is not used for routing.
- Live discovery drift is reported.
- Strict mode fails startup on live-vs-registry mismatch.
- Existing registry routing tests pass.
- New tests prove live discovery cannot override registry.

## Validation

Run:

    uv run pytest tests/test_route_resolver.py tests/test_tool_registry.py -v

Also verify:

    rg -n "discovery.*override|live.*override|highest priority" docs shared tests

No active docs should describe live discovery as the routing override source.
