# Title
Refactor AgentSession to remove direct RAG-layer responsibility

## Summary
Refactor the agent layer so that `AgentSession` is responsible only for session/message/note persistence in `session.sqlite`.
Remove any direct dependency on RAG-layer tables (`documents`, `chunks`, `chunks_fts`, `chunks_vec`) from `AgentSession`.

## Background
Current documentation is inconsistent about whether `AgentSession` directly accesses RAG-layer tables.
This creates an unclear responsibility boundary between the agent layer and the RAG layer.

## Problem
`AgentSession` should not depend on the internal schema of `rag.sqlite`.
RAG document maintenance should be handled by a dedicated service boundary such as `DbMaintenanceService` or an MCP-based RAG interface.

## Required Changes
- Audit `agent/session.py` and identify all logic that directly depends on RAG-layer tables or schema details.
- Move RAG document listing / deletion / maintenance behavior out of `AgentSession`.
- Route `/db` commands that operate on RAG data through a dedicated service layer or MCP call path.
- Ensure `AgentSession` only owns session/message/note persistence responsibilities.

## Acceptance Criteria
- `AgentSession` no longer depends on RAG table names or RAG schema details.
- `/db clean`, `/db stats`, `/db urls`, and related RAG operations run through a dedicated maintenance/service boundary.
- Documentation is updated so that all agent-layer documents describe the same responsibility boundary.
- Future RAG schema refactors can be performed without modifying `AgentSession`.

## Out of Scope
- Redesigning the full RAG pipeline.
- Changing the functional behavior of RAG indexing/search itself.

## Notes
Prefer a strict separation:
- `AgentSession`: session.sqlite only
- RAG maintenance: dedicated service or MCP
