# Add runtime availability metadata to /v1/tools responses

## Priority

High

## Target files

- mcp_servers/file/read_server.py
- mcp_servers/file/write_server.py
- mcp_servers/file/delete_server.py
- mcp_servers/git/server.py
- mcp_servers/file/read_tools.py
- mcp_servers/file/write_tools.py
- mcp_servers/file/delete_tools.py
- mcp_servers/git/tools.py
- tests/mcp_servers/test_tool_schema.py
- tests/mcp_servers/test_tools_endpoint.py

## Reason

`enabled` and `disabled_reason` are runtime state and cannot be represented as static tool metadata only.

The Agent will build RuntimeToolRegistry from `/v1/tools`. Therefore, `/v1/tools` must clearly distinguish between:

- tools implemented by the server
- tools currently executable under the server's current configuration
- tools implemented but currently disabled
- the reason a tool is disabled

Disabled tools must not be omitted from `/v1/tools`, because omission makes it impossible to distinguish between "not implemented" and "implemented but disabled".

## Intent

Make `/v1/tools` return all implemented tools and express current runtime availability using `enabled` and `disabled_reason`.

## Scope

Every tool returned by `/v1/tools` must include:

- `enabled`
- `disabled_reason`

Enabled tools should return:

- `enabled: true`
- `disabled_reason: ""`

Disabled tools should return:

- `enabled: false`
- `disabled_reason: "<standard reason>"`

## Runtime rules for file servers

For file-read, file-write, and file-delete:

When `_cfg.allowed_dirs` is empty, every tool must be returned with:

- `enabled: false`
- `disabled_reason: "allowed_dirs is empty"`

When `_cfg.allowed_dirs` is non-empty, every tool must be returned with:

- `enabled: true`
- `disabled_reason: ""`

## Runtime rules for git server

When `_cfg.allowed_repo_paths` is empty, every git tool must be returned with:

- `enabled: false`
- `disabled_reason: "allowed_repo_paths is empty"`

When `_cfg.allowed_repo_paths` is non-empty and `_cfg.read_only` is true, only git write tools must be returned with:

- `enabled: false`
- `disabled_reason: "read_only=true"`

The following git write tools are affected by `read_only=true`:

- git_add
- git_commit
- git_checkout
- git_pull
- git_push

Git read tools remain enabled when `_cfg.allowed_repo_paths` is non-empty.

## Precedence rule

For git tools, `allowed_repo_paths is empty` takes precedence over `read_only=true`.

## Standard disabled_reason values

Use these exact initial values:

- `allowed_dirs is empty`
- `allowed_repo_paths is empty`
- `read_only=true`
- `command_allowlist is empty`
- `workflow_allowlist is empty`

## Acceptance criteria

- `/v1/tools` always returns all implemented tools.
- Disabled tools are not omitted.
- Every returned tool has boolean `enabled`.
- Every returned tool has string `disabled_reason`.
- Enabled tools have empty `disabled_reason`.
- Disabled tools have non-empty `disabled_reason`.
- File tools are disabled when `allowed_dirs` is empty.
- Git tools are disabled when `allowed_repo_paths` is empty.
- Git write tools are disabled when `read_only` is true and `allowed_repo_paths` is non-empty.
- Disabled reason strings match the standard values exactly.
