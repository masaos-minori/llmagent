## Summary
Add CI checks to detect malformed Markdown and stale RAG documentation patterns before they regress.

## Background
The latest docs contain malformed Markdown headings and stale issue-routing content. These documents are used as AI implementation references, so structural quality matters.

## Problem
Manual review is not enough to prevent broken headings, malformed tables, stale artifact paths, or non-canonical command names from reappearing.

## Required Changes
- Add Markdown lint or equivalent docs-render validation.
- Detect:
  - broken headings
  - malformed tables
  - unclosed inline code
  - JSON examples not wrapped in fenced code blocks
- Add forbidden/stale pattern scans for:
  - active `rag-src/*.txt`
  - active `rag-src/chunk/*.txt`
  - active `rag-src/registered/*.txt`
  - `{stem}-0000.txt`
  - `/db fts-rebuild`
- Add a lightweight known-issues consistency check:
  - no resolved issue remains under active issues
  - no stale issue ID routing remains in document guide
- Document how to run the checks locally.

## Acceptance Criteria
- Markdown structure regressions fail CI.
- Stale `.txt` artifact references fail CI unless marked historical.
- Non-canonical `/db fts-rebuild` references fail CI.
- The checks can be run locally by developers.
- Existing docs pass after fixes from dependent issues.

## Out of Scope
- Full documentation site generation.
- Heavy semantic validation of every statement.

## Components
- docs lint configuration
- CI workflow
- optional scripts such as `scripts/check_docs_consistency.py`
- `03_rag_*.md`

## AI Implementation Instruction
Implement this as a docs quality gate.

Strict rules:
- Do not make brittle checks that fail on legitimate historical notes.
- Allow explicit markers such as `legacy`, `historical`, or `archive only` where appropriate.
- Keep checks fast enough for CI.
- If a lint tool requires configuration, commit the configuration file.
