# [High] Clarify responsibility boundary between config/agent.toml and config/*_mcp_server.toml

## Labels
priority/high, area/config, area/mcp, area/docs, architecture

## Summary
Clarify the configuration ownership boundary between the agent process and MCP server-local configuration.

The confirmed specification is:

- `config/agent.toml` contains only agent-side MCP lifecycle and transport entries.
- `config/*_mcp_server.toml` contains each MCP server's own application configuration.

## Correct Ownership Model

`config/agent.toml` owns only:

- `mcp_servers.<key>.startup_mode`
- `mcp_servers.<key>.transport`
- `mcp_servers.<key>.url`
- `mcp_servers.<key>.healthcheck_mode`
- `mcp_servers.<key>.cmd`

`config/*_mcp_server.toml` owns server-local application configuration, such as:

- allowlists
- denylists
- resource limits
- audit paths
- GitHub `allowed_repos`
- GitHub `allowed_repos_mode`
- shell `command_allowlist`
- file server `allowed_dirs`

## Background
Some documentation says all configuration keys are defined in `config/agent.toml`, while other sections still describe server-local `*_mcp_server.toml` files. This creates ambiguity.

## Required Changes

- Replace overly broad statements such as “all configuration keys are defined in config/agent.toml”.
- Introduce a clear two-layer model:
  - agent process configuration
  - MCP server-local application configuration
- State that MCP servers read only their own `*_mcp_server.toml` files.
- State that `agent.toml` does not own MCP server-local policy settings.
- State that MCP server lifecycle/transport changes require full agent restart.
- Ensure process isolation documentation remains consistent.

## Acceptance Criteria
- Documentation clearly separates agent-side MCP entries from MCP server-local settings.
- MCP server-local config examples are not moved into `agent.toml`.
- `agent.toml` examples include only the confirmed MCP lifecycle/transport fields.
- Operators can determine which file to edit for any MCP-related setting.
- `/reload` vs restart-required behavior is documented consistently.
