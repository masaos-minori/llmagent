# Define an end-to-end correlation model for MCP tool-call observability

## Summary
Document how to trace one MCP tool call across agent-side execution, server-side audit logs, transport errors, health-state changes, and watchdog actions.

## Background
The MCP docs already define rich observability data, including:
- `X-Session-Id`
- `X-Request-Id`
- per-server health states
- transport vs tool error distinction
- watchdog behavior

However, they do not describe one end-to-end tracing workflow.

## Objective
Make it possible to trace one tool call from dispatch to audit log to health consequences.

## In Scope
- `04_mcp_02_protocol_and_transport.md`
- `04_mcp_03_routing_lifecycle_and_execution.md`
- `04_mcp_06_configuration_and_operations.md`

## Required Changes
- Define the canonical correlation keys, including at least:
  - `X-Session-Id`
  - `X-Request-Id`
  - `server_key`
  - `tool_name`
- Explain where each key is created and where it appears
- Add one success-path example and one failure-path example
- Clarify how tool errors differ from transport errors in tracing

## Acceptance Criteria
- A reader can trace one tool call from agent dispatch to server audit output
- The relationship between audit logs, transport errors, and health-state changes is explicit
- The documented model supports operator incident investigation

## Out of Scope
- Distributed tracing implementation
- New log storage backends

## Notes
Prefer an example-driven explanation, not just a field list.
