# Restrict runtime tool routing source of truth to live discovery and ToolRegistry

## Priority

High

## Goal

Ensure runtime tool routing uses only the approved routing sources.

Allowed runtime routing sources:

1. live discovery map
2. `ToolRegistry`

Disallowed runtime routing sources:

- `config/mcp_servers.toml` `tool_names`
- `tool_constants.py` frozensets

## Background

`ToolRegistry` is the primary source of truth for MCP tool ownership. Config `tool_names` are drift validation metadata, not runtime routing input. `tool_constants.py` is registry seed/classification data, not a runtime routing fallback.

## Scope

Affected files:

- `shared/route_resolver.py`
- `shared/tool_registry.py`
- `shared/tool_constants.py`
- `shared/tool_routing_validation.py` after rename
- tests for route resolution

## Implementation requirements

- Inspect `ToolRouteResolver`.
- Remove runtime fallback to config `tool_names`, if present.
- Remove runtime fallback to `tool_constants.py` frozensets, if present.
- Keep live discovery as highest-priority runtime routing source.
- Keep `ToolRegistry` as primary registry routing source.
- Keep config and live response comparisons as validation-only behavior.

## Acceptance criteria

- Runtime route resolution checks live discovery first.
- Runtime route resolution checks `ToolRegistry` second.
- Runtime route resolution does not use config `tool_names`.
- Runtime route resolution does not use `tool_constants.py` frozensets directly.
- Unknown tool behavior is explicit and tested.
- Drift validation remains available outside the runtime execution path.

## Validation plan

- Add or update `ToolRouteResolver` tests.
- Test live discovery override.
- Test registry fallback.
- Test unknown tool.
- Test config `tool_names` mismatch does not affect runtime route resolution.
- Test constants are not used directly by route resolver.

## Out of scope

- Changing `ToolRegistry` seed data.
- Changing live discovery protocol.
