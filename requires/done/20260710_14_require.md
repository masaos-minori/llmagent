# [High] Clarify ToolRegistry ownership responsibilities and cleanup compatibility surface

## Background

`shared/tool_registry.py` is the canonical ownership and routing registry for MCP tools. It maps each tool name to exactly one `server_key`.

However, the current documentation and code comments can be improved:

- The module docstring references outdated or non-existent function names such as `compare_registry_vs_config()`.
- ToolRegistry can be misread as a full schema registry because `ToolDefinition` has `description` and `input_schema` fields.
- Validation helpers are re-exported from `tool_registry.py` for backward compatibility, but the canonical validation module is `shared.tool_routing_validation`.
- There is a no-op `if TYPE_CHECKING: pass` block.

## Goal

Make ToolRegistry's responsibility explicit: ownership and routing only. Clarify validation helper ownership and remove misleading or obsolete documentation.

## Scope

- `shared/tool_registry.py`
- `shared/tool_routing_validation.py`
- MCP routing documentation
- Tool schema export policy documentation
- New tool registration documentation

## Tasks

- [ ] Replace outdated docstring references to `compare_registry_vs_config()` with `validate_routing_against_config()`.
- [ ] Replace outdated docstring references to `compare_registry_vs_live()` with `validate_routing_against_live()`.
- [ ] Remove or correct any reference to `compare_config_vs_live()` if no such function exists.
- [ ] Clarify that ToolRegistry is the canonical ownership and routing registry.
- [ ] Clarify that live `/v1/tools` is validation-only and must not affect routing decisions.
- [ ] Clarify that config `tool_names` is not a routing input.
- [ ] Clarify that `ToolDefinition.description` and `ToolDefinition.input_schema` are not populated by default registry population.
- [ ] Document the canonical source for LLM-visible tool schemas.
- [ ] Add a future-use comment for `description` and `input_schema`, or remove them if they are not needed.
- [ ] Add a comment near validation helper re-exports explaining they are backward compatibility aliases.
- [ ] Recommend importing validation helpers from `shared.tool_routing_validation` in new code.
- [ ] Decide whether a removal timeline is needed for compatibility re-exports.
- [ ] Remove the no-op `if TYPE_CHECKING: pass` block if still present.
- [ ] Search existing imports and optionally migrate internal callers to the canonical validation module.

## Acceptance Criteria

- [ ] ToolRegistry docstring references only existing APIs.
- [ ] Documentation clearly states that ToolRegistry owns routing and ownership only.
- [ ] Documentation no longer implies that ToolRegistry is the full schema source of truth.
- [ ] New code has a clear canonical import path for routing validation helpers.
- [ ] Backward compatibility remains intact unless an explicit migration is approved.
- [ ] No runtime behavior changes are introduced.

## Notes

This issue is primarily documentation and maintainability cleanup. It prevents AI agents and maintainers from calling non-existent APIs or misusing ToolRegistry as a schema registry.
