# Title
Make `/db` command targets explicit by database scope

## Summary
Restructure or clarify `/db` commands so users can immediately tell whether an operation targets RAG, session, or workflow storage.

## Background
The system uses multiple SQLite databases, but `/db` command descriptions blur the distinction between them.

## Problem
Operators should not need implicit implementation knowledge to understand which database a command affects.

## Required Changes
- Clarify or redesign `/db` commands so the target database is explicit.
- Consider scoped forms such as:
  - `/db rag ...`
  - `/db session ...`
  - `/db workflow ...`
- Update help text and documentation accordingly.
- Preserve the distinction between normal behavior and admin-only maintenance access.

## Acceptance Criteria
- Users can tell which DB a `/db` command affects without hidden knowledge.
- Help text and docs match actual behavior.
- Maintenance-access exceptions remain clearly described.

## Out of Scope
- Replacing SQLite
- Rebuilding the full maintenance subsystem

## AI Implementation Instruction
Implement this as an explicitness and operator-clarity task.

Strict rules:
- Do not hide database scope behind shorthand behavior.
- Preserve existing maintenance capabilities unless clearly deprecated.
- Make admin-only access semantics explicit.
- If there is uncertainty about backward-compatible command naming, mark it as: Needs confirmation

Deliverables:
1. Clearer `/db` command structure or semantics
2. Updated help text and docs
3. Explicit database targeting behavior
