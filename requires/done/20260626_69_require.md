# Title
Add startup safety guard for unauthenticated public Event Bus binding

## Priority
Medium

## Labels
- enhancement
- eventbus

## Dependencies
- Issue 8

## Summary
Add a startup safety guard to reduce the risk of exposing the unauthenticated Event Bus API on public or LAN interfaces.

## Background
The Event Bus has no authentication or ACL. Docs state that access control is enforced at the network boundary and recommend binding to `127.0.0.1`.

## Problem
If the server binds to `0.0.0.0` by mistake, the unauthenticated API may become reachable from the local network or the internet.

## Required Changes
- Detect public bind addresses such as `0.0.0.0` or `::`.
- Emit a clear startup warning or fail-fast by default.
- Consider explicit override such as `allow_public_bind=true`.
- Log:
  - no authentication enabled
  - bind address
  - whether public bind is allowed
- Update deployment docs with reverse proxy + authentication guidance.

## Acceptance Criteria
- Unsafe public binding is visible at startup.
- Public binding requires explicit acknowledgement or fails fast.
- Docs clearly state that the Event Bus must not be exposed directly.
- Tests cover safe bind, unsafe bind, and explicit override.

## Out of Scope
- Implementing API-key auth.
- Implementing mTLS.
- Adding ACLs.

Implement this as a startup safety guard.
