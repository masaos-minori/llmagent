# [Medium] Clarify ToolExecutor transport, health, lifecycle, and retry contracts

## Background

`HttpTransport` has explicit retry and error-boundary behavior:

- retryable statuses: `429`, `502`, `503`, `504`
- retry max: 3
- timeout: non-retryable
- tool-level `is_error=True`: returned as a tool result, not a transport failure
- malformed response / request error / HTTP status failure: transport error

`ToolExecutor` also has lifecycle and health-gate behavior, and a duplicated `LifecycleProtocol` definition exists between `tool_executor.py` and `tool_lifecycle.py`.

`repeated_tool_error_threshold` is stored by ToolExecutor but does not appear to drive behavior.

## Goal

Make transport, health, lifecycle, and retry contracts explicit and reduce ambiguity.
