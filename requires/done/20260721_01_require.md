# [Critical] Fix remaining stale `cmd` paths in `config/agent.toml` for 6 MCP servers

## Priority

Critical

## Summary

`config/agent.toml`'s `[mcp_servers.<name>]` `cmd` entries for **6 servers** — `shell`, `git`, `github`,
`cicd`, `rag_pipeline`, `mdq` — still launch a bare `server.py` file that was deleted by an earlier
rename cleanup (commit `abcf0820`). Every one of these servers uses `startup_mode = "subprocess"`, so
each of these 6 `cmd` entries will fail with `FileNotFoundError` the moment
`scripts/agent/http_lifecycle.py::HttpLifecycleManager.start()` tries to launch it — the server never
comes up.

This is a re-filing of `issues/20260720-070458_mcp_server_agent_toml_cmd_points_to_deleted_files.md`,
which originally identified **8** affected servers. Since that issue was filed:

- `web_search`'s entry was already fixed (as a side effect of a later, unrelated browser-mcp merge) —
  confirmed: `config/agent.toml`'s `web_search` block now correctly reads
  `.../web_search/web_search_server.py`.
- `browser` no longer exists as a separate server (merged into `web_search` in the same change) — its
  entry was removed entirely, so it is no longer part of this gap.
- The other 6 servers were never touched and remain broken today. Verified directly:

```
config/agent.toml:292  cmd = [..., "scripts/mcp_servers/shell/server.py"]         (deleted; live file: shell_server.py)
config/agent.toml:304  cmd = [..., "scripts/mcp_servers/git/server.py"]           (deleted; live file: git_server.py)
config/agent.toml:357  cmd = [..., "scripts/mcp_servers/github/server.py"]        (deleted; live file: github_server.py)
config/agent.toml:366  cmd = [..., "scripts/mcp_servers/cicd/server.py"]          (deleted; live file: cicd_server.py)
config/agent.toml:375  cmd = [..., "scripts/mcp_servers/rag_pipeline/server.py"]  (deleted; live file: rag_pipeline_server.py)
config/agent.toml:387  cmd = [..., "scripts/mcp_servers/mdq/server.py"]           (deleted; live file: mdq_server.py)
```

`deploy/deploy.sh` mirrors `scripts/` to `/opt/llm/scripts/` with `rsync --delete`, so this is not just a
repo-state issue — the next production deploy removes any previously-deployed bare `server.py`, breaking
these 6 servers live.

## Reason for change

Every subprocess-mode MCP server that cannot start is a hard production outage for that server's tools
(`shell_run`; `git_status`/`git_commit`/etc.; all `github_*` tools; `trigger_workflow`/`get_workflow_*`;
`rag_run_pipeline`/`rag_debug_pipeline`/etc.; all MDQ tools `search_docs`/`get_chunk`/`outline`/
`index_paths`/`refresh_index`/`stats`/`grep_docs`). This is not a hypothetical risk — the referenced
files are confirmed absent from disk today.

## Intent

After this change:

- All 6 remaining `[mcp_servers.<name>]` `cmd` entries point at their actual `<name>_server.py` file.
- A regression test fails if any future rename/cleanup reintroduces a `cmd` path pointing at a
  nonexistent script, for every subprocess-mode server, not just these 6.
- `issues/20260720-070458_mcp_server_agent_toml_cmd_points_to_deleted_files.md`'s status is updated to
  resolved.

## Required changes

### 1. Fix `config/agent.toml`

Change only the final path segment of each of the 6 `cmd` arrays listed above, from `<name>/server.py`
to `<name>/<name>_server.py`. No other field (`startup_mode`, `url`, `auth_token`, `role`, `tool_names`)
changes.

### 2. Add the regression test

Per `implementations/20260720-134218_test_mcp_server_cmd_paths.py.md`'s procedure (re-verify against
current source first): build the real config via `agent.config_builders.build_agent_config()`, filter to
`startup_mode == StartupMode.SUBPROCESS` entries, and assert each `cmd[-1]` (mapped from the
`/opt/llm/scripts/` deploy prefix back to this repo's own `scripts/` directory) exists as a file.

### 3. Final repo-wide gate

`grep -n 'models\.py\|tools\.py\|service\.py\|/server\.py"' config/agent.toml` must return zero matches
after the fix — confirms no stale bare-filename reference remains anywhere in the file, not just the 6
known entries.

### 4. Close out the tracking issue

Update `issues/20260720-070458_mcp_server_agent_toml_cmd_points_to_deleted_files.md`'s `## Status`
section to resolved, per `implementations/20260720-134259_...md.md`'s procedure.
