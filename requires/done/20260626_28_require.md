# [Medium] Define MDQ tool error taxonomy and error handling

## Priority
Medium

## Labels
`mcp`, `mdq`, `errors`, `observability`, `reliability`

## Dependencies
- Align mdq-mcp with MCPServer base class behavior

## Background
Production MCP operations require clear distinction between tool-level errors and transport/server errors. MDQ should not convert all domain failures into HTTP 500 responses.

## Task
Create explicit MDQ domain exceptions and map them to appropriate MCP tool responses.

## Implementation Requirements
Add domain error classes such as:
- `MdqValidationError`
- `MdqAuthorizationError`
- `MdqNotFoundError`
- `MdqIndexNotReadyError`
- `MdqDatabaseError`
- `MdqConsistencyError`

Define behavior:
- Validation errors return tool error.
- Not found returns tool error.
- Unauthorized path returns tool error.
- DB unavailable may affect health.
- Unexpected exceptions are logged and handled consistently.

## Acceptance Criteria
- Tool-level errors return `is_error=true`.
- Transport-level failures remain distinguishable.
- Logs include error kind.
- Tests cover validation, not found, unauthorized path, and DB failures.
