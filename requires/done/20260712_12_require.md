# [Docs][全体] front matter の related/source が分割前の旧ファイル名を参照している(壊れたリンク、13グループ)

## 背景

docsディレクトリでは、肥大化したファイルを `-partN` や `_NN` 形式の複数ファイルに分割する
運用が行われている(例: commit `49154b4a` "docs: split oversized docs into part1/part2, clean up
stale implementation-doc duplicates")。しかし分割後、分割元ファイル自身のfront matter
(`related:` / `source:`)が、分割によって存在しなくなった旧ファイル名を参照したまま残って
いるケースが多数見つかった。

対照的に、各ドメインの索引的ドキュメント(例: `01_overview.md`)は分割後の実ファイル名に
正しく更新されている。front matter のメタデータ更新だけが漏れている。

## 目的

front matter の `related:` / `source:` が実在するファイルのみを参照するように修正し、
ドキュメント間ナビゲーション・RAGインデックス・関連ツールでの参照解決を正しくする。
再発防止のため、機械的な実在性チェックも追加する。

既存の整合性チェックツールはこの種の壊れたリンクを検出しない:
- `tools/validate_docs_structure.py` の `check_front_matter()`(53〜67行)は `related`
  フィールドの**存在**のみを検証し、その値(参照ファイル名)が実在するかは検証していない。
- 一方、同ファイルの `check_links()`(86〜95行)は本文中の `[text](file.md)` 形式リンクを
  `resolved.is_file()` で実在性検証している。front matter の `related:`/`source:` には
  この検証ロジックが適用されていない(検出漏れの原因)。
- `check_docs_consistency.py` はRAG固有の文字列パターン検査(stale artifact path等)が
  スコープであり、front matter/一般リンクの実在性検証は対象外。
- 本文中の「## Related Documents」節はバッククォート表記(`` `file.md` ``)でリンク形式で
  ないため、`check_links()` の正規表現(`LINK_RE`)にもマッチしない。front matterの
  `related:`と同様、実在性検証の対象外になっている。

置き換え方針の前例: 既に正しく分割済みの `03_rag_02_02_ingestion_pipeline-crawler-part1.md` /
`-part2.md` は、front matterで死んだ旧ファイル名を参照し直しておらず、関連性のある実在ファイルへの
参照に置き換えている。複数パートがある場合の `source:` はpart1のファイル名を代表として使う
規約になっている(両ファイルとも `source: ...crawler-part1.md`)。

## 提案設計

1. 上記13グループについて、各参照元ファイルの `related:`(該当する場合 `source:`)から、
   存在しないファイル名を削除する。無条件の「実ファイル名一覧への置き換え」は行わず、
   `03_rag_02_02_ingestion_pipeline-crawler-part1/2.md` の前例に倣い、削除のみを基本とする
   (死んだ参照先が指していた話題が今も文書内で重要なら、関連する実在ファイルへの参照に
   個別に差し替えてもよいが、機械的な一括置換はしないこと)。
   - `source:` フィールドを複数パートで揃える場合は、part1 のファイル名を代表として使う
     (crawler-part1/2の規約に合わせる)。
   - 各ドメインの索引ドキュメント(`01_overview.md`、`03_rag_00_document-guide.md`、
     `04_mcp_00_document-guide.md`、`05_agent_00_document-guide.md` 等)は既に正しい実ファイル名
     一覧を保持しているため、迷った場合の正として参照する。
2. `03_rag_91_design_notes-part1.md` / `-part2.md` で `03_rag_91_design_notes.md` が
   重複記載されている箇所は、重複除去も合わせて行う。
3. 再発防止のため、`tools/validate_docs_structure.py` に、front matter `related:`/`source:` の
   参照先ファイルが `docs/` 配下に実在するかを検証するチェックを新規追加する
   (`check_links()` と同じ `resolved.is_file()` パターンを再利用する)。
   - 重大度はまず WARNING とする(`check_mcp_docs_consistency.py` の `check_tool_counts()` が
     採用している "Reports WARNING (not ERROR) to avoid brittleness" の方針に倣う)。

## 効果

- ドキュメント間ナビゲーションの信頼性向上。
- 将来同様の分割作業を行った際に、チェックで漏れを機械的に検出できるようになる。

## リスク

| リスク | 可能性 | 対策 |
|---|---|---|
| 死んだ参照を削除するだけで済むかは全13グループ共通ではない可能性がある | 低 | crawler-part1/2の前例(削除のみ、置き換えなし)を基本方針とし、個別差し替えは内容が明確に必要な場合のみに限定する |

## 検証方法

```bash
uv run python tools/check_docs_consistency.py
uv run python tools/validate_docs_structure.py

# front matter 参照先実在性の手動再検証(修正前後で比較)
python3 - <<'PY'
import re, os, glob
existing = set(os.path.basename(f) for f in glob.glob("docs/*.md"))
for path in sorted(glob.glob("docs/*.md")):
    text = open(path, encoding="utf-8").read()
    m = re.match(r"^---\n(.*?)\n---\n", text, re.S)
    if not m:
        continue
    for ref in re.findall(r"-\s*([A-Za-z0-9_\-]+\.md)", m.group(1)):
        if ref not in existing:
            print(path, "->", ref)
PY
```

## AI実装者への注記

- 分割元ファイル(実体が既に存在しない旧ファイル名)自体を新規作成する対応は取らないこと。
  あくまで参照側の front matter から死んだ参照を削除する。
- `03_rag_02_02_ingestion_pipeline-crawler-part1.md` / `-part2.md` を前例として参照すること。
  死んだ参照は削除するだけでよく、無理に置き換え先を探して機械的に埋める必要はない。
- 新規チェックは `tools/check_docs_consistency.py` ではなく `tools/validate_docs_structure.py`
  に追加すること(後者は既に `check_links()` で本文リンクの実在性を検証しており、スコープが
  一致する)。
- 各ドメインの `*_00_document-guide.md` や `01_overview.md` など索引ドキュメントの記載を
  正として突き合わせること。
