# Title
Unify config loading, reload behavior, and restart-required semantics

## Summary
Create a single source of truth for configuration loading and runtime reload behavior so operators and AI systems can reliably know what changes apply immediately and what requires restart.

## Background
Configuration semantics are currently spread across multiple documents.
Some settings are hot-reloadable, some require restart, and some are partially reloadable.

## Problem
This creates unnecessary ambiguity and increases the chance of incorrect operational assumptions.

## Required Changes
- Classify important settings into:
  - startup-only
  - hot-reloadable
  - restart-required
  - partially reloadable
- Present this classification in one authoritative table.
- Make `/reload` output clearly report:
  - applied
  - deferred
  - restart required
  - ignored / unchanged
- Clarify the ownership and behavior of `common.toml`.

## Acceptance Criteria
- Operators can determine reload behavior from one place.
- `/reload` output is explicit and actionable.
- Documentation and runtime behavior match.

## Out of Scope
- Complete redesign of configuration file structure

## AI Implementation Instruction
Implement this as a configuration clarity and operator-experience task.

Strict rules:
- Do not redesign the full config model unless necessary.
- Focus on clarifying runtime semantics, not adding new features.
- Do not leave overlapping, contradictory descriptions in multiple docs.
- If a config field’s runtime semantics remain uncertain, mark it as: Needs confirmation

Deliverables:
1. One authoritative reload behavior matrix
2. Improved `/reload` output semantics
3. Updated docs aligned with runtime behavior
