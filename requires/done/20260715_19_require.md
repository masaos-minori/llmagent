# Normalize shared tool constants and side-effect helper behavior

## Priority

High

## Type

Implementation / Refactoring / Bug Fix / Testing

## Target Files

- `shared/tool_constants.py`
- `shared/tool_executor_helpers.py`
- `tests/test_tool_executor_helpers.py`
- `tests/test_tool_constants.py` if available

## Summary

Normalize shared MCP tool classification and fix helper behavior so that side-effect detection, scheduling, caching, approval, and diagnostics all use consistent definitions.

This issue consolidates several closely related cleanup tasks in shared tool classification code.

## Background

The project already centralizes many MCP tool names in `shared/tool_constants.py`. This is the right direction, but some tool families are still too coarse-grained.

Currently, these aggregate sets exist:

- `CICD_TOOLS`
- `RAG_TOOLS`
- `MDQ_TOOLS`

However, they do not distinguish read-only tools from write/admin tools.

As a result, downstream code has to guess which tools have side effects, or it misses them entirely. This directly affects:

- ToolExecutor cache safety
- scheduler serialization
- approval and risk policy
- dry-run policy
- diagnostics
- tests

There are also helper implementation bugs:

- `tool_hash_key()` is documented as hashing the tool name and args, but the implementation does not include the tool name.
- `format_transport_error()` builds a summary without including the actual field values expected by tests.
- `tool_constants.py` docstring incorrectly suggests that live `/v1/tools` discovery participates in runtime routing.
- `WRITE_TOOLS` means file write tools, but the name can be mistaken for all write tools.

## Why This Needs to Be Fixed

The same tool classification must drive all safety-sensitive behavior. If cache bypass, scheduler serialization, approval checks, and risk classification each infer side effects differently, the system becomes inconsistent.

Examples of current or likely inconsistencies:

- `trigger_workflow` is operationally side-effecting but may not be included in side-effect detection.
- `rag_delete_document` mutates the RAG index but may be treated like a normal RAG query.
- `fts_rebuild` mutates the MDQ FTS index but may not be treated as a write/admin operation.
- Tests expect `tool_hash_key()` to distinguish tool names, but implementation ignores them.
- Transport error summaries lose useful information even though the detail JSON contains it.

## Fix Intent

The intent is to make shared tool classification the single source for side-effect-related behavior.

After this change:

- Tool families should expose read/write/admin subsets.
- `is_side_effect()` should cover all known mutation tools.
- Helper functions should match their documented behavior and tests.
- Naming and docstrings should reduce future misuse.

## Required Changes

### 1. Add read/write/admin classifications

Add explicit classifications for CICD, RAG, and MDQ tools.

Recommended additions:

- `CICD_WRITE_TOOLS`
- `CICD_READ_TOOLS`
- `RAG_WRITE_TOOLS`
- `RAG_READ_TOOLS`
- `MDQ_WRITE_TOOLS`
- `MDQ_ADMIN_TOOLS`
- `MDQ_READ_TOOLS`

Recommended initial classification:

- `CICD_WRITE_TOOLS`: `trigger_workflow`
- `RAG_WRITE_TOOLS`: `rag_delete_document`
- `MDQ_WRITE_TOOLS`: `index_paths`, `refresh_index`, `fts_rebuild`
- `MDQ_ADMIN_TOOLS`: `fts_consistency_check`, `fts_rebuild`

### 2. Expand `_SIDE_EFFECT_TOOLS`

Update `_SIDE_EFFECT_TOOLS` in `shared/tool_executor_helpers.py` to include:

- `CICD_WRITE_TOOLS`
- `RAG_WRITE_TOOLS`
- `MDQ_WRITE_TOOLS`

At minimum, these must return `True` from `is_side_effect()`:

- `trigger_workflow`
- `rag_delete_document`
- `index_paths`
- `refresh_index`
- `fts_rebuild`

### 3. Fix `tool_hash_key()`

Include the tool name in the hashed input.

Required behavior:

- Same tool name and same args produce the same key.
- Different tool names with the same args produce different keys.

### 4. Fix `format_transport_error()`

The one-line summary must include:

- `source`
- `phase`
- `kind`
- `status_code`
- `retryable`
- `partial`

The `detail` field should remain JSON-encoded and retain all fields.

### 5. Correct routing authority wording

Update `shared/tool_constants.py` docstring so it clearly states:

- ToolRegistry is the sole runtime routing authority.
- Live `/v1/tools` discovery is for startup validation and drift detection only.
- Config `tool_names` are validation metadata, not routing inputs.

### 6. Add `FILE_WRITE_TOOLS` alias

Add:

- `FILE_WRITE_TOOLS = WRITE_TOOLS`

This preserves compatibility while making the file-specific meaning explicit.

## Acceptance Criteria

- CICD, RAG, and MDQ tools have explicit read/write/admin classifications.
- `is_side_effect("trigger_workflow")` returns `True`.
- `is_side_effect("rag_delete_document")` returns `True`.
- `is_side_effect("index_paths")` returns `True`.
- `is_side_effect("refresh_index")` returns `True`.
- `is_side_effect("fts_rebuild")` returns `True`.
- Read-only tools remain non-side-effecting.
- `tool_hash_key("tool_a", args) != tool_hash_key("tool_b", args)` when args are identical.
- `format_transport_error()` summary includes source, phase, kind, status code, retryable, and partial values.
- `tool_constants.py` no longer implies live `/v1/tools` is used for runtime routing.
- Existing imports of `WRITE_TOOLS` continue to work.
- `FILE_WRITE_TOOLS` is available as an alias.

## Suggested Tests

- `test_trigger_workflow_is_side_effect`
- `test_rag_delete_document_is_side_effect`
- `test_mdq_write_tools_are_side_effects`
- `test_read_only_tools_are_not_side_effects`
- `test_tool_hash_key_differs_for_different_tool_names`
- `test_format_transport_error_summary_includes_all_fields`
- `test_file_write_tools_alias_matches_write_tools`
- `test_tool_constants_routing_docstring_does_not_reference_live_routing`

## Non-Goals

- Do not change runtime routing behavior.
- Do not remove existing aggregate sets such as `RAG_TOOLS`, `CICD_TOOLS`, or `MDQ_TOOLS`.
- Do not rename `WRITE_TOOLS` in a breaking way.

## Implementation Notes

Prefer adding more precise constants while preserving existing compatibility.

Downstream code should import shared constants rather than redefining local tool-name sets.
