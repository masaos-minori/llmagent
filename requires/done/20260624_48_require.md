# Unify schema-initialization and migration strategy across rag/session/workflow DBs

## Summary
Bring `rag`, `session`, and `workflow` schema initialization under one coherent strategy.

## Background
`create_schema.py` initializes rag and session schemas, while `workflow_schema.py` uses a separate initialization path and separate ad hoc migration handling. This creates multiple schema lifecycle patterns in the same DB layer.

## Objective
Standardize how schemas are:
- initialized
- kept idempotent
- migrated incrementally
- logged and error-handled

## In Scope
- `create_schema.py`
- `workflow_schema.py`
- `helper.py` if shared DB-opening policy should be reused during workflow init

## Required Changes
- Define one schema lifecycle approach for all three DBs
- Align workflow schema init with the rag/session pattern where appropriate
- Consolidate or standardize migration handling
- Ensure schema creation entrypoints are discoverable and consistent

## Acceptance Criteria
- Rag/session/workflow schema setup follows the same conceptual lifecycle model
- Migration logic is not split into inconsistent one-off patterns without explanation
- Developers can identify one standard approach for DB schema evolution

## Out of Scope
- Full DB migration framework introduction
- Changing workflow table design

## Implementation Notes
The goal is consistency of lifecycle policy, not necessarily one giant file. The design should make it obvious where schema initialization and incremental evolution belong for each DB.
