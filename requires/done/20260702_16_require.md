# [Medium] Strengthen RAG quality regression tests beyond embed-failure checks

## Priority
Medium

## Background
`tests/test_rag_quality_regression.py` exists, but the fetched test primarily verifies the embed-failure path where no embed server is configured and `reranked == []`. This is useful, but it does not validate retrieval quality, expected source ranking, or RRF scoring behavior.

Evidence:
- `tests/test_rag_quality_regression.py` exists and currently checks embed failure diagnostics.
- The query pipeline docs describe RRF, no-RRF behavior, semantic cache, and retrieval quality tradeoffs.

## Goal
Add deterministic quality assertions for RAG retrieval behavior.

## Scope
Affected files:
- `tests/test_rag_quality_regression.py`
- `docs/03_rag_03_query_pipeline.md`

## Implementation Tasks
- Add a normal retrieval fixture with fixed documents and fixed embeddings.
- Assert expected URL or chunk appears in top N.
- Assert `use_rrf=True` produces non-zero `rrf_score`.
- Assert `use_rrf=False` produces `rrf_score == 0.0`.
- Assert semantic cache hit returns the same context string on repeated query.
- Keep existing embed failure regression test.

## Open Questions / Unknowns
- Need to confirm current test utilities for in-memory rag.sqlite schema.
- Need to decide whether to mock `RagRepository` or build a real in-memory DB.
- Need stable fixture content that does not depend on external embed service.

## Acceptance Criteria
- Test fails when RRF scoring is disabled accidentally.
- Test fails when expected document is no longer retrieved.
- Existing embed failure behavior remains covered.

## Validation
- Run `uv run pytest tests/test_rag_quality_regression.py -v`.
