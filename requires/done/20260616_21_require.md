# [Agent][Data Layer] Move RAG DB operations out of AgentSession

## Summary
`AgentSession` currently owns session persistence, but it also accesses RAG-layer tables for `/db` commands. This mixes agent-layer persistence with RAG-layer data ownership.

## Problem
The docs say that `session.sqlite` is owned by the agent layer, while `rag.sqlite` is owned by the RAG layer.
However, `AgentSession` still performs RAG-related operations such as document cleanup and stats access.
This creates a responsibility boundary leak and makes future refactoring harder.

## Scope
- `agent/session.py`
- `/db` command handlers
- RAG-related DB access paths

## Task
- remove direct RAG-table operations from `AgentSession`
- move RAG document cleanup and stats access behind a RAG-facing service or MCP boundary
- keep `AgentSession` responsible only for sessions, messages, and notes

## Expected Behavior
- the agent layer does not directly own RAG DB operations
- RAG data access is handled by a dedicated RAG service boundary
- responsibility boundaries are cleaner and easier to maintain

## Acceptance Criteria
- [ ] `AgentSession` no longer deletes or inspects RAG-layer tables directly
- [ ] `/db clean` and related commands use a RAG-facing service path
- [ ] docs and implementation agree on ownership boundaries
