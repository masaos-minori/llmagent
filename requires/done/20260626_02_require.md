## Summary
Resolve the inconsistency between diagnostic persistence in `messages` and diagnostic persistence in a separate diagnostics store.

## Background
Current docs describe diagnostics in conflicting ways.
One document states that diagnostics are stored in a separate `session_diagnostics` path/table.
Another document states that diagnostic records are stored as `role="diagnostic"` rows in `messages`.
Partial completion handling also references `save_diagnostic()` in a way that does not fully resolve the ambiguity.

## Problem
It is unclear where diagnostic data actually lives, how it is restored, and whether it is excluded from normal session restore.
This ambiguity affects session reload, partial completion handling, and operator expectations.

## Required Changes
- Choose one canonical diagnostic persistence model.
- Align:
  - `save_diagnostic()`
  - `DiagnosticStore`
  - `fetch_messages()`
  - partial completion persistence
  - session restore behavior
- Update all affected docs to match the implemented model.
- Add tests that verify the chosen persistence and restore behavior.

## Acceptance Criteria
- Diagnostic persistence is explained in one consistent way across docs and code.
- Session restore behavior is unambiguous.
- Partial completion and guard-hint persistence follow the same documented model.

## Out of Scope
- Full redesign of the observability subsystem
- Changing what counts as a diagnostic event
