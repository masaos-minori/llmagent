# Agent Startup Processing — Workflow Task Created After LLM Turn May Lose Context

## Summary

In `_handle_workflow_engine()`, the workflow task is initialized inside the turn handler, not at startup. If the workflow engine fails mid-turn after creating the task, the task record exists in the database but has no associated workflow execution context. The `_deactivate_workflow()` method resets `ctx.workflow.active = False` but doesn't clean up orphaned task records.

## Severity

Medium

## Files

- `scripts/agent/orchestrator.py:184-234`

## Details

In `Orchestrator._handle_workflow_engine()`:

```python
async def _handle_workflow_engine(self, line: str, ctx: AgentContext, turn_started_at: float) -> None:
    assert self._workflow_def is not None
    session_id = _format_session_id(ctx.session.session_id) or "none"
    store = StateStore()
    answer: str = ""
    error_kind: str | None = None
    is_partial: bool = False
    try:
        (workflow_id, task) = self._init_workflow_task(ctx, session_id, ctx.turn.pending_approval_task_id)
        ctx.turn.pending_approval_task_id = None
        self._activate_workflow(ctx, task)
        engine = WorkflowEngine(self._workflow_def, store, tracer=self._tracer)
        
        async def plan_fn() -> str | None: ...
        async def execute_fn() -> str | None: ...
        async def verify_fn() -> str | None: ...
        
        await engine.run(task, plan_fn, execute_fn, verify_fn)
    except WorkflowPendingApprovalError as exc:
        self._handle_workflow_approval_pending(exc, session_id)
    except WorkflowHaltError as exc:
        self._handle_workflow_halt(exc)
    finally:
        self._deactivate_workflow(ctx)
        store.close()
```

If `engine.run()` fails after `execute_fn()` completes (e.g., due to a tool execution failure), the task record is persisted in the database via `audit_stage_completed()` called within `execute_fn()`. However, `_deactivate_workflow()` only clears in-memory state:

```python
def _deactivate_workflow(self, ctx: AgentContext) -> None:
    ctx.workflow.active = False
    ctx.workflow.current_task_id = None
    ctx.workflow.workflow_id = None
```

It does not update the task status in the database. On the next turn, `_init_workflow_task()` will create a new task record, leaving the previous one in an inconsistent state (status = "execute" but no active workflow). This can lead to:

1. Orphaned task records accumulating over time
2. Incorrect workflow tracking in `/stats` output
3. Potential confusion when reviewing audit logs

## Reproduction Steps

1. Configure a workflow with multiple stages
2. Trigger a turn that causes a tool execution failure during the "execute" stage
3. Check the workflow database for task records — observe orphaned tasks with status="execute" but no corresponding workflow_id on subsequent turns

## Expected Behavior

When a workflow engine fails, the current task's status should be updated to reflect the failure (e.g., status = "failed"), preventing orphaned records and ensuring accurate workflow tracking.

## Proposed Fix

Update the task status in the finally block before deactivating:

```python
finally:
    # Update task status before deactivating to prevent orphaned records
    try:
        if task.task_id:
            store.update_task_status(task.task_id, "failed" if error_kind else "completed")
    except Exception as e:
        logger.warning("Failed to update task status on engine failure: %s", e)
    self._deactivate_workflow(ctx)
    store.close()
```

Alternatively, ensure `WorkflowEngine.run()` always updates the task status before raising exceptions, rather than relying on the caller to do so.
