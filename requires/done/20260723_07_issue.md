# Agent Startup Processing — Missing Error Handling for Session Start Failure

## Summary

If `ctx.session.start()` raises an exception (e.g., DB connection error, schema missing), the exception propagates through `_run_repl_loop` → `run()` → `StartupOrchestrator.run()` rollback logic, potentially leaving resources in inconsistent state. The session_id check at line 419-423 happens after `start()` succeeds, so it cannot protect against this case.

## Severity

Medium

## Files

- `scripts/agent/repl.py:413-431`

## Details

In `AgentREPL._run_repl_loop()`:

```python
async def _run_repl_loop(self) -> None:
    ctx = self._ctx
    try:
        self._print_startup_banner()
        ctx.session.start()  # Can raise sqlite3.Error or RuntimeError
        if (
            ctx.services_required.tools is not None
            and ctx.session.session_id is not None
        ):
            ctx.services_required.tools.set_session_id(str(ctx.session.session_id))
        await self._repl_loop()
    except RuntimeError as e:
        self._view.write_fatal(str(e))
        raise
    finally:
        self._persist_session_diagnostics(ctx)
        await self._persist_session_memories(ctx)
        await self._close_resources()
```

`session.start()` can raise:
- `sqlite3.Error` — DB connection failure
- `RuntimeError` — "Session schema missing" (line 103-105 in session.py)

When this happens, the exception propagates through the `except RuntimeError` block, which writes a fatal message and re-raises. The `finally` block runs `_close_resources()` which calls `svc.lifecycle.shutdown_all()` and `svc.http.aclose()`. However, if the DB was already partially initialized (e.g., WAL checkpoint at line 260-264 fails), the final state depends on timing.

More critically, if `session.start()` fails because the schema is missing, the operator sees a fatal error but the root cause ("Run: bash deploy/init_db.sh") is buried in the exception chain. The error message is opaque without reading the source code.

## Reproduction Steps

1. Stop the agent database or remove the session table
2. Start the agent REPL
3. Observe the error message — it shows "Session schema missing" but does not clearly indicate how to fix it

## Expected Behavior

The error message should clearly guide the operator to run `deploy/init_db.sh` without requiring them to read the source code. Additionally, the shutdown sequence should handle partial initialization gracefully.

## Proposed Fix

Add explicit handling for the schema-missing case before entering the REPL loop:

```python
async def _run_repl_loop(self) -> None:
    ctx = self._ctx
    try:
        self._print_startup_banner()
        try:
            ctx.session.start()
        except RuntimeError as e:
            if "no such table" in str(e):
                self._view.write_fatal(
                    "Session schema missing. Run: bash deploy/init_db.sh to initialize the database."
                )
            else:
                self._view.write_fatal(str(e))
            raise
        if (
            ctx.services_required.tools is not None
            and ctx.session.session_id is not None
        ):
            ctx.services_required.tools.set_session_id(str(ctx.session.session_id))
        await self._repl_loop()
    except RuntimeError as e:
        self._view.write_fatal(str(e))
        raise
    finally:
        self._persist_session_diagnostics(ctx)
        await self._persist_session_memories(ctx)
        await self._close_resources()
```
