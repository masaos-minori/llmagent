# [High] Strengthen MDQ authorization and configuration safety

## Priority

High

## Summary

Strengthen MDQ's runtime authorization and configuration safety.

This issue consolidates:

- read-time `allowed_dirs` enforcement
- deny-all visibility when `allowed_dirs` is empty
- fail-fast validation for MDQ configuration values

## Target files

- `scripts/mcp_servers/mdq/auth.py`
- `scripts/mcp_servers/mdq/search.py`
- `scripts/mcp_servers/mdq/service.py`
- `scripts/mcp_servers/mdq/db_grep.py`
- `scripts/mcp_servers/mdq/health_check.py`
- `scripts/mcp_servers/mdq/models.py`
- `scripts/mcp_servers/mdq/server.py`
- `config/mdq_mcp_server.toml`
- `tests/test_mdq_service.py`
- `tests/test_mdq_read_authorization.py`
- `tests/test_mdq_config.py`
- `tests/test_mdq_health.py`

## Background

MDQ already has a fail-closed path authorization function.

`authorize_path()` denies access when `allowed_dirs` is empty and resolves paths before checking whether they are under an allowed directory.

Path-based tools such as `index_paths`, `refresh_index`, and `outline` use this authorization path.

However, read-time tools such as `search_docs`, `get_chunk`, and `grep_docs` can return content already stored in the SQLite index without re-checking whether the result path is still allowed by the current `allowed_dirs` configuration.

This creates a runtime authorization gap:

- a file may have been indexed while its directory was allowed
- later, `allowed_dirs` may be narrowed or set to an empty list
- old indexed content may still be returned from the database

This is not a database migration issue. It is a runtime authorization enforcement issue.

Separately, `allowed_dirs = []` is a valid fail-closed state, but operators cannot easily see this deny-all state in health output.

Also, `MdqService.__init__()` reads many config values directly from the raw config dictionary. Many of these values are enforced later, but invalid types or invalid ranges are not consistently rejected at startup.

## Reason for change

MDQ handles documentation search and chunk retrieval. Even though it is not a general file reader, returning stale indexed content from paths no longer allowed by configuration can violate operator expectations.

The server should enforce the current authorization policy at read time, not only at index time.

Operators should also be able to distinguish:

- MDQ is broken
- MDQ is healthy but configured to deny all paths
- MDQ configuration is invalid and should fail fast

Invalid config should not cause surprising runtime behavior such as zero-second timeouts, negative limits, huge output sizes, or malformed glob settings.

## Intent

Make MDQ's runtime behavior match the current configuration at all times.

The intended behavior is:

- current `allowed_dirs` controls all returned content
- `allowed_dirs = []` is clearly visible as deny-all
- invalid config fails early
- existing fail-closed security posture is preserved
- no database migration is introduced

## Required changes

### 1. Enforce `allowed_dirs` on read-time tools

For `search_docs`:

- filter search results by `source_path`
- call `authorize_path(Path(source_path), service.allowed_dirs)` for each result
- do not return unauthorized results
- do not leak unauthorized source path, heading, snippet, or chunk ID

For `get_chunk`:

- fetch `source_path` together with chunk content
- authorize `source_path` before returning content
- return `MdqAuthorizationError` when the chunk exists but is no longer authorized

For `grep_docs`:

- if explicit `paths` are supplied, authorize each requested path
- if no path filter is supplied, limit scanning to currently authorized `source_path` values
- do not return unauthorized matches

### 2. Expose deny-all state

Add to health details:

- `allowed_dirs_count`
- `deny_all`

Also log a startup warning when `allowed_dirs` is empty.

Suggested meaning:

- MDQ is running
- path-based operations will be denied
- configure `allowed_dirs` to enable indexing and path operations

Do not expose full allowed paths in health unless existing security policy allows it.

### 3. Add MDQ config validation

Introduce an explicit `MdqConfig` model or equivalent validator.

Validate:

- `allowed_dirs` is `list[str]`
- `include_globs` is `list[str]`
- `exclude_globs` is `list[str]`
- `max_snippet_chars > 0`
- `max_chunk_chars > 0`
- `max_file_chars > 0`
- `search_timeout_sec > 0`
- `max_results_limit > 0`
- `max_chars_per_chunk > 0`
- `max_total_result_chars > 0`
- `max_outline_items > 0`
- `max_grep_matches > 0`
- `max_chars_per_match > 0`
- `context_before >= 0`
- `context_after >= 0`
- `enable_grep` is boolean

Keep current valid defaults unchanged.

## Acceptance criteria

- `search_docs` does not return content from paths outside current `allowed_dirs`.
- `get_chunk` rejects unauthorized chunks even when the chunk exists in the database.
- `grep_docs` does not search or return unauthorized content.
- `allowed_dirs = []` prevents read-time content disclosure.
- `/health.details.deny_all` is true when `allowed_dirs` is empty.
- `/health.details.allowed_dirs_count` reflects the current config.
- Invalid config values fail during service construction or startup.
- Existing valid MDQ config still works.
- Existing service tests continue to pass.

## Test plan

Add tests for:

- index file under allowed directory, then set `allowed_dirs = []`, then verify `search_docs` returns no unauthorized content
- index file under allowed directory, then set `allowed_dirs = []`, then verify `get_chunk` raises `MdqAuthorizationError`
- index file under allowed directory, then set `allowed_dirs = []`, then verify `grep_docs` does not return matches
- `grep_docs(paths=[...])` rejects unauthorized paths
- health details for empty and non-empty `allowed_dirs`
- invalid config values for numeric fields
- invalid config values for list fields
- invalid `enable_grep` type

## Notes for implementer

Do not solve read-time authorization by deleting or migrating database rows.

The goal is runtime enforcement of the current authorization policy.

Avoid leaking unauthorized metadata such as source paths or chunk IDs.
