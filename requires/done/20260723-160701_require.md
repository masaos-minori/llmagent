# Requirement: Add Governance Policy for Removing Implementation-derived Details

## Priority

High

## Target files

- `docs/00_governance_06_ai-reading-metadata.md`

## Background

The design documentation currently contains many details that can be confirmed directly from source code or configuration files. These include:
- Complete source file listings
- Public method tables
- Function signatures
- Constructor parameter tables
- Public attribute lists
- TypedDict, dataclass, DTO, and JSON field listings
- CLI argument tables
- Configuration key tables
- Import examples
- Module-level constant listings

These details are expensive to maintain and can easily become outdated when implementation changes. The documentation should not become a stale copy of the source tree. It should instead focus on design intent, responsibility boundaries, architectural constraints, invariants, failure behavior, operational constraints, known issues, and unresolved confirmation items.

## Problem

Implementation-derived details in design documents create several problems:
1. They require frequent updates when implementation changes
2. They increase the cognitive load for readers who must distinguish design-relevant from implementation-referential content
3. AI agents may treat copied implementation details as authoritative even when the implementation has changed
4. Long reference tables make it harder to find actual design decisions

## Reason for change

Create a governance rule before removing or compressing content. This rule will guide future AI agents and maintainers so that documentation cleanup is consistent, safe, and reviewable. The policy must not encourage blind deletion — it must clearly distinguish between implementation-derived details and design-relevant information.

## Implementation intent

Extend `00_governance_06_ai-reading-metadata.md` with a Japanese policy that defines how to handle implementation-derived details in design documents. Edit only this one file. Write the new policy section in Japanese. Preserve all existing content and add the new section as an extension of the AI-readable documentation guidance.

## Implementation instructions

Add a new Japanese section:

`## 実装参照で確認できる情報の記載方針`

The new section must define the following categories:

### Information Usually Removed or Compressed

Document that the following should normally be removed, compressed, or replaced with source references:
- Complete public method lists
- Function signatures listed only as API reference material
- Constructor parameter tables
- Public attribute tables
- Dataclass, TypedDict, DTO, and Pydantic model field listings
- Complete directory file lists
- Complete CLI argument listings
- Complete configuration key listings
- JSON examples that simply mirror DTO or schema fields
- Module import lists
- Module-level constant listings
- Tables that only map file names to one-line implementation descriptions

### Information That Must Be Kept

Document that the following must remain in design documents:
- Design intent
- Responsibility boundaries
- Architectural constraints
- Non-negotiable invariants
- Failure behavior
- Fail-fast and fail-open behavior
- Security constraints
- Operational constraints
- Data consistency rules
- Code/document mismatches
- Known Issues
- Needs Confirmation items
- Deprecated behavior that is still relevant for migration or compatibility
- Behavior verified by tests
- Operationally observed behavior

### Decision Categories

Add a short decision model for documentation cleanup:
- **Remove**: Content that is purely implementation-derived with no design relevance
- **Compress**: Content that is partially design-relevant but too verbose in its current form
- **Replace with source reference**: Content that is accurate but better confirmed from source code
- **Keep**: Content that is design-relevant and cannot be derived from source alone
- **Move to Known Issues**: Content that represents a documented mismatch between design and implementation
- **Move to Needs Confirmation**: Content that represents an unresolved uncertainty about design intent
