# Add guardrails to keep MDQ `TOOL_LIST`, dispatch table, and ToolRegistry aligned

## Priority

High

## Target files

- `mcp_servers/mdq/server.py`
- `mcp_servers/mdq/tools.py`
- `shared/tool_constants.py`
- `shared/tool_registry.py`
- `shared/tool_routing_validation.py`
- `tests/test_mdq_*.py`
- `tests/test_tool_routing*.py`

## Type

Regression prevention / routing consistency

## Background

MDQ has multiple tool-definition layers:

- `tools.py` defines LLM-visible tool schemas through `TOOL_LIST`
- `server.py` defines runtime dispatch through `_DISPATCH_TABLE`
- `tool_constants.py` defines tool sets used to populate `ToolRegistry`
- `ToolRegistry` is the primary routing authority
- Live `/v1/tools` is used for startup validation and drift detection

A previous inconsistency existed where `tools.py` exposed admin tools that were not present in `_DISPATCH_TABLE`.

This issue adds tests and guardrails so the same class of issue does not recur.

## Problem

Without explicit consistency tests, new tools can be added to one layer but not another.

This causes:

- Schema/runtime mismatch
- Routing drift
- Unknown tool errors
- Incorrect safety tier validation
- Confusing startup diagnostics
- LLM-visible tools that cannot execute

## Intent

Enforce MDQ tool consistency automatically.

The intent is to keep each layer focused on its own responsibility while preventing accidental divergence.

## Tasks

- [ ] Add a test that every `TOOL_LIST` entry exists in `_DISPATCH_TABLE`
- [ ] Add a test that every `_DISPATCH_TABLE` entry exists in `TOOL_LIST`
- [ ] Add a test that every supported MDQ tool is registered in `ToolRegistry`
- [ ] Add a test that no removed MDQ tools remain in `ToolRegistry`
- [ ] Add a test that `MDQ_TOOLS` matches the supported MDQ tool names
- [ ] Add a test that write tools have `is_write=True` where applicable
- [ ] Add a test that serial tools have `requires_serial=True` where applicable
- [ ] Add drift validation expectations for MDQ live discovery
- [ ] Update documentation describing the single responsibility of each layer

## Acceptance criteria

- MDQ schema and dispatch cannot drift silently
- Registry and runtime tool lists remain aligned
- Removed compatibility tools cannot reappear unnoticed
- CI fails when a tool is added to only one layer
- Documentation clearly explains where tool schema, dispatch, and routing ownership live

## Implementation notes

Do not merge all tool definitions into one file unless there is a separate architectural decision. This issue focuses on validation guardrails.
