## Goal

Define canonical source rules for resolving conflicts between documents and between code and documents in the LLM agent design documentation set.

## Scope

Create a governance document (`docs/00_governance_02_canonical-source-rule.md`) that defines:
- How to identify canonical sources for each area
- Rules for resolving document-to-document conflicts
- Classification of code-vs-document conflicts
- Registration rules for Known Issues when conflicts are found
- Resolution workflow with record-keeping

## Assumptions

- Each area already has its own document-guide identifying local canonical sources.
- No single document is universally canonical across all areas.
- Code is not always correct — it can be outdated, deviated from design, provisional, or buggy.

## Design

### Canonical Documents by Area

Each area maintains its own canonical document list via its document-guide. This document provides the framework for defining them but does not list specific documents.

### Conflict Resolution Rule

When two documents conflict:
1. Identify the canonical source for each area involved
2. If both are canonical, defer to the most recently reviewed version
3. If one is canonical and the other is not, the canonical source wins
4. Record the resolution in Known Issues

### Code vs Document Conflict Rule

Classify as one of:
- Document is outdated — update document to match code
- Code reflects current specification — update document
- Code deviates from design intent — escalate for decision
- Provisional implementation — mark as such in Known Issues
- Implementation bug — fix code and update documents
- Design intent uncertain — add to Known Issues for investigation

### Known Issues Registration Rule

Register a Known Issue when:
- The conflict cannot be resolved immediately
- The classification is unclear
- A decision requires stakeholder input
- The resolution affects multiple areas

### Resolution Workflow

1. Detect conflict (document-to-document or code-vs-document)
2. Classify according to rules above
3. Determine if immediate resolution is possible
4. If yes, resolve and record
5. If no, register as Known Issue and follow Known Issues process

## Implementation steps

1. Create `docs/00_governance_02_canonical-source-rule.md` with the sections listed below.
2. Define canonical source identification method per area.
3. Specify conflict resolution rules with clear precedence.
4. Define code-vs-document conflict classification categories.

## Risks

- Medium risk: Over-specifying canonical sources could lock in incorrect decisions. Mitigation: Define the framework without hard-coding specific documents.
- Low risk: Classification categories may overlap. Mitigation: Make categories mutually exclusive where possible.
