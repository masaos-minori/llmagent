# [Agent][LLM][History] Separate incomplete LLM output from normal conversation history

## Summary
When an LLM streaming failure happens after partial text has already been received, the system saves an `[INCOMPLETE: ...]` assistant message into the conversation history.

## Problem
This preserves failure context, but it also injects incomplete content into future turns.
The same problem exists for synthetic tool error messages used during failed tool continuation.

## Scope
- `agent/orchestrator.py`
- history persistence behavior
- tool result store usage

## Task
- store incomplete outputs and synthetic continuation errors in a diagnostic channel separate from normal user-facing history
- keep them visible through debug or `/tool show` style inspection
- avoid feeding incomplete assistant output back into future LLM turns unless explicitly requested

## Expected Behavior
- incomplete generations are preserved for debugging
- incomplete generations do not pollute future conversational context by default

## Acceptance Criteria
- [ ] incomplete assistant output is no longer added to normal history by default
- [ ] synthetic tool continuation errors are stored separately from normal history
- [ ] diagnostic data remains accessible for inspection
- [ ] docs explain the new persistence behavior

- In-memory tool result storage
# [Agent][Persistence] Replace silent persistence skips with visible warnings or strict mode

## Summary
Some persistence paths skip invalid conditions silently, such as invalid roles or missing session context.

## Problem
Silent skip behavior makes runtime more forgiving, but it also hides logic bugs and state corruption.

## Scope
- `agent/session.py`
- persistence error handling
- logging and metrics for skipped writes

## Task
- emit warnings when persistence is skipped because of invalid role or missing session state
- add counters or metrics for skipped persistence operations
- add an optional strict mode that treats these conditions as errors

## Expected Behavior
- unexpected persistence skips are visible
- production can use strict persistence rules if desired
- debugging persistence bugs becomes easier

## Acceptance Criteria
- [ ] skipped writes are logged
- [ ] skipped writes increment a visible metric or counter
- [ ] strict persistence mode exists
- [ ] docs describe both permissive and strict behavior

- AgentSession API failure notes
