## Goal

Add governance document defining cross-cutting rules for maintaining the LLM agent design documentation set.

## Scope

Create a single governance document (`docs/00_governance_01_documentation-governance.md`) that defines:
- Document classes (types) used across the design documentation set
- Update rules: when and which documents must be updated when changes occur
- Review rules: conditions requiring review before merging
- Change impact rules: how to determine affected documents based on change type
- Non-goals: what this document explicitly excludes

## Assumptions

- The `docs/` directory uses a flat structure (no subdirectories).
- Each area has its own document-guide, Known Issues, and design notes.
- This document covers cross-cutting concerns only, not area-specific details.

## Unknowns

- None identified. All scope items are defined in the issue.

## Design

### Document Classes

Define the following document classes:
- Overview: high-level architecture and system description
- Specification: detailed design specifications
- Guide: operational guidance and conventions
- Governance: cross-cutting rules and policies
- Reference: API references, schemas, and technical reference material
- Operations: deployment, monitoring, and troubleshooting
- Note: temporary working documents and decisions

### Update Rule

When a change occurs, update the following documents based on change type:
- Architecture change → Overview + area Specification
- Configuration change → area Specification + Operations
- MCP tool specification change → MCP area Specification + Governance
- Database schema change → Shared/DB Specification + Operations
- Safety boundary change → Governance + area Specification

### Review Rule

Review is required when:
- Affecting safety boundaries or security controls
- Changing database schemas
- Modifying cross-cutting governance rules
- Introducing breaking changes to documented APIs

### Change Impact Rule

Determine affected documents by:
1. Identifying the change category
2. Mapping to affected areas
3. Listing specific documents in each area that require updates

## Risks

- Medium risk: Defining too many document classes could create confusion. Mitigation: Keep classes minimal and mutually exclusive.
- Low risk: Update rules may miss edge cases. Mitigation: Document principles rather than exhaustive lists; allow human judgment.
