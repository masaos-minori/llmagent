# Title
Verify SQLite shared connection thread-safety and add DB access serialization if needed

## Priority
High

## Labels

## Dependencies

## Summary
Validate the Event Bus SQLite access model and add synchronization or a safer connection strategy if FastAPI can execute DB operations across threads.

## Background
The persistence doc states that a shared SQLite connection is opened at startup with `check_same_thread=False` and is safe because FastAPI runs DB operations on a single async event loop thread. The known-issues document still lists FastAPI thread pool worker usage as unconfirmed.

## Problem
If any DB operation runs in a worker thread or concurrently without proper synchronization, the shared SQLite connection can cause data races, lock errors, or corruption risk.

## Required Changes
- Confirm whether Event Bus routes are `async def` or `def`.
- Confirm whether DB code can run inside FastAPI's thread pool.
- If a shared connection remains:
  - serialize writes with `asyncio.Lock`, or
  - introduce a dedicated DB worker, or
  - use per-request/per-operation connections.
- Add stress tests for concurrent:
  - publish
  - replay
  - subscribe
  - ack
  - DLQ promotion
- Update docs to describe the final DB access model.

## Acceptance Criteria
- DB access model is verified and documented.
- Concurrent publish/replay/ack/DLQ operations do not produce thread errors, lock errors, or lost writes.
- Known issue about FastAPI thread pool usage is resolved or updated with a safe interpretation.
- Tests cover concurrent DB usage.

## Out of Scope
- Replacing SQLite with another database.
- Distributed Event Bus clustering.
