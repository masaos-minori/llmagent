## Title
[Low] Document DB store public API boundaries

## Priority
Low

## Labels
documentation, db, architecture, cleanup

## Dependencies
None

## Background
The DB store layer is split into:
- `db/store.py` as the stable public import surface
- `db/store_protocols.py` for Protocol definitions
- `db/store_impl.py` for SQLite implementations

Documentation should reflect this boundary so future implementation work does not import concrete internals unnecessarily.

## Scope
Update:
- DB layer documentation
- Reference API documentation
- Developer guide if present

## Required Changes
- Document that callers should import from `db.store`.
- Identify `store_protocols.py` as the extension point for storage contracts.
- Identify `store_impl.py` as the SQLite implementation layer.
- Warn against direct imports from implementation modules unless intentional.

## Acceptance Criteria
- DB store module boundaries are clear.
- Future implementation tasks know where to add protocols and implementations.
