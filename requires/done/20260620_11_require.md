# Title
Unify the responsibility boundary between AgentSession and RAG maintenance operations

## Summary
Refactor the agent layer so that `AgentSession` is responsible only for session/message/note persistence in `session.sqlite`, while RAG maintenance responsibilities are handled through `DbMaintenanceService` or MCP-based service boundaries.

## Background
Current documentation is inconsistent about whether `AgentSession` directly accesses RAG-layer tables.
Some documents describe `AgentSession` as touching RAG data for `/db` convenience commands, while others describe RAG access as going through `DbMaintenanceService`.

## Problem
This inconsistency weakens architectural boundaries and makes future RAG schema refactoring riskier than necessary.

## Required Changes
- Audit `agent/session.py` for any direct dependency on RAG-layer tables or schema details.
- Move RAG-related `/db` maintenance behavior out of `AgentSession` and into `DbMaintenanceService` or MCP-mediated service calls.
- Ensure `AgentSession` owns only session/message/note persistence responsibilities.
- Align all relevant documentation to describe the same boundary model.

## Acceptance Criteria
- `AgentSession` no longer depends on RAG schema details.
- RAG maintenance operations are routed through a dedicated maintenance or service boundary.
- Documentation consistently describes `AgentSession` as session-layer persistence only.

## Out of Scope
- Redesigning the RAG retrieval pipeline itself
- Removing `/db` maintenance commands entirely

## AI Implementation Instruction
Implement this as a boundary-hardening task, not as a broad feature redesign.

Strict rules:
- Do not add new direct DB access paths.
- Do not move session persistence responsibilities out of `AgentSession`.
- Prefer moving RAG-specific logic away from `AgentSession`, not the other way around.
- If you find ambiguous existing behavior, document it clearly and mark it as: Needs confirmation
- Update docs and implementation together so the boundary is consistent everywhere.

Deliverables:
1. Refactored ownership boundary in code
2. Updated docs reflecting the corrected boundary
3. A short note listing any remaining ambiguous edge cases
