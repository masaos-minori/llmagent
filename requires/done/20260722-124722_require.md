## Background

MCP documentation needs organization to ensure consistency between implementation and docs, eliminate duplicates, and remove references to deleted features.

## Problem

Multiple inconsistencies exist between current implementation and documentation:
- RuntimeToolRegistry may not be documented as the sole source for runtime routing
- ToolRegistry descriptions may cause misreading as routing source
- tool_constants.py may be described as routing input incorrectly
- /v1/tools response shape may not match current discovery implementation
- enabled/disabled_reason/enabled_for_llm descriptions may not match implementation
- disabled_reason may be described as a direct RuntimeTool field (it's not)
- Git MCP availability metadata and disabled call response spec differences are unclear
- web-search/browser_fetch availability metadata may be stated without implementation verification
- LLM-visible vs runtime-routable distinction may not be explained
- DAG scheduling metadata source may be incorrectly attributed to RuntimeToolRegistry
- New tool addition procedure may not assume live /v1/tools discovery
- Deleted MCP features/configs/endpoints/old routing models may remain in docs
- Duplicate MCP routing explanations across multiple sections

## Reason for Change

Ensure documentation accurately reflects current implementation, eliminates duplicates, and removes references to deleted features.

## Implementation Intent

Organize MCP documentation by consolidating related content into appropriate files and removing outdated information.

## Implementation Instructions

### Step 1: Consolidate routing details

Move all routing details to `04_mcp_03_01_dispatch-and-routing.md`. Ensure RuntimeToolRegistry is documented as the sole source for runtime routing. Remove any ToolRegistry descriptions that imply it's a routing source. Remove any tool_constants.py routing input descriptions.

### Step 2: Consolidate ToolRegistry description

Move all ToolRegistry descriptions to `04_mcp_03_02_tool-registry.md`. Clarify its role vs RuntimeToolRegistry.

### Step 3: Consolidate /v1/tools and availability metadata

Move all /v1/tools and availability metadata content to `04_mcp_03_06_tool-runtime-availability-metadata.md`. Update /v1/tools response shape to match current discovery implementation. Fix enabled/disabled_reason/enabled_for_llm descriptions to match implementation. Remove any description implying disabled_reason is a direct RuntimeTool field.

### Step 4: Add new tool registration procedure

Add or update new tool registration procedure in `04_mcp_03_05_lifecycle-and-new-server.md` or `04_mcp_06_14_new-tool-registration-procedure.md`. Ensure it assumes live /v1/tools discovery.

### Step 5: Review server-specific catalogs

Review `04_mcp_04_01_web-search-file-read-github.md` and `04_mcp_04_05_git.md`. Document Git MCP availability metadata and disabled call response spec differences. Do not state web-search/browser_fetch availability metadata without implementation verification.

### Step 6: Review failure diagnosis and startup validation

Review `04_mcp_06_09_mcp-failure-diagnosis.md` and `04_mcp_06_11_startup-validation-behavior-tool_definitions_strict.md`. Ensure LLM-visible vs runtime-routable distinction is explained. Ensure DAG scheduling metadata source is correctly attributed.

### Step 7: Clean up known issues

Keep only unresolved issues that affect current specification in `04_mcp_90_inconsistencies_and_known_issues.md`.

### Step 8: Update cross-references

Update File Index, AI Query Routing Table, Related Documents, and internal links after consolidation.
