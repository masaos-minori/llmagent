# Title
Enforce ack-only consumer offset advancement

## Priority
High

## Labels
- bug
- eventbus
- offsets
- delivery-semantics
- at-least-once
- reliability

## Dependencies
- None

## Summary
Make consumer offset advancement strictly ack-only. Offsets must advance only when the consumer explicitly acknowledges an event.

## Background
The delivery semantics document defines an ack-only offset model. However, the HTTP API document also states that `/subscribe` writes the current sequence to the offset file on disconnect.

## Problem
Writing offsets on disconnect can acknowledge events that the consumer has not processed. This weakens the at-least-once delivery guarantee and may cause message loss after consumer crashes.

## Required Changes
- Inspect `/subscribe` disconnect handling.
- Remove any automatic offset write on disconnect.
- Ensure offsets are updated only by `POST /events/{event_id}/ack?consumer_id=...`.
- Update HTTP API docs to remove "write offset on disconnect" behavior.
- Add tests for:
  - receive event without ack, disconnect, reconnect, event is replayed
  - ack event, disconnect, reconnect, replay starts after acked seq
  - since_seq override behavior remains explicit and documented

## Acceptance Criteria
- Offsets advance only after successful ack.
- Consumer crash before ack causes replay of the unacked event.
- Documentation consistently describes at-least-once delivery.
- Tests prove reconnect behavior with and without ack.

## Out of Scope
- Implementing exactly-once delivery.
- Adding consumer-side deduplication beyond event_id semantics.
