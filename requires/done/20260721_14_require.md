# [Critical] Add `from __future__ import annotations` to `scripts/shared/tool_executor.py` to fix startup-breaking `NameError`

## Priority

Critical

## Labels

priority/critical, area/shared, area/agent-startup, bugfix, regression

## Summary

`scripts/shared/tool_executor.py` raises an unhandled `NameError: name 'RuntimeToolRegistry' is not
defined` the instant it is imported, because it uses a `TYPE_CHECKING`-only-imported name
(`RuntimeToolRegistry`) as a bare (non-string) runtime method annotation without
`from __future__ import annotations` at the top of the file. Every module that imports
`agent.factory`, `agent.context`, or `agent.repository_gateway` — i.e. the normal agent-startup import
chain — currently crashes as a result. Add the missing `from __future__ import annotations` import;
no other change is required.

## Background

Full root-cause analysis, reproduction steps, and impact assessment are recorded in
`issues/20260721-120414_tool_executor_missing_future_annotations_nameerror.md`. Summary:

- Commit `3c4fbe06` ("feat: make RuntimeToolRegistry sole routing authority; merge browser-mcp into
  web-search-mcp") added:
  ```python
  if TYPE_CHECKING:
      from shared.runtime_tool_registry import RuntimeToolRegistry
  ...
  def set_runtime_registry(self, registry: RuntimeToolRegistry) -> None:
  ```
  to `scripts/shared/tool_executor.py`, but this file has no `from __future__ import annotations`, so
  the annotation is evaluated eagerly at class-body execution time — while `RuntimeToolRegistry` is
  never imported outside the `TYPE_CHECKING` guard.
- Every other file in the codebase using the identical `if TYPE_CHECKING:` guard pattern for a runtime
  annotation (e.g. `scripts/agent/tool_policy.py`, `scripts/agent/repository_gateway.py`) already has
  `from __future__ import annotations` at the top — this file is the outlier that's missing it.
- Reproduced directly on current `master` (2026-07-21):
  `PYTHONPATH=scripts .venv/bin/python3 -c "from agent import factory"` raises the `NameError`
  immediately.
- This is the confirmed root cause of the full test suite's 25 failed / 17 errors observed during
  unrelated implementation work today — isolated by re-running with an unrelated file temporarily
  removed and observing an identical failure/error count.

This bug was discovered incidentally while implementing an unrelated set of plans
(`implementations/done/20260720-134152_agent.toml.md`,
`implementations/done/20260720-134218_test_mcp_server_cmd_paths.py.md`,
`implementations/done/20260720-134259_20260720-070458_mcp_server_agent_toml_cmd_points_to_deleted_files.md.md`)
and is out of scope for those; it needs its own planning/implementation pass.

## Required Changes

- Add `from __future__ import annotations` as the first import in `scripts/shared/tool_executor.py`
  (matching the placement convention used in `scripts/agent/tool_policy.py` and
  `scripts/agent/repository_gateway.py` — immediately after the module docstring, before other
  imports).
- Do not change the `TYPE_CHECKING` guard, the `RuntimeToolRegistry` import, or the
  `set_runtime_registry` method signature itself — the annotation stays a bare (non-string) name; only
  its evaluation timing changes.
- After the fix, confirm no other annotation in the file relied on eager evaluation in a way that
  would break under deferred evaluation (e.g. any runtime use of `typing.get_type_hints()` on this
  module, or `inspect.signature()` callers that resolve annotations at runtime) — grep
  `get_type_hints\|inspect\.signature` against callers of `ToolExecutor` before concluding the change
  is side-effect-free.

## Acceptance Criteria

- `PYTHONPATH=scripts .venv/bin/python3 -c "import shared.tool_executor"` succeeds with no exception.
- `PYTHONPATH=scripts .venv/bin/python3 -c "from agent import factory"` succeeds with no exception.
- `PYTHONPATH=scripts .venv/bin/python3 -c "from agent import context, repository_gateway"` succeeds
  with no exception.
- `uv run pytest -q` — the 25 failed / 17 errors attributable to this `NameError` clear. Any residual
  failures unrelated to this root cause are triaged separately, not silently absorbed into this fix.
- `uv run mypy scripts/` — no new errors introduced (deferred annotation evaluation must not change
  type-checking results, since mypy already treats annotations as deferred by default).
- `uv run ruff check scripts/` / `ruff format scripts/` — clean.
- A regression test (or an existing test extended) asserts that `shared.tool_executor` and
  `agent.factory` import successfully — e.g. a smoke-import test — so this exact failure mode cannot
  silently regress again undetected by the normal test suite.

## Target Files

- `scripts/shared/tool_executor.py`
- `tests/` (new or extended smoke-import regression test)

## Non-Goals

- Do not refactor `ToolExecutor`, `ToolRouteResolver`, or `RuntimeToolRegistry` routing logic — this is
  a single-line import-ordering fix, not a design change.
- Do not add `from __future__ import annotations` to unrelated files as part of this requirement, even
  if similar risk exists elsewhere — scope this to the one file confirmed broken today. A broader
  audit for the same class of bug (bare `TYPE_CHECKING`-only annotation without deferred evaluation)
  across the rest of `scripts/` can be filed as a separate, follow-on requirement if desired.
- Do not address the unrelated pre-existing lint/architecture findings noted during today's
  investigation (`rag_pipeline_server.py` formatting/import order, `eventbus` layering violations,
  bandit medium/low findings) — those are out of scope for this fix.
