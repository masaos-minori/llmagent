# Add ADR for Mandatory Workflow Execution

## Summary

Create an Architecture Decision Record that explains why workflow execution is mandatory and why workflow bypass modes are not supported.

## Reason for Change

The current documentation clearly states that workflow definitions are required and that workflow execution cannot be disabled.

However, the architectural rationale is not sufficiently documented.

This creates confusion for new contributors and operators because even simple agent usage appears to depend on workflow infrastructure.

Without an explicit ADR, maintainers may incorrectly assume that workflow execution is an optional orchestration feature rather than a core control plane for task execution, approvals, retries, idempotency, and auditability.

## Intent

The intent is to document workflow as a mandatory execution boundary.

The ADR should explain that workflow execution is not merely an optional feature.
It is the system-level mechanism that provides:

- Task state tracking
- Approval gates
- Execution ordering
- Retry management
- Idempotency controls
- Auditability
- Recovery support
- Consistent handling of tool side effects

The ADR should make it clear that direct tool execution bypassing workflow is intentionally disallowed for safety, consistency, and observability reasons.

## Scope

Create a new ADR document.

Suggested filename:

- `ADR-Workflow-Mandatory.md`

Also update references from:

- Architecture overview
- Workflow documentation
- Deployment documentation
- Operations documentation where appropriate

## Required Sections

### Context

Describe the architectural context:

- The system executes LLM-planned tasks.
- Some tools have side effects.
- Some operations require approval.
- Tool execution must be observable and recoverable.
- A direct LLM-to-tool path would make auditing and recovery harder.

### Decision

State the decision clearly:

- Workflow execution is mandatory.
- Workflow definitions are required deployment artifacts.
- Workflow bypass mode is not supported.
- Optional workflow mode is not supported.
- Direct execution fallback is not supported.

### Rationale

Explain why workflow is always required.

Include the following rationale:

- All side-effecting operations must be traceable.
- Approval state must survive process boundaries.
- Retry and idempotency behavior must be centralized.
- Partial task completion must be inspectable.
- Recovery requires persisted task and attempt state.
- Tool execution should not depend solely on LLM conversational state.

### Alternatives Considered

Document rejected alternatives:

- Direct tool execution without workflow
- Optional workflow mode
- Workflow disabled for local mode
- Workflow fallback when workflow definition is missing
- Per-tool ad hoc approval without workflow state

For each alternative, explain why it was rejected.

### Consequences

Document operational consequences:

- Deployment must include workflow definition files.
- Startup must fail if mandatory workflow artifacts are missing or invalid.
- Workflow schema must be initialized before service startup.
- Operators must treat workflow failures as platform failures.
- Simple chat and tool-backed tasks share the same execution control plane.

### Non-Goals

Clarify what this ADR does not attempt to solve:

- It does not define every workflow stage.
- It does not redesign approval policy.
- It does not introduce EventBus integration.
- It does not change runtime behavior.

## Implementation Notes for AI Agent

- This is a documentation task.
- Do not change workflow runtime code unless a separate implementation issue requires it.
- Use clear architectural language.
- Separate facts from assumptions.
- If historical motivation is unknown, write `Historical motivation: Unknown`.
- Do not claim past incidents unless documented.
- Do not include EventBus-related recommendations.

## Acceptance Criteria

- ADR document is created.
- ADR clearly states that workflow execution is mandatory.
- ADR explains the design intent behind mandatory workflow execution.
- ADR documents rejected alternatives.
- ADR documents operational consequences.
- Architecture and workflow documentation link to the ADR.
- No unsupported historical claims are introduced.
