# MCP /v1/tools: Clarify schema_version, capabilities, and Deferred Fields

## Priority

Medium

## Target Files

- `docs/04_mcp_02_01_endpoints-and-transport.md` — separate current vs deferred fields
- `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` — clarify deferred design options

## Background

The documentation needs a sharper boundary between current /v1/tools fields and deferred future options. Some fields are currently supported while others are planned but not yet implemented.

## Problem

API consumers and server authors need to know which fields are safe to rely on today. Deferred design options should not be confused with implemented API behavior.

## Reason for Change

API consumers and server authors need to know which fields are safe to rely on today. Deferred design options should not be confused with implemented API behavior.

## Implementation Intent

Document current fields separately from deferred fields in both endpoint documentation and availability metadata documentation.

## Implementation Instructions

### Step 1: Update endpoints-and-transport.md

In the /v1/tools section, add two clear subsections:

```markdown
### Current fields

These fields are currently supported by the discovery service:

- `schema_version` — top-level discovery-time metadata
- `tools` — array of tool definitions
- `name` — per-tool identifier
- `description` — per-tool description
- `inputSchema` — per-tool schema (or `input_schema`)
- `status` — per-tool status
- `enabled` — maps to RuntimeTool.enabled_for_llm
- `disabled_reason` — reason when disabled
- `is_write` — whether the tool performs writes
- `requires_serial` — whether the tool requires serial execution
- `resource_scope` — resource scope constraints
- per-tool `capabilities` — optional capability flags

### Deferred fields

These fields are planned but NOT yet implemented:

- `include_disabled` — query parameter to include disabled tools
- `disabled_code` — structured code for disabled reasons
- top-level `capabilities` — if not already implemented
```

### Step 2: Update tool-runtime-availability-metadata.md

In the Future / deferred design options section, explicitly mark:
- `include_disabled` as not yet implemented
- `disabled_code` as not yet implemented
- top-level capabilities as either documented as unsupported OR implemented and documented

## Acceptance Criteria

1. Implemented fields and deferred fields are clearly separated
2. include_disabled is not presented as available unless implemented
3. disabled_code is not presented as available unless implemented
4. Per-tool capabilities behavior is documented
5. Top-level capabilities are either documented as unsupported or implemented and documented
