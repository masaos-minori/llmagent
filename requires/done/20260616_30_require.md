# [Agent][Context] Improve token budgeting accuracy beyond chars // 4

## Summary
When exact token usage is not available, the agent falls back to `chars // 4` as a token estimate.

## Problem
This estimate is simple, but it can be inaccurate for multilingual text, tool payloads, and structured conversation history.
That can make compression timing unstable or misleading.

## Scope
- `HistoryManager`
- `/context` budget reporting
- token counting fallback logic

## Task
- improve the fallback token estimator
- distinguish history text budget from tool payload budget where practical
- keep accurate counting when usage or `/tokenize` is available

## Expected Behavior
- token budget estimation is more stable
- compression behavior is less dependent on a rough character heuristic
- `/context` shows more trustworthy budget information

## Acceptance Criteria
- [ ] fallback token estimate is more accurate than `chars // 4`
- [ ] budget reporting makes tool payload impact more visible
- [ ] docs explain exact vs estimated token counting clearly
