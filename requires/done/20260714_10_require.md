# [Medium] Normalize MCP security policy documentation

## Labels
priority/medium, area/security, area/docs, area/github-mcp, area/cicd-mcp, area/shell-mcp

## Summary
Normalize security policy documentation across fail-open/fail-closed settings, production recommendations, and known issue status.

This issue consolidates overlapping security documentation fixes for:

- `tool_definitions_strict` default versus production recommendation
- `shell_sandbox_backend` default versus production requirement
- CI/CD `workflow_allowlist` behavior
- GitHub `allowed_repos_mode`
- stale `db_allowlist` references

## Background
The security documentation contains several policy ambiguities:

- defaults and production recommendations are sometimes conflated
- `shell_sandbox_backend` is described as both `"none"` default and `"firejail"` production default
- `workflow_allowlist` has a known issue around RuntimeError versus warning behavior
- GitHub fail-open/fail-closed summary mentions `allowed_repos_mode`, but the GitHub catalog does not fully document it
- `db_allowlist` appears in notes despite no current DB MCP server being listed

## Required Changes

### 1. `tool_definitions_strict`
Inspect implementation default and document separately:

- default value
- production recommended value
- production validation behavior, if enforced

Suggested model:

- Default: local/development compatibility value
- Production recommended: `true`

### 2. `shell_sandbox_backend`
Document default and production requirement separately.

Suggested model:

- Default: `shell_sandbox_backend = "none"`
- Production requirement: must be `"firejail"`
- `"none"` is rejected at startup when `security_profile="production"`

### 3. CI/CD `workflow_allowlist`
Confirm actual behavior for empty `workflow_allowlist`.

Unless implementation says otherwise, document:

- empty list emits startup warning
- `trigger_workflow` is denied for all workflows
- no startup RuntimeError solely because the allowlist is empty

Update known issue status accordingly.

### 4. GitHub `allowed_repos_mode`
If supported, document:

- `allowed_repos_mode = "fail_closed"`:
  - `allowed_repos=[]` denies all GitHub write operations

- `allowed_repos_mode = "fail_open"`:
  - `allowed_repos=[]` allows all repositories
  - not allowed or strongly discouraged in production

Document startup audit or production validation behavior if implemented.

### 5. `db_allowlist`
Resolve stale `db_allowlist` references.

If no DB MCP server exists:

- remove `db_allowlist` from active docs

If it exists:

- restore catalog, configuration, and security documentation

## Acceptance Criteria
- Defaults and production recommendations are not conflated.
- Shell sandbox docs agree across catalog, security model, and defaults table.
- CI/CD `workflow_allowlist` RuntimeError versus warning behavior is resolved.
- GitHub `allowed_repos_mode` is documented wherever fail-open/fail-closed behavior is discussed.
- Stale `db_allowlist` references are removed or fully restored.
- Pre-production checklist reflects current implemented controls.
