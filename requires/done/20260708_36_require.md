# M-4: Keep production MCP auth validation startup-only

## Priority

Medium

## Background

Production security profile requires `auth_token` on HTTP MCP servers. This validation must remain startup-only.

Reload must not become a way to partially repair or overwrite MCP auth configuration at runtime.

## Target files

- `agent/repl_health.py`
- `agent/services/config_reload.py`
- tests for security validation and reload

## Implementation tasks

- Preserve production-mode startup validation for missing MCP `auth_token`.
- Ensure `/reload` does not apply MCP `auth_token` changes.
- Ensure `/reload` reports auth token changes as restart-required.
- Add documentation that production auth validation is enforced at startup.
- Do not introduce runtime token swapping to bypass startup validation.

## Acceptance criteria

- Production startup still fails if any HTTP MCP server lacks `auth_token`.
- Reloading a config with a new token reports restart-required.
- Reloading a config with a new token does not change running config.
- Operators must restart to apply auth token changes.

## Test plan

- In production mode, start with missing token and verify startup validation fails.
- In a running session, reload with changed token.
- Verify restart-required output.
- Verify running token is unchanged.
