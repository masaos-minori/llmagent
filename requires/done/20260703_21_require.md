# Refactor plugin tool registration to enforce return contract at registration time

## Priority

High

## Goal

Make plugin tool return contract validation fail fast at registration time.

The canonical plugin tool contract is:

- Return type annotation must be `tuple[str, bool]`
- The function must be an async plugin tool handler
- Invalid plugin tools must not be registered

## Background

Plugin tool validation must be canonical at registration time, not delayed until first execution.

Current or target behavior:

- `@register_tool()` validates the return annotation.
- Missing return annotation is a contract violation.
- Invalid return annotation is a contract violation.
- Runtime validation may remain as defensive validation in `PluginToolInvoker`.

## Scope

Affected files:

- `shared/plugin_registry.py`
- `shared/plugin_tool_invoker.py` if introduced in a separate task
- plugin-related tests
- agent documentation for plugin extension behavior
- known issues documentation if it still states call-time-only validation

## Implementation requirements

- Update `register_tool(name)` in `shared/plugin_registry.py`.
- Validate return annotation using `typing.get_type_hints(fn)`.
- Accept `tuple[str, bool]` as the canonical contract.
- If project policy allows `typing.Tuple[str, bool]`, normalize using `typing.get_origin()` and `typing.get_args()`.
- Reject missing annotations.
- Reject any return annotation other than `tuple[str, bool]` or its explicitly supported equivalent.
- Ensure invalid tools are not inserted into the plugin tool registry.
- Preserve strict and non-strict plugin load behavior:
  - `plugin_strict=false`: record plugin load failure and continue.
  - `plugin_strict=true`: aggregate failures and raise `PluginLoadError`.

## Acceptance criteria

- A plugin tool annotated as `-> tuple[str, bool]` registers successfully.
- A plugin tool without return annotation fails registration.
- A plugin tool annotated as `-> tuple[str, str]` fails registration.
- Invalid plugin tools are not present in the registry.
- `ToolExecutor.execute()` no longer owns plugin return contract validation details.
- Defensive runtime validation remains possible in `PluginToolInvoker`.
- Documentation no longer contradicts the registration-time validation model.

## Validation plan

- Add or update unit tests for `shared/plugin_registry.py`.
- Test valid annotation registration.
- Test missing annotation rejection.
- Test invalid annotation rejection.
- Test strict plugin loading behavior.
- Test non-strict plugin loading behavior.

## Out of scope

- Changing plugin tool precedence.
- Changing plugin command registration.
- Adding backward-compatible validation shims.
