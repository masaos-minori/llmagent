## Summary
Resolve conflicting statements about mdq-mcp being production-ready versus having experimental or `stub: true` markers.

## Background
The document guide and server catalog say mdq-mcp is production-ready. The security model still contains wording that FTS5 search is functional but not production-validated and that health/tool metadata include `stub: true`.

## Problem
Operators and AI routing logic cannot reliably determine whether mdq-mcp should be used in production.

## Required Changes
- Inspect mdq-mcp health endpoint and tool metadata.
- Determine whether `stub: true` or similar experimental markers remain.
- If mdq-mcp is production-ready:
  - remove `stub: true`
  - remove outdated non-production wording
  - update known issues if needed
- If mdq-mcp is not production-ready:
  - downgrade production-ready wording in all docs.
- Add tests for health metadata and tool metadata consistency.
- Ensure MDQ/RAG boundary docs reflect the final status.

## Acceptance Criteria
- mdq-mcp status is consistently documented as either production-ready or experimental.
- Health endpoint and tool metadata match the documented status.
- No stale `stub` wording remains if production-ready.
- Tests protect status metadata from regressing.

## Out of Scope

- Migrating MDQ data to RAG.
