## Summary
Clarify what causes events to enter the DLQ and how `retry_count` should be incremented.

## Background
DLQ promotion occurs when `retry_count >= max_retry` and `dlq_at IS NULL`. Current docs state that `retry_count` is incremented only on DLQ requeue, not during normal delivery.

## Problem
Operators may expect normal consumer delivery failures to increment retry_count and eventually move events to DLQ. The current model appears to only DLQ events that have already been manually requeued enough times.

## Required Changes
- Decide whether DLQ is for:
  - manual requeue exhaustion only, or
  - consumer delivery failure tracking.
- If consumer failure tracking is required:
  - design explicit nack/failure API
  - define retry_count increment rules
- If current behavior remains:
  - document that normal streaming failure does not increment retry_count
  - explain why DLQ promotion may not happen for unacked events
- Add warning to requeue response when event will be re-promoted on next DLQ loop.
- Update known issues if behavior remains surprising.

## Acceptance Criteria
- DLQ promotion criteria are unambiguous.
- `retry_count` ownership is documented.
- Operators understand the difference between unacked replay and DLQ promotion.
- Tests cover requeue, re-promotion, and max_retry edge cases.

## Out of Scope
- Implementing full consumer group failure tracking unless explicitly approved.
- Exactly-once delivery.
