# Add a startup validator for cross-file RAG configuration consistency

## Summary
Implement a startup-time configuration validator that checks consistency across RAG-related config files.

## Background
Configuration is intentionally split across multiple TOML files, but cross-file validation is not clearly documented.

## Objective
Fail fast or warn early when RAG configuration values are mutually inconsistent.

## In Scope
- Config loading path for:
  - `config/common.toml`
  - `config/agent.toml`
  - `config/rag_pipeline.toml`
- Relevant service startup path
- Related operational documentation

## Required Changes
- Build an integrated configuration validation step
- Validate at least:
  - embedding dimensions
  - embedding endpoint settings
  - `rag_service_url` related constraints
  - semantic cache configuration sanity
  - dangerous quality settings such as `use_rrf=false`
- Decide which problems are:
  - warnings
  - hard errors
- Update documentation to explain validation behavior

## Acceptance Criteria
- Misconfigured cross-file settings are detected at startup
- Validation output is actionable
- Operator-facing behavior is documented
- Validation is deterministic and testable

## Out of Scope
- Redesigning the config file layout
- Secret management changes

## Implementation Notes
Prefer explicit validation messages tied to concrete config keys and file names.
