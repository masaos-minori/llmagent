## Background

The MCP watchdog is a background asyncio task (`watchdog_loop()` in `repl_health.py`) that polls each MCP server's `GET /health` endpoint on an interval (`mcp_watchdog_interval`, currently `60` in `config/agent.toml`, security-profile default `30.0` for `production` / `0.0` for `local`) and, based on the `reachable` / `restart_recommended` fields in the health response, restarts subprocess-mode servers up to `mcp_watchdog_max_restarts` (default `3`) times. This is documented in `docs/04_mcp_03_04_tool-call-tracing-and-watchdog.md` and `docs/04_mcp_06_12_watchdog-configuration-monitoring.md`.

The user has decided this feature is unnecessary and requested its complete removal.

## Problem

Keeping the watchdog in the codebase while it is unwanted carries the usual cost of dead/unwanted functionality: config keys that must be maintained (`mcp_watchdog_interval`, `mcp_watchdog_max_restarts`), a background task started/stopped on every REPL run, a dedicated test file, a doc-consistency check (`check_watchdog_restarts_on_dependency_failure`), and multiple documentation pages describing behavior nobody wants to keep.

Note (assumption, not confirmed): removing the watchdog removes the only automatic recovery path for a crashed subprocess-mode MCP server. After removal, a crashed MCP server stays unreachable until an operator manually restarts the agent process or the server. This is a factual consequence of the removal, not a reason against it — flagging it so the decision is made with full knowledge of the operational impact.

## Reason for change

Explicit user request: the watchdog feature is not needed and should be removed.

## Design intent

Remove the watchdog loop, its configuration surface, its status/display integration, and its dedicated tests and documentation, while leaving the rest of the MCP health-check subsystem intact:

- `GET /health` endpoint on each MCP server, `HealthRegistry` (HEALTHY/DEGRADED/UNAVAILABLE/HALF_OPEN state machine), and the tool-execution-layer `record_success()` / `record_failure()` calls are used independently by `/mcp status` (`McpStatusService.probe_all()`) and by tool dispatch error handling. These must NOT be removed as part of this change unless a separate decision is made to remove them too.
- `HealthRegistry.record_restart_exhausted()` (`scripts/shared/mcp_health.py`) currently has no caller other than the watchdog. Decide whether to delete it outright or keep the state/reason (e.g. for `/mcp status` display of a prior watchdog-exhausted state) — see Unknowns.

## Acceptance criteria

- No references to `mcp_watchdog_interval`, `mcp_watchdog_max_restarts`, `watchdog_loop`, or `_watchdog_check_http` remain in `scripts/`, `config/`, or `tools/`
- `/mcp status` no longer displays a `Watchdog` line
- `tests/test_watchdog.py` is deleted; remaining tests pass without watchdog config fields
- `tools/check_mcp_docs_consistency.py` passes with no watchdog-related check left dangling (either the check is removed, or updated to assert the feature's absence)
- `docs/*` contain no stale references implying the watchdog still exists
- `HealthRegistry`, `/health` endpoints, and `/mcp status` probing continue to work (manual, non-watchdog health check path is unaffected)
- Full test suite and toolchain validation (`rules/toolchain.md`) pass

## Non-goals

- Do not remove the `/health` endpoint, `HealthRegistry`, or `McpStatusService.probe_all()` (used by `/mcp status`)
- Do not remove tool-execution-layer `record_success()` / `record_failure()` calls
- Do not change subprocess lifecycle management (`start_http_subprocess()`, `ensure_ready()`, manual restart via `/mcp` commands) beyond what is strictly needed to detach it from the watchdog

## Unknowns

- Whether `HealthRegistry.record_restart_exhausted()` and its associated state/reason should be deleted entirely or kept as a harmless no-longer-invoked capability
- Whether any external process supervisor (systemd, container healthcheck, etc.) is expected to take over crash recovery for subprocess-mode MCP servers in production once the watchdog is removed
- Whether the removal should be reflected as "removed" notes in docs (kept as historical record) or the pages should be deleted outright, per this project's documentation convention for removed features
