# Ensure routing drift validation runs at startup and fails in strict mode

## Priority

High

## Type

Design / Implementation / Startup Validation

## Background

The MCP tool routing model uses `ToolRegistry` as the sole routing authority. Tool names from server config and live `/v1/tools` responses are validation inputs only and must not affect routing decisions.

The startup validation layer already has functions for checking routing drift and live tool drift, but startup wiring must be verified and enforced.

## Problem

If routing drift validation is not always executed during startup, the system may start with inconsistent tool state.

Examples of unsafe drift include:

- A tool exists in `ToolRegistry` but is not exposed by any MCP server.
- A live MCP server exposes a tool that is not registered.
- A tool is visible to the LLM but cannot be routed.
- A tool lacks a declared safety tier.
- Config `tool_names` no longer matches the registry.

## Goal

Guarantee that routing drift is detected at startup and that strict mode treats drift as a fatal startup error.

## Scope

Affected areas:

- `agent/repl_health.py`
- `shared/tool_registry.py`
- `shared/route_resolver.py`
- `shared/tool_routing_validation.py`
- Startup orchestration
- CI validation tests

## Implementation Tasks

- Verify that startup sequence always calls:
  - `check_routing_drift(ctx)`
  - `check_routing_drift_vs_live(ctx, strict=...)`
  - `check_tool_definitions_startup(ctx)`
- Integrate `validate_all_routing()` into startup validation.
- Ensure strict mode raises `RuntimeError` on any routing drift.
- Ensure non-strict mode returns structured warnings.
- Add CI checks for registry, live tools, LLM tool definitions, and safety tiers.
- Ensure error messages include server key, tool name, and mismatch direction.

## Acceptance Criteria

- Startup validation always checks registry vs config drift.
- Startup validation always checks registry vs live `/v1/tools` drift when servers are reachable.
- Strict mode fails startup on drift.
- Non-strict mode logs warnings and returns a structured warning result.
- All LLM-visible tools are routable.
- All registered tools have a safety tier.

## Validation Plan

Add tests for:

- Tool in registry but missing from live server.
- Tool exposed by live server but missing from registry.
- Tool configured for LLM but missing from live server.
- Tool missing safety tier.
- All servers unreachable in strict mode.
- All servers unreachable in non-strict mode.
- Partial server unreachable with reachable servers still validated.

## Risks

Strict drift validation may reveal existing inconsistencies and cause startup failures. This is intended behavior for environments that require correctness.

## Out of Scope

- Changing routing authority away from `ToolRegistry`.
- Using live `/v1/tools` for routing.
- Documentation-only updates.
