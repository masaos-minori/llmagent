# Protect reset_registry as a test-only API

## Priority

Medium

## Type

Maintainability / Safety / Test Infrastructure

## Background

`ToolRegistry` is a global singleton populated from `tool_constants.py`.

A reset function exists for tests so that unit tests can isolate registry state.

## Problem

If registry reset is accidentally called from production code, routing state may be recreated during runtime, potentially causing inconsistent tool ownership or drift behavior.

## Goal

Make registry reset clearly test-only and prevent production use.

## Scope

Affected areas:

- `shared/tool_registry.py`
- Tests using registry reset
- Static analysis checks
- Import usage scan

## Implementation Tasks

- Rename `reset_registry()` to `_reset_registry_for_testing()` or move it to a test helper module.
- Keep a temporary compatibility wrapper only if necessary.
- Add a static check that forbids production code from calling registry reset.
- Allow usage only in pytest fixtures or test modules.
- Add deprecation warning if the old public function remains temporarily.

## Acceptance Criteria

- Production code does not call registry reset.
- Tests still have a supported way to reset registry state.
- CI fails if non-test code imports or calls the reset API.
- The test-only nature of the API is obvious from the function name and location.

## Validation Plan

Add tests or static checks for:

- Allowed usage in test files.
- Forbidden usage in production modules.
- Registry reset still works for test isolation.
- Backward compatibility wrapper behavior if retained.

## Risks

Existing tests may need import updates. This is acceptable because the API should not be public runtime surface.

## Out of Scope

- Removing the global singleton registry.
- Changing routing rules.
- Documentation-only updates.
