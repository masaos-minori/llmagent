## [Medium][eventbus][health] Define clear HTTP status and body semantics for health, readiness, and degraded states

### Summary
The current health endpoint always returns HTTP 200 and encodes problems only in the response body.

### Problem
This weakens integration with infrastructure health checks and makes severe degradation harder to classify automatically.

### Scope
- app.py
- /health
- operational health semantics
- troubleshooting and deployment docs

### Task
- define separate semantics for:
  - healthy
  - degraded
  - unavailable / not ready
- decide when HTTP 200 vs HTTP 503 (or similar) should be used
- keep backward compatibility in mind if health consumers already exist
- update docs and tests for the new behavior

### Expected Result
The health endpoint communicates severe failures and degraded states in a way that infrastructure can interpret reliably.

### Acceptance Criteria
- health semantics are explicitly defined
- HTTP status behavior is documented and implemented
- body fields remain clear and consistent
- tests verify healthy/degraded/unavailable responses
