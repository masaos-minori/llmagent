# Add WorkflowExecutionPolicy to centralize workflow mode fallback and fail-closed decisions

## Priority

Medium

## Goal

Centralize workflow execution policy decisions so `workflow_mode` behavior is not scattered across startup and orchestration code.

## Background

Workflow behavior includes:

- `workflow_mode=disabled`
- `workflow_mode=auto`
- `workflow_mode=required`
- startup validation
- startup degradation
- turn-time fail-closed behavior
- `WorkflowCreationError` handling

These decisions should be controlled by one policy object.

## Scope

Affected files:

- `agent/workflow_execution_policy.py` new file, or equivalent
- `agent/startup.py`
- `agent/orchestrator.py`
- workflow tests
- configuration tests if needed

## Implementation requirements

Create:

    class WorkflowExecutionPolicy:
        def is_workflow_enabled(self) -> bool:
            ...

        def requires_startup_definition(self) -> bool:
            ...

        def allow_startup_fallback(self) -> bool:
            ...

        def allow_turn_fallback(self, error: Exception) -> bool:
            ...

        def fail_closed_on_creation_error(self) -> bool:
            ...

Required behavior:

- `workflow_mode=disabled`
  - Always use direct LLM path.
  - Do not require workflow definition.
- `workflow_mode=required`
  - Require workflow definition at startup.
  - Fail startup on loader or state store initialization failure.
  - Do not fall back to direct execution on `WorkflowCreationError`.
- `workflow_mode=auto`
  - Allow startup degradation for missing workflow definition.
  - Do not allow fallback that bypasses required tool approval.
  - Clearly define turn-time fallback and fail-closed behavior.

## Acceptance criteria

- `startup.py` does not scatter raw `workflow_mode` string comparisons.
- `orchestrator.py` does not scatter raw `workflow_mode` string comparisons.
- `WorkflowCreationError` does not fall back to unsafe direct execution.
- `workflow_mode=disabled` always uses direct LLM path.
- `workflow_mode=required` fails startup when required workflow resources are missing.
- `workflow_mode=auto` has explicit startup and turn-time behavior.

## Validation plan

- Add tests for each workflow mode.
- Test missing workflow definition in each mode.
- Test workflow creation failure.
- Test direct path only in disabled mode.
- Test fail-closed behavior where required.

## Out of scope

- Changing workflow database schema.
- Changing workflow approval semantics.
