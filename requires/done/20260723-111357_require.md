# Requirement: Treat Memory Injection Failure as Non-Fatal Warning During Startup

## Priority

Medium

## Target files

- `scripts/agent/startup.py`

## Background

In `StartupOrchestrator._setup_prompt()`, the memory layer is an optional service (`ctx.services_required.memory`). When it exists, `on_session_start()` injects semantic memories into the initial system prompt. However, if `on_session_start()` raises an exception (e.g., DB connection error, network timeout), the entire startup sequence fails because no try/except wraps this call. Deployments without memory enabled continue silently without any indication that the memory feature is unavailable.

## Problem

When the memory backend is unreachable during startup, the exception propagates through `StartupOrchestrator.run()`:

```python
try:
    await self._start_servers()
    _servers_started = True
    await self._check_services()
    await self._recover_pending_approvals()
    await self._setup_prompt()  # <-- exception here
except Exception as setup_err:
    if _servers_started:
        try:
            await self._ctx.services_required.lifecycle.shutdown_all()
        except Exception as shutdown_err:
            logger.error("CRITICAL: Startup rollback FAILED...")
    raise setup_err
```

This causes three issues:
1. Production deployments with memory enabled fail hard — no fallback path
2. Deployments without memory enabled continue silently — no warning about the missing feature
3. The operator sees a cryptic raw exception (e.g., "connection refused") without guidance

## Reason for change

Since the memory layer is explicitly optional (disabled via `use_memory_layer=False`), its failure should not prevent the agent from starting. The agent should operate normally without memory snippets when the memory backend is unavailable, just as it does when the memory layer is disabled entirely.

## Implementation intent

Wrap the memory injection call in `StartupOrchestrator._setup_prompt()` in a try/except block. On failure, log a warning and display a non-fatal warning message to the operator, then continue with the base system prompt (without memory snippets).

## Implementation instructions

1. In `StartupOrchestrator._setup_prompt()`, wrap the existing `if ctx.services_required.memory is not None:` block in a try/except:
   ```python
   if ctx.services_required.memory is not None:
       try:
           memory_snippets = ctx.services_required.memory.on_session_start(
               ctx.session.session_id,
           )
           if memory_snippets:
               ...  # existing truncation and prompt injection logic
       except Exception as exc:
           logger.warning(
               "Memory injection failed during startup: %s; continuing without memory",
               exc,
           )
           self._view.write_warning(f"{OutputTag.NON_FATAL} Memory injection failed: {exc}")
   ```
2. Do NOT modify `session.start()` behavior — that is handled separately.
3. Ensure the warning uses `OutputTag.NON_FATAL` to distinguish from fatal errors.

## Acceptance criteria

- [ ] When `on_session_start()` fails during startup, the agent continues operating with the base system prompt (no memory snippets)
- [ ] A non-fatal warning is displayed to the operator indicating memory injection failed
- [ ] No raw exception strings or tracebacks shown to the operator
- [ ] Resource cleanup proceeds normally via the finally block after the warning
- [ ] No regression when memory layer is disabled (`use_memory_layer=False`)
- [ ] No regression when memory layer succeeds normally

## Tests

- Unit test: Mock `memory.on_session_start()` to raise `ConnectionRefusedError`, verify startup continues and writes a non-fatal warning
- Unit test: Verify the base system prompt is used when memory injection fails
- Integration test: Stop the embedding service, start REPL with `use_memory_layer=True`, verify startup completes with a warning
