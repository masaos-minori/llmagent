# [High] Invalidate the rag-pipeline MCP server's semantic cache after CLI ingestion

## Labels
priority/high, area/rag, area/mcp, area/cache, consistency

## Summary
`rag-ingest` (the CLI entry point in `scripts/rag/ingestion/ingester.py`) currently has no way to invalidate the semantic cache held by the long-running `rag-pipeline` MCP server process (port 8010). After a CLI ingestion run, the MCP server keeps serving stale cached query results until the cache entries expire or are evicted on their own.

This is tracked as **OPEN-01** in `docs/03_rag_90_inconsistencies_and_known_issues-part2.md`. The document's current "Recommended action" (pass `pipeline.semantic_cache.invalidate` as the `on_ingest_complete` callback) only works when the caller and the `RagPipeline` instance share the same process. It does not work for `rag-ingest`, because the CLI and the MCP server are separate OS processes — an in-process callback in the CLI process cannot reach an object living in the MCP server process.

## Background
- The current specification is:
  - `RagIngester.ingest_all(force, on_ingest_complete)` accepts an optional callback and invokes it after ingestion completes (`scripts/rag/ingestion/ingester.py`, `ingest_all()` around lines 91-130).
  - `main()` (lines 605-620) calls `ingester.ingest_all(args.force)` without passing `on_ingest_complete`.
  - `RagPipelineMCPService` (`scripts/mcp_servers/rag_pipeline/service.py`) owns the single, process-local `RagPipeline` instance (`self._pipeline`, constructed in `start()`) whose `invalidate_cache()` (line 213, called from `fmt_delete_document()`) clears `semantic_cache`.
  - The MCP server (`scripts/mcp_servers/rag_pipeline/server.py`) runs as its own long-lived HTTP process on port 8010 and exposes `/v1/call_tool`, `/v1/tools`, `/health`, `/rag_run_pipeline`, `/rag_debug_pipeline` — there is currently no endpoint to invalidate the cache from outside the process.
  - `ConfigLoader.restrict_to("ingester.toml")` (`ingester.py` line 624) restricts the CLI process to loading only `config/ingester.toml`. It cannot read `config/agent.toml`, where the MCP server URL is currently defined (`rag_service_url = "http://127.0.0.1:8010"`, `config/agent.toml` line 52). `config/ingester.toml` does not currently define any MCP server URL.
  - The CLI already owns a reusable `httpx.Client` (`self._client`, `ingester.py` line 75) used for embedding requests, so it has an established HTTP call pattern to extend.
- Impact: after `rag-ingest` runs (e.g. a scheduled or manual content refresh), any RAG query served by the MCP server may return results derived from a stale semantic cache until the cache naturally expires. This can surface outdated or removed content to end users.

## Required Changes

- Add an HTTP endpoint to the `rag-pipeline` MCP server (`scripts/mcp_servers/rag_pipeline/server.py`) that invalidates the process-local semantic cache, e.g. `POST /rag_invalidate_cache`, delegating to the existing `RagPipelineMCPService` (which already exposes `invalidate_cache()` via `_pipeline_or_raise().invalidate_cache()`, see `service.py` line 213).
- Add a new config key to `config/ingester.toml` for the MCP server's base URL (e.g. `rag_pipeline_service_url`), since the CLI process cannot read `config/agent.toml` under `ConfigLoader.restrict_to("ingester.toml")`.
- Update `RagIngester` (or `main()`) to call the new endpoint via the existing `httpx.Client` after `ingest_all()` completes successfully, replacing (or complementing) the currently-unused `on_ingest_complete` parameter for the CLI entry point.
- Define and implement the failure behavior when the MCP server is unreachable or not running (e.g. in local/dev environments where the server isn't started): this must not fail the CLI ingestion run. Log a `WARNING` and continue — do not raise.
- Keep the existing `on_ingest_complete` parameter and its in-process semantics intact for any other same-process callers; this change only adds an out-of-process path for the CLI entry point.
- Update `docs/03_rag_90_inconsistencies_and_known_issues-part2.md`: replace or supplement the OPEN-01 "Recommended action" to describe the new HTTP-based invalidation path, and mark OPEN-01 as resolved once implemented (following the same resolved-entry style already used for OPEN-02 in the same file).

## Acceptance Criteria
- Running `rag-ingest` (CLI) against a running `rag-pipeline` MCP server results in the server's `semantic_cache` being invalidated immediately after ingestion completes, without requiring a server restart.
- If the MCP server is unreachable when the CLI attempts invalidation, `rag-ingest` logs a `WARNING` and exits normally (ingestion success/failure reporting is unaffected).
- `config/ingester.toml` documents the new MCP server URL key with a comment, following the existing style of other keys in that file.
- `docs/03_rag_90_inconsistencies_and_known_issues-part2.md` OPEN-01 is updated to reflect the resolved, process-boundary-aware invalidation mechanism.
- No change to the semantics of `on_ingest_complete` for existing in-process callers.

## Non-Goals
- Do not change the semantic cache's TTL/eviction policy — this issue is only about triggering invalidation after CLI ingestion, not about cache expiry design.
- Do not introduce an EventBus-based (`scripts/eventbus/`) notification path for this issue; use a direct HTTP call to the existing MCP server as the simpler, in-scope solution.
- Do not modify the `on_ingest_complete` callback's existing in-process contract or remove it.
- Do not address `docs/03_rag_90_inconsistencies_and_known_issues-part1.md` (`DESIGN-2`, `DESIGN-3`) — these are confirmed design decisions, not inconsistencies, and are out of scope for this issue.
