# L-2: Classify startup messages as fatal or non-fatal

## Priority

Low

## Labels

observability, startup, logging, developer-experience

## Background

Startup currently emits many warnings.

Some are non-fatal in local development, such as:

- Plugin skipped in fail-open mode
- LLM service unreachable in local profile
- MCP servers unreachable in local profile

Other issues are fatal or startup-blocking, such as:

- Missing `sessions` table in `session.sqlite`

The current output makes it difficult to distinguish immediate blockers from degraded optional services.

## Goal

Improve startup diagnostics by clearly classifying messages as fatal or non-fatal.

## Scope

### In scope

- Improve startup warning wording
- Distinguish local-profile degraded services from fatal initialization errors
- Add actionable message for missing session schema

### Out of scope

- Changing readiness policy
- `/exit` handling
- sqlite-vec path handling

## Required behavior

Examples:

- Plugin skip: mark as non-fatal when plugin loading mode is fail-open
- LLM or MCP unreachable in local profile: mark as non-fatal degraded mode
- Missing session schema: mark as fatal and suggest session schema initialization command

## Acceptance criteria

- Startup output makes it clear which messages require immediate action
- Local degraded mode warnings are not confused with fatal errors
- Missing `sessions` table produces an actionable message

## Validation

Run agent in local profile with:

- No MCP servers running
- No LLM server running
- An initialized session DB

Expected result:

- Agent starts
- Warnings are classified as non-fatal

Run agent with missing session schema.

Expected result:

- Fatal error tells user how to initialize session schema

## Affected files

- startup / health check modules
- session startup path
- operations documentation

## Dependencies

Recommended after:
