# Title
Reduce startup risk around `workflow_mode="required"` in production defaults

## Priority
Medium

## Labels
- bug
- agent
- workflow
- startup
- configuration
- operations

## Dependencies
- None

## Summary
Reduce the operational trap created by the production default `workflow_mode="required"` when no valid workflow definition is deployed.

## Background
The config docs state that:
- dataclass default is `workflow_mode="auto"`
- production default in `common.toml` is `workflow_mode="required"`
- environments that deploy `common.toml` must also deploy a valid workflow definition, or startup fails

## Problem
A missing workflow definition can make the entire agent fail at startup, even for otherwise unrelated features.
This creates a large behavior gap between local/dev and production environments.

## Required Changes
- Improve startup validation and error reporting for missing workflow definitions.
- Make production requirements more explicit in operations docs.
- Make the startup error clearly identify:
  - missing workflow definition
  - active mode
  - why startup is blocked
- Consider adding a preflight validation message earlier in startup.

## Acceptance Criteria
- Workflow-definition absence is easier to detect before or at startup.
- Startup failure messages are actionable.
- The local/dev vs production difference is clear in docs and runtime behavior.

## Out of Scope
- Redesigning workflow mode semantics
- Changing production default automatically
