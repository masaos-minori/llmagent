# Clarify that DAG scheduling metadata currently comes from configured tool_definitions

## Priority

Medium

## Scope

Documentation-only unless a future design chooses to unify scheduler metadata with RuntimeToolRegistry.

## Target documentation sections

- `04_mcp_03_01_dispatch-and-routing.md`
  - `直列化メカニズムは2つ存在する`
  - `副作用検出`
  - `DAGスケジューリング`
- `04_mcp_06_14_new-tool-registration-procedure.md`

## Problem

The documentation may imply that RuntimeToolRegistry metadata from `/v1/tools` directly drives DAG scheduling.

Current implementation does not do that.

## Current implementation behavior

DAG scheduling uses metadata built from `ctx.cfg.tool.tool_definitions`.

The scheduler reads or derives:

- `function.requires_serial`
- `function.resource_scope`
- `function.is_write`
- side-effect status from `is_side_effect()`
- shell-specific serial behavior

RuntimeToolRegistry metadata is used for routing and LLM visibility, but it is not currently the direct input to DAG scheduling.

## Required documentation changes

Add a clear explanation:

- RuntimeToolRegistry controls runtime routing and LLM visibility.
- DAG scheduling currently uses configured LLM tool definitions.
- Updating `/v1/tools.requires_serial` alone does not necessarily change DAG scheduling.
- New tool registration may require updates in both `/v1/tools` metadata and `config/agent.toml` tool definition metadata.

## Reason for change

This prevents implementers from assuming that live MCP metadata automatically controls scheduling. Without this clarification, new tools may be incorrectly parallelized or unnecessarily serialized.

## Acceptance criteria

- Documentation clearly separates:
  - Runtime routing metadata.
  - LLM-visible tool schema.
  - DAG scheduling metadata.
- New tool registration docs include all required metadata update paths.
- No section implies that `RuntimeTool.requires_serial` alone controls scheduling today.
