# Dead code removal: gitops_force_push_blocked / gitops_protected_branches

## Priority

High

## Target files

- `scripts/agent/config_dataclasses.py`
- `scripts/agent/config_builders.py`
- `docs/05_agent_06_02_tool-execution-and-approval-approval.md`
- `docs/05_agent_06_03_tool-execution-and-approval-concurrency-safety.md`

## Background

The docs audit (2026-07-21) identified that `gitops_force_push_blocked` and `gitops_protected_branches` are referenced in documentation (`05_agent_06_02_tool-execution-and-approval-approval.md:106-107`, `05_agent_06_03_tool-execution-and-approval-concurrency-safety.md:36`) as part of the approval gate's branch escape level evaluation. However, verification against current source shows these fields are NOT used anywhere in the actual implementation.

## Problem

Both fields exist in `scripts/agent/config_dataclasses.py` with default values:

```python
gitops_force_push_blocked: bool = True       # Line 351
gitops_protected_branches: list[str] = ...    # Lines 353-355
```

But in `scripts/agent/config_builders.py`, their loading is commented out:

```python
# REMOVED: gitops_force_push_blocked=bool(cfg.get("gitops_force_push_blocked", True)),
# REMOVED: gitops_protected_branches=list(cfg.get("gitops_protected_branches", ["main", "master"])),
```

A full grep across `scripts/` confirms zero references outside these four locations. These fields are dead code — they cannot affect runtime behavior because:
1. They are never loaded from config (commented out in config_builders.py)
2. They are never read by any logic (no references in tool_policy.py or tool_approve)

This creates a critical inconsistency: documentation describes functionality that does not exist. If an operator relies on `gitops_protected_branches` to protect branches like "main"/"master", those protections are silently absent.

## Reason for change

This is a security-relevant discrepancy. Documentation claims protections exist that do not actually exist. Either the code needs to be restored to implement the documented behavior, or the documentation and dead code need to be removed to prevent false confidence.

## Implementation intent

Remove the dead code from `config_dataclasses.py` and update documentation to accurately reflect the current state of branch protection. Do NOT restore the code unless there is explicit business justification — the current design appears intentional (the REMOVED comment suggests deliberate removal).

## Implementation instructions

### Step 1: Remove dead code from config_dataclasses.py

Delete lines 351-355:

```python
    # Block all GitHub write operations globally when True
    gitops_push_blocked: bool = False
    # Block github_push_files with force=True
    gitops_force_push_blocked: bool = True          # ← DELETE
    # Protected branch names; push/merge to these requires high-risk approval
    gitops_protected_branches: list[str] = field(   # ← DELETE
        default_factory=lambda: ["main", "master"]   # ← DELETE
    )                                                 # ← DELETE
```

Keep `gitops_push_blocked` (line 349) only if it has active usage — verify first.

### Step 2: Update documentation

Update `docs/05_agent_06_02_tool-execution-and-approval-approval.md` and `docs/05_agent_06_03_tool-execution-and-approval-concurrency-safety.md` to remove references to `gitops_force_push_blocked` and `gitops_protected_branches`. Replace with accurate descriptions of what branch protections currently exist.

### Step 3: Verify no other references

Run:

```bash
grep -rn "gitops_force_push_blocked\|gitops_protected_branches" . --include="*.md" --include="*.py"
```

Ensure all references are cleaned up.
