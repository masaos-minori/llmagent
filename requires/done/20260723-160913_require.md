## Background

Several RAG design documents contain extensive implementation reference material including public method tables, constructor parameter tables, public attribute lists, function signatures, import examples, TypedDict field lists, DTO field tables, dataclass field tables, full JSON payload examples, and diagnostics field listings. These details are useful in source-level API references but make design documents long and difficult to maintain. They can also become stale when implementation changes.

## Problem

Design documents that duplicate every source-level signature and field list create several problems:
1. They require frequent updates when implementation changes
2. They increase cognitive load for readers who must distinguish design-relevant from implementation-referential content
3. AI agents may treat copied implementation details as authoritative even when the implementation has changed
4. Long reference tables make it harder to find actual design decisions

## Reason for change

A design document should focus on design-level information, not exhaustively copy source-level API references. This task compresses implementation-derived API signatures, public method tables, DTO field listings, TypedDict listings, dataclass listings, and JSON examples into concise Japanese design summaries while preserving caller-visible behavior, responsibility boundaries, failure behavior, data ownership, persistence semantics, search-quality-sensitive fields, compatibility-sensitive fields, security-sensitive fields, and important invariants.

## Implementation intent

Keep RAG documentation focused on design-level information. Stop duplicating every source-level signature and field list. Replace exhaustive signatures and field lists with concise Japanese design summaries. Add source references where appropriate.

## Implementation instructions

For each target document:

1. Identify implementation-derived API and data model details
2. Classify each candidate before editing:
   - implementation-derived detail
   - caller-visible behavior
   - design boundary
   - invariant
   - failure behavior
   - data ownership rule
   - unresolved uncertainty
3. Remove or compress only implementation-derived details
4. Replace exhaustive signatures and field lists with concise Japanese design summaries
5. Preserve major entry points
6. Preserve caller-visible behavior
7. Preserve data ownership and compatibility constraints
8. Preserve invariants and failure behavior
9. Add source references where appropriate

Suggested API replacement sections:
- `## 主要入口`
- `## 呼び出し側に影響する挙動`
- `## 責務境界`
- `## 失敗時の挙動`
- `## 実装詳細の参照先`

Suggested data model replacement sections:
- `## データ構造の目的`
- `## 設計上重要なフィールド`
- `## 永続化・検索・互換性に関わる制約`
- `## 使用してはいけない境界`
- `## 実装詳細の参照先`

Content to remove or compress:
- Complete method signature tables
- Constructor parameter tables
- Complete public attribute lists
- Import statement examples
- Helper method catalogs
- Full DTO field tables
- Complete TypedDict field lists
- Complete dataclass field lists
- Full JSON payload examples
- Diagnostics field listings that simply mirror implementation
- Repeated field descriptions already available from source code

Content to keep:
- Main public entry points
- Externally observable behavior
- Error behavior
- Fallback behavior
- Cache invalidation responsibility
- `None` versus empty string behavior
- Database failure behavior
- Process-local configuration behavior
- File lifecycle where operationally relevant
- Data ownership boundaries
- Compatibility-sensitive fields
- Persistence-critical fields
- Search-quality-sensitive fields
- Security-sensitive fields
- Known Issues
- Needs Confirmation items

Specific preservation requirements — preserve these RAG design constraints where currently documented:
- `RagPipeline.augment()` is the main query pipeline entry point
- Empty string is a valid result where documented
- `None` triggers fallback where documented
- Database connection failure must not be silently swallowed where documented
- Cache invalidation is caller responsibility where documented
- Agent does not directly own RAG core logic when MCP is the boundary
- Process-local configuration files must remain documented
- `content` is the LLM-facing text where documented
- `normalized_content` is for FTS5/search indexing and must not replace LLM context text
- `chunks_vec → chunks → documents` deletion order must remain documented
- Local file freshness behavior must remain documented where design-relevant

Work document by document. If unsure whether content is design-relevant, keep it and mark for later review rather than deleting it.
