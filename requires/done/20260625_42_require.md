## [Medium][eventbus][observability] Add health and runtime metrics for subscribers, queue backlog, notify lag, and slow consumers

### Summary
Moving live delivery to an in-memory broker changes the runtime bottlenecks and observability needs.

### Problem
Current health reporting cannot show broker pressure, subscriber backlog, or push-delivery lag.

### Scope
- /health
- runtime metrics
- push broker observability
- operations docs

### Task
- add metrics for:
  - active subscriber count
  - per-subscriber or broker queue backlog
  - publish-to-notify lag
  - slow consumer count
- surface the relevant metrics in health or runtime diagnostics
- document how to interpret the new signals
- add tests where practical for degraded broker states

### Expected Result
Operations can detect push-path degradation before it turns into visible SSE issues.

### Acceptance Criteria
- subscriber and broker pressure are observable
- notify lag is observable
- slow consumers are observable
- docs explain the new metrics clearly
