# Strengthen the public contract visibility of db/store_protocols.py

## Summary
Make the protocol-layer contract in `db/store_protocols.py` easier to read and verify against its SQLite implementations.

## Background
The DB layer exposes a protocol/implementation split, but the protocol surface is currently harder to inspect than the implementation surface. This weakens the value of the abstraction boundary.

## Objective
Make the store protocols function as a clear public contract, not just as a thin internal typing aid.

## In Scope
- `store_protocols.py`
- `store.py`

## Required Changes
- Ensure each protocol exposes a readable and complete method contract
- Make the intended mapping from protocol to implementation easier to discover
- Add or improve module-level comments so users know:
  - which file defines protocols
  - which file defines SQLite implementations
  - which file is the stable import surface

## Acceptance Criteria
- A reader can understand the protocol API without diving into implementation first
- The abstraction boundary is clearer from the exported surface
- `store.py` remains a stable import entrypoint with better discoverability

## Out of Scope
- Adding new storage backends
- Reorganizing all DB modules

## Implementation Notes
Prefer contract clarity. The protocol layer should be the first place a reader can understand the supported storage operations.
