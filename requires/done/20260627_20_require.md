## Summary
Standardize MCP audit log format across protocol docs, operations docs, server logs, and agent tracing guidance.

## Background
The protocol docs describe audit logs as key=value lines. The operations docs show JSON grep patterns and JSON log examples.

## Problem
Operators may use grep patterns or parsers that do not match actual logs, making failure diagnosis unreliable.

## Required Changes
- Inspect actual audit logging implementation.
- Choose a canonical format:
  - JSON Lines, or
  - key=value lines.
- Update all docs and grep examples to match the canonical format.
- Standardize required fields:
  - `session`
  - `request`
  - `server_key`
  - `tool_name` or `action`
  - `target`
  - `outcome`
  - `detail`
  - `error_type`
- Ensure X-Session-Id and X-Request-Id can correlate agent, transport, and server logs.
- Add audit log format tests.

## Acceptance Criteria
- Docs grep examples work against actual logs.
- Agent audit logs and MCP server audit logs share correlation fields.
- Tool errors and transport errors are distinguishable.
- Audit log tests validate format and required fields.

## Out of Scope
- Building a full log aggregation system.
- Changing log storage paths unless needed for consistency.

## Components
- `mcp/audit.py`

- `04_mcp_06_configuration_and_operations.md`

## AI Implementation Instruction
Implement this as an observability consistency task.

Strict rules:
- Do not keep two incompatible canonical formats.
- Prefer machine-readable JSON Lines if current implementation supports it.
- Preserve correlation keys.
- If different logs intentionally use different formats, document each clearly with examples.
