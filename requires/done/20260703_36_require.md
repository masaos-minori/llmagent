# Split Git and GitHub tool constants into read/write/dangerous classifications

## Priority

Medium

## Goal

Improve classification precision for approval policy and scheduling behavior.

## Background

Current aggregate Git and GitHub tool sets mix read-only and write-capable operations.

This makes approval and side-effect classification less precise.

## Scope

Affected files:

- `shared/tool_constants.py`
- `shared/tool_registry.py`
- `shared/tool_executor_helpers.py`
- `agent/tool_policy.py`
- tests for constants and classification

## Implementation requirements

Add Git classifications:

    GIT_READ_TOOLS = frozenset({
        "git_status",
        "git_log",
        "git_diff",
        "git_branch",
        "git_show",
    })

    GIT_WRITE_TOOLS = frozenset({
        "git_add",
        "git_commit",
        "git_checkout",
        "git_pull",
        "git_push",
    })

Add GitHub classifications:

- `GITHUB_READ_TOOLS`
- `GITHUB_WRITE_TOOLS`
- `GITHUB_DANGEROUS_TOOLS`

Decide whether to keep aggregate sets:

- `GIT_TOOLS`
- `GITHUB_TOOLS`

Recommended:

- Keep aggregate sets as derived unions if still needed by registry seeding.

## Acceptance criteria

- Registry seeding remains correct.
- Existing known Git tools are still registered.
- Existing known GitHub tools are still registered.
- `is_side_effect()` follows the new write/dangerous classification where appropriate.
- Approval policy uses the new classifications.
- Tests cover read/write/dangerous classification.

## Validation plan

- Test aggregate Git set equals read union write.
- Test aggregate GitHub set equals read union write union dangerous.
- Test registry population still includes all tools.
- Test approval classification for representative read/write/dangerous tools.

## Out of scope

- Adding new Git/GitHub tool names unrelated to existing constants.
- Changing tool server ownership.
