## Title
[High] Do not persist LLM transport errors as normal assistant messages

## Priority
High

## Labels
bug, llm, persistence, diagnostics, session-history

## Dependencies
None

## Background
LLM transport errors are returned as `TurnResult(action="fail", answer=summary, exception=e)`. The orchestrator may save `result.answer` as a normal assistant message even when it represents an error summary.

This risks polluting conversation history and future LLM context with transport error diagnostics.

## Scope
Update:
- `agent/turn_result.py`
- `agent/llm_turn_runner.py`
- `agent/orchestrator.py`
- `agent/diagnostic_store.py`
- LLM error handling tests

## Required Changes
- Add an explicit persistence flag to `TurnResult`, for example:
  - `persist_as_assistant: bool = True`
- Return `persist_as_assistant=False` for LLM transport failures.
- In the orchestrator, only save assistant messages when `persist_as_assistant=True`.
- Keep partial and non-partial transport errors in:
  - `session_diagnostics`
  - `tool_results`, where applicable
- Ensure `turn_end.partial_completion` reflects actual partial completion state.

## Tests to Add or Update
- Add tests verifying:
  - Transport error summaries are not saved as assistant messages.
  - Diagnostics are saved to `session_diagnostics`.
  - Restored session history does not include transport diagnostics.
  - Partial completion audit state is accurate.

## Acceptance Criteria
- Transport errors do not enter normal `messages` conversation history.
- Diagnostic information remains retrievable.
- `fetch_messages()` does not restore diagnostic content into LLM context.
