# Title
Enforce MDQ / RAG data ownership boundaries with automated checks

## Labels
architecture
validation

## Dependencies
- Formalize the MDQ / RAG responsibility boundary
- Standardize the MDQ database path to `mdq.sqlite`

## Components
- mcp/mdq/
- mcp/rag_pipeline/
- Agent / application code
- test / lint / static validation

## Summary
Add enforceable checks to protect the data ownership boundary between MDQ and RAG and to prevent normal application code from introducing direct DB access.

## Background
The intended design is:
- MDQ owns `mdq.sqlite`
- RAG owns `rag.sqlite`
- neither system directly accesses the other’s DB
- the agent layer normally uses MCP tool calls instead of direct sqlite access

Admin-only maintenance access is an exception, not the normal integration model.

## Problem
If this boundary is enforced only by convention, future code changes may introduce:
- cross-DB dependencies
- direct sqlite access from application code
- hidden ownership leaks that are hard to detect during review

## Required Changes
- Add checks that ensure MDQ code does not directly depend on `rag.sqlite`.
- Add checks that ensure RAG code does not directly depend on `mdq.sqlite`.
- Add validation or review rules that detect forbidden direct sqlite access in normal application code.
- Clearly distinguish allowed admin-only maintenance access from forbidden normal runtime direct access.

## Acceptance Criteria
- Cross-DB dependency violations can be detected automatically.
- Forbidden direct DB access in normal application code can be detected automatically or via an enforceable validation rule.
- The exception path for admin maintenance is explicitly documented and distinguished from normal behavior.

## Out of Scope
- Removing admin maintenance access entirely
- Replacing MCP with another integration architecture

## AI Implementation Instruction
Implement this as a boundary-enforcement and regression-prevention task.

Strict rules:
- Do not add any new direct DB access path.
- Do not weaken the MCP-first principle for normal application behavior.
- Treat admin maintenance access as an explicit exception, not as a pattern to reuse.
- Prefer lightweight and enforceable checks:
  - focused tests
  - grep-based validation
  - lint rules
  - code review guardrails
- Keep false positives manageable, but do not make the rule meaningless.

Required analysis:
- Search all references to `mdq.sqlite`, `rag.sqlite`, and direct `sqlite3` usage.
- Distinguish clearly between:
  - allowed admin-only maintenance access
  - forbidden normal application access

If an existing use is ambiguous:
- do not guess,
- do not silently whitelist it,
- mark it as: Needs confirmation

Deliverables:
1. Automated validation for MDQ / RAG ownership boundaries
2. Detection of forbidden direct DB access in normal application code
3. Documentation of allowed vs forbidden access paths
