# Require: Add session_id guard to AgentSession.save_diagnostic

## Priority

Medium

## Target files

- `scripts/agent/session.py`

## Background

`AgentSession.__init__()` initializes `DiagnosticStore(self.session_id)` when `self.session_id` is still `None`. The `save_diagnostic()` method passes `self.session_id` directly to `DiagnosticStore.save()` without checking if it is `None`. Unlike `replace_messages()` which has a guard against this condition, `save_diagnostic()` silently ignores the call when `session_id` is `None`.

## Problem

When `DiagnosticStore.save()` is called with `session_id=None`, the diagnostic data cannot be associated with any specific session. The call is silently ignored without any warning or log output. Operators cannot tell why diagnostics are not being saved.

## Reason for change

`save_diagnostic()` should behave consistently with `replace_messages()` by checking `session_id` before attempting to save. When `session_id` is `None`, a WARNING-level log should be emitted so operators can identify the issue.

## Implementation intent

Add a `session_id` guard to `save_diagnostic()` similar to the existing guard in `replace_messages()`. Emit a WARNING log when `session_id` is `None` and skip the save operation.

## Implementation instructions

In `scripts/agent/session.py`, modify the `save_diagnostic()` method around line 66-70:

```python
def save_diagnostic(self, content: str) -> None:
    """Persist a diagnostic event to the session_diagnostics table via DiagnosticStore; never written to messages."""
    if self.session_id is None:
        logger.warning(
            "save_diagnostic called before session.start(); skipping persist"
        )
        return
    self._diagnostic_store.save(
        self.session_id, kind="llm_transport_error", content=content
    )
```

## Acceptance criteria

- When `session_id` is `None`, a WARNING log is emitted and no save attempt is made
- When `session_id` is set, behavior is unchanged (diagnostic saves correctly)
- The warning message is consistent with the one used in `replace_messages()`
- No regression in diagnostic saving after session start

## Tests

- Unit test: Call `save_diagnostic()` before `session.start()`, verify WARNING log is emitted
- Unit test: Call `save_diagnostic()` after `session.start()`, verify data is saved correctly
- Integration test: Verify no diagnostic data is lost during normal operation
