# [High] Make MCP tool-discovery failure visible/fatal now that routing has no fallback

## Priority

High

## Summary

`scripts/shared/route_resolver.py::ToolRouteResolver.resolve()` was recently changed to make
`RuntimeToolRegistry` the **sole** routing authority (no more fallback to `ToolRegistry`'s compile-time
frozensets). This was an intentional, correct change (see `requires/done/20260720_01_require.md`), but
it removed a de facto safety net: if MCP tool discovery fails entirely at startup, no tool can be routed
for the rest of the session — and today that failure is caught and silently downgraded to a "skipped"
line in the startup log, not surfaced as an error.

Verified directly in current source (`scripts/agent/startup.py`, `StartupOrchestrator._check_services()`):

```python
try:
    discovery = await McpToolDiscoveryService(ctx).discover_all()
    # ... per-outcome fatal/warning/ok handling ...
except Exception:  # noqa: BLE001
    pipeline.add_skipped(
        "mcp_tool_discovery", f"MCP tool discovery skipped: {exc}"
    )
```

If `discover_all()` itself raises (e.g. every MCP server is unreachable at startup, or a bug in discovery
itself — as opposed to a per-server outcome that the `try` block already handles via
`add_fatal`/`add_warning`), `ToolExecutor.set_runtime_registry()` is never called, and
`ctx.services_required.tools._resolver._runtime_registry` stays `None` for the entire session. The REPL
still starts (this failure path is non-fatal), but **every single tool call** for that session now
raises `ValueError` from `ToolRouteResolver.resolve()` — a full-session tool-call outage with only a
generic "skipped" line in the startup log to signal it, easy to miss in interactive use.

Under the previous two-tier routing model, this same failure mode left the agent in a degraded-but-usable
state: `ToolRegistry`'s frozensets could still resolve tool names for whichever backing servers happened
to be reachable. Sole-authority routing turned a partial degradation into a total outage without
upgrading the failure's visibility to match.

## Target files

- `scripts/agent/startup.py` (`StartupOrchestrator._check_services()`'s discovery exception handling)
- Possibly `scripts/agent/repl_health.py` (if a post-startup health-gate approach is chosen instead of/in
  addition to a startup-time fatal check)
- `tests/test_startup.py` or wherever `StartupOrchestrator._check_services()` is currently tested

## Already-identified context

This gap was found and documented during the planning cycle for
`requires/done/20260720_01_require.md` (routing sole-authority change) and filed as
`issues/20260720-135023_risks.md`, explicitly as an out-of-scope follow-up (the sole-authority
requirement itself listed `mcp_tool_discovery.py`/`startup.py` under "What NOT to change"). That issue
file already contains a full writeup — read it first; this requirement formalizes its "Recommended
follow-up" section into an actionable spec. Do not duplicate the investigation there.

## Reason for change

A full-session tool-call outage that is only visible as a generic "skipped" pipeline line is exactly the
kind of failure that goes unnoticed until a user reports "nothing works" with no clear signal why. Now
that there is no partial-degradation fallback, this failure mode deserves failure-mode-appropriate
visibility (fatal in production, or an unmistakable warning), not the same treatment as a merely
best-effort diagnostic.

## Intent

After this change, a total MCP tool-discovery failure at startup either:

(a) fails startup fast with a clear fatal message in production security profile (mirroring the existing
`SecurityProfile.PRODUCTION` fatal-on-subprocess-failure pattern already used elsewhere in
`_start_servers()`), or

(b) is surfaced as a distinct, high-visibility `WARNING`/`FATAL` pipeline outcome that is unambiguous
about impact ("all tool calls will fail this session"), not lumped in with routine per-server
"unreachable" warnings.

Pick one approach (or combine, e.g. fatal in production, prominent warning in local/dev profiles) — see
Unknowns below.

## Required changes

### 1. Decide and implement the failure-visibility strategy

Choose one (or a combination) of the three options from `issues/20260720-135023_risks.md`'s
"Recommended follow-up":

1. In `SecurityProfile.PRODUCTION`, make `mcp_tool_discovery` failures fatal (raise instead of
   `pipeline.add_skipped`) — mirrors the existing fatal-on-subprocess-failure pattern in `_start_servers()`.
2. Add an explicit startup/post-startup health check (e.g. in `agent/repl_health.py`) asserting
   `ctx.services_required.runtime_tools` (or equivalent) is non-empty after discovery, surfaced as
   `FATAL` or a prominent `WARNING` pipeline outcome.
3. At minimum, upgrade the log level/wording of the "skipped" case so it unambiguously states that
   **all** tool calls will fail for the session, not just discovery-dependent diagnostics.

### 2. Add test coverage for total discovery failure

Extend `StartupOrchestrator._check_services()`'s existing tests with a case that forces
`McpToolDiscoveryService.discover_all()` to raise, and asserts the chosen behavior (fatal in production /
distinct warning otherwise).

## Unknowns to resolve during planning

- Whether production-fatal (option 1) is acceptable given the agent's other startup failure precedents,
  or whether the project prefers to keep the agent always-starts-something (option 2/3) — resolve by
  checking how other `_check_services()` steps in `startup.py` already handle similarly severe failures
  (e.g. `_start_servers()`'s fatal-on-subprocess-failure precedent cited above) and matching that
  project convention rather than inventing a new one.

## Acceptance criteria

- A discovery-failure scenario (e.g. all MCP servers unreachable at startup, or `discover_all()` raising
  directly) either fails startup fast with a clear fatal message in production profile, or is surfaced as
  an unmistakable, high-visibility warning distinct from routine per-server "unreachable" warnings.
- Existing tests for `StartupOrchestrator._check_services()` gain a case covering total discovery
  failure and assert the chosen behavior.
- No change to the already-correct per-outcome handling inside the `try` block (fatal/warning/ok for
  individual server discovery results) — this requirement is scoped to the `except Exception` path only.

## Test plan

- New/extended unit test(s) in `tests/test_startup.py` forcing `McpToolDiscoveryService.discover_all()`
  to raise and asserting the chosen fatal/warning behavior.
- `uv run pytest tests/test_startup.py -v`
- `uv run pytest -q` (full suite, confirm no regressions)

## Notes for implementer

### Out of scope

- Any change to `McpToolDiscoveryService` itself, or to `ToolRouteResolver`/`ToolExecutor`'s
  sole-authority routing behavior — this requirement only addresses how a total discovery failure is
  surfaced at startup, not the routing model itself.
- The already-correct per-server discovery outcome handling (`add_fatal`/`add_warning`/`add_ok` inside
  the `try` block) — unchanged.
