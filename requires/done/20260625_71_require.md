## [High][mcp][rag-pipeline] Remove the backward-compatibility `POST /v1/search` endpoint from rag-pipeline-mcp

### Summary
`rag-pipeline-mcp` still exposes `POST /v1/search` as a backward-compatibility endpoint for external RAG service delegation. 【1-3c9e72】

### Problem
This leaves two entry paths for the same subsystem:
- the canonical MCP tool API (`POST /v1/call_tool`)
- the backward-compatibility `/v1/search` endpoint

Keeping both paths increases maintenance cost, weakens API clarity, and delays full migration to the MCP-native tool interface. 【1-3c9e72】【3-37d99d】

### Scope
- `scripts/mcp/rag_pipeline/*`
- `rag-pipeline-mcp` server routing
- server tests
- MCP server catalog docs

### Task
- remove `POST /v1/search` from `rag-pipeline-mcp`
- keep only the canonical MCP endpoints:
  - `POST /v1/call_tool`
  - `GET /v1/tools`
  - `GET /health`
- ensure all RAG access goes through MCP tools such as:
  - `rag_run_pipeline`
  - `rag_debug_pipeline`
  - `rag_list_documents`
  - `rag_delete_document`
- remove any code paths, handlers, or tests that still depend on `/v1/search`

### Expected Result
`rag-pipeline-mcp` exposes only the canonical MCP API surface, with no backward-compatibility HTTP search endpoint remaining. 【1-3c9e72】【3-37d99d】

### Acceptance Criteria
- `POST /v1/search` no longer exists
- all supported RAG behavior is reachable through MCP tool calls only
- tests no longer depend on `/v1/search`
- docs no longer describe `/v1/search` as an available endpoint
