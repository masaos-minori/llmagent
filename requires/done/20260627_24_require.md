# Title
Align MCP response truncation metadata and visible byte count

## Priority
Low

## Labels
- mcp
- truncation
- protocol
- documentation
- tests

## Dependencies
- None

## Summary
Ensure MCP response truncation metadata and suffix text accurately describe the visible output size.

## Background
The protocol docs describe `total_bytes` and `actual_visible_bytes`, but also note that the suffix says it is showing the fixed 512 KB limit rather than the actual truncated byte count.

## Problem
Operators and LLM consumers may misunderstand how much content is actually visible, especially with multi-byte UTF-8 text.

## Required Changes
- Inspect `_truncate_with_meta()` in `mcp/server.py`.
- Confirm how `actual_visible_bytes` is calculated.
- Decide whether suffix text should use:
  - the configured limit, or
  - `actual_visible_bytes`.
- Update implementation or docs so they agree.
- Add tests for:
  - under-limit response
  - over-limit ASCII response
  - over-limit UTF-8 multibyte response
  - valid UTF-8 after truncation

## Acceptance Criteria
- Truncation metadata and suffix text are consistent.
- UTF-8 output is not corrupted.
- Tests verify byte counts and visible text behavior.
- Documentation explains the final behavior clearly.

## Out of Scope
- Changing the global max response size.
- Adding streaming responses.

## Components
- `mcp/server.py`
- `04_mcp_02_protocol_and_transport.md`
- truncation tests

## AI Implementation Instruction
Implement this as a protocol metadata cleanup.

Strict rules:
- Do not break UTF-8 text.
- Do not report misleading visible byte counts.
- Preserve the 512 KB limit unless separately requested.
