# Remove workflow_mode and make workflow execution always required

## Priority

High

## Type

Breaking Change / Workflow / Configuration

## Background

The current implementation supports `workflow_mode` values such as `auto`, `required`, and `disabled`.

The target architecture no longer supports execution modes. Workflow execution must always be enabled and required.

Migration compatibility is not required.

## Problem

Keeping `workflow_mode` allows legacy behavior to remain visible and potentially reachable.

Specifically:

- `disabled` allows the workflow engine to be bypassed.
- `auto` allows startup degradation.
- `required` becomes the only meaningful value.

If only one mode is valid, the mode field itself is unnecessary and misleading.

## Goal

Remove `workflow_mode` entirely and make workflow execution mandatory by design.

## Scope

Affected areas:

- Agent configuration dataclasses
- Config builder / config validation
- Orchestrator constructor
- Workflow execution policy
- `/config`
- `/context`
- `workflow_status()`
- Tests referencing workflow modes

## Implementation Tasks

- Remove `AgentConfig.workflow_mode`.
- Remove `AgentConfig._validate_workflow_mode()`.
- Remove `WorkflowExecutionPolicy`.
- Remove `workflow_mode` from `Orchestrator.__init__()`.
- Reject config files that still contain `workflow_mode`.
- Remove workflow mode display from `/config`, `/context`, and `workflow_status()`.
- Delete or rewrite tests that rely on `workflow_mode`.
- Ensure workflow execution is always initialized before any user turn.

## Acceptance Criteria

- `workflow_mode` no longer exists in `AgentConfig`.
- Config containing `workflow_mode` fails with `ConfigLoadError`.
- Orchestrator no longer receives or stores workflow mode.
- There is no runtime branch based on workflow mode.
- `/config` and `/context` do not display workflow mode.
- All user turns require a valid workflow definition and workflow state store.

## Validation Plan

Add tests for:

- Config with `workflow_mode="auto"` fails.
- Config with `workflow_mode="disabled"` fails.
- Config with `workflow_mode="required"` fails if the field is still present.
- Config without `workflow_mode` succeeds when workflow prerequisites are valid.
- Orchestrator cannot be constructed without workflow readiness.
- No workflow mode appears in command output.

## Risks

This is a breaking change. Existing configs must remove `workflow_mode`.

## Out of Scope

- Backward compatibility.
- Gradual deprecation.
- Migration scripts.
- Dev-only workflow bypass.
