# [High] Define SemanticCache capacity semantics and extend RAG config validation for flat and nested config

## Background

`SemanticCache.prune()` currently uses `self._entries[-self._max_size:]`.

When `semantic_cache_max_size=0`, Python treats `-0` as `0`, which can keep all entries instead of disabling or clearing the cache.

There are no existing `tests/test_rag_cache*.py` files, so RAG cache tests must be added as a new file.

`RagConfigValidator` currently reads mostly from nested `cfg["rag"]`, while the MCP RAG service passes a flat `module_cfg` dictionary to `RagPipeline`. Existing validator tests cover only the nested config shape.

## Goal

Define cache size semantics, prevent the `-0` slicing bug, add cache tests, and make config validation effective for both nested and flat RAG config shapes.

## Scope

- `rag/cache.py`
- `shared/config_validator.py`
- `tests/test_rag_cache.py`
- `tests/test_config_validator.py`
- RAG configuration documentation
