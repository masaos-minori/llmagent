# Requirement: Run RAG Consistency Validation Regardless of Security Profile

## Priority

Low

## Target files

- `scripts/agent/startup.py`

## Background

In `StartupOrchestrator._validate_rag_consistency()`, there is an early return when `ctx.cfg.mcp.security_profile != SecurityProfile.PRODUCTION`. This means non-production deployments never validate RAG consistency — specifically, they do not check whether the RAG database embedding dimensions match the configured model's embedding dimensions.

## Problem

When a developer switches embedding models between sessions (e.g., `text-embedding-3-small` → `text-embedding-3-large`), the old RAG database still contains vectors from the previous model with different dimensions. Without dimension validation, similarity search returns meaningless results because cosine similarity across incompatible dimensions is invalid. In non-production mode, no warning or error is emitted — the operator sees normal-looking behavior but gets wrong answers.

Note: The specific methods `get_embedding_dimension()` and `model_embedding_dim` referenced in the original issue do not currently exist in `rag_maintenance_service.py`. This suggests either the feature was not yet implemented or the issue description predates the current implementation. The underlying concern — non-production mode skipping RAG validation — remains valid.

## Reason for change

RAG consistency must be validated regardless of security profile. Non-production mode should at minimum emit a warning when RAG dimensions are incompatible, so operators know to take corrective action.

## Implementation intent

Remove the security profile guard from `_validate_rag_consistency()` and emit warnings based on severity level: fatal in production mode, non-fatal warning in non-production mode.

## Implementation instructions

1. Remove the early return `if ctx.cfg.mcp.security_profile != SecurityProfile.PRODUCTION: return` from `_validate_rag_consistency()`.
2. Add conditional logic inside the dimension mismatch handler:
   - In production mode: log ERROR and display FATAL message (current behavior)
   - In non-production mode: log WARNING and display NON_FATAL warning message
3. If `get_embedding_dimension()` and `model_embedding_dim` methods do not exist on `rag_maintenance`, implement them first before adding the dimension comparison logic. These would need to read the vector dimension from the RAG database schema and compare it against the configured model's embedding dimension.

## Acceptance criteria

- [ ] RAG dimension validation runs in both production and non-production modes
- [ ] In production mode, dimension mismatch produces a FATAL error and halts startup
- [ ] In non-production mode, dimension mismatch produces a NON_FATAL warning and allows startup to continue
- [ ] No regression in existing RAG functionality

## Tests

- Unit test: Mock RAG maintenance service to report incompatible dimensions, verify non-production mode emits a warning
- Integration test: Configure different embedding models between sessions, start agent in non-production mode, verify warning is displayed
