# [Low] Clarify MDQ parser scope, audit-log policy, and config documentation

## Priority

Low

## Summary

Clarify MDQ's longer-term parser scope, audit-log policy, and configuration documentation.

This issue consolidates lower-priority documentation and design decisions:

- fenced code block indexing behavior
- dedicated MDQ audit log evaluation
- moving removed config-key history into documentation
- future Markdown feature coverage

## Target files

- `scripts/mcp_servers/mdq/parser.py`
- `tests/test_mdq_service.py`
- `tests/test_mdq_parser_code_fences.py`
- `tests/test_mdq_parser_markdown_features.py`
- `config/mdq_mcp_server.toml`
- `docs/04_mcp_04_04_mdq.md`
- `docs/04_mcp_05_05_mdq-enforcement-and-lockdown.md`
- `docs/04_mcp_06_07_reading-audit-logs.md`
- optionally `docs/mdq_config_history.md`

## Background

MDQ's parser is ATX-heading focused and intentionally limited.

Existing tests confirm that `#` inside fenced code blocks is not treated as a heading.

However, current parser behavior appears to skip fenced code block content entirely rather than preserving it as searchable section content.

That is a product/design decision and should be explicit.

Separately, MDQ currently uses shared audit logging through `server_key = "mdq"`. A dedicated `mdq_audit.log` may or may not be needed.

The config file also contains long historical notes about removed keys. These notes are useful, but they can make the active config harder to scan.

## Reason for change

These are not urgent runtime correctness issues, but they affect maintainability, documentation clarity, and future design decisions.

Unclear parser scope can lead to accidental behavior changes.

Unclear audit-log policy can lead to reintroducing dead config keys without implementation.

A long config file can make active settings harder for operators to understand.

## Intent

Make MDQ's documented behavior match implementation and future direction.

Avoid accidental expansion of parser functionality or reintroduction of removed config keys.

## Required changes

### 1. Decide fenced code block content behavior

Choose one behavior.

Option A: include code block content

- Preserve fenced code block text as section content.
- Do not treat headings inside code fences as headings.
- Code snippets become searchable.

Option B: exclude code block content

- Keep current behavior if intentional.
- Document that fenced code block content is omitted from indexed text.
- Tests should lock this behavior.

### 2. Evaluate dedicated MDQ audit log

Document whether shared audit is sufficient.

If dedicated audit is not needed:

- document shared audit as authoritative

If dedicated audit is needed:

- create a separate implementation issue
- do not re-add `audit_log_path` without actual write implementation

### 3. Move removed config-key history into docs

Consider moving long historical notes about removed keys out of the active config file.

Examples of removed historical keys:

- `audit_log_path`
- `concurrency_limit`
- `max_search_results`
- embedding-related keys
- summary-cache keys
- `enable_refresh`
- `status`

Keep a short pointer in the config file if useful.

### 4. Clarify future Markdown feature coverage

Document support or non-support for:

- Setext headings
- GFM tables
- YAML frontmatter metadata search
- tags or categories from frontmatter
- HTML headings
- MDX

## Acceptance criteria

- Fenced code block content behavior is documented and tested.
- Dedicated MDQ audit-log decision is documented.
- Removed config-key history is preserved in documentation if moved.
- Active config file remains operator-friendly.
- Markdown parser support scope is explicit.
- No parser behavior changes occur without tests.

## Test plan

If parser behavior changes, add tests for:

- code fence with `# Not a heading`
- code fence containing a unique searchable term
- indexing and searching the unique term
- mixed Markdown text and fenced code content

If only documentation changes are made, run documentation linting if available.

## Notes for implementer

This is a low-priority design and documentation cleanup issue.

Do not expand parser behavior without explicit approval.
