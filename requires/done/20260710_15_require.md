# [High] Add deterministic ToolRegistry snapshot tests for server groups, tool counts, and ordering

## Background

The implementation derives MCP tool ownership from `ToolRegistry`, which is populated from `tool_constants.py` frozensets.

The current implementation appears to register the following server groups:

- `file_read`
- `file_write`
- `file_delete`
- `rag_pipeline`
- `cicd`
- `mdq`
- `git`
- `shell`
- `github`
- `web_search`

Documentation and tests should prevent drift between implementation, server catalog, and tool counts.

`ToolRegistry.get_tool_names(server_key)` currently returns names in registration order. Default registry population registers sorted tool names, but the ordering contract is not explicitly documented.

## Goal

Ensure ToolRegistry output is deterministic and protected by snapshot-style tests.

## Scope

- `shared/tool_registry.py`
- `shared/tool_constants.py`
- `tests/test_tool_registry.py`
- `tests/test_tool_constants.py`
- MCP documentation
- Documentation generation, if applicable

## Tasks

- [ ] Add a test for `get_registry().get_servers()`.
- [ ] Assert the expected server key list.
- [ ] Add a test for `get_registry().get_all_tool_names()`.
- [ ] Assert the expected total tool count.
- [ ] Add per-server tool count assertions.
- [ ] Decide whether `get_tool_names(server_key)` should guarantee sorted order.
- [ ] If ordering is guaranteed, document it in the method docstring.
- [ ] If ordering is not guaranteed, update tests to sort before comparison.
- [ ] Add a unit test for the chosen ordering behavior.
- [ ] Update MCP documentation to match implementation-derived server and tool counts.
- [ ] If documentation claims an additional server or tool, identify whether implementation or documentation is wrong.
- [ ] Ensure documentation generation does not depend on unspecified ordering.

## Acceptance Criteria

- [ ] CI fails if a tool is added without updating expected counts.
- [ ] CI fails if a server group is added or removed without updating expectations.
- [ ] Documentation server count matches ToolRegistry.
- [ ] Documentation tool count matches ToolRegistry.
- [ ] Ordering behavior is explicit.
- [ ] Tests do not accidentally rely on undocumented ordering.
- [ ] Test failure messages clearly indicate which count, server key, or ordering assumption drifted.

## Notes

Treat ToolRegistry and `tool_constants.py` as the implementation source for ownership tests. Do not hardcode stale documentation values without verifying implementation.
