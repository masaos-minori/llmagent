# [High] Promote DESIGN-3 to a confirmed RAG design decision

## Goal

Promote DESIGN-3, "Separation of responsibilities among `documents`, `chunks`, `chunks_fts`, `chunks_vec`", from a needs-confirmation item to a confirmed RAG design decision.

## Background

The RAG docs consistently describe `documents` as URL-level metadata, `chunks` as canonical chunk text and position data, `chunks_fts` as a derived FTS5/BM25 index, and `chunks_vec` as a derived sqlite-vec KNN index. 【1-54c80f】【2-58c0bb】

## Scope

Documentation and design invariant clarification only.

Do not change runtime behavior in this issue unless tests reveal a contradiction.

## Implementation Tasks

- Update `03_rag_90_inconsistencies_and_known_issues.md`.
- Change DESIGN-3 type from `Needs confirmation` to `Confirmed design decision`.
- Define the table responsibility boundaries:
  - `documents`: canonical URL/document metadata.
  - `chunks`: canonical chunk text and position data.
  - `chunks_fts`: derived FTS5/BM25 full-text index.
  - `chunks_vec`: derived sqlite-vec KNN vector index.
- Document that application code should treat `documents` and `chunks` as canonical data.
- Document that `chunks_fts` and `chunks_vec` are derived indexes.
- Document that `chunks_fts` and `chunks_vec` have different sync mechanisms:
  - `chunks_fts`: trigger-based sync.
  - `chunks_vec`: ingestion-time insert and explicit delete.

## Acceptance Criteria

- DESIGN-3 is no longer listed as `Needs confirmation`.
- DESIGN-3 is documented as a confirmed design decision.
- Canonical tables and derived indexes are clearly distinguished.
- Docs no longer imply that `chunks_fts` or `chunks_vec` are canonical data stores.
