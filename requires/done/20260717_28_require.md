# [High] Replace fragile MDQ audit detail parsing with structured metadata

## Priority

High

## Summary

Make MDQ audit details accurate by deriving them from structured metadata instead of parsing formatted output strings.

## Target files

- `scripts/mcp_servers/mdq/server.py`
- `scripts/mcp_servers/mdq/service.py`
- `scripts/mcp_servers/mdq/search.py`
- `scripts/mcp_servers/mdq/indexer.py`
- `scripts/mcp_servers/mdq/db_grep.py`
- `scripts/mcp_servers/mdq/audit_target.py`
- `tests/test_mdq_audit.py`
- `tests/test_mdq_service.py`

## Background

MDQ currently enriches audit detail in `server.py` by inspecting formatted output strings.

Examples:

- `search_docs.result_count` is inferred using a separator count.
- `index_paths.indexed_count` is inferred by searching for text such as `Indexed:`.
- `refresh_index` properties are extracted from formatted text.

These assumptions do not fully match current output formats.

For example, `search_docs` output is not actually separated by that marker, and `index_paths` returns a simple completion string.

## Reason for change

Audit logs are used for troubleshooting, security review, and operational accountability.

If audit detail is computed by fragile output parsing, audit fields such as result count, indexed count, deleted count, match count, and truncation state may be inaccurate.

This weakens observability and can mislead operators.

## Intent

Make audit metadata accurate and implementation-friendly.

Audit detail should be derived from structured execution metadata, not from user-facing formatted strings.

## Required changes

### 1. Introduce structured internal metadata

Add an internal structure such as:

- text output
- metadata dictionary
- error type
- duration milliseconds

Do not necessarily change the public MCP response shape.

### 2. Add metadata for search_docs

Capture:

- query preview
- result count
- shown count
- truncated flag
- duration milliseconds
- total count if available

### 3. Add metadata for index_paths

Capture:

- input path count
- indexed file count
- skipped file count
- failed file count
- duration milliseconds

If `index_paths` currently returns only `"Indexing complete"`, preserve that external text while creating internal metadata.

### 4. Add metadata for refresh_index

Capture:

- indexed count
- skipped count
- deleted count
- failed count
- elapsed seconds

### 5. Add metadata for grep_docs

Capture:

- pattern preview
- path filter count
- match count
- truncated flag
- grep enabled

### 6. Update audit emission

Update `server.py` so audit detail uses structured metadata.

Avoid parsing formatted text to infer operational detail.

## Acceptance criteria

- `search_docs.result_count` in audit is accurate.
- `index_paths.indexed_count` in audit is accurate.
- `refresh_index` audit fields are accurate.
- `grep_docs.match_count` in audit is accurate.
- Authorization errors and validation errors are audited with useful error details.
- Existing user-facing tool output remains compatible unless explicitly changed.
- Tests cover success and error audit paths.

## Test plan

Add tests for:

- audit detail on `search_docs`
- audit detail on `index_paths`
- audit detail on `refresh_index`
- audit detail on `grep_docs`
- audit detail on authorization error
- audit detail on validation error
- no fragile dependency on formatted text markers

## Notes for implementer

Do not compute audit fields by parsing final formatted strings.

If a full metadata refactor is too large, introduce a small internal result wrapper first and migrate tool handlers incrementally.
