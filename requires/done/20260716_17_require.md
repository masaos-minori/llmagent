## Background

The Agent documentation states that the split documentation set is the formal reference and that older monolithic or removed files should no longer be used. It also includes a documentation consistency checklist covering command references, database schema references, and diagnostics references.

The documentation also contains many `Current behavior` notes that compare current implementation with simplified diagrams, older descriptions, or intended architecture.

Both areas affect AI implementation readiness. If documentation is stale or ambiguous, AI implementation agents can follow obsolete references or treat temporary differences as stable design.

## Problem

Manual documentation maintenance is prone to drift. Broken links, removed file references, stale command references, obsolete diagnostics references, and unresolved `Current behavior` notes can accumulate.

A `Current behavior` note can mean several different things:

- accepted current specification,
- temporary implementation mismatch,
- documentation mismatch,
- issue already tracked,
- obsolete note that should be removed.

Without triage and validation, maintainers and AI agents cannot know which interpretation is intended.

## Reason for change

These documents are intended to guide implementation. Therefore, documentation must remain mechanically checkable where possible and explicitly triaged where semantic judgment is required.

This issue improves documentation reliability by combining automated CI checks with a structured process for classifying current-behavior notes.

## Design intent

Create two mechanisms:

1. CI checks for objective documentation drift.
2. A triage workflow for subjective or semantic current-behavior notes.

Objective checks should catch:

- broken internal links,
- references to removed legacy files,
- slash command documentation drift,
- DB schema documentation drift where feasible,
- obsolete diagnostics references.

Current-behavior notes should be classified into:

- accepted specification,
- implementation fix required,
- documentation fix required,
- issue already tracked,
- obsolete and removable.

## Implementation instructions

### Documentation validation CI

- Add or extend a documentation validation script.
- Detect broken internal links in Agent and MCP documentation.
- Prohibit references to removed legacy documentation files.
- Compare slash command documentation with `_COMMANDS`.
- Compare DB documentation references with `schema_sql.py` and initialization scripts where feasible.
- Detect obsolete diagnostics references.
- Add validation to CI.
- Ensure error output identifies file path, section, and offending reference.

### Current behavior triage

- Extract all `Current behavior` sections from Agent and MCP documentation.
- Create a triage report with file, section, short summary, and classification.
- Promote accepted current behavior into normal documentation.
- Convert implementation mismatches into GitHub issues.
- Convert documentation mismatches into documentation tasks.
- Remove obsolete notes after verification.
- Add guidance for future `Current behavior` notes.

## Acceptance criteria

- CI fails on broken internal documentation links.
- CI fails on references to removed legacy documentation files.
- CI detects slash command documentation drift.
- CI detects obsolete diagnostics references.
- Documentation validation output is actionable.
- All current-behavior notes are classified.
- Accepted behavior is promoted into normal documentation where appropriate.
- Temporary mismatches are tracked as issues or documentation tasks.
- Obsolete notes are removed after verification.

## Non-goals

- Do not rewrite the entire documentation set.
- Do not require semantic verification of every paragraph.
- Do not automatically delete current-behavior notes without review.
- Do not resolve every implementation mismatch inside this issue.
