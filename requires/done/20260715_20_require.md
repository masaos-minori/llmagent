# Make ToolExecutor cache safe for side-effecting tools and preserve cache attribution

## Priority

High

## Type

Implementation / Safety / Observability / Testing

## Target Files

- `shared/tool_executor.py`
- `shared/tool_cache.py`
- `shared/transport_dto.py`
- `shared/tool_executor_helpers.py`
- `tests/test_tool_executor.py`

## Summary

Update ToolExecutor caching so that only safe read-only tool calls are cached, while side-effecting tools always execute live.

Also improve cached result attribution so diagnostics and audit logs can identify which server produced the cached result.

## Background

ToolExecutor currently applies TTL caching to successful tool results. This is useful for read-only tools but unsafe for tools with side effects.

For example, caching is appropriate for tools such as:

- `read_text_file`
- `list_directory`
- `search_files`
- `search_docs`
- `get_workflow_status`

Caching is unsafe for tools such as:

- `write_file`
- `delete_file`
- `shell_run`
- `git_push`
- `github_merge_pull_request`
- `trigger_workflow`
- `rag_delete_document`
- `fts_rebuild`

If a side-effecting tool call is cached, a later call with the same arguments may return the cached text without actually performing the operation. That is unsafe and misleading.

Cached results also currently lose attribution data such as `server_key`, because `CacheEntry` stores only output, error flag, and timestamp.

## Why This Needs to Be Fixed

This is a correctness and safety issue.

A cached result for a mutation tool can create a false impression that an operation was executed when it was not. This can be especially dangerous for:

- file writes/deletes
- Git and GitHub mutations
- shell execution
- CI/CD workflow triggers
- index rebuilds
- document deletion

Separately, losing `server_key` on cache hits weakens observability. If a cached result is later audited or inspected, there is no clear attribution to the MCP server that originally produced it.

## Fix Intent

The cache should be read-only by default.

The intended behavior is:

- Plugin tools keep their existing bypass behavior.
- Side-effecting MCP tools bypass cache completely.
- Read-only MCP tools continue to use cache and stampede protection.
- Cache hits preserve enough metadata to remain observable.

## Required Changes

### 1. Bypass cache for side-effecting tools

Update `ToolExecutor.execute()` so that side-effecting MCP tools call `_raw_execute()` directly.

Required behavior:

- If a plugin returns a result, return it.
- Else if `is_side_effect(tool_name)` is true, call `_raw_execute()`.
- Else call `_execute_with_cache()`.

### 2. Preserve cache for read-only tools

Read-only tools must continue to support:

- TTL cache
- LRU eviction
- inflight future sharing
- successful-result-only caching

### 3. Preserve server attribution on cache hits

Enhance cache storage so cached results can retain attribution.

Recommended approach:

- Add `server_key` to `CacheEntry`.
- Store `result.server_key` when caching a successful result.
- Return cache hit results with the original `server_key`.

Optional improvement:

- Set `source="cache"` for cache hit results.
- If `source="cache"` is introduced, update audit and tests accordingly.

### 4. Update tests that currently cache write tools

Existing cache tests that use `write_file` should be changed to a read-only tool, such as `read_text_file`.

Add explicit tests proving that side-effecting tools bypass cache.

## Acceptance Criteria

- Side-effecting tools are never cached.
- Side-effecting tools do not share inflight cache futures.
- Repeated side-effecting tool calls execute repeatedly.
- Read-only tools still use cache.
- Stampede protection still works for read-only tools.
- Cached results preserve `server_key`, or the decision not to preserve it is explicitly documented.
- Existing ToolExecutor error classification behavior remains unchanged.
- Tests no longer depend on caching `write_file`.

## Suggested Tests

- `test_side_effect_tool_bypasses_cache`
- `test_write_file_bypasses_cache`
- `test_delete_file_bypasses_cache`
- `test_shell_run_bypasses_cache`
- `test_trigger_workflow_bypasses_cache`
- `test_fts_rebuild_bypasses_cache`
- `test_read_text_file_uses_cache`
- `test_read_only_cache_stampede_protection`
- `test_cache_hit_preserves_server_key`

## Non-Goals

- Do not remove caching entirely.
- Do not change plugin execution semantics.
- Do not change transport error classification in this issue unless required by cache attribution changes.

## Implementation Notes

Side-effect detection should come from shared classification logic, not a local list in ToolExecutor.

If `source="cache"` is added, update any tests or audit code that currently expects an empty source on cache hits.
