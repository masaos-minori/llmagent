# Reduce documentation drift by auto-generating API, config, and CLI reference sections

## Summary
Reduce future documentation inconsistency by auto-generating stable reference sections from source definitions where practical.

## Background
The current RAG documentation is rich but already shows drift in naming, file references, and behavior descriptions.

## Objective
Move stable reference material to generated output so that hand-written documents focus on design intent and caveats.

## In Scope
- Reference sections for:
  - config keys
  - CLI arguments
  - selected API signatures
- Relevant RAG documentation files
- Documentation build or generation workflow if one exists

## Required Changes
- Identify reference sections that can be generated safely
- Prefer generation for:
  - config key lists
  - CLI argument lists
  - selected public API signatures
- Keep hand-written sections for:
  - design rationale
  - operational caveats
  - known issues
- Document the generation workflow

## Acceptance Criteria
- At least one stable documentation section is generated from source definitions
- The generation path is reproducible
- Hand-written and generated sections have a clear boundary
- The RAG document guide reflects the new maintenance model

## Out of Scope
- Full site-generation platform migration
- Rewriting all documents into a new docs framework

## Implementation Notes
Start small. Focus first on the sections most likely to drift, especially config and CLI references.
