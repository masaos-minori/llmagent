# [High] Normalize MDQ search production behavior and remove unsupported hybrid/embedding surface

## Priority

High

## Type

Implementation / Search Behavior / Configuration Cleanup / Validation

## Target files

- `mcp_servers/mdq/search.py`
- `mcp_servers/mdq/service.py`
- `mcp_servers/mdq/tools.py`
- `mcp_servers/mdq/models.py`
- `config/mdq_mcp_server.toml`
- Relevant MDQ search tests

## Consolidated from

- Remove MDQ hybrid / embedding search stub from production surface
- Clarify `max_search_results` and `max_results_limit` responsibilities
- Keep RAG/session vector schema separate from MDQ hybrid cleanup

## Background

`search.py` contains hybrid/vector search scaffolding. `_search_vector()` explicitly states that embedding generation and vector search would require LLM API integration and currently returns an empty result list. `service.py` reads embedding-related settings such as `use_embedding`, `vector_table`, `embedding_model`, and `embedding_dims`. `tools.py` exposes `search_docs.mode` with wording that mentions unsupported hybrid behavior. 【5-3ed2c1】【6-51b4fa】【2-03d74e】

`service.py` also reads both `max_search_results` and `max_results_limit`, while `tools.py` exposes `max_results_limit` as a request-level field for `search_docs`. 【6-51b4fa】【2-03d74e】

Separately, `schema_sql.py` defines vector tables such as `chunks_vec` and `memories_vec` for RAG/session schemas. These are not the same as MDQ hybrid search and must not be removed as part of MDQ search cleanup. 【7-409e83】

## Problem

The MDQ search surface currently suggests that hybrid or embedding-backed search may exist, but the actual vector path is a placeholder. This can mislead users, tests, and configuration owners into believing that enabling embedding-related options activates real hybrid search.

At the same time, search result limit configuration is unclear because both `max_search_results` and `max_results_limit` exist without an explicitly documented responsibility split.

## Reason for change

A production tool schema should only advertise behavior that is actually implemented. Returning empty vector results from an unsupported hybrid path is ambiguous and can hide configuration errors.

Search limit semantics also need to be deterministic. Without a clear distinction between default result count and hard cap, request-level overrides may behave inconsistently or bypass intended limits.

## Design intent

MDQ search should expose only supported production behavior. Unsupported modes must fail clearly and deterministically. Search limit configuration should have a single documented model, such as:

- one setting for default result count,
- one setting for maximum server-side cap,
- request-level overrides allowed only within the cap.

The cleanup must not affect RAG/session vector schema, because those tables are defined for different subsystems. 【7-409e83】

## Implementation instructions

### Search mode cleanup

- Review `SearchDocsRequest.mode` in `models.py`.
- Restrict accepted modes to implemented production modes only.
- Remove or clearly reject unsupported `hybrid` mode.
- Update `tools.py` schema description so it does not advertise unsupported hybrid behavior.
- Remove `_search_vector()` and `_merge_hybrid()` from production flow, or move them behind a clearly experimental path that cannot be enabled accidentally.
- If `use_embedding` is not implemented for MDQ search, remove the config read or make it fail fast with a clear validation error.
- Add deterministic validation errors for unsupported modes.

### Search limit cleanup

- Identify all uses of `max_search_results`.
- Identify all uses of `max_results_limit`.
- Decide the canonical responsibility of each setting.
- If one setting is obsolete, remove it or mark it deprecated.
- Ensure request-level limits cannot exceed the server-side cap.
- Update schema descriptions and config comments to match actual behavior.

### Scope protection

- Do not remove `chunks_vec` or `memories_vec` from RAG/session schemas.
- Do not treat generic RAG/session vector storage as MDQ hybrid search.
- Document the subsystem boundary if needed.

## Required tests

Add or update tests for:

- supported search modes,
- unsupported search modes,
- disabled or removed MDQ embedding configuration,
- default result limit,
- request-level result limit,
- server-side cap enforcement,
- schema description consistency if applicable.

## Acceptance criteria

- `search_docs` does not silently execute an unsupported hybrid path.
- Unsupported modes fail with a clear validation error.
- Tool schema only describes implemented production capabilities.
- Search result limit behavior is deterministic and documented.
- Request-level overrides cannot bypass server-side caps.
- RAG/session vector tables remain untouched.

## Non-goals

- Do not implement full hybrid search in this issue unless all required embedding generation, vector search, ranking merge, schema, config, tests, and docs are included.
- Do not remove RAG/session vector schema.
- Do not modify unrelated RAG pipeline behavior.
