 # Structured Output First Migration

You are required to produce a concrete implementation proposal for migrating the system to a **Structured Output First** architecture.
You must treat this as a **mandatory architectural redesign**, not as an optional improvement or a local refactor.

## Mission

You must redesign the system so that:
- LLM responses,
- tool call responses,
- workflow responses, and
- MCP payloads

are no longer interpreted primarily through natural language.

From this point forward, these outputs must be handled as **schema-driven structured data** based on explicit JSON contracts.

Do not rely on natural-language interpretation for decision signals such as `needs_more_context`.
Do not infer decisions from prose.
Do not preserve loosely structured response patterns.

Instead, represent all decisions explicitly as structured JSON fields and typed action/result objects.

## Non-Negotiable Design Requirements

### 1. Structured Output First
You must make structured JSON the default output format for all machine-interpreted responses.
Natural language is allowed only as a final presentation layer.

You must use structured output for:
- branching
- retry control
- validation
- persistence
- replay
- audit logging
- orchestration

You must not use free-form prose as the canonical source for machine decisions.

### 2. Replace flag-based decisions with action-oriented contracts
You must not model decisions as isolated booleans such as `needs_more_context`.

You must replace such flags with explicit action/result objects that include fields such as:
- `action`
- `reason`
- `required_context`
- `next_step`
- `confidence`

This contract must be reusable across:
- LLM output handling
- RAG decision logic
- tool call dispatch
- workflow branching
- MCP payload processing

### 3. MCP payloads must be schema-driven
You must not interpret MCP payloads through natural language.

You must define explicit:
- request schemas
- response schemas
- error schemas

At minimum, MCP contracts must make the following explicit:
- tool selection
- authentication requirement
- payload version
- action name
- input structure
- output structure
- error structure

Unknown tool names, unknown tiers, and unknown metadata must fail fast through schema validation.
Do not allow fail-open behavior.

## Current-State Assessment You Must Address

The current codebase already shows partial movement toward typed I/O, but the final boundaries still collapse into strings, unvalidated dictionaries, or direct display rendering.

Examples of the current problems include:
- returning `list[str]` instead of structured response objects,
- building snippets as raw strings instead of typed records,
- returning `dict[str, Any]` from internal collection functions,
- converting structured internal results into human-readable text too early,
- relying on direct `print()` instead of presenter/output-port rendering.

You must explicitly identify these weak points and treat them as migration targets.

## Required Proposal Scope

Your proposal must cover all of the following implementation areas.

### A. LLM prompt / parser implementation
You must:
- redesign prompts so that their primary purpose is to generate schema-compliant JSON,
- replace natural-language parser logic with JSON decoding plus schema validation,
- classify parse failures explicitly, for example:
  - `invalid_schema`
  - `missing_field`
  - `unexpected_enum`
  - `invalid_payload_type`

You must not leave LLM output as unstructured text if downstream layers are typed.

### B. RAG decision logic
You must:
- replace boolean-style `needs_more_context` logic with action-based structured responses,
- include fields such as:
  - `action: "retrieve_more_context"`
  - `reason`
  - `query_rewrite`
  - `required_scope`
  - `priority`
- structure the reason why additional context is required

You must not express RAG decisions only through human-readable text.

### C. MCP payload schema
You must:
- define MCP request, response, and error payloads explicitly,
- require fields such as:
  - `action`
  - `payload_version`
  - `tool_name`
  - `auth_mode`
  - `input`
  - `output`
  - `errors`
- enforce schema validation as the first validation gate

You must not allow unknown tools, unknown tiers, or unknown metadata to pass silently.

### D. Workflow definitions
You must:
- redesign workflow responses and workflow steps into explicit structured action/state definitions,
- include fields such as:
  - `action`
  - `input`
  - `output`
  - `error`
  - `transition`
- represent workflow progression as a schema-compliant action chain
- represent human approval and retry conditions structurally

You must not express workflow progression as vague natural-language instructions.

## Required Common Contract

You must define and use a common action/result schema similar to the following:

```json
{
  "action": "retrieve_more_context | call_tool | continue | ask_user | fail",
  "reason": "string",
  "required_context": ["string"],
  "payload": {},
  "errors": [],
  "confidence": 0.0
}
```

You must explain how this shared contract is reused across:
- LLM decision outputs
- tool call results
- workflow branches
- MCP payloads

## Output Responsibility Separation

You must stop collapsing internal structured results into display strings too early.

You must keep internal boundaries JSON/DTO based.
Only the final presentation layer may convert structured outputs into human-readable text.

You must move rendering responsibility into a presenter or output-port layer.
You must reduce or eliminate direct `print()` usage in machine-facing paths.

## Required Implementation Roadmap

Your proposal must include a phased execution plan with at least the following stages.

### Phase 1: Contract definition
You must:
- define the common action schema,
- define separate schemas for:
  - LLM responses
  - MCP payloads
  - workflow steps
- define fail-fast validation error categories

### Phase 2: Parser replacement
You must:
- inventory all natural-language parsing points,
- replace them with JSON decode + schema validation,
- add telemetry for parser failure and schema mismatch

### Phase 3: Workflow / MCP integration
You must:
- unify tool call and workflow responses under the action contract,
- introduce versioned schema handling for MCP payloads,
- enforce schema validation for authentication, tool selection, and metadata

### Phase 4: Output-layer separation
You must:
- reduce dependence on:
  - `list[str]`
  - `dict[str, Any]`
  - direct `print()`
- route final rendering through presenters or output ports

## Required Benefits Analysis

You must explain the expected benefits of this migration, including at least:
- more stable branching logic,
- deterministic parsing and retry behavior,
- better auditability and replayability,
- stronger consistency across LLM, RAG, MCP, tool call, and workflow layers,
- better maintainability and long-term extensibility.

## Final Instruction

You must treat this migration as a **system-wide contract redesign**.
Do not frame it as a cosmetic cleanup.
Do not frame it as an optional future enhancement.
Do not propose preserving natural-language-first logic where machine decisions depend on it.

You must present the migration in the following priority order:
1. Structured Output First
2. Unified action-oriented contracts
3. JSON Schema validation at every I/O boundary

Your output must be written as a concrete implementation proposal that can be used directly as a design and execution guide.
Return the proposal in Markdown format.
