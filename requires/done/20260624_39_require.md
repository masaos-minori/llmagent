# Add a unified turn-state mutation table for errors, diagnostics, and history changes

## Summary
Refactor the turn-flow and state/persistence docs so that all mid-turn error artifacts and history mutations are shown in one explicit state-transition model.

## Background
The current turn-flow and persistence chapters describe the right concepts, but the error-path mutations are spread across prose sections. This makes it hard to answer simple questions like:
- what enters conversation history
- what stays diagnostic-only
- what persists to DB
- what appears in `/tool`

## Objective
Document turn-time state mutation as a table-driven model.

## In Scope
- `05_agent_03_turn-processing-flow.md`
- `05_agent_04_state-and-persistence.md`

## Required Changes
- Add one state-transition table covering at least:
  - normal assistant response
  - partial LLM completion
  - pre-stream LLM failure
  - tool continuation failure
  - guard hint event
  - immediate user-message rollback
- For each case, show:
  - history mutation
  - DB persistence
  - diagnostic artifact write
  - operator visibility surface

## Acceptance Criteria
- A reader can trace each error-path artifact without jumping across multiple prose sections
- Historical context mutation and diagnostic isolation are clearly separated
- Turn-flow and persistence chapters describe the same behavior model

## Out of Scope
- Changing error handling
- Changing guard semantics

## Notes
Use a structured state-transition table instead of only prose.
