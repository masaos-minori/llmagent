# Add ToolResultCache and move TTL/LRU cache management out of ToolExecutor

## Priority

High

## Goal

Move tool result cache key generation, TTL validation, LRU update, store, and eviction logic out of `ToolExecutor`.

`ToolExecutor` should no longer directly manage an `OrderedDict[str, CacheEntry]`.

## Background

`shared/tool_cache.py` currently contains only `CacheEntry`. Cache behavior currently remains inside `ToolExecutor`, including:

- cache key generation
- TTL validation
- expired entry deletion
- LRU update
- storing successful results
- max-size eviction

## Scope

Affected files:

- `shared/tool_cache.py`
- `shared/tool_executor.py`
- tests for cache behavior

## Implementation requirements

Add `ToolResultCache` to `shared/tool_cache.py`.

Expected API:

    class ToolResultCache:
        def make_key(self, tool_name: str, args: dict[str, Any]) -> str:
            ...

        def get_result(self, key: str) -> ToolCallResult | None:
            ...

        def store_if_success(self, key: str, result: ToolCallResult) -> None:
            ...

        def clear(self) -> None:
            ...

        def apply_config(self, ttl: float, max_size: int) -> None:
            ...

Required behavior:

- Cache key must remain `f"{tool_name}:{_json_dumps(args)}"`.
- Only successful results are cached.
- Plugin tools are not cached.
- Cache hit result must use `request_id=""`.
- Cache hit result must use `server_key=""`.
- Cache hit result must preserve `error_type="tool"` only when cached result is an error, though error results should not normally be stored.
- TTL expiration must remove expired entries.
- Cache hit must update LRU ordering.
- Max-size eviction must remove the least recently used entry.

Update `ToolExecutor._execute_with_cache()` to delegate cache behavior to `ToolResultCache`.

## Acceptance criteria

- `ToolExecutor` no longer directly stores `OrderedDict[str, CacheEntry]`.
- `_store_and_evict()` is removed or fully replaced by `ToolResultCache.store_if_success()`.
- `_execute_with_cache()` remains responsible for calling `_execute_with_stampede_protection()` on cache miss.
- `ToolExecutor.stat_cache_hits` may remain in `ToolExecutor` for Phase 1.
- Existing cache behavior remains unchanged.

## Validation plan

- Add `tests/shared/test_tool_result_cache.py`.
- Test cache key generation.
- Test cache hit result shape.
- Test TTL expiration.
- Test LRU move-on-hit.
- Test max-size eviction.
- Test error result is not stored.
- Test plugin path bypasses cache.

## Out of scope

- Moving stampede protection into the cache class.
- Changing cache key format.
- Adding persistent cache.
