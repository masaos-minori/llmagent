## Title: [MEDIUM] Add Connection Pooling / Reuse Options to `SQLiteHelper`

### Context
`SQLiteHelper` opens and closes a SQLite connection *per-query* (`Opened per-query (not connection pool)`).

### Problem Statement
While WAL mode is active, the constant overhead of opening and closing file handles causes unnecessary I/O performance bottlenecks under high-frequency workloads, such as rapid chat log insertion or heavy FTS5 indexing.

### Requirements
- **Connection Management Refactor**: Add a connection reuse mechanism to `SQLiteHelper`. Support an optional session-scoped or thread-local persistent connection mode.
- **Context Manager Compatibility**: Ensure that the existing context manager syntax (`with SQLiteHelper().open(...) as db:`) remains fully compatible, minimizing structural changes across dependent consumer modules.

### Constraints
- Do NOT cause multi-threaded write conflicts; strictly manage isolation or lock ownership when a shared connection is used for writing.
- Ensure that connections are explicitly closed when the session/REPL terminates.

### Acceptance Criteria (Definition of Done)
- [ ] Profiling shows a reduction in file handle openings during multi-turn chats.
- [ ] All existing database unit tests pass without modifying their core query assertions.
