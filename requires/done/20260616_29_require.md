# [Agent][Config] Unify config loading and make reload results explicit

## Summary
Configuration is loaded from many files, but not all relevant config sources are part of the same loading path.
At the same time, `/reload` updates only part of the total runtime configuration.

## Problem
This makes it easy to misunderstand what is actually reloaded, what is only read indirectly, and what still requires restart.

## Scope
- `build_agent_config()`
- config loading path
- `/reload` reporting
- config-to-runtime synchronization

## Task
- make configuration boundaries more explicit
- include independently loaded config sources in the visible config model where practical
- make `/reload` print:
  - what changed
  - what was applied
  - what still requires restart

## Expected Behavior
- operators can clearly see reloadable vs non-reloadable settings
- configuration ownership is easier to understand
- fewer hidden config sources exist

## Acceptance Criteria
- [ ] reload output shows applied vs restart-required changes
- [ ] config ownership is documented clearly
- [ ] indirectly loaded config sources are visible or explicitly declared
- [ ] confusing key-to-field name mappings are documented
