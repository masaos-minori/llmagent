# Title
Make memory-layer activation and startup behavior production-friendly

## Priority
Medium

## Labels
- enhancement
- memory
- startup

- operations

## Dependencies
- Issue 1 recommended

## Summary
Improve the activation and startup story for the memory layer so operators can clearly understand whether it is disabled, degraded, FTS-only, or fully active. The current design uses `use_memory_layer=False` by default and treats activation as startup-only.

## Background
The current configuration marks `use_memory_layer` as startup-only, and the system can degrade gracefully when embeddings are unavailable by falling back to FTS-only retrieval. `/memory status` already exists, but startup and runtime status can be more explicit.

## Problem
Persistent Semantic Memory is hard to operate as a real feature if activation state is technically present but operationally ambiguous.

## Required Changes
- Add startup readiness checks for the memory subsystem.
- Make disabled / degraded / FTS-only / hybrid modes explicit in startup and `/memory status`.
- Clarify fail-fast vs degraded behavior when memory is enabled but incomplete.

## Acceptance Criteria
- Operators can tell which memory mode is currently active.
- A half-configured memory layer is easier to detect.
- Config docs and runtime behavior match.

## Out of Scope
- Fully automatic memory setup
- Changing startup-only policy itself

## AI Implementation Instruction
Implement this as a **startup and activation clarity task**.

Strict rules:
- Do not hide degraded mode behind vague warnings.
- Do not silently change startup-only behavior.
- Preserve graceful degradation, but make it explicit.
