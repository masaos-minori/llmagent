# [High] Remove silent fallback from workflow execution in production

## Summary
Make workflow execution the single production execution path.
Do not silently fall back to direct execution when workflow prerequisites are missing.

## Problem
The current runtime may switch between:
- workflow execution
- direct legacy execution

depending on `workflow_mode`, config availability, and workflow DB availability.

This changes runtime behavior and safety boundaries in ways that are hard to reason about.

## Goal
Use workflow execution as the primary production path.
Fail fast when workflow prerequisites are missing.

## Scope
Target files may include:
- `agent/orchestrator.py`
- `agent/config.py`
- workflow config loading
- startup validation paths

## Required Changes
1. Treat `workflow_mode=required` as the default production behavior.
2. Allow `workflow_mode=auto` only for local/dev compatibility.
3. Replace silent fallback with explicit, user-visible failure in production.
4. Add startup or pre-turn validation for workflow prerequisites.
5. Update logs and operator-facing messages to clearly describe why execution is blocked.

## Acceptance Criteria
- Production mode never silently falls back from workflow execution to direct execution.
- Missing workflow config or unavailable workflow DB produces a hard failure.
- Local/dev compatibility mode remains explicit and documented.
- Tests cover both required and auto modes.

## Out of Scope
- Approval model redesign
- Workflow phase expansion
