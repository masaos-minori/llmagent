# File Read Server — Tool Definition and Routing Consistency

## 1. Objective

Resolve the tool definition mismatch and routing drift for the `file_read` MCP server. Two independent issues combine to create the current state: missing `[[tool_definitions]]` blocks (LLM-facing descriptions) and missing entries in the `tool_names` list (routing drift detection).

---

## 2. Current-State Assessment

### 2.1 Symptom

Agent startup logs show three types of warnings:

```
WARNING: Tool definitions in config do not match any reachable MCP server
WARNING: Tool definitions on running MCP servers not found in config
WARNING [check_routing_drift] Routing drift [file_read]: [file_read] tool 'get_file_info' in registry but not in config
WARNING [check_routing_drift] Routing drift [file_read]: [file_read] tool 'list_directory_with_sizes' in registry but not in config
WARNING [check_routing_drift] Routing drift [file_read]: [file_read] tool 'read_media_file' in registry but not in config
WARNING [check_routing_drift] Routing drift [file_read]: [file_read] tool 'read_multiple_files' in registry but not in config
```

These warnings indicate that the LLM may be missing available tools or attempting to call non-existent ones.

### 2.2 Root Cause

Two independent issues combine to create the current state:

#### Issue A: Missing `[[tool_definitions]]` blocks (LLM-facing descriptions)

The File Read server exposes 9 tools total, but `config/agent.toml` only defines 5 `[[tool_definitions]]` blocks:

| Config-defined tools | On which server? | Status |
|---------------------|------------------|--------|
| `read_text_file`, `list_directory`, `directory_tree`, `search_files`, `grep_files` | file_read (port 8005) | Defined |
| **`list_directory_with_sizes`**, **`read_media_file`**, **`read_multiple_files`**, **`get_file_info`** | file_read (port 8005) | **Missing** |

Current config only defines: `read_text_file`, `list_directory`, `directory_tree`, `search_files`, `grep_files` (5 tools). Actual server exposes 9 tools total.

**Source file:** `scripts/mcp_servers/file/read_tools.py`

| Tool | Description | Source line |
|------|-------------|-------------|
| `list_directory_with_sizes` | Return directory entries with sizes including stat size for directories | Line 54 |
| `read_media_file` | Return a media file base64-encoded with its MIME type | Line 76 |
| `read_multiple_files` | Retrieve multiple files in a single request | Line 83 |
| `get_file_info` | Return metadata (size, timestamps, permissions) for a file or directory | Line 104 |

#### Issue B: Missing `tool_names` entries (routing drift detection)

The `tool_names` list in `[mcp_servers.file_read]` only lists 5 of the 9 actual tools:

| Setting | Content |
|---------|---------|
| `agent.toml:338` `tool_names` | `["read_text_file", "list_directory", "directory_tree", "search_files", "grep_files"]` |
| Actual server tools (read_tools.py) | 上記5つ + `list_directory_with_sizes`, `read_media_file`, `read_multiple_files`, `get_file_info` |

The routing drift check compares the registry (populated from `tool_constants.py` frozensets) against the `tool_names` list in the server config. Since the 4 tools are missing from `tool_names`, they appear as drift.

#### Why `tool_definitions_strict = false` and `routing_drift_strict = false` Matter

Both settings default to `false`:

```toml
# config/agent.toml:59
tool_definitions_strict = false

# config_dataclasses.py:195
routing_drift_strict = false
```

This means mismatches produce **warnings**, not blocking errors. The agent continues operation despite the mismatch. However, this creates silent degradation where the LLM may not know about available tools.

## 3. Improvement Principles

### 3.1 Core Principles

- **Achieve full parity.** Every tool exposed by a running MCP server should have corresponding definitions in `config/agent.toml`.
- **Address both layers simultaneously.** Fixing only `tool_names` or only `[[tool_definitions]]` leaves one problem unresolved.
- **Preserve existing behavior where possible.** Do not change the server's internal logic or API contracts.
- **Normalize the operational model.** All MCP servers should use consistent lifecycle management.

After applying the fix, consider enabling strict mode:

```toml

```

This would cause the agent to fail fast if there's a mismatch between config and server tool lists, preventing silent degradation. However, this requires ensuring all MCP servers are started before validation runs.

### 6.2 Enable `routing_drift_strict = true`

Similarly, enable strict mode for routing drift:


This would cause the agent to fail fast if there's a mismatch between the registry and config.

Consider building a CLI tool that reads tool definitions from running MCP servers and generates the `[[tool_definitions]]` blocks automatically. This would prevent future drift by design.

## 7. Risks and Mitigations

### Risk 1: Adding tool definitions without corresponding server being healthy

**Mitigation**: The LLM can only call tools on reachable servers. If a server is not running, its tools won't be available regardless of whether they're defined in config. The `tool_definitions_strict = false` setting already handles this gracefully.

### Risk 2: Tool parameter schema mismatch between config and server

**Mitigation**: The tool definitions in `config/agent.toml` use LLM-facing descriptions (Japanese), while the actual tool schemas come from the MCP server at runtime via `/v1/tools`. There is no direct schema enforcement between them. Ensure the parameter names match when writing definitions.

### Risk 3: Configuration drift between deploy and runtime

**Mitigation**: The deployment script (`deploy/deploy.sh`) copies `config/agent.toml` to `/opt/llm/config/`. Changes must be applied through the standard deployment workflow to ensure consistency.

### Risk 4: Missing additional tools discovered after initial fix

**Mitigation**: After applying the fix, monitor the agent startup logs for any remaining warnings. Run periodic audits using the same validation logic that produces the warnings.

### 8.2 Standardize `tool_names` Maintenance Process

When adding new tools to an MCP server, ensure the following checklist:

1. Update `tool_constants.py` frozenset (registry population)
2. Add `[[tool_definitions]]` block in `agent.toml` (LLM-facing description)
3. Add tool name to server's `tool_names` list (drift detection)
4. Add to `tool_safety_tiers` section (safety classification)

### 8.3 Add Pre-flight Validation

Before starting any MCP server, validate:
- All tools in `tool_names` exist in the registry
- All tools in the registry for this server exist in `tool_names`
- Required environment variables are set
- Required files exist (e.g., `sqlite-vec/vec0.so`)
- Ports are available
- Dependencies are installed

This would prevent the kind of drift detected by `check_routing_drift()`.
