# Resolve the contract gap around messages.tool_call_id

## Summary
Decide whether `messages.tool_call_id` is a live part of the public storage contract or an unused schema remnant, and align schema and store API accordingly.

## Background
The `messages` table defines a `tool_call_id` column, but the current `SQLiteSessionStore` write/read methods do not expose or persist that field.

This creates a contract mismatch between schema and store implementation.

## Objective
Make the `tool_call_id` contract explicit and consistent.

## In Scope
- `schema_sql.py`
- `store_impl.py`
- `models.py` if DTO expansion is needed

## Required Changes
- Decide one of:
  - keep `tool_call_id` and add it to the store API and DTOs, or
  - deprecate/remove it as unused schema residue
- Align `message_save()` and `message_list()` with the final decision
- Update related row models if required

## Acceptance Criteria
- Schema and session-store API expose the same message contract
- `tool_call_id` is either fully supported or explicitly abandoned
- There is no silent schema/API mismatch for message persistence

## Out of Scope
- Redesigning all message DTOs across upper layers
- Changing OpenAI tool-call semantics

## Implementation Notes
Do not leave partially supported columns in the schema without a documented reason. The persistence contract must be explicit.
