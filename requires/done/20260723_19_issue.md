# Add Governance Policy for Removing Implementation-derived Details

## Priority

High

## Summary

Extend `00_governance_06_ai-reading-metadata.md` with a Japanese policy that defines how to handle implementation-derived details in design documents.

This policy must explain when implementation-derived information should be removed, compressed, replaced with source references, or kept as design-relevant documentation.

## Reason for Change

The design documentation currently contains many details that can be confirmed directly from source code or configuration files.

Examples include:

- Complete source file listings
- Public method tables
- Function signatures
- Constructor parameter tables
- Public attribute lists
- TypedDict, dataclass, DTO, and JSON field listings
- CLI argument tables
- Configuration key tables
- Import examples
- Module-level constant listings

These details are expensive to maintain and can easily become outdated when implementation changes.

The documentation should not become a stale copy of the source tree. It should instead focus on design intent, responsibility boundaries, architectural constraints, invariants, failure behavior, operational constraints, known issues, and unresolved confirmation items.

## Implementation Intent

The intent is to create a governance rule before removing or compressing content.

This rule will guide future AI agents and maintainers so that documentation cleanup is consistent, safe, and reviewable.

The policy must not encourage blind deletion. It must clearly distinguish between implementation-derived details and design-relevant information.

## Required Changes

Edit only:

- `00_governance_06_ai-reading-metadata.md`

Add a new Japanese section:

- `## 実装参照で確認できる情報の記載方針`

The new section must define the following categories.

### Information Usually Removed or Compressed

Document that the following should normally be removed, compressed, or replaced with source references:

- Complete public method lists
- Function signatures listed only as API reference material
- Constructor parameter tables
- Public attribute tables
- Dataclass, TypedDict, DTO, and Pydantic model field listings
- Complete directory file lists
- Complete CLI argument listings
- Complete configuration key listings
- JSON examples that simply mirror DTO or schema fields
- Module import lists
- Module-level constant listings
- Tables that only map file names to one-line implementation descriptions

### Information That Must Be Kept

Document that the following must remain in design documents:

- Design intent
- Responsibility boundaries
- Architectural constraints
- Non-negotiable invariants
- Failure behavior
- Fail-fast and fail-open behavior
- Security constraints
- Operational constraints
- Data consistency rules
- Code/document mismatches
- Known Issues
- Needs Confirmation items
- Deprecated behavior that is still relevant for migration or compatibility
- Behavior verified by tests
- Operationally observed behavior

### Decision Categories

Add a short decision model for documentation cleanup:

- Remove
- Compress
- Replace with source reference
- Keep
- Move to Known Issues
- Move to Needs Confirmation
