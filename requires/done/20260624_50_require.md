# Make cross-table memory deletion explicitly transactional

## Summary
Express the intended atomicity of `SQLiteMemoryDeleteStore.delete_memories_before()` through an explicit transaction boundary.

## Background
The implementation claims all deletions are atomic, but the code currently performs multiple cross-table delete steps without an explicit `begin_immediate()` or equivalent transaction block.

Even if current sqlite behavior may provide practical transaction grouping, the implementation does not make that intent explicit.

## Objective
Make cross-table memory deletion explicitly atomic in code, not just in comments.

## In Scope
- `store_impl.py`
- `helper.py` if transaction helpers need to be reused or extended

## Required Changes
- Wrap the multi-step deletion logic in an explicit transaction boundary
- Ensure rollback behavior is clear on intermediate failure
- Preserve the current deletion semantics across:
  - `memories`
  - `memories_fts`
  - `memories_vec`

## Acceptance Criteria
- The code explicitly expresses atomic cross-table deletion
- Rollback behavior is deterministic on failure
- The implementation matches its own “atomic” claim

## Out of Scope
- Changing memory retention policy
- Replacing SQLiteMemoryDeleteStore

## Implementation Notes
Use the existing helper transaction API if it fits cleanly. The important part is that atomicity becomes visible in code review.
