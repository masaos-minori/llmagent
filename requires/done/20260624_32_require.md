# Fix workflow database ownership and `/db` scope documentation

## Summary
Unify the documentation for `workflow.sqlite` ownership, maintenance scope, and `/db` command coverage.

## Background
The current Agent docs describe `workflow.sqlite` as a separate platform database, but another section states that workflow tables live in `session.sqlite` and are not separately exposed via `/db`. This is a direct documentation conflict.

## Objective
Document one authoritative model for:
- whether `workflow.sqlite` is a separate database
- who owns it
- whether `/db` can target it
- how it is verified operationally
- how workflow persistence differs from session persistence

## In Scope
- `05_agent_04_state-and-persistence.md`
- `05_agent_09_data-layer.md`
- `05_agent_10_operations-and-observability.md`

## Required Changes
- Compare all workflow DB descriptions
- Define the final ownership and access model
- Update all schema and maintenance sections accordingly
- Clarify whether `/db` supports:
  - session only
  - rag only
  - workflow too
  - no workflow access
- Ensure operational verification examples match the final model

## Acceptance Criteria
- `session.sqlite`, `rag.sqlite`, and `workflow.sqlite` ownership is documented without contradiction
- `/db` scope is explicit
- Verification and troubleshooting commands reflect the same final DB model
- There is no self-contradictory statement about workflow tables living somewhere else

## Out of Scope
- Implementing new `/db workflow` commands
- Database migration work
