# Requirement: Aggregate Resource Close Errors During Shutdown

## Priority

Low

## Target files

- `scripts/agent/repl.py`

## Background

In `AgentREPL._close_resources()`, each resource close operation has its own try/except block:
- WAL checkpoint: caught separately with `sqlite3.Error`
- Memory persistence (`_persist_session_memories`): caught internally with `(RuntimeError, sqlite3.Error, OSError)`
- Diagnostics persistence (`_persist_session_diagnostics`): caught internally
- Lifecycle shutdown (`svc.lifecycle.shutdown_all()`): no explicit catch
- HTTP client close (`svc.http.aclose()`): no explicit catch

Each individual failure logs its own error message. However, there is no aggregation of all failures — if multiple resources fail during shutdown, the operator sees multiple separate log entries without a consolidated view.

Note: The original issue described a single try/except wrapping all four operations, which no longer exists in the current codebase. Each resource now has its own try/except. The underlying concern — lack of visibility into multiple simultaneous failures — remains valid.

## Problem

When multiple resources fail during shutdown, operators see separate log entries without context about how many total failures occurred. This makes it harder to assess the severity of shutdown issues and identify patterns (e.g., DB connection lost causing both memory and diagnostics persistence to fail).

## Reason for change

Shutdown failures should produce a consolidated summary so operators can quickly understand the scope of the problem. A single log entry listing all failures provides better situational awareness than scattered individual errors.

## Implementation intent

Add an error collection mechanism to `_close_resources()` that aggregates all failures and emits a summary log entry at the end.

## Implementation instructions

1. In `AgentREPL._close_resources()`, add an `errors: list[tuple[str, str]] = []` list at the start.
2. After each resource close operation's try/except block, append any captured exception to the `errors` list instead of logging immediately.
3. At the end of the method, if `errors` is non-empty, emit a summary log entry:
   ```python
   if errors:
       summary = "; ".join(f"{name}: {err}" for name, err in errors)
       logger.error("Resource close errors (%d): %s", len(errors), summary)
   ```
4. Keep individual error logs for traceability — the summary supplements, not replaces, them.

## Acceptance criteria

- [ ] When multiple resources fail during shutdown, a summary log entry lists all failures
- [ ] Individual error logs remain unchanged for traceability
- [ ] No regression in normal operation when all resources close successfully
- [ ] No regression when a single resource fails

## Tests

- Unit test: Mock two resources to raise exceptions during close, verify the summary log contains both errors
- Integration test: Lock the SQLite database during shutdown, verify both WAL checkpoint and diagnostics persistence failures appear in the summary
