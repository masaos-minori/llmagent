# Add retrieval quality regression tests for major RAG execution modes

## Summary
Create regression tests that detect retrieval-quality drift across important RAG execution modes.

## Background
The RAG system has multiple quality-affecting branches such as MQE, RRF fusion, reranking, refiner behavior, semantic cache, and remote HTTP fallback.

## Objective
Introduce a repeatable quality regression harness for the retrieval pipeline.

## In Scope
- Test suite under `tests/`
- Fixtures for representative indexed content
- Retrieval evaluation for key execution modes

## Required Changes
- Add a small gold query set or equivalent deterministic assertion corpus
- Cover at least:
  - `use_rrf=true` vs `use_rrf=false`
  - `use_rerank=true` vs `use_rerank=false`
  - remote result vs in-process fallback behavior
  - semantic cache hit path
  - refiner enabled vs disabled where practical
- Define deterministic expected outcomes or relative assertions

## Acceptance Criteria
- Tests fail when a major retrieval branch regresses materially
- Test inputs are deterministic
- The quality harness documents what it does and does not guarantee
- The suite can run in CI without external ambiguity

## Out of Scope
- Human-judged large-scale benchmark creation
- Production telemetry analysis

## Implementation Notes
If exact numeric quality thresholds are not yet available, start with deterministic expected hit inclusion and ranking sanity assertions.
