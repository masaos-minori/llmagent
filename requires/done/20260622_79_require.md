# Title
Expose memory embedding fallback and circuit-breaker state in normal runtime views

## Summary
Make memory embedding state and fallback mode visible so operators can tell when retrieval has degraded to FTS-only behavior.

## Background
The memory embedding client supports retry and circuit-breaker behavior, and hybrid retrieval can fall back to FTS-only mode when embedding is unavailable.

## Problem
Operators may not realize memory retrieval quality has degraded because fallback behavior is not visible enough in normal runtime views.

## Required Changes
- Expose fields such as:
  - embedding enabled/disabled
  - circuit open/closed
  - retrieval mode (hybrid vs FTS-only)
- Surface this in a normal operator-facing path such as `/stats` or `/memory status`.
- Emit a visible signal when retrieval falls back to FTS-only mode.

## Acceptance Criteria
- Operators can see when memory retrieval is degraded.
- Fallback from hybrid retrieval to FTS-only is observable.
- Runtime visibility does not require deep implementation knowledge.

## Out of Scope
- Changing embedding API behavior
- Redesigning retrieval scoring

## AI Implementation Instruction
Implement this as a visibility and diagnosability task.

Strict rules:
- Do not change fallback semantics unless necessary.
- Surface degraded state in normal runtime paths, not only deep debug output.
- Keep status simple and operator-friendly.
- If exact circuit-breaker semantics are unclear, mark them as: Needs confirmation

Deliverables:
1. Runtime memory-status visibility
2. Fallback-mode visibility
3. Operator-friendly degraded-state indicators
