# Document and tighten plugin-registry global state lifecycle rules

## Summary
Make the lifecycle of global plugin-registry state explicit and easier to reason about.

## Background
The plugin registry now exposes structured load results and a testing reset hook, but the registry itself still lives in module-global mutable state.

This is acceptable, but its runtime lifecycle and reset assumptions are not yet obvious from the API surface.

## Objective
Clarify how global registry state is expected to behave across:
- startup
- repeated loads
- tests
- future reload scenarios

## In Scope
- `plugin_registry.py`

## Required Changes
- Add explicit lifecycle notes for:
  - initial load
  - repeated load expectations
  - test isolation
  - `_reset_for_testing()`
- Clarify whether load is intended to be one-shot per process
- Clarify how `_last_load_result` should be interpreted after multiple loads

## Acceptance Criteria
- The runtime and test lifecycle of plugin registry state is easier to understand
- Global state assumptions are documented, not implicit
- Future reload work has a clearer baseline contract

## Out of Scope
- Removing global registries
- Implementing hot-reload for plugins

## Implementation Notes
This task is mainly about lifecycle clarity, not architectural replacement.
