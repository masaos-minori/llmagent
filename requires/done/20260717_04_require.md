# Use RuntimeToolRegistry for routing and MCP execution

## Priority

High

## Target files

- shared/route_resolver.py
- shared/tool_executor.py
- shared/tool_transport_invoker.py
- shared/http_transport.py
- shared/runtime_tool_registry.py
- shared/transport_dto.py
- tests/shared/test_route_resolver.py
- tests/shared/test_tool_executor.py

## Reason

Current routing uses static ToolRegistry as the routing authority. This conflicts with the design where MCP servers own tool definitions and Agent builds a runtime index from `/v1/tools`.

ToolExecutor should route only to MCP transports using RuntimeToolRegistry.

## Intent

Make RuntimeToolRegistry the execution routing source.

## Scope

Change routing from:

- static ToolRegistry
- plugin-first execution

to:

- RuntimeToolRegistry
- MCP transport execution only

## Expected flow

1. ToolExecutor receives `tool_name` and `args`.
2. RuntimeToolRegistry resolves `tool_name` to `server_key`.
3. ToolTransportInvoker invokes MCP transport with `server_key`, `tool_name`, and `args`.
4. HttpTransport posts to `/v1/call_tool`.

## Error handling

- Unknown tool name should return deterministic tool error.
- Missing transport for a resolved server key should return deterministic tool error.
- Transport errors should preserve existing transport error behavior.

## Acceptance criteria

- Static ToolRegistry is no longer the execution routing authority.
- ToolExecutor routes using RuntimeToolRegistry.
- ToolExecutor executes MCP tools only.
- Existing HttpTransport behavior remains compatible.
