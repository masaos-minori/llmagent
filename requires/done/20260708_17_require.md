# M-7: Decide migration policy for existing tool_results tables

## Priority

Medium

## Background

Existing `session.sqlite` databases may already contain the `tool_results` table. New schemas should not create it, but deleting existing user data is potentially destructive.

A migration policy is required.

## Target files

- migration docs
- optional db migration scripts
- operations documentation

## Implementation tasks

- Decide whether existing `tool_results` tables should be:
  - left in place but unused
  - dropped by an explicit opt-in migration
  - archived before dropping
- Recommended initial policy:
  - leave existing tables in place
  - remove all runtime references
  - document that the table is deprecated and unused
- If a DROP migration is later added, make it explicit and opt-in.
- Ensure the app does not fail if the table exists.
- Ensure the app does not fail if the table does not exist.

## Acceptance criteria

- Runtime code works with databases that have `tool_results`.
- Runtime code works with databases that do not have `tool_results`.
- No automatic destructive migration runs.
- Migration policy is documented.

## Test plan

- Run the app against an old database with `tool_results`.
- Run the app against a fresh database without `tool_results`.
- Verify both start and execute turns successfully.
- Verify no runtime SQL references `tool_results`.
