## [High][eventbus][dlq] Promote events to the DLQ inline as soon as the retry limit is reached

### Summary
The current DLQ design waits for a periodic background loop before promoting eligible events.

### Problem
Events that have already reached the retry limit can remain visible as live events until the next DLQ loop tick.
This creates incorrect live visibility and delays isolation of terminal failures.

### Scope
- app.py
- dlq.py
- retry-limit handling
- DLQ promotion flow
- delivery failure path

### Task
- move threshold-based DLQ promotion into the failure-handling path
- promote to DLQ immediately when the retry limit is reached
- keep the background DLQ loop only for recovery/sweep purposes
- ensure inline promotion writes both durable state and DLQ artifact consistently
- add tests for immediate promotion on threshold reach

### Expected Result
Events that hit the retry limit are isolated immediately instead of remaining live until the next periodic sweep.

### Acceptance Criteria
- retry-limit-reached events are promoted inline
- no 60-second delay remains for threshold-based promotion
- live event queries no longer show terminal-failure events after threshold reach
- tests verify immediate DLQ promotion
