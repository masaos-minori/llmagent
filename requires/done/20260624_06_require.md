# Implement explicit RAG index repair commands for FTS and vector consistency

## Summary
Add formal repair operations for RAG index consistency instead of relying only on read-only consistency checks and manual recovery.

## Background
The current system documents consistency checks for `chunks`, `chunks_fts`, and `chunks_vec`, but the recovery path is not fully standardized.

## Objective
Provide explicit, documented repair commands that can rebuild or reconcile derived RAG indexes safely.

## In Scope
- `scripts/db/maintenance.py`
- `/db` command implementation under `scripts/agent/commands/`
- RAG maintenance service code if present
- `03_rag_05_configuration_and_operations.md`

## Required Changes
- Add or formalize the following repair operations:
  - `fts-rebuild`
  - `vec-rebuild`
  - `reconcile-url <url>`
- Define safe behavior for each command
- Ensure repair logic respects the canonical ownership model:
  - `chunks` as canonical data
  - `chunks_fts` and `chunks_vec` as derived data
- Update operational documentation with invocation steps and safety notes

## Acceptance Criteria
- An operator can repair FTS index drift without manual SQL
- An operator can repair vector index drift without ad hoc force re-ingestion
- URL-scoped repair is supported for targeted recovery
- Documentation clearly explains when to use each repair command

## Out of Scope
- Distributed or multi-node database support
- Replacing SQLite

## Implementation Notes
Handle repair in a deterministic and auditable way. Prefer rebuild-from-canonical-data over mutation of derived tables when possible.
