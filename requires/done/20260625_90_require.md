# Title
Enforce a local-only guarantee for Persistent Semantic Memory

## Priority
High

## Labels
- enhancement
- memory
- security
- local-only
- configuration
- observability

## Dependencies
- None

## Summary
Add a technical local-only guarantee so the memory layer does not send conversation history, reasoning, failures, or project-specific knowledge to external cloud endpoints. The current design uses an embedding client with `embed_url`, but local-only behavior is not enforced by configuration or startup validation.

## Background
The current memory subsystem is optional, controlled by `use_memory_layer`, and embedding is enabled through `memory_embed_enabled` plus `embed_url`. `EmbeddingClient` is HTTP-based and does not currently distinguish local endpoints from remote ones. Also, `use_memory_layer` is startup-only, not dynamically enabled by `/reload`.

## Problem
The requirement is “fully local persistent memory,” but the current implementation relies on operator discipline rather than enforceable technical guarantees. A misconfigured embedding endpoint could send memory-related data outside the local environment.

## Required Changes
- Add a local-only validation rule for memory embedding endpoints.
- Reject or fail-fast when `memory_embed_enabled=True` and `embed_url` is not local.
- Introduce an explicit config policy such as `memory_local_only`.
- Surface local-only guarantee status in startup output and `/memory status`.
- Document what “local-only” means in configuration and operations docs.

## Acceptance Criteria
- External embedding endpoints can be blocked by configuration policy.
- `/memory status` or startup output clearly shows local-only guarantee status.
- Configuration and ops docs explain the guarantee and the failure mode.

## Out of Scope
- Replacing the embedding model
- Replacing the memory backend
- Full memory architecture redesign
