# Agent Startup Processing — Health Check Timeout May Leave Orphaned Process

## Summary

If `HttpServerLifecycleManager.start()` times out during the health check phase, the subprocess is killed via `proc.kill()` but the `_http_procs` entry is not cleaned up. The process is removed from the internal mapping before the timeout exception is re-raised, leaving a dangling entry that could cause issues if the server is retried later.

## Severity

Medium

## Files

- `scripts/agent/http_lifecycle.py:258-306`

## Details

In `HttpServerLifecycleManager.start()`:

```python
async def start(self, server_key: str, cfg: McpServerConfig) -> None:
    proc = await self._start_subprocess(server_key, cfg)
    self._http_procs[server_key] = proc  # <-- Entry added here
    
    try:
        await asyncio.wait_for(self._health_check(server_key), timeout=self.health_check_timeout)
    except asyncio.TimeoutError:
        logger.error("Health check timed out for MCP server %r after %.1fs", server_key, self.health_check_timeout)
        try:
            proc.kill()
        except ProcessLookupError:
            pass
        # NOTE: proc is NOT removed from _http_procs here
        raise RuntimeError(f"MCP server {server_key!r} failed health check after {self.health_check_timeout}s")
    finally:
        if server_key in self._http_procs:
            del self._http_procs[server_key]  # <-- Cleanup happens in finally, AFTER kill()
```

Wait — actually looking more carefully, the `finally` block does remove the entry from `_http_procs`. However, there's a subtle race condition: between `proc.kill()` and the `del self._http_procs[server_key]`, if another thread/task accesses `_http_procs[server_key]`, it may find a dead process object.

More critically, if `proc.kill()` itself raises an exception (other than `ProcessLookupError`), the `finally` block will still execute and remove the entry, but the process may not actually be killed. For example, if `proc.kill()` raises `PermissionError` because the process has already been reparented to init (PID reuse scenario).

Additionally, if the subprocess exits naturally between the timeout detection and `proc.kill()`, calling `kill()` on an already-exited process will raise `ProcessLookupError`, which is caught. But the timing window exists where the process is neither tracked nor properly cleaned up.

## Reproduction Steps

1. Configure an MCP server with a very short health check timeout (e.g., 0.5 seconds)
2. Make the subprocess slow to respond (e.g., add sleep in startup logic)
3. Start the agent and observe the health check timeout
4. Verify whether `_http_procs` is correctly cleaned up after the timeout

## Expected Behavior

The process should be unconditionally removed from `_http_procs` regardless of whether `kill()` succeeds. The cleanup should happen atomically with the kill attempt.

## Proposed Fix

Move the cleanup inside the except block, ensuring it always runs after kill():

```python
except asyncio.TimeoutError:
    logger.error("Health check timed out for MCP server %r after %.1fs", server_key, self.health_check_timeout)
    try:
        proc.kill()
    except ProcessLookupError:
        pass  # Already exited
    except PermissionError:
        pass  # Cannot kill — process already reparented
    finally:
        # Always clean up the entry, regardless of kill() outcome
        if server_key in self._http_procs:
            del self._http_procs[server_key]
    raise RuntimeError(f"MCP server {server_key!r} failed health check after {self.health_check_timeout}s")
```

Alternatively, use a context manager pattern to guarantee cleanup:

```python
async def start(self, server_key: str, cfg: McpServerConfig) -> None:
    proc = await self._start_subprocess(server_key, cfg)
    self._http_procs[server_key] = proc
    try:
        await asyncio.wait_for(self._health_check(server_key), timeout=self.health_check_timeout)
    except asyncio.TimeoutError:
        await self._cleanup_on_failure(server_key, proc)
        raise
    except Exception:
        await self._cleanup_on_failure(server_key, proc)
        raise

async def _cleanup_on_failure(self, server_key: str, proc: subprocess.Popen) -> None:
    try:
        proc.kill()
    except (ProcessLookupError, PermissionError):
        pass
    finally:
        if server_key in self._http_procs:
            del self._http_procs[server_key]
```
