# Narrow the configuration source used by build_db_config()

## Summary
Reduce the configuration blast radius of `db/config.py::build_db_config()` by limiting it to an explicit DB-owned config source.

## Background
`build_db_config()` currently consumes the merged output of `ConfigLoader().load_all()`. This means DB path and SQLite policy values can be influenced by any file participating in the global config merge if duplicate keys ever appear.

For low-level DB configuration, this source boundary is too broad.

## Objective
Make DB configuration sourcing explicit, minimal, and semantically owned.

## In Scope
- `config.py`
- `config_loader.py` if helper changes are needed

## Required Changes
- Replace broad `load_all()` dependency with a narrower config-loading path for DB settings
- Clearly define where these fields come from:
  - `rag_db_path`
  - `session_db_path`
  - `workflow_db_path`
  - `sqlite_vec_so`
  - `sqlite_timeout`
  - `sqlite_busy_timeout_ms`
  - `embedding_dims`
- Preserve existing field defaults where appropriate
- Ensure the new source boundary is documented in code comments or docstrings

## Acceptance Criteria
- `DbConfig` no longer depends on unrelated merged config by default
- DB-owned config keys have an explicit source of truth
- Future duplicate-key collisions from unrelated files cannot silently override DB config
- Behavior remains backward-compatible unless a deliberate migration is documented

## Out of Scope
- Redesigning the full config hierarchy
- Changing runtime config reload behavior

## Implementation Notes
Prefer explicit semantic ownership over convenience. DB configuration should not be an accidental by-product of a global config merge.
