# Make workflow approval recovery fail-closed when multiple pending approvals exist

## Priority

High

## Type

Design / Implementation / Safety

## Background

Workflow-level approvals are persisted in `workflow.sqlite`. On startup, the agent can recover a pending approval from the database.

The current recovery model appears to restore the latest global pending approval. `/approve` and `/reject` can resolve a pending workflow approval gate.

## Problem

If multiple pending approvals exist, resolving the latest approval implicitly is unsafe. An operator may approve or reject a task different from the one they intended.

Approval is a safety boundary between the execute and verify stages. Ambiguous approval resolution must therefore fail closed.

## Goal

Prevent accidental approval or rejection of the wrong workflow task.

## Scope

Affected areas:

- `agent/startup.py`
- `agent/workflow/state_store.py`
- `/approve` command handler
- `/reject` command handler
- Workflow approval audit events
- Workflow approval tests

## Implementation Tasks

- Inspect `StateStore.find_latest_pending_approval()`.
- Determine whether multiple pending approvals can exist.
- If multiple pending approvals exist, make `/approve` and `/reject` fail with a clear error.
- Add explicit approval ID support:
  - `/approve <approval_id> [reason]`
  - `/reject <approval_id> [reason]`
- Preserve backward compatibility only when exactly one pending approval exists.
- Emit audit events when approvals are resolved.
- Include the following fields in approval audit events:
  - `approval_id`
  - `task_id`
  - `workflow_id`
  - `session_id`
  - `decision`
  - `reason`

## Acceptance Criteria

- If zero pending approvals exist, `/approve` and `/reject` clearly report that no approval is pending.
- If exactly one pending approval exists, `/approve [reason]` and `/reject [reason]` may continue to work.
- If multiple pending approvals exist, approval without ID fails closed.
- Approval with explicit `approval_id` resolves the intended approval only.
- Audit logs contain enough identifiers to reconstruct the approval decision.

## Validation Plan

Add tests for:

- No pending approval.
- One pending approval.
- Multiple pending approvals without explicit ID.
- Multiple pending approvals with explicit ID.
- Approval after agent restart.
- Rejection after agent restart.
- Audit event content.

## Risks

Adding explicit ID syntax may affect operator workflows. Mitigate by keeping the legacy syntax only for the single-pending-approval case.

## Out of Scope

- Redesigning the workflow engine.
- Implementing multi-user approval queues.
- Documentation-only updates.
