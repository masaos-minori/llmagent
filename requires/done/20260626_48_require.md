# Title
Clarify diagnostics for HTTP RAG `remote_empty` results

## Priority
Medium

## Labels
- enhancement
- rag
- http-mode
- diagnostics
- observability

## Dependencies
- None

## Summary
Clarify the diagnostics model for HTTP RAG `remote_empty` results so a successful empty remote response is not confused with fallback behavior.

## Background
In HTTP RAG mode, `remote_empty` means the remote service returned HTTP 200 but no context. This is a valid success case and does not trigger in-process fallback.

## Problem
The current model can report `status=success` while also using `fallback_reason="http_remote_empty"`, which is semantically confusing.

## Required Changes
- Decide whether `remote_empty` is represented as:
  - success with a non-fallback reason field, or
  - a separate explicit status/kind.
- Avoid using `fallback_reason` for success-empty cases.
- Update `/rag search --debug` output to show success-empty clearly.
- Update docs to state that `remote_empty` does not run in-process fallback.

## Acceptance Criteria
- Success-empty and fallback are clearly distinguishable.
- Diagnostics field names match semantics.
- Debug output is understandable to operators.

## Out of Scope
- Changing HTTP fallback policy itself.
- Changing remote API response format unless required.
