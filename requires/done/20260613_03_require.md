## 改善要望

### 1. 後方互換レイヤの全廃

- `config.py` の flat config 互換、`context.py` の flat runtime-context 互換、`repl_debug.py`、`context_detection.py`、未使用 `rag_debug.py` など、互換のためだけに残る実装を削除する。複数の呼び方や古い経路が残ること自体が、変更影響範囲の拡大、型安全性低下、保守コスト増大の原因である。
- 既存 API は `cfg.llm / cfg.rag / cfg.tool / ...` と `ctx.conv / ctx.turn / ctx.stats / ctx.services / ...` に統一する。呼び出し側全面修正を前提にし、互換維持より構造簡素化を優先する。

### 2. hidden global state の排除

- `db/helper.py`、`db/tool_results.py`、各所の module-global `_cfg` キャッシュ、`_audit_logger_instance`、process-scope warning suppression を廃止する。設定再読込、テスト差し替え、複数インスタンス並行動作で不整合が起きるためである。
- 設定取得は explicit dependency injection、または共有 ConfigLoader / config service に集約する。logger も runtime assembly 時に構築し、task-local context へ移行する。

### 3. 責務分離の徹底

- `tool_executor.py`、`llm_client.py`、`MemoryLayer`、`cmd_mcp.py`、`cmd_context.py`、`cmd_ingest.py`、`cmd_rag.py`、`mcp/server.py`、`mcp/installer.py` などの責務過多モジュールを分割する。routing、transport、cache、plugin、error model、validation、template rendering、CLI interaction、audit を分離する。
- Command 層は薄くし、アプリケーションロジックは service、表示は renderer、永続化は store/repository に寄せる。private attribute 直接書換や mixin 間暗黙依存を禁止する。

### 4. エラー処理と障害契約の統一

- `orchestrator.py` と `llm_turn_runner.py` の LLM 失敗処理を一元化し、partial completion、mid-turn transport failure、synthetic tool-error injection の扱いを共通ポリシー化する。
- `format_transport_error()`、LLM transport error、tool transport error を別々の文字列表現で持たず、`retryable / partial / source / phase` を持つ構造化エラー型へ統一する。
- DB 失敗時に `None` や空配列へ静かにフォールバックしない。少なくとも audit event、failure metrics、履歴への明示注記を追加する。特に `tool_results` は `/tool show` の信頼性に直結するため、保存失敗を隠さない。
- `git_helper.py` の `except Exception: return None` も禁止し、reason-coded result を返す。

### 5. 型・スキーマ・バリデーション強化

- `shared/types.py` の `LLMMessage.role: str`、tool call `list[dict]`、`mcp/models.py` の `args: dict[str, Any]` のような緩い型をやめ、Literal / Enum / 専用 request schema へ変更する。
- `agent/memory/types.py` で `memory_type`、`source_type`、`importance`、`created_at` の invariant を型層で保証する。正規化を store 側へ押し込まない。
- `mcp_config.py` の `transport`、`startup_mode`、`healthcheck_mode` を enum 化し、設定エラーを早期検出する。

### 6. ルーティング・安全性判定の単一ソース化

- ツールの副作用判定を `WRITE_TOOLS | DELETE_TOOLS | {"shell_run"}` の固定集合で持たず、`ToolSpec` のような単一メタモデルに統合する。capability、risk、backend を別 set 群で並行管理しない。
- `route_resolver.py` は config-first 化し、static fallback は strict mode で禁止、warning mode では audit を出す。新規 tool が fallback だけで成功しないようにする。
- 承認モデルは `require_approval_tools` のツール名ベースから、args、対象リソース、操作種別を含む危険度ベースへ変更する。dry-run / preview を強制し、承認結果を構造化 audit log に保存する。

### 7. ツール実行の依存関係解析と部分並列化

- `asyncio.gather()` 全並列と `serial_tool_calls` 全直列の二択をやめ、`side_effect`、`resource_scope`、`requires_serial` をメタに持たせて実行計画を生成する。同一 resource への write は直列、それ以外は並列とする。
- これにより、速度と安全性の両立を図る。依存判定不備による順序バグを防ぐため、ツール定義メタ管理を正式化する。
