# Title
Expose `dry_run` in MCP tool input schemas for all supported tools

## Summary
Update MCP tool definitions so that every tool which already supports `dry_run` in its request model or service layer also exposes `dry_run` in its published `inputSchema`.

## Background
Several tools already implement `dry_run`, but this capability is hidden from the LLM because it is missing from the MCP schema.

## Problem
The runtime supports a safer preview path, but the LLM cannot reliably discover or use it through tool definitions.

## Required Changes
- Add `dry_run` to the MCP `inputSchema` for all affected tools.
- Ensure tool descriptions clearly explain that `dry_run` performs a preview without side effects.
- Verify that schema, request model, and service behavior are aligned.

## Affected Tools
- `write_file`
- `move_file`
- `delete_file`
- `delete_directory`
- `trigger_workflow`

## Acceptance Criteria
- All affected tools expose `dry_run` through tool schema output.
- LLMs can discover `dry_run` directly from tool definitions.
- No tool supports `dry_run` only implicitly while hiding it from schema.

## Out of Scope
- Changing the meaning of existing `dry_run` behavior
- Redesigning approval flow beyond schema visibility

## AI Implementation Instruction
Implement this as a schema-correctness and LLM-discoverability task.

Strict rules:
- Do not change business semantics unless required for schema alignment.
- Do not leave any model/schema mismatch in place.
- Keep tool descriptions short and explicit.
- If a tool’s runtime behavior is inconsistent with the documented `dry_run` contract, mark it as: Needs confirmation

Deliverables:
1. Updated schemas for all affected tools
2. Aligned request model / service / schema behavior
3. Short documentation update for each affected tool
