# [Medium] Decouple `tool_constants.py` from routing responsibilities

## Priority
Medium

## Background
`tool_constants.py` currently documents multiple responsibilities, including static routing. After routing fallback removal, it should no longer be described or used as a routing source. `tool_executor.py` still legitimately uses tool constants for side-effect detection, which should remain. 【8-ba6137】【1-ff2e06】

## Goal
Remove routing responsibility from `tool_constants.py` while preserving classification use cases.

## Scope
Affected files:
- `shared/tool_constants.py`
- `shared/tool_registry.py`
- `shared/tool_executor.py`
- `agent/tool_policy.py`
- docs

## Implementation Tasks
- Remove `shared/route_resolver.py — static routing` from the `tool_constants.py` docstring.
- Keep `tool_constants.py` responsibilities limited to:
  - registry seed data, if still required
  - side-effect classification
  - approval/risk policy classification
- Ensure `route_resolver.py` does not import `tool_constants.py`.
- If responsibilities remain too broad, create a follow-up issue to split the module.

## Acceptance Criteria
- `tool_constants.py` is not a routing source.
- Side-effect and policy classification usages remain intact.
- Module docstring matches actual responsibilities.
- Tests pass.

## Validation
- Run: `rg -n "tool_constants.*routing|route_resolver.*tool_constants|static routing" shared docs tests`
