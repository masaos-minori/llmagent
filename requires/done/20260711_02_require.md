# MCP Server Startup Mode — RAG Pipeline and CICD Servers

## 1. Objective

Resolve the degraded state of the RAG Pipeline and CICD MCP servers caused by `startup_mode = "none"` and missing `GITHUB_TOKEN`. Both servers cannot start automatically on agent startup, and the watchdog cannot auto-recover them when they go down.

---

## 2. Current-State Assessment

### 2.1 Symptom

Two independent symptoms occur across two servers:

**RAG Pipeline server:**
- Watchdog reports `'rag_pipeline' is not a subprocess-mode server; manual intervention required`
- RAG functionality unavailable until manually restarted
- Occurred 55 times since first reported

**CICD server:**
- Watchdog reports `'cicd' requires operator action` with `status: degraded`
- All four CICD tools unavailable (workflow trigger, run listing, status check, logs retrieval)
- Manual intervention required every time the server goes down

### 2.2 Error Logs

**RAG Pipeline:**
```
Watchdog: 'rag_pipeline' is not a subprocess-mode server; manual intervention required
```

**CICD:**
```
Watchdog: 'cicd' requires operator action: {
  'status': 'degraded',
  'ready': False,
  'liveness': True,
  'restart_recommended': False,
  'operator_action_required': True,
  'dependencies': {'github_token': 'not_set'},
  'details': {'service': 'cicd-mcp'}
}
```

### 2.3 Root Cause

Two independent issues combine to create the current degraded state:

#### Issue A: `startup_mode = "none"` (Both Servers)

Both servers have `startup_mode = "none"` in `config/agent.toml`:

| Server | Port | Line | Setting |
|--------|------|------|---------|
| `cicd` | 8012 | 366 | `startup_mode = "none"` |
| `rag_pipeline` | 8010 | 378 | `startup_mode = "none"` |

With `startup_mode = "none"`:
- The agent does not attempt to start the server automatically
- The watchdog does not attempt to restart the server
- Tool calls are rejected with a "disabled" error instead of failing silently

**Affected components:**

| Component | File | Behavior with `startup_mode = "none"` |
|-----------|------|---------------------------------------|
| Lifecycle router | `scripts/agent/factory.py:85-89` | Skips `ensure_ready()` call |
| Watchdog | `scripts/agent/repl_health.py:603-617` | Logs warning instead of restarting |
| Lifecycle restart | `scripts/agent/factory.py:134-140` | Returns without restarting |

#### Issue B: Missing `GITHUB_TOKEN` (CICD Only)

All four CICD tools require GitHub API access, which needs a valid `GITHUB_TOKEN`:

| Tool | Description | Token Required |
|------|-------------|----------------|
| `trigger_workflow` | Trigger GitHub Actions workflow dispatch | Yes (POST) |
| `get_workflow_runs` | List recent workflow runs | Yes (GET) |
| `get_workflow_status` | Get workflow run status | Yes (GET) |
| `get_workflow_logs` | Retrieve workflow job logs | Yes (GET) |

When `GITHUB_TOKEN` is not set, the `/health` endpoint returns HTTP 503 with `dependencies: {"github_token": "not_set"}`.

**Configuration files involved:**

```toml
# config/agent.toml:366
startup_mode = "none"

# config/cicd_mcp_server.toml:27
github_token = ""
```

Both are empty — no token is configured anywhere.

### 2.4 Why `"none"` Was Chosen

Two different reasons were cited:

1. **cicd**: GITHUB_TOKEN not set → `/health` always returns 503 → stalls agent startup for the full `startup_timeout_sec`. Setting `startup_mode = "none"` avoids this stall.

2. **rag_pipeline**: No specific technical reason given. Only "temporarily out of dev scope" — likely a decision made during development to avoid startup complexity.

### 2.5 Comparison with Other Servers

Other MCP servers also depend on `GITHUB_TOKEN` but handle it differently:

| Server | Startup Mode | Health When Token Unset | Behavior |
|--------|-------------|------------------------|----------|
| `github` | subprocess | 503 + `github_token: not_set` | Starts, reports degraded |
| `cicd` | none | N/A | Never starts, reports disabled |
| `web_search` | subprocess | N/A | No token dependency |

The `github` server demonstrates that even with a missing token, the server can start and report its degraded state — it doesn't need `startup_mode = "none"`.

### 2.6 Affected Components

**Configuration:**

| File | Line | Setting |
|------|------|---------|
| `config/agent.toml` | 366 | `cicd.startup_mode = "none"` |
| `config/agent.toml` | 378 | `rag_pipeline.startup_mode = "none"` |
| `config/cicd_mcp_server.toml` | 27 | `github_token = ""` |

## 3. Improvement Principles

### 3.1 Core Principles

- **Normalize the operational model.** All MCP servers should use the same lifecycle management approach unless there is a compelling technical reason not to.
- **Eliminate manual intervention requirements.** Auto-recovery should work for all servers.
- **Address both root causes simultaneously.** Fixing only the token or only the startup mode leaves one problem unresolved.
- **Preserve existing behavior where possible.** Do not change the server's internal logic or API contracts.

---

## 4. Target Architecture

### 4.1 Desired State After Fix

After applying both fixes, the following flow will occur for both servers:

```
Agent REPL startup
  ↓
Lifecycle.ensure_ready("rag_pipeline")    ← Issue A fix
Lifecycle.ensure_ready("cicd")            ← Issue A fix
  ↓
HttpServerLifecycleManager.start()
  ├─ Spawn subprocess: uv run python scripts/mcp_servers/<server>/server.py
  ├─ Wait for /health to return 200
  └─ Register process snapshot
  ↓
Watchdog loop
  ├─ Probe /health every N seconds
  ├─ If unhealthy → HttpServerLifecycleManager.restart()
  └─ If max_restarts reached → log warning + operator_action_required
```

### 4.2 Comparison: Before vs After

| Aspect | Before (`startup_mode = "none"`) | After (`startup_mode = "subprocess"`) |
|--------|----------------------------------|---------------------------------------|
| Agent startup | Server not started automatically | Server started automatically |
| Watchdog recovery | Warning logged, no action | Subprocess terminated and restarted |
| Manual intervention | Always required | Only after max_restarts exceeded |
| Process visibility | None | Available via `get_process_info()` |
| Graceful shutdown | Not handled | Handled via `shutdown_all()` |

## 5. Recommended Fix

### 5.1 Implementation

Apply both fixes simultaneously for each server:

#### Fix A: Change `startup_mode` to `"subprocess"` for rag_pipeline

Modify `config/agent.toml`:

```diff
 [mcp_servers.rag_pipeline]
 transport        = "http"
 url              = "http://127.0.0.1:8010"
-# Temporarily out of dev scope: startup_mode="none" disables subprocess spawn and
-# health check entirely; tool calls are rejected with a clear "disabled" error.
-startup_mode     = "none"
+startup_mode     = "subprocess"
 healthcheck_mode = "http"
 auth_token       = ""
 role             = "query"
 tool_names       = ["rag_run_pipeline", "rag_debug_pipeline", "rag_list_documents", "rag_delete_document"]
 cmd              = ["uv", "run", "--directory", "/opt/llm", "python", "/opt/llm/scripts/mcp_servers/rag_pipeline/server.py"]
```

#### Fix B: Set `GITHUB_TOKEN` for cicd

Choose one of the following options:

**Option B1: Environment variable (recommended)**

```bash

```


## 6. Alternative Approaches

### 6.1 Keep `startup_mode = "none"` for cicd

If the CICD server is truly out of scope for the current development cycle, keep the current setting. However, document the reason clearly and consider adding a TODO item for future resolution.

### 6.2 Make `/health` return 200 even without GitHub token

Modify the CICD server's health endpoint to return 200 OK when the service itself is healthy, regardless of GitHub API availability. Then change `startup_mode` to `"subprocess"` without needing to set the token. This would allow the server to start and report its degraded state (similar to how the `github` server behaves), but the tools would still fail when called.

This approach has limited value because all four tools require the token — there is no partial functionality to expose.

### 6.3 Add conditional startup based on token presence

Implement logic in the lifecycle manager to skip starting servers whose required tokens are not configured. This would prevent the "exited early" errors seen with the GitHub MCP server while avoiding the need to set `startup_mode = "none"`.

This is a more complex architectural change that requires careful design and testing. Consider as a long-term improvement rather than an immediate fix.

## 7. Risks and Mitigations

### Risk 1: Token exposure if committed to repository

**Mitigation**: Use environment variables or secret management systems. Never commit actual token values to version control.

### Risk 2: Startup timeout if server fails to start

**Mitigation**: The `startup_timeout_sec` parameter already handles this. If the server doesn't become healthy within the timeout period, the agent will log an error and continue without the affected functionality.

### Risk 3: Port conflict if server is already running

**Mitigation**: The agent's lifecycle manager checks if the port is already in use before starting. If a conflict occurs, the agent will log an error and skip starting the server.

### Risk 4: Configuration drift between deploy and runtime

**Mitigation**: The deployment script (`deploy/deploy.sh`) copies `config/agent.toml` to `/opt/llm/config/`. Changes must be applied through the standard deployment workflow to ensure consistency.

### Risk 5: Token expiration

**Mitigation**: Monitor token expiration dates and rotate proactively. Consider adding a watchdog alert for expiring tokens.

### Risk 6: Server crashes due to missing dependencies

**Mitigation**: Ensure all dependencies are installed before starting the agent:

```bash
cd /opt/llm && uv sync --dev
```

Check that the server can start independently:

```bash
cd /opt/llm && uv run python scripts/mcp_servers/rag_pipeline/server.py &
curl http://127.0.0.1:8010/health
kill %1
```

## 9. Conclusion

The RAG Pipeline and CICD MCP servers have two independent root causes: missing `GITHUB_TOKEN` (CICD only) and `startup_mode = "none"` (both servers). Both must be addressed together to restore full functionality. Setting the token resolves the health check degradation, and changing `startup_mode` restores auto-recovery capability. The implementation is straightforward — modify two configuration settings per server — and aligns both servers' operation with other MCP servers in the system.
