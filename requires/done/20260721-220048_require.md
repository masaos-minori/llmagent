# Require: Make startup rollback failure fatal when shutdown_all fails

## Priority

High

## Target files

- `scripts/agent/startup.py`

## Background

During `StartupOrchestrator.run()`, if an exception occurs after some MCP subprocesses have started (`_servers_started=True`), a rollback via `shutdown_all()` is attempted. If this rollback itself fails, only a WARNING-level log is emitted and the original exception is re-raised. The MCP subprocesses remain running as orphans.

## Problem

When `shutdown_all()` fails during rollback, MCP subprocesses are left running as orphans. Only a WARNING log is emitted, which may not be noticed by operators. Orphaned processes cause port conflicts and resource leaks. The scenario:
1. HTTP subprocess MCP server A starts successfully
2. HTTP subprocess MCP server B fails to start (e.g., startup_timeout exceeded)
3. `shutdown_all()` is called to clean up server A
4. `shutdown_all()` raises an exception (e.g., SIGINT handler restoration fails with ValueError)
5. Server A's process remains running as an orphan

## Reason for change

Rollback failures must not be silently ignored. Operators need immediate visibility into situations where cleanup cannot complete, and the system should fail fast rather than leave orphaned processes.

## Implementation intent

Change the rollback failure handling from WARNING to FATAL level, ensuring operators are immediately notified. Also consider making the rollback failure itself a fatal condition that prevents the agent from starting.

## Implementation instructions

In `scripts/agent/startup.py`, around line 64-72, replace:

```python
except Exception:
    if _servers_started:
        try:
            await self._ctx.services_required.lifecycle.shutdown_all()
        except Exception as shutdown_err:
            logger.warning(
                "Startup rollback: shutdown_all() failed: %s", shutdown_err
            )
    raise
```

with:

```python
except Exception:
    if _servers_started:
        try:
            await self._ctx.services_required.lifecycle.shutdown_all()
        except Exception as shutdown_err:
            logger.error(
                "CRITICAL: Startup rollback FAILED — subprocesses may be orphaned: %s",
                shutdown_err,
            )
            raise RuntimeError(
                f"Startup rollback failed: {shutdown_err}. "
                "Subprocesses may be orphaned."
            ) from shutdown_err
    raise
```

Key changes:
1. Use `logger.error` instead of `logger.warning` for higher visibility
2. Re-raise as `RuntimeError` instead of letting the original exception propagate, making it clear this is a critical failure
3. Include context about potential orphaned subprocesses in the error message

## Acceptance criteria

- When `shutdown_all()` succeeds during rollback, behavior is unchanged (no additional logging)
- When `shutdown_all()` fails during rollback, ERROR-level log is emitted and a RuntimeError is raised
- The RuntimeError message includes information about potential orphaned subprocesses
- No orphaned MCP subprocesses remain after a failed startup
- Original exception propagation is preserved when no subprocesses were started

## Tests

- Unit test: Mock `shutdown_all()` to raise an exception, verify RuntimeError is raised with correct message
- Unit test: Mock `shutdown_all()` to succeed, verify original exception propagates without modification
- Integration test: Simulate SIGINT handler failure during shutdown, verify no orphaned processes remain
