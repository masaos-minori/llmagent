# Promote strict plugin MCP-conflict rejection rules into the main extension docs

## Summary
Update the main Agent extension documentation so that strict plugin conflict rejection behavior is documented as part of the primary plugin model.

## Background
A dedicated design note describes how strict mode should aggregate MCP tool-conflict rejections into `PluginLoadError`, but the main extension chapter does not yet fully reflect that guarantee.

## Objective
Document the final strict-mode behavior for plugin loading and MCP-tool name conflicts.

## In Scope
- `05_agent_11_extension-points.md`
- Cross-reference from `05_agent_90_inconsistencies_and_known_issues.md` if needed

## Required Changes
- Update the plugin conflict section to describe strict-mode behavior explicitly
- Clarify whether MCP tool conflicts contribute to startup failure in strict mode
- Clarify the difference between:
  - import failure
  - command shadow
  - MCP tool conflict reject
- Ensure the extension docs and known-issues docs do not diverge

## Acceptance Criteria
- Strict plugin loading guarantees are documented in the main extension chapter
- MCP tool conflict handling is explicit
- A CI/operator reader can infer fail-fast behavior from the main docs alone

## Out of Scope
- Plugin loader implementation changes
- New plugin conflict policy design

## Notes
Keep the extension chapter normative. Do not rely on a narrow design-note file as the only source for core strict-mode behavior.
