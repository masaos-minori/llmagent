## [Low][ops][observability] Add runtime metrics and alerts for publish latency, SSE lag, DLQ loop duration, and offload saturation

### Summary
Fixing the blocking issue in code is not enough.
Operations needs visibility into regressions and overload conditions.

### Problem
Without runtime metrics and alertable signals, performance regressions may go unnoticed until clients observe broken streaming behavior.

### Scope
- operations docs
- runtime observability
- metrics/alerts guidance

### Facts
- Event Bus handles publish, replay, SSE subscribe, offset checkpoints, and DLQ processing in one runtime. 【1-2f69b2】【4-1b37a6】【1-6bdda3】
- Current docs describe validation status and deployment, but not stall-oriented runtime metrics. 【6-b6fcb8】

### Task
- define runtime metrics for:
  - publish p95 / p99 latency
  - SSE delivery lag
  - DLQ loop duration
  - threadpool queue/saturation if available
- add operator guidance for thresholds and alerts
- document how to distinguish load saturation from DB unavailability

### Expected Result
Operations can detect and react to runtime degradation before it becomes a visible outage.

### Acceptance Criteria
- docs list the recommended metrics
- alerting guidance is documented
- metrics cover both availability and latency degradation
