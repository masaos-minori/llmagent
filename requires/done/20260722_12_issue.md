## 重点確認項目

以下を重点的に確認してください。

- ingestion pipeline の実行ファイル、設定ファイル、入出力 JSON 形式が実装と一致しているか
- `crawler.toml` / `chunk_splitter.toml` / `ingester.toml` の説明が現行実装と一致しているか
- 存在しない共通設定ファイルの説明が残っていないか
- query pipeline のステージ説明が現行実装と一致しているか
- DTO / TypedDict / dataclass のフィールド説明が現行コードと一致しているか
- DB スキーマ説明が現行 schema と一致しているか
- 削除済み項目や旧仕様説明が残っていないか
- DESIGN notes に残すべき不変条件と削除すべき履歴説明が混在していないか
- 同じ pipeline / config / DTO 説明が複数章に重複していないか

## 修正方針

- システム概要は `03_rag_01_*` に残す。
- ingestion 詳細は `03_rag_02_*` に集約する。
- query pipeline 詳細は `03_rag_03_*` に集約する。
- DTO / DB / 型は `03_rag_04_*` に集約する。
- 設定・実行手順は `03_rag_05_*` に集約する。
- 未解決の不整合は `03_rag_90_*` に集約する。
- 設計上の不変条件は `03_rag_91_*` に残す。
- 現行仕様に不要な履歴説明は削除する。

基本方針は以下です。

1. 実装とドキュメントが矛盾する場合は、実装を正本としてドキュメントを修正する。
2. 現行仕様に影響しない過去経緯・移行履歴・旧仕様説明は削除する。
3. 実装で削除済みの機能、API、CLIコマンド、設定キー、DBカラム、型、関数、ファイルの説明は削除する。
4. 同一内容の説明が複数箇所にある場合は、正典となる1箇所に集約し、他は削除または参照リンクだけにする。
5. 判断できない場合は推測せず、必要な実装ファイル・設定ファイル・テストファイルを列挙する。
6. Known Issues に残すのは、現行仕様に影響する未解決事項だけにする。
7. 修正後は File Index、AI Query Routing Table、Related Documents、内部リンクも整合させる。
