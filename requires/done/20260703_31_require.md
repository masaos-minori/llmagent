# ToolTransportInvoker statistics ownership

## Priority

Medium

## Decision

`ToolTransportInvoker` owns transport-related statistics.
`ToolExecutor` exposes read-only properties for compatibility.

## Goal

Keep statistics ownership aligned with the component that generates the statistics.

## Background

After extracting `ToolTransportInvoker`, these events belong to the transport invocation layer:

- successful transport call
- tool-level error result from transport
- transport-level error
- health success
- health failure

Therefore, `ToolTransportInvoker` should own:

- `stat_tool_errors`
- `stat_transport_errors`

## Implementation requirements

In `ToolTransportInvoker`:

    @property
    def stat_tool_errors(self) -> dict[str, int]:
        return dict(self._stat_tool_errors)

    @property
    def stat_transport_errors(self) -> dict[str, int]:
        return dict(self._stat_transport_errors)

In `ToolExecutor`:

    @property
    def stat_tool_errors(self) -> dict[str, int]:
        return self._transport_invoker.stat_tool_errors

    @property
    def stat_transport_errors(self) -> dict[str, int]:
        return self._transport_invoker.stat_transport_errors

Properties should return defensive copies.

## Acceptance criteria

- `ToolTransportInvoker` mutates tool and transport error stats.
- `ToolExecutor` does not directly mutate transport stats.
- Existing callers can still read `executor.stat_tool_errors`.
- Existing callers can still read `executor.stat_transport_errors`.
- External callers cannot mutate internal stats dictionaries through returned references.
- `/stats` behavior remains unchanged.

## Validation plan

- Add tests for stats ownership.
- Test tool-level error increments `stat_tool_errors`.
- Test transport error increments `stat_transport_errors`.
- Test returned dict is a copy.
- Test existing `ToolExecutor` property access works.

## Out of scope

- Introducing a new shared `ToolExecutionStats` object.
- Callback or recorder-based stats updates.
