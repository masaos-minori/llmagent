# 改善項目一覧

## 1. 全体共通の改善項目

### 1-1. 後方互換コードの削除
- legacy entry point、compatibility wrapper、re-export shim、旧戻り値契約、旧 alias を削除すること。
- `IF NOT EXISTS`、旧スキーマ救済、`_get_cfg() + {}` fallback、invalid JSON の `{}` 化、unknown role/tier の温情吸収を削除すること。

### 1-2. 失敗戦略の統一
- `except Exception` を禁止し、HTTP / JSON / SQLite / I/O / transport / LLM / validation などの具体例外へ分解すること。
- `assert` を業務ロジックから除去し、前提条件違反は明示例外で扱うこと。
- `None`、空配列、空文字、`False`、`"?"`、warning ログだけでの継続をやめ、正常系と異常系を型と例外で分離すること。

### 1-3. 型と契約の厳格化
- `dict[str, Any]`、`tuple[bool, str]`、`object`、生文字列ステータスを内部主要契約として使わず、DTO / Enum / 専用例外へ置換すること。
- LLM / HTTP / JSON / SQLite row の decode 直後に schema 検証し、不一致は即時失敗とすること。
- `str(args.get(...))`、`str(msg.get(...))`、`.get(..., default)` による曖昧補正を廃止し、期待型以外は例外にすること。

### 1-4. 出力責務の分離
- `print()` と `input()` を command / service / domain から除去し、UI/CLI 用 OutputPort / Renderer / Presenter に集約すること。
- 監査ログ、承認結果、実行結果、表示用ビューは専用 DTO を定義し、表示文言は presentation 層で組み立てること。

## 2. Command 層の改善項目

### 2-1. command handler の責務縮小
- command handler は「入力受付」「引数解析」「service 呼び出し」「renderer 委譲」のみを担当させること。直接状態変更、業務ロジック、表示整形、監査イベント組み立てを持たせないこと。
- `cmd_context.py`、`cmd_db.py`、`cmd_memory.py`、`cmd_notes.py`、`cmd_session.py`、`cmd_tooling.py`、`cmd_config.py`、`cmd_debug.py`、`cmd_ingest.py`、`cmd_mcp.py` の各 command を result DTO ベースへ統一すること。

### 2-2. 引数解析と command spec の共通化
- `args.split()`、`isdigit()`、`dispatch.get(sub)` のような ad-hoc 解析を廃止し、subcommand spec と parser schema に統一すること。未知 subcommand、未知 flag、不足引数、型不正は fail-fast で扱うこと。
- `registry.py` の `_COMMANDS` を唯一の command 仕様定義へ昇格し、help / usage / handler / validator / renderer 連携を一元管理すること。

### 2-3. formatter の再設計
- `formatter.py` は print helper ではなく OutputPort / TableRenderer / TextRenderer / JsonRenderer に再編すること。
- config、context、db、tool、notes、mcp、ingest の出力整形を個別 renderer へ分離すること。

### 2-4. 個別 command の改善
- `cmd_config.py` は stats 収集を ViewModel DTO 化し、reload は request DTO + validator + typed outcome へ置き換えること。`except Exception` を廃止すること。
- `cmd_context.py` は `/clear`、`/undo`、`/system` を service 化し、history / system prompt の直接操作を排除すること。context state は typed DTO で返すこと。
- `cmd_db.py` は辞書 dispatch と緩い flag 解釈を廃止し、DbCommandRequest DTO と破壊的操作用の request/result DTO を導入すること。
- `cmd_memory.py` は subcommand spec 化し、pin / unpin / delete / prune を service に移すこと。audit event を command で直組みしないこと。
- `cmd_notes.py` は if/elif dispatch と `arg.isdigit()` ベース validation を廃止し、request/result DTO 化すること。
- `cmd_session.py` は `tuple[bool, str]` 戻り値依存をやめ、SessionCommandResult DTO を経由すること。load / delete / rename を service 経由へ統一すること。
- `cmd_tooling.py` は tool result を dict 参照せず ToolResultView DTO で扱い、invalid JSON を `{}` に縮退しないこと。`/tool` と `/plan` の責務分離を行うこと。
- `cmd_ingest.py` は ingest / rag / compact / export を request/result DTO 化し、進捗・失敗表示を renderer 経由へ移すこと。
- `cmd_mcp.py` は install/status の入力を厳密検証し、next steps 出力を render plan 化すること。
- `cmd_debug.py` は直接 print と OSError 文字列化を廃止し、diagnostic result DTO + renderer へ変更すること。

## 3. Service 層の改善項目

### 3-1. service 戻り値の DTO 化
- `context_view.py`、`db_maintenance_service.py`、`session_restore.py`、`undo_service.py`、`conversation_service.py`、`session_title.py`、`mcp_status.py`、`mcp_install.py`、`ingest_workflow.py`、`config_reload.py`、`export_formatter.py` の戻り値を DTO に統一すること。
- tuple / dict / object / 生文字列に依存した command 側の制御を廃止すること。

### 3-2. fallback と曖昧継続の削除
- `session_title.py` のタイトル fallback、`mcp_status.py` の未知 tier 許容、`context_view.py` の token 推定 fallback、`mcp_install.py` の unknown role -> generic を削除すること。
- `config_reload.py` の raw dict 直適用と `.get()` ベース補正をやめ、strict request DTO + validator + typed outcome にすること。

### 3-3. I/O 依存の分離
- `mcp_install.py` の CLI 入出力、`export_formatter.py` の書き込み結果表示を Port 経由へ分離すること。
- service が presentation text を直接持たず、render plan や view model を返す構造へ変更すること。

## 4. DB 層の改善項目

### 4-1. 設定と接続の DI 化
- DB 設定は `DbConfig` に集約し、module-level cache や ConfigLoader 直読みを廃止すること。
- `helper.py` は接続制御専用とし、`transaction()` context manager を導入すること。sqlite-vec は必要時のみロードすること。

### 4-2. スキーマと migration の整理
- `create_schema.py` から migration 救済、`IF NOT EXISTS`、旧スキーマ共存コードを削除し、最新スキーマのみを生成するモジュールへ変更すること。
- `schema_version` を使わない場合は削除し、使うなら正式運用へ統一すること。

### 4-3. 書き込みの厳格化
- `store.py` は `INSERT OR REPLACE` をやめ、`ON CONFLICT DO UPDATE` に統一すること。FTS / vec 同期は warning 継続ではなく全体失敗にすること。
- `maintenance.py` の checkpoint / vacuum / purge / recover は fail-fast の typed DTO へ変更すること。unsafe copy backup を禁止すること。
- `tool_results.py` の例外握りつぶしを禁止し、summary / full text / retention の扱いを明確化すること。

## 5. Memory 層の改善項目

### 5-1. 型再設計
- `types.py` を復元・再設計し、memory record、query、hit、neighbor、embedding response、JSONL record、consistency report、injection snippet、extraction candidate を DTO / Enum / Exception で表現すること。
- `models.py`、`enums.py`、`exceptions.py`、`ports.py` を追加し、memory 層の public API を strict DTO ベースへ統一すること。

### 5-2. 永続化の一貫化
- malformed JSONL 行の読み飛ばし、quarantine 継続、SQLite/vec 失敗時の warning 継続を廃止すること。
- JSONL と SQLite の関係を設計で固定し、分断書き込みを解消すること。event log→再投影、または単一正本＋再構築戦略へ変更すること。

### 5-3. strict validation と fail-fast
- `mapper.py` は raw dict / sqlite row を曖昧に受けず、validated row DTO のみを扱うこと。`float()/bool()` の無条件 coercion、`.get(..., default)` 補完を廃止すること。
- `embedding_client.py` は HTTP / transport / JSON / schema error を分類し、invalid response・timeout・circuit open・disabled を曖昧に success=False へ潰さないこと。
- `retriever.py` は invalid timestamp を 0.0 扱いせず、schema error にすること。search failure を empty hit に見せないこと。
- `extract.py` は固定英語キーワードと長さ閾値に依存せず、ExtractionDecision / Candidate DTO を返す構造へ変更すること。
- `ingestion.py` は抽出・dedup・永続化・linking を stage 分離し、manual write と自動 ingestion を統一路経にすること。
- `injection.py` は空クエリと embedding failure を fail-fast 扱いし、snippet fallback を DTO 化すること。

## 6. RAG 層の改善項目

### 6-1. stage モデル化
- `pipeline.py`、`stage.py`、`types.py` は mutable bag / TypedDict 中心構造をやめ、stage input/output/result/trace DTO を導入すること。
- `PipelineStage.run()` は typed request context を受け、failure reason と observability を返す契約へ変更すること。

### 6-2. repository / llm / pipeline の責務分離
- `repository.py` は tokenization fallback、DB failure fallback、raw row 返却を廃止し、row mapper・dedup・full document fetch を分離すること。
- `llm.py` は MQE / rerank / summarize / refine / embedding の request builder、response parser、error mapping を分離し、malformed response fallback を削除すること。
- `pipeline.py` は safe wrapper を削除し、expand → retrieve → merge/dedup → rerank → assemble → refine → format を明示 phase 化すること。

### 6-3. ingestion 系 strict 化
- crawler / chunk splitter / ingester は crawl JSON、chunk JSON、URL parse、language detection、chunk pair、registration result を strict DTO 化すること。`str(entry[0])` や `None` fallback を削除すること。

## 7. MCP / Installer / File / GitHub / Tool 実行基盤の改善項目

### 7-1. MCP 共通基盤の改善
- `server.py` は HTTP / stdio / auth / truncation / audit / dispatch を分離し、`ToolArgs = dict[str, Any]` と broad exception を除去すること。
- `dispatch.py` は unknown tool / validation failure / handler failure / internal failure を result model で区別すること。
- `tool_validators.py` は未登録時挙動と危険 tool の validator を厳格化し、失敗コード付き例外または result を返すこと。
- `audit.py` は logger helper ではなく監査イベントモデル + backend port に再編すること。

### 7-2. Installer の改善
- `installer_templates.py` を server skeleton / tool schema / endpoint / startup / config snippet に分割し、role ごとの mapping で生成すること。
- `installer_validation.py` は `str | None` 返却を廃止し、validation / normalization failure を専用例外へ変更すること。
- `installer_writer.py` は検証 / 競合検出 / 原子的書き込み / 権限設定 / dry-run / 生成後検証へ分割すること。
- `installer_port.py` はポート探索・衝突判定・設定読み取りを strict に扱い、曖昧 fallback をやめること。

### 7-3. File MCP の改善
- `common.py` に path/size/type/symlink/path traversal の共通 policy を定義し、read/write/delete で共有すること。HTTPException を domain exception へ置換すること。
- read/write/delete の models は `_get_cfg()` fallback を廃止し、strict config 注入へ変更すること。success / failure は discriminated result へ置き換えること。
- read/write/delete の service は formatter・singleton・動的 dispatch を外し、DI 化と atomic I/O を行うこと。dry-run 契約を統一すること。

### 7-4. GitHub MCP の改善
- `service.py` から `_get_cfg()`、HTTPException、動的 dispatch、formatter、監査直書きを除去し、typed config 注入 + domain exception + mapper/formatter/policy 分離に変更すること。
- `tools.py` は request model 由来の schema 生成へ移行し、server/service/tool 定義の整合性を自動検証すること。
- `server.py` は declarative route registration と common exception handler を導入し、thin transport にすること。

### 7-5. MCP サーバ群横断の改善
- cicd/shell/file/git/sqlite/rag_pipeline/web_search 各 MCP で `_get_cfg()` + `{}` fallback を廃止し、typed config dataclass を起動時注入すること。
- service 層から HTTPException を除去し、domain exception を server 層で HTTP 変換すること。git service の `"[ERROR] ..."` 文字列返却も例外化すること。

### 7-6. Tool 実行基盤の改善
- `tool_runner.py` は assert、invalid JSON fallback、永続化失敗握りつぶし、直接 print を削除し、ToolCallRequest / ToolExecutionResult を導入すること。
- `tool_approval.py` は ApprovalDecision の TypedDict を廃止し、ApprovalOutcome DTO + Enum 化すること。dry-run preview failure を黙殺しないこと。
- `tool_policy.py` は bool / 生文字列 risk/tier を廃止し、Enum + 具体例外へ変更すること。unknown tier を fail-fast にすること。
- `tool_scheduler.py` は list/dict 前提 metadata を typed ToolMeta へ置き換え、metadata 欠落時は即失敗にすること。
- `tool_result_formatter.py` は `str(args.get(...))` を廃止し、typed preview dispatcher と再帰マスキングへ変更すること。
- `tool_audit.py` は audit logger 未設定時に黙って return せず、AuditUnavailableError または typed failure で扱うこと。
- `tool_loop_guard.py` は raw dict ではなく ToolCallRequest ベースの fingerprint を使い、GuardDecision DTO を返すこと。
- `stdio_lifecycle.py` は no-op と broad exception を廃止し、LifecycleConfigurationError と strict state machine を導入すること。

## 8. Agent Runtime の改善項目

### 8-1. REPL / health / factory / lifecycle の再分離
- `repl.py` は startup / shutdown / input loop の最小責務だけを持つ thin coordinator 化を行うこと。`"?"` 返却や startup warning 直表示を削除すること。
- `repl_health.py` は health probe、tool definition check、watchdog、restart policy を分離し、warning 文字列ではなく structured diagnostic result を返すこと。
- `factory.py` は wiring を分解し、部分構築成功のまま継続しない fail-fast builder 群へ再設計すること。
- `http_lifecycle.py` / `stdio_lifecycle.py` / `lifecycle.py` / `lifecycle_protocol.py` は state machine と protocol を厳密化し、restart / ensure_ready / health poll / shutdown の失敗契約を統一すること。

### 8-2. Context / Orchestrator / TurnRunner / History の再整理
- `config.py` は load → normalization → field validation → cross-field validation の一方向パイプラインへ整理すること。
- `context.py` は shared mutable bag をやめ、ConversationState / TurnState / RuntimeStats / AppServices を責務ごとに分離すること。
- `orchestrator.py` と `llm_turn_runner.py` は turn lifecycle・tool handling・streaming・error mapping・view interaction を分離し、厳密な結果型で接続すること。
- `history.py` は圧縮失敗を warning 継続しない。履歴圧縮、保護ターン、予算計測、選定を分離すること。
- `history_selection_policy.py` は純粋関数 / 純粋オブジェクト化し、副作用を持たせないこと。
- `cli_view.py` は表示専用にし、Writer / Reader / View contract を Protocol 化すること。

### 8-3. Repository の fail-fast 化
- `session.py` から note/document/message/session title/undo の責務を分離し、session façade を薄くすること。
- `session_message_repo.py`、`document_repo.py`、`note_repo.py` は broad exception と `None/[]/False` fallback を廃止し、typed model と DB provider 注入へ変更すること。

## 9. テストと静的検査の改善項目
- `mypy --strict`、`ruff check`、`pytest` を完了条件に含めること。
- 異常系テストとして、invalid reload config、unknown role/tier/tool、invalid JSONL、invalid embedding response schema、invalid timestamp、DB failure、tool args malformed JSON、audit logger unavailable、stdio lifecycle misconfiguration、MCP auth failure、port collision、partial write failure などを追加すること。
