# Fix tool_hash_key to include tool name in failed-call fingerprint

## Priority

High

## Goal

Prevent failed-call tracking and retry guard fingerprint collisions between different tools with identical arguments.

## Background

`tool_hash_key(name, args)` is used for failed-call tracking, not for cache keys.

The fingerprint must include both:

- tool name
- serialized arguments

Without the tool name, different tools with the same arguments may share the same fingerprint.

## Scope

Affected files:

- `shared/tool_executor_helpers.py`
- tests for tool fingerprinting
- any tests for retry/dedup guards if affected

## Implementation requirements

Update `tool_hash_key()` to include `name` in the hash input.

Expected behavior:

    def tool_hash_key(name: str, args: dict[str, object]) -> str:
        payload = f"{name}:{_json_dumps(args)}"
        return hashlib.md5(
            payload.encode(),
            usedforsecurity=False,
        ).hexdigest()

Keep the note that this is a non-security hash.

## Acceptance criteria

- `tool_hash_key("read_text_file", {"path": "a"})` differs from `tool_hash_key("delete_file", {"path": "a"})`.
- Same tool name and same args produce the same fingerprint.
- Same tool name with args in different key order produces the same fingerprint if `_json_dumps()` sorts keys.
- Cache key behavior is unchanged.
- No security purpose is implied.

## Validation plan

- Add or update tests for `tool_hash_key()`.
- Test different tools with identical args.
- Test same tool with identical args.
- Test deterministic output for sorted args.

## Out of scope

- Changing cache key format.
- Replacing MD5 with a cryptographic hash.
