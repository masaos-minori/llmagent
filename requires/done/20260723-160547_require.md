## Background

The documentation already contains useful information such as:
- Implementation notes
- Current behavior
- Inferred intent
- Responsibility boundaries
- Failure behavior
- Needs confirmation items

However, section names and structure differ across documents.

## Problem

When current behavior, inferred design intent, and unresolved uncertainty are mixed together, AI agents and reviewers may mistake uncertain interpretation for confirmed specification. A consistent section structure will make the documents easier to read, review, and use for implementation planning.

## Reason for change

This task improves clarity without changing technical content. The documentation should clearly distinguish:
- What the implementation currently does
- Why the design appears to exist
- What the component owns or does not own
- How failures are handled
- What still needs confirmation

This supports safer AI-assisted implementation and documentation maintenance.

## Implementation intent

Improve readability and reduce ambiguity in major design documents. This is not a rewrite — it is a targeted restructuring of existing content where the separation improves clarity. Do not force all five sections into every document if a section is not relevant.

## Implementation instructions

Apply the following Japanese headings where appropriate:

- `## 現在の実装挙動` — Document what the implementation currently does, verified from source code
- `## 設計意図` — Document why the design appears to exist, based on evidence labels
- `## 責務境界` — Document what the component owns or does not own
- `## 失敗時の挙動` — Document how failures are handled
- `## 未確認事項` — Document what still needs confirmation, linked to the Needs Confirmation inventory

Use the existing evidence labels defined in `00_governance_03_evidence-labels.md`. Work document by document. Preserve existing content and reorganize only where it improves clarity. Do not over-template guide documents. Keep edits small, reviewable, and evidence-aware.

## Acceptance criteria

- [ ] Selected major documents more clearly separate current behavior from design intent
- [ ] Responsibility boundaries are explicit where relevant
- [ ] Failure behavior is clearly separated where relevant
- [ ] Unresolved or uncertain items are placed under `未確認事項` or linked to the Needs Confirmation inventory
- [ ] Existing evidence labels are preserved or normalized
- [ ] No new evidence label system is introduced
- [ ] Existing useful content is preserved
- [ ] No document is rewritten from scratch
