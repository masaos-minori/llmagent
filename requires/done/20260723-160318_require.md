# Requirement: Improve Needs Confirmation and Evidence Label Operations

## Priority

High

## Target files

- `docs/00_governance_07_needs-confirmation-inventory.md`
- Selected major design documents (when evidence-label clarification is necessary)

## Background

The documentation set already defines evidence labels in `00_governance_03_evidence-labels.md`. It also has `00_governance_07_needs-confirmation-inventory.md`, which tracks unresolved confirmation items such as NC-001 through NC-017. However, the inventory needs stronger prioritization and duplicate management.

Some major design documents also use informal expressions such as "implementation note", "current behavior", "inferred intent", or "Needs confirmation" without consistently connecting them to the established evidence label policy.

## Problem

The current Needs Confirmation inventory is useful as a list, but it is less useful as an execution plan. The main gaps are:
- No clear priority field
- No explicit duplicate or related-item management
- No resolution target field
- No indication of whether an item blocks implementation or documentation confidence

This makes it difficult for AI agents and maintainers to decide which uncertainty should be resolved first.

## Reason for change

Unconfirmed items should not remain as passive notes. Items that may affect runtime behavior, persistence, workflow execution, startup behavior, or data correctness need to be prioritized. Evidence labels should also be used consistently enough that AI agents can distinguish confirmed facts from inferred intent and unresolved uncertainty.

## Implementation intent

Turn the Needs Confirmation inventory into an actionable uncertainty-management tool. This task does not resolve individual NC items — it improves the structure needed to manage them.

## Implementation instructions

### 1. Update `00_governance_07_needs-confirmation-inventory.md`

Add Japanese definitions for the following fields:

- **Priority**: Classification of urgency for resolving the item
- **Related NC**: Links to other NC items that share the same root cause
- **Resolution Target**: Expected method of resolution (e.g., source verification, operator confirmation, test addition)
- **Blocking**: Whether this item blocks downstream implementation or documentation confidence

Add Japanese priority values:

#### High
Items affecting runtime behavior, data loss risk, security behavior, startup failure, workflow execution, persistence correctness, or production operation.

#### Medium
Items involving API ownership ambiguity, suspected dead code, unclear operational behavior, unclear diagnostics, or ambiguous configuration behavior.

#### Low
Items involving wording issues, legacy docstring uncertainty, future-use uncertainty, or low-impact documentation ambiguity.

### 2. Apply initial prioritization

Where enough information is already available, add priority metadata to existing NC entries. At minimum, review and classify:

- **NC-003**: ETagManager / DocumentManager `doc_id` behavior → classify based on impact assessment
- **NC-008**: `workflow_id` usage in multi-workflow scenarios → classify based on impact assessment
- **NC-010 and NC-014**: duplicate `gen_rag_reference.py` output target issue → mark as related or duplicate-root items without removing either ID

### 3. Audit selected major design documents for evidence-label usage

Review only major design documents that contain important factual or inferred claims. Initial target documents:
- `01_overview-arch-01-process.md`
- `01_overview-arch-02-pipelines.md`
- `03_rag_03_01_query_pipeline-overview.md`
- `04_mcp_00_document-guide.md`
- `05_agent_00_document-guide.md`
- `90_shared_00_document-guide.md`

For these documents:
- Normalize evidence labels only where necessary
- Do not label every sentence
- Preserve readability
- Ensure unresolved uncertainty is either described with required fields or linked to the Needs Confirmation inventory
