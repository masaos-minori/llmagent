# Title
Unify config loading, reload behavior, and restart-required semantics

## Summary
Create one authoritative model for config loading and runtime reload behavior so operators and AI systems can reliably know what applies immediately and what requires restart.

## Background
Config semantics are described across multiple docs.
Some settings are hot-reloadable, some require restart, and some are only partially reloadable.

## Problem
This makes configuration changes easy to misinterpret and increases the chance of operational mistakes.

## Required Changes
- Classify major settings into:
  - startup-only
  - hot-reloadable
  - restart-required
  - partially reloadable
- Present this classification in one authoritative table.
- Make `/reload` output clearly report:
  - applied
  - deferred
  - restart required
  - unchanged / ignored
- Clarify the role of `common.toml`.

## Acceptance Criteria
- Operators can determine reload behavior from one place.
- `/reload` output is explicit and actionable.
- Documentation and runtime behavior match.

## Out of Scope
- Complete redesign of config file layout

## AI Implementation Instruction
Implement this as a configuration-clarity task.

Strict rules:
- Do not redesign the whole config system unless necessary.
- Focus on runtime semantics clarity, not feature growth.
- Remove contradictory descriptions from active docs.
- If a field’s runtime semantics remain uncertain, mark it as: Needs confirmation

Deliverables:
1. One authoritative reload-behavior matrix
2. Improved `/reload` output semantics
3. Docs aligned with runtime behavior
