## [High][eventbus][db] Redesign SQLite connection ownership for threadpool-based execution

### Summary
The current design uses one shared SQLite connection and justifies it by saying all DB operations run on one async event-loop thread.

### Problem
That justification stops being valid once DB work is offloaded to a threadpool.
Connection ownership must be redesigned explicitly.

### Scope
- db.py
- app.py
- lifecycle initialization
- DB connection strategy docs

### Facts
- `open_db()` returns one shared `sqlite3.Connection`. 【2-f75a0d】【1-6bdda3】
- The current docstring explicitly justifies the shared connection with single-thread event-loop execution. 【2-f75a0d】【2-d76c04】
- Threadpool usage is still marked as an unconfirmed item in known issues. 【5-bd74c0】

### Task
- define a new connection-ownership policy for offloaded DB execution
- choose and implement one of:
  - per-call connection
  - thread-confined connection
  - shared connection with explicit serialization lock
- update lifecycle initialization and cleanup accordingly
- document the selected model and its reasoning
- add tests for concurrent publish/subscribe access

### Expected Result
The Event Bus has an explicit, safe, and documented SQLite connection model for threadpool-based execution.

### Acceptance Criteria
- connection ownership is documented and enforced in code
- the old single-thread safety assumption is removed
- concurrency behavior is covered by tests
- no ambiguous connection-sharing model remains
