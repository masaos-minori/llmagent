# Require: Make duplicate tool exclusion FATAL in LOCAL mode

## Priority

Medium

## Target files

- `scripts/agent/services/mcp_tool_discovery.py`

## Background

When multiple MCP servers report the same tool name, the tool is excluded from the registry. Under the current unified severity scheme (`is_fatal = strict or (security_profile == PRODUCTION)`), this finding is logged as WARNING in LOCAL mode but as FATAL in PRODUCTION mode. The WARNING is displayed via `write_warning()` in the startup banner.

## Problem

In LOCAL mode, duplicate tool exclusions are only WARNING-level. When many warnings appear during startup, operators may miss the fact that a tool has been silently excluded from the registry. This can lead to confusion when tool calls fail unexpectedly.

## Reason for change

Duplicate tool exclusions are a significant operational issue — a tool is completely unavailable after registration. Even in LOCAL mode, this should be treated as a FATAL finding so operators cannot overlook it.

## Implementation intent

Make duplicate tool exclusions always produce FATAL findings regardless of security profile or strictness setting.

## Implementation instructions

In `scripts/agent/services/mcp_tool_discovery.py`, around line 293-309, replace the conditional severity with an unconditional FATAL for duplicate tool exclusions:

```python
for name, group in by_name.items():
    server_keys = sorted({server_key for server_key, _, _ in group})
    if len(server_keys) > 1:
        status = StartupCheckStatus.FATAL  # Always fatal — tool is unusable
        msg = (
            f"duplicate tool name {name!r} reported by multiple servers: "
            f"{', '.join(server_keys)} — excluded from registry"
        )
        findings.append(
            StartupCheckOutcome(source=_SOURCE, status=status, message=msg)
        )
        continue
```

Remove the dependency on `self._is_fatal_severity()` for this specific finding type.

## Acceptance criteria

- Duplicate tool exclusions always produce FATAL findings regardless of security profile or strictness
- WARNING-level findings for other issues (drift, etc.) remain unchanged
- Startup fails when duplicate tools are detected (FATAL causes startup abort)
- No regression in PRODUCTION mode behavior (was already FATAL)

## Tests

- Unit test: Mock two MCP servers reporting the same tool, verify FATAL finding is produced in LOCAL mode
- Unit test: Verify other findings (drift, etc.) still use `_is_fatal_severity()` for their severity
- Integration test: Start agent with conflicting MCP servers, verify startup fails with clear error
