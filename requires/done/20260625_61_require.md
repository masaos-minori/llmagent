## [High][agent][cli] Remove all /db flat-form compatibility aliases and keep only scoped /db commands

### Summary
The current CLI still supports flat `/db` commands as compatibility aliases.

### Problem
The command surface has two parallel entry paths for the same maintenance actions:
- flat compatibility aliases such as `/db stats`
- scoped commands such as `/db session stats` and `/db rag stats`

This keeps backward-compatibility behavior alive in the runtime and documentation, increases maintenance cost, and weakens the scope boundary between session DB operations and RAG DB operations.

### Scope
- agent/commands/*
- agent/commands/registry.py
- CLI routing for `/db`
- docs/05_agent_07_cli-and-commands.md
- docs/05_agent_04_state-and-persistence.md

### Facts
- The CLI docs explicitly label the flat `/db` forms as `Flat forms (compatibility aliases)`. 【1-819e5c】
- The state/persistence docs also describe flat `/db` forms as compatibility aliases alongside the scoped `/db rag ...` and `/db session ...` commands. 【2-3cddb8】

### Task
- remove support for all flat `/db` compatibility aliases
- keep only scoped commands:
  - `/db rag ...`
  - `/db session ...`
- update command dispatch so flat alias forms are rejected with a clear validation error
- update all CLI and state/persistence docs to remove alias references
- remove or update tests that still expect flat alias behavior

### Explicit Removal Targets
- `/db stats`
- `/db urls`
- `/db clean`
- `/db rebuild-fts`
- `/db health`
- `/db checkpoint`
- `/db vacuum`
- `/db purge`
- `/db recover`
- `/db consistency`

### Expected Result
The CLI exposes exactly one command model for DB operations: scoped commands only.

### Acceptance Criteria
- all flat `/db` compatibility aliases are removed
- only `/db rag ...` and `/db session ...` remain valid
- flat alias calls fail with a clear migration error message
- docs no longer mention flat compatibility aliases
- tests no longer depend on flat alias behavior
