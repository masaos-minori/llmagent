# Align DbConfig mutability semantics with its documentation

## Summary
Resolve the mismatch between the documented “immutable” nature of `DbConfig` and its current mutable dataclass implementation.

## Background
`DbConfig` is described as immutable in comments/docstrings, but the implementation is a regular mutable dataclass.

## Objective
Make the documented and implemented mutability model consistent.

## In Scope
- `config.py`

## Required Changes
- Choose one of:
  - make `DbConfig` actually immutable (`frozen=True` or equivalent), or
  - update the surrounding documentation/comments so it is clearly documented as mutable
- Ensure downstream usage is compatible with the chosen direction

## Acceptance Criteria
- `DbConfig` mutability is no longer ambiguous
- Code comments and actual behavior match
- Reviewers can tell whether mutation is allowed by design

## Out of Scope
- Redesigning all config DTOs
- Adding runtime mutability guards elsewhere

## Implementation Notes
Prefer explicitness. If mutability is allowed, document why. If immutability is desired, enforce it in code.
