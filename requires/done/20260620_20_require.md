# Title
Expose memory embedding fallback and circuit-breaker state in normal runtime views

## Summary
Make memory-layer embedding state and fallback mode visible so operators can tell when retrieval has degraded to FTS-only behavior.

## Background
The memory embedding client includes retry and circuit-breaker behavior, and hybrid retrieval can fall back to FTS-only mode when embedding is disabled or unavailable.

## Problem
Operators may not realize memory retrieval quality has degraded, because fallback behavior is not sufficiently visible in normal runtime views.

## Required Changes
- Expose fields such as:
  - embedding enabled/disabled
  - circuit open/closed
  - retrieval mode (hybrid vs FTS-only)
- Surface this information in at least one normal operator-facing path such as `/stats` or `/memory status`.
- Emit a visible signal when retrieval falls back to FTS-only mode.

## Acceptance Criteria
- Operators can see when memory retrieval is running in degraded mode.
- Fallback from hybrid retrieval to FTS-only is observable.
- Runtime visibility does not require deep debugging knowledge.

## Out of Scope
- Changing embedding API behavior
- Redesigning retrieval scoring

## AI Implementation Instruction
Implement this as a visibility and diagnosability task.

Strict rules:
- Do not change fallback semantics unless necessary.
- Make degraded state visible in normal runtime paths, not only debug internals.
- Keep the status simple and operator-friendly.
- If exact circuit-breaker state semantics are unclear, mark them as: Needs confirmation

Deliverables:
1. Runtime memory-status visibility
2. Fallback-mode visibility
3. Operator-friendly indicators of degraded retrieval
