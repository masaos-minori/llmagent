# Declare `shared/tool_registry.py` as the single routing source of truth

## Summary
Clarify the routing ownership model by making `shared/tool_registry.py` the explicit single source of truth and reclassifying other routing inputs as compatibility or validation layers.

## Background
Routing currently uses live discovery, the tool registry, config `tool_names`, and static fallback. The priority is documented, but the maintenance model is still more complex than necessary.

## Objective
Define which routing data source is authoritative and which ones are:
- mandatory
- compatibility-only
- validation-only
- fallback-only

## In Scope
- `04_mcp_03_routing_lifecycle_and_execution.md`
- `04_mcp_06_configuration_and_operations.md`
- `config/agent.toml` documentation sections that describe MCP server tool mapping

## Required Changes
- Explicitly declare `shared/tool_registry.py` as the canonical routing source
- Reclassify config `tool_names` as either:
  - required metadata, or
  - optional validation material
- Reclassify static fallback as a compatibility layer
- Update "new tool" and "new server" procedures accordingly
- Ensure all documentation uses the same routing ownership model

## Acceptance Criteria
- The routing source of truth is explicit and unambiguous
- The role of `tool_names` is clearly documented
- The role of static fallback is clearly documented
- New tool registration steps reflect the final ownership model

## Out of Scope
- Refactoring transport code
- Removing existing fallback behavior unless separately approved

## Notes
Prefer a maintenance model that minimizes required manual synchronization points.
