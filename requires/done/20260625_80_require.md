## [High][rag][imports] Remove the backward-compatibility `scripts/rag/llm.py` re-export module and migrate all imports to the real implementation modules

### Summary
`scripts/rag/llm.py` is explicitly documented as a backward-compatibility re-export module.
The real implementations now live in:
- `scripts/rag/llm_client.py`
- `scripts/rag/llm_prompts.py` 【1-f0779f】

### Problem
Keeping the re-export module preserves two parallel import paths for the same functionality.
That increases maintenance cost, weakens dependency clarity, and makes code search and refactoring harder.

### Scope
- `scripts/rag/llm.py`
- all RAG import sites using `rag.llm`
- docs that still show `from rag.llm import ...`
- tests that rely on the compatibility path

### Task
- remove `scripts/rag/llm.py`
- migrate all imports to the real implementation modules:
  - `scripts/rag/llm_client.py`
  - `scripts/rag/llm_prompts.py`
- update all examples and documentation to use the real module paths
- remove tests that depend on the compatibility re-export
- add negative checks or code search validation so the old import path is not reintroduced

### Expected Result
RAG has exactly one import path for LLM-related implementation code, with no backward-compatibility re-export module remaining. 【1-f0779f】

### Acceptance Criteria
- `scripts/rag/llm.py` no longer exists
- all code imports the real implementation modules directly
- docs no longer instruct users to import from `rag.llm`
- tests no longer depend on the compatibility re-export path
