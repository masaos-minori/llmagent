# Title
Improve RAG consistency check repair guidance

## Priority
Medium

## Labels
- enhancement
- rag
- operations
- consistency
- diagnostics

## Dependencies
- Issue 5 recommended

## Summary
Improve RAG consistency diagnostics so operators can move from detection to repair without manual database investigation.

## Background
`check_rag_consistency()` detects differences among `chunks`, `chunks_fts`, and `chunks_vec`, but it is read-only and does not identify repair targets in enough detail.

## Problem
The current output can say that inconsistencies exist, but it does not clearly identify affected URL/doc/chunk targets or the best repair action.

## Required Changes
- Keep `check_rag_consistency()` read-only.
- Add recommended actions for:
  - `fts_gap`
  - `fts_orphan_count`
  - `orphan_vec_count`
- Expose affected `chunk_id`, `doc_id`, and URL where feasible.
- Add fix hints to `/db consistency` output.
- Document when to use FTS rebuild vs `ingester.py --force`.

## Acceptance Criteria
- Operators can identify what to repair after a failed consistency check.
- `/db consistency` output includes actionable fix hints.
- The repair flow is documented clearly.

## Out of Scope
- Automatic destructive repair.
- Replacing consistency checks with a background job.

## Components
- `db/maintenance.py`
- `/db consistency`
- `scripts/rag/ingestion/ingester.py`
- `03_rag_05_configuration_and_operations.md`

## AI Implementation Instruction
Implement this as an operational diagnostics improvement.

Strict rules:
- Do not make the consistency check mutate data.
- Provide repair hints without performing destructive actions automatically.
- Include affected identifiers when possible.
- If affected URL resolution is expensive or ambiguous, mark it as: `Needs confirmation`.
