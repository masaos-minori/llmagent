# Agent Startup Processing — Session Diagnostics Persistence Fails Silently

## Summary

If `_persist_session_diagnostics()` raises an exception during shutdown, the exception is silently swallowed. This can leave diagnostic data lost when the agent shuts down, making post-mortem analysis difficult.

## Severity

Low

## Files

- `scripts/agent/repl.py:243-253`

## Details

In `AgentREPL._close_resources()`:

```python
async def _close_resources(self) -> None:
    ctx = self._ctx
    try:
        self._persist_session_diagnostics(ctx)
        await self._persist_session_memories(ctx)
        await self._close_http_clients(ctx)
        await self._shutdown_lifecycle_manager(ctx)
    except Exception as e:
        logger.error("Error closing resources: %s", e)
```

`_persist_session_diagnostics()` writes diagnostic data to the session record:

```python
def _persist_session_diagnostics(self, ctx: AgentContext) -> None:
    if ctx.session.session_id is None:
        return
    diagnostics = ctx.session.diagnostics
    if diagnostics:
        ctx.session.update_diagnostics(diagnostics)
```

Potential failure modes:
1. `update_diagnostics()` raises `sqlite3.IntegrityError` — duplicate key violation
2. `update_diagnostics()` raises `sqlite3.OperationalError` — DB locked during concurrent write
3. `update_diagnostics()` raises `RuntimeError` — session schema missing (same as `start()`)

When any of these occur, the exception propagates up to `_close_resources()` which catches it and logs. But the log message "Error closing resources" is too generic — it doesn't indicate which specific resource failed, making debugging harder. Additionally, if multiple resources fail during close, only the first exception is logged; subsequent failures are masked.

## Reproduction Steps

1. Lock the SQLite database (e.g., another process holds a transaction open)
2. Shut down the agent REPL
3. Observe that the error message is generic ("Error closing resources") without specifying which operation failed

## Expected Behavior

Each resource close operation should have its own try/except block with specific error messages, allowing operators to identify which resource failed and take corrective action.

## Proposed Fix

Separate resource cleanup into individual try/except blocks:

```python
async def _close_resources(self) -> None:
    ctx = self._ctx
    errors = []
    
    try:
        self._persist_session_diagnostics(ctx)
    except Exception as e:
        logger.error("Failed to persist session diagnostics: %s", e)
        errors.append(("diagnostics", str(e)))
    
    try:
        await self._persist_session_memories(ctx)
    except Exception as e:
        logger.error("Failed to persist session memories: %s", e)
        errors.append(("memories", str(e)))
    
    try:
        await self._close_http_clients(ctx)
    except Exception as e:
        logger.error("Failed to close HTTP clients: %s", e)
        errors.append(("http_clients", str(e)))
    
    try:
        await self._shutdown_lifecycle_manager(ctx)
    except Exception as e:
        logger.error("Failed to shutdown lifecycle manager: %s", e)
        errors.append(("lifecycle", str(e)))
    
    if errors:
        summary = "; ".join(f"{name}: {err}" for name, err in errors)
        logger.error("Resource close errors: %s", summary)
```
