# [High] Align ToolRegistry-only routing contract, documentation, and registry tests

## Background

`ToolRegistry` is the sole routing authority for MCP tools.

`ToolRouteResolver.resolve()` resolves a tool name through `ToolRegistry.get_server_for_tool()` and raises `ValueError` for unknown tools. Config `tool_names` and live `/v1/tools` responses are validation metadata only, not routing inputs.

Existing documentation still contains stale wording that may imply a live `/v1/tools` discovery fallback or a two-step routing cascade.

## Goal

Make the routing contract unambiguous across implementation docs, generated docs, and regression tests.

## Scope

- `shared/tool_registry.py`
- `shared/route_resolver.py`
- `shared/tool_constants.py`
- `shared/tool_routing_validation.py`
- `90_99.md`
- `tests/test_tool_registry.py`
- `tests/test_tool_registry_counts.py`

- `tests/test_tool_executor_routing.py`

## Notes for AI Implementer

Do not introduce a discovery-map fallback. If routing fails, the correct behavior is to raise `ValueError` for unknown tools.
