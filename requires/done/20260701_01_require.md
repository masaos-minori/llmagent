## Issue 1: [High] Remove diagnostics.jsonl from implementation and active documentation

### Goal

Remove the legacy `diagnostics.jsonl` runtime diagnostics path and make `session_diagnostics` the only supported diagnostics persistence mechanism.

### Background

Diagnostics are implemented through `DiagnosticStore`, which writes diagnostic records into the `session_diagnostics` table in `session.sqlite`.
`AgentSession.save_diagnostic()` delegates to `DiagnosticStore` and does not write diagnostics into `messages`.

### Scope

- Remove any remaining `diagnostics.jsonl` write path.
- Remove `diagnostics.jsonl` from active documentation.
- Replace JSONL operational examples with SQL examples against `session_diagnostics`.

### Implementation Tasks

- Search the implementation for the following strings:
  - `diagnostics.jsonl`
  - `_persist_session_diagnostics`
  - `Path(session_db_path).parent`
  - session-end JSONL diagnostic summary writes
- If any JSONL diagnostics write path exists, remove it.
- Keep diagnostics persistence through `DiagnosticStore.save()` only.
- Update the following documents:
  - `05_agent_04_state-and-persistence.md`
  - `05_agent_10_operations-and-observability.md`
  - `05_agent_90_inconsistencies_and_known_issues.md`
- Remove operational examples that read `diagnostics.jsonl`.
- Add SQL examples for `session_diagnostics`.

### Acceptance Criteria

- No implementation path writes `diagnostics.jsonl`.
- Active docs state that diagnostics are persisted only in `session_diagnostics`.
- No active documentation describes `diagnostics.jsonl` as supported, active, deprecated, or dual-written.
- Partial completion, transport error, and guard hint documentation points to `session_diagnostics`.

### Validation

- Grep returns no active runtime write path for `diagnostics.jsonl`.
- Run diagnostic persistence tests if available.
