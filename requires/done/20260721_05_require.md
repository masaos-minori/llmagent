# [High] Restore `[[tool_definitions]]` coverage for git/mdq/file_write tools; fix inert `routing_drift_strict`

## Priority
High

## Labels
priority/high, area/config, area/mcp, area/tool-routing, consistency

## Summary
Two independent, verified gaps in `config/agent.toml`'s tool-routing configuration:

1. **19 registered tools have no `[[tool_definitions]]` entry** (the LLM-facing function-calling
   schema), spanning three MCP servers: `file_write` (`edit_file`, `move_file`), `git` (all 10 tools:
   `git_status`, `git_log`, `git_diff`, `git_branch`, `git_show`, `git_add`, `git_commit`,
   `git_checkout`, `git_pull`, `git_push`), and `mdq` (all 7 tools: `search_docs`, `get_chunk`,
   `outline`, `index_paths`, `refresh_index`, `stats`, `grep_docs`). All 19 are correctly registered in
   `[mcp_servers.*].tool_names` and `[tool_safety_tiers]`, but absent from the 45-entry
   `[[tool_definitions]]` array.
2. **`routing_drift_strict = true` has no effect.** `scripts/agent/startup.py:252` calls
   `check_routing_drift(ctx)` without passing `strict=`, so `check_routing_drift()`
   (`scripts/agent/repl_health.py:365`, default `strict: bool = False`) always runs in non-strict mode
   regardless of the `routing_drift_strict` config value. Setting `routing_drift_strict = true` in
   `config/agent.toml` does not produce the documented "startup `RuntimeError` on drift" behavior
   (`docs/05_agent_08_03_configuration-tools-memory.md` line 32) — drift is always logged as a
   non-fatal warning, regardless of the configured value.

## Background

### Gap 1 — missing tool_definitions
`requires/done/20260711_04_require.md` previously found and fixed an identical class of gap for the
`file_read` server (4 missing tools: `list_directory_with_sizes`, `read_media_file`,
`read_multiple_files`, `get_file_info`) — confirmed fixed in current `config/agent.toml` (all 9
`file_read` tools now have `[[tool_definitions]]` entries). That requirement's own "Other Servers — No
Issues Found" section (dated 2026-07-11) stated Git (10/10) and MDQ (9/9, though 2 of those —
`fts_consistency_check`/`fts_rebuild` — are not present in current `[mcp_servers.mdq].tool_names` at
all; see Unknowns) were fully covered at the time. As of 2026-07-21, current source shows Git and MDQ
`tool_definitions` coverage has regressed to zero, and File Write's `edit_file`/`move_file` were never
covered by that requirement (out of its scope, since it only covered `file_read`). This requirement
extends the same fix pattern to the three affected servers.

Effective impact today is bounded by the fact that a live `RuntimeToolRegistry` built from `/v1/tools`
discovery normally supplies these tools' actual schemas dynamically at runtime
(`scripts/agent/config_dataclasses.py`, comment near line 189-192) — but when discovery falls back to
the static `config/agent.toml` list (discovery failure, or a mismatch masked by
`tool_definitions_strict=false`), these 19 tools become invisible to the LLM's function-calling
payload with no error raised anywhere in the startup path.

### Gap 2 — inert routing_drift_strict
This is a distinct, more fundamental bug than the "should production require strict=true" design
question already covered by `requires/done/20260709_03_require.md` / `requires/done/20260709_08_require.md`
(which propose a `ProductionConfigValidator` that makes `routing_drift_strict=false` fatal *in
production*). Those two requirements assume that setting `routing_drift_strict=true` actually changes
`check_routing_drift()`'s behavior at startup; this requirement reports that it currently does not,
for any value of `security_profile`, because the config value never reaches the function call. Fixing
this wiring bug is a prerequisite for either of those two requirements' production-hardening logic to
have any real effect once implemented — otherwise the new `ProductionConfigValidator` would validate a
config flag that the actual drift check still ignores.

## Required Changes

### 1. Add missing tool_definitions (19 tools)
Add `[[tool_definitions]]` blocks for `edit_file`, `move_file` (file_write server) and all 10 `git_*`
tools (git server) and all 7 mdq tools (`search_docs`, `get_chunk`, `outline`, `index_paths`,
`refresh_index`, `stats`, `grep_docs`) in `config/agent.toml`, following the existing style/format used
for the other 45 entries (Japanese `description`, `required` params matching each tool's actual
parameter contract). Source the real parameter shapes from:
- `scripts/mcp_servers/file/write_tools.py` (`edit_file`, `move_file`)
- `scripts/mcp_servers/git/git_tools.py` (all 10 git tools)
- `scripts/mcp_servers/mdq/mdq_tools.py` (all 7 mdq tools)

### 2. Fix `routing_drift_strict` wiring
Pass `strict=ctx.cfg.tool.routing_drift_strict` into the `check_routing_drift(ctx)` call at
`scripts/agent/startup.py:252`, so the configured value actually reaches `check_routing_drift()`'s
`strict` parameter instead of silently defaulting to `False`.

### 3. Regression test
Add a test asserting that `routing_drift_strict=true` with an injected drift condition raises/aborts
startup, and `routing_drift_strict=false` with the same drift condition only warns — exercising the
real `startup.py` call site end-to-end, not a direct call to `check_routing_drift(strict=True)` alone
(a test of that form already passes today and would not have caught this bug).
