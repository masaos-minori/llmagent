# [Medium] Clarify approval resume semantics, idempotency expectations, and approval lookup naming

## Background

`/approve <approval_id> [reason]` marks a workflow approval as approved and sets `ctx.turn.pending_approval_task_id`.

The Orchestrator uses `pending_approval_task_id` on the next turn to resume an existing workflow task rather than creating a new task.

This means the current behavior is effectively:

- `/approve` resolves the approval gate.
- Workflow resumes on the next user turn.
- Previously completed stages should be skipped by idempotency.

`WorkflowEngine._gate_approval()` checks the latest approval record and branches by status:

- `approved` → pass to `verify`
- `pending` → raise `WorkflowPendingApprovalError`
- `rejected` → mark task halted and raise `WorkflowHaltError`

However, `approval_ops.get_pending_approval()` is misleadingly named because it returns the latest approval record for a task and does not filter by `status='pending'`.

`/reject` also marks the task as halted directly, while the engine also handles rejected approvals as halted during gate evaluation. This may be intentional defensive duplication, but it should be documented.

## Goal

Make approval resume behavior explicit, ensure idempotent resume is tested, and clarify approval lookup naming.
