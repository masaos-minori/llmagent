# H-2: Relax `_get_handler()` typing to support no-argument exact handlers

## Priority

High

## Labels

bug, typing, agent, cli, command-registry

## Background

`CommandRegistry._get_handler()` currently declares all command handlers as callables that accept a single `str` argument.

This is incorrect because exact-match commands can have no-argument handlers. For example, `_cmd_help()` is implemented as a no-argument method.

This type mismatch encouraged the previous incorrect dispatch behavior where exact commands were called as `handler("")`.

## Goal

Align `_get_handler()` typing with the actual command handler model.

## Scope

### In scope

- Update `_get_handler()` return type in `scripts/agent/commands/registry.py`
- Remove the misleading `Callable[[str], ...]` assumption
- Keep runtime behavior fail-fast for missing handlers

### Out of scope

- Full command signature framework redesign
- `/exit` handling
- plugin dispatch changes

## Required change

Change `_get_handler()` from a single-argument callable model to a generic callable model.

Recommended shape:

- Return `Callable[..., Any]`
- Remove the unused `is_async` parameter if it is not used by the method
- Keep the missing-handler `AttributeError`

## Acceptance criteria

- `_get_handler()` no longer implies every command handler accepts `str`
- `CommandRegistry.dispatch()` can call exact handlers with no arguments
- `CommandRegistry.dispatch()` can call prefix handlers with `args: str`
- Static typing does not force exact command handlers to accept unused args

## Validation

Run command-related tests.

Manual validation:

- `/help` runs
- `/compact` runs
- `/db help` runs

## Affected files

- `scripts/agent/commands/registry.py`
