# [Medium] Clarify watchdog operational state, restart exhaustion, and shared HealthRegistry wiring

## Background

ToolExecutor and watchdog share the same `McpServerHealthRegistry` instance. This shared state affects dispatch gating, restart behavior, and operational status reporting.

Current documentation and operational visibility should clarify:

- ToolExecutor transport failures and watchdog probes update the same registry.
- Watchdog `record_success()` resets failure count and degraded reason.
- Restart limit exhaustion should have a clear HealthRegistry and operator-facing outcome.
- `/mcp status` or equivalent diagnostics should expose enough information for operators.

## Goal

Make shared HealthRegistry wiring and watchdog operational behavior clear to maintainers and operators.

## Scope

- `agent/factory.py`
- `agent/repl_health.py`
- MCP lifecycle documentation
- MCP operations documentation
- MCP status reporting if available

## Tasks

- [ ] Document that ToolExecutor and watchdog share the same `McpServerHealthRegistry`.
- [ ] Explain that transport failures and watchdog probes update the same state.
- [ ] Explain how shared state affects dispatch gating.
- [ ] Document that HTTP 200 health probes call `record_success()`.
- [ ] Document that `record_success()` resets failure count, unavailable timestamp, and degraded reason.
- [ ] Define expected behavior when watchdog restart limit is reached.
- [ ] Decide whether restart limit reached should call `record_failure()`.
- [ ] Decide whether restart limit reached should record a reason such as `restart_limit_reached`.
- [ ] Ensure status output can show restart exhaustion or manual intervention need.
- [ ] Add tests for restart limit reached behavior if not already covered.

## Acceptance Criteria

- [ ] Operators understand why watchdog state affects tool dispatch.
- [ ] Maintainers understand why replacing the registry object would break consistency.
- [ ] Restart limit exhaustion has a deterministic and documented outcome.
- [ ] Status output or logs make restart exhaustion actionable.
- [ ] Documentation matches implementation.

## Notes

This issue is mainly operational clarity and diagnostics. It may require a small implementation change if restart exhaustion is currently ambiguous.
