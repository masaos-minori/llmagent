# Unify `file://` re-ingestion behavior across RAG documents

## Summary
Unify the documented behavior of local file ingestion and re-ingestion via `file://` across all RAG design documents.

## Background
The current documentation is inconsistent about whether `file://` re-ingestion is fully implemented and how freshness detection works. This causes ambiguity for implementers and operators.

## Objective
Produce one canonical and internally consistent specification for:
- first-time `file://` ingestion
- freshness detection for local files
- automatic re-ingestion on file change
- `--force` behavior for local files

## In Scope
- `03_rag_02_ingestion_pipeline.md`
- `03_rag_05_configuration_and_operations.md`
- `03_rag_90_inconsistencies_and_known_issues.md`

## Required Changes
- Compare all `file://` related sections and identify contradictions
- Rewrite the affected sections so that they describe the same behavior
- Standardize the terminology for:
  - SHA-256 hash usage
  - mtime / last_modified handling
  - automatic re-ingestion
  - `--force` semantics
- Ensure all examples use the same behavior model

## Acceptance Criteria
- No contradiction remains across the three documents about `file://` ingestion
- Freshness detection for local files is described identically in all three documents
- `--force` behavior for local files is described identically in all three documents
- The final wording clearly distinguishes HTTP behavior from local file behavior

## Out of Scope
- Code changes in `scripts/rag/ingestion/crawler.py`
- Code changes in `scripts/rag/ingestion/ingester.py`
