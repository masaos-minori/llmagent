# Unify GitHub policy and approval behavior

## Priority

High

## Type

Bug Fix / Security / Refactoring / Testing

## Target Files

- `agent/tool_policy.py`
- `agent/tool_approval.py`
- `shared/tool_constants.py`
- `tests/test_tool_policy.py`
- `tests/test_tool_approval.py`

## Summary

Unify GitHub mutation classification across policy and approval code, fix repository allowlist enforcement, and clarify dry-run failure behavior for high-risk GitHub tools.

## Background

GitHub write tool classification is currently duplicated in policy and approval code.

The duplicated sets are not identical:

- `tool_policy.py` includes some write tools that `tool_approval.py` does not.
- `github_create_issue` and `github_add_issue_comment` are treated inconsistently.
- GitHub dangerous tools must be handled consistently across all approval and policy paths.

There is also a repository allowlist bug in `check_allowed_repo()`:

- The function reads `owner` and `repo`.
- It should compare `owner/repo` against the allowlist.
- Instead, the current comparison checks whether a slash is in the allowlist.

This means the allowlist logic does not correctly enforce the configured repository set.

Finally, dry-run preview failures are currently downgraded to text-only previews. That may be acceptable for low-risk tools, but it is questionable for high-risk GitHub mutation tools.

## Why This Needs to Be Fixed

GitHub operations are external side effects. They can create branches, modify files, delete files, create issues, update pull requests, and merge pull requests.

Inconsistent classification means one enforcement layer can reject a tool while another layer allows it. This weakens the safety model and makes behavior harder to reason about.

The repository allowlist bug is especially important because it directly affects whether GitHub mutation tools are allowed to target a repository.

If dry-run is configured but silently falls back to text-only preview, operators may believe they received a stronger preview than they actually did.

## Fix Intent

The intent is to make GitHub mutation handling consistent and explicit:

- One shared GitHub mutation set should be used across policy and approval.
- Repository allowlist checks should use exact `owner/repo` matching.
- GitOps blocking should apply to the same shared mutation set unless deliberately scoped otherwise.
- Dry-run failure behavior should be explicit and tested, especially for high-risk mutation tools.

## Required Changes

### 1. Use shared GitHub mutation constants

Remove local GitHub write sets from:

- `agent/tool_policy.py`
- `agent/tool_approval.py`

Use the shared set:

- `GITHUB_WRITE_TOOLS | GITHUB_DANGEROUS_TOOLS`

from `shared.tool_constants`.

### 2. Fix repository allowlist comparison

Update `check_allowed_repo()` so it checks:

- `f"{owner}/{repo}" in allowed`

Required behavior:

- Empty allowlist denies GitHub mutation tools.
- Missing `owner` or `repo` denies GitHub mutation tools.
- Matching `owner/repo` allows the operation.
- Non-GitHub mutation tools are not affected.

### 3. Align `gitops_push_blocked`

Apply `gitops_push_blocked` consistently to the shared GitHub mutation set.

If some tools, such as issue creation or issue comments, should not be blocked by `gitops_push_blocked`, document that explicitly and consider renaming or splitting the setting.

### 4. Review dry-run failure behavior

For high-risk GitHub mutation tools, decide whether dry-run failure should block approval instead of falling back to text-only preview.

At minimum:

- document current behavior,
- add tests,
- audit dry-run failure events.

Recommended stricter behavior:

- For high-risk mutation tools, dry-run returned error should block approval.
- Dry-run unsupported may either block or fall back depending on explicit configuration.

## Acceptance Criteria

- `tool_policy.py` and `tool_approval.py` use the same shared GitHub mutation set.
- Local duplicated GitHub write sets are removed.
- `github_create_issue` and `github_add_issue_comment` are handled consistently.
- `github_delete_file` and `github_merge_pull_request` are included as dangerous mutation tools.
- Repository allowlist uses exact `owner/repo` matching.
- Existing `TestCheckAllowedRepo` tests pass.
- New tests cover `gitops_push_blocked` for the shared GitHub mutation set.
- Dry-run fallback behavior for high-risk tools is documented and tested.
- No GitHub mutation tool bypasses preflight checks due to set mismatch.

## Suggested Tests

- allowed repo is accepted
- non-allowed repo is denied
- empty allowlist denies GitHub mutation tools
- missing owner or repo denies GitHub mutation tools
- `github_create_issue` uses the same mutation set in policy and approval
- `github_add_issue_comment` uses the same mutation set in policy and approval
- `github_delete_file` is treated as dangerous
- `github_merge_pull_request` is treated as dangerous
- `gitops_push_blocked=true` blocks shared GitHub mutation tools
- high-risk dry-run failure behavior is covered

## Non-Goals

- Do not implement GitHub server-side dry-run support in this issue unless required by tests.
- Do not change GitHub MCP tool APIs except where needed for policy consistency.
- Do not relax repository allowlist semantics.

## Implementation Notes

If issue creation and issue comments are intentionally allowed during GitOps push blocking, create a distinct constant or configuration name that reflects the narrower behavior.

Otherwise, treat all GitHub mutation tools consistently.
