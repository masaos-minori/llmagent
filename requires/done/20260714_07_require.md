# [High] Consolidate MCP configuration ownership and secret-based authentication model

## Labels
priority/high, area/config, area/security, area/mcp, documentation, production-hardening

## Summary
Consolidate MCP configuration documentation around the confirmed ownership model and replace raw `auth_token` configuration with secret references.

This issue merges all overlapping configuration documentation fixes for:

- `agent.toml [mcp_servers.*]` field ownership
- MCP server-local configuration ownership
- secret-based MCP Bearer authentication
- `tool_definitions` location
- `tool_names` as validation metadata only
- `security_profile` config path
- stale split-config file names
- stale `routing.md` references in setup procedures

## Confirmed Specification

`config/agent.toml [mcp_servers.<key>]` may contain only agent-side lifecycle and transport fields:

- `startup_mode`
- `transport`
- `url`
- `healthcheck_mode`
- `cmd`

MCP server-local application configuration belongs in `config/*_mcp_server.toml`.

Raw bearer tokens must not be stored in either `agent.toml` or `*_mcp_server.toml` for production.

Use secret references instead:

- `auth_token_env`
- `auth_token_file`

## Background
Current documentation mixes several models:

- Some sections list `auth_token`, `tool_names`, `env`, `role`, timeouts, and other fields under `agent.toml [mcp_servers.*]`.
- Auth migration docs imply raw `auth_token` values are configured in `agent.toml`.
- New tool procedures still reference old split files such as `config/tools_definitions.toml`.
- Routing docs correctly say `ToolRegistry` is the sole routing authority, but other sections still imply `tool_names` is a routing-related config field.
- Some procedures still reference `routing.md` instead of the canonical MCP routing document.

These must be corrected as one coherent configuration ownership update.

## Required Changes

### 1. Agent-side MCP fields
Document that `agent.toml [mcp_servers.<key>]` contains only:

- `startup_mode`
- `transport`
- `url`
- `healthcheck_mode`
- `cmd`

Remove the following from the documented `agent.toml [mcp_servers.*]` field list unless implementation explicitly requires otherwise:

- `auth_token`
- `tool_names`
- `env`
- `role`
- `idle_timeout_sec`
- `startup_timeout_sec`
- `call_timeout_sec`

### 2. Server-local configuration
Document that `config/*_mcp_server.toml` owns server-local application settings, such as:

- allowlists
- denylists
- limits
- audit paths
- GitHub `allowed_repos`
- GitHub `allowed_repos_mode`
- shell `command_allowlist`
- file server `allowed_dirs`
- secret references such as `auth_token_env` or `auth_token_file`

### 3. Secret-based MCP authentication
Replace raw `auth_token` documentation with secret references.

Supported preferred fields:

- `auth_token_env`
- `auth_token_file`

Required behavior:

- server side resolves the expected Bearer token from `auth_token_env` or `auth_token_file`
- agent side resolves the same token through an authorized config/secret resolution path
- agent injects `Authorization: Bearer <resolved-token>` for MCP HTTP requests
- production startup fails when an enabled HTTP MCP server has no resolved token
- unresolved env var is fatal in production
- unreadable secret file is fatal in production
- raw `auth_token` should be rejected, forbidden, or migration-warning-only
- local/development may allow missing token with warning

### 4. `tool_definitions`
Document `config/agent.toml::tool_definitions` as the canonical location for LLM-visible tool schemas.

Remove active procedure references to `config/tools_definitions.toml`.

### 5. `tool_names`
Document `tool_names` only as optional startup drift validation metadata if it remains supported.

It must not be documented as:

- a routing input
- a required new-tool step for routing
- a valid confirmed `agent.toml [mcp_servers.*]` field

Routing remains controlled only by ToolRegistry.

### 6. `security_profile`
Document `security_profile` as an agent-wide setting in `config/agent.toml`.

It is not a per-server `[mcp_servers.*]` field.

### 7. Stale file names
Replace stale `routing.md` references with:

- `04_mcp_03_routing_lifecycle_and_execution.md`

Replace active old split config references with current `config/agent.toml` paths.

## Acceptance Criteria
- `agent.toml [mcp_servers.*]` docs list only the confirmed five fields.
- Raw `auth_token` is no longer recommended in any config file.
- `auth_token_env` and/or `auth_token_file` are documented as the production-safe pattern.
- Production validation is described as validating resolved secrets, not plain config values.
- `tool_definitions` documentation points to `config/agent.toml`.
- `tool_names` is clearly validation metadata only and not a routing source.
- `security_profile` is documented as agent-wide.
- No active procedure references `config/tools_definitions.toml` as canonical.
- No active procedure references `routing.md` as current documentation.
- MCP server process isolation remains documented: servers read only their own `*_mcp_server.toml`.
