# Title
Clarify and expose the empty-output fallback path for `use_refiner=True`

## Summary
Document and instrument the case where `_augment_refiner()` returns an empty string and the pipeline falls back to raw chunks.

## Background
The docs acknowledge that an empty-string return from the refiner causes a fallback to raw chunks, but the conditions that produce an empty string are not documented.

## Problem
This makes it hard to understand or debug retrieval-quality changes when the refiner path quietly degrades.

## Required Changes
- Identify the implementation conditions under which `_augment_refiner()` returns an empty string instead of raising an exception.
- Document those conditions explicitly.
- Make raw-chunk fallback visible in stage results, logs, or diagnostics.
- Distinguish refiner exceptions from empty-output fallback.

## Acceptance Criteria
- Empty-output fallback conditions are documented.
- Operators can tell when raw chunks were returned because of refiner fallback.
- Failure and fallback paths are distinguishable.

## Out of Scope
- Redesigning the refiner prompt/model
- Removing refiner fallback behavior

## AI Implementation Instruction
Implement this as a fallback-clarification and observability task.

Strict rules:
- Do not remove fallback behavior unless another issue explicitly changes it.
- Distinguish empty-output fallback from exception-based fallback.
- Keep fallback visible in diagnostics.
- If the implementation path remains ambiguous, mark it as: Needs confirmation

Deliverables:
1. Documented empty-output fallback semantics
2. Visible diagnostics for the refiner fallback path
3. Clear distinction between failure and fallback cases
