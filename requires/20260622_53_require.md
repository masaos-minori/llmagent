# Title
Require sandboxed shell execution in production deployments

## Summary
Harden production shell execution by requiring an OS-level sandbox instead of allowing `shell_sandbox_backend="none"`.

## Background
Shell execution is currently fail-closed at the allowlist level, but once a command is allowed, it may still execute without OS-level isolation.

## Problem
This default is too permissive for production environments and increases blast radius if a command is allowed.

## Required Changes
- In production mode, require a sandbox backend such as `firejail`.
- Treat `shell_sandbox_backend="none"` as a startup failure or equivalent hard error in production.
- Ensure runtime health/status output clearly exposes the active sandbox backend.
- Update tool guidance to prefer structured `argv` execution over raw command strings.

## Acceptance Criteria
- Production deployments cannot use unsandboxed shell execution by default.
- Runtime diagnostics clearly show the active sandbox backend.
- Documentation reflects the hardened production behavior.

## Out of Scope
- Replacing Firejail with another sandbox technology
- Reworking shell allowlist semantics
