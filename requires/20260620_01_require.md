## Summary
Formalize the responsibility boundary between MDQ and RAG as an explicit architectural specification.

The intended model is:
- MDQ = Markdown-only, structure-aware retrieval and context compression
- RAG = multi-format, semantic retrieval, production search path
- Agent = MCP-mediated access path as the default integration model

## Background
MDQ is intended for Markdown-only content, especially when structural understanding matters (headings, sections, outlines, hierarchy).
RAG is intended for multi-format content and production-oriented retrieval, including semantic search and more general ingestion/search workflows.

Each system owns its own database, schema, and indexing logic.
They should not access each other’s database directly.

The agent layer should access both systems through MCP tool calls only.
Direct DB access is an admin-only maintenance exception, not a normal application path.

## Problem
If this boundary remains implicit, implementation and operational behavior will drift.
That drift will cause confusion in routing, ownership, maintenance, and future migration decisions.

## Required Changes
- Create a formal boundary specification for MDQ, RAG, and Agent responsibilities.
- Explicitly define MDQ as:
  - Markdown-only
  - structure-aware
  - low-to-moderate volume focused
  - experimental
- Explicitly define RAG as:
  - multi-format
  - semantic retrieval capable
  - production-oriented
- Explicitly define Agent integration rules:
  - MCP tool calls are the default and expected access path
  - direct sqlite access is not allowed in normal application code
  - `/db`-style direct maintenance access is an explicit admin-only exception

## Acceptance Criteria
- A single architectural document defines the MDQ / RAG / Agent boundary.
- The boundary is consistent across architecture docs, implementation notes, and operational guidance.
- The rules clearly distinguish normal MCP-mediated access from admin-only maintenance access.
- Future work can reference this document as the source of truth for ownership decisions.

## Out of Scope
- Rewriting MDQ internals
- Rewriting the RAG pipeline
- Making MDQ production-ready

## AI Implementation Instruction
Implement this as a boundary-definition and documentation-hardening task.

Strict rules:
- Do not redesign retrieval algorithms.
- Do not introduce new runtime access paths.
- Do not add direct sqlite access to normal application code.
- Preserve the intended split exactly:
  - MDQ = Markdown-only, structure-aware, experimental
  - RAG = multi-format, semantic retrieval, production path
  - Agent = MCP-mediated integration path
