# [Medium] Define `RagConfig` Protocol as the canonical RAG runtime config contract

## Priority
Medium

## Difficulty
Medium

## Background
`shared/types.py` defines `RagConfig` as a structural Protocol consumed by `RagPipeline`.

It allows both Agent config objects and MCP adapter objects to satisfy the same config contract without importing Agent-layer classes into the RAG layer.

## Goal
Make `RagConfig` the single canonical runtime configuration contract for the RAG pipeline.

## Scope
Affected files:

- `shared/types.py`
- `scripts/mcp/rag_pipeline/service.py`
- `scripts/mcp/rag_pipeline/models.py`
- `scripts/rag/pipeline.py`
- RAG documentation

## Implementation Tasks
- Document `RagConfig` as the canonical runtime config interface.
- Ensure `build_rag_cfg_adapter(cfg)` satisfies every field in `RagConfig`.
- Ensure `RagPipeline` only depends on the `RagConfig` Protocol, not concrete Agent config classes.
- Remove or mark non-runtime config DTOs as ingestion-only if duplicated.
- Add tests validating that the MCP config adapter satisfies the required fields.

## Acceptance Criteria
- `RagPipeline` can be constructed with any object satisfying `RagConfig`.
- RAG MCP config adapter passes a structural field coverage test.
- No RAG runtime code reads configuration from module globals.
- Documentation clearly distinguishes:
  - runtime config contract: `RagConfig`
  - MCP config file DTO
  - ingestion config DTOs

## Validation
- Add a test that creates a minimal config adapter and constructs `RagPipeline`.
- Add a test that fails if `build_rag_cfg_adapter()` misses a required `RagConfig` field.
