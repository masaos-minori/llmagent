# Task: Fix Lifecycle and Interrupt Handling for Workflow Pending Approval State

## Overview and Context
When a workflow enters a pending approval state, the `WorkflowPendingApprovalError` is raised, setting the volatile `ctx.workflow.approval_pending` flag and storing the task ID in `ctx.turn.pending_approval_id`. This state is transient and resides in memory. Currently, there is an absence of a strict interrupt guard to block arbitrary prompts, and if the system or REPL restarts while pending, the memory flags disappear while the row remains 'pending' in `workflow.sqlite`, leading to inconsistent state deadlocks.

## Acceptance Criteria (AC)
- [ ] In `Orchestrator.handle_turn()`, if `ctx.workflow.approval_pending` is True, reject any input that is not a valid `/approve` or `/reject` command.
- [ ] Return a clear message to the user: "[workflow] Approval required. Please use /approve [reason] or /reject [reason]."
- [ ] Implement a state rehydration routine during system startup inside `StartupOrchestrator.run()`.
- [ ] When the REPL boots up, scan `workflow.sqlite` for any record with `status='pending'` and automatically restore the corresponding memory context flags in `AgentContext`.

## Technical Specifications and Implementation Steps
1. Target Files: `agent/orchestrator.py`, `agent/startup.py`
2. Step 1 (Interrupt Guard): Add a conditional check at the entry point of `Orchestrator.handle_turn()` to evaluate `ctx.workflow.approval_pending` and short-circuit regular prompt parsing.
3. Step 2 (Database Scan): In `StartupOrchestrator`, query the `workflow.sqlite` database using `SQLiteHelper` to retrieve outstanding pending records.
4. Step 3 (Rehydration): Instantiate or populate the memory structures for `ctx.workflow` using the scanned `task_id`.

## Validation and Testing Plan
- Command: `pytest tests/agent/test_workflow_approval_lifecycle.py`
- Required Test Cases:
  - `test_arbitrary_prompt_blocked_during_pending`: Verifies regular inputs are rejected with the strict instruction string.
  - `test_startup_rehydration_restores_pending_state`: Mocks a pending record in SQLite, executes the startup routine, and verifies memory flags match the database record.

## Constraints and Core Rules
- Do not let unhandled non-command inputs mutate any other state when a workflow is pending approval.
- Ensure type definitions for state rehydration are fully verified with Mypy/Pyright.
