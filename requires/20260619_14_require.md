# Title
Change CI/CD workflow allowlist behavior to fail-closed in production

## Summary
Make workflow execution authorization fail-closed in production so that unlisted workflows are not implicitly allowed.

## Background
`repo_allowlist` is fail-closed, but `workflow_allowlist` is currently fail-open.
That means all workflows may be reachable unless explicitly restricted.

## Problem
Workflow execution is a sensitive action.
Allowing all workflows by default creates an unnecessary risk surface.

## Required Changes
- Change production behavior so that an empty `workflow_allowlist` does not imply allow-all.
- Decide whether this should be:
  - a hard fail at startup, or
  - a runtime deny-unless-listed policy
- Update docs and safety notes accordingly.
- Ensure the operator-facing startup audit highlights this setting clearly.

## Acceptance Criteria
- In production mode, unconfigured workflow execution is not implicitly allowed.
- Documentation no longer presents workflow allowlists as permissive by default in production.
- Runtime behavior is clear and auditable.

## Out of Scope
- Replacing CI/CD MCP architecture.
- Adding support for non-GitHub CI providers.

## Notes
Preserve explicit authorization boundaries for workflow-triggering operations.
