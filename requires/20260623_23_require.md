# Title
Upgrade `/health` from shallow to runtime-aware health reporting

## Summary
Make the health endpoint reflect DB readiness, schema readiness, and background-task status instead of returning a fixed `"ok"` only.

## Background
The current `/health` endpoint simply returns `{"status": "ok"}`.
It does not report DB availability, schema state, or whether the DLQ background task is alive.

## Problem
Shallow health makes it hard to distinguish “process is running” from “system is actually ready and healthy.”

## Required Changes
- Add a lightweight DB readiness check.
- Reflect relevant runtime state such as schema load success and `_dlq_task` status.
- Consider separating readiness vs liveness if appropriate.
- Strengthen smoke tests to verify real health semantics.

## Acceptance Criteria
- `/health` reports meaningful runtime status, not just a fixed response.
- Operators can detect startup or background-task problems early.

## Out of Scope
- Full monitoring system integration

## Components
- `scripts/eventbus/app.py`
- `scripts/eventbus/db.py`
- Health / operations documentation
- Smoke test / deployment validation

## AI Implementation Instruction
Implement this as an operability and health-signal improvement.

Strict rules:
- Do not turn `/health` into a heavy endpoint.
- Keep health checks lightweight but meaningful.
- If readiness/liveness split is not clearly needed, mark it as: Needs confirmation
