# [rag][query][http] Document _augment_http() fallback conditions clearly

## Summary
The HTTP fallback behavior is implemented, but the conditions are still not documented clearly enough.

## Problem
It is hard to know exactly when `_augment_http()` returns:
- `None`
- `""`
- a normal context string

This makes remote-RAG behavior harder to reason about.

## Scope
- `rag/pipeline.py`
- external RAG service behavior
- HTTP fallback documentation

## Task
- document the exact fallback contract for `_augment_http()`
- explain what causes:
  - `None`
  - empty string
  - normal successful result
- clarify how transport errors, 4xx, 5xx, and parse errors are handled

## Expected Result
Remote RAG behavior is easier to understand and operate safely.

## Acceptance Criteria
- [ ] `_augment_http()` return meanings are documented clearly
- [ ] fallback-trigger conditions are documented
- [ ] remote empty-result behavior is distinguished from failure
