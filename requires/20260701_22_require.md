## Issue 2: Remove blocking `time.sleep()` from SIGTERM handler

## Title
Remove blocking `time.sleep()` spin from SIGTERM handler to prevent asyncio event loop stall

## Summary
- The `_sigterm_handler` registered via `loop.add_signal_handler()` contains a busy-spin using `time.sleep(0.1)` that blocks the asyncio event loop for up to 10 seconds.
- During this time, no async tasks (including the in-progress turn) can make progress, defeating the purpose of graceful shutdown.
- The existing `asyncio.wait_for()` timeout in `_repl_loop()` already handles the graceful wait; the spin is redundant.

## Background
Signal handlers registered with `loop.add_signal_handler()` are invoked synchronously in the asyncio event loop thread. Any blocking call inside them stalls the loop. The intent was to wait for the current turn to complete before initiating shutdown, but the implementation blocks the loop that would allow the turn to complete.

## Problem
In `repl.py._sigterm_handler()`:
```python
if self._ctx.conv.is_processing:
    deadline = time.monotonic() + 10.0
    while self._ctx.conv.is_processing and time.monotonic() < deadline:
        time.sleep(0.1)  # BLOCKS THE EVENT LOOP
```
This blocks for up to 10 seconds. The current turn cannot complete because the event loop is stalled.

## Required Changes
- `scripts/agent/repl.py` — `run()._sigterm_handler()`: Remove the `while ... time.sleep(0.1)` spin entirely.
- `_sigterm_handler` should only: set `ctx.conv.shutdown_requested = True` and call `self._shutdown_event.set()`.
- The graceful completion wait is already handled by `_repl_loop()`'s `asyncio.wait_for(..., timeout=_GRACEFUL_TIMEOUT)` which detects `shutdown_requested=True`.

## Acceptance Criteria
- `_sigterm_handler` does not call `time.sleep()` or any other blocking operation.
- After SIGTERM, the in-progress turn is allowed to complete within `_GRACEFUL_TIMEOUT` (10 seconds).
- SIGTERM causes the REPL loop to exit cleanly via the `shutdown_requested` flag check.
- `_close_resources()` is still called at shutdown.

## Out of Scope
- Changes to the shutdown timeout value (`_GRACEFUL_TIMEOUT`).
- Changes to `_close_resources()` behavior.
- Windows-specific signal handling (`signal.signal()` fallback path) — apply the same change to that path.
