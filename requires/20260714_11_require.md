# [Medium] Split audit log documentation by agent-side, shared MCP, and per-server logs

## Labels
priority/medium, area/audit, area/observability, area/docs

## Summary
Clarify MCP and agent audit logging by separating agent-side audit logs, shared MCP server audit logs, and per-server audit logs.

## Background
Some documentation says every POST `/v1/call_tool` emits a JSON-lines audit record.

Other sections state that some servers write per-server audit logs in custom formats and that some servers do not currently have dedicated audit writer code.

These statements should be reconciled.

## Required Changes

### 1. Agent-side audit log
Document:

- path: `/opt/llm/logs/audit.log`
- format: JSON-lines
- event examples: `tool_exec`
- generated by agent-side execution/audit layer

### 2. Shared MCP server audit log
Document:

- path: `/opt/llm/logs/audit.log`
- format: JSON-lines
- event examples: `mcp_tool_exec`
- only servers that actually use shared MCP audit write these records

### 3. Per-server audit logs
Document server-specific audit logs separately.

Examples may include:

- GitHub audit log
- shell audit log
- file-delete audit log
- MDQ audit log

Clarify that format may be server-specific and may not include all cross-log correlation fields.

### 4. Correlation limitations
Document which logs support:

- `X-Session-Id`
- `X-Request-Id`
- `request_id`
- `mcp_request_id`

Clarify when correlation with MCP server access logs is not possible.

### 5. git-mcp audit status
If git-mcp has an audit config key but no writer implementation, document it as:

- reserved
- not implemented
- future work

## Acceptance Criteria
- Documentation no longer says every server always writes shared JSON-lines audit records unless true.
- Agent-side audit and MCP server-side audit are clearly separated.
- Per-server audit formats are documented separately.
- Cross-log correlation limitations are clear.
- git-mcp audit status is documented accurately.
- Reading audit logs guide matches implementation.
