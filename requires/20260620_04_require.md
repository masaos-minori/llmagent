## Dependencies
- Formalize the MDQ / RAG responsibility boundary

## Components
- mdq-mcp tool metadata
- rag-pipeline-mcp tool metadata
- server catalog
- usage / operator guidance

## Summary
Make the MDQ / RAG responsibility split visible in MCP tool descriptions, server descriptions, and supporting docs.

## Background
MDQ and RAG serve different purposes:
- MDQ is for Markdown-only, structure-aware retrieval
- RAG is for multi-format, semantic, production-oriented retrieval

This difference must be visible at the point of tool selection, not only in deep architecture documentation.

## Problem
If tool descriptions are vague, users and LLMs may select the wrong path:
- Markdown structural queries may be sent to RAG unnecessarily
- production-oriented general retrieval may be sent to MDQ incorrectly

## Required Changes
- Update MDQ tool descriptions to explicitly state:
  - Markdown-only
  - structure-aware retrieval
  - experimental status
- Update RAG tool descriptions to explicitly state:
  - multi-format retrieval
  - semantic / embedding-based retrieval
  - production-ready role
- Align related docs so the same distinction appears consistently in:
  - tool metadata
  - server docs
  - usage guidance

## Acceptance Criteria
- A user or LLM can tell when to use MDQ vs RAG by reading the available tool descriptions.
- Tool metadata and documentation use consistent wording.
- MDQ is clearly described as Markdown-structural and experimental.
- RAG is clearly described as the general-purpose production retrieval path.

## Out of Scope
- Renaming tools
- Changing retrieval algorithms

## AI Implementation Instruction
Implement this as a metadata-and-documentation clarity task.

Strict rules:
- Do not rename tools unless explicitly required elsewhere.
- Do not change search behavior.
- Make the role split visible exactly where tools are selected.
- Keep wording short, precise, and operationally useful.

Required emphasis:
- MDQ:
  - Markdown-only
  - structure-aware retrieval
  - experimental
- RAG:
  - multi-format
  - semantic retrieval
  - production path

Also:
- Update all AI-facing and user-facing descriptions consistently.
- Do not leave conflicting wording in older docs if they are still part of the active doc set.
- If any wording would overstate readiness or support level, tone it down and mark the ambiguity as: Needs confirmation

Deliverables:
1. Updated MCP tool descriptions
2. Updated related documentation
3. Consistent wording across metadata and docs
