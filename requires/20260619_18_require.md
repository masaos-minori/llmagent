# Title
Unify mdq-mcp documentation as functional but experimental

## Summary
Update all mdq-mcp documentation so that it consistently describes the server as functionally implemented but experimental, rather than as a non-functional stub.

## Background
Some docs still imply that mdq-mcp tools return only stub behavior, while the known-issues section clarifies that FTS5 indexing and search are actually implemented.

## Problem
This inconsistency can cause AI systems and operators to misunderstand whether mdq-mcp is usable at all.

## Required Changes
- Update all mdq-mcp references to a consistent message:
  - functionally implemented
  - experimental / not production-validated
- Remove or rewrite outdated "stub only" wording.
- Align the document guide, security notes, and server catalog.

## Acceptance Criteria
- No active documentation describes mdq-mcp as non-functional if the implementation is functional.
- All docs consistently distinguish functional capability from production readiness.
- AI-facing notes no longer mislead tool selection.

## Out of Scope
- Making mdq-mcp production-ready.
- Replacing mdq-mcp with rag-pipeline-mcp.

## Notes
Keep the nuance: usable for experimentation, not recommended as the production search path.
