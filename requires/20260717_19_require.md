# Document MCP runtime availability metadata and disabled tool behavior

## Priority

Medium

## Target files

- docs/mcp_tool_metadata.md
- docs/mcp_server_authoring.md
- docs/runtime_tool_registry.md
- MCP server README files
- docs or README sections that mention requires_config

## Reason

MCP server authors need clear guidance for the new metadata contract.

`requires_config` is removed with no compatibility period. Tool availability is now expressed through `config_dependent`, `enabled`, and `disabled_reason`.

The default `/v1/tools` behavior is to return all tools, including disabled tools.

## Intent

Document the new MCP tool metadata model and its relationship with RuntimeToolRegistry.

## Scope

Document:

- removal of `requires_config`
- meaning of `config_dependent`
- meaning of `enabled`
- meaning of `disabled_reason`
- standard `disabled_reason` values
- rule that `/v1/tools` always returns all implemented tools
- rule that disabled tools are not omitted
- rule that disabled tools are not LLM-visible
- rule that disabled tools must be rejected before dispatch
- RuntimeToolRegistry handling expectations

## Acceptance criteria

- Documentation no longer describes `requires_config` as active metadata.
- Documentation includes example `/v1/tools` response with enabled and disabled tools.
- Documentation explains how MCP servers should compute `enabled`.
- Documentation explains how RuntimeToolRegistry should treat disabled tools.
- MCP server README files are consistent with the new behavior.
