# Title
Clarify the behavior and quality implications of `use_rrf=False`

## Summary
Make it explicit that `use_rrf=False` means a simplified dedup-based fallback path rather than full ranking fusion.

## Background
The docs describe `use_rrf=False` as switching from RRF merge to a simpler dedup path, but the practical retrieval-quality implications are not especially visible.

## Problem
Operators and AI systems may misinterpret the setting as a minor optimization switch instead of a meaningful quality tradeoff.

## Required Changes
- Clarify in docs and diagnostics what `use_rrf=False` actually does.
- Explain the quality implications compared with full RRF.
- Surface the active fusion mode in debug or status output if appropriate.

## Acceptance Criteria
- The meaning of `use_rrf=False` is obvious in documentation.
- Operators can distinguish RRF mode from simplified dedup mode.
- The retrieval-quality tradeoff is explained.

## Out of Scope
- Replacing fusion logic
- Designing new ranking algorithms

## AI Implementation Instruction
Implement this as a clarity and diagnostics task.

Strict rules:
- Do not redesign fusion behavior.
- Focus on accurate explanation and observable mode reporting.
- If intended user-facing wording is still unclear, mark it as: Needs confirmation

Deliverables:
1. Clear documentation of `use_rrf=False`
2. Optional mode visibility in runtime/debug output
3. Explanation of quality tradeoffs
