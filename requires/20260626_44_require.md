# Title
Strengthen RAG quality regression tests with meaningful assertions

## Priority
High

## Labels

## Dependencies
- None

## Summary
Replace weak RAG regression assertions with meaningful checks that detect ranking, fallback, semantic cache, and diagnostics regressions.

## Background
Current RAG quality tests include assertions such as `len(result.reranked) >= 0`, which always pass and do not validate retrieval quality.

## Problem
The existing tests behave more like smoke tests than regression tests. They cannot detect ranking degradation, broken RRF behavior, semantic-cache regressions, or missing fallback diagnostics.

## Required Changes
- Replace always-true assertions with meaningful expectations.
- For known queries, assert expected `chunk_id`, URL, title, or ranking properties.
- Add tests that distinguish `use_rrf=True` from `use_rrf=False`.
- Verify semantic cache hit behavior through diagnostics or cache state.
- Verify embed failure fallback diagnostics, not only empty results.

## Acceptance Criteria
- Tests fail when RAG ranking or fallback behavior regresses.
- RRF and no-RRF behavior are distinguishable in tests.
- Semantic cache and embed-failure fallback behavior are explicitly validated.

## Out of Scope
- Building a large benchmark suite.
- Replacing unit tests with full evaluation metrics.

## AI Implementation Instruction
Implement this as a regression-test hardening task.

Strict rules:
- Do not use assertions that are always true.
- Use deterministic fixtures and expected outcomes.
- Validate diagnostics where fallback behavior is expected.
- If expected ranking cannot be determined from current fixtures, mark it as: `Needs confirmation`.
