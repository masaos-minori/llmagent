# Clarify timestamp policy for newly recreated DBs only

## Priority

Medium

## Summary

Clarify timestamp format policy under the no-migration approach.

## Background

The schema currently mixes:

- SQLite `datetime('now')`
- SQLite `strftime('%Y-%m-%dT%H:%M:%SZ', 'now')`
- Python-side `datetime.now(UTC).isoformat()`

Since existing DB migration is not supported, timestamp normalization should apply only to newly recreated DBs unless a separate manual migration is performed.

## Goal

Document timestamp policy without implying automatic conversion of existing records.

## Target files

- `90_shared_04_db_architecture_and_schema.md`
- `schema_sql.py`
- `state_store.py`

## Scope

### In scope

- New DB timestamp policy.
- Documentation of existing mixed behavior.
- No-migration clarification.

### Out of scope

- Automatic timestamp conversion.
- Existing DB migration.

## Implementation tasks

- Decide the canonical timestamp format for newly recreated DBs.
- Document that existing DB timestamps are not converted automatically.
- If changing DDL defaults, update `schema_sql.py`.
- If changing workflow timestamp generation, update `state_store.py`.

## Acceptance criteria

- Docs clearly distinguish:
  - new DB timestamp format
  - existing DB values
  - no automatic migration
- No documentation implies existing timestamps are rewritten.

## Validation

- Review all timestamp defaults in `schema_sql.py`.
- Review `_now()` in `state_store.py`.

## Confirmation required

- Confirm preferred canonical timestamp format for new DBs:
  - `YYYY-MM-DDTHH:MM:SSZ`
  - Python ISO format with `+00:00`
