# [Medium] Keep stdio transport but separate it from routing fallback

## Priority
Medium

## Background
stdio transport is intentionally retained for local testing, CI, embedded subprocesses, and low-concurrency workloads. It should not be described or treated as a fallback routing mechanism.

## Goal
Keep stdio transport as an explicit transport option while separating it from routing compatibility cleanup.

## Scope
Affected files:
- `shared/tool_executor.py`
- `shared/mcp_config.py`
- `mcp/server.py`
- `agent/factory.py`
- `docs/04_mcp_01_system_overview.md`
- `docs/04_mcp_02_protocol_and_transport.md`
- `docs/04_mcp_03_routing_lifecycle_and_execution.md`

## Implementation Tasks
- Keep `StdioTransport`.
- Keep `TransportType.STDIO`.
- Keep `StartupMode.ONDEMAND`.
- Keep `HealthcheckMode.PING_TOOL`.
- Keep stdio lifecycle support in the agent factory/lifecycle layer.
- Update docs to describe stdio as:
  - local testing transport
  - CI transport
  - embedded subprocess transport
  - low-concurrency explicit transport
- Remove any wording that implies stdio is a routing fallback or backward compatibility layer.

## Acceptance Criteria
- stdio functionality remains available.
- stdio is documented as an explicit transport, not a fallback.
- Routing docs no longer mix transport selection with route resolution.
- HTTP remains documented as the production default.

## Validation
- Run existing stdio transport tests if available.
- Run MCP config tests for stdio modes.
- Run `rg -n "stdio.*fallback|fallback.*stdio|compat.*stdio" docs shared mcp scripts`.
