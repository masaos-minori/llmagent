# Title
Document and validate the event ack endpoint as a first-class API

## Priority
High

## Dependencies
- Issue 2

## Summary
Promote `POST /events/{event_id}/ack?consumer_id=...` to a fully documented and tested first-class API.

## Background
The offset model depends on explicit acknowledgements, but the HTTP API and reference docs do not fully document the ack endpoint behavior.

## Problem
Consumer recovery cannot be implemented reliably unless the ack API contract is explicit. Missing documentation also makes offset semantics hard to test and integrate.

## Required Changes
- Define the ack endpoint:
  - path
  - query parameters
  - response body
  - error cases
- Ack must update the consumer offset to the event's `seq`.
- Return a clear error when:
  - `consumer_id` is missing
  - `event_id` does not exist
  - `consumer_id` is invalid
- Document whether acking an older event may move offsets backwards. Prefer monotonic offset advancement.
- Add ack endpoint tests.

## Acceptance Criteria
- `POST /events/{event_id}/ack?consumer_id=...` is documented in HTTP API and Reference API docs.
- Ack writes the expected seq to the offset file.
- Missing consumer_id is rejected.
- Unknown event_id behavior is documented and tested.
- Offset advancement is monotonic unless explicitly confirmed otherwise.

## Out of Scope
- Batch ack.
- Negative ack / nack.
- Exactly-once delivery guarantees.
