# Standardize mdq-mcp wording: distinguish "experimental" from "non-functional stub"

## Summary
Unify the wording around mdq-mcp so that documents clearly distinguish an experimental implementation from a non-functional placeholder.

## Background
Different MCP documents use different language around mdq-mcp, including "experimental", "stub", and wording that can be interpreted as "not implemented". This can mislead readers about actual usability.

## Objective
Use one consistent terminology model for mdq-mcp status.

## In Scope
- `04_mcp_00_document-guide.md`
- `04_mcp_04_server_catalog.md`
- `04_mcp_07_mdq_rag_boundary.md`
- `04_mcp_90_inconsistencies_and_known_issues.md`

## Required Changes
- Define approved status terms such as:
  - `experimental but functional`
  - `not production-validated`
  - `placeholder / missing implementation`
- Apply the same terms consistently across all mdq-related sections
- Ensure that `stub: true` is explained consistently everywhere
- Remove wording that implies mdq-mcp is unusable if the documented intent is only "not production-ready"

## Acceptance Criteria
- All mdq-mcp references use consistent terminology
- `stub` is not used in a way that contradicts "functional but experimental"
- The document guide no longer sends readers to a misleading conclusion

## Out of Scope
- Changing mdq-mcp implementation status
- Rewriting MDQ architecture

## Notes
If some mdq features are truly missing while others are functional, describe that split explicitly instead of using a single overloaded word.
