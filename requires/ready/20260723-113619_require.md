# Requirement: Create MCP Tool Ownership Matrix, Service Boundaries, and Capability Mapping Documentation

## Priority

Medium

## Target files

- `docs/04_mcp_01_tool_ownership_matrix.md` (new)
- `docs/04_mcp_02_service_boundaries.md` (new)
- MCP document guide or index (update existing)

## Background

The current MCP documentation describes servers and runtime tool discovery but lacks a single canonical view of which tool belongs to which MCP server. This creates maintenance risks: developers cannot quickly identify tool ownership, responsibility boundaries erode over time, workflow/approval behavior cannot be traced back to specific tools, and new tools have unclear placement guidance.

The most important unclear areas are:
- Local Git operations versus GitHub API operations
- GitHub repository operations versus CI/CD operations
- File read, write, and delete responsibilities
- RAG versus MDQ responsibilities
- Tool risk classification and approval requirements

## Problem

Without a consolidated ownership map, developers face several issues:
1. Cannot quickly identify the owning MCP server for a given tool
2. Tool ownership becomes ambiguous as new tools are added
3. Responsibility boundaries between similar MCP servers erode over time
4. Workflow and approval behavior cannot be traced back to specific MCP tools
5. Runtime discovery explains how tools are discovered, but not who owns each tool
6. No clear guidance on where a new tool should be added

## Reason for change

Making MCP ownership explicit and reviewable allows maintainers to answer key questions from documentation alone: which MCP server owns a given tool, which capability depends on a given MCP server, which workflow stage normally invokes the tool, which tools require approval, which operations are outside a given MCP server's responsibility, and where a new tool should be added.

## Implementation intent

Create two new documentation files plus update the MCP document guide/index:
1. `04_mcp_01_tool_ownership_matrix.md` — canonical tool-to-MCP-server mapping table
2. `04_mcp_02_service_boundaries.md` — per-server responsibility/non-responsibility definitions
3. Update MCP document guide/index to link to the new documents

This is a documentation-only task — no runtime behavior changes.

## Implementation instructions

### 1. Tool Ownership Matrix (`04_mcp_01_tool_ownership_matrix.md`)

Create a markdown table with columns: Tool Name, Owning MCP Server, Capability Group, Risk Tier, Approval Required, Typical Workflow Stage, Notes.

Use the following authoritative source for tool names — `scripts/shared/tool_constants.py`:

**File Management:**
- `list_directory`, `list_directory_with_sizes`, `directory_tree`, `read_text_file`, `read_media_file`, `read_multiple_files`, `search_files`, `grep_files`, `get_file_info` → `file-read-mcp`
- `write_file`, `edit_file`, `create_directory`, `move_file` → `file-write-mcp`
- `delete_file`, `delete_directory` → `file-delete-mcp`

**Repository Operations:**
- `git_status`, `git_log`, `git_diff`, `git_branch`, `git_show` → `git-mcp` (READ)
- `git_add`, `git_commit`, `git_checkout`, `git_pull`, `git_push` → `git-mcp` (WRITE)

**GitHub Operations:**
- `github_search_repositories`, `github_list_branches`, `github_list_commits`, `github_get_commit`, `github_search_code`, `github_get_file_contents`, `github_list_issues`, `github_get_issue`, `github_search_issues`, `github_list_pull_requests`, `github_get_pull_request`, `github_search_pull_requests` → `github-mcp` (READ)
- `github_create_branch`, `github_create_or_update_file`, `github_push_files`, `github_create_issue`, `github_add_issue_comment`, `github_create_pull_request`, `github_update_pull_request` → `github-mcp` (WRITE)
- `github_delete_file`, `github_merge_pull_request` → `github-mcp` (DANGEROUS)

**CI/CD Operations:**
- `trigger_workflow` → `cicd-mcp` (WRITE)
- `get_workflow_runs`, `get_workflow_status`, `get_workflow_logs` → `cicd-mcp` (READ)

**Internet Search:**
- `search_web`, `browser_fetch` → `web-search-mcp`

**RAG Retrieval:**
- `rag_run_pipeline`, `rag_debug_pipeline`, `rag_list_documents` → `rag-pipeline-mcp` (READ)
- `rag_delete_document` → `rag-pipeline-mcp` (WRITE)

**Markdown Query:**
- `search_docs`, `get_chunk`, `outline`, `index_paths`, `refresh_index`, `stats`, `grep_docs` → `mdq-mcp`

**Shell Execution:**
- `shell_run` → `shell-mcp`

Risk tiers should follow existing classifications: READ_TOOLS → LOW, WRITE_TOOLS → MEDIUM, DELETE_TOOLS → HIGH, GITHUB_DANGEROUS_TOOLS → HIGH, CICD_WRITE_TOOLS → HIGH.

Approval requirements should follow existing `cfg.approval.tool_safety_tiers` logic — high-risk tools require approval, low-risk tools do not.

Typical workflow stages: plan → READ_TOOLS, execute → WRITE_TOOLS, verify → READ_TOOLS.

### 2. Service Boundary Definitions (`04_mcp_02_service_boundaries.md`)

For each MCP server, document: Responsibilities, Explicit non-responsibilities, Allowed operation types, Forbidden operation types, Ownership rationale.

At minimum define boundaries for all 10 servers listed above.

Key boundary rules:
- Local repository operations belong to `git-mcp`.
- GitHub remote API operations belong to `github-mcp`.
- GitHub Actions operations belong to `cicd-mcp`.
- File reading, writing, and deletion remain separate services.
- RAG handles semantic retrieval only.
- MDQ handles Markdown-structured querying only.

### 3. Capability Dependency Mapping

Create a mapping linking: Capability, Required MCP server, Optional MCP server, Degraded behavior, Unavailable behavior.

Example capabilities: Repository Operations, GitHub Operations, CI/CD Operations, File Management, Internet Search, RAG Retrieval, Markdown Query, Shell Execution.

### 4. Mermaid Diagrams

Add three Mermaid diagrams using GitHub-compatible syntax:
- Tool to MCP ownership
- Capability to MCP dependency
- Workflow stage to tool to MCP relationship

### 5. Document Guide/Index Update

Update the MCP document guide or index to link to the new documents.
