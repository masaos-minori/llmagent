# Requirement: Improve Fail-Open Visibility and Runtime Tool Availability Handling

## Priority

Medium

## Target files

Implementation-dependent — likely affected areas:
- Startup orchestration module
- MCP health tracking module
- Runtime tool discovery module
- Runtime tool registry construction module
- CLI status command handler
- Operations documentation (`docs/*.md`)

Exact file names must be verified before making changes.

## Background

In local mode, the system may continue startup even when one or more MCP servers fail. This fail-open behavior is useful for development, allowing partial functionality to operate while missing dependencies are addressed. However, the current implementation does not clearly communicate this degraded state to users or operators.

The LLM receives a list of available tools at planning time based on what was discovered during startup. If MCP servers fail after tool discovery but before execution, the LLM may attempt to use tools that are no longer available. Similarly, if MCP servers fail during startup, their tools may still appear in the planning context even though they cannot be executed.

Production mode already uses stricter fail-fast behavior, so this gap only affects local mode.

## Problem

When MCP servers fail to start in local mode:

- Users do not receive clear indication that some capabilities are unavailable.
- The LLM may attempt to use tools from failed MCP servers, causing repeated execution failures.
- Tool loop guards mitigate repeated failures but do not solve the underlying availability mismatch.
- Workflow tasks may fail after approval even though the missing dependency was known at startup.
- No post-startup way to query which capabilities are degraded or unavailable.

## Reason for change

Make fail-open state explicit, visible, and operationally safe without removing the fail-open behavior itself. Users should clearly see which capabilities are unavailable, and the system should prevent the LLM from selecting tools that are known to be unavailable.

## Implementation intent

1. Print a concise startup readiness report summarizing MCP health and capability status.
2. Expose the same readiness information through a CLI command for persistent visibility.
3. Log readiness information during startup for operator observability.
4. Prevent tools from unavailable MCP servers from being registered in the runtime tool registry.
5. Exclude unavailable tools from the LLM planning context.
6. Update operations documentation to explain fail-open behavior and its implications.

## Implementation instructions

### 1. Startup Readiness Report

Create a centralized readiness reporting function that aggregates MCP health status and capability-level summaries. At startup, call this function and print the result.

The report must include:
- MCP healthy count
- MCP degraded count
- MCP unavailable count
- Workflow readiness
- RAG readiness if available
- Memory readiness if available
- Capability-level summary where possible

Example output shape:

```
System Readiness Report

MCP:
  Healthy: 8
  Degraded: 1
  Unavailable: 2

Capabilities:
  Repository Operations: unavailable
  File Management: healthy
  RAG Retrieval: healthy
  Memory Services: degraded
```

Centralize the readiness logic — do not duplicate it across multiple places.

### 2. Persistent Visibility

Expose the same readiness information through a CLI command. Use the existing command structure where appropriate. Suggested command options: `/mcp status`, `/health`, `/status`.

Implement using the existing CLI command pattern rather than creating a new one.

### 3. Logging

Write readiness information to logs during startup. The log must clearly state:
- Which MCP servers are unavailable
- Which capabilities are affected
- Whether the process is continuing because of local fail-open behavior

Use structured logging (e.g., `structlog`) consistent with the project's logging conventions.

### 4. Runtime Tool Registration Rule

Runtime tool registry construction must follow this rule:
- Healthy MCP server: register discovered tools.
- Degraded MCP server: exclude tools unless explicit policy allows inclusion (conservative default).
- Unavailable MCP server: do not register its tools.

If degraded-state semantics are unclear, implement the conservative behavior: exclude degraded MCP server tools unless explicit policy allows inclusion. Document any unknowns as `Unknown` and implement only clearly safe behavior.

### 5. Planning Context Safety

Tools that cannot be executed must not be advertised to the LLM as available tools. This prevents the model from selecting tools that are known to be unavailable. Ensure the planning context builder filters out tools from unavailable MCP servers.

### 6. Documentation

Update documentation to explain:
- Why local mode can continue after MCP startup failure
- How fail-open state is displayed
- How unavailable MCP tools are handled
- Difference between local and production behavior
- Operator action when required capabilities are unavailable

Do not modify documentation files during this document-only phase — mark them as out-of-scope for this step and note them for the implementation phase.

### General constraints

- Preserve production fail-fast semantics — do not change production behavior.
- Prefer conservative runtime safety: do not expose unavailable tools.
- Avoid duplicating readiness logic in multiple places; centralize if practical.
- If exact degraded-state semantics are unclear, document them as `Unknown` and implement only clearly safe behavior.

## Acceptance criteria

- [ ] Startup output clearly reports degraded or unavailable MCP services.
- [ ] Local fail-open continuation is explicitly visible to the user.
- [ ] Readiness information is available after startup through a CLI command.
- [ ] Readiness information is written to logs.
- [ ] Tools from unavailable MCP servers are not registered in the runtime tool registry.
- [ ] Unavailable tools are not advertised to the LLM planning context.
- [ ] Production fail-fast behavior remains intact.
- [ ] Documentation is updated.

## Tests

- Unit test: Verify that tools from an unavailable MCP server are excluded from the runtime tool registry.
- Unit test: Verify that tools from a degraded MCP server are excluded by default (conservative behavior).
- Unit test: Verify that tools from a healthy MCP server are included in the runtime tool registry.
- Unit test: Verify that the startup readiness report generates correct counts for each MCP health category.
- Integration test: Verify that the CLI status command returns the same readiness information as the startup report.
- Integration test: Verify that the LLM planning context does not include tools from unavailable MCP servers.
- Manual test: Simulate MCP server startup failure and verify that the readiness report, logs, and CLI command all reflect the degraded state accurately.
