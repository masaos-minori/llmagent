# MCP DAG Scheduling: Clarify That Metadata Currently Comes From Configured Tool Definitions

## Priority

Medium

## Target Files

- `docs/04_mcp_03_01_dispatch-and-routing.md` — clarify DAG scheduling data source
- `docs/04_mcp_06_14_new-tool-registration-procedure.md` — add metadata update paths

## Background

The documentation may imply that RuntimeToolRegistry metadata from /v1/tools directly drives DAG scheduling. Current implementation does not do that.

## Problem

Without clarification, implementers may assume that live MCP metadata automatically controls scheduling. This can lead to new tools being incorrectly parallelized or unnecessarily serialized.

## Reason for Change

This prevents implementers from assuming that live MCP metadata automatically controls scheduling. Without this clarification, new tools may be incorrectly parallelized or unnecessarily serialized.

## Implementation Intent

Add a clear explanation separating RuntimeToolRegistry's role (routing + LLM visibility) from DAG scheduling's data source (configured LLM tool definitions).

## Implementation Instructions

### Step 1: Update dispatch-and-routing.md

In the DAGスケジューリング section, add:

```markdown
### Data source for DAG scheduling

DAG scheduling currently uses metadata built from `ctx.cfg.tool.tool_definitions`, NOT from RuntimeToolRegistry.

The scheduler reads or derives:
- `function.requires_serial`
- `function.resource_scope`
- `function.is_write`
- side-effect status from `is_side_effect()`
- shell-specific serial behavior

RuntimeToolRegistry controls runtime routing and LLM visibility, but it is not currently the direct input to DAG scheduling.
```

### Step 2: Add clarification about /v1/tools impact

Add:

```markdown
Important: updating `/v1/tools.requires_serial` alone does not necessarily change DAG scheduling. Changes to scheduling behavior require updates in both `/v1/tools` metadata AND `config/agent.toml` tool definition metadata.
```

### Step 3: Update new-tool-registration-procedure.md

In the new tool registration section, add:

```markdown
### Metadata update paths

When registering a new tool, you may need to update metadata in two places:

1. `/v1/tools` response — for routing and LLM visibility
2. `config/agent.toml` tool definitions — for DAG scheduling

These are separate concerns and must be updated independently.
```

## Acceptance Criteria

1. Documentation clearly separates: runtime routing metadata, LLM-visible tool schema, DAG scheduling metadata
2. New tool registration docs include all required metadata update paths
3. No section implies that RuntimeTool.requires_serial alone controls scheduling today
