# Requirement: Clean Up _http_procs After Health Check Timeout

## Priority

Medium

## Target files

- `scripts/agent/http_lifecycle.py`

## Background

In `HttpServerLifecycleManager.start()`, when a health check times out, the subprocess is terminated via `_terminate_with_timeout()` and then `HttpStartupError` is raised. However, the `_http_procs[server_key]` entry is never removed from the internal mapping after a timeout. The `_terminate_with_timeout()` method terminates and kills the process but does not clean up the `_http_procs` dictionary.

Note: The original issue described a `finally` block that removes the entry, which does not exist in the current codebase. The actual problem is worse — the entry is NEVER removed after a health check timeout, leaving a stale reference to a dead process object.

## Problem

When an MCP subprocess fails its health check:
1. `_http_procs[server_key]` still contains the dead process object
2. Subsequent `verify_running()` checks may incorrectly report the server as running (if `proc.poll()` returns None due to race conditions)
3. If the server is retried later, `start()` will overwrite the entry, but the old process may still be running under the hood
4. Memory leak: the dead process object accumulates in the dictionary over time

## Reason for change

Stale process references in `_http_procs` can cause incorrect state reporting and resource leaks. The entry must be unconditionally removed regardless of whether the process was successfully killed.

## Implementation intent

After `_terminate_with_timeout()` completes in the health check timeout path, explicitly remove the `_http_procs[server_key]` entry. Also add cleanup to other paths where `_terminate_with_timeout()` is called without subsequent dictionary cleanup.

## Implementation instructions

1. In `HttpServerLifecycleManager.start()`, after line 300 (`await self._terminate_with_timeout(proc, server_key)`), add:
   ```python
   self._http_procs.pop(server_key, None)
   ```
2. Review all call sites of `_terminate_with_timeout()` to ensure they clean up `_http_procs`:
   - Line 300 (health check timeout) — needs fix above
   - Line 325 (early exit path) — already has cleanup at line 322
   - Line 375 (shutdown_all) — already has cleanup at line 368
3. Consider extracting the cleanup logic into a helper method `_cleanup_server_entry(server_key)` to avoid duplication.

## Acceptance criteria

- [ ] When a health check times out, `_http_procs[server_key]` is removed from the dictionary
- [ ] No stale process objects remain in `_http_procs` after any failure path
- [ ] No regression in normal operation when health checks succeed
- [ ] No regression when subprocess exits naturally before health check completes

## Tests

- Unit test: Mock health check to always fail with timeout, verify `_http_procs` is empty after start() raises
- Integration test: Configure an MCP server with a very short health check timeout, start agent, verify no orphaned entries in `_http_procs`
