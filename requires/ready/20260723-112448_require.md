# Requirement: Narrow Broad Exception Handlers in RAG Validation to Prevent Silent Failures

## Priority

High

## Target files

- `scripts/agent/startup.py`

## Background

In `StartupOrchestrator._validate_rag_consistency()`, there is a broad `except Exception` clause that catches ALL exceptions during RAG consistency validation. The original issue references `get_embedding_dimension()` and `model_embedding_dim` methods which do not currently exist in `rag_maintenance_service.py`. This suggests either the feature was not yet implemented or the issue description predates the current implementation. However, the underlying concern â€” a broad exception handler swallowing errors and allowing startup to continue â€” remains valid for any future RAG validation logic.

## Problem

A broad `except Exception` handler swallows all exceptions regardless of type, including:
1. `sqlite3.DatabaseError` â€” corrupted RAG database
2. `ConnectionRefusedError` â€” embedding service unreachable
3. `ValueError` â€” invalid dimension value returned from the database
4. `RuntimeError` â€” schema missing in RAG database

When any of these occur, the operator sees only "Failed to validate RAG consistency: <error>" in logs, with no indication of what went wrong or how to fix it. The startup proceeds normally, and subsequent RAG queries may return meaningless results.

## Reason for change

Broad exception handlers hide critical failures. RAG consistency must be validated with specific error handling per exception type, ensuring operators receive actionable information when RAG fails.

## Implementation intent

Narrow the exception handling to catch specific exception types and handle each appropriately: fatal for dimension mismatch, non-fatal warning for database accessibility issues, and logging for other unexpected errors.

## Implementation instructions

1. Replace the broad `except Exception` with specific handlers:
   - `sqlite3.DatabaseError` / `ConnectionRefusedError`: log CRITICAL, display NON_FATAL warning, disable RAG for this session (`ctx.services_required.rag_maintenance = None`)
   - `ValueError`: log ERROR, display NON_FATAL warning about invalid dimension value
   - `RuntimeError` (dimension mismatch): log ERROR, display FATAL message, halt startup
   - `Exception` (catch-all): log WARNING, allow startup to continue for truly minor issues
2. If `get_embedding_dimension()` and `model_embedding_dim` methods do not exist on `rag_maintenance`, implement them first before adding the dimension comparison logic. These would need to read the vector dimension from the RAG database schema and compare it against the configured model's embedding dimension.
3. Ensure the dimension mismatch check raises an exception (not just logs) so the startup sequence halts.

## Acceptance criteria

- [ ] Dimension mismatch produces a FATAL error and halts startup
- [ ] Database accessibility failures produce a NON_FATAL warning and disable RAG for the session
- [ ] Invalid dimension values produce a NON_FATAL warning
- [ ] Minor/unexpected issues produce a WARNING-level log entry without halting startup
- [ ] No regression in existing RAG functionality
