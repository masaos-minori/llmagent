# Shared / DB Documentation Organization

## Priority

Medium

## Target Files

- `docs/90_shared_00_document-guide.md` — update guide if needed
- `docs/90_shared_01_*` — review and update
- `docs/90_shared_02_*` — consolidate types/DTOs here
- `docs/90_shared_03_*` — consolidate shared runtime (ToolExecutor/ToolRegistry/RuntimeToolRegistry) here
- `docs/90_shared_04_*` — consolidate DB schema here
- `docs/90_shared_05_*` — consolidate DB operations here
- `docs/90_shared_90_inconsistencies_and_known_issues.md` — keep only unresolved issues

## Background

Shared/DB documentation needs organization to ensure consistency between implementation and docs, eliminate duplicates, and remove references to deleted features.

## Problem

Multiple inconsistencies exist between current implementation and documentation:
- shared/ vs db/ responsibility descriptions may not match current implementation
- import direction/importlinter constraint descriptions may not match implementation/config
- LLMMessage/RuntimeTool/RuntimeToolRegistry/ToolSpec/ToolCallResult field descriptions may not match current code
- ToolExecutor/ToolRegistry/RuntimeToolRegistry responsibility descriptions may conflict with MCP-side docs
- ToolResultCache and unused components may have excessive descriptions
- Descriptions of deleted legacy sources or old config files may remain unnecessarily
- DB schema/helper/maintenance/recovery descriptions may not match current implementation
- Duplicate explanations between Shared side and Agent/MCP side

## Reason for Change

Ensure documentation accurately reflects current implementation, eliminates duplicates, and removes references to deleted features.

## Implementation Intent

Organize Shared/DB documentation by consolidating related content into appropriate files and removing outdated information.

## Implementation Instructions

### Step 1: Review shared/db responsibilities

Verify shared/ vs db/ responsibility descriptions against current implementation. Verify import direction/importlinter constraints against implementation and config.

### Step 2: Review type/DTO field descriptions

Verify LLMMessage/RuntimeTool/RuntimeToolRegistry/ToolSpec/ToolCallResult field descriptions against current code. Consolidate into `90_shared_02_*`.

### Step 3: Review shared runtime descriptions

Verify ToolExecutor/ToolRegistry/RuntimeToolRegistry responsibility descriptions against MCP-side docs. Remove any conflicts. Consolidate into `90_shared_03_*`.

### Step 4: Reduce excessive component descriptions

Reduce descriptions of unused components like ToolResultCache that are not used in production.

### Step 5: Clean up legacy source descriptions

Remove descriptions of deleted legacy sources or old config files that no longer apply.

### Step 6: Review DB schema and operations

Verify DB schema/helper/maintenance/recovery descriptions against current implementation. Consolidate schema into `90_shared_04_*`, operations into `90_shared_05_*`.

### Step 7: Eliminate cross-domain duplication

For MCP-specific behavior, add reference links to MCP docs instead of duplicating in Shared docs. For Agent-specific behavior, add reference links to Agent docs instead of duplicating in Shared docs.

### Step 8: Clean up known issues

Keep only unresolved issues that affect current specification in `90_shared_90_*`.

### Step 9: Output deliverables

Provide:
- Modified target files
- Chapters/sections modified
- Explanations that differ from implementation
- Duplicated explanations across documents
- Consolidation destinations
- Deleted history explanations
- Additional implementation files needed

### Step 10: Update cross-references

Update File Index, AI Query Routing Table, Related Documents, and internal links after consolidation.

## Acceptance Criteria

1. shared/ vs db/ responsibility descriptions match current implementation
2. Import direction/importlinter constraint descriptions match implementation/config
3. Type/DTO field descriptions match current code
4. ToolExecutor/ToolRegistry/RuntimeToolRegistry responsibility descriptions don't conflict with MCP docs
