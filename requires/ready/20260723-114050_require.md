# Requirement: Create Centralized Failure Modes and Operational Readiness Document

## Priority

Medium

## Target files

- `docs/05_failure_modes_and_operational_readiness.md` (new)
- Operations documentation (update links)
- Index documents (update links)

## Background

Normal operation is documented in detail across multiple documents, but failure and degradation behavior is scattered. Operators need one place to understand what happens when parts of the system are unavailable. Important examples include MCP server startup failure, workflow definition missing or invalid, workflow database schema mismatch, RAG database failure, embedding service failure, memory layer degraded mode, tool discovery failure, and partial system startup in local mode. The current documentation describes some of these behaviors individually but does not provide a single operational model that explains system readiness from the user's point of view.

## Problem

Without a centralized failure modes document:
1. Operators must search across multiple documents to understand failure behavior
2. The relationship between service failures and user-visible capability impact is unclear
3. There is no capability-oriented readiness model — only service-oriented status
4. Recovery actions are not consolidated
5. New operators have no single reference for diagnosing degraded systems

## Reason for change

Moving from service-oriented status to capability-oriented readiness allows operators to understand what user-visible capability is affected when a part of the system becomes unavailable. For example, `git-mcp` unavailable means Repository Operations may be unavailable, not just "a service is down."

## Implementation intent

Create a centralized document explaining failure modes, degradation behavior, and capability-oriented readiness. This is a documentation-only task — it prepares the operational model that implementation can follow but does not require implementing readiness reporting.

## Implementation instructions

### Document (`05_failure_modes_and_operational_readiness.md`)

Create the document with the following sections:

**1. Failure Mode Overview** — Document failure categories:
- Startup failure
- Runtime service failure
- Configuration failure
- Database failure
- Tool discovery failure
- LLM service failure
- Embedding service failure
- Degraded memory mode
- RAG search degradation

**2. MCP Failure Behavior** — Document per startup_mode:
- `startup_mode = none`: expected operator action
- `startup_mode = persistent`: expected operator action
- `startup_mode = subprocess`: expected operator action
- fail-fast behavior in production mode
- fail-open behavior in local mode

**3. Workflow Failure Behavior** — Document:
- Missing workflow definition
- Invalid workflow definition
- Missing workflow database
- Workflow schema mismatch
- Required table missing
- Expected startup behavior
- Recovery action

**4. RAG Failure Behavior** — Document:
- Embedding API unavailable
- RAG database unavailable
- Vector extension unavailable
- FTS search failure
- Rerank failure
- Refiner fallback
- Expected degraded behavior

**5. Memory Layer Failure Behavior** — Document the known memory activation modes:
- `disabled`: no semantic retrieval, no FTS
- `fts-only`: FTS-based retrieval only, no vector similarity
- `degraded`: hybrid search partially functional
- `hybrid`: full functionality

**6. Capability Readiness Model** — Define capability readiness states:
- `healthy`: all required services available
- `degraded`: optional services unavailable, core functionality reduced
- `unavailable`: required services unavailable, feature unusable
- `unknown`: unable to determine status

For each capability, document: Required services, Optional services, Degraded conditions, Unavailable conditions, Recovery actions.

Suggested capabilities: Workflow Execution, File Management, Repository Operations, GitHub Operations, CI/CD Operations, RAG Retrieval, Markdown Query, Memory Services, Internet Search, Shell Execution.

**7. Operator-Facing Examples** — Include user-facing examples such as:
- "Repository Operations unavailable because git-mcp is unavailable."
- "RAG Retrieval degraded because embedding service is unavailable."
- "Memory Services running in fts-only mode."

### Documentation Updates

Update references in operations documentation and indexes to link to the new document. Cross-link to existing deployment and troubleshooting runbooks.
