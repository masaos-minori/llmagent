# MCP Tool Visibility: Clarify LLM Visibility vs Runtime Routability vs Availability Metadata

## Priority

High

## Target Files

- `docs/04_mcp_03_01_dispatch-and-routing.md` — add two-stage explanation section
- `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` — clarify enabled/disabled_reason fields
- `docs/04_mcp_06_09_mcp-failure-diagnosis.md` — document failure mode

## Background

The documentation does not clearly separate three different concepts:
1. Tool visibility to the LLM
2. Runtime routability through ToolRouteResolver
3. Runtime availability metadata from /v1/tools

These concepts are related but not identical.

## Problem

Without clear separation of these concepts, operators and developers cannot diagnose why a tool can appear in the LLM payload but fail at execution time. This leads to misdiagnosis of routing failures as LLM or approval issues.

## Reason for Change

Operators and developers need to understand why a tool can appear in the LLM payload but fail at execution time. Without this distinction, routing failures may be misdiagnosed as LLM or approval issues.

## Implementation Intent

Add a clear two-stage explanation to the documentation: LLM-visible tools vs runtime-routable tools, plus document the important failure mode where RuntimeToolRegistry is missing.

## Implementation Instructions

### Step 1: Update dispatch-and-routing.md

In the ツール呼び出しディスパッチフロー section, add:

```markdown
### Two-stage tool resolution

#### Stage 1: LLM visibility

A tool is LLM-visible if it remains in the tool payload after `LLMTurnRunner._filter_disabled_tool_definitions()` runs.

- Uses `RuntimeToolRegistry.llm_tool_definitions()` to filter
- If `RuntimeToolRegistry` is None, no filtering occurs and all configured tool definitions pass through

#### Stage 2: Runtime routability

A tool is runtime-routable only if `ToolRouteResolver.resolve()` can find it in `RuntimeToolRegistry`.

- If a tool is not present in the registry, execution fails with `Unknown tool`
```

### Step 2: Update tool-runtime-availability-metadata.md

Clarify the field mapping:

```markdown
- `/v1/tools.enabled` -> `RuntimeTool.enabled_for_llm`
- `/v1/tools.disabled_reason` -> part of the `/v1/tools` contract, but not currently a first-class `RuntimeTool` field
```

If direct diagnostic access is required, note the need for a future implementation task to add `RuntimeTool.disabled_reason`.

### Step 3: Update mcp-failure-diagnosis.md

Document the critical failure mode:

```markdown
### Failure mode: LLM sees tool but execution fails

If `RuntimeToolRegistry` is missing, a tool may still be visible to the LLM because filtering is skipped, but execution will fail with `Unknown tool`.
```

## Acceptance Criteria

1. Documentation clearly separates LLM visibility from runtime routability
2. The failure mode "LLM can see a tool but execution fails with Unknown tool" is documented
3. The docs state that `enabled` maps to `enabled_for_llm`
4. The docs do not imply that `RuntimeTool.disabled_reason` exists today
5. A future task is documented if first-class disabled reason support is desired
