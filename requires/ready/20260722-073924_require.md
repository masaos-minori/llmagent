# Session Memory Injection: Token Explosion on Long Sessions

## Priority

Low

## Target Files

- `scripts/agent/startup.py` (lines 338-346)

## Background

When a new session starts, `StartupOrchestrator._setup_prompt()` injects relevant semantic memories into the initial system prompt via `MemoryService.on_session_start()`. There is no limit on the number of snippets returned.

## Problem

In `_setup_prompt()` at lines 338-346:

```python
if ctx.services_required.memory is not None:
    memory_snippets = ctx.services_required.memory.on_session_start(
        ctx.session.session_id,
    )
    if memory_snippets:
        memory_block = "\n\n[Relevant memories]\n" + "\n".join(
            f"- {snippet.text}" for snippet in memory_snippets
        )
        initial_prompt = initial_prompt + memory_block
```

For long-running sessions with many interactions, `on_session_start()` may return a large number of memory snippets. These are concatenated directly into the system prompt without any token count check or limit. This can cause:
- System prompt exceeding LLM context window limits
- LLM returning errors due to oversized input
- Unpredictable behavior when context is silently truncated

## Reason for Change

- Long sessions accumulate memories that exceed typical LLM context windows (e.g., 4096 tokens)
- No upper bound on injected content creates operational risk
- Users experience confusing errors instead of graceful degradation

## Implementation Intent

Add a configurable limit on the number of memory snippets injected per session start. The default should be reasonable for most use cases while preventing token explosion.

## Implementation Instructions

Refactor lines 338-346 to add a snippet limit:

```python
if ctx.services_required.memory is not None:
    memory_snippets = ctx.services_required.memory.on_session_start(
        ctx.session.session_id,
    )
    if memory_snippets:
        # Limit to prevent token explosion
        max_snippets = ctx.cfg.memory.max_retrieved_memories or 10
        limited_snippets = memory_snippets[:max_snippets]
        memory_block = "\n\n[Relevant memories]\n" + "\n".join(
            f"- {snippet.text}" for snippet in limited_snippets
        )
        initial_prompt = initial_prompt + memory_block
```

Key points:
- Add a configuration parameter `memory.max_retrieved_memories` with a sensible default (10)
- Slice the snippets list before joining to enforce the limit
- Consider adding a warning log when snippets are truncated:
  ```python
  if len(memory_snippets) > max_snippets:
      logger.warning(
          "Truncated %d memory snippets to %d for session %s",
          len(memory_snippets), max_snippets, ctx.session.session_id
      )
  ```

## Acceptance Criteria

1. Memory snippets are limited to a configurable maximum per session start
2. Default limit prevents token explosion for typical LLM context windows
3. A warning is logged when snippets are truncated
4. Existing sessions with fewer than the limit continue to work unchanged

## Tests

- Unit test: mock `on_session_start()` to return more snippets than the limit, verify only the first N are injected
- Integration test: create a session with accumulated memories, verify system prompt size stays within bounds
