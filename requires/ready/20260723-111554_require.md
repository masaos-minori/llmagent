# Requirement: Update Workflow Task Status on Engine Failure to Prevent Orphaned Records

## Priority

Medium

## Target files

- `scripts/agent/orchestrator.py`

## Background

In `Orchestrator._handle_workflow_engine()`, when a workflow engine fails mid-turn after the "execute" stage completes, the task record is persisted in the database with status="execute". However, `_deactivate_workflow()` only clears in-memory state (`ctx.workflow.active = False`) and does not update the task status in the database. On the next turn, `_init_workflow_task()` creates a new task record, leaving the previous one in an inconsistent state (status="execute" but no active workflow).

## Problem

When the workflow engine fails after persisting the "execute" stage, the following issues occur:
1. Orphaned task records accumulate over time with status="execute" but no corresponding active workflow
2. `/stats` output shows incorrect workflow tracking data
3. Audit logs are confusing — they show tasks stuck in "execute" state without explanation

## Reason for change

Workflow task records must accurately reflect their lifecycle state. An orphaned "execute" status record misrepresents reality and makes debugging harder. Updating the task status to "failed" on engine failure ensures accurate tracking.

## Implementation intent

In `Orchestrator._handle_workflow_engine()`, add logic in the `finally` block to update the task status to "failed" if `error_kind` is set, preventing orphaned records.

## Implementation instructions

1. In `Orchestrator._handle_workflow_engine()`, modify the `finally` block to update the task status:
   ```python
   finally:
       # Update task status before deactivating to prevent orphaned records
       try:
           if task.task_id:
               if error_kind is not None:
                   store.update_task_status(task.task_id, "failed")
               else:
                   store.update_task_status(task.task_id, "completed")
       except Exception as e:
           logger.warning("Failed to update task status on engine exit: %s", e)
       self._deactivate_workflow(ctx)
       store.close()
   ```
2. Ensure `store` is accessible in the `finally` block — it already is since it's defined at method scope.
3. Do NOT modify `WorkflowEngine.run()` — the fix should be in the caller, not the engine itself, to keep concerns separated.

## Acceptance criteria

- [ ] When workflow engine fails, the current task's status is updated to "failed" in the database
- [ ] When workflow engine succeeds, the current task's status is updated to "completed" in the database
- [ ] No orphaned task records remain after workflow failures
- [ ] `/stats` output reflects correct workflow tracking data
- [ ] No regression in normal operation when workflow succeeds

## Tests

- Unit test: Mock `engine.run()` to raise an exception after `execute_fn()` completes, verify task status is updated to "failed"
- Unit test: Verify task status is updated to "completed" when engine.run() succeeds
- Integration test: Configure a workflow with multiple stages, trigger a tool execution failure during "execute" stage, verify the task record shows status="failed"
