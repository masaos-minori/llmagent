# MCP Startup Validation: Align Severity Documentation and FATAL Output Behavior

## Priority

Medium

## Target Files

- `docs/04_mcp_06_11_startup-validation-behavior-tool_definitions_strict.md` — update severity descriptions
- Agent CLI output or startup display documentation — add FATAL display description
- Any section describing [fatal], [non-fatal], or startup validation messages — update consistency

## Background

Startup validation uses statuses such as WARNING, FATAL, and SKIPPED. However, current startup display code sends FATAL outcomes through the warning output path while using a fatal tag. CLIView exposes a dedicated write_fatal() method.

## Problem

This creates ambiguity between semantic severity and display method. Operators should not have to infer severity only from embedded text. Fatal startup validation outcomes should be visually distinct and consistently documented.

## Reason for Change

Operators should not have to infer severity only from embedded text. Fatal startup validation outcomes should be visually distinct and consistently documented.

## Implementation Intent

Implement Option A (preferred): change the implementation to use write_fatal() for StartupCheckStatus.FATAL, then document all three status meanings and their display behavior.

## Implementation Instructions

### Step 1: Implement Option A — use write_fatal() for FATAL

Update the startup display code to call `write_fatal()` instead of `write_warning()` when the status is `StartupCheckStatus.FATAL`.

### Step 2: Document all startup validation severities

In startup-validation-behavior-tool_definitions_strict.md, add clear descriptions:

```markdown
### Startup validation statuses

#### WARNING
A non-critical issue. The system continues operating but the operator should be aware.
Example: optional server discovery failed.

#### FATAL
A critical issue that prevents normal operation. The system may be partially functional.
Displayed via write_fatal() for visual distinction.
Example: required server discovery failed.

#### SKIPPED
Discovery was skipped entirely. In local mode, this may indicate a full-session tool-call outage.
Example: MCP discovery skipped due to missing configuration.
```

### Step 3: Document production vs local behavior for MCP discovery

Clarify how MCP discovery behaves differently between production and local modes.

### Step 4: Update tests if needed

If the output method or text changes, update corresponding tests.

## Acceptance Criteria

1. Startup validation severity documentation matches implementation
2. FATAL display behavior is explicit
3. Tests are updated if output method or text changes
4. The docs clarify that mcp_tool_discovery skipped in local mode can still indicate a full-session tool-call outage
