# RuntimeToolRegistry enabled_for_llm wiring gap (requirement 15 follow-up)

## Priority

Critical

## Target files

- `scripts/agent/services/mcp_tool_discovery.py`
- `scripts/shared/runtime_tool.py`
- `scripts/mcp_servers/file/{read,write,delete}_server.py`
- `scripts/mcp_servers/git/git_server.py`
- `docs/04_mcp_03_06_tool-runtime-availability-metadata.md`

## Background

Requirements 14-20 (authored 2026-07-17) defined the RuntimeToolRegistry / MCP tool-availability-metadata migration:

- Requirement 14: rename `requires_config` → `config_dependent`
- Requirement 15: add `enabled`/`disabled_reason` to `/v1/tools`
- Requirement 17: RuntimeToolRegistry (disabled-tool visibility, hidden from LLM)
- Requirement 19: document the contract in `docs/04_mcp_03_06_tool-runtime-availability-metadata.md`
- Requirement 20: evaluate two further options (`include_disabled` query param, `disabled_code` structured enum)

The consolidated risk analysis in `requires/derived/20260718-122348_mcp_runtime_availability_metadata_landing_risks.md` identified that requirements 14-17 had landed but one critical gap remained: `enabled_for_llm` was never wired from the live `/v1/tools` response into `build_runtime_tool()`.

Since then, requirements 14-17 have been fully landed (verified against current source):
- Requirement 14: `config_dependent` is the live field name (13 matches); zero `requires_config` references remain
- Requirement 15: all 4 originally-targeted handlers plus more compute and return `(enabled, disabled_reason)` via `availability_flags()`/per-tool helpers
- Requirement 17: `McpToolDiscoveryService` is wired into `startup.py`/`context.py`; `RuntimeToolRegistry` is the sole routing authority
- Requirement 19: target doc exists and has been kept current through several rounds of edits since

However, the critical gap identified in the derived risks note has not been closed.

## Problem

`build_runtime_tool()` defaults `enabled_for_llm` to `False` when omitted. The call site in `mcp_tool_discovery.py:_dedupe_and_build()` (line 311-322) never passes `enabled_for_llm`:

```python
built[name] = build_runtime_tool(
    name=name,
    server_key=server_key,
    server_url=server_url,
    description=str(entry.get("description", "")),
    input_schema=entry.get("inputSchema", entry.get("input_schema")),
    raw_definition=entry,
    status=str(entry.get("status", "active")),
    is_write=entry.get("is_write"),
    requires_serial=entry.get("requires_serial"),
    capabilities=tuple(entry.get("capabilities", []) or []),
)
```

This means every discovered tool has `enabled_for_llm=False`, so `_filter_disabled_tool_definitions()` filters ALL of them out of the LLM's tool payload once `ctx.services_required.runtime_tools` is populated — which `startup.py:230` does on every successful discovery (the normal case, not an edge case). This is plausibly a full agent tool-calling outage under ordinary operation, not a cosmetic metadata gap.

Additionally, `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` line 22 states: "`enabled`/`disabled_reason` fields are not yet implemented in MCP server responses." This is outdated — these fields ARE implemented in the MCP server responses (as verified by grep across `scripts/mcp_servers/`), but the discovery service simply doesn't read them.

## Reason for change

Without this fix, the RuntimeToolRegistry's core purpose — filtering disabled tools from the LLM-facing tool list — is inverted: instead of hiding disabled tools, it hides ALL tools. This is a production-blocking issue.

## Implementation intent

Wire the `enabled` field from each tool's `/v1/tools` response into `enabled_for_llm` during discovery, so that:
- Tools with `enabled=true` are exposed to the LLM
- Tools with `enabled=false` are filtered out before reaching the LLM's tool-calling surface
- The `disabled_reason` field is preserved for diagnostics but not passed to the LLM

## Implementation instructions

### Step 1: Pass `enabled` to `build_runtime_tool()`

In `mcp_tool_discovery.py:_dedupe_and_build()`, add `enabled_for_llm` parameter:

```python
built[name] = build_runtime_tool(
    name=name,
    server_key=server_key,
    server_url=server_url,
    description=str(entry.get("description", "")),
    input_schema=entry.get("inputSchema", entry.get("input_schema")),
    raw_definition=entry,
    status=str(entry.get("status", "active")),
    is_write=entry.get("is_write"),
    requires_serial=entry.get("requires_serial"),
    enabled_for_llm=bool(entry.get("enabled", True)),  # ← NEW
    capabilities=tuple(entry.get("capabilities", []) or []),
)
```

Rationale: `enabled` defaults to `True` if absent (backward compatibility with servers that don't send it yet), and the value is cast to bool for type safety.

## Acceptance criteria

- Every discovered tool's `enabled_for_llm` reflects the actual `enabled` value from its `/v1/tools` response
- Tools with `enabled=true` are visible to the LLM's tool-calling surface
- Tools with `enabled=false` are filtered out before reaching the LLM's tool-calling surface
- Backward compatibility: tools from servers that don't send `enabled` still work (default to `True`)
- Documentation accurately reflects the current implementation state
