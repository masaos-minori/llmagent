# Requirement: Create ADR for Mandatory Workflow Execution

## Priority

Medium

## Target files

- `ADR-Workflow-Mandatory.md` (new)
- Architecture overview documentation (update links)
- Workflow documentation (update links)
- Deployment documentation (update links if appropriate)

## Background

The current documentation clearly states that workflow definitions are required and that workflow execution cannot be disabled. However, the architectural rationale is not sufficiently documented. This creates confusion for new contributors and operators because even simple agent usage appears to depend on workflow infrastructure. Without an explicit ADR, maintainers may incorrectly assume that workflow execution is an optional orchestration feature rather than a core control plane for task execution, approvals, retries, idempotency, and auditability.

## Problem

Without an explicit ADR documenting why workflow execution is mandatory:
1. New contributors and operators are confused about why even simple agent usage depends on workflow infrastructure
2. Maintainers may incorrectly assume workflow execution is an optional feature
3. The system-level mechanism providing task state tracking, approval gates, execution ordering, retry management, idempotency controls, auditability, recovery support, and consistent handling of tool side effects is not clearly communicated
4. Future proposals to add "optional workflow" modes lack context for rejection

## Reason for change

Documenting workflow as a mandatory execution boundary prevents future misunderstandings and provides a reference point for evaluating any proposal to relax the mandatory workflow constraint.

## Implementation intent

Create a new ADR document explaining why workflow execution is mandatory and why workflow bypass modes are not supported. Update references in architecture overview, workflow documentation, deployment documentation, and operations documentation where appropriate.

## Implementation instructions

### ADR Document (`ADR-Workflow-Mandatory.md`)

Create the ADR with the following sections:

**Context:** Describe the architectural context — the system executes LLM-planned tasks, some tools have side effects, some operations require approval, tool execution must be observable and recoverable, a direct LLM-to-tool path would make auditing and recovery harder.

**Decision:** State clearly — workflow execution is mandatory, workflow definitions are required deployment artifacts, workflow bypass mode is not supported, optional workflow mode is not supported, direct execution fallback is not supported.

**Rationale:** Explain why workflow is always required:
- All side-effecting operations must be traceable
- Approval state must survive process boundaries
- Retry and idempotency behavior must be centralized
- Partial task completion must be inspectable
- Recovery requires persisted task and attempt state
- Tool execution should not depend solely on LLM conversational state

**Alternatives Considered:** Document rejected alternatives:
- Direct tool execution without workflow — rejected because auditing and recovery become impossible
- Optional workflow mode — rejected because inconsistent enforcement of approval gates
- Workflow disabled for local mode — rejected because safety guarantees must be uniform across environments
- Workflow fallback when workflow definition is missing — rejected because silent degradation hides configuration errors
- Per-tool ad hoc approval without workflow state — rejected because approval state cannot survive process boundaries

**Consequences:** Document operational consequences:
- Deployment must include workflow definition files
- Startup must fail if mandatory workflow artifacts are missing or invalid
- Workflow schema must be initialized before service startup
- Operators must treat workflow failures as platform failures
- Simple chat and tool-backed tasks share the same execution control plane

**Non-Goals:** Clarify what this ADR does not attempt to solve:
- Does not define every workflow stage
- Does not redesign approval policy
- Does not introduce EventBus integration
- Does not change runtime behavior
