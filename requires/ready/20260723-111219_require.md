# Requirement: Handle Session Start Failure Gracefully in REPL Loop

## Priority

Medium

## Target files

- `scripts/agent/repl.py`

## Background

In `AgentREPL._run_repl_loop()`, `ctx.session.start()` can raise two types of exceptions:
- `RuntimeError` — when the session schema is missing (already includes a helpful message: "Session schema missing. Run: bash deploy/init_db.sh to initialize the database.")
- `sqlite3.Error` — for other DB failures (connection refused, disk full, etc.)

However, only `RuntimeError` is caught by the `except RuntimeError` handler. `sqlite3.Error` propagates uncaught, bypassing the user-facing error message entirely. Even for the `RuntimeError` case, the outer handler writes the raw exception string, which may include the full traceback rather than the clean message from `session.start()`.

## Problem

When `session.start()` fails due to a missing schema or DB connection error, the operator receives an opaque error message. For the "no such table" case, the message is partially helpful but lacks context about what to do next. For other DB errors, no guidance is provided. Additionally, the exception propagates through the outer `except RuntimeError` block, which writes the raw exception string — losing the structured message from `session.start()`.

## Reason for change

Startup failures must produce actionable error messages so operators can resolve issues without reading source code. Currently, both the "no such table" case and other DB errors produce confusing output that requires investigation.

## Implementation intent

Wrap `ctx.session.start()` in a dedicated try/except that catches both `RuntimeError` and `sqlite3.Error`, and emits a user-friendly message for each case before re-raising. This ensures the outer handler always sees a consistent error format.

## Implementation instructions

1. In `AgentREPL._run_repl_loop()`, add a nested try/except around `ctx.session.start()`:
   - Catch `RuntimeError` and check if the message contains "no such table" — if so, emit a clear fatal message directing the operator to run `bash deploy/init_db.sh`
   - Catch `sqlite3.Error` and emit a fatal message indicating the database is unavailable, suggesting they check DB connectivity or run `bash deploy/init_db.sh`
   - Re-raise after writing the message so the outer `except RuntimeError` block handles cleanup
2. Consider whether the outer `except RuntimeError` handler should also catch `sqlite3.Error` to prevent double-handling. If the inner handler re-raises, the outer handler may still receive it depending on the exception type.
3. Do NOT modify `session.py` — the existing "no such table" error message there is already correct.

## Acceptance criteria

- [ ] When session schema is missing, the operator sees a clear fatal message: "Session schema missing. Run: bash deploy/init_db.sh to initialize the database."
- [ ] When DB connection fails, the operator sees a clear fatal message about database unavailability
- [ ] No raw exception strings or tracebacks shown to the operator
- [ ] Resource cleanup proceeds normally via the finally block after the error message
- [ ] No regression in normal operation when session.start() succeeds

## Tests

- Unit test: Mock `session.start()` to raise `RuntimeError("Session schema missing. Run: bash deploy/init_db.sh...")`, verify the REPL loop writes the expected fatal message
- Unit test: Mock `session.start()` to raise `sqlite3.Error("database is locked")`, verify the REPL loop writes a database-unavailable message
- Integration test: Remove the session table, start REPL, verify the startup banner shows the correct error message
