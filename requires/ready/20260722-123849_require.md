# MCP git-mcp: Align Availability Metadata and Disabled-Call Behavior with Documented Contract

## Priority

High

## Target Files

- `docs/04_mcp_04_05_git.md` — update git-mcp section
- `docs/04_mcp_03_06_tool-runtime-availability-metadata.md` — clarify disabled_reason values
- `docs/04_mcp_02_03_audit-logging-and-errors.md` — update dispatch_tool helper

## Background

git-mcp implements runtime availability metadata in /v1/tools, but the documentation must be updated to reflect this as current behavior. Additionally, the disabled /v1/call_tool response appears to omit the disabled reason even though it is computed.

## Problem

The server already exposes useful availability metadata, but if direct disabled calls hide the reason, operators lose the most important diagnostic detail. The documentation and implementation should agree on the disabled-tool contract.

## Reason for Change

The server already exposes useful availability metadata, but if direct disabled calls hide the reason, operators lose the most important diagnostic detail. The documentation and implementation should agree on the disabled-tool contract.

## Implementation Intent

Update git-mcp documentation to reflect current /v1/tools behavior, and fix or document the /v1/call_tool disabled response behavior.

## Implementation Instructions

### Step 1: Update git-mcp documentation (04_mcp_04_05_git.md)

In the ツール section, add:

```markdown
### Availability metadata

git-mcp currently returns `enabled` and `disabled_reason` from `/v1/tools`:

- If `allowed_repo_paths` is empty: all git tools are disabled; `disabled_reason` = `allowed_repo_paths is empty`
- If `read_only=true`: git write tools are disabled; `disabled_reason` = `read_only=true`
- Read-only git tools remain enabled when `allowed_repo_paths` is non-empty
- `allowed_repo_paths is empty` takes precedence over `read_only=true`
```

### Step 2: Fix or document /v1/call_tool disabled response

Check the current implementation of git-mcp's /v1/call_tool handler. 

If the disabled reason is being lost:
- Fix the code to include the reason in the response: `Tool disabled: <reason>`

If fixing is deferred:
- Document the current behavior as a known deviation:
  ```markdown
  ### Known deviation: disabled call reason omission
  
  When a git tool is disabled via /v1/call_tool, the response omits the disabled reason.
  Expected: `Tool disabled: allowed_repo_paths is empty`
  Actual: `Tool disabled: `
  ```

### Step 3: Update validation error handling

Verify that git-mcp includes validation error details in its responses. If not, either fix or document the deviation.

### Step 4: Add tests

Add tests covering:
- Empty `allowed_repo_paths`
- `read_only=true` for write tools
- Enabled read-only tools when repository paths are configured

## Acceptance Criteria

1. git-mcp catalog accurately describes current /v1/tools availability metadata
2. Disabled git tool calls include the concrete reason, or the deviation is documented as known
3. Validation errors either include details or the docs explicitly state that details are omitted
4. Tests cover: empty allowed_repo_paths, read_only=true for write tools, enabled read-only tools when repository paths are configured

## Tests
