# Discovery failure must be fatal under PRODUCTION security profile

## Priority

Medium

## Target files

- `scripts/agent/startup.py`
- `scripts/agent/repl_health.py`
- `scripts/agent/services/mcp_tool_discovery.py`
- `tests/test_startup.py` (or equivalent)

## Background

Requirement 17's plan (`plans/20260720-134821_plan.md`) unified routing authority to `RuntimeToolRegistry`, removing the `ToolRegistry` fallback. This eliminated the safety net that previously allowed partial tool availability when discovery failed.

The derived risk in `requires/derived/20260720-135023_risks.md` identified that if discovery throws (e.g., every MCP server is unreachable at startup, or a bug in discovery itself), `discovery.registry` is never built, `set_runtime_registry()` is never called, and `ctx.services_required.tools._resolver._runtime_registry` stays `None` for the entire agent session. The REPL loop still starts (this failure is not fatal), but under sole-authority routing every single tool call for that session will immediately raise `ValueError` from `ToolRouteResolver.resolve()`.

Under the current (pre-change) two-tier behavior, this same failure mode still leaves the agent partially functional: `ToolRegistry`'s compile-time frozensets keep resolving tool names to server keys, so at least the tools whose backing servers happen to be reachable can still be invoked. Sole-authority routing turns a partial/degraded-mode failure into a full-session tool-call outage, with only a non-fatal "skipped" line in the startup log to signal it — easy to miss in interactive use.

## Problem

`scripts/agent/startup.py`'s `StartupOrchestrator._check_services()` step 4 wraps `McpToolDiscoveryService(ctx).discover_all()` in a bare `except Exception` that is **non-fatal**:

```python
except Exception:  # noqa: BLE001
    pipeline.add_skipped(
        "mcp_tool_discovery", f"MCP tool discovery skipped: {exc}"
    )
```

When discovery fails, the agent starts silently crippled — every tool call will fail with `ValueError` from `ToolRouteResolver.resolve()`, but there is no clear indication at startup why.

## Reason for change

Sole-authority routing eliminates the safety net. Without this fix, a discovery failure produces a silent, catastrophic failure mode where all tool calls fail but the agent appears to start normally.

## Implementation intent

Make MCP tool discovery failures fatal under the `PRODUCTION` security profile, so the agent refuses to start rather than starting silently crippled. Under `LOCAL` profile, keep as WARNING but make the warning unmistakable.

## Implementation instructions

### Option A: Make discovery fatal under PRODUCTION (recommended)

In `startup.py:_check_services()`, change the exception handler to check the security profile:

```python
except Exception as exc:  # noqa: BLE001
    if self._ctx.cfg.mcp.security_profile == SecurityProfile.PRODUCTION:
        pipeline.add_fatal(
            "mcp_tool_discovery", f"MCP tool discovery failed: {exc}"
        )
    else:
        pipeline.add_warning(
            "mcp_tool_discovery", f"MCP tool discovery skipped: {exc}"
        )
```

### Option B: Add explicit health gate

Add an explicit post-discovery health check in `agent/repl_health.py`:

```python
def assert_discovery_succeeded(pipeline: StartupPipeline) -> None:
    """Assert that discovery succeeded and runtime_tools is non-empty."""
    ctx = get_current_context()
    if ctx.services_required.runtime_tools is None:
        if ctx.cfg.mcp.security_profile == SecurityProfile.PRODUCTION:
            pipeline.add_fatal(
                "discovery_validation",
                "RuntimeToolRegistry not populated — all tool calls will fail",
            )
        else:
            pipeline.add_warning(
                "discovery_validation",
                "RuntimeToolRegistry not populated — all tool calls will fail",
            )
```

Call this after `_check_services()` completes.

### Option C: Upgrade log level / wording (minimum viable)

At minimum, upgrade the "skipped" case to ERROR level and clarify the impact:

```python
except Exception as exc:  # noqa: BLE001
    logger.error(
        "CRITICAL: MCP tool discovery failed — %s. All tool calls will fail.",
        exc,
    )
    pipeline.add_skipped(
        "mcp_tool_discovery",
        f"MCP tool discovery FAILED — all tool calls will fail: {exc}",
    )
```

**Recommendation:** Option A is preferred because it mirrors the existing `SecurityProfile.PRODUCTION` fatal-on-subprocess-failure pattern already used elsewhere in `startup.py` (`_start_servers()`).

## Acceptance criteria

- Under `PRODUCTION` security profile, a discovery failure causes startup to fail fast with a clear fatal message
- Under `LOCAL` security profile, a discovery failure produces a high-visibility WARNING distinct from routine per-server "unreachable" warnings
- Existing tests for `StartupOrchestrator._check_services()` gain a case covering total discovery failure and assert the chosen behavior

## Tests

- Unit test for `StartupOrchestrator._check_services()` with mocked discovery raising an exception
- Assert that under `PRODUCTION` profile, the pipeline outcome is FATAL
- Assert that under `LOCAL` profile, the pipeline outcome is WARNING (not SKIPPED)
- Integration test verifying that a fully unreachable MCP server cluster causes startup failure under `PRODUCTION` profile
