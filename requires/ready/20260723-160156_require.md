# Requirement: Strengthen Governance Rules for Canonical Entry Points and AI-readable Markdown

## Priority

High

## Target files

- `docs/00_governance_02_canonical-source-rule.md`
- `docs/00_governance_06_ai-reading-metadata.md`

## Background

The documentation set already has governance documents for canonical source rules and AI reading metadata. However, two gaps remain:

1. AI agents need a concise list of the first document to consult for each major area.
2. Markdown formatting rules for AI readability are not yet clearly documented in the AI reading metadata guidance.

## Problem

The current canonical source rule correctly delegates detailed canonical ownership to each area's document guide. However, without a compact area entry-point table, AI agents may start from the wrong file or read too many documents.

Also, some documentation includes patterns that are difficult for AI/RAG systems to parse reliably, such as:
- Shell commands written inline with prose
- JSON examples not fenced as code blocks
- Python signatures not clearly marked
- Body-level keywords written as Markdown headings
- Overuse of HTML tables where Markdown tables would be sufficient

## Reason for change

This change improves document retrieval accuracy, reduces unnecessary context loading, and prevents AI agents from misreading prose as commands or headings. It also avoids creating redundant new governance files by extending existing governance documents.

## Implementation intent

Strengthen the existing governance layer without changing the documentation hierarchy. This task must:
- Keep the folder structure flat
- Avoid creating `00_canonical_sources.md`
- Avoid creating a new Markdown formatting governance file
- Avoid changing `00_index.md`
- Avoid introducing ADR documents

## Implementation instructions

### 1. Update `00_governance_02_canonical-source-rule.md`

Add a Japanese section:

`## 領域別の正典入口`

The section must explain that:
- This table does not replace each area's document guide
- Detailed canonical source ownership remains delegated to each area's `*_00_document-guide.md`
- The table only identifies the first document to consult

Include the following area entry points:

| Area | Entry Point |
|---|---|
| Overview | `01_overview.md` |
| Deployment | `02_deployment-part1.md` |
| RAG | `03_rag_00_document-guide.md` |
| MCP | `04_mcp_00_document-guide.md` |
| Agent | `05_agent_00_document-guide.md` |
| EventBus | `06_eventbus_00_document-guide.md` |
| Shared/DB | `90_shared_00_document-guide.md` |
| Governance | `00_governance_01_documentation-governance.md` |

### 2. Update `00_governance_06_ai-reading-metadata.md`

Add a Japanese section:

`## Markdown記法ルール`

Include guidance for:

#### コードブロック

- Shell commands should use fenced code blocks with `bash`
- JSON examples should use fenced code blocks with `json`
- Python imports, type examples, and function signatures should use fenced code blocks with `python`
- Commands must not be placed on the same line as prose

#### 表

- Prefer Markdown tables for small tables
- Use HTML tables only when Markdown tables are insufficient
- Do not place long code examples inside tables

#### Keywords

- Prefer front matter `tags` and `keywords`
- If a body-level `Keywords` section is used, write keywords as a bullet list
- Do not write each keyword as a Markdown heading

#### 文書境界

- Each document should start with valid front matter
- Combined documentation files should keep document boundaries identifiable
