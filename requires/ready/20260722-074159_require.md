# Lifecycle Manager UNKNOWN State Transition Validation Gap

## Priority

Low

## Target Files

- `scripts/agent/factory.py` (lines 63, 67-69)

## Background

`LifecycleManager.__init__()` initializes `_states` as an empty dictionary. When a server key is not yet present, `get_transport_state()` returns `LifecycleState.UNKNOWN`. The code then attempts to validate the `UNKNOWN → STARTING` transition but does not handle it specially — it relies on `assert_valid_transition()` to either accept or reject it.

## Problem

In `factory.py` at lines 63 and 67-69:

```python
self._states: dict[str, LifecycleState] = {}

def _set_state(self, server_key: str, new_state: LifecycleState) -> None:
    current = self._states.get(server_key, LifecycleState.UNKNOWN)
    try:
        assert_valid_transition(current, new_state)
    except ValueError:
        _logger.warning(
            "Lifecycle: invalid state transition %r -> %r for %r",
            current,
            new_state,
```

The issue is:
1. For a brand-new server that has never been seen before, `current` will be `LifecycleState.UNKNOWN`
2. `assert_valid_transition(LifecycleState.UNKNOWN, LifecycleState.STARTING)` may raise `ValueError` if this transition is not explicitly allowed
3. If it raises, the warning is logged but the state is NOT updated — the server remains in an undefined state
4. There is no explicit handling for the "first time seeing this server" case

This creates ambiguity: is `UNKNOWN` a valid starting state, or should we treat missing keys differently?

## Reason for Change

- Unclear whether `UNKNOWN → STARTING` is intentional or a gap in the state machine
- Servers that have never been started may fail to initialize properly
- Debugging is difficult because the root cause is hidden behind a vague warning message

## Implementation Intent

Make the "first time seeing this server" case explicit by treating missing keys differently from `UNKNOWN`. A missing key means "never existed," while `UNKNOWN` means "exists but state is unknown." These should be handled separately.

## Implementation Instructions

Refactor `_set_state()` to distinguish between "never seen" and "unknown state":

```python
def _set_state(self, server_key: str, new_state: LifecycleState) -> None:
    """Transition the server state with validation."""
    # Distinguish between "never seen" and "unknown state"
    if server_key not in self._states:
        # First time seeing this server — allow any initial state
        logger.info("Lifecycle: initializing %s to %r", server_key, new_state)
        self._states[server_key] = new_state
        return
    
    current = self._states[server_key]
    try:
        assert_valid_transition(current, new_state)
    except ValueError:
        logger.warning(
            "Lifecycle: invalid state transition %r -> %r for %r",
            current,
            new_state,
            server_key
        )
        raise
```

Key points:
- Add explicit check for `server_key not in self._states` before accessing
- Log an info-level message when initializing a new server's state
- Remove the default value `LifecycleState.UNKNOWN` from `.get()` call
- Keep the existing warning behavior for actual invalid transitions

## Acceptance Criteria

1. First-time servers can be initialized to any valid starting state without validation errors
2. Existing servers still go through proper state transition validation
3. Info-level logging occurs when a new server's state is initialized
4. No behavioral change for servers with known states

## Tests

- Unit test: verify first-time server initialization works without validation errors
- Unit test: verify existing server state transitions still require valid transitions
- Integration test: start MCP server that was never previously running, verify correct state progression
