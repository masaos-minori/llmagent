## Background

A tool can be visible to the LLM but still fail at execution time with `Unknown tool`. This happens when RuntimeToolRegistry is missing or incomplete — LLM tool payload filtering may not remove the tool, but runtime routing still cannot resolve it.

## Problem

This failure mode is confusing because it looks like an LLM or approval problem, but the true cause is missing runtime routing metadata. Operators lack a diagnostic path for this scenario.

## Reason for Change

This failure mode is confusing because it looks like an LLM or approval problem, but the true cause is missing runtime routing metadata. Operators lack a diagnostic path for this scenario.

## Implementation Intent

Add a diagnosis section for the "LLM called a tool, but execution failed with Unknown tool" failure mode, including recommended diagnosis steps and clarification about startup discovery outcomes.

## Implementation Instructions

### Step 1: Add diagnosis section to mcp-failure-diagnosis.md

Add a new section:

```markdown
### LLM called a tool, but execution failed with Unknown tool

**Possible causes:**
- RuntimeToolRegistry is missing or incomplete
- Discovery was FATAL, WARNING, or SKIPPED
- Duplicate tool name detection excluded the tool
- ToolExecutor.set_runtime_registry() was not called

**Diagnosis steps:**
1. Check startup output for `mcp_tool_discovery` outcomes
2. Check whether discovery was FATAL, WARNING, or SKIPPED
3. Check whether the owning server's /v1/tools response includes the tool
4. Check whether duplicate tool name detection excluded the tool
5. Check whether ctx.services_required.runtime_tools was populated
6. Check whether ToolExecutor.set_runtime_registry() was called
7. Check whether the tool exists in RuntimeToolRegistry
8. Do not rely only on config/agent.toml tool_definitions
```

### Step 2: Update dispatch-and-routing.md

Cross-reference the diagnosis section from the two-stage tool resolution explanation.

### Step 3: Update startup-validation-behavior-tool_definitions_strict.md

Document that in local mode, MCP discovery failure may appear as `SKIPPED`, but it can still mean all tool calls will fail for that session. Mark this as operationally high impact.
