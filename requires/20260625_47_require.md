## [High][eventbus][api] Add pagination and streaming-safe chunking to /replay and /dlq instead of using unbounded fetchall()

### Summary
The current `/replay` and `/dlq` endpoints load all matching rows into memory at once.

### Problem
Unbounded `fetchall()` is unsafe for large event volumes and can cause memory pressure, unstable response times, and poor operational behavior.

### Scope
- app.py
- /replay
- /dlq
- replay and DLQ response design
- API documentation

### Task
- replace unbounded `fetchall()` with chunked iteration or page-limited retrieval
- add request parameters for page size, limit, or cursor-based pagination where appropriate
- make SSE replay stream rows incrementally instead of loading the full result set first
- define API limits and defaults explicitly
- update docs and tests accordingly

### Expected Result
Replay and DLQ listing remain stable even when the event table or DLQ grows large.

### Acceptance Criteria
- `/replay` no longer loads all rows eagerly
- `/dlq` no longer loads all rows eagerly
- pagination or chunked retrieval is implemented and documented
- tests cover large-result behavior
