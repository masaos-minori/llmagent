# Document when stdio transport is acceptable and when it should be avoided

## Summary
Clarify the operational boundary of stdio transport so that engineers understand it is supported but not the default production path.

## Background
The MCP documents explain stdio transport and its serialization constraints, while also stating that all current production servers use HTTP. The intended use boundary can be documented more explicitly.

## Objective
Define when stdio is acceptable, when it is discouraged, and when HTTP is preferred.

## In Scope
- `04_mcp_01_system_overview.md`
- `04_mcp_02_protocol_and_transport.md`
- `04_mcp_03_routing_lifecycle_and_execution.md`

## Required Changes
- Add an explicit "When to use stdio" subsection
- Add an explicit "When not to use stdio" subsection
- State that per-instance serialization limits stdio concurrency
- Clarify that HTTP is the current production-default transport model
- Keep future scaling options as non-committed planning notes

## Acceptance Criteria
- The document set clearly describes stdio as a supported but constrained transport
- Readers can identify whether stdio is suitable for a low-concurrency tool
- The production-preferred path is unambiguous

## Out of Scope
- Implementing stdio multiplexing
- Replacing stdio with HTTP everywhere

## Notes
Do not over-promise future stdio scaling work. Keep future options clearly labeled as planning notes.
