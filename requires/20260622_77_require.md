# Title
Improve tool scheduling to become dependency-aware instead of round-wide serialization

## Summary
Refine tool execution scheduling so independent read-only calls are not unnecessarily serialized just because a side-effect tool appears in the same round.

## Background
Current behavior serializes the whole tool-call round when any side-effect tool is present.

## Problem
This preserves safety but increases latency and reduces concurrency even when some calls are logically independent.

## Required Changes
- Separate read-only tool calls from side-effect tool calls where safe.
- Preserve ordering guarantees for side-effect tools.
- Extend DAG-oriented execution support if useful.
- Make scheduling decisions visible in debug/trace output.

## Acceptance Criteria
- Independent read-only tools are not always forced into sequential execution.
- Safety guarantees for side-effect tools remain intact.
- Execution order is explainable for debugging.

## Out of Scope
- Workflow-engine redesign
- Distributed orchestration

## AI Implementation Instruction
Implement this as a scheduling-granularity improvement.

Strict rules:
- Do not weaken side-effect safety.
- Prefer explicit dependency handling over hidden heuristics.
- Keep resulting behavior explainable.
- If dependency rules are ambiguous, mark them as: Needs confirmation

Deliverables:
1. Improved scheduling behavior
2. Debug/trace visibility for execution-order decisions
3. Preserved side-effect safety
