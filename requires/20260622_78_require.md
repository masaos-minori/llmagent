# Title
Add operator-facing consistency check and rebuild flows for the memory layer

## Summary
Expose memory-layer consistency checking and rebuild operations so operators can detect and repair JSONL-to-SQLite divergence.

## Background
The memory layer uses JSONL as the append-only source of truth and SQLite as the searchable index.
Consistency-related APIs exist, but operator-facing flows are weak or unclear.

## Problem
When JSONL and SQLite drift, operators lack a standard supported path to diagnose and repair the issue.

## Required Changes
- Add an operator-facing consistency-check flow such as `/memory check-consistency`.
- Add an operator-facing rebuild or reindex flow from JSONL to SQLite.
- Expose mismatch signals through observability or diagnostics.

## Acceptance Criteria
- Operators can check JSONL / SQLite consistency without custom scripts.
- Operators can trigger a supported rebuild or reindex flow.
- Consistency mismatch is visible through CLI or normal diagnostics.

## Out of Scope
- Changing extraction heuristics
- Changing memory schema design

## AI Implementation Instruction
Implement this as an operability and recovery-path task.

Strict rules:
- Preserve JSONL as the canonical source of truth unless another issue changes that design.
- Do not add silent self-healing without operator visibility.
- Prefer explicit operator-invoked checks and rebuild flows.
- If rebuild semantics are unclear, mark them as: Needs confirmation

Deliverables:
1. Operator-facing consistency check
2. Operator-facing rebuild/reindex path
3. Visibility of consistency problems
