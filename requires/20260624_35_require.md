# Consolidate operator-facing observability for partials, fallback truncation, and session diagnostics

## Summary
Create one coherent operator-facing observability model for partial completions, fallback truncation, and session-end diagnostics.

## Background
The current docs describe visibility through several channels:
- `/stats`
- `/context`
- `tool_result_store`
- `diagnostics.jsonl`

The functionality is useful, but the operator guidance is fragmented.

## Objective
Explain in one place what each user-facing observability surface is for.

## In Scope
- `05_agent_10_operations-and-observability.md`
- `05_agent_15_fallback-truncation-stats.md`
- `05_agent_17_partial-llm-completion-visibility.md`

## Required Changes
- Create a single observability matrix covering:
  - partial completions
  - fallback truncation count
  - compression count
  - diagnostic artifacts
- Clarify which surface is best for:
  - live session inspection
  - post-mortem review
  - per-turn warning visibility
- Cross-link specialized design notes instead of duplicating them

## Acceptance Criteria
- Operators can tell where to inspect each diagnostic signal
- `/stats` vs `/context` vs tool-result artifacts are clearly differentiated
- Specialized design-note files no longer act as hidden normative sources

## Out of Scope
- UI redesign
- Adding new metrics

## Notes
Prefer a visibility matrix over prose-only explanation.
