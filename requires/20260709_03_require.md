# [Medium] Enforce production fail-fast settings with a dedicated ProductionConfigValidator

## Labels
priority/medium, area/config-validation, area/security, production-hardening

## Summary
Add a dedicated production configuration validator to reject fail-open or non-strict settings in production.

This validator must be separate from the existing RAG-specific validator.

## Background
The existing `RagConfigValidator` is scoped to RAG cross-file consistency. It should not be expanded with unrelated production hardening rules.

Create a dedicated validator for production profile hardening.

## Required Changes

### New Validator
Add:

- `shared/production_config_validator.py`
- `tests/test_production_config_validator.py`

### Settings to Enforce
When `security_profile == "production"`:

- `plugin_strict` must be `true`
- `tool_definitions_strict` must be `true`
- `routing_drift_strict` must be `true`, or routing drift must otherwise be fatal

### Runtime Integration
- Call the production validator during startup.
- Fail startup with `RuntimeError` on production policy violations.
- Keep local/development behavior unchanged.

### Documentation
- Update `04_mcp_05_security_and_safety_model.md`:
  - Correct any statement claiming `tool_definitions_strict` default is `true` if implementation default is `false`.
  - State that production requires strict settings.
- Update `04_mcp_06_configuration_and_operations.md` Pre-Production checklist:
  - Add `plugin_strict=true`
  - Add `routing_drift_strict=true`
- Update `05_agent_08_configuration.md` with production recommended values.
- Update `05_agent_11_extension-points.md` to state that production must not use `plugin_strict=false`.

### Tests
Add production profile tests.

Test cases:

- production + `plugin_strict=false` => error
- production + `tool_definitions_strict=false` => error
- production + `routing_drift_strict=false` => error or forced fatal behavior
- local + non-strict settings => warning or no production error
- production + all strict settings true => valid

## Acceptance Criteria
- Production profile rejects non-strict startup configuration.
- Local/development behavior remains unchanged.
- `RagConfigValidator` remains RAG-specific.
- Tests cover production and local behavior.
- Documentation clearly distinguishes implementation defaults from production requirements.

## Non-Goals
- Do not modify `RagConfigValidator` responsibilities.
- Do not change CI default behavior except where production profile explicitly requires strict mode.
- Do not remove local/development fail-open behavior.
