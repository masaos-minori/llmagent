# [Medium] Add canonical vs derived table classification to RAG data model docs

## Goal

Make the RAG data model explicitly distinguish canonical tables from derived indexes.

## Background

The current data model documents the four RAG-owned tables, but DESIGN-3 should be made more explicit: `documents` and `chunks` are canonical data, while `chunks_fts` and `chunks_vec` are derived indexes. 【2-58c0bb】【1-54c80f】

## Scope

Documentation only.

## Implementation Tasks

- Update `03_rag_04_data_model_and_interfaces.md`.
- Add a canonical/derived classification near the RAG-owned table list.
- Mark these as canonical:
  - `documents`
  - `chunks`
- Mark these as derived indexes:
  - `chunks_fts`
  - `chunks_vec`
- Explain that application code should mutate canonical data and let triggers or ingestion logic maintain derived indexes.
- Explain that `chunks_fts` is trigger-synced.
- Explain that `chunks_vec` is maintained by ingestion and explicit delete logic.

## Acceptance Criteria

- The data model clearly identifies canonical tables and derived indexes.
- `chunks_fts` and `chunks_vec` are not described as source-of-truth tables.
- The sync mechanism for each derived index is documented.

## Validation

- Review `03_rag_04_data_model_and_interfaces.md`.
- Confirm the table responsibility model matches DESIGN-3.
