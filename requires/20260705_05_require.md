# Persist workflow stage-level progress for failed or partial workflow runs

## Priority

High

## Type

Design / Implementation / Recovery

## Background

The workflow engine executes stages such as `plan`, `execute`, `verify`, and optional retry handling.

Workflow state is persisted in `workflow.sqlite` using tables such as:

- `tasks`
- `attempts`
- `processed_events`
- `approvals`
- `artifacts`

## Problem

When a workflow fails after some stages have completed, the final task status may be persisted, but stage-level completion details may not be sufficiently queryable from the database.

Relying only on audit logs makes recovery, duplicate detection, and replay analysis difficult.

## Goal

Make workflow partial failure state reconstructable from the database.

## Scope

Affected areas:

- `agent/workflow/workflow_engine.py`
- `agent/workflow/state_store.py`
- Workflow schema
- Workflow audit events
- Artifact persistence
- Workflow recovery tests

## Implementation Tasks

- Inspect the `attempts` table schema.
- Ensure attempts persist:
  - `stage_id`
  - `status`
  - `started_at`
  - `ended_at`
  - `error_kind`
  - `error_detail`
- Ensure `stage_completed` audit events can be correlated with attempt records.
- Persist `workflow_id`, `task_id`, `stage_id`, and `attempt` on artifact records.
- Review `processed_events` idempotency keys.
- Ensure idempotency keys include enough context:
  - `workflow_id`
  - `task_id`
  - `stage_id`
  - `attempt`
- Add tests for failed workflow runs at each stage.

## Acceptance Criteria

- Operators can query the database to determine which stage failed.
- Completed stages are visible in `attempts`.
- Failed stages include error classification.
- Artifacts are linked to workflow, task, stage, and attempt.
- Re-running a failed workflow does not silently duplicate external side effects.
- Audit logs and database records can be correlated.

## Validation Plan

Add tests for:

- Failure during `plan`.
- Failure during `execute`.
- Failure during `verify`.
- Failure after artifact creation.
- Retryable stage failure.
- Non-retryable stage failure.
- Database query reconstruction of workflow state.

## Risks

Schema changes may require migrations. Ensure migrations are additive and backward-compatible where possible.

## Out of Scope

- Implementing full automatic workflow resume.
- Changing approval semantics.
- Documentation-only updates.
