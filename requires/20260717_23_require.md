# [High] Add reliable audit, health state, and lightweight metrics for web-search-mcp

## Priority

High

## Summary

Make `web-search-mcp` observable through audit, health, and lightweight runtime metrics.

This issue combines the following work:

- Always emit audit logs for both success and failure
- Include classified error information in audit
- Track recent upstream provider health
- Track lightweight process-local metrics

## Target files

- `scripts/mcp_servers/web_search/server.py`
- `scripts/mcp_servers/web_search/health.py`
- `scripts/mcp_servers/web_search/metrics.py`
- `scripts/mcp_servers/web_search/formatters.py`
- `scripts/mcp_servers/web_search/search_provider.py`
- `tests/test_web_search_audit.py`
- `tests/test_web_search_health.py`
- `tests/test_web_search_metrics.py`
- `tests/test_web_search_server.py`

## Reason for change

`server.py::call_tool()` currently writes audit logs after dispatch returns.

If dispatch raises an exception, audit logging may be skipped.

The `/health` endpoint currently does not reflect DuckDuckGo provider state.

The implementation logs individual search latency and result count, but does not maintain aggregate state such as total searches, failed searches, or last failure timestamp.

## Intent

Make web-search-mcp operationally transparent without introducing a heavy observability framework.

The server should record:

- what was attempted
- whether it succeeded
- why it failed
- whether provider failures are repeated
- whether the upstream provider appears degraded

## Required changes

### 1. Always audit tool calls

Refactor `server.py::call_tool()` so audit is emitted for:

- successful search
- zero-result search
- validation error
- unknown tool
- timeout error
- provider/network/parse error
- unexpected error

Use `try` / `except` / `finally` or an equivalent structure.

Preserve existing HTTP and MCP error behavior.

### 2. Improve audit detail

Include useful fields where available:

- server key
- tool name
- outcome
- error type
- provider
- requested max_results
- result count
- latency milliseconds
- query preview
- optional query hash

Prefer:

- query preview: first 80 characters
- query hash: SHA-256 of normalized query

Avoid storing sensitive full query text unless policy explicitly allows it.

### 3. Track provider health state

Add a small health state module or object that records:

- provider name
- last success timestamp
- last failure timestamp
- last error type
- consecutive failure count

The `/health` response should include provider details.

Do not perform a live external search on every `/health` call by default.

### 4. Add lightweight metrics

Track:

- queries_total
- queries_succeeded
- queries_failed
- average latency in milliseconds
- last_success_at
- last_failure_at
- last_error_type

Metrics must not store full query text.

## Acceptance criteria

- Audit is emitted for successful search.
- Audit is emitted for zero-result search.
- Audit is emitted for validation failure.
- Audit is emitted for unknown tool.
- Audit is emitted for provider timeout/failure.
- Audit includes classified error type.
- `/health` includes provider state.
- Repeated provider failures can make health degraded.
- Successful provider call can reset or improve provider health state.
- Metrics update on success and failure.
- Metrics do not expose full query text.
- Shared MCP health response shape remains compatible.

## Test plan

Add tests for:

- audit on successful search
- audit on zero-result search
- audit on validation error
- audit on unknown tool
- audit on provider timeout
- health after success
- health after one provider failure
- health after repeated provider failures
- health after recovery
- metrics counters on success/failure
- no full query stored in metrics

## Notes for implementer

Keep the implementation lightweight.

Health should represent recent known state. It should not perform expensive external probing by default.

Do not swallow exceptions only to write audit logs.
