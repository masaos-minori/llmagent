# [Medium] Add HTTP RAG mode fallback and `remote_empty` regression tests

## Priority
Medium

## Background
HTTP mode delegates to remote RAG via `/v1/call_tool`. The docs distinguish `remote_nonempty`, `remote_empty`, and `in_process_fallback`. Existing tests confirm the canonical endpoint and result parsing, but fetched tests do not show coverage for `remote_empty` or fallback classification.

Evidence:
- HTTP mode docs define `remote_empty` as success, not fallback.
- `tests/test_rag_pipeline_service.py` confirms `/v1/call_tool` and result parsing.

## Goal
Lock down HTTP RAG result classification and fallback behavior.

## Scope
Affected files:
- `tests/test_rag_pipeline_service.py`
- `tests/test_rag_pipeline_mcp_service.py`
- `scripts/rag/pipeline_service.py`
- `scripts/rag/pipeline.py`
- `docs/03_rag_03_query_pipeline.md`

## Implementation Tasks
- Add test for HTTP 200 with empty or missing `result`:
  - returns `""`
  - does not trigger in-process fallback.
- Add test for HTTP 4xx:
  - no retry
  - returns `None`
  - marks fallback reason.
- Add test for HTTP 5xx after retries:
  - returns `None`
  - fallback reason set.
- Add test for JSON parse error.
- Verify diagnostics include:
  - `result_source`
  - `http_result_kind`
  - `remote_status_code`
  - `remote_latency_ms`
  - `fallback_reason`

## Open Questions / Unknowns
- Need full `pipeline.py` HTTP-mode classification logic.
- Need to decide whether production can disable in-process fallback.
- Need to confirm whether `remote_empty` should emit an info log or debug-only log.

## Acceptance Criteria
- `remote_empty` is tested as success.
- HTTP error paths are tested as fallback triggers.
- Diagnostics make remote/local/fallback distinction clear.

## Validation
- Run `uv run pytest tests/test_rag_pipeline_service.py tests/test_rag_pipeline_mcp_service.py -v`.
