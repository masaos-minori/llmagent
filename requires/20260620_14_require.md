# Title
Harden plugin loading and plugin tool shadowing behavior for production use

## Summary
Make plugin loading fail fast in controlled environments and prevent accidental or invisible shadowing of MCP tools by plugin tools.

## Background
The current plugin system loads plugin files in fail-open mode by default.
Import failures are logged and skipped.
Plugin tools are evaluated before MCP tools and may shadow MCP tools depending on configuration.

## Problem
This weakens reproducibility, makes startup behavior less trustworthy in production/CI, and creates a hidden control-plane split between MCP tools and plugin tools.

## Required Changes
- Make strict plugin loading the default or strongly enforced behavior in production and CI-like environments.
- Reject MCP-name conflicts by default.
- Allow plugin override of MCP tools only through explicit opt-in.
- Expose plugin load results (loaded / failed / skipped / overridden) through operator-visible status output.

## Acceptance Criteria
- Plugin import failure is not silently ignored in strict environments.
- MCP tool shadowing cannot occur accidentally.
- Operators can inspect plugin load status without reading raw logs only.

## Out of Scope
- Removing the plugin system entirely
- Replacing the decorator-based plugin API

## AI Implementation Instruction
Implement this as a safety and operability hardening task.

Strict rules:
- Do not weaken MCP tool visibility or safety controls.
- Preserve plugin support, but make risky behavior explicit and controlled.
- Favor fail-fast in controlled environments over silent partial startup.
- Keep override behavior opt-in, not implicit.
- If compatibility trade-offs remain, document them and mark unresolved points as: Needs confirmation

Deliverables:
1. Safer plugin loading defaults or environment-aware enforcement
2. Explicitly controlled MCP tool override behavior
3. Operator-visible plugin load status reporting
