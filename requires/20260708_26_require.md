# H-4: Remove deferred auth_token reload for MCP servers

## Priority

High

## Background

`auth_token` is copied into `HttpTransport` at initialization. Updating only `ctx.cfg.mcp.mcp_servers[key].auth_token` during reload does not update existing transports.

The current deferred behavior makes runtime config look updated while actual HTTP Authorization headers still use the old transport snapshot.

## Target files

- `agent/services/config_reload.py`
- reload command tests
- security-related tests if present

## Implementation tasks

- Remove runtime assignment of `auth_token` during reload.
- Do not execute behavior equivalent to:
  - `old_srv.auth_token = new_srv.auth_token`
- Remove deferred reporting for MCP auth token changes.
- Detect `auth_token` differences.
- Add `mcp/<server_key>.auth_token` to `ConfigReloadOutcome.needs_restart`.
- Do not add auth token changes to `ConfigReloadOutcome.deferred`.
- Do not add auth token changes to `ConfigReloadOutcome.applied`.

## Acceptance criteria

- `auth_token` changes require full agent restart.
- `/reload` reports auth token changes as `[RESTART]`.
- `/reload` never reports MCP auth token changes as `[DEFER]`.
- Running config auth token remains unchanged after `/reload`.
- Existing transports continue to use startup auth token until restart.

## Test plan

- Reload with changed `auth_token`.
- Verify `needs_restart` contains `mcp/<server_key>.auth_token`.
- Verify `deferred` is empty for MCP auth token changes.
- Verify running config auth token is unchanged.
- Verify `/reload` output contains `[RESTART]`, not `[DEFER]`.
