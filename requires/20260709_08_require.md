# [High] Add production hardening and backward-compatibility validation

## Labels
priority/high, area/config-validation, area/security, production-hardening, backward-compatibility

## Summary
Add a dedicated production validation layer that detects unsafe or compatibility-oriented settings and makes them fatal in production where appropriate.

This issue consolidates production validation for:

- non-strict startup settings
- `tool_safety_tiers` consistency
- GitHub MCP `allowed_repos_mode="fail_open"`
- legacy `use_tool_dag=false`
- `allowed_tools=[]` allow-all visibility

## Background
The current configuration supports several local/development compatibility modes. These are useful in development but should be explicit, audited, and in some cases rejected in production.

Current `agent.toml` uses `security_profile="local"`, `plugin_strict=false`, `tool_definitions_strict=false`, `allowed_tools=[]`, and empty MCP `auth_token` values, which confirms that local/dev compatibility modes are actively used in current configuration. 【1-14e820】

`plugin_strict=false` allows plugin load failures to be logged and skipped instead of failing startup; `plugin_strict=true` raises an aggregated `PluginLoadError`. 【2-a4e3df】

`tool_definitions_strict=false` and `routing_drift_strict=false` downgrade startup validation problems to warnings rather than fatal errors. 【3-ad9603】【4-8938fe】

`allowed_repos_mode="fail_open"` allows all repositories when `allowed_repos=[]` and is explicitly documented as local/development backward compatibility while prohibited in production. 【1-62b04c】【3-5d2763】

`allowed_tools=[]` means the tool whitelist layer allows all tools. Other approval and safety controls still apply separately. 【1-5c05eb】

`use_tool_dag=false` is documented as legacy non-production scheduling without resource-scoped parallelism. 【3-ad9603】【4-8938fe】

## Required Changes

### 1. Add ProductionConfigValidator
Create a dedicated validator:

- `shared/production_config_validator.py`
- `tests/test_production_config_validator.py`

Do not merge production hardening checks into `RagConfigValidator`.

### 2. Enforce strict startup settings in production
When `security_profile == "production"`, fail startup if:

- `plugin_strict=false`
- `tool_definitions_strict=false`
- `routing_drift_strict=false`

Local/development behavior must remain warning-only or permissive.

### 3. Validate tool_safety_tiers bidirectionally
Validate both directions:

- registered tool exists but has no safety tier
- safety tier key exists but is not a registered tool

Behavior:

- local/development: warning
- production: fatal `RuntimeError`

### 4. Validate GitHub MCP fail-open mode
When GitHub MCP config has `allowed_repos_mode="fail_open"`:

- production: fatal `RuntimeError`
- local/development: warning

Keep runtime GitHub MCP behavior unchanged.

### 5. Validate legacy tool scheduling mode
When `use_tool_dag=false`:

- production: fatal or startup audit error
- local/development: warning

Do not remove `use_tool_dag` in this issue.

### 6. Make allowed_tools=[] visible
When `allowed_tools=[]`, emit an explicit startup audit message that the tool whitelist layer is in allow-all mode.

Decide and document whether production treats this as:

- warning only
- policy violation

### 7. Integrate into startup
Call the new validator in the startup path before the REPL becomes available.

Validation failures in production must be startup-fatal.

## Tests
Add tests for:

- production + `plugin_strict=false` => fatal
- production + `tool_definitions_strict=false` => fatal
- production + `routing_drift_strict=false` => fatal
- production + `allowed_repos_mode="fail_open"` => fatal
- local + `allowed_repos_mode="fail_open"` => warning
- missing `tool_safety_tiers` entry in production => fatal
- unknown `tool_safety_tiers` key in production => fatal
- local missing/unknown safety tier cases => warning
- production + `use_tool_dag=false` => fatal or startup audit error
- local + `use_tool_dag=false` => warning
- `allowed_tools=[]` emits explicit audit/validation output

## Acceptance Criteria
- `ProductionConfigValidator` exists and is used at startup.
- Production rejects unsafe or compatibility-oriented settings where specified.
- Local/development compatibility remains available.
- Missing and unknown `tool_safety_tiers` keys are detected.
- GitHub `fail_open` remains available only outside production.
- `allowed_tools=[]` is visible in startup audit.
- `use_tool_dag=false` is clearly isolated as legacy non-production behavior.
- Tests cover production and local/development behavior.
