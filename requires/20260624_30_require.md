# Resolve session-title generation failure behavior and document one fallback policy

## Summary
Define one authoritative behavior for session title generation failure.

## Background
The current docs disagree on whether session-title generation failure:
- leaves the title empty, or
- always falls back to a title derived from the first user input.

This affects UX, persistence guarantees, and session discoverability.

## Objective
Choose and document one final behavior for:
- asynchronous title generation
- failure handling
- fallback title generation
- user-visible behavior
- persistence guarantees

## In Scope
- `05_agent_04_state-and-persistence.md`
- `05_agent_90_inconsistencies_and_known_issues.md`
- Any cross-reference in `05_agent_03_turn-processing-flow.md` or `05_agent_13_reference-api.md`

## Required Changes
- Compare all session-title descriptions
- Define the final failure model
- If fallback is intended:
  - specify truncation rule
  - specify persistence timing
  - specify logging behavior
- If no fallback is intended:
  - specify visible outcome clearly
- Update all affected documents to match the same behavior

## Acceptance Criteria
- There is one documented session-title failure policy
- The docs state whether a session title is guaranteed to become non-empty
- The asynchronous nature of title generation is still documented
- The known-issues file no longer conflicts with the main state/persistence document

## Out of Scope
- Replacing background-task title generation
- Adding user prompts for title correction

## Notes
Use explicit behavioral language. Avoid describing both possibilities as if they were simultaneously valid.
