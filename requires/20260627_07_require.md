## Summary
Verify that RRF disabled mode and Refiner fallback diagnostics behave as documented, and add tests to prevent silent degradation.

## Background
The query pipeline documentation now describes `use_rrf=False` as a significant quality degradation with `fusion_mode="dedup_only"` and debug warnings. It also describes Refiner fallback reasons and counters.

## Problem
These observability guarantees are valuable only if implementation and tests enforce them. Otherwise, no-RRF mode and Refiner failures can silently degrade output quality.

## Required Changes
- Verify `use_rrf=False` emits:
  - `fusion_mode="dedup_only"`
  - debug text indicating rank signal is disabled
  - log or startup warning if documented
- Verify `use_rrf=True` emits `fusion_mode="rrf"`.
- Verify Refiner fallback diagnostics include:
  - `refiner_returned_empty`
  - `refiner_exception`
  - `fallback_reasons`
  - `refiner_fallback_count`
  - `refiner_exception_count`
- Add tests for empty Refiner output.
- Add tests for Refiner exception.
- Add tests for no-retry behavior if documented and feasible.

## Acceptance Criteria
- No-RRF mode is visible in diagnostics and debug output.
- Refiner fallback is visible in diagnostics and stage results.
- Refiner failures are not silent.
- Tests protect documented observability behavior.

## Out of Scope
- Redesigning RRF scoring.
- Adding Refiner retries unless separately approved.
- Changing the Refiner prompt.

## AI Implementation Instruction
Implement this as a diagnostics verification and regression-test task.

Strict rules:
- Do not hide no-RRF mode as normal success without context.
- Do not let Refiner failures disappear without diagnostics.
- Preserve raw fallback as degraded but safe behavior.
- If startup warning behavior is not implemented, either implement it or update docs and mark as: `Needs confirmation`.
