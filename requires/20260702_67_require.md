# M-2: Clarify non-destructive compatibility repair policy in schema creation

## Priority

Medium

## Labels

docs, db, schema, maintenance

## Background

`create_schema.py` creates current schemas and also applies limited compatibility repair helpers.

Current helper functions include:

- `_migrate_rag_schema`
- `_migrate_add_undone_column`
- `_migrate_session_schema`
- `_migrate_workflow_schema`

The module states that destructive migrations are not performed, but the presence of `_migrate_*` helpers can be confused with a broader migration policy.

## Goal

Clarify that schema creation may apply non-destructive, idempotent compatibility repairs, while destructive migrations remain out of scope.

## Scope

### In scope

- Update comments and docs around `create_schema.py`
- Clarify what compatibility repair means
- Clarify what is not supported

### Out of scope

- Removing existing `_migrate_*` helpers
- Adding destructive migrations
- Changing schema DDL

## Required documentation

Document the following policy:

- `create_schema()` creates current schema objects with `IF NOT EXISTS`
- Limited compatibility repairs may add missing columns or rebuild specific local tables when safe
- Destructive data migrations are not performed
- Large schema transitions should be handled as explicit maintenance tasks

## Acceptance criteria

- The schema initialization policy is clear
- Readers understand why `_migrate_*` helpers exist
- Docs do not imply full migration framework support
- Docs do not imply that all migrations have been removed

## Validation

Review documentation and module comments.

No runtime behavior change required.

## Affected files

- `scripts/db/create_schema.py`
- DB architecture or operations documentation

## Dependencies

None
