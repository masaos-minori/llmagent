# H-1: Replace MCP runtime reload with restart-required classification

## Priority

High

## Background

MCP server definitions must be treated as restart-time snapshots.

The current reload path mutates running MCP server config in `ConfigReloadService._apply_mcp_url_reload()`. It applies HTTP URL changes at runtime and stores `auth_token` / `startup_mode` as deferred changes.

This creates inconsistent runtime state because ToolExecutor and HttpTransport are initialized from server configs at startup, while watchdog and lifecycle code may observe later mutations.

## Target files

- `agent/services/config_reload.py`

## Implementation tasks

- Replace `_apply_mcp_url_reload()` with a classification-only function such as `_classify_mcp_server_changes()`.
- Do not apply MCP server definition changes at runtime.
- Remove all direct mutations of existing `McpServerConfig` objects:
  - `old_srv.url = new_srv.url`
  - `old_srv.auth_token = new_srv.auth_token`
  - `old_srv.startup_mode = new_srv.startup_mode`
- Use `_build_mcp_servers(new_cfg)` only to build a comparison model.
- Report MCP server definition changes through `ConfigReloadOutcome.needs_restart`.
- Do not append MCP server entries to `ConfigReloadOutcome.applied`.
- Do not append MCP server entries to `ConfigReloadOutcome.deferred`.

## Acceptance criteria

- `/reload` does not mutate `ctx.cfg.mcp.mcp_servers`.
- MCP server definition changes are reported only as restart-required.
- No MCP server change is reported as applied at runtime.
- No MCP server change is reported as deferred.
- Existing LLM, RAG, tool cache, SSE, memory, and approval reload behavior remains unchanged.

## Test plan

- Change an MCP server URL and run reload classification.
- Verify `needs_restart` contains the changed MCP field.
- Verify `applied` contains no MCP server entry.
- Verify `deferred` contains no MCP server entry.
- Verify the original `ctx.cfg.mcp.mcp_servers` object and field values remain unchanged.
