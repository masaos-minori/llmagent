# Title
Unify transport error handling and HealthRegistry update path

## Priority
High

## Labels
- bug
- mcp
- transport
- health-registry
- watchdog
- reliability

## Dependencies
- None

## Summary
Ensure transport failures are consistently classified and always update the MCP server health registry.

## Background
The protocol docs describe `HttpTransport` as catching HTTP/request errors and returning an error result. The routing docs describe a flow where `TransportError` is raised and `ToolExecutor` calls `HealthRegistry.record_failure()`.

## Problem
If transport failures are swallowed into normal tool results without health updates, the watchdog and circuit-breaker may not reflect server availability.

## Required Changes
- Inspect `shared/tool_executor.py`.
- Determine whether transport failures are represented as:
  - exceptions, or
  - `ToolCallResult(is_error=True, error_type="transport")`.
- Ensure `ToolExecutor` always calls `HealthRegistry.record_failure(server_key)` for transport failures.
- Ensure tool-level errors call `record_success()` or otherwise do not degrade server health.
- Define behavior for:
  - HTTP 4xx
  - HTTP 5xx
  - timeout
  - connection refused
  - DNS/network errors
  - malformed responses
- Update docs to describe the actual flow.
- Add tests for DEGRADED, UNAVAILABLE, HALF_OPEN, and recovery transitions.

## Acceptance Criteria
- Transport failures affect HealthRegistry state.
- Tool-level errors do not mark the server unavailable.
- Watchdog restart decisions are based on transport availability.
- Docs and implementation describe the same failure flow.
- Tests cover success, tool error, transport error, and recovery.

## Out of Scope
- Restarting servers on tool-level errors.
- Changing tool validation behavior.

## AI Implementation Instruction
Implement this as a transport reliability and health-state correction.

Strict rules:
- Do not classify reachable tool errors as transport failures.
- Do not allow transport failures to bypass HealthRegistry.
- Preserve `request_id=""` for failures where no server response was received.
- If the current implementation uses a different error model, align docs and tests to it.
