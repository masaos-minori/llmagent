## Summary
Add lightweight CI checks to prevent MCP documentation drift in startup mode values, tool counts, routing authority, fail-open/fail-closed policies, and known issues.

## Background
Several MCP inconsistencies are documentation drift rather than isolated implementation bugs. Similar drift can reappear without automated checks.

## Problem
Manual review is not enough to keep docs, registry, config, and operations guidance synchronized.

## Required Changes
- Add a docs consistency check script or CI job.
- Check that documented startup modes are valid.
- Check registry-derived tool count against documented count if feasible.
- Detect unsupported config values such as `startup_mode="external"` if not implemented.
- Detect fail-open wording for `workflow_allowlist`.
- Detect stale routing authority language if ToolRegistry is canonical.
- Check that active inconsistencies are represented in known issues.
- Document how to run checks locally.

## Acceptance Criteria
- CI fails when unsupported startup modes appear in docs.
- CI fails when MCP tool counts drift from registry-derived counts.
- CI detects reintroduced fail-open wording for cicd workflow allowlist.
- CI remains fast and developer-friendly.
- Developers can run the check locally.

## Out of Scope
- Full documentation generation.
- Deep semantic validation of all documentation.

## Components
- CI workflow
- optional `scripts/check_mcp_docs_consistency.py`
- `04_mcp_*.md`
- tool registry/config files

## AI Implementation Instruction
Implement this as a documentation quality gate.

Strict rules:
- Keep checks lightweight and deterministic.
- Avoid brittle checks that fail on legitimate historical notes unless they are marked active guidance.
- Prefer explicit allowlists for intentional exceptions.
- Update docs before enabling CI if current docs fail.
