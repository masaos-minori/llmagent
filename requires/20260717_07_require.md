# Use RuntimeToolRegistry for policy and approval classification

## Priority

High

## Target files

- agent/tool_policy.py
- agent/tool_approval.py
- agent/repository_gateway.py
- shared/runtime_tool_registry.py
- tests/agent/test_tool_policy.py
- tests/agent/test_tool_approval.py
- tests/agent/test_repository_gateway.py

## Reason

Policy classification currently depends on static tool sets such as WRITE_TOOLS, DELETE_TOOLS, SHELL_TOOLS, and GitHub mutation tools.

With RuntimeToolRegistry, operation type and risk should be derived from runtime tool metadata and Agent policy.

## Intent

Keep RepositoryGateway as the enforcement boundary, but replace static classification inputs with RuntimeToolRegistry metadata.

## Scope

Refactor classification to use RuntimeToolRegistry for:

- operation type
- safety tier
- approval requirement
- write/delete/API-write classification

Runtime metadata should support classifications such as:

- READ
- WRITE
- DELETE
- EXECUTE
- API_WRITE
- UNKNOWN

## Safe defaults

Unknown tools should be classified conservatively:

- high risk or dangerous safety tier
- approval required
- LLM exposure disabled unless explicitly allowed

## Acceptance criteria

- tool_policy no longer primarily depends on static tool_constants.
- RepositoryGateway continues to enforce write/delete/API-write boundaries.
- Unknown tools are handled fail-safe.
