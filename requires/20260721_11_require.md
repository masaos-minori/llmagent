## Title

Unify startup config validation with reload validation and fix missing defaults

## Priority

High

## Target files

- `agent/config_builders.py`
- `agent/config_dataclasses.py`
- `agent/services/typed_validators.py`
- `agent/services/config_validators.py`
- `tests/test_agent_config*.py`
- Existing tests that call `build_agent_config()`

## Background

Runtime reload uses strict typed validators such as `_get_bool()`, `_get_int()`, `_get_list()`, and `_get_dict()`.

Startup config building still relies on implicit conversions such as:

- `bool(cfg.get(...))`
- `int(...)`
- `float(...)`
- `list(...)`
- `dict(...)`

In addition, `tool_concurrency_limits` and `tool_safety_tiers` are built using `dict(cfg.get(...))`, which can fail when the keys are omitted.

## Problem

Startup and reload validation behave differently.

This can lead to unsafe or confusing behavior:

- string `"false"` becomes `True`
- strings can be converted into lists character by character
- missing optional dictionaries can fail during startup
- security, approval, memory, and tool settings may be misinterpreted

## Reason for change

Configuration validation must be deterministic, strict, and consistent between startup and reload.

Operators should receive clear startup errors for invalid configuration rather than silent coercion or late runtime failures.

## Implementation intent

Make startup config loading as strict and predictable as runtime reload.

Also ensure minimal valid configuration can start by using correct defaults for optional dictionaries.

## Implementation instructions

- Introduce startup-safe typed config extraction helpers.
- Reuse `typed_validators.py` where appropriate, or add startup equivalents that raise `ConfigLoadError`.
- Replace implicit `bool(...)`, `list(...)`, and `dict(...)` conversions in `config_builders.py`.
- Use `{}` as the default for:
  - `tool_concurrency_limits`
  - `tool_safety_tiers`
- Validate `tool_concurrency_limits` as a mapping of string keys to positive integers.
- Validate `tool_safety_tiers` values as one of:
  - `READ_ONLY`
  - `WRITE_SAFE`
  - `WRITE_DANGEROUS`
  - `ADMIN`
- Add validators for key memory settings:
  - `memory_retention_days >= 1`
  - `memory_embed_dim >= 1`
  - `memory_embed_timeout_sec > 0`
  - `memory_max_inject_semantic >= 0`
  - `memory_max_inject_episodic >= 0`
- Include config key name, expected type or range, actual type, and actual value in errors.
- Review stale test fixture keys and remove obsolete settings where practical.

## Acceptance criteria

- Startup config validation rejects invalid types.
- Startup and reload validation semantics are consistent.
- `"false"` is rejected as a bool value instead of becoming `True`.
- Missing `tool_concurrency_limits` does not break startup.
- Missing `tool_safety_tiers` does not break startup.
- Invalid concurrency limits fail fast.
- Invalid memory numeric values fail fast.
- Minimal valid config can start successfully.
- Existing production-like config still loads successfully.

## Tests

Add or update tests for:

- Invalid bool type at startup.
- Invalid list type at startup.
- Invalid dict type at startup.
- Bool is not accepted as int.
- Missing `tool_concurrency_limits`.
- Missing `tool_safety_tiers`.
- Invalid safety tier value.
- Invalid memory retention days.
- Invalid memory embedding timeout.
- Minimal startup config.
