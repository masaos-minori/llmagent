## Title
[Medium] Document DLQ promotion as inline-on-nack plus background safety sweep

## Priority
Medium

## Labels
documentation, eventbus, dlq, operations

## Dependencies
None

## Background
The current implementation promotes events to the DLQ inline when `/nack` causes `delivery_failure_count >= max_retry`.

The background DLQ loop calls `sweep_orphans()` and acts as a safety net for events that reached the retry threshold but were not promoted inline.

Documentation should not describe the DLQ background loop as the primary promotion path.

## Scope

- `docs/06_eventbus_02_http_api_and_runtime.md`
- `docs/06_eventbus_04_dlq_offsets_and_delivery_semantics.md`

## Required Changes
- Document primary DLQ promotion path:
  - `/nack` increments `delivery_failure_count`
  - when the threshold is reached, inline promotion runs immediately
- Document background DLQ loop as safety sweep.
- Explain that non-zero `sweep_orphans()` results may indicate an inline promotion issue.
- Document that requeue does not reset `delivery_failure_count`.

## Acceptance Criteria
- DLQ promotion docs match implementation.
- Operators understand when DLQ files are created.
- The background loop role is accurately described.
