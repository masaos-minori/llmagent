## Title
[High] Define and implement the idempotency behavior for Event Bus ack endpoints

## Priority
High

## Labels
bug, api-design, eventbus, offsets, ack

## Dependencies
None

## Background
The current ack implementation updates only events where `acked_at IS NULL`. Repeated ack attempts return failure and the HTTP API returns 404 with `event not found or already acked`.

This behavior is currently covered by tests, but it is ambiguous for clients. A retrying consumer cannot distinguish an unknown event from an already-acked event.

## Scope
Update:
- `scripts/eventbus/db.py`
- `scripts/eventbus/app.py`
- `tests/test_eventbus_ack_nack.py`
- `tests/test_eventbus_concurrent.py`
- Event Bus HTTP API documentation

## Required Decision
Choose one behavior and implement it consistently:

### Option A: Keep current strict behavior
- First ack returns 200.
- Repeated ack returns 404.
- Unknown event returns 404.
- Documentation must clearly state `already acked` is treated as failure.

### Option B: Make ack idempotent, recommended
- First ack returns 200 with `acked: true`.
- Repeated ack returns 200 with `already_acked: true`.
- Unknown event returns 404.
- This is safer for retrying consumers.

## Recommended Implementation
Use Option B.

## Required Changes for Option B
- Add DB-level logic to distinguish:
  - event does not exist
  - event exists and is already acked
  - event exists and was newly acked
- Return:
  - `{"event_id": "...", "acked": true, "seq": N}` for first ack
  - `{"event_id": "...", "acked": true, "already_acked": true, "seq": N}` for repeated ack
- Decide offset behavior for repeated ack:
  - if `consumer_id` is provided, writing the same offset again is acceptable
- Update existing tests that expect only one successful ack.

## Tests to Add or Update
- `test_ack_first_time_returns_200`
- `test_ack_repeated_returns_200_already_acked`
- `test_ack_unknown_event_returns_404`
- Update concurrent ack test to accept idempotent repeated ack behavior if Option B is selected.

## Acceptance Criteria
- Ack behavior is explicitly defined and tested.
- Unknown event and already-acked event are distinguishable.
- Consumer retry behavior is safe and documented.
