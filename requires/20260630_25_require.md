# [High] Remove config `tool_names` from routing resolution

## Priority
High

## Background
`route_resolver.py` still lists config `tool_names` in the routing priority, and route resolver tests include cases where config-only tools are resolved through `tool_names`. This keeps config-driven routing compatibility alive. 【1-acccf4】【2-7f91a3】

## Goal
Make `tool_names` validation metadata only, not a routing input.

## Scope
Affected files:
- `shared/route_resolver.py`
- `shared/tool_registry.py`
- `shared/mcp_config.py`
- `tests/test_route_resolver.py`
- MCP/Agent routing documentation

## Implementation Tasks
- Remove all `McpServerConfig.tool_names` lookup from `ToolRouteResolver.resolve()`.
- Keep `tool_names` only for drift validation.
- Keep `validate_routing_against_config()` if useful, but ensure it is independent from routing.
- Update tests so that tools listed only in config `tool_names` do not resolve.
- Remove config-driven routing wording from documentation.
- Document `tool_names` as optional validation metadata.

## Acceptance Criteria
- Changing `tool_names` does not change routing behavior.
- A config-only custom tool fails unless it exists in live discovery or `ToolRegistry`.
- `tool_names` remains usable for drift validation.
- Active docs no longer describe `tool_names` as a routing layer.

## Validation
- Run: `pytest tests/test_route_resolver.py -v`
- Run: `rg -n "tool_names.*fallback|tool_names.*routing|config-driven" shared docs tests`
