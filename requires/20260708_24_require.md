# H-2: Prevent /reload from mutating ctx.cfg.mcp.mcp_servers

## Priority

High

## Background

`ctx.cfg.mcp.mcp_servers` is shared by multiple runtime components. ToolExecutor, lifecycle routing, watchdog, and config display all reference MCP server configuration in different ways.

Reload-time mutation is unsafe because some components use startup snapshots while others read live config references.

## Target files

- `agent/services/config_reload.py`
- tests for config reload

## Implementation tasks

- Ensure `/reload` never modifies:
  - the `ctx.cfg.mcp.mcp_servers` dictionary
  - any existing `McpServerConfig` object inside that dictionary
- Treat new MCP config built from `_build_mcp_servers(new_cfg)` as comparison-only.
- Do not add new server keys to running config.
- Do not remove deleted server keys from running config.
- Do not update existing server fields.
- Add tests proving all relevant fields remain unchanged after reload:
  - `transport`
  - `url`
  - `startup_mode`
  - `healthcheck_mode`
  - `call_timeout_sec`
  - `startup_timeout_sec`
  - `tool_names`
  - `auth_token`
  - `role`
  - `cmd`
  - `env`

## Acceptance criteria

- MCP server config remains a restart-time runtime snapshot.
- Reload classification can detect changed MCP config without modifying running config.
- Watchdog and lifecycle continue to see the startup-time MCP config after `/reload`.
- `/config` continues to display the actual running MCP config, not pending restart config.

## Test plan

- Capture all `ctx.cfg.mcp.mcp_servers` field values before reload.
- Reload with changed URL, token, startup mode, command, env, timeouts, and tool names.
- Verify field values after reload are identical to before reload.
- Verify restart-required entries are still reported.
