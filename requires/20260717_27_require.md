# [High] Fix MDQ freshness and search result correctness

## Priority

High

## Summary

Fix MDQ freshness detection and clarify search result limiting/counting behavior.

This issue consolidates:

- stale index timestamp comparison in `outline()`
- `search_docs.limit` and `max_results_limit` semantics
- displayed result count versus total matched count

## Target files

- `scripts/mcp_servers/mdq/service.py`
- `scripts/mcp_servers/mdq/search.py`
- `scripts/mcp_servers/mdq/health_check.py`
- `scripts/mcp_servers/mdq/models.py`
- `scripts/mcp_servers/mdq/tools.py`
- `tests/test_mdq_service.py`
- `tests/test_mdq_stale_detection.py`

## Background

MDQ stores file modification timestamps as `mtime_ns`, which are nanoseconds.

MDQ stores `indexed_at` using `time.time()`, which is seconds.

The health check compares these values by converting `indexed_at` to nanoseconds.

However, `MdqService.outline()` compares `mtime_ns` directly to `indexed_at`, mixing units.

Search result limiting also has overlapping controls:

- `SearchDocsRequest.limit`
- request-level `max_results_limit`
- service-level `max_results_limit`

The SQL query uses `req.limit`, then the formatted output applies additional truncation.

Additionally, `_search_docs_structured()` reports `total = len(results)` after SQL limit is applied. This can make the displayed "found" count represent retrieved count rather than true total matches.

## Reason for change

Incorrect stale detection can produce misleading warnings.

Ambiguous search limit semantics make it hard for users and agents to understand how many results are requested, retrieved, displayed, and capped.

Misleading total counts can cause downstream agents to make poor decisions about whether additional narrowing or chunk retrieval is needed.

## Intent

Make freshness and search result reporting accurate and predictable.

The intended behavior is:

- stale detection uses consistent timestamp units everywhere
- search result limits have clear responsibilities
- output distinguishes displayed results from total matches, or avoids claiming exact totals when not computed

## Required changes

### 1. Fix stale timestamp comparison

- Extract stale detection into a shared helper.
- Use `mtime_ns > int(indexed_at * 1e9)` or equivalent.
- Use the same helper in `health_check.py` and `service.py`.

### 2. Clarify search limit semantics

Define clear semantics for:

- `limit`: requested number of results
- service config `max_results_limit`: server-side hard cap
- request-level `max_results_limit`: decide whether to keep it, rename it, or deprecate it

Use a single effective SQL limit.

Suggested behavior:

- `effective_limit = min(requested_limit, service.max_results_limit)`

If request-level `max_results_limit` remains, document exactly how it interacts with `limit`.

### 3. Improve count reporting

Choose one approach.

Option A: exact count

- run a count query for the FTS match
- return both `matched_count` and `shown_count`

Option B: honest bounded count

- avoid exact count query
- change wording so output does not imply exact total
- example: "Showing 10 matching results" instead of "10 found"

## Acceptance criteria

- `outline()` does not show stale warning immediately after indexing an unchanged file.
- `outline()` shows stale warning after the file is modified after indexing.
- `/health` and `outline()` use the same stale logic.
- SQL result limiting uses a clearly defined effective limit.
- Default search behavior is documented.
- Result output does not misrepresent a limited result set as exact total matches.
- Tests cover stale and non-stale cases.
- Tests cover default limit, explicit limit, config cap, and truncation wording.

## Test plan

Add tests for:

- index file and immediately call `outline()`
- index file, modify it, then call `outline()`
- health stale count for unchanged and changed files
- default search limit
- explicit `limit`
- `limit` greater than config cap
- search with more matches than displayed
- output header wording
- truncation message consistency

## Notes for implementer

Do not add database migration.

If exact total count is expensive, prefer honest wording over inaccurate totals.
