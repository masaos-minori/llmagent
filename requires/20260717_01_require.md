# Remove plugin subsystem completely

## Priority

High

## Target files

- shared/plugin_registry.py
- shared/plugin_auto_discover.py
- shared/plugin_registries.py
- shared/plugin_result.py
- shared/plugin_tool_invoker.py
- agent/commands/cmd_plugins.py
- agent/commands/registry.py
- agent/commands/command_defs_list.py
- agent/commands/__init__.py
- agent/config_dataclasses.py
- agent/config_builders.py
- agent/services/config_validators.py
- agent/services/config_reload.py
- shared/production_config_validator.py
- shared/transport_dto.py
- config/agent.toml
- plugin-related tests
- plugin-related docs
- plugins/

## Reason

Plugin support is being fully removed.

The current plugin subsystem provides three extension points: plugin commands, plugin tools, and plugin pipeline hooks. It also uses global plugin registries and auto-discovery by importing `plugins/*.py`. Keeping any part of this system would leave unsupported runtime extension paths in the application.

Plugin tools also bypass MCP routing entirely and execute local Python functions directly. This conflicts with the new direction where all tools are provided by MCP servers and indexed through RuntimeToolRegistry.

## Intent

Remove all plugin-related runtime, configuration, command, tool execution, diagnostics, and documentation paths.

After this work, the Agent should no longer load plugins, dispatch plugin commands, execute plugin tools, expose `/plugin`, or validate plugin-specific configuration.

## Scope

Remove the following plugin infrastructure:

- Plugin registry
- Plugin auto-discovery
- Plugin internal registries
- Plugin load result types
- Plugin tool invoker
- `/plugin` command
- `_PluginsMixin`
- `CommandRegistry._dispatch_plugin()`
- Plugin command fallback in `CommandRegistry.dispatch()`
- `plugin_strict`
- `plugin_tool_override`
- plugin-related config validators
- plugin-related production validation
- plugin-related docs and tests

Update `ToolCallResult` documentation and tests to remove active plugin-specific semantics:

- `source="plugin"`
- `error_type="plugin_contract"`

## Implementation notes

CommandRegistry should dispatch built-in commands only. If no built-in command matches, `dispatch()` should return `False`.

ToolExecutor should become MCP-only. It should not import or call PluginToolInvoker.

Configuration reload should no longer report or validate plugin settings.

## Acceptance criteria

- Application starts without any plugin modules.
- CommandRegistry has no plugin imports.
- `/plugin` is removed from command definitions and help output.
- ToolExecutor has no plugin execution path.
- Plugin config fields are no longer accepted.
- Plugin-specific tests are removed or migrated.
- Documentation no longer describes plugin support.
