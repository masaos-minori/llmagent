## Dependencies

## Summary
Define and enforce consistent semantics for MCP `/health` response fields: `status`, `ready`, and `dependencies`.

## Background
The docs show a common health shape with `status`, `ready`, `dependencies`, and `details`. Some examples show dependency problems while still returning `status="ok"` and `ready=true`.

## Problem
It is unclear whether dependency failures should degrade status, set `ready=false`, or remain informational. Watchdog and `/mcp` status behavior may become inconsistent.

## Required Changes
- Define canonical semantics:
  - `status="ok"`
  - `status="degraded"`
  - `status="unhealthy"`
  - `ready=true/false`
- Define whether dependency states such as `"not configured"` or `"check failed"` affect readiness.
- Apply semantics consistently across all MCP servers.
- Document whether HTTP status code is always 200 or changes on unhealthy states.
- Clarify what watchdog uses:
  - HTTP response status
  - body `status`
  - body `ready`
- Update `/mcp` status display logic if needed.
- Add health tests for each server category.

## Acceptance Criteria
- Operators can determine readiness from `/health`.
- Watchdog behavior is documented and matches implementation.
- Dependency failures are not silently treated as healthy unless explicitly documented.
- Health examples match actual semantics.

## Out of Scope
- Replacing all health endpoints with a new API.
- Adding external monitoring integrations.
