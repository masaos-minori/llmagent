# Implement MCP tool discovery and build RuntimeToolRegistry at startup

## Priority

High

## Target files

- agent/services/mcp_tool_discovery.py
- agent/startup.py
- agent/factory.py
- agent/context.py
- agent/repl_health.py
- shared/runtime_tool.py
- shared/runtime_tool_registry.py
- shared/mcp_config.py
- tests/agent/services/test_mcp_tool_discovery.py
- tests/agent/test_startup.py

## Reason

The Agent already probes MCP `/v1/tools` for validation, but the result is not used as the execution index. Tool discovery should become the source of RuntimeToolRegistry construction.

RuntimeToolRegistry must be built once at startup. `/reload` must not rediscover tools or rebuild the registry.

## Intent

Make startup discovery explicit and authoritative for runtime tool indexing.

## Startup flow

Update startup to perform:

1. Load Agent configuration.
2. Build AgentContext.
3. Build ToolExecutor, Transport, and Lifecycle.
4. Start MCP subprocess servers.
5. Check MCP readiness.
6. Discover MCP tools via `/v1/tools`.
7. Build RuntimeToolRegistry.
8. Detect duplicate tool names.
9. Apply Agent policy.
10. Store RuntimeToolRegistry in AppServices.
11. Continue to prompt and REPL setup.

## Discovery validation

Validate each MCP `/v1/tools` response:

- response is JSON object
- `tools` is a list
- each tool has non-empty `name`
- each tool has `description`
- each tool has an object schema via `inputSchema` or equivalent
- optional `status`, `is_write`, `requires_serial`, and `resource_scope` are type-checked

## Duplicate tool rule

If multiple MCP servers expose the same tool name:

- production: fatal startup error
- local: warning is acceptable, but automatic priority selection should be avoided

## Acceptance criteria

- RuntimeToolRegistry exists before the REPL loop starts.
- RuntimeToolRegistry is built from MCP `/v1/tools`.
- Startup reports malformed tool schemas clearly.
- Duplicate tool names are detected during registry construction.
- `/reload` does not perform MCP tool discovery.
