# Title
Implement freshness detection for local `file://` ingestion

## Summary
Add incremental change detection for local `file://` ingestion so changed files are reprocessed without requiring `--force`.

## Background
Local file ingestion is supported, but current behavior still requires `--force` for re-embedding after file changes.
The docs already describe a proposed mtime + SHA-256 strategy, but it is not implemented yet.

## Problem
This makes local-file workflows error-prone because changed files are easy to miss unless operators remember to use `--force`.

## Required Changes
- Implement freshness detection for local `file://` targets using file metadata and/or content hashing.
- Reuse existing metadata columns where appropriate for local-file freshness tracking.
- Detect unchanged local files and skip re-embedding safely.
- Detect changed local files and trigger re-crawl, re-chunk, and re-embed automatically.
- Expose local-file freshness outcomes in logs or operational status.

## Acceptance Criteria
- Local file changes can be detected without `--force`.
- Unchanged local files are skipped safely.
- Changed local files are re-ingested automatically.
- Runtime output clearly distinguishes skipped vs changed vs re-ingested local files.

## Out of Scope
- Remote HTTP freshness logic
- Unrelated schema redesign

## AI Implementation Instruction
Implement this as an incremental-ingestion and operator-safety task.

Strict rules:
- Do not break existing `--force` behavior.
- Prefer deterministic freshness checks (mtime and/or hash), not heuristics.
- Reuse existing metadata columns if that remains design-consistent.
- Do not silently accept stale content when a file clearly changed.
- If storage semantics for mtime or content hash remain ambiguous, mark them as: Needs confirmation

Deliverables:
1. Freshness detection for `file://` ingestion
2. Clear runtime logs/status for local file decisions
3. Updated docs reflecting the implemented behavior
