# Title
Refactor `cmd_tooling.py` to use typed DTOs for tool result decoding

## Summary
Replace raw `dict[str, Any]` handling in `cmd_tooling.py` with typed DTOs for decoded tool arguments and tool result views.

## Background
The current plan identifies `_decode_args()` and `_to_tool_result_view()` as weakly typed and still dependent on raw dict structures.

## Problem
Raw dict usage at this boundary makes validation weak and hides the true contract of tool result decoding.

## Required Changes
- Add a typed `DecodedArgs` wrapper for decoded tool arguments.
- Change `_decode_args(raw: str | None)` to return `DecodedArgs`.
- Validate decoded JSON values and reject unsupported runtime types.
- Change `_to_tool_result_view()` to accept a typed row object instead of a raw dict.
- Introduce `ToolResultRow` if needed for this conversion path.

## Acceptance Criteria
- `_decode_args()` no longer returns a raw `dict[str, Any]`.
- `_to_tool_result_view()` no longer depends on an untyped dict input.
- Type validation is explicit and failure behavior is predictable.

## Out of Scope
- Refactoring tool execution itself
- Changing stored tool-result persistence format

## Components
- `scripts/agent/commands/cmd_tooling.py`
- `scripts/agent/commands/models.py` or related DTO module

## AI Implementation Instruction
Implement this as a typed-boundary refactor.

Strict rules:
- Do not silently coerce unknown JSON value types into strings.
- Validate all decoded values before wrapping them.
- Keep the UI behavior stable while improving the internal contract.
- If the correct placement of `ToolResultRow` is unclear, mark it as: `Needs confirmation`
