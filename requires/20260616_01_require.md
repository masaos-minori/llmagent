# [MCP][Health] Fix McpServerHealthRegistry state transitions

## Summary
`McpServerHealthRegistry` does not transition in practice because `ToolExecutor._raw_execute()` checks `is_unavailable()` but does not call `record_failure()` or `record_success()`.

## Problem
The current implementation makes the health registry effectively dead code:
- failures do not increment the failure counter
- successful calls do not reset the counter
- `DEGRADED` and `UNAVAILABLE` states are never reached

## Scope
- `shared/tool_executor.py`
- `shared/mcp_config.py`

## Task
- add `record_failure(server_key)` after transport-level failures
- add `record_success(server_key)` after successful transport responses
- decide whether `is_error=true` should affect health state or remain separate from transport failures

## Expected Behavior
- repeated transport failures move the server from `HEALTHY` to `DEGRADED` and then `UNAVAILABLE`
- successful calls reset failure state to `HEALTHY`
- `is_unavailable()` can actually block dispatch

## Acceptance Criteria
- [ ] failure count increases on transport failure
- [ ] failure count resets on success
- [ ] `UNAVAILABLE` state is reachable after repeated failures
- [ ] dispatch is blocked when the server is marked unavailable

## Notes
Keep tool-level errors and transport-level failures clearly separated unless there is a strong reason to merge them.
