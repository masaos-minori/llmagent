## Title: [HIGH] Fix Lifecycle and Interrupt Handling for Workflow Pending Approval State

### Context
When a workflow enters a pending approval state (`WorkflowPendingApprovalError`), the context flags `ctx.workflow.approval_pending`. However, this state is currently transient (memory-only) and vulnerable to interruptions like user slash commands or REPL restarts.

### Problem Statement
1. If a user inputs a normal prompt or a non-approval slash command while a workflow is `pending_approval`, the system behavior is undefined. The state may get corrupted or stuck.
2. If the REPL restarts during a pending approval, the in-memory flag is lost, leaving stale `pending` status rows in `workflow.sqlite`.

### Requirements
- **Interrupt Guard**: Modify `Orchestrator.handle_turn()` to check if `ctx.workflow.approval_pending` is True. If a user sends any input other than `/approve` or `/reject`, block the execution and return a clear user-facing warning.
- **State Recovery**: Add a recovery routine during REPL initialization. Scan `workflow.sqlite` for `ApprovalRecord` entries where `status='pending'`. If found, re-hydrate the transient `ctx.workflow.approval_pending` flag and log a warning to prompt user action.

### Constraints
- Do NOT use arbitrary `time.sleep()` for waiting or syncing states.
- Do NOT bypass the `WorkflowEngine` lifecycle; state restoration must be done through official APIs or dedicated managers.

### Acceptance Criteria (Definition of Done)
- [ ] Mypy and Pyright type-checking passes cleanly.
- [ ] Entering a non-approval command during pending approval safely triggers a block message without changing the task state.
- [ ] Restarting the REPL automatically detects a pending database workflow and restores the context state correctly.
