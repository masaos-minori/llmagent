# [Medium] Strengthen ToolExecutor error classification tests

## Goal

Add regression tests to lock down the boundary between tool-level errors and transport-level errors.

## Background

The current implementation distinguishes:

- tool-level errors:
  - HTTP 200 response received
  - `is_error=true`
  - HealthRegistry records success
- transport-level errors:
  - HTTP failure, timeout, request error, malformed response, retry exhaustion
  - `HttpTransport` raises `TransportError`
  - `ToolExecutor` records transport failure

These behaviors are important for watchdog, health state, audit logging, and operational diagnosis.

## Scope

Tests only, unless tests reveal implementation defects.

Update:

- `tests/test_tool_executor.py`

## Test Cases

Add tests for:

- HTTP 200 + `is_error=false`
  - returns success
  - `error_type == ""`
  - HealthRegistry records success
- HTTP 200 + `is_error=true`
  - returns tool error
  - `error_type == "tool"`
  - `stat_tool_errors` increments
  - HealthRegistry records success
- HTTP 500
  - classified as transport error
  - `error_type == "transport"`
  - `stat_transport_errors` increments
  - HealthRegistry records failure
- HTTP 503 retry exhaustion
  - retries according to configured retry policy
  - final result is transport error
- timeout
  - classified as transport error
  - non-retryable according to current implementation
- malformed response
  - non-dict JSON, missing `result`, or non-bool `is_error`
  - classified as transport error
- cache hit
  - no HealthRegistry update
  - `request_id == ""`

## Acceptance Criteria

- Tool-level and transport-level errors are tested separately.
- HealthRegistry behavior is tested.
- Error counters are tested.
- Retry behavior is tested.
- Cache-hit behavior remains unchanged.

## Validation

Run:

    uv run pytest tests/test_tool_executor.py -v
