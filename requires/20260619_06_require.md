# Title
Make non-primary LLM generation parameters configurable

## Summary
Move LLM parameters for history compression, session title generation, and tool-result summarization into explicit configuration.

## Background
Normal response generation parameters are configurable, but some secondary use cases still rely on fixed constants or separately handled values.

## Problem
This creates inconsistent tunability across different LLM use cases and makes model-specific adjustment harder during operations.

## Required Changes
- Add explicit config fields for:
  - history compression temperature / max tokens
  - session title generation temperature / max tokens
  - tool result summarization temperature / max tokens
- Decide and document whether each field is hot-reloadable or restart-required.
- Ensure runtime services consume the unified configuration instead of embedded constants.

## Acceptance Criteria
- All major LLM use cases have explicit configuration.
- Behavior does not depend on hidden compile-time constants where configuration should exist.
- Documentation explains which parameters affect which use case.
- Runtime behavior matches configured values.

## Out of Scope
- Changing model provider protocols.
- Implementing per-request dynamic policy selection.

## Notes
Aim for consistency across all LLM-powered workflows.
