# Title
Make memory retrieval branch-aware and use stored `branch` context explicitly

## Priority
Medium

## Labels
- enhancement
- agent
- memory
- retrieval
- branch-awareness
- context-boundary

## Dependencies
- None

## Summary
Use the stored `branch` field as a real retrieval and injection boundary instead of leaving it as mostly passive metadata.

## Background
The memory model already stores `project`, `repo`, and `branch`.
Retriever APIs visibly use `project` and `repo`, but branch-aware retrieval is not clearly established at the same level.

## Problem
Without branch-aware filtering, decisions, failures, and temporary workarounds from one branch may leak into another branch’s working context.

## Required Changes
- Add branch-aware filtering to retrieval and injection flows.
- Define how current branch context is resolved.
- Distinguish global memories from branch-specific memories in ranking or filtering.
- Add tests that ensure unrelated branch memories are not injected into the current task.

## Acceptance Criteria
- `branch` is no longer stored-but-unused in practice.
- Retrieval and injection can explain project/repo/branch filtering clearly.
- Cross-branch memory leakage is reduced and covered by tests.

## Out of Scope
- Full git integration redesign
- Replacing memory metadata model

## Components
- `scripts/agent/memory/retriever.py`
- `scripts/agent/memory/injection.py`
- memory context resolution logic
- related tests
- memory docs

## AI Implementation Instruction
Implement this as a branch-boundary retrieval task.

Strict rules:
- Do not keep `branch` as passive metadata if boundary control is the goal.
- Do not rank global and branch-local memories identically by default.
- Define the source of current branch context explicitly.
- If current branch context cannot be resolved safely from existing design, mark it as: `Needs confirmation`
