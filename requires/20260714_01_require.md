# [High] Remove obsolete direct-execution fallback documentation and enforce workflow-required semantics

## Labels
priority/high, area/workflow, area/docs, consistency

## Summary
Update the documentation to consistently state that workflow support is mandatory.

The current specification is:

- The agent requires `config/workflows/default.json` at startup.
- If the workflow definition is missing or invalid, startup fails.
- There is no direct-execution fallback.

## Background
Some documentation still describes a fallback to the traditional direct-execution flow when `config/workflows/default.json` is missing or the workflow database is unavailable.

That fallback is obsolete. The correct behavior is startup-fatal workflow validation.

## Required Changes

- Remove all documentation that says the agent falls back to direct execution when workflow loading fails.
- State clearly that `config/workflows/default.json` is required at startup.
- State clearly that missing or invalid workflow definition causes startup failure.
- Remove any remaining references to degraded workflow mode.
- Remove any remaining references to `workflow_mode="disabled"` or `workflow_mode="auto"` as valid behavior.
- Ensure the startup validation and troubleshooting documentation uses the same wording.

## Suggested Replacement Text
The agent requires `config/workflows/default.json` at startup. If the workflow definition is missing or invalid, startup fails with a RuntimeError. There is no direct-execution fallback.

## Acceptance Criteria
- No documentation claims that workflow failure falls back to direct execution.
- Workflow startup failure behavior is described consistently across turn flow, configuration, operations, and reference docs.
- `workflow_mode` is not described as a valid runtime or config option.
- The required workflow definition path is documented.
- Troubleshooting documentation tells operators to deploy/fix the workflow definition and restart the agent.
