# H-1: Redirect HTTP MCP subprocess stderr to per-server log files

## Priority

High

## Background

`HttpServerLifecycleManager.start()` currently starts HTTP MCP subprocesses with `stderr=subprocess.PIPE`. This creates two risks:

- Startup timeout path may block when reading from `proc.stderr.read()` while the process is still alive.
- Long-running MCP servers may block if stderr output fills the pipe buffer.

The agreed approach is "Option A": redirect stderr to a per-server log file.

## Target files

- `agent/http_lifecycle.py`

## Implementation tasks

- Replace `stderr=subprocess.PIPE` with per-server stderr log file redirection.
- Use a dedicated log directory, for example `/opt/llm/logs/mcp`.
- Create a safe file name from `server_key`.
- Open the stderr log file in append-binary mode.
- Pass the file handle to `subprocess.Popen(..., stderr=stderr_file, ...)`.
- Ensure the stderr file handle is closed on:
  - startup failure
  - restart
  - shutdown
  - cleanup after already-exited processes
- Replace `StartupFailure.stderr_full` collection with tailing the per-server stderr log file.
- Limit captured stderr tail size, for example 64 KiB.

## Acceptance criteria

- `HttpServerLifecycleManager.start()` no longer uses `stderr=subprocess.PIPE`.
- Startup timeout no longer performs blocking reads from a live process pipe.
- Startup failure still includes useful stderr diagnostics from the server log tail.
- Normal long-running MCP subprocesses cannot block due to an undrained stderr pipe.
- Stderr log files are created under a predictable per-server path.
- File handles are closed during restart and shutdown.

## Test plan

- Add a fake or temporary subprocess test that writes repeatedly to stderr and verify it does not block.
- Simulate startup timeout and verify `HttpStartupError.failure.stderr_full` contains stderr log tail.
- Simulate early exit and verify stderr log tail is captured.
- Verify stderr file handle cleanup on restart and shutdown.
