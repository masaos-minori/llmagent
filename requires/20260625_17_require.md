## [High][eventbus][performance] Add high-load integration tests to verify SSE continuity and publish latency under concurrency

### Summary
The blocking-I/O issue is not just a code-style problem.
It is a runtime bottleneck risk that must be validated under concurrent load.

### Problem
Without concurrency tests, the system can regress into event-loop stalls while still passing functional tests.

### Scope
- tests/test_eventbus*.py
- publish/replay/subscribe/DLQ concurrency scenarios
- latency and stall detection

### Facts
- `/subscribe` is polling-based SSE and repeatedly queries SQLite. 【1-2f69b2】【1-6bdda3】
- `/publish` writes to SQLite and then appends JSONL. 【1-2f69b2】【1-6bdda3】
- DLQ processing is a concurrent async task using synchronous DB/file operations today. 【1-6bdda3】【1-835298】【4-1b37a6】

### Task
- add concurrent load tests with multiple subscribers and publishers
- verify SSE delivery continues during publish bursts
- verify publish latency does not spike catastrophically under subscription load
- verify DLQ loop execution does not starve API handlers
- add thresholds for acceptable lag or timeout

### Expected Result
The Event Bus has regression coverage for event-loop-stall-sensitive workloads.

### Acceptance Criteria
- tests cover concurrent publish + subscribe
- tests cover DLQ loop interaction under load
- SSE continuity is asserted
- latency regression thresholds are enforced in CI
