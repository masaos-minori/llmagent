# Enforce safe serialization for write/delete tool calls in DAG scheduler

## Priority

High

## Type

Design / Implementation / Test

## Background

The tool execution layer supports DAG-based scheduling for tool calls. The scheduler is expected to group tool calls by `resource_scope`, `is_write`, and `requires_serial`.

Write/delete/shell tools can cause destructive side effects. Therefore, tool calls that mutate the same resource must not be executed concurrently.

The expected safety rules are:

- Tools with `requires_serial=True` must run as a global serial barrier.
- `shell_run` must always run as a serial barrier.
- Write/delete tools targeting the same non-empty `resource_scope` must be serialized.
- Write/delete tools without a `resource_scope` must run in a conservative write-first batch.
- Read-only tools may run concurrently when no side-effect ordering constraint applies.

## Problem

The current design documentation and scheduler comments indicate that same-resource write tools should be serialized, but the actual implementation of `build_execution_groups()` must be verified.

If same-resource write calls are grouped in a batch that is later executed with `asyncio.gather()`, the scheduler may allow concurrent writes to the same file, directory, repository, or database resource.

## Goal

Guarantee that destructive or state-mutating tool calls are never executed concurrently when they target the same logical resource.

## Scope

Affected areas:

- `agent/tool_scheduler.py`
- Tool batch execution code that consumes scheduler output
- Tool metadata generation for `ToolSpec`
- Audit logging for tool execution rounds
- Tests for DAG scheduling and execution ordering

## Implementation Tasks

- Inspect `build_execution_groups()` in `agent/tool_scheduler.py`.
- Verify how groups and concurrent batches are consumed by the executor.
- Ensure that same-`resource_scope` write/delete calls are placed in sequential batches, not in the same concurrently gathered group.
- Ensure `requires_serial=True` creates a single-tool serial batch.
- Ensure `shell_run` always creates a serial barrier.
- Ensure write/delete calls without `resource_scope` run before read-only calls.
- Add audit metadata for scheduling decisions:
  - `resource_scope`
  - `is_write`
  - `requires_serial`
  - `scheduling_decision`
  - `trigger_tool`

## Acceptance Criteria

- Same-resource write/delete tool calls are never executed concurrently.
- `shell_run` always runs alone.
- `requires_serial=True` always runs alone.
- Read-only tool calls still run concurrently when safe.
- Audit logs show why a tool call was serialized.
- Existing non-DAG execution behavior remains unchanged when DAG scheduling is disabled.

## Validation Plan

Add tests for the following cases:

- Multiple `write_file` calls targeting the same file path.
- `move_file` and `delete_file` targeting the same directory tree.
- Multiple GitHub write operations targeting the same repository.
- `shell_run` mixed with read-only tools.
- Write tools without `resource_scope`.
- Mixed read-only tools that should still run concurrently.

## Risks

Incorrect serialization may reduce concurrency. However, correctness and safety are more important than parallelism for destructive operations.

## Out of Scope

- Changing the public tool definition schema unless required.
- Rewriting the whole tool execution layer.
- Documentation-only updates.
