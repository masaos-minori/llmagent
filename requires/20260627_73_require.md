# Consolidate workflow schema DDL into a single canonical source

## Priority
High

## Labels
- db
- schema
- workflow
- maintainability
- high-priority

## Dependencies
None

## Background
Workflow schema DDL exists in more than one place. `db/schema_sql.py` exposes `build_workflow_schema_sql()`, while `db/workflow_schema.py` also defines its own workflow DDL.

This creates schema drift risk when workflow tables are changed.

## Scope
Modify:
- `db/schema_sql.py`
- `db/workflow_schema.py`
- `db/create_schema.py`
- `tests/test_create_schema.py`

## Required Changes
- Treat `db/schema_sql.py::build_workflow_schema_sql()` as the only canonical workflow DDL source.
- Remove duplicated workflow DDL from `db/workflow_schema.py`.
- Update `workflow_schema.init_schema()` to call `build_workflow_schema_sql()`.
- Extract or share workflow schema migration logic so both paths behave consistently:
  - `workflow_schema.init_schema()`
  - `create_schema.create_workflow_schema()`
- Ensure both paths create the same tables and apply the same migrations.

## Tests
Update `tests/test_create_schema.py`.

Add or update tests for:
- `create_workflow_schema()` creates all expected workflow tables.
- `workflow_schema.init_schema()` creates the same tables.
- Both schema creation paths behave consistently.
- The test workflow DDL patch actually creates the expected tables.

Expected tables:
- `tasks`
- `attempts`
- `processed_events`
- `artifacts`
- `approvals`

## Acceptance Criteria
- Workflow DDL exists in one canonical location.
- Both workflow initialization paths produce the same schema.
- Workflow schema tests pass reliably.
- Future workflow schema changes require editing only one DDL source.
