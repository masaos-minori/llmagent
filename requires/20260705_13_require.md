# Do not silently skip security audit failures in production mode

## Priority

High

## Type

Security / Implementation / Startup Validation

## Background

The startup security audit checks MCP-related security defaults such as:

- HTTP server auth tokens
- Shell sandbox backend
- Shell command allowlist
- Git allowed repository paths
- GitHub allowed repositories
- CI/CD workflow allowlist
- GitHub force-push and PR review settings

In production mode, missing HTTP auth tokens are already treated as fatal.

## Problem

Some server-specific config load failures are silently ignored. This means production startup can continue even when the security audit could not inspect a critical MCP server configuration.

Silent security audit failures can hide dangerous states.

## Goal

Make security audit failures visible and fatal in production mode.

## Scope

Affected areas:

- `agent/repl_health.py`
- `audit_security_defaults()`
- Server config model loading
- Startup validation tests

## Implementation Tasks

- Replace broad `except Exception: pass` blocks with explicit warning or error handling.
- In production mode, treat security audit config load failures as fatal.
- In local mode, log warnings and return them in the warning list.
- Ensure `security_lockdown_enabled=True` suppresses deny-all warnings only.
- Ensure `security_lockdown_enabled=True` does not suppress:
  - config load failures
  - production auth enforcement
  - shell sandbox enforcement
  - fail-open warnings
- Review production policy for:
  - `github.allow_force_push=true`
  - `github.require_pr_review=false`

## Acceptance Criteria

- Config load failure for shell security audit is fatal in production.
- Config load failure for git security audit is fatal in production.
- Config load failure for GitHub security audit is fatal in production.
- Config load failure for CI/CD security audit is fatal in production.
- Local mode logs warnings instead of silently skipping.
- Lockdown mode does not bypass production enforcement.

## Validation Plan

Add tests for:

- Production mode with missing GitHub config.
- Production mode with invalid Git config.
- Production mode with invalid CI/CD config.
- Local mode with invalid server config.
- `security_lockdown_enabled=True` with deny-all settings.
- `security_lockdown_enabled=True` with config load failure.
- Force-push enabled warning or failure behavior.

## Risks

Production startup may fail in environments that previously ignored broken security config. This is the intended safer behavior.

## Out of Scope

- Redesigning server config files.
- Removing intentional lockdown support.
- Documentation-only updates.
