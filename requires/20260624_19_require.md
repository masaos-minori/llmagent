# Define a cross-layer audit-correlation model for MCP tool calls

## Summary
Document how to correlate a single MCP tool call across agent-side logs, transport diagnostics, server-side audit logs, and health-state changes.

## Background
The protocol and operations documents describe rich observability fields, but the end-to-end correlation model is not documented as one clear tracing workflow.

## Objective
Make it possible to trace one tool call across:
- agent execution logs
- server audit logs
- transport failures
- health registry state changes
- watchdog actions

## In Scope
- `04_mcp_02_protocol_and_transport.md`
- `04_mcp_03_routing_lifecycle_and_execution.md`
- `04_mcp_06_configuration_and_operations.md`

## Required Changes
- Define the canonical correlation keys, including at least:
  - `X-Session-Id`
  - `X-Request-Id`
  - `server_key`
  - `tool_name`
- Explain where each key is generated and where it appears
- Add an operator-facing example that traces one failed and one successful tool call
- Clarify how agent-side and server-side logs relate

## Acceptance Criteria
- A reader can trace one tool call from agent dispatch to server audit line
- The correlation keys and their lifecycle are documented explicitly
- Transport errors, tool errors, and watchdog actions can be connected using the documented model

## Out of Scope
- Distributed tracing implementation
- Log backend integration changes

## Notes
Prefer a concrete example-driven explanation instead of only listing fields.
