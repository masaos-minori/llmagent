# Title
Clarify the responsibility boundary between `messages` history and `ToolResultStore`

## Summary
Document clearly that the `messages` table is the conversation-flow history, while `tool_results` is the supplementary archive for full tool output.

## Background
The current documentation implies:
- `messages` stores the LLM conversation flow, including tool-related messages
- `tool_results` stores the full tool result text
Only a summary or truncated version of the tool result appears in message history

## Problem
Without an explicit boundary, implementers may confuse these roles.
They may incorrectly treat `tool_results` as a replacement for `messages`, or they may push full tool output into conversation history.

## Required Changes
- Document that `messages` is the authoritative conversation-flow history.
- Document that `tool_results` is a supplementary full-output archive.
- Clarify that `tool_results` does not replace `messages`.
- Clarify how `/tool show <id>` relates to the archived full output.
- Clarify that the message history may contain only summary or truncated tool output.

## Acceptance Criteria
- The roles of `messages` and `tool_results` are explicit and non-overlapping.
- Implementers know to query:
  - `messages` for conversation history
  - `ToolResultStore.get(id)` for full tool output
- The documentation no longer leaves the boundary ambiguous.

## Out of Scope
- Redesigning session persistence
- Changing summarization or truncation policy itself

## AI Implementation Instruction
Implement this as a responsibility-clarification and documentation task.

Strict rules:
- Do not treat `tool_results` as a replacement for `messages`.
- Do not start storing full tool outputs in `messages` by default.
- Keep the distinction explicit in docs and user-facing command explanations.
- If an existing runtime path already mixes these roles in an unclear way, mark it as: Needs confirmation

## Notes
Safe interpretation:
- `messages` = conversation flow (what the LLM sees)
- `tool_results` = full output archive (retrievable via `/tool show <id>`)
