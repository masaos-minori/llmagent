# Title
Harden plugin loading and tool override behavior for production and CI

## Summary
Change plugin loading and plugin tool override behavior so that production and CI environments fail fast on plugin errors and avoid accidental shadowing of MCP tools.

## Background
Plugin files are currently loaded in fail-open mode by default.
Import failures are logged and skipped.
Plugin tools are checked before MCP tools and can shadow MCP behavior.

## Problem
This creates safety, reproducibility, and operational risks:
- a broken plugin may be silently ignored
- a plugin tool may unexpectedly replace an MCP tool
- runtime behavior may differ between environments without obvious signals

## Required Changes
- Make strict plugin loading the recommended or default behavior for production/CI.
- Require explicit opt-in for plugin tool override of MCP tools.
- Reject MCP-name conflicts by default.
- Add visible reporting of plugin load results (loaded / failed / skipped / overridden).

## Acceptance Criteria
- Plugin import failure can fail startup in strict environments.
- MCP tool shadowing requires explicit configuration and is clearly logged.
- Operators can inspect plugin load status without reading raw startup logs only.
- Default behavior favors safety over silent permissiveness.

## Out of Scope
- Designing a full plugin marketplace or package manager.
- Replacing the current decorator-based plugin API.

## Notes
Prefer explicit failure over silent partial startup in controlled environments.
