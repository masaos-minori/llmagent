# Title
Make `/db` command targets explicit by database scope

## Summary
Restructure or clarify `/db` commands so users can immediately tell whether an operation targets RAG, session, or workflow storage.

## Background
The system uses multiple SQLite databases, but current `/db` descriptions blur the distinction between them.

## Problem
Operators should not need implicit implementation knowledge to know which database a command affects.

## Required Changes
- Clarify or redesign `/db` commands so the target database is explicit.
- Consider scoped forms such as:
  - `/db rag ...`
  - `/db session ...`
  - `/db workflow ...`
- Update help text and docs accordingly.
- Preserve the distinction between normal usage and admin-only maintenance access.

## Acceptance Criteria
- Users can tell which DB a `/db` command affects without hidden knowledge.
- Help text and docs match actual behavior.
- Admin-only maintenance semantics remain explicit.

## Out of Scope
- Replacing SQLite
- Rebuilding the full maintenance subsystem

## AI Implementation Instruction
Implement this as an explicitness and operator-clarity task.

Strict rules:
- Do not hide database scope behind shorthand behavior.
- Preserve maintenance capabilities unless clearly deprecated.
- If backward-compatible command naming is unclear, mark it as: Needs confirmation

Deliverables:
1. Explicit database-targeting behavior
2. Updated help/docs
3. Clear maintenance-scope semantics
