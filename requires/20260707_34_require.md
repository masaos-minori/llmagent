# M-5: Align /mcp status health parsing with watchdog health parsing

## Priority

Medium

## Background

Watchdog reads `/health` body fields such as `restart_recommended` and `operator_action_required`. `/mcp status` currently extracts only `details.sandbox_backend`.

This creates inconsistent health interpretation between monitoring and status display.

## Target files

- `agent/repl_health.py`
- `agent/services/mcp_status.py`
- optionally a new shared helper module

## Implementation tasks

- Parse these `/health` response fields in MCP status:
  - `restart_recommended`
  - `operator_action_required`
  - `reason`
  - `message`
- Consider extracting common health probing logic used by both watchdog and status.
- Preserve current sandbox backend extraction.
- Populate new `McpProbeResult` fields.

## Acceptance criteria

- `/mcp status` displays restart recommendation when provided.
- `/mcp status` displays operator action requirement when provided.
- Watchdog and status use consistent interpretation of `/health` body.
- Existing availability behavior remains unchanged.

## Test plan

- Test health body with `restart_recommended=true`.
- Test health body with `operator_action_required=true`.
- Test health body with reason/message.
- Test malformed JSON still results in safe fallback.
