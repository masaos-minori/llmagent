# [Medium] Clarify migration helper responsibilities in create_schema.py

## Goal

Clarify the responsibility of `_migrate_*` helpers in `db/create_schema.py`.

## Background

`create_schema.py` currently says it creates the latest schema only and has no migration logic. However, it contains and calls migration helpers such as `_migrate_rag_schema()`, `_migrate_session_schema()`, `_migrate_add_undone_column()`, and `_migrate_workflow_schema()`. 【2-418bb2】

## Scope

Documentation and comments.
Small refactoring is allowed if it improves clarity.
Do not introduce a full migration framework in this issue.

## Implementation Tasks

- Update the module docstring in `db/create_schema.py`.
- Remove or correct the phrase `No migration logic`.
- Clarify that `create_schema.py`:
  - creates current schemas
  - applies narrowly scoped compatibility repairs
  - does not perform destructive migrations
- Document each `_migrate_*` helper:
  - `_migrate_rag_schema()`
  - `_migrate_session_schema()`
  - `_migrate_add_undone_column()`
  - `_migrate_workflow_schema()`
- Clarify that destructive schema cleanup must be handled by explicit SQL or a separate migration procedure.
- Update Shared/DB docs if they describe schema creation as DDL-only.

## Acceptance Criteria

- `create_schema.py` docstring matches actual behavior.
- Migration helper responsibilities are explicit.
- No destructive migration behavior is added.
- Docs do not contradict the implementation.

## Validation

- Review `create_schema.py` docstring and comments.
- Run schema creation tests.
