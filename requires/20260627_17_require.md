# Title
Unify new MCP tool and server registration procedure

## Priority
Medium

## Labels
- documentation
- mcp
- tool-registration
- routing
- developer-experience

## Dependencies
- Issue 2

## Summary
Create one consistent procedure for adding MCP tools and servers, aligned with the actual ToolRegistry and tool_constants model.

## Background
The current docs mix references to adding tools in `shared/tool_registry.py`, adding frozensets in `shared/tool_constants.py`, config `tool_names`, and GitHub prefix exceptions.

## Problem
AI implementers may update the wrong file or omit required registration steps, causing routing drift.

## Required Changes
- Define the canonical new-tool procedure:
  1. Add tool name to `shared/tool_constants.py`.
  2. Confirm registry auto-populates.
  3. Implement server `dispatch()`.
  4. Expose tool in `/v1/tools`.
  5. Add LLM schema to `config/tools_definitions.toml` if it should be exposed to the LLM.
  6. Add `tool_safety_tiers`.
  7. Optionally add config `tool_names` for drift validation only.
- Correct any wording that says frozensets are added to `shared/tool_registry.py`.
- Document GitHub prefix routing as an explicit exception if still supported.
- Update new-server checklist accordingly.

## Acceptance Criteria
- New-tool and new-server instructions no longer contradict routing authority.
- Required vs optional steps are clearly labeled.
- GitHub prefix exception is documented or removed.
- Tests or validation commands are included.

## Out of Scope
- Changing the routing model itself.
- Removing GitHub prefix routing unless separately approved.
