# Task: Clean Up Layering Violation in db/maintenance.py

## Overview and Context
The system architecture mandates a strict separation of data ownership boundaries between the Agent runtime sub-layer and the RAG sub-layer. Currently, `db/maintenance.py` handles database routines for `session.sqlite` but contains functions like `rotate_rag_db` and `vacuum_db` targeted directly at RAG data files. This direct file manipulation violates the modular isolation policy.

## Acceptance Criteria (AC)
- [ ] Completely extract all RAG-centric database manipulation routines (such as `rotate_rag_db`) out of `db/maintenance.py`.
- [ ] Relocate these capabilities inside the RAG layer domain under `RagMaintenanceService`.
- [ ] If the Agent CLI or an external routine needs to invoke a RAG database vacuum, it must interact via an exposed public interface of the RAG module boundary rather than accessing the database files directly.

## Technical Specifications and Implementation Steps
1. Target Files: `db/maintenance.py`, `rag/maintenance.py`
2. Step 1 (Code Extraction): Cut the definition blocks of `rotate_rag_db` and associated utility strings out of the agent module space.
3. Step 2 (RAG Implementation): Insert these routines into `rag/maintenance.py`, exposing them cleanly through `RagMaintenanceService`.
4. Step 3 (Reference Update): Refactor any administration tool macros or CLI call paths to access the new module path.

## Validation and Testing Plan
- Command: `pytest tests/db/test_maintenance_boundaries.py`
- Required Test Cases:
  - `test_agent_maintenance_has_no_rag_dependencies`: Validates via AST analysis that `db/maintenance.py` contains no structural imports or literal string expressions pointing to RAG database assets.

## Constraints and Core Rules
- Maintain encapsulation. The Agent layer must treat the RAG layer data tier as an opaque service box.
