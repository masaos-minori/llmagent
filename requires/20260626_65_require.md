## Summary
Unify the canonical DLQ requeue endpoint path and document its behavior consistently.

## Background
The DLQ semantics document uses `POST /dlq/{event_id}/requeue`, while the HTTP API document has an incomplete `POST /dlq/` entry.

## Problem
Client implementers and operators cannot know which requeue endpoint is canonical. The requeue behavior around `retry_count` and `dlq_at` must also be consistent.

## Required Changes
- Use `POST /dlq/{event_id}/requeue` as the canonical endpoint unless implementation proves otherwise.
- Update HTTP API and Reference API docs.
- Define behavior for:
  - unknown event_id
  - event not currently in DLQ
  - event already at or above `max_retry`
  - repeated requeue requests
- Keep or remove any legacy alias intentionally.
- Add tests for requeue path and retry_count behavior.

## Acceptance Criteria
- One requeue path is canonical.
- Docs and runtime agree.
- Requeue clears `dlq_at` and increments `retry_count` as documented.
- Edge cases are covered by tests.

## Out of Scope
- Redesigning DLQ promotion criteria.
- Adding automatic consumer failure tracking.
