# [Medium] Remove `/set temperature|max_tokens` and `/db`; move `/export` and `/db session` subcommands under `/session`

## Labels
priority/medium, area/agent, area/cli, cleanup

## Summary
This requirement implements a set of already-decided slash-command changes (final target
behavior specified by the user, not an open design question):
- Delete `/set temperature <f>` and `/set max_tokens <n>` entirely.
- Move `/export md [file]` → `/session export markdown [file]`.
- Move `/export json [file]` → `/session export json [file]`.
- Move `/db session stats|health|checkpoint|vacuum|purge|recover` → `/session
  stats|health|checkpoint|vacuum|purge|recover` (same argument shapes).
- Remove `/db` entirely (confirmed by the user) — once its only subcommand scope
  (`session`) moves to `/session`, `/db` has no remaining purpose and is deleted,
  not kept as a stub.
- Delete unnecessary/dead test cases (confirmed by the user) — specifically the
  pre-existing failing `/db rag` tests in `tests/test_agent_cmd_db.py` (left over
  from the incomplete `846dc93b` cleanup) and the dead `_db_stats()` method in
  `db_stats_display.py`, both of which become moot once `/db` is removed.

Investigation confirms all six affected commands exist today exactly as described.
`/db rag` was already removed on 2026-07-14 (commit `846dc93b`), leaving `/db` with
only a `session` scope — the original reason for the `session` qualifier
(disambiguating from `rag`) no longer applies, which supports both collapsing it
into `/session` and removing `/db` outright. That same prior removal left
`tests/test_agent_cmd_db.py` with 22 pre-existing failing tests testing the
already-deleted `/db rag` behavior; since the whole file is about a command that no
longer exists after this change, it is deleted rather than partially patched.

## Background
Confirmed in code (all paths read in full unless noted):

- **`/set`**: `CommandDef` at `scripts/agent/commands/command_defs_list.py:119-125`.
  Handler `_cmd_set` in `_ConfigSetMixin`
  (`scripts/agent/commands/cmd_config_set.py:66-93`) branches on exactly two params,
  `temperature` (`_set_temperature`, lines 36-49) and `max_tokens`
  (`_set_max_tokens`, lines 51-64) — no other sub-options exist, so removal has no
  spillover to other `/set` behavior (there is none).
- **`/export`**: `CommandDef` at `command_defs_list.py:91-97`. Handler `_cmd_export`
  in `_RagExportMixin` (`scripts/agent/commands/cmd_rag_export.py:27-42`) recognizes
  only `"md"` and `"json"` (line 37); any other token is treated as the output
  filename. No third format (e.g. `txt`) exists. Actual rendering is delegated to
  `scripts/agent/services/export_formatter.py`'s `render_export()`/`write_export()`
  (lines 47-52, 67-83), which are format-agnostic and can be reused unchanged.
- **`/db session ...`**: `CommandDef` for `/db` at `command_defs_list.py:112-118`
  (help string is already stale — it still advertises `rag stats|urls|clean|...`,
  which no longer exists). `_cmd_db` (`scripts/agent/commands/cmd_db.py:33-47`)
  dispatches only `session` and `help` — the `rag` branch was deleted in commit
  `846dc93b`. `_cmd_db_session` (`cmd_db.py:49-63`) dict-dispatches the six
  subcommands to `DbSessionOps` (`scripts/agent/commands/db_session_ops.py:21-69`)
  and `_db_session_stats` (`db_stats_display.py:29-38`), which call
  `DbMaintenanceService` (`scripts/agent/services/db_maintenance_service.py`,
  docstring: "maintenance operations on session.sqlite only"). None of this logic
  touches RAG or workflow tables — it is already scoped to `session.sqlite` only,
  consistent with living under `/session`.
- **`/session` already exists** and has no name collision with the incoming
  subcommands: `_SessionMixin`/`_cmd_session`
  (`scripts/agent/commands/cmd_session.py:121-152`) currently handles `list [n]`,
  `load <id>`, `rename <title>`, `delete <id>` via an if/elif chain (not dict
  dispatch, unlike `/db session`). None of `export`, `stats`, `health`,
  `checkpoint`, `vacuum`, `purge`, `recover` collide with `list`/`load`/`rename`/`delete`.
- **Removing `/set` does not remove the underlying config values.**
  `llm_temperature`/`llm_max_tokens` remain valid `LLMConfig` fields
  (`scripts/agent/config_dataclasses.py:108-109`) and are already reachable at
  runtime through `/reload` independently of `/set`:
  `scripts/agent/services/config_reload.py`'s `_apply_llm_prompt_params`
  (lines 333-338) applies both fields from `config/agent.toml` and calls the same
  `ctx.services_required.llm.apply_config(...)` path that `/set` uses. So the only
  capability lost is a single in-session quick-tweak; the value itself stays
  configurable via `config/agent.toml` + `/reload`.
- **Readline tab completion is a separate hardcoded list**:
  `scripts/agent/repl.py:61-81` (`AgentREPL.SLASH_COMMANDS`) includes `"/set"`
  (line 76) and is consumed by `CLIView.setup_readline()`
  (`scripts/agent/cli_view.py:63-88`). This list is already out of sync with
  `_COMMANDS` (missing `/plan`, `/mdq`, `/memory`, `/audit`, `/plugin`, `/skill`) and
  is the exact subject of the existing, separate requirement
  `requires/20260715_10_require.md` ("Use command metadata as the single source of
  truth for dispatch, help, completion..."), whose scope explicitly covers
  eliminating this list generically. This requirement only needs to remove the
  `"/set"` entry (and `"/db"` if item 3's Unknown resolves to removing `/db`
  entirely) — it does not need to solve the general drift problem, which is already
  tracked separately.
- **Documentation locations**: `docs/05_agent_07_08_cli-and-commands-slash-commands-session-mcp.md`
  (lines 35-38 for `/session`, line 42 for `/export`, lines 68-69 for `/set`) and
  `docs/05_agent_07_09_cli-and-commands-slash-commands-context-db.md` (lines 39-50
  document a stale `/db rag` table that no longer matches implementation; lines
  52-61 document the `/db session` table that this requirement moves).
  `docs/05_agent_07_07_cli-and-commands-migration-notes.md` (lines 42-57) still
  references `/db rag recover`/`/db rag stats` as current, which is also stale.
- **Test coverage requiring changes**: `tests/test_cmd_config_refactor.py` (entire
  file, 148 lines) is the sole test file for `/set` and should be removed with it.
  `tests/test_cmd_registry_ingest_removal.py:25-29`
  (`test_export_still_registered`) asserts `/export` exists as its own `CommandDef`
  and must be updated once it moves under `/session`. In `tests/test_agent_cmd_db.py`,
  the currently-passing tests for `/db session` (`TestDbHealth` line 463,

## Required Changes

   - Delete `scripts/agent/commands/cmd_config_set.py` (`_ConfigSetMixin`,

## Acceptance Criteria
- `/set temperature ...` and `/set max_tokens ...` no longer exist as commands
  (attempting them produces the same "unknown command" behavior as any other
  non-existent slash command).
- `/session export markdown [file]` and `/session export json [file]` produce
  output identical to what `/export md [file]` and `/export json [file]` produced
  before this change.
- `/session stats|health|checkpoint|vacuum|purge|recover [...]` produce output
  identical to what `/db session stats|health|checkpoint|vacuum|purge|recover [...]`
  produced before this change, with the same argument parsing (`--max-sessions`,
  `--max-age-days`, checkpoint `[MODE]`, recover `[backup-path]`).
- `llm_temperature`/`llm_max_tokens` remain changeable via `config/agent.toml` +
  `/reload`, unaffected by `/set`'s removal.
- No references to `/set`, `/export` (as a top-level command), `/db`, or `/db
  session` remain anywhere in the codebase or docs after the change; the moved
  content appears under `/session` instead.
- `/db` (any subcommand, including `/db help`) no longer exists as a command.
- `tests/test_agent_cmd_db.py` no longer exists; its `/db rag` coverage is gone
  (not fixed-in-place) and its `/db session` coverage is superseded by equivalent
  `/session`-targeting tests in `tests/test_agent_cmd_session.py`.
- The full test suite has no failing tests attributable to the removed `/db rag`
  coverage (they are deleted, not left failing).
- `CommandRegistry.__init__`'s fail-fast handler validation still passes (no
  `CommandDef` references a missing handler).
