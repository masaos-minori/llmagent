# Aggregate startup validation results into a unified validation pipeline

## Priority

Medium

## Type

Design / Implementation / Operations

## Background

Startup validation currently includes multiple independent checks such as:

- LLM and embedding service readiness
- Workflow definition preflight
- Tool definition validation
- Routing drift validation
- Security defaults audit
- MCP server health and tool discovery checks

Each check may emit warnings, skip validation, or raise fatal errors.

## Problem

When startup validation is distributed across many functions, operators may see only the first fatal error and miss other important startup problems.

The rules for what is fatal, warning, or skipped are also harder to reason about.

## Goal

Create a unified startup validation pipeline that aggregates all startup validation results.

## Scope

Affected areas:

- Startup orchestration
- `agent/repl_health.py`
- Workflow preflight checks
- Tool definition validation
- Routing drift validation
- Security audit
- Startup log formatting
- Startup validation tests

## Implementation Tasks

- Introduce a structured result type such as `StartupValidationResult`.
- Represent each validation outcome as one of:
  - fatal
  - warning
  - skipped
  - ok
- Include source check name, message, and remediation hint.
- Run all safe validations before failing startup.
- At the end, raise a combined startup error if fatal items exist.
- Preserve strict mode behavior for fatal checks.
- Make startup logs show a clear validation summary.

## Acceptance Criteria

- Startup validation reports all warnings and fatal items together.
- Fatal validation still prevents startup.
- Non-fatal warnings are preserved.
- Skipped validations explain why they were skipped.
- Operator-facing error messages include remediation hints.
- Existing startup checks are integrated without weakening safety.

## Validation Plan

Add tests for:

- All validations pass.
- Single fatal readiness failure.
- Multiple fatal failures.
- Warnings only.
- Skipped live routing validation due to all servers unreachable in non-strict mode.
- Strict mode routing drift fatal.
- Missing workflow definition in required mode.
- Security audit failure in production mode.

## Risks

Changing startup error formatting may require test updates. Keep underlying failure semantics unchanged.

## Out of Scope

- Changing individual validation policies.
- Removing strict mode.
- Documentation-only updates.
