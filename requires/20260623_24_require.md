# Title
Make polling-based subscribe behavior visible and measurable

## Summary
Improve observability for the current polling-based subscribe model so latency and query cost are visible in normal operations.

## Background
`/subscribe` is implemented using periodic polling, with `poll_interval_ms` loaded from config and defaulting to 500 ms.
The plan also notes that push/WAL-hook behavior is not implemented yet.

## Problem
The current subscribe path works, but its latency and DB polling cost are not visible enough for operators.

## Required Changes
- Surface the effective polling interval in startup or status output.
- Add basic metrics or logs for subscribe query frequency and observed delivery delay.
- Document clearly that the current implementation is polling-based.

## Acceptance Criteria
- Operators can see the configured subscribe polling interval.
- Latency and polling behavior are measurable in runtime diagnostics.
- Documentation no longer makes subscribe look more real-time than it currently is.

## Out of Scope
- Replacing polling with push/WAL-hook behavior in this issue

## Components
- `scripts/eventbus/app.py`
- `scripts/eventbus/config.py`
- Event Bus runtime / operations docs
- Optional observability hooks or metrics layer
