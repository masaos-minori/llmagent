# [db][memory] Document that MemoryDeleteStore Protocol exists for future backend replacement

## Summary
`MemoryDeleteStore` and `SQLiteMemoryDeleteStore` are split into protocol and concrete implementation,
but the reason for that abstraction is not obvious enough from the main docs.

## Problem
The current system is SQLite-based in practice, so the protocol layer can look unnecessary.
Without explanation, developers may not understand why both the protocol and the SQLite implementation exist.

## Scope
- `db/store.py`
- `db/store_protocols.py`
- `db/store_impl.py`
- shared/db documentation

## Task
- document clearly that `MemoryDeleteStore` exists to preserve future backend replacement options
- explain that the current primary implementation is `SQLiteMemoryDeleteStore`
- make it clear that the protocol is an extensibility boundary, not a sign that multiple backends already exist
- keep the SQLite concrete implementation as the practical main path in current docs

## Expected Result
Developers understand why the protocol exists,
and they also understand that SQLite remains the main implementation today.

## Acceptance Criteria
- [ ] docs clearly explain the purpose of the protocol abstraction
- [ ] docs clearly identify `SQLiteMemoryDeleteStore` as the current main implementation
- [ ] the abstraction is easier to understand without reading source code deeply
