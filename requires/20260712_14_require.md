# [Docs][MCP] 04_mcp_03_03_transport-and-health.md がサイズ上限を超過している

## 背景

本プロジェクトのdocsは、LLM/RAGでの参照コストを抑えるため、1ファイルあたり8KB(8192 byte)を
上限として `-partN` 形式に分割する運用を取っている(`tools/validate_docs_structure.py` の
`MAX_SIZE = 8192` で機械的に検証)。`docs/04_mcp_03_03_transport-and-health.md` はこの上限を
超過している。

## 目的

サイズ上限を超過しているファイルを、既存の分割運用(part1/part2)に揃えて分割する。

## 現状(調査結果)

- `uv run python tools/validate_docs_structure.py` を実行すると、以下の1件が検出される:
  - `04_mcp_03_03_transport-and-health.md: size 8721 bytes exceeds 8192 byte limit`
- 実測: 8721 byte、138行(`wc -c` / `wc -l` で確認)。上限を約6.5%超過している。
- 同系列の `04_mcp_03_01_dispatch-and-routing.md`〜`04_mcp_03_05_lifecycle-and-new-server.md` は
  既に分割・命名済みであり、このファイルのみが `part1`/`part2` 分割の対象として残っている。
- 見出し構成(H2)は次の3つ: `HttpTransport (shared/tool_executor.py)`(20行目)、
  `McpServerHealthRegistry (shared/mcp_config.py)`(37行目)、
  `エンドツーエンドのツール呼び出し追跡`(79行目、`### 相関キー`/`### 成功パスの例` を含む)。
  2番目と3番目のH2境界(78/79行目付近)で分割すると、内容のまとまりを保ったまま2分割できる。
- 現在このファイルを参照しているのは以下7ファイル(front matter/本文内の参照を含む):
  `04_mcp_00_document-guide.md`, `04_mcp_02_01_endpoints-and-transport.md`,
  `04_mcp_02_03_audit-logging-and-errors.md`, `04_mcp_03_01_dispatch-and-routing.md`,
  `04_mcp_03_02_tool-registry.md`, `04_mcp_03_04_tool-call-tracing-and-watchdog.md`,
  `04_mcp_03_05_lifecycle-and-new-server.md`

## 提案設計

1. `docs/04_mcp_03_03_transport-and-health.md` の内容を、`McpServerHealthRegistry` 節と
   `エンドツーエンドのツール呼び出し追跡` 節の間(H2境界、78/79行目付近)で
   `04_mcp_03_03_transport-and-health-part1.md`(HttpTransport + McpServerHealthRegistry) /
   `-part2.md`(エンドツーエンドのツール呼び出し追跡)に分割する
   (他ファイルの分割慣習に合わせる)。
2. 分割後、front matter (`related:`/`source:`)を実ファイル名に揃える(別課題「front matter の
   壊れたリンク解消」と同様の不具合を新たに作らないよう注意する。`source:` はpart1のファイル名を
   代表として使う既存規約に合わせる)。
3. 上記7ファイルのうち、当該ファイルへのリンク・参照を含むものを、分割後のファイル名
   (part1/part2、内容に応じて使い分け)に更新する。
