# H-8: Ensure watchdog never observes pending MCP reload config

## Priority

High

## Background

The watchdog loop reads `ctx.cfg.mcp.mcp_servers` live during runtime. If reload mutates MCP server config, watchdog may use pending restart config while ToolExecutor still uses startup transport snapshots.

This must not happen.

## Target files

- `agent/services/config_reload.py`
- `agent/repl_health.py`
- watchdog tests if present

## Implementation tasks

- Ensure MCP reload classification does not mutate `ctx.cfg.mcp.mcp_servers`.
- Keep watchdog behavior based on running config only.
- Do not introduce any pending MCP config overlay into watchdog.
- Document that watchdog restart uses startup-time MCP server configuration.
- Add tests proving reload with changed MCP URL/startup mode does not alter watchdog input config.

## Acceptance criteria

- Watchdog always probes running MCP server URLs.
- Watchdog restart decisions continue to use running startup-mode values.
- Pending restart MCP config does not affect watchdog behavior.
- Reload classification has no side effect on watchdog-visible MCP config.

## Test plan

- Capture `ctx.cfg.mcp.mcp_servers` before reload.
- Reload with changed MCP URL and startup mode.
- Verify watchdog-visible config remains unchanged.
- Verify restart-required entries are still reported.
