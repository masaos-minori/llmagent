## [High][eventbus][architecture] Split subscribe delivery into durable replay from SQLite and live push from memory

### Summary
The current subscribe flow uses SQLite for both replay and live notification.

### Problem
Using one polling path for both replay and live delivery creates avoidable latency and unnecessary database load.

### Scope
- app.py
- subscribe()
- replay path
- offset restore path
- live notification path

### Task
- redesign `/subscribe` as a hybrid model:
  - replay from SQLite until the subscriber catches up
  - then switch to in-memory live delivery
- preserve `since_seq` and `consumer_id` resume semantics
- add a resync path for recovery if the in-memory notification stream is missed
- document the replay-to-live transition clearly
- add integration tests for reconnect and resume

### Expected Result
Durable replay remains correct while live delivery becomes low-latency and push-based.

### Acceptance Criteria
- replay still works from SQLite using `since_seq`
- live delivery no longer depends on SQLite polling once caught up
- reconnect/resume behavior remains correct
- tests cover replay-to-live handoff
