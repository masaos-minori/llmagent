# Title
Harden shell execution defaults for production deployments

## Summary
Change production-oriented shell MCP behavior so that sandboxing is required and unsafe fallback behavior is not silently accepted.

## Background
`shell-mcp` currently defaults to `shell_sandbox_backend = "none"`.
That means commands run with the agent process user and permissions unless an external sandbox is explicitly enabled.

## Problem
This default is too weak for production-grade deployments.
A validated allowlist alone is not a substitute for OS-level isolation.

## Required Changes
- Require a sandbox backend such as `firejail` in production mode.
- Treat `shell_sandbox_backend = "none"` as a production startup failure or equivalent hard error.
- Keep local/development behavior configurable, but make the risk explicit.
- Ensure `/health` or equivalent runtime diagnostics clearly expose the active sandbox backend.

## Acceptance Criteria
- Production deployments cannot run shell execution with `shell_sandbox_backend = "none"` without an explicit override strategy.
- Runtime diagnostics clearly indicate whether sandboxing is active.
- Documentation reflects the stricter production behavior.

## Out of Scope
- Replacing Firejail with a different sandbox technology.
- Redesigning shell command allowlist semantics.

## Notes
Favor explicit secure failure over permissive fallback in production.
