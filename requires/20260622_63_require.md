# Title
Add a forward-looking design note for stdio transport scalability limits

## Summary
Document the current scalability limits of stdio transport and outline future options if stdio-based servers become more common.

## Background
Current stdio transport behavior serializes calls per transport instance.
This is acceptable today, but it may become a bottleneck if stdio transport usage grows.

## Problem
Without a documented note, future scaling pain may arrive without any shared design expectation.

## Required Changes
- Add a design note describing the current serialization behavior of stdio transport.
- Describe potential future options such as:
  - worker parallelism
  - multiplexed request handling
  - transport redesign
- Keep this as a planning note, not as a current redesign commitment.

## Acceptance Criteria
- The current stdio transport limitation is documented clearly.
- Future maintainers know what scaling options exist.
- No immediate behavior change is introduced unless explicitly requested elsewhere.

## Out of Scope
- Implementing multiplexed stdio now
- Replacing stdio transport
