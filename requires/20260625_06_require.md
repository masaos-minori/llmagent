## Summary
Several MCP tools support `dry_run` at the implementation/model level, but the LLM-exposed `/v1/tools` input schema is missing that argument.

## Problem
If `dry_run` is not exposed in `inputSchema`, the LLM cannot use it directly, and the safety model that depends on dry-run previews becomes inconsistent.

## Scope
- file-write-mcp
  - write_file
  - move_file
- file-delete-mcp
  - delete_file
  - delete_directory
- cicd-mcp
  - trigger_workflow

## Facts
- `file-write-mcp` documents `dry_run` support for write and move operations.
- `file-delete-mcp` documents `dry_run` support for delete operations.
- `cicd-mcp` documents `trigger_workflow` as supporting `dry_run`.
- The agent approval flow depends on `approval_dry_run_tools` for pre-execution preview behavior.

## Task
- update the `/v1/tools` metadata generation for the affected servers
- ensure `dry_run` appears in the exposed `inputSchema` for all affected tools
- verify that the schema matches the actual request model accepted by the server
- add regression tests for `/v1/tools` output

## Expected Result
The affected tools expose `dry_run` consistently in `/v1/tools`, and the LLM can use the feature safely.

## Acceptance Criteria
- `write_file` input schema includes `dry_run`
- `move_file` input schema includes `dry_run`
- `delete_file` input schema includes `dry_run`
- `delete_directory` input schema includes `dry_run`
- `trigger_workflow` input schema includes `dry_run`
- tests fail if any of these fields disappear again
