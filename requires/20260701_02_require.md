## Issue 2: [High] Document physical removal policy for legacy notes table

### Goal

Remove backward-compatibility wording around the legacy `notes` table and document the final policy: existing databases should remove it manually with SQL.

### Background

The current session schema does not create a `notes` table. The current `session.sqlite` schema includes session, message, tool result, memory, and diagnostic tables only.

### Scope

- Do not add a new CLI command.
- Do not add `DbMaintenanceService.drop_legacy_notes_table()`.
- Do not modify `cmd_db.py`.
- Update documentation to state that `notes` is not part of the current schema.
- Provide direct SQL for existing DB cleanup.

### Implementation Tasks

- Remove wording such as:
  - `existing databases retain the table for backward compatibility`
  - `notes table retained for compatibility`
- Add the following manual cleanup SQL to the relevant operations or data-layer documentation:

```sql
DROP TABLE IF EXISTS notes;
```

- If old deployments may have created FTS-related note tables, instruct operators to inspect the DB explicitly before dropping additional objects.
- Do not add speculative DROP statements unless the old schema is verified.
- Confirm that `schema_sql.py` remains unchanged unless a hidden `notes` definition is found elsewhere.

### Acceptance Criteria

- Active docs clearly state that the legacy `notes` table is not part of the current `session.sqlite` schema.
- Existing DB cleanup is documented as a manual SQL operation.
- No `/db session drop-legacy-notes` command is introduced.
- No new maintenance service method is introduced for this one-time cleanup.

### Validation

- Confirm `init_db.sh` expected session tables do not include `notes`.
- Grep active docs for stale `notes` backward-compatibility wording.
