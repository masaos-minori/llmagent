# [High] Reconcile MCP server catalog counts and startup-mode semantics

## Labels
priority/high, area/mcp, area/catalog, area/lifecycle, documentation

## Summary
Reconcile MCP server count, tool count, and startup-mode descriptions across the MCP documentation set.

## Background
The overview states that there are 11 MCP servers and 66 tools, while the visible server catalog lists fewer active servers and the tool total does not consistently match.

The catalog also lists startup modes such as `persistent`, while lifecycle documentation defines `none`, `persistent`, and `subprocess`, with omitted `startup_mode` defaulting to `none`.

## Required Changes

### 1. Server and tool counts
- Recount active MCP servers from implementation and config.
- Recount MCP tools from `shared/tool_constants.py` and ToolRegistry.
- Update all documentation to use the same counts.
- Remove stale server entries if a server no longer exists.
- Add missing server catalog entries if an active server is omitted.

### 2. Startup-mode semantics
Clarify that catalog startup-mode values are deployment/configured values, not universal defaults.

Document mode definitions:

- `subprocess`: agent-managed HTTP server; agent starts it and may restart it
- `persistent`: externally managed HTTP server; agent connects and probes it
- `none`: disabled; tool calls are rejected before network access

### 3. Current deployment values
Update catalog startup modes to match current `config/agent.toml`.

If a server is disabled, explicitly mark it as `startup_mode="none"`.

## Acceptance Criteria
- MCP server count is consistent across guide, overview, and catalog.
- MCP tool count matches ToolRegistry or `shared/tool_constants.py`.
- Server catalog and system overview agree.
- Startup mode documentation distinguishes default behavior from configured deployment values.
- `startup_mode="none"` is documented as disabled.
- Operators understand when to use `subprocess` versus `persistent`.
