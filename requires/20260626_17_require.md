# [High] Redesign MDQ SQLite schema for production indexing

## Priority
High

## Labels
`mcp`, `mdq`, `sqlite`, `schema`, `fts5`, `migration`

## Dependencies
- Replace line-based Markdown parser with hierarchy-aware parser

## Background
The current schema only stores `sections(id, file_path, heading, content, file_mtime)`. Production MDQ requires document metadata, stable chunk IDs, heading paths, line ranges, token counts, hashes, and FTS consistency.

## Task
Create a production MDQ schema and migration path.

## Implementation Requirements
Add or migrate to production tables such as:
- `documents`
- `chunks`
- `chunks_fts`
- optional `index_state`

Each chunk must store:
- stable `chunk_id`
- `doc_id`
- `source_path`
- `heading`
- `heading_path`
- `heading_level`
- `ordinal`
- `content`
- `normalized_content`
- `start_line`
- `end_line`
- `char_count`
- `token_count`
- `content_hash`
- `tags_json`
- `indexed_at`

Each document must store:
- `doc_id`
- `source_path`
- `mtime_ns`
- `size_bytes`
- `content_hash`
- `indexed_at`

## Acceptance Criteria
- Existing minimal schema can be migrated or rebuilt safely.
- FTS5 table is synchronized with chunks.
- Schema supports stable chunk lookup.
- Schema supports incremental indexing.
- `stats` can report document, chunk, FTS, and stale counts.
