# Title
Clarify RAG-owned database boundaries and remove Agent session ownership ambiguity

## Priority
High

## Labels
- bug
- rag
- database
- architecture
- documentation
- ownership-boundary

## Dependencies
- None

## Summary
Clarify that the RAG layer owns only RAG tables and does not own Agent session tables. Remove or rewrite any wording that implies `sessions` or `messages` are part of the RAG-owned schema.

## Background
The RAG system overview defines Agent REPL as out of scope and says the Agent uses RAG through MCP. However, the RAG data model still contains wording that can be read as Agent session tables residing in the same SQLite file for operational convenience.

## Problem
This creates a dangerous ownership ambiguity. AI implementers may incorrectly add or maintain Agent session tables from the RAG schema or migration path.

## Required Changes
- Inspect the actual schema initialization and migration code.
- Confirm whether `rag.sqlite` is RAG-only or physically shares a SQLite file with Agent session tables.
- Define RAG-owned tables strictly as:
  - `documents`
  - `chunks`
  - `chunks_fts`
  - `chunks_vec`
- Remove wording that implies RAG owns `sessions` or `messages`.
- If Agent session tables are mentioned, keep only a short cross-reference to the Agent data-layer documentation.
- Update `03_rag_90_inconsistencies_and_known_issues.md` by deleting the resolved/stale SPEC entry after the docs are corrected.
- Add or update tests to ensure RAG schema initialization does not require Agent session tables.

## Acceptance Criteria
- RAG docs no longer imply that Agent session tables are RAG-owned.
- RAG schema initialization succeeds without creating or requiring `sessions` or `messages`.
- Any Agent session schema reference is clearly marked as external to the RAG layer.
- The stale known-issues entry is removed after the correction is complete.

## Out of Scope
- Changing the Agent session schema.
- Moving Agent persistence responsibilities into RAG.
- Redesigning the shared database architecture unless the implementation proves the docs are wrong.
