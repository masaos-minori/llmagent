## Title: [MEDIUM] Fix FTS5 Index Synchronicity and Remove Manual Rebuild Reliance

### Context
`rag.sqlite` relies on SQLite database triggers (`chunks_ai`, `chunks_au`, `chunks_ad`) to automatically sync data from the `chunks` table into the `chunks_fts` virtual table. However, documentation points out that counts can mismatch, requiring manual invocation of `/db rebuild-fts`.

### Problem Statement
Relying on manual `/db rebuild-fts` command implies that bulk operations or error rollbacks are bypassing or mismanaging triggers, leading to data degradation over time.

### Requirements
- **Root Cause Analysis & Fix**: Investigate how data insertion/updates (e.g., during bulk ingestion) are handled. Fix any scenarios (such as uncaught exceptions or transaction boundary issues) causing `chunks_fts` to drift out of sync.
- **Automated Validation**: Add an automated assertion or periodic lightweight check within `RagMaintenanceService` to log a SEVERE warning if `SELECT COUNT(*) FROM chunks` does not match `chunks_fts`.

### Constraints
- Do NOT drop and rebuild the index on every insert. The fix must enforce incremental consistency via correct transactional/trigger behavior.

### Acceptance Criteria (Definition of Done)
- [ ] Bulk indexing workflows leave the database in a state where `chunks` and `chunks_fts` row counts are perfectly identical.
- [ ] Unit tests covering aborted transactions confirm that FTS5 records rollback accurately alongside standard records.
