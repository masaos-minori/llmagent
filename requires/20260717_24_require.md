# [Medium] Refactor web-search-mcp into service, provider, formatter, health, and metrics layers

## Priority

Medium

## Summary

Refactor `web-search-mcp` to separate provider execution, business logic, formatting, health, metrics, and HTTP/audit concerns.

This issue should preferably be implemented after the behavior fixes in Issues 1 to 3.

## Target files

- `scripts/mcp_servers/web_search/service.py`
- `scripts/mcp_servers/web_search/formatters.py`
- `scripts/mcp_servers/web_search/search_provider.py`
- `scripts/mcp_servers/web_search/health.py`
- `scripts/mcp_servers/web_search/metrics.py`
- `scripts/mcp_servers/web_search/server.py`
- `tests/test_web_search_service.py`
- `tests/test_web_search_formatters.py`
- `tests/test_web_search_provider.py`
- `tests/test_web_search_health.py`
- `tests/test_web_search_metrics.py`

## Reason for change

`formatters.py` currently owns too many responsibilities:

- request model construction
- provider call
- latency measurement
- logging
- empty-result handling
- response formatting
- dispatch table definition

This makes it harder to add timeout, error classification, audit, health state, and metrics without creating coupling.

## Intent

Introduce clearer internal boundaries.

Recommended layering:

- `server.py`
  - HTTP endpoints
  - exception handling
  - audit

- `service.py`
  - request validation
  - provider invocation
  - latency measurement
  - empty-result semantics
  - metrics updates
  - health state updates

- `search_provider.py`
  - DuckDuckGo-specific provider interaction

- `formatters.py`
  - `SearchResponse` to LLM-readable text

- `health.py`
  - provider health state

- `metrics.py`
  - lightweight counters and latency aggregation

## Required changes

### 1. Add service layer

Create `service.py` for orchestration.

Responsibilities:

- build or accept `SearchRequest`
- call provider
- measure latency
- handle zero-result success
- update metrics
- update health state
- return structured result and metadata as needed

### 2. Simplify formatters.py

Keep formatter responsibilities focused on converting structured search responses to text.

Avoid provider-specific operational logic in formatters.

### 3. Keep DuckDuckGo-specific code in provider layer

`search_provider.py` should be the only layer that directly depends on DuckDuckGo-specific library behavior.

Other layers should depend on domain-level provider functions or a provider interface.

### 4. Optional SearchProvider interface

Introduce a small interface or protocol if helpful.

Do not add additional external providers in this issue.

## Acceptance criteria

- Existing behavior remains unchanged except for fixes from Issues 1 to 3.
- `formatters.py` no longer owns provider execution logic.
- Provider behavior can be tested independently.
- Formatter behavior can be tested with static `SearchResponse`.
- Health and metrics updates are triggered from service layer.
- Server remains responsible for audit and HTTP exception mapping.
- No circular imports are introduced.

## Test plan

Add or update tests for:

- service success
- service zero-result success
- service provider timeout
- service provider error
- formatter normal output
- formatter zero-result output
- provider-level error translation
- server-level audit and exception mapping

## Notes for implementer

Keep commits separated if possible:

1. behavior fixes
2. observability changes
3. refactor

This makes review and rollback easier.
