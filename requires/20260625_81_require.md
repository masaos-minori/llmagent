## [High][rag][types] Remove the legacy `PipelineStageResult` type and standardize on `StageResult` only

### Summary
The docs explicitly state that `PipelineStageResult` is a legacy type retained for backward compatibility.
They also state that the current pipeline uses `StageResult` from `scripts/rag/stage.py`. 【2-222f47】【1-f0779f】

### Problem
Keeping both the legacy and current stage-result types creates duplicate semantics for one concept.
This complicates typing, diagnostics, tests, and future schema changes.

### Scope
- `scripts/rag/types.py`
- `scripts/rag/stage.py`
- pipeline diagnostics paths
- docs that reference `PipelineStageResult`
- tests using the old type

### Task
- remove the `PipelineStageResult` legacy type
- make `StageResult` the only supported stage-result representation
- migrate all references from `PipelineStageResult` to `StageResult`
- update diagnostics, debug output, and docs to use one result model only
- remove or rewrite tests that still depend on the legacy type

### Expected Result
There is one canonical stage-result type in the RAG system: `StageResult`. 【2-222f47】【1-f0779f】

### Acceptance Criteria
- `PipelineStageResult` is removed
- all code paths use `StageResult`
- docs no longer describe `PipelineStageResult` as available
- tests no longer depend on the legacy type
