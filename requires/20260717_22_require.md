# [High] Fix provider execution semantics, empty result handling, timeout, and error classification

## Priority

High

## Summary

Improve provider execution behavior for `search_web`.

This issue combines the following work:

- Treat empty result sets as successful zero-result responses
- Add explicit MCP-level timeout around DuckDuckGo calls
- Introduce classified web search errors
- Preserve proper provider failure behavior
- Add provider and formatter tests

## Target files

- `scripts/mcp_servers/web_search/search_provider.py`
- `scripts/mcp_servers/web_search/formatters.py`
- `scripts/mcp_servers/web_search/models.py`
- `config/web_search_mcp_server.toml`
- `tests/test_web_search_provider.py`
- `tests/test_web_search_formatters.py`
- `tests/test_web_search_server.py`

## Reason for change

The current formatter layer has a branch that appears intended to return `No search results found.` when a search returns zero results.

However, the lower-level search function raises `WebSearchUpstreamError` when the provider result list is empty.

This incorrectly treats a valid zero-result search as an upstream provider failure.

In addition, the current provider implementation executes DuckDuckGo search through `asyncio.to_thread()`, but there is no explicit MCP-level timeout around the full provider call.

Provider failures are also aggregated into a broad upstream error, making operational diagnosis imprecise.

## Intent

Make provider execution semantics precise:

- empty result set means successful zero-result response
- timeout means classified timeout error
- network/provider failure means classified provider error
- malformed provider output means parse/provider error
- all cases are testable independently

## Required changes

### 1. Treat empty result list as success

Remove the behavior that raises `WebSearchUpstreamError` for an empty result list.

Return a successful response with an empty results list.

The formatted output should remain:

- `No search results found.`

### 2. Add explicit provider timeout

Add config key:

- `search_timeout_sec`

Recommended default:

- `10.0`

Validate that the value:

- is numeric
- is greater than 0
- does not exceed a reasonable upper bound

Wrap the provider call with `asyncio.wait_for()`.

### 3. Add specific domain exceptions

Add domain-level exceptions such as:

- `WebSearchTimeoutError`
- `WebSearchNetworkError`
- `WebSearchProviderError`
- `WebSearchParseError`

Keep `WebSearchUpstreamError` only as a base class or compatibility wrapper if needed.

### 4. Validate provider result shape

Handle malformed provider data before constructing `SearchResult`.

Decide and test behavior for:

- missing title
- missing URL
- missing body
- invalid result object
- provider-specific unexpected response

## Acceptance criteria

- Empty provider result returns successful zero-result response.
- Empty result does not produce HTTP 502.
- Formatted result for zero results is `No search results found.`
- Provider timeout is enforced by MCP code.
- Timeout raises a classified timeout error.
- Provider runtime failure is distinguishable from timeout.
- Malformed provider response is classified.
- Existing successful search formatting remains unchanged.

## Test plan

Add tests for:

- provider returns one result
- provider returns empty list
- provider call exceeds timeout
- provider raises runtime error
- provider returns malformed result object
- formatter output for zero results
- formatter output for normal results

## Notes for implementer

Cancelling `asyncio.to_thread()` does not necessarily kill the underlying thread immediately. The MCP request should still return promptly.

Do not turn empty result sets into exceptions.
