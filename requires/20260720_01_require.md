# [High] Unify routing authority: RuntimeToolRegistry becomes sole routing source

## Priority

High

## Summary

Eliminate the dual-routing-layer ambiguity in `ToolRouteResolver.resolve()` by making `RuntimeToolRegistry` the sole routing authority and moving `ToolRegistry` to a pure drift-detection role.

Currently `resolve()` checks both registries sequentially:

```python
def resolve(self, tool_name: str) -> str:
    if (key := self._lookup_runtime_registry(tool_name)) is not None:
        return key
    if (key := self._lookup_legacy_registry(tool_name)) is not None:
        return key
    raise ValueError(f"Unknown tool: {tool_name!r}")
```

This creates four problems:

1. **Ambiguous fallback condition**: It is unclear whether "fallback to legacy" means "RuntimeToolRegistry is unavailable" or "the specific tool is absent from RuntimeToolRegistry". Both paths produce the same result (legacy lookup succeeds), but they have different operational meanings.

2. **Inconsistent error messages**: The `strict_mode` error message references "frozenset in `shared/tool_constants.py`", which only applies to `ToolRegistry`, not `RuntimeToolRegistry`. After this change the message would need to differ based on which registry was consulted — a fragility.

3. **State loss on wiring**: `set_runtime_registry()` replaces the entire resolver instance (`tool_executor.py:76-84`), discarding `discovery_map`, `warn_on_missing`, and other constructor parameters. This is a hidden side effect.

4. **Documentation inconsistency**: Docs describe two registries but do not explain when each is authoritative vs fallback.

## Reason for change

The current design treats both registries as equally valid routing sources, but their roles are fundamentally different:

| Aspect | RuntimeToolRegistry | ToolRegistry |
|---|---|---|
| Data source | Live /v1/tools discovery | Compile-time frozensets |
| Data type | RuntimeTool (rich metadata) | ToolDefinition (name + server_key) |
| Purpose | Routing + runtime metadata | Drift detection + seed data |
| Availability | Populated after discovery | Always available |

Routing via `ToolRegistry` is a backward-compatible convenience, not a primary feature. Once `McpToolDiscoveryService.discover_all()` completes successfully, `RuntimeToolRegistry` contains all known tools. The legacy path exists only for edge cases where discovery fails entirely.

## Intent

After this change:

- `RuntimeToolRegistry` is the sole routing authority for `resolve()`.
- `ToolRegistry` is removed from the resolution chain and used exclusively for drift detection.
- Error messages reference only `RuntimeToolRegistry`.
- `set_runtime_registry()` updates the existing resolver instead of replacing it.
- All docs reflect the unified authority model.

## Required changes

### 1. Simplify ToolRouteResolver.resolve()

Remove `_legacy_registry` field and `_lookup_legacy_registry()` method. Make `RuntimeToolRegistry` the only routing source:

```python
class ToolRouteResolver:
    def __init__(
        self,
        server_configs: dict[str, McpServerConfig],
        *,
        discovery_map: dict[str, str] | None = None,
        warn_on_missing: bool = False,
        strict_mode: bool = False,
        known_tools: frozenset[str] | None = None,
        runtime_registry: RuntimeToolRegistry | None = None,
    ) -> None:
        # ... validation_data ...
        self._runtime_registry: RuntimeToolRegistry | None = runtime_registry
        # No longer initialize _legacy_registry
        # ...

    def resolve(self, tool_name: str) -> str:
        if (key := self._lookup_runtime_registry(tool_name)) is not None:
            return key
        if self._strict_mode:
            self._raise_strict_error(tool_name)
        if self._warn_on_missing:
            logger.warning(
                "ToolRouteResolver: tool %r not found; "
                "ensure MCP servers are healthy and discovery completed.",
                tool_name,
            )
        raise ValueError(f"Unknown tool: {tool_name!r}")

    def _lookup_runtime_registry(self, tool_name: str) -> str | None:
        if self._runtime_registry is not None:
            return self._runtime_registry.resolve(tool_name)
        return None
```

### 2. Update set_runtime_registry() to mutate instead of replace

Change `ToolExecutor.set_runtime_registry()` to update the existing resolver's internal state rather than creating a new resolver instance:

```python
def set_runtime_registry(self, registry: RuntimeToolRegistry) -> None:
    """Wire RuntimeToolRegistry into the resolver after discovery completes."""
    if hasattr(self._resolver, 'set_runtime_registry'):
        self._resolver.set_runtime_registry(registry)
    else:
        # Fallback: replace resolver (backward compat for old callers)
        self._resolver = ToolRouteResolver(
            self._server_configs,
            discovery_map={},
            runtime_registry=registry,
        )
```

Add a corresponding method to `ToolRouteResolver`:

```python
def set_runtime_registry(self, registry: RuntimeToolRegistry) -> None:
    """Update the runtime registry without recreating the resolver."""
    self._runtime_registry = registry
```

### 3. Fix error messages

Update `strict_mode` error message to reference `RuntimeToolRegistry` instead of "any registry" and remove the frozenset hint:

```python
def _raise_strict_error(self, tool_name: str) -> None:
    raise ValueError(
        f"ToolRouteResolver: tool {tool_name!r} not found in RuntimeToolRegistry "
        f"and strict_mode=True; check MCP server health and discovery status"
    )
```

### 4. Remove legacy registry references from diagnostics

Update `_log_routing_coverage()` to no longer check the legacy registry:

```python
def _log_routing_coverage(self, known_tools: frozenset[str]) -> None:
    mapped: list[str] = []
    unmapped: list[str] = []
    for tool_name in sorted(known_tools):
        if self._lookup_runtime_registry(tool_name) is not None:
            mapped.append(tool_name)
        else:
            unmapped.append(tool_name)
    # ... rest unchanged
```

### 5. Update startup.py wiring

No functional change needed — `startup.py` already calls `set_runtime_registry()` after discovery. The difference is that now the resolver mutation is clean (no instance replacement).

### 6. Update cmd_mcp.py diagnostics display

The `/mcp status` command already displays RuntimeToolRegistry diagnostics alongside the MCP server table. No change needed unless we want to clarify the display to indicate this is the sole routing authority.

### 7. Update all documentation

- Remove all references to "two-layer routing" or "fallback to ToolRegistry"
- Clarify that `ToolRegistry` is used for drift detection only
- Update diagrams showing routing flow
- Remove "unimplemented" annotations from docs that previously noted un-wired RuntimeToolRegistry

## Acceptance criteria

- `ToolRouteResolver.resolve()` raises `ValueError` when `RuntimeToolRegistry` is None and the tool is unknown.
- `ToolRouteResolver.resolve()` returns correct server key when `RuntimeToolRegistry` is populated.
- `ToolRouteResolver.resolve()` falls through to legacy `ToolRegistry` when `RuntimeToolRegistry` is None and the tool exists there.
- `set_runtime_registry()` does not replace the resolver instance (verified by checking resolver identity before/after).
- `strict_mode` error message references `RuntimeToolRegistry`, not "any registry" or "frozenset".
- All existing tests pass (114 tests).
- All docs updated to reflect single-authority model.

## Notes for implementer

### Migration risk assessment

Low risk because:
- `RuntimeToolRegistry` is always populated after discovery completes (which happens before any tool call)
- The legacy fallback path already handles the case where `RuntimeToolRegistry` is None
- No production caller passes `known_tools` to `__init__`, so `_log_routing_coverage()` is not currently active

### What NOT to change

- Do not modify `McpToolDiscoveryService` — it already builds and returns `RuntimeToolRegistry` correctly
- Do not modify `startup.py` wiring logic — it already calls `set_runtime_registry()`
- Do not remove `ToolRegistry` entirely — it is still needed for drift detection

### Future work

After this change is complete, consider:
1. Deprecating `_legacy_registry` parameter in `ToolRouteResolver.__init__()`
2. Removing `discovery_map` parameter (it was originally intended for legacy routing)
3. Adding a deprecation warning when legacy registry is consulted
