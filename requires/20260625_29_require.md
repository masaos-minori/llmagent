## [Medium][eventbus][offsets] Separate delivered offsets from acknowledged offsets

### Summary
The current offset model stores stream position, not confirmed successful processing position.

### Problem
A consumer can advance the saved offset after receiving an event even if processing has not succeeded.
This makes offset persistence unsuitable as a reliable completion signal.

### Scope
- app.py
- offsets.py
- subscribe flow
- checkpoint semantics
- delivery acknowledgement model

### Task
- redesign offset semantics so that durable offsets reflect successful processing
- either:
  - store separate delivered and acknowledged offsets
  - or persist offsets only after explicit acknowledgement
- update subscribe/checkpoint behavior to match the new semantics
- add tests for crash recovery and partial processing cases

### Expected Result
Offset persistence no longer confuses “delivered” with “successfully processed”.

### Acceptance Criteria
- offset semantics are explicitly defined
- successful processing and mere delivery are not conflated
- crash recovery behavior is tested
- offset persistence matches the chosen ack model
