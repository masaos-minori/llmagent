# Update memory branch behavior documentation to match implementation

## Priority

Medium

## Labels

docs, memory, retrieval, consistency

## Background

Memory retrieval uses branch information in two ways:

1. As a retrieval filter
2. As a scoring signal

The current implementation filters retrieval results when a non-empty branch is provided. It includes:

- global memories where `branch = ''`
- memories whose branch matches the current branch

Other branch-specific memories are excluded.

After retrieval, FTS hits are rescored with context matching. A branch match receives a stronger context boost than project or repo match.

Some documentation still describes branch behavior as ranking-only. That is not accurate for the current implementation.

## Goal

Update memory documentation so it accurately describes branch filtering and scoring behavior.

## Scope

### In scope

- Update memory documentation
- Update known-issues documentation if it still says branch is ranking-only
- Add or update tests for branch-aware memory retrieval if missing

### Out of scope

- Changing branch retrieval behavior
- Changing scoring constants
- Changing memory storage schema

## Correct behavior to document

When `branch` is non-empty:

- Include memories with `branch = ''`
- Include memories with `branch = current branch`
- Exclude memories from other branches
- Apply context boost during rescoring

Branch match boost is stronger than project or repo match boost.

## Acceptance criteria

- Documentation no longer states that branch affects ranking only
- Documentation describes global memory inclusion
- Documentation describes current-branch filtering
- Documentation describes context boost
- Tests cover:
  - empty branch query
  - current branch query
  - global memory inclusion
  - other branch exclusion

## Validation

Review memory docs and known-issues docs.

Run memory retrieval tests if available.

## Affected files

- Memory documentation

- Optional memory retrieval tests
