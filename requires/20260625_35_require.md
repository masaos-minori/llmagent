## [High][eventbus][subscribe] Replace polling-based live SSE delivery with in-memory push notification

### Summary
The current `/subscribe` endpoint behaves like SSE externally, but internally it uses repeated SQLite polling.

### Problem
Polling-based live delivery introduces unnecessary latency and scales poorly with subscriber count.
Subscribers should be notified immediately when a new event is published.

### Scope
- app.py
- subscribe flow
- live delivery path
- in-memory subscriber notification layer

### Task
- introduce an in-memory Pub/Sub layer for live event notification
- stop using SQLite polling as the primary live-delivery mechanism
- notify active subscribers immediately after publish succeeds
- keep the delivery contract compatible with existing SSE clients
- add a feature flag only if a phased rollout is necessary

### Expected Result
Live subscribers receive new events immediately without waiting for polling intervals.

### Acceptance Criteria
- `/subscribe` no longer depends on periodic polling for normal live delivery
- new events can be pushed to active subscribers immediately
- SSE client behavior remains compatible
- tests verify immediate live delivery
