# Unify strict tool-definition validation behavior across MCP docs

## Summary
Consolidate the behavior of `tool_definitions_strict` and live `/v1/tools` validation into one coherent specification.

## Background
The current documentation spreads startup-validation behavior across:
- `04_mcp_90_inconsistencies_and_known_issues.md`
- `04_mcp_06_configuration_and_operations.md`
- `04_mcp_03_routing_lifecycle_and_execution.md`

This makes it difficult to determine the actual startup-failure contract.

## Objective
Define one authoritative behavior model for:
- tool name mismatch
- partial server unreachability
- full server unreachability
- strict vs non-strict mode behavior

## In Scope
- `04_mcp_90_inconsistencies_and_known_issues.md`
- `04_mcp_06_configuration_and_operations.md`
- `04_mcp_03_routing_lifecycle_and_execution.md`

## Required Changes
- Move normative strict-validation behavior into one primary document
- Keep the known-issues file as a status tracker, not a second spec source
- Ensure all documents describe the same behavior for:
  - mismatch
  - partial unreachable
  - all unreachable
  - validation skipped

## Acceptance Criteria
- A reader can determine startup behavior from one primary source
- The known-issues file no longer duplicates normative startup behavior
- Strict-mode semantics are internally consistent across docs

## Out of Scope
- Rewriting the underlying tool-definition validation implementation
- Adding new validation modes

## Notes
Use a behavior table. Avoid ambiguous phrasing such as “may fail” if the intended behavior is deterministic.
